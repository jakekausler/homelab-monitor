1. STAGE-001-012 — First built-in host collector plugins metrics

STAGE-001-012 is the proof that the plugin framework laid down in STAGE-001-006 through 011 can host a real, useful collector. A single in-process BaseCollector emits ~145 host metrics per tick using psutil, fed through a memory-retaining writer that keeps the latest value per (name, label-set) so a dashboard tile can read current state without hitting an external time-series store.

Sourceapps/monitor/homelab_monitor/plugins/collectors/builtin/host.py

HostCollector.run() dispatches to ten private helpers (_collect_cpu, _collect_load_average, _collect_memory, _collect_swap, _collect_disk_usage, _collect_disk_io, _collect_net_io, _collect_uptime, _collect_process_states, _collect_top_processes). Each helper catches its own psutil errors and returns a (metrics_emitted, errors) tuple; one bad section never kills the whole tick. KeyboardInterrupt and SystemExit are deliberately not caught so cooperative shutdown is respected.

The retaining writer (D1, D4)

MemoryRetainingMetricsWriter(InMemoryMetricsWriter) in kernel/plugins/io.py extends the append-only writer from STAGE-001-006 with a latest-value-by-(name, frozen-labels) dict and a replace_family(name, entries) method. write_* calls auto-stamp ts = utc_now_iso() at write time (D3 — the collector does not pass ts). For top-N process families, replace_family() atomically wipes prior entries for that family from the latest-value map and writes the new top-N — mirrors how topk() would behave on real VM (which arrives in STAGE-001-015) and bounds snapshot size to ≤ 2 × top_n_processes.

    {"name": "homelab_host_cpu_percent", "value": 12.5,
     "labels": {"cpu": "all"}, "kind": "gauge", "ts": "..."},

Auth: Depends(require_session()) — cookie-only; no token; CSRF not enforced for GET. Top-level ts is the snapshot capture time; each entry carries its own ts (most recent write for that (name, labels) tuple). The Overview tile in STAGE-001-014 filters with entries.find(e => e.name === 'homelab_host_cpu_percent' && e.labels.cpu === 'all'). Pydantic models are MetricsSnapshotEntry + MetricsSnapshotResponse in kernel/api/schemas.py with extra="forbid".

First-tick caveat: psutil.cpu_percent(interval=None) returns 0.0 on the first call after module import (no prior delta to compare). For ~10 seconds after process boot every CPU metric reports 0. Documented in _collect_cpu; consumers must tolerate it.

2. STAGE-001-013 — Alert ingestor + first inproc-dashboard channel alerts api

STAGE-001-013 builds the alert pipeline that every later stage feeds into: an Alertmanager v2 webhook receiver with fingerprint-based dedup, an alert repository with cursor-paginated listing, an outcome tracking surface (ack / dismiss), and the first dispatcher channel that fans alerts out to the SSE broker. It also closes the STAGE-001-008 quarantine TODO by emitting a synthetic alert when a collector enters quarantine.

The five endpoints

MethodPathAuthStatusNotes
POST/api/alerts/ingestcookie+CSRF OR token w/alerts:ingest:write202Returns {received, ingested}
GET/api/alertssession200Cursor pagination; filters: status, severity, source_tool, fingerprint; total omitted by design
GET/api/alerts/{id}session200 / 404Includes outcome history + raw payload
POST/api/alerts/{id}/acksession+CSRF200Inserts ACKED outcome row + sets ack_at/ack_by

Dedup semantics (D4)

While alerts.resolved_at IS NULL, a matching fingerprint UPDATEs last_seen_at only — opened_at is unchanged and severity / labels / annotations are pinned at first fire (severity escalations on the same fingerprint do NOT propagate; upstream rules should encode severity in the fingerprint hash if escalation matters). On the resolution payload, resolved_at is set. On the NEXT firing of the same fingerprint with the prior row already resolved, a fresh row is INSERTed. Each fire→resolve→fire cycle is operationally a new incident; outcome history attaches to its row, not aggregated across cycles.

Migration 0005 (additive)

Adds 10 columns to alerts (source_tool, severity, status, opened_at, last_seen_at, resolved_at, ack_at, ack_by, runbook_id, payload_json) and 3 to alert_outcomes (outcome, decided_at, decided_by), plus indexes on (source_tool, opened_at), (status, opened_at), and alert_outcomes.alert_id. Tables were empty stubs from migration 0001; no backfill needed.

Dispatcher contract (D3)

AlertDispatcher at apps/monitor/homelab_monitor/kernel/dispatch/dispatcher.py iterates a static channel list constructed in lifespan startup. Per-channel delivery failures are caught, logged at WARNING with channel kind, and surfaced as DeliveryResult(ok=False, error=str(exc)). A per-kind failure counter (delivery_failures property) tracks hits for future metrics surfacing. The dispatcher NEVER raises — one bad channel cannot break others. Static channel list at boot keeps STAGE-014 consumers wired through a single /api/events connection that filters by topic.

Logging discipline: the ingestor MUST NOT log full alert payloads at INFO level (label dicts may carry hostnames, internal IPs, secrets). Only fingerprints + counts at INFO; full payloads land in alerts.payload_json for operator inspection via GET. A test asserts no INFO log record contains a full label dict.

5. STAGE-001-016 — VictoriaLogs + vector logs

STAGE-001-016 stands up the log pipeline. VL replaces the in-memory logs writer; vector tails Docker container stdout/stderr (and optionally journald) and ships to VL with per-stream throttle. After this stage every container's logs are queryable via LogsQL, and a planted message in any container is findable.

The MultiplexLogsWriter (D1)

Mirrors STAGE-001-015's metrics multiplex pattern verbatim. Keep InMemoryLogsWriter AND add new VictoriaLogsWriter at apps/monitor/homelab_monitor/kernel/logs/vl_writer.py; MultiplexLogsWriter in kernel/logs/multiplex.py fans every ingest() call to BOTH writers (in-memory first, then VL HTTP). In-memory writer remains the source for any future snapshot/recent-events endpoints with zero VL round-trip; VL writer is best-effort (HTTP failures logged but never fail the collector).

VictoriaLogsWriter internals

Bridges the sync LogsWriter Protocol to async HTTP via a bounded asyncio.Queue(maxsize=10000) + a single background worker started by lifespan. ingest(stream, line, ts=None) is non-blocking put_nowait; on QueueFull the line is dropped, dropped_count is incremented, a warning is logged. The worker waits up to batch_timeout_s=1.0 for the first event, then drains up to batch_size=100 without waiting, then POSTs the batch as NDJSON to /insert/jsonline.

VL event fieldSource
_msgThe log line text
_stream_idThe stream argument
_timeRFC3339 UTC; auto-stamped via utc_now_iso() if not supplied
VL only accepts RFC3339 timestamps. Refinement of STAGE-001-018 caught a bug in vl_planter.py that used epoch milliseconds for _time; VL v0.30.0 returned cannot unmarshal rfc3339 timestamp. Fixed to ts.strftime("%Y-%m-%dT%H:%M:%S.%f")[:-3] + "Z". The kernel writer correctly uses utc_now_iso() by default.

Per-stream caps (D2 — defense in depth)

  • Layer 1 — vector throttle at ingest time: 50 lines/sec/stream default; configurable via env.
  • Layer 2 — in-process collector: log_stream_budget (60s interval) queries VL /select/logsql/stats for bytes/day per stream and emits homelab_log_stream_bytes_today{host, service} + homelab_log_stream_lines_per_sec{host, service}.
  • vmalert rule (lands in STAGE-001-018) fires on per-stream overrun.

6. STAGE-001-017 — Alertmanager + vmalert (metrics) + first rule alerts metrics

STAGE-001-017 wires real alerting end to end. Alertmanager + vmalert (metrics) are added as sidecars; vmalert evaluates rules against VictoriaMetrics and routes to AM; AM webhooks back to /api/alerts/ingest (the path built in STAGE-001-013) using a render-on-boot Bearer token.

The render-on-boot mechanism

The challenge: AM needs a Bearer token to authenticate to the monitor's ingestor, but a real token can never be baked into a docker image or live in plaintext on disk indefinitely. Solution: render the AM config at lifespan startup with a token minted on first boot and reused thereafter.

Token bootstrap (Q2)

On first boot, ensure_ingest_token() in apps/monitor/homelab_monitor/kernel/alertmanager/render.py looks up an existing API token row named alertmanager-ingest AND its plaintext under secret name alertmanager-ingest-token:

  • Both present → emit alertmanager.bootstrap.token_reused, return the plaintext.
  • Both absent → mint via make_api_token() with Scope.ALERTS_INGEST_WRITE + who="system:alertmanager-bootstrap" (distinguishable in audit log); call secrets_repo.set(...); emit alertmanager.bootstrap.token_minted. Plaintext NEVER logged.
  • Inconsistent pair (one present, one absent) → delete the orphan, then mint fresh. Required because api_tokens.name and secrets.name both have UNIQUE indexes; a half-existing pair would collide on re-insert.

Atomic config render

render_config() reads the template, substitutes ${ALERTMANAGER_INGEST_TOKEN}, and atomic-replaces the output: writes to a sibling .tmp file via tempfile.mkstemp, os.replaces into place. AM picking up a SIGHUP never sees a partial file. The rendered file is group-owned by GID 2000 (group amconfig, baked into the AM image's group_add) with mode 0o640 so AM (running as nobody with supplementary GID 2000) can read it but no other host user can read the bearer token. Unit-test environments without the amconfig group keep the process default GID and 0o640.

8. STAGE-001-019 — Karma + kthxbye alerts ui

Karma (alert lifecycle UI) and kthxbye (auto-extends silences while alerts fire) join the stack. Karma is hosted on the docker network only; the monitor reverse-proxies /api/karma/* behind cookie-session auth so the iframe at /alerts is gated by our login.

Karma at port 8081 (not 8080)

Karma's default 8080 conflicts with pihole-unbound (network_mode: host) on this homelab. Override: 

# deploy/karma/karma.yml
listen:
  address: 0.0.0.0
  port: 8081
  prefix: /api/karma/

alertmanager:
  servers:
    - name: default
      uri: http://alertmanager:9093
      timeout: 10s
      proxy: false
      readonly: false

ui:
  refresh: 10s

listen.prefix: /api/karma/ means Karma expects requests under that path; the proxy forwards the FULL incoming path verbatim, not just the captured suffix.

The reverse-proxy router

apps/monitor/homelab_monitor/kernel/api/routers/karma.py mounts a catch-all /api/karma/{path:path} handler for every method (GET / POST / PUT / PATCH / DELETE / HEAD / OPTIONS). Five layers of defense:

  1. Auth — require_session_no_csrf() dependency (cookie session required; CSRF EXEMPT — see Origin check below). Token auth not exposed; Karma is interactive-UI-only.
  2. Path validation — regex ^[A-Za-z0-9._/~+\-%]*$; reject literal .. segment anywhere; reject null byte. Anything outside the safe set returns 400 invalid_path.
  3. Origin / Referer same-origin assertion on POST/PUT/PATCH/DELETE. Reverse-proxy aware: when HOMELAB_MONITOR_TRUST_FORWARDED_HEADERS=1 is set, honors X-Forwarded-Proto; always reads Host. Defense against the standard browser-attack model — does NOT defend against a misconfigured upstream proxy that forwards client-supplied X-Forwarded-Proto verbatim.
  4. Header allow-list both directions: request headers forwarded upstream are limited to accept, accept-encoding, accept-language, cache-control, content-type, content-length, if-modified-since, if-none-match, user-agent. Cookie and Authorization are DELIBERATELY absent (those are ours, not Karma's). Response headers relayed back: content-type, content-length, content-encoding, cache-control, etag, last-modified, vary. Set-Cookie and X-Frame-Options are NEVER relayed.
  5. Streaming response with deferred aclose via an _aiter_with_close generator so the upstream connection is properly released after iteration.

Why CSRF-exempt is safe here

The iframe at /alerts cannot attach the X-CSRF-Token header — it lives on a same-origin sandboxed document with allow-same-origin but no access to our top-frame JS context that mints CSRF requests. The Origin / Referer same-origin check substitutes: browsers cannot let scripted JS forge the Origin header on cross-origin requests.

The CspHeadersMiddleware

New pure-ASGI middleware (NOT BaseHTTPMiddleware) at kernel/api/middleware.py injects Content-Security-Policy: frame-ancestors 'self' on all responses unless upstream already set one. This is the modern X-Frame-Options replacement; ensures only same-origin pages can frame the monitor. Comprehensive CSP deferred to a future hardening epic.

Iframe sandbox policy

<iframe sandbox="allow-scripts allow-same-origin allow-forms"> — minimum needed for Karma's same-origin React SPA + XHR-with-cookies + ack/silence form submissions. allow-popups, allow-modals, allow-top-navigation intentionally omitted.

kthxbye flag-ordering invariant

kthxbye CRASHES if extend-by is not strictly greater than extend-if-expiring-in. Refinement caught this in BOTH compose files: prod 5m/6m and test 10s/15s were both invalid. Final values: prod -extend-by=10m -extend-if-expiring-in=6m -interval=1m -max-duration=24h; test -extend-by=30s -extend-if-expiring-in=15s -interval=5s -max-duration=2m. e2e-tester caught the crash immediately on bringing up the test rig.

Origin check made reverse-proxy aware (Refinement)

The original _verify_origin compared Origin to f"{request.url.scheme}://{request.url.netloc}" — wrong when behind nginx, traefik, or vite-dev's proxy. Karma iframe POST to /api/karma/api/v2/silences got 403 cross_origin_blocked. Fix: use Host header (forwarded verbatim by HTTP/1.1 proxies) + optional X-Forwarded-Proto (gated by HOMELAB_MONITOR_TRUST_FORWARDED_HEADERS=1); fall back to request.url.{scheme,netloc} only when those headers are absent. Vite dev proxy correspondingly switched to changeOrigin: false.

Karma healthcheck removed (Refinement)

Karma's image is distroless — no wget, curl, or sh. The original CMD-SHELL wget ... healthcheck stuck the container at "starting" forever. No service depends_on Karma's health, so removing the healthcheck is safe; readiness is observed via the monitor's reverse-proxy returning 502 if Karma is briefly down.

11. EPIC-001 closure

21/21 stages complete. EPIC-001 — "Foundation" — is closed. What's now possible end-to-end:

  • A real homelab metric (host CPU) flows from psutil → Prometheus registry → vmagent scrape → VictoriaMetrics → MetricsQL → vmalert → Alertmanager → monitor ingest → SSE broker → React UI. Latency budget < 60s confirmed by the canonical e2e test.
  • Logs flow from any docker container → vector → VictoriaLogs → LogsQL → vmalert-logs → Alertmanager → monitor ingest. Same fan-out infrastructure as metrics, distinguished only by source_tool=vmalert-logs on the alert row.

  • kthxbye keeps silences alive while alerts continue firing — solving the "silence expired but alert still active → re-page noise" failure mode that vanilla AM has.
  • The full stack runs in CI on every PR via a 14-service test rig in < 8 minutes with BuildKit GHA layer cache. Required for merge to main.
  • Backups are operator-driven via POST /api/admin/backup or hm backup run — SQLite + VM snapshot files, 7-daily retention by default, never includes the master key.
  • Disk-budget self-monitoring emits per-slot bytes, derived budget, and used%; vmalert fires at 70/85/95 thresholds; the auto-shrink decision-only path is in place but no destructive action is taken in v1.
  • Developers can spin up the full hybrid stack with make dev (docker sidecars + host uvicorn + host pnpm dev) or the full prod-image variant with make dev-prod.

13. Cross-cutting invariants reinforced in stages 12-21

  • Async-only kernel. Every IO path goes through asyncio + aiosqlite; no blocking calls inside ASYNC collectors. STAGE-001-016's VictoriaLogsWriter is the canonical example of bridging a sync Protocol (LogsWriter.ingest) to an async HTTP client via a bounded queue + background worker.
  • audit_log singular. The table name is audit_log, NOT audit_logs (singular by convention, not by accident). STAGE-001-013 Finalize fixed a typo that referenced audit_logs in a mark_resolved code path.
  • Render-on-boot pattern (STAGE-001-017) is the canonical way to inject runtime-minted secrets into sidecar config files: lifespan looks up an existing token-secret pair OR mints a fresh pair atomically with audit log; renders the template with tempfile.mkstemp + os.replace for atomicity; calls the sidecar's /-/reload endpoint with warn-on-failure (file is source of truth). NEVER raises — degrades gracefully so lifespan continues.
  • Reverse-proxy router pattern (STAGE-001-019, reused verbatim in STAGE-001-020): catch-all /{path:path} + path validation regex + Origin/Referer same-origin assertion + explicit request/response header allow-lists + require_session_no_csrf() for iframe-driven flows + streaming response with deferred aclose.
  • Multiplex writer pattern (STAGE-001-015 metrics, STAGE-001-016 logs): keep the existing in-memory writer; add a real backend writer; fan every write_* / ingest() call to BOTH. Preserves test ergonomics and provides an in-memory buffer that rides out backend outages.
  • Static config files over code-generated. Both STAGE-001-015 (vmagent scrape) and STAGE-001-016 (vector.toml) explicitly rejected dynamic config generation. Hand-authored YAML/TOML is reviewable in git; reload = container restart; no monitor↔sidecar lifecycle coupling.
  • GHCR-published custom images. The custom Grafana image (STAGE-001-020 deliverable) is published via a separate GitHub Actions workflow on changes to deploy/grafana/**; CI integration job pulls from GHCR with a local-build fallback for the very first run before the workflow has run. Pattern is reusable for any future image we need to bake plugins or config into.
  • Fixture-driven integration tests. STAGE-001-021's fixture-host (FastAPI / Prometheus exporter with a /control endpoint to set arbitrary metric values) and noisy-logger (POST /log emits to stdout for vector) are the canonical pattern for deterministically tripping vmalert rules in tests. Direct VL POSTs via vl_planter.py are the helper for log-derived alerts where the vector→VL path isn't under test.
  • Origin/Referer same-origin defense for CSRF-exempt iframe routes. The Karma + Grafana proxies cannot use the standard CSRF cookie pattern because the iframe document can't read our top-frame CSRF cookie via JS. Origin/Referer same-origin enforcement (browsers cannot let scripted JS forge those headers cross-origin) substitutes — gated by HOMELAB_MONITOR_TRUST_FORWARDED_HEADERS=1 when behind a trusted reverse proxy.
  • Dedup by fingerprint while firing; new row on re-fire after resolve (STAGE-001-013 D4). Each fire→resolve→fire cycle is operationally a new incident; outcome history attaches to its row, not aggregated across cycles. Severity / labels / annotations are PINNED at first fire — upstream rules must encode severity in the fingerprint hash if escalation matters.
  • Severity vocabulary locked: info, warning, error, critical (per spec §8.2). Rules must use one of these four; missing severity defaults to warning with a structured WARNING log noting the misconfigured rule.
  • RFC3339 timestamps for VictoriaLogs (refinement-phase bug from STAGE-001-018). Epoch milliseconds and other formats are silently rejected by VL with an opaque error. utc_now_iso() in the kernel writer correctly produces the right format; helpers must use ts.strftime("%Y-%m-%dT%H:%M:%S.%f")[:-3] + "Z" when constructing timestamps manually.
  • Health-check 127.0.0.1, not localhost for busybox-based images. Found in STAGE-001-018 Refinement: VM, VL, AM, vmalert images don't have localhost in /etc/hosts; healthchecks using it stall containers in "Created". Use http://127.0.0.1:<port>/... everywhere.
  • kthxbye flag invariant (STAGE-001-019 Refinement): extend-by MUST be strictly greater than extend-if-expiring-in or kthxbye crashes at startup. Both compose files must satisfy this (prod 10m/6m, test 30s/15s).
  • Subagent-driven implementation discipline (CLAUDE.md): main agent never reads/edits/writes source files directly; all execution work delegated to subagents. STAGE-001-013 was split into Spec A + Spec B sub-builds to stay within the per-spec tool-use budget; STAGE-001-021 further split into Specs A + B + C executed sequentially.
