Architectural Overview (EPIC-002 — Heartbeat receiver + cron registry + cron auto-discovery)

This picks up where overview-stages-12-21.html left off (EPIC-001 closed: a synthetic CPU spike becomes a firing alert in the UI within ~60 seconds). EPIC-002 delivers the cron-monitoring half of the system across twelve stages: a Healthchecks.io-style heartbeat receiver, a fingerprint-keyed cron registry, automatic discovery of crons from /etc/cron.* and user crontabs, B-mode log-scrape that tracks exit codes of unmodified crons, one-click wrapper install/remove helpers, and vmalert cron-health alerting. By the end of STAGE-002-010, the Inventory screen exposes a Crons tab listing every known cron with last-seen state, lateness, and recent runs; vmalert fires when a cron is late or failing; and the operator can opt any local cron into active heartbeat with a one-click wrapper install.

1. Subsystem architecture

EPIC-002 adds three new data-flow paths to the EPIC-001 kernel, all converging on two SQLite tables: crons (the registry) and heartbeats_state (derived observational state per cron). A cron row is content-addressable by its fingerprint — a SHA256 of the 4-tuple (host, source_path, schedule, command) — so every path that observes the same cron converges on the same row without a coordination protocol.

Cron monitoring is hybrid by design (spec §2 / Q5). Three observation modes feed the same registry:

  • B (default) — log-scrape. A cron-log-collector plugin watches journald/syslog via VictoriaLogs and matches dispatch lines to crons by a fingerprint heuristic. Works on any cron with no modification.
  • A (gold standard) — active heartbeat. The cron is wrapped to POST /api/hb/{fingerprint}/start then .../ok or .../fail. Most reliable; requires touching the cron.
  • C (bootstrap helper) — discovery + wrapper. The monitor reads the host's crontabs, auto-generates a per-job wrapper script, and offers it as a one-click install from the dashboard.
Public-release default policy (spec §2 Q30). The open-source release is conservative: discovery records every cron without modifying it, and B-mode log-scrape always runs. Active heartbeat (A) and wrapper install (C) are strictly opt-in per cron, through the dashboard. No existing user cron is ever modified silently. The host-specific homelab-monitor-overrides repo can be more aggressive; that wiring is a later epic, not EPIC-002.

2. STAGE-002-001 — Heartbeat receiver + crons/heartbeats_state schema heartbeat db

The foundation stage. It replaced the auth-only heartbeat router stub from STAGE-001-021 with a real receiver that persists per-cron state, and landed the migration adding the behavioral columns to crons (cadence, grace, integration_mode) and heartbeats_state (current_state, last_ok_at, current_streak, expected_next_at). The /hb/{id}/start, /hb/{id}/ok, and /hb/{id}/fail endpoints update those tables, audit every state transition, and the receiver 404s on an unknown id — at this stage the only creation path was the registry CRUD coming in STAGE-002-002.

Heartbeat endpoints accept API tokens only — no anonymous posts. Tokens carry the heartbeat:write scope (Scope.HEARTBEAT_WRITE) and are rate-limited per token. The schema shipped here is later substantially reshaped by the STAGE-003–006 redesign block; the receiver semantics it established (audited transitions, derived heartbeats_state) survive intact.

3. STAGE-002-002 — Cron registry CRUD API + Inventory "Crons" tab cron registry ui

Built the first user-facing surface for the registry: a typed CRUD API for crons rows (list, get, create, update, soft-delete) plus a "Crons" tab in the Inventory screen showing every known cron with last-seen state, expected next run, current streak, and integration mode. As originally shipped it allowed manual cron creation, used a UUID primary key, and carried an integration_mode field (observe/heartbeat/both).

Superseded by the redesign block. Manual create, the UUID PK, and integration_mode were all removed or replaced in STAGE-002-003 and 002-004. EPIC-002 was reshaped on 2026-05-11 by the cron-derived-state-redesign brainstorm: four new stages (003–006) introduce the fingerprint identity model and the derived-state UI, and the original 003–006 were renumbered 007–010.

4. STAGE-002-003 — Schema redesign: fingerprint identity db

Replaced the UUID-keyed cron schema with a fingerprint-keyed one. After this stage crons.fingerprint is the primary key — a SHA256 hex digest of (host, source_path, schedule, command); integration_mode is gone; archived_at is renamed hidden_at; new columns source_path and wrapper_installed_at are present. The heartbeat receiver and CronRepo were updated to use the fingerprint as the cron identifier. This was a destructive migration — dev seed data was dropped and re-seeded by the migration's own upgrade() (no production data existed).

The fingerprint algorithm (kernel/cron/fingerprint.py) serializes the 4-tuple as JSON with sort_keys=True, separators=(",",":"), and ensure_ascii=False, then SHA256s the UTF-8 bytes. JSON-canonical rather than delimiter-joined: cron commands legitimately contain pipes, colons, and quotes, so any delimiter scheme would collide. See cron-identity.md for the full derivation.

5. STAGE-002-004 — API removal: drop manual create api

The "no add" enforcement stage. It stripped the manual-create path entirely: the POST /api/crons endpoint, the CronCreate Pydantic schema, the AddCronModal UI component, the "+ Add cron" button were all removed, and CronForm collapsed to edit-only mode. After this stage the only ways a crons row can come into existence are auto-discovery (STAGE-002-007) and the /register endpoint (STAGE-002-005). Manual-add cannot return without a deliberate code change reverting the deletion.

6. STAGE-002-005 — /register endpoint heartbeat api

Added POST /api/hb/{fingerprint}/register — the heartbeat-side creation path. A wrapper script (or any heartbeat client) POSTs {host, source_path, schedule, command, wrapper: bool}; the server recomputes the fingerprint from the body and rejects with 422 (fingerprint_mismatch) if it does not match the URL. The endpoint is an idempotent upsert: first call for a fingerprint INSERTs (201), subsequent calls RETURN the existing row (200) and never touch operator-edited fields — the only column that may mutate on re-register is wrapper_last_seen_at, and only when the body carries wrapper: true.

/start, /ok, and /fail continue to 404 on unknown fingerprints — creation happens only via /register or discovery. This reverses STAGE-002-001's "404 on unknown" stance for the register path specifically, justified by the derived-state model where the heartbeat wrapper is a legitimate authority on what it wraps. Stage decisions D1–D10 are locked in epics/EPIC-002-heartbeat-cron/STAGE-002-005.md.

7. STAGE-002-006 — UI redesign: 4-panel detail page ui

Redesigned the cron detail page to reflect the derived-state model. The page is split into four clearly-labeled panels:

PanelContents
Disk sourceRead-only: host, source_path, schedule, command — the fingerprint inputs.
Monitoring policyEditable: name, expected_grace_seconds, enabled.
Heartbeat stateRead-only with diagnostics: current state, last ok/fail, streak, lateness.
ActionsHide/Unhide; Install heartbeat (disabled in this stage, enabled in STAGE-002-009 for local-host crons only).

The Crons toolbar replaced the include_archived filter with include_hidden and replaced the integration_mode filter with a wrapper_installed filter. The list view shows a "Remote (no disk source)" badge for rows where source_path IS NULL. After this stage the UI matches the derived-state mental model; downstream stages extend it without further structural change.

8. STAGE-002-007 — Cron auto-discovery discovery cron registry

Stood up the cron-discoverer builtin plugin — a standard in-process BaseCollector (interval 300s default, concurrency group discovery) that scans the host's crontab files and writes directly into the crons registry. It populates host, source_path, schedule, command, cadence_seconds, schedule_canonical, and last_discovered_at. There is no suggestion-queue interaction — per the derived-state redesign, discovered crons appear in the registry as derived state the moment a scan sees them.

Sources scanned: /etc/crontab and /etc/cron.d/* (system crontabs, world-readable, bind-mounted read-only under /host/etc/...) plus per-user crontabs. The 0600 spool files are unreadable by the non-root container, so discovery reads a world-readable snapshot directory (HM_CRON_SNAPSHOT_DIR) populated by a root-side hm-crontab-snapshot script via crontab -l -u <user>. The DB stores host paths, not container paths: /etc/crontab, /etc/cron.d/certbot, or the logical identifier crontab:<user>. Audit verb: crons.discover.

9. STAGE-002-007A — Auto-soft-delete + auto-restore discovery cron registry

An inserted sub-stage. Before it, a discovered cron whose crontab line was later deleted left an orphan row forever. STAGE-002-007A introduced automatic soft-delete: after each scan, CronRepo.reconcile_soft_deletes() compares the DB rows for each cleanly inspected source file against the fingerprints the scan actually found. A row whose fingerprint is absent from a clean scan gets soft_deleted_at = now and a crons.soft_delete audit row; when the fingerprint reappears on a later clean scan, the row is auto-restored (soft_deleted_at = NULL, crons.restore).

The reachability gate is the safety invariant. Soft-delete runs only for source files that were cleanly read and parsed this scan. An unreadable file (a 0600 crontab the container UID cannot read, a parse error) is never treated as a clean empty scan — "we couldn't see the file" is not "the cron is gone." Reconciliation is per-host and per-source-file, and remote-only crons with source_path IS NULL are never auto-soft-deleted.

soft_deleted_at (system-controlled, set by discovery) is distinct from hidden_at (operator-controlled suppression). Both columns can coexist on one row. The same restore also fires when a soft-deleted cron is re-registered via /register.

10. STAGE-002-008 — B-mode log-scrape log-scrape heartbeat

Shipped the log-scrape pipeline: vector + journald + a cron-log-collector plugin that watches VictoriaLogs for cron output and updates heartbeats_state rows. After this stage every cron in the registry — wrapper or not — gets last_seen_state updates from log evidence whenever the host runs the cron.

Reworked from the original STAGE-002-004 plan in two ways. First, no integration_mode gating — the derived-state model has no such field, so log-scrape always runs for any registered cron; if a cron also has heartbeat evidence the timestamps tie-break naturally (most recent wins). Second, logs are matched to crons by a fingerprint heuristic: the collector parses a log line into (host, command_extracted) and updates every crons row where host and command match. Parsing is idempotent — a per-cron high-water mark plus per-line content hash guarantees re-processing the same syslog line never double-counts. A NEUTRAL observed run writes the cron.observed_run audit verb and bumps observed_runs_total without changing current_state.

11. STAGE-002-009 — Wrapper install helpers wrapper api

Delivered the C-mode bootstrap helper: a bash wrapper template (cron-with-heartbeat.sh.tmpl) that wraps a host cron command and POSTs /start then /ok|/fail heartbeats. The fingerprint, heartbeat URL base, and token file path are baked into the wrapper at install time. Three install paths exist: the UI "Install heartbeat wrapper" button (an InstallHeartbeatModal dry-run-preview → confirm flow), the CLI hm cron install-wrapper, and a standalone install_wrapper_remote.py for hosts not running the monitor stack.

The monitor container has zero host-write capability. Wrapper install routes through a host-side cron-apply executor: the monitor writes only an IPC request file into /var/lib/homelab-monitor/cron-apply/{requests,results}/; a root systemd oneshot service (scripts/hm-cron-apply.sh, triggered by a .path watcher with a 60s .timer safety-net) performs the privileged crontab rewrite, writes the shared wrapper script at /usr/local/bin/cron-with-heartbeat.sh, and writes the shared token at /etc/homelab-monitor/heartbeat.token. The executor re-derives the wrapped line itself, writes to fixed paths only, and snapshots + rolls back on any failure.

A wrapped crontab line is unwrapped before fingerprinting, so a cron keeps its original fingerprint across install — no spurious soft-delete. The is_local computed field on CronOut gates the UI Install button to local-host crons. The new env var HOMELAB_MONITOR_PUBLIC_URL supplies the heartbeat URL base baked into the wrapper. This stage also retired the fragile crontab-ACL discovery mechanism (vixie-cron rejects an ACL-carrying crontab) in favour of the root-side hm-crontab-snapshot systemd units.

12. STAGE-002-009A — Wrapper removal helpers wrapper ui

The inverse of STAGE-009. An inserted sub-stage that reverts a wrapped crontab line <schedule> /usr/local/bin/cron-with-heartbeat.sh -- <original> back to <schedule> <original> — byte-exact original command. Three uninstall paths mirror the install paths: the UI cron detail page now has an Install/Remove toggle (RemoveHeartbeatModal with an un-wrap diff preview), the CLI hm cron uninstall-wrapper, and the remote script's --uninstall mode. The new endpoint POST /api/crons/{fingerprint}/uninstall-wrapper routes through the same host-side executor via a new unwrap-crontab operation. A typed NotWrappedError → HTTP 409 mirrors AlreadyWrappedError.

This stage added the wrapper_installed boolean column (alembic migration 0013) — derived by the discoverer from the crontab line's actual wrapped state. The UI toggle keys on wrapper_installed, not on wrapper_last_seen_at (which is NULL for an installed-but-never-fired wrapper and would misalign the toggle). The cron-apply executor now refreshes the crontab snapshot inline after every wrap/unwrap, and the monitor requests a one-shot cron-discovery on startup so wrapper_installed converges within seconds of a restart instead of up to 300s.

13. STAGE-002-010 — vmalert cron-health alerting alerting metrics

The EPIC-002 closer. It adds cron-health alerting end to end: a metrics collector, vmalert rules, a new Alertmanager channel, and a wrapper-health badge in the UI.

HeartbeatStateCollector

A scheduler-registered collector (kernel/metrics/heartbeat_collector.py) that every 30 seconds emits six Prometheus metrics per non-hidden, non-soft-deleted cron:

MetricMeaning
homelab_heartbeat_seconds_since_last_okAge of the last successful run.
homelab_heartbeat_seconds_since_last_failAge of the last failed run.
homelab_heartbeat_current_streakConsecutive ok (positive) or fail (negative) count.
homelab_heartbeat_expected_next_secondsTime to the lateness threshold; negative when overdue.
homelab_heartbeat_last_duration_secondsDuration of the last observed run.
homelab_heartbeat_logscrape_count_since_last_heartbeatLog-scrape runs observed since the last real heartbeat — the wrapper-staleness signal.

The collector uses replace_family() for an atomic per-family series swap, so hidden / soft-deleted crons' series are dropped rather than persisting forever (a stale-gauge bug found and fixed in Refinement 3a). @reboot lateness is handled inside the collector: it emits a negative expected_next_seconds when last_ok_at predates host boot time, reusing the HeartbeatStale alert ladder with no separate @reboot rule. Host boot time is read from /host/proc/stat via host_boot_time.py — a /proc:/host/proc:ro bind mount was added for this (and it fixed a real host.py bug where psutil.boot_time() returned container uptime).

vmalert rules — heartbeats.yaml

deploy/vmalert/metrics/heartbeats.yaml ships 6 alert rules + 4 recording rules: HeartbeatStale_Warning/_Error/_Critical, HeartbeatFailed, HeartbeatFlapping, and WrapperPossiblyStale. The first five carry routing_channel=cron-health; WrapperPossiblyStale carries routing_channel=monitoring-health and fires when wrapper_installed="yes" AND the log-scrape count is ≥ 3 for 60s — i.e. the cron is still running (log-scrape sees it) but the wrapper has stopped reporting.

The logscrape_runs_since_heartbeat counter

A new counter column on heartbeats_state (alembic migration 0014, chains from 0013). It is incremented in record_observed_run and reset to 0 by record_ok / record_fail / record_start — so a non-zero value means the cron has run (per log-scrape) without the wrapper firing. It backs both the WrapperPossiblyStale alert and the wrapper_health API field.

monitoring-health channel + wrapper-health badge

A new monitoring-health route and monitoring-health-channel receiver were added to the Alertmanager template, keeping wrapper-degradation alerts on a separate channel from ordinary cron-health alerts. The API gained a wrapper_health enum field (ok/stale/unknown) on CronWithStateOut, computed via _compute_wrapper_health against a _WRAPPER_STALE_THRESHOLD = 3. The cron-detail Actions panel shows a wrapper-health badge whenever the cron's wrapper is installed, and the UI label "Next due" was renamed "Overdue after" to accurately describe the lateness threshold.

14. Key design decisions

  • Fingerprint identity — content-addressable cron rows. A cron is the SHA256 of (host, source_path, schedule, command), JSON-canonical. Discovery and the /register handshake both compute the same fingerprint from the same source data, so they converge on one row with no coordination protocol. Editing the schedule or command produces a new fingerprint; the old row goes stale. The user cannot create cron rows manually — rows exist only because discovery or a wrapper handshake created them. name is deliberately not in the tuple, so the operator can rename freely without changing identity.
  • Derived-state model for heartbeats_state. Every value in heartbeats_state can be recomputed from the events log (heartbeat receiver writes + B-mode collector writes). It is treated as a materialized view, never the source of truth. The same principle drove the STAGE-003–006 redesign that removed manual cron creation: the registry is derived from observed reality, not operator data entry.
  • Hybrid B / A / C cron-monitoring model. B-mode log-scrape (default, works on any cron unmodified), A-mode active heartbeat (gold standard, requires wrapping), and C-mode discovery + wrapper bootstrap. All three feed the one fingerprint-keyed registry. Log-scrape always runs regardless of whether a wrapper is installed; when both exist, the more recent timestamp wins and WrapperPossiblyStale cross-checks them.
  • Public-A-default policy (spec §2 Q30). The open-source release observes existing crons without modifying them — discovery records, B-mode scrapes, but A/C modifications are strictly opt-in per cron through the dashboard. No user cron is edited silently. The host-specific homelab-monitor-overrides repo is the place for more aggressive B/C application; that wiring is out of scope for EPIC-002.
  • Zero host-write capability in the monitor container. All privileged host writes (crontab rewrites, the wrapper script, the shared token) are performed by a root-side systemd executor (hm-cron-apply.sh) triggered through an IPC request/result directory. The monitor only ever writes a request file. The executor re-derives every line itself, writes to fixed paths only, and snapshots + rolls back on failure.
  • Reachability gate on auto-soft-delete. A cron is soft-deleted only when its fingerprint is absent from a cleanly scanned, readable source file. An unreadable file is never treated as an empty scan. This prevents a transient permission error or parse failure from wiping the registry.
  • Idempotent log-scrape. Re-processing the same journald/syslog line never double-counts — guaranteed by a per-cron high-water mark plus a per-line content hash. The logscrape_runs_since_heartbeat counter is the wrapper-staleness signal that connects log evidence to wrapper-health alerting.
  • Unwrap-before-fingerprint convergence. Installing or removing the heartbeat wrapper rewrites a crontab line's command, which would change the fingerprint if hashed directly. Both the discovery parser and the wrapper installer strip the wrapper invocation prefix before fingerprinting, so a cron's identity is stable across install → uninstall → re-install.

15. EPIC-002 closure

All twelve stages — STAGE-002-001 through 002-010, with 002-007A and 002-009A inserted as sub-stages — are Complete. The cron-monitoring half of homelab-monitor is in place: a Healthchecks.io-style heartbeat receiver, a fingerprint-keyed cron registry populated entirely by auto-discovery and the /register handshake, B-mode log-scrape that tracks unmodified crons, one-click wrapper install/remove helpers, and vmalert cron-health alerting routed across the cron-health and monitoring-health channels. The Inventory → Crons tab presents every known cron through the 4-panel detail page.

EPIC-003 and later can now rely on an authoritative cron registry, a heartbeat receiver that backs Healthchecks-style integrations from any source (not just crons — also long-running jobs, backups, and the local-watchdog itself), and the foundation for the digest's "Cron heartbeat report" section.
