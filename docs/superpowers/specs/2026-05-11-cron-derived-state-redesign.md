# Cron registry → derived state: identity, registration, alerts redesign

**Status:** Approved 2026-05-11. Implementation begins after STAGE-002-002 ships (Refinement → Finalize → Complete).

**Author:** brainstorm session 2026-05-11.

**Scope:** Replaces the manual-create model of the cron registry with a derived-state model in which crons exist ONLY because they were auto-discovered or because a heartbeat wrapper POSTed metadata. Reworks EPIC-002 stages 003–006 and inserts four new stages (003 schema, 004 API/UI removal, 005 register endpoint, 006 UI redesign).

## Why

The current cron registry (STAGE-002-002) allows manual creation of cron rows. This is wrong: a "cron" is not an arbitrary object the user invents — it is a row that exists on disk in some crontab file, or a signal we receive from a wrapper script attached to such a row. Manual registration produces orphan rows that don't correspond to any real cron, creates pollution risk if a typo'd name is added, and breaks the user's mental model ("I am monitoring my real crons" vs "I am cataloging crons in a database").

The registry must instead be derived state:

- Crons appear because **auto-discovery** found them on disk (local-host scanning in STAGE-002-007, cross-host SSH/agent in EPIC-015 / EPIC-017).
- Crons may also appear because **a heartbeat wrapper registered** them — for a remote host the monitor cannot scan today, the wrapper is the only seed.
- The user cannot manually create rows. The user can only edit a small set of policy fields (`name`, `expected_grace_seconds`, `enabled`, `hidden_at`).

## Decisions

### Identity

Each cron is identified by a **SHA256 fingerprint** of `(host, source_path, schedule, command)`. Fingerprint is the primary key on the `crons` table. URLs use the fingerprint (`POST /api/hb/{fingerprint}/ok`). When a crontab edit changes the schedule or command, the fingerprint changes — the old fingerprint goes stale, the new one appears.

Rationale: content-addressable identity. Same fingerprint → same cron. Discovery and heartbeat wrappers both compute the same fingerprint from the same source data, so they converge on the same row without a coordination protocol.

Trade-off accepted: if a user edits the crontab, the existing fingerprinted row becomes stale and a new row appears. The UI surfaces this as "the old cron stopped reporting; a new cron appeared" — which is accurate.

### Wrapper script knows its fingerprint via install-time bake

The wrapper script that the cron daemon invokes embeds the fingerprint as a constant (`HEARTBEAT_ID="sha256:abc..."`) at install time. The wrapper script is generated by `hm cron install-wrapper`. The wrapper itself is bash; the user's underlying script is language-agnostic (the wrapper just `exec`s it). At runtime, the wrapper has the fingerprint baked in and POSTs to `/api/hb/{fingerprint}/{start,ok,fail}`.

Alternative considered (runtime fingerprint lookup via /proc walking) was rejected: fragile, container-hostile, and depends on cron daemon flavor.

### Layered install paths

- **Local-host (monitor's own machine):** UI button "Install heartbeat" → the monitor writes the wrapper file, rewrites the crontab, calls `/register`. One click. Ships in STAGE-002-009 (reworked from STAGE-002-005).
- **SSH-reachable hosts:** UI button → SSH push variant. **Deferred to EPIC-017** (SSH probe framework). EPIC-002's "Install" button is disabled for non-local hosts with a tooltip pointing at EPIC-017.
- **Unreachable hosts (Synology, NAS, foreign-network containers):** User downloads `hm cron install-wrapper` CLI binary onto the host and runs it locally. The CLI scans the crontab, computes fingerprints, writes wrappers, calls `/register` against the monitor (authenticated). EPIC-002 ships the CLI; the binary itself is built by EPIC-019 release pipeline.

### Heartbeat receiver behavior

- `POST /api/hb/{fingerprint}/{start,ok,fail}` — receiver **always 404s on unknown fingerprints**. The 404 is the protocol — it tells the wrapper "you have not registered yet; call /register first."
- `POST /api/hb/{fingerprint}/register` — body carries metadata `{host, source_path, schedule, command, wrapper: bool}`. The server recomputes the fingerprint from the body and rejects with 422 if the URL fingerprint doesn't match. On success, INSERT or idempotent-UPDATE a `crons` row. Records `wrapper_installed_at = utc_now_iso()` if `wrapper: true`. Audit verb: `crons.register`.

A conflicting metadata for an existing fingerprint (same fingerprint, different host/source_path/etc.) is impossible by construction (the fingerprint IS the metadata hash) — defensive 409 is included for the "the server's hash function evolved" scenario but should never fire under correct clients.

### Grace and lateness

`late` state fires when `now > last_ok_at + cadence_seconds + grace_seconds`. Computed by vmalert (reworked STAGE-002-010). Cadence is derived from the schedule (croniter average interval); user-editable expected_grace_seconds. The semantic is "expected interval plus a user-tunable slop window."

### Mode

`integration_mode` is **dropped entirely**. Both data sources (log-scrape and heartbeats) always run when present. The cron is in whatever state the available evidence supports.

If a cron HAS been heartbeated AND log-scrape is also seeing it run, that's a positive cross-check. If a cron was heartbeated previously AND log-scrape now sees 3+ runs without a corresponding heartbeat, that's a wrapper-health concern (alerts via a separate channel — see below). If neither evidence source has signal, the cron is `unknown`.

### Wrapper health monitoring

A new vmalert rule fires `WrapperPossiblyStale` when `wrapper_installed_at IS NOT NULL` AND `log_scrape_count_since_last_heartbeat >= 3`. Alertmanager routes the alert to a `monitoring-health` channel separate from cron-health alerts. This is a "your monitoring is broken, not your cron" signal. Ships in reworked STAGE-002-010.

### Hidden replaces archive

The existing `archived_at` column is renamed `hidden_at`. Semantically: "hide from default views and suppress alerts." Discovery does NOT auto-unhide. Re-hiding is a PATCH. The user uses this to say "I know this cron exists; I don't want to monitor it." It does NOT remove the cron from disk.

### Name field

Defaults to command basename (e.g., `backup.sh` for `/opt/scripts/backup.sh`). User-editable on the detail page. `NOT NULL` constraint preserved. Discovery sets it on row insert; never re-derives after that (user edits stick).

### Remote-only crons (heartbeating without local discovery)

A cron may exist in the registry with `source_path IS NULL` because it heartbeated from a host the monitor cannot scan locally. UI surfaces these with a banner: "Remote cron — disk source not available until cross-host discovery ships in EPIC-015/017."

The cron's `host` field is the value the wrapper claimed in its `/register` payload. We trust the wrapper because it authenticated with a valid `heartbeat:write` token.

## Schema changes (STAGE-002-003 reworked, see also new stage 002-003)

`crons` table (after migration):

| Column | Type | Notes |
|---|---|---|
| `fingerprint` | TEXT PK | SHA256 hex of `host\|source_path\|schedule\|command` |
| `name` | TEXT NOT NULL | Defaults to command basename; user-editable |
| `host` | TEXT NOT NULL | Read-only after creation |
| `command` | TEXT NOT NULL | Read-only after creation |
| `schedule` | TEXT | Read-only after creation; NULL for cadence-only (rare) |
| `schedule_canonical` | TEXT | Derived from schedule |
| `cadence_seconds` | INTEGER | Derived from schedule via croniter |
| `expected_grace_seconds` | INTEGER NOT NULL | User-editable, default 300 |
| `enabled` | INTEGER NOT NULL | User-editable, default 1 |
| `last_seen_state` | TEXT enum | Derived |
| `created_at` | TEXT | UTC ISO-8601 |
| `updated_at` | TEXT | UTC ISO-8601 |
| `hidden_at` | TEXT NULL | Was `archived_at`; user-editable |
| `source_path` | TEXT NULL | NEW. `/etc/crontab`, `/etc/cron.d/foo`, or `crontab:user`. NULL for remote-only crons. |
| `wrapper_installed_at` | TEXT NULL | NEW. Set when `/register` called with `wrapper: true`. Gates wrapper-health alerts. |

Dropped: `id` (UUIDv7 PK), `integration_mode`, `archived_at` (replaced with `hidden_at`).

Migration is destructive: the dev test data is just seed; no production data exists. STAGE-002-003 (new) does a clean schema replacement; legacy UUID-keyed rows are not migrated.

## API surface changes

**Removed:**
- `POST /api/crons` (manual create) — gone in STAGE-002-004 (new).
- `CronCreate` Pydantic schema — gone.
- `AddCronModal` + "Add cron" button in UI — gone.

**Added:**
- `POST /api/hb/{fingerprint}/register` — token-only (`Scope.HEARTBEAT_WRITE`). Idempotent upsert. Records `wrapper_installed_at` if `wrapper: true`. Ships in STAGE-002-005 (new).

**Changed:**
- `POST /api/hb/{fingerprint}/{start,ok,fail}` — path param semantic renamed (cron_id → fingerprint). Unchanged behavior except: 404 on unknown is now part of the protocol, not a footnote.
- `PATCH /api/crons/{fingerprint}` — editable fields shrink to `{name, expected_grace_seconds, enabled, hidden_at}`. All other fields read-only.

**Unchanged:**
- `GET /api/crons`, `GET /api/crons/{fingerprint}`, `GET /api/crons/{fingerprint}/preview-runs?count=N`, `DELETE /api/crons/{fingerprint}` (still soft-delete; semantic is "hide").

**Audit verb taxonomy:**
- `crons.register` — created via `/register`
- `crons.discover` — created via discovery (002-007)
- `crons.update` — PATCH to a policy field
- `crons.hide` — PATCH `hidden_at` set
- `crons.unhide` — PATCH `hidden_at` cleared

## UI changes (STAGE-002-006 new)

- Remove "+ Add cron" button; remove `AddCronModal` component; remove `mode='create'` from `CronForm`.
- `CronForm` becomes edit-only. Fields shown: `name`, `expected_grace_seconds`, `enabled`. Fields shown read-only with copy buttons: `host`, `source_path`, `command`, `schedule` (+ `schedule_canonical` tooltip).
- `CronDetail` page becomes four panels:
  1. **Disk source (read-only):** host, source_path (with remote-banner if NULL), schedule, command, last_discovered.
  2. **Monitoring policy (editable):** name, expected_grace_seconds, enabled.
  3. **Heartbeat state:** current_state, streak, last_*_at timestamps, last_duration, last_exit_code, wrapper_installed_at (with "Wrapper installed via UI/CLI" or "No wrapper — ad-hoc curl").
  4. **Actions:** Hide / Unhide button (replaces "Archive"). "Install heartbeat" button — ENABLED for local host (STAGE-002-009); DISABLED with tooltip pointing at EPIC-017 for remote hosts. Wrapper-health badge if applicable.
- `CronsToolbar` filters: replace `integration_mode` filter with `wrapper_installed: yes/no/all`. Rename "include_archived" → "include_hidden".
- List view: "Remote (no disk source)" badge appears in the host column for `source_path IS NULL` rows.

## Downstream epic notes

### EPIC-011 (Discovery & suggestions UX)

Cron discovery deliberately does NOT use the suggestion queue. Crons appear directly in the registry as derived state. Suggestions are reserved for resources that ARE valid for the user to either accept or ignore — e.g., new hosts, new services, new mounts. For crons, the only "ignore" affordance is the `hidden_at` column.

### EPIC-015 (Netdata + future remote agents)

This epic absorbs the **agent-push cross-host discovery** model. The `hm-agent` binary scans local crontabs on the host where it runs and POSTs a manifest to `/api/discovery/report`. The agent is the same Netdata-companion concept the epic already plans. Cron discovery is one of several reports the agent emits.

Also: this epic owns the **host registration / enrollment** flow (per-host enrollment tokens, `hm agent enroll <token>`). Once a host is enrolled, its agent posts cron discoveries + Netdata metrics + future container/service discoveries through the same authenticated channel.

### EPIC-017 (SSH probe framework)

This epic absorbs **SSH-pull cron discovery** (monitor SSHes into a registered host and runs the same scanner that 002-007 runs locally), and the **"Install heartbeat" SSH-push variant** (UI button installs the wrapper on a remote host via SSH). The local-only "Install heartbeat" button from STAGE-002-009 becomes the SSH-push button enabled for any host with SSH credentials configured. Hosts the user has enrolled via EPIC-015 are also reachable here once the SSH probe is set up.

### EPIC-019 (Polish + release)

Confirm the `hm cron install-wrapper` CLI ships as a standalone downloadable binary (per platform) for unreachable hosts. Document the one-line install command on the release page.
