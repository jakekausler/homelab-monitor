## [STAGE-005-006] Entity-availability collector

**Summary:** Added the first real HA collector, `HaEntityAvailableCollector` — polls `/api/states`, applies a domain allow/deny filter, and emits per-entity availability + staleness metrics through the STAGE-005-004 cardinality cap (its first real consumer). Refinement against real HA drove a cardinality-cap default revision.

**Files — Backend (new):**
- `apps/monitor/homelab_monitor/plugins/collectors/integrations/homeassistant/ha_entity_available.py` — `HaEntityAvailableCollector` (name `ha_entity_available`, interval 30s, timeout 10s, concurrency_group `homeassistant`). Emits `homelab_ha_entity_available{entity_id,domain}` (1.0 real value / 0.0 for `unavailable`/`unknown`/empty, case-sensitive), `homelab_ha_entity_last_changed_seconds{entity_id,domain}` (now_utc − last_changed, clamped ≥0, emitted for ALL filtered entities incl. unavailable, tzinfo-guarded), and a self-diagnostic `homelab_ha_entity_last_changed_parse_errors{}` (only when >0). Domain allow-list (`DEFAULT_DOMAIN_ALLOW`) + optional deny as first-line cardinality control; the 005-004 cap (`cap_for(family)`) as safety net. `ctx.ha is None` / `get_states()→HaError` → failed `CollectorResult(ok=False)`. `extract_domain` inline (`entity_id.partition(".")[0]`).
- `apps/monitor/tests/test_ha_entity_available_collector.py` — 16 tests.

**Files — Backend (modified):**
- `apps/monitor/homelab_monitor/plugins/collectors/integrations/homeassistant/__init__.py` — registered `HaEntityAvailableCollector` in `_HA_COLLECTORS`.
- `apps/monitor/homelab_monitor/kernel/config.py` — `_DEFAULT_CARDINALITY_FAMILIES` (built-in per-family default cap **2500** for `homelab_ha_entity_available` + `homelab_ha_entity_last_changed_seconds`); the loader MERGES it under user YAML (`{**defaults, **yaml_families}`) so user per-family overrides win and the global `default` stays 500.
- `apps/monitor/tests/test_config.py` — cardinality-caps tests for the new defaults + the partial-override merge invariant + env-default-doesn't-touch-families.
- `apps/monitor/tests/test_ha_integration_bundle.py` — bundle test made membership-based + a `test_register_all_registers_ha_entity_available`.

**Refinement (3a+3b) note:** Backend host-integration stage, no viewport prompt. 3a: both `ha_up` + `ha_entity_available` registered. 3b (REAL HA, 2496 entities): collector ran `ok=True`, 0 errors; domain filter → 1906 entities. **Refinement-driven design revision:** the Design assumed the domain filter alone would land under the 500 cap; real data (1906 > 500) showed the cap firing + suggestion on first run (poor out-of-box UX). Fixed by shipping the 2500 built-in default (config layer, no collector change); re-validated against real HA → all 1906 emit, 0 dropped, 0 suggestions. The cap remains a safety net (still fires above 2500) and is fully user-tunable. ~5000 HA series is negligible for VictoriaMetrics.

**Code Review:** Opus APPROVED (single pass — no impl code changed after review; only this changelog note added). No Critical/Important findings; the collector's availability/staleness/parse-error branches and the config MERGE verified correct + well-tested. All findings reviewer-classified optional → reasoned no-action: (a) getattr duck-typed config fields mirror the `host.py` precedent; (b) per-tick YAML re-read is negligible; (c) the test `SimpleNamespace + # type: ignore[arg-type]` ctx double is justified/scoped. **Documented layering note (reviewer-recommended):** `_DEFAULT_CARDINALITY_FAMILIES` hard-codes HA-specific metric family names in the generic kernel `config.py` — a mild coupling of the generic cap-config to a specific integration, accepted per the existing `DEFAULT_ERROR_PATTERNS` precedent; the long-term home may be per-integration default registration.
