## [STAGE-004-043] Create-alert-from-query UX

**Summary:** Adds a reusable Create-Alert modal launched from the Logs Explorer to author vmalert log-alert rules, with a guided Simple mode and a raw Advanced mode, a live YAML preview, and validate-before-persist so unloadable rules can't break vmalert. Consumed by STAGE-042's persistence API.

**Key changes:**

**Frontend:**
- New `<CreateAlertModal>` (reusable; STAGE-044 will reuse via initialValues + initialMode props): two-pane (form + live YAML preview). **Simple mode** (default) — guided structured inputs (service dropdown, "message contains", threshold, window) compiled into safe LogsQL; the user never types pipe syntax; free-text is LogsQL-escaped. **Advanced mode** — raw LogsQL expr editor (CodeMirror) + a link to the LogsQL reference + advisory warnings.
- Launch button "Create alert from this query" in the Logs Explorer (plain-text query → Simple; LogsQL query → Advanced).
- React-query hooks `useUserRules`/`useCreateUserRule` (`apps/ui/src/api/userRules.ts`).
- Client-side YAML preview (`alertRuleYaml.ts`) mirroring STAGE-042's renderer; read-only CodeMirror YAML pane (`YamlPreview`/`YamlPreviewImpl`) + `@codemirror/lang-yaml` dep.

**Backend:**
- New backend heuristic expr validator (`expr_validate.py`) — rejects vmalert-unloadable exprs at POST/PATCH with 400 `invalid_expr`: logsql requires a `| stats` pipe; `| filter <reserved-word>` (e.g. `count`) rejected; unbalanced quotes/parens rejected. Pure Python (no docker/vmalert dependency; env-portable).
- Per-rule-file rendering: user rules now render to one `<rule_name>.yaml` per rule (was one aggregate file per kind), with orphan reconcile (deleted/disabled/renamed rules' files removed). `render_dirs_from_env` replaces `render_paths_from_env`.
- STAGE-042 expr-validation fix: the `expr` field wrongly forbade `"`/`\` (used the inline forbidden-char pattern); expr is a YAML block scalar where those are safe + REQUIRED for LogsQL. Now uses the block pattern (`allow_newlines=True`). One-line fix + test.
- Reserved-keyword alias fix: the scaffold/builder used `count` (a reserved LogsQL word) as the stats alias → vmalert rejected `| filter count:`. Renamed to `match_count`.
- `for_duration` compound durations: backend regex widened to `^(\d+[smhd])+$|^0s$` to accept `2h30m` (matches the frontend + the advertised placeholder); backend `invalid_rule` 400s now surface inline in the modal.

**Important notes:**
- One bad rule no longer disables all user rules: validate-before-persist (the heuristic validator) keeps unloadable rules out of the vmalert glob (which rejects the whole glob on any parse error).
- Exact-parser vmalert `-dryRun` validation + a user-rules MANAGEMENT UI (list/enable/disable/edit/delete) are DEFERRED to STAGE-004-043A.
- Rig-verified: real Home Assistant model-signature queries + special-char "contains" inputs persist and load in vmalert health=ok; invalid exprs rejected 400.

**Out-of-band (separate concern, same commit):**
- Models tab detail-pane horizontal scroll (`ModelsDebugPage.tsx` min-w-0); eslint `react-hooks/incompatible-library` pinned to warn (`eslint.config.js`) to stop its severity floating across the plugin's ^7.x range and intermittently blocking commits.

## [STAGE-004-043A] User-rules management UI + prod vmalert dry-run validation

**Summary:** Added a management surface for user-authored alert rules (previously write-only from the UI) plus a best-effort vmalert exact-parser dry-run validation layer. Includes per-rule vmalert health aggregation, CRUD UI (list/enable/disable/edit/delete), and optional dry-run validation gated by environment flag.

**Key changes:**

**Backend:**
- New `GET /api/logs/user-rules-health` endpoint — aggregates per-rule vmalert load health (ok/err/unknown + last_error) by fanning out to both vmalert instances (`vmalert-logs:8880`, `vmalert-metrics:8880`), filtering to the `user-rules-logs`/`user-rules-metrics` groups. Degraded-200 contract: returns partial/empty health when a vmalert instance is unreachable, never 500. Session-auth required.
- New `kernel/logs/vmalert_dryrun.py` — fail-open vmalert `-dryRun -rule.validateExpressions` runner (`run_vmalert_dryrun`, `DryRunResult`, `DryRunRunner` protocol). Availability-detects docker CLI + image + mount; skips (never raises) on any miss/timeout/OSError. Wired into the repo `_validate_and_render_check` seam via an injected `dryrun_runner` (defaults None), ordered after the heuristic floor. Gated behind `HOMELAB_MONITOR_VMALERT_DRYRUN_ENABLED` (default off); image/timeout/mount from env. A dry-run failure maps to the existing 400 `invalid_expr` (check="dryrun").
- New Pydantic models `LogUserRuleHealth` + `LogUserRulesHealthResponse`. Public `GROUP_LOGS_NAME`/`GROUP_METRICS_NAME` aliases exported from `user_rules_render.py`.

**Frontend:**
- New "Alert Rules" management tab (`/logs/user-rules`, `routes/logs/UserRulesTab.tsx`) mirroring the SilenceAllowlist pattern: desktop table + mobile cards listing rule_name/kind/severity/enabled + a vmalert HEALTH column (joined by rule_name; missing → "unknown"). Per-row enable/disable toggle, edit, and delete (confirm).
- `CreateAlertModal` gained edit mode via an optional `editRuleId` prop: PATCHes only the 5 mutable fields (expr/severity/summary/description/for_duration), renders rule_name read-only, excludes the edited rule from the duplicate-name check, swaps title/button to "Edit alert rule"/"Save changes". The create path is byte-stable (STAGE-044 reuses it).
- New react-query hooks in `api/userRules.ts`: `useUserRulesHealth` (20s refetchInterval), `usePatchUserRule`, `useDeleteUserRule`, `useEnableUserRule`, `useDisableUserRule`. Route + tab registration in `router.tsx` / `LogsLayout.tsx`.

## [STAGE-004-044] Create-alert-from-signature + saved-query shortcuts (EPIC-004 final)

**Summary:** Added "Create alert" shortcuts from the Signature Catalog and Saved Queries into the shared CreateAlertModal, pre-filling sensible defaults per source. Frontend-only stage completing EPIC-004.

**Key changes:**

**Frontend:**
- New pure module `apps/ui/src/routes/logs/alertPrefill.ts`: `prefillFromSignature` (advanced-mode LogsQL count-threshold from `scaffoldLogsqlExpr(templateToLogsQl(template_str))`, severity warning, for 1m), `prefillFromSavedQuery` (mirrors 043 mode-detection), `deriveRuleName` (deterministic `^[a-zA-Z_][a-zA-Z0-9_]*$` sanitizer — same source → same name → powers dedupe via the modal's existing dup-name warning), and `composeServicesExpr`.
- **Service-scope fix (found in Refinement):** `prefillFromSavedQuery` now bakes the saved query's `selected_services` into the alert expr via `composeServicesExpr`, replicating the backend `_compose_services_expr` exactly (`service:"<svc>" AND source_type:"<src>"` clauses, OR'd then AND'd with the query; empty base → just the services clause). Service-scoped saved queries (single or multi-service, e.g. "nginx errors" → docker:nginx; "infra services overview" → 3 services) open in Advanced mode with the full service filter in the (editable) expr. Without this, alerts from service-scoped saved queries fired on ALL services.
- Extracted `scaffoldLogsqlExpr` into a leaf module `apps/ui/src/components/logs/alertExpr.ts` (re-exported from CreateAlertModal — existing importers unbroken) so the pure prefill module + its test don't transitively pull CodeMirror (jsdom-hostile).
- Wired "Create alert" buttons: `SignatureDetailPage.tsx` (Bell, in the detail action row) and `SavedQueriesPanel.tsx` (per-row Bell action). Both open the shared CreateAlertModal in create mode.
- **Latent 409 race fix (found in Refinement):** the modal's 300ms client-side uniqueness debounce timer could fire after the 409 catch set the name-conflict error and clobber it back to null (only under `make verify`'s concurrent backend-pytest + UI-vitest CPU contention). Fixed: track the debounce timer in a ref; a `setAuthoritativeNameError` helper cancels the pending debounce before setting the submit-path error (client dup backstop + 409 catch).
