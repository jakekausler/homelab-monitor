2026-06-23 4c8c752 EPIC-006/STAGE-006-014: DNS health probe (composite up + direct :53 probe)

Added the `pihole_dns_health` collector (`name="pihole_dns_health"`, 30s, `concurrency_group="pihole"`) — the epic's truest "is Pi-hole actually resolving" signal. It sends a real DNS A-record query for `dns.google.com` from the monitor over UDP directly to Pi-hole's `:53` listener (host LAN IP `192.168.2.148:53`) and emits a DNS-decisive composite `homelab_pihole_up`. Because the probe originates from the monitor and is independent of the Pi-hole REST API, it catches the "API up but DNS wedged" failure that an API-only check would mask.

The DNS query is a new pure-stdlib async UDP helper in `kernel/dns/` (`resolve_a(resolver_ip, qname, *, port=53, timeout_seconds) -> DnsProbeResult`): it builds a minimal A-record query (12-byte header + QNAME-encoded question, QTYPE=A, QCLASS=IN), sends it over `asyncio` datagram transport, and parses only the QR/RCODE/TC bits + ANCOUNT (no answer-record decompression). It NEVER raises — timeout, socket error, malformed/too-short response, ID mismatch, and a non-ASCII qname all map to a typed `DnsProbeResult`. It is parameterized by resolver IP so STAGE-006-015's Pi-hole-vs-WAN split-check can reuse it unchanged.

Metrics: `homelab_pihole_up` (gauge 1/0, always emitted — 1 iff the probe resolves: QR set, RCODE NOERROR, ANCOUNT≥1, not truncated); `homelab_pihole_dns_probe_seconds` (latency gauge — emitted only on a real round-trip, i.e. when a response was received even SERVFAIL/NXDOMAIN/REFUSED/no-answer/truncated; omitted on timeout/socket-error/malformed/id-mismatch so latency graphs aren't polluted); and `homelab_pihole_dns_probe_result{outcome}` (one-hot gauge, exactly one series per run; outcome ∈ ok/timeout/servfail/nxdomain/refused/malformed/socket_error/id_mismatch/truncated/no_answer). The composite is DNS-decisive only — the API-reachability signal is NOT folded in (STAGE-006-016 builds the richer "DNS up but API down" view from the separate `homelab_collector_run_*` / `homelab_pihole_api_took_seconds` series).

Config: added `PiholeConfig.dns_host` (env `HOMELAB_MONITOR_PIHOLE_DNS_HOST`, deriving from the base_url hostname when empty) + `dns_port` (env `HOMELAB_MONITOR_PIHOLE_DNS_PORT`, default 53), giving 015 a clean home for a second resolver. The resolver host/port are injected into the collector post-construction in `lifespan.py` (mirroring the 013 UnboundStatsCollector wiring). Backend-only; no migration, no API/frontend change. 100% branch coverage; validated live on the prod rig (`make dev-prod` + collector retry → `homelab_pihole_up 1.0`, `outcome="ok"`, ~1ms latency against the real Pi-hole DNS, proving UDP egress from the bridge network works). Also fixed a pre-existing seed-dependent test-isolation flake in `tests/test_api_docker_pull_and_restart.py` (its lifespan-enabled tests start the full scheduler; on a `PYTHONHASHSEED` where the unbound collector's hash-offset is 0, its docker-exec tick fired inside the test's strict httpx mock window — the autouse mock now tolerates the `/containers/.../exec` + `/exec/.../start|json` endpoints, optional+reusable).
