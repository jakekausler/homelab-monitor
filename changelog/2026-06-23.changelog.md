2026-06-23 ce24b8b EPIC-006/STAGE-006-010: Pi-hole FTL diagnostic-messages collector

Adds `PiholeFtlMessagesCollector` (`pihole_ftl_messages`), a single-endpoint collector polling Pi-hole v6 `/api/info/messages` every 60s to surface FTL's diagnostic messages (inaccessible adlists, high-load warnings, config issues, etc.) as metrics.

Metrics emitted:
- `homelab_pihole_messages_count` (gauge) = total active-message count (`len(messages)`), ALWAYS emitted including `0` when there are no messages — so the planned `PiholeMessagesPresent` alert can fire on `> 0` without needing `absent()`.
- `homelab_pihole_messages{type}` (gauge) = per-type COUNT, grouped by each message's `type` field (duplicate types collapse: two `type="LIST"` messages → `{type="LIST"}=2`, not two series). Present types only (no zero-fill, no hardcoded FTL enum). A non-string/missing `type` falls back to `{type="unknown"}`; non-dict message entries are skipped.
- `homelab_pihole_api_took_seconds{endpoint="info/messages"}` per run.

The message text bodies (`plain`/`html`) are deliberately NOT emitted as metric labels (cardinality); they are fetched at panel render-time / alert-annotation render-time (a future UI/alert stage's concern). No cardinality cap (the FTL message-type enum is a small bounded internal set).

Design note: the card named the per-type metric `homelab_pihole_message{type}` (singular), which implied one series per message; live recon showed duplicate types must collapse to a count, so the metric was corrected to the plural `homelab_pihole_messages{type}` with per-type-count semantics. Documented semantic: `messages_count` = total list length (includes any malformed entries); the per-type breakdown groups only well-formed entries, so sum(per-type) may be < count if malformed entries are present.

10 unit tests at 100% branch (happy path, empty list, malformed payload/entry, unknown-type fallback, all failure paths). Live-validated on prod against the real Pi-hole v6.6.2: 3 messages → `messages_count=3`, correctly grouped to `{type="LIST"}=2` + `{type="LOAD"}=1`.

2026-06-23 ebd4e3f EPIC-006/STAGE-006-009: Pi-hole FTL-health + DB collector

Adds `PiholeFtlHealthCollector` (`pihole_ftl_health`), a two-endpoint collector polling Pi-hole v6 `/api/info/ftl` (FTL process health) and `/api/info/database` (long-term DB metrics) every 60s. Mirrors the `gravity.py` two-endpoint pattern with per-endpoint resilience (`ok = ftl_ok or db_ok`).

Metrics emitted:
- From `/api/info/ftl` (fields nested under the top-level `ftl` object): `homelab_pihole_ftl_uptime_seconds`, `_ftl_cpu_percent`, `_ftl_memory_percent`, `_privacy_level`, `_dnsmasq_cache_insertions`, `_dnsmasq_cache_evictions`.
- From `/api/info/database`: `homelab_pihole_db_size_bytes` and `homelab_pihole_db_queries_total` (the on-disk total `queries_disk`, ~11.6M live — not the in-memory window).
- `homelab_pihole_api_took_seconds{endpoint}` per endpoint.

Does NOT re-emit `homelab_pihole_gravity_domains` (STAGE-006-007 owns it) and does NOT emit host CPU/mem (node-level scope-out). 16 unit tests at 100% branch (full two-endpoint resilience matrix + a regression test pinning the `payload["ftl"]` nesting).

Refinement caught a real bug: the collector initially read FTL fields off the top-level payload, but Pi-hole v6 nests them under an `ftl` key, so all 6 FTL metrics were silently absent in production despite green unit tests (the fixtures were mis-shaped to match the buggy code). Fixed to descend into `payload["ftl"]`; fixtures re-nested; regression test added. Live-validated on prod (10/10) against the real Pi-hole v6.6.2: FTL process counters + the ~1.1 GB DB size + ~11.6M on-disk query count all confirmed.

Two follow-ups tracked (no silent deferral): `homelab_pihole_query_logging_enabled` deferred to STAGE-006-018 (lives in `/api/config`, not `/api/info/ftl`); and a latent `PiholeRestClient._get()` gap (does not detect a 200-response carrying an `{"error": ...}` body — 5-collector blast radius) recorded as a STAGE-006-018 deliverable + regression item.

2026-06-23 ee3239a EPIC-006/STAGE-006-008: Pi-hole blocking-state collector

Adds `PiholeBlockingCollector` (`pihole_blocking`), a single-endpoint collector polling Pi-hole v6 `/api/dns/blocking` every 30s to surface whether DNS blocking is currently active — a common foot-gun is disabling blocking temporarily and forgetting.

Metrics emitted:
- `homelab_pihole_blocking_enabled` — gauge `1.0` iff the API `blocking` field equals `"enabled"`, else `0.0` (fail-closed: `disabled`/`failed`/`unknown`/non-string/missing all map to `0.0`, i.e. "not protecting").
- `homelab_pihole_blocking_timer_seconds` — gauge of remaining seconds, emitted ONLY when a temporary-disable timer is active; OMITTED (never zeroed) when no timer is set, so the "disabled indefinitely" case is distinguishable from "disabled with auto-re-enable" via series presence/`absent()`. A legitimate `timer == 0` (0s remaining) IS emitted as `0.0`.
- `homelab_pihole_api_took_seconds{endpoint="dns/blocking"}` — per-call API latency (standard).

Mirrors the `upstreams.py` collector pattern (object-local → isinstance → cast payload narrowing, no source `# pyright: ignore`, `_parsing.as_float` reuse). Registered in the Pi-hole bundle `__init__.py`. 14 unit tests at 100% branch coverage, plus a `timer=0`-boundary contract test. Live-validated on the prod stack against the real Pi-hole (v6.6.2): `_blocking_enabled=1` matching the live enabled state, `_api_took` ~43µs, `_blocking_timer_seconds` correctly absent (live `timer=null`), self-metric `homelab_collector_run_success_total{name="pihole_blocking"}` incrementing.

Consumed by the planned STAGE-006-016 alerts `PiholeBlockingDisabled` (`_blocking_enabled == 0` for >5m) and `PiholeBlockingDisabledIndefinitely` (`absent(_blocking_timer_seconds)` for >15m — `for:` must exceed VM staleness; note recorded on the 016 card).
