## STAGE-001-020: Grafana + dashboards-as-code provisioning

**Commit**: `0c6abfd` (implementation)

### Summary

Embedded Grafana at `/metrics` via a sandboxed same-origin iframe + provisioned VictoriaMetrics + VictoriaLogs datasources + a host-overview dashboard. The monitor reverse-proxies Grafana at `/api/grafana/{path:path}` with the same cookie-session auth + Origin/Referer enforcement + reverse-proxy-aware Host validation as the STAGE-019 Karma proxy. Grafana's default `X-Frame-Options: DENY` is stripped at the proxy so the iframe renders.

### Highlights

- **Backend**: new `grafana` router (catch-all proxy mirroring karma.py with grafana_unavailable error code, X-Frame-Options strip, env-driven HOMELAB_MONITOR_GRAFANA_URL/TIMEOUT_S). 100% kernel coverage maintained.
- **UI**: new `/metrics` route (full-bleed iframe in kiosk mode + "Open in Grafana" escape link), Metrics sidebar item enabled.
- **Compose**: pinned `grafana/grafana-oss:12.3.3` as internal-only sidecar. Anonymous Viewer auth (auth boundary at monitor proxy). Plugin install: `victoriametrics-logs-datasource 0.26.3` via `GF_INSTALL_PLUGINS` (first-boot HTTPS dependency).
- **Provisioning**: VictoriaMetrics (Prometheus type) + VictoriaLogs (native plugin — NOT Loki; VL lacks full Loki API parity) + host-overview dashboard with 6 panels (CPU%, load avg, memory%, disk% per mountpoint, network throughput, uptime). Schema v40, Grafana 12.x compatible.
- **Tests**: 24 backend unit tests + 3 slow integration tests + 6 UI tests added. Backend: 1107 passing, 100% coverage. UI: 110 passing.
