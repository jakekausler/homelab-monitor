## STAGE-003-010: Pull & Restart action — compose-aware + audit + confirm

**Epic:** EPIC-003 (Docker)
**Date:** 2026-05-26
**Commit:** `4e46d5f`

### Summary

First kernel-driven Docker WRITE operation. A confirm-gated per-container button executes `docker compose -p <project> -f <file> {pull|build} <service> && docker compose ... up --force-recreate -d <service>` from inside the monitor container against the host's compose file, captures stdout/stderr, writes a `compose_actions` audit row + `audit_log` row, emits Prometheus metrics, and refreshes the UI. The docker socket mount widens from `:ro` to RW; the compose file + build contexts are mounted RO. Confirm phrase is the literal string `"pull"` (server-validated). Concurrency: per-container `asyncio.Lock` + 409 Conflict on duplicate submit. Phase state machine: `pulling`/`building` → `restarting` → `success`/`failed`/`timeout`/`killed`. After a successful action, the appropriate image-update refresher (registry digest OR local-build source hash) is invoked so the badge clears within ~2s instead of waiting for the next scheduled tick. Auth: session OR new `Scope.DOCKER_WRITE`.

This stage establishes the audit + confirm + capture-transcript pattern that EPIC-009 (auto-fix subsystem) will reuse for Claude-driven runbook real-runs. The `compose_actions` table is kept separate from EPIC-009's future `runbook_runs` (cross-epic carry-forward documented).

### Behavior changes

- **NEW**: `POST /api/integrations/docker/containers/{name}/pull-and-restart` — triggers the action; gated by `require_user_or_token({Scope.DOCKER_WRITE})`; body `{"confirm_phrase": "pull"}` (case-insensitive, trimmed); returns 202 + `{action_id, state}`. Returns 409 if any row with `state in ('pulling','building','restarting','running')` already exists for this container.
- **NEW**: `GET /api/integrations/docker/compose-actions/{id}` — single action detail (stdout/stderr included). Session auth.
- **NEW**: `GET /api/integrations/docker/compose-actions?container={name}&limit=N` — list recent actions for a container. Session auth.
- **NEW**: `Scope.DOCKER_WRITE = "docker:write"` — admins can mint API tokens with this scope to trigger Pull & Restart programmatically.
- **NEW**: `homelab_compose_action_success_total{container,action}` + `homelab_compose_action_failed_total{container,action,reason}` Prometheus counters.
- **NEW**: `DockerComposeActionFailed` vmalert rule (warning severity, fires on failure increase in 10m window).
- **NEW**: `compose_actions` table (migration 0028) with `state` widened in 0029 (`pulling`/`restarting` instead of `running`) and again in 0030 (`building` added for locally-built containers).
- **NEW**: Per-container `asyncio.Lock` dict on `ComposeActionRunner` (mirrors `Scheduler._group_locks`). Lock acquired inside the background task; API-side 409 guard prevents two concurrent triggers from reaching the runner.
- **NEW**: `LocalBuildUpdateCollector.refresh_container(container_name)` — on-demand refresher invoked after a successful local-build rebuild; resets `baseline_source_hash` + `baseline_image_id` to current values and writes `update_available=0`. Mirrors `image_update_collector.refresh_container` (added in earlier wave for remote-image refresh after pull).
- **NEW**: `compose_actions.state` phase column. Initial state set to `pulling` (remote-image) or `building` (local-build). Transitions to `restarting` before `up --force-recreate -d` runs. Terminal: `success`/`failed`/`timeout`/`killed`.
- **NEW**: UI "Rebuild & Restart" button label for locally-built containers; "Pull & Restart" for remote-image. Confirm phrase is `"pull"` for both (API contract uniformity). Modal title/description reflect the action type.
- **NEW**: `useGetComposeAction(actionId)` polls every 2s while `state` is non-terminal; stops on terminal.
- **NEW**: Optimistic local state in `ActionsCell` bridges the gap between mutation success and the first poll showing the new row.
- **NEW**: On terminal state transition, `ActionsCell` invalidates `dockerImageUpdateQueryKeys.summary` + `dockerImageUpdateQueryKeys.detail(containerName)` so the badge clears immediately.
- **NEW**: Toast notifications on terminal state (success / failed / timeout / killed) — guarded by an `invalidatedRef` to fire at most once per `action_id`.
- **CHANGED**: Docker socket mount widens from `:ro` to RW in `deploy/compose/docker-compose.yml`. Compose file mounts RO at `/host-compose`. Security boundary is at the docker socket level; accepted per EPIC-003 cross-stage criterion #18.
- **CHANGED**: Monitor container `group_add` gains `homelab-compose` GID (default 994, configurable via `HM_HOST_COMPOSE_GID`). Container user needs read access to `/storage/docker/compose/.env` for compose env interpolation.
- **CHANGED**: Monitor container env adds `HOME=/tmp` + `DOCKER_CONFIG=/tmp/.docker`. Docker CLI requires a writable HOME for its config dir.
- **CHANGED**: `apps/monitor/Dockerfile` installs `docker-ce-cli`, `docker-compose-plugin`, `docker-buildx-plugin` from Docker's official APT repo (~50 MB image growth).

### Architecture decisions (locked in STAGE-003-010.md)

Original scope:
- D-COMPOSE-AWARE — `docker compose -f <file> pull <svc> && docker compose -f <file> up -d <svc>`, NOT raw `docker pull`. Preserves env, volumes, networks, dependencies.
- D-INSIDE-MONITOR — action runs INSIDE the monitor container. NOT via a separate fixer-runner pattern (reserved for EPIC-009 Claude runs).
- D-CONFIRM-API-SIDE — confirm gate enforced API-side, not just UI-side. Even a hand-crafted curl must pass.
- D-CONFIRM-EXTENDS-SPEC-7-2 — Pull & Restart extends spec §7.2's confirm-on-destructive list (the list was illustrative).
- D-AUDIT-EVERY-ATTEMPT — audit_log row + compose_actions row for EVERY attempt, regardless of outcome.
- D-COMPOSE-ACTIONS-OWN-TABLE — `compose_actions` is its own table, not a reuse of EPIC-009's future `runbook_runs`.
- D-SOCKET-RW-WIDEN — socket mount widens from `:ro` to RW this stage. Security boundary documented + accepted.
- D-AUTO-UPDATE-OFF — auto-update is permanently OFF. Always manual.

Design phase decisions (user-selected):
- Confirm UX: static phrase `"pull"` (typed in modal, server-validated). Lower friction for single-user homelab.
- Auth: Session OR new `Scope.DOCKER_WRITE` on API tokens. Future-proofs bot/CLI flows.
- Concurrency: per-container `asyncio.Lock` via `dict[str, asyncio.Lock]` (mirrors `scheduler._group_locks`).
- Compose path discovery: reuse `BuildSourcesLoader` from STAGE-003-009.
- Output storage: TEXT columns on `compose_actions`, truncated at 1M chars on write.
- Background task: `asyncio.create_task()` + module-level `set[asyncio.Task]` + shutdown cancellation.
- Timeout: 5 min default (`HOMELAB_MONITOR_COMPOSE_ACTION_TIMEOUT_SECONDS=300`). SIGTERM, 10s grace, then SIGKILL.

ADDENDUM Q2 — Container resolution uses `com.docker.compose.service` label via `socket_client.inspect_container`. Robust to docker-compose project-name prefixes + v1/v2 naming.

Refinement-wave decisions:
- 409 Conflict on duplicate submit (per-container, checked at API layer before reaching the runner).
- Phase column splits `running` into `pulling`/`building` (initial) + `restarting` (post-pull). Migration 0029 widens CHECK; migration 0030 adds `building`.
- Optimistic UI bridge — local React state stores `optimisticActionId` to show "Pulling…"/"Building…" instantly without waiting for first poll.
- Cache invalidation on terminal transition — invalidates `dockerImageUpdateQueryKeys.summary` + `.detail(name)` so badge clears within ~2s.
- Post-success refresher dispatched on `resolved.is_local_build` flag: registry-digest refresher for remote-image, source-hash refresher for local-build.
- `docker compose up -d` → `docker compose up --force-recreate -d` (handles already-running-container name conflicts).
- Compose project name extracted from container's `com.docker.compose.project` label (NOT the cwd basename). Passed as `-p <project>` flag.
- Build vs pull detection from `ResolvedCompose.is_local_build` (from the compose service's `build_context`). Local-build runs `docker compose build`; remote-image runs `docker compose pull`.
- Auto-generated `docker-compose.override.yml` from `build-sources.yaml`'s `build_context_roots` — bind-mounts host build-context paths AT the same path inside the container so `docker compose build` resolves them correctly. Operator runs `make generate-build-mounts` when `build-sources.yaml` changes; must invoke compose with both `-f` flags explicitly (auto-loading disabled by `-f` of the main file).
- Dedicated `homelab-compose` host group + `chmod 0640` on `.env` files listed in `scripts/host-setup-shared-files.conf`. Provisioned via `scripts/host-setup.sh`.

### Code review

Extended `insert_audit(...)` to accept `audit_id: str | None`; pre-resolution failure path now generates `audit_id = uuid7()` upfront and passes it to BOTH `update_terminal_state` (via `audit_log_id`) AND `_write_audit_row` (via `audit_id`). Audit row + compose_actions row now correctly cross-reference end-to-end.

  - **I7 (relative path in conf):** `scripts/host-setup-shared-files.conf` referenced `docker-compose.override.yml` as a relative path; `readlink -f` resolved against wrong cwd. Fixed to absolute path.
  - **I8 (hardcoded desktop user):** `scripts/host-setup.sh` had `COMPOSE_GROUP_DESKTOP_USER="jakekausler"` hardcoded. Now env-overridable via `COMPOSE_GROUP_DESKTOP_USER=… sudo bash scripts/host-setup.sh`, defaulting to `$SUDO_USER` if available.

### Operational notes

- Operator runs `sudo bash scripts/host-setup.sh` once to provision the `homelab-compose` group + apply 0640 to `.env`.
- Operator runs `make generate-build-mounts` after editing `build-sources.yaml` to refresh `docker-compose.override.yml`.
- Operator MUST invoke docker compose with both `-f` flags explicitly: `docker compose -f deploy/compose/docker-compose.yml -f deploy/compose/docker-compose.override.yml ...`. The `-f` flag of the main file disables auto-loading of override files.
- `HM_HOST_COMPOSE_GID` in `.env` defaults to 994 (a sentinel matching the test host). Operators on other hosts MUST run `getent group homelab-compose | cut -d: -f3` after `host-setup.sh` and set the value accordingly.
