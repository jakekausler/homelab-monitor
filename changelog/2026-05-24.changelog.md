## STAGE-003-008: Image-update detection (registry digest)

**Epic:** EPIC-003 (Docker)
**Date:** 2026-05-24
**Commit:** `e003b48`

### Summary

Adds diun-style image-update detection. A new `ImageUpdateCollector` runs on a 6h cadence (env-configurable), and for every running container whose image was pulled from a registry, compares the local `RepoDigest` against the registry's current digest for the container's tag (HEAD manifest with dual Docker v2 + OCI v1 Accept header). Differences emit `homelab_image_update_available{name,image,current_digest,latest_digest} 1`; an `info`-severity vmalert routes to dashboard-only. The container grid shows a blue "Update available" pill (desktop + mobile parity); a click-through drill-down at `/integrations/docker/containers/$name/image-update` shows full digest detail. A new shared `digest.ts` helper truncates SHA digests consistently across drill-down AND main grid. The collector is READ-ONLY тАФ no `docker pull` (deferred to STAGE-003-010).

### Behavior changes

- **NEW**: `homelab_image_update_available{name,image,current_digest,latest_digest}` gauge тАФ 1 when registry digest != local digest, 0 otherwise.
- **NEW**: `homelab_image_update_last_checked_seconds{name}` gauge тАФ epoch of last successful check (drives staleness alerts).
- **NEW**: `homelab_registry_rate_limit_remaining{registry}` gauge тАФ Docker Hub `RateLimit-Remaining` parsed from manifest response headers.
- **NEW**: `homelab_image_update_check_skipped{reason}` counter тАФ increments per skip; `reason="rate_limit"` for hard-cap skips.
- **NEW**: `DockerImageUpdateAvailable` vmalert rule (`info` severity, `for: 30m` flap suppression).
- **NEW**: `GET /api/integrations/docker/image-updates/summary` тАФ aggregate summary for the grid + rate-limit banner.
- **NEW**: `GET /api/integrations/docker/containers/{name}/image-update` тАФ per-container detail for the drill-down.
- **NEW**: Image-update background events loop тАФ second filtered subscriber on the Docker socket events stream (`filters={"type":["image"]}`); on `image pull` events triggers an out-of-band collector run.
- **NEW**: Per-container image-update state persisted in new `image_update_state` table (migration 0026), keyed by `container_name`, mirroring `docker_override_ownership` precedent.

### Architecture decisions (locked in STAGE-003-008.md)

- D-REGISTRY-READ-ONLY тАФ image-update detection is PURELY READ; no `docker pull`
- D-DIGEST-COMPARISON тАФ by cryptographic `RepoDigest`, not tag string
- D-PUBLIC-IMAGES-V1 тАФ anonymous bearer-token flow only; private registry credentials deferred
- D-PER-IMAGE-FAILURE-ISOLATION тАФ one image's failure does NOT abort the tick
- D-RATE-LIMIT-AWARE тАФ `RateLimit-Limit` / `RateLimit-Remaining` headers parsed, surfaced as gauge metric
- D-INFO-SEVERITY тАФ image-update alerts are `info` (dashboard-only / Karma, not paging)
- D-STATE-SIDECAR-TABLE тАФ new `image_update_state` table (migration 0026)
- D-LATEST-ALWAYS-CHECK тАФ `:latest` / `:main` / `:edge` / rolling tags ARE checked
- D-RATE-LIMIT-HARD-CAP-PLUS-BANNER тАФ skip remaining checks + UI banner when `RateLimit-Remaining < 10`
- D-BADGE-TEXT-PLUS-BLUE-PILL тАФ blue `<Link>` pill rendered on BOTH desktop row AND mobile card
- D-EVENTS-FILTERS-KWARG тАФ `DockerSocketClient.events(filters=...)` kwarg drives the new image-events subscriber
- D-LABEL-KEY-NAME тАФ metrics use `name=` (matching STAGE-003-004 convention), NOT `container=`
- D-SUMMARY-SIBLING-ENDPOINT тАФ `/image-updates/summary` is a sibling of `/probes/summary` (not an extension)
- D-PER-CONTAINER-ENDPOINT тАФ `/containers/{name}/image-update` for drill-down
- D-IMAGE-REF-PARSER-INHOUSE тАФ in-house parser; no new dependency

## STAGE-003-009: Image-update detection (locally-built images)

**Epic:** EPIC-003 (Docker)
**Date:** 2026-05-24
**Commit:** `304356b`

### Summary

Extends image-update detection to containers built from local source (no registry digest available). A new `LocalBuildUpdateCollector` runs on a 30-min cadence (env-configurable), and for every running container whose image was locally built, hashes the build context (Dockerfile + `COPY`/`ADD` source paths, filtered via `.dockerignore`-style pathspec rules) and compares against a stored baseline. Baseline tracking is pinned to the container's `image_id` so `available=true` persists across collector ticks until a real rebuild lands. Build context discovery is driven by a hot-reloadable `build-sources.yaml` (pydantic-validated) supporting multiple compose files and explicit hostтЖТcontainer path remapping, so operators with build contexts at absolute host paths outside the default mount can map them in without a code change. The container grid's "Update available" pill now distinguishes registry vs local-build updates via a `source` discriminator; the drill-down page surfaces the build-context path + source hash + baseline source hash. Operator default behavior is unchanged when no `build-sources.yaml` is present.

Two startup-tick improvements also landed: `Scheduler.await_immediate_run(...)` gives lifespan code a way to await the first run of an immediately-scheduled job, and `image_update_checker` + `local_build_update_checker` are both awaited at startup so operators see populated data within seconds of deploy (vs the 6h / 30min default cadences).

### Behavior changes

- **NEW**: `homelab_local_build_update_available{name,image,source_hash,baseline_source_hash} 1` gauge тАФ 1 when current source hash != baseline, 0 otherwise.
- **NEW**: `homelab_local_build_last_checked_seconds{name}` gauge тАФ epoch of last successful check.
- **NEW**: `homelab_local_build_update_check_skipped{reason}` counter тАФ `reason="context_missing"`, `"limit_exceeded"`, `"compose_unreadable"`, etc.
- **NEW**: `/api/integrations/docker/image-updates/summary` and `/containers/{name}/image-update` payloads gain a `source: "registry" | "local_build"` discriminator, plus `baseline_source_hash`, `compose_service`, and `build_context_path` fields for local-build entries.
- **NEW**: Local-build entries surface in the existing blue "Update available" pill on desktop row AND mobile card; the drill-down route at `/integrations/docker/containers/{name}/image-update` renders source-hash + baseline + build-context-path + compose-service when `source="local_build"`.
- **NEW**: `build-sources.yaml` operator config (mounted at `/config/docker/build-sources.yaml`); hot-reloaded by a 30s loop. Supports `compose_files: [...]` (1+ paths) and `path_remaps: [{host: ..., container: ...}, ...]` (0+ entries). Absent or empty file = default behavior preserved.
- **NEW**: Per-container build-hash state persisted in new `docker_build_hashes` table (migration 0027), keyed by `container_name` with `source_hash`, `baseline_source_hash`, `baseline_image_id`, `image_id`, `last_checked_at`, and `last_error` columns.
- **NEW**: `Scheduler.await_immediate_run(job_name)` тАФ returns an `asyncio.Future` resolved when the job's next tick completes (success or failure). Used by lifespan to gate startup until first collector ticks have populated state.
- **CHANGED**: `Scheduler._tick` now resolves any pending immediate-run Future via a new `_signal_awaitable_done` helper on every early-return path (job not found, job disabled, exception) тАФ not just the success path. Prevents `await_immediate_run` callers from hanging.

### Architecture decisions (locked in STAGE-003-009.md)

Original scope:
- D-NEW-BUILD-HASHES-TABLE тАФ separate `docker_build_hashes` table (not extension of `image_update_state`), keyed by `container_name`
- D-PATHSPEC-LIBRARY тАФ adopt the `pathspec` library for `.dockerignore`-style filtering (Git-style + Docker-style semantics)
- D-HASHING-LIMITS-STAGEDOC-PLUS-ABORT тАФ per-file (`MAX_FILE_BYTES`) + total (`MAX_TOTAL_BYTES`) hashing budgets; ABORT and record `last_error` when exceeded, never silently truncate
- D-FIRST-CHECK-BASELINE тАФ on first check for a container, current hash IS the baseline (`available=0`); only subsequent ticks can flip to `available=1`
- D-COMPOSE-DIR-WHOLE-MOUNT тАФ mount the operator's compose dir read-only at a known container path (`/compose`); pair with `path_remaps` for contexts outside the dir
- D-SEED-UDO-VIEWER тАФ dev rig seeds a synthetic `udo-viewer` build context (Dockerfile + tiny source tree) so the local-build path is exercisable on a clean rig

Mid-Refinement scope expansions:
- D-BASELINE-TRACKING-WITH-IMAGE-ID тАФ baseline is preserved across ticks; only reset when `image_id` changes (signaling a real rebuild). Without this, `available=true` self-erased on the next tick because the "current" hash was being written into the baseline slot.
- D-BUILD-SOURCES-YAML-CONFIG тАФ operator-supplied `/config/docker/build-sources.yaml` (pydantic schema, hot-reloaded every 30s) is the single source of truth for which compose files to read and how to remap host paths
- D-MULTI-COMPOSE тАФ `compose_files` is a LIST; supports operators with multiple compose stacks under different directories
- D-PATH-REMAP-EXPLICIT тАФ `path_remaps` is explicit (`host` тЖТ `container`) rather than implicit; no auto-detection or glob inference
