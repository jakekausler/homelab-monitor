## STAGE-003-008: Image-update detection (registry digest)

**Epic:** EPIC-003 (Docker)
**Date:** 2026-05-24
**Commit:** `e003b48`

### Summary

Adds diun-style image-update detection. A new `ImageUpdateCollector` runs on a 6h cadence (env-configurable), and for every running container whose image was pulled from a registry, compares the local `RepoDigest` against the registry's current digest for the container's tag (HEAD manifest with dual Docker v2 + OCI v1 Accept header). Differences emit `homelab_image_update_available{name,image,current_digest,latest_digest} 1`; an `info`-severity vmalert routes to dashboard-only. The container grid shows a blue "Update available" pill (desktop + mobile parity); a click-through drill-down at `/integrations/docker/containers/$name/image-update` shows full digest detail. A new shared `digest.ts` helper truncates SHA digests consistently across drill-down AND main grid. The collector is READ-ONLY — no `docker pull` (deferred to STAGE-003-010).

### Behavior changes

- **NEW**: `homelab_image_update_available{name,image,current_digest,latest_digest}` gauge — 1 when registry digest != local digest, 0 otherwise.
- **NEW**: `homelab_image_update_last_checked_seconds{name}` gauge — epoch of last successful check (drives staleness alerts).
- **NEW**: `homelab_registry_rate_limit_remaining{registry}` gauge — Docker Hub `RateLimit-Remaining` parsed from manifest response headers.
- **NEW**: `homelab_image_update_check_skipped{reason}` counter — increments per skip; `reason="rate_limit"` for hard-cap skips.
- **NEW**: `DockerImageUpdateAvailable` vmalert rule (`info` severity, `for: 30m` flap suppression).
- **NEW**: `GET /api/integrations/docker/image-updates/summary` — aggregate summary for the grid + rate-limit banner.
- **NEW**: `GET /api/integrations/docker/containers/{name}/image-update` — per-container detail for the drill-down.
- **NEW**: Image-update background events loop — second filtered subscriber on the Docker socket events stream (`filters={"type":["image"]}`); on `image pull` events triggers an out-of-band collector run.
- **NEW**: Per-container image-update state persisted in new `image_update_state` table (migration 0026), keyed by `container_name`, mirroring `docker_override_ownership` precedent.

### Architecture decisions (locked in STAGE-003-008.md)

- D-REGISTRY-READ-ONLY — image-update detection is PURELY READ; no `docker pull`
- D-DIGEST-COMPARISON — by cryptographic `RepoDigest`, not tag string
- D-PUBLIC-IMAGES-V1 — anonymous bearer-token flow only; private registry credentials deferred
- D-PER-IMAGE-FAILURE-ISOLATION — one image's failure does NOT abort the tick
- D-RATE-LIMIT-AWARE — `RateLimit-Limit` / `RateLimit-Remaining` headers parsed, surfaced as gauge metric
- D-INFO-SEVERITY — image-update alerts are `info` (dashboard-only / Karma, not paging)
- D-STATE-SIDECAR-TABLE — new `image_update_state` table (migration 0026)
- D-LATEST-ALWAYS-CHECK — `:latest` / `:main` / `:edge` / rolling tags ARE checked
- D-RATE-LIMIT-HARD-CAP-PLUS-BANNER — skip remaining checks + UI banner when `RateLimit-Remaining < 10`
- D-BADGE-TEXT-PLUS-BLUE-PILL — blue `<Link>` pill rendered on BOTH desktop row AND mobile card
- D-EVENTS-FILTERS-KWARG — `DockerSocketClient.events(filters=...)` kwarg drives the new image-events subscriber
- D-LABEL-KEY-NAME — metrics use `name=` (matching STAGE-003-004 convention), NOT `container=`
- D-SUMMARY-SIBLING-ENDPOINT — `/image-updates/summary` is a sibling of `/probes/summary` (not an extension)
- D-PER-CONTAINER-ENDPOINT — `/containers/{name}/image-update` for drill-down
- D-IMAGE-REF-PARSER-INHOUSE — in-house parser; no new dependency
- D-COLLECTOR-CADENCE-6H — 6h tick default via `HOMELAB_MONITOR_IMAGE_UPDATE_INTERVAL_SECONDS`
