2026-06-18 5cd2484 EPIC-017/STAGE-017-007: ssh.yaml framework-health alert rules

Added `deploy/vmalert/metrics/ssh.yaml` (group `ssh_health`) — the framework-health alert rules over the SSH-probe metrics. Three per-target rules, auto-loaded by vmalert via the `/etc/vmalert/metrics/*.yaml` glob:
- `SshTargetUnreachable` (warning) — `homelab_ssh_up == 0` for 5m; routes to the `monitoring-health` channel.
- `SshHostKeyMismatch` (critical) — `homelab_ssh_host_key_mismatch == 1`, fires immediately (`for: 0s`); default routing (security finding, not monitoring degradation).
- `SshProbeStale` (warning) — `homelab_ssh_last_success_age_seconds > 900` for 5m; routes to `monitoring-health`. Silent when the age series is absent (post-monitor-restart / never-succeeded) by design.

Each rule's `annotations.description` carries the card-required remediation (re-install authorized_keys incl. the appliance firmware-wipe note via `hm ssh-probe install-instructions <target>`; out-of-band host-key verification + `hm ssh-probe capture-hostkey <target>` for a mismatch; the in-memory-state-resets-on-restart note for stale). NO `absent()` anywhere — on a multi-target metric `absent()` collapses labels and breaks per-target alerting (it's valid in `vl_health.yaml` only because VictoriaLogs is single-target).

Validation: vmalert loads all 3 rules with `health=ok` (no parse error); a synthetic-injection live-replay on the prod rig proved `SshHostKeyMismatch` fires end-to-end to Alertmanager with the `{{ $labels.target }}`-rendered summary, `SshTargetUnreachable` goes pending on `homelab_ssh_up==0`, and `SshProbeStale` correctly stays silent with no age series. `make verify` green (no Python coverage impact — YAML only).
