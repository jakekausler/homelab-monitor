## [STAGE-005-042] HA cadence detail API + Status-tab idle cadence widget

**Summary**

The in-app HA cadence surface (split off from STAGE-005-038): a new `GET /api/integrations/home-assistant/cadence` endpoint + a Status-tab widget listing idle (>24h or never-triggered) automations and idle scripts, mirroring the `/entities` endpoint + `HaEntitiesDrill` pattern. Final stage of EPIC-005 (Home Assistant integration) ‚Äî completes the epic at 42/42.

**Backend** ‚Äî `GET /cadence` returns two named collections (`automations`, `scripts`) from VictoriaMetrics per-series gauges. Automation universe = `homelab_ha_automation_enabled` (LEFT JOIN `homelab_ha_automation_last_triggered_seconds`; absent ‚Üí `last_triggered_age_seconds: null` = never-triggered). Include iff `enabled AND (age is None OR age > 86400)`. Scripts: universe = `homelab_ha_script_last_triggered_seconds`, include iff `age > 86400`. Sorted most-idle-first (null age first, then age DESC), top-N capped, `*_total`/`*_returned` per collection, `filtered_to="idle_24h"`. friendly_name enriched via one live HA `get_states()` (HA-down ‚Üí null, never 502; VM-down ‚Üí 502).

**Refinement scope change (user)** ‚Äî disabled automations EXCLUDED entirely (the card's "idle + disabled" became "idle only"): the predicate dropped the `not enabled` inclusion clause, `filtered_to` ‚Üí `"idle_24h"`, the frontend "disabled" badge removed, the card retitled "Idle automations". ## [STAGE-005A-001] Host machine threshold + anomaly rules

**Summary**

Backfilled 11 host-machine vmalert alert rules (the first EPIC-005A stage) over metrics EPIC-001's host collector already emits but had no alerts. Rules in new `deploy/vmalert/metrics/host_anomalies.yaml`: critical CPU/mem tiers (>95%, above the existing >90% warnings), load-vs-core-count (`scalar(2x/4x * count(cpu))`), a swap+RAM AND-gate (`HostMemoryPressure`), zombie-process count, reboot detection (uptime<600s), and rolling-baseline anomaly rules for CPU, memory, disk-IO, and net throughput (the house `clamp_min(K * avg_over_time(...), floor)` idiom). All carry `source_tool: vmalert-metrics` + `target_kind: host`; severity vocabulary info|warning|error|critical.

Two pre-existing bugs were root-caused and fixed correctly in-scope (not deferred):
- **Memory-division label mismatch:** `homelab_host_memory_bytes{type="used"} / {type="total"}` matches nothing in MetricsQL (operands differ on `type`), so the division returned empty and the rule could never fire. Fixed with `/ ignoring(type)` on the 3 new memory rules AND on the pre-existing `HostHighMemory` in `host.yaml` ‚Äî the host's primary memory alert had never fired in production.
- **Cumulative-counter inflation:** the host collector passed psutil's absolute cumulative counter to `write_counter()` (which `.inc()`s) every tick, inflating `homelab_host_disk_io_bytes_total` / `homelab_host_net_bytes_total` to ~31 PB (rate ~2 TB/s). Fixed by adding a new `write_counter_absolute` MetricsWriter method (Gauge-backed SET semantics) across the writer subsystem and switching the 4 host disk/net emit sites to it. Post-rebuild the metric reads ~22 TB (kernel truth) with sane MB/s rates.

**Modified files**

- `deploy/vmalert/metrics/host_anomalies.yaml` (NEW) ‚Äî 11 alert rules.
- `deploy/vmalert/metrics/__tests__/host_anomalies.tests.yaml` (NEW) ‚Äî forward-compatible shortened-window test mirror (fire/no-fire per rule; NOT CI-run).
- `deploy/vmalert/metrics/host.yaml` ‚Äî fixed pre-existing `HostHighMemory` label-mismatch (`/ ignoring(type)`).
- `apps/monitor/homelab_monitor/kernel/plugins/io.py`, `.../kernel/metrics/prometheus_writer.py`, `.../kernel/metrics/multiplex.py`, `.../kernel/plugins/process_context.py` ‚Äî added `write_counter_absolute` (Gauge-backed SET) across the MetricsWriter protocol + all implementors.

- `apps/monitor/tests/test_metrics_prometheus_writer.py`, `test_plugins_io.py`, `test_process_context.py`, `test_metrics_multiplex.py`, `test_host_collector.py`, `test_heartbeat_collector.py` ‚Äî new/updated tests for set-semantics (proving SET not INC) + the 2 type-checked test doubles.

**Design decisions**

- D-CPU-TIERS: critical tiers in the new file; `host.yaml` warning rules frozen. D-LOAD-MULTIPLIER: 15m period, 2x/4x cores, `scalar(count(...))` for engine portability. D-SWAP: reformulated to a swap+RAM AND-gate (the box idles at 93% swap / 20% RAM, so static swap-fullness would false-fire). D-PROCESS: zombie-only (`>20`). D-REBOOT: absolute `uptime<600`. D-ANOMALY-FORM: `clamp_min` idiom with floors (CPU 3x/40, mem 2x/0.6, IO/net 3x/50 MB/s). D-IO-NET: authored in-scope (tuned in 3b), NOT deferred. D-TEST-EXECUTION: forward-compatible mirror + manual promtool/live-replay validation (vmalert has no unit-test mode).
