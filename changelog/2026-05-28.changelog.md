## STAGE-004-001 — Vector multi-line log stitching

Lands Vector's `docker_logs` multiline codec that coalesces multi-line log events (Python tracebacks, Java/Go/Ruby/Node.js/.NET/C++/Rust/PHP stack traces, kernel oops, sshd PAM stanzas) into a single VictoriaLogs record instead of one record per physical line. First stage in EPIC-004 so all downstream stages (LogViewer, Drain clustering, anomaly rules) receive coherent events from the start.

### Added

- **Vector `docker_logs` multiline codec** (`deploy/vector/vector.toml.template` [`sources.docker_logs.multiline` block]):
  - `mode = "continue_through"` — accumulates lines matching a continuation pattern
  - `start_pattern` enumerates event-start markers (RFC3164/5424 timestamps, exception-class lines, `Traceback`/`panic`/`Fatal`/`Stack trace` keywords, etc.) for 9 languages
  - `condition_pattern` enumerates continuation markers (indented lines `\s+`, `Caused by:`, stack frames, hex addresses, trailing exception terminators like `at ...`)
  - Verified live against Vector 0.41.1 for Python, Java, Go, Ruby, Node.js, .NET, C++, Rust, PHP
  - Vector's Rust regex crate has no lookaround support; patterns use positive enumeration, enforced by a `_assert_no_lookarounds` test
- **Test fixture Vector config** (`deploy/compose/test-fixtures/vector.toml`) — carries an identical multiline block; a drift-guard unit test (`test_vector_template.py::test_fixture_vector_multiline_matches_production`) enforces byte-identity with the production template to prevent config skew
- **Integration test suite** (`apps/monitor/tests/integration/test_vector_multiline.py`) — 11 tests covering:
  - Python traceback (multi-line with local variables)
  - Java stack trace (wrapped exception chain with `Caused by:`)
  - Go panic (with goroutine details + memory addresses)
  - Ruby exception (with backtrace + system context)
  - Node.js error (with stack frames)
  - .NET exception (with line numbers)
  - C++ crash (with demangled symbols)
  - Rust panic (with location + backtrace)
  - PHP Fatal error (with context lines)
  - Two negative controls (single-line logs, timestamps that look like exceptions)
  - Each test uses tail-fragment assertions proving real stitching (not just record count matching)
- **Journald source limitation** — Vector's `journald` source type does NOT support a multiline sub-table; journald delivers discrete entries already and multiline is inapplicable

### Test Infrastructure (folded into Finalize)

- **Fixed `make test-fast` marker bug**: CLI `-m "not slow"` was overriding `pyproject.toml`'s `-m 'not integration'`, leaking integration tests into the fast suite (expected ~14s, actual ~14min). Now `-m "not slow and not integration"` enforces both filters.
- **Session-cached rig health-probe gate** (`apps/monitor/tests/integration/helpers/rig_health.py` + `require_rig_components` decorator):
  - Integration tests skip fast (~14s for full set, was ~14min per test when rig was absent) when their rig components are unreachable, instead of blocking 30s on `Rig.boot()` per test
  - Partial-rig aware: only probes named components; fail-loud on unknown names
  - 2s per-component probe; preserves real 30s token-boot budget when rig is present
  - `noisy-logger` service got a healthcheck + `service_healthy` dependency to close a probe race
- **Fixed pre-existing docker-compose.test.yml issue**: `vmalert-metrics` was missing `-remoteWrite.url` flag (broke rig boot since STAGE-003 added recording rules). Corrected in test compose file.

## STAGE-004-002 — Backend LogLine shape convergence

Converges three divergent log endpoint response shapes into a single `LogLine` model. All endpoints now return `.lines: list[LogLine]` with consistent fields (timestamp, message, stream, severity, host, service, fields dict). Precedes the shared `<LogViewer>` component (STAGE-004-003), giving it one uniform contract to render.

### Added

- **`apps/monitor/homelab_monitor/kernel/logs/models.py`** — new shared LogLine model:
  - `LogLine = {timestamp: str, message: str, stream: str, severity: str, host: str | None, service: str | None, fields: dict[str, Any]}` with Pydantic validation
  - `from_victorialogs_line(VlLogLine) -> LogLine` mapper that converts VictoriaLogs records to the converged shape
  - `_normalize_severity(raw: str | int) -> str` helper that canonicalizes severity to lowercase (debug/info/notice/warn/error/critical/alert/emergency), accepts syslog numerics (0-7) and aliases (warning→warn, err→error, crit→critical, panic→emergency, emerg→emergency), preserves raw value in `fields.severity_raw`
  - `host` and `service` extracted from VL fields dict with fallbacks: host defaults to `host` field or `_HOSTNAME` label; service defaults to `service` field or `SYSLOG_IDENTIFIER` label
  - Full VL fields dict retained in `fields` for downstream extensibility (anomaly rules, drill-down filters)

### Changed

- **`/api/integrations/docker/containers/{name}/logs`** — response now returns `.lines: list[LogLine]` instead of `.lines: list[ContainerLogLine]`; severity/host/service promoted from scattered nested fields
- **`/api/crons/{fp}/runs/{run_id}/log`** — outer response field renamed `.entries` → `.lines`; response now returns `list[LogLine]` instead of `.entries: list[RunLogLine]`; severity/host/service normalized
- **`/api/logs/query`** — renamed `.entries` → `.lines`; response now returns `list[LogLine]` instead of `.entries: list[LogsQueryEntry]`; severity normalized; all VL fields merged into single `fields` dict
- **Deleted old inner types** — `ContainerLogLine`, `RunLogLine`, `LogsQueryEntry` (replaced by single `LogLine` contract)
- **Frontend response handling**:
  - `apps/ui/src/components/DockerContainerLogsViewerBody.tsx` — updated to use `entry.message` instead of `entry.line`
  - `apps/ui/src/components/CronRunLogViewer.tsx` — updated to handle `.lines` instead of `.entries`
  - Test mocks updated to match new shape
  - OpenAPI schema regenerated; frontend types regenerated (`pnpm openapi`)
