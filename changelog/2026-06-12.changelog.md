## [STAGE-005-011] Repairs collector (websocket)

**Summary:** Added `HaRepairsCollector` — second websocket-consuming collector. Surfaces HA's Repairs registry via per-tick `send_command("repairs/list_issues")` snapshot RPC (no subscription; A1 pattern matches 010's config-entry bridge). Emits per-repair-issue state as `homelab_ha_repair_issue{domain,issue_id,severity}` = 1.0 for each active, non-ignored repair issue. Severity enum (critical|error|warning) is rule-friendly; free-text fields (translation_key, translation_placeholders, is_fixable, descriptions, learn_more_url, breaks_in_ha_version) kept PANEL-ONLY per metric-hygiene convention. Global 500 cardinality cap sufficient (repairs are tens at most). Alert-vs-panel discrimination deferred to 015 (panel-only this stage).

**Files — Backend (new):**
- `apps/monitor/homelab_monitor/plugins/collectors/integrations/homeassistant/ha_repairs.py` — `HaRepairsCollector` (name `ha_repairs`, interval 60s, timeout 15s, concurrency_group `homeassistant`, injected `_ws` from lifespan). Per-tick `send_command("repairs/list_issues")` snapshot RPC (A1 pattern). Defensive parse guards: list shape from bare list vs `{"issues":[...]}` dict (real HA returns dict wrapping array); empty-list / missing-issues-key handling. Emits 1 family: `homelab_ha_repair_issue{domain,issue_id,severity}` (1.0 per active + non-ignored issue). Filters: `active is False` (HA pre-filters; guard identity) and `ignored is True` (skipped). CappedEmitter, global default cap 500. `ctx._ws is None` → failed CollectorResult(ok=False); `HaError` result → same. Labels from `domain` (string), `issue_id` (string), and `severity` (enum string, one of critical|error|warning).
- `apps/monitor/tests/test_ha_repairs_collector.py` — 36 tests covering list/dict payload shapes (bare-list, dict-wrapped, empty list, missing issues key), active-false skip, ignored-true skip, severity enum cardinality, missing domain/issue_id/severity (defensive defaults), HaError result, `_ws` None, cardinality cap, bundle test.

**Files — Backend (modified):**
- `apps/monitor/homelab_monitor/plugins/collectors/integrations/homeassistant/__init__.py` — registered `HaRepairsCollector` in `_HA_COLLECTORS` (now 7th).
- `apps/monitor/homelab_monitor/kernel/api/lifespan.py` — added `c._ws = app.state.ha_ws_client` injection arm for `HaRepairsCollector`.
- `apps/monitor/tests/test_ha_integration_bundle.py` — added `test_register_all_registers_ha_repairs`.

**Design decisions:**
- **D-REPAIR-METRIC-SHAPE (Option A):** Single per-issue 1.0 gauge (vs multi-state enum or cardinality-per-severity). Matches 010/006/008 metric style; simpler for alerting (presence of a label combo = issue exists).
- **D-REPAIR-ACTIVE-IGNORED (filtering):** Only emit active + non-ignored issues. HA pre-filters active-only on the API side (`ignored` is computed `dismissed_version is not None`); the guard is an identity filter, defensive. Reduces cardinality noise (dismissed repairs don't alert).
- **D-REPAIR-FREEBIES-PANEL-ONLY:** Translation keys, placeholders, is_fixable, full descriptions, learn_more_url, breaks_in_ha_version all deferred to the HA panel context. Per 008/010 discipline: metrics only for alertable state (repair exists yes/no).
- **D-WS-SNAPSHOT-BRIDGE (A1):** Reuses 010's `send_command("repairs/list_issues")` recipe: one-shot RPC, no subscription. Architecture matches 010; no new complexity.

**Refinement 3a+3b (real HA):** Collector registered/injected/live in prod rig; drop gauge present in VictoriaMetrics. Real `repairs/list_issues` confirmed to return dict `{"issues":[...]}` (the dict-wrapped branch is the real production path). Real HA had 0 active repair issues during Refinement; per-issue field contract recovered + confirmed against HA core source (`homeassistant/repairs/websocket_api.py` + `homeassistant/helpers/issue_registry.py`): `domain` (string) / `issue_id` (string) / `severity` (enum: critical|error|warning) are correct literal keys. `ignored` present (computed from `dismissed_version is not None`). `active` NOT serialized in the API response (HA pre-filters active-only server-side), so the `active is False` identity guard correctly emits (never triggers, by design). Real cardinality 0 → global 500 cap ample. Dict-wrapping defensive parse confirmed necessary by contract inspection (real path is dict-wrapped array).

## [STAGE-005-012] Persistent notifications collector (websocket)

**Summary:** Added `HaPersistentNotificationCollector` — third websocket-consuming collector. Surfaces HA's persistent notifications (the notification "bell") via per-tick `persistent_notification/get` snapshot RPC (no subscription; A1 pattern matches 010/011). Emits per-notification presence as `homelab_ha_persistent_notification{notification_id}` = 1.0 for each active notification. **PRIVACY constraint (non-negotiable):** notification title/message NEVER enter metric labels, events, or logs; only stable low-churn `notification_id` slug becomes a label. Global 500 cardinality cap sufficient (active notifications typically low-digit). Alert deferred to 015 (panel-only this stage).

**Files — Backend (new):**
- `apps/monitor/homelab_monitor/plugins/collectors/integrations/homeassistant/ha_persistent_notification.py` — `HaPersistentNotificationCollector` (name `ha_persistent_notification`, interval 60s, timeout 15s, concurrency_group `homeassistant`, injected `_ws` from lifespan). Per-tick `persistent_notification/get` snapshot RPC (A1 pattern). Defensive parse guards: list shape from bare list vs `{"notifications":[...]}` dict; empty-list handling. Emits 1 family: `homelab_ha_persistent_notification{notification_id}` (1.0 per active notification). CappedEmitter, global default cap 500. `ctx._ws is None` → failed CollectorResult(ok=False); `HaError` result → same. Labels from `notification_id` only (stable slug, never includes title/message). **012 stores nothing:** no in-memory body dict, no SQLite cache, no disk persistence. Body-sourcing for future HA panel deferred to STAGE-005-021 (D-PANEL-NOTIF-EXPOSURE). **Privacy mechanics:** run() returns events=[] (no event carries body text); no logger imports/calls in module (no log leaks); `_notification_labels` helper extracts only `notification_id` from each notification dict.
- `apps/monitor/tests/test_ha_persistent_notification_collector.py` — 30 tests covering list/dict payload shapes (bare-list, dict-wrapped, empty list, missing notifications key), notification_id extraction, privacy assertions (3 sentinel-based tests: label-content NEVER includes title/message, events=[], no logs of parsed notifications), HaError result, `_ws` None, cardinality cap, bundle test, 100% branch coverage.

**Files — Backend (modified):**
- `apps/monitor/homelab_monitor/plugins/collectors/integrations/homeassistant/__init__.py` — registered `HaPersistentNotificationCollector` in `_HA_COLLECTORS` (now 8th).
- `apps/monitor/homelab_monitor/kernel/api/lifespan.py` — added `c._ws = app.state.ha_ws_client` injection arm for `HaPersistentNotificationCollector`.
- `apps/monitor/tests/test_ha_integration_bundle.py` — added `test_register_all_registers_ha_persistent_notification`.

**Design decisions:**
- **D-NOTIF-METRIC-SHAPE (Option A):** Single per-notification 1.0 gauge keyed by `notification_id` (count derived via `count()`). Matches 010/011/008/006 metric style; simplest for alerting (presence = notification exists).
- **D-NOTIF-PRIVACY (non-negotiable):** Only `notification_id` (stable low-churn slug) becomes a label. Title/message are sensitive context-dependent data (e.g., "Device lost power", "SSL cert expiring in 3 days"); NEVER leak into metrics, events, or logs. Per metric-hygiene discipline: alerts are binary state (notification exists yes/no), not content-based.
- **D-NOTIF-NO-PERSISTENCE:** Rejected docker-panel's historical body caching (privacy regression vs "never disk-persisted" promise). Rejected in-memory body dict (speculative pre-work, defers to 021's needs). Only the gauge; body is panel-specific, fetched on-demand at UI render time (STAGE-005-021 responsibility).
- **D-WS-SNAPSHOT-BRIDGE (A1):** Reuses 010/011's one-shot RPC recipe: `persistent_notification/get`, no subscription. Architecture matches 010/011; no new complexity.

**Refinement 3a+3b (real HA):** Collector registered/injected/live in prod rig; drop gauge present in VictoriaMetrics. Real `persistent_notification/get` confirmed to return **bare list** (NOT a dict — defensive dict-branch handles but is fallback). Real per-notification field contract recovered + confirmed: `{notification_id, title, message, created_at}`. `notification_id` a stable slug (e.g., http-login, invalid_config — low-digit cardinality in practice). **PRIVACY validated end-to-end on REAL HA:** Created sentinel notification (title HMTEST-TITLE-SENTINEL, body HMTEST-BODY-SENTINEL-XYZ), drove collector against live WebSocket, verified all 4 PRIVACY assertions: (1) gauge emitted with ONLY notification_id label, (2) sentinel title/message NEVER in any metric label, (3) events=[], (4) ZERO sentinel text occurrences in monitor container logs.
