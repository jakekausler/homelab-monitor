## [STAGE-005-020] Push-back-to-HA event firer

**Summary:** The outbound-event half of bidirectional HA integration (scenario #24), completing the bidirectional picture alongside 017 (push notifications OUT) and 019 (events IN). On an opted-in alert firing/resolving, the monitor POSTs an event to Home Assistant's event bus so the operator's own HA automations can react (run a scene, announce on speakers). Implemented as a new dispatch Channel `HAEventChannel` (kind="ha_event") mirroring 017's HAPushChannel — it reuses the dispatcher's failure-isolation + the 018 `accepts()` gate; EPIC-012 can later select it via routing_rules.

**Files — Backend (new):**
- `apps/monitor/homelab_monitor/kernel/dispatch/channels/ha_event.py` — `HAEventChannel` (kind="ha_event"). Constructor DI: shared `HomeAssistantRestClient` + `event_type: str` + `public_url_provider: Callable[[], str | None] = get_public_url`. `accepts(event)` has TWO gates: global enable (`if not self._event_type: return False` — empty event_type = channel OFF) AND per-alert opt-in (`event.labels.get("push_to_ha") == "true"`). **Design decisions:**
- **D-PUSHBACK-SHAPE → new `HAEventChannel` in the dispatcher.** The only trigger is an alert firing/resolving — exactly what `AlertDispatcher.dispatch(event)` already fans out — so a channel is the minimal correct shape and the clean mirror of 017. Rejected: firing outside the dispatcher (would duplicate the fan-out + re-implement failure isolation).
- **D-PUSHBACK-MECHANISM → `fire_event(...) -> None | HaError`, success = `None`.** Nothing consumes the fire-event response body; `isinstance(result, HaError)` is all the channel needs, and `None`-on-success is the tightest path for 100% coverage. Uses the shared `_status_error` 2xx-band helper (consistent with every other client method). Rejected: a new `HaEventResult` dataclass (an unused `message` field + coverage burden).
- **D-PUSHBACK-OPTIN → two gates in `accepts()` (global `event_type != ""` AND per-alert label `push_to_ha == "true"`).** Default `event_type=""` (OFF, open-source-safe); env `HOMELAB_MONITOR_HA_EVENT_TYPE`; recommended `homelab_monitor_alert`. The label is the per-alert opt-in (operator adds it in the vmalert rule / Alertmanager config) — forward-compatible with EPIC-012's `tag_match`, exactly the "tag / label" mechanism the 018 hand-off pointed at. Rejected: severity-based gate (card says per-alert opt-in, not severity); a config allow-list (routing-table territory, EPIC-012 owns it).
- **D-PUSHBACK-PAYLOAD → flat `dict[str, str]` (the fire-event body IS the event data, no envelope).** `url` omitted when `get_public_url()` is None (type-safety — never None in a `dict[str, str]`).
- **D-PUSHBACK-RESOLVED → fire on BOTH; `status` field is the branch.** A resolved event closes the loop (HA turns the alarm scene back off). Both pass the same `accepts()` gate (resolved carries the same labels incl. `push_to_ha`).
- **D-LOOP-SAFETY → no loop.** Inverse of the loop 019 avoided: 020 fires on alert events; 019 (inbound webhook) sinks HA events to `audit_log` ONLY and never creates alerts. So 020 → HA event → (an HA automation may POST back to 019) → audit_log → dead end (no alert created → nothing re-enters `dispatch()` → 020 cannot re-fire). A pathological indirect loop requires three deliberate operator config choices (operator's misconfiguration, not a 020 defect).
