## [STAGE-004-025] drain3 wrapper + SQLite persistence (Drain log-clustering library)

## [STAGE-004-026] Periodic batch consumer (DrainConsumer: VL → DrainEngine cycle)

**Summary:** Introduced DrainConsumer, a continuous in-process periodic service that drives the STAGE-004-025 Drain library: every ~5 min (HOMELAB_MONITOR_DRAIN_INTERVAL_S) it queries VictoriaLogs for log lines newer than a global watermark, feeds them through DrainEngine.add_line, persists clustering state via engine.snapshot(), and advances the watermark. The watermark is a single value in the app_settings kv table (key `drain.cycle_watermark_ms`); per-model cursors are still maintained by the engine. Cold-start seeds the watermark from the max per-model cursor (resume after restart) or `now - ingest_lag_grace` (does NOT replay history). Partial cycles cap at batch_max_lines=50000 and resume from the newest processed line; an ingest-lag grace margin (30s default) avoids skipping late-arriving lines. The consumer is env-gated (HOMELAB_MONITOR_DRAIN_ENABLED) and spawned as a lifespan task. NO metrics emission and NO refresh endpoint yet (STAGE-004-027).

**Key changes:**
- `kernel/logs/drain_consumer.py` (NEW): DrainConsumer (start_task/stop_task/run_forever/run_once), DrainCycleResult frozen dataclass, WATERMARK_KEY, watermark read/seed/persist. run_once catches VictoriaLogsClientError → snapshot + cycle_status='failed' + watermark NOT advanced (idempotent retry); run_forever is a cancellable while-loop mirroring OverrideLoader.
- Inclusive-boundary correctness: VL `_time:[start,end]` is inclusive both ends, so the resume query start is built at NANOSECOND precision `_ns_to_iso(watermark_ms*1e6 + 1)` (reusing kernel/logs/pagination) to exclude the already-processed boundary millisecond — prevents line_count/cluster.size double-counting and a same-millisecond-burst infinite-reprocessing wedge, with no data loss.
- `kernel/config.py`: DrainConfig (interval_seconds=300, batch_max_lines=50000, ingest_lag_grace_seconds=30, enabled=True) + load_drain_config() with clamps.
- `kernel/logs/drain_persistence.py`: get_max_cursor() (MAX(last_processed_ts) across drain_models) for cold-start resume.
- `kernel/api/lifespan.py`: env-gated #7g block constructing DrainEngine(SqlitePersistence(repo)) + VL client + the consumer; start_task on startup, stop_task first in shutdown.

## [STAGE-004-027] Drain signature metrics emission + manual refresh API

**Summary:** Extended the STAGE-026 DrainConsumer with (A) Drain signature metrics emission to VictoriaMetrics after each cycle, and (B) a manual refresh API. After each run_once cycle (inline, after snapshot), the consumer emits 6 metrics + a cardinality-warn gauge: cycle counters (lines/new-templates), a cycle-duration summary, and per-signature gauges (count per (service_key, template_hash, severity), cumulative total, first-seen-ns) sourced from the per-cycle SignatureEvent stream — severity comes from the per-line LogLine.severity ('unknown' if None); gauges emit only for templates touched this cycle (no all-models iterator). Metrics are written directly (single source + unique labels → no gauge clobber). The manual refresh API exposes POST /api/logs/signatures/refresh (202 + cycle_id, runs a cycle in a fire-and-forget background task; 409 if a cycle is already running; 503 if drain disabled) + GET /api/logs/signatures/refresh/{cycle_id} (200 {status: running|done|failed, result?, error?}; 404 unknown/expired). A re-entrancy guard (asyncio.Lock in run_once + CycleInProgressError, atomic check-then-acquire) ensures manual + periodic cycles never overlap. Cycle results live in an in-process CycleStatusStore (1h TTL, prune-on-access). NO UI (028), NO anomaly rules (035).
