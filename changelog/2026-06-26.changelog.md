Added the first Alertmanager `inhibit_rules:` block plus the full vmalert rule chain it operates on, so a Synology /rackstation NFS mount going down suppresses the now-redundant container-down alerts for containers that depend on those mounts.

- NEW `deploy/vmalert/metrics/synology.yaml` — three rules: `SynologyMountDown` (source: `homelab_synology_mount_up == 0`, `for: 2m`, labels severity:critical / source_tool:vmalert-metrics / category:storage / integration:synology, with the per-mount `mount` label flowing through from the metric); `PlexContainerDown` and `PodcastFeedContainerDown` (targets, modeled on the existing `PiholeContainerDown`: `absent(running) or state=~"exited|dead|restarting"`, `name="plex"`/`"podcast-feed"`, labels including `target_kind:docker_container` / `integration:docker` / `depends_on_rackstation:"true"`). plex and podcast-feed are the only two containers that mount /rackstation.
- MODIFIED `deploy/alertmanager/alertmanager.yml.template` — the first `inhibit_rules:` block: source `alertname="SynologyMountDown"`, target `depends_on_rackstation="true"`, no `equal:` (per-mount equality is structurally impossible since one container mounts many shares, and a NAS-layer outage is realistically rackstation-wide).
- NEW `deploy/vmalert/metrics/__tests__/synology.tests.yaml` — promtool golden tests for all three rules (fire / no-fire, exact rendered annotations).
- MODIFIED `Makefile` — new `make alertmanager-check` target running `amtool check-config` on the template (amtool tolerates the unsubstituted `${ALERTMANAGER_INGEST_TOKEN}` placeholder as an opaque bearer string, so it lints the raw template directly).
