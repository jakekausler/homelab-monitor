2026-06-26 5e50970 EPIC-008/STAGE-008-018: NFS mount-health collector

Added `SynologyMountHealthCollector` (`synology_mount_health`, 60s) — a host-filesystem collector that probes the monitor host's NFS mounts (the `/rackstation/*` Synology shares) and emits the root-cause "is this mount alive" signal that the STAGE-008-019 Docker-alert suppression will consume. The probe is NON-HANGING: it reads `/proc/self/mountinfo` (never blocks) to find present mounts, then runs `os.statvfs` per mount in a dedicated bounded `ThreadPoolExecutor(max_workers=4)` wrapped in a 5s `asyncio.wait_for`, so a wedged hard-NFS mount is reported down within the timeout instead of wedging the collector (a wedged mount leaks at most 4 threads, never starving the default pool). Per configured mount it emits `homelab_synology_mount_up{mount}` (seeded 0, overwritten to 1 only on a responsive probe — a missing or hung mount stays 0 so a vmalert/inhibit rule can fire), `homelab_synology_mount_probe_seconds{mount}`, and `homelab_synology_mount_free_bytes{mount}` / `homelab_synology_mount_total_bytes{mount}` on success.

The configured mount list is a new `synology_mounts` field, default EMPTY (open-source-safe: the collector no-ops with no config); the host-override declares the real `/rackstation/*` paths. Exposing the mounts to the container requires a new compose bind `${HM_RACKSTATION_SRC:-/dev/null}:/rackstation:ro,rslave` on the monitor service — `:ro` upholds the read-only privilege invariant and `rslave` propagation is required so the NFS submounts are visible inside the container (a plain bind shows only the empty parent dir); the `/dev/null` public default is inert (verified). Validated live against the real 9 `/rackstation/*` NFS mounts (responsive mounts report ~52 TB volumes; a bogus path reports down), and the prod monitor was recreated with the bind (healthy, /rackstation visible via rslave). 100% branch coverage on the new module.

Per-camera all-time `homelab_synology_ss_events_total{camera}` (from the `evt_cam` composite `<id>-<name>` keys); seeded always-emit system rollups `ss_events_today` (system-wide, from the NAS-LOCAL today date key), `ss_events_total_all`, `ss_recordings_total` (cheap top-level total), `ss_recordings_bytes_total`; per-camera `ss_recordings_count{camera}` + `ss_recordings_bytes{camera}` aggregated over the full recording list. Feeds the STAGE-008-022 "camera connected but not recording / went silent" integrity rule.

The `date` map is keyed in NAS-local time, so `events_today` uses a LOCAL (America/New_York) date key, not UTC — a UTC key would silently mis-read during the nightly offset window; covered by a unit test (frozen now where ET date ≠ UTC date) and validated live.

Scope-outs (rationale, not silent): am/pm event split (YAGNI), per-camera-per-day events (does not exist in the DSM API), the `limit` param on the recording list (the full list is needed for per-camera bytes anyway; trivial at homelab scale). No metric-name collision with STAGE-008-015 (`ss_camera_*`/`ss_info_*`) or STAGE-008-017 (`ss_license_*`).

Refinement validated the real collector against the live Surveillance Station: 22 gauges / 9 families for the full 2650-record payload in 0.74s, with the per-camera recording counts AND bytes summing exactly to their system totals (804+1098+748=2650; 1023192+1023920+1025248=3072360).
