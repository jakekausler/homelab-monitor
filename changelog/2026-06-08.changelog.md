## [STAGE-004-038] Anomaly Type D: Log signature silence detection + allowlist

**Summary:** Log signature silence detection — when a recurring log signature stops appearing for 15-60 minutes (configurable), a backend collector emits an alert signaling a service/job may have stopped. An operator-managed allowlist suppresses the silence alert for expected silences (scheduled maintenance, cron jobs with known quiet windows, etc.). Suppression uses schedule kinds (always / window / cron) and is checked per-signature at collection time, ensuring suppressed signatures never fire alerts. Frontend Silence Allowlist tab provides full CRUD with an optional signature picker and responsive UI refinements.

**Key changes:**

**Backend:**
- New `SilenceDetectionCollector` (BaseCollector) — DB-reader that ticks every 60s, reads `log_signatures` + the new `log_signature_silence_allowlist` table, emits gauge `homelab_log_signature_silent{service_key, template_hash}=1` for signatures silent within `[silent_min_seconds=900, silent_max_seconds=3600]` (15-60 min, configurable) that are not suppressed and not allowlist-covered. Emits mandatory `homelab_collector_run_silence_detection` self-metric.
- New `is_silence_allowed(schedule_kind, schedule_value, expires_at)` evaluator (kernel/logs/silence_schedule.py) — schedule kinds always (silence always allowed) / window (ISO 8601 range, e.g. `2025-12-25/2025-12-26`) / cron (silence allowed EXCEPT within `cron_grace_seconds=900` after previous cron fire — i.e. alert can only fire just after a scheduled job's run window ends).
- New `SilenceAllowlistRepository` (kernel/logs/silence_allowlist_repo.py) + table `log_signature_silence_allowlist` (migration 0040) — fields service_key, template_hash, schedule_kind (enum: always/window/cron), schedule_value (JSON string), window_start, window_end (ISO strings for window kind), cron_expression, cron_grace_seconds, expires_at, created_by, created_at. UNIQUE (service_key, template_hash) enforces one allowlist entry per signature.
- New REST API (routers/logs.py): GET/POST (CSRF)/DELETE at `/api/logs/signatures/silence-allowlist`. SilenceAllowlistCreateRequest validates schedule_kind/value cross-consistency (e.g. window kind requires window_start+window_end, cron kind requires cron_expression), rejects unsupported `@reboot` cron, validates ISO window ranges, validates expires_at is in future. List response includes is_expired computed flag.
- New `SilenceDetectionConfig` in kernel/config.py (HOMELAB_MONITOR_SILENCE_*_SECONDS, HOMELAB_MONITOR_SILENCE_GRACE_SECONDS; defaults 900/3600/900).
- New vmalert-metrics rule deploy/vmalert/metrics/signature_silent.yaml (dumb `homelab_log_signature_silent == 1` with labels category=log-anomaly/anomaly_kind=signature_silent/severity=warning). Note: `.yaml` (the glob skips `.yml`).

**Frontend:**
- New SilenceAllowlistTab (Logs → Silence Allowlist) with rhf + zod form: schedule-kind-aware conditional fields (always kind shows just expires_at; window kind shows ISO window inputs; cron kind shows cron expression + grace_seconds input). List section displays service_key + signature, schedule summary (e.g. "every Tue 02:00-04:00"), expiry, delete button.
- Optional signature picker (autocomplete from GET `/api/logs/signatures`, autofills service_key + template_hash) — user can paste/search rather than copy-paste from Signatures tab.
- Refinement additions (user-requested per dual sign-off):
  - "Template Hash" → "Signature" relabel in Signatures and Models tabs (visible labels only; API/DB field `template_hash` unchanged).
  - Description blocks atop Silence Allowlist / Signatures / Models tabs (brief explainers of the feature's purpose).
  - Local-timezone helper text on window/expires datetime inputs (`Displayed in <system timezone>`).
- Bug fix: vertical-scroll clipping in the Logs tab-content area — SilenceAllowlistTab now uses internal `flex h-full min-h-0 flex-col` + `overflow-auto` scroll region (matching Signatures/Models tabs, inherited from LogsExplorerBody pattern).

**Important clarification:** The silence allowlist suppresses ONLY the "signature went silent" alert (`homelab_log_signature_silent` → vmalert rule `SignatureSilent`). It does NOT affect log storage/ingestion, signature catalog updates (NewLogSignature, SignatureSpike, ServiceErrorRateSpike, etc.), or any other system behavior — purely suppresses one alert type.
