### Decisions locked (recorded in stage tracking)

- D1 Full 64-char SHA256 hex fingerprint (matches `alerts/fingerprinting.py` precedent ‚Äî same project, same hash function, same hex format)
- D2 NULL `source_path` for remote crons (matches kernel "absent" convention; `min_length=1` on `CronCreate` prevents empty-string confusion)

### Refinement summary

e2e-tester ran 6 backend scenarios on real DB (backend-only stage; UI changes are mechanical field renames with no UX delta):

1. **Fingerprint round-trip** ‚Äî 64-char hex; same tuple ‚Üí same fp; varying any of (host, source_path, schedule, command) ‚Üí distinct fp; NULL ‚âÝ `""` source_path
2. **Migration 0008 fresh DB** ‚Äî schema correct; 4 seed rows present with env var set; 1 row has `source_path = NULL` (remote-host); `audit_log` has 0 `crons.discover` rows (D5)
3. **Heartbeat receiver on fingerprint URL** ‚Äî 204 on success; `heartbeats_state` upsert + `crons.last_seen_state` mirror + `audit_log` row all correct
4. **Receiver 404 on unknown fingerprint** ‚Äî all-zeros fingerprint returns HTTP 404 with `not_found` error code
5. **Hide ‚Üí 404, unhide ‚Üí 204** ‚Äî D2a regression: `_SELECT_CRON_SQL` filters `AND hidden_at IS NULL`; unhiding restores 204
6. **API CRUD with fingerprint URLs + PATCH whitelist 422** ‚Äî GET list/get use new field shape; PATCH accepts whitelisted fields; PATCH rejects non-whitelisted field (`host`) with HTTP 422 + `extra_forbidden`
