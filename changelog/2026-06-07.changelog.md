## [STAGE-004-031A] "Show surrounding logs" in the Logs Explorer

**Summary:** From a selected log line in the Logs Explorer, reconstruct incident context as an in-place "surrounding-logs mode" showing ~100 lines before + the anchor + ~100 after, with bidirectional accumulating load-older/newer, anchor highlighting, exit, and re-anchor. Completes the model→explorer→surrounding-logs debug flow (step 3 after STAGE-030/031).

**Key changes:**
- Backend `GET /api/logs/window` (anchor-centered, two-sided fetch via a `LogWindowFetcher` singleton; merge + dedup + `anchor_index`).
- `LogWindowFetcher.fetch()` reworked to oversample + sort ascending + keep the N lines NEAREST the anchor (side-aware), since VictoriaLogs returns newest-first and one-sided windows otherwise dropped anchor-adjacent lines.
- `LogWindowFetcher` singleton wired in lifespan + test lifespan; `get_log_window_fetcher` DI dependency; `LogWindowResponse` schema.
- Frontend in-place surrounding-logs mode in `LogsExplorerBody` (replaced the original modal): enter from the Field Inspector, exit button, bidirectional accumulating scroll, re-anchor, anchor highlight via the backend `anchor_index`.
- Surrounding window uses `expr='*'` (ignores the Explorer's active filter) so it shows full context regardless of the active query.
- Scope defaults to All services; only-this-service narrows only when the line has a raw `fields.service` + real `source_type` (never emits a dead `source_type:"unknown"` clause).
- `min-w-max` on the log row so the selected/hover highlight spans the full line width (also fixes normal-mode selection).
- Template "Open in Explorer" (`templateToLogsQl`) anchors on the single longest literal run instead of AND-chaining every segment (fixes empty results when a trailing template segment carried a non-printable ANSI byte). Applied to ModelsDebugPage + SignatureDetailPage.
