## STAGE-002-011 ‚Äî cron_runs table + per-run history schema + heartbeat run_id threading

First of five appended EPIC-002 stages (EPIC-002 was re-opened on 2026-05-19 to add cron run history & run-log viewing ‚Äî see docs/superpowers/specs/2026-05-19-cron-run-logs-design.md). Backend foundation: a new per-cron-invocation `cron_runs` table and the heartbeat-receiver changes that let wrapper-installed (A-mode) crons synchronously record run rows.

### Added

- **New `cron_runs` SQLite table** via alembic migration `0015_cron_runs.py` (chains from `0014`) ‚Äî one row per cron invocation, 16 columns:
  - `run_id` TEXT PRIMARY KEY
  - `cron_fingerprint` ‚Äî which cron spawned this run
  - `source` ‚Äî e.g., "heartbeat", "logscrape"
  - `state` ‚Äî `running`/`ok`/`fail`
  - `started_at`, `ended_at` ‚Äî UTC timestamps
  - `duration_seconds` ‚Äî computed as `ended_at - started_at`
  - `exit_code` ‚Äî process exit code (nullable)
  - `vl_window_start`, `vl_window_end` ‚Äî VictoriaLogs window for this run's logs
  - `overlapping` ‚Äî boolean flag (written by later stages)
  - `enriched_at` ‚Äî timestamp when logs/anomalies were enriched (written by STAGE-002-013/014)
  - `line_count`, `byte_count` ‚Äî log metrics (written by later stages)
  - `content_digest` ‚Äî hash of log contents (written by later stages)
  - `anomaly_flags` ‚Äî detected anomaly bitmask (written by later stages)
  - Plus 3 indexes: `ix_cron_runs_fingerprint_started` on `(cron_fingerprint, started_at DESC)`, a partial `ix_cron_runs_enrich_queue` on `enriched_at IS NULL AND state != 'running'`, `ix_cron_runs_fingerprint_state`
- **`CronRunRepository`** (`apps/monitor/homelab_monitor/kernel/cron/run_repository.py`) ‚Äî new repository class with:
  - `CronRunRecord` and `CronRunListPage` dataclasses
  - `insert_run(run_id, cron_fingerprint, source, started_at)` ‚Äî INSERT OR IGNORE (idempotent on duplicate `run_id`)
  - `close_run(run_id, state, ended_at, exit_code)` ‚Äî ON CONFLICT DO UPDATE UPSERT with lost-`/start` derivation of `started_at` and `vl_window_start` as `ended_at - duration_seconds`
  - `get_run(run_id)` ‚Äî fetch single run
  - `list_runs(cron_fingerprint, state_filter=None, cursor=None, limit=50)` ‚Äî cursor pagination with state filtering and fingerprint isolation
- **`get_cron_run_repo` FastAPI dependency** (`kernel/api/dependencies.py`) and `CronRunRepository` construction + `app.state.cron_run_repo` wiring in `lifespan.py`

### Changed

- **Heartbeat endpoint query schemas** ‚Äî all three heartbeat endpoints `POST /api/hb/{fingerprint}/start|ok|fail` gained an optional `run_id` query param:
  - `HeartbeatStartQuery`, `HeartbeatOkQuery`, `HeartbeatFailQuery` classes updated
  - `HeartbeatOkQuery` additionally gained optional `exit_code` param
  - `run_id` validated as charset `[A-Za-z0-9._-]`, length 1-64 ‚Äî HTTP 422 on violation
- **Heartbeat receiver logic** ‚Äî when a heartbeat call carries a `run_id`, the receiver synchronously creates (on `/start`) or closes (on `/ok`|`/fail`) a `cron_runs` row IN ADDITION TO the existing `heartbeats_state` update:
  - `/start` with `run_id`: INSERT into `cron_runs` with `state='running'`, `source='heartbeat'`, `started_at=now()`
  - `/ok` with `run_id`: UPSERT `cron_runs` closing the run with `state='ok'`, `ended_at=now()`, `exit_code=0` (default)
  - `/fail` with `run_id`: UPSERT `cron_runs` closing the run with `state='fail'`, `ended_at=now()`, `exit_code=<actual>`
  - When `run_id` is absent: heartbeat behavior is byte-for-byte unchanged (no `cron_runs` row created)

## STAGE-002-012 ‚Äî wrapper rewrite: generic shared script, run-UUID capture, per-line tagging, Vector transform

Second of the five appended EPIC-002 stages. Rewrites the cron heartbeat wrapper into a generic shared script that captures per-run stdout/stderr with exact run-UUID attribution, adds a Vector transform to route that output into VictoriaLogs, and adds a `wrapper_format_version` column + a format-migration path so pre-existing wrapped crons can be upgraded onto the new convention.

### Added

- **Generic wrapper template** (`apps/monitor/homelab_monitor/data/cron-with-heartbeat.sh.tmpl`) ‚Äî a POSIX-sh script that is byte-identical for every cron: the cron fingerprint is a positional argument (`$1`), the heartbeat URL base is read at runtime from `/etc/homelab-monitor/wrapper.env`. Per invocation it generates a run UUID, logs `HM_RUN_START`/`HM_RUN_END` boundary markers to journald (`logger --tag hmrun`), runs the real command with each output line prefixed `HM_RUN=<uuid> ` (so VictoriaLogs can attribute lines to a run), and threads `run_id` onto the heartbeat `/start|/ok|/fail` calls. Embeds a `WRAPPER_FORMAT_VERSION` constant (`1.0.0`).
- **Exit-code preservation through the `logger` pipe** via a tempfile technique (`{ cmd; echo $? > codefile; } | <tee+logger loop>`) ‚Äî pure POSIX sh (no `pipefail`/`$PIPESTATUS`), trap-cleaned; the real command's exit code is always returned.
- **New Vector transform branch** (`deploy/vector/vector.toml.template`) ‚Äî `hmrun_filter` + `hmrun_shaped`: parses the `HM_RUN=<uuid>` per-line prefix into a regular `run_id` VictoriaLogs field and strips the prefix from the stored message body; the `drop_noise` transform excludes `hmrun` lines.
- **`WriteWrapperEnvOp` IPC operation** (+ `OP_WRITE_WRAPPER_ENV`) ‚Äî the install path writes `/etc/homelab-monitor/wrapper.env` (mode 644) via the host cron-apply executor.
- **New `wrapper_format_version` TEXT column** on the `crons` table (alembic migration `0016_crons_wrapper_format_version.py`, chains from `0015`).
- **New `format_outdated` member** on the `WrapperHealth` Literal ‚Äî a cron whose installed wrapper predates the run-log-capable format reports `wrapper_health: "format_outdated"`; the CronDetail wrapper-health badge renders it as "Re-install to enable run logs".
- **`is_legacy_wrapped` helper** + `WRAPPER_ENV_PATH` / `WRAPPER_FORMAT_VERSION` constants in `wrapper_constants.py`.

### Changed

- **`wrapper_constants.py`** ‚Äî `is_wrapped` / `unwrap_command` now recognize BOTH the new generic wrapper format (`<base> <fingerprint> -- <cmd>`) AND the legacy baked-fingerprint format (`<base> -- <cmd>`).
- **The wrapper install path** (`install.py`) ‚Äî `_find_matching_line` returns a `line_is_wrapped` flag + the pre-unwrap `raw_command`; `build_install_kit` re-wraps a legacy-format line onto the new convention (rather than rejecting it) and rejects a current-format line as already-wrapped; `_rewrite_line` replaces the legacy wrapper segment in place (no double-wrap); `_build_wrapper_content` produces the generic wrapper (no per-cron substitution). Install rejects a `public_url` containing a newline.
- **The host cron-apply executor** `scripts/hm-cron-apply.sh` ‚Äî the new `<fingerprint> --` crontab-line shape, a `verify_wrap` that accepts both a fresh wrap and a legacy‚Üícurrent re-wrap, and the `write-wrapper-env` op handler.
- **The cron parser's `parse_one_line`** gained an `unwrap` parameter; `CronRecord` carries `wrapper_format_version`.

### Fixed

- **The format-migration path** (found broken during Refinement 3b on the real prod host): `is_wrapped`/`unwrap_command` recognized only the new wrapper format, so legacy-wrapped crontab lines were invisible to discovery and install ‚Äî the real wrapped cron got soft-deleted, a phantom cron appeared, and install double-wrapped. Fixed via dual-format recognition; the fix auto-heals the registry on the next discovery scan (the real cron is restored, the phantom soft-deleted) ‚Äî confirmed on the prod host.
- **Pre-existing VictoriaLogs ingestion bug**: the Vector `[sinks.vl]` URI was missing `?_msg_field=message`, so VictoriaLogs silently discarded every event (VL only indexes the `_msg` field; Vector emits `message`). VictoriaLogs had ingested ZERO indexed events until this fix.

### Deferred / noted

- A monitor-only `docker compose up --build` does not recreate the Vector container; `docker compose restart vector` is needed for Vector to pick up a re-rendered config (noted in the changelog as an operational observation).

## STAGE-002-013 ‚Äî VictoriaLogsClient + CronRunReconciler + B-mode log-scrape run rows

Third of the five appended EPIC-002 stages. Delivered the asynchronous half of cron run history via a new `VictoriaLogsClient` (bounded VL query primitive with truncated flag and canonical escaping), a new `CronRunReconciler` (scheduler-registered `BaseCollector`, 30 s tick, three phases: window-finalize ‚Üí enrich ‚Üí prune), and the B-mode `cron_runs` row create/close extension of cron_events log parsing.

### Added

- **`VictoriaLogsClient`** (`apps/monitor/homelab_monitor/kernel/logs/victorialogs_client.py`) ‚Äî bounded VictoriaLogs query primitive with:
  - `VlQueryLimits` config (max-lines, max-bytes, HTTP timeout)
  - `truncated` flag on response (query hit limit)
  - `logsql_quote_phrase` helper ‚Äî escapes `\` then `"` per Go-style quoted-string rules for safe LogsQL injection
  - `build_amode_query` ‚Äî constructs UUID-exact query: `SYSLOG_IDENTIFIER:hmrun AND run_id:<uuid>`
  - `build_bmode_query` ‚Äî constructs cron-fingerprint heuristic query for log-scrape attribution
  - `VictoriaLogsClientError` exception for query failures
- **`CronRunReconciler`** (`apps/monitor/homelab_monitor/kernel/metrics/cron_run_reconciler.py`) ‚Äî new `BaseCollector` subclass mirroring `HeartbeatStateCollector`:
  - Runs every 30 seconds via scheduler tick
  - Three-phase loop per tick: window-finalize (stamp `vl_window_end` on runs ready for enrichment) ‚Üí enrich (query VL, update log metrics + content digest) ‚Üí prune (enforce 30-day age limit + 50k-per-cron hard cap via two-pass algorithm)
  - Per-run isolation in enrich phase: one bad VL query cannot poison the entire batch
  - Content-digest normalization: strips timestamps, PIDs, and standalone integers for stable comparison across runs
- **B-mode log-scrape run creation/closure** in `_process_one` (`apps/monitor/homelab_monitor/kernel/api/routers/cron_events.py`):
  - `OBSERVED_RUN` event now creates a `cron_runs` row with `source=logscrape`
  - `STATE_OK`/`STATE_FAIL` correlates and closes the most-recent open B-mode run by cron fingerprint
  - New `_close_bmode_run` helper

### Changed

- **`CronRunRepository`** (`apps/monitor/homelab_monitor/kernel/cron/run_repository.py`) ‚Äî added 7 new methods:
  - `list_open_bmode_runs(state_filter=None, cursor=None, limit=50)` ‚Äî pagination over runs still in "running" state
  - `find_open_run_by_fingerprint(cron_fingerprint)` ‚Äî find most-recent open run for a fingerprint
  - `list_runs_needing_enrich(cursor=None, limit=50)` ‚Äî cursor pagination over runs with `enriched_at IS NULL AND state != 'running'`
  - `finalize_bmode_run(run_id, vl_window_end)` ‚Äî stamp `vl_window_end` on a run
  - `set_overlapping(run_id, overlapping)` ‚Äî flag runs that overlap with others
  - `set_enrichment(run_id, line_count, byte_count, content_digest, anomaly_flags, enriched_at)` ‚Äî bulk update enrichment fields
  - `prune_runs(retention_days, max_per_cron)` ‚Äî two-pass prune (age, then per-cron cap)
- **`VlQueryLimits` and `CronRunReconcilerConfig` dataclasses** in `apps/monitor/homelab_monitor/kernel/config.py` with env var loaders:
  - `HOMELAB_MONITOR_VL_QUERY_MAX_LINES`, `_MAX_BYTES`, `_TIMEOUT_SECONDS`
  - `HOMELAB_MONITOR_CRON_RUN_RETENTION_DAYS`, `_MAX_ROWS_PER_CRON`, `_BMODE_TIMEOUT_HOURS`, `_ENRICH_GRACE_SECONDS`
- **`/api/logs/query` endpoint** (`apps/monitor/homelab_monitor/kernel/api/routers/logs.py`) ‚Äî refactored to delegate to `VictoriaLogsClient` as the single VL-query code path
- **`lifespan.py`** ‚Äî registers `CronRunReconciler` via `loader.register` after `HeartbeatStateCollector`

### Fixed

- **BUG-1: Invalid LogsQL for cron commands with backslashes/quotes** ‚Äî `build_bmode_query` produced invalid LogsQL for real-world commands like `certbot` cron containing `\!`. VictoriaLogs returned HTTP 400. Fixed by introducing `logsql_quote_phrase` which escapes `\` (first) then `"` per Go-style quoted-string rules; applied to both `build_amode_query` (defense-in-depth) and `build_bmode_query`.
- **BUG-2: Single bad run query aborts entire enrich phase** ‚Äî one failed per-run VL query aborted the entire enrich phase every tick, leaving 7000+ rows unenriched. Fixed by wrapping each per-run enrich body in a `try/except Exception` that logs `cron_run_reconciler.enrich_run_skipped` and continues, so one bad run can no longer poison the whole batch.
