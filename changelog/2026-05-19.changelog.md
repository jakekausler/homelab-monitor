# 2026-05-19

## STAGE-002-011 ‚Äî cron_runs table + per-run history schema + heartbeat run_id threading

First of five appended EPIC-002 stages (EPIC-002 was re-opened on 2026-05-19 to add cron run history & run-log viewing ‚Äî see docs/superpowers/specs/2026-05-19-cron-run-logs-design.md). Backend foundation: a new per-cron-invocation `cron_runs` table and the heartbeat-receiver changes that let wrapper-installed (A-mode) crons synchronously record run rows.

### Added

- **New `cron_runs` SQLite table** via alembic migration `0015_cron_runs.py` (chains from `0014`) ‚Äî one row per cron invocation, 16 columns:
  - `run_id` TEXT PRIMARY KEY
  - `cron_fingerprint` ‚Äî which cron spawned this run
  - `source` ‚Äî e.g., "heartbeat", "logscrape"
  - `state` ‚Äî `running`/`ok`/`fail`
  - `started_at`, `ended_at` ‚Äî UTC timestamps
  - `duration_seconds` ‚Äî computed as `ended_at - started_at`
  - `exit_code` ‚Äî process exit code (nullable)
  - `vl_window_start`, `vl_window_end` ‚Äî VictoriaLogs window for this run's logs
  - `overlapping` ‚Äî boolean flag (written by later stages)
  - `enriched_at` ‚Äî timestamp when logs/anomalies were enriched (written by STAGE-002-013/014)
  - `line_count`, `byte_count` ‚Äî log metrics (written by later stages)
  - `content_digest` ‚Äî hash of log contents (written by later stages)
  - `anomaly_flags` ‚Äî detected anomaly bitmask (written by later stages)
  - Plus 3 indexes: `ix_cron_runs_fingerprint_started` on `(cron_fingerprint, started_at DESC)`, a partial `ix_cron_runs_enrich_queue` on `enriched_at IS NULL AND state != 'running'`, `ix_cron_runs_fingerprint_state`
- **`CronRunRepository`** (`apps/monitor/homelab_monitor/kernel/cron/run_repository.py`) ‚Äî new repository class with:
  - `CronRunRecord` and `CronRunListPage` dataclasses
  - `insert_run(run_id, cron_fingerprint, source, started_at)` ‚Äî INSERT OR IGNORE (idempotent on duplicate `run_id`)
  - `close_run(run_id, state, ended_at, exit_code)` ‚Äî ON CONFLICT DO UPDATE UPSERT with lost-`/start` derivation of `started_at` and `vl_window_start` as `ended_at - duration_seconds`
  - `get_run(run_id)` ‚Äî fetch single run
  - `list_runs(cron_fingerprint, state_filter=None, cursor=None, limit=50)` ‚Äî cursor pagination with state filtering and fingerprint isolation
- **`get_cron_run_repo` FastAPI dependency** (`kernel/api/dependencies.py`) and `CronRunRepository` construction + `app.state.cron_run_repo` wiring in `lifespan.py`

### Changed

- **Heartbeat endpoint query schemas** ‚Äî all three heartbeat endpoints `POST /api/hb/{fingerprint}/start|ok|fail` gained an optional `run_id` query param:
  - `HeartbeatStartQuery`, `HeartbeatOkQuery`, `HeartbeatFailQuery` classes updated
  - `HeartbeatOkQuery` additionally gained optional `exit_code` param
  - `run_id` validated as charset `[A-Za-z0-9._-]`, length 1-64 ‚Äî HTTP 422 on violation
- **Heartbeat receiver logic** ‚Äî when a heartbeat call carries a `run_id`, the receiver synchronously creates (on `/start`) or closes (on `/ok`|`/fail`) a `cron_runs` row IN ADDITION TO the existing `heartbeats_state` update:
  - `/start` with `run_id`: INSERT into `cron_runs` with `state='running'`, `source='heartbeat'`, `started_at=now()`
  - `/ok` with `run_id`: UPSERT `cron_runs` closing the run with `state='ok'`, `ended_at=now()`, `exit_code=0` (default)
  - `/fail` with `run_id`: UPSERT `cron_runs` closing the run with `state='fail'`, `ended_at=now()`, `exit_code=<actual>`
  - When `run_id` is absent: heartbeat behavior is byte-for-byte unchanged (no `cron_runs` row created)
