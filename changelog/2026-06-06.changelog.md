## [STAGE-004-028] Signature catalog backend + list/drill-in UI

**Summary:** Added a Signatures catalog to /logs: a SQLite `log_signatures` table kept in sync by the drain consumer (touched-only upsert), a REST surface (list / get / patch label+status / live VL samples), and a frontend Signatures tab with a deep-linked full-page detail view.

**Key changes:**
- Migration `0034_log_signatures` (composite PK template_hash+service_key; indexes on service_key/status/last_seen_at). EXPECTED_TABLES → 32.
- `SignatureCatalogSync` collaborator: after each drain cycle, upserts touched signatures — `total_count` accumulates per-cycle line deltas, `first_seen_at` INSERT-only, `last_seen_at` = newest line ts; user `label`/`status` never overwritten by sync. Wired into `DrainConsumer` (runs on normal + VL-failure partial paths; failure never breaks the cycle).
- `SignaturesRepository` (list with service/status/label_q filters + pagination + whitelisted sort; get; update_label; set_status).
- Endpoints (router order after 027's `/signatures/refresh`): `GET /api/logs/signatures`, `GET/PATCH /api/logs/signatures/{template_hash}/{service_key}`, `GET .../samples`. PATCH uses `model_fields_set` (omitting label does not clear it; CSRF-protected). Samples query VL live via a best-effort template→LogsQL `_msg` conjunction; graceful degrade (`template_too_generic` / `vl_unavailable`, never 500).
- 4 Pydantic response/request models; openapi.json regenerated.
- Frontend: `/logs` becomes a tabbed layout (Query / Signatures, URL `/logs/query` vs `/logs/signatures`, index redirect preserves deep-link search params). Signatures tab: desktop table + mobile cards, opaque sticky header, filters. Drill-in is a deep-linked full-page route `/logs/signatures/{template_hash}/{service_key}` with a Back button (selection survives refresh) — label edit, status toggle, live samples, Open-in-Explorer. react-query hooks in `apps/ui/src/api/signatures.ts`.
- Open-in-Explorer deep-link builds an `_msg:"seg" AND ...` conjunction from the template's non-wildcard segments with NO service/source_type scoping (the catalog service_key spans docker/systemd/cron source_types, so a hardcoded `docker:` scope wrongly returned zero logs for non-docker services).

**Files:**

- Backend (modified): apps/monitor/homelab_monitor/kernel/api/{lifespan.py, routers/logs.py, schemas.py}, apps/monitor/homelab_monitor/kernel/logs/drain_consumer.py
- Frontend (new): apps/ui/src/api/signatures.ts, apps/ui/src/routes/logs/{LogsLayout.tsx, SignaturesTab.tsx, SignatureDetailPage.tsx}
- Frontend (modified): apps/ui/src/router.tsx, apps/ui/src/routes/logs/LogsExplorerPage.tsx
- Contract: packages/shared-types/openapi.json (regenerated)

## [STAGE-004-029] Signature annotations

**Summary:** Added timestamped plain-text annotations per log signature — a chronological (newest-first) list of notes shown in the signature detail page, persisted in SQLite, cascade-deleted with the parent signature.

**Key changes:**
- Migration `0035_log_signature_annotations` — table `log_signature_annotations(id, template_hash, service_key, note, author, created_at)` with a composite FK `(template_hash, service_key) → log_signatures ON DELETE CASCADE` (cascade fires natively via `PRAGMA foreign_keys=ON`) + indexes on `(template_hash, service_key)` and `created_at`. `created_at` is ISO-8601 UTC TEXT. EXPECTED_TABLES → 33.
- `AnnotationsRepository` (kernel/logs/signature_annotations_repo.py): `create` (lastrowid re-read), `list_for_signature` (ORDER BY created_at DESC, id DESC), `delete` (scoped by id + composite key → bool). `author` is the denormalized session username (Decision A2 — no created_by FK).
- Nested endpoints (declared after `/samples`): `GET /api/logs/signatures/{template_hash}/{service_key}/annotations` (returns `[]` for unknown signature, no 404), `POST` (201; author from session; explicit signature-existence pre-check → 404 if absent; CSRF), `DELETE .../annotations/{annotation_id}` (204; scoped delete → 404 on miss or cross-signature; CSRF).
- Validation: `note` Pydantic `Field(min_length=1, max_length=2000)` + a field_validator that strips and rejects whitespace-only (stores the stripped value).
- 3 Pydantic models (AnnotationResponse / AnnotationListResponse / AnnotationCreateRequest); openapi.json regenerated.
- Frontend: a new "Annotations" section (`SignatureAnnotations.tsx`) in the signature detail page after "Recent Samples" — a newest-first list (author · relative-time · note) with per-item delete (× → window.confirm), and an add-form (textarea maxLength 2000 + Add button disabled while empty/whitespace OR pending; cleared on success). react-query hooks in `apps/ui/src/api/annotations.ts` (invalidate-on-success). Delete button disabled while a delete is in flight.

**Files:**
- Backend (new): apps/monitor/alembic/versions/0035_log_signature_annotations.py, apps/monitor/homelab_monitor/kernel/logs/signature_annotations_repo.py
- Backend (modified): apps/monitor/homelab_monitor/kernel/api/{routers/logs.py, schemas.py}
- Frontend (new): apps/ui/src/api/annotations.ts, apps/ui/src/routes/logs/SignatureAnnotations.tsx
- Frontend (modified): apps/ui/src/routes/logs/SignatureDetailPage.tsx
- Contract: packages/shared-types/openapi.json (regenerated)

**Tests:** backend repo + API tests (incl. a real-DB ON DELETE CASCADE test, cross-signature delete scoping, author-from-session, whitespace-strip, 422 validation, 404-on-absent-signature, CSRF/auth) + migration count (100% kernel coverage); frontend hook + component tests. Backend e2e Refinement validated 40/40 scenarios against a real backend (incl. cascade firing on the real DB connection path).
