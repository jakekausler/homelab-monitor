## [STAGE-005A-004] Container lifecycle + resource rules + probe enablement

**Summary**

Added the fourth EPIC-005A stage: container lifecycle/resource alerts plus first-party probe enablement. EPIC-003 shipped the Docker collector, cadvisor, and a fully-built per-container probe subsystem, but none of those signals were alerted and no probes were configured (so `homelab_probe_up` was empty and `DockerProbeFailing` was dormant). New `deploy/vmalert/metrics/container_lifecycle.yaml` with 6 rules (all `source_tool: vmalert-metrics` + `target_kind: docker_container` + `integration: docker`):
- `ContainerRestartLoop` (warning, for 0m) — `increase(homelab_container_restart_count[1h]) > 3`. Container restarting repeatedly.
- `DockerImageUpdateCheckSkipped` (info, for 0m) — `homelab_image_update_check_skipped == 1`. Registry image-update check skipped (rate-limit/cooldown); per the locked epic decision this alerts on the SKIPPED signal, not `rate_limit_remaining`. Absent-until-event metric.
- `ContainerHighCpuShare` (warning, for 15m) — `rate(container_cpu_usage_seconds_total{name!=""}[5m]) / count(homelab_host_cpu_percent{cpu!="all"}) > 0.5`. One container using >50% of host cores.
- `ContainerHighMemoryShare` (warning, for 15m) — `container_memory_working_set_bytes{name!=""} / on() group_left() homelab_host_memory_bytes{type="total"} > 0.5`. One container using >50% host RAM.
- `ContainerCriticalMemoryShare` (critical, for 10m) — same join `> 0.85`. One container using >85% host RAM (OOM risk).
- `DockerProbeSlow` (warning, for 5m) — `homelab_probe_duration_seconds > 2`. Complements the existing `DockerProbeFailing`.

Container CPU/mem alerts are SHARE-OF-HOST (locked epic decision option b): no per-container limits exist (`container_spec_memory_limit_bytes`/`machine_cpu_cores` absent), so % is computed against host total cores / host total RAM. The long `for:` durations are the anti-false-fire guard (healthy baseline is tiny — largest container ~1% of host RAM).

PROBE ENABLEMENT: added `homelab-monitor.http.health` Docker labels to four first-party sidecars in `deploy/compose/docker-compose.yml` (monitor `http://container:9090/api/healthz`, grafana `http://container:3000/api/health`, victoriametrics `http://container:8428/health`, alertmanager `http://container:9093/-/healthy`). These are host-agnostic and ship in the public repo, so `homelab_probe_up` emits for every user and `DockerProbeFailing`/`DockerProbeSlow` go live. New `docs/admin/docker-probes.md` documents the first-party probes plus the operator host-override path (`homelab-monitor.<kind>.<name>` labels or `/config/plugins/docker/<container>.yaml` override files).

Two card deliverables were dropped as Design FINDINGS (not deferrals — evaluated and determined the rules should not exist):
- **Bad-exit-code rule DROPPED:** `homelab_container_last_exit_code` is a persisted gauge holding the LAST exit code (history, not a live event); it would false-fire forever on any container whose last exit was non-clean (live: gm-redis=128, campaign_playwright=1). The crash EVENT is already alerted by `ContainerCrashed`.
- **Stuck-non-running rule DROPPED:** "should be running" is not derivable from any metric (no restart-policy metric) and host-specific container names cannot ship in the public repo. A configured probe IS the open-source-safe "expected up" declaration — `DockerProbeFailing` covers it.

`fixture.yaml` (`FixtureHostHighCPU`) was KEPT (the card's "remove leaked fixture" premise was incorrect): it is an intentional test-only artifact used by `apps/monitor/tests/integration/test_canonical_e2e.py` as the canonical end-to-end smoke test; it stays inactive in prod because `fixture_cpu_percent` never exists there.

D-005A-004-RUNNING-WHITELIST: stuck-non-running rule dropped (no restart-policy metric; host names can't ship public; DockerProbeFailing is the open-source-safe expected-up signal). ## [STAGE-005A-003] Collector health rules

**Summary**

Added the third EPIC-005A stage: collector self-monitoring alerts in a new `deploy/vmalert/metrics/collector_health.yaml`. The collector framework emits `homelab_collector_run_*` self-metrics for 31 collectors but none were alerted. 4 rules (all `source_tool: vmalert-metrics` + `target_kind: collector`, keyed on `name`):
- `collector_failure_rate_elevated` (warning, for 15m) — `sum without(reason)(rate(failure_total[15m])) / (rate(success_total[15m]) + sum without(reason)(rate(failure_total[15m]))) > 0.5`. Catches a collector failing >50% of runs (unhealthy but below the quarantine budget).
- `collector_silent_death` (critical, for 30m) — `rate(success_total{name!~"noop|image_update_checker"}[1h]) == 0`. Fast-cadence collectors (~30 run ≤5min); ~90min detection.
- `collector_silent_death_slow` (critical, for 30m) — `rate(success_total{name=~"image_update_checker"}[12h]) == 0`. The slow ~6-7h registry checker; [12h] window > cadence so it never false-fires; ~12.5h detection.
- `collector_duration_anomaly` (info, for 15m) — mean run duration `> clamp_min(3 * avg_over_time(mean[6h:15m]), 1)` (no histogram buckets exist → mean-via-sum/count; per-collector 6h baseline).

`last_error_age_seconds` evaluated and intentionally NOT alerted (redundant with the failure-rate rule — documented D-005A-003-LAST-ERROR-AGE).

Two bugs found + fixed during the stage:
- **Build — label mismatch in the failure-rate rule:** `failure_total` carries `{name,reason}`, `success_total` carries `{name}`; a bare `rate(success)+rate(failure)` matched nothing → empty → rule never fired. Fixed with `sum without(reason)(...)` collapsing failures to per-`{name}` (one alert per collector; the STAGE-005A-001 label-mismatch class).
- **Refinement — chronic false-fire on `image_update_checker`:** a single `[1h]` silent-death rule flagged this healthy collector, which runs every ~6-7h by design (rate-limited registry checks). Fixed by splitting into the fast/slow cadence-class pair above (the slow [12h] rule covers it without false-firing).

**Severity note:** the two silent-death rules were initially authored as `severity: error`, but the runtime `Severity` enum only supports `info|warning|critical` (`error` from any rule is silently coerced to `warning` at ingest — a latent system-wide bug). Per user decision: (1) the silent-death rules use VALID severities — a fully-dead collector is page-now → `critical` (vs `collector_failure_rate_elevated` warning, `collector_duration_anomaly` info); AND (2) `error` was removed as an alert severity EVERYWHERE in the codebase (not just this stage) — see the separate "[CHORE] Remove `error` as a valid alert severity" entry below. No `severity: error` remains in any vmalert rule file, Python alert-severity Literal, or the frontend.

**Modified files**

- `deploy/vmalert/metrics/collector_health.yaml` (NEW) — 4 alert rules.
- `deploy/vmalert/metrics/__tests__/collector_health.tests.yaml` (NEW) — test mirror (FIRE + NO-FIRE per rule; noop + image_update_checker exclusion cases; slow-rule [12h] cases; rule 4 by-design-empty under promtool).

**Design decisions**

D-005A-003-FAILURE-RATE: failure-ratio > 0.5 (cadence-normalized; catches unhealthy-not-quarantined). D-005A-003-SILENT-DEATH: cadence-class split (fast [1h] / slow [12h]) after Refinement found the single-window form false-fires on the 6-7h collector. D-005A-003-LAST-ERROR-AGE: dropped as redundant. D-005A-003-DURATION: per-collector mean-duration baseline anomaly (no histogram buckets). Silent-death severity = critical (user decision; error not a valid enum member).

**Verification**

`make verify` GREEN — backend 4926 pass / 2 known-environmental skips / 100% kernel coverage; pyright/tsc/ruff clean; UI 1541 pass. All 4 rules load `health=ok` on the live prod rig. Fire/no-fire: `promtool test rules collector_health.tests.yaml` → SUCCESS (rules 1+2a+2b promtool-validatable; rule 4 the `[6h:15m]` subquery validated via live-VictoriaMetrics replay — fires on a mean-duration spike, silent below the 1s floor). 3b on real data: zero false-fire at the real baseline (image_update_checker no longer flagged after the cadence-class split); rules confirmed watching real data (31 success series, 30 mean-duration series). ## [CHORE] Remove `error` as a valid alert severity (codebase-wide)

**Summary**

Discovered during STAGE-005A-003 code review: the `Severity` enum (`kernel/alerts/types.py`) defines only `info|warning|critical`, but the spec §8.2 referenced `error` and ~9 vmalert rules (across prior epics) + several Python Literals + the alert-create form still used/allowed `error`. At ingest, `_resolve_severity` silently coerced `error` → `warning`, so every `error`-severity alert was degraded (stored/displayed as warning). Per user decision, `error` is removed as a valid ALERT severity everywhere (the canonical vocab is `info|warning|critical`); LOG-LINE `error` (the syslog level — signature ranking, log histogram buckets, LogsQL `severity:error` query tokens, log-viewer tint) is a separate concept and was left untouched.

**Severity remap (rules):** rules whose `error` middle tier would collapse to duplicate an existing warning sibling were bumped to `critical`; standalone error rules → `warning` (or `critical` if inherently page-now):
- → `critical`: `HaBatteryCritical`, `KernelOOM`, `SelfDiskError` (sibling SelfDiskWarn is warning), `HeartbeatStale_Error` (sibling HeartbeatStale_Warning is warning).
- → `warning`: `HaConfigEntryError`, `HaAutomationScriptFailure`, `HaIntegrationSetupFailure`, `HeartbeatFailed`, `CronRunFailed`.
Alert NAMES (SelfDiskError, HeartbeatStale_Error, etc.) and annotation prose were NOT changed (renaming breaks Alertmanager fingerprints/silences); only the `labels.severity` value changed.

**Modified files**
- vmalert rules (severity label only): `deploy/vmalert/metrics/home-assistant.yaml`, `deploy/vmalert/metrics/self_disk.yaml`, `deploy/vmalert/metrics/heartbeats.yaml`, `deploy/vmalert/logs/home-assistant.yaml`, `deploy/vmalert/logs/system.yaml`, `deploy/vmalert/logs-test/home-assistant.yaml`, `deploy/vmalert/logs-test/system.yaml`, `deploy/vmalert/metrics/__tests__/home-assistant.tests.yaml`.
