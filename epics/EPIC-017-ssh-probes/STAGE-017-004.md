# STAGE-017-004: `hm ssh-probe keygen` + `capture-hostkey`

## Status: Not Started

## Overview

The first half of the setup tooling — `hm ssh-probe keygen <target>` generates a fresh per-target ed25519
keypair, stores the private key in secrets, and prints the public key for manual install; `capture-hostkey
<target>` connects (read-only) and pins the target's host key into config/secrets for the 001 verification.

**Vertical slice:** `hm ssh-probe keygen udm` produces `ssh_probe_key_udm` in secrets + prints a public key;
`hm ssh-probe capture-hostkey udm` records the UDM's host key; neither prints the private key.

**Spec references:** spec §2 Q29 (per-target keys, setup instructions), §7.3 (secrets; `hm secrets` CLI
precedent), §3.1 (CLI). EPIC-017 "CLI & config". STAGE-017-002 (config + secret model), STAGE-017-001
(host-key pinning consumes the captured key).

## Cross-cutting standards

- ed25519; private key written via `AsyncSecretsRepository.set(..., who=)`; **private key NEVER printed/logged**
  (assert).
- `capture-hostkey` is read-only (a connect to read the host key); it does NOT install anything on the target.
- CLI mirrors the `hm secrets` subcommand structure.

## Concrete deliverables

### 1. `hm ssh-probe keygen <target>`

Generate ed25519 keypair; store private key as `ssh_probe_key_<target>` in secrets; print the public key (for
the user to install per 005's instructions). Idempotency/rotation behavior decided at Design (e.g. refuse to
overwrite without `--rotate`).

### 2. `hm ssh-probe capture-hostkey <target>`

Connect to the target (host/port from 002 config) and pin its host key into the target config/secrets so 001
can verify it on every probe. Read-only.
