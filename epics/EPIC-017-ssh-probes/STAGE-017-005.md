# STAGE-017-005: `hm ssh-probe install-instructions` (account-mode-aware) + `test` (verify restriction)

## Status: Complete

## Overview

The second half of the setup tooling — `install-instructions <target>` prints the account-mode-aware setup the
user performs by hand (the framework never writes to a target's auth config), and `test <target>` connects,
runs the forced command, and **verifies the restriction actually holds** (an arbitrary command is refused/
overridden). This is the safety-critical tooling that makes the forced-command boundary real and checkable.

**Vertical slice:** `hm ssh-probe install-instructions udm` prints the `appliance` recipe (authorized_keys
line with `command="..."` + hardening flags + the firmware-update persistence warning); for a `dedicated-user`
target it prints the create-user + install-script + sudoers-line + authorized_keys-line recipe. `hm ssh-probe
test <target>` connects, runs the pinned command, and asserts an arbitrary command does NOT run.

**Spec references:** spec §2 Q29 (setup instructions per remote target; forced commands), §7.2/§7.4 (least
privilege). EPIC-017 "Account model" + "Transport & key model" (manual install; verify the restriction).
STAGE-017-001/002/004 (transport, config, keygen/host-key).

## Cross-cutting standards

- **Instructions NEVER include the private key** — only the public key + steps. Hardening flags
  (`no-port-forwarding,no-pty,no-X11-forwarding,no-agent-forwarding`) included in the authorized_keys line.
- **Account-mode-aware output:**
  - `appliance`: paste this authorized_keys line (with the fixed inlined compound command) into the privileged
    user's authorized_keys + a **firmware-update persistence warning** (re-apply after upgrades).
  - `dedicated-user`: create the low-priv user, install the read-only script (path + content), add the
    NOPASSWD sudoers line (mechanism; specific commands are the consumer epic's), add the authorized_keys line
    pinned to the script.
- `test` is the restriction verifier — it MUST confirm an arbitrary command does not execute (the forced
  command runs instead). This is a real safety check, not just "did it connect."
- Framework ships only the MECHANISM + the trivial exemplar script body; real script bodies/sudoers are
  consumer epics'.

## Concrete deliverables

### 1. `hm ssh-probe install-instructions <target>`

Render the account-mode-aware setup recipe (appliance vs dedicated-user, per above), including the
sudoers-line-generation mechanism for dedicated-user targets and the persistence warning for appliance
targets.

### 2. `hm ssh-probe test <target>`

Connect (per 002 config, 001 transport), run the forced command, capture output, AND verify the restriction:
attempt an arbitrary command and assert it is refused/overridden. Report pass/fail with detail.

### Design Decisions (selected by coordinator, pre-authorized)

| Decision | Selection |
| --- | --- |
| A. appliance forced-command source + line | Render `config.forced_command` VERBATIM inside `command="..."`. If neither `forced_command` nor `script_id` is set → render placeholder `command="<CONFIGURE forced_command IN ssh-targets.yaml>"` + a stderr note (NOT exit 1 — install-instructions is a read-only render useful pre-config). If `script_id` IS set on an appliance target (wrong mode) → error, exit 1. Exact line: `command="<FORCED_COMMAND>",no-port-forwarding,no-pty,no-X11-forwarding,no-agent-forwarding <PUBLIC_KEY> hm-probe-<target>` (flags comma-joined to `command="..."` as ONE options field, space before the key; comment `hm-probe-<target>`). |
| B. dedicated_user render (mechanism + exemplar) | Concrete copy-pasteable 5-step recipe: (1) create low-priv user (`config.user`); (2) install read-only script at `/home/<user>/hm-probe.sh` (mode 0755, owned by `<user>`), body = the TRIVIAL exemplar `#!/bin/sh` + comment + `uptime` (the framework's one shipped body, consumed by 006); (3) sudoers MECHANISM as a clearly-marked PLACEHOLDER — `<user> ALL=(root) NOPASSWD: <ABSOLUTE_PATHS_OF_READ_ONLY_COMMANDS>` to `/etc/sudoers.d/hm-probe-<user>` (validated via `visudo -cf`), with a note "the exemplar uptime needs NO sudo; add only when a real probe (EPIC-008) needs privileged reads"; (4) authorized_keys line forcing the SCRIPT PATH (`command="/home/<user>/hm-probe.sh",<hardening> <PUBKEY> hm-probe-<target>`); (5) "verify with `hm ssh-probe test <target>`". Framework ships MECHANISM + trivial exemplar ONLY — real privileged commands/script bodies are EPIC-008 (placeholder rendered, NOT invented here). NOT a descope — the mechanism + exemplar are fully delivered; EPIC-008 owns the privileged content. |
| C. hardening flags + persistence warning | Hardening flags `no-port-forwarding,no-pty,no-X11-forwarding,no-agent-forwarding` in BOTH modes' authorized_keys lines. Firmware-update PERSISTENCE WARNING is APPLIANCE-ONLY (UniFi-OS overlayfs `/root/.ssh/authorized_keys` is wiped on firmware upgrade → re-apply; meaningless on Synology's persistent ext4 home, so omitted for dedicated_user). |
| D. `test` logic + exit codes | Resolve target → RAW config (needs `host_key`/`forced_command`); pre-check: target-not-in-config → exit 1; `host_key` empty → exit 1 ("run capture-hostkey first"); key secret missing → exit 1 ("run keygen first"); MasterKeyError → exit 1. Then build a one-off factory, `async with factory.open(target) as conn: result = await conn.run("echo HM_PROBE_RESTRICTION_CHECK_<target>")`. **Marker ABSENT from stdout → restriction HOLDS** (the forced command overrode the arbitrary command): print the forced command's actual output (operator visibility) + `PASS: forced-command restriction enforced` → **exit 0**. **Marker PRESENT → restriction NOT enforced** (operator pasted the key WITHOUT `command=`): `FAIL: restriction NOT enforced ... re-check the authorized_keys command="..." option` → **exit 3** (the safety failure this command exists to catch). Connection/auth/host-key errors (SshTransportError taxonomy) → exit 1; `HostKeyMismatch` additionally prints a `CRITICAL: host key mismatch — possible MITM` line. **Exit-code scheme: 0 = restriction holds; 1 = could not test (config/key/connection/auth/host-key); 3 = connected but restriction BROKEN.** (Distinct codes so a wrapping script separates "couldn't check" from "checked and broken".) |
| E. forced-command test fixture | New SHARED fixture `ssh_test_server_forced_command` in `tests/ssh/conftest.py` using `import_authorized_keys('command="echo HM_FORCED_OK" <client_pubkey>')` — exercises the REAL asyncssh server-side forced-command override (reuses the existing echoing `_handle_process`; asyncssh sets `process.command` to the forced command). PASS-path test (forced fixture → marker absent → exit 0) + NEGATIVE/safety test (existing plain-echo `ssh_test_server` → marker echoed back → exit 3, proving `test` actually CATCHES an unenforced restriction, not a no-op) + error-path tests (no host_key → 1; no key → 1; host-key-mismatch → 1). |
| F. placement + shared helpers | Both commands extend `cli/ssh_probe.py` (reuse `_valid_target`/`build_secrets_repo`). Add `_build_target_factory(target, repo)` helper (`AsyncSshClientFactory(resolve=lambda tid: params.get(tid), secrets_for=...)`). Add a RAW-config loader `load_ssh_target_configs() -> dict[str, SshTargetConfig]` to `kernel/ssh/config.py` (the existing `load_ssh_targets()` projects to `SshTargetParams` which DROPS `forced_command`/`script_id`/`concurrency_group` — install-instructions + test's report need the un-projected config-only fields). Build phase: verify whether a raw accessor already exists before adding. |

**Security invariants:** the private key is NEVER printed — install-instructions derives the PUBLIC key via `asyncssh.import_private_key(<secret PEM>).export_public_key().decode().strip()` and prints only that (test asserts no `PRIVATE KEY` in output). `test` is the safety verifier: it makes the forced-command boundary real + checkable (exit 3 on a broken restriction).

**Public-key derivation (verified):** `asyncssh.import_private_key(pem).export_public_key().decode().strip()` → bare `ssh-ed25519 AAAA...`. **Forced-command override (verified):** asyncssh sets `process.command` to the `command="..."` option server-side, ignoring the client request — so `test`'s marker-absent assertion is sound.
