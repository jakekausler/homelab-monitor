# STAGE-017-005: `hm ssh-probe install-instructions` (account-mode-aware) + `test` (verify restriction)

## Status: Not Started

## Overview

The second half of the setup tooling — `install-instructions <target>` prints the account-mode-aware setup the
user performs by hand (the framework never writes to a target's auth config), and `test <target>` connects,
runs the forced command, and **verifies the restriction actually holds** (an arbitrary command is refused/
overridden). This is the safety-critical tooling that makes the forced-command boundary real and checkable.

**Vertical slice:** `hm ssh-probe install-instructions udm` prints the `appliance` recipe (authorized_keys
line with `command="..."` + hardening flags + the firmware-update persistence warning); for a `dedicated-user`
target it prints the create-user + install-script + sudoers-line + authorized_keys-line recipe. `hm ssh-probe
test <target>` connects, runs the pinned command, and asserts an arbitrary command does NOT run.

**Spec references:** spec §2 Q29 (setup instructions per remote target; forced commands), §7.2/§7.4 (least
privilege). EPIC-017 "Account model" + "Transport & key model" (manual install; verify the restriction).
STAGE-017-001/002/004 (transport, config, keygen/host-key).

## Cross-cutting standards

- **Instructions NEVER include the private key** — only the public key + steps. Hardening flags
  (`no-port-forwarding,no-pty,no-X11-forwarding,no-agent-forwarding`) included in the authorized_keys line.
- **Account-mode-aware output:**
  - `appliance`: paste this authorized_keys line (with the fixed inlined compound command) into the privileged
    user's authorized_keys + a **firmware-update persistence warning** (re-apply after upgrades).
  - `dedicated-user`: create the low-priv user, install the read-only script (path + content), add the
    NOPASSWD sudoers line (mechanism; specific commands are the consumer epic's), add the authorized_keys line
    pinned to the script.
- `test` is the restriction verifier — it MUST confirm an arbitrary command does not execute (the forced
  command runs instead). This is a real safety check, not just "did it connect."
- Framework ships only the MECHANISM + the trivial exemplar script body; real script bodies/sudoers are
  consumer epics'.

## Concrete deliverables

### 1. `hm ssh-probe install-instructions <target>`

Render the account-mode-aware setup recipe (appliance vs dedicated-user, per above), including the
sudoers-line-generation mechanism for dedicated-user targets and the persistence warning for appliance
targets.

### 2. `hm ssh-probe test <target>`

Connect (per 002 config, 001 transport), run the forced command, capture output, AND verify the restriction:
attempt an arbitrary command and assert it is refused/overridden. Report pass/fail with detail.
