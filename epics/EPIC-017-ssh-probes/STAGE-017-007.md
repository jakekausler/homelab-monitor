## Concrete deliverables

### 1. `deploy/vmalert/metrics/ssh.yaml`

- `SshTargetUnreachable` (warning) — `homelab_ssh_up == 0` sustained.
- `SshHostKeyMismatch` (**critical**) — `homelab_ssh_host_key_mismatch == 1` (MITM signal).
- `SshProbeStale` (warning) — `homelab_ssh_last_success_age_seconds` beyond N intervals.

**Annotation remediation guidance (required):** Every rule's `annotations.description` must include actionable "what to check when this fires" remediation steps, matching the project's vmalert convention (a `summary` naming the target via `{{ $labels.target }}` + a `description` with troubleshooting guidance, like the existing `disk_fill.yaml` "Investigate disk usage" pattern). Per-rule remediation must cover at minimum:
- **SshTargetUnreachable** (`homelab_ssh_up == 0`): check that the probe's forced-command key is still installed in the target's `authorized_keys`; **on appliance/UniFi-OS targets, a firmware update WIPES `/root/.ssh/authorized_keys` (overlayfs) — re-apply the authorized_keys line** (`hm ssh-probe install-instructions <target>`); verify the target is powered on / reachable on its host:port; re-run `hm ssh-probe test <target>` to confirm the restriction + connectivity.
- **SshHostKeyMismatch** (`homelab_ssh_host_key_mismatch == 1`): CRITICAL/possible-MITM — do NOT auto-trust; verify the target's host key out-of-band; if the target was legitimately rebuilt/reinstalled, re-run `hm ssh-probe capture-hostkey <target>`, verify the fingerprint, and update `host_key` in the `ssh_targets` config; otherwise investigate for interception.
- **SshProbeStale** (`homelab_ssh_last_success_age_seconds` beyond threshold, accounting for the metric being absent before first success / after a monitor restart): check whether the monitor process restarted (in-memory last-success state resets); confirm the probe is still scheduled/registered; check the same connectivity/key items as SshTargetUnreachable.

Rationale: these remediation steps were identified during STAGE-017-006 Refinement (the firmware-wipe failure mode for the appliance authorized_keys is a real, recurring operational gotcha worth surfacing directly in the alert).

## Design Phase

### Selected Approach

**Decided 2026-06-18 (self-selected per pre-authorization; brainstormer Opus generated options).**

Three framework-health rules in `deploy/vmalert/metrics/ssh.yaml`, single group `ssh_health` (`interval: 30s`), mirroring the `disk_fill.yaml`/`vl_health.yaml` convention. Auto-loaded by vmalert via the `/etc/vmalert/metrics/*.yaml` glob (no registration). All rules are per-target (target label intact) and deliberately avoid `absent()`.

| Decision | Selection |
| --- | --- |
| SshTargetUnreachable | `expr: homelab_ssh_up == 0`, `for: 5m`, **warning**. Per-target (clean target label). NO `absent()` — SSH is multi-target so `absent()` collapses to a label-less series and only fires when ALL targets vanish; a never-emitting probe is a collector-registration failure owned by the `homelab_collector_run_*` collector-health rules, not this file. |
| SshHostKeyMismatch | `expr: homelab_ssh_host_key_mismatch == 1`, `for: 0s`, **critical**. The mismatch gauge is always-present (0/1) and deterministic (not flap-prone); fire immediately — it's the highest-stakes (possible-MITM) signal. |
| SshProbeStale | `expr: homelab_ssh_last_success_age_seconds > 900`, `for: 5m`, **warning**. Keys on the AGE metric (per-target, label intact). SILENT when age is absent (post-monitor-restart / never-succeeded) BY DESIGN — absent age = no measured staleness; the "currently down" case is owned by SshTargetUnreachable. NO `absent()` (would collapse labels). Threshold 900s (15min) sits well clear of the 5m Unreachable window so the two don't overlap-spam. |
| Labels | All three: `severity` (per above) + `source_tool: vmalert-metrics` + `target_kind: ssh_target`. `routing_channel: monitoring-health` on SshTargetUnreachable + SshProbeStale ONLY (probe-health = "your monitoring is degraded", matching the wrapper-health route → `monitoring-health-channel` receiver). SshHostKeyMismatch carries NO routing_channel (default routing) — it's a SECURITY finding about the target, not monitoring degradation. |
| Annotations | Each rule's `description` (`>-` folded scalar) names the target via `{{ $labels.target }}` and carries the card-required remediation: unreachable → check authorized_keys still installed + APPLIANCE FIRMWARE UPDATE WIPES /root/.ssh/authorized_keys (re-apply via `hm ssh-probe install-instructions <target>`) + verify reachable + `hm ssh-probe test`; mismatch → CRITICAL possible-MITM, verify host key OUT-OF-BAND, if legit-rebuilt re-run `hm ssh-probe capture-hostkey <target>` + update host_key, else investigate interception; stale → uses `{{ $value }}` for the age, notes the in-memory-state-resets-on-restart caveat, confirm probe still scheduled, check same connectivity/key items. |

**Key landmine (locked):** Do NOT use `absent()` anywhere in this file. `absent(metric)` on a MULTI-target series collapses to a single label-less series (fires only when EVERY target's series is gone, with no target label) — breaking per-target identity, annotation templating, and `group_by`. The `vl_health.yaml` `or absent(...)` precedent is valid ONLY because VictoriaLogs is a single target. All three SSH rules use plain per-target comparisons.

**Validation plan:** No `promtool` in `make verify` (rules validated at runtime by vmalert). The exprs are plain comparisons (no `absent()`/subquery), so promtool COULD parse them, but the gate is: (Build) the file is valid YAML + vmalert loads it without error; (Refinement 3a) promtool/`docker compose config` validates structure + expr parse; (Refinement 3b) live-replay on the prod rig — confirm SshTargetUnreachable/SshProbeStale fire when a real target is down (age climbing past 900s) and confirm SshProbeStale does NOT spuriously fire on a monitor restart (age absent), and that SshHostKeyMismatch path is correct. Severity labels drive display; `routing_channel` drives Alertmanager routing.
