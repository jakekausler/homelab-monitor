# STAGE-017-001: asyncssh transport — `SshClientFactory` + `SshConnection` impl + host-key pinning

## Status: Complete

## Overview

The foundation stage — fill the kernel's SSH stubs with a real **asyncssh**-based implementation. The kernel
already declares `CollectorContext.ssh: SshClientFactory` and `SshClientFactory.open(target_id)` returning an
async-context-manager yielding an (empty) `SshConnection` Protocol, both marked "real impl: EPIC-017". This
stage adds the `asyncssh` dependency and implements: the factory (resolve target → connect with the
per-target key + pinned host key), the connection (run the pinned forced command → typed output), open-per-run
(no pool), and per-target host-key verification.

**Vertical slice:** a unit test opens a connection to a mock/fixture SSH server via `ctx.ssh.open(target_id)`,
runs the pinned command, gets typed stdout/exit back, and the connection closes; host-key verification accepts
the pinned key and rejects a mismatched one.

**Spec references:** spec §2 Q29 (SSH model), §3.1 (`SshClientFactory` in `CollectorContext`), §5.4
(concurrency model — `concurrency_group`). EPIC-017 "Transport & key model". Kernel stubs at
`apps/monitor/homelab_monitor/kernel/plugins/{context,io}.py`.

## Cross-cutting standards

- `asyncssh` added to `apps/monitor/pyproject.toml`; SSH probes run `run_kind=ASYNC`.
- **Open-per-run, no pool.** Per-target `concurrency_group` (so a target is never hit by parallel connects).
- **Pinned host-key verification — NEVER blanket `known_hosts=None`.** A mismatch is surfaced as a distinct
  error type (consumed as `homelab_ssh_host_key_mismatch` in 003), NOT swallowed as a generic failure.
- Private key read via `ctx.secrets.get(...)`; never logged.
- Typed error mapping (connection refused, host-key mismatch, auth failure, timeout).

## Concrete deliverables

### 1. asyncssh dependency

Add `asyncssh` to backend deps; lock.

### 2. `SshClientFactory` + `SshConnection` impl

Implement `open(target_id) -> AbstractAsyncContextManager[SshConnection]`: resolve the target config (002
provides the model; this stage can use a minimal inline shape or interface), connect with the per-target
ed25519 key + pinned host key, on a configurable non-standard port. Fill the empty `SshConnection` Protocol
with a narrow connect-and-run-the-pinned-command → typed output (stdout/stderr/exit) surface. Open-per-run;
close on context exit.

### 3. Host-key verification

Verify the server host key against the per-target pinned key; raise a distinct `HostKeyMismatch` error on
mismatch (never auto-trust).

## Design Phase

### Design decisions locked (2026-06-17)

**Q1 SshConnection shape:** single `async def run(command: str = "") -> SshCommandResult`. `SshCommandResult` = frozen slots dataclass (stdout:str, stderr:str, exit_status:int; asyncssh None exit → -1). Uses asyncssh `conn.run(command, check=False, encoding="utf-8")`; non-zero exit is NOT an exception (probe interprets exit codes); exceptions reserved for transport failures.

**Q2 target params + resolver seam:** define `SshTargetParams` frozen slots dataclass in `kernel/ssh/params.py` (host, port, user, key_secret_name, pinned_host_key: str|None, account_mode). `AsyncSshClientFactory` is constructed with an injected `resolve: Callable[[str], SshTargetParams | None]` (STAGE-017-002 supplies the real config-backed resolver later; this stage uses a hand-built dict resolver in tests) + a `secrets_for: Callable[[str], str | None]` seam backed by the UNFILTERED `ttl_resolver.current().get(name)` (matching the HA `token_provider` lifespan pattern, since the factory is a singleton and CollectorContext.secrets is a per-collector filtered view). Replaces the existing `_StubSshFactory` in lifespan with `resolve=lambda tid: None` for now.

**Q3 host-key pinning:** `pinned_host_key` = OpenSSH public-key line string. Pin via asyncssh `known_hosts` using the CALLBACK form `known_hosts=lambda host, addr, port: ([key], [], [])` first (version-stable; tuple form is a fallback). Empty/None pinned key → raise `HostKeyNotPinned` BEFORE connecting (never `known_hosts=None`, never auto-trust). A target with no captured host key (pre-017-004) is correctly unprobeable.

**Q4 error hierarchy:** `kernel/ssh/errors.py` → `SshTransportError` base (carries target_id) + subclasses `HostKeyNotPinned`, `HostKeyMismatch` (distinct → drives `homelab_ssh_host_key_mismatch` later), `SshConnectionRefused`, `SshAuthError`, `SshTimeout`. Map asyncssh.HostKeyNotVerifiable→HostKeyMismatch, asyncssh.PermissionDenied→SshAuthError, ConnectionRefusedError/gaierror/OSError→SshConnectionRefused, asyncio.TimeoutError→SshTimeout, other asyncssh.Error→SshTransportError. Connect+host-key+auth happen inside `open()`'s `__aenter__`, so those errors raise on entering `async with factory.open(target_id)`; `run()` raises only command-phase failures.

**Q5 test strategy:** hybrid → a REAL in-process loopback `asyncssh.listen(...)` server fixture (`tests/ssh/conftest.py`, `ssh_test_server` yielding port + server host pubkey line + client key PEM) for the security-critical paths (host-key ACCEPT, host-key MISMATCH reject, HostKeyNotPinned, auth-ok, auth-reject, run-empty, run-selector + exit codes); real triggers for refused (connect to unbound 127.0.0.1 port) and timeout (short connect_timeout). Mock only where a live trigger is genuinely flaky.

**Module layout:** new `kernel/ssh/` dir: `params.py` (SshTargetParams), `errors.py` (the hierarchy), `result.py` (SshCommandResult — split out so io.py's Protocol return-type + the future SshProbe base import it WITHOUT pulling asyncssh; avoids an import cycle), `client.py` (`_AsyncSshConnection` impl of run() + `AsyncSshClientFactory` with `open()` as an `@asynccontextmanager` that connects/pins/maps-errors/async-with-closes per run). Fill `kernel/plugins/io.py` `SshConnection` Protocol with `async def run(command: str = "") -> SshCommandResult`. Add `asyncssh>=2.18` to `apps/monitor/pyproject.toml` flat dependencies (python >=3.12). Delete `_StubSshFactory` in `kernel/api/lifespan.py`.

**Build-phase live-verify (risk #1):** the exact asyncssh `known_hosts` pin API must be confirmed against the installed asyncssh version at Build (callback form first; tuple form fallback). Do NOT assume.

**Acceptance test (vertical slice):** a unit test opens a connection to the loopback server via the factory, runs the pinned command, gets typed stdout/exit, the connection closes; host-key verification ACCEPTS the pinned key and REJECTS a mismatched one.
