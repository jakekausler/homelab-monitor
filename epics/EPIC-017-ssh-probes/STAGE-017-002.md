# STAGE-017-002: `ssh_targets:` config model + per-target key secret model

## Status: Complete

## Overview

The declarative target model — a pydantic-validated `ssh_targets:` config section describing each SSH target,
and the per-target key secret convention. The framework resolves a `target_id` → its host/port/account_mode/
user/key/pinned-host-key/forced-command via this model. Public default is empty (no targets); the user's
overrides repo declares the UDM + Synology.

**Vertical slice:** a config with two `ssh_targets` entries (one `appliance`, one `dedicated-user`) validates;
an invalid entry (bad account_mode, missing key-ref) is rejected with a clear error; the factory (001) can
resolve a target_id to its connection parameters; the per-target key secret name convention is documented.

**Spec references:** spec §6.5 (application configuration; per-plugin config; pydantic validation), §7.3
(secrets). EPIC-017 "CLI & config" + "Transport & key model". STAGE-017-001 (the factory that consumes this).

## Concrete deliverables

### 1. `ssh_targets:` config model

A pydantic model: per-target `id`, `host`, `port`, `account_mode` (`appliance`|`dedicated-user`), `user`,
`key_secret_ref` (default `ssh_probe_key_<id>`), `host_key` (pinned; populated by `capture-hostkey`),
`forced_command` or `script_id`, `concurrency_group`. Validate (account_mode enum, required fields per mode).
Empty public default.

### 2. Per-target key secret convention

Document + wire `ssh_probe_key_<target>` as the secret name holding the per-target private key; the factory
(001) reads it via `ctx.secrets.get`. The key is generated/stored by the CLI in 004.

## Design Phase

**Status**: [x] Design (Complete)

### Design Decisions (selected by coordinator, pre-authorized)

| Decision | Selection |
| --- | --- |
| A. Model shape & location | New module `kernel/ssh/config.py`; pydantic `BaseModel` with `model_config = {"extra": "forbid"}`, one model per target; YAML `ssh_targets:` is a **list of entries**, each carrying an `id`. Loader `load_ssh_targets(...)` returns `dict[str, SshTargetParams]` keyed by id (raises on duplicate id). Fallback cascade mirrors `load_redact_patterns`/`load_logs_config`: missing file → `{}`, section absent → `{}`, null → `{}`, explicit `[]` → `{}`. |
| B. Naming reconciliation | Operator-facing YAML uses the card's field names: `account_mode`, `key_secret_ref`, `host_key`. `account_mode` accepts `appliance` and `dedicated-user` (hyphen) and is **normalized to `dedicated_user`** (underscore) before building `SshTargetParams`. At the config→params projection, `key_secret_ref` → `key_secret_name`, `host_key` → `pinned_host_key`. The frozen `SshTargetParams` contract (STAGE-017-001) is untouched. |
| C. Richer config vs YAGNI | Config model carries `forced_command` / `script_id` / `concurrency_group` **now** (consumed by 017-003/005/006). Validate `forced_command` XOR `script_id` (at most one). Resolver projects DOWN to the narrower `SshTargetParams` (these extra fields are not part of the transport contract). **Rationale: this is a correctness requirement, not YAGNI — `extra="forbid"` would otherwise REJECT the card's documented YAML.** No scope deferred. |
| D. `pinned_host_key` (`host_key`) format | **Bare-only**: store the bare OpenSSH pubkey `ssh-<type> <base64>[ comment]`. Validate it starts with a known key-type token and has a base64 second field. **Reject** a `ssh-keyscan`/known_hosts line (leading hostname token) with a clear error. Matches the 017-001 prod-refinement format contract and what 017-004 `capture-hostkey` writes. |
| E. Resolver wiring | Load **once at startup** in `lifespan.py` (replacing `resolve=lambda tid: None` at the `# TODO(STAGE-017-002)` marker); `resolve` closes over the in-memory id→params dict (`targets.get`). Mirrors the once-at-startup `ha_config` pattern; validation errors fail-fast at startup. |

**Key secret-name convention:** `key_secret_ref` defaults to `ssh_probe_key_<id>` when omitted. 017-002 does NOT write secrets — it only fixes the name convention; the factory's existing `secrets_for` seam reads the key. The key itself is generated/stored by the 017-004 CLI.
