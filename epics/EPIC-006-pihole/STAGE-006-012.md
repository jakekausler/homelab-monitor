# STAGE-006-012: Per-client (Tier 2) collector

## Status: Complete

## Overview

The Tier-2 per-client DNS collector: top talkers, per-client query/block counts, and top blocked/permitted
domains — capped + loopback-attributed via the 006-004 helper. This is the built-in default per-client
visibility (the user's own devices). Feeds Grafana row 5, the panel clients widget, and the
`PiholeClientFlooding` alert. It also produces the join-ready per-client series the unified view (006-027)
and Unifi (007) consume.

**Vertical slice:** `homelab_pihole_client_queries{client_ip,client_name,client_kind}` /
`_client_blocked{...}` + `_top_blocked_domain{domain}` / `_top_permitted_domain{domain}` in VM, capped to
top-N, with loopback attributed to the host `192.168.2.148`.

**Spec references:** EPIC-006 "Privacy tiers" (Tier 2 default), "Client-join contract", "Metric families"
(Per-client). Consumes 006-004 (cap + loopback attribution). Alert: `PiholeClientFlooding` (warning,
rolling-baseline + warm-up).

## Concrete deliverables

### 1. Collector

Poll `/api/stats/top_clients` (and `?blocked=true` for per-client blocked) + `/api/stats/top_domains`
(+ `?blocked=true`) + the network table (~60s). Pass entries through the 006-004 helper. Emit:
- `homelab_pihole_client_queries{client_ip,client_name,client_kind,host_lan_ip?,client_mac?}`
- `homelab_pihole_client_blocked{...}`
- `homelab_pihole_top_blocked_domain{domain}`, `homelab_pihole_top_permitted_domain{domain}`
All capped to top-N (default ~50, configurable) with the cap layer's deterministic eviction.

### 2. Join-ready emission

Ensure the client series carry the labels the unified view (006-027) needs to time-window-join on IP to
Unifi. This stage is the canonical EPIC-006 emitter of per-client identity-by-IP.

## Design Phase

### Design Notes

**Approach selected (coordinator selected own recommendation; brainstormer consulted — Opus):** Tier-2 per-client collector, 5-endpoint, with internal classification+cap reuse.

**Recon-first (live Pi-hole v6 API ground truth, captured in Design):**
- `GET /api/stats/top_clients` → `{"clients":[{"name","ip","count"}], "total_queries","blocked_queries","took"}`; default top-N=10.
- `?blocked=true` is a VALUE-ONLY filter (identical envelope/per-entry keys; only `count` + ranking change).
- `GET /api/stats/top_domains` → `{"domains":[{"domain","count"}], ...}`.
- `count=` IS honored by this FTL build: `count=60` returned 49 clients / 60 domains (default no-count = 10). Confirmed live.
- `GET /api/network/devices` → `{"devices":[{"hwaddr","macVendor","ips":[{"ip","name",...}]}], "took"}`; MAC↔IP lives ONLY here. ~9 of ~49 live clients carry a MAC. Names are null/empty for ~all clients.
- Live self-client is `pi.hole`/`ip="::"` → classifies as `resolver_self` (`::` in `_LOOPBACK_IPS`, name in resolver-names).

**Locked decisions:**
1. **5 GETs**: top_clients, top_clients?blocked, top_domains, top_domains?blocked, network/devices. **4 primary `*_ok` flags** (clients, clients_blocked, domains, domains_blocked) → `ok = any(4 primary)`. A 5th `devices_ok` flag gates MAC-map building but is EXCLUDED from `ok` (MAC is optional; network/devices down → no MAC labels, run still succeeds). Per-endpoint resilience mirrors `gravity.py` (api_took emitted per endpoint before payload-shape check; mis-shaped payload = sub-skip, flag stays True).
2. **Helper extension** (`kernel/pihole/client.py`): add `blocked: bool = False, count: int | None = None` params to `stats_top_clients` and `stats_top_domains`; conditionally pass `params={"blocked":"true","count":str(count)}`; vary the endpoint label so api_took distinguishes all four (`stats/top_clients`, `stats/top_clients_blocked`, `stats/top_domains`, `stats/top_domains_blocked`). Backward-compatible defaults. Reuse the existing `network_devices()` helper for the 5th GET (label `network/devices`).
3. **MAC enrichment INCLUDED in this stage** (NOT deferred). Rationale: the ternary no-silent-deferral rule — RETRACT is invalid (a live data source exists at `/api/network/devices`); DEFER-to-027 is invalid (STAGE-006-027 EXPLICITLY rejects Pi-hole MAC enrichment as a non-goal: "improving Pi-hole MAC coverage via host-side ARP ... explicitly rejected"; 027's join is IP-keyed, MAC is "corroborating secondary key only"). The 012 card itself lists `client_mac?` as a deliverable label and "~9 with MAC" as a 3b expectation. The `RawClient.mac → ClassifiedClient.client_mac` pass-through already exists. So including it here is scope COMPLETION, not creep. Build an `ip→hwaddr` map (flatten `devices[].ips[].ip → devices[].hwaddr`, `setdefault` first-wins on dup IP; one MAC may map to multiple IPs — fine), populate `RawClient.mac = ip_mac.get(ip)` (None when absent → label omitted).
4. **Emission model**: classify ONCE via `classify_clients(raw, host_lan_ip=cfg.host_lan_ip, cap=cap)` on the unfiltered top_clients (requested with `count=cap+10`); reuse the returned `kept` (loopback-exempt + capped LAN) for BOTH client families. `client_queries` uses each client's total `value`; `client_blocked` looks up the blocked count by IP from the top_clients?blocked response (`blocked_by_ip.get(ip, 0.0)` → symmetric membership). Domains: two independent `cap_domains((domain,count), cap)` calls (blocked / permitted). **4 drop-gauge series** `homelab_metric_family_dropped_series{family}` always emitted (even 0), one per family: `homelab_pihole_client_queries`, `homelab_pihole_client_blocked` (same dropped count as queries — shared classification), `homelab_pihole_top_blocked_domain`, `homelab_pihole_top_permitted_domain`. **Labels: OMIT `host_lan_ip`/`client_mac` when None** (matches the card's `?` notation + Prometheus idiom — a deliberate, documented divergence from gravity's `""`-default, which exists for always-conceptually-present fields; host_lan_ip/client_mac are structurally absent for whole client classes). Always-present labels: `client_ip`, `client_name`, `client_kind`.
5. **Cap config** (`kernel/config.py`): add `"pihole_client_queries": 50` and `"pihole_top_domains": 50` to `_DEFAULT_CARDINALITY_FAMILIES`. Collector: `cap = load_cardinality_caps_config().cap_for("pihole_client_queries")` for the client pair and `cap_for("pihole_top_domains")` for the domain pair (mirrors UniFi `client_stats.py` `_CAP_FAMILY` precedent). Request `count = cap + 10` from the API so the cap is exercisable. `host_lan_ip` from `PiholeConfig.host_lan_ip` (default `""` → loopback "unattributed").

**Metric families (pinned — literal-name test required):**
- `homelab_pihole_client_queries{client_ip,client_name,client_kind,host_lan_ip?,client_mac?}`
- `homelab_pihole_client_blocked{client_ip,client_name,client_kind,host_lan_ip?,client_mac?}`
- `homelab_pihole_top_blocked_domain{domain}`
- `homelab_pihole_top_permitted_domain{domain}`
- shared `homelab_pihole_api_took_seconds{endpoint}` (5 endpoint labels)
- `homelab_metric_family_dropped_series{family}` (4 series)

**Collector identity:** new file `plugins/collectors/integrations/pihole/top_clients.py` (distinct from `kernel/pihole/clients.py` — no collision), class `PiholeClientsCollector`, `name="pihole_clients"`, `interval=60s`, `concurrency_group="pihole"`. One-line `_PIHOLE_COLLECTORS` registration + import.

**No deferrals.** Self-contained: helper extension + cap-config families + new collector + tests. No migration, no openapi/frontend change (pure collector + kernel-helper changes; openapi-export/generate-ui-types hooks will no-op).

**Brainstormer skip rationale:** NOT skipped — full Opus brainstormer consulted twice (initial 5-decision analysis + a follow-up resolving the MAC deferral after I verified 027 disowns MAC). Coordinator selected the recommendations (accepted all; the MAC decision flipped to "include now" once 027's non-goal was confirmed).

### Refinement Notes

**Backend stage — validated via subagents (3a unit/fixtures + 3b live prod), NO user prompt (not a frontend stage).**

**3a (unit/fixtures):** covered by the Build test suite — 28 collector tests, 100% branch: loopback (`pi.hole`/`::` → resolver_self when host_lan_ip set, unattributed when empty), LAN clients, MAC-present/absent, cap-exercising (>cap synthetic input → dropped>0 + drop gauge), per-endpoint error + non-dict-payload sub-skips, flatten edge cases. Re-confirmed green by `make verify`.

**3b (live prod, `homelab-monitor` @ :29090, real Pi-hole):** rebuilt prod via `make dev-prod`, triggered the collector, validated emitted metrics in VictoriaMetrics against live Pi-hole ground truth.

**3b caught a REAL bug unit tests missed (the value of live validation):** the original Design decision (OMIT `host_lan_ip`/`client_mac` labels when None — the Prometheus idiom) was INCOMPATIBLE with the `prometheus_writer`, which stores the FIRST-seen label-key set as the per-metric schema and SILENTLY DROPS any series with a different key set (`labelnames_mismatch`). Result: ~37 of 49 client series were dropped — only the ~12 clients WITH a MAC reached VM. **FIX (Design Decision 4 label-presence REVERSED):** `host_lan_ip`/`client_mac` are now ALWAYS-present label keys with `""` default (stable 5-key set on every series) — matching gravity.py's `""`-default convention. Collector source + 8 test value-lookup dicts updated; `make verify` re-green (5776 passed, 100% branch). Rebuilt prod with the fix; re-validated 3b.

**3b PASS (post-fix, proven same-scrape-tick by a full debugger pass):**
- Client series recovered 12 → ~52 (full live universe; cap=50 LAN + loopback exempt). `labelnames_mismatch` for pihole_client metrics GONE.
- Collector healthy: `homelab_collector_run_success_total{name="pihole_clients"}` > 0, error_total absent.
- All 5 api_took endpoint labels present; 4 drop-gauge families present (domain families dropped=10 each — 60 live domains > cap 50; client families dropped=0 — 49 < cap 50).
- Spot-checks matched live (e.g. 192.168.2.100 queries/blocked within live delta); top blocked domain `mobile.events.data.microsoft.com` matched.

**Two real-world semantics documented (NOT bugs — see regression.md items 5 & 6):**
1. **VictoriaMetrics drops empty-string labels on storage** (`""` label == absent label per Prometheus data model). The collector ALWAYS emits the 5-key set at the EXPOSITION layer (verified via the live `/metrics`: every `homelab_pihole_client_queries` line carries all 5 keys); VM then strips whichever are `""`. So VM-stored series show a variable label set per client — this is EXPECTED, not a regression. Verify the stable label set at the exposition layer, not VM storage.
2. **`client_mac` may be `ip-<addr>` (e.g. `ip-::`)** — this is Pi-hole FTL's own synthetic hardware-address for MAC-less/loopback clients, passed through verbatim from the API (`client_mac=rc.mac`). Exhaustive grep + git history confirmed NO `ip-` synthesis in this repo. Not a bug.
