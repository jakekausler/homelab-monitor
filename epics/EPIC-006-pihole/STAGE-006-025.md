# STAGE-006-025: Tier-3 query-feed shipper (/api/queries → VictoriaLogs)

## Status: Complete

## Overview

The Tier-3 forensic capability: a log shipper that streams Pi-hole's per-query feed (`/api/queries`) into
VictoriaLogs, enabling line-level forensic search in the embedded LogViewer (incident timelines, malware/
exfil DNS pattern hunting, per-query DNSSEC/EDE). **Toggleable, default-OFF in the public release, ON for
this user** (`pihole_stream_query_feed`). Has its own VL stream byte/line cap + configurable retention.
This is NOT a metrics collector — it's a log shipper (per spec §6.3 log streams).

**Vertical slice:** when `pihole_stream_query_feed` is enabled, per-query lines (timestamp, query, client,
status, reply, dnssec, ede) appear in VictoriaLogs under a dedicated stream (capped + retention-bounded),
searchable in the LogViewer (006-024); when disabled (default), nothing is shipped.

**Spec references:** EPIC-006 "Privacy tiers" (Tier 3 toggleable). spec §6.3 (VL streams + per-stream caps),
§9.2 (Logs). The aggregate roll-ups (heatmaps) over this feed are DEFERRED to EPIC-020 STAGE-020-009 (this
stage only SHIPS the line stream; it does not build aggregate views).

- Own VL stream byte/line cap (the EPIC-004 per-stream cap mechanism) + configurable retention (e.g. 14-30
  days, NOT 365) — disk-budget-aware (`HOMELAB_MONITOR_DISK_BUDGET_GB`).
- Treat query data as PII: dedicated stream label, restricted, deliberate retention. Do NOT also ingest the
  raw `pihole.log` text (this API feed is the structured equivalent).
- Use `/api/queries` cursor pagination to stream incrementally without re-fetching (track the cursor);
  reuse the RO session.

## Concrete deliverables

### 1. Query-feed shipper

A collector/shipper (toggle-gated) that pages `/api/queries` (cursor-based) on a cadence, transforms each
query record into a structured VL log line (timestamp, domain, client (Tier-2 attribution applies —
loopback→host), status, reply, dnssec status, ede code, reply_time), and ingests into a dedicated VL stream
(`service:"pihole-queries"` or similar — Design picks the label). Tracks the cursor to avoid duplication.

### 2. Caps + retention + toggle

Apply a VL per-stream byte/line cap; configurable retention for this stream; the whole shipper is gated by
`pihole_stream_query_feed` (no-op + nothing shipped when off). Document the disk-budget impact.

### 3. Hand-off to EPIC-020

Note (in the stream's docs) that aggregate roll-ups (heatmaps, NXDOMAIN/SERVFAIL-by-domain, etc.) over this
stream are EPIC-020 STAGE-020-009's job — this stage only produces the searchable line stream + the
LogViewer view (006-024).

### (Added — deferred from STAGE-006-024) Expose query-feed shipper state to the frontend

Add a `query_feed_streaming: bool` field to `PiholeOverviewResponse`, derived from the `pihole_stream_query_feed` config flag (default `false`). Reports whether the monitor's query-feed shipper is active — distinct from Pi-hole's own `query_logging_enabled` / `privacy_level`. This lets the frontend (STAGE-006-024's `PiholeLogsTab.tsx`) gate the Tier-3 toggle on the real shipper state.

### (Added — deferred from STAGE-006-024) Wire the Tier-3 per-query-feed view toggle into `PiholeLogsTab.tsx`

Wire the Tier-3 per-query-feed view toggle into the existing `apps/ui/src/routes/integrations/PiholeLogsTab.tsx` (the host FTL LogViewer built in STAGE-006-024). When `query_feed_streaming` is `true`, render a toggle that switches the LogViewer's `servicesCsv` between the docker-stdout FTL stream (`docker:pihole-unbound`, built in 024) and the query-feed VL stream (label finalized in THIS stage's Design). When `false` (default public), the toggle is HIDDEN and the tab shows only the FTL viewer (024's behavior). THIS stage's Design finalizes the query-feed VL stream label and MUST use that same label for both the shipper and this toggle (single source of truth). This is a FRONTEND deliverable → 025's Refinement gains a Desktop + Mobile viewport sign-off for the toggle.

## Design Phase

### Design Notes

**Approach selected** (coordinator selected from brainstormer options; no user prompt per pre-authorization). All backend contracts verified at SOURCE level + the live `/api/queries` shape confirmed against the prod rig before design.

**Decision 1 — VL stream label + frontend queryability (the load-bearing decision; verified sound at source).**
The monitor writes to VictoriaLogs DIRECTLY via `VictoriaLogsWriter.ingest(stream, line, ts)` (`apps/monitor/homelab_monitor/kernel/logs/vl_writer.py`) — vector is NOT in this path. BUT the frontend's `GET /api/logs/query?services=<src>:<svc>` filter (`_compose_services_expr`, `routers/logs.py` ~line 252) builds `service:"X" AND source_type:"Y"` — and `ingest()` today sets ONLY `_msg`/`_stream_id`/`_time` (no `service`/`source_type`), so a monitor-written stream is INVISIBLE to that filter. **Resolution (Option A):** extend `ingest()` with OPTIONAL `service`/`source_type` params (backward-compatible — the one existing caller `subprocess_runner.py:180` and all tests omit them, no breakage; `LogsWriter` Protocol satisfied). The shipper writes `_stream_id="pihole-queries"`, `service="pihole-queries"`, `source_type="pihole"`. The frontend toggle uses `servicesCsv="pihole:pihole-queries"`. VERIFIED at source: the flusher POSTs the full event dict via `json.dumps` so extra fields survive to VL; `_EXCLUDED_FIELDS` excludes only `_stream_id`/`_time`/`_msg` (so `service`/`source_type` ARE queryable); `_compose_services_expr` is generic (no docker hardcoding). Define the label ONCE as a shared backend constant near the pihole collectors (e.g. `PIHOLE_QUERY_FEED_STREAM = "pihole-queries"` + `PIHOLE_QUERY_FEED_SOURCE_TYPE = "pihole"`); the frontend hardcodes the matching `"pihole:pihole-queries"` (consistent with how 024 hardcodes `docker:pihole-unbound`). **Build GATE (#1 correctness risk):** before wiring the toggle, prove the live round-trip — a monitor-written event with `service`/`source_type` is actually returned by `/api/logs/query?services=pihole:pihole-queries`.

**Decision 2 — Cursor / incremental dedup model.** Durable high-water query ID persisted in the existing `app_settings` KV (`app_settings_repository.py`; reachable via `ctx.db`) under a key like `pihole.query_feed.last_id`. Each tick: read last_id, page `/api/queries` (cursor is ID-based — response sorted ID-descending; `PiholeRestClient.queries(params)` already exists), ship records with `id > last_id`, write back the new max. Survives restart → no dup re-ship and no silent gap. **First run (no stored key): record the current max ID as the baseline and ship NOTHING/only that page — do NOT backfill the full history** (respects the PII/disk standard). Collector cadence: interval 60s, `concurrency_group="pihole"` (matches `UnboundStatsCollector`). Page with `length` large enough to drain a tick; keep paging within one `run()` until fewer-than-`length`, up to a per-tick page cap (safety) so a burst can't run unbounded.

**Decision 3 — Caps + retention.** VictoriaLogs has NO native per-stream retention (only global `-retentionPeriod`, default 30d via `load_vl_retention_days()`). The query-feed stream INHERITS the global 30d retention (documented honestly). The card's "per-stream byte/line cap" is delivered as an IN-SHIPPER daily byte cap reusing the existing `LogStreamBudgetConfig.bytes_per_day_per_stream` (500 MiB) default, optionally tunable via a new env knob (`HOMELAB_MONITOR_PIHOLE_QUERY_FEED_MAX_BYTES_PER_DAY`). On cap-hit: advance the high-water ID (DROP for the rest of the UTC day, do NOT backlog — an audit feed that silently backlogs then floods is worse), emit a `homelab_collector_run_*` self-observability signal, and log once. A TRUE per-stream retention prune job is NOT built this stage (VL can't enforce per-stream retention; a prune job is real new machinery, YAGNI for a single-user homelab) — documented as a limitation; if the user later wants query-feed retention shorter than global, that's a future stage (not a dropped card deliverable — the card's "configurable retention" is honestly satisfied by global retention + the documented in-shipper byte cap).

**Decision 4 — Structured log-line schema.** Each VL line is a JSON-object string (so VL parses fields for LogsQL filtering — the card wants `domain:`/`client:`/`status:`/`dnssec:`/`ede:` queryable). `_time` (the ingest `ts`) = the real query time (epoch→ISO-8601 UTC), NOT ship time. Field map from `/api/queries` record → line: `id`→`query_id`, `time`→`_time`(+raw), `domain`→`domain`, `client.ip`/`client.name`→ run through the Tier-2 attribution helper `classify_one` (`pihole/clients.py` ~line 106; loopback→host) emitting `client_ip`/`client_name`/`client_kind`(+attributed host), `status`→`status`, `type`→`query_type`, `reply.type`→`reply_type`, `reply.time`→`reply_time`, `dnssec`→`dnssec`, `ede.code`→`ede_code`, `ede.text`→`ede_text`, `upstream`→`upstream`, `cname`→`cname`, `list_id`→`list_id`. Define a fresh narrow dataclass/TypedDict from `PiholeResponse.payload` (no existing typed model). PII is inherent to an audit feed — bounded by feed-OFF-by-default, the daily byte cap, and 30d retention.

**Decision 5 — Frontend toggle wiring (the 024-deferred deliverable).** In `apps/ui/src/routes/integrations/PiholeLogsTab.tsx`, read `usePiholeOverview().data.query_feed_streaming`. When FALSE (default public) → current tab verbatim (no selector; FTL `docker:pihole-unbound` + the errors-only↔all toggle). When TRUE → a segmented STREAM SELECTOR ("Service log" = `docker:pihole-unbound` ↔ "Query feed" = `pihole:pihole-queries`); the errors-only toggle is scoped to FTL mode ONLY (in Query-feed mode it's hidden and the expr defaults to `*` — errors-only is meaningless for structured query records). Swapping the selector swaps `servicesCsv` (the query key includes `services` → auto-refetch; an empty stream returns empty, no error). Zero regression when the flag is off.

**Backend field:** add `query_feed_streaming: bool` to `PiholeOverviewResponse` (CONFIG-derived from the new `pihole_stream_query_feed` flag — NOT a VM query). The flag = `stream_query_feed_enabled: bool = False` on `PiholeConfig` + `load_pihole_config()` reading `HOMELAB_MONITOR_PIHOLE_STREAM_QUERY_FEED`. The collector no-ops (returns ok CollectorResult, 0 emitted) when the flag is off.

**This is a MIXED stage:** backend log shipper (collector + VL-writer extension + config flag + caps + cursor) → e2e-tester validation in Refinement; PLUS the small frontend toggle → Desktop+Mobile viewport sign-off. The impl touches a backend route/model (`query_feed_streaming` on the overview) → openapi-export + generate-ui-types run; 100% kernel branch coverage required for the shipper/flag/cap/cursor code. **PII caution for Refinement:** enabling `pihole_stream_query_feed` ships the user's real DNS history to VL — confirm with the user before flipping the flag ON on their live host.
