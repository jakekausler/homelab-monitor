# STAGE-006-014: DNS health probe (composite up + direct :53 probe)

## Status: Complete

## Overview

The most important signal in the epic: a robust "is Pi-hole actually resolving" probe that does NOT route
its liveness check through Pi-hole in a way that masks the failure. Actively send a real DNS query to
Pi-hole's `:53` listener (resolve `dns.google.com`) and measure success + latency, and combine with the
API-reachability signal into a composite `homelab_pihole_up`. This catches "API up but DNS wedged" â€” the
truest LAN-can-resolve signal. (The Pi-hole-vs-WAN disambiguation is 006-015; the container signal is
EPIC-003.)

**Vertical slice:** `homelab_pihole_up` (composite 1/0) + `homelab_pihole_dns_probe_seconds` in VM; a
forced Pi-hole DNS failure drives `up=0` while the API may still respond, proving the probe is independent.

**Spec references:** EPIC-006 "Metric families" (DNS health probe) + cross-stage criterion ("the 'is
Pi-hole alive' check NEVER routes through Pi-hole in a masking way"). Q4 = all four detection signals.
Alerts: `PiholeDown` (composite up==0, critical), `PiholeDnsProbeFailing` (direct :53 fails, critical).

## Cross-cutting standards

- The probe queries Pi-hole's listener DIRECTLY (a real A-record lookup of `dns.google.com` against
  Pi-hole `:53`), not via any path that would hang ambiguously.
- Composite `up`: combine API-reachability (collector success) + the `:53` probe; Design defines the exact
  composition (e.g. up=1 requires BOTH). Container-down (EPIC-003) and split-divergence (006-015) are
  SEPARATE alert inputs, not folded into `up` here.
- `dns.google.com` is the fixed probe domain (decided).

## Concrete deliverables

### 1. Direct DNS probe

~60s. Send an A-record query for `dns.google.com` to Pi-hole `:53` (use a DNS client lib; Design picks).
Emit `homelab_pihole_dns_probe_seconds` (latency; high/absent on failure) and a probe-success boolean.

### 2. Composite up

Emit `homelab_pihole_up` (1/0) combining API reachability + the probe. This is the metric `PiholeDown`
fires on. Document the composition so operators understand what `up=0` means.

## Design Phase

### Design Notes

**Approach selected (coordinator selected own recommendation; full Opus brainstormer consulted â€” DNS-query mechanism + composite semantics are novel).** A new pihole-bundle collector that performs an INDEPENDENT real DNS A-record query to Pi-hole's `:53` listener over the monitorâ†’LAN network path, using a pure-stdlib async UDP DNS helper in a NEW `kernel/dns/` module. Emits the composite `homelab_pihole_up` (DNS-decisive) + `homelab_pihole_dns_probe_seconds`.

**Recon-first (live source + codebase ground truth, captured in Design):**
- NO existing DNS-query code in the repo; `dnspython` is NOT a dependency (verified `pyproject.toml` + `uv.lock`). The only raw async socket networking is the TCP probe's `asyncio.open_connection` in `kernel/docker/probe_executor.py`.
- The ProbeSupervisor framework (http/tcp/metrics/exec kinds) is CONTAINER-LABEL-DRIVEN â€” probes attach to a monitored docker container and resolve `host`/`container` sentinels against it; it has no UDP DNS kind. A bare LAN IP `:53` does not correspond to a monitored container, so extending ProbeSupervisor is an awkward fit. A TCP-connect to `:53` only proves the port is open, not that DNS resolves â€” insufficient for the card's "actually resolving" requirement.
- The pihole collector bundle (`plugins/collectors/integrations/pihole/`) is the established home; `UnboundStatsCollector` (013) is the sibling pattern. Register a new collector by appending to `_PIHOLE_COLLECTORS` in the bundle `__init__.py` + an import.
- `PiholeConfig.base_url = "http://192.168.2.148:8080"` (host LAN IP; container cannot reach localhost); `host_lan_ip` exists but there is NO DNS host/port config field yet.

**Mechanism options weighed (full Opus brainstormer):**
1. **Pure-stdlib async UDP DNS helper (`kernel/dns/`)** â€” SELECTED. Zero new dep; query originates from the monitor over the real bridgeâ†’LAN path to `192.168.2.148:53` (truest LAN-can-resolve signal, genuinely independent of the REST API); cleanest 015 reuse (parameterized by resolver IP); 100%-branch coverage is clean with owned wire-format + byte-string fixtures. Cost: ~100-150 lines of careful wire-format code, mitigated by a MINIMAL parser (header flags + rcode + ANCOUNT + TC bit only â€” no answer-record decompression).
2. **Add `dnspython`** â€” rejected. New kernel dependency in a repo that scrutinizes deps, for ~2% of the library's surface (single fixed A-record probe needs no TCP-fallback/EDNS/search-domains); 100% branch coverage means mocking a third-party exception taxonomy (comparable test effort, brittle coupling, less clarity).
3. **docker-exec `dig`/`nslookup` inside `pihole-unbound`** â€” rejected (disqualifying). Wrong network vantage: the query would originate from the Pi-hole host over loopback, testing the resolver daemon but NOT the monitorâ†’Pi-hole `:53` LAN path â€” a loopback dig can succeed while the LAN path is wedged, exactly the failure the probe exists to catch. Also couples DNS liveness to the docker socket + container co-location; `dig`/`nslookup` presence in the image is unconfirmed; worse 015 vantage.

**Locked decisions:**
1. **DNS query primitive:** a standalone, reusable PURE helper `async def resolve_a(resolver_ip: str, qname: str, *, port: int = 53, timeout_seconds: float) -> DnsProbeResult` in a NEW `kernel/dns/` module. Parameterized by resolver IP so STAGE-006-015 (split-check vs a WAN upstream like 1.1.1.1) imports it unchanged from the same monitor vantage. Builds a minimal A-record query (12-byte header + QNAME-encoded question) over `loop.create_datagram_endpoint(...)`; parses header flags (rcode, TC bit) + ANCOUNT only (no answer-record decompression). `DnsProbeResult` carries `(ok, rcode, truncated, latency_seconds, error)`. Never raises â€” maps timeout / OSError / malformed to a typed result.
2. **Composite `homelab_pihole_up` semantics â€” DNS-decisive, NOT DNS-AND-API:** `up=1` iff the direct `:53` probe resolves (rcode NOERROR + ANCOUNTâ‰¥1); `up=0` on timeout / network-unreachable / SERVFAIL / REFUSED / NXDOMAIN / malformed/truncated. The API-reachability signal is NOT folded into `up` inside this collector â€” folding it in would re-couple the probe to the very thing it must be independent of (the point is "API up but DNS wedged" â†’ up=0). The API signal already exists separately (`homelab_collector_run_success_total{name=...}`, `homelab_pihole_api_took_seconds`); STAGE-006-016 alert rules build the richer "DNS up but API down" view from the separate series. Keeps the collector single-responsibility; `PiholeDown` (composite up==0) fires precisely when LAN resolution is broken.
3. **Probe target:** resolve `dns.google.com` (A record), fixed per the card.
4. **Metrics (pin in a `test_metric_name_constants_match_contract` literal test):** `homelab_pihole_up` (gauge 1/0, always emitted); `homelab_pihole_dns_probe_seconds` (gauge, latency â€” emitted ONLY on a completed round-trip; OMITTED on hard error so latency graphs are not polluted, with `up=0` + the outcome metric carrying the failure); `homelab_pihole_dns_probe_result{outcome="ok|timeout|servfail|nxdomain|refused|malformed|..."}` (gauge 1.0 â€” lets `PiholeDnsProbeFailing` distinguish failure modes from the generic `homelab_collector_run_*`).
5. **DNS host config:** add an optional `dns_host` / `dns_port` to `PiholeConfig` defaulting to `urlparse(base_url).hostname` + 53 (env-configurable), giving STAGE-006-015 a clean config home for the WAN upstream. Final decision (derive-vs-add-field) confirmed at Build against `config.py`.
6. **Collector shape:** new file `plugins/collectors/integrations/pihole/dns_health.py`, class `PiholeDnsHealthCollector`, `name="pihole_dns_health"`, `interval=30s`, `timeout=15s`, `concurrency_group="pihole"`; appended to `_PIHOLE_COLLECTORS`. Uses `ctx.vm.write_gauge` / `ctx.log`. The DNS query needs no docker socket (it is a pure UDP socket from the monitor) â†’ NO lifespan-injection change required (simpler than 013). Reads the resolver IP from `PiholeConfig` (injected via the existing pihole-config path, or constructed in run() â€” confirm at Build).

**Build-time recon flags (must verify early in Build):**
1. **UDP egress from the bridge network to `192.168.2.148:53`** â€” almost certainly fine (the container already reaches `192.168.2.148:8080` over the same LAN path) but UDP firewall rules can differ from TCP; verify with a real probe early in Build. Fallback if blocked: TCP-53 query, or (last resort, weaker vantage) Option C docker-exec.
2. **Latency metric type:** confirm a plain gauge (matches the existing `*_seconds` pattern) is sufficient for the 016 alert rules; the card only says "probe latency", so a gauge is almost certainly correct.

**No deferrals.** Backend-only; no migration, no API/frontend change. `kernel/dns/` is a new module reusable by 015.

**Files:** CREATE `kernel/dns/` module (DNS helper) + its test; CREATE `plugins/collectors/integrations/pihole/dns_health.py` + `tests/test_dns_health_collector.py`; MODIFY `plugins/collectors/integrations/pihole/__init__.py` (import + `_PIHOLE_COLLECTORS` append); possibly MODIFY `kernel/config.py` (`PiholeConfig` dns_host/dns_port â€” confirmed at Build).
