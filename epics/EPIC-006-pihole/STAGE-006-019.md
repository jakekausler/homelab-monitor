# STAGE-006-019: Generic container lifecycle actions (restart/start/stop) — EPIC-003 back-fill

## Status: Not Started

## Overview

**This is a Pi-hole-AGNOSTIC, generic capability that back-fills functionality missing from EPIC-003.**
EPIC-003 built container monitoring (status, health, restart-counts, image-update digests, the "Pull &
Restart" update action) but NOT a plain operator "restart/start/stop this container" control. Build that
generic action — `POST /api/docker/containers/{id}/{restart|start|stop}` — confirm-gated + audited,
surfaced as buttons on ANY container's inventory/detail view. The Pi-hole panel's container-control widget
(006-023) consumes it for `pihole-unbound`. The code lives in the shared Docker/inventory surface, not in
Pi-hole-specific code; it is merely authored during this epic because the need surfaced here.

**Vertical slice:** the three actions work against any container (via the Docker socket EPIC-003 already
mounts), each confirm-gated + audited; an e2e restarts a throwaway test container and asserts the lifecycle
transition + the audit row.

**Spec references:** spec §3.1 (audit_log), §7.2 (confirm-on-destructive). EPIC-003 (container monitoring +
the Docker socket). EPIC-006 "Operator panel" (the container-control widget consumes this). A
cross-reference note is added to EPIC-003 pointing here.

**REUSE EPIC-003 STAGE-003-010 machinery (do NOT reinvent):** EPIC-003 already shipped the compose-aware
"Pull & Restart" write action, which introduced (a) the confirm-on-destructive UI component, (b) the
`compose_actions` audit table, and (c) the RW docker socket access (EPIC-003 criterion #18). This stage's
generic restart/start/stop actions MUST reuse all three — the same confirm component, the same audit
table/pattern (or a clearly cross-referenced sibling), and the existing RW socket. This is an extension of
003-010's surface to the plain lifecycle verbs, not a new write path.

## Cross-cutting standards

- `require_user_or_token` (scoped) + confirm-on-destructive + `audit_log` on EVERY action.
- Reuse EPIC-003's Docker client/socket access — do NOT add a new Docker access path.
- Generic + Pi-hole-agnostic: the action takes a container id/name and works for any container; no Pi-hole
  coupling in this code. Document it as an EPIC-003 back-fill.
- Scope: restart + start + stop (no kill/remove/recreate — out of scope).

## Concrete deliverables

### 1. Generic lifecycle endpoint(s)

`POST /api/docker/containers/{id}/restart`, `/start`, `/stop` (or one endpoint with an action param) using
the existing Docker client. Each: confirm-on-destructive + `audit_log` (who/what/when, container id+name,
action). Surface action buttons on the container inventory list + container detail view (the EPIC-003 UI
surface).

### 2. Pi-hole panel consumption

The Pi-hole panel (006-023) gets restart/start/stop buttons for `pihole-unbound` that call this generic
action. (The button UI lands in 006-023; this stage provides the backend + the generic inventory/detail
buttons.)

### 3. EPIC-003 cross-reference

Add a note to EPIC-003.md recording that the generic container-lifecycle actions were back-filled here
(traceability).
