# STAGE-006-019: Generic container lifecycle actions (restart/start/stop) — EPIC-003 back-fill

## Status: Complete

## Overview

**This is a Pi-hole-AGNOSTIC, generic capability that back-fills functionality missing from EPIC-003.**
EPIC-003 built container monitoring (status, health, restart-counts, image-update digests, the "Pull &
Restart" update action) but NOT a plain operator "restart/start/stop this container" control. Build that
generic action — `POST /api/docker/containers/{id}/{restart|start|stop}` — confirm-gated + audited,
surfaced as buttons on ANY container's inventory/detail view. The Pi-hole panel's container-control widget
(006-023) consumes it for `pihole-unbound`. The code lives in the shared Docker/inventory surface, not in
Pi-hole-specific code; it is merely authored during this epic because the need surfaced here.

**Vertical slice:** the three actions work against any container (via the Docker socket EPIC-003 already
mounts), each confirm-gated + audited; an e2e restarts a throwaway test container and asserts the lifecycle
transition + the audit row.

**Spec references:** spec §3.1 (audit_log), §7.2 (confirm-on-destructive). EPIC-003 (container monitoring +
the Docker socket). EPIC-006 "Operator panel" (the container-control widget consumes this). A
cross-reference note is added to EPIC-003 pointing here.

**REUSE EPIC-003 STAGE-003-010 machinery (do NOT reinvent):** EPIC-003 already shipped the compose-aware
"Pull & Restart" write action, which introduced (a) the confirm-on-destructive UI component, (b) the
`compose_actions` audit table, and (c) the RW docker socket access (EPIC-003 criterion #18). This stage's
generic restart/start/stop actions MUST reuse all three — the same confirm component, the same audit
table/pattern (or a clearly cross-referenced sibling), and the existing RW socket. This is an extension of
003-010's surface to the plain lifecycle verbs, not a new write path.

## Design Phase

### Design Notes

**Approach selected (coordinator decision, grounded on recon + brainstormer Opus):** Generic, Pi-hole-agnostic container lifecycle WRITE actions, reusing the STAGE-006-018 write-action pattern (scope+confirm+audit). EPIC-003 back-fill, authored in EPIC-006 because the need surfaced here. Scope-disciplined: ONLY scope+confirm+audit (NOT the auto-fix harness — allow-list-per-alert/rate-limit/cooldown/kill-switch are a separate future epic).

**Recon (verified vs live source):**
- DockerSocketClient (`kernel/docker/socket_client.py`) has NO restart/start/stop yet → 019 ADDS them. It talks httpx over the unix socket and RAISES typed `DockerSocketError` (subclasses `DockerSocketConnectionError`/`DockerSocketProtocolError`); it does NOT return result objects. Docker Engine API: `POST /containers/{id}/restart` (optional `{"t":sec}`), `/start`, `/stop` (optional `{"t":sec}`); success = 204 (and 304 = already-in-state = idempotent success).
- `Scope.DOCKER_WRITE` already exists → REUSE (these are docker actions, not a new scope).
- The docker router (`kernel/api/routers/docker.py`, prefix `/integrations/docker`) already has `pull-and-restart` (async-202 via ComposeActionRunner — a DIFFERENT, compose-aware path); 019 adds plain synchronous socket-based lifecycle endpoints to the SAME router.
- `TargetsRepository.list_docker_containers(include_hidden=False)` is the container registry → validate the target exists (404 if unknown); match by `name`, use the row's `container_id` for the socket call. This is the safety guard (only act on KNOWN/discovered containers).
- Audit: reuse `kernel/db/audit.py` `insert_audit(conn, audit_id=uuid7(), who, what, before, after, ip)` + the `audit_log` table (NOT compose_actions — that's compose-specific).
- The STAGE-006-018 pattern to mirror: body-only confirm-validator Depends (runs BEFORE the socket-client dependency → bad confirm = 400 not 503), `require_user_or_token({Scope.DOCKER_WRITE})`, `_who`/`_client_ip` helpers, transactional `insert_audit`, 502 on `DockerSocketError`, audit-on-success-only.

**Design decisions (selected):**
1. API surface: THREE endpoints `POST /api/integrations/docker/containers/{name}/restart` `/start` `/stop` (FOLLOWING THE CARD'S LITERAL URLS + the docker router's per-action `pull-and-restart` convention — OVERRULED the brainstormer's single-`/lifecycle` suggestion: the card explicitly specifies the 3 URLs, and same-router consistency + clean per-action panel-button URLs win; 3× boilerplate mitigated by a shared internal `_lifecycle_action(...)` helper the 3 thin endpoints call).
2. Socket methods RAISE `DockerSocketError`, return `None`; success = status in (204, 304). Signatures: `restart_container(container_id, *, timeout_seconds=None) -> None`, `start_container(container_id) -> None`, `stop_container(container_id, *, timeout_seconds=None) -> None`. Router catches `DockerSocketError` → 502.
3. Confirm: a `confirm_phrase` body field on each action, validated by a body-only Depends (runs before the socket-client dependency); phrase = the action name (`restart`/`start`/`stop`), `.strip().lower()`. Required on ALL THREE (uniform, matching 018). 400 on mismatch.
4. Synchronous 200 response `{action, container_name, container_id, audit_id}` (mirror 018's synchronous shape, NOT pull-and-restart's async-202 — a single socket call needs no action-state table). Do NOT expose Docker's `t` grace param (use Docker default; scope creep otherwise).
5. Validation + audit: validate by name via `list_docker_containers` (404 if unknown), use `container_id` for the socket call. ONE `inspect_container` before the action to capture `before={"state": <status>}`; `after={"action": <action>}`; `what="docker.container.<action>"`. Skip the post-action inspect (target state is deterministic). If the pre-inspect itself raises `DockerSocketError` → 502 (do not proceed).
6. Idempotency: 304 (already-in-state) = success. The audit `before.state` will show it was already in the target state (useful signal).
7. EPIC-003 cross-reference: at Finalize, add a one-line back-fill note at the bottom of `epics/EPIC-003-*/EPIC-003.md` (traceability — a card deliverable).

**Brainstormer used:** yes (Opus); recommendations selected EXCEPT Point 1 (endpoint shape) — overruled to follow the card's literal 3-endpoint URLs.

**Build note (planner):** the socket client's existing ~5s httpx timeout may be too short for `restart` (Docker's default 10s stop-grace before re-start). The new write methods likely need a LONGER per-call timeout (e.g. 30-60s) so a real restart doesn't time out. Confirm + set an appropriate timeout for the write calls.

**Validation plan:** 3a — unit/e2e against a fake DockerSocketClient + seeded targets_repo (confirm enforced, scope rejects unprivileged → 403, 404 unknown container, 502 on DockerSocketError, audit written, the confirm-400-beats-503 ordering). 3b — restart a real NON-CRITICAL container on the prod rig; confirm lifecycle transition + audit row. CAREFUL: real container lifecycle control — do NOT leave any container stopped; restart (not stop) a safe target, or stop→start a throwaway, confirming it ends RUNNING.

**3b/regression flags:** (1) real Docker 204-vs-304 status codes for start-already-running / stop-already-stopped (confirm idempotent-success mapping); (2) a real restart/stop completes within the chosen socket write-timeout; (3) the target container ends in RUNNING state (leave nothing stopped).

### Build Notes

**Implemented (planner Opus spec → scribe + 1 fix cycle):**
- `kernel/docker/socket_client.py` — added `restart_container(container_id, *, timeout_seconds=None)`, `start_container(container_id)`, `stop_container(container_id, *, timeout_seconds=None)`. RAISE `DockerSocketError` subclasses (house style); success = status in (204, 304) (304 = already-in-state idempotent success). Each uses a per-call `timeout=httpx.Timeout(_WRITE_TIMEOUT_SECONDS=30.0, connect=_REQUEST_TIMEOUT_SECONDS)` — the existing 5s default is too short for a restart (Docker's 10s stop-grace). `t` included only when `timeout_seconds` is not None.
- `kernel/api/routers/docker.py` — added 3 synchronous endpoints `POST /api/integrations/docker/containers/{name}/restart|start|stop` (mirroring the docker router's per-action `pull-and-restart` URL convention + the STAGE-006-018 sync pattern, NOT pull-and-restart's async-202). A shared `_lifecycle_action(...)` helper backs all 3 thin endpoints: validates the container exists via `targets_repo.list_docker_containers` (404 if unknown / no container_id), `inspect_container` for the before-state (502 if it raises), calls the matching socket method (502 if it raises), writes a transactional `insert_audit(audit_id=uuid7(), what="docker.container.<action>", before={"state":<status>}, after={"action":<action>}, who, ip)`, returns `ContainerLifecycleResponse(action, container_name, container_id, audit_id)`. Each endpoint: a body-only confirm-validator Depends (phrase == action name, runs BEFORE the socket dependency → bad confirm 400 before socket-None 503), `require_user_or_token({Scope.DOCKER_WRITE})` (REUSED — no new scope). Added a `_get_docker_socket_client_required` provider that 503s on None (the existing provider returns None). Request model `ContainerLifecycleRequest(confirm_phrase)`.
- Regenerated `packages/shared-types/openapi.json` + `apps/ui/src/api/schema.ts` (new routes/models).
- Tests: `test_docker_socket_client.py` (+23: per-method success 204 / idempotent 304 / ConnectError / other-httpx-error / unexpected-status / t-included-vs-omitted / per-call-timeout-applied) + new `test_api_docker_lifecycle.py` (router: 3 success + confirm-fail-400 + confirm-400-beats-socket-503 ordering + 403 scope-reject + 404 unknown-container + 502-on-action + 502-on-pre-inspect + audit-on-success-only + `_who` User/Token branches + `_client_ip` branches). No new scope, no CLI, no action-state table, no allow-list (scope-disciplined).

**No Protocol entry needed:** `DockerSocketClient` is a concrete class (no Protocol/ABC) and the write methods are router-only.
