## Concrete deliverables

### 1. Rich per-client DNS fields on the Client page

Extend the `DnsEnrichment` contract (OpenAPI + Pydantic + the `ClientDnsSection` populated branch) and the `/clients/{mac}` aggregation to add query_volume, block_rate, top_permitted vs top_blocked split, recent-blocks list, and per-client DNSSEC/SERVFAIL — sourced from the `pihole-queries` VL feed. (STAGE-006-027 intentionally shipped only top_domains/blocked_count/last_query_at to respect EPIC-007's narrow slot.)

### 2. VL forensic index promotion

Promote `client_ip` (and likely `domain`/`status`) from JSON-in-`_msg` to INDEXED VictoriaLogs fields in `query_feed.py`, so exact per-client LogsQL (`client_ip:"…"`) works without phrase-match-and-post-filter, and the proxy can support server-side filtering. Enables precise, cheap per-client forensic drill-downs.

### 3. `unifi_name` metric backfill (~30 days)

Reconstruct historical per-client counts from the `pihole-queries` VL feed + the `unifi_client_observations` history and write NEW `homelab_pihole_client_queries{client_ip,unifi_name,…}` / `_client_blocked{…}` series for past timestamps via VM `/api/v1/import/prometheus`. HONEST CAVEATS (document, do not engineer away): VM existing samples are immutable (old unlabeled series coexist forever); only ~30 days back (VL retention); byte-cap days have gaps → undercount; pruned-observation clients won't resolve a name. (STAGE-006-027 shipped going-forward label emit ONLY; user declined backfill there as lossy.)

### Design Notes

**Scope (A2): build deliverables #1 (rich per-client DNS fields) + #2 (VL forensic index promotion). RE-DEFER #3 (the ~30-day unifi_name metric backfill) → tracked in the new STAGE-006-029.**

**Why re-defer #3:** the user EXPLICITLY DECLINED the backfill during STAGE-006-027 ("option 1 is fine" = going-forward label only). Newly-verified reality makes it LOSSIER than they were told: `unifi_client_observations` is pruned (~24h DNS cutoff), so the historical IP→name mapping for old IPs is ALREADY GONE — most reconstructed series couldn't even be name-resolved. Combined with VM sample immutability (old unlabeled series coexist FOREVER in every Grafana panel), the backfill would be mostly-unlabeled, undercounted, permanent. Honoring the prior decline; tracked (not dropped) in STAGE-006-029 (Not Started, optional).

**Deliverable #1 — rich per-client DNS fields (frontend + aggregation; the one deliberate EPIC-007 extension):**
- ADDITIVELY extend `DnsEnrichment` (Pydantic in `integrations_unifi.py` + OpenAPI + the TS `Schema<'DnsEnrichment'>`): KEEP the existing `top_domains`/`blocked_count`/`last_query_at`; ADD as OPTIONAL fields — `query_volume`, `block_rate`, `top_permitted[]`, `top_blocked[]`, `recent_blocks[{domain,time}]`, `servfail_count`, `dnssec_bogus_count`. Additive = non-breaking (EPIC-007's existing `ClientDnsSection` render keeps working).
- Aggregation: add accumulators to the EXISTING `_enrich_client_dns` per-client loop (keeps 027's phrase-match — see decision D). Sources (all confirmed live): query_volume = count(query_id); block_rate = blocked-status / total; top_permitted/top_blocked = partition by status + group by domain; recent_blocks = blocked records sorted by time desc (cap N); servfail = `reply_type=="SERVFAIL"`; dnssec_bogus = `dnssec=="BOGUS"`. (SERVFAIL/BOGUS are 0 on a healthy host — honest-empty most of the time.)
- Render (`ClientDnsSection.tsx`, EPIC-007-owned — the deliberate extension): extend the dl with Query volume + Block rate; show a top-permitted / top-blocked split; a "Recent blocks" list; and SERVFAIL / DNSSEC-bogus badges rendered ONLY WHEN NON-ZERO (honest empty = no dead "0" rows). FRONTEND change → Desktop+Mobile viewport sign-off in Refinement.

**Deliverable #2 — VL forensic index promotion (backend/shipper; decoupled from #1):**
- Promote `client_ip` to an INDEXED VL field: add `client_ip: str | None = None` to `vl_writer.ingest()`, set `event["client_ip"]`, pass it from `query_feed.py`'s `ingest(...)` call. **client_ip ONLY** (domain is high-cardinality with no current consumer; status only if trivially zero-cost). The `/api/logs/query` proxy already supports `client_ip:"X"` server-side once indexed.
- Transition (C-b): indexed-only going forward; OLD records have client_ip in `_msg` only and age out in ~30d (VL retention). Document the boundary — exact `client_ip:"X"` filtering covers only records shipped AFTER this stage, NOT all history.
- **DECOUPLED from #1 (D2):** `_enrich_client_dns` KEEPS its proven 027 phrase-match-and-post-filter (so the user-facing Client page does NOT regress/undercount during the ~30d rollover). #2's value is enabling FUTURE exact per-client forensic search (LogViewer-style), not rewiring #1. Optional `# TODO` in `_enrich_client_dns` noting the phrase-match can swap to the indexed filter once all records are indexed.
- BACKEND/shipper change → e2e validation (re-validate the shipper against live Pi-hole; confirm `client_ip:"X"` works server-side on new records). No viewport sign-off.

**Biggest risk:** the #2 transition boundary being misread as "exact client_ip filtering works on all history" — it only works on records shipped after 028. Mitigated by D2 (the user-facing page stays on phrase-match; #2's exact filter is explicitly future-forensic-scoped, ~30d-bounded). Document it honestly.

**Deferrals (tracked):** #3 backfill → STAGE-006-029 (created this Design). Also future: dnssec_secure_count as a card field (low value), indexed status/domain forensic filters, the LogViewer-style exact per-client forensic UI (all enabled by #2; future stages when a consumer exists).
