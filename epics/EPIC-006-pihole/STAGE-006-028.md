## Concrete deliverables

### 1. Rich per-client DNS fields on the Client page

Extend the `DnsEnrichment` contract (OpenAPI + Pydantic + the `ClientDnsSection` populated branch) and the `/clients/{mac}` aggregation to add query_volume, block_rate, top_permitted vs top_blocked split, recent-blocks list, and per-client DNSSEC/SERVFAIL — sourced from the `pihole-queries` VL feed. (STAGE-006-027 intentionally shipped only top_domains/blocked_count/last_query_at to respect EPIC-007's narrow slot.)

### 2. VL forensic index promotion

Promote `client_ip` (and likely `domain`/`status`) from JSON-in-`_msg` to INDEXED VictoriaLogs fields in `query_feed.py`, so exact per-client LogsQL (`client_ip:"…"`) works without phrase-match-and-post-filter, and the proxy can support server-side filtering. Enables precise, cheap per-client forensic drill-downs.

### 3. `unifi_name` metric backfill (~30 days)

Reconstruct historical per-client counts from the `pihole-queries` VL feed + the `unifi_client_observations` history and write NEW `homelab_pihole_client_queries{client_ip,unifi_name,…}` / `_client_blocked{…}` series for past timestamps via VM `/api/v1/import/prometheus`. HONEST CAVEATS (document, do not engineer away): VM existing samples are immutable (old unlabeled series coexist forever); only ~30 days back (VL retention); byte-cap days have gaps → undercount; pruned-observation clients won't resolve a name. (STAGE-006-027 shipped going-forward label emit ONLY; user declined backfill there as lossy.)
