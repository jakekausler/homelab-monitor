# STAGE-006-017: vmalert-LOGS rules (FTL-log alerts over docker-stdout)

## Status: Not Started

## Overview

Author the 4 logs-based vmalert rules over the Pi-hole container's docker-stdout stream (where FTL.log
lines already land via the EPIC-004 vector pipeline — NO new mount). Catches FTL rate-limit events,
FTL errors/warnings, gravity-update failures, and DB-maintenance anomalies.

**Vertical slice:** `deploy/vmalert/logs/pihole.yaml` rules load on vmalert-logs; a planted FTL log line
(e.g. a `RATE_LIMIT` message) fires the rule; healthy logs are silent.

**Spec references:** EPIC-006 "Alert catalog" (logs rules) + "Verified deployment reality" (FTL lines on
docker-stdout; no new mount). Uses the vector `service` label pinned in 006-001. Pattern: EPIC-005
STAGE-005-014 (vmalert-LOGS over `service:"homeassistant"`).

## Concrete deliverables

### 1. The 4 rules

`deploy/vmalert/logs/pihole.yaml`: PiholeFtlRateLimit (FTL `RATE_LIMIT` line → warning), PiholeFtlError
(FTL ERROR/WARNING lines → warning), PiholeGravityUpdateFailedLog (gravity-run failure lines → warning),
PiholeDbMaintenanceAnomaly (DB-vacuum errors / unusual row-deletion → info). LogsQL over the pinned service
label; tune patterns against real FTL log lines (the live host's FTL.log shows gravity summaries, DB-vacuum,
rate-limit, API warnings — good corpus).

### 2. Test mirror

`deploy/vmalert/logs-test/pihole.tests.yaml` (or the project's logs-test convention) — planted FTL lines
fire the rules; healthy lines silent.
