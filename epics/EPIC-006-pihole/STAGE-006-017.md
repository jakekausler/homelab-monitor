## Design Phase

### Design Notes

**Approach selected (coordinator decision, grounded on live VictoriaLogs corpus + brainstormer):** 4 vmalert-LOGS rules in `deploy/vmalert/logs/pihole.yaml` over `service:="pihole-unbound"`, mirroring the EPIC-005 home-assistant container-logs sibling EXACTLY (group `type: vlogs`, prod `interval: 30s` / `_time:5m`, test mirror `interval: 5s` / `_time:30s`). Labels on every rule: `severity` + `source_tool: vmalert-logs` + `target_kind: container` + `integration: pihole`. Each LogsQL expr ends with `| stats by (host) count() as <name> | filter <name>:>0` (the `| stats` pipe is mandatory). Each rule gets `summary` / `description` ({{ $value }}, {{ $labels.host }}) / `explorer` (`/logs?logsql=...`) annotations.

**Recon (recon-first, verified against live source + live VictoriaLogs):**
- Logs-rule harness is NOT promtool (promtool cannot parse LogsQL). Validation = (a) Python `tests/test_expr_validate.py` LogsQL syntax check (balanced quotes/parens, `| stats` pipe present, reserved fields); (b) vmalert-logs `/api/v1/rules` health=ok; (c) integration test plants log lines via `plant_log_lines(host,service,severity,message,count,vl_url)` and asserts the alert reaches the test webhook. Test mirror file is the BASE name `deploy/vmalert/logs-test/pihole.yaml` (NOT `.tests.yaml`).
- Live service label confirmed: `service="pihole-unbound"`; every record carries a `severity` field (info/warning/error). vmalert-logs has `-configCheckInterval=30s` hot-reload (no rebuild needed for a rules-only change).
- Real 24h corpus: routine `INFO: Optimized database in X s`, `INFO: Gravity database has been updated`, recurring benign `WARNING:` lines (CAP perms, NTP, API bad_request, host-load, and the persistently-inaccessible NRD blocklist warning `WARNING: List with ID N ... was inaccessible during last gravity run` — this RECURS ~2×/gravity-run and is the steady-state baseline, NOT an anomaly). No `ERROR:`-severity lines and no rate-limit lines in the window.

**The 4 rules:**
| Rule | severity | LogsQL `_msg` match | prod `for:` |
|---|---|---|---|
| `PiholeFtlRateLimit` | warning | `"Rate-limiting"` | 2m |
| `PiholeFtlError` | warning | `"ERROR:"` (FTL's own error-level prefix; silent on the many benign `WARNING:` lines) | 2m |
| `PiholeGravityUpdateFailedLog` | warning | a gravity whole-run-FAILURE token (e.g. `"Unable to update gravity"`), explicitly NOT the recurring per-list `"inaccessible during last gravity run"` baseline | 0m |
| `PiholeDbMaintenanceAnomaly` | info | SQLite/DB-failure phrase set (`"database is locked"` OR `"disk I/O error"` OR `"unable to open database"` OR `"Failed to optimize"` OR `"SQLITE_"`); silent on the routine `Optimized database in X s` success | 0m |

All prod thresholds are `filter X:>0` (each is a discrete failure EVENT, not a rate). Test-mirror thresholds `:>0`, `for:0m`, `_time:30s`, `interval:5s` (plant 1 line → fires within one 5s interval), matching the HA sibling.

**Brainstormer used:** yes (Opus) — for the contested LogsQL design points (severity-field vs `ERROR:`-substring; the gravity "inaccessible"-baseline trap; the DB-anomaly vs routine-optimize distinction). Recommendations selected as above.

**Validation plan:** 3a — `test_expr_validate.py` syntax + a planted-line fire test per rule (defeats the loads-but-never-fires trap: a `stats|filter` rule that loads health=ok is NOT proof it can fire). 3b — rules load health=ok on the LIVE `vmalert-logs` container; confirm no false-fire at the real baseline (the recurring `inaccessible` warning must NOT trip PiholeGravityUpdateFailedLog; the routine `Optimized database` line must NOT trip PiholeDbMaintenanceAnomaly).

**3b/regression flags (inferred FTL phrasings — patterns are matched against PLANTED lines at 3a, not induced live failures, because inducing a real gravity/DB failure on the user's live Pi-hole is destructive and forbidden):**
- `PiholeGravityUpdateFailedLog` whole-run-failure phrase is INFERRED (`Unable to update gravity`); the live corpus only shows the success line + the per-list inaccessible warning. Exact FTL whole-run-failure wording to be confirmed if/when a real failure ever lands (regression item).
- `PiholeFtlRateLimit` `Rate-limiting` token and `PiholeFtlError` `ERROR:` prefix are FTL-format-inferred (unobserved in the window) — validated via planted lines at 3a.
