## Overview

Author the 22 metrics-based vmalert rules over the Wave-B/C metrics, in `deploy/vmalert/metrics/pihole.yaml`
(+ test mirror) and the unbound rules. Severities per the EPIC-006 catalog (info|warning|critical).
Absolute-threshold rules carry the load immediately; anomaly rules use the rolling-baseline idiom with
warm-up.

**Vertical slice:** all 22 rules load `health=ok` on the live vmalert-metrics; the absolute-threshold ones
(PiholeDown, BlockingDisabled, GravityStale, AdlistFailing, MessagesPresent, the SERVFAIL/bogus absolute
forms) fire correctly on planted conditions and are silent at the real baseline; anomaly rules promtool-
validate where possible (live-replay for any subquery forms).

**Spec references:** EPIC-006 "Alert catalog" (metrics rules table). Inherits the user-authored-MetricsQL-
rule machinery (005-005) so the user can add/tune Pi-hole thresholds without code. Pattern precedent:
EPIC-005A vmalert-metrics stages + the rolling-baseline `clamp_min(K*avg_over_time(...))` idiom.

## Design Phase

### Design Notes

**Approach selected (coordinator selected own recommendation; full Opus brainstormer consulted — 22 novel PromQL rules over the 005–015 contracts with real false-fire/evaluability risks).** Author 22 vmalert-metrics alert rules in a NEW `deploy/vmalert/metrics/pihole.yaml` (three groups) + a promtool test mirror `deploy/vmalert/metrics/__tests__/pihole.tests.yaml`. Absolute thresholds preferred throughout (a noisy rule is a FAILED rule); 12 rules ship sane defaults and are tuned at 3b.

**Recon-first (verified against source):**
- Rule-file format mirrors `deploy/vmalert/metrics/unifi.yaml`; test-mirror format mirrors `__tests__/unifi.tests.yaml`. `pihole.yaml` does NOT exist yet (create fresh).
- Validation: `docker run --rm --entrypoint promtool -v $(pwd)/deploy/vmalert/metrics:/rules prom/prometheus:v2.47.0 test rules /rules/__tests__/pihole.tests.yaml`. Run MANUALLY in Build — `make verify` does NOT run vmalert tests (consistent with every other `__tests__/*.tests.yaml`). promtool CANNOT evaluate MetricsQL-only constructs (`[Wd:Ns]` subqueries, `label_value()`); such rules would be LIVE-REPLAY-ONLY (EPIC-005A pattern) — but this design uses NO such constructs (all 22 are promtool-validatable absolute/threshold/comparison/and-join/rate-on-real-counter forms).
- Anomaly/floor idiom (`clamp_min(K*avg_over_time(...[W]), FLOOR)`) exists in `host_anomalies.yaml` but is NOT used here — absolute thresholds chosen instead (modest homelab traffic; noisy = failed).
- Metric contracts confirmed for all 005–015 (see expr table). EPIC-003 container metric: `homelab_container_status{name,id,state}` (value always 1.0; state in label).

**Locked decisions:**
1. **Labels (all 22 rules):** `severity` (info|warning|critical ONLY) + `source_tool: vmalert-metrics` + `category: dns` + `integration: pihole`. Chosen over the card's `target_kind` because Pi-hole is an INTEGRATION (like EPIC-007 unifi, which uses `category`+`integration`); `target_kind` is the host-platform convention. Unbound rules keep `integration: pihole` (single integration surface; unbound runs in the `pihole-unbound` container).
2. **Structure:** one file `pihole.yaml`, THREE groups (`interval: 30s` each): `pihole` (16 rules: all `homelab_pihole_*` + PiholeContainerDown + PiholeApiSlow), `unbound` (4 rules), `dns` (2 rules: PiholeDnsProbeFailing, PiholeDnsSplitDivergence). `homelab_dns_resolution_*` is cross-epic (shared with EPIC-016 ISP/WAN) — PiholeDnsSplitDivergence lives HERE (Pi-hole is the subject; `path="pihole"` failing while `path="direct"` works); EPIC-016 owns rules where `path="direct"` itself fails. Document this boundary in the file header.
3. **Severity split: 8 critical / 10 warning / 4 info = 22.** RESOLVES the card's ambiguous "deferred SERVFAIL/bogus absolute forms": `UnboundServfailSpike` + `UnboundDnssecBogusSpike` are those two deferred items, authored as CRITICAL absolute-threshold rules (NOT separate warning rolling-baselines). Rationale: a rolling-baseline over a near-zero SERVFAIL/bogus counter is a false-fire trap; absolute thresholds are correct and avoid redundant paging. Every card-named rule has exactly one home — no silent deferral.
4. **Unbound-extended-off honest-empty:** the homelab runs unbound WITHOUT extended stats, so `answer_rcode`/`answer_bogus_total`/`cache_hit_ratio`/`query_type` series are ABSENT at baseline. The 4 unbound rules are plain threshold comparisons → an absent series yields an empty vector → the rule NEVER fires (correct: loads health=ok, future-proof if extended is enabled). NO `absent()`-guard (which would wrongly invert to fire-on-absence). Test mirror plants synthetic extended series to prove fire + a no-fire case with series absent.
5. **Pi-hole *_total are 24h-rolling GAUGES (no rate()/increase()); unbound *_total are REAL COUNTERS (rate() SAFE).** Block-rate signals use `homelab_pihole_percent_blocked` directly. Upstream rules MUST exclude `upstream=~"cache|blocklist"` (always-populated pseudo-upstreams).

**THE 22 RULES (name | severity | expr | for):**
pihole group —
- PiholeDown | crit | `homelab_pihole_up == 0` | 0m
- PiholeContainerDown | crit | `absent(homelab_container_status{name="pihole-unbound",state="running"}) or homelab_container_status{name="pihole-unbound",state=~"exited|dead|restarting"} == 1` | 2m
- PiholeBlockRateCollapsed | crit | `homelab_pihole_percent_blocked < 1 and homelab_pihole_query_frequency > 0` | 15m | tune floor 1%
- PiholeUpstreamAllDown | crit | `absent(homelab_pihole_upstream_queries{upstream!~"cache|blocklist"} > 0)` | 10m (NOT the count()==0 form — count of empty = empty, never fires)
- PiholeBlockingDisabled | warn | `homelab_pihole_blocking_enabled == 0` | 5m
- PiholeBlockingDisabledIndefinitely | warn | `homelab_pihole_blocking_enabled == 0 and absent(homelab_pihole_blocking_timer_seconds)` | 15m
- PiholeUpstreamDown | warn | `homelab_pihole_upstream_queries{upstream!~"cache|blocklist"} == 0` | 15m | tune
- PiholeGravityStale | warn | `homelab_pihole_gravity_last_update_age_seconds > 691200` (8d; age metric needs no TZ math) | 0m
- PiholeGravityDomainsDropped | warn | `homelab_pihole_gravity_domains < 1000` | 5m | tune floor
- PiholeAdlistFailing | warn | `homelab_pihole_adlist_status{status!="ok"} == 1` | 5m
- PiholeMessagesPresent | warn | `homelab_pihole_messages_count > 0` | 5m (type/count in annotation)
- PiholeDbTooLarge | warn | `homelab_pihole_db_size_bytes > 5368709120` (5 GB) | 10m | tune
- PiholeClientFlooding | warn | `homelab_pihole_client_queries > 50000` | 15m | **MUST tune at 3b (highest false-fire risk; 24h-rolling gauge, high-cardinality labels name the client)**
- PiholeUpdateAvailable | info | `homelab_pihole_update_available == 1` | 0m
- PiholeBlockRateSpike | info | `homelab_pihole_percent_blocked > 60 and homelab_pihole_query_frequency > 0` | 15m | tune
- PiholeApiSlow | info | `max(homelab_pihole_api_took_seconds) > 2` | 10m | tune
unbound group —
- UnboundServfailSpike | crit | `rate(homelab_unbound_answer_rcode{rcode="SERVFAIL"}[5m]) > 1` | 10m | tune (inert while extended off)
- UnboundDnssecBogusSpike | crit | `rate(homelab_unbound_answer_bogus_total[5m]) > 0.1` | 10m | tune (inert while extended off)
- UnboundRecursionSlow | warn | `homelab_unbound_recursion_time_seconds{quantile="0.95"} > 1` | 15m | tune (quantile is a LABEL — plain threshold, no histogram_quantile)
- UnboundCacheHitLow | info | `homelab_unbound_cache_hit_ratio < 0.5 and homelab_unbound_cache_hit_ratio > 0` | 30m | tune (the `>0` guards a freshly-restarted unbound)
dns group —
- PiholeDnsProbeFailing | crit | `homelab_pihole_dns_probe_result{outcome!="ok"} == 1` | 2m (distinct from PiholeDown's up==0 — keys on the probe-outcome series)
- PiholeDnsSplitDivergence | crit | `homelab_dns_resolution_up{path="pihole"} == 0 and on() homelab_dns_resolution_up{path="direct"} == 1` | 5m (**`and on()` is load-bearing** — the naive `and` on differing path selectors yields empty)

**Build-time / 3b verification flags:**
1. **promtool-evaluability of 3 exprs (verify when the test mirror runs in Build):** rule 4 `absent(... > 0)` (NOT count==0); rule 22 `and on()` (verify the rendered firing sample under promtool; fallback `unless on()` form if it misbehaves); rule 2 `absent() or =~` — CONFIRM the actual non-running `state` label vocabulary EPIC-003 emits (`exited|dead|restarting` is the default deny-list; validate against the live container at 3b).
2. **12 must-tune-at-3b thresholds** ship sane defaults in Build; Refinement 3b tunes vs the real Pi-hole baseline + confirms NO false-fire at baseline (a noisy rule is a FAILED rule). PiholeClientFlooding is the #1 false-fire candidate.
3. **3b also:** all 22 rules load `health=ok` on the live vmalert-metrics; the unbound rules correctly stay inert (extended off); plant conditions to confirm the absolute-threshold rules fire.

**No deferrals.** Backend/YAML-only; no code, no migration, no API/frontend change. EPIC-016 STAGE-016-005 consumes `homelab_dns_resolution_*`; EPIC-016 owns the `path="direct"`-fails (WAN) rules.

**Files:** CREATE `deploy/vmalert/metrics/pihole.yaml` (3 groups, 22 rules) + `deploy/vmalert/metrics/__tests__/pihole.tests.yaml` (promtool fire + no-fire per rule; synthetic plants for the unbound-extended rules).
