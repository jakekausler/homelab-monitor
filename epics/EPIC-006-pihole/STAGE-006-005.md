# STAGE-006-005: Core query-stats collector

## Status: Complete

## Overview

The first real Pi-hole collector: poll `/api/stats/summary` and emit the core query metrics that feed
Grafana row 1-3 and most alerts. This is the headline "what is Pi-hole doing" data — totals, block
percentage, query frequency, unique domains/clients, and the single-aspect breakdowns (by type/status/
reply).

**Vertical slice:** `homelab_pihole_queries_total` / `_blocked_total` / `_percent_blocked` /
`_query_frequency` etc. + `query_by_type{type}` / `query_by_status{status}` / `query_by_reply{reply}`
appear in VictoriaMetrics for the live Pi-hole; collector shows healthy in the status surface.

**Spec references:** spec §6.2 (`homelab_pihole_*`). EPIC-006 "Metric families" (Core query stats row).
Research: summary figures are last-24h ROLLING totals (not lifetime counters) — handle as window-gauges;
unbound (006-013) provides the clean monotonic counters for `rate()`.

## Concrete deliverables

### 1. Collector

Poll `/api/stats/summary` (~30s). Emit: `homelab_pihole_queries_total`, `_blocked_total`,
`_forwarded_total`, `_cached_total`, `_percent_blocked`, `_query_frequency`, `_unique_domains`,
`_unique_clients`, `_active_clients`, `_total_clients`; and the breakdowns
`homelab_pihole_query_by_type{type}` (A/AAAA/PTR/HTTPS/SVCB/…), `_query_by_status{status}`
(GRAVITY/FORWARDED/CACHE/REGEX/…), `_query_by_reply{reply}` (NODATA/NXDOMAIN/IP/CNAME/SERVFAIL/…).
Capture the response `took` → feed the API-latency metric (the dedicated took metric is emitted centrally
by the client/collector base; Design decides whether per-collector or one shared took collector — EPIC-006
lists `homelab_pihole_api_took_seconds{endpoint}`).

### 2. Status enum handling

The summary `status`/`reply` enum sets overlap and have many values — emit them faithfully by label; do
NOT naively sum across categories (research caveat). Document the enum set seen live.

## Design Phase

**Status**: [x] Design

### Design Notes

**Selected approach: Option A (adapted) — faithful-summary collector mirroring the Unifi house style, with `unique_clients` RETRACTED (no source).** (Brainstormer produced 3 options; coordinator selected Option A adapted, without user prompt, per backend-stage pre-authorization. Brainstormer used because there were real scope/contract decisions.)

**Backend Contract Verification (live recon, ground truth):** Captured the REAL `/api/stats/summary` from the live Pi-hole v6.6.2. Shape: `queries.{total,blocked,percent_blocked,unique_domains,forwarded,cached,frequency}` + `queries.types{16 keys}` + `queries.status{20 keys}` + `queries.replies{15 keys}`; `clients.{active,total}`; `gravity.{domains_being_blocked,last_update}`; top-level `took`. Pi-hole emits the FULL enum every tick (zero-count labels included) so the ~51 enum series are a STABLE bounded set.

**Decision A — `homelab_pihole_unique_clients` RETRACTED (documented, NOT silently dropped).** The metric has NO source anywhere in Pi-hole v6: `/api/stats/summary` provides only `clients.active` and `clients.total` (no `clients.unique`); verified across ALL 15 PiholeClient GET helpers — none returns a distinct-client COUNT; STAGE-006-012 (per-client Tier-2) emits only per-client top-N series, NOT an aggregate unique count, and no other EPIC-006 stage claims the metric. Because NO stage or endpoint can ever own it, this is a card-deliverable RETRACTION with cause, not a deferral-to-a-future-stage (there is no possible owner). `clients.active` (`homelab_pihole_active_clients`, distinct clients in 24h) and `clients.total` (`homelab_pihole_total_clients`, distinct clients ever seen) ARE emitted and cover the distinct-client information Pi-hole actually exposes. Aliasing `unique_clients` to either would emit a misleading always-equal duplicate series into an alerting path — rejected.

**Decision B — gravity left ENTIRELY to STAGE-006-007.** `/api/stats/summary` also returns `gravity.domains_being_blocked`/`gravity.last_update`, but STAGE-006-007 OWNS the gravity family (`homelab_pihole_gravity_domains` / `homelab_pihole_gravity_last_update_age_seconds`) via its own endpoints (`/api/info/ftl` + `/api/lists`) with a TZ-guarded age computation. 005 parse-skips the `gravity` object to avoid two collectors writing the same family (double-write/ownership confusion). NOT a deferral — 007 already owns it.

**Decision C — NO cardinality cap.** The by-type/status/reply families are a STABLE bounded enum (~51 series, full key set every tick); the aggregate gauges are single-series. The EPIC-005 `CardinalityCapper` is reserved for UNBOUNDED per-client/per-domain families (006-012); applying it to bounded enums is unnecessary.

**Decision D — API latency: `homelab_pihole_api_took_seconds{endpoint="stats/summary"}`** emitted by this collector from `resp.took_seconds`, counted in `metrics_emitted`. Exact mirror of the established Unifi pattern (`homelab_unifi_api_took_seconds{endpoint}`); rejected a shared central took-collector (unjustified new abstraction). Resolves the card's "Design TBD".

**Decision E — collector structure.** New `plugins/collectors/integrations/pihole/stats_summary.py`, class `PiholeStatsSummaryCollector`: `name="pihole_stats_summary"`, `interval=timedelta(seconds=30)` (card-specified, tighter than Unifi's 60s), `timeout=timedelta(seconds=15)`, `concurrency_group="pihole"` (introduce the convention, mirroring Unifi's `"unifi"`). Reuses the `unifi/_parsing.py` house style (`as_float`/`emit_numeric`) plus a local `_emit_enum(ctx, name, obj, label_key, emitted)` helper for the {label:count} objects (defer a dedicated `pihole/_parsing.py` until a second collector needs it). `resp.payload` is typed `object` (no Any) — narrowed with isinstance guards; defensive narrowing (non-dict payload, missing `queries`/`clients`, wrong scalar types) yields per-field omission for 100% branch coverage; the typed `bad_response` PiholeError already covers the 2xx-wrong-shape path at the client layer. Error path mirrors Unifi: `isinstance(result, PiholeError)` → `CollectorResult(ok=False, metrics_emitted=0, errors=[result.message], ...)`.

**Metrics emitted (final set):** `homelab_pihole_queries_total` (=queries.total), `_blocked_total` (queries.blocked), `_forwarded_total` (queries.forwarded), `_cached_total` (queries.cached), `_percent_blocked` (queries.percent_blocked), `_query_frequency` (queries.frequency), `_unique_domains` (queries.unique_domains), `_active_clients` (clients.active), `_total_clients` (clients.total), `_query_by_type{type}` (queries.types), `_query_by_status{status}` (queries.status), `_query_by_reply{reply}` (queries.replies), `_api_took_seconds{endpoint}`. All summary scalars are 24h-ROLLING window-gauges (emit as gauges; downstream must NOT `rate()` them — to be noted in the alert-rule stages 016/017). `homelab_pihole_unique_clients` NOT emitted (retracted, see Decision A).

**Placeholder removal (STAGE-006-002 scaffolding):** DELETE `plugins/collectors/integrations/pihole/placeholder.py`; remove the `PiholePlaceholderCollector` import + entry from `_PIHOLE_COLLECTORS` in the bundle `__init__.py` (replace with `PiholeStatsSummaryCollector` so the list stays non-empty for the 100%-branch gate); rewrite the placeholder/`homelab_pihole_bundle_loaded` tests in `apps/monitor/tests/test_pihole_integration_bundle.py` to assert real-collector registration; and REMOVE the two placeholder/bundle_loaded regression items from `epics/EPIC-006-pihole/regression.md` (replace with a "pihole_stats_summary collector present + emits core metrics" regression item in Refinement).

**Refinement:** 3a (captured `/api/stats/summary` fixture — assert metric shape + the 51 enum labels) + 3b (live summary on prod `:29090` after `make dev-prod` rebuild; sanity-check values vs the Pi-hole web UI). No migration (alembic HEAD `0044`), no openapi/frontend change.
