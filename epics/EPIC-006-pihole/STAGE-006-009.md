## Design Phase

**Status**: [x] Design

### Design Notes

**Two-endpoint collector** (`/api/info/ftl` + `/api/info/database`), mirroring the STAGE-006-007 gravity collector's structure (per-endpoint resilience: `ok = ftl_ok or db_ok`, ok=False only if BOTH endpoints error).

Live recon (Pi-hole v6.6.2, captured this session) revealed several card/API mismatches; the card was WRONG about three fields. Final metric contract:

FROM `/api/info/ftl` (the `ftl` object):
- `homelab_pihole_ftl_uptime_seconds` ← `ftl.uptime` (FLOAT seconds, e.g. 569475247.28 — narrow via `as_float`, not int).
- `homelab_pihole_ftl_cpu_percent` ← `ftl["%cpu"]` (the key is literally `%cpu` — bracket access, not dotted).
- `homelab_pihole_ftl_memory_percent` ← `ftl["%mem"]`.
- `homelab_pihole_privacy_level` ← `ftl.privacy_level` (int 0-3).
- `homelab_pihole_dnsmasq_cache_insertions` ← `ftl.dnsmasq.dns_cache_inserted` (int). dnsmasq is a FLAT object — NO `dnsmasq.cache.*` nesting.
- `homelab_pihole_dnsmasq_cache_evictions` ← `ftl.dnsmasq.dns_cache_live_freed` (int).
- Does NOT touch `ftl.database.gravity` — STAGE-006-007 already owns that as `homelab_pihole_gravity_domains`. No double-emit.

FROM `/api/info/database`:
- `homelab_pihole_db_size_bytes` ← top-level `size` (int bytes; ~1.03 GB live).
- `homelab_pihole_db_queries_total` ← top-level `queries_disk` (the FULL on-disk total row count, 11.6M live — NOT `queries`, which is the recent in-memory window). `_total` + the `PiholeDbTooLarge` growth alert both want the on-disk total.

PLUS `homelab_pihole_api_took_seconds{endpoint=...}` for each endpoint ("info/ftl", "info/database").

Card mismatches and their no-silent-deferral resolutions:
1. **`homelab_pihole_query_logging_enabled` (card metric) does NOT exist in `/api/info/ftl`.** The field lives in Pi-hole's `/api/config` endpoint (`config.dns.queryLogging`), for which there is currently NO client helper. Pure retraction is INVALID (a data source exists). DEFERRED to **STAGE-006-018** (Pi-hole write endpoints — it owns the `/api/config` RW surface; adding a `config()` GET helper + emitting `homelab_pihole_query_logging_enabled` there is cohesive with that stage's config work). A tracked deliverable note has been added to the STAGE-006-018 card.
2. **`homelab_pihole_dnsmasq_cache_*` card named "insertions/evictions/hits/misses".** Only insertions+evictions exist in `/api/info/ftl` (renamed accordingly above). The cache HITS/MISSES signal is NOT retracted and NOT newly deferred — it is ALREADY DELIVERED by STAGE-006-005 (the stats-summary collector) as `homelab_pihole_cached_total` (cache hits, from `queries.cached`), `homelab_pihole_forwarded_total` (≈ misses, from `queries.forwarded`), and `homelab_pihole_query_by_status{status}` (incl. `status=CACHE`/`CACHE_STALE`/`FORWARDED`). 016/026 should consume those existing STAGE-005 metrics for cache hit/miss; this stage adds only the insertions/evictions counters from the FTL process view.

Scope-out (per card): host CPU/mem/temps NOT emitted (node-level + EPIC-005A collectors own those) — Build adds a test asserting these are absent. Bool/numeric narrowing: `as_float` for all numeric fields (uptime/%cpu/%mem/insertions/evictions/size/queries_disk); `privacy_level` numeric via `as_float`. No bool fields remain in scope (query_logging_enabled deferred).

Brainstormer (Opus) consulted on the three mismatches; recommendations adopted with one correction (the cache-hit source is `/api/stats/summary` already-owned-by-005, confirmed via live recon + reading stats_summary.py, NOT a new retraction).

planner-lite is sufficient at Build (clone of the 007 two-endpoint shape).

### Refinement Notes

**3a (unit/fixtures):** 16-case unit suite (was 15 + a regression test added during this Refinement), 100% branch, full two-endpoint matrix.

**3b (live, prod stack) — caught a REAL bug.** First 3b run: 7/9 PASS but FTL metrics (uptime/cpu/memory/privacy_level/dnsmasq cache insertions/evictions) were ABSENT in VM. Root cause (diagnosed via debugger, NOT the e2e-tester's first guess): Pi-hole v6 `/api/info/ftl` returns `{"ftl": {...}, "took": ...}` — fields nested under a top-level `"ftl"` key. `_emit_ftl` read them off the TOP level (`payload.get("uptime")` etc.) → all None → silently skipped. `gravity.py` reads `payload["ftl"][...]` correctly, which is why gravity worked and exposed the inconsistency. The bug passed Build because the test fixture `_FTL_PAYLOAD` was mis-shaped (top-level) to match the buggy code — collector + fixture wrong in the same direction.

Fix: `_emit_ftl` now descends into `payload["ftl"]` (guard: non-dict `ftl` → sub-skip, mirroring gravity); all 4 ftl fixtures re-nested under `"ftl"`; added regression test `test_ftl_payload_missing_ftl_key` (valid dict, no `"ftl"` key → 0 ftl metrics, ok stays True). `make verify` re-run PASS, 100% branch.

Re-validated 3b live after the fix: **10/10 PASS.** All 6 FTL metrics now present with sane values matching the live Pi-hole (privacy_level exact, dnsmasq_cache_insertions VM 395876 vs live 395914 = timing lag). Regression metrics (db_size ~1.1GB, db_queries_total ~11.6M = queries_disk, gravity_domains 5.86M, api_took both endpoints) all unaffected.

**Pre-existing flaky test noted:** `tests/test_scheduler.py::test_process_run_kind` failed once in a full `make verify` run, passed in isolation and on re-run — order-dependent flake, unrelated to this stage (the handoff-noted scheduler hash-decorrelation flake). Tracked as a regression item; not fixed here (unrelated area, intermittent, real fix = substantial test-isolation work).

**Cross-cutting client gap discovered (tracked, NOT fixed here):** `PiholeRestClient._get()` only maps HTTP 4xx/5xx to `PiholeError`; it does NOT detect a 200 response carrying an `{"error": {...}}` body. If Pi-hole ever returns 200-with-error-envelope (e.g. a session expiry returning 200), ALL 5 pihole collectors would treat it as success and silently emit nothing. This is a client-level hardening with a 5-collector blast radius — recorded as a deliverable note on STAGE-006-018 (which owns client error-handling work) + a regression item. NOT bundled into this collector-local fix.
