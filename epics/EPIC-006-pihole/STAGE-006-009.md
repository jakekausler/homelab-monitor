# STAGE-006-009: FTL health + DB collector

## Status: Not Started

## Overview

Poll `/api/info/ftl` + `/api/info/database` and emit FTL process health + long-term-DB metrics. Feeds
Grafana row 7 and the `PiholeDbTooLarge` alert. Note: host CPU/mem/temps from `/api/info/system` +
`/api/info/sensors` are deliberately SKIPPED (they duplicate the node-level + EPIC-005A-009 collectors,
since Pi-hole is on the monitor's own host) — this collector covers FTL-process and DB specifics only.

**Vertical slice:** `homelab_pihole_ftl_uptime_seconds` / `_ftl_cpu_percent` / `_ftl_memory_percent` /
`_db_size_bytes` / `_db_queries_total` / `_privacy_level` / `_query_logging_enabled` (+ dnsmasq cache
counters) in VM.

**Spec references:** EPIC-006 "Metric families" (FTL health; SKIP host system/sensors). Alert:
`PiholeDbTooLarge` (the `/tmp`-full-wipes-config risk research flagged). The live DB is ~1.05 GB.

## Concrete deliverables

### 1. FTL process

From `/api/info/ftl` (~60s): `homelab_pihole_ftl_uptime_seconds`, `_ftl_cpu_percent`, `_ftl_memory_percent`,
`_privacy_level`, `_query_logging_enabled` (1/0). Also the dnsmasq cache counters present in the FTL
response (cache insertions/evictions/hits/misses) → `homelab_pihole_dnsmasq_cache_*`.

### 2. Database

From `/api/info/database` (~5-15min — it's slow-changing): `homelab_pihole_db_size_bytes`,
`homelab_pihole_db_queries_total` (on-disk row count). These drive `PiholeDbTooLarge`.
