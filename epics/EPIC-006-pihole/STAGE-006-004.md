# STAGE-006-004: Per-client cardinality + loopback-attribution helper

## Status: Not Started

## Overview

Build the shared helper the per-client collector (006-012) and the unified view (006-027) depend on:
(1) apply the reusable cardinality cap (from EPIC-005 STAGE-005-004) to per-client/top-N series so client
data can't explode the metric store, and (2) implement the **loopback-attribution** rules — Pi-hole sees
the host's own DNS as `127.0.0.1`/`::`/`pi.hole`, which must be KEPT, tagged by origin, and attributed to
the host device `192.168.2.148`.

**Vertical slice:** a helper that, given Pi-hole client entries, returns capped+tagged client series:
loopback clients get `client_kind="local"` (host-app DNS) or `client_kind="resolver_self"`
(Pi-hole→unbound internal) and are stamped with the configured `pihole_host_lan_ip`; LAN clients get
`client_kind="lan"` keyed by IP; the top-N cap is applied with stable eviction. Unit-tested with fixtures
covering loopback, resolver-self, and ordinary LAN clients.

**Spec references:** EPIC-006 "Client-join contract" (load-bearing) + "Privacy tiers" (Tier 2 capped).
EPIC-005 STAGE-005-004 (the reusable cardinality cap this consumes). EPIC-006 "Credentials & config"
(`pihole_host_lan_ip`).

## Cross-cutting standards

- Consume EPIC-005's cardinality cap — do NOT reimplement it.
- Loopback is KEPT, never dropped; attributed to the host `192.168.2.148`; the host is a first-class device.
- Non-goal asserted in tests/docs: Pi-hole cannot split loopback DNS by process/container (only source IP
  is visible) — attribute to the host as a whole.

## Concrete deliverables

### 1. Loopback-attribution tagging

Given a client identifier (IP/name), classify:
- `127.0.0.1` / `::1` / `::` / the `pi.hole` self-client → `client_kind` in {`local`, `resolver_self`}
  (Design defines the exact discriminator — e.g. the `pi.hole`/unbound-upstream client is `resolver_self`,
  everything else loopback is `local`), and stamp `host_lan_ip = pihole_host_lan_ip` (config; default empty
  in public → unattributed, `192.168.2.148` in the user override).
- Everything else → `client_kind="lan"`, keyed by IP (+ resolved name when present, + MAC when Pi-hole
  happens to have it — the ~19% case, as a secondary corroborating field).

### 2. Cardinality cap application

Apply EPIC-005's cap to the per-client + top-domain series (default ~top-50 each, configurable), with the
deterministic stable-sort eviction the cap layer provides. Over-budget emits the cap's over-budget
suggestion event (inherited behavior).

### 3. Join-ready output shape

Emit client series labeled so the unified view (006-027) and Unifi (007) can time-window-join on IP:
`{client_ip, client_name, client_kind, host_lan_ip?, client_mac?}`. Document this as the EPIC-006 side of
the client-join contract.
