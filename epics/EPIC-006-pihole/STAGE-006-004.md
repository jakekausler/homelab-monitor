# STAGE-006-004: Per-client cardinality + loopback-attribution helper

## Status: Complete

## Overview

Build the shared helper the per-client collector (006-012) and the unified view (006-027) depend on:
(1) apply the reusable cardinality cap (from EPIC-005 STAGE-005-004) to per-client/top-N series so client
data can't explode the metric store, and (2) implement the **loopback-attribution** rules — Pi-hole sees
the host's own DNS as `127.0.0.1`/`::`/`pi.hole`, which must be KEPT, tagged by origin, and attributed to
the host device `192.168.2.148`.

**Vertical slice:** a helper that, given Pi-hole client entries, returns capped+tagged client series:
loopback clients get `client_kind="local"` (host-app DNS) or `client_kind="resolver_self"`
(Pi-hole→unbound internal) and are stamped with the configured `pihole_host_lan_ip`; LAN clients get
`client_kind="lan"` keyed by IP; the top-N cap is applied with stable eviction. Unit-tested with fixtures
covering loopback, resolver-self, and ordinary LAN clients.

**Spec references:** EPIC-006 "Client-join contract" (load-bearing) + "Privacy tiers" (Tier 2 capped).
EPIC-005 STAGE-005-004 (the reusable cardinality cap this consumes). EPIC-006 "Credentials & config"
(`pihole_host_lan_ip`).

## Cross-cutting standards

- Consume EPIC-005's cardinality cap — do NOT reimplement it.
- Loopback is KEPT, never dropped; attributed to the host `192.168.2.148`; the host is a first-class device.
- Non-goal asserted in tests/docs: Pi-hole cannot split loopback DNS by process/container (only source IP
  is visible) — attribute to the host as a whole.

## Concrete deliverables

### 1. Loopback-attribution tagging

Given a client identifier (IP/name), classify:
- `127.0.0.1` / `::1` / `::` / the `pi.hole` self-client → `client_kind` in {`local`, `resolver_self`}
  (Design defines the exact discriminator — e.g. the `pi.hole`/unbound-upstream client is `resolver_self`,
  everything else loopback is `local`), and stamp `host_lan_ip = pihole_host_lan_ip` (config; default empty
  in public → unattributed, `192.168.2.148` in the user override).
- Everything else → `client_kind="lan"`, keyed by IP (+ resolved name when present, + MAC when Pi-hole
  happens to have it — the ~19% case, as a secondary corroborating field).

### 2. Cardinality cap application

Apply EPIC-005's cap to the per-client + top-domain series (default ~top-50 each, configurable), with the
deterministic stable-sort eviction the cap layer provides. Over-budget emits the cap's over-budget
suggestion event (inherited behavior).

### 3. Join-ready output shape

Emit client series labeled so the unified view (006-027) and Unifi (007) can time-window-join on IP:
`{client_ip, client_name, client_kind, host_lan_ip?, client_mac?}`. Document this as the EPIC-006 side of
the client-join contract.

## Design Phase

**Status**: [x] Design

### Design Notes

**Selected approach: Option A — kernel-resident pure classifier with structurally cap-exempt loopback.**

**Decision A — Module location:** New file `apps/monitor/homelab_monitor/kernel/pihole/clients.py`. Rationale: the client-join contract is consumed by THREE places — 006-012 (pihole bundle collector), 006-027 (cross-epic pihole+unifi unified view), and EPIC-007 (Unifi). Two of those are OUTSIDE the pihole plugin bundle, so the contract must live in the shared kernel layer, mirroring the existing `kernel/unifi/identity.py` (host-attribution precedent already lives in kernel, not in a plugin bundle). The `unifi/_parsing.py` / `ha/_shared.py` precedent does NOT apply — those exist because they need `CollectorContext`; this classifier is pure (no I/O, no context), which is also why it is trivially 100%-branch-coverable with fixtures.

**Decision B — client_kind taxonomy:** `ClientKind = Literal["local", "resolver_self", "lan", "unattributed"]` (string-union, matching kernel house style; no Enum). Deterministic classification on `(ip, name)`, evaluated in order:
1. Normalize: strip/lowercase name; canonicalize IP string.
2. Loopback branch — IP in {"127.0.0.1", "::1", "::"} OR name == "pi.hole":
   - `resolver_self` if hostname matches the resolver self-identity set (default `{"pi.hole", "localhost"}`); else `local`. (Name is the only deterministic signal Pi-hole exposes; unbound shares the `pihole-unbound` container so IP alone cannot separate resolver-self from host-app DNS — the card explicitly accepts host-granularity attribution. Best-effort heuristic; falls back to `local` when no resolver-name match.)
   - host_lan_ip stamping: if `pihole_host_lan_ip` empty → `client_kind` stays the loopback kind but host_lan_ip is NOT stamped and the kind is recorded as `unattributed` (loopback KEPT, just unattributed); if non-empty → stamp `host_lan_ip = pihole_host_lan_ip`.
3. Else → `lan`, keyed by IP (+ name + MAC when available).
Branch set for 100% coverage: loopback-by-IP / loopback-by-name(pi.hole) / resolver_self-vs-local / empty-host-ip→unattributed / non-empty→stamped / lan.

**Decision C — Config:** Add `host_lan_ip: str = ""` to `PiholeConfig` in `kernel/config.py`, env `HOMELAB_MONITOR_PIHOLE_HOST_LAN_IP`, EMPTY default (per the card: empty → unattributed loopback in the public release; the host user overrides to `192.168.2.148`). Do NOT reuse `UnifiConfig.host_lan_ip` — its non-empty default contradicts the card and would couple Pi-hole behavior to whether Unifi is configured.

**Decision D — Output shape + cap interaction:** Classify-THEN-cap, with loopback STRUCTURALLY cap-exempt (partitioned out before the capper, so loopback/host clients can NEVER be evicted). Return frozen slotted dataclasses:
- `ClassifiedClient(client_ip: str, client_name: str, client_kind: ClientKind, host_lan_ip: str | None, client_mac: str | None)`
- A result wrapper exposing kept clients (exempt loopback + capped lan survivors) and the dropped (lan) count.
Mechanism: classify all → partition loopback from lan → `CardinalityCapper(cap).apply()` on ONLY the lan candidates → concatenate exempt-loopback + lan-survivors. Reuses the existing `kernel/metrics/cardinality.py` `CardinalityCapper` unchanged. Top-domain series capped via the same `CardinalityCapper` (no exemption — domains have no loopback concept). Cap value sourced from `load_cardinality_caps_config().cap_for(<family>)`; default-50 per the card is configurable via the existing per-family cap mechanism (add a pihole family entry if needed at Build).

**Build-phase note to confirm:** document the `resolver_self` vs `local` name heuristic and its `local` fallback in the module docstring; it is best-effort (Pi-hole exposes only source IP + optional hostname, unbound shares the container).

**No migration, no openapi/frontend change** (pure helper). Refinement is 3a fixtures-only (006-012 does the live 3b).
