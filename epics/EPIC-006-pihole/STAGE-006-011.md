## Overview

Poll `/api/info/version` and emit update-available flags per component (core/web/ftl/docker) by diffing
each component's local vs remote version. Feeds Grafana row 7 + the panel version widget + the

## Concrete deliverables

### 1. Collector

Poll `/api/info/version` (~1h). For each component (core/web/ftl/docker), compare `local.version` vs
`remote.version`; emit `homelab_pihole_update_available{component}` = 1 when they differ, 0 when equal.
Optionally also emit the current version strings as an info-gauge (`homelab_pihole_version_info{component,
version}` value 1) for the panel/dashboard, bounded cardinality (4 components).

## Design Phase

**Status**: [x] Design

### Design Notes

**Single-endpoint collector** (`/api/info/version`), mirroring `blocking.py`'s structure. Brainstormer (Opus) consulted on 4 decisions; all recommendations adopted.

Live recon (Pi-hole v6.6.2, this session) confirmed the card's contract and revealed ONE implementation gotcha (shape divergence, not a contract change):
- `/api/info/version` returns `{"version": {<component>: {...}}, "took": float}`.
- **CRITICAL shape divergence:** core/web/ftl have `version.<c>.local` and `version.<c>.remote` as OBJECTS each with a `.version` string key (+ branch/hash/date noise). But `docker` has `version.docker.local` and `version.docker.remote` as BARE STRINGS (no `.version` sub-key). A uniform `version[c].local.version` accessor BREAKS for docker.
- Live values: core `v6.4.2`==`v6.4.2` (no update), web `v6.5` != `v6.5.1` (UPDATE), ftl `v6.6.2`==`v6.6.2` (no update), docker `2026.05.0` != `2026.06.0` (UPDATE). All 4 present. `v`-prefix on core/web/ftl, not docker. web.local is 2-segment (`v6.5`) vs remote 3-segment (`v6.5.1`).

Metric contract (final â€” matches card):
- `homelab_pihole_update_available{component}` (gauge 1/0) = `1.0` when local != remote (plain string inequality â€” NO semver; the v-prefix + segment-count differences make semver fragile, and the card spec is literally "1 when they differ"). Emitted ONLY when BOTH local and remote are present non-empty strings; if either is missing/null/empty â†’ emit NO series for that component (absence = "unknown", which honestly does not fire the `PiholeUpdateAvailable` alert â€” never emit a false `0` that would claim "up to date").
- `homelab_pihole_version_info{component, version}` (info-gauge, value always `1.0`, the LOCAL/installed version string as the `version` label) â€” emitted whenever `local` is present (decoupled from remote, so the dashboard shows the installed version even if remote hasn't been fetched). Local-only (matches the card's exact `{component, version}` label set); the remote/latest string is fetched at panel render-time (like STAGE-010's message text), NOT a metric.
- `homelab_pihole_api_took_seconds{endpoint="info/version"}` per run.

Design decisions (brainstormer-recommended, adopted):
1. Shape handling: a `_extract_versions(name, component_obj) -> tuple[str | None, str | None]` helper that branches on RUNTIME SHAPE (`isinstance(local, dict)` â†’ read `["version"]`; `isinstance(local, str)` â†’ use the bare string directly) â€” data-shape-driven, not component-name-driven, so docker isn't special-cased by name and a future bare-string component still parses. pyright-clean isinstance narrowing, no Any.
2. Comparison + null edge: string `!=`; skip `update_available` when either version missing (honest "unknown"); emit `version_info` whenever local present.
3. info-gauge: LOCAL version only (card's exact label set; remote available at render-time â€” YAGNI on a separate remote metric).
4. Iteration: iterate the keys actually PRESENT in `payload["version"]` (present-only convention, consistent with STAGE-010 group-by-present and stats_summary enum iteration â€” handles an absent docker on a bare-metal Pi-hole and any future 5th component gracefully). Guard: `version_obj = payload.get("version"); if isinstance(version_obj, dict): iterate .items()`.

No cardinality cap (4 components Ã— slowly-rolling version labels = bounded; identical to the uncapped Unifi `firmware_info` precedent; the version label only churns on actual upgrades). Skip `branch`/`hash`/`date` noise (YAGNI; none in the contract). Info-gauge precedent: Unifi `device.py` already emits `write_gauge(metric, 1.0, {"version": <str>})`.

Acceptance fixtures (exercise both shapes + both outcomes): web `v6.5`/`v6.5.1` (object shape, update=1), docker `2026.05.0`/`2026.06.0` (bare-string shape, update=1), core `v6.4.2`/`v6.4.2` (object shape, update=0), ftl match (update=0), plus a missing-remote component (update_available skipped, version_info still emitted) and an absent-component case.

planner-lite is sufficient at Build (single-endpoint clone + the shape-branching extractor + info-gauge).

**Operational observation (tracked, NOT a defect):** the version collector has a 3600s (1h) interval with the scheduler's randomized initial offset (`hash(name) % 3600`), so it does NOT run on scheduler startup â€” the version/update metrics are absent for up to ~1h after a monitor restart until the first scheduled run. The two sibling slow update-checkers (`image_update_checker`, `local_build_update_checker`) DO run on startup, but via explicit per-collector `await_immediate_run()` blocks hardcoded in `kernel/api/lifespan.py` (a cross-cutting ~15-line block per collector, NOT a one-line ClassVar opt-in). Adding a startup-run for `pihole_version` (and conceptually the other slow pihole collectors) is therefore a cross-cutting lifespan change deliberately OUT OF SCOPE for this stage (the card specified "1h cadence" and that is delivered correctly).
