# STAGE-006-008: Blocking-state collector

## Status: Complete

## Overview

Poll `/api/dns/blocking` and emit the blocking-enabled state + any temporary-disable timer. This is the
data behind the "someone hit disable and walked away" alerts and the panel's live auto-re-enable
countdown. Small but high-value (with the user's forced-DNS topology, blocking-off = real protection loss).

**Vertical slice:** `homelab_pihole_blocking_enabled` (1/0) + `_blocking_timer_seconds` (remaining, or
absent when permanently enabled/disabled) in VM; reflects the live state.

**Spec references:** EPIC-006 "Metric families" (Blocking state). Alerts: `PiholeBlockingDisabled`
(>5m, warning), `PiholeBlockingDisabledIndefinitely` (no timer, >15m, warning).

## Design Phase

**Status**: [x] Design

### Design Notes

**Selected approach: Option A (minimal — two contract metrics + api_took, fail-closed enum mapping).**

Live recon of `/api/dns/blocking` (Pi-hole v6.6.2, captured this session) confirmed the exact shape:
`{"blocking": <string-enum>, "timer": <number|null>, "took": <float>}`. `blocking` is a STRING enum (`"enabled"`/`"disabled"`; FTL may also report transitional `"failed"`/`"unknown"`), NOT a bool. `timer` is `null` when no temporary disable is active. Current live state: `enabled`, timer `null`. The card's named endpoint and metric contract were correct.

Encoding decisions:
- `homelab_pihole_blocking_enabled` = `1.0` iff `blocking == "enabled"`; `0.0` for `disabled`/`failed`/`unknown`/any unrecognized string (fail-closed — anything other than "enabled" means "not protecting", the safe alerting default).
- `homelab_pihole_blocking_timer_seconds` emitted ONLY when `timer` is a non-null number (narrowed via `_parsing.as_float`); OMITTED entirely when null/missing — never emit `0` (a `0` would read as "0s remaining"). When no timer is active the series simply has no sample and goes stale.
- `homelab_pihole_api_took_seconds{endpoint="dns/blocking"}` emitted every run (standard).

Brainstormer (Opus) considered three options; A selected (recommendation adopted). Option B (a bounded one-hot `homelab_pihole_blocking_state{state}` exposing the raw enum) was DEFERRED as speculative/YAGNI — both 016 alerts (`PiholeBlockingDisabled`, `PiholeBlockingDisabledIndefinitely`) are fully satisfiable from `_blocking_enabled` + presence/absence of `_blocking_timer_seconds` (confirmed). Option B is a non-breaking ADDITIVE change if a later dashboard genuinely needs to distinguish `failed` vs `disabled`; if added it MUST fold unrecognized values into `state="unknown"` (never raw-label passthrough — that breaks the bounded-cardinality guarantee). This deferral has no tracked owner stage because it is a hypothetical-future enhancement, not a dropped card deliverable — the card's full metric contract IS delivered by Option A.

Collector shape: `PiholeBlockingCollector`, `name="pihole_blocking"`, 30s interval / 15s timeout, `concurrency_group="pihole"`, single endpoint via `ctx.pihole.dns_blocking()`. Mirrors STAGE-006-006 (`upstreams.py`): object-local → isinstance → cast payload narrowing, reuse `_parsing.py`, literal-name contract test, 100% branch coverage of error/malformed/unknown-enum paths.

Cross-stage contract note recorded on the STAGE-006-016 card: `PiholeBlockingDisabledIndefinitely` must use `absent(homelab_pihole_blocking_timer_seconds)` whose truth keys off VM series staleness (~5m default), so its `for:` (15m) must exceed the staleness window to avoid a just-expired timer briefly reading as "indefinite".
