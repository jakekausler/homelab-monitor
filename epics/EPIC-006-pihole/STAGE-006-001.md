# STAGE-006-001: Pi-hole v6 client + RO/RW app-password secrets + lifespan wiring + vector service-label pin

## Status: Not Started

## Overview

The first foundation stage of the Pi-hole integration bundle. Build a real Pi-hole **v6** REST client
(the v5 PHP `api.php` model is gone — v6 uses session auth) and wire it into the collector context so
every Wave-B/C collector can reach Pi-hole. Establish the two-credential, least-privilege model
(read-only app password for collectors, write-scoped app password for the later control actions) and
pin the exact vector `service` label for the `pihole-unbound` container's docker-stdout (so the Wave-D
log rules and the panel's `<LogViewer>` target the right stream).

**Vertical slice:** after this stage a smoke test (and the collector status API) shows the monitor
authenticating to Pi-hole v6 (`POST /api/auth` → SID) with the RO app password from the secret store
and successfully calling `GET /api/info/version`; the resolved vector service label for `pihole-unbound`
is documented and asserted.

**Spec references:** master spec §2 (integration_bundle), §5.2 (`CollectorContext` client injection
pattern — `ctx.ha` precedent), §6.2 (`homelab_pihole_*`). EPIC-006 "Verified deployment reality" +
"Credentials & config". EPIC-005 STAGE-005-001 is the structural precedent (an integration's API client
constructed, credentialed via the secret store, injected into context).

## Cross-cutting standards (from EPIC-006 cross-stage acceptance criteria)

- App passwords NEVER logged (assert in tests).
- ONE reused session (`X-FTL-SID` header — the space after the colon is mandatory); re-auth only on 401;
  back off on 429; poll ≤5min keeps the session alive. Do NOT auth-per-scrape (intermittent app-password
  bug + session-slot limit).
- Pi-hole 5xx/timeout never propagates as our 5xx — the client surfaces a typed error the collector turns
  into a failed run (failure budget takes over).
- All internal timestamps UTC.

## Concrete deliverables

### 1. Real Pi-hole v6 client

A new `PiholeClient` (Design picks the exact module path within the integration bundle area, e.g.
`apps/monitor/homelab_monitor/kernel/pihole/client.py` or under the bundle — keep any Protocol in
`io.py`, the impl separate; mirror how the HA client was structured in 005-001). Uses the shared
`httpx.AsyncClient` from the context. Responsibilities:
- **Session management:** `POST /api/auth {"password": <ro-app-pw>}` → store `session.sid`; attach
  `X-FTL-SID: <sid>` to every subsequent GET; on a 401 re-auth once; on 429 back off. `DELETE /api/auth`
  on shutdown to free the session slot. Track session validity; the reuse model keeps it alive at ≤5min
  polling.
- **Typed GET helpers** for the endpoints Wave B/C need (return parsed models, not raw JSON):
  `info_version()`, `info_ftl()`, `info_database()`, `info_messages()`, `info_system()` (available but
  unused — host metrics skipped), `stats_summary()`, `stats_upstreams()`, `stats_query_types()`,
  `stats_top_clients()`, `stats_top_domains()`, `stats_recent_blocked()`, `dns_blocking()`, `lists()`,
  `network_devices()`, `queries(params)`. (Not all are consumed in THIS stage — this establishes the
  client surface; collectors land in Wave B/C.)
- **`took` capture:** expose each response's `took` float so the API-latency metric collector can read it.
- Typed error mapping (auth failure, 429, 5xx, timeout, JSON-shape mismatch).

### 2. Two-credential secret + config wiring

- Two secret-store keys: `pihole_api_password_ro` (collectors) and `pihole_api_password_rw` (Wave-E write
  actions). This stage wires + smoke-tests the RO password only; the RW one is stored but first exercised
  in 006-018.
- `pihole_url` config (default `http://localhost:8080`), read via the established `getattr(ctx.config, ...)`
  pattern. Inject a live `PiholeClient` into the collector context (the `ctx.ha=None`→real precedent).
- Document the exact Pi-hole UI steps to generate the two app passwords (Settings → API → app passwords)
  + the two secret names, in the integration README. Do NOT touch the user's live main password.

### 3. Pin the vector service label

Determine + document the exact vector `service` label under which `pihole-unbound` container stdout lands
in VictoriaLogs (the way EPIC-005 pinned `homeassistant`). Query VL live in Design (`docker logs` shows
FTL lines on stdout → already ingested). Record the label so 006-017 (log rules) and 006-024 (LogViewer)
use the correct scope. Likely `pihole-unbound` or `pihole` — verify, do not assume.
