## Design Phase

**Status**: [x] Design

### Design Notes

**Selected approach:** Single `deploy/grafana/dashboards/pihole.json` dashboard-as-code, 7 collapsible rows over AGGREGATE `homelab_pihole_*` / `homelab_unbound_*` metrics, embedded as a Pi-hole tab in the Metrics screen (`MetricsLayout.tsx`). Mirrors `home-assistant.json` / `homelab-unifi.json` (schemaVersion 40, datasource `{type:prometheus, uid:victoriametrics}`, rows as section headers with content panels as peers in the outer `panels` array).

**Live-verified in Design (prod rig @ 29090):** all sampled `homelab_pihole_*`/`homelab_unbound_*` metrics return real series (up=1, percent_blocked=47.65, queries_total=127084, gravity_domains=5.8M, unbound cache_hit_ratio, recursion quantiles). `/api/grafana/` proxy returns 200; existing HA/Unifi embeds use `/api/grafana/d/{uid}/{uid}?kiosk`. Grafana queries VM directly server-side (`victoriametrics:8428`); NO backend route change needed.

**Key decisions:**
- Row order topâ†’bottom: 1 Health & blocking, 2 Query volume over time, 3 Query composition, 4 Gravity/blocklists, 5 Clients (Tier 2), 6 Unbound, 7 FTL/process & API. Rows 1â€“3 `collapsed:false`; rows 4â€“7 `collapsed:true`.
- **Tables (not barcharts) for composition (row 3), top-N clients/domains (row 5), adlist status (row 4), RCODEs + API latency (rows 6â€“7).** Deliberate: avoids the documented 4-attempt barchart-axis pitfall (memory `project_grafana_barchart_category_axis`); 16-category by-type reads better as a sorted table; tables are a proven repo pattern (`homelab-unifi.json`).
- `rate(...[$__rate_interval])` for over-time counter panels (queries/blocked/cached/forwarded, unbound hits/misses); raw values for headline stats/totals.
- Unbound recursion latency filters `{quantile=~"0.5|0.95|0.99"}` â€” the metric also carries STRING quantile values `avg`/`median` that would poison the panel.
- Adlist status + answer-RCODE rendered value-in-label: the `_adlist_status` value is always 1, meaning lives in the `status` label (3/5 adlists currently download_failed/parse_failed) â†’ table with a color-background Status column.
- `percent_blocked` left GREEN-throughout (no yellow/red thresholds): a high block-rate is normal/desirable for Pi-hole; coloring it "bad" would mislead. (Selector override of the brainstormer's 60/90 thresholds.)
- Self-observability stats (`query_feed_shipped_total` rate, `query_feed_cap_hit_total`) included in row 7 per the "plugins observe themselves" convention.

**Resolve in Build (live `curl` first, not design-time guesses):**
- Collapse mechanics: confirm `collapsed:true` rows hide their peer (outer-array) content panels in schemaVersion 40 (existing dashboards use `collapsed:false` only). Test one collapsed row; fall back to all-`collapsed:false` if peers don't hide.
- Row 5 client/domain metric shape: verify whether `homelab_pihole_client_queries`/`client_blocked`/`top_*_domain` carry a per-client/per-domain label (â†’ tables) or are scalar top-N gauges (â†’ name-stat panels).
- Adlist label names: confirm exact `{list, address, status}` spelling; adjust organize transform keys.

**Frontend embed (pure-frontend, no backend change):** create `apps/ui/src/routes/metrics/MetricsPiholeTab.tsx` (mirrors `MetricsHomeAssistantTab.tsx`, src `/api/grafana/d/pihole/pihole?kiosk`); add `{ path:'/metrics/pihole', label:'Pi-hole' }` to the `TABS` array in `MetricsLayout.tsx`; register the route in `router.tsx` (new `metricsPiholeRoute` + add to `metricsLayoutRoute.addChildren`); add colocated `MetricsPiholeTab.test.tsx` mirroring `MetricsHomeAssistantTab.test.tsx`.

**Deferrals:** None. Query-feed roll-up Grafana panels belong to EPIC-020 STAGE-020-009; per-client rich drill-down belongs to the uncounted STAGE-006-027.
