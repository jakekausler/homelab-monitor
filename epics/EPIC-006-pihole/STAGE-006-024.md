## Overview

`service` label (the FTL log lines: gravity runs, rate-limits, errors). And — when Tier 3 is enabled —

the same viewer.

## Design Phase

### Design Notes

**Approach selected** (coordinator selected from brainstormer options; no user prompt per pre-authorization). Contracts verified LIVE against the prod rig (29090) before design.

**Decision 1 — Tier-3 per-query-feed toggle: DEFERRED to STAGE-006-025 (selected Option A).**
024 builds ONLY the docker-stdout FTL `<LogViewer>` (deliverable #1), which is fully data-ready now (`service:"pihole-unbound"`, 237 live VL hits, real FTL content). The Tier-3 query-feed toggle (deliverable #2) depends ENTIRELY on STAGE-006-025 (the NEXT, higher-numbered stage, Not Started) for three things that DO NOT EXIST yet: (a) the `pihole_stream_query_feed` config flag (defined in 025), (b) the query-feed VictoriaLogs stream — whose exact label 025's card leaves as "`service:"pihole-queries"` or similar — Design picks the label" (NOT finalized), and (c) exposing the flag's on/off state to the frontend (in NEITHER stage's current scope). Per Sequential Dependency Ordering, 024 must NOT depend on later-numbered work to function, and building the toggle now would be dead/untestable UI against an unfinalized stream label on every host until 025 ships. So the toggle is tracked into its true owner (025) — see TASK 2. 024 builds the HOST component `PiholeLogsTab.tsx` (FTL viewer); 025 wires the toggle INTO it once the stream + flag exist.

**Decision 2 — FTL docker-stdout LogViewer (ships now).** Near-verbatim port of the `HomeAssistantLogsTab.tsx` precedent (STAGE-005-025): `servicesCsv = identitiesToServicesCsv([{ source_type:'docker', service:'pihole-unbound' }])`; `useLogsQuery(expr, startIso, endIso, servicesCsv)` (infinite query) → an adapter mapping to `UseLogsResult` → `<LogViewer useLogs={...} headerSlot timezone wrap fillHeight emptyStateCopy unavailableCopy />`. Match HA's time-window span, timezone, and wrap defaults (Build reads HA's literal values). Single-service (like HA). `fillHeight` so the viewer fills the Logs-tab area.

**Decision 2b — errors-only toggle uses TEXT-MATCH, NOT `severity:` (verified live).** HA's errors-only toggle uses `severity:error OR severity:warn`, but a LIVE check proved FTL docker-stdout lines all default to `severity:"info"` — Vector's `docker_severity_extract` regex does not parse FTL's `[57/T70] WARNING:` thread-ID format, so `severity:error OR severity:warn` returns EMPTY on real pihole-unbound data. Therefore 024's errors-only toggle uses a LogsQL TEXT-MATCH expression (e.g. `WARNING OR ERROR OR CRITICAL`) ↔ `*` (all), which matches the real FTL message text. (Fixing Vector's severity regex is out of scope for this frontend stage — not pursued.) Build verifies the exact text-match LogsQL syntax against the live stream during pre-flight.

**Backend contract:** no new backend in 024. Consumes the existing `GET /api/logs/query` (params `expr`, `start`, `end`, `services`) — same endpoint HA uses. The query-feed stream + flag are 025's job.
