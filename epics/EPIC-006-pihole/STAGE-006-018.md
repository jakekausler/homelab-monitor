# STAGE-006-018: Pi-hole write endpoints (blocking enable/disable, gravity-update)

## Status: Complete

## Overview

The first WRITE actions of the epic: backend endpoints that toggle Pi-hole blocking and trigger a gravity
update, using the write-scoped app password, gated by confirm-on-destructive + audit-log. These power the
panel's actionable controls (006-022). This is the one place the integration stops being purely
observational — handled with the spec's confirm + audit pattern.

**Vertical slice:** `POST /api/integrations/pihole/blocking {enable|disable, timer?}` and
`POST /api/integrations/pihole/gravity/update` work against the live Pi-hole (using `pihole_api_password_rw`),
each requiring the confirm step and writing an `audit_log` row; an e2e test exercises enable→disable→enable
and a gravity update.

**Spec references:** spec §7.2 (confirm-on-destructive for state-changing actions; API tokens/scopes),
§3.1 (audit_log). EPIC-006 "Operator panel + write controls" + "Credentials & config" (RW app password).
Q9 decision = write-capable. Reuses the 006-001 client's write path with the RW credential.

This is a cross-cutting client fix with a 5-collector blast radius — implement with care + its own tests. Discovered when STAGE-006-009's e2e probe transiently saw a 200-unauthorized body (the persistent 009 bug turned out to be a separate payload-path issue, but this client gap is real and latent).

## Concrete deliverables

### 1. Blocking endpoint

`POST /api/integrations/pihole/blocking` → calls Pi-hole `POST /api/dns/blocking {"blocking":bool,
"timer":sec?}` with the RW session. Body: enable/disable + optional timer. Confirm + audit.

### 2. Gravity-update endpoint

`POST /api/integrations/pihole/gravity/update` → triggers a gravity rebuild (the v6 API action for
`pihole -g`; Design confirms the exact endpoint/action). Confirm + audit. Note: re-running won't fix a
permanently-dead adlist URL — it retries transient failures (document this).

### 3. Audit + scopes

Each action writes `audit_log` (who/what/when/before/after). Scope the endpoints (a write scope) so a
read-only session/token can't invoke them.

## Design Phase

### Design Notes

**Approach selected (coordinator decision, grounded on live Pi-hole v6 OpenAPI + brainstormer Opus):** Scope-disciplined manual operator write endpoints mirroring the existing docker.py `pull-and-restart`/probe-toggle precedent — scoped-auth + confirm-phrase + transactional audit. The card scope is spec §7.2 (confirm-on-destructive) + §3.1 (audit_log); the FULL Claude-auto-fix safety harness (allow-list-per-alert, rate-limit, cooldown, dry-run+ack, kill-switch, homelab-fixer OS user) is a SEPARATE future epic and is explicitly OUT of scope here. Deliberately NOT building an action-state table / background runner / rate-limiter / cooldown / kill-switch.

**Recon (recon-first, verified vs live source + live Pi-hole v6 OpenAPI specs):**
- Live v6 write contracts: blocking = `POST /api/dns/blocking` body `{"blocking": bool, "timer": int|null}` → `{"blocking": "enabled"|"disabled"|..., "timer": float|null, "took"}`; gravity = `POST /api/action/gravity` returns STREAMING `text/plain` (newline-delimited `pihole -g` output, long-running, NOT JSON); config = `GET /api/config` → `config.dns.queryLogging` (boolean).
- Existing precedent: docker.py `pull-and-restart` (confirm_phrase Pydantic field + `require_user_or_token({Scope.DOCKER_WRITE})` + transactional `insert_audit(conn, who, what, before, after, ip)`); audit framework `kernel/db/audit.py` REUSED.
- Client: `PiholeRestClient(password_provider=Callable[[], str|None])`; RO collectors use a provider returning `pihole_api_password_ro`. No write methods exist yet. `PiholeClient` Protocol in `kernel/plugins/io.py` lists GET methods (must stay in lockstep). `_get()` does session-auth (X-FTL-SID) + 401-retry-once; it currently checks HTTP status ONLY (a 200-with-`{"error":{...}}`-body slips through). `pihole_api_password_rw` secret is NOT yet wired in the secrets/config loader (only `_ro` exists) — 018 wires it.

**Design decisions (selected):**
1. RO/RW split: a SECOND long-lived RW `PiholeRestClient` constructed in lifespan (`password_provider` → `pihole_api_password_rw`), injected into the new pihole router via a Request-based provider (docker `_get_compose_action_runner` pattern); own X-FTL-SID session; `aclose()` in lifespan finally; structurally cannot leak into the RO collector path.
2. Surfaces split: `config()` (a GET) added to BOTH the read Protocol (`io.py`) AND the concrete client (a collector consumes it); `set_blocking(*, blocking, timer)` + `gravity_update()` added to the CONCRETE client ONLY (only the router calls them — keeps the read Protocol honest, no write stubs in collector doubles). Factor a private `_request`/`_post` reusing the `_get` re-auth-retry pattern.
3. Gravity: SYNCHRONOUS — `gravity_update()` drains the text/plain stream to completion, tail-parses success/failure markers, returns a `PiholeResponse(payload={"success": bool, "log_tail": [...]}, ...)`. No action-state table (over-build vs card). Generous client timeout (~120s).
4. Confirm: `confirm_phrase: str` literal (mirror docker), required on ALL THREE actions (disable/enable/gravity); 400 on mismatch.
5. API surface: ONE `POST /api/integrations/pihole/blocking` with body `{action: "enable"|"disable", timer: int|null, confirm_phrase}` (translate `action`→`blocking: bool` at the boundary) + `POST /api/integrations/pihole/gravity/update` (body `{confirm_phrase}`). Responses echo the REAL new state Pi-hole reports + an `audit_id` (pass explicit `audit_id=uuid7()` into `insert_audit` and echo it). Add `Scope.PIHOLE_WRITE = "pihole:write"`.
6. Query-logging metric: NEW tiny `pihole_config` collector (mirror `blocking.py`), RO client `ctx.pihole.config()`, emits `homelab_pihole_query_logging_enabled` as a bare fail-closed 1/0 gauge (1.0 only when `config.dns.queryLogging is True`, else 0.0) + `homelab_pihole_api_took_seconds{endpoint="config"}`. Registered in the bundle `__init__.py`.
7. No CLI (matches the docker write precedent; card lists only API + audit + scope).

**Cross-stage deliverable 2 (`_get()` hardening):** restructure `_get` so a 200-response carrying `{"error":{"key":...,"message":...}}` shares the SAME re-auth-retry guard as a real HTTP 401: classify a raw response into ok|needs_reauth|error; `key=="unauthorized"` → reason `auth` + one re-auth+retry (then give up); any other error key → reason `bad_response`. Stay within the existing `PiholeErrorReason` Literal (NO new member). All 11 collectors already branch on `isinstance(result, PiholeError)` → no collector logic change; the new branches each need a dedicated client unit test for the 100% coverage gate.

**Brainstormer used:** yes (Opus) — for the RO/RW split, the Protocol surface boundary, the gravity-streaming approach, the confirm mechanism, the metric-emission home, and the `_get` re-auth restructure.

**Validation plan:** 3a — unit/e2e against a mock Pi-hole write API: confirm enforced, audit row written, scope rejects unprivileged callers, the `_get` error-envelope branches, the gravity tail-parse. 3b — against the LIVE Pi-hole (after `make dev-prod` rebuild for the new routes + RW secret): real enable→disable→enable (toggle BACK to original state — non-destructive), a real gravity update, confirm `audit_log` rows written. CAREFUL: writes against the user's real DNS — leave blocking in its original state.

**3b/regression flags (live-confirm at 3b):** (1) `pihole_api_password_rw` secret wired + a real RW app-password authenticates against live v6; (2) whether RW endpoints require the CSRF token (`PiholeSession.csrf` is parsed on login but never sent — confirm); (3) real gravity duration + exact `pihole -g` success/failure marker lines (the tail-parse) + adequate timeout; (4) the exact `{"error":{"key":...}}` envelope shape + the auth `key` value (`unauthorized`).
