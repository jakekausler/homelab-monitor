# STAGE-006-018: Pi-hole write endpoints (blocking enable/disable, gravity-update)

## Status: Not Started

## Overview

The first WRITE actions of the epic: backend endpoints that toggle Pi-hole blocking and trigger a gravity
update, using the write-scoped app password, gated by confirm-on-destructive + audit-log. These power the
panel's actionable controls (006-022). This is the one place the integration stops being purely
observational — handled with the spec's confirm + audit pattern.

**Vertical slice:** `POST /api/integrations/pihole/blocking {enable|disable, timer?}` and
`POST /api/integrations/pihole/gravity/update` work against the live Pi-hole (using `pihole_api_password_rw`),
each requiring the confirm step and writing an `audit_log` row; an e2e test exercises enable→disable→enable
and a gravity update.

**Spec references:** spec §7.2 (confirm-on-destructive for state-changing actions; API tokens/scopes),
§3.1 (audit_log). EPIC-006 "Operator panel + write controls" + "Credentials & config" (RW app password).
Q9 decision = write-capable. Reuses the 006-001 client's write path with the RW credential.

## Cross-stage deliverable notes (from upstream stages)

- **(deferred from STAGE-006-009 FTL/DB collector)** Add a `homelab_pihole_query_logging_enabled` (1/0) metric. The field is NOT in `/api/info/ftl` (where 009 looked) — it lives in Pi-hole's `/api/config` endpoint (`config.dns.queryLogging` or equivalent). STAGE-006-018 owns the `/api/config` RW surface, so it should add the `config()` GET client helper (alongside the RW config writes) and emit `homelab_pihole_query_logging_enabled` from the read. Confirm the exact config field path via live recon at 018 Design. This metric feeds Grafana row 7 (privacy/logging state).
- **(deferred from STAGE-006-009 Refinement)** Harden `PiholeRestClient._get()` to detect a 200-response whose JSON body is an `{"error": {"key": ..., "message": ...}}` envelope (Pi-hole v6 sometimes returns 200 with an error body instead of 401, e.g. on session expiry). Currently `_get()` only maps HTTP 4xx/5xx to `PiholeError`, so a 200-error-body is treated as a successful response and ALL pihole collectors silently emit nothing. Map a 200-error-body to a `PiholeError` (reason `auth` for `key=="unauthorized"`, else `bad_response`/`api_error`), and for an unauthorized error trigger the existing single re-auth+retry path. This is a cross-cutting client fix with a 5-collector blast radius — implement with care + its own tests. Discovered when STAGE-006-009's e2e probe transiently saw a 200-unauthorized body (the persistent 009 bug turned out to be a separate payload-path issue, but this client gap is real and latent).

## Cross-cutting standards

- `require_user_or_token` (scoped) + confirm-on-destructive + `audit_log` on EVERY write (per spec §7.2).
- Uses the RW app password ONLY here; collectors never touch it. Note the Pi-hole quirk that RO app
  passwords aren't a hard server boundary — the protection is that ONLY these endpoints use RW.
- Toggling blocking briefly changes protection state — the confirm step makes it deliberate; the action is
  audited.

## Concrete deliverables

### 1. Blocking endpoint

`POST /api/integrations/pihole/blocking` → calls Pi-hole `POST /api/dns/blocking {"blocking":bool,
"timer":sec?}` with the RW session. Body: enable/disable + optional timer. Confirm + audit.

### 2. Gravity-update endpoint

`POST /api/integrations/pihole/gravity/update` → triggers a gravity rebuild (the v6 API action for
`pihole -g`; Design confirms the exact endpoint/action). Confirm + audit. Note: re-running won't fix a
permanently-dead adlist URL — it retries transient failures (document this).

### 3. Audit + scopes

Each action writes `audit_log` (who/what/when/before/after). Scope the endpoints (a write scope) so a
read-only session/token can't invoke them.
