## Overview

**CROSS-EPIC ENHANCEMENT (rewritten 2026-06-17 Unifi brainstorm).** EPIC-007 OWNS and BUILDS the client
view — the persistent `unifi_clients` registry, the **Network → Clients tab**, and the per-client **Client
page** (identity + connection + uptime + bandwidth + DPI + lease). This stage does NOT build a separate
"merged view." Instead it **ENHANCES EPIC-007's existing Client page + Clients-tab rows in place** with
Pi-hole DNS behavior, by plugging into the documented **DNS-enrichment extension point** EPIC-007 exposes
(EPIC-007 STAGE-007-022). One view, enhanced — not two views.

**DO NOT BUILD until EPIC-007 has landed** its client view + registry + extension point. The whole-epic
build order is EPIC-017 → EPIC-007 → EPIC-006 → EPIC-008, so by the time this stage runs EPIC-007's Client
page exists and this is a clean additive plug-in.

**Vertical slice (when built):** each client's Client page (and optionally a Clients-tab column) gains a
DNS-behavior section — query volume, block rate, top blocked/permitted domains, recent blocks, per-client
DNSSEC/SERVFAIL — joined time-windowed by IP→MAC onto EPIC-007's registry. The host `192.168.2.148` (a
first-class Client page in EPIC-007) shows its own DNS behavior (loopback-attributed per 006-004).

**Spec references:** EPIC-007 "Build order + client-identity ownership" + STAGE-007-022 (the DNS-enrichment

## Cross-cutting standards (the load-bearing client-join contract)

- **EPIC-007 is the identity authority** (the `unifi_clients` registry, MAC↔current-IP, time-stamped IP↔MAC
  observations). **Pi-hole's canonical key is IP.** Join **time-windowed by IP-at-time-of-observation** —
  NEVER a static IP→device map (DHCP IPs rotate). Pi-hole-supplied MAC (~19%) is a corroborating secondary
  key only.
- **Plug into EPIC-007's extension point — do NOT fork the Client page / Clients table.** This stage adds a
  DNS-behavior section/column via the documented contract; it does not modify EPIC-007's components' internals
  or create a parallel client view.
- **Loopback (`127.0.0.1`/`::`/`pi.hole`) is attributed to the host `192.168.2.148`** (via the 006-004
  `pihole_host_lan_ip` mapping + `client_kind` tags); the host is the first-class Client page EPIC-007 built;
  its DNS behavior renders there.
- **Non-goal:** improving Pi-hole MAC coverage via host-side ARP (EPIC-007/Unifi is the identity source —
  explicitly rejected).

## Concrete deliverables

### 1. DNS-behavior join

A backend that joins Pi-hole per-client DNS series (006-012, keyed by IP, loopback-attributed) onto the
EPIC-007 `unifi_clients` registry, time-windowed by IP via EPIC-007's registry query contract. Produces the
per-client DNS-behavior payload the extension point consumes.

### 2. Client-page / Clients-tab enrichment (via EPIC-007's extension point)

Populate the EPIC-007 Client page DNS-enrichment slot (query volume, block rate, top blocked/permitted
domains, recent blocks, DNSSEC/SERVFAIL) and, if useful, a Clients-tab column. Uses the documented contract;
renders cleanly for clients with no Pi-hole data (e.g., the bypass device).

### 3. IP→name replacement sites (must enrich both in this stage)

- **Grafana `pihole.json` client tables:** The panels "Top talkers (by client)" and "Top blocked per client" currently render `client_ip` only (Pi-hole's `client_name` label is empty for ~all LAN clients). This stage must replace/enrich these two panels' IP columns with the Unifi-join device name (name when available, IP otherwise), matching the enriched Client page.
- **Pi-hole integration page client name-mapping:** The Pi-hole integration page (`PiholeClientsWidget.tsx` and any clients table) must use the Unifi-join names, not just Pi-hole's mostly-empty `client_name` label. Consistent with the Client page enrichment above.

### Design Notes

**Verified-reality recon (source + live prod @ 29090):** EPIC-007's DNS-enrichment extension point is DONE and ready for data — `apps/ui/src/components/clients/ClientDnsSection.tsx` already renders the populated branch (`top_domains`, `blocked_count`, `last_query_at`); `NetworkClientPage.tsx` already mounts `<ClientDnsSection dns={data.dns ?? null}/>`. The backend `GET /api/integrations/unifi/clients/{mac}` returns `UnifiClientDetail` with `dns=None` HARDCODED (~line 1388) — everything else (identity/series/dpi) works live. The typed `DnsEnrichment` contract (OpenAPI + Pydantic) is exactly `{top_domains: string[], blocked_count: int|null, last_query_at: string|null}`.

**Selected approach (decisions A–E, selected by coordinator; D + backfill confirmed with user):**
- **A1 — fill ONLY the existing 3 DnsEnrichment fields.** Respect EPIC-007's deliberately-narrow contract (pure additive plug-in, zero frontend/contract change). Richer per-client fields (query_volume, block_rate, top_permitted/blocked split, recent_blocks, DNSSEC/SERVFAIL) DEFERRED → STAGE-006-028.
- **B1 — per-client data from the `pihole-queries` VL feed.** Feed is HEALTHY/dense (~1.2 records/s). `client_ip` is NOT an indexed VL field (whole record is JSON in `_msg`; only `service`/`source_type` indexed; the proxy rejects LogsQL `unpack_json` pipes with 502). So query `/api/logs/query?expr="<client_ip>"&services=pihole:pihole-queries&start=…&end=…`, parse each line's JSON, POST-FILTER on the parsed `client_ip` (phrase-match is a pre-filter only), and aggregate: `top_domains` (ranked, capped), `blocked_count` (count status=blocked in window), `last_query_at` (max time → ISO). Honest semantics: this is RECENT-window behavior, not lifetime — surface that. VL `client_ip` index promotion DEFERRED → STAGE-006-028.
- **C — add `find_ips_for_mac(mac, window≈24h)` to `unifi_clients_repository.py`** (the reverse of the existing `find_mac_by_ip_at`), querying `unifi_client_observations` for time-windowed IP spans. Join IP-time-windowed (NEVER a static current IP — DHCP rotates); window ≈24h (aligned to VL's practical horizon), as a module constant. Loopback (`127.0.0.1`/`::`/`pi.hole`) is already attributed to the host `192.168.2.148` (006-004); the host's Client page queries its LAN IP as a normal client_ip — verify the host owns a registry/observation entry for its LAN IP (Refinement test).
- **D1 — emit a `unifi_name` label on the per-client Pi-hole metrics** (`homelab_pihole_client_queries`/`_client_blocked`) in `top_clients.py`, so Grafana panels 26/27 can display device names (Grafana queries VM directly and CANNOT reach the SQLite registry, so a metric label is the only path). Resolve the name via a ONCE-PER-RUN `UnifiClientRepo(ctx.db).list_clients()` snapshot → IP→name map (the existing `active_client` pattern; NO per-emit DB read in the scrape path). Then update Grafana `pihole.json` panels 26/27 to surface the `unifi_name` label (prefer name, fall back to client_ip). **Going-forward ONLY — NO backfill** (user-confirmed: VM samples are immutable so existing history cannot gain the label; the ~30-day reconstructed-series backfill was declined as lossy/caveat-heavy). The ~30-day backfill DEFERRED → STAGE-006-028.
- **#3b (Pi-hole integration page) is ALREADY DONE:** `PiholeClientsWidget` + `/api/integrations/pihole/clients` were Unifi-joined in STAGE-006-023 (name resolves Unifi-name > hostname > pihole-name; live names populate: Desktop / Jake's Galaxy / Bedroom Echo). 027 only VERIFIES + documents this — no new code. (The card's #3 note predated 023.)
- **E — Clients-tab DNS column DEFERRED.** Per-row DNS = N registry+VL joins per list load (pathological). Per STAGE-007-022 this is a documented comment-hook only (per-row DNS belongs on the detail page), not a tracked deliverable. No new stage required for it.

**Build surface:** backend = new repo method `find_ips_for_mac`; populate `UnifiClientDetail.dns` in the `/clients/{mac}` endpoint (VL-feed aggregation, honest empty state → `dns=None` when no records); `unifi_name` label in `top_clients.py` (once-per-run registry snapshot). dashboard = `pihole.json` panels 26/27 show `unifi_name`. frontend = NONE (slot already renders). Tests: repo method, the dns-aggregation, the label-emit. No OpenAPI change for the slot (DnsEnrichment already in the schema); openapi MAY change if the endpoint response shape is touched — verify.

**Deferrals (tracked in STAGE-006-028, an uncounted enhancement stage created this Design):** (1) rich DnsEnrichment fields on the Client page (needs a contract+component extension); (2) VL `client_ip`/domain/status indexed-field promotion in `query_feed.py` for exact forensic LogsQL; (3) the ~30-day reconstructed metric backfill of `unifi_name`. The Clients-tab column stays a 007-022 documented comment-hook (no stage).
