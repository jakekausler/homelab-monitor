## Cross-cutting standards (the load-bearing client-join contract)

- **Unifi is the identity authority** (reliable MAC↔current-IP). **Pi-hole's canonical key is IP** (~19% MAC
  coverage). Join **time-windowed by IP-at-time-of-observation** — NEVER a static IP→device map (DHCP IPs
  rotate). Pi-hole-supplied MAC is a corroborating secondary key only.
- **Loopback (`127.0.0.1`/`::`/`pi.hole`) is KEPT and attributed to the host `192.168.2.148`** (via the
  006-004 `pihole_host_lan_ip` mapping + `client_kind` tags); the host is a first-class top-tier device,
  never excluded. `client_kind="resolver_self"` (Pi-hole→unbound internal) is distinguished from
  `client_kind="local"` (host-app DNS) but both attribute to the host.
- **Non-goal:** improving Pi-hole MAC coverage via host-side ARP population (Unifi is the better source —
  explicitly rejected).

## Concrete deliverables (to be finalized in the Unifi brainstorm)

### 1. Merged client model + join

A backend that joins Pi-hole per-client DNS series (006-012, keyed by IP, loopback-attributed) onto Unifi's

### 2. Network → Clients UI

The merged view (one row per device, both data sources). EPIC-007 builds its own Unifi-centric views; this
view is the cross-source merge. Exact UI placement + the 006/007 seam decided in the Unifi brainstorm.

### 3. Stage-number reconciliation

The Unifi brainstorm confirms whether this stays STAGE-006-027, renumbers, or moves into EPIC-007. Update
the epic tables accordingly at that time.
