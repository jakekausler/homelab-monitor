## Design Phase

### Design Notes

**Approach selected** (coordinator selected from brainstormer options; no user prompt per pre-authorization).

Backend contract verified LIVE against the prod rig (29090) before design. All four placeholder sections in `PiholeOverviewTab.tsx` are filled 1:1 (Decision 3 / Option 3A — matches STAGE-006-021/022 one-widget-per-PanelSection precedent).

**Decision 1 — Unbound metrics gap (selected 1A: extend backend this stage).**
The card's Unbound sub-card success criteria list cache-hit ratio, recursion p50/p95, DNSSEC secure/bogus, SERVFAIL rate, extended-stats-enabled. The LIVE `/api/integrations/pihole/unbound` endpoint (STAGE-006-020) returns ONLY 7 fields (cache_hit_ratio, queries_total, cache_hits_total, cache_misses_total, prefetch_total, requestlist_current, extended_stats_enabled) — it does NOT include p50/p95, DNSSEC secure/bogus, or SERVFAIL. Those three ARE collected as VictoriaMetrics series (`homelab_unbound_recursion_time_seconds{quantile}`, `homelab_unbound_answer_secure_total`, `homelab_unbound_answer_bogus_total`, `homelab_unbound_answer_rcode{rcode="SERVFAIL"}`), emitted only when `extended_stats_enabled` (TRUE on this host). This codebase has NO frontend-side PromQL path (verified) — all VM-derived values are pre-computed by backend REST endpoints via `vm_instant_query()`. Therefore surfacing the three metrics requires extending the `/unbound` backend endpoint with ~4 additional `vm_instant_query()` calls + 4 new response fields. Selected because: (a) 023 is already not pure-frontend (it must add a container-lifecycle mutation hook integrating a real write contract), so "keep it pure frontend" is moot; (b) the metrics are already collected — the backend work is small and additive on an endpoint that already uses the helper; (c) avoids a second Design→Finalize cycle re-touching the same endpoint + sub-card. **Consequence:** the impl commit now touches a backend route/model → the openapi-export + generate-ui-types scripts run (stage `packages/shared-types/openapi.json` + `apps/ui/src/api/schema.ts`), and Build must satisfy the 100% kernel branch-coverage gate for the new fields. Honest empty state when `extended_stats_enabled` is false: hide the three extended metrics and show an "Unbound extended stats disabled" note (open-source-safe). Nothing is deferred under 1A.

**Decision 2 — Clients widget + per-client drill-down (selected 2A: top-talkers table now; rich drill-down deferred to STAGE-006-027).**
The verified contract provides ONLY two aggregate `/clients` lists (blocked=false total + blocked=true blocked). There is NO per-client recent-queries / top-domains endpoint, and `/recent-blocked` has no client filter. The card's "recent queries / top domains, live-queried per client" drill-down has no backing data and would require NEW backend work (per-client FTL query-log access + privacy gating). 023 ships the merged top-talkers table only: join the two `/clients` calls on `client`, compute queries / blocked / block% per client. **Deferral (tracked, not prose):** the rich per-client drill-down is owned by the EXISTING **STAGE-006-027** (cross-epic EPIC-007 Client-page enhancement), whose card already lists "query volume, block rate, top blocked/permitted domains, recent blocks, per-client DNSSEC/SERVFAIL" powered by STAGE-006-025's VictoriaLogs query feed. No new stage authored; STAGE-006-027 is the concrete tracked owner.

**Decision 3 — Layout (selected 3A).** Fill the four existing `<PanelSection>` placeholders 1:1: "Upstreams & Unbound" (upstreams table + Unbound sub-card), "Clients" (top-talkers table), "Recent blocked" (string-list feed), "Version & container" (versions/updates table + restart/start/stop control row). Container-control buttons live in "Version & container"; each opens the generic `ConfirmPhraseDialog` with `expectedPhrase` = the action word ("restart"/"start"/"stop", matching the backend's `.strip().lower()` comparison). Target container hardcoded `pihole-unbound` (single combined container; Pi-hole is not a separately registered container).

**Decision 4 — Container-control write/refetch freshness (selected 4B).** Container status comes from the ~30s-lagged `homelab_container_status` collector metric; the lifecycle response returns only `{action, container_name, container_id, audit_id}` (no resulting running/exited state). onSuccess patches a transient "Restarting…/Starting…/Stopping…" indicator via `setQueryData` + `invalidateQueries({ refetchType: 'none' })`, mirroring the existing `useBlockingMutation` pattern; the existing 30s `refetchInterval` reconciles to authoritative collector state. Avoids 4A (immediate refetch of stale metric shows pre-action state) and 4C (fabricating a running/exited state the response never returns).

**New frontend surface:** `api/pihole.ts` adds `useUpstreams`, `useUnbound`, `useClients` (×2 or parameterized), `useRecentBlocked` reads + query keys; a container-lifecycle mutation hook (new — none exists in `docker.ts`; only `useStartPullAndRestart`). New widgets under `apps/ui/src/routes/integrations/`: upstreams/unbound card, clients top-talkers table, recent-blocked feed, version+container widget with control buttons.

**Backend-contract verification:** done LIVE (29090) — `/upstreams` (real 127.0.0.1#5335), `/unbound` (extended_stats_enabled true), `/clients` (top talker 192.168.2.100; card's "192.168.2.148 top device" is a descriptive example, not an invariant — widget renders whatever the live top talker is), `/recent-blocked` (live domains), `overview.versions`/`updates_available` (core/docker/ftl/web; updates: docker, web). Container write contract read from source (`integrations_docker.py`): phrase compared `.strip().lower()` == action word, Scope DOCKER_WRITE.
