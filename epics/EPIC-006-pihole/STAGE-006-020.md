# STAGE-006-020: Backend panel data endpoint(s)

## Status: Complete

## Overview

The backend that powers the Integrations ‚Üí Pi-hole panel: typed FastAPI endpoint(s) aggregating the panel's
read data (status strip, blocking state, gravity/adlists, FTL messages, upstreams+unbound, clients,
recent-blocked, versions, privacy level) ‚Äî some served from stored metrics, some live-queried from the
Pi-hole API at view-time (per the Tier-2 live-query decision). Mirrors EPIC-005 STAGE-005-021 (HA panel
data endpoint).

**Vertical slice:** `GET /api/integrations/pihole/*` returns the panel's data (typed, OpenAPI-described);
the frontend (006-021+) can render against it; live-query fields (top clients, recent blocked, messages
text) reflect current Pi-hole state.

**Spec references:** spec ¬ß9.2 (Integrations = plugin-provided panel), ¬ß3.1 (API layer typed routes +
OpenAPI). EPIC-006 "Operator panel". EPIC-005 STAGE-005-021 (panel data endpoint precedent).

## Cross-cutting standards

- Typed routes; OpenAPI schema regenerated; `require_user_or_token` (read scope) on the read endpoints.
- Live-query fields use the RO session; respect the refresh cadence (the frontend drives refetch; the
  backend doesn't poll on its own beyond serving requests).
- Privacy-level field included so the frontend can conditionally show the privacy banner.

### Cross-stage deliverable notes (from upstream stages)

- **(from STAGE-006-011 Refinement)** The `pihole_version` collector (and the other slow-cadence pihole collectors: e.g. `pihole_gravity` 30s is fine, but `pihole_version` at 3600s) does NOT run on scheduler startup ‚Äî its metrics are absent for up to ~1h after a monitor restart, so the version widget will show no version data until the first scheduled run. The sibling update-checkers (`image_update_checker`, `local_build_update_checker`) run on startup via explicit `await_immediate_run()` blocks in `kernel/api/lifespan.py` (a ~15-line per-collector block; there is NO one-line ClassVar opt-in). When building the version widget / panel-data endpoint, EVALUATE whether to add a startup-run hook for `pihole_version` (so the widget has data promptly after a restart) ‚Äî either by adding the lifespan block for `pihole_version`, or by refactoring to a `run_on_startup: ClassVar[bool]` opt-in on BaseCollector (the cleaner cross-cutting fix). This is a startup-latency UX concern, not a correctness bug (the collector works correctly when it runs).

## Concrete deliverables

### 1. Panel data endpoint(s)

`GET /api/integrations/pihole/overview` (status strip + blocking + versions + gravity summary + messages
count + privacy level), and the live-query sub-endpoints the widgets need: `.../clients` (top talkers,
live), `.../recent-blocked` (live feed), `.../messages` (full text, live), `.../adlists` (per-adlist health),
`.../upstreams` + `.../unbound` (resolver card). Design decides the exact split (one fat endpoint vs several
focused ones ‚Äî EPIC-005 used focused ones for drill-downs).

### 2. Data sourcing

Aggregate-from-stored-metrics where history matters (block %, query volume), live-query the Pi-hole API
where "now" matters (top clients, recent blocked, messages text, blocking timer). Document which field
comes from where.

## Design Phase

### Design Notes

**Selected approach (recon-verified against live source + brainstormer-assisted; coordinator selected):**

Extend the EXISTING router `apps/monitor/homelab_monitor/kernel/api/routers/integrations_pihole.py` (prefix `/integrations/pihole`, already registered in `app.py`, currently holds the 018 blocking/gravity POST routes). Mirrors the EPIC-005 HA panel pattern (`integrations_home_assistant.py`: `/summary` + focused sub-endpoints, VM-for-almost-everything, live-only-for-text/now).

**Decision 1 ‚Äî Endpoint split: focused endpoints (Option 1A).** A pure-VM `/overview` plus focused sub-endpoints, mirroring EPIC-005. Isolates failures (a Pi-hole API blip on a live endpoint doesn't break the VM-backed header strip), enables lazy-load aligned to the 021‚Äì024 tab structure, keeps each route's test surface small.

**Decision 2 ‚Äî Data sourcing: VM-maximal (Option 2A).** Collectors keep VM gauges fresh at 30‚Äì60s, so VM is near-live for almost everything. Live-query the Pi-hole RO API ONLY where staleness is user-visibly wrong: `/clients` (spec'd live Tier-2), `/recent-blocked` (a feed needs "now"), `/messages` (VM stores only the count; bodies require a live call ‚Äî the `/notifications` analogue). `/overview` stays pure-VM so the header strip is fast and resilient.

**Decision 3 ‚Äî `pihole_version` startup hook: one-off lifespan block (Option 3A).** Resolves the deferred-from-STAGE-006-011 startup-latency item (this stage OWNS it). Add a ~13-line `await_immediate_run("pihole_version", trigger=TriggerContext(kind="manual", request_id=None), timeout=30.0)` block in `kernel/api/lifespan.py` next to the existing `image_update_checker` / `local_build_update_checker` blocks, with a TODO noting "if a 3rd collector needs startup-run, promote to a generic `run_on_startup: ClassVar[bool]` opt-in on BaseCollector (Decision 3B)". Only 1 slow-cadence collector (`pihole_version`, 3600s) needs it today ‚Äî YAGNI over the cross-cutting base-class refactor.

**Endpoints (all GET, `Depends(require_session())`, read scope, GET CSRF-exempt; Pydantic models `ConfigDict(extra="ignore")`):**

| Endpoint | Source | Returns |
| --- | --- | --- |
| `GET /overview` | VM | up, blocking_enabled, blocking_timer_seconds, percent_blocked, query_frequency, messages_count, privacy_level, query_logging_enabled, gravity_domains, versions[{component,version}], updates_available[{component}] (nullable = metric absent; honest empty states ‚Äî versions[] empty until the version collector has run, updates_available[] empty when both versions not present) |
| `GET /adlists` | VM | rows[{list,address,status,enabled,domains?}], gravity_domains, gravity_last_update_age_seconds |
| `GET /upstreams` | VM | rows[{upstream,queries}] |
| `GET /unbound` | VM | cache_hit_ratio, queries_total, cache_hits_total, cache_misses_total, prefetch_total, requestlist_current (all nullable), extended_stats_enabled (None/legit-absent when unbound extended stats disabled) |
| `GET /clients` | LIVE (RO Pi-hole API) | rows[{client,name?,count}], returned; `?blocked=` + `?count=` (clamped) |
| `GET /recent-blocked` | LIVE | rows[], returned |
| `GET /messages` | LIVE | rows[{id,type,message,timestamp,url?}], total, returned |

**DI / contracts (recon-verified):**
- VM endpoints reuse the existing `get_vm_url` + `get_http_client` providers (dependencies.py) and `vm_count` / `vm_instant_query` (vm_query.py); VM unreachable ‚Üí `HttpProblem(502, "upstream_unavailable")` (matches HA precedent ‚Äî 502, not 503).
- Live endpoints obtain the RO client via a local `_get_pihole_ro_client(request)` reading the already-existing `app.state.pihole_client` (RO, `pihole_api_password_ro`; long-lived singleton wired in lifespan) ‚Äî 503 if uninitialized (mirrors the existing `_get_pihole_rw_client`). `PiholeError ‚Üí HttpProblem(502, "upstream_unavailable")`; defensive per-field `isinstance` parsing (mirror HA `/notifications`).
- OpenAPI regenerated (new response_models) ‚Üí `packages/shared-types/openapi.json` (+ `apps/ui/src/api/schema.ts` if it changes) staged in the impl commit.

**Out of scope for 020 (tracked owners):** the 024 embedded LogViewer / Tier-3 query feed (separate VictoriaLogs path ‚Äî STAGE-006-024/025); the 022 blocking-toggle MUTATION (a STAGE-006-022 write that should echo authoritative post-toggle state, so 020 needn't live-read blocking).

**Refinement plan:** 3a ‚Äî endpoints tested with mocked `vm_instant_query`/`vm_count` + mocked Pi-hole RO client (typed shape + OpenAPI correct, 502/503 error paths asserted). 3b ‚Äî real panel data on the prod rig (`make dev-prod` rebuild); live-query fields reflect current Pi-hole state; `/overview` versions populated promptly after restart (validates the 3A startup hook).
