# STAGE-006-006: Upstreams collector

## Status: Complete

## Overview

Poll `/api/stats/upstreams` and emit per-upstream query counts. On this deployment the upstreams are
unbound (`127.0.0.1#5335`) plus Pi-hole's cache/blocklist pseudo-upstreams. Feeds Grafana row 3 (by
upstream) and the `PiholeUpstreamDown` / `PiholeUpstreamAllDown` alerts.

**Vertical slice:** `homelab_pihole_upstream_queries{upstream}` appears in VM with one series per forward
destination; the unbound upstream is visible.

**Spec references:** EPIC-006 "Metric families" (Upstreams). Alert catalog: `PiholeUpstreamDown` (warning),
`PiholeUpstreamAllDown` (critical).

## Concrete deliverables

### 1. Collector

Poll `/api/stats/upstreams` (~30s). Emit `homelab_pihole_upstream_queries{upstream}` per destination
(including the cache/blocklist pseudo-upstreams Pi-hole reports). Document the pseudo-upstream entries so
the `UpstreamAllDown` rule (which should fire only when the REAL resolver share → 0) doesn't misfire on a
pseudo-upstream.

## Design Phase

**Status**: [x] Design

### Design Notes

**Selected approach: Option A (card-literal) — single `homelab_pihole_upstream_queries{upstream}` metric, mirroring the STAGE-006-005 collector pattern.** (Brainstormer produced 3 options + recommended Option B (add an `is_pseudo` indicator metric); coordinator selected Option A instead, because the card itself frames pseudo-upstream handling as DOCUMENTATION ("Document the pseudo-upstream entries so the `UpstreamAllDown` rule doesn't misfire"), NOT as a metric to emit — and STAGE-006-016 assumes no pseudo-indicator metric. Adding `is_pseudo` would be speculative scope (YAGNI) for a consumer (016) that doesn't exist yet and may choose name-exclusion anyway. Backend-stage selection made without user prompt, per pre-authorization.)

**Backend Contract Verification (live recon, ground truth):** Captured the REAL `/api/stats/upstreams` from live Pi-hole v6.6.2. Shape: `{upstreams: [ {ip, name, port, count, statistics:{response, variance}}, ... ], total_queries, forwarded_queries, took}`. Each upstream entry: `ip` (str), `name` (str friendly hostname), `port` (int — REAL has a real port e.g. 5335; PSEUDO has `port == -1`), `count` (int, 24h-rolling per-upstream query total), `statistics.{response,variance}` (floats; response = mean resolve seconds, 0 for pseudo). Live data: 3 entries — `127.0.0.1#5335` (unbound, real, count 6893) + 2 pseudo (`cache` count 36276, `blocklist` count 48852, both `port=-1`). Low cardinality.

**Decision A — `{upstream}` label rule:** `port == -1` (pseudo) → `upstream = ip` (`"cache"` / `"blocklist"`, NO `#-1` suffix); else (real) → `upstream = f"{ip}#{port}"` (e.g. `127.0.0.1#5335`). Use `ip#port`, NOT `name` (`name="localhost"` is less informative/unique than `127.0.0.1#5335`). Defensive branch set for 100% coverage (upstreams is a LIST of dicts — narrow EACH entry, skip malformed): (1) `upstreams` missing/not-a-list → emit nothing, ok=True with just api_took; (2) entry not a dict → skip; (3) `ip` missing/not-str → skip; (4) `port` missing/not-int → skip (do NOT fall through to a bare-ip label — would collide a real upstream with a pseudo); (5) `port == -1` → pseudo label branch; (6) `port != -1` → real label branch; (7) `count` non-numeric (`as_float` None) → skip that entry.

**Decision B — single `{upstream}` label, NO `kind`/`is_pseudo`.** Pseudo-vs-real is the consumer's (STAGE-006-016) concern, handled by name-exclusion. CONTRACT FOR 016 (documented here + in the module docstring so it is NOT a silent handoff): the `PiholeUpstreamAllDown` / `PiholeUpstreamDown` alert rules MUST exclude the pseudo-upstreams `upstream="cache"` and `upstream="blocklist"` (Pi-hole pseudo-upstreams are identified by `port == -1`; on this deployment they surface as exactly those two `upstream` label values) so the "all real upstreams down" alert fires only when the REAL resolver share → 0, not when cache/blocklist traffic is nonzero.

**Decision C — `statistics.response` (per-upstream mean response time) NOT emitted.** The card names only `homelab_pihole_upstream_queries`. Per-upstream latency is genuinely unowned (verified: STAGE-006-013 emits unbound-internal recursion time, STAGE-006-014 emits a single `:53` probe — neither is per-`{upstream}`), but YAGNI: no current consumer. This is NOT a deferral of a card deliverable (the card never asked for it) — it is a documented observation that the data exists at `/api/stats/upstreams` `statistics.response` if a future stage ever needs a per-upstream "slow upstream" signal.

**Decision D — top-level `total_queries`/`forwarded_queries` NOT emitted.** STAGE-006-005 already owns the totals (`homelab_pihole_queries_total` / `homelab_pihole_forwarded_total` from `/api/stats/summary`). Re-emitting here would create double-ownership; skip.

**Decision E — exact names.** `M_UPSTREAM_QUERIES = "homelab_pihole_upstream_queries"` (singular), label key `"upstream"` (singular); reuse `M_API_TOOK = "homelab_pihole_api_took_seconds"` with `{"endpoint": "stats/upstreams"}`. Add a `test_metric_name_constants_match_contract` literal-name test mirroring the one added in STAGE-006-005 (guards against constant-value drift, the gap that let the 005 naming bug through).

**Structure:** New `plugins/collectors/integrations/pihole/upstreams.py`, class `PiholeUpstreamsCollector`: `name="pihole_upstreams"`, `interval=timedelta(seconds=30)`, `timeout=timedelta(seconds=15)`, `concurrency_group="pihole"`. Mirrors `stats_summary.py` exactly (ctx.pihole None guard → `stats_upstreams()` → PiholeError branch → api_took emit → `raw_payload: object`→isinstance(dict)→cast → iterate the `upstreams` list narrowing each entry → `emit_numeric`/`write_gauge` with the `{upstream}` label → CollectorResult, `emitted[0]` counter). Reuses `pihole/_parsing.py` (`as_float`/`emit_numeric`). Module docstring documents: the `count` is a 24h-rolling window-gauge (NOT a counter — downstream must not `rate()` it), the pseudo-upstream contract for 016, and that `statistics.response` is intentionally not emitted. One-line `_PIHOLE_COLLECTORS` edit in the bundle `__init__.py` (+ import). Cardinality cap NOT applied (low, bounded — per the card).

**Refinement:** 3a (captured `/api/stats/upstreams` fixture — assert the `{upstream}` series incl. `127.0.0.1#5335` + cache + blocklist) + 3b (live on prod `:29090` after `make dev-prod`; confirm `homelab_pihole_upstream_queries{upstream="127.0.0.1#5335"}` present in VM). No migration (alembic HEAD `0044`), no openapi/frontend change.
