# STAGE-006-007: Gravity + per-adlist health collector

## Status: Complete

## Overview

Monitor Pi-hole's blocklist engine ("gravity"): the aggregate domain count + last-update age, AND
**per-adlist health** (each adlist's enabled state, domain contribution, and last-download status). The
live host currently has 2 failing adlists (inaccessible during the last gravity run) — this collector +
the `PiholeAdlistFailing` alert make that specific and actionable. Feeds Grafana row 4 and the gravity/
adlist alerts.

**Vertical slice:** `homelab_pihole_gravity_domains` + `_gravity_last_update_age_seconds` +
`_adlist_domains{list}` / `_adlist_enabled{list}` / `_adlist_status{list}` appear in VM; the 2 failing
adlists show a failed status by name.

**Spec references:** EPIC-006 "Metric families" (Gravity + per-adlist). Q5 decision = option B (essential +
per-adlist). Alerts: `PiholeGravityStale` (TZ-guarded), `PiholeGravityDomainsDropped`, `PiholeAdlistFailing`.

## Cross-cutting standards

- Reuse client session; failure budget; self-metrics.
- `gravity_last_update_age` computed as `now - last_update` — Design must TZ-guard (a stale-looking
  timestamp can be a timezone artifact, not a real failure; research-flagged) so `PiholeGravityStale`
  doesn't false-fire.
- Per-adlist cardinality: bounded by the number of adlists (small) — label by list id/address; cap if a
  user has a huge adlist count (configurable).

## Concrete deliverables

### 1. Gravity aggregate

From `/api/info/ftl` (gravity domain count + last_update) — emit `homelab_pihole_gravity_domains` and
`homelab_pihole_gravity_last_update_age_seconds` (TZ-correct).

### 2. Per-adlist health

From `/api/lists` (~5min) — emit per-adlist `homelab_pihole_adlist_domains{list}`,
`_adlist_enabled{list}` (1/0), `_adlist_status{list}` (enum: ok / failed / …; map the v6 list-status field).
The `list` label = the adlist id or address (Design picks the stable key; address is human-meaningful).
Surface the failing-adlist case (the live 2) as a `failed` status.

## Design Phase

**Status**: [x] Design

### Design Notes

**Selected approach: Option B (adjusted) — two-endpoint Gravity + per-adlist collector with one-hot status labels.** (Brainstormer produced 3 options + recommended Option B; coordinator adopted Option B with two YAGNI/churn adjustments noted below. Backend-stage selection made without user prompt, per pre-authorization.)

**Backend Contract Verification (live recon, ground truth — OVERRODE the card's endpoint assumption):**
- `/api/info/ftl` → `{ftl:{...}, took}`. Gravity DOMAIN COUNT is at `ftl.database.gravity` (int, live=5863264). **CRITICAL: `/api/info/ftl` has NO last-update timestamp anywhere** — the card assumed it did; it does NOT. `ftl.database` is counts only.
- `/api/lists` → `{lists:[...], took}`. 5 adlists. Each: `id`(int), `address`(str URL), `enabled`(BOOL true/false), `status`(INT code), `number`(int per-list domain count), `date_updated`(epoch SECONDS — last successful pull of THIS list), `date_added`/`date_modified`(epoch), `comment`(str), `type`/`invalid_domains`/`abp_entries`/`groups`. Status int codes seen: {1,3}. The 2 known-failing adlists (id=4, id=5) carry `status=3` (stale date_updated 2025-08 vs the 3 OK lists' 2026-06-21).
- ALL timestamps are Unix EPOCH SECONDS → `datetime.fromtimestamp(epoch, UTC)` is unambiguous UTC; **NO timezone artifact possible**. The card's "TZ-guard" concern was predicated on an assumed ISO timestamp; epoch eliminates it. Only the clock-skew clamp (`max(0.0)`) is needed. (Record this so STAGE-006-016 does NOT over-engineer a TZ guard into PiholeGravityStale.)

**Decision A — gravity `last_update_age_seconds` SOURCE (resolves the card's contract gap).** `/api/info/ftl` has no timestamp, so the age is DERIVED from `/api/lists`: `now - max(date_updated across ALL adlists)`. Rationale: "gravity stale" means the whole gravity run hasn't happened recently; a FAILING list keeps its OLD `date_updated` (it didn't pull cleanly) so it never drags the max down — max-across-all correctly tracks "last time gravity successfully pulled anything" (on the live host this = the 3 healthy lists' 2026-06-21, fresh, while the 2 failed lists stay at their 2025-08 stamps). If NO adlist has a parseable `date_updated` (empty list / all entries malformed), the age metric is SKIPPED entirely (NOT emitted as 0 — a 0 would look "updated right now" and false-clear PiholeGravityStale). Clamp `max(age, 0.0)` for clock skew.

**Decision B — TWO endpoints, per-endpoint resilience (first two-endpoint collector).** Calls BOTH `info_ftl()` (authoritative `gravity_domains` from `ftl.database.gravity`; `sum(number)` is NOT a substitute — it double-counts overlapping domains, ~6.4M raw vs the dedup'd 5.86M) AND `lists()` (per-adlist + derived age). Emits `homelab_pihole_api_took_seconds` TWICE — `{endpoint="info/ftl"}` and `{endpoint="lists"}`. Partial-failure policy: emit whatever succeeded; append each endpoint's `PiholeError.message` to `errors`; `ok=False` ONLY if BOTH endpoints error (or `ctx.pihole is None`). Rationale: mirrors the existing collectors' "partial = still ok=True" convention and maximizes alert reliability (one flaky endpoint shouldn't suppress the other's metrics + the collector health signal).

**Decision C — `_adlist_status` one-hot STRING label (matches the card's "enum: ok/failed/…" + the `query_by_status{status}` precedent).** Emit `homelab_pihole_adlist_status{list=<id>, status=<name>} = 1.0` where the v6 int code maps `0→not_run, 1→ok, 2→download_failed, 3→parse_failed`, and any UNKNOWN future code → `status="unknown_<n>"` (a covered fallthrough branch so a new v6 code never silently vanishes). STAGE-006-016's `PiholeAdlistFailing` matches `homelab_pihole_adlist_status{status!="ok"} == 1` (single-metric threshold — dodges 016's §4.1 empty-join trap; confirmed 016 is encoding-neutral and assumes only the ok/failed enum semantics).

**Decision D — series keyed by `id`; carry `address` as the SINGLE descriptive label (coordinator adjustment vs brainstormer's D3).** Series key = adlist `id` (stable, unique, immutable). Carry `address` as an additional label on each adlist metric so the alert/dashboard can name the failing list ("by name" requirement — the NRD GitHub URLs identify the failing id=4/id=5). DEVIATION FROM BRAINSTORMER: do NOT also carry `comment` as a label — `comment` is user-editable (high label churn) and `address` already satisfies "by name" with far lower churn; tripling churn surface across 3 metrics for a redundant human label is rejected (YAGNI). So each adlist metric = `{list=<id>, address=<url>}` (+ `status=<name>` on the status metric). `MAX_ADLISTS` hard cap constant (per the card's "cap if a user has a huge adlist count") with a covered truncation branch — kept as a hard module constant, NOT settings-plumbing (YAGNI on configurability; the card's "configurable" is deferred until a real multi-hundred-adlist user exists). At 5 adlists the cap never triggers in prod; the branch is covered by a synthetic over-cap test.

**Decision E — exact names + structure.** New `plugins/collectors/integrations/pihole/gravity.py`, class `PiholeGravityCollector`: `name="pihole_gravity"`, `interval=timedelta(seconds=30)`, `timeout=timedelta(seconds=15)`, `concurrency_group="pihole"`. Constants: `M_GRAVITY_DOMAINS="homelab_pihole_gravity_domains"`, `M_GRAVITY_AGE="homelab_pihole_gravity_last_update_age_seconds"`, `M_ADLIST_DOMAINS="homelab_pihole_adlist_domains"`, `M_ADLIST_ENABLED="homelab_pihole_adlist_enabled"`, `M_ADLIST_STATUS="homelab_pihole_adlist_status"`, `M_API_TOOK="homelab_pihole_api_took_seconds"`; label key `"list"`. Reuses `pihole/_parsing.py` `as_float`/`emit_numeric`. `enabled` (bool) → emit 1.0/0.0 (non-bool → skip or default; specify at Build). Age: `datetime.now(UTC)` inline + `datetime.fromtimestamp(epoch, UTC)` + `max(0.0)`; tested by range-assert (`datetime.now(UTC) - timedelta(...)` fixtures, ±range), NOT a clock injection (matches the HA `ha_entity_available` test pattern). Add a `test_metric_name_constants_match_contract` literal-name test (the 005/006 drift guard). One-line `_PIHOLE_COLLECTORS` registration + import.

**100% branch set:** ctx.pihole None; ftl PiholeError; ftl payload non-dict; `ftl`/`database` missing-or-non-dict; `gravity` missing/non-numeric; lists PiholeError; lists payload non-dict; `lists` missing/not-a-list; empty list; per-adlist entry-not-dict / missing-id / number missing-or-non-numeric / enabled true/false/non-bool / status valid-int/missing/non-int / unknown-status-code fallthrough / date_updated valid/missing/non-numeric; age has-at-least-one-valid-date_updated vs none-valid (skip) ; clock-skew clamp; MAX_ADLISTS truncation; both-fail vs one-fail vs both-succeed ok/errors matrix.

**Refinement:** 3a (fixtures: healthy gravity, a stale-gravity TZ/age case, a failed-adlist entry, both-endpoint-error case) + 3b (live on prod `:29090` after `make dev-prod`: real `gravity_domains` ~5.86M, derived age fresh, the 2 live failing adlists id=4/id=5 surfaced as `status="parse_failed"` with their `address` labels).
