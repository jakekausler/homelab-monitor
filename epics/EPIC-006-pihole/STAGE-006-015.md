# STAGE-006-015: DNS split-check collector (Pi-hole vs direct 1.1.1.1)

## Status: Complete

## Overview

Resolve a known domain via Pi-hole AND directly via 1.1.1.1 (bypassing Pi-hole), emitting both latencies
so a failure can be attributed: Pi-hole-broken (Pi-hole path fails, direct succeeds) vs WAN/internet-down
(both fail). This is the disambiguation signal behind `PiholeDnsSplitDivergence` and is **shared with
EPIC-016** (the same `homelab_dns_resolution_seconds{path}` family). Authored here because Pi-hole is the
subject; EPIC-016 reuses it for WAN health.

**Vertical slice:** `homelab_dns_resolution_seconds{path="pihole"}` and `{path="direct"}` in VM; when
Pi-hole is forced down, the pihole path fails while direct succeeds — proving the split distinguishes
Pi-hole vs WAN.

**Spec references:** spec §2 Q21 (DNS resolution health: split via Pi-hole and direct to 1.1.1.1). EPIC-006
"Metric families" (DNS split-check, shared w/ EPIC-016). EPIC-016 STAGE-016-005 (same dual-path test).
Alert: `PiholeDnsSplitDivergence` (critical — confirms Pi-hole, not WAN, is the problem).

## Cross-cutting standards

- The "direct" path MUST bypass Pi-hole entirely (query 1.1.1.1 directly) — never via Pi-hole (circular-
  dependency rule, shared with EPIC-014/016).
- Emit a shared metric family (`homelab_dns_resolution_seconds{path}`) so EPIC-016 consumes it without
  duplication. Coordinate the metric name with EPIC-016 STAGE-016-005 (they are the same collector concept;
  decide in Design whether this stage IS 016-005 or a shared module both reference — avoid two implementations).

## Concrete deliverables

### 1. Dual-path resolver

~60s. Resolve a known domain (e.g. `dns.google.com`) via Pi-hole `:53` and via `1.1.1.1:53` directly. Emit
`homelab_dns_resolution_seconds{path="pihole"}` and `{path="direct"}` (latency; failure = high/absent +
a per-path success boolean).

### 2. Divergence signal

The `PiholeDnsSplitDivergence` rule (006-016) fires when pihole-path fails AND direct-path succeeds. This
stage emits the clean per-path signals; the rule logic is in 006-016. Document the both-fail case = WAN
problem (NOT a Pi-hole alert — that's EPIC-016's WAN-down territory).

## Design Phase

### Design Notes

**Approach selected (coordinator selected own recommendation; full Opus brainstormer consulted — new shared cross-epic metric family + WAN-interception correctness risk are novel).** A NEW standalone collector `PiholeDnsSplitCollector` resolves `dns.google.com` via TWO paths each cycle — Pi-hole's `:53` AND directly via `1.1.1.1:53` (bypassing Pi-hole) — reusing the STAGE-006-014 `resolve_a` helper, and emitting the NEW shared family `homelab_dns_resolution_*{path="pihole"|"direct"}`. 015 is a STATELESS dual-probe emitter; the divergence/both-fail LOGIC lives in STAGE-006-016 PromQL.

**Recon-first (live source + codebase ground truth, captured in Design):**
- `resolve_a(resolver_ip, qname, *, port=53, timeout_seconds) -> DnsProbeResult(ok, rcode, truncated, latency_seconds, error)` EXISTS (`kernel/dns/resolver.py`, built in 014), never raises, parameterized by resolver IP, re-exported from `kernel/dns/__init__.py`. 015 calls it ONCE per path. No DNS reinvented.
- The 014 collector `PiholeDnsHealthCollector` (`plugins/collectors/integrations/pihole/dns_health.py`) is the structural template: `_OUTCOME_BY_ERROR` (10 outcome tokens), `_RESPONSE_OUTCOMES` (the latency-emit gate), `_PROBE_QNAME = "dns.google.com"`, `ctx.vm.write_gauge`, post-construction `_dns_host`/`_dns_port` injection in `lifespan.py`.
- `PiholeConfig` already has `dns_host` (derives Pi-hole IP from base_url hostname when empty) + `dns_port` (014). NO config for a direct/WAN resolver yet.
- The `homelab_dns_resolution_*` family does NOT exist anywhere in the repo (collectors/vmalert/grafana/tests) — 015 creates it fresh.
- **OWNERSHIP RESOLVED (the card's open cross-stage question):** STAGE-006-015 is the SOLE OWNER of the dual-path resolver + the `homelab_dns_resolution_*` family. EPIC-016 (entirely Not Started) STAGE-016-005 CONSUMES the family for WAN-health context — it does NOT build a second implementation. Authored in EPIC-006 because Pi-hole is the subject. No duplication.
- 016 alert contract: `PiholeDnsSplitDivergence` (critical) = "pihole-path-fail AND direct-ok". 015 must emit a per-path SUCCESS signal (not just latency) so the PromQL is clean.

**Locked decisions:**
1. **Collector (Decision A → A1):** NEW standalone `PiholeDnsSplitCollector`, file `plugins/collectors/integrations/pihole/dns_split.py`, `name="pihole_dns_split"`, `interval=60s`, `timeout=15s`, `concurrency_group="pihole"`. Single-responsibility; 014 stays the Pi-hole-liveness collector. The cross-epic `homelab_dns_*` family is a separate concern → separate collector (the distinct prefix signals this). Appended to `_PIHOLE_COLLECTORS`. (The one extra Pi-hole UDP query/min vs reusing 014 is negligible.)
2. **Direct-resolver config (Decision B → B2):** add `direct_dns_host: str = "1.1.1.1"` + `direct_dns_port: int = 53` to `PiholeConfig` (env `HOMELAB_MONITOR_PIHOLE_DIRECT_DNS_HOST` / `HOMELAB_MONITOR_PIHOLE_DIRECT_DNS_PORT`; default literal `1.1.1.1`/53 IS the fallback — no derive-from-base_url). Chosen over hardcoding because WAN `:53` interception is a real correctness risk (see Build flags) — an env override turns a redeploy into a config change; and it mirrors the 014 `dns_host`/`dns_port` precedent exactly (near-zero novelty, already-tested shape, with config tests).
3. **Metrics (Decision C → C-i + C-ii; pin in a `test_metric_name_constants_match_contract` literal test):**
   - `homelab_dns_resolution_up{path}` (gauge 1/0) — ALWAYS emitted for BOTH paths every cycle (1 iff `DnsProbeResult.ok`). PRIMARY success signal → makes the 016 rule a trivial `up{path="pihole"}==0 and up{path="direct"}==1`.
   - `homelab_dns_resolution_seconds{path}` (gauge, latency) — emitted ONLY on a real round-trip (outcome in `_RESPONSE_OUTCOMES`); OMITTED on no-response (timeout/socket_error/malformed/id_mismatch). Never a sentinel/lie. Failure detection keys off `_up`, not latency presence.
   - `homelab_dns_resolution_probe_result{path,outcome}` (gauge 1.0, one-hot) — ALWAYS one series per path per cycle (`outcome = _OUTCOME_BY_ERROR[result.error]`). Shipped NOW (not deferred) because adding to a cross-epic-shared family later is a breaking-shape change, the data is already in `resolve_a`, and 014 proved the pattern. Diagnostic for 016 dashboards (why a path failed).
   - `path ∈ {"pihole","direct"}`. Stable label-key sets per family (prometheus_writer first-seen-frozenset rule): `_up`/`_seconds` carry exactly `{path}`; `_probe_result` carries exactly `{path, outcome}` (outcome always present).
   - **Shared-constant extraction:** extract 014's `_OUTCOME_BY_ERROR` + `_RESPONSE_OUTCOMES` + `_PROBE_QNAME` into ONE shared module that BOTH collectors import (avoids cross-collector duplication of the dicts) — exact location pinned at Build.
4. **Semantics (Decision D):** 015 computes NO divergence — it emits raw per-path series; 016 owns the `PiholeDnsSplitDivergence` PromQL. Both-fail (WAN down) emits both `_up=0` (no special handling; 016 simply doesn't fire because `direct==1` is false). Probe BOTH paths UNCONDITIONALLY (the direct host is never empty; the Pi-hole `dns_host` is derived/non-empty) — emit `up=0` per-path on failure rather than short-circuiting. Both paths resolve the SAME qname `dns.google.com` (apples-to-apples). Per-probe `timeout_seconds=5.0`; two sequential 5s-worst-case probes fit the 15s collector timeout.
5. **Lifespan injection:** mirror the 014 `isinstance(c, PiholeDnsHealthCollector)` block with a new `isinstance(c, PiholeDnsSplitCollector)` block injecting `_pihole_host`/`_pihole_port` (from `pihole_config.dns_host`/`dns_port`) + `_direct_host`/`_direct_port` (from the new `pihole_config.direct_dns_host`/`direct_dns_port`), with `# pyright: ignore[reportPrivateUsage]`. lifespan.py is in the 100%-coverage gate → add a `test_lifespan_wires_pihole_dns_split_collector` startup test.

**Build-time recon flags (must verify early in Build / 3b):**
1. **UDP egress to a PUBLIC IP** (`1.1.1.1:53`) from the bridge network → NAT → WAN (analogous to 014's egress check, but to the WAN). If blocked, the direct path reads as timeout and the split is useless.
2. **WAN `:53` interception (CRITICAL correctness risk):** if this network DNAT-redirects all outbound `:53` to the local resolver, the "direct" `1.1.1.1` probe is silently answered by Pi-hole/router — violating the bypass guarantee INVISIBLY. Verify in 3b with a Cloudflare-identifying query (e.g. `dig @1.1.1.1 ch txt id.server` should return a Cloudflare server, NOT the local resolver). If interception is found, Decision B2's configurability lets us move to a non-:53 anchor without a redeploy.
3. Confirm the collector is REGISTERED in `_PIHOLE_COLLECTORS` (injection alone doesn't schedule it).

**No deferrals.** Backend-only; no migration, no API/frontend change. The `homelab_dns_resolution_*` family is owned here and consumed by EPIC-016 STAGE-016-005 (Not Started) + alerted by STAGE-006-016 (`PiholeDnsSplitDivergence`).

**Files:** CREATE `plugins/collectors/integrations/pihole/dns_split.py` + `tests/test_pihole_dns_split_collector.py`; CREATE or designate a shared-constants module (extract `_OUTCOME_BY_ERROR`/`_RESPONSE_OUTCOMES`/`_PROBE_QNAME` from 014) + update 014 to import from it; MODIFY `plugins/collectors/integrations/pihole/__init__.py` (import + `_PIHOLE_COLLECTORS` append); MODIFY `kernel/config.py` (+ `tests/test_config.py`) for the direct-resolver fields; MODIFY `kernel/api/lifespan.py` (+ `tests/test_api_lifespan.py`) for the injection block.
