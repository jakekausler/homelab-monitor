# STAGE-006-015: DNS split-check collector (Pi-hole vs direct 1.1.1.1)

## Status: Not Started

## Overview

Resolve a known domain via Pi-hole AND directly via 1.1.1.1 (bypassing Pi-hole), emitting both latencies
so a failure can be attributed: Pi-hole-broken (Pi-hole path fails, direct succeeds) vs WAN/internet-down
(both fail). This is the disambiguation signal behind `PiholeDnsSplitDivergence` and is **shared with
EPIC-016** (the same `homelab_dns_resolution_seconds{path}` family). Authored here because Pi-hole is the
subject; EPIC-016 reuses it for WAN health.

**Vertical slice:** `homelab_dns_resolution_seconds{path="pihole"}` and `{path="direct"}` in VM; when
Pi-hole is forced down, the pihole path fails while direct succeeds — proving the split distinguishes
Pi-hole vs WAN.

**Spec references:** spec §2 Q21 (DNS resolution health: split via Pi-hole and direct to 1.1.1.1). EPIC-006
"Metric families" (DNS split-check, shared w/ EPIC-016). EPIC-016 STAGE-016-005 (same dual-path test).
Alert: `PiholeDnsSplitDivergence` (critical — confirms Pi-hole, not WAN, is the problem).

## Cross-cutting standards

- The "direct" path MUST bypass Pi-hole entirely (query 1.1.1.1 directly) — never via Pi-hole (circular-
  dependency rule, shared with EPIC-014/016).
- Emit a shared metric family (`homelab_dns_resolution_seconds{path}`) so EPIC-016 consumes it without
  duplication. Coordinate the metric name with EPIC-016 STAGE-016-005 (they are the same collector concept;
  decide in Design whether this stage IS 016-005 or a shared module both reference — avoid two implementations).

## Concrete deliverables

### 1. Dual-path resolver

~60s. Resolve a known domain (e.g. `dns.google.com`) via Pi-hole `:53` and via `1.1.1.1:53` directly. Emit
`homelab_dns_resolution_seconds{path="pihole"}` and `{path="direct"}` (latency; failure = high/absent +
a per-path success boolean).

### 2. Divergence signal

The `PiholeDnsSplitDivergence` rule (006-016) fires when pihole-path fails AND direct-path succeeds. This
stage emits the clean per-path signals; the rule logic is in 006-016. Document the both-fail case = WAN
problem (NOT a Pi-hole alert — that's EPIC-016's WAN-down territory).
