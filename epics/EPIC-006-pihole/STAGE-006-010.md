# STAGE-006-010: FTL diagnostic-messages collector

## Status: Complete

## Overview

Poll `/api/info/messages` and emit Pi-hole's own diagnostic-message feed as a first-class metric. This is
the highest-signal "Pi-hole is telling you it's unhappy" source — it mirrors the web UI's red-dot warnings
(gravity issues, rate-limit events, config problems, inaccessible adlists). The live host currently
reports 2 such messages (the inaccessible adlists). Feeds the `PiholeMessagesPresent` alert + the panel's
messages widget.

**Vertical slice:** `homelab_pihole_messages_count` + `homelab_pihole_message{type}` in VM; the live 2
messages are reflected; the alert annotation can carry the message text.

**Spec references:** EPIC-006 "Metric families" (FTL messages, first-class). Q5 decision = surface FTL
messages first-class. Alert: `PiholeMessagesPresent` (warning, with message text in annotation).

## Concrete deliverables

### 1. Collector

Poll `/api/info/messages` (~60s; or `/messages/count` for the cheap count + `/messages` for detail). Emit
`homelab_pihole_messages_count` and `homelab_pihole_message{type}` (1 per active message, labeled by its
type/category — NOT by the free-text body, to bound cardinality). Map the v6 message-type enum.

### 2. Text availability for the panel

The full message text is surfaced live in the panel's FTL-messages widget (006-022), fetched at view-time
— not stored as a high-cardinality metric label. This stage emits the count + type; the panel renders text.

## Design Phase

**Status**: [x] Design

### Design Notes

**Single-endpoint collector** (`/api/info/messages`), mirroring `blocking.py`'s structure. Brainstormer (Opus) consulted on 4 decisions; all recommendations adopted.

Live recon (Pi-hole v6.6.2, this session) corrected the card on several points:
- `/api/info/messages` returns `{"messages": [ {id:int, timestamp:int-epoch, type:str, plain:str, html:str}, ... ], "took": float}` — a LIST of message objects.
- Live state: **3** messages (card said 2): 2× `type="LIST"` (inaccessible-adlist warnings) + 1× `type="LOAD"` (high-load warning). The inaccessible-adlist type is `"LIST"`, NOT `"INACCESSIBLE_ADLIST"` as the card assumed. These card factual errors have no code impact (the collector reads whatever types are live).
- A dedicated `/api/info/messages/count` endpoint exists (`{"count":N,"took":...}`, total only, no per-type breakdown).

Metric contract (final):
- `homelab_pihole_messages_count` (gauge) = `len(messages)`, **always emitted** (0 when the list is empty — so the `PiholeMessagesPresent` alert can fire on `> 0` with no `absent()` needed).
- `homelab_pihole_messages{type}` (gauge) = per-type COUNT, grouped from the message list (duplicate types collapse: 2×LIST → `{type="LIST"}=2`). Emitted for PRESENT types only (no zero-fill, no hardcoded FTL enum — consistent with 005/006/007 enum collectors). **NAME CORRECTED from the card's `homelab_pihole_message{type}` (singular) to plural `homelab_pihole_messages{type}`** — the card's singular implied one-series-per-message, which recon disproved (duplicate types must collapse to a count; sibling families like `query_by_type`/`queries_total` use plural for grouped counts). The literal-name contract test asserts the plural.
- `homelab_pihole_api_took_seconds{endpoint="info/messages"}` once per run.

Design decisions (brainstormer-recommended, adopted):
1. Count source: SINGLE `/api/info/messages` call; derive both metrics from one snapshot (`len(messages)` for total). The separate `/messages/count` call is redundant (we must fetch the full list anyway for the per-type breakdown) and risks total≠sum(per-type) if FTL state changes between two calls.
2. Per-type metric: group the list by `type` into `{type: count}`, emit present types only. Map a non-string/odd `type` value to a stable fallback (e.g. `unknown_<...>`, mirroring gravity's `_status_name` defensive pattern) so a malformed entry can't crash or produce a bad label.
3. Empty state: always emit `_messages_count = 0`; no per-type series when empty.
4. No cardinality cap (FTL message-type enum is a small fixed internal set; the `type` source is not user/network input — consistent with the other pihole enum collectors).

Implementation note: the existing `stats_summary._emit_enum` consumes a dict-of-counts; this collector must FIRST aggregate the message LIST into a `{type: count}` dict (group-by the `type` field, with the unknown-type fallback), then emit. A small local aggregation helper (or inline group-by) is needed — not present in the exemplars.

Text bodies (`plain`/`html`) are NOT emitted as metric labels (cardinality). They are fetched at panel render-time / alert annotation render-time (a future UI/alert stage's concern, not this collector).

Future candidate (NOT a card deliverable, NOT a deferral of required scope — recorded as an idea only): an `oldest_message_age_seconds` gauge (now − min(timestamp)) could flag a stale unresolved warning. The card does not ask for it and no alert consumes it; build-only-what's-asked, so it is intentionally NOT built. If a future stage wants it, it's a non-breaking additive metric.

planner-lite is sufficient at Build (single-endpoint clone + a small group-by helper).
