# STAGE-006-010: FTL diagnostic-messages collector

## Status: Not Started

## Overview

Poll `/api/info/messages` and emit Pi-hole's own diagnostic-message feed as a first-class metric. This is
the highest-signal "Pi-hole is telling you it's unhappy" source — it mirrors the web UI's red-dot warnings
(gravity issues, rate-limit events, config problems, inaccessible adlists). The live host currently
reports 2 such messages (the inaccessible adlists). Feeds the `PiholeMessagesPresent` alert + the panel's
messages widget.

**Vertical slice:** `homelab_pihole_messages_count` + `homelab_pihole_message{type}` in VM; the live 2
messages are reflected; the alert annotation can carry the message text.

**Spec references:** EPIC-006 "Metric families" (FTL messages, first-class). Q5 decision = surface FTL
messages first-class. Alert: `PiholeMessagesPresent` (warning, with message text in annotation).

## Cross-cutting standards

- Reuse client session; failure budget; self-metrics.
- The message TEXT may need to reach the alert annotation — decide whether text rides a label (cardinality
  risk if message text is high-variety) or is fetched at alert-render time. EPIC-006 lists
  `message{type}` (type as the label, low-cardinality) + a count; the panel shows full text live.

## Concrete deliverables

### 1. Collector

Poll `/api/info/messages` (~60s; or `/messages/count` for the cheap count + `/messages` for detail). Emit
`homelab_pihole_messages_count` and `homelab_pihole_message{type}` (1 per active message, labeled by its
type/category — NOT by the free-text body, to bound cardinality). Map the v6 message-type enum.

### 2. Text availability for the panel

The full message text is surfaced live in the panel's FTL-messages widget (006-022), fetched at view-time
— not stored as a high-cardinality metric label. This stage emits the count + type; the panel renders text.
