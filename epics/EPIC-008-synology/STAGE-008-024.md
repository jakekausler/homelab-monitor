**Design Notes** (recorded 2026-06-27):

**Approach:** Author the BACKEND panel data endpoints (FastAPI `response_model`s) that feed the frontend Synology + Surveillance Integration pages (built in stages 025-029). Mirror the existing UniFi panel router pattern (`integrations_unifi.py`): routes with `response_model=`, deps `_user: Depends(require_session())` + `vm_url: Depends(get_vm_url)` + `http_client: Depends(get_http_client)`, data via `vm_instant_query(http_client, vm_url, promql)` in parallel `asyncio.gather`, Pydantic models with `ConfigDict(extra="ignore")`, honest-empty-state (absent VM vector â†’ 0/None/[]/False, never 502 on empty), VM-transport-failure â†’ `HttpProblem(502, "upstream_unavailable")`. Register in app.py via `include_router(..., prefix="/api")`. No existing synology panel endpoint â€” all new.

**Endpoint set (~8 endpoints, TWO routers â€” selected via Opus brainstormer, recommendations adopted per pre-authorization):**

`kernel/api/routers/integrations_synology.py` (prefix `/integrations/synology`):

**`data_available` flag (the collector-down vs genuinely-empty discriminator):** An empty VM vector can't distinguish "collector not running" from "genuinely zero." For the 3 families that are ABSENT-today-due-to-stale-image (surveillance, SSH-probe, mount-health), each relevant endpoint adds ONE `vm_instant_query` against the collector self-metric `homelab_collector_run_success_total{name="<collector>"}` and sets a per-section `data_available: bool` (the self-metric is mandatory per the "plugins observe themselves" invariant, so it's guaranteed present). This lets the frontend show "no data yet" vs "0 cameras". For genuinely-configurable-zero families (backup `no_backup_configured=1`, replication `replication_available=0`) use their OWN configured-state metric, NOT the collector probe (don't over-add `data_available`).
