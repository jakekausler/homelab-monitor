# STAGE-008-006: Pool & RAID collector (status, rebuild progress, scrub)

## Status: Complete

## Overview

Emit the pool/RAID/scrub slice of the `load_info` response (the same `Storage.CGI.Storage load_info` fetch
008-005 uses; this collector parses `storagePools[]`). This is the array-health and scrub signal — RAID
degraded/rebuild progress and btrfs scrub state — which on this DSM lives in the pool object (recon: there
is NO separate `Scrubbing` API; scrub state = `storagePools[].scrubbingStatus` + `progress.percent`).

**Vertical slice:** after this stage `homelab_synology_pool_*` (RAID status, normal/designed disk count,
rebuild `progress.percent`, `scrubbingStatus`, unverified-disk attention) is emitted for the live pool
(`reuse_1`, RAID6, 8/8 disks, scrub `ready`).

**Spec references:** §6.2. EPIC-008 "Verified deployment reality" (pool/scrub shape — scrub in the pool
object). EPIC-007 STAGE-007-005 (parsing a slice of a fat response) is the precedent.

## Concrete deliverables

### 1. Pool & RAID collector

`pool.py`: parses `storagePools[]`, emits:
- `homelab_synology_pool_status{pool,status}` (1/0 per state incl. `has_unverified_disk`),
  `_disk_failure_number{pool}`, `_writable{pool}`.
- RAID: `homelab_synology_raid_status{pool,raid}` (1=normal from `raidStatus`),
  `_normal_disk_count{pool}` / `_designed_disk_count{pool}` (degraded when normal < designed),
  `_has_parity{pool}`, `_crashed_reason{pool}`.
- Rebuild/resync: `homelab_synology_pool_progress_percent{pool}` (-1 = idle), `_progress_step{pool}`.
- Scrub: `homelab_synology_pool_scrub_status{pool,state}` (from `scrubbingStatus`),
  `_scrub_can_do_manual/_schedule` flags; scrub last-done/next-schedule if present.
- Unverified-disk attention flag (drives the info-severity `SynologyUnverifiedDisk` alert).

### 2. Tests

3a fixture (the recon pool shape: RAID6, 8/8, `scrubbingStatus=ready`, `has_unverified_disk`,
`progress.percent=-1`) + a synthetic degraded/rebuilding fixture; 3b live emission.

## Design Phase

**Status**: [x] Design (Complete)

**Design Notes** (selected 2026-06-23; brainstormer used for the real architectural choices, recommendation adopted):

Grounded against the live `storagePools[]` payload (`/tmp/synology_load_info_raw.json`). The stage card's field-name guesses were a prior; several were corrected against ground truth (RAID fields are nested in a `raids[]` LIST, not on the pool object; `is_writable` not `writable`; scrub flags in `data_scrubbing`; `progress.percent` is a string "-1").

Metric design (all mirror the STAGE-008-005 storage collector pattern: BaseCollector shape, `_shared.py` helpers, CappedEmitter cap-routing, concrete-writer test scaffolding):

- **Pool status (state-set, double-emit like 005's volume):** `homelab_synology_pool_status{pool,status}=1.0` for BOTH top-level `status` AND nested `space_status.status`.
- **Scrub status (state-set):** `homelab_synology_pool_scrub_status{pool,state}=1.0` from top-level `scrubbingStatus` (label key `state`).
- **Pool scalars:** `homelab_synology_pool_disk_failure_number{pool}` (int via `as_float`), `homelab_synology_pool_writable{pool}` (from `is_writable` via local `_bool_to_gauge`).
- **Unverified-disk attention (dedicated boolean gauge):** `homelab_synology_pool_unverified_disk{pool}` = 1.0 if `status=="has_unverified_disk"` OR `space_status.show_attention is True` OR `space_status.show_danger is True`, else 0.0. Emit-always. (Stable forward-contract for the future info-severity `SynologyUnverifiedDisk` alert in stage 021/022; the state-set obs remains for diagnostics.)
- **RAID (per-raid, `raid` label = `raidPath` e.g. `/dev/md2`; pool can have multiple `raids[]` sets):** `homelab_synology_raid_status{pool,raid}` (mapped 1→1.0 normal, else 0.0, from int `raidStatus`), `homelab_synology_raid_normal_disk_count{pool,raid}` (`normalDevCount`), `homelab_synology_raid_designed_disk_count{pool,raid}` (`designedDiskCount`; degraded when normal<designed), `homelab_synology_raid_has_parity{pool,raid}` (bool `hasParity`), `homelab_synology_raid_crashed_reason{pool,raid}` (raw int `raidCrashedReason` via `as_float`, 0=no crash, emit-always).
- **Rebuild/resync:** `homelab_synology_pool_progress_percent{pool}` (`progress.percent` string via `as_float`, emit literally; -1=idle), `homelab_synology_pool_progress_step{pool,step}=1.0` (state-set from string `progress.step`).
- **Scrub flags & timing:** `homelab_synology_pool_scrub_can_do_manual{pool}` / `homelab_synology_pool_scrub_can_schedule{pool}` (from `data_scrubbing.can_do_manual`/`.can_do_schedule` via `_bool_to_gauge`); `homelab_synology_pool_scrub_last_done_timestamp{pool}` / `homelab_synology_pool_scrub_next_schedule_timestamp{pool}` (epoch ints via `as_float`, emit literally incl. 0 — the alert layer owns never-scrubbed interpretation).

Helper decision (YAGNI / rule-of-three NOT met — 006 is only consumer #2): COPY the navigators (`_as_dict`/`_as_list_of_dicts`/`_nested`) + `_bool_to_gauge` LOCAL into `pool.py`; add a local `_raid_status_to_gauge` (1→1.0/else→0.0/non-int→None). Do NOT extract to `_shared.py` and do NOT touch 005's committed file. Revisit extraction at stage 009 if it becomes the 3rd consumer.
