# STAGE-008-016: Surveillance events & recordings collector

## Status: Complete

## Overview

Collect Surveillance Station activity: per-camera event/motion counts (`SYNO.SurveillanceStation.Event
CountByCategory` — recon: per-camera + per-day breakdowns, e.g. Driveway 745 / Backyard 804 / Doorbell 1097)
and recording totals/size (`SYNO.SurveillanceStation.Recording List` — 2645 clips). Kept as AGGREGATE
counters only (never per-event series). This feeds the "camera online but not recording / went silent"
integrity signal (STAGE-008-022).

**Vertical slice:** after this stage `homelab_synology_ss_events_total{camera}` + `_events_today` +
`homelab_synology_ss_recordings_total` + `_recordings_bytes{camera}` are emitted for the live SS at 5m.

**Spec references:** §6.2. EPIC-008 "Verified deployment reality" (Event.CountByCategory per-camera/per-day;
Recording.List total/size). EPIC-008 cardinality decision (SS events as aggregate counters only).

  recording footprint).

plus a "camera dropped to 0 events" synthetic case; 3b live emission.

## Design Phase

**Status**: [x] Design (Complete)

**Design Notes** (2026-06-26):

**Selected approach: Option A — full-list faithful, aggregate-only (mirror cameras.py exactly).** Brainstormer (Opus) presented 3 options grounded in the LIVE captured payloads; Option A selected (backend collector, user pre-authorized design recommendation). Rationale: delivers every carded metric exactly (Option B dropped `_recordings_bytes{camera}`; Option C made it silently-sampled), per-camera bytes REQUIRES iterating the recording list anyway (so a limit buys nothing without dropping the metric), cost is a non-issue at homelab scale (2645 dicts ≈ sub-ms parse on a 5m cadence), mirrors the just-built cameras.py exactly (lowest review friction, no client-helper churn), and the seeded system rollups let STAGE-008-022's integrity rule alert.

**Collector:** new `apps/monitor/homelab_monitor/plugins/collectors/integrations/synology/events.py` — `SynologyEventsCollector(BaseCollector)` (name `synology_events`, interval 300s/5m per the epic cadence "SS-events 5m", timeout 30s, concurrency_group "synology"). Co-equal 2-fetch `ss_event_count_by_category()` + `ss_recording_list()` (both helpers + their scaffolding-parametrize tuples ALREADY exist — NO client.py / test_synology_client.py change). `ok = event_resp is not None or rec_resp is not None`. Per-camera / per-recording parse isolated. Mirrors cameras.py (`_Built` slots + seeded baselines, `_fetch` wrapper, `_emit` family() closure, run() skeleton, metrics_emitted accounting). Uses `_shared.py` helpers (as_dict/as_list_of_dicts/nested/as_float/bytes_field/capped_emitter/cap_for_synology/fetch_or_result).

**Metric set (Option A) — AGGREGATE COUNTERS ONLY (privacy lock; NEVER per-event/per-recording series):**
- Per-camera all-time events (emit-on-presence, label `camera`=name): `homelab_synology_ss_events_total{camera}` — from `evt_cam["0"]` per-camera composite key `"<id>-<name>"` (the `"-1"` group-total key SKIPPED; split on FIRST `-` via `split("-", 1)` since names can contain hyphens; label `camera`=name, fall back to id).
- System rollups (SEEDED 0.0, always-emit — alertable): `homelab_synology_ss_events_today` (from `date[<today-LOCAL>]["-1"]`, system-wide), `homelab_synology_ss_events_total_all` (the grand `total` / `date["-1"]`), `homelab_synology_ss_recordings_total` (from the cheap recording-list top-level `total` field — NO list iteration needed for the count), `homelab_synology_ss_recordings_bytes_total` (sum of all sizeByte, system-wide).
- Per-camera recordings (emit-on-presence, label `camera`=cameraName): `homelab_synology_ss_recordings_count{camera}` (count per cameraId from iterating the list), `homelab_synology_ss_recordings_bytes{camera}` (sum sizeByte per camera via bytes_field).

**Card-vs-reality deltas (LIVE capture 2026-06-26 → /tmp/synology_ss_event_count_raw.json + /tmp/synology_ss_recording_list_raw.json — Lesson #3):**
- `Event CountByCategory` is NOT by event-type category. It segments by CAMERA (`evt_cam["0"]`, composite `"<id>-<name>"` keys) and by DATE (`date["YYYY/MM/DD"]={"-1":dayTotal,"am":n,"pm":n}`). NO motion/alarm/manual categories. NO per-camera-per-day cross-dimension (per-camera is all-time only; per-day is all-cameras only). So `_events_total{camera}` is all-time-per-camera and `_events_today` is system-wide — they live in DIFFERENT dimensions; emit both honestly, do NOT invent a per-camera-today.
- `Recording List` returns ALL 2645 records (no pagination), each with `cameraId`(1/2/3), `cameraName`, `sizeByte`(BYTES), `id`, `filePath`, `height`/`width`, codecs, `locked`(bool). Aggregate (count + sum bytes per camera); NEVER emit per-recording. The top-level `total` gives the count cheaply without iterating.

**THREE load-bearing Build requirements (beyond a plain cameras.py copy):**
1. The `"-1"` skip + composite-key `split("-", 1)` (names can contain hyphens → split-once; reject non-int counts in isolation; do NOT invent event-type categories).
2. **today-key TIMEZONE:** `date[<today>]` MUST be keyed in the NAS-local / DISPLAY TZ (`America/New_York`), NOT raw `datetime.now(UTC)` — else `events_today` reads the wrong/missing key for ~4h nightly. This is the ONE place the all-internal-UTC rule yields to a foreign-system key match (document in the module docstring). When the computed key is absent (NAS hasn't logged today yet), `events_today` stays at its seeded 0.0 — correct + alertable.
3. `as_float` rejects bool — feed `sizeByte` to `bytes_field`, never feed `locked` to as_float. Document the tiny-`sizeByte` caveat (max ~1384 B live — likely manifest/stub files, may not reflect actual storage) in the docstring; emit as-is.

**SCOPE-OUTS (documented, not silent):** am/pm event split (YAGNI — no downstream consumer; no owning stage); per-camera-per-day events (does NOT exist in the DSM API — never card it); the `limit`/`offset` param on `ss_recording_list` (not needed under Option A — the full list is required for per-camera bytes anyway; a future high-volume stage would add it if ever needed). NO metric-name collision with STAGE-008-015 (`ss_camera_*`/`ss_cameras_*`/`ss_info_*`) or STAGE-008-017 (`ss_license_*`/`ss_homemode_*`) — this stage owns `ss_events_*`/`ss_recordings_*`.

**Downstream:** STAGE-008-022 (vmalert integrity rule) consumes these gauges — it joins per-camera `ss_camera_connected{camera}` (015) against `ss_recordings_count{camera}` / `ss_events_total{camera}` (016) on the `camera` label, plus the seeded `events_today` system gauge, to fire "camera connected but not recording / went silent". The seeded always-emit system rollups are what 022 alerts on.

**Expected metrics_emitted for the live 3-camera fixture: 18** (9 per-camera series across 3 on-presence families [events_total, recordings_count, recordings_bytes] + 4 seeded system rollups + 7 family-increments = 16, + 2 api_took). Confirm in the Build acceptance test.
