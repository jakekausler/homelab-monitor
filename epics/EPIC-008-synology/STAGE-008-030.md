## Design Phase

**Status**: [x] Design (Complete)

**Design Notes**:

Selected approach (Design phase, 2026-06-28; brainstormer Opus presented options, recommendations selected; validated against existing pihole.json + homelab-unifi.json):

- **Panel mix (Decision A → A1): stat-glance + table-detail + minimal timeseries.** Mirror the house pattern in deploy/grafana/dashboards/pihole.json + homelab-unifi.json (schemaVersion 40, datasource uid "victoriametrics", row panels type:"row"). Scalar/binary signals = stat panels; per-instance data (8 disks sda-sdh, 9 /rackstation mounts, 1 volume, 14 snapshot shares, 5 security severities) = TABLES with an "organize" transform (one row per instance, columns for temp/status/etc) — NOT barcharts. Genuinely time-varying scalars (cpu_load1/5/15, disk/sys temps) = timeseries. NO BARCHARTS: neither reference dashboard uses one, and the barchart x-axis tax (first-STRING-field + format:table + organize + xField + label_replace) cost 4 attempts a prior epic (project_grafana_barchart_category_axis.md); tables are the robust idiom for per-instance label data.

- **Glance-first (Decision B → B1): a top non-collapsed "Synology — Alerts at a glance" stat strip** above the 7 collapsible rows, so critical 0/1 signals stay visible even when detail rows are collapsed. Inverted thresholds: health_ok=0 → red "ALERT" (LIVE health_ok=0!); dsm_update_available=1 → red (LIVE 1); no_backup_configured=1 → red (LIVE 1); pool has_unverified_disk → warn (LIVE present); security_status=0 → green "Good" (LIVE 0). The same metrics also appear in their detail rows (acceptable duplication for glance-first).

- **7 collapsible rows** (mirroring the card): (1) health & storage (health_ok, volume_used_percent/bytes, pool_status, need_reboot, sys_temp, dsm_update), (2) disks & SMART (per-disk temp/status/smart_status/utilization/unc_count table), (3) system & UPS (cpu load/percent, mem%, ups_charge/status/on_battery), (4) backup & replication (no_backup_configured, replication_available, snapshot_count per-share table, backup counts), (5) security (security_status/safe, findings_total per-severity table, last_scan_age), (6) mount-health (mount_up/free_bytes/total_bytes per-mount table, 9 mounts), (7) SSH-probe (ssh_up + last_success_age filtered to target="synology").

- **Honest empty/edge states (project ETHOS), against LIVE reality:** health_ok=0 reads as ALERT (not green); disk_remain_life is LIVE -1 on ALL 8 disks (unsupported/not-reported) → value-mapping -1 → "n/a" + noValue "—" (NEVER "0% life"); ssh_uptime_seconds has ONLY a target="udm" series (NO synology series) → OMIT a synology uptime panel; use ssh_last_success_age_seconds{target="synology"} (LIVE 0s) instead. ALL synology ssh panels MUST filter target="synology" (a udm target also exists on these metrics).

- **Time range + slow-metric handling (Decision C → C1): default `now-6h`** + the existing lastNotNull/instant:true stat reducer keeps the hourly (3600s) collectors (dsm_update_available, security_*) populated. Both references use now-1h; bump to 6h to comfortably span an hourly scrape. `last_over_time([2h])` held in RESERVE — apply only to a specific panel if it is observed blanking in Refinement (the stale-read lesson REG-008-027-STALE-READ).

- **The `.tsx` Metrics-tab embed (global Metrics page, NOT a Synology-page tab — per the locked scope correction):** new apps/ui/src/routes/metrics/MetricsSynologyTab.tsx using `GrafanaDashboardEmbed src="/api/grafana/d/synology/synology?kiosk"` (mirroring MetricsPiholeTab); add `{ path: '/metrics/synology', label: 'Synology' }` to the TABS array in apps/ui/src/routes/metrics/MetricsLayout.tsx; add metricsSynologyRoute (path 'synology') to router.tsx + addChildren. Dashboard uid = "synology".

- **TWO-PART GATE PATH:** (1) the `deploy/grafana/dashboards/synology.json` is provisioned-as-code (the provider watches the dir, 30s updateInterval) → REFRESH-ONLY: no image rebuild, no re-provision, the user just refreshes Grafana (feedback_grafana_json_refresh_only.md). (2) the `.tsx` Metrics-tab embed is baked into the monitor image → REQUIRES rebuild + sha256-prove served==built + Desktop + Mobile viewport sign-off on the dashboard tab. So Refinement has a JSON-validation half (subagent: dashboard loads in Grafana, panels populate against live data — the readability pass) + a frontend half (viewport sign-off of the embedded tab).

- **Readability pass (card requirement, no silent omission):** every emitted Synology + ssh-probe + mount metric is surfaced on a panel. Verified the live metric surface (all 7 rows have live series). Any metric that turns out empty/edge gets honest handling (the disk_remain_life=-1 / empty-synology-uptime cases above), NOT silent omission.

- **Refinement plan:** drop synology.json → Grafana auto-provisions (30s) → a subagent validates the dashboard loads + every panel populates against live data (the readability iteration; no rebuild). THEN rebuild the prod image (--no-cache) for the .tsx tab embed + sha256-prove + PROMPT the user for Desktop + Mobile viewport sign-off of the Synology tab on the global Metrics page at http://192.168.2.148:29090. Regression items added in Refinement.
