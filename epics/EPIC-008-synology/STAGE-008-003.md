# STAGE-008-003: Cardinality-cap reuse hook + shared parse / typed-error helpers

## Status: Complete

## Overview

Establish the shared helper layer every Synology collector will use, so Waves B/C/E don't each re-derive it:
(1) wire the reusable cardinality cap (EPIC-005-004) for the `homelab_synology_*` metric families as a
guardrail, and (2) extract `_shared.py` helpers for the common collector mechanics — calling a typed
`SynologyClient` GET, turning a typed `SynologyError`/DSM-error-code into a failed `CollectorResult`,
emitting `homelab_synology_api_took_seconds{api}` from each response, and the `ok=True`-even-when-NAS-sad
convention. This keeps each later collector a thin, single-purpose file.

**Vertical slice:** after this stage `_shared.py` exposes the helper(s) a collector calls to (a) fetch +
parse a DSM response with uniform typed-error→failed-run handling, (b) emit the API-latency metric, and (c)
respect the cardinality cap; a unit test exercises each helper against a fake client.

**Spec references:** §5.4 (concurrency + failure budget), §6.2 (metric naming + cardinality). EPIC-005
STAGE-005-004 (cardinality cap) + STAGE-005-007 (the `_shared.py` extraction precedent). EPIC-006/007
`_shared.py` are the most recent copies.

## Cross-cutting standards (from EPIC-008 cross-stage acceptance criteria)

- Reuses the EPIC-005-004 cardinality cap (no new cap implementation) — guardrail for the naturally-bounded
  Synology series.
- The `homelab_synology_api_took_seconds{api}` emission is centralized here so every collector gets the free
  latency signal uniformly.
- Typed-error → failed-run mapping is uniform; `ok=True` when the NAS is sad but the probe itself succeeded;
  `ok=False` only when the client errors.

## Concrete deliverables

### 1. Cardinality-cap reuse hook

Wire the EPIC-005-004 cap for the `homelab_synology_*` families (default cap as a guardrail; the bounded
series — 8 disks, 1 volume/pool, 3 cameras, ~17 packages — are well under it). Document the cap is a safety
net, not an expected-active limit.

### 2. `_shared.py` helpers

Helpers for: a typed `SynologyClient` GET + parse with uniform `SynologyError`/DSM-error-code →
failed-`CollectorResult` mapping; `homelab_synology_api_took_seconds{api}` emission; small parsers reused
across collectors (e.g. DSM enum-string → bool/state, byte/percent helpers). Mirror `_shared.py` in the
HA/Pi-hole/Unifi bundles.

### 3. Tests

Unit tests for each helper against a fake `SynologyClient` (success, typed-error reasons, DSM-error-code
mapping, latency emission, cap behavior).

## Design Phase

**Design Notes** (decided 2026-06-23):

Selected approach: **Option B (trimmed)** — a thin `_shared.py` that centralizes the seams the card names, scoped by YAGNI to what the immediate next collectors (005 storage, 009 UPS/health) actually need.

`_shared.py` helpers to BUILD this stage:
- `client_unconfigured_result(start) -> CollectorResult` — `ctx.synology is None` → ok=False, errors=["synology client not configured"].
- `failed_result(error: SynologyError, start) -> CollectorResult` — typed-error → ok=False, errors=[error.message]. (Centralizes the error→failed-run mapping that HA/Pihole/Unifi do inline — card-specified divergence.)
- `fetch_or_result(ctx, response: SynologyResponse | SynologyError, start, emitted: list[int]) -> SynologyResponse | CollectorResult` — on error returns `failed_result`; on success emits `homelab_synology_api_took_seconds{api=<endpoint>}`, bumps `emitted[0]`, returns the response. (Centralizes the took-emission the card asks for; Unifi/Pihole emit it inline.)
- `as_float(v) -> float | None`, `bytes_field(v) -> float | None`, `percent_field(v) -> float | None` — universal numeric parsers (disk temp/capacity, UPS charge/load percents).
- `capped_emitter(ctx, events) -> CappedEmitter` + `cap_for_synology(family) -> int` — thin wrappers over `CappedEmitter(writer=ctx.vm, events=events)` and `load_cardinality_caps_config().cap_for(family)`. 005 (per-disk/per-volume families) is the cap consumer.

Cardinality-cap hook decision: **default-500, NO registration.** Unlisted families auto-get `CardinalityCapsConfig.default` (=500) via `cap_for()`; the verified-tiny Synology series (8 disks, 1 vol/pool, 3 cameras, ~17 packages) are far under 500, so 500 is an honest runaway guardrail. NOT adding `homelab_synology_*` entries to `_DEFAULT_CARDINALITY_FAMILIES` — a tighter cap (e.g. 50) would be a false guardrail that silently drops legitimate series if the homelab grows. The "hook" is satisfied by collectors routing through the `capped_emitter`/`cap_for_synology` helpers (same pattern as Unifi `client_stats`). Documented as a safety net, not an expected-active limit.

DEFERRALS (no silent drop):
- Generic enum→gauge-value mapper (`state_to_value(mapping=...)`): DEFERRED to STAGE-008-005 (storage; it knows the exact per-disk `smart_status`/status enum shape) and STAGE-008-009 (UPS/SystemHealth status enums). Each builds the CONCRETE parser it needs against its known field values, not a speculative generic mapper here. Reason: YAGNI + 100% branch coverage (concrete parsers are trivially coverable; a generic mapping-driven one invites untested branches).
- Whole-lifecycle collector template (`run_simple_synology_collector`): NOT built. Revisit only if 3+ collectors prove identical payload shape (earliest realistic: after 007/008). The 13 collectors are heterogeneous (005 dual-family, 011 snapshot/replication, 013 connections, 015-017 surveillance) — a single template would fight the non-uniform ones.

CARD DEVIATIONS (card is a strong prior, not ground truth — verified vs real code):
- The card says "Mirror `_shared.py` in the HA/Pi-hole/Unifi bundles." Pi-hole has NO `_shared.py` (only `_parsing.py`); HA `_shared.py` is the real structural model. Drift noted; using HA as the precedent.
- Centralized took-emission + centralized error→`CollectorResult` go BEYOND what HA/Unifi/Pihole extracted (they do both inline). This is intentional and card-specified (the card explicitly names both as deliverables) — a justified divergence from the inline precedent, not a mistake.

3b (live) decision: `_shared.py` has no independently live-testable surface beyond a trivial direct call; the live exercise is DEFERRED to STAGE-008-005 (the first real collector, which calls these helpers against the live DSM). Named here, not a silent drop. Refinement for 003 = 3a only (full unit coverage against a fake `SynologyClient` + fake `InMemoryMetricsWriter`: success / each typed-error reason / DSM-error-code mapping / latency emission / cap behavior / None-client guard).

## Refinement Phase

**Refinement Notes** (2026-06-23):

3a (realistic-integration exercise, via e2e-tester) — 42/42 checks PASS, zero issues. The existing unit suite (28 tests) was confirmed green, then a throwaway script drove the REAL `_shared.py` helpers through a realistic collector lifecycle against a fake `SynologyClient` + real `MetricsWriter`:
- Happy path (storage_load_info-shaped payload, 2 vols + 3 disks, mixed int/numeric-string fields): `fetch_or_result` returns the response + emits `homelab_synology_api_took_seconds{api=...}` (=0.137) + increments `emitted`; `bytes_field`/`percent_field`/`as_float` parse real fields to floats, and return None for missing/garbage so a collector can skip emitting those series.
- NAS-sad-but-reachable (volume status "has_unverified_disk"): shared layer is payload-agnostic → returns the response (NOT a CollectorResult), so the collector emits ok=True. The `ok=True`-when-sad convention is enforced at the correct layer.
- Error paths (`unreachable` no-status, `api_error` DSM code 105, plus `failed_result`/`client_unconfigured_result` direct): all → ok=False CollectorResult, errors=[message], metrics_emitted=0, NO gauge, `emitted` unchanged.
- Cardinality cap (realistic): `cap_for_synology("homelab_synology_disk_temp_celsius")`=500 confirmed at runtime; 8 obs (under cap) → all 8 survive + `homelab_metric_family_dropped_series`=0 + no SuggestionEvent; 600 obs (over cap) → exactly 500 survive, dropped=100, 1 SuggestionEvent(severity="warning"). The guardrail actually guards. Survivor set deterministic (sorted label-set order).
