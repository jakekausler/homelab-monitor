# STAGE-008-019: Alertmanager inhibit-rules — mount-down suppresses dependent ContainerDown

## Status: Complete

## Overview

Wire the cross-signal suppression the epic's "mount-health before container probes" criterion requires: when
`homelab_synology_mount_up == 0` (a `/rackstation` mount is down — the root cause from STAGE-008-018), the
dependent EPIC-003 `ContainerDown` alerts for containers that mount `/rackstation` are INHIBITED, so the
mount-down alert surfaces as the single root cause instead of a storm of expected container-sad alerts. This
uses Alertmanager's standard inhibit-rules (the proper home for cross-signal suppression), not custom code.

**Vertical slice:** after this stage, with a `/rackstation` mount down, `SynologyMountDown` fires and the
dependent `ContainerDown` alerts are suppressed (verified via Alertmanager); when the mount is up, container
alerts behave normally.

**Spec references:** EPIC-008 cross-stage criterion + Amendment 8. EPIC-003 (Docker container alerts —
the inhibited target). Alertmanager inhibit-rules.

## Cross-cutting standards (from EPIC-008 cross-stage acceptance criteria)

- Suppression via Alertmanager inhibit-rules (declarative config), not bespoke code.
- The inhibit source (`SynologyMountDown` / `mount_up==0`) and target (`ContainerDown` for
  `/rackstation`-dependent containers) are matched on shared labels — Design defines the matching labels
  (e.g. a `depends_on_mount` / `mount` label, or the container→mount mapping).
- Validated against real Alertmanager behavior (a down mount actually inhibits the dependent container
  alerts; an unrelated container alert is NOT inhibited).

## Concrete deliverables

### 1. Inhibit-rule config

Add an Alertmanager inhibit-rule (in the deploy config) so `SynologyMountDown` (source) inhibits
`ContainerDown` (target) for the dependent containers. Design defines how dependency is expressed (a label on
the container alert / a static mapping of which containers depend on `/rackstation`).

### 2. Dependency mapping

Identify (live, in instance A) which Docker containers mount `/rackstation/*` and encode the inhibit match
(by container label or an explicit list). Document the mapping.

### 3. Tests / validation

3a: config-level golden test of the inhibit-rule YAML (mirrors the vmalert golden-test discipline). 3b
(instance A): real Alertmanager — a down mount inhibits the dependent ContainerDown; an unrelated alert is
not.

**DECISION 3 — Dependency expressed as a COARSE boolean flag (not per-mount `equal:`):** The two target alerts (`PlexContainerDown`/`PodcastFeedContainerDown`) carry a `depends_on_rackstation: "true"` label. The inhibit rule: `source_matchers: [alertname="SynologyMountDown"]`, `target_matchers: [depends_on_rackstation="true"]`, NO `equal:`. Rationale: per-mount `equal: [mount]` is STRUCTURALLY IMPOSSIBLE (plex mounts MANY /rackstation shares; one alert can't carry multiple mount values for `equal:`), AND a /rackstation problem at homelab scale is realistically NAS-/NFS-layer-wide, so "any rackstation mount down → all rackstation-dependent containers suspect" is the CORRECT model, not a compromise. Target alerts modeled on `container_lifecycle.yaml`/`PiholeContainerDown` (`absent(running) or state=~"exited|dead|restarting"`, `name="plex"`/`"podcast-feed"`) carrying `severity: critical, source_tool: vmalert-metrics, target_kind: docker_container, integration: docker, depends_on_rackstation: "true"`.

**DECISION 4 — Testing (nothing fires on real data in prod):** (a) promtool golden tests in `__tests__/synology.tests.yaml` for all 3 rules (fire on `mount_up==0` / `absent`+`exited` container states; no-fire on healthy).
