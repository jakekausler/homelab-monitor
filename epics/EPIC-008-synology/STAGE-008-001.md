## Overview

The first foundation stage of the Synology integration bundle. Build a real Synology **DSM v7** REST client
(`SynologyClient`) and wire it into the collector context so every Wave-B/C/E collector can reach the NAS.
DSM v7 uses session auth (`SYNO.API.Auth` version 7 â†’ `sid`); all calls route through `/webapi/entry.cgi`
over a self-signed HTTPS endpoint. Establish the single-credential, observe-only model (one admin DSM
password for read-only collection) and the `verify=False` self-signed transport.

**Vertical slice:** after this stage a smoke test (and the collector status surface) shows the monitor
authenticating to DSM v7 (login â†’ `sid`, **no `session=` param**) with the password from the secret store
and successfully calling `SYNO.Core.System` v3 `info` (model/firmware/uptime/`sys_temp`).

## Cross-cutting standards (from EPIC-008 cross-stage acceptance criteria)

- DSM password NEVER logged (assert in tests); read per-request from the TTL-cached resolver, never stored
  on the client.
- ONE reused session: login once â†’ store `sid`; attach `_sid` to every call; re-auth on **error 119**
  (session expired); back off on rate-limit; logout on shutdown to free the slot. Do NOT auth-per-call.
- Self-signed TLS â†’ `verify=False` (pin `CN=synology`).
- DSM 5xx/timeout/119 never propagates as our 5xx â€” the client surfaces a typed error the collector turns
  into a failed run (failure budget takes over).
- Observe-only â€” the client exposes ONLY read GET helpers; no write methods.
- All internal timestamps UTC.

## Concrete deliverables

### 1. Real Synology DSM v7 client

A new `SynologyClient` (Design picks the exact module path â€” mirror `kernel/pihole/` + `kernel/unifi/`; keep
any Protocol in `plugins/io.py`, impl separate). Likely a DEDICATED `httpx.AsyncClient` with `verify=False`
(like Unifi's self-signed client), NOT the shared pool. Responsibilities:
- **Session management:** login via `SYNO.API.Auth` version 7 (`method=login`, `account`, `passwd`,
  `format=sid`, **NO `session=` param** â€” sending one returns error 402, recon-confirmed) â†’ store
  `self._sid`; attach `_sid=<sid>` to every subsequent `entry.cgi` call; on **error 119** re-auth once
  (single-flight, `asyncio.Lock`) and retry; `method=logout` on shutdown (`aclose()`, best-effort,
  errors suppressed).
- **Typed GET helpers** (return parsed models, not raw JSON) establishing the client surface Wave B/C/E
  need: `system_info()`, `system_utilization()`, `system_health()`, `hardware_fanspeed()`,
  `need_reboot()`, `storage_load_info()`, `ups_get()`, `upgrade_check()`, `package_list()`,
  `package_server_list()`, `backup_task_list()`, `backup_repository_list()`, `share_snapshot_list(name)`,
  `security_scan_status()`, `current_connection_list()`, plus the SS family (`ss_info()`,
  `ss_camera_list()`, `ss_event_count_by_category()`, `ss_recording_list()`, `ss_license()`,
  `ss_homemode()`, `ss_log_list()`). Only `system_info()` (+ auth/session machinery) is exercised THIS
  stage; the rest are SCAFFOLDING (`# SCAFFOLDING: consumed in STAGE-008-0XX`).
- **`took`/latency capture:** expose each response's latency so the API-latency metric can read it (emit
  `homelab_synology_api_took_seconds{api}` â€” wired in Wave B; client exposes the value here).
- **Typed error mapping:** `reason: Literal["unreachable","timeout","auth","http_error","bad_response",
  "rate_limited","api_error"]` (`api_error` carries the DSM error code, e.g. 119/105/402) + message +
  optional status/code. Map DSM `{"error":{"code":N}}` envelopes to typed errors (119â†’re-auth, othersâ†’
  api_error).

**Status**: [x] Design (Complete)

**Design Notes**:

**Design completed.** Approach: a `SynologyRestClient` mirroring the verified Pi-hole (session-auth) + Unifi (self-signed `verify=False`, monotonic latency) integration-client precedents, adapted for DSM v7's single-CGI + JSON-error-code model. Stage-consistency check PASSED (description â†” deliverables coherent; no copy-paste artifacts). Brainstormer (Opus) consulted for the six novel DSM-specific decisions; recommendations selected as-is (Design autonomy â€” no user prompt).

**Decisions selected (D1â€“D6):**
- **D1 â€” Request construction:** a single private `_get(api, version, method, params=None) -> SynologyResponse | SynologyError` that builds `/webapi/entry.cgi` and injects `api`/`version`/`method` + `_sid` query params; the ~20 typed helpers are branchless one-line wrappers. `endpoint` label = `"{api}/{method}"` for the latency metric. (DSM has ONE path varying only by query params â€” Unifi's prefix-builder split has nothing to split.)
- **D2 â€” Session/re-auth:** Pi-hole single-flight pattern (`self._sid`, `asyncio.Lock`, `_ensure_session(failed_sid)` double-check, `_login()` only under lock, retry-once), BUT the re-auth trigger is **DSM body error code 119** (session expired) checked AFTER JSON parse / BEFORE wrapping â€” NOT HTTP 401 (DSM returns HTTP 200 with `{"error":{"code":119}}`). Second 119 â†’ `auth` error. Real HTTP non-2xx still â†’ `http_error`.
- **D3 â€” Error model:** `SynologyErrorReason = Literal["unreachable","timeout","auth","rate_limited","http_error","bad_response","api_error"]` (adds `api_error` vs the 6-reason precedent). Frozen `SynologyError(reason, message, status: int | None = None)`; `status` carries the DSM error code for `api_error` / DSM-derived `auth`. Mapping: connectâ†’unreachable; read-timeoutâ†’timeout; HTTP non-2xxâ†’http_error(status); 429â†’rate_limited; non-JSON/bad-shapeâ†’bad_response; DSM 400â†’auth(400); DSM 119 post-retryâ†’auth(119); DSM 105â†’api_error(105, permission â€” SID valid, account lacks rights, do NOT re-auth); DSM 402â†’api_error(402, "should never happen â€” we never send session="); any other DSM code Nâ†’api_error(N). Password never in any message.
- **D4 â€” Timing:** monotonic client-side timer (Unifi pattern; DSM has NO `took` field) around the successful data-returning HTTP call ONLY â€” re-auth round-trips excluded (timer lives in the per-call helper). `SynologyResponse(payload: object, took_seconds: float, endpoint: str)` frozen/slots.
- **D5 â€” Login parsing:** store ONLY `data.sid` (defensive `isinstance` narrowing like Pi-hole's `_parse_session`); IGNORE `synotoken` (it's the CSRF header for write calls; observe-only â†’ YAGNI). Missing/empty sid or `success:false` â†’ `auth`.
- **D6 â€” Version handling:** HARD-CODE the recon-confirmed versions as literal args in each helper (Auth v7, `SYNO.Core.System` v3 method `info`, etc.); NO `SYNO.API.Info` discovery call (single-target homelab monitor; versions pinned by recon; YAGNI). Comment each helper citing the recon doc.

**Wiring (locked):** `SynologyConfig` + `load_synology_config()` in `kernel/config.py` (env `HOMELAB_MONITOR_SYNOLOGY_URL`, default `https://192.168.2.4:5001`, `rstrip("/")`); a DEDICATED `httpx.AsyncClient(verify=False)` (Unifi-style, self-signed `CN=synology`) constructed once in lifespan with `password_provider=lambda: ttl_resolver.current().get("synology_dsm_password")`; new optional `synology: SynologyClient | None = None` field LAST on `CollectorContext` (after `pihole`); `ctx_factory` injects `synology=synology_client`; guarded best-effort `aclose()` (logout `method=logout` + clear `_sid`, errors suppressed) in the shutdown `finally` BEFORE the dedicated http client's `aclose()`. `SynologyClient` Protocol in `plugins/io.py` (return-not-raise, mirror `UnifiClient`).

**Smoke slice:** only `system_info()` (`SYNO.Core.System` v3 `info`) exercised; the ~20 other helpers are SCAFFOLDING tagged `# SCAFFOLDING: consumed in STAGE-008-0XX`.
