# STAGE-008-018: NFS/`/rackstation` mount-health collector

## Status: Complete

## Overview

Build the mount-health collector that validates the monitor host's NFS/SMB mounts from the Synology
(`/rackstation/*`) are alive and responsive — emitting the root-cause signal that the Docker-alert
suppression (STAGE-008-019) consumes. The check must be NON-HANGING (a stale NFS mount blocks naive
`stat`/`statfs` forever), so it runs the probe with a hard timeout and treats a hang as "down." This is the
"mount-health before container probes" criterion: if `/rackstation` is gone, dependent containers are
*expected* to be sad, so this signal is the root cause.

**Vertical slice:** after this stage `homelab_synology_mount_up{mount}` + a mount-probe-latency metric are
emitted for each configured `/rackstation/*` mount at 60s, with a hung mount detected as down (not a wedged
collector).

**Spec references:** §6.2. EPIC-008 cross-stage criterion ("mount-health before container probes"). EPIC-008
Amendment 8 (built in Wave F, before rules/UI/dashboards). EPIC-003 (Docker — the suppression consumer, in
008-019).

## Cross-cutting standards (from EPIC-008 cross-stage acceptance criteria)

- Self-metrics; `ok=True` when a mount is down (data, not probe failure); `ok=False` only if the probe
  mechanism itself errors.
- **Non-hanging:** the responsiveness check runs with a hard timeout (e.g. a thread/subprocess `statfs` with
  cancellation); a hang → `mount_up=0`, never a wedged collector or blocked event loop.
- Configurable mount list (the `/rackstation/*` mounts this host depends on); public default empty.
- All timestamps UTC.

## Concrete deliverables

### 1. Mount-health collector

`mount_health.py`: for each configured mount, perform a timeout-bounded responsiveness probe (e.g.
`os.statvfs` in a cancellable thread, or a bounded subprocess) and emit:
- `homelab_synology_mount_up{mount}` (1 responsive / 0 down-or-hung),
- `homelab_synology_mount_probe_seconds{mount}` (probe latency),
- `homelab_synology_mount_free_bytes{mount}` / `_total_bytes{mount}` when responsive.

### 2. Config

A `synology_mounts:` (or reuse an existing mount-config) list of mounts to probe; public default empty,
user override declares `/rackstation/*`.

### 3. Tests

3a: responsive mount, down mount, HUNG mount (probe must time out and report down without hanging),
missing-mount. 3b (instance A): real `/rackstation/*` mounts probed on the host.

## Design Phase

**Status**: [x] Design (Complete)

**Design Notes** (recorded 2026-06-26):

**Selected approach:** brainstormer Option set A/C/A/down-as-0 (live-grounded against the real host + prod container). This is a HOST-FILESYSTEM collector (non-hanging mount-liveness probe), NOT a DSM-API collector — different shape from the Wave-E SS collectors.

**LIVE GROUND TRUTH (verified against the real host + prod `homelab-monitor` container):**
- The host has 9 real NFS v3 mounts under `/rackstation/*` (e.g. `/rackstation/Movies`, `/rackstation/Surveillance`, `/rackstation/Media/TV` [nested deeper], `/rackstation/Pictures`, `/rackstation/Documents`, `/rackstation/YouTube`, `/rackstation/Books`, `/rackstation/HomeVideos`, `/rackstation/Roms`) from Synology 192.168.2.4, all proto=tcp/hard/sec=sys.
- **CRITICAL: the prod monitor container CANNOT currently see `/rackstation`** — there is NO bind-mount and the container's `/proc/mounts` has ZERO NFS mounts. A collector inside the container cannot probe `/rackstation/*` until a compose bind-mount lands. The container only has narrow read-only host binds (`/host/etc:ro`, `/host/proc:ro`, `/host-compose:ro`, `/host-crontab-snapshot:ro`) + `/host-ipc:rw` (the documented invariant: only `/host-ipc` is rw).

**DECISION 1 — Compose exposure (IN SCOPE this stage, override-only):** Add `${HM_RACKSTATION_SRC:-/dev/null}:/rackstation:ro,rslave` to the `monitor` service volumes in the PUBLIC `deploy/compose/docker-compose.yml`. The host-override repo sets `HM_RACKSTATION_SRC=/rackstation`; the public default `/dev/null` keeps a generic user's `docker compose up` safe (no rslave-on-missing-source error). `:ro` honors the privilege invariant (statvfs is read-only); `rslave` propagation is REQUIRED because the NFS submounts under `/rackstation` are SEPARATE mounts — a plain bind shows only the empty parent dir. **BUILD MUST EMPIRICALLY VERIFY** that `/dev/null:/rackstation:ro,rslave` is inert (does not error container start); if it errors, fall back to gating the whole volume line behind a compose override fragment (documented fallback).

**DECISION 2 — Non-hanging probe (mountinfo presence-check → bounded statvfs in a dedicated executor, 5s timeout):** Parse `/proc/self/mountinfo` (cheap, NEVER hangs) to get the set of present mounts. For each CONFIGURED mount: if absent → leave `mount_up=0` (no thread spawned). If present → run `os.statvfs(path)` in a collector-owned dedicated `ThreadPoolExecutor(max_workers=4)` wrapped in `asyncio.wait_for(..., timeout=5)`. A present-but-wedged hard NFS mount can leak at most 4 threads (bounded), never starving the shared default pool. The 5s probe timeout is well under the 60s cadence (a healthy LAN statvfs is <50ms). Rationale vs alternatives: a plain `asyncio.to_thread`+`wait_for` on the SHARED pool risks unbounded thread-pool exhaustion over time on a wedged hard mount; a per-cycle subprocess is more moving parts + harder to unit-test.

**DECISION 3 — Config + host.py relationship:** NEW dedicated config field `synology_mounts: list[str]` (default `[]`, open-source-safe), read via `getattr(ctx.config, "synology_mounts", [])`. Do NOT reuse host.py's `extra_mountpoints` (different semantics: host.py = disk-fullness trend on all mountpoints; this = mount LIVENESS + timeout-safe bytes). KEEP host.py untouched — its `homelab_host_disk_bytes` (fullness trend) and this collector's `homelab_synology_mount_{free,total}_bytes` (timeout-safe, alongside the liveness verdict, consumed by 019) are NOT interchangeable.

**DECISION 4 — Missing-vs-down + seeding (Wave-B always-emit):** Configured-but-absent mount → `mount_up=0` (treat absent as down — mandatory: 019 inhibits ContainerDown on `mount_up==0`, and "absence is hostile to alerting"). Seeding model:
- `homelab_synology_mount_up{mount}` — SEEDED 0 per configured mount at cycle start, OVERWRITTEN to 1 only on a successful responsive probe (a fully-failed round still emits 0 per mount). ALWAYS-emitted / alertable.
- `homelab_synology_mount_free_bytes{mount}` / `homelab_synology_mount_total_bytes{mount}` — emit-on-success ONLY (a 0-byte seed would falsely look like a full disk; `mount_up` is the alertable signal, not bytes).
- `homelab_synology_mount_probe_seconds{mount}` — emit on every probe ATTEMPT for a PRESENT mount (responsive OR timed-out), so hang duration is observable. Not emitted for an absent mount (no probe ran; `mount_up=0` captures it).

**FINAL METRIC SET (exact names, per the card):**
| metric | labels | type | classification |
| --- | --- | --- | --- |
| `homelab_synology_mount_up` | `{mount}` | gauge 1/0 | seeded 0 per configured mount → 1 on responsive probe |
| `homelab_synology_mount_probe_seconds` | `{mount}` | gauge (s) | emit on every probe attempt (present mounts) |
| `homelab_synology_mount_free_bytes` | `{mount}` | gauge (bytes) | emit-on-success only |
| `homelab_synology_mount_total_bytes` | `{mount}` | gauge (bytes) | emit-on-success only |

**Collector shape:** a `BaseCollector` subclass (file + exact name/registration to be confirmed by Build planner against the real registration site — likely `builtin/` or the synology integration dir; metrics use the `homelab_synology_mount_*` prefix per the card+spec since 019's Synology inhibit-rules consume them). `interval=timedelta(seconds=60)`, `timeout=timedelta(seconds=30)`, `concurrency_group="host"`, `run_kind=RunKind.ASYNC`. Emit via `ctx.vm.write_gauge`.

**Consumed downstream:** STAGE-008-019 (Alertmanager inhibit-rules) consumes `mount_up==0` to inhibit EPIC-003 `ContainerDown` alerts (confirmed `homelab_container_status{name,state}` pattern exists in deploy/vmalert).

**TRACKED FOLLOW-UP (no silent deferral):** host.py's `psutil.disk_usage("/rackstation")` has NO timeout protection and will HANG the `host` collector on a wedged NFS mount (a real latent bug). It is OUT OF SCOPE for this stage (this stage adds the timeout-safe liveness probe under a separate metric; refactoring host.py is unrelated). Captured as a regression.md follow-up item in this stage's Refinement naming `builtin/host.py` as the owning file — to be addressed by a future host-collector-hardening stage IF it warrants one (it has not caused a real failure; the host collector's 5s scheduler timeout records a "timeout" failure but the blocking psutil thread still leaks). Do NOT fix host.py here.

**Scope-outs (YAGNI):** no per-mount writability check (the card asks liveness + bytes; `:ro` bind can't validate writability and `:rw` would break the privilege invariant — out of scope, documented rationale). No SMB/CIFS-specific handling (all 9 are NFS v3). No auto-discovery of mounts (explicit configured list only — open-source-safe empty default).
