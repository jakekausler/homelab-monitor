# STAGE-008-012: Updates & packages collector

## Status: Complete

## Overview

Collect update-availability + package-health: DSM update availability (`SYNO.Core.Upgrade.Server` v4
`check` — recon found an update available, `isSecurityVersion:true`), installed packages + running/stopped
status (`SYNO.Core.Package list`), and available package updates (`SYNO.Core.Package.Server list`). These are
standing-advisory signals (info severity per the hybrid philosophy) plus the package-down signal (a critical
package stopped is a warning).

**Vertical slice:** after this stage `homelab_synology_dsm_update_available` (+ `is_security`),
`homelab_synology_package_status{package,state}`, and `homelab_synology_package_update_available{package}` are
emitted for the live NAS at 1h.

**Spec references:** §6.2. EPIC-008 "Verified deployment reality" (Upgrade.Server v4 `check`; Package /
Package.Server shapes).

### 1. Updates & packages collector

`updates.py`: calls `client.upgrade_check()` + `client.package_list()` + `client.package_server_list()`,
emits:
- `homelab_synology_dsm_update_available` (1/0), `homelab_synology_dsm_update_is_security` (1/0),
  `homelab_synology_dsm_update_info{available_version}` (label).
- `homelab_synology_package_status{package,state}` (running/stopped/...), `_package_count`.
- `homelab_synology_package_update_available{package}` (1 per package with an update; from the
  installed-vs-available diff).

### 2. Tests

+ none. 3b live emission.

## Design Phase

**Status**: [x] Design (Complete)

**Design Notes**:

Co-equal graceful-empty multi-fetch collector (mirrors `backup.py`/`replication.py`). New file
`plugins/collectors/integrations/synology/updates.py`. Cadence 1h (interval=3600s), timeout 30s,
`concurrency_group="synology"`. Local `_fetch` wrapper + co-equal combine; always calls `_emit`.

**Live recon (POPULATED ground truth — a DSM update IS available; first non-empty live stage):**
- All 3 client helpers EXIST (no scaffolding gap): `upgrade_check()` (`SYNO.Core.Upgrade.Server` v4 `check`),
  `package_list()` (`SYNO.Core.Package` v1 `list`), `package_server_list()` (`SYNO.Core.Package.Server` v2
  `list`).
- **upgrade_check** payload `{"update": {available: True, version: "DSM 7.3.2-86009 Update 3", type: "nano",
  reboot: "now", restart: "some", version_details: {isSecurityVersion: True, major:7, minor:3, micro:2,
  nano:3, buildnumber:86009, os_name:"DSM"}}}`. `update.available` (bool) = PRIMARY signal;
  `version_details.isSecurityVersion` (bool) = security classifier. No-update shape UNKNOWN (only captured
  the update-available case) → handle absent/falsy/non-dict defensively.
- **package_list** payload `{"packages": [{id, name, version, timestamp, additional:{install_type}}]}` (~17
  pkgs). **NO per-package running/stopped STATUS field and NO per-package update field** — just
  id/name/version/timestamp/install_type.
- **package_server_list** payload `{"packages": [{id, dname, version (LATEST AVAILABLE), is_security_version,
  beta, ...}], "beta_packages":[...], ...}` — the Synology package CATALOGUE.
- Saved: `/tmp/synology_upgrade_raw.json`, `/tmp/synology_package_list_raw.json`,
  `/tmp/synology_package_server_raw.json`.

**Selected design (brainstormer options; coordinator selected per pre-authorization):**
1. **DSM update signal:** `homelab_synology_dsm_update_available` (1.0/0.0 via `bool_to_gauge(update.available)`,
   None→0.0) + `homelab_synology_dsm_update_is_security` (1.0/0.0 via
   `bool_to_gauge(nested(update,"version_details","isSecurityVersion"))`, None→0.0) — BOTH ALWAYS emitted
   (Wave-B: a `>0`/`==1` rule needs the 0 baseline). `homelab_synology_dsm_update_info{available_version,type}`
   identity series emitted ONLY when update_available==1 and a non-empty version string exists. A single
   `as_dict(nested(payload,"update"))` guard covers all no-update shapes (absent key / available:False /
   non-dict). Parse bools via `bool_to_gauge` (NOT `as_float` — it rejects bools by design).
2. **package_status MISMATCH (card-vs-reality):** the card asked for `package_status{package,state}`
   (running/stopped) but the live `SYNO.Core.Package list` has NO status field. **DROP the running/stopped
   state-set** (don't fabricate a state-set with no source). Emit instead
   `homelab_synology_package_info{package,name,version}=1.0` (identity series, `package`=stable `id`) +
   `homelab_synology_package_count` (always-emitted scalar). Package presence/disappearance IS the alertable
   signal (info series goes absent + count delta). **Tracked scope-out (NOT a silent deferral):** a REGRESSION
   item (added in Refinement) names the revisit trigger — "if a status-bearing API (`SYNO.Core.Package status`
   or `additional=[\"status\"]` on the list call) is confirmed via live recon, add the running/stopped
   state-set". Did NOT spend Build budget on the unconfirmed status probe.
3. **Per-package update detection (derived join):** build `{server.id: entry}` from the NON-beta
   `package_server_list.packages[]` (ignore `beta_packages[]`, skip entries with `beta:True`); for each
   installed package, look up by `id`, emit `homelab_synology_package_update_available{package}` = 1.0 if the
   server `version` is a non-empty string AND `!= installed.version`, else 0.0 — ALWAYS per installed package
   (Wave-B 0-baseline; ~17 mostly-0 series, trivial cardinality). A package with no server match → 0.0. Plus
   `homelab_synology_packages_with_updates_count` (always-emitted scalar). STRING inequality, NOT
   version-tuple parsing (YAGNI for a 1h advisory; the non-beta catalogue lists latest-stable so `!=` reliably
   means newer).
4. **Field-name constants (live-VERIFIED this time, higher confidence than 010/011):** centralize
   `_UPGRADE_FIELDS` (available/version/type), `_PKG_FIELDS` (id/name/version), `_SERVER_FIELDS`
   (id/version/is_security_version/beta) per the exemplar; `nested(update,"version_details",
   "isSecurityVersion")` for the security path; `as_dict`/`as_list_of_dicts` navigation; require a non-empty
   `str` id as the package key (mirror `_job_key`'s guard; skip records with no usable id); no label-value
   sanitizing (spaces in "DSM 7.3.2-86009 Update 3" are valid VM label values).

All follow the locked Wave-B alerting philosophy. These are STANDING-ADVISORY signals (info severity per the
hybrid philosophy); STAGE-008-022 owns the consuming alert rules.

**Known scope-out (tracked, NOT a silent deferral):** the card's `package_status{package,state}`
running/stopped state-set is DROPPED — the live `SYNO.Core.Package list` exposes no run-state field. Owned by
a regression item (added in Refinement) naming the revisit trigger (a confirmed status-bearing API). Package
presence is covered by `package_info` + `package_count`.

**Files:** `updates.py` (new) + add `SynologyUpdatesCollector` to `_SYNOLOGY_COLLECTORS` in
`synology/__init__.py` + `tests/test_synology_updates_collector.py` (new) + add
`_EXPECTED_UPDATES_INTERVAL = 3600` + `test_register_all_registers_updates` to
`tests/test_synology_integration_bundle.py`. No migration / openapi / frontend. (All 3 client helpers already
exist — no client.py change needed.)

**Brainstormer:** used (real choices — the package_status data-gap + update-detection join). Coordinator
selected all four recommendations; the package_status scope-out tracked via regression item.
