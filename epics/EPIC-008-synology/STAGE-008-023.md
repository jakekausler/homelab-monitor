## Design Phase

**Status**: [x] Design (Complete)

**Design Notes** (recorded 2026-06-27):

**Approach:** Author vmalert-LOGS alert rules in `deploy/vmalert/logs/synology.yaml` (+ a `deploy/vmalert/logs-test/synology.yaml` mirror with shortened windows for the integration-test harness) over the `service="synology-*"` VictoriaLogs syslog stream forwarded by STAGE-008-020. Rules grounded against the REAL 020-captured/fixture line content. Logs rules use `source_tool: vmalert-logs` (NOT vmalert-metrics), `integration: synology`, `target_kind: nas` (or host — match the existing logs-rule convention), severity per hybrid model, plus an `explorer:` annotation. LogsQL shape: `_time:Nm service:="<bucket>" "<phrase>" | stats by (host) count() as <var> | filter <var>:>N`, with `for:` debounce.

**Rules to author (4 of the 5 card rules; real phrases from 020 fixtures):**
- `SynologyFailedLoginBurst` — `_time:5m service:="synology-auth" "failed to sign in" | stats by (host) count() as fail_count | filter fail_count:>10` — for: 5m — warning. The failure phrase `failed to sign in` is the REAL DSM "Connection" wording (survives redaction; only bracketed identity fields are redacted). The `service:="synology-auth"` filter already scopes to the Connection bucket. Threshold mirrors the existing SshFailedLoginBurst (system.yaml); tunable in Refinement (a NAS baseline is quieter than an SSH host, so >10 is conservative). Live-confirmable in Refinement (failed logins are user-triggerable).
- `SynologySmartEventLog` — `_time:Nm service:="synology-smart" "SMART" | stats by (host) count() as smart_count | filter smart_count:>0` — warning. Keyed on the BUCKET (`service:="synology-smart"`), which is the stable contract the 020 vector parser promises (category-word→bucket). Fixture phrase: "SMART warning detected".
- `SynologyPackageEventLog` — `_time:Nm service:="synology-package" ("updated" OR "installed" OR "failed") | stats by (host) count() as pkg_count | filter pkg_count:>0` — info. Keyed on the `synology-package` bucket. Fixture phrase: "has been updated".
- `SynologyLogErrorRate` — phrase-based over all `synology-*`: `_time:10m service:~"synology-.*" ("fail" OR "failure" OR "critical") | stats by (host) count() as err_count | filter err_count:>20` — for: 10m — info. Phrase-based (NOT severity-based) because DSM emits every line at PRI=14 (severity=info) per 020 — so `syslog_severity_name` is uniformly "info" and useless as a filter. Curated tokens avoid the bare-"error" "no errors" false-positive class. All four numbers are Refinement-tunable placeholders (no live data today).
- (The exact `_time:` windows + thresholds for SMART/package/error-rate are finalized by the planner against the existing logs-rule precedent.)

**logs-test mirror** (`deploy/vmalert/logs-test/synology.yaml`): REQUIRED deliverable (the integration-test harness — test_vmalert_*_logs_pipeline.py — plants synthetic lines via the VL ingest API and runs a vmalert-logs-test container over the mirror rules). Mirror = same rules with `_time:5m`→`_time:30s`, thresholds lowered to fire on planted fixtures (e.g. `>10`→`>2`, `>0` stays), `for:`→`0m`, group `interval:30s`→`5s`. Satisfied-by-reuse of the pihole/unifi/system mirror pattern, but it's a concrete file to author.

**Test mechanism (NOT promtool):** logs rules are validated by pytest INTEGRATION tests (`apps/monitor/tests/integration/test_vmalert_*_logs_pipeline.py`), `@pytest.mark.slow`, requiring `make integration` / the rig. Build's 3a validation = author + add a synology logs-pipeline integration test (or extend the existing pattern) + run it against the rig. There is NO promtool/static golden-test for LogsQL.

**Scope dispositions (no-silent-deferral ledger):**

Fallback re-interpretation noted but NOT auto-adopted: a "successful auth immediately after a failed-login burst" detector is authorable but is a different rule family (and largely duplicates SynologyFailedLoginBurst); not built this stage.
- SMART/package real-category-word risk — **SATISFIED-BY-REUSE of OPEN regression 020#4** (cross-reference, not duplicate). The rules are authored against the stable bucket contract; live correctness of the bucket names depends on 020#4.
- Severity-based error detection (vs phrase-based) for SynologyLogErrorRate — **DEFERRED** to the same 020 live-confirmation: revisit if DSM is ever observed emitting varied syslog severities (currently uniform PRI=14).
