# STAGE-008-010: Hyper Backup collector (graceful-empty + NoBackupConfigured)

## Status: Complete

## Overview

The epic's single most important collector: pull Hyper Backup job status (`SYNO.Backup.Task list` +
`SYNO.Backup.Repository list`) and emit per-job last-run/result/size/next-run so the `BackupFailed` /
`BackupMissing` rules (STAGE-008-022) can fire. False negatives are unacceptable. Recon found HyperBackup
**installed + running but with 0 configured jobs**, so this collector must handle the empty case gracefully
AND emit a distinct `SynologyNoBackupConfigured` signal (running-but-zero-tasks is itself worth knowing).

**Vertical slice:** after this stage `homelab_synology_backup_*` is emitted per job (none live yet) and
`homelab_synology_backup_configured_count` + a `no_backup_configured` signal reflect the empty state; when
the user adds a Hyper Backup job, it lights up with no code change.

**Spec references:** §6.2. EPIC-008 cross-stage criterion ("backup-status is the single most important
rule; test the empty / failed / zero-size-but-ran cases"). EPIC-008 "Verified deployment reality" (Backup.Task
list empty; the per-job field set for when jobs exist).

## Cross-cutting standards (from EPIC-008 cross-stage acceptance criteria)

- `synology` concurrency group; self-metrics + API latency; 5m.
- **False negatives unacceptable** — test the empty case, the failed case, and the ran-but-zero-size case.
- Graceful-empty: 0 jobs → `homelab_synology_backup_configured_count=0` + `no_backup_configured=1`, NOT a
  collector failure.
- All timestamps UTC (last-run / next-run epochs → UTC).

## Concrete deliverables

### 1. Hyper Backup collector

`backup.py`: calls `client.backup_task_list()` (+ `backup_repository_list()`), emits per job:
- `homelab_synology_backup_last_result{job,result}` (success/fail), `_last_run_timestamp{job}` (UTC),
  `_last_run_age_seconds{job}`, `_next_run_timestamp{job}`, `_data_size_bytes{job}`, `_status{job,state}`
  (backuping/waiting/error/...), `_enabled{job}`.
- Aggregate: `homelab_synology_backup_configured_count`, `homelab_synology_no_backup_configured` (1 when
  HyperBackup running + 0 tasks), `homelab_synology_backup_repository_count`.
- **Zero-size guard:** expose `_data_size_bytes` so a "ran but produced 0 bytes" rule is possible
  (STAGE-008-022 owns the rule; this collector ensures the field is present even when 0).

### 2. Tests

3a fixtures: empty (the live state), one-job-success, one-job-failed, ran-but-zero-size, multi-job mixed.
3b live emission (empty/`NoBackupConfigured` path; if the user has added a job by then, validate the populated
path too).

## Design Phase

**Status**: [x] Design (Complete)

**Design Notes**:

Co-equal multi-fetch collector (mirrors `ups.py`): fetches `client.backup_task_list()`
(`SYNO.Backup.Task/list` v1) + `client.backup_repository_list()` (`SYNO.Backup.Repository/list` v1).
Both client helpers already exist (verified — no scaffolding gap). `ok = task_resp is not None or
repo_resp is not None` (ok=False only if BOTH fail); always calls `_emit()`. Local `_fetch` wrapper
(records errors, returns None on failure) copied from `ups.py`. Cadence 5m (interval=300s), timeout 30s,
`concurrency_group="synology"`.

**Live capture (ground truth, EMPTY — 0 jobs/repos configured live):**
- `backup_task_list` payload: `{"task_list": [], "total": 0, "is_data_restoring": false,
  "is_downloading": false, "is_lun_restoring": false, "is_restoring": false,
  "is_snapshot_restoring": false}`.
- `backup_repository_list` payload: `{"offset": 0, "repo_list": [], "total": 0}`.
Saved to `/tmp/synology_backup_task_raw.json` + `/tmp/synology_backup_repository_raw.json`.

**Selected design (brainstormer options; coordinator selected per pre-authorization):**
1. **Field-name strategy:** centralized `_FIELDS` constant map (logical name → inferred DSM key) parsed via
   the tolerant `as_list_of_dicts`/`nested`/`as_float`/`bytes_field` helpers + a local `_parse_dsm_time`
   accepting epoch-int OR `"%Y-%m-%d %H:%M:%S"` string. Inferred populated-job fields (UNVERIFIED — no live
   job): `task_id, name, state, enabled, data_size, last_bkp_time, last_bkp_result, next_bkp_time`.
   Tolerant helpers degrade wrong guesses to None (metric skipped) rather than crash. Repo path is
   COUNTED ONLY (`len(repo_list)`) so its inner shape is irrelevant.
2. **`no_backup_configured` derivation:** `=1.0` iff the task fetch SUCCEEDED (response, not error) AND
   parsed task count == 0; `=0.0` otherwise (incl. on fetch failure). Success-implies-running: a successful
   `SYNO.Backup.Task` response is positive evidence the package is installed+running. Always emitted.
3. **Job label identity:** `task_id` (stringified) is the universal `{job}` label on every per-job metric;
   a separate `homelab_synology_backup_info{job,name}=1.0` identity series carries the human name (mirrors
   `system.py`/`ups.py` `_info`). Rename-stable + cardinality-bounded. Anchor fallback: id-then-name if
   `task_id` absent.
4. **`last_result`:** card's `last_result{job,result="success"|"fail"}=1.0` state-set PLUS an always-emitted
   scalar `homelab_synology_backup_last_result_ok{job}` (1.0 success / 0.0 fail; omit on unknown). The
   scalar is what STAGE-008-022's BackupFailed rule alerts on; the state-set drives the dashboard.
5. **`status`:** `status{job,state=<active>}=1.0` state-set PLUS an always-emitted
   `homelab_synology_backup_status_error{job}` (1.0 when active state ∈ a collector-owned `_ERROR_STATES`
   constant `{"error","unavailable"}`, else 0.0). Keeps the error-state taxonomy in the collector, not in
   the PromQL rule.

All five follow the locked Wave-B alerting philosophy (always-emitted alertable gauge + labeled detail).
Aggregates ALWAYS emitted (even empty): `backup_configured_count`, `no_backup_configured`,
`backup_repository_count` — these are the empty live path's alertable output.

**Known limitation (NOT a silent deferral):** the populated-job field names are INFERRED, not live-verified
(0 jobs configured on the NAS). The empty/`NoBackupConfigured` path is the live-validated 3b path; the
populated path is fixture-only until a real job exists. Mitigation: `_FIELDS` + `_ERROR_STATES` centralized
constants make a post-live correction a one-line edit. A REGRESSION item (added in Refinement) will name the
trigger: "verify Hyper Backup populated-job field names against a real job when one is configured." This is
endorsed by the locked architecture ("backup/replication built fully now, graceful-empty").

**Files:** `plugins/collectors/integrations/synology/backup.py` (new) + add `SynologyBackupCollector` to
`_SYNOLOGY_COLLECTORS` in `synology/__init__.py` + `tests/test_synology_backup_collector.py` (new) +
add `_EXPECTED_BACKUP_INTERVAL = 300` + `test_register_all_registers_backup` to
`tests/test_synology_integration_bundle.py`. No migration / openapi / frontend.

**Brainstormer:** used (real design choices on field-name strategy + signal shapes). Coordinator selected
all five recommendations.
