## Overview

Collect the remaining Surveillance Station status signals: license usage (`SYNO.SurveillanceStation.License
Load` ‚Äî recon: 3/3 used, max 90) and HomeMode armed/disarmed state (`SYNO.SurveillanceStation.HomeMode

**Design Notes** (recorded 2026-06-26):

**Selected approach:** Option B (state core + curated config/trend context) from the Design brainstorm, with one refinement: ALL scalars are seeded `[({}, 0.0)]` (not emit-on-success), because both License.Load and HomeMode.GetInfo return a single flat scalar object (no per-entity families, no cardinality risk), so seeding gives a uniform parse + dashboard-stable / alertable-on-fetch-failure series consistent with the Wave-B "absence is hostile to alerting" philosophy and the cameras.py/events.py exemplars.

**LIVE PAYLOAD CAPTURE (3b, against the real NAS 192.168.2.4) drove this design ‚Äî card field names were a prior, NOT ground truth:**

**CLIENT BUG FOUND + FIXED IN THIS STAGE:** The scaffolding helper `SynologyRestClient.ss_license()` called `SYNO.SurveillanceStation.License` **version "2"**, which returns DSM **error 103 (Unknown API)** on the live NAS (Surveillance Station 9.2.4-11880). The correct version is **"1"**. This stage fixes `client.py` (`ss_license` version `"2"` ‚Üí `"1"`) AND updates the matching parametrize tuple in `test_synology_client.py::test_scaffolding_helper_sends_correct_query` (per the rule: a client-helper change carries its scaffolding-test tuple in the SAME stage). `ss_homemode()` was already correct at v1 GetInfo.

**SS.License Load v1 ‚Äî REAL keys (captured live):** `key_total`=3 (PURCHASED license seats ‚Äî authoritative), `key_used`=3 (seats consumed), `key_max`=90 (max cameras the SS package SUPPORTS ‚Äî equals SS.Info.maxCameraSupport, a hardware ceiling, NOT purchased seats), `localCamCnt`=3, plus YAGNI-dropped keys: `localKeyTotal`, `license_num`, `keyPostponedCnt`, `localAcsCnt`/`localIOCnt`/`localSpeakerCnt`/`localTransCnt`, `slaveDsCnt`, `blCMSLosthost` (true is NORMAL for a standalone non-CMS NAS ‚Äî not alertable), `success` (envelope flag).

**SS.HomeMode GetInfo v1 ‚Äî REAL keys (captured live):** `on`=false (PRIMARY state: false=disarmed/normal recording, true=armed/home-privacy mode), `notify_on`, `mode_schedule_on`, `rec_schedule_on`, `streaming_on`, plus YAGNI-dropped keys: `mode_schedule`/`rec_schedule` (336-char bitmap STRINGS ‚Äî not gauges), `mode_schedule_next_time`, `onetime_enable_on`/`onetime_disable_on` + their epoch fields, `actrule_on`, `dual_rec_off`, `reason`, geofence ints (`geo_*`, unconfigured/0), `custom*_det`/`custom*_di`, `last_update_time` (microsecond epoch ‚Äî no alert/panel consumer at homelab scale), and string/array fields (`cameras`, `actrules`, `io_modules`, `stream_profile`, `wifi_ssid`, `notify_event_list`, `rec_sch_custom_det_app_list`).

**RESOLVED ‚Äî the inherited STAGE-008-015 naming caveat:** Confirmed against the live License.Load payload that DSM DOES expose purchased-seat counts (`key_total`/`key_used`) distinct from `key_max`/`maxCameraSupport`. Reconciliation: 015's `homelab_synology_ss_info_license_*` stays as the CHEAP SS.Info-derived sanity numbers (`ss_info_license_used` from the DSM-typo'd `liscenseNumber`, `ss_info_license_max` from `maxCameraSupport`); 017's `homelab_synology_ss_license_*` is the AUTHORITATIVE breakdown. `ss_license_total` (=purchased seats) vs `ss_license_max` (=max-supported-cameras ceiling) is the disambiguation so the two prefixes never imply contradictory meanings.

**RESOLVED ‚Äî the consistency-check schedule-flags ambiguity:** The card asked whether schedule flags are one combined flag or separate. Live payload shows them as separate boolean fields; this stage emits `ss_homemode_schedule_on` (mode_schedule_on) + `ss_homemode_rec_schedule_on` (rec_schedule_on) as distinct gauges. The card's earlier `ss_homemode_schedule_enabled`/`ss_license_local_cam_count`/`ss_license_total` naming is superseded by the live-grounded names below.

**FINAL METRIC SET (11 gauges, ALL seeded `[({}, 0.0)]`, single series each ‚Äî no labels):**

| metric name | source key | type | derivation |
| --- | --- | --- | --- |
| `homelab_synology_ss_license_total` | `key_total` | int (`as_float`) | purchased license seats (authoritative) |
| `homelab_synology_ss_license_used` | `key_used` | int (`as_float`) | seats consumed by cameras |
| `homelab_synology_ss_license_max` | `key_max` | int (`as_float`) | max cameras SS package supports (90; hardware ceiling) |
| `homelab_synology_ss_license_exhausted` | DERIVED | bool (0/1) | `1.0` iff BOTH `used` and `total` parse non-None AND `used > total` (over-subscribed = real fault); `used==total` (the NORMAL fully-licensed state) ‚Üí `0.0`; partial payload ‚Üí stays seeded `0.0` (no false positive) |
| `homelab_synology_ss_license_camera_count` | `localCamCnt` | int (`as_float`) | sanity cross-check vs `ss_cameras_total` |
| `homelab_synology_ss_homemode_on` | `on` | bool (`bool_to_gauge`) | PRIMARY state: 1=armed (home/privacy), 0=disarmed |
| `homelab_synology_ss_homemode_notify_on` | `notify_on` | bool | home-mode push notifications enabled |
| `homelab_synology_ss_homemode_schedule_on` | `mode_schedule_on` | bool | schedule-driven home-mode switching active |
| `homelab_synology_ss_homemode_rec_schedule_on` | `rec_schedule_on` | bool | recording schedule active |
| `homelab_synology_ss_homemode_streaming_on` | `streaming_on` | bool | live streaming enabled |
| `homelab_synology_api_took_seconds{api=...}` √ó2 | ‚Äî | float | per successful fetch (emitted by the shared `fetch_or_result`/`_fetch` helper) |

**Collector shape:** `class SynologyLicenseCollector(BaseCollector)`, file `ss_license_homemode.py`, `name="synology_license"`, `interval=timedelta(seconds=300)` (5m), `timeout=timedelta(seconds=30)`, `concurrency_group="synology"`. 2-fetch co-equal (`ss_license()` + `ss_homemode()`), independent fetch isolation, `ok = license_resp is not None or homemode_resp is not None`. Mirrors cameras.py/events.py. New interval constant `_EXPECTED_LICENSE_INTERVAL = 300` + a register test in the bundle test; class added to `_SYNOLOGY_COLLECTORS`.

**Alertable bad-states surfaced for downstream vmalert (consumed in STAGE-008-022, NOT built here):** `ss_license_exhausted == 1` (over-subscribed) and `ss_homemode_on == 1` (armed when unexpected). Both seeded so a rule can fire even on total fetch failure. `blCMSLosthost` deliberately NOT alertable (true is normal for a standalone NAS) ‚Äî documented YAGNI scope-out, not a deferral.

**Scope-outs (YAGNI, documented rationale ‚Äî NOT deferrals needing a tracking stage):** all device-count keys except `localCamCnt`, `license_num`, `keyPostponedCnt`, `localKeyTotal`, `blCMSLosthost`, geofence, override flags/epochs, `reason`, `mode_schedule_next_time`, `dual_rec_off`, `actrule_on`, `custom*`, `last_update_time`, and all string/array fields. Reason: zero downstream consumer at homelab scale (single user, 3 cameras, standalone non-CMS, fully licensed); emitting them adds dead series + 100%-coverage burden with no alert/panel value.
