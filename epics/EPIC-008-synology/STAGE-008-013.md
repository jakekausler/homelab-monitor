# STAGE-008-013: Security Advisor status + active-connections collector

## Status: Complete

## Overview

Collect the structured security STATUS signals (not log events — those come via syslog per Amendment 5):
Security Advisor result (`SYNO.Core.SecurityScan.Status` method **`system_get`** — recon: `sysStatus=risk`
currently, per-category fail counts) and the active-connection count (`SYNO.Core.CurrentConnection list` —
how many sessions, who/from-where is a view-time re-query on the panel, not a stored series). Per Amendment 5:
do NOT API-poll anything that arrives via syslog; Security Advisor scan-status is a periodic scan result (not
a log event) so it is API-sourced; login *events* are VL-derived (STAGE-008-023).

**Vertical slice:** after this stage `homelab_synology_security_status` +
`homelab_synology_security_findings{category,severity}` + `homelab_synology_active_connections` are emitted
for the live NAS.

**Spec references:** §6.2. EPIC-008 Amendment 5 (events→syslog, status→metric; single source per signal).
EPIC-008 "Verified deployment reality" (SecurityScan.Status `system_get`; CurrentConnection shape).

  (privacy + cardinality).

## Concrete deliverables

### 1. Security + connections collector

`security.py`: calls `client.security_scan_status()` (`system_get`) + `client.current_connection_list()`,
emits:
- `homelab_synology_security_status` (0=safe / 1=warning / 2=risk, from `sysStatus`),
  `homelab_synology_security_findings{category,severity}` (per-category fail counts:
  malware/network/systemCheck/update/userInfo × danger/warning/risk/outOfDate/info),
  `homelab_synology_security_last_scan_age_seconds`.
- `homelab_synology_active_connections` (count from `CurrentConnection total`). (The who/from-IP/service list
  is NOT emitted as series — surfaced live on the panel in STAGE-008-027.)

### 2. Tests

3a fixtures: SecurityScan `system_get` (risk with a userInfo finding; safe); CurrentConnection list. 3b live
emission.

## Design Phase

**Status**: [x] Design (Complete)

**Design Notes**:

Co-equal multi-fetch collector (mirrors `updates.py`/`backup.py`). New file
`plugins/collectors/integrations/synology/security.py`. Cadence 1h (interval=3600s), timeout 30s,
`concurrency_group="synology"`. Local `_fetch` wrapper + co-equal combine; always calls `_emit`.

**Live recon (POPULATED ground truth):**
- Both client helpers EXIST (no scaffolding gap): `security_scan_status()`
  (`SYNO.Core.SecurityScan.Status` v1 `system_get` — confirmed `system_get`, NOT `get`),
  `current_connection_list()` (`SYNO.Core.CurrentConnection` v1 `list`). Both APIs are v1-only (maxVersion 1).
- **security payload** top keys: `items`, `lastScanTime`, `startTime`, `success`, `sysProgress`, `sysStatus`.
  `sysStatus="risk"` (live overall posture). `lastScanTime="1782090303"` (bare Unix epoch STRING).
  `sysProgress=100`. **`items` is a DICT KEYED BY CATEGORY** (NOT a list — verified against the captured
  payload; the recon agent mischaracterized it): `items = {"malware": {...}, "network": {...},
  "systemCheck": {...}, "update": {...}, "userInfo": {...}}`. Each category value:
  `{category, failSeverity (str), fail: {danger, info, outOfDate, risk, warning} (int counts), progress,
  total, runningItem, waitNum}`. LIVE: malware/network/systemCheck/update all `failSeverity="safe"` (all
  fail.*=0); `userInfo` `failSeverity="risk"`, `fail.risk=1` → overall `sysStatus="risk"`. NO per-individual-
  check list — only the 5 category aggregates.
- **connection payload** top keys: `items` (list), `systime`, `total`. Live `total=5`. Per-connection detail
  (who/from/service) is NOT emitted as series (privacy + cardinality; the panel re-queries live in
  STAGE-008-027) — only the top-level `total` COUNT is emitted (no need to walk `items`).
- Saved: `/tmp/synology_security_raw.json`, `/tmp/synology_connection_raw.json`,
  `/tmp/synology_api_info_security.json`.

**Selected design (brainstormer options; coordinator selected per pre-authorization):**
1. **`security_status` numeric mapping:** `_STATUS_MAP = {"safe":0.0,"warning":1.0,"risk":2.0,"danger":3.0}`;
   `security_status = _STATUS_MAP.get(sysStatus.lower())`; an UNKNOWN non-empty string clamps to 3.0
   (fail-loud — an unknown security status must never read as safe or disappear). The `security_status`
   0-baseline is seeded at **2.0** (risk), so absent/failed status reads "not clearly safe" (conservative,
   not 0/safe). PLUS an always-emit `homelab_synology_security_safe` boolean = 1.0 iff `sysStatus=="safe"`
   else 0.0 (seeded 0.0) — the clean alert primitive (`security_safe == 0` fires regardless of the exact
   status enum). Guard `isinstance(sysStatus, str)` before `.lower()`.
2. **`security_findings{category,severity}` matrix:** hardcode `_CATEGORIES =
   ("malware","network","systemCheck","update","userInfo")` and `_SEVERITIES =
   ("danger","warning","risk","outOfDate","info")`; iterate the 5×5 GRID (not the live `fail` keys) so all 25
   series exist even when a 0-count key is omitted — emit `homelab_synology_security_findings{category,
   severity}` = `as_float(nested(items_dict, cat, "fail", sev)) or 0.0` for every cell (0 included; Wave-B
   absence-is-hostile). Preserve `outOfDate` camelCase verbatim as the label value. PLUS an aggregate
   `homelab_synology_security_findings_total{severity}` = sum across the 5 categories per severity (5 series)
   — the alert-ergonomic rollup ("any risk finding anywhere"). Parse `items` via `as_dict(nested(payload,
   "items"))` then `nested(items_dict, cat, "fail", sev)` — NOT `as_list_of_dicts` (items is a dict).
3. **`security_last_scan_age_seconds`:** local `_parse_dsm_time` (copy from updates.py/backup.py — its
   `as_float` numeric-string path parses the bare epoch "1782090303"); `age = max(0.0, now - scan_time)`,
   `now = datetime.now(UTC).timestamp()` (`from datetime import UTC` IS needed here). Emit-if-present (skip
   age + timestamp if lastScanTime absent/unparseable). PLUS raw `homelab_synology_security_last_scan_timestamp`
   (epoch) for dashboards (pairs with age, mirrors backup's last_run_timestamp+age). **SCOPE-OUT (YAGNI, with
   rationale — NOT a card deliverable):** `scan_in_progress`/`sysProgress` is NOT emitted — no alert consumer
   (scans are advisory + infrequent; staleness is covered by `last_scan_age`). Not on the card, so no
   card-deliverable is dropped.
4. **`active_connections`:** `homelab_synology_active_connections = as_float(nested(payload,"total"))` emitted
   ONLY when the connection fetch SUCCEEDED and `total` parses — a DELIBERATE break from the seed-0-baseline
   pattern: `active_connections=0` is a meaningful real state ("nobody logged in"), distinct from "the fetch
   failed"; seeding 0 would conflate them (a failed fetch reading as "no connections"). This gauge is a
   trend/panel metric, NOT an alertable bad-state gauge (no connection alert on the card), so the always-emit-
   baseline rationale doesn't apply. Count ONLY — no per-connection detail (card-confirmed; STAGE-008-027
   handles the who/from-IP/service live panel re-query). Documented in the module docstring so a future reader
   doesn't "fix" it to a seeded baseline.
5. **Cadence:** ONE collector at 1h (3600s), mirroring `synology_updates` (also 1h/advisory). The card's
   "connections 60s" desideratum is a documented DESIGN RESOLUTION (NOT a deferral): the active-connection
   COUNT IS emitted, just at 1h not 60s; the 60s freshness was for the live operator panel, which STAGE-008-027
   provides via on-demand re-query (named). A split into two collectors (1h security + 60s connections) was
   rejected — doubles the file/test surface to gain a faster count series whose only real-time consumer
   re-queries live anyway. Noted in the module docstring.
6. **Field constants + parse:** `_SEC_FIELDS` (items/sysStatus/lastScanTime/fail) + `_CONN_FIELDS` (total) +
   the hardcoded `_CATEGORIES`/`_SEVERITIES` grid tuples. `items` (security) is a category-keyed DICT →
   `as_dict`/`nested`; connections `items` is a list we do NOT iterate (read top-level `total`). `nested`
   tolerates missing intermediate dicts (returns None → `or 0.0`). `from datetime import UTC` imported (for
   the scan-age now()).

All follow the locked Wave-B alerting philosophy (always-emit alertable gauge + labeled detail). These are
STANDING-ADVISORY/status signals (Security Advisor scan-status is API-sourced per Amendment 5; login EVENTS
are syslog→VL in STAGE-008-023, NOT API-polled here). STAGE-008-022 owns the consuming alert rules.

**Additions beyond the card (each justified, none speculative):** `security_safe` boolean (clean alert
primitive), `security_findings_total{severity}` rollup (alert ergonomics), `security_last_scan_timestamp`
(dashboard pairing). **Scoped out (YAGNI + rationale, not a card deliverable):** `scan_in_progress`.
**Deliberate pattern break:** `active_connections` is emit-on-success not seeded-0.

**Key parse gotcha for Build:** security `items` is a category-keyed DICT (verified against the captured
payload — the recon report wrongly called it a list); connections `items` is a list we don't iterate.

**Files:** `security.py` (new) + add `SynologySecurityCollector` to `_SYNOLOGY_COLLECTORS` in
`synology/__init__.py` + `tests/test_synology_security_collector.py` (new) + add
`_EXPECTED_SECURITY_INTERVAL = 3600` (or reuse the 3600 constant) +
`test_register_all_registers_security` to `tests/test_synology_integration_bundle.py`. No migration / openapi
/ frontend. (Both client helpers already exist — no client.py change needed.)

**Brainstormer:** used (real choices — status mapping, findings matrix, the connections-fetch-fail nuance,
cadence). Coordinator selected all six recommendations; verified the `items`-is-a-dict shape against the live
payload before adopting (Lesson #3).
