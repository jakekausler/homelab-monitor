# STAGE-008-033: Host-collector hardening — timeout-wrap host.py's blocking psutil probes

## Status: Complete

## Overview

`builtin/host.py`'s `HostCollector` probes its configured `extra_mountpoints` (default `["/rackstation"]`) via `psutil.disk_usage(mountpoint)` — a BLOCKING call with NO timeout. On a wedged hard-NFS mount this HANGS the host collector: the scheduler's per-collector timeout records a "timeout" failure, but the underlying blocking psutil thread leaks (it stays blocked until the NFS export recovers). This is a real latent bug discovered during STAGE-008-018 (which added the timeout-safe liveness probe under a SEPARATE metric `homelab_synology_mount_up`, but deliberately left host.py untouched as out-of-scope).

Harden `host.py` so its blocking filesystem probes are non-hanging, mirroring the pattern STAGE-008-018 established in `synology_mount_health.py`: a `/proc/self/mountinfo` presence-check (cheap, never hangs) before probing, then the blocking `psutil.disk_usage` run in a BOUNDED dedicated `ThreadPoolExecutor` wrapped in an `asyncio.wait_for` timeout, so a wedged hard mount is reported (or skipped) within the timeout instead of leaking unbounded threads.

**Vertical slice:** after this stage, a wedged `/rackstation` (or any `extra_mountpoints`) NFS mount no longer hangs `HostCollector` or leaks threads — the disk-usage probe times out gracefully (the mountpoint is skipped / its bytes omitted) and the collector continues.

**Spec references:** STAGE-008-018 (the timeout-safe statvfs pattern in `synology_mount_health.py` — mountinfo presence-check → bounded dedicated-executor probe → `asyncio.wait_for`). The STAGE-008-018 regression.md item #6 (the discovery + technical spec). `builtin/host.py` (the owning file; the `psutil.disk_usage` call site ~line 111-123 / `_collect_disk_usage`).

## Concrete deliverables

### 1. Non-hanging disk-usage probe in host.py

Wrap `HostCollector`'s `psutil.disk_usage(mountpoint)` (the `extra_mountpoints` probe in `_collect_disk_usage`) so a wedged hard-NFS mount cannot hang the collector or leak unbounded threads: a `/proc/self/mountinfo` presence-check first (skip a mount that's not present), then the blocking `psutil.disk_usage` in a BOUNDED dedicated `ThreadPoolExecutor` under an `asyncio.wait_for(timeout=...)`. On timeout/OSError: skip the mountpoint (omit its `homelab_host_disk_bytes`) + record the error, and continue. Mirror `synology_mount_health.py`'s helpers/shape. Consider whether the `psutil.disk_partitions` loop's per-partition `disk_usage` calls need the same treatment (the partition loop also calls disk_usage — scope: at minimum the `extra_mountpoints` NFS probe; evaluate the partition loop too).

### 2. Tests

3a: unit tests — responsive mount (bytes emitted), wedged mount (probe times out → mountpoint skipped, no hang, error recorded, no thread leak), missing mount (presence-check skip). Mirror `test_synology_mount_health_collector.py`'s injection seam (fake mountinfo + fake/blocking disk_usage, no real /proc, no real threads). 3b (instance A): real `/rackstation` probe on the host stays responsive; verify the timeout mechanism doesn't regress normal disk-usage emission.

## Design Phase

**Status**: [x] Design (Complete)

**Design Notes** (2026-06-29):

**Selected approach:** Option A (Minimal Mirror) from the brainstormer — wrap ONLY the critical blocking `psutil.disk_usage` call, mirroring STAGE-008-018's `synology_mount_health.py` idiom exactly. NOT Option B (adds a probe-duration metric — rejected: violates the card's "no metric-surface change" mandate) and NOT Option C (also wraps disk_partitions — rejected: over-engineering a third-order risk, diverges from 018 which wraps exactly one blocking call).

**What gets wrapped (the one critical call):** `psutil.disk_usage(raw_mp)` in `_emit_disk_mountpoint` (host.py ~line 274) — the only call that stats a (potentially wedged) NFS filesystem. Mechanism (mirror 018): a MODULE-LEVEL dedicated bounded `_EXECUTOR = ThreadPoolExecutor(max_workers=4, thread_name_prefix="host-disk")`; a module-level `_disk_usage(path)` seam run in that executor via `await asyncio.wait_for(loop.run_in_executor(_EXECUTOR, _disk_usage, raw_mp), timeout=_PROBE_TIMEOUT)` with `_PROBE_TIMEOUT: Final[float] = 5.0`; a `/proc/self/mountinfo` presence-check (`parse_mountinfo`/`read_mountinfo`/`present_mounts`, copied from 018) that skips probing a path NOT currently mounted (cheap, never hangs — avoids even spawning a thread for an absent path). `_emit_disk_mountpoint` (and the two loops in `_collect_disk_usage`) become `async`/awaited. Catch `(TimeoutError, OSError)` → return the existing `(0, error_string)` shape; skip the mountpoint's `homelab_host_disk_bytes`; continue.

**NOT wrapped (YAGNI, documented):** `psutil.disk_partitions` (~line 221, enumeration — rarer hang, third-order risk; 018 wraps exactly one call) and `psutil.disk_io_counters` (~line 299 — reads kernel counters, never stats a mount). Wrapping these is out of scope for THIS stage and not deferred to any future stage (no concrete need exists — they're genuinely lower-risk; if a real wedge on disk_partitions is ever observed, a new stage would own it). This is a YAGNI-scope-out-with-rationale, not a silent drop.

**No metric-surface change (honors the card):** host.py keeps emitting ONLY `homelab_host_disk_bytes{mountpoint,type}`. NO new probe-duration metric (unlike 018's M_PROBE_SECONDS). The timeout remains OBSERVABLE: a timeout records an error in the result's `errors` list → flows into the framework's `homelab_collector_run_*` self-metrics + flips `ok=False` on the partition path. (If probe-latency observability is ever wanted, that's a documented future metric addition, not part of 033.)

**ok-semantics (preserve host.py's existing two-loop asymmetry — locked):** a `disk_usage` timeout on a PARTITION-loop mount records an error → `ok=False` (legitimately signals an unexpected stall on an allowlist-included mount). A timeout on an EXTRA_MOUNTPOINTS entry is DISCARDED → `ok=True`, mount skipped (preserves the existing "extra_mountpoints silently skips unmounted/inaccessible paths" contract). Do NOT convert host.py to 018's blanket always-ok — host.py is a multi-subsystem collector where ok=False legitimately means "something I expected to work didn't."

**Card-vs-reality (honest framing — verified against code + live prod 2026-06-29):** the card frames the hang as a real bug, but reality is it's LATENT/conditional. host.py's default `disk_mountpoint_relabel` allowlist (`/config`→`/`, `/host-compose`→`/storage`) means it currently probes ONLY local `/` and `/storage` — NOT `/rackstation` NFS. Live confirms host disk metrics cover only local FS; the host collector is healthy (178/178 runs, ~18ms avg, 0 errors). The wedge-hang triggers only if an operator opts an NFS mount into the relabel allowlist or extra_mountpoints reaches a wedged path. So this hardening is correct DEFENSE-IN-DEPTH for that opt-in path, not a fix for a live fire. The presence-check + executor wrap change NOTHING about which mountpoints emit in the healthy default case.

**Brainstormer:** used (Opus) — 3 options (A minimal / B +probe-metric / C +disk_partitions); selected A myself (pre-authorized) for no-metric-change + mirror-018 + YAGNI + cheapest 100%-branch coverage.

**Test plan (new file `apps/monitor/tests/test_host_collector.py`, clone 018's `test_synology_mount_health_collector.py` seam):** monkeypatch module-level `_disk_usage`/`present_mounts`/`read_mountinfo` — NO real /proc, NO real threads, NO real 5s wait (a fake raising TimeoutError drives the timeout branch). Per-branch coverage for 100%: (1) present+responsive → bytes emitted; (2) mountinfo-absent → skip, no probe; (3) wait_for TimeoutError → error branch, BOTH loops (partition records→ok=False; extra_mountpoints discards→ok=True); (4) OSError from _disk_usage → same error branch; (5) usage.total==0 skip; (6) friendly-in-`seen` dedup skip. Existing host.py subsystems (cpu/mem/net/temp/etc) keep current coverage.
