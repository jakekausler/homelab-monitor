**Design Notes** (recorded 2026-06-27):

**Approach:** Author the second tranche of Synology vmalert-METRICS alert rules — protect/maintain (backup, updates/packages), Security Advisor, and surveillance-integrity (cameras/license) — grounded against the LIVE production NAS (captured via a throwaway container running current collector source against the real DSM, since the running prod image is stale and lacks the surveillance + full backup collectors). Hybrid severity model (info=standing advisory / warning=problem / critical=hardware/data-loss). Protect/security/updates rules append NEW groups to existing `deploy/vmalert/metrics/synology.yaml`; surveillance rules go in a SEPARATE new file `deploy/vmalert/metrics/synology-surveillance.yaml` (the epic ships two Grafana dashboards + two integration pages — Synology and Surveillance — so `surveillance/*` is a real architectural seam). Existing 5 groups in synology.yaml (synology, synology_dependents, synology_storage, synology_hardware, synology_system) are UNTOUCHED (append-only). `SynologyMountDown` is NOT re-authored (already in 019).

**Rules to author (real metric exprs, verified against live capture):**

protect/security/updates groups (appended to synology.yaml):
- `SynologyBackupFailed` — `homelab_synology_backup_last_result_ok == 0` — critical — forward-looking (emit-on-presence; 0 HyperBackup jobs on NAS today so dead-safe; goes live the moment a job exists, no privilege dependency).
- `SynologyNoBackupConfigured` — `homelab_synology_no_backup_configured == 1` — **info** (card said warning; DOWNGRADED — "no backup configured" is a standing advisory state per hybrid model, consistent with update-available/security-advisor-not-safe at info). Fires today (no_backup_configured==1, always-seeded).
- `SynologyUpdateAvailable` — `homelab_synology_dsm_update_available == 1 and homelab_synology_dsm_update_is_security == 0` — info. Mutually exclusive with the security rule (both bare scalars, `and` matches directly). Suppressed today (the pending DSM update IS a security update).
- `SynologyDsmSecurityUpdateAvailable` — `homelab_synology_dsm_update_available == 1 and homelab_synology_dsm_update_is_security == 1` — warning (**NEW, beyond card** — an available security patch is an unpatched-vuln problem, not a standing advisory). Fires today (DSM 7.3.2 Update 3 is a security update).
- `SynologyPackageUpdateAvailable` — `count(homelab_synology_package_update_available == 1) > 0` — info. Single quiet rollup over the per-package metric (NOT the contradicted `packages_with_updates_count` rollup, which reads 0 while 4 packages have updates). Fires today (4 packages: FileStation/HybridShare/QuickConnect/SurveillanceStation).
- `SynologySecurityAdvisorNotSafe` — `homelab_synology_security_safe == 0` — info (standing advisory per hybrid model). Fires today (1 real risk-severity finding in userInfo; security_safe==0).

surveillance group (new file synology-surveillance.yaml, category: surveillance):
- `SurveillanceCameraOffline` — `homelab_synology_ss_camera_connected == 0` — **warning** (card said critical; DOWNGRADED — camera-offline is a problem/coverage-gap, not NAS hardware failure or data-loss; recordings already captured aren't lost). Uses the clean bool `ss_camera_connected{camera}` (NOT `ss_camera_status`, whose value 3 = intentionally-disabled and must not alert). No-fire today (all 3 cameras connected).
- `SurveillanceCameraRecordingDisabled` — `homelab_synology_ss_camera_recording_retention_mode{mode="none"} == 1` — warning (narrowed from card's "SurveillanceCameraNotRecording"; see deferral below). No-fire today (all 3 cameras in `size` retention mode).
- `SurveillanceLicenseExhausted` — `homelab_synology_ss_info_license_used >= homelab_synology_ss_info_license_max` — warning. Fires over the CAMERAS-collector SS.Info source (used=3/max=90), NOT the dead `ss_license_exhausted` (see REG-008-LICENSE-METRICS). Both bare scalars → plain `>=` matches. No-fire today (3 >= 90 false).

**PromQL notes:** `ss_info_license_used/max`, `dsm_update_available/is_security`, `no_backup_configured`, `security_safe` are all bare collector-emitted scalars with identical label sets → binary ops (`>=`, `and`) match without `on()/group_left()`. `ss_camera_connected`/`recording_retention_mode` carry `{camera}` (+ `{mode}`); `package_update_available` carries `{package}` (aggregated away by `count()`).

**Card-vs-reality deltas (live capture corrected the card):**
1. License: dedicated `ss_license_*` metrics are permanently 0.0 (DSM error 103 — account lacks SurveillanceStation License/Load privilege). Real license counts come from the cameras collector's `ss_info_license_used/max`. SurveillanceLicenseExhausted repointed accordingly.
2. Camera-recording signal: there is NO "is recording now" metric. The recording signal is `ss_camera_recording_retention_mode{mode}` (days/size/none) + `recording_keep_*`. The card's "SurveillanceCameraNotRecording" is split: config-disabled subset shipped now (SurveillanceCameraRecordingDisabled); the connected-but-silently-not-recording TREND half is deferred (see below).
3. Replication is a PERMISSIONS artifact, not unconfigured: replication_available==0/snapshot_count==0 are DSM error 103 (account lacks Btrfs/Replication privilege), and snapshot_latest_age_seconds is ABSENT. SynologyReplicationLag's only data source is unreadable → NOT authored (see deferral).
4. Two card severities downgraded to match the hybrid model: NoBackupConfigured warning→info; SurveillanceCameraOffline critical→warning.
5. `packages_with_updates_count` rollup reads 0 while 4 packages have updates (collector quirk) → use per-package metric aggregated via count(), not the rollup.

**Scope dispositions (no-silent-deferral ledger):**
- `SynologyReplicationLag` — **NOT AUTHORED** (its sole data source `snapshot_latest_age_seconds` is unreadable: DSM account lacks Btrfs/Replication privilege; authoring it would be silently dead). Tracked home: regression item **REG-008-REPLICATION-BLOCKED** (added in Refinement) — names the privilege-grant trigger; when granted, verify the collector emits snapshot_latest_age_seconds then author the rule (suggested 26h threshold). Owner: that regression item; promote to a stage only if replication monitoring becomes a priority.
- `SurveillanceRecordingStorageLow` — **SATISFIED-BY-REUSE** by existing `SynologyVolumeNearFull` (>=80%) / `SynologyVolumeAlmostFull` (>=90%) in synology.yaml (append-only). Existing 5 groups in synology.yaml (synology, synology_dependents, synology_storage, synology_hardware, synology_system) are UNTOUCHED. Recording storage lives on a Synology volume, and a volume-full condition is exactly when recording stops. No recording-quota metric exists (YAGNI until one does). Cited, not duplicated.
- `SurveillanceCameraNotRecording` (trend / silent-failure half) — **DEFERRED**: connected-but-no-recent-recordings detection needs a `recordings_count` delta or `last_recording_age_seconds` metric that events.py does not yet emit. Tracked home: regression item **REG-008-SURVEILLANCE-RECORDING-TREND** (added in Refinement) — owns adding the metric + the trend rule. The config-disabled subset ships now as SurveillanceCameraRecordingDisabled.
- Dead `ss_license_*` metrics — tracked via regression item **REG-008-LICENSE-METRICS** (added in Refinement): the four `ss_license_*` metrics are permanently 0.0 (missing privilege); they are a trap for future rule authors and MUST NOT back any rule; `ss_info_*` is canonical. Trigger: if the privilege is granted, decide whether to delete the dead metrics. Owner: next Synology-collector stage touching the SS collector.
