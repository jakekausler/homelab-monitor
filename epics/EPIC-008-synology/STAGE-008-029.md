**Design Notes**:

Selected approach (Design phase, 2026-06-28; brainstormer Opus presented options, recommendations selected):

- **Placement (Decision 1 → 1a):** BOTH the per-camera activity widgets AND the embedded `<LogViewer>` land on the Surveillance page's NEW "Activity" tab — the slot STAGE-008-028 scaffolded (`// SCAFFOLDING: STAGE-008-029 adds an "Activity" tab` in SurveillanceLayout.tsx TABS, line 8/9). CARD-VS-ARCHITECTURE RECONCILIATION: the 029 card's title/Overview/Deliverable-2 say the LogViewer goes "on the Synology page", but the card's OWN vertical-slice says "the Surveillance page shows per-camera activity"; Amendment 7 + STAGE-008-028 established a SEPARATE "Surveillance" integration page and scaffolded its Activity tab for exactly this. The "Synology page" phrasing is stale pre-Amendment-7 wording (same drift 028's card had). Resolved: both widgets + LogViewer on the Surveillance → Activity tab, cohesive in one route. The LogViewer carries an honest caption clarifying it surfaces DSM-WIDE syslog (auth/SMART/package), not surveillance-specific lines.

- **Component map:** new `SurveillanceActivityTab.tsx` (activity widgets + embedded LogViewer); add `{ path: '/integrations/surveillance/activity', label: 'Activity' }` to SurveillanceLayout TABS; add `surveillanceActivityRoute` (path 'activity') to router.tsx + addChildren. The LogViewer adapter mirrors the `PiholeLogsTab.tsx` precedent (caller-provided `useLogs` adapter calling `useLogsQuery(expr, startIso, endIso, servicesCsv)` against `/api/logs/query`, the EPIC-004 STAGE-004-003 embedding contract). Activity widgets consume the existing `useSurveillanceCameras()` scalars (events_today 144 live, events_total_all, recordings_total 2646 live) + per-camera recordings_count, PLUS the folded-in recording bytes (see below).

- **Recording-storage enrichment (Decision 2 → 2a) — FOLD BACKEND INTO 029 (mirrors 028's model/ip fold-in):** the card asks for recording footprint / recording-storage. The bytes metrics EXIST live but the /cameras endpoint exposes only COUNTS. Live verified: `homelab_synology_ss_recordings_bytes{camera=...}` per-camera (Backyard 0.97MB / Doorbell 0.98MB / Driveway 0.97MB) + `homelab_synology_ss_recordings_bytes_total`. Resolution: ADD nullable `recordings_bytes` to CameraRow (per-camera) + `recordings_bytes_total` scalar to SurveillanceCameras; the /cameras endpoint queries the bytes metrics and joins per-camera by camera name (same idiom as 028's _info join; fast 60s/300s collector → bare instant query OK, no last_over_time). The activity widgets render bytes via the existing formatBytes helper. This makes 029 frontend + a small backend change → touches kernel/api (triggers openapi-export + generate-ui-types hooks; packages/shared-types/openapi.json regenerated + MUST be staged; apps/ui/src/api/schema.ts gitignored/build-generated) and REQUIRES the full backend 100%-branch coverage gate (NOT the frontend-only shortcut).

- **LogViewer filter (Decision 3 → 3a):** the adapter passes LogsQL `expr: service:~"synology-.*"` (regex) — captures the live `synology-auth` stream (99 lines, logins/connections) now + synology-smart/package/system/other automatically as they populate. Future-proof vs a brittle explicit CSV. LIVE REALITY: only `synology-auth` is currently populated; there are NO surveillance/camera lines in syslog (surveillance is metrics-only). Honest empty-state: the LogViewer's project-standard "no data yet" copy handles a sparse stream so it reads as honest, not broken; the DSM-wide caption frames why it's mostly auth.

- **Card-vs-contract deltas (both resolved, no silent drop):** (1) recording-storage — folded into 029 as the bytes enrichment above. (2) syslog filter — card's `service:"synology-*"` glob → real LogsQL regex `service:~"synology-.*"` (the logs `services` CSV param needs exact source_type:service pairs; the regex lives in `expr`). Both documented here, not silently changed.

- **Scope-out (YAGNI with rationale, NOT a deferred card deliverable):** a dedicated DSM-wide Synology "Logs" tab on the Synology integration page (brainstormer option 1b) is NOT built — the card's LogViewer requirement is FULLY satisfied by the Surveillance Activity tab embed. Rationale: the streams are DSM-wide but the card scopes the LogViewer to surveillance/Synology context, and a separate Synology Logs tab would add a 3rd Synology tab + split 029 across two pages for no current benefit (synology-smart/package streams are empty). If a future need for DSM-wide log search on the Synology page arises, it would be a NEW tracked stage. This is a documented YAGNI scope-out, not a dropped deliverable.

- **Refinement plan:** FRONTEND stage → rebuild apps/ui/dist + rebuild prod monitor image `--no-cache` + sha256-prove served==built + grep the new `/integrations/surveillance/activity` route into the bundle, THEN PROMPT the user for Desktop + Mobile viewport sign-off at http://192.168.2.148:29090.
