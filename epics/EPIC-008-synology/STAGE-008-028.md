The status strip lives in the Layout (above the Outlet) so 029's Activity tab inherits the license/HomeMode chips without duplication. Brainstormer (Opus) presented Options A (placeholder stub tab) / B (single page, 029 retrofits shell) / C; C selected (A is the acceptable fallback; B was weakest — it defers the exact scaffolding 029 needs, doubling viewport sign-off).

- **Component map:** `SurveillanceLayout.tsx` (shell + strip) + `SurveillanceCamerasTab.tsx` (camera grid) + `apps/ui/src/api/surveillance.ts` (`useSurveillanceSummary()` + `useSurveillanceCameras()`, useQuery + apiClient.GET + unwrap + refetchInterval 30_000). - **Backend contract verified LIVE (against the running prod container + Pydantic models + schema.ts):** `GET /api/integrations/surveillance/summary` → `SurveillanceSummary` {license_used, license_max, homemode_on, cameras_total, cameras_connected_total, cameras_disconnected_total, data_available}; `GET /api/integrations/surveillance/cameras` → `SurveillanceCameras` {cameras: CameraRow[], events_today, events_total_all, recordings_total, data_available}. Live values confirmed: 3 cameras (Backyard/Doorbell/Driveway, all connected), license 3/90, HomeMode off. `data_available` = TRUE iff `homelab_collector_run_success_total{name="synology_cameras"}` > 0.

- **CARD-VS-CONTRACT DELTA — RESOLVED by folding a backend enrichment INTO 028 (user-approved 2026-06-28).** The 028 card asks the grid to show camera "name, model/ip", but the existing `CameraRow` only exposes camera/connected/status/recordings_count — model/ip/vendor were NOT in the contract and NO existing stage owned adding them (silent gap between STAGE-008-024 and 028). The metric `homelab_synology_ss_camera_info` DOES carry per-camera labels {camera,id,ip,model,vendor,mac} (live: e.g. Backyard 192.168.2.99 Generic_ONVIF/ONVIF, Doorbell 192.168.2.215, Driveway 192.168.2.103). Resolution: ADD nullable `model`/`ip`/`vendor` to `CameraRow`; the `/cameras` endpoint (`integrations_surveillance.py`) reads the `_info` metric labels and joins by `camera` name; the grid renders them. This makes 028 frontend + a small backend change → it touches `kernel/api/` (triggers the openapi-export + generate-ui-types pre-commit hooks; `packages/shared-types/openapi.json` is regenerated + MUST be staged; `apps/ui/src/api/schema.ts` is gitignored/build-generated) and REQUIRES the full backend 100%-branch coverage gate (NOT the frontend-only shortcut). Matches this epic's established pattern (026/027 folded backend fixes into frontend stages + tracked them).

- **Honest empty states:** three states per the project ethos — positive (show value), null (muted "—", unknown), zero (honest "N clear"/"0" caption). `data_available=false` → "no data yet" (collector absent); `data_available=true` + empty → genuinely zero. Camera model/ip/vendor are nullable post-enrichment → render "—" when a label is absent (never fabricate).

- **Stale-read check (REG-008-027-STALE-READ):** the surveillance collectors are FAST — `synology_cameras` 60s, `synology_events` 300s, `synology_license` 300s — all ≤5min, so the `/summary` + `/cameras` bare instant queries are correct (no `last_over_time` needed). The new `_info` read is from the same 60s `synology_cameras` collector → also fast → bare instant query OK.

- **Refinement plan:** FRONTEND stage → rebuild `apps/ui/dist` + rebuild prod monitor image `--no-cache` + sha256-prove served==built + grep the new `/integrations/surveillance` route into the bundle, THEN PROMPT the user for Desktop + Mobile viewport sign-off at http://192.168.2.148:29090.
