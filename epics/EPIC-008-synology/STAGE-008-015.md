## Design Phase

**Status**: [x] Design (Complete)

**Design Notes** (2026-06-26):

**Selected approach: Option A тАФ faithful per-camera split + SS.Info rollup, label-carrier identity, top-level seeded alertable rollups.** Brainstormer (Opus) presented 3 options (grounded in the LIVE captured payloads); Option A selected (backend collector тАФ no UX prompt; user pre-authorized design recommendation). Rationale: faithful to live data (emits the REAL recording signal тАФ keep-SIZE active, keep-days mode OFF on this NAS тАФ not the card's guessed field that would emit nothing), fully alert-ready (per-camera `_connected` bad-state + seeded top-level rollups so "all cameras gone" still fires even on fetch failure), clean label hygiene (single `_info` carrier holds ip/model/mac/id/vendor, not smeared across families), zero metric-name collision with STAGE-008-016 (events/recordings) or STAGE-008-017 (authoritative license).

**Collector:** new `apps/monitor/homelab_monitor/plugins/collectors/integrations/synology/cameras.py` тАФ `SynologyCameraCollector(BaseCollector)` (name `synology_cameras`, interval 60s per the epic cadence "SS-cameras 60s", timeout 30s, concurrency_group "synology"). Co-equal 2-fetch: `ss_camera_list()` + `ss_info()` (both client helpers already exist + already covered in test_synology_client.py scaffolding parametrize тАФ no client change). `ok = camera_resp is not None or info_resp is not None`. Per-camera parse isolated. Closest exemplars: storage.py (two-slice) + security.py (seeded always-emit baseline + co-equal ok). Uses _shared.py helpers (as_dict/as_list_of_dicts/nested/as_float/bool_to_gauge/capped_emitter/cap_for_synology/fetch_or_result).

**Metric set (Option A):**
- Per-camera (emit-on-presence, label `camera`=newName): `homelab_synology_ss_camera_connected{camera}` (1.0 if status==1 else 0.0 тАФ alertable bad-state), `homelab_synology_ss_camera_status{camera}` (raw int 1/2/3), `homelab_synology_ss_camera_info{camera,id,model,vendor,ip,mac}`=1.0 (label-carrier identity), `homelab_synology_ss_camera_fps{camera}` (stream1.fps), `homelab_synology_ss_camera_resolution_pixels{camera}` (width├Чheight parsed from stream1.resolution "WxH"), `homelab_synology_ss_camera_resolution{camera,resolution}`=1.0 (string label-carrier), `homelab_synology_ss_camera_recording_keep_days{camera}` (recordingKeepDays, always), `homelab_synology_ss_camera_recording_keep_size_mb{camera}` (recordingKeepSize), `homelab_synology_ss_camera_recording_retention_mode{camera,mode}`=1.0 (modeтИИ{days,size,none} from enableRecordingKeepDays/enableRecordingKeepSize).
- Top-level rollups (SEEDED 0.0, always-emit тАФ alertable even on fetch fail / 0 cameras): `homelab_synology_ss_cameras_total`, `homelab_synology_ss_cameras_connected_total`, `homelab_synology_ss_cameras_disconnected_total`.
- From SS.Info (seeded 0.0 scalars + version carrier): `homelab_synology_ss_info_camera_number` (cameraNumber), `homelab_synology_ss_info_license_used` (liscenseNumber тАФ DSM typo field), `homelab_synology_ss_info_license_max` (maxCameraSupport), `homelab_synology_ss_info_version{version}`=1.0 (assembled "9.2.4-11880" from version.major/minor/small/build).

**Card-vs-reality deltas (LIVE capture 2026-06-26 тЖТ /tmp/synology_ss_camera_list_raw.json + /tmp/synology_ss_info_raw.json тАФ Lesson #3):**
- Camera name field is `newName` (NOT `name`). `id` int, `model`="Generic_ONVIF", `vendor`="ONVIF", `ip`, `status` int (all 1=connected; DSM docs 2=disconnected/3=disabled).
- NO `enabled` boolean field exists. fps/resolution are NESTED under `stream1` (not top-level).
- `recordingKeepDays`=30 exists but is INACTIVE (enableRecordingKeepDays=false; keep-SIZE active instead).
- SS.Info: `cameraNumber`=3, `liscenseNumber`=3 (DSM TYPO "liscense"), `maxCameraSupport`=90, version=object to assemble.

**SCOPE-OUT (documented, not silent): `_enabled` DROPPED.** Rationale: no backing field in the live SS.Camera payload (the card assumed a boolean that doesn't exist). The useful substitute IS emitted: `_connected` (status-derived alertable) + `_recording_retention_mode` state-set. No future stage owns a camera `enabled` field (it does not exist in DSM) тАФ justified non-build, not a deferral.

**DEFERRALS (named owners): events/recordings (Event CountByCategory, Recording List) тЖТ STAGE-008-016; the AUTHORITATIVE per-license-key breakdown (SS.License Load) тЖТ STAGE-008-017.** STAGE-008-015 emits only an SS.Info-derived cameras-vs-license SANITY rollup under the `ss_info_*` prefix (distinct from 017's `ss_license_*`), so no collision.

**Minor accepted redundancy:** `_resolution` emitted twice тАФ `_resolution_pixels` (numeric, alertable on resolution-drop) + `_resolution{resolution}` (string carrier for display; preserves the Doorbell portrait "1920x2560" info a pixel count hides).
