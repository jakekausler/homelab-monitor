# STAGE-008-005: Storage collector (volumes + disks + SMART status)

## Status: Complete

## Overview

The first real collector: pull `SYNO.Storage.CGI.Storage` method `load_info` (ONE call = volumes + disks +
pools + caches) and emit per-volume and per-disk metrics. This is the workhorse storage signal — volume
usage/status and per-disk health/temperature. (The pool/RAID slice of the same response is its own collector
in STAGE-008-006 to keep each file single-purpose; both read the same one cached fetch via the shared
client.)

**Vertical slice:** after this stage `homelab_synology_volume_*` (usage/status/fs) and
`homelab_synology_disk_*` (model/temp/`smart_status`/`unc`/`remain_life`/`sb_days_left`) are emitted for the
live NAS (1 volume, 8 disks) at 5-min cadence, in the `synology` concurrency group.

**Spec references:** §6.2 (`homelab_synology_*`). EPIC-008 "Verified deployment reality" (the load_info
shape). EPIC-007 STAGE-007-005 (combined device collector parsing one fat response) is the structural
precedent.

## Concrete deliverables

### 1. Storage collector

`storage.py` in the bundle: calls `client.storage_load_info()`, parses `volumes[]` + `disks[]`, emits:
- **Per-volume:** `homelab_synology_volume_used_bytes{volume}`, `_total_bytes{volume}`,
  `_used_percent{volume}`, `_status{volume,status}` (1/0 per state incl. `has_unverified_disk`/
  `fs_almost_full`), `_fs_type{volume,fs}`, writable/encrypted/locked/inode-full flags.
- **Per-disk:** `homelab_synology_disk_temp_celsius{disk,model}`, `_smart_status{disk}` (1=normal),
  `_status{disk}`, `_unc_count{disk}` (uncorrectable), `_remain_life{disk}` (where trustable; -1 = N/A),
  `_sb_days_left{disk}` (bad-sector days-left), size, slot.

### 2. Tests

3a fixture-based unit tests (the recon `load_info` JSON shape: 1 volume, 8 disks, mixed sizes, normal SMART,
sdc 50 °C, `has_unverified_disk`) covering parse + emit + graceful-missing-field; 3b live emission check.

## Design Phase

**Design Notes** (decided 2026-06-23):

Selected approach (brainstormer abf7378f9f30067ef). First real Synology Wave-B collector + removes the 002 placeholder.

PARSE — Option 1 (defensive-parse now, capture+validate live at 3b): recon confirmed the LEAF field names but NOT the JSON nesting. A live Design-time capture was NOT possible — `synology_dsm_password` is absent from instance B's secret store (B has only alertmanager-ingest-token + cron-events-ingest-token; STAGE-008-001's live smoke used a transient /tmp creds file, never persisted). User chose Option 1 (no password reset mid-Design). So: design a DEFENSIVE parse against recon-confirmed field names + documented DSM 7.x `load_info` structure, build the 3a fixture by hand from recon, and CAPTURE THE REAL `load_info` PAYLOAD AT REFINEMENT 3b → validate nesting + commit it as the real fixture. The parse uses a small `_nested(rec, *path)` navigator + the `_shared.py` None-tolerant parsers (`as_float`/`bytes_field`/`percent_field`), so a wrong nesting guess degrades to "metric absent," not a crash/pyright failure.

3b PREREQUISITE (user action): the live 3b exercise requires `synology_dsm_password` stored in instance B's secret store. User will reset the DSM `homelab-monitor` account password after lunch (2026-06-23) and store it via `docker exec -i homelab-monitor-b-monitor hm secrets set synology_dsm_password --from-stdin`. Until then 3b's live capture is blocked; 3a (fixture-based) proceeds independently. This one-time secret setup unblocks EVERY subsequent Synology collector's 3b too.

COLLECTOR — `storage.py` mirrors `integrations/unifi/client_stats.py`: `SynologyStorageCollector(BaseCollector)`, ClassVars `name="synology_storage"`, `interval=timedelta(seconds=300)` (5m), `timeout=timedelta(seconds=30)`, `concurrency_group="synology"`. `run(ctx)`: `start=time.monotonic()`; guard `ctx.synology is None` → `client_unconfigured_result(start)`; `emitted=[0]`; `resp=fetch_or_result(ctx, await ctx.synology.storage_load_info(), start, emitted)`; `if isinstance(resp, CollectorResult): return resp`; parse `resp.payload` (narrow `object`→dict via isinstance + `cast`, NO bare Any); emit via `capped_emitter`; return `CollectorResult(ok=True, metrics_emitted=emitted[0], events=events, duration_seconds=...)`. `ok=True` even when NAS degraded; `ok=False` only on client error.

ENUM HANDLING (resolves the 003 "generic enum mapper" deferral): the two shapes diverge, so NO shared helper — local to `storage.py`:
- Volume `status` → STATE-SET pattern: `homelab_synology_volume_status{volume,status} 1` for the observed state string (e.g. status="has_unverified_disk"). Do NOT enumerate all DSM states with 0/1 each (unbounded, version-dependent, untestable). Emit 1 for the observed state; stale series stop being written on change.
- Disk `smart_status`/`status` → SCALAR map: `normal`→1.0, any-other-string→0.0, non-str→None (3 branches, all testable). Resist a richer normal/warning/critical ladder (recon saw only "normal"; YAGNI + speculative coverage). Binary 1=healthy/0=not.
Note in 003-deferral-resolution: "no shared enum helper needed — shapes diverge; local scalar+state-set maps. Revisit extraction at 009 if a 3rd consumer wants the scalar map (rule of three)."

CARDINALITY — cap-route ALL per-disk + per-volume families through `capped_emitter(ctx, events)` + `cap_for_synology(family)` (default 500). 8 disks / 1 volume are far under, but the guardrail + the free always-written `homelab_metric_family_dropped_series{family}` gauge matches the card ("cardinality cap as guardrail") + the unifi precedent; `metrics_emitted` accounting = `emit_family() return + 1 per family`.

LABELS — disk key `{disk}` = the DSM disk `id` field (verbatim, confirm exact value at 3b; recon showed device-style `sdc`); `temp` family adds `{disk,model}` (model bounded, ~8 distinct, useful for "which model runs hot"); other disk families `{disk}`-only. Volume key `{volume}` = volume id (`volume_1`); `fs_type` adds `{fs}`; `status` adds `{status}`.

METRICS — CORE (recon-backed, emit-always): per-volume `homelab_synology_volume_used_bytes{volume}`, `_total_bytes{volume}`, `_used_percent{volume}`, `_status{volume,status}` (state-set=1), `_fs_type{volume,fs}` (info=1); per-disk `homelab_synology_disk_temp_celsius{disk,model}`, `_smart_status{disk}` (1=normal), `_status{disk}`, `_unc_count{disk}`, `_remain_life{disk}` (-1=N/A emitted literally), `_sb_days_left{disk}`. OPTIONAL (emit-if-present, None→skip, graduate to CORE if 3b live capture confirms): volume writable/encrypted/locked/inode-used-percent; disk size_bytes/slot. NO pool/RAID/storagePools metrics — that's STAGE-008-006 (same load_info response, different slice).

PLACEHOLDER REMOVAL (002 deferral, in-scope): delete `integrations/synology/placeholder.py`; delete `apps/monitor/tests/test_synology_placeholder_collector.py`; swap the `_SYNOLOGY_COLLECTORS` entry in `integrations/synology/__init__.py` from `SynologyPlaceholderCollector` to `SynologyStorageCollector` (update the import). ALSO RETARGET `apps/monitor/tests/test_synology_integration_bundle.py` (it currently asserts `SynologyPlaceholderCollector` registers — point those assertions at `SynologyStorageCollector`; do NOT just delete them).

TESTS (3a, 100% branch coverage): fixture-based unit tests mirroring `client_stats` test structure — classvars; `ctx.synology is None`→ok=False; `SynologyError`→ok=False; malformed-payload (payload not a dict)→ok=True with drop-gauges-at-0; happy-path fixture (1 degraded volume so `status` emits a real state; 8 disks incl. one `remain_life=-1`, one `smart_status != "normal"`, one missing an optional field → exercise None-skip + the 0.0 + the -1 pass-through) asserting exact series + `metrics_emitted`. Build the 3a fixture hand-built from recon field names; 3b replaces/validates it with the captured real payload.
