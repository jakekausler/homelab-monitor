**Backend changes (extend `kernel/api/routers/integrations_synology.py` `/hardware`):**
- `SynologySshProbe` += 4 EPIC-017 probe-state fields (resolves GAP A): `up: bool` (from `homelab_ssh_up{target="synology"}`, absent→False), `host_key_mismatch: bool` (`homelab_ssh_host_key_mismatch{target="synology"}`, absent→False), `last_success_age_seconds: float | None` (`homelab_ssh_last_success_age_seconds{target="synology"}` — NO synology sample yet because no prior SSH success, so it is None now → frontend renders "never succeeded"), `probe_duration_seconds: float | None` (`homelab_ssh_probe_duration_seconds{target="synology"}`). This fully satisfies the EPIC-017 UI-mandate ("Synology page renders its own probe state: last-success age, host-key status, duration") — the metrics existed but had NO frontend path until now.
- `DiskRow` += `model: str` (resolves GAP B-model) — read the `model` label off the already-queried `homelab_synology_disk_temp_celsius{disk,model}` series (build a model_idx like the existing smart_idx/temp_idx; fall back to disk id), and `remain_life: float | None` (resolves GAP B-remain_life) — add a `_Q_DISK_REMAIN_LIFE` query for `homelab_synology_disk_remain_life{disk}` (LIVE value is -1 for all 8 disks = N/A for HDDs; frontend renders -1 as "N/A"/"—").

**Card-vs-contract gaps — ALL resolved by the backend extension (NO scope-out, card fully satisfied):**
- GAP A (SSH-probe-state card: last-success age / host-key / duration / up): data existed in VM but no frontend path — RESOLVED by adding the 4 fields to SynologySshProbe.
- GAP B (`remain_life` + per-disk `model`): the brainstormer initially assumed these weren't collected; recon CORRECTED this — both ARE already collected (`homelab_synology_disk_temp_celsius{model}` label + `homelab_synology_disk_remain_life`), just not surfaced in DiskRow. RESOLVED by adding the 2 DiskRow fields (no new DSM calls, existing metrics). All 8 disks live: models WD221KFGX/WD100EFAX/WD101EFBX/ST10000VN0008/HAT3310; remain_life all -1 (HDD N/A).
- GAP C (SMART-attr drill table attr_id/attr_name/raw/worst/threshold): SmartAttrRow already matches — no change. Data currently EMPTY (SSH down) → honest-empty-state.

**Frontend (Hardware-tab body, mirrors UnifiOverviewTab.tsx):** add `useSynologyHardware()` + `useSynologyDiskSmart(disk)` hooks to `api/synology.ts` (mirror useUnifiDevices). Tab body = `<div className="h-full space-y-4 overflow-y-auto p-4">` with `PanelSection` cards wrapped in `QueryState`:
- **System** (FULL-WIDTH): model/serial/firmware, `formatUptime(uptime_seconds)`, sys_temp (thresholds), `need_reboot` badge, `health_ok` badge (false→WARN amber, NOT critical — API reachable), fans (state+value chips).
- **2-col grid:** Volumes (volume, `formatPct(used_percent)`+bar, status chips) | RAID/Pool (pool, status, raid_status).
- **2-col grid:** UPS (connected/on_battery/charge%) | SSH-probe (load1/cpu_temp/mdstat + the 4 probe-state fields; honest-empty when up=false).
- **Disks SMART grid** (FULL-WIDTH `<table>`): 8 rows — disk, model, status badge, smart_status badge, temp (colored), remain_life (N/A when -1), smart_attr_failing badge, + a drill affordance per row opening a Dialog.

**Threshold/badge mappings (match the 025 strip):** temp °C green<45 / amber 45-55 / red>55; volume used_percent ok<80 / warn 80-90 / crit>90 (live ~80.7→warn); disk status/smart_status float 1.0→green "Healthy", null→muted "Unknown", other→red "Fault"; smart_attr_failing true→red "Failing"/false→green "OK"; health_ok false→amber WARN; need_reboot true→amber; UPS connected=false→red, on_battery=true→amber, charge<100→amber/100→green; volume/pool status strings via a status→tone lookup (pool_normal/normal→green, has_unverified_disk/fs_almost_full→amber, unknown→muted+raw text).

**Honest-empty-state (SSH probe + smart-attrs):** key off `ssh_probe.up` + `ssh_probe_data_available`. data_available=true & up=false → "SSH probe connection down — load, CPU temp, and per-disk SMART details unavailable. Last collector run succeeded; SSH connection to the NAS is failing." vs data_available=false → "No SSH probe data collected." remain_life=-1 → render "N/A".

**Choice 4 — drill-down = Dialog** (NOT a sub-route): a modal Dialog triggered from each disk row, lazy-fetching `useSynologyDiskSmart(disk)` on open (only the clicked disk — avoids 8 idle queries). Renders the SmartAttrRow table (attr_id/attr_name/raw/worst/threshold, null→"—"). Empty-state inside Dialog: data_available=true & attrs=[] → "SMART attributes unavailable — SSH probe connection to the NAS is down. Detailed attributes will populate once the SSH probe recovers." Need to verify a Dialog/Sheet component exists in components/ui/ at Build (if none, use the existing modal precedent or fall back to an expandable row / DrillList — Build planner decides from real components).

**Known pre-existing gap (NOT a 026 deliverable, tracked as a note):** the Unifi/EPIC-007 page does NOT render its SSH-probe-state despite the same EPIC-017 mandate — pre-existing EPIC-007 debt, out of 026 scope. Recorded as a regression/known-gap item naming EPIC-007 (026 only owns the Synology page's probe-state).

Root causes (all on the NAS, fixed with the user): (a) `homelab-probe` home dir `/var/services/homes/homelab-probe` missing → created (chown homelab-probe:users, 700); (b) login shell was `/sbin/nologin` → set to `/bin/sh` via `sed -i` on /etc/passwd (DSM `synouser` has no --setsh); (c) the deployed forced-command script `/usr/local/bin/hm-probe-synology.sh` had a corrupted/truncated hwmon loop → redeployed clean. RESULT: `homelab_ssh_up{target="synology"}=1`, load1/cpu_temp/mdstat/uptime live, AND full SMART attribute data for all 8 disks now populates (synodisk readable by the probe user — SMART is NOT honest-empty after all). 4. **SSH-probe-script canonicalization (durability fix, folded into 026 per user direction)** — the probe script had NO canonical repo source (only a NAS artifact + a test fixture), which is why it drifted/corrupted. Added `deploy/ssh-probes/hm-probe-synology.sh` (canonical LF-clean source), embedded it as `_SYNOLOGY_PROBE_SCRIPT` in `cli/ssh_probe.py` with a byte-equality test enforcing single-source-of-truth, added a Synology-specific `install-instructions` branch emitting the script + the DSM nologin `/bin/sh` shell-setting step + DSM-revert caveat, added a `sh -n` lint + marker-presence test, and updated `docs/ssh-probe-setup.md` to reference the canonical file.
