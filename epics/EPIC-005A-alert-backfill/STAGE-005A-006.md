## Design Phase

**Status**: [x] Design (Complete)

### Design Notes

All card claims re-verified against the LIVE prod rig (VM) + the EPIC-004 source + existing rules before locking. Ground truth:
- Gap A: `homelab_log_signature_cardinality_warn` is a SINGLE instance-level gauge (value 0/1; 1=over budget) — NO `service_key` label (the alert can't name the offending service; annotation points operator to investigate). Currently 0.
- Gap B (drain health): `homelab_drain_cycle_lines_total` (counter), `homelab_drain_cycle_duration_seconds` (summary pair `_count`+`_sum`, NO buckets; mean = rate(_sum)/rate(_count)), `homelab_drain_cycle_new_templates_total` (counter). All single-instance scalars. **CRITICAL CADENCE FINDING:** the drain runs on a FIXED 5-MINUTE interval (`interval_seconds=300` in `apps/monitor/homelab_monitor/kernel/logs/drain_consumer.py`); `_count` increments per CYCLE (~1529 lines/cycle, ~1 cycle/5min — NOT per-line). So `increase(_count[15m])` = ~3 at baseline; a `[15m]==0` stall window is too tight (3 cycles, marginal on restart). Use `[20m]` (4 cycles, 2-cycle buffer) → fires only after ~25min silence = 5× the interval = unambiguous stall.
- Gap C: `homelab_metric_family_dropped_series` — 14 series, label `family` (all HA metric families), value = dropped-series count per family, all 0 now.
- Gap D (per-service volume): the per-signature volume metric is `homelab_log_signature_count{service_key, template_hash, severity}` — a PER-CYCLE GAUGE (lines matched this cycle; gappy — zero-line cycles emit no sample). Use `sum_over_time(...[5m])` (NOT rate/increase — it's a gauge). `homelab_log_signature_total` is a GAUGE set to cumulative cluster_size (NOT a counter — rate() WRONG). The signature_spike.yml.tmpl (a per-SIGNATURE renderer template, NOT loaded) uses `sum_over_time(homelab_log_signature_count[5m]) > 5*max(avg_over_time(...[7d:5m]),10)`.
- Logs-arriving gate (for drain-stall, to avoid false-fire when there's legitimately no traffic): `sum(homelab_log_stream_lines_per_sec) > 1` (98-series gauge per exported_service; sum > 1 proves logs flowing; currently ~5/s).
- Existing rules NOT to double-fire: `ServiceErrorRateSpike` (`homelab_container_error_rate`, error-RATE per container), `NewLogSignature` (`homelab_log_signature_new == 1`, new pattern, DB-driven), `SignatureWentSilent` (`homelab_log_signature_silent == 1`). The new per-service spike keys on signature VOLUME aggregated per `service_key` — a distinct axis.
- LABEL CONVENTION (verified from existing log rules — the card said `routing_channel` but the REAL convention is): `severity` + `source_tool: vmalert-metrics` + `category: log-anomaly` + `anomaly_kind: <X>` + `target_kind: <Y>`. Group interval 30s/60s.

Decisions (brainstormer-assisted; coordinator picked winners, no user prompt — backend-only, no in-app surface):

- **D-005A-006-CARDINALITY-WARN → `homelab_log_signature_cardinality_warn == 1` for 5m, warning.** `anomaly_kind: cardinality_budget_exceeded`, `target_kind: log_signature`. The 5m `for:` absorbs a transient single-cycle blip. Promtool-validatable.

- **D-005A-006-DRAIN-STALL-FORM → ship TWO rules (B1 + B2), drop the too-slow form.** B1 (the dominant failure, **critical**): `increase(homelab_drain_cycle_duration_seconds_count[20m]) == 0 and sum(homelab_log_stream_lines_per_sec) > 1` for 5m — the cycle loop is dead while logs are still arriving; critical because a dead drain blinds the entire log-anomaly layer. `anomaly_kind: drain_stall`, `target_kind: drain_pipeline`. B2 (warning): `increase(homelab_drain_cycle_lines_total[20m]) == 0 and increase(homelab_drain_cycle_duration_seconds_count[20m]) > 0 and sum(homelab_log_stream_lines_per_sec) > 1` for 5m — loop alive but processing nothing (the `_count > 0` guard makes B1/B2 mutually exclusive). `anomaly_kind: drain_no_progress`, `target_kind: drain_pipeline`. **[20m] window (corrected from the card's 15m after the 5-min-cadence finding).** The too-slow rule (mean-duration threshold) DROPPED — mean is 0.351s with no SLA; any threshold is arbitrary and duration is the noisiest signal (Design finding, decided-not-to-build). Both B1/B2 promtool-validatable (no subquery).

- **D-005A-006-DROPPED-SERIES → `homelab_metric_family_dropped_series > 0` for 5m, warning, `{{ $labels.family }}`.** `anomaly_kind: metric_series_dropped`, `target_kind: metric_family`. `for: 5m` (not a count threshold) is the noise filter — suppresses transient churn while catching small-but-sustained shedding (small-sustained is the insidious case a `> N` threshold would mask). Meta-monitoring (the monitor protecting itself). Promtool-validatable.

- **D-005A-006-SPIKE-FORM → metrics adaptive per-service baseline (Form A); LogsQL absolute-threshold form REJECTED.** `sum by (service_key) (sum_over_time(homelab_log_signature_count[5m])) > clamp_min(4 * avg_over_time((sum by (service_key)(sum_over_time(homelab_log_signature_count[5m])))[7d:5m]), 50)` for 10m, warning, `{{ $labels.service_key }}`. `anomaly_kind: service_volume_spike`, `target_kind: log_service`. Each service spikes vs its OWN 7-day history — the only design that survives wildly-varying per-service baselines without per-service hand-tuning. K=4× (per-service totals are smoother than per-signature, so a tighter multiplier than the renderer's 5×); floor=50 lines/5m (the critical anti-false-fire knob — prevents a quiet service tripling from 10→40 from firing; also cold-start protection until 7d history accrues); for:10m (two 5m windows = sustained, not a one-off batch). Does NOT double-fire ServiceErrorRateSpike (error-rate, different metric/axis) or the per-signature renderer rules (per-signature, not per-service). The LogsQL absolute-threshold form (`_time:5m | stats by (service_key) count() | filter lines:>N`) was REJECTED: a single absolute threshold is simultaneously too low for chatty services (constant noise) and too high for quiet ones (misses their spike) — a guaranteed false-fire generator (Design finding). **Live-replay-only** (the `[7d:5m]` subquery is not promtool-evaluatable).

FILE DECISION: all 5 rules are METRICS rules (MetricsQL) → a SINGLE file `deploy/vmalert/metrics/log_drain_health.yaml` + `__tests__/log_drain_health.tests.yaml`. The card's "optional `deploy/vmalert/logs/log_anomalies.yaml`" (LogsQL) is NOT created — the per-service spike is better as a metrics rule with an adaptive baseline than a LogsQL absolute-threshold rule (Design finding, the LogsQL form was evaluated and rejected, not deferred).

Net deliverables: `log_drain_health.yaml` with 5 rules (CardinalityBudgetExceeded, DrainStall [critical], DrainNoProgress, MetricFamilyDroppedSeries, ServiceLogVolumeSpike) + `__tests__/log_drain_health.tests.yaml` (promtool FIRE+NO-FIRE for the 4 promtool-validatable rules; ServiceLogVolumeSpike's `[7d:5m]` subquery commented out as live-replay-only with recipe preserved). All rules: source_tool: vmalert-metrics, category: log-anomaly, + per-rule anomaly_kind/target_kind.
