# STAGE-005A-006: Logging + Drain anomaly rules (LOGS + metrics)

## Status: Not Started

## Overview

EPIC-004 built the logs pipeline (Drain clustering, signature catalog, anomaly detection) and shipped several anomaly alerts (`NewLogSignature`, `SignatureWentSilent`, `ServiceErrorRateSpike`, the log-stream budget rules). But a few EPIC-004 signals never got a rule, and the user explicitly called out **log anomalies** as a priority. This stage fills the remaining logging/Drain gaps — a mix of LogsQL (`type: vlogs`) and metrics rules.

**Verified live (scoping session):**
- `homelab_log_signature_cardinality_warn` EXISTS (1 series, gauge=1 when distinct-signature count exceeds threshold) — NO alert.
- `homelab_drain_cycle_duration_seconds_count` / `homelab_drain_cycle_lines_total` / `homelab_drain_cycle_new_templates_total` all EXIST (1 series each) — NO alerts (Drain consumer health).
- `homelab_metric_family_dropped_series` EXISTS (14 series, label `family`) — emitted by the `CappedEmitter` cardinality guard when a metric family exceeds its series budget — NO alert.
- `homelab_log_signature_new` is EMPTY at baseline (only emits when a new signature appears) — `NewLogSignature` already alerts on it (dormant-but-correct, like the collector-failure rules).
- The per-service error-rate spike (`ServiceErrorRateSpike` on `homelab_container_error_rate`) already exists. The signature-SPIKE rule has a `signature_spike.yml.tmpl` template + a `signature_spike.tests.yaml` test, but NO loaded `.yaml` rule (the spike rule is backend-only / not wired as a live coarse rule).

## Stage Flags

- Has Input Forms: No.
- Touches secrets / production data: No (rules over existing log metrics + LogsQL over the existing log stream; same privacy posture as the existing log rules — no free-text exfiltration).
- Host-integration concerns: Yes — validated against the prod rig's real log stream + Drain metrics (3a + 3b).

## Dependencies

- EPIC-004 (Drain consumer, signature catalog, `homelab_log_*` + `homelab_drain_cycle_*` metrics; the `signature_spike.yml.tmpl`).
- EPIC-001 cardinality guard (`homelab_metric_family_dropped_series`).
- vmalert-metrics + vmalert-logs + Alertmanager.

## Concrete deliverables

A metrics rule file `deploy/vmalert/metrics/log_drain_health.yaml` (+ test mirror) and possibly a logs rule file `deploy/vmalert/logs/log_anomalies.yaml` (+ `logs-test/` mirror):

1. **Signature-cardinality-warn.** `homelab_log_signature_cardinality_warn == 1` for `5m` → warning. Distinct signature count exceeded its budget (a service is generating too many distinct log shapes — log explosion / a noisy new failure mode).
2. **Drain-cycle stall.** The Drain consumer stopped processing. Two candidate forms (Design picks):
   - `rate(homelab_drain_cycle_lines_total[15m]) == 0` while logs are still arriving → the consumer is stuck. (Confirm "logs still arriving" via a log-stream metric to avoid firing when there's genuinely no log traffic.)
   - `homelab_drain_cycle_duration_seconds > <N>` → a cycle is taking abnormally long (rebuild storm). Use the histogram `_count`/`_sum` for a mean, or a threshold on the gauge if it's emitted as a gauge.
   Severity warning. This catches a silently-dead log-clustering pipeline.
3. **Metric-family cardinality drop.** `homelab_metric_family_dropped_series > 0` for `5m` → warning, `{{family}}`. The cardinality guard dropped series from a metric family (its series budget was exceeded — VM is being protected, but the operator is now BLIND to the dropped series and should investigate / raise the cap). This is a meta-monitoring signal (the monitor protecting itself).
4. **Per-service log-signature spike (the user's "logging anomalies" — coarse, always-on).** Ship the coarse per-service spike rule that the `signature_spike.yml.tmpl` template anticipates: a service suddenly emitting far more total signature volume than its baseline. Form: a LogsQL (`type: vlogs`) per-service rate-vs-baseline, OR a metrics rule over a per-signature/per-service volume metric aggregated per `service_key`. **METRIC-NAME CAVEAT (verify in Design):** the scoping session confirmed `homelab_log_signature_cardinality_warn`, `homelab_log_signature_silent`, and `homelab_log_signature_new` (conditional) exist; it did NOT confirm a per-signature `count`/`total` volume metric. EPIC-004's STAGE-004-027 ("metrics emission for signatures") is the likely source — **read it + query VM in Design to find the ACTUAL per-signature volume metric name** (it may be `homelab_log_signature_count` / `homelab_log_signature_total`, but DO NOT assume; the LogsQL form sidesteps this by counting log lines directly). Decide the coarse-but-shippable form in Design (the epic notes: ship the coarse per-service spike; full dynamic per-signature wiring is a future enhancement, possibly via EPIC-005-005 user-rule machinery). Do NOT duplicate the existing `ServiceErrorRateSpike` (that's error-rate; this is total-signature-volume spike).

## Decisions to confirm in Design

- **D-005A-006-DRAIN-STALL-FORM** — `rate(lines_total)==0` (gated on logs-arriving) vs duration-threshold vs both. Confirm whether `drain_cycle_duration_seconds` is a gauge or histogram (the `_count` suffix in VM suggests histogram — use `rate(_sum)/rate(_count)` for mean).
- **D-005A-006-SPIKE-FORM** — the coarse per-service signature-spike: LogsQL vs metrics-over-`log_signature_count`; per-`service_key` aggregation; baseline window + multiplier. Must NOT overlap `ServiceErrorRateSpike` (error-rate) or `NewLogSignature` (new-signature). Read the `signature_spike.yml.tmpl` + `signature_spike.tests.yaml` first to reuse the intended shape.
- **D-005A-006-CARDINALITY-WARN** — confirm `homelab_log_signature_cardinality_warn` semantics (gauge=1 when over budget) and pick a `for:` window.
- **D-005A-006-DROPPED-SERIES** — confirm `homelab_metric_family_dropped_series` semantics (value = count of dropped series per family?) and threshold (`> 0` any drop, or `> N`).

## Acceptance criteria

- Each new rule has a test mirror (metrics `__tests__/` and/or logs `logs-test/`) + integration test (fires on planted condition, silent at baseline).
- The signature-spike rule does NOT double-fire with `ServiceErrorRateSpike` / `NewLogSignature` (distinct fingerprints).
- Drain-stall rule does NOT false-fire when there's legitimately no log traffic.
- Dropped-series and cardinality-warn rules validated against the real (14-series) dropped-family data + the real cardinality-warn metric.
- Severity vocab; `source_tool: vmalert-metrics` or `vmalert-logs`; `category: log-anomaly` consistent with the existing log rules.
