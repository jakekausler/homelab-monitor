# STAGE-005A-005: Cron / heartbeat duration-anomaly rules

## Status: Complete

## Overview

EPIC-002's heartbeat/cron monitoring is well-alerted for **staleness/overdue/failure** (`HeartbeatStale_*`, `HeartbeatFailed`, `CronRunFailed`, `HeartbeatFlapping`, `WrapperPossiblyStale` — all live). The one gap is **run-duration anomaly**: a cron job that suddenly runs 10× longer than usual (hung/degraded) or suspiciously fast (exited early / did nothing) — neither is caught by the existing staleness rules.

This stage adds ONLY the duration-anomaly rule. It is the smallest stage in the epic.

**Verified live (scoping session):** `homelab_heartbeat_last_duration_seconds` EXISTS (labels include `name`, `host`, `fingerprint`; the 2026-06-15 snapshot saw ~2 series for the wrapper-installed scripts e.g. `backup.sh`, `rtlamr-watchdog.sh` — **re-verify the live series + which jobs have a duration metric in Design**, the set grows as wrappers are installed). There is an existing recording rule `homelab:heartbeat_runtime_p95_seconds = quantile(0.95, homelab_heartbeat_last_duration_seconds)` in `heartbeats.yaml` (confirm in Design) — the anomaly rule can build on the per-job duration directly, or reference a rolling baseline.

- Host-integration concerns: Yes — the rolling-baseline threshold tuned against REAL cron run-durations on the prod rig (the two real wrapped scripts).

- vmalert-metrics + Alertmanager.

## Concrete deliverables

A new rule file `deploy/vmalert/metrics/cron_duration.yaml` (+ `__tests__/cron_duration.tests.yaml`) containing:

1. **Cron run-duration anomaly (too slow).** A wrapped script whose latest duration is far above its own rolling baseline. Form (rolling-baseline, per `name`/`fingerprint`):
   `homelab_heartbeat_last_duration_seconds > clamp_min(<multiplier> * avg_over_time(homelab_heartbeat_last_duration_seconds[<baseline-window>]), <floor-seconds>)` for `<for>` → warning, `{{name}}`. The `clamp_min` floor prevents firing on a job that normally takes 1s suddenly taking 5s (still trivial). Tune multiplier (~5×), baseline window (e.g. 6h-1d), and floor in Refinement.
2. **(Design decides) too-fast anomaly.** A job that suddenly finishes far faster than baseline (e.g. < 0.2× baseline) can mean it exited early / did no work. Lower value; include only if it doesn't false-fire (many jobs legitimately have variable fast runs). Decide in Design — may DROP as too noisy.

## Decisions to confirm in Design

- **D-005A-005-BASELINE** — rolling-baseline window + multiplier + floor for the too-slow rule (must align with STAGE-005-013's pattern if baseline-relative). Tune against the 2 real wrapped scripts' duration history.
- **D-005A-005-TOO-FAST** — include the too-fast anomaly or DROP (noise risk). If dropped, document why (not a tracked deferral — it's a "decided not to build" not a "deferred").
- **D-005A-005-SCOPE** — only the wrapper-installed jobs have a meaningful `last_duration_seconds` (B-mode log-scrape jobs may not). Confirm the rule scopes to `wrapper_installed="yes"` or whichever subset has the duration metric.

## Acceptance criteria

- Test mirror (plants a baseline + a spiked duration → fires; baseline-only → silent).
- Tuned against the real wrapped scripts (whichever have a live `last_duration_seconds` series) so it doesn't false-fire on their normal variance.
- Does NOT overlap the existing staleness/failure rules (those fire on overdue/fail; this fires on duration anomaly while the job still succeeds).
- Severity vocab + `source_tool: vmalert-metrics` + a cron/heartbeat `routing_channel` label consistent with the existing heartbeat rules (`cron-health`).

## Design Phase

**Status**: [x] Design (Complete)

### Design Notes

All card claims re-verified against the LIVE prod rig before locking. Ground truth:
- `homelab_heartbeat_last_duration_seconds` has exactly 2 series, both `wrapper_installed="yes"`: `backup.sh` (cadence 86400s/daily, runs ~20411s = 5.67h, <1% variance — rock-stable) and `rtlamr-watchdog.sh` (cadence 300s/5min, avg ~25s but min 3.9s / max 74s = ~19× natural spread — highly variable). Labels: {name, host, fingerprint, cadence_seconds, wrapper_installed, instance, job, service}.
- `wrapper_installed="yes"` is NATIVE on this metric (no join needed). `cadence_seconds` is a LABEL (values "86400"/"300"), not a value-gauge — no `homelab_heartbeat_cadence_seconds`/`_interval_seconds` metric exists.
- **VictoriaMetrics `label_value()` IS supported live** — `label_value(homelab_heartbeat_last_duration_seconds{wrapper_installed="yes"}, "cadence_seconds")` returns 86400/300, and the comparison `metric > 0.5 * label_value(...)` parses, runs, and is correctly EMPTY at baseline (backup 0.236, rtlamr 0.100 of cadence — both well under 0.5). Caveat: `label_value()` is a VM-only function → promtool CANNOT eval it → the cadence-relative rule is LIVE-REPLAY-ONLY (got:[] under promtool by design, like STAGE-005A-004's R3).
- CRITICAL emission semantic: `homelab_heartbeat_last_duration_seconds` is a PERSISTED LAST-VALUE GAUGE (the collector re-emits the DB's last_duration_seconds every 30s until the next run). So `avg_over_time(metric[6h])` for the DAILY job ≈ the current value (self-comparison) — the rolling baseline is weak for a once-daily persisted gauge; `[1d]` window is the minimum that can ever hold two runs.
- No existing rule overlaps: the 7 heartbeat rules (`HeartbeatStale_*`, `HeartbeatFailed`, `CronRunFailed`, `HeartbeatFlapping`, `WrapperPossiblyStale`) fire on staleness/overdue/failure/flapping — NONE on run-duration. routing_channel for the cron family is `cron-health`, target_kind `cron`, source_tool `vmalert-metrics`. Group `homelab.heartbeats`, interval 30s.

Decisions (brainstormer-assisted; coordinator picked winners, no user prompt — backend-only, no in-app surface):

- **D-005A-005-BASELINE → ship TWO complementary rules (Option C, both expressible).** (1) A CADENCE-RELATIVE hang backstop `homelab_heartbeat_last_duration_seconds{wrapper_installed="yes"} > 0.5 * label_value(homelab_heartbeat_last_duration_seconds{wrapper_installed="yes"}, "cadence_seconds")` for 15m, warning — a job eating ≥50% of its own interval is anomalous; works identically for the rock-stable daily job AND the variable 5-min job, robust to the persisted-gauge problem, no false-fire (backup 23.6%, rtlamr 10% at baseline). LIVE-REPLAY-ONLY (label_value is VM-only → promtool can't eval). (2) A ROLLING-BASELINE degradation rule `homelab_heartbeat_last_duration_seconds{wrapper_installed="yes"} > clamp_min(3 * avg_over_time(homelab_heartbeat_last_duration_seconds{wrapper_installed="yes"}[1d]), 120)` for 15m, warning — catches gradual sub-cadence creep the cadence rule misses. floor=120s is the load-bearing number: it keeps rtlamr's 74s natural max silent (74 < 120) while still flagging a 5-min watchdog crossing 2min; K=3 house multiplier; window [1d] (min window where the daily job can hold two runs). This rule IS promtool-validatable (plain avg_over_time, no label_value). The two rules are complementary: cadence-relative = reliable hang detector, rolling-baseline = degradation detector.

- **D-005A-005-TOO-FAST → DROP (decided-not-to-build, not a deferral).** A too-fast rule (`metric < frac * baseline`) is unconditionally noisy on rtlamr-watchdog.sh, which legitimately runs at 0.16× its avg (3.9s vs 25s) on every OK check. To not fire on rtlamr you'd need frac < 0.156, which grazes its true min. The only meaningful too-fast case (backup.sh finishing in 1min instead of 5.67h) is already covered by the existing heartbeat staleness/failure rules (a too-short run trips overdue/streak signals). No coverage gap. Not deferred — evaluated and determined it should not exist.

- **D-005A-005-SCOPE → `wrapper_installed="yes"` only.** Native on the metric; last_duration is only meaningful when the wrapper records it. No cadence-threshold filter (both 300s and 86400s jobs are validly monitored; the 120s floor already neutralizes the short-cadence job's variance). No per-job exclusion (the floor neutralizes rtlamr without naming it — a rule that must name a series to stay quiet is wrong).

FOLLOW-UP NOTE (not a tracked deferral): a future collector change could emit `homelab_heartbeat_cadence_seconds` as a value-gauge, which would let the cadence-relative rule use a native join instead of `label_value()` and become promtool-validatable. Not needed now — `label_value()` works live.

Net deliverables: `deploy/vmalert/metrics/cron_duration.yaml` with 2 rules (CronRunDurationNearCadence cadence-relative live-replay-only; CronRunDurationAnomalous rolling-baseline promtool-validatable) + `__tests__/cron_duration.tests.yaml` (promtool FIRE+NO-FIRE for the rolling-baseline rule; cadence-relative commented out as live-replay-only with recipe). All rules: severity warning, source_tool: vmalert-metrics, target_kind: cron, routing_channel: cron-health.

BUILD-TIME RESTRUCTURE (the SPLIT): the two rules were initially authored in one `cron_duration.yaml`. promtool rejects the ENTIRE rule file when any rule uses an unknown function — RULE 2's VM-only `label_value()` is unparseable by promtool, which blocked the promtool test of RULE 1 (the promtool-validatable rule sharing the file). Root cause confirmed: `promtool test rules` loads the referenced rule file in full and fails to parse `label_value()`. Fix: split RULE 2 into its own `cron_duration_cadence.yaml` so the test mirror's `rule_files: ../cron_duration.yaml` points at a file containing only RULE 1. Both files load independently in prod vmalert (the `-rule=/etc/vmalert/metrics/*.yaml` glob picks up any .yaml; vmalert DOES support label_value). This mirrors the repo's file-isolation precedent (e.g. signature_spike.yml.tmpl is excluded from the glob). RULE 1 file: promtool check + test → SUCCESS; RULE 2 file: valid YAML, validated via live-replay in Refinement (label_value is VM-only).

D-005A-005-TOO-FAST: too-fast rule DROPPED (Design finding — noise on rtlamr's 0.16× natural fast exits; backup-too-fast already covered by existing heartbeat staleness rules).

Pre-checks: `promtool check rules cron_duration.yaml` → SUCCESS (1 rule); `promtool test rules __tests__/cron_duration.tests.yaml` → SUCCESS; `cron_duration_cadence.yaml` → valid YAML (group cron_duration_cadence / rule CronRunDurationNearCadence / interval 30s).
