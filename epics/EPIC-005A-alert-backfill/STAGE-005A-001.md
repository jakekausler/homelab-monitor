# STAGE-005A-001: Host machine threshold + anomaly rules

## Status: Complete

## Overview

Backfill vmalert (metrics) rules for host-machine signals that EPIC-001's `host` collector already emits but that have no alert. The existing `host.yaml` covers only `HostHighCPU` and `HostHighMemory` (both `>90%`, `for: 10m`). This stage adds the remaining host-resource gaps as **threshold-for-duration** rules, plus lightweight **rolling-baseline anomaly** variants for CPU/memory (the EPIC-005-013 z-score pattern, NOT Netdata/EPIC-015 ML).

Disk-fill rules (host filesystems + per-slot budgets) are explicitly OUT of this stage — they are STAGE-005A-002.

**Verified against the live rig (this scoping session):** all metrics below EXIST in VM. `homelab_host_cpu_percent` has `cpu="all"` + `cpu="0".."15"` (16 cores). `homelab_host_memory_bytes{type}` ∈ {used, total, available}. `homelab_host_swap_bytes{type}` ∈ {used, total}. `homelab_host_load_average{period}` ∈ {1m, 5m, 15m}. `homelab_host_processes_total{state}` ∈ {running, sleeping, zombie}. `homelab_host_uptime_seconds` single series. `homelab_host_disk_io_bytes_total{direction,disk}`, `homelab_host_net_bytes_total{direction,iface}` are counters.

## Stage Flags

- Has Input Forms: No.
- Touches secrets / production data: No (vmalert rules over existing host metrics).
- Host-integration concerns: Yes — thresholds tuned against REAL host data on the prod rig (Refinement 3a synthetic/fixture + 3b real-data). A noisy rule is a failed rule.

## Dependencies

- EPIC-001 (host collector emits all these metrics — confirmed live).

## Concrete deliverables

A new rule file `deploy/vmalert/metrics/host_anomalies.yaml` (and its shortened-window test mirror under `deploy/vmalert/metrics/__tests__/host_anomalies.tests.yaml`) containing:

1. **Host CPU sustained (warning + critical tiers).** Keep the existing `HostHighCPU` (`avg_over_time(homelab_host_cpu_percent{cpu="all"}[5m]) > 90`, `for: 10m`, warning) OR move/extend it here; ADD a `critical` tier (e.g. `> 95` for `10m`). Decide in Design whether to extend the existing `host.yaml` rule or add the critical tier here (avoid duplicate-fire). The user wants the "above a threshold for a certain time" form — the existing `for: 10m` already satisfies it; this adds the higher tier.
2. **Host memory sustained (warning + critical tiers).** Existing `HostHighMemory` (`used/total > 0.9`, `for: 10m`). ADD critical tier (`> 0.95` for `10m`).
3. **Swap actively used.** `homelab_host_swap_bytes{type="used"} / homelab_host_swap_bytes{type="total"} > 0.5` for `10m` → warning ("host is swapping — memory pressure"). Tune the ratio in Refinement.
4. **Load average high relative to core count.** Core count derived from `count(homelab_host_cpu_percent{cpu!="all"})` (=16; `machine_cpu_cores` is ABSENT on this rig — do NOT use it). Rule: `homelab_host_load_average{period="5m"} > 1.0 * count(homelab_host_cpu_percent{cpu!="all"})` for `10m` → warning (5m load exceeds core count = saturation). Confirm the multiplier in Design/Refinement (1.0×cores warning, maybe 2.0×cores critical).
5. **Process-count explosion.** `homelab_host_processes_total` (sum across states, or `{state="running"}`) anomaly: `sum(homelab_host_processes_total) > clamp_min(2 * avg_over_time(sum(homelab_host_processes_total)[1h]), <floor>)` for `10m` → warning (process count 2× its 1h baseline). Tune.
6. **Reboot detection.** `homelab_host_uptime_seconds` resets to ~0 on reboot. Rule: `homelab_host_uptime_seconds < 600` (uptime under 10 min = recent boot) → info/warning, `for: 0m`. (A `rate(...) < 0` form also works but the absolute-low form is simpler and less flappy.) Decide form in Design.
7. **CPU rolling-baseline anomaly (lightweight, EPIC-005-013 pattern).** CPU% far above its own rolling baseline even if under the static 90% (e.g. a box that normally sits at 10% suddenly at 60%). Form: z-score or ratio-to-baseline over a rolling window, mirroring STAGE-005-013. Severity info/warning. This is the user-called-out "machine-metric anomaly." NOT ML (that's EPIC-015).
8. **Memory rolling-baseline anomaly** — same pattern for `homelab_host_memory_bytes{type="used"}`.
9. **(Design decides, lower priority) disk-IO / network saturation** on `rate(homelab_host_disk_io_bytes_total[5m])` and `rate(homelab_host_net_bytes_total{iface!="lo"}[5m])` — sustained high throughput. These lack an obvious absolute threshold (hardware-dependent); prefer a rolling-baseline anomaly form or DEFER to a follow-up if no sensible default emerges. Document the decision.

## Acceptance criteria

- Each new rule has a shortened-window test mirror under `__tests__/` and an integration test asserting it FIRES on a planted condition and does NOT fire on baseline.
- No duplicate alerts: confirm no existing rule already covers the same fingerprint (re-verify the live rule set in Design — the audit is a point-in-time snapshot).
- Severity vocabulary `info|warning|error|critical` only; every rule carries `source_tool: vmalert-metrics` + a `target_kind: host` label per the existing convention.
- Anomaly rules are lightweight vmalert (rolling-baseline / rate-of-change), NOT ML (EPIC-015's job).
- Thresholds tuned against real prod-rig data in Refinement and documented. Conservative defaults + a note that EPIC-005-005 user-rule machinery lets the user tighten.

**Decisions (brainstormer → main-agent picks; user confirmed the two structural ones):**

- **D-005A-001-CPU-TIERS → Option (a).** Freeze `host.yaml` (keeps `HostHighCPU` + `HostHighMemory` warning rules untouched). Add CRITICAL tiers in the new `host_anomalies.yaml`: `HostCriticalCPU` (>95% 5m-avg, 10m) and `HostCriticalMemory` (used/total>0.95, 10m). Both-tier firing at >95% is intentional severity tiering (distinct fingerprints; Alertmanager keeps them distinct). Rejected (b) move+delete (churns proven rules, risks the load test) and (c) extend-in-place (pollutes the frozen threshold file; card names `host_anomalies.yaml` as the deliverable).

- **D-005A-001-LOAD-MULTIPLIER → 15m period, 2×cores warning / 4×cores critical, for 10m.** `homelab_host_load_average{period="15m"} > {2,4} * count(homelab_host_cpu_percent{cpu!="all"})`. 15m is the least-flappy sustained-saturation horizon (baseline 0.09×cores → huge headroom). `count()` yields an empty labelset (scalar-broadcast vs the single 15m series — same shape as the working `HaUpdatesPending` count() rule). Rejected 5m period (flappy on backup/build transients).

- **D-005A-001-SWAP → composite AND-gate (reformulated).** Original static `swap%>0.5` would FIRE IMMEDIATELY at the 93%-swap baseline (a failed rule). Growth-anomaly is defeated by physics (swap already near-full → no headroom to grow). Author `HostMemoryPressure` (warning, for 10m): `(homelab_host_swap_bytes{type="used"}/homelab_host_swap_bytes{type="total"} > 0.5) and (homelab_host_memory_bytes{type="used"}/homelab_host_memory_bytes{type="total"} > 0.85)`. Silent today (mem 20% ≪ 85% kills the conjunction); fires only on genuine memory pressure (mem high AND swap engaged). Rejected (a) drop-entirely and (b) growth-anomaly (no headroom).

- **D-005A-001-PROCESS-EXPLOSION → zombie-only absolute.** Author `HostZombieProcesses` (warning, for 10m): `homelab_host_processes_total{state="zombie"} > 20`. The total metric (~2 at baseline, coarse) is too flappy for a 2× clamp_min and its semantics don't mean "explosion"; zombies>20 is an unambiguous reap-failure defect signal (baseline 0). Rejected (a) total-count floor (suspect semantics) and (c) full deferral (zombie sub-signal is reliable today).

- **D-005A-001-REBOOT-FORM → absolute-low.** Author `HostRecentlyRebooted` (info, for 0m): `homelab_host_uptime_seconds < 600`. Clears naturally as uptime climbs; no counter-reset edge cases. Rejected negative-rate form (window-sensitive, fiddly thresholding).

- **D-005A-001-ANOMALY-FORM → house clamp_min idiom with floors, info severity.** `HostCPUAnomalous` (info, for 15m): `homelab_host_cpu_percent{cpu="all"} > clamp_min(3 * avg_over_time(homelab_host_cpu_percent{cpu="all"}[6h]), 40)` (floor 40 forces "3× baseline AND ≥40% absolute"). `HostMemoryAnomalous` (info, for 15m): ratio form `mem_used/mem_total > clamp_min(2 * avg_over_time((ratio)[6h]), 0.6)`; **abs-bytes fallback** if vmalert balks on the inline division-subquery: `homelab_host_memory_bytes{type="used"} > clamp_min(2 * avg_over_time(homelab_host_memory_bytes{type="used"}[6h]), 40*1024*1024*1024)`. 6h window (daily rhythm, leaner than 7d). Conservative early-warnings, not pagers. Rejected K=5/no-floor (idle-box false-fire) and [7d:5m] window (heavier, over-smooths).

- **D-005A-001-IO-NET → AUTHOR IN THIS STAGE (user directive — not deferred).** The obstacle was only picking a non-flappy floor from one snapshot; that is a Refinement-3b tuning problem (tune against real `rate()` data), not a reason to defer. ### Refinement Notes

**3a (fire/no-fire validation).** vmalert has NO unit-test mode (`-rule.testFile` does not exist — the original header comment was wrong, now corrected to a promtool invocation). Validated fire/no-fire two ways:
- **promtool** (`prom/prometheus:v2.47.0`, same rule-test schema) validates 7 of 11 rules cleanly: HostCriticalCPU, HostCriticalMemory, HostMemoryPressure, HostZombieProcesses, HostRecentlyRebooted, HostCPUAnomalous, HostMemoryAnomalous (fire-on-planted + silent-at-baseline).
- **Live-VictoriaMetrics replay** (disposable v1.107.0 container, the engine prod runs) validates the 4 MetricsQL-only rules promtool can't evaluate: HostHighLoad, HostCriticalLoad (count()-scalar-broadcast), HostDiskIOAnomalous, HostNetThroughputAnomalous (inline rate()[6h:5m] subqueries). 9/9 replay scenarios pass (fire on spike, silent on baseline, iface!="lo" excludes loopback).

**Bug found + fixed #1 — memory-division label mismatch (3 new rules + 1 pre-existing).** `homelab_host_memory_bytes{type="used"} / homelab_host_memory_bytes{type="total"}` matches NOTHING in MetricsQL (operands differ on `type`), so the division returned EMPTY and the rules could never fire. Fixed with `/ ignoring(type)` on HostCriticalMemory, HostMemoryPressure (both swap + mem operands), HostMemoryAnomalous (outer ratio + subquery). **Also fixed the IDENTICAL pre-existing bug in `host.yaml` HostHighMemory** (the host's primary memory alert had never fired in production) — per the project's fix-pre-existing-failures directive. Verified on live VM: the fixed division now returns 0.211 (21% mem) / 0.933 (93% swap); rules correctly stay silent at this healthy baseline.

**Bug found + fixed #2 — cumulative-counter inflation (disk-IO + net collector).** `apps/monitor/homelab_monitor/plugins/collectors/builtin/host.py` passed psutil's ABSOLUTE cumulative counter to `write_counter()` (which does `.inc()`) every tick, so `homelab_host_disk_io_bytes_total` / `homelab_host_net_bytes_total` accumulated `n_ticks × cumulative` (disk write read ~31 PB, rate ~2 TB/s — physically impossible). Fixed correctly (NOT deferred — user directive) by adding a new MetricsWriter method `write_counter_absolute` (Gauge-backed SET semantics) across the writer subsystem (io.py InMemory + MemoryRetaining, prometheus_writer.py, multiplex.py, process_context.py BufferingMetricsWriter, + the two type-checked test doubles) and switching the 4 host disk/net emit sites to it. `make verify` GREEN (4927 passed, 100% kernel coverage). Rig rebuilt + monitor recreated: disk-IO raw now 22.1 TB (matches `/proc/diskstats` kernel truth), per-tick deltas ~12-24 MB (real disk writes), net raw 2.2 MB / 0.018 MB/s — all sane. HostDiskIOAnomalous/HostNetThroughputAnomalous now operate on correct data.

**3b (live rig, real data).** After the rebuild: all `host` + `host_anomalies` rules load `health=ok` (no lastError); ZERO host alerts firing/pending at the real baseline (correct — mem 21%, 15m-load ≪ 32 cores, uptime 22.9d, swap-engaged-but-mem-low so the AND-gate stays silent). Net-throughput floor (50 MB/s) confirmed sane vs current peak ~42 MB/s. Disk-IO floor (50 MB/s) now meaningful against the corrected metric.
