# STAGE-005A-003: Collector health rules

## Status: Not Started

## Overview

"All plugins observe themselves" (spec §16) — the collector framework emits `homelab_collector_run_*` self-metrics, but none of them are alerted. When a collector starts failing, runs too long, or goes stale, nothing pages. This stage backfills vmalert rules so the system's own self-observation actually surfaces.

**Quarantine is intentionally NOT covered here** (user decision, this scoping session): the failure-budget machinery (`kernel/scheduler/failure_budget.py`) ALREADY synthesizes a `collector_quarantined` `AlertFiringEvent` (severity warning, with reason/timestamps/consecutive_failures) when a collector exceeds its budget. A `homelab_collector_quarantine_count` gauge + vmalert rule would duplicate that existing alert with no new signal — DROPPED. This stage alerts on the *run* metrics, which are NOT yet alerted.

**CRITICAL data finding (scoping session) — metric-absent-until-first-failure:** on a healthy rig, `homelab_collector_run_failure_total` and `homelab_collector_run_last_error_age_seconds` are **EMPTY in VM** — they are only emitted once a collector has actually failed at least once. `homelab_collector_run_success_total` and `homelab_collector_run_duration_seconds_count` DO exist (one series per registered collector — the 2026-06-15 snapshot saw ~31; **re-verify the exact series + names live in Design**). So the failure/error-age rules must be written to handle the metric being absent (the rule simply won't fire until the first failure emits the series, which is correct behavior — but the test mirror must PLANT the series to prove the rule fires).

**Histogram-component caveat (verify in Design):** only `homelab_collector_run_duration_seconds_count` was confirmed live. The companion `_sum` and `_bucket` series (needed for `histogram_quantile`/mean) are ASSUMED — **confirm `_sum`/`_bucket` exist in VM before writing a `histogram_quantile` or rate-of-sum/count expr** (read the collector's duration-metric emission in EPIC-001 to confirm the histogram is actually emitted with buckets, not just a `_count`).

## Concrete deliverables

A new rule file `deploy/vmalert/metrics/collector_health.yaml` (+ `__tests__/collector_health.tests.yaml`) containing:

1. **Sustained collector failure rate.** `rate(homelab_collector_run_failure_total[15m]) > 0` (or `increase(...[15m]) > N`) for `5m` → error, label `target_kind: collector`, `{{name}}` in annotation. A collector failing repeatedly. (Metric absent until first failure — rule is dormant-but-correct; test mirror plants it.)
2. **Collector run-duration / timeout.** Duration anomaly on `homelab_collector_run_duration_seconds` — either a static "a collector run took > its timeout" threshold, or a rolling-baseline "this collector suddenly runs 5× longer." Decide form in Design (the metric is a histogram — `homelab_collector_run_duration_seconds_count`/`_sum`/`_bucket`; use `histogram_quantile` for p95 or a rate-of-sum/count for mean). Severity warning.
3. **Last-error-age staleness.** `homelab_collector_run_last_error_age_seconds` — when present, a SMALL value means a collector errored very recently. Rule: `homelab_collector_run_last_error_age_seconds < 300` (errored within the last 5 min) → warning. (Again absent until a failure exists.) Confirm the metric's semantics in Design (is it "seconds since last error" — small = recent error = bad?).
4. **(Design decides) success-rate dropout.** Alternatively/additionally: a collector whose `homelab_collector_run_success_total` STOPPED incrementing (`rate(...[15m]) == 0` for a collector that should be running every tick) = the collector silently stopped. This catches a stuck collector that isn't emitting failures either. This uses the always-present `success_total` (one series per registered collector) so it's testable at baseline. Strongly consider including this — it may be the most valuable rule here (catches silent death, not just loud failure).

## Decisions to confirm in Design

- **D-005A-003-DURATION-FORM** — histogram p95 (`histogram_quantile(0.95, rate(...bucket[5m]))`) vs mean (`rate(_sum)/rate(_count)`) vs rolling-baseline for the run-duration rule. Read STAGE-001's collector-duration metric emission to confirm the histogram bucket layout.
- **D-005A-003-SILENT-DEATH** — include the `rate(success_total) == 0` silent-collector-death rule (recommended — uses always-present metric, catches the failure mode the failure-counter misses). Confirm which collectors are "should run every tick" vs intermittent (don't false-fire on a collector that legitimately runs rarely).
- **D-005A-003-ERROR-AGE-SEMANTICS** — confirm `last_error_age_seconds` direction (small = recent error).
- **D-005A-003-ABSENT-METRIC** — confirm the failure/error-age rules behave correctly when the series is absent (dormant, not erroring); the test mirror must plant the series.

## Acceptance criteria

- Test mirror PLANTS the failure / error-age / duration series (they're absent at baseline) and asserts the rules fire; asserts silent at healthy baseline.
- The silent-death rule (if included) tested against the always-present `success_total`.
- No double-fire with the existing `collector_quarantined` synthetic alert (these alert on run-metrics, quarantine alerts on budget-exhaustion — confirm they don't overlap fingerprints).
- Severity vocab + `source_tool: vmalert-metrics` + `target_kind: collector` labels.
- Thresholds tuned; conservative defaults to avoid paging on a single transient failure (hence the `for:` windows).
