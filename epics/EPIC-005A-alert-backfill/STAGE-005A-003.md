# STAGE-005A-003: Collector health rules

## Status: Complete

## Overview

"All plugins observe themselves" (spec §16) — the collector framework emits `homelab_collector_run_*` self-metrics, but none of them are alerted. When a collector starts failing, runs too long, or goes stale, nothing pages. This stage backfills vmalert rules so the system's own self-observation actually surfaces.

**Quarantine is intentionally NOT covered here** (user decision, this scoping session): the failure-budget machinery (`kernel/scheduler/failure_budget.py`) ALREADY synthesizes a `collector_quarantined` `AlertFiringEvent` (severity warning, with reason/timestamps/consecutive_failures) when a collector exceeds its budget. A `homelab_collector_quarantine_count` gauge + vmalert rule would duplicate that existing alert with no new signal — DROPPED. This stage alerts on the *run* metrics, which are NOT yet alerted.

**CRITICAL data finding (scoping session) — metric-absent-until-first-failure:** on a healthy rig, `homelab_collector_run_failure_total` and `homelab_collector_run_last_error_age_seconds` are **EMPTY in VM** — they are only emitted once a collector has actually failed at least once. `homelab_collector_run_success_total` and `homelab_collector_run_duration_seconds_count` DO exist (one series per registered collector — the 2026-06-15 snapshot saw ~31; **re-verify the exact series + names live in Design**). So the failure/error-age rules must be written to handle the metric being absent (the rule simply won't fire until the first failure emits the series, which is correct behavior — but the test mirror must PLANT the series to prove the rule fires).

**Histogram-component caveat (verify in Design):** only `homelab_collector_run_duration_seconds_count` was confirmed live. The companion `_sum` and `_bucket` series (needed for `histogram_quantile`/mean) are ASSUMED — **confirm `_sum`/`_bucket` exist in VM before writing a `histogram_quantile` or rate-of-sum/count expr** (read the collector's duration-metric emission in EPIC-001 to confirm the histogram is actually emitted with buckets, not just a `_count`).

## Concrete deliverables

A new rule file `deploy/vmalert/metrics/collector_health.yaml` (+ `__tests__/collector_health.tests.yaml`) containing:

1. **Sustained collector failure rate.** `rate(homelab_collector_run_failure_total[15m]) > 0` (or `increase(...[15m]) > N`) for `5m` → error, label `target_kind: collector`, `{{name}}` in annotation. A collector failing repeatedly. (Metric absent until first failure — rule is dormant-but-correct; test mirror plants it.)
2. **Collector run-duration / timeout.** Duration anomaly on `homelab_collector_run_duration_seconds` — either a static "a collector run took > its timeout" threshold, or a rolling-baseline "this collector suddenly runs 5× longer." Decide form in Design (the metric is a histogram — `homelab_collector_run_duration_seconds_count`/`_sum`/`_bucket`; use `histogram_quantile` for p95 or a rate-of-sum/count for mean). Severity warning.
3. **Last-error-age staleness.** `homelab_collector_run_last_error_age_seconds` — when present, a SMALL value means a collector errored very recently. Rule: `homelab_collector_run_last_error_age_seconds < 300` (errored within the last 5 min) → warning. (Again absent until a failure exists.) Confirm the metric's semantics in Design (is it "seconds since last error" — small = recent error = bad?).
4. **(Design decides) success-rate dropout.** Alternatively/additionally: a collector whose `homelab_collector_run_success_total` STOPPED incrementing (`rate(...[15m]) == 0` for a collector that should be running every tick) = the collector silently stopped. This catches a stuck collector that isn't emitting failures either. This uses the always-present `success_total` (one series per registered collector) so it's testable at baseline. Strongly consider including this — it may be the most valuable rule here (catches silent death, not just loud failure).

## Design Phase

**Status**: [x] Design (Complete)

### Design Notes

**Live-rig verification (this Design session):**
- Collector-identity label is **`name`** (NOT `collector`/`collector_name`). 31 collectors registered.
- `homelab_collector_run_success_total{name}` — ALWAYS PRESENT (31 series, monotonic counter).
- `homelab_collector_run_duration_seconds_count{name}` + `homelab_collector_run_duration_seconds_sum{name}` — ALWAYS PRESENT (31 series each).
- **`homelab_collector_run_duration_seconds_bucket` DOES NOT EXIST** (0 series) → `histogram_quantile` is IMPOSSIBLE; the only duration signal is the MEAN via `rate(_sum[w]) / rate(_count[w])`.
- `homelab_collector_run_failure_total{name,reason}` — **DORMANT** (absent until a collector's first failure; currently 2 series: ha_battery, ha_persistent_notification, reason="result_error").
- `homelab_collector_run_last_error_age_seconds{name}` — **DORMANT** (currently 2 series, ~10.5h old).
- Existing coverage: ZERO collector-health vmalert rules (genuine gap). `collector_quarantined` is synthesized in `failure_budget.py` as an AlertFiringEvent at the HARD failure budget (5) — labels `collector_name` + source `scheduler` (distinct fingerprint from these vmalert rules which use `name` + `source_tool: vmalert-metrics`). Do NOT duplicate it.

**Decisions (brainstormer → main-agent picks; no user prompt per standing instruction):**

- **D-005A-003-FAILURE-RATE → (b) failure RATIO.** `collector_failure_rate_elevated` (warning, for 15m): `rate(homelab_collector_run_failure_total[15m]) / (rate(homelab_collector_run_success_total[15m]) + rate(homelab_collector_run_failure_total[15m])) > 0.5`. Catches a collector failing >50% of runs — "persistently erroring but recovering between runs, never hits the 5-in-a-row quarantine budget." Dormant-metric semantics correct: a stale one-off failure (counter stuck at 1, no new increments) → rate≈0 → won't fire. Rejected (a) `rate(failure_total)>0` (fires on any failure, too noisy) and (c) `increase(failure_total)>N` (ignores cadence; drifts toward duplicating quarantine).

- **D-005A-003-SILENT-DEATH → (a) blanket long-window, exclude noop.** `collector_silent_death` (error, for 30m): `rate(homelab_collector_run_success_total{name!="noop"}[1h]) == 0`. The highest-value rule. `success_total` always present → a stopped counter yields rate==0 exactly. `[1h]` window + `for 30m` = ~1.5h of silence before firing, conservatively longer than any legit collector cadence (no per-collector cadence metric exists, so the window must be conservative). `noop` excluded (deliberate no-op, not a health signal). `error` severity (a dead collector is worse than an erroring-but-running one). Rejected (b) hardcoded exclusion allow-list (brittle, drifts, can mask real deaths) and (c) `increase()==0` (rate is the repo idiom).

- **D-005A-003-LAST-ERROR-AGE → DROP (documented Design decision, NOT silent scope removal).** `last_error_age_seconds` is dormant and counts UP from the error moment; its only actionable form (`< X` = recent error) is STRICTLY DOMINATED by `collector_failure_rate_elevated`, which is cadence-normalized and suppresses one-off errors. A standalone low-age rule would re-fire on single recent errors the ratio rule deliberately ignores (re-introducing the exact noise rejected in D-FAILURE-RATE option a). This is a deliberate redundancy FINDING exercised under the card's "Decisions to confirm in Design" latitude — the recent-error signal is owned by `collector_failure_rate_elevated`; we are concluding the metric shouldn't have its own rule, NOT deferring it to a future stage (no tracking stage needed — if a distinct stale-error signal is ever wanted it's a new request). Rejected (a) `last_error_age < 300` (redundant + noisier) and (c) large-age stale signal (a recovered collector is not actionable).

- **D-005A-003-DURATION → (b) rolling-baseline mean-duration anomaly, info.** `collector_duration_anomaly` (info, for 15m): `(rate(homelab_collector_run_duration_seconds_sum[15m]) / rate(homelab_collector_run_duration_seconds_count[15m])) > clamp_min(3 * avg_over_time((rate(sum[15m]) / rate(count[15m]))[6h:15m]), 1)`. No buckets → no histogram_quantile; mean is the only duration signal. 31 collectors have wildly different normal durations → an absolute threshold is wrong; a per-collector 6h rolling baseline (K=3, the STAGE-005A-001 clamp_min idiom) catches a collector slowing vs ITSELF. `clamp_min(..., 1)` floor of 1s prevents a fast collector (0.01s→0.05s) from firing on noise. `info` severity (soft signal; a hung collector is caught harder by silent-death). Rejected (a) absolute threshold (one number can't fit host vs noop) and (c) drop (the baseline anomaly is a cheap info-level early degradation indicator nothing else covers).

- **D-005A-003-DORMANT-TEST → plant dormant series; classify validation.** The test mirror PLANTS `failure_total` + paired `success_total` (drive the ratio past 0.5 + a no-fire ratio<0.5 baseline); plants flat `success_total` (drive silent-death + an incrementing no-fire + a flat `noop` exclusion no-fire); plants `duration_sum`+`duration_count` baseline-then-step (drive the anomaly + a sub-floor fast-collector no-fire). No last_error_age planting (rule dropped). **Validation:** `collector_failure_rate_elevated` + `collector_silent_death` are promtool-validatable (plain rate/ratio/threshold). `collector_duration_anomaly` uses an `avg_over_time(...[6h:15m])` SUBQUERY → NOT promtool-validatable → validate via the STAGE-005A-001 live-VictoriaMetrics-replay path.

**Final rule set for `deploy/vmalert/metrics/collector_health.yaml` (3 rules; + `__tests__/collector_health.tests.yaml`). All carry `source_tool: vmalert-metrics` + `target_kind: collector`, keyed on `name`:**
1. `collector_failure_rate_elevated` (warning, for 15m) — failure ratio > 0.5. [promtool-validatable]
2. `collector_silent_death` (error, for 30m) — `rate(success_total{name!="noop"}[1h]) == 0`. [promtool-validatable]
3. `collector_duration_anomaly` (info, for 15m) — mean-duration > clamp_min(3 × 6h-baseline, 1s). [live-replay validation — subquery]

No edits to existing files. No recording rules. `last_error_age_seconds` rule intentionally not authored (see D-005A-003-LAST-ERROR-AGE).

**Bug found + fixed in 3b — `collector_silent_death` chronically false-fired on `image_update_checker`.** 3b found the rule PENDING for `image_update_checker`. Investigation (config + 24h VM history): `image_update_checker` runs every ~6-7h BY DESIGN (configured 6h interval; hits rate-limited container registries), is HEALTHY, but the `[1h]`+30m window flagged it as silent. It is the ONLY collector slower than ~1h (all other 30 run every 10s–5min; `local_build_update_checker` runs ~10m, safe). A noisy rule is a failed rule, so this needed fixing. **Fix (brainstormer → two-tier cadence-class split):** the original `collector_silent_death` became a FAST rule `rate(success_total{name!~"noop|image_update_checker"}[1h]) == 0 for 30m` (~90min detection for the 30 fast collectors) PLUS a new SLOW rule `collector_silent_death_slow` = `rate(success_total{name=~"image_update_checker"}[12h]) == 0 for 30m` (~12.5h detection; the [12h] window safely exceeds the ~7h cadence so it never false-fires on the healthy collector, yet still covers it if genuinely stuck). Every collector now has silent-death coverage with zero false-fire. LINKED PAIR: a future >1h-cadence collector must be added to BOTH the fast rule's `name!~` exclusion AND the slow rule's `name=~` set. Known edge: on a FRESH deploy the slow rule's `[12h]` rate reads 0 until 12h of history accrues — a one-time ~12.5h post-deploy window where it could fire spuriously (accepted/documented).
