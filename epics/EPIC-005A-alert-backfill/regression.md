## STAGE-005A-001 — Host machine threshold + anomaly rules

- `deploy/vmalert/metrics/host_anomalies.yaml` loads on the live vmalert with all 11 rules `health=ok` (check `docker exec homelab-vmalert-metrics sh -c "wget -qO- http://127.0.0.1:8880/api/v1/rules"`). No false-fire at a healthy baseline (check `/api/v1/alerts` — no `target_kind=host` alert firing/pending).
- Memory-ratio rules must use `/ ignoring(type)` — a bare `homelab_host_memory_bytes{type="used"} / homelab_host_memory_bytes{type="total"}` matches nothing and the rule silently never fires. Applies to HostCriticalMemory, HostMemoryPressure, HostMemoryAnomalous (host_anomalies.yaml) AND HostHighMemory (host.yaml). Regression check: query the division on the live VM and confirm it returns a value (not empty).
- `homelab_host_disk_io_bytes_total` / `homelab_host_net_bytes_total` must read SANE absolute values (kernel-truth ~TB/GB, not PB/EB) and `rate()` in MB/s (not TB/s). Regression check: `rate(homelab_host_disk_io_bytes_total[5m])` per-series should be plausible disk throughput. If inflated, the host collector regressed to passing psutil cumulative values through `write_counter` (.inc) instead of `write_counter_absolute` (Gauge set).
- Rule-test mirror `__tests__/host_anomalies.tests.yaml`: run `docker run --rm --entrypoint promtool -v $(pwd)/deploy/vmalert/metrics:/rules prom/prometheus:v2.47.0 test rules /rules/__tests__/host_anomalies.tests.yaml` — the 9 promtool-validatable rules must pass; the 2 MetricsQL-only rules (HostDiskIOAnomalous, HostNetThroughputAnomalous, which use a `rate(...)[w:s]` subquery) show `got:[]` under promtool by design (validated via live-MetricsQL replay instead). The load rules (HostHighLoad/HostCriticalLoad) use `scalar(count(...))` and ARE promtool-validatable.

## STAGE-005A-002 — Disk-fill rules (host fs + per-slot budgets)

Regression check: `/api/v1/rules` shows the 4 rules health=ok; `/api/v1/alerts` shows no disk_fill alert firing/pending.
- Host-fs rules MUST use `/ ignoring(type)` (used/total differ on `type` → bare division empty). Per-slot rules MUST NOT use ignoring (used_bytes and budget_bytes share the `{slot}` label → native join) and MUST keep `{slot!="vl"}` (vl has dedicated VictoriaLogsDisk* rules) + the `(homelab_self_disk_budget_bytes > 0)` divide-by-zero guard. Regression check: query both exprs on the live VM — host-fs returns a value per mountpoint, per-slot returns a value per non-vl slot with `slot="vl"` ABSENT.
- Rule-test mirror `__tests__/disk_fill.tests.yaml`: `docker run --rm --entrypoint promtool -v $(pwd)/deploy/vmalert/metrics:/rules prom/prometheus:v2.47.0 test rules /rules/__tests__/disk_fill.tests.yaml` → SUCCESS (all 4 rules promtool-validatable; FIRE + NO-FIRE + 2 vl-exclusion cases). FIRE-case `exp_annotations` assert BOTH the summary AND the rendered `description` (which embeds the computed % value).
- Do NOT double-fire the `vl` slot: the generic SelfDiskSlotFill rules exclude `slot="vl"`; `VictoriaLogsDiskWarning/Critical` in `vl_health.yaml` own the vl slot. Regression check: a high-usage vl slot fires VictoriaLogsDisk* but NOT SelfDiskSlotFill*.

## STAGE-005A-003 — Collector health rules

- `deploy/vmalert/metrics/collector_health.yaml` loads on the live vmalert with all 4 rules `health=ok` (collector_failure_rate_elevated, collector_silent_death, collector_silent_death_slow, collector_duration_anomaly) and does NOT false-fire at the real baseline. Regression check: `/api/v1/rules` shows the 4 rules health=ok; `/api/v1/alerts` shows no collector_health alert firing/pending (all 31 collectors succeeding regularly; none failing >50%; image_update_checker NOT flagged despite its 6-7h cadence).
- `collector_failure_rate_elevated` MUST use `sum without(reason)(rate(failure_total[15m]))` (NOT a bare `rate(failure_total)`): `failure_total` carries `{name,reason}` while `success_total` carries `{name}`, so a bare `rate(success)+rate(failure)` matches nothing → empty → rule never fires (the STAGE-005A-001 label-mismatch class). Regression check: query the denominator `rate(success_total[15m]) + sum without(reason)(rate(failure_total[15m]))` on the live VM — it must return a value (non-empty) for collectors that have failed.
- `collector_silent_death` is a CADENCE-CLASS PAIR with `collector_silent_death_slow`. The fast rule `{name!~"noop|image_update_checker"}[1h]` covers the ~30 collectors running ≤5min; the slow rule `{name=~"image_update_checker"}[12h]` covers the ~6-7h registry checker. `image_update_checker` runs every ~6-7h BY DESIGN (rate-limited) — it MUST stay excluded from the fast rule or it chronically false-fires. Regression check: `image_update_checker` does NOT appear in firing/pending collector_silent_death alerts. A future >1h-cadence collector must be added to BOTH rules (fast exclusion + slow inclusion).
- `collector_duration_anomaly` uses a `[6h:15m]` subquery → NOT promtool-validatable (shows empty under promtool by design); validated via live-VictoriaMetrics replay (fires on a mean-duration spike > clamp_min(3×6h-baseline, 1s); silent below the 1s floor). The other 3 rules ARE promtool-validatable: `docker run --rm --entrypoint promtool -v $(pwd)/deploy/vmalert/metrics:/rules prom/prometheus:v2.47.0 test rules /rules/__tests__/collector_health.tests.yaml` → SUCCESS.
