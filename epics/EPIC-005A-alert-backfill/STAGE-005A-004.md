# STAGE-005A-004: Container lifecycle + resource rules + probe enablement

## Status: Not Started

## Overview

EPIC-003 shipped the Docker collector + cadvisor + per-container probes, but several container signals have no alert, the per-container RESOURCE metrics (cadvisor) are entirely unalerted, and the live `DockerProbeFailing` alert is **dormant** because no probes are actually configured to run. **EPIC-003 is complete**, so all remaining container-monitoring wiring lands HERE (user decision, this scoping session) — including making probes actually run.

This is the largest alerting stage. It groups three related container concerns:
1. **Container lifecycle** rules (restart loops, bad exit codes, stuck non-running, registry-skip).
2. **Container resource** rules (per-container CPU/memory as **share-of-host**, user choice (b)).
3. **Probe enablement** — configure label-based probes so `homelab_probe_up` actually emits and `DockerProbeFailing` becomes live, not dormant.

**Verified live (scoping session):**
- `homelab_container_restart_count` (41 series, label `name`), `homelab_container_last_exit_code` (41), `homelab_container_status{state}` (41; states include running/exited/paused/restarting/dead), `homelab_image_update_check_skipped{reason,registry}` (emitted on rate-limit/cooldown skip).
- cadvisor: `container_cpu_usage_seconds_total{name,image,id}` (37), `container_memory_working_set_bytes{name}` (37). **`container_spec_memory_limit_bytes` and `machine_cpu_cores` are ABSENT** (no per-container limits set) — so memory % and CPU normalization-by-cores are NOT computable. Use **share-of-host** instead: container working-set ÷ `homelab_host_memory_bytes{type="total"}`, and container CPU cores ÷ `count(homelab_host_cpu_percent{cpu!="all"})` (=16).
- `homelab_probe_up` / `homelab_probe_duration_seconds` are EMITTED by `probe_supervisor.py` but EMPTY in VM (no probes discovered/configured). The `DockerProbeFailing` alert (`homelab_probe_up == 0`) is correct-but-dormant.

## Concrete deliverables

A new rule file `deploy/vmalert/metrics/container_lifecycle.yaml` (+ `__tests__/container_lifecycle.tests.yaml`) and the probe-enablement work:

### Lifecycle rules
1. **Restart loop.** `increase(homelab_container_restart_count[1h]) > N` (N tuned, e.g. 3) for `0m` → warning, `{{name}}`. A container restarting repeatedly.
2. **Bad exit code.** `homelab_container_last_exit_code != 0 and homelab_container_last_exit_code != 137 and homelab_container_last_exit_code != 143` (137=SIGKILL/OOM, 143=SIGTERM are "normal stop") → warning, `{{name}}`. Confirm the exit-code allowlist in Design.
3. **Stuck non-running (whitelist).** `homelab_container_status{state="exited"} == 1` for `5m` on a container in a configured "should be running" whitelist → warning. Requires a whitelist (no auto-detection of "expected running"). Decide the whitelist mechanism in Design (a config list, like watched-dirs; default empty or a sensible set). Without a whitelist this rule can't avoid false-firing on intentionally-stopped containers.
4. **Registry rate-limit skip (user choice: the skipped one).** `homelab_image_update_check_skipped == 1` (or `count(...) > 0`) → info/warning, `{{registry}}`/`{{reason}}`. An image-update check was skipped due to rate-limiting/cooldown. (User chose the consequence signal, NOT `rate_limit_remaining < N`.)

### Resource rules (share-of-host — user choice (b))
5. **Container CPU share-of-host.** `rate(container_cpu_usage_seconds_total{name!=""}[5m]) / count(homelab_host_cpu_percent{cpu!="all"}) > <frac>` for `10m` → warning, `{{name}}`. A single container consuming > X% of the box's cores, sustained. (`name!=""` excludes the root cgroup.) Tune `<frac>` (~0.5 = half the cores) in Refinement.
6. **Container memory share-of-host.** `container_memory_working_set_bytes{name!=""} / on() group_left() homelab_host_memory_bytes{type="total"} > <frac>` for `10m` → warning, `{{name}}`. A single container using > X% of host RAM, sustained. Tune `<frac>` (~0.25) in Refinement. (Use `working_set_bytes`, not `usage_bytes` — excludes reclaimable cache.) Confirm the cross-metric join (`on() group_left()` to broadcast the scalar host-total across container series) works in MetricsQL.

### Probe enablement (make DockerProbeFailing live)
7. **Configure label-based probes so `homelab_probe_up` actually emits.** EPIC-003's probe auto-config is label-driven; nothing is labeled, so no probes run. This stage configures a default/sensible set of probes (per Design — likely HTTP/TCP probes on key containers) so the probe supervisor emits `homelab_probe_up`/`homelab_probe_duration_seconds`, making the existing `DockerProbeFailing` alert (and a NEW `DockerProbeSlow` on `homelab_probe_duration_seconds`) actually fire on real failures. Decide in Design: which containers get probes, and whether this is config (labels/overrides) or a code change. This is the "EPIC-003 is done so wire it here in full" deliverable.
8. **`DockerProbeSlow` alert** — `homelab_probe_duration_seconds > <N>` for `5m` → warning, once probes emit.

## Decisions to confirm in Design

- **D-005A-004-EXIT-ALLOWLIST** — which exit codes are "normal" (0, 137, 143?) and excluded from the bad-exit alert.
- **D-005A-004-RUNNING-WHITELIST** — mechanism + default for the "should be running" whitelist (stuck-non-running rule). Config list pattern.
- **D-005A-004-RESOURCE-FRAC** — share-of-host fractions (CPU ~0.5, mem ~0.25) and the cross-metric join syntax (verify `on()/group_left()` against MetricsQL on the rig).
- **D-005A-004-PROBE-SCOPE** — which containers get default probes, probe kinds (HTTP/TCP/exec), config-vs-code, and whether any probe-config UI is touched. (This is the biggest open question — probe enablement could itself be sizeable; Design may split it into a sub-stage if it balloons.)
- **D-005A-004-FIXTURE-CLEANUP** — this stage (or the epic's first stage) removes the leaked `deploy/vmalert/metrics/fixture.yaml` `FixtureHostHighCPU` rule (fires on `fixture_cpu_percent` which doesn't exist in prod) OR confirms it's intentionally a test artifact that shouldn't load in prod. Decide where the fixture cleanup lands.

## Acceptance criteria

- Test mirror + integration tests for every lifecycle + resource rule (fire on planted condition, silent at baseline).
- Resource rules validated against REAL container behavior — no false-fire from normal usage (homeassistant legitimately uses more than a sidecar).
- Probe enablement: `homelab_probe_up` actually emits for the configured probes on the prod rig; `DockerProbeFailing` demonstrably fires when a probed container is unhealthy (planted) and is silent when healthy.
- The dormant→live transition for `DockerProbeFailing` documented in regression.md (it was correct-but-dormant; now it has real data).
- `fixture.yaml` cleanup resolved (removed or confirmed test-only-not-loaded-in-prod).
- Severity vocab + `source_tool: vmalert-metrics` + `target_kind: docker_container` labels.
