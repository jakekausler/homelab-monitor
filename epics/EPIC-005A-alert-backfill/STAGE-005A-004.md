# STAGE-005A-004: Container lifecycle + resource rules + probe enablement

## Status: Complete

## Overview

EPIC-003 shipped the Docker collector + cadvisor + per-container probes, but several container signals have no alert, the per-container RESOURCE metrics (cadvisor) are entirely unalerted, and the live `DockerProbeFailing` alert is **dormant** because no probes are actually configured to run. **EPIC-003 is complete**, so all remaining container-monitoring wiring lands HERE (user decision, this scoping session) — including making probes actually run.

This is the largest alerting stage. It groups three related container concerns:
1. **Container lifecycle** rules (restart loops, bad exit codes, stuck non-running, registry-skip).
2. **Container resource** rules (per-container CPU/memory as **share-of-host**, user choice (b)).
3. **Probe enablement** — configure label-based probes so `homelab_probe_up` actually emits and `DockerProbeFailing` becomes live, not dormant.

**Verified live (scoping session):**
- `homelab_container_restart_count` (41 series, label `name`), `homelab_container_last_exit_code` (41), `homelab_container_status{state}` (41; states include running/exited/paused/restarting/dead), `homelab_image_update_check_skipped{reason,registry}` (emitted on rate-limit/cooldown skip).
- cadvisor: `container_cpu_usage_seconds_total{name,image,id}` (37), `container_memory_working_set_bytes{name}` (37). **`container_spec_memory_limit_bytes` and `machine_cpu_cores` are ABSENT** (no per-container limits set) — so memory % and CPU normalization-by-cores are NOT computable. Use **share-of-host** instead: container working-set ÷ `homelab_host_memory_bytes{type="total"}`, and container CPU cores ÷ `count(homelab_host_cpu_percent{cpu!="all"})` (=16).
- `homelab_probe_up` / `homelab_probe_duration_seconds` are EMITTED by `probe_supervisor.py` but EMPTY in VM (no probes discovered/configured). The `DockerProbeFailing` alert (`homelab_probe_up == 0`) is correct-but-dormant.

## Concrete deliverables

A new rule file `deploy/vmalert/metrics/container_lifecycle.yaml` (+ `__tests__/container_lifecycle.tests.yaml`) and the probe-enablement work:

### Lifecycle rules
1. **Restart loop.** `increase(homelab_container_restart_count[1h]) > N` (N tuned, e.g. 3) for `0m` → warning, `{{name}}`. A container restarting repeatedly.
2. **Bad exit code.** `homelab_container_last_exit_code != 0 and homelab_container_last_exit_code != 137 and homelab_container_last_exit_code != 143` (137=SIGKILL/OOM, 143=SIGTERM are "normal stop") → warning, `{{name}}`. Confirm the exit-code allowlist in Design.
3. **Stuck non-running (whitelist).** `homelab_container_status{state="exited"} == 1` for `5m` on a container in a configured "should be running" whitelist → warning. Requires a whitelist (no auto-detection of "expected running"). Decide the whitelist mechanism in Design (a config list, like watched-dirs; default empty or a sensible set). Without a whitelist this rule can't avoid false-firing on intentionally-stopped containers.
4. **Registry rate-limit skip (user choice: the skipped one).** `homelab_image_update_check_skipped == 1` (or `count(...) > 0`) → info/warning, `{{registry}}`/`{{reason}}`. An image-update check was skipped due to rate-limiting/cooldown. (User chose the consequence signal, NOT `rate_limit_remaining < N`.)

### Resource rules (share-of-host — user choice (b))
5. **Container CPU share-of-host.** `rate(container_cpu_usage_seconds_total{name!=""}[5m]) / count(homelab_host_cpu_percent{cpu!="all"}) > <frac>` for `10m` → warning, `{{name}}`. A single container consuming > X% of the box's cores, sustained. (`name!=""` excludes the root cgroup.) Tune `<frac>` (~0.5 = half the cores) in Refinement.
6. **Container memory share-of-host.** `container_memory_working_set_bytes{name!=""} / on() group_left() homelab_host_memory_bytes{type="total"} > <frac>` for `10m` → warning, `{{name}}`. A single container using > X% of host RAM, sustained. Tune `<frac>` (~0.25) in Refinement. (Use `working_set_bytes`, not `usage_bytes` — excludes reclaimable cache.) Confirm the cross-metric join (`on() group_left()` to broadcast the scalar host-total across container series) works in MetricsQL.

### Probe enablement (make DockerProbeFailing live)
7. **Configure label-based probes so `homelab_probe_up` actually emits.** EPIC-003's probe auto-config is label-driven; nothing is labeled, so no probes run. This stage configures a default/sensible set of probes (per Design — likely HTTP/TCP probes on key containers) so the probe supervisor emits `homelab_probe_up`/`homelab_probe_duration_seconds`, making the existing `DockerProbeFailing` alert (and a NEW `DockerProbeSlow` on `homelab_probe_duration_seconds`) actually fire on real failures. Decide in Design: which containers get probes, and whether this is config (labels/overrides) or a code change. This is the "EPIC-003 is done so wire it here in full" deliverable.
8. **`DockerProbeSlow` alert** — `homelab_probe_duration_seconds > <N>` for `5m` → warning, once probes emit.

### Design Notes

All card claims re-verified against the LIVE prod rig (VictoriaMetrics + vmalert `/api/v1/rules`) before locking. Key verifications:
- Memory share-of-host join `container_memory_working_set_bytes{name!=""} / on() group_left() homelab_host_memory_bytes{type="total"}` returns **36 non-empty series** (the #1 recurring bug — confirmed working). CPU share `rate(container_cpu_usage_seconds_total{name!=""}[5m]) / count(homelab_host_cpu_percent{cpu!="all"})` also 36 non-empty series. Host denominators: `homelab_host_memory_bytes{type="total"}`=66752569344 (single series, only `type` differentiates); `count(homelab_host_cpu_percent{cpu!="all"})`=16.
- `homelab_container_status{name,state}` value is always 1 (presence flag); `state` ∈ {running,exited,created}. Only labels are {name,id,instance,job,service,state} — NO restart-policy label.
- `homelab_container_last_exit_code{name}` is a persisted gauge holding the LAST exit code (history, not event): live values include 128 (gm-redis, exited) and 1 (campaign_playwright, exited).
- `homelab_image_update_check_skipped{reason,registry}` EMPTY (absent until a skip event).
- `homelab_probe_up` / `homelab_probe_duration_seconds` EMPTY (no probes configured → `DockerProbeFailing` dormant-but-correct).
- Probe subsystem is FULLY BUILT in EPIC-003 (ProbeSupervisor, label_parser, probe_executor http/tcp/exec/metrics, probe_resolver sentinel, probe_targets DB, full CRUD API + complete UI: ProbeListPanel/AddProbeModal/EditProbeModal). Nothing to build — only config to add.
- No existing rule covers restart_count, exit_code, stuck-non-running, cadvisor per-container CPU/mem, image_update_check_skipped, or DockerProbeSlow. `DockerProbeFailing` exists (`homelab_probe_up == 0` for 2m).
- Confirmed sidecar health endpoints (for probe labels): monitor `/api/healthz`:9090, grafana `/api/health`:3000, victoriametrics `/health`:8428, alertmanager `/-/healthy`:9093. The `container` sentinel resolves to the probed container's own IP; probe HTTP client runs from the monitor process. Compose `labels:` reach the discoverer via `Config.Labels`.

Decisions (brainstormer-assisted; coordinator picked winners, no user prompt — no in-app React surface so no Desktop+Mobile approval needed):

- **D-005A-004-EXIT-ALLOWLIST → DROP the standalone bad-exit-code rule (FINDING, not deferral).** `homelab_container_last_exit_code` is a persisted gauge holding the LAST exit code — it describes the past, not a live event, and would false-fire forever on any container whose last exit was non-clean (live: gm-redis=128, campaign_playwright=1). The crash EVENT is already alerted by `ContainerCrashed` (`increase(homelab_container_crash_total[5m])>0`) and the new restart-loop rule. A persisted-gauge exit-code rule cannot satisfy the no-false-fire-at-baseline criterion. Not deferred — evaluated and determined the rule should not exist.

- **D-005A-004-RUNNING-WHITELIST → DROP the standalone stuck-non-running rule (FINDING, not deferral), coupled with D-PROBE-SCOPE.** "Should be running" is not derivable from any current metric (no restart-policy metric — verified) and baking host-specific container names into this PUBLIC repo is disqualified (open-source-safe defaults). A configured probe IS the canonical, host-safe, per-container declaration of "I expect this up" — `DockerProbeFailing` already covers it. A container intentionally stopped has no probe → correctly silent. Replaced by probe enablement (D-PROBE-SCOPE), not deferred.

- **D-005A-004-RESOURCE-FRAC → mem warn `>0.5` for 15m + mem critical `>0.85` for 10m; CPU warn `>0.5` for 15m (single-tier).** Healthy baseline is tiny (largest mem container ~0.01 of host), so huge headroom. Share-of-host means "one container eating half the box" — operator-actionable → severity warning (critical for mem `>0.85`). `for:15m` is the anti-false-fire guard (resource share is the spikiest signal; long `for` filters transient backup/restart/cache-warm spikes). No CPU critical tier (CPU is spikier; risks false-fire on legit batch work). Joins verified non-empty live.

- **D-005A-004-PROBE-SCOPE → enable probes via PUBLIC compose http labels on first-party sidecars + document the /config host-override path; include DockerProbeSlow; NO SPLIT.** Add `homelab-monitor.http.health` labels to monitor (`http://container:9090/api/healthz`), grafana (`http://container:3000/api/health`), victoriametrics (`http://container:8428/health`), alertmanager (`http://container:9093/-/healthy`) in `deploy/compose/docker-compose.yml` — host-agnostic, ships in the public repo, makes `homelab_probe_up` emit for every user and `DockerProbeFailing` demonstrable universally. http kind only (every first-party service has a health endpoint); exec excluded (double-gated/risky). Document the `/config/plugins/docker/<container>.yaml` override path for the user's real services. Add `DockerProbeSlow` (`homelab_probe_duration_seconds > 2` for 5m, warning) — trivial once probes emit, completes the probe-alerting surface. **NO 004a/004b split**: the original split presumed probe-subsystem BUILD work; the subsystem is complete (EPIC-003), so this stage is just lifecycle/resource rules + compose labels + DockerProbeSlow + docs — one cohesive session-sized stage. No in-app React surface added (probe UI already exists) → no Desktop+Mobile Refinement prompt.

- **D-005A-004-FIXTURE-CLEANUP → KEEP `fixture.yaml`; confirm intentionally test-only (REVERSED the "dead rule" premise).** `FixtureHostHighCPU` is NOT dead: `apps/monitor/tests/integration/test_canonical_e2e.py` actively uses it as the canonical end-to-end smoke test (`rig.wait_for_alert("FixtureHostHighCPU")`), and `apps/monitor/tests/fixtures/fixture-host/fixture_host.py` emits `fixture_cpu_percent` in the TEST stack. The rule stays inactive in PROD only because the fixture metric never exists there — that is deliberate (a shared rules-glob test artifact, not a leak). Deleting it would break `test_canonical_e2e.py`. Resolution: keep it, confirmed intentionally test-only.

Net rule deliverables (down from the card's 8, after the two FINDINGS): `container_lifecycle.yaml` with (1) restart-loop `increase(homelab_container_restart_count[1h])>3` warning `{{name}}`; (2) image-update-skipped `homelab_image_update_check_skipped == 1` info `{{registry}}`/`{{reason}}`; (3) CPU share warn `>0.5` for 15m; (4) mem share warn `>0.5` for 15m; (5) mem share critical `>0.85` for 10m; (6) `DockerProbeSlow` `homelab_probe_duration_seconds > 2` for 5m warning. Plus compose probe labels, an override-config docs note, and the test-mirror `__tests__/container_lifecycle.tests.yaml`. Labels on all rules: severity, source_tool: vmalert-metrics, target_kind: docker_container, integration: docker.
