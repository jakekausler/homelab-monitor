# STAGE-005A-009: Host temperature metrics + sustained alerts + Grafana panel

## Status: Complete

## Overview

Add host thermal monitoring: a new collector emits per-sensor temperatures, vmalert rules alert on
**sustained** over-temperature (the user's core ask) AND on rolling-baseline thermal anomalies, and a
single time-series Grafana panel plots all sensors. This is a beyond-rules stage (like 004/008): it
adds a Python collector helper + a new metric family + vmalert rules + a dashboard panel.

**Verified against the live rig (this scoping session):** `psutil.sensors_temperatures()` on this host
returns three chips — `k10temp` (label `Tctl`, AMD CPU package, ~54°C), `nvme` (label `Composite`, SSD,
~46°C, self-reports `high=89.85`/`crit=93.85`), `amdgpu` (label `edge`, GPU, ~42°C). No fans
(`sensors_fans()` empty), no battery. **Sensors are readable inside the running `homelab-monitor`
container with NO compose change** — Docker mounts a default read-only sysfs (`/sys/class/hwmon` is
present), and hwmon is a host-global (non-namespaced) subsystem, so the existing container already sees
host sensors. (Design 3b MUST re-confirm this in the actual prod container, not just an exec probe; if a
future base-image/runtime change ever hides `/sys/class/hwmon`, the fallback is a `/sys:/sys:ro` mount on
the monitor service, mirroring cadvisor — but it is NOT needed today.)

User-confirmed scope (2026-06-15): **both** sustained absolute-threshold alerts AND rolling-baseline
anomaly rules; **plus** a single-line time-series Grafana panel on `host-overview` showing all
temperature entries.

## Concrete deliverables

### A. Collector — new metric family
A `_collect_temperatures()` helper in `apps/monitor/homelab_monitor/plugins/collectors/builtin/host.py`,
following the exact `_collect_cpu` pattern (try/except wrapper, returns `tuple[int, list[str]]`):
- Emit `homelab_host_temperature_celsius{chip, sensor}` as a **gauge** (`ctx.vm.write_gauge`), one series
  per (chip, sensor-label). E.g. `{chip="k10temp", sensor="Tctl"}`, `{chip="nvme", sensor="Composite"}`,
  `{chip="amdgpu", sensor="edge"}`. Use the psutil label when present; fall back to a synthesized
  `sensor="<index>"` when the label is empty (some chips emit unlabeled entries).
- **Optionally** also emit `homelab_host_temperature_high_celsius` / `_critical_celsius{chip,sensor}` when
  psutil reports `high`/`critical` (the nvme does) — these let rules/dashboards reference hardware-declared
  limits instead of guessed constants. Decide in Design (D-009-HW-LIMITS).
- **Platform/availability guard:** wrap with `hasattr(psutil, "sensors_temperatures")` AND try/except —
  `sensors_temperatures()` is Linux/FreeBSD-only and can raise or return `{}`. On absence: emit nothing,
  return `(0, [])` (NOT an error — a host without sensors is normal, not a failure). Mirror the defensive
  style of the existing `# pragma: no cover` psutil-failure branches.
- **Config (`HostCollectorConfig`):** add fields following the existing baked-default pattern
  (`disk_mountpoint_relabel`, `exclude_disk_devs`): `exclude_temp_sensors: list[str]` (chip or chip:sensor
  globs to drop) and optionally `temp_sensor_relabel: dict[str,str]` for friendly names. Decide the exact
  field set in Design (D-009-CONFIG). Read via `getattr(ctx.config, ..., default)` in `run()`.
- **100% kernel coverage:** every new branch (empty-label fallback, exclude match, hasattr-false, psutil
  raises, high/crit present-vs-absent) needs a test input that hits it. Extend `test_host_collector.py`
  with a `sensors_temperatures` stub in the psutil-patch helper (mock returning the real-shaped dict:
  `{"k10temp":[("Tctl",54.0,(None,None,None))], "nvme":[("Composite",46.0,(89.85,93.85,...))], ...}` —
  confirm the exact psutil `shwtemp` namedtuple shape in Design).

### B. vmalert rules — `deploy/vmalert/metrics/host_temperature.yaml` (+ `__tests__/host_temperature.tests.yaml`)
All rules carry `source_tool: vmalert-metrics` + `target_kind: host`. Severity vocab info|warning|error|critical.

**Sustained absolute-threshold (the core ask) — per-sensor, because safe ranges differ wildly:**
1. **CPU sustained-hot** (warning + critical) — `avg_over_time(homelab_host_temperature_celsius{chip="k10temp"}[5m]) > <warn>` for 10m, and a critical tier. Tune `<warn>/<crit>` in 3b (AMD Tctl typically throttles ~95°C; conservative defaults e.g. 85 warn / 92 critical — confirm against the real idle baseline ~54°C).
2. **NVMe sustained-hot** (warning + critical) — use the hardware-declared limits where available:
   data-driven form `homelab_host_temperature_celsius{chip="nvme"} > homelab_host_temperature_high_celsius{chip="nvme"}` for 10m (if B.A emits the limit metric), OR absolute `> 80` warn / `> 90` critical for 10m. Decide in Design (D-009-HW-LIMITS).
3. **GPU sustained-hot** (warning + critical) — `amdgpu` `{chip="amdgpu"}` sustained threshold (amdgpu edge throttles ~95-110°C depending on model; conservative warn/crit, tuned in 3b).

**Rolling-baseline thermal anomaly (the "drift unusually hot vs its own baseline" ask):**
4. **Per-sensor thermal anomaly** (info) — the house idiom over ALL sensors, with a floor so a cool sensor
   doubling from 20°C→40°C doesn't fire: `homelab_host_temperature_celsius > clamp_min(<K> *
   avg_over_time(homelab_host_temperature_celsius[6h]), <floor>)` for 15m. Pick `<K>` (e.g. 1.3-1.5 — temps
   don't swing like CPU%, so a smaller multiplier than the CPU 3×) and `<floor>` (e.g. 60°C) in Design;
   tune in 3b. This is per-series (the `{chip,sensor}` labels carry through), so one rule covers all sensors.

**Acceptance:** every rule has a fire + no-fire case in the test mirror. Anomaly FIRE plants a 6h baseline
then a spike. Per-sensor threshold rules plant the relevant `{chip=...}` series. Validate per STAGE-005A-001's
proven method: promtool for the threshold/anomaly rules it can evaluate; live-MetricsQL replay for any rule
promtool can't (the anomaly `[6h:5m]`-style subquery if used). Confirm rules load `health=ok` on the live rig
and do NOT false-fire at the real idle baseline (3b).

**Decisions:**
- **D-009-CONFIG** → CHOSEN: `exclude_temp_sensors: list[str]` ONLY (glob/exact chip or chip:sensor to drop). REJECTED: adding `temp_sensor_relabel` (YAGNI — labels already readable; cosmetic renaming belongs in Grafana legendFormat, not the metric, where it would complicate alert exprs).
- **D-009-HW-LIMITS** → CHOSEN: emit `homelab_host_temperature_high_celsius` + `homelab_host_temperature_critical_celsius{chip,sensor}` from psutil's high/crit fields, None-GUARDED (only emit when not None → only nvme emits them on this host). The NVMe sustained ALERT stays HARDCODED-absolute (`> 80` warn / `> 90` crit), DECOUPLED from the limit metric. REJECTED: a data-driven cross-metric join (`...{chip="nvme"} > homelab_host_temperature_high_celsius{chip="nvme"}`) — that is the epic's #1 §4.1 empty-join trap (different metric names, silent empty = dead rule); matches the STAGE-008 hardcoded-per-path precedent. The limit metrics are emitted for DASHBOARD reference value, never for the alert.
- **D-009-THRESHOLDS** → CHOSEN: CPU (k10temp) warn 85 / crit 92; NVMe (nvme) warn 80 / crit 90; GPU (amdgpu) warn 90 / crit 100; anomaly K=1.4, floor=60. Conservative starting points above idle (54/46/42) below throttle (~95/90/95-110); K small because temps don't swing like CPU% (thermal mass); floor=60 prevents a cool sensor's benign warmup from firing. ALL tuned in 3b against real load data (a noisy thermal rule is a failed rule).
- **D-009-SENSOR-LABEL** → CHOSEN: labels `{chip, sensor}`, sensor = the psutil `label` field when non-empty, else a synthesized `sensor="<index>"` (the entry's position in the chip's list). REJECTED: skipping unlabeled entries (silent data loss; cheap insurance against dropping a real sensor on other hardware). This host's 3 chips all report labels.
- **D-009-COMPOSE** → CHOSEN: NO compose change (sensors readable via the container's default sysfs — confirmed live). 3b re-confirms in the rebuilt prod container. Fallback `/sys:/sys:ro` ONLY if 3b shows hwmon absent (not expected).

**Rule plan (`deploy/vmalert/metrics/host_temperature.yaml` + `__tests__/host_temperature.tests.yaml`, labels `severity` + `source_tool: vmalert-metrics` + `target_kind: host`):**

Annotations use `{{ $labels.chip }}` / `{{ $labels.sensor }}` / `{{ $value }}`.
