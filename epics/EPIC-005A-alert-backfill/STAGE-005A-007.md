# STAGE-005A-007: Build / config health + self-disk-shrink rules

## Status: Not Started

## Overview

A cluster of small "a subsystem silently stopped working / the monitor is acting on its own storage" health signals that EPIC-001/003 emit but never alert on. All four metrics below EXIST live and are one-rule-away from coverage. Severity is low (mostly warning/info) — these surface silent degradation, not emergencies.

Per-slot self-disk fill is OUT of this stage (it's in STAGE-005A-002); this stage covers the self-disk *shrink action* event plus the build/config health gauges.

**Verified live (scoping session):**
- `homelab_self_disk_shrink_total` EXISTS (counter, label `tier`) — emitted by `self_disk.py` when `used_pct > 95%` and the 5-min cooldown elapsed (the auto-shrink DECISION fired; the actual VM-retention reduction is a future TODO, currently metric-only signaling). NO alert.
- `homelab_build_sources_config_loaded` EXISTS (1 series, gauge) — health indicator for the build-sources config load. NO alert.
- `homelab_docker_compose_readable` EXISTS (1 series, gauge) — whether the monitor can read the compose file. NO alert.
- `homelab_build_source_hash_skipped_total` EXISTS (gauge despite the `_total` name, labels `reason`/`name`) — emitted by `local_build_update_collector.py` when a local-build update check skips (context_missing / context_too_large / permission_denied). NO alert.

## Stage Flags

- Has Input Forms: No.
- Touches secrets / production data: No (rules over existing self-metrics).
- Host-integration concerns: Yes — the build/config-readable gauges depend on the real host compose/build-context paths being mounted (validated on the prod rig, 3a + 3b). The shrink rule needs a planted critical-disk condition to test.

## Dependencies

- EPIC-001 (`self_disk` collector → `homelab_self_disk_shrink_total`).
- EPIC-003 (build-sources / local-build-update collectors → `homelab_build_*`, `homelab_docker_compose_readable`).
- vmalert-metrics + Alertmanager.

## Concrete deliverables

A new rule file `deploy/vmalert/metrics/self_and_build.yaml` (+ `__tests__/self_and_build.tests.yaml`) containing:

1. **Self-disk auto-shrink fired (user: include).** `increase(homelab_self_disk_shrink_total[10m]) > 0` → warning (or error), `{{tier}}`. The monitor's self-disk hit critical and the auto-shrink decision triggered — "my own storage is in trouble." Note: this fires at the same 95% tier as `SelfDiskCritical`, but is a DISTINCT event (the shrink decision, with the `tier` label) vs the threshold crossing. Design confirms whether warning or error and whether to dedupe vs SelfDiskCritical (they convey different things; both firing is acceptable, but confirm).
2. **Build-sources config load failure (user: include).** `homelab_build_sources_config_loaded == 0` for `5m` → warning. The build-sources config failed to load → the local-build-update feature is broken.
3. **Compose file unreadable (user: include).** `homelab_docker_compose_readable == 0` for `5m` → warning. The monitor can't read the compose file → degraded Docker features (locally-built image detection, etc.).
4. **Build-source hash skipped (user: include).** `homelab_build_source_hash_skipped_total == 1` (or `count(...) > 0`) for `5m` → info/warning, `{{name}}`/`{{reason}}`. A local-build update check couldn't compute the source hash (context missing / too large / permission denied) → the operator should fix the build context.

## Decisions to confirm in Design

- **D-005A-007-SHRINK-SEVERITY** — warning vs error for the shrink-fired rule; dedupe-vs-SelfDiskCritical decision (likely keep both — distinct signals).
- **D-005A-007-FOR-WINDOWS** — `for:` windows for the `== 0` health gauges (5m proposed — avoid firing on a transient single-tick blip).
- **D-005A-007-SKIPPED-SEVERITY** — info vs warning for the build-hash-skipped rule; whether to alert per-reason or aggregate.
- **D-005A-007-GAUGE-SEMANTICS** — confirm `build_source_hash_skipped_total` is a gauge (resets per tick) so the rule uses `== 1`/`count() > 0`, not `increase()` (the `_total` name is misleading; verify against the collector).

## Acceptance criteria

- Test mirror (plants config-load-failure / compose-unreadable / skip / shrink → fires; healthy → silent).
- The config-readable rules validated on the prod rig where the real compose/build-context paths ARE mounted (so the gauges read 1 at baseline, 0 only on real failure).
- Shrink rule does NOT spuriously fire (it only emits on a real >95% critical-disk event).
- Severity vocab + `source_tool: vmalert-metrics` + appropriate `target_kind`/`integration` labels.
