# STAGE-001-021: Full integration test rig + canonical e2e test

## Status: Not Started

## Overview

Finalize the integration test rig that future epics rely on. Expand `docker-compose.test.yml` to include every sidecar from EPIC-001 (VM, VL, vmagent, vmalert ×2, Alertmanager, Karma, kthxbye, Grafana, vector, monitor) plus a fixture target stack that produces predictable metrics, logs, and heartbeats. After this stage, EPIC-001 is complete: the foundation is proven end-to-end with an automated test that catches regressions in the alert pipeline.

## Concrete deliverables

1. **`docker-compose.test.yml` finalization**
   - All sidecars from STAGE-001-015 through 020 included.
   - Plus a "fixture target" container (`fixture-host`): exposes a Prometheus `/metrics` endpoint with controllable values driven by env (e.g., `FIXTURE_CPU_PERCENT=95` makes the host-cpu metric report 95). Used to deterministically trip rules.
   - Plus a "noisy logger" container (from STAGE-001-016) that emits planted log lines on demand.
   - Plus an `integration-tests` container that runs pytest against the live rig and exits with the suite's exit code.
   - `--abort-on-container-exit --exit-code-from integration-tests` is the canonical run mode.
2. **Canonical e2e test** at `apps/monitor/tests/integration/test_canonical_e2e.py`
   - Steps:
     1. Boot the rig (handled by the docker-compose call; the test assumes services are up via healthchecks).
     2. Set the fixture `cpu_percent` to 95 via the fixture target's control endpoint.
     3. Wait up to 60s for the alert to appear in `/api/alerts` (poll).
     4. Assert: `severity=warning`, `source_tool=vmalert-metrics`, `alertname=HostHighCPU`.
     5. Reset `cpu_percent` to 5.
     6. Wait up to 60s for the alert to enter `resolved` state.
     7. Assert: `resolved_at` is populated and a `alert.resolved` SSE event was observed.
   - Test imports a small `Rig` helper that wraps polling, fixture control, and SSE consumption.
3. **Additional integration tests** (each is short, focused on one path)
   - **Heartbeat path stub** — POST to `/hb/test/ok` returns 401 without a token; with a valid token, returns 204 and updates a row. (The full heartbeat receiver behavior lands in EPIC-002, but the auth on the endpoint is here.)
   - **Logs path** — plant a log line via the noisy logger; assert `/api/logs/query` returns it.
   - **Logs alert path** — plant 11 SSH-failed-password lines; assert `SshFailedLoginBurst` fires and reaches `/api/alerts` with `source_tool=vmalert-logs`.
   - **Karma ack path** — programmatic ack via Karma's API creates a silence in AM; the alert moves to silenced state.
   - **Quarantine path** — induce a collector to fail repeatedly; verify the quarantine alert appears in `/api/alerts` (this exercises the path where the scheduler emits via the ingestor's internal API, not via vmalert).
4. **CI integration**
   - `ci.yml` adds an `integration` job that runs the rig and asserts exit code 0.
   - The `integration` job is required for PR merge (branch protection).
   - The job uses BuildKit cache for fast image builds.
5. **Tooling — centralized dev + prod docker workflows**

   **Single source of truth for ports/env:** `deploy/dev/dev.env.example` (checked in) + `deploy/dev/dev.env` (gitignored, user-generated from example). Lists every port number the dev stack uses (UI, backend, AM, Karma, kthxbye, VM, VL, vmagent, vector, vmalert ×2) plus all `HOMELAB_MONITOR_*` env vars and the master key. The same file is sourced by both the dev script and the test-rig script so port choices don't drift.

   **`scripts/reset-test-rig.sh`** — tears down `docker-compose.test.yml` volumes and rebuilds. Used both locally and (optionally) in CI when builds drift.

   **`scripts/dev-up.sh [--clean] [--prod]`** — single entrypoint for local development:
   - Reads `deploy/dev/dev.env` (creates from `.env.example` on first run; aborts if master key not set).
   - **Dev mode (default)**: hybrid stack. Brings up sidecars in docker (AM, Karma, kthxbye, VM, VL, vmagent, vmalert ×2, vector) on the dev network with port-publishing per `dev.env`. Runs the monitor backend on host via `uv run uvicorn ... --reload` so code edits hot-reload. Runs UI dev server on host via `pnpm dev`. Creates default `admin` user if none exists (idempotent). Prints status banner with all URLs + login creds.
   - **`--prod` mode**: no host processes; brings up the full `docker-compose.yml` stack with the monitor image built from local Dockerfile (`docker compose build monitor`). Validates that the prod image runs end-to-end. Used to verify deployment correctness before merge.
   - **`--clean` flag**: kills any existing dev processes (uvicorn, pnpm dev) AND tears down docker stack + volumes before bringing up. The "clear cached code" path.

   **`scripts/dev-down.sh`** — graceful tear-down of whatever `dev-up.sh` started. Detects mode from running containers/processes.

   **`Makefile` targets**:
   - `make dev` → `scripts/dev-up.sh` (dev mode, no clean)
   - `make dev-clean` → `scripts/dev-up.sh --clean` (kill + restart, dev mode)
   - `make dev-prod` → `scripts/dev-up.sh --prod` (prod-image variant)
   - `make dev-down` → `scripts/dev-down.sh`

   **Prod Dockerfile correctness — must be validated by `make dev-prod`:**
   - `.dockerignore` re-includes `apps/monitor/alembic/` (currently missing — discovered in STAGE-019 Refinement).
   - Container user (uid 1000) must own `/data` volume mount: add `RUN chown -R homelab:homelab /data` after USER switch in `apps/monitor/Dockerfile`, OR use a tini-based init script that chowns at startup.
   - `make dev-prod` MUST succeed end-to-end (login, navigate to /alerts, see Karma) as the acceptance test for the prod variant. Failure here blocks STAGE-021 completion.

   **Documentation:**
   - Update `CLAUDE.md` Local Refinement section to point at `make dev` / `make dev-prod` instead of the manual recipe.
   - Add `docs/dev/local-environment.md` covering: when to use dev vs `--prod`, port map, troubleshooting (port collisions, host-network containers, vmagent visibility).

## Decisions in this stage

- **Test rig boots ALL sidecars** — yes. Future epics will add more services that need to participate; making this the canonical complete rig means future epics can extend without bringing up parallel rigs.
- **Polling-based assertions** with sensible timeouts — yes; alert pipelines are inherently asynchronous and arbitrary sleeps are flaky.
- **CI runtime budget** — target ≤ 5 minutes for the integration job; if it grows, parallelize sub-suites later.
