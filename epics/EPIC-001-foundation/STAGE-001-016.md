# STAGE-001-016: VictoriaLogs + vector

## Status: Build (Complete)

## Overview

Stand up the log pipeline. Add VictoriaLogs and vector as sidecars. Replace the in-memory `LogsWriter` with a real VictoriaLogs-backed implementation. Configure vector to tail Docker container stdout/stderr and the host's journald, and ship to VL with per-stream caps to enforce the disk budget.

After this stage, every container's logs are queryable in VL via LogsQL, and a planted error message in any container is findable from the dashboard's eventual Logs explorer.

**Spec references:** §3.2 (VL + vector sidecars), §6.3 (VictoriaLogs streams), §6.4 (disk budget), §10.1 (compose layout), §13.2 (integration tests).

## Concrete deliverables

1. **VictoriaLogs sidecar** — added to `docker-compose.yml`
   - Image pinned (e.g., `victoriametrics/victoria-logs:v0.x.x`).
   - Args: configured retention period from env; HTTP API on 9428.
   - Volume `data/vl/`.
2. **vector sidecar**
   - Image: `timberio/vector` pinned.
   - Config at `deploy/vector/vector.toml`:
     - **Sources:**
       - `docker_logs`: tails all containers in the `homelab-monitor` compose project AND optionally the user's other compose projects (configurable via env `VECTOR_DOCKER_INCLUDE`). Default: only the homelab-monitor project — extending to host-wide is opt-in.
       - `journald`: tails `/var/log/journal` (host-mounted read-only). Optional; gated by env `VECTOR_JOURNALD_ENABLED`.
     - **Transforms:**
       - `add_labels`: enriches every event with `host`, `service` (derived from container name or systemd unit), and `severity` (parsed best-effort).
       - `parse_json`: best-effort JSON parsing (containers logging JSON get fields lifted; non-JSON passes through).
       - `drop_noise`: a baseline filter that drops obvious noise (e.g., kubelet healthz, docker daemon ping, monitor's own debug logs). Configurable.
     - **Sinks:**
       - `vl`: writes to `http://victorialogs:9428/insert/jsonline`. With per-stream rate-limiting via vector's transforms.
3. **Real `LogsWriter`** at `apps/monitor/homelab_monitor/kernel/logs/vl_writer.py`
   - Implements the protocol from STAGE-001-006.
   - Used by collectors that produce log lines (rare; most logs come from vector, not from collectors).
4. **Per-stream caps**
   - Vector's `throttle` transform enforces lines/sec per `service` label.
   - Hard byte-cap per stream per day enforced by a small in-process collector (`log_stream_budget`) that queries VL's `/select/stream/_/stats` for current bytes-today and emits a metric; a vmalert rule can fire on overrun (rule lands in STAGE-001-018).
   - Defaults: 50 lines/sec per stream; 500 MB/day per stream. Configurable via `homelab-monitor.yaml`.
5. **API integration**
   - `GET /api/logs/query` — accepts a LogsQL expression, time range; returns matching lines (paginated). Auth required.
   - `GET /api/logs/streams` — lists known streams (host, service pairs) with last-seen and lines/sec.
6. **Disk budget integration**
   - The disk-budget self-monitor from STAGE-001-015 now also accounts for `data/vl/`.
   - Auto-shrink at 95% trims VL retention before SQLite/audit.
7. **Test rig extension** — `docker-compose.test.yml`
   - Adds VL + vector + a "noisy" fixture container that emits 1 line/sec with a known pattern.
   - Integration test: vector ships line → VL query returns it.
8. **Tests**
   - Unit: `vl_writer` posts JSONL to the right endpoint.
   - Integration: planted log line ↔ `/api/logs/query` returns it.
   - Throttle: a flooded stream gets throttled; remaining lines preserved within budget.
   - Per-stream byte cap: synthetic large line bursts trip the budget metric.

### Design Decisions (selected 2026-05-08)

1. **LogsWriter architecture**: Option A — MultiplexLogsWriter.
   - Keep `InMemoryLogsWriter` AND add new `VictoriaLogsWriter` (HTTP push to VL `/insert/jsonline`).
   - New `MultiplexLogsWriter` fans out every `ingest()` call to BOTH writers (in-memory first, then VL HTTP).
   - The in-memory writer remains the source for any future snapshot/recent-events endpoints (zero VL round-trip).
   - VL writer best-effort: HTTP failures logged but don't fail the collector.
   - Rationale: directly mirrors STAGE-015 `MultiplexMetricsWriter` pattern; preserves 20+ existing tests using `InMemoryLogsWriter()`; provides in-memory buffer that rides out VL outages.

2. **Per-stream caps mechanism**: Option A — Both layers.
   - Layer 1: vector `throttle` transform at ingest time (50 lines/sec/stream default; configurable via env).
   - Layer 2: in-process `log_stream_budget` collector (60s interval) queries VL `/select/logsql/stats` for bytes/day per stream; emits `homelab_log_stream_bytes_today{host, service}` metric.
   - vmalert rule (in `deploy/vmalert/logs/`) fires on per-stream byte overrun (lands in STAGE-001-018).
   - Defaults: 50 lines/sec/stream, 500 MB/day/stream. Configurable in `homelab-monitor.yaml`.
   - Rationale: defense in depth — throttle prevents bursts at ingest; budget collector catches slow-bleeding streams with per-stream attribution.

3. **Logs API contract**: Option A — Thin proxy.
   - `GET /api/logs/query?expr=...&start=...&end=...&limit=...` proxies to VL `/select/logsql/query`. Returns VL native response shape directly.
   - `GET /api/logs/streams` returns `[{host, service, last_seen, lines_per_sec, bytes_today}]`. Backed by the budget collector's in-process state (not a per-request VL stats query).
   - Both endpoints require cookie session OR API token auth.
   - Rationale: UI consumption is a future stage; v1 is operator/developer query; VL native shape is fine for now. Revisit with normalized `/api/logs/recent` when UI stage demands.

4. **Vector config approach**: Option A — Static hand-authored TOML.
   - `deploy/vector/vector.toml` checked into the repo.
   - Sources: `docker_logs` (containers in `homelab-monitor-net` by default; override via `VECTOR_DOCKER_INCLUDE` env), `journald` (opt-in via `VECTOR_JOURNALD_ENABLED`).
   - Transforms: `add_labels` (host/service/severity), `parse_json` (best-effort), `drop_noise` (baseline filter), `throttle` (per-service).
   - Sink: `vl` → `http://victorialogs:9428/insert/jsonline`.
   - No code-generated config. No SIGHUP / monitor-driven reload. Reload = container restart.
   - Rationale: same as STAGE-015 vmagent decision (rejected dynamic generation); avoids monitor↔vector lifecycle coupling.

### Cross-cutting Decisions

- **Disk-budget integration**: monitor's compose service gains a new bind mount `data_vl:/var/vl-data:ro` so STAGE-015A's `SelfDiskCollector` "vl" slot reports real bytes (currently 0).
- **Auto-shrink ordering**: at >95% disk used, the SelfDiskCollector emits `homelab_self_disk_shrink_total{tier="vl"}` (and "vm", "sqlite") — STAGE-016 doesn't change auto-shrink behavior; v1 stays decision-only.
- **VL ingest auth**: VL's `/insert/jsonline` is unauthenticated within the compose network (vector is in-network). NOT exposed externally.
- **VL query auth**: `/select/logsql/query` IS proxied through monitor's authenticated `/api/logs/query`; VL itself is not exposed externally.
- **VL image**: `victoriametrics/victoria-logs:v1.x.x` (latest stable v1, pinned in compose).
- **VL retention**: configurable via `HOMELAB_MONITOR_VL_RETENTION_DAYS` env (default 30 days).
- **Vector image**: `timberio/vector:0.x.x` pinned.
