# STAGE-001-015: VictoriaMetrics + vmagent

## Status: Not Started

## Overview

Replace the in-memory `MetricsWriter` with a real VictoriaMetrics-backed implementation. Add VM and vmagent as sidecar containers. Generate vmagent's scrape config from the monitor's collector registry. Wire the dashboard's Overview tile to read from VM (via the monitor's existing `/api/metrics/snapshot`, now implemented as a thin VM query proxy).

After this stage, metrics persist, can be queried with MetricsQL, and survive restarts. The follow-on stage **STAGE-001-015A** layers backup, disk-budget enforcement, and the test rig extension on top — those are split out so this stage stays single-session-sized.

## Concrete deliverables

1. **Compose file** — production at `deploy/compose/docker-compose.yml`
   - Services: `monitor`, `victoriametrics`, `vmagent`. Network: `homelab-monitor-net`.
   - VM image pinned (e.g., `victoriametrics/victoria-metrics:v1.107.x`). Single binary; flags include retention period (configurable via env), snapshot endpoint enabled.
   - vmagent image pinned. Args: `-promscrape.config=/etc/vmagent/scrape.yaml`, `-remoteWrite.url=http://victoriametrics:8428/api/v1/write`, `-promscrape.config.strictParse=false`.
   - Volumes: `data/vm/` for VM, `data/vmagent/` for vmagent buffer.
   - The `monitor` Dockerfile (at `apps/monitor/Dockerfile`) is finalized in this stage: multi-stage build (Python deps via uv → slim runtime; UI bundle copied from `apps/ui/dist`).
2. **Real `MetricsWriter`** at `apps/monitor/homelab_monitor/kernel/metrics/vm_writer.py`
   - Implements `MetricsWriter` protocol.
   - Two write paths:
     - **Direct push** via VM's `/api/v1/import/prometheus` for events that don't fit the scrape model (e.g., heartbeat lateness, alert outcomes — though most of those land in EPIC-002+).
     - **Scrape exposure** for collector metrics: the monitor itself exposes `/metrics` in Prometheus text format, scraped by vmagent. The `MetricsWriter` accumulates current values into an in-memory registry that `/metrics` serializes on request.
   - Decision in Design phase: use `prometheus_client` Python library for the registry (battle-tested), or a custom minimal registry. Recommend `prometheus_client`.
3. **vmagent scrape config generation**
   - The monitor writes `/etc/vmagent/scrape.yaml` (a docker volume shared with vmagent) on startup and whenever the collector registry changes.
   - Initial config scrapes:
     - `homelab-monitor-monitor:9090/metrics` (the monitor itself; exposed at the same port as the API but on a different path)
     - We do NOT yet scrape `node_exporter` / `cadvisor` here — those are added in EPIC-003 (Docker) and a future host epic.
   - After writing, monitor calls `POST http://vmagent:8429/-/reload`.
4. **API integration**
   - `/api/metrics/snapshot` (added in STAGE-001-012 against the in-memory writer) is updated to query VM via `/api/v1/query` (instant query). The endpoint returns a normalized snapshot for the dashboard.
   - New `GET /api/metrics/range` (auth-required) for range queries — accepts a PromQL/MetricsQL expression, start, end, step. Used by the Overview tile sparkline.
5. **Tests**
   - Unit: `vm_writer` correctly serializes counters, gauges, summaries to Prometheus text.
   - Smoke integration: stand up VM + monitor in a temporary compose; planted metric in `/metrics`; assert it's queryable in VM via `/api/v1/query`. (Full integration test rig finalization is STAGE-001-015A and STAGE-001-021.)
   - Range query path: planted metric ↔ `/api/metrics/range` returns expected series.
   - Reload: change scrape config in-place; vmagent picks it up after `/-/reload`.

## Decisions in this stage

- **Which metric format** — Prometheus text via `prometheus_client`. VM accepts Prometheus, OpenMetrics, InfluxDB line protocol, and Graphite; we standardize on Prometheus throughout for consistency with the exporter ecosystem.
- **Direct-push vs scrape** — both. Most metrics use the scrape model (vmagent pulls from monitor's `/metrics`); direct push is reserved for one-off events like heartbeat lateness.
- **Snapshot retention** — keep last 7 daily snapshots in `/storage/backup/homelab-monitor/vm/`. Configurable.
