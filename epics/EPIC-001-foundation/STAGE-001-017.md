# STAGE-001-017: Alertmanager + vmalert (metrics) + first rule

## Status: Not Started

## Overview

Add Alertmanager and vmalert (metrics) as sidecars. Configure vmalert to evaluate rules against VictoriaMetrics and route to Alertmanager. Configure Alertmanager to webhook back to our `/api/alerts/ingest`. Author the first vmalert rule (host metric threshold). Verify the full path: synthetic threshold trip → vmalert → Alertmanager → ingestor → in-process dashboard channel → SSE → dashboard live update.

After this stage, real alerts flow end-to-end with no manual webhook posting.

## Concrete deliverables

1. **Alertmanager sidecar** in compose
   - Image pinned (e.g., `prom/alertmanager:v0.27.x`).
   - Config at `deploy/alertmanager/alertmanager.yml` (template; rendered by the monitor on startup so secrets like the API token can be injected without baking them into the image):
     - One default route → webhook receiver.
     - Webhook receiver URL: `http://homelab-monitor-monitor:9090/api/alerts/ingest`.
     - `Authorization: Bearer ${ALERTMANAGER_INGEST_TOKEN}` header (token created via `hm api-token create --scope alerts:ingest:write` and stored in the secrets store; the monitor renders the AM config with the token at startup).
     - Default `group_by`, `group_wait`, `group_interval`, `repeat_interval` configured per spec §8.4.
2. **vmalert (metrics) sidecar**
   - Image pinned.
   - Args: `-datasource.url=http://victoriametrics:8428`, `-notifier.url=http://alertmanager:9093`, `-rule=/etc/vmalert/metrics/*.yaml`, `-evaluationInterval=30s`.
   - Volume `deploy/vmalert/metrics/` mounted at `/etc/vmalert/metrics/`.
3. **First rule** at `deploy/vmalert/metrics/host.yaml`
   - Group `host`:
     - **`HostHighCPU`** — `avg_over_time(homelab_host_cpu_percent{cpu="all"}[5m]) > 90` for 10m → severity=warning, source_tool=vmalert-metrics, target_kind=host, summary, description.
     - **`HostHighMemory`** — `homelab_host_memory_bytes{type="used"} / homelab_host_memory_bytes{type="total"} > 0.9` for 10m → severity=warning.
     - **`CollectorQuarantined`** — `homelab_collector_quarantine_count > 0` → severity=warning, source_tool=vmalert-metrics, target_kind=collector. Replaces (de-duplicates with) the synthetic alert the scheduler emits internally — the scheduler keeps its synthetic alert as a faster path; vmalert's version provides cross-restart durability via the persisted `consecutive_failures` counter.
4. **Alertmanager render-on-boot**
   - The monitor reads `alertmanager.yml.template`, substitutes `${ALERTMANAGER_INGEST_TOKEN}` from the secrets store, writes the result to a docker volume Alertmanager reads. Calls Alertmanager's `/-/reload` after.
   - On token rotation, the same flow re-runs.
5. **Ingestor lockdown**
   - `/api/alerts/ingest` now strictly requires the token's `alerts:ingest:write` scope. The dashboard's "synthetic test post" (used in STAGE-001-013) gets its own scope `alerts:ingest:dev` that's only minted in dev mode.
6. **Tests**
   - Integration: bring up the test rig; force `homelab_host_cpu_percent` high (use a fixture metric exporter); vmalert fires within evaluation+wait windows; Alertmanager webhook hits the monitor; alert appears in `/api/alerts`; SSE event delivered.
   - Resolved path: metric returns to normal; vmalert sends `status=resolved`; alert row updated; resolved SSE event delivered.
   - Auth: an unauthenticated POST to `/api/alerts/ingest` returns 401; a wrong-scope token returns 403.

## Decisions in this stage

- **Token lives in secrets store** — yes; never in plaintext in alertmanager.yml on disk.
- **Reload on token rotation** — yes; the rotate flow includes re-rendering the Alertmanager config and POSTing to AM `/-/reload`.
- **Severity vocabulary** locked — `info`, `warning`, `error`, `critical` (per spec §8.2). Rules must use one of these.
