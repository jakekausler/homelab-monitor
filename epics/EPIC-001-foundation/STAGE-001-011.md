# STAGE-001-011: Local auth (bcrypt + sessions + CSRF)

## Status: Not Started

## Overview

Replace the `X-Auth: dev` placeholder with real authentication: local users stored in SQLite (bcrypt password hashes), signed cookie sessions, CSRF protection on state-changing requests, login/logout endpoints, and per-API-token scoped programmatic access.

After this stage, every API call (except `/api/healthz`, `/api/version`, the heartbeat endpoints, and the alert webhook ingestor) requires either a valid session cookie or a valid API token. A first user can be bootstrapped via `hm user create`.

**Spec references:** §7.1 (Authentication), §7.2 (Authorization & confirm-on-destructive — most of which lands in later stages, but the auth scaffolding is here), §6.1 (users, sessions, api_tokens tables — created in STAGE-001-004).

## Concrete deliverables

1. **User model & repository**
   - `users` table is already present from STAGE-001-004's initial migration.
   - Repository methods: `create_user(username, password)`, `verify_password(username, password)`, `change_password(user_id, new)`, `list_users()`, `delete_user(user_id)`.
   - bcrypt cost factor 12 (configurable, but default 12).
2. **Session middleware**
   - Sessions stored in `sessions` table (id, user_id, expires_at, created_ip, csrf_token).
   - Cookie name: `homelab_monitor_session`. Attributes: HttpOnly, SameSite=Lax (matches LAN-only architecture; users access via single domain), Secure when `HOMELAB_MONITOR_HTTPS_ONLY_COOKIES=true` (default true; set false in dev).
   - Session value is a signed token (HMAC over `session_id`) using a key derived from the master key + a session-purpose label.
   - Session TTL default: 7 days idle. Configurable in `homelab-monitor.yaml`.
3. **CSRF protection**
   - Double-submit cookie pattern: server sets `homelab_monitor_csrf` (HttpOnly=false, readable by JS) at session start; client must echo its value in `X-CSRF-Token` header for state-changing requests.
   - Enforcement on all `POST/PUT/PATCH/DELETE` to `/api/*` except API-token-authenticated calls.
4. **Endpoints**
   - `POST /api/auth/login` → `{username, password}` → 200 with `{user: {id, username}}` on success; sets cookies. Rate-limited (5 attempts per 5 minutes per IP, in-process token bucket).
   - `POST /api/auth/logout` → clears session row + cookies; 204.
   - `GET /api/auth/me` → returns current user or 401.
   - `POST /api/auth/change-password` → requires current password; rotates session afterward.
5. **API tokens** (programmatic access)
   - `api_tokens` table from STAGE-001-004 holds scoped tokens.
   - `hm api-token create --scope heartbeat:write` prints the token once (only).
   - Token format: `homelab_<env-prefix>_<base64url-30-bytes>`; only the SHA-256 hash is stored.
   - Auth via `Authorization: Bearer ...` header.
   - Scopes are an enum: `heartbeat:write`, `alerts:ingest:write`, `read:status`. (Adding scopes is a later additive change.)
   - **Heartbeat and alert-ingest endpoints accept tokens; do NOT require a session.** This is critical — those endpoints are called by external services (cron jobs, Alertmanager).
6. **CLI**
   - `hm user create USERNAME` → prompts for password, creates user.
   - `hm user list` → list users (no hashes).
   - `hm user passwd USERNAME` → set password (admin override; logs to audit).
   - `hm user delete USERNAME` → deletes user.
   - `hm api-token create --scope SCOPE [--name NAME]` → prints token once.
   - `hm api-token list`, `hm api-token revoke ID`.
7. **Auth dependency** in FastAPI
   - `Depends(get_current_user)` returns the authenticated user or raises 401.
   - `Depends(require_scope("..."))` for token-protected endpoints.
   - The `/api/healthz`, `/api/version`, `/api/auth/login` endpoints are unauthenticated.
8. **Replace dev placeholder**
   - The `HOMELAB_MONITOR_DEV_AUTH` env var is removed; the placeholder header is no longer accepted under any circumstance.
9. **Tests**
   - Login success / wrong password / nonexistent user.
   - Session cookie issued; subsequent request with cookie authorizes.
   - Session TTL: a session past TTL is rejected.
   - CSRF: a state-changing request without the header is 403; with it, 200.
   - Rate limit: 6th login attempt within 5 minutes returns 429.
   - API token: valid token + correct scope authorizes; wrong scope returns 403; revoked token returns 401.
   - Audit log row written for: user create/delete, password change, token create/revoke.

## Decisions in this stage

- **No password complexity rules in v1** — single-user homelab; bcrypt + length-only check (≥ 12 chars). Documented as a future hardening item.
- **No 2FA in v1** — same reasoning. Confirm-on-destructive (typed phrase) substitutes for sensitive actions.
