# STAGE-001-011: Local auth (bcrypt + sessions + CSRF)

## Status: Design (Complete)

## Overview

Replace the `X-Auth: dev` placeholder with real authentication: local users stored in SQLite (bcrypt password hashes), signed cookie sessions, CSRF protection on state-changing requests, login/logout endpoints, and per-API-token scoped programmatic access.

After this stage, every API call (except `/api/healthz`, `/api/version`, the heartbeat endpoints, and the alert webhook ingestor) requires either a valid session cookie or a valid API token. A first user can be bootstrapped via `hm user create`.

**Spec references:** §7.1 (Authentication), §7.2 (Authorization & confirm-on-destructive — most of which lands in later stages, but the auth scaffolding is here), §6.1 (users, sessions, api_tokens tables — created in STAGE-001-004).

## Concrete deliverables

1. **User model & repository**
   - `users` table is already present from STAGE-001-004's initial migration.
   - Repository methods: `create_user(username, password)`, `verify_password(username, password)`, `change_password(user_id, new)`, `list_users()`, `delete_user(user_id)`.
   - bcrypt cost factor 12 (configurable, but default 12).
2. **Session middleware**
   - Sessions stored in `sessions` table (id, user_id, expires_at, created_ip, csrf_token).
   - Cookie name: `homelab_monitor_session`. Attributes: HttpOnly, SameSite=Lax (matches LAN-only architecture; users access via single domain), Secure when `HOMELAB_MONITOR_HTTPS_ONLY_COOKIES=true` (default true; set false in dev).
   - Session value is a signed token (HMAC over `session_id`) using a key derived from the master key + a session-purpose label.
   - Session TTL default: 7 days idle. Configurable in `homelab-monitor.yaml`.
3. **CSRF protection**
   - Double-submit cookie pattern: server sets `homelab_monitor_csrf` (HttpOnly=false, readable by JS) at session start; client must echo its value in `X-CSRF-Token` header for state-changing requests.
   - Enforcement on all `POST/PUT/PATCH/DELETE` to `/api/*` except API-token-authenticated calls.
4. **Endpoints**
   - `POST /api/auth/login` → `{username, password}` → 200 with `{user: {id, username}}` on success; sets cookies. Rate-limited (5 attempts per 5 minutes per IP, in-process token bucket).
   - `POST /api/auth/logout` → clears session row + cookies; 204.
   - `GET /api/auth/me` → returns current user or 401.
   - `POST /api/auth/change-password` → requires current password; rotates session afterward.
5. **API tokens** (programmatic access)
   - `api_tokens` table from STAGE-001-004 holds scoped tokens.
   - `hm api-token create --scope heartbeat:write` prints the token once (only).
   - Token format: `homelab_<env-prefix>_<base64url-30-bytes>`; only the SHA-256 hash is stored.
   - Auth via `Authorization: Bearer ...` header.
   - Scopes are an enum: `heartbeat:write`, `alerts:ingest:write`, `read:status`. (Adding scopes is a later additive change.)
   - **Heartbeat and alert-ingest endpoints accept tokens; do NOT require a session.** This is critical — those endpoints are called by external services (cron jobs, Alertmanager).
6. **CLI**
   - `hm user create USERNAME` → prompts for password, creates user.
   - `hm user list` → list users (no hashes).
   - `hm user passwd USERNAME` → set password (admin override; logs to audit).
   - `hm user delete USERNAME` → deletes user.
   - `hm api-token create --scope SCOPE [--name NAME]` → prints token once.
   - `hm api-token list`, `hm api-token revoke ID`.
7. **Auth dependency** in FastAPI
   - `Depends(get_current_user)` returns the authenticated user or raises 401.
   - `Depends(require_scope("..."))` for token-protected endpoints.
   - The `/api/healthz`, `/api/version`, `/api/auth/login` endpoints are unauthenticated.
8. **Replace dev placeholder**
   - The `HOMELAB_MONITOR_DEV_AUTH` env var is removed; the placeholder header is no longer accepted under any circumstance.
9. **Tests**
   - Login success / wrong password / nonexistent user.
   - Session cookie issued; subsequent request with cookie authorizes.
   - Session TTL: a session past TTL is rejected.
   - CSRF: a state-changing request without the header is 403; with it, 200.
   - Rate limit: 6th login attempt within 5 minutes returns 429.
   - API token: valid token + correct scope authorizes; wrong scope returns 403; revoked token returns 401.
   - Audit log row written for: user create/delete, password change, token create/revoke.

## Design Decisions (Locked 2026-05-06)

**D1 — Cookie session value: signed token format `<session_id>.<HMAC>`**
- Cookie value is the session_id (UUID hex, 32 chars) followed by a `.` separator and an HMAC-SHA256 truncated to a fixed length over the session_id. HMAC key derived from the master key via HKDF with a session-purpose label (e.g., `homelab.session.v1`).
- Server validates HMAC FIRST (cheap, in-memory, no DB hit). Only on HMAC pass does the server look up the session row by id. Defense-in-depth + log-spam protection from random `/api/*` scanners.
- Spec phrasing "Sessions revocable via Settings → Auth" requires DB-backed sessions (rules out self-contained signed JSON cookies).
- Cookie name: `homelab_monitor_session`. Attributes: `HttpOnly; SameSite=Lax; Secure (when HOMELAB_MONITOR_HTTPS_ONLY_COOKIES=true, default true; set false in dev); Path=/`.

**D2 — Auth detection: thin resolver middleware + per-route `require_*` dependencies**
- Replace `DevAuthMiddleware` with a thin `AuthMiddleware` that resolves the user-or-None from cookie/token/Authorization header and sets `request.state.user`. NO enforcement in middleware (other than the universal exempt-list for `/api/healthz`, `/api/version`, `/api/openapi.json`, docs URLs, `/api/auth/login`).
- Routes declare their auth posture via FastAPI dependencies:
  - `Depends(require_session())` — session cookie REQUIRED; CSRF check on state-changing methods (POST/PUT/PATCH/DELETE) reads `X-CSRF-Token` header and compares to session.csrf_token.
  - `Depends(require_token_scope("heartbeat:write"))` — API token REQUIRED with the named scope. CSRF skipped (tokens are CSRF-immune).
  - `Depends(require_user_or_token({"alerts:ingest:write"}))` — accept either; if cookie auth used, also enforce CSRF.
- CSRF check lives inside the `require_session()` dependency itself (NOT in middleware). It runs BEFORE handler body-binding and only reads request.headers, no body access.
- Why not monolithic AuthMiddleware: path-prefix routing logic outside FastAPI's router fights the framework; scope checks duplicate logic; per-route deps are FastAPI-idiomatic and CRG-graphable.

**D3 — API token format: `homelab_<env-prefix>_<base64url-30>` + UNIQUE index**
- Token format: `homelab_<HOMELAB_MONITOR_TOKEN_PREFIX>_<base64url-30-bytes>`. Default prefix is `prod`; operator can override (e.g., set to `dev` in dev container). Catches the cross-environment "I pasted prod into dev curl" mishap. Aligns with the homelab-monitor-overrides split where dev/prod containers run side-by-side.
- Storage: SHA-256 hex of the full token in `api_tokens.hash`. Token plaintext NEVER persisted.
- New migration `0004_api_tokens_hash_unique.py`: adds `UNIQUE INDEX api_tokens_hash_idx ON api_tokens(hash)`. SHA-256 collisions are astronomically unlikely; the UNIQUE catches duplicate-insert bugs (e.g., re-running create after a partial transaction) at write time rather than producing two rows that both match at lookup. Lookup becomes O(log n) instead of O(n).
- `hm api-token create` prints the token plaintext ONCE to stdout with a clearly-marked block; CLI help warns the token is irretrievable after exit.

**D4 — Rate-limit storage: in-process dict behind a `LoginRateLimiter` Protocol**
- Spec-blessed in-process token bucket: `dict[str, list[float]]` keyed by IP, sliding-window of 5 attempts per 5 minutes. Lost on restart (acceptable for homelab single-process; attacker who can restart the container has higher-privilege exec already).
- Behind a `LoginRateLimiter` Protocol with a single method `check_and_record(ip: str) -> bool` returning `True` if the attempt is allowed. In-process implementation injected via FastAPI dependency for production. Tests inject a deterministic-clock fake.
- Future SQLite-backed variant can be swapped in without touching the login route.

**D5 — Bootstrap UX: CLI-only with three-signal warning**
- `hm user create USERNAME` prompts for password TWICE (confirmation, like `passwd`); creates user; writes to audit_log atomically.
- Lifespan emits a structured-JSON warning at startup when zero users exist: `lifespan.bootstrap_required` event + reason field. Operator sees this in `docker compose logs` immediately on first run.
- `/api/version` response gains a new field `users_configured: bool` so external monitoring (and the future React frontend) can detect the pre-bootstrap state without authenticating. (NOT exposed on `/api/healthz` to keep the high-frequency healthcheck payload minimal.)
- No web bootstrap endpoint. The "first-run web bootstrap" pattern was rejected — it's the kind of "just for setup" code that survives for years and shows up in CVEs. Three independent signals (CLI, log, version field) cover the operator UX without exposing any unauthenticated write path.

**+ AccessLogMiddleware: log auth path**
- AccessLog emits `auth=session` (with `user_id=<int>`) or `auth=token:<token-name>` per request. Enables post-incident grep ("which token hit /hb/foo at 03:00?") and audit-driven token rotation.
- Applies AFTER the auth resolver middleware sets request.state.user; pulls from request.state.

## Acceptance Criteria

- `POST /api/auth/login` with valid `{username, password}` returns 200 + `{user: {id, username}}` + sets `homelab_monitor_session` and `homelab_monitor_csrf` cookies. Invalid credentials return 401 with uniform error envelope.
- `POST /api/auth/login` with 6 attempts within 5 minutes from the same IP returns 429.
- `GET /api/auth/me` returns 200 with `{user: {id, username}}` when session cookie is valid; 401 otherwise.
- `POST /api/auth/logout` deletes the session row + clears cookies; returns 204.
- `POST /api/auth/change-password` requires current password; on success rotates session (invalidates all sessions for that user, issues a fresh one).
- `POST /api/collectors/{name}/retry` (a state-changing endpoint requiring session) requires `X-CSRF-Token` matching session.csrf_token; otherwise 403 with envelope `{error: {code: "csrf_mismatch", ...}}`.
- API token with `heartbeat:write` scope authorizes `/hb/<id>/start` (placeholder accept; STAGE-013 implements full heartbeat semantics). API token with wrong scope returns 403. Revoked token returns 401.
- `hm user create alice` prompts for password twice, creates user with bcrypt cost 12, writes audit row `user.create`.
- `hm api-token create --scope heartbeat:write --name alertmanager` prints token once + writes audit row.
- `/api/version` includes `users_configured: false` when zero users exist; flips to `true` after first user is created.
- Lifespan emits `lifespan.bootstrap_required` warning when 0 users; absent when ≥1 user.
- AccessLog includes `auth=session(user_id=...)` or `auth=token:<name>` field per request.
- `HOMELAB_MONITOR_DEV_AUTH` env var no longer recognized anywhere; `X-Auth: dev` header has no effect.
- All existing tests using `X-Auth: dev` are migrated to the new `authenticated_client` fixture pattern.
- 100% kernel coverage on new `kernel/auth/` modules. ruff + pyright (strict) clean. `make verify` green.

### Scenario 2 — Direct service-layer interaction (37 assertions)
Pre-bootstrap state: GET /api/healthz returns 200 (auth-exempt); GET /api/version returns 200 with `users_configured: false`; GET /api/auth/me and GET /api/collectors return 401.

Login flow:
- Valid creds → 200 + body `{user: {id, username}}` + sets `homelab_monitor_session` + `homelab_monitor_csrf` cookies

Logout:
