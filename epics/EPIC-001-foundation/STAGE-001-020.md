# STAGE-001-020: Grafana + dashboards-as-code provisioning

## Status: Refinement (Complete)

## Overview

Add Grafana as a sidecar with provisioned datasources (VM and VL) and dashboards-as-code under `deploy/grafana/dashboards/`. Embed Grafana into the dashboard's Metrics screen via iframe, behind our auth proxy. Wire the Metrics sidebar item live. Provide a single starter dashboard ("Host overview") that's auto-loaded on first boot.

After this stage, opening Metrics in the dashboard shows the Grafana host-overview dashboard with live VM data; editing JSON in the repo and restarting the Grafana container picks up the change (or hot-reload via Grafana's provisioning watch).

## Concrete deliverables

1. **Grafana sidecar** in compose
   - Image pinned (e.g., `grafana/grafana-oss:11.x`).
   - Volumes:
     - `data/grafana/` for Grafana's own DB (default SQLite).
     - `deploy/grafana/provisioning/` mounted at `/etc/grafana/provisioning/` (datasources + dashboards provisioning yaml).
     - `deploy/grafana/dashboards/` mounted at `/var/lib/grafana/dashboards/`.
   - Env: `GF_SECURITY_ADMIN_USER` and `GF_SECURITY_ADMIN_PASSWORD` rendered from secrets at boot (similar to AM token rendering in STAGE-001-017). The admin user is for emergency direct access; day-to-day use is via the dashboard reverse proxy where Grafana is configured for anonymous viewer (single-user homelab).
   - `GF_AUTH_ANONYMOUS_ENABLED=true`, `GF_AUTH_ANONYMOUS_ORG_ROLE=Viewer`.
2. **Provisioning files**
   - `deploy/grafana/provisioning/datasources/victoriametrics.yaml` — Prometheus-type datasource pointing at `http://victoriametrics:8428`.
   - `deploy/grafana/provisioning/datasources/victorialogs.yaml` — Loki-compatible datasource pointing at `http://victorialogs:9428` (VL is Loki-compatible enough for basic queries).
   - `deploy/grafana/provisioning/dashboards/dashboards.yaml` — file provider pointing at `/var/lib/grafana/dashboards`.
3. **First dashboard** at `deploy/grafana/dashboards/host-overview.json`
   - Panels: CPU%, memory %, disk usage per mountpoint, load averages, uptime, top-N processes (CPU and memory), network throughput.
   - All panels query VM via the metric names defined in STAGE-001-012.
4. **Reverse-proxy route**
   - The monitor proxies `/grafana/*` → Grafana, gated by our cookie session (anonymous Viewer role inside Grafana means no internal Grafana login).
5. **Metrics screen** at `apps/ui/src/routes/metrics.tsx`
   - Renders an iframe of `/grafana/d/host-overview/host-overview?kiosk` with a "Quick PromQL" panel at the top for ad-hoc queries (calls our `/api/metrics/range` from STAGE-001-015 so it works without leaving our auth context).
   - Sidebar "Metrics" item enabled.
6. **Tests**
   - Integration: bring up the rig; assert the dashboard provisions on first boot (HTTP 200 on `/grafana/api/dashboards/uid/host-overview`).
   - Snapshot: edit `host-overview.json`, restart Grafana, assert the panel count or title change is reflected.

## Out of scope for this stage

- Per-integration dashboards (HA, Pi-hole, Synology, Unifi — those land in their respective epics)
- Alertmanager-driven dashboards (we use Karma for alerts; Grafana is metrics-only)
- Multiple users / RBAC inside Grafana (single-user; anonymous viewer suffices)

## Decisions in this stage

- **Anonymous Viewer in Grafana** — yes; the auth boundary is the monitor's reverse proxy, not Grafana itself. Documented as a security note.
- **Datasource type for VL** — Loki-compatible. VictoriaLogs offers a Loki-compatible API endpoint sufficient for basic LogsQL panels.
- **Dashboards-as-code in JSON** — yes; the project ships `.json` files, not Jsonnet/Grafonnet, to keep contributor barrier low. Future epics may layer Grafonnet if growth requires.

## Design Phase

- **Approaches to present:**
  - (a) Grafana iframe + provisioning + JSON dashboards (recommended)
  - (b) Build our own metric viewer (rejected)
- **UI Options Presented:**
  - Iframe with vs without "Quick PromQL" bar above
- **User Choice:**
  - **(A) Image pin:** `grafana/grafana-oss:12.1.10` — 13 months baked, on still-maintained v12 branch.
  - **(B) VictoriaLogs datasource:** Native `victoriametrics-logs-datasource` plugin v0.26.3. **CORRECTION TO SPEC:** stage file originally said "Loki-compatible datasource"; upstream research confirmed VictoriaLogs does NOT implement enough of the Loki HTTP API for Grafana integration (`/loki/api/v1/index/volume` missing; tracked at VictoriaMetrics/VictoriaLogs#250). The native plugin is the supported path.
  - **(C) Plugin install:** `GF_INSTALL_PLUGINS=victoriametrics-logs-datasource 0.26.3` env var. Requires outbound HTTPS at first boot; persisted in `data_grafana` volume after.
  - **(D) Quick PromQL panel:** **DEFERRED** to a follow-up UI stage. v1 ships iframe-only Metrics screen (mirrors STAGE-019's iframe-only Alerts screen baseline). Grafana's Explore view inside the iframe covers ad-hoc PromQL needs in the interim.
  - **(E) Live streaming:** Disabled in Grafana via `[live] disable = true` (env: `GF_LIVE_DISABLE=true`). Karma proxy is HTTP-only; extending it to handle WebSocket upgrades is unnecessary because Grafana panels can refresh on their own intervals.
  - **(F) Sub-path routing:** Preserve sub-path at the proxy (`/api/grafana/{path}` forwarded as-is); set `GF_SERVER_SERVE_FROM_SUB_PATH=true` and `GF_SERVER_ROOT_URL=http://127.0.0.1/api/grafana/`. Mirrors STAGE-019's karma router prefix-preservation pattern.
  - **(G) Kiosk mode:** Embed iframe with `?kiosk` URL param (full kiosk, no Grafana nav). Provide an "Open in Grafana" escape hatch link (opens `/api/grafana/` without kiosk in a new tab) for users needing Grafana's full UI (alerts admin, plugin management).
- **Seed Data Agreed:** existing test rig fixtures + first dashboard `host-overview.json` with CPU%, memory%, disk%, load averages, uptime, top-N processes, network throughput.
- **Session Notes:** Most decisions transfer directly from STAGE-019 (Karma + kthxbye iframe pattern). Brainstormer (Opus) presented options for 7 architecturally-sensitive decisions; user accepted the recommended package across the board. The most material correction is decision (B): the spec's "Loki-compatible datasource" wording is incorrect — VL doesn't have full Loki API parity, and the dedicated VictoriaLogs Grafana plugin is the supported path. STAGE-021's `make dev-prod` deliverable includes Grafana so the integration capstone validates this stage end-to-end. Anonymous-Viewer auth in Grafana, dashboards-as-code as JSON (not Jsonnet), and provisioning paths under `deploy/grafana/provisioning/{datasources,dashboards}/` are LOCKED per stage spec — no brainstorming needed.
