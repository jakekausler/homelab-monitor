# STAGE-001-007: In-process plugin loader + scheduler

## Status: Not Started

## Overview

Implement the runtime that discovers in-process Python collector plugins (built-ins via filesystem, third-party via entry points), loads their configs, and schedules them according to their declared interval. Each tick honors `run_kind` (ASYNC inside the FastAPI loop, THREAD via `asyncio.to_thread`, PROCESS via a `ProcessPoolExecutor`) and per-collector timeout.

This is the first stage where the system "runs" anything. After this stage, starting the monitor wires up the noop collector and we can see ticks fire on schedule.

## Concrete deliverables

1. **Plugin discovery** at `apps/monitor/homelab_monitor/kernel/plugins/loader.py`
   - **Built-in scan:** walks `homelab_monitor/plugins/collectors/builtin/` and `homelab_monitor/plugins/collectors/integrations/` for modules exposing a class that subclasses `BaseCollector` (or otherwise satisfies the `Collector` Protocol).
   - **Entry-point scan:** loads any package registered under entry-point group `homelab_monitor.collectors`. (Subprocess plugins via manifest land in STAGE-001-009.)
   - **Config loading:** for each discovered collector, looks under `/config/plugins/collectors/<name>.{yaml,toml}` for a config file. If none, uses the collector's declared default config. Validates against the collector's pydantic config model.
   - Returns a list of `LoadedCollector(definition, config)` records.
2. **Scheduler** at `apps/monitor/homelab_monitor/kernel/scheduler/scheduler.py`
   - Async scheduler driven by `asyncio.gather` of per-collector loops.
   - Each collector loop: sleep until next tick â†’ invoke `_run_with_kind` â†’ record metrics â†’ schedule next tick.
   - **`_run_with_kind`** dispatches based on `run_kind`:
     - `ASYNC`: `await collector.run(ctx)` directly within a `asyncio.timeout(timeout)` block.
     - `THREAD`: `await asyncio.to_thread(_sync_runner, collector, ctx)`. The `_sync_runner` runs the collector's `run` via `asyncio.run` if it's coroutine-only, or directly if a sync entry is provided. Timeout enforced by joining a `Future` with a deadline; if missed, `_sync_runner` raises and the thread is left to expire.
     - `PROCESS`: shared `ProcessPoolExecutor`. Collector + serializable subset of context shipped. Result returned. Timeout: `concurrent.futures.Future` with `timeout=` and on timeout we cancel the future and kill the worker process if needed.
   - **Tick precision:** scheduler uses absolute deadlines (not "sleep interval") so jitter doesn't accumulate.
   - **Initial offset:** when many collectors share the same interval, scheduler offsets each by `(hash(name) % interval)` to avoid simultaneous-tick storms.
3. **Self-metrics emission** at every tick:
   - `homelab_collector_run_success_total{name}`
   - `homelab_collector_run_failure_total{name, reason}`
   - `homelab_collector_run_duration_seconds{name}` (summary)
   - `homelab_collector_run_last_error_age_seconds{name}` (gauge updated on tick)
   These are emitted via the `MetricsWriter` stub for now; real VM-backed emission lands in STAGE-001-012/015.
4. **Lifecycle**
   - `Scheduler.start()` and `Scheduler.stop()`.
   - Graceful shutdown: cancels all tick loops; in-flight tasks given `shutdown_grace_seconds` (default 30) before forced cancel.
5. **Tests** (in `tests/unit/scheduler/`)
   - **Tick precision:** schedule a noop collector at 100ms; over a 1.5s test window, expect 14â€“16 ticks (jitter tolerance).
   - **Multiple collectors with different intervals:** all fire correctly.
   - **Per-collector timeout:** a collector that sleeps `interval + 1s` is cancelled and a failure metric is emitted.
   - **THREAD run_kind:** a sync collector runs without blocking the loop.
   - **PROCESS run_kind:** a CPU-loop collector runs in a worker process; main loop continues.
   - **Initial offset:** two collectors with identical intervals don't have correlated tick times.
   - **Graceful shutdown:** `stop()` cancels everything within grace; pending ticks not lost (recorded as "shutdown" outcome, not "failed").

## Decisions in this stage

- **`ProcessPoolExecutor` size** â€” `min(2, os.cpu_count())` by default; configurable via `homelab-monitor.yaml`.
- **THREAD timeout strategy** â€” best-effort: we don't kill worker threads (Python doesn't support that cleanly). If a thread runs past its timeout, the scheduler logs and emits the failure metric, but the thread continues until completion. THREAD plugins should use cooperative cancellation if they need true timeouts. PROCESS run_kind is the escape hatch.
- **Whether to checkpoint in-flight runs** â€” no for this stage. Restart loses in-flight ticks; next scheduled tick re-runs.
