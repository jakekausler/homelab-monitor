# STAGE-001-007: In-process plugin loader + scheduler

## Status: Complete

## Overview

Implement the runtime that discovers in-process Python collector plugins (built-ins via filesystem, third-party via entry points), loads their configs, and schedules them according to their declared interval. Each tick honors `run_kind` (ASYNC inside the FastAPI loop, THREAD via `asyncio.to_thread`, PROCESS via a `ProcessPoolExecutor`) and per-collector timeout.

This is the first stage where the system "runs" anything. After this stage, starting the monitor wires up the noop collector and we can see ticks fire on schedule.

**Spec references:** §5.1 (kernel vs plugin layer), §5.4 (concurrency model), §5.5 (plugin discovery & install), §3.1 (collector scheduler).

## Concrete deliverables

1. **Plugin discovery** at `apps/monitor/homelab_monitor/kernel/plugins/loader.py`
   - **Built-in scan:** walks `homelab_monitor/plugins/collectors/builtin/` and `homelab_monitor/plugins/collectors/integrations/` for modules exposing a class that subclasses `BaseCollector` (or otherwise satisfies the `Collector` Protocol).
   - **Entry-point scan:** loads any package registered under entry-point group `homelab_monitor.collectors`. (Subprocess plugins via manifest land in STAGE-001-009.)
   - **Config loading:** for each discovered collector, looks under `/config/plugins/collectors/<name>.{yaml,toml}` for a config file. If none, uses the collector's declared default config. Validates against the collector's pydantic config model.
   - Returns a list of `LoadedCollector(definition, config)` records.
2. **Scheduler** at `apps/monitor/homelab_monitor/kernel/scheduler/scheduler.py`
   - Async scheduler driven by `asyncio.gather` of per-collector loops.
   - Each collector loop: sleep until next tick → invoke `_run_with_kind` → record metrics → schedule next tick.
   - **`_run_with_kind`** dispatches based on `run_kind`:
     - `ASYNC`: `await collector.run(ctx)` directly within a `asyncio.timeout(timeout)` block.
     - `THREAD`: `await asyncio.to_thread(_sync_runner, collector, ctx)`. The `_sync_runner` runs the collector's `run` via `asyncio.run` if it's coroutine-only, or directly if a sync entry is provided. Timeout enforced by joining a `Future` with a deadline; if missed, `_sync_runner` raises and the thread is left to expire.
     - `PROCESS`: shared `ProcessPoolExecutor`. Collector + serializable subset of context shipped. Result returned. Timeout: `concurrent.futures.Future` with `timeout=` and on timeout we cancel the future and kill the worker process if needed.
   - **Tick precision:** scheduler uses absolute deadlines (not "sleep interval") so jitter doesn't accumulate.
   - **Initial offset:** when many collectors share the same interval, scheduler offsets each by `(hash(name) % interval)` to avoid simultaneous-tick storms.
3. **Self-metrics emission** at every tick:
   - `homelab_collector_run_success_total{name}`
   - `homelab_collector_run_failure_total{name, reason}`
   - `homelab_collector_run_duration_seconds{name}` (summary)
   - `homelab_collector_run_last_error_age_seconds{name}` (gauge updated on tick)
   These are emitted via the `MetricsWriter` stub for now; real VM-backed emission lands in STAGE-001-012/015.
4. **Lifecycle**
   - `Scheduler.start()` and `Scheduler.stop()`.
   - Graceful shutdown: cancels all tick loops; in-flight tasks given `shutdown_grace_seconds` (default 30) before forced cancel.
5. **Tests** (in `tests/unit/scheduler/`)
   - **Tick precision:** schedule a noop collector at 100ms; over a 1.5s test window, expect 14–16 ticks (jitter tolerance).
   - **Multiple collectors with different intervals:** all fire correctly.
   - **Per-collector timeout:** a collector that sleeps `interval + 1s` is cancelled and a failure metric is emitted.
   - **THREAD run_kind:** a sync collector runs without blocking the loop.
   - **PROCESS run_kind:** a CPU-loop collector runs in a worker process; main loop continues.
   - **Initial offset:** two collectors with identical intervals don't have correlated tick times.
   - **Graceful shutdown:** `stop()` cancels everything within grace; pending ticks not lost (recorded as "shutdown" outcome, not "failed").

- Concurrency groups (STAGE-001-008) — every tick is currently free to run concurrently with every other tick.
- Failure budget / quarantine (STAGE-001-008).

- Real metrics destination (STAGE-001-012/015).
- Plugin hot-reload (deferred; restart for now).

## Decisions in this stage

- **`ProcessPoolExecutor` size** — `min(2, os.cpu_count())` by default; configurable via `homelab-monitor.yaml`.
- **THREAD timeout strategy** — best-effort: we don't kill worker threads (Python doesn't support that cleanly). If a thread runs past its timeout, the scheduler logs and emits the failure metric, but the thread continues until completion. THREAD plugins should use cooperative cancellation if they need true timeouts. PROCESS run_kind is the escape hatch.
- **Whether to checkpoint in-flight runs** — no for this stage. Restart loses in-flight ticks; next scheduled tick re-runs.

## Design Phase

- **Approaches to present:**
  - (a) per-collector asyncio.Task with absolute-deadline ticks (recommended)
  - (b) global priority-queue scheduler with one driver task
  - (c) APScheduler library
- **UI Options Presented:** N/A in this stage.
- **User Choice:** (a) per-collector asyncio.Task with absolute-deadline ticks
- **Seed Data Agreed:** N/A — purely algorithmic stage; tests use NoopCollector + new test-only collectors
- **Session Notes:**
  - **D1 driver shape:** Per-collector asyncio.Task with absolute-deadline tick loop. Pattern: `sleep_until(deadline); await _tick(c); deadline += interval; if deadline < now: deadline = now + interval` (drift guard skips forward instead of catch-up storm). Cancellation = single `task.cancel()` per collector. Concurrency-groups (STAGE-001-008) slot in as `async with self._group_locks[c.concurrency_group]:` inside `_tick` with no driver changes.
  - **D2 clock source:** `asyncio.get_running_loop().time()` everywhere in scheduler — same monotonic clock asyncio.sleep uses internally. Convert to UTC datetime ONLY at metric/log emission boundaries. Avoids NTP/DST jumps.
  - **D3 PROCESS run-kind context:** Pure-result workers — worker gets `(collector_cls, ProcessCollectorContext(config, secrets))`, returns `(CollectorResult, buffered_metrics_list)`. `ProcessCollectorContext` is a separate slim dataclass with ONLY picklable fields. Worker uses a `BufferingMetricsWriter` (appends to list) instead of the live `MetricsWriter`. Parent process replays buffered metrics through the real writer. Single pickle hop in, one out — extends naturally to STAGE-001-009 subprocess pattern.
  - **D4 self-metrics ownership:** Scheduler-owned. The `_tick` wrapper unconditionally emits the 4 mandated `homelab_collector_run_*` metrics for every collector regardless of what the collector does internally. Collectors retain `ctx.metrics` for their own domain metrics (`homelab_<name>_*`). "Plugins observe themselves" satisfied at the system level via uniform scheduler instrumentation — plugin authors can't forget self-metrics.
  - **D5 shutdown outcome:** New 5th self-metric `homelab_collector_run_shutdown_total{name}` distinct from `failure_total`. In-flight ticks cancelled during graceful stop increment shutdown_total only — `failure_total` stays semantically "real problems" so alertmanager rules can fire on `rate(failure_total[5m]) > 0` without filtering. Stage file's deliverables list (currently 4 self-metrics) is expanded to include this 5th counter; the self-metrics emission section in Concrete Deliverables item 3 is updated accordingly.
  - **Forward-compat verified:**
    - STAGE-001-008 concurrency groups → add per-group `asyncio.Lock` inside `_tick` before `_dispatch`. No driver changes.
    - STAGE-001-008 quarantine → check `c.name in self._quarantined` inside per-collector loop; skip `_tick`. No dispatch changes.
    - STAGE-001-009 subprocess plugins → new `RunKind.SUBPROCESS` arm in `_dispatch`; the buffer-and-replay model from PROCESS generalizes.
    - STAGE-001-010 FastAPI lifespan → just calls `Scheduler.start()` and `Scheduler.stop()`.

- **Session Notes:**
  - **Round-1 finding (debugger-lite):** Test wall-time too short for THREAD test (200ms vs 100ms interval ≈ borderline) and for the PROCESS test (cold-start latency); pickle failure on dynamically-created `_make_collector(...)` class because it routed through the `abc` module (BaseCollector is ABC) instead of the test module's `__name__`. Replaced dynamic class with explicit `class _ProcessTestCollector(BaseCollector):` definition for module-level pickling.
  - **Round-2 critical bug (debugger-lite):** Drift-skip logic at scheduler.py used `if deadline < now:` which fired on the FIRST iteration (microsecond delta between assigning `deadline` and re-fetching `now`). This pushed the first tick to `now + interval` AFTER one full interval, breaking both the timeout test and the in-flight-shutdown test. Fixed to `if deadline + interval < now:` so only TRUE drift > 1 interval triggers skip-forward.
  - **Round-2 typing fix:** `Scheduler.stop()` cancellation flow originally awaited in-flight tasks within the grace timeout — they completed successfully via `asyncio.timeout()` instead of being cancelled, so `shutdown_total` never fired. Fixed to cancel in-flight tasks IMMEDIATELY before the grace gather; the grace timer now waits for cancellation propagation, not natural completion.
  - **Round-3 typing fix:** `_process_collector_impl` was incorrectly typed `ctx: CollectorContext`; the worker actually receives `ProcessCollectorContext` (which has `metrics`, not `vm`). Pyright complained about `ctx.metrics` access on CollectorContext; round-2 mistakenly "fixed" it by changing to `ctx.vm`, which broke the test runtime (ProcessCollectorContext has no `vm` field). Round-3 reverted the field rename AND tightened the parameter type to `ProcessCollectorContext`, with a `# type: ignore[arg-type]` at the wrapper class call site.
  - **Hash-collision flake:** `test_initial_offset_decorrelates_ticks` was inherently flaky because `hash("alpha") % 3 == hash("beta") % 3` happens 1-in-3 with PYTHONHASHSEED randomized. Added a deterministic `pytest.skip()` guard that mirrors the scheduler's exact offset formula; when collision occurs, the test skips with reason rather than failing.
  - **Coverage gate:** Six lines marked `# pragma: no cover` (or `# pragma: no branch`) for genuinely defensive paths: subprocess `_process_runner` body (parent-process coverage doesn't aggregate), gauge/summary metric replay branches in PROCESS dispatch (test only emits counter), TimeoutError second-pass cancel guard in `stop()` (cancellation already propagated above), drift > 1 interval skip-forward (timing-dependent), `_stopping` mid-tick break (timing race), CancelledError re-raise outside shutdown (external cancel of tick task), and `NotImplementedError` for unknown RunKind (StrEnum exhaustively dispatched). Each pragma carries a one-line rationale comment.
