# STAGE-001-018: vmalert (logs) + first log-derived rule

## Status: Design (Complete)

## Overview

Add the second vmalert instance, this one targeting VictoriaLogs. Author the first log-derived rule. After this stage, log-pattern alerts flow the same path as metric alerts (Alertmanager → ingestor → channels), with `source_tool=vmalert-logs`.

**Spec references:** §3.2 (vmalert ×2), §6.3 (logs streams), §4.3 (log flow), §13.2 (integration tests).

## Concrete deliverables

1. **vmalert (logs) sidecar** in compose
   - Same image as the metrics one. Args: `-datasource.url=http://victorialogs:9428`, `-notifier.url=http://alertmanager:9093`, `-rule=/etc/vmalert/logs/*.yaml`. The two vmalert services are named `vmalert-metrics` and `vmalert-logs`.
2. **First rule** at `deploy/vmalert/logs/system.yaml`
   - Group `system_logs`:
     - **`KernelOOM`** — LogsQL `{service="kernel"} ~ "Out of memory.*kill_process"` count > 0 over 5m → severity=critical, source_tool=vmalert-logs, target_kind=host.
     - **`SshFailedLoginBurst`** — LogsQL `{service="sshd"} ~ "Failed password"` count > 10 over 5m → severity=warning, source_tool=vmalert-logs, target_kind=host. (Useful even on day one — guards against drive-by SSH scans on hosts with internet exposure.)
     - **`LogStreamBudgetOverrun`** — bound to the `log_stream_budget` metric from STAGE-001-016 (this rule is metrics-side actually; cross-listed here for completeness — it lives in `deploy/vmalert/metrics/log_stream_budget.yaml`).
3. **Test rig extension**
   - `docker-compose.test.yml` adds a fixture container that, on demand (env-controlled), emits planted log lines matching the rules.
4. **Tests**
   - Integration: planted "Out of memory: Killed process 1234 (badproc)" → vmalert-logs fires within eval+wait → alert appears in `/api/alerts` with `source_tool=vmalert-logs`.
   - Integration: 11 "Failed password" lines in 5m → SshFailedLoginBurst fires.
   - Resolved path: lines stop, alert auto-resolves.
   - Source tagging assertion: `source_tool` is correctly set to `vmalert-logs` (not `vmalert-metrics`).

## Out of scope for this stage

- Drain log clustering / signature-based rules (EPIC-004; this stage is hand-written rules)
- HA / Pi-hole / Synology / Unifi log rules (those land in their respective epics)
- Log redaction policy (EPIC-004)

## Decisions in this stage

- **Two vmalert instances** confirmed (vs one). The two-instance pattern is canonical for vmalert + VL.
- **Initial rules conservative** — high-confidence patterns only (OOM, SSH burst). Wide pattern matching belongs in the Drain-driven approach (EPIC-004).

## Design Phase

- **Approaches to present:**
  - (a) two vmalert instances (recommended; spec)
  - (b) one combined instance via a query-router (doesn't exist as a stable upstream component)
- **UI Options Presented:** N/A
- **User Choice:** vmalert-logs sidecar reusing the vmalert-metrics + Alertmanager + AlertDispatcher infrastructure from STAGE-001-017. Locked decisions:
  - **Q1 — Test fixture**: Direct curl from pytest helper. Test plants log lines via httpx POST to `http://victorialogs:9428/insert/jsonline` directly from the integration-tests container with explicit `_time` fields for deterministic timestamps. Helper at `apps/monitor/tests/integration/helpers/vl_planter.py` reusable for STAGE-021. No extra fixture container, no vector pipeline coupling — vmalert-logs is what's under test; vector → VL path is STAGE-021 scope.
  - **Q2 — Eval windows**: Test-specific rule dir at `deploy/vmalert/logs-test/` with shorter windows (`count > 0 over 30s` for OOM; `count > 2 over 30s` for SshFailedLoginBurst) mirroring host.yaml's "fast-test convention" pattern. Production keeps canonical 5m windows in `deploy/vmalert/logs/`. Integration tests use the test-windows rule file. The slow integration test is gated by `@pytest.mark.slow` so `make test-fast` excludes it; the default `make verify` runs it (per the hybrid testing strategy locked in STAGE-017 Q4).
  - **Q3 — LogStreamBudgetOverrun**: `homelab_log_stream_lines_per_sec > 45` for 5m, severity=warning, source_tool=vmalert-metrics, target_kind=log_stream. 45 = 90% of vector's hardcoded 50/s throttle. The LogStreamBudgetCollector emits exactly two gauges (`homelab_log_stream_bytes_today`, `homelab_log_stream_lines_per_sec`) with `{host, service}` labels; no threshold gauge exists. Bytes-today has no daily budget defined anywhere — don't alert on it.
  - **Q4 — Architectural extras**: (a) Land `system.yaml` + vmalert-logs service in SAME commit so the rules dir is never empty at first boot. (b) Mirror vmalert-metrics healthcheck (`wget http://localhost:8880/-/health`). vmalert is stateless; no persistent volume. (c) Build-phase verify VL `-datasource.url` form (VictoriaLogs v0.30.0 may expect `/select/logsql` path suffix vs bare URL). (d) Build-phase verify Alertmanager routing accepts the new vmalert-logs notifier without changes (existing flat routing should work).
- **Seed Data Agreed:** Three log-derived rules in `deploy/vmalert/logs/system.yaml` (KernelOOM, SshFailedLoginBurst) + one cross-listed metrics rule (`LogStreamBudgetOverrun` at `deploy/vmalert/metrics/log_stream_budget.yaml`).
- **Session Notes:**
  - STAGE-017 left baseline at 1041 backend tests / 100% kernel coverage; verified clean before Design.
  - vmalert-logs reuses Alertmanager from STAGE-017. The same `data_alertmanager_config` rendered config + AM webhook receiver handles both metric and log-derived alerts. Both vmalert instances point to `-notifier.url=http://alertmanager:9093`.
  - vector.toml labels (`host`, `service`, `severity`) confirmed as the LogsQL selector keys for rule expressions.
  - `homelab_collector_quarantine_count` metric remains deferred (commented in host.yaml). Out of scope for STAGE-018.
