# STAGE-001-012: First built-in `host` collector

## Status: Build (Complete)

## Overview

Land the first real built-in collector: `host`. It uses `psutil` to gather CPU, RAM, disk, network, load average, uptime, and per-process top-N for the host the monitor runs on. Metrics flow through the (still in-memory) `MetricsWriter` stub. The collector status is visible via `/api/collectors`. Self-metrics from the scheduler emit alongside.

This stage proves: the kernel + plugin framework can run a real collector that produces real metrics. The rest of the monitor's metric flow (real VM destination, alert routing, dashboards) builds on this single collector before we add more.

## Concrete deliverables

1. **Collector** at `apps/monitor/homelab_monitor/plugins/collectors/builtin/host.py`
   - Class `HostCollector(BaseCollector)`.
   - `name = "host"`, `interval = 10s`, `timeout = 5s`, `concurrency_group = "host"`, `run_kind = ASYNC`, `trust_level = BUILTIN`.
   - Uses `psutil` for:
     - `homelab_host_cpu_percent` (gauge, labels: `cpu="all"|N`)
     - `homelab_host_load_average` (gauge, labels: `period="1m"|"5m"|"15m"`)
     - `homelab_host_memory_bytes{type="used"|"available"|"total"}` (gauge)
     - `homelab_host_swap_bytes{type="used"|"total"}` (gauge)
     - `homelab_host_disk_bytes{mountpoint, type="used"|"total"}` (gauge) — iterates `psutil.disk_partitions(all=False)` plus an explicit list from config (e.g., `/rackstation`)
     - `homelab_host_disk_io_bytes_total{disk, direction="read"|"write"}` (counter)
     - `homelab_host_net_bytes_total{iface, direction="rx"|"tx"}` (counter)
     - `homelab_host_uptime_seconds` (gauge)
     - `homelab_host_processes_total{state="running"|"sleeping"|"zombie"}` (gauge)
     - `homelab_host_top_processes_cpu_percent{name, pid}` (gauge; top 10 by CPU)
     - `homelab_host_top_processes_memory_bytes{name, pid}` (gauge; top 10 by RSS)
   - Default config (overridable via `/config/plugins/collectors/host.yaml`):
     - `extra_mountpoints: ["/rackstation"]` (notable because of the Synology mount)
     - `top_n_processes: 10`
     - `exclude_disk_devs: ["loop", "ram"]` (regex prefixes)

## Decisions in this stage

- **Top-N processes** — included. Useful for "what's eating my CPU right now" without a full processes-by-pid scrape.
- **Per-CPU vs aggregate** — both. Aggregate has `cpu="all"`; per-core has `cpu="0"|"1"|...`.
- **Keep filesystems with usage > 0%** — explicit filter; don't emit for empty `/dev/loopN` etc.

### Design phase decisions (locked 2026-05-06)

**D1 — Writer architecture: Subclass `MemoryRetainingMetricsWriter(InMemoryMetricsWriter)`.**

The new class extends the existing append-only `InMemoryMetricsWriter` with a latest-value-by-(name, frozen-labels) dict and a `replace_family(name, entries)` method (see D4). Drop-in replacement at the lifespan registration site; existing consumers (`/api/collectors`, `/api/healthz`, scheduler self-metrics) read `.recorded` and the `last_tick_at_for` / `last_error_for` helpers unchanged. One concrete instance lives at `app.state.metrics_writer`.

**D2 — Snapshot endpoint shape: Flat array.**

```json
{
  "ts": "2026-05-06T12:34:56.789Z",
  "entries": [
    {"name": "homelab_host_cpu_percent", "value": 12.5, "labels": {"cpu": "all"}, "kind": "gauge", "ts": "2026-05-06T12:34:56.500Z"},
    ...
  ]
}
```

Top-level `ts` is the snapshot capture time. Each entry carries its own `ts` (most-recent write for that (name, labels) tuple). STAGE-014 Overview tile filters with `entries.find(e => e.name === 'homelab_host_cpu_percent' && e.labels.cpu === 'all')`. Pydantic response model: `MetricsSnapshotResponse`.

**D3 — Per-tick timestamp: Writer auto-stamps `ts` inside `write_*` methods.**

Each `write_gauge / write_counter / write_summary` call records `ts = utc_now_iso()` at the time of the write. The collector does NOT pass `ts`. Protocol unchanged. No backdating support (non-goal for STAGE-012).

**D4 — Top-N processes cardinality: Epoch semantics via `replace_family(name, entries)`.**

For top-N families (`homelab_host_top_processes_cpu_percent`, `homelab_host_top_processes_memory_bytes`), the collector calls `ctx.vm.replace_family(name, [(value, labels), ...])` which atomically wipes prior entries for that family from the latest-value map and writes the new top-N. Snapshot is bounded to ≤ 2 × `top_n_processes` for top-N families. Mirrors how `topk()` would behave on real VM (STAGE-015). Other metrics use the standard `write_*` methods (latest value per (name, labels) is overwritten in place).

**D5 — Config loading: Pydantic defaults + inline lifespan registration (no YAML loader yet).**

`HostCollectorConfig(CollectorConfig)` Pydantic subclass carries spec defaults (`extra_mountpoints=["/rackstation"]`, `top_n_processes=10`, `exclude_disk_devs=["loop", "ram"]`). Registered inline in lifespan exactly like `NoopCollector`. A `# TODO(STAGE-014): load from /config/plugins/collectors/host.yaml when YAML config-loading lands` comment goes at the registration site. Real YAML config-loading is deferred to STAGE-014/015.

### Other locked specifics (from stage spec — confirmed in Design)

- **Collector path:** `apps/monitor/homelab_monitor/plugins/collectors/builtin/host.py` (file, not directory; new tree under `plugins/`, NOT `kernel/plugins/`)
- **ClassVars:** `name="host"`, `interval=timedelta(seconds=10)`, `timeout=timedelta(seconds=5)`, `concurrency_group="host"`, `run_kind=RunKind.ASYNC`, `trust_level=TrustLevel.BUILTIN`
- **Metric prefix:** `homelab_host_*` (NOT `host_*`)
- **CPU labels:** both aggregate `cpu="all"` AND per-core `cpu="0"|"1"|...`
- **Disk filter:** skip mountpoints with `usage == 0%` OR device name matching `^(loop|ram)` regex prefix (per `exclude_disk_devs` config)
- **Snapshot endpoint auth:** `Depends(require_session())` (cookie-only; no token; CSRF NOT enforced for GET)

### Test plan (Build phase)

- **Unit (psutil mocked):** `monkeypatch.setattr(psutil, "cpu_percent", ...)` etc. Assert each metric family appears with correct labels and kind. One test per metric family.
- **Unit (writer):** `replace_family()` atomicity; per-entry `ts` auto-stamping; `.recorded` still grows append-only.
- **Integration (real psutil):** assert types only (values vary by host). One test that drives a single tick and inspects the writer.
- **API:** `GET /api/metrics/snapshot` returns 200 with response model shape; 401 without session cookie; CSRF NOT enforced (GET); empty entries list when no metrics written yet.
- **Stress:** 100 sequential ticks against `MemoryRetainingMetricsWriter`; assert `len(writer._entries)` grows linearly (counters/gauges expected) but `len(writer._latest)` for top-N families remains bounded to `2 * top_n_processes` (epoch semantics verified).
- **Adversarial (per STAGE-011 lesson):** snapshot endpoint with no cookie → 401; with valid session no CSRF header → 200 (GET-only).

## Build Phase Notes (Completed 2026-05-06)

### Implementation summary
- New collector: `apps/monitor/homelab_monitor/plugins/collectors/builtin/host.py` (`HostCollector` + `HostCollectorConfig`). Run() refactored into 10 private helper methods (`_collect_cpu`, `_collect_load_average`, etc.) for complexity compliance.
- New retaining writer: `MemoryRetainingMetricsWriter(InMemoryMetricsWriter)` in `kernel/plugins/io.py`. Adds `_latest` dict + `replace_family()` method for epoch-semantics top-N families. Auto-stamps per-entry `ts` via `utc_now_iso()` (note: produces `+00:00` offset format, NOT `Z`-suffix).
- New endpoint: `GET /api/metrics/snapshot` at `kernel/api/routers/metrics.py`. Auth via `Depends(require_session())`. Response shape: `{ts, entries: [{name, value, labels, kind, ts}, ...]}`.
