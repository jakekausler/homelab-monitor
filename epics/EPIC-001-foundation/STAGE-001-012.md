# STAGE-001-012: First built-in `host` collector

## Status: Not Started

## Overview

Land the first real built-in collector: `host`. It uses `psutil` to gather CPU, RAM, disk, network, load average, uptime, and per-process top-N for the host the monitor runs on. Metrics flow through the (still in-memory) `MetricsWriter` stub. The collector status is visible via `/api/collectors`. Self-metrics from the scheduler emit alongside.

This stage proves: the kernel + plugin framework can run a real collector that produces real metrics. The rest of the monitor's metric flow (real VM destination, alert routing, dashboards) builds on this single collector before we add more.

## Concrete deliverables

1. **Collector** at `apps/monitor/homelab_monitor/plugins/collectors/builtin/host.py`
   - Class `HostCollector(BaseCollector)`.
   - `name = "host"`, `interval = 10s`, `timeout = 5s`, `concurrency_group = "host"`, `run_kind = ASYNC`, `trust_level = BUILTIN`.
   - Uses `psutil` for:
     - `homelab_host_cpu_percent` (gauge, labels: `cpu="all"|N`)
     - `homelab_host_load_average` (gauge, labels: `period="1m"|"5m"|"15m"`)
     - `homelab_host_memory_bytes{type="used"|"available"|"total"}` (gauge)
     - `homelab_host_swap_bytes{type="used"|"total"}` (gauge)
     - `homelab_host_disk_bytes{mountpoint, type="used"|"total"}` (gauge) â€” iterates `psutil.disk_partitions(all=False)` plus an explicit list from config (e.g., `/rackstation`)
     - `homelab_host_disk_io_bytes_total{disk, direction="read"|"write"}` (counter)
     - `homelab_host_net_bytes_total{iface, direction="rx"|"tx"}` (counter)
     - `homelab_host_uptime_seconds` (gauge)
     - `homelab_host_processes_total{state="running"|"sleeping"|"zombie"}` (gauge)
     - `homelab_host_top_processes_cpu_percent{name, pid}` (gauge; top 10 by CPU)
     - `homelab_host_top_processes_memory_bytes{name, pid}` (gauge; top 10 by RSS)
   - Default config (overridable via `/config/plugins/collectors/host.yaml`):
     - `extra_mountpoints: ["/rackstation"]` (notable because of the Synology mount)
     - `top_n_processes: 10`
     - `exclude_disk_devs: ["loop", "ram"]` (regex prefixes)
2. **`MetricsWriter` stub upgrade** at `apps/monitor/homelab_monitor/kernel/plugins/io.py`
   - The `InMemoryMetricsWriter` from STAGE-001-006 already exists. This stage adds a fixture-friendly **default** stub `MemoryRetainingMetricsWriter` used at process boot when no real backend is configured. It retains the last value per (name, label_set) so `/api/metrics/snapshot` can return current state.
   - **Real VM-backed `MetricsWriter` lands in STAGE-001-015**, NOT here. The plugin must not assume the backend.
3. **Status surface**
   - `/api/collectors` from STAGE-001-010 already exists; verify the `host` collector appears with: name, interval, last_run_at, last_status, last_error, next_run_at, last_duration_seconds, quarantined.
   - Add `GET /api/metrics/snapshot` returning the in-memory writer's current state â€” useful for dev/debug and the dashboard live tile (STAGE-001-014). The endpoint is auth-required.
4. **Tests**
   - Run the collector against a fixture `psutil` (or against the real one in CI's Linux runner; both are acceptable).
   - Assert each expected metric family appears.
   - Assert label sets are sensible (e.g., disk metrics for `/` mountpoint).
   - Assert `homelab_collector_run_*` self-metrics emit.
   - Stress: 100 ticks in a row don't leak memory (rough check on the in-memory writer's footprint).

## Decisions in this stage

- **Top-N processes** â€” included. Useful for "what's eating my CPU right now" without a full processes-by-pid scrape.
- **Per-CPU vs aggregate** â€” both. Aggregate has `cpu="all"`; per-core has `cpu="0"|"1"|...`.
- **Keep filesystems with usage > 0%** â€” explicit filter; don't emit for empty `/dev/loopN` etc.
