# STAGE-001-008: Concurrency groups + failure budget + quarantine

## Status: Not Started

## Overview

Add the two pieces of behavioral discipline that make the scheduler safe under load: **concurrency groups** prevent thundering-herd against a single shared backend (e.g., the UDM controller, the HA API, the Synology DSM), and **failure budgets with quarantine** prevent a broken plugin from running every tick forever and flooding alerts.

After this stage, plugins in the same `concurrency_group` are serialized, and a plugin that fails N consecutive times is quarantined (paused), an alert is emitted, and the dashboard offers a "retry now" action.

**Spec references:** §5.4 (concurrency model — `concurrency_group`), §5.7 (per-collector failure budget), §3.1 (collector scheduler).

## Concrete deliverables

1. **Concurrency-group enforcement** in `scheduler.py`
   - The scheduler maintains an `asyncio.Lock` per `concurrency_group`.
   - Before invoking `_run_with_kind`, the scheduler acquires the group lock; if held, the tick is queued (with a deadline tied to `interval / 2` — if it can't run in that window, it's skipped and a `homelab_collector_skipped_total{reason="group_busy"}` metric is incremented).
   - **Group default:** if a collector doesn't declare a group, it gets its own (effectively unbounded concurrency relative to others).
   - **Group ordering:** within a group, ticks run in FIFO order.
2. **Failure budget tracker** at `apps/monitor/homelab_monitor/kernel/scheduler/failure_budget.py`
   - Per-collector counter of consecutive failures.
   - Quarantine threshold default: 5 consecutive failures (configurable per collector via `quarantine_after`).
   - Cooldown after quarantine: collector is paused; ticks are NOT scheduled while quarantined.
   - Successful run resets counter to 0.
3. **Quarantine state** persisted in `collectors` table:
   - Columns added (additive migration `0002_collector_quarantine.py`): `quarantined_at` (nullable ISO timestamp), `quarantine_reason` (nullable text), `consecutive_failures` (int default 0).
   - State survives restart so we don't accidentally hammer a known-broken backend on boot.
4. **Quarantine alert**
   - When a collector enters quarantine, the scheduler emits a synthetic alert via the (future) alert ingestor — for this stage, since the alert ingestor isn't built until STAGE-001-013, **the quarantine event is logged at WARNING level + recorded as a row in `audit_log`**, and the alert emission is wired in STAGE-001-013 (a TODO comment with stage reference is acceptable).
5. **Retry-now API** — extend the (placeholder) plugin status surface with a method `clear_quarantine(collector_name)`. The HTTP endpoint lands in STAGE-001-010 (`POST /api/collectors/{name}/retry`). For this stage, the method exists on the scheduler and is unit-tested.
6. **Tests**
   - **Group serialization:** two collectors in the same group with overlapping run times — verify they never overlap (using a synchronization primitive that fails the test if it does).
   - **Group skip-on-busy:** a slow collector holds the group; a fast collector's tick is skipped; metric incremented.
   - **Failure budget:** a collector that fails on every tick enters quarantine after exactly N failures.
   - **Quarantine persistence:** restart the scheduler; quarantined collector remains quarantined.
   - **Recovery:** clear_quarantine resets and the collector starts ticking again.
   - **Reset on success:** a single success resets the counter.

## Decisions in this stage

- **`quarantine_after` default = 5.** Documented; per-plugin overrides supported.
- **Group skip vs queue tradeoff** — we *queue* ticks but enforce a deadline equal to `interval / 2` so a stalled group doesn't grow an unbounded backlog. After deadline, ticks are *skipped* and counted.
