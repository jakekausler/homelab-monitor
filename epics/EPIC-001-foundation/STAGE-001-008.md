# STAGE-001-008: Concurrency groups + failure budget + quarantine

## Status: Complete

## Overview

Add the two pieces of behavioral discipline that make the scheduler safe under load: **concurrency groups** prevent thundering-herd against a single shared backend (e.g., the UDM controller, the HA API, the Synology DSM), and **failure budgets with quarantine** prevent a broken plugin from running every tick forever and flooding alerts.

After this stage, plugins in the same `concurrency_group` are serialized, and a plugin that fails N consecutive times is quarantined (paused), an alert is emitted, and the dashboard offers a "retry now" action.

**Spec references:** §5.4 (concurrency model — `concurrency_group`), §5.7 (per-collector failure budget), §3.1 (collector scheduler).

## Concrete deliverables

1. **Concurrency-group enforcement** in `scheduler.py`
   - The scheduler maintains an `asyncio.Lock` per `concurrency_group`.
   - Before invoking `_run_with_kind`, the scheduler acquires the group lock; if held, the tick is queued (with a deadline tied to `interval / 2` — if it can't run in that window, it's skipped and a `homelab_collector_skipped_total{reason="group_busy"}` metric is incremented).
   - **Group default:** if a collector doesn't declare a group, it gets its own (effectively unbounded concurrency relative to others).
   - **Group ordering:** within a group, ticks run in FIFO order.
2. **Failure budget tracker** at `apps/monitor/homelab_monitor/kernel/scheduler/failure_budget.py`
   - Per-collector counter of consecutive failures.
   - Quarantine threshold default: 5 consecutive failures (configurable per collector via `quarantine_after`).
   - Cooldown after quarantine: collector is paused; ticks are NOT scheduled while quarantined.
   - Successful run resets counter to 0.
3. **Quarantine state** persisted in `collectors` table:
   - Columns added (additive migration `0002_collector_quarantine.py`): `quarantined_at` (nullable ISO timestamp), `quarantine_reason` (nullable text), `consecutive_failures` (int default 0).
   - State survives restart so we don't accidentally hammer a known-broken backend on boot.
4. **Quarantine alert**
   - When a collector enters quarantine, the scheduler emits a synthetic alert via the (future) alert ingestor — for this stage, since the alert ingestor isn't built until STAGE-001-013, **the quarantine event is logged at WARNING level + recorded as a row in `audit_log`**, and the alert emission is wired in STAGE-001-013 (a TODO comment with stage reference is acceptable).
5. **Retry-now API** — extend the (placeholder) plugin status surface with a method `clear_quarantine(collector_name)`. The HTTP endpoint lands in STAGE-001-010 (`POST /api/collectors/{name}/retry`). For this stage, the method exists on the scheduler and is unit-tested.
6. **Tests**
   - **Group serialization:** two collectors in the same group with overlapping run times — verify they never overlap (using a synchronization primitive that fails the test if it does).
   - **Group skip-on-busy:** a slow collector holds the group; a fast collector's tick is skipped; metric incremented.
   - **Failure budget:** a collector that fails on every tick enters quarantine after exactly N failures.
   - **Quarantine persistence:** restart the scheduler; quarantined collector remains quarantined.
   - **Recovery:** clear_quarantine resets and the collector starts ticking again.
   - **Reset on success:** a single success resets the counter.

## Decisions in this stage

- **`quarantine_after` default = 5.** Documented; per-plugin overrides supported.
- **Group skip vs queue tradeoff** — we *queue* ticks but enforce a deadline equal to `interval / 2` so a stalled group doesn't grow an unbounded backlog. After deadline, ticks are *skipped* and counted.

- **Session Notes:**
  - **D1 Lock instantiation + 'default' string:** Lazy `self._group_locks.setdefault(group_key, asyncio.Lock())` in `_tick`. Magic-string interpretation: `group_key = c.name if c.concurrency_group == "default" else c.concurrency_group`. STAGE-007 BaseCollector default of "default" stays as-is (zero API churn); the scheduler resolves it to the per-collector name so collectors that don't override get their own lock. Document magic-string in BaseCollector docstring.
  - **D2 Skip-on-busy:** `asyncio.wait_for(lock.acquire(), timeout=interval/2)`. asyncio.Lock provides FIFO waiter queue for free; wait_for adds the deadline. On TimeoutError, emit new 6th scheduler self-metric `homelab_collector_skipped_total{name, reason}` with reason="group_busy" and return without running the tick. CPython 3.11+ properly removes the waiter on timeout cancellation.
  - **D3 Counter persistence policy:** In-memory authoritative `_consecutive_failures: dict[str, int]`. Persist on failure (UPDATE counter), quarantine entry (UPDATE counter + quarantined_at + reason + audit), and quarantine clear (UPDATE all to NULL/0 + audit). NEVER persist on success — counter resets in-memory only. Cuts steady-state DB writes by ~99%; the only material outcome (quarantined-vs-not) is always durable. On restart, in-memory counter loads from persisted DB value (slightly stale, but errs over-conservative toward earlier quarantine).
  - **D4 Orchestration:** New module `apps/monitor/homelab_monitor/kernel/scheduler/failure_budget.py` with `FailureBudget` class. Methods: `__init__(repo, clock, default_threshold=5)`, `load_state()`, `is_quarantined(name)`, `consecutive_failures(name)`, `record_success(name)`, `record_failure(name, reason, threshold=None)`, `clear_quarantine(name, by="operator")`. Quarantine gate at TOP of `Scheduler._tick`: `if self._failure_budget.is_quarantined(c.name): return` (also emit `skipped_total{name, reason="quarantined"}` for visibility). `record_success` resets counter but does NOT auto-clear quarantine — only `clear_quarantine` does. `clock` injected for testability. Per-collector `quarantine_after` override read from `LoadedCollector.config.quarantine_after` (new optional field on `CollectorConfig`).
  - **D5 Audit helper:** New public helper `apps/monitor/homelab_monitor/kernel/db/audit.py` exporting `insert_audit(conn, *, who, what, before_json, after_json, ip=None, when=None)`. failure_budget.py imports and uses it inside `async with repo.transaction() as conn:` for atomicity (UPDATE collectors + audit_log INSERT in same transaction). Do NOT modify secrets/repository.py's private `_insert_audit` in this stage (STAGE-005 scope); add a TODO referencing the new module for future consolidation. Audit event names: `collector.quarantine_entered`, `collector.quarantine_cleared`. who="scheduler" for system-initiated quarantine; who=by-parameter for manual clear (default "operator", STAGE-001-010 will pass real values).
  - **D6 Migration:** New file `apps/monitor/homelab_monitor/kernel/db/migrations/0003_collector_quarantine.py` (NOT 0002 — stage file's "0002_collector_quarantine.py" is wrong; 0002_secrets_columns.py already exists). Adds 3 columns to collectors: `quarantined_at TEXT NULL`, `quarantine_reason TEXT NULL`, `consecutive_failures INTEGER NOT NULL DEFAULT 0`. Adds `CREATE UNIQUE INDEX ix_collectors_name_unique ON collectors(name)`. up() runs `SELECT name, COUNT(*) FROM collectors GROUP BY name HAVING COUNT(*) > 1` defensively; raises if duplicates found before adding the index. Migration test fragility: existing tests use `pending[-1] == "0001"` style (robust); no breakage expected.
  - **Forward-compat verified:**
    - STAGE-001-009 subprocess plugins ALSO have `concurrency_group` and ALSO need failure budget. The implementation is RunKind-agnostic by design: group locks keyed by name, FailureBudget keys by collector name. STAGE-009 plugs in via the same `_tick` flow.
    - STAGE-001-010 `POST /api/collectors/{name}/retry` calls `Scheduler.clear_quarantine(name)` which delegates to `self._failure_budget.clear_quarantine(name, by="operator")`.
    - STAGE-001-013 alert dispatch: TODO comment in `record_failure` at the quarantine-transition site (`# TODO(STAGE-013): emit alert here`).
  - **Additional flags for Build:**
    - `CollectorConfig.quarantine_after: int | None = None` field added (resolves to default_threshold=5 if None).
    - 6th self-metric `homelab_collector_skipped_total{name, reason}` registered alongside the 5 existing scheduler-owned metrics. Reasons: "group_busy", "quarantined". Future stages may add reasons.
    - `Scheduler.clear_quarantine(name)` public method, thin delegation to FailureBudget.
    - Audit event names follow `collector.<verb>_<state>` namespace for future events.
