# STAGE-001-004: SQLite + Alembic + first migration

## Status: Build

## Overview

Stand up the operational database. Add SQLite via SQLAlchemy Core (NOT the ORM ‚Äî see spec ¬ß6.1) behind a narrow `SqliteRepository` facade, configure Alembic for migrations, and produce the first migration that creates the *initial* set of tables: `users`, `sessions`, `audit_log`, `api_tokens`, **plus minimal-schema stubs** for the tables future stages will use so additive migrations stay focused: `targets`, `collectors`, `crons`, `heartbeats_state`, `alerts`, `alert_outcomes`, `runbooks`, `runbook_runs`, `secrets`, `channels`, `routing_rules`, `digest_configs`, `maintenance_windows`, `suggestions`, `tool_scorecards`.

**"Minimal-schema stub" rule:** the initial migration creates each future table with ONLY its identity, name/key columns, and `created_at`. Behavioral or feature-specific columns are added by the stage that implements that behavior. Examples: `collectors` has `id, name, config, created_at` here; `quarantined_at`, `quarantine_reason`, `consecutive_failures` are added by STAGE-001-008's `0002_collector_quarantine.py` when the quarantine logic lands. This way every additive migration ships alongside the code that uses it, while still avoiding "table doesn't exist yet" headaches in tests that span subsystems.

**Spec references:** ¬ß6.1 (SQLite tables), ¬ß6.5 (config: `HOMELAB_MONITOR_AUTO_MIGRATE`), ¬ß16 (UTC timestamps).

## Concrete deliverables

1. **DB connection layer** at `apps/monitor/homelab_monitor/kernel/db/`
   - `engine.py` ‚Äî creates an `AsyncEngine` for SQLite (URL: `sqlite+aiosqlite:///{data_dir}/homelab-monitor.db`); WAL pragma applied on connect; foreign keys ON; busy timeout 5000ms.
   - `repository.py` ‚Äî abstract `SqliteRepository` facade. No raw cursor leakage to callers; methods return dataclasses or pydantic models.
   - `migrations.py` ‚Äî wraps Alembic invocations: `migrate_to_head()`, `current_revision()`, `pending_revisions()`.
2. **Alembic** at `apps/monitor/alembic/`
   - `alembic.ini` checked in, with `script_location = alembic` and `sqlalchemy.url = ${HOMELAB_MONITOR_DB_URL}`.
   - `env.py` configured to read the DB URL from env (so tests use in-memory or temp-file DBs cleanly).
   - First migration: `0001_initial_schema.py` creating all tables described above with the columns from spec ¬ß6.1.
3. **Startup integration**
   - On app boot (a stub `main()` for now ‚Äî actual FastAPI shell lands in STAGE-001-010), the migration runner is invoked. Behavior controlled by env `HOMELAB_MONITOR_AUTO_MIGRATE` (default `true`). When `false`, refuses to start if pending migrations exist; instructs user to run `hm migrate`.
4. **CLI** ‚Äî extend `hm` with: `hm migrate` (apply pending), `hm migrate status` (show current vs head revisions), `hm migrate history` (list revisions).
5. **Tests** at `apps/monitor/tests/unit/db/`
   - Migrate up to head from empty DB; verify all tables exist with expected columns.
   - Round-trip: `alembic downgrade -1` then `alembic upgrade head`.
   - Repository facade smoke tests with a fresh in-memory DB.
6. **Audit log helper** ‚Äî small utility `audit.write(who, what, before, after)` that the rest of the codebase uses for state-changing operations. Lands in the kernel; later stages route their writes through it.

## Schema notes

- **All timestamps stored as ISO-8601 UTC strings** (`TEXT` columns). Rationale: sqlite has no native timestamptz; storing as ISO strings avoids the timezone-loss problem with the `TIMESTAMP` type in many SQLite setups. Repository helpers convert to/from `datetime` objects.
- **Primary keys** ‚Äî UUIDv7 strings everywhere except `users.id` (INTEGER autoincrement is fine for a tiny user table). UUIDv7 chosen so primary keys carry creation-time ordering (helpful for the audit log and runbook runs).
- **Indexes** ‚Äî at least the indexes named in spec ¬ß6.1 (alerts.fingerprint, alerts(source_tool, opened_at), runbook_runs(runbook_id, started_at), targets(kind, name), crons(host, command)).
- **Foreign keys** declared on all relations; `PRAGMA foreign_keys = ON` enforced on every connection.

### Locked decisions (Design ‚Äî 2026-05-05)

**Schema strategy:** Option (a) ‚Äî all 19 tables created in `0001_initial_schema.py`. Behavioral columns added by later stages via additive migrations (e.g., STAGE-001-008 ‚Üí `0002_collector_quarantine.py`).

**Tables fully defined per spec ¬ß6.1:** `users`, `sessions`, `audit_log`, `api_tokens`.

**Tables created as minimal stubs (id, name/key, created_at only):** `targets`, `collectors`, `crons`, `heartbeats_state`, `alerts`, `alert_outcomes`, `runbooks`, `runbook_runs`, `secrets`, `channels`, `routing_rules`, `digest_configs`, `maintenance_windows`, `suggestions`, `tool_scorecards`.

**Engine:** Async via `sqlalchemy.ext.asyncio.create_async_engine` + `aiosqlite`. Alembic env.py uses `connection.run_sync(...)` inside an async wrapper (the standard async-Alembic pattern).

**Pragmas applied at connect:** `journal_mode=WAL`, `foreign_keys=ON`, `busy_timeout=5000`. Wired via `@event.listens_for(engine.sync_engine, "connect")` listener.

**Timestamps:** All TEXT columns storing ISO-8601 UTC strings. Helper `kernel/db/time.py` provides `utc_now_iso() -> str`.

**Primary keys:** UUIDv7 (str) on every table except `users.id` (INTEGER autoincrement, since bcrypt-keyed sessions need a stable lookup target). UUIDv7 generated via the `uuid-utils` PyPI package ‚Äî adds as a runtime dep in `apps/monitor/pyproject.toml`.

**Repository facade:** Thin generic `SqliteRepository` at `kernel/db/repository.py` with `execute(stmt) -> Result`, `fetch_one(stmt)`, `fetch_all(stmt)`, and `transaction()` async-context-manager. Each future stage adds its own narrow query helpers as plain functions in its own module ‚Äî no per-table classes upfront.

**Audit helper:** `kernel/db/audit.py` exposes `async def audit_write(repo, who, what, before=None, after=None, ip=None) -> None`. Inserts one row into `audit_log`.

**Auto-migrate env var:** `HOMELAB_MONITOR_AUTO_MIGRATE` (default `"true"`). On boot, kernel checks Alembic head vs current DB. If equal, proceed. If pending and env is `"true"`, run `upgrade head` automatically. If pending and env is `"false"`, raise `MigrationsPendingError` and refuse to start.

**CLI subcommands** added to `homelab_monitor.cli.main`:
- `hm migrate` ‚Äî runs `alembic upgrade head`
- `hm migrate status` ‚Äî prints current revision + pending revisions
- `hm migrate history` ‚Äî prints `alembic history`
