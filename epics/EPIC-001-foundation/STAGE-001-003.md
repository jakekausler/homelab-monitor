# STAGE-001-003: CI + Code Review Graph + Dependabot

## Status: Refinement

## Overview

Wire backend and frontend verify pipelines into GitHub Actions; install and initialize Code Review Graph (CRG) so MCP-based code review is available from the very next stage; configure Dependabot (or Renovate — Design phase decides) for upstream dependency updates. After this stage, every PR runs the full verify pipeline and CRG provides blast-radius/semantic-search MCP tools to Claude Code throughout the project.

**Spec references:** §10.5 (CI / release), `project_repo_tooling.md` (CRG details).

## Concrete deliverables

1. **GitHub Actions workflows** (`.github/workflows/`)
   - `ci.yml` — runs on `pull_request` and on `push` to non-main branches:
     - Job `backend`: checkout → set up Python 3.12 → install uv (or chosen tool) → `make verify-backend` (ruff + black + pyright + pytest with coverage)
     - Job `frontend`: checkout → set up Node 20 + pnpm → `pnpm --filter ui run verify`
     - Job `crg-build`: checkout → install code-review-graph → `code-review-graph build` → upload graph artifact (so other jobs/reviewers can fetch it)
     - All jobs run in parallel where possible; PR is required to be green.
   - `release.yml` — runs on tags `v*`:
     - Build multi-arch container images for the backend (amd64 + arm64) and push to GHCR
     - Build the UI bundle, upload as a release asset (or include in the backend image at `/app/ui/`)
     - Auto-generate release notes from `CHANGELOG.md`
   - `codeql.yml` — GitHub's default CodeQL on the Python+TypeScript matrix.
2. **Dependabot config** at `.github/dependabot.yml`
   - Ecosystems: `pip` (root and `apps/monitor/`), `npm` (root and `apps/ui/`), `github-actions`, `docker` (when Dockerfiles land later)
   - Schedule: weekly
   - Group minor + patch updates per ecosystem to reduce PR noise
3. **Code Review Graph**
   - Add to backend dev deps: `code-review-graph` (or instruct via README + a `make crg-init` target).
   - `make crg-init` runs `pip install code-review-graph && code-review-graph install && code-review-graph build`.
   - `crg-daemon` is documented in the README as the recommended local mode (auto-rebuilds graph on file change).
   - `.code-review-graph/` confirmed gitignored.
   - CI's `crg-build` job validates that `code-review-graph build` succeeds on the current source.
4. **Branch protection** (via the GitHub UI; documented in `docs/repo-setup.md`)
   - `main` requires: status checks `backend`, `frontend`, `crg-build`, `codeql`; up-to-date branch; signed commits not required (homelab project).
5. **README updates**
   - Sections: "Getting started" with `make crg-init` after first clone; "CI" describing the pipeline; "Releases" describing the tag flow.

## Decisions in this stage

- Dependabot vs Renovate: **recommend Dependabot** for simplicity; revisit if grouping isn't sufficient.
- Whether to gate merges on coverage drops vs only failures: **recommend hard 100% kernel gate, no soft thresholds for plugins yet**.
- Whether release.yml runs in this stage or a later stage: **the workflow file lands here; it activates only on tag pushes, so the project doesn't actually publish anything yet**.

## Out of scope for this stage

- Production publishing of container images (no tags created yet)
- Self-hosted runner (use GitHub-hosted)
- Coverage badges (cosmetic, deferred)
- Lockfile auto-update beyond Dependabot's defaults

The unplanned mini-stage already wired `make crg-init` and `make setup`'s daemon-start hook locally — CI builds the graph fresh per run. CRG must be in the required-checks list to gate merges.

## Build Phase

- **Components Created:**
  - `.github/workflows/ci.yml` — three parallel jobs (`backend`, `frontend`, `crg-build`) running on `pull_request` + `push` to non-main branches.
  - `.github/workflows/release.yml` — dormant scaffolding for `v*` tags. Builds multi-arch Docker images to GHCR, generates GH release notes. Inert until first tag (Dockerfile lands in STAGE-001-015).
