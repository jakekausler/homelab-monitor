# STAGE-001-019: Karma + kthxbye

## Status: Not Started

## Overview

Add Karma (alert lifecycle UI) and kthxbye (auto-extends silences while alerts fire). Embed Karma into the dashboard's Alerts screen via iframe. Wire the Alerts sidebar item live. After this stage, the user can ack and snooze alerts from the dashboard with the same ergonomics as standalone Karma — and acks behave correctly (silence stays in place until the underlying alert resolves, courtesy of kthxbye).

## Concrete deliverables

1. **Karma sidecar** in compose
   - Image pinned (e.g., `ghcr.io/prymitive/karma:v0.x.x`).
   - Config at `deploy/karma/karma.yml`:
     - `alertmanager.servers: [- name: default, uri: http://alertmanager:9093]`
     - `authentication.basicAuth.users` is NOT used; we put Karma behind our own dashboard's auth via reverse proxy on the monitor (see Embedding below).
     - `ui.refresh: 10s`, default columns, group_by labels.
2. **kthxbye sidecar**
   - Image pinned. Config: AM URL, default silence duration to extend by, max-extend cap. Runs as a long-lived process polling AM.
3. **Embedding Karma in our UI**
   - Karma is hosted on the docker network only; not exposed to the LAN directly.
   - The monitor's FastAPI proxies `/karma/*` → Karma's `/...` after authenticating the request with our cookie session (so Karma is only reachable through our auth).
   - The Alerts screen renders an `<iframe src="/karma/" />` with appropriate sandbox attributes.
4. **Alerts screen** at `apps/ui/src/routes/alerts.tsx`
   - Layout: Karma iframe occupies the main pane.
   - Right-side drawer (collapsible) shows "Selected alert detail" populated from our enriched data — but the iframe-to-parent communication is via `postMessage` from a small script we inject at proxy time (or, simpler: clicking an alert in Karma updates the URL hash; the parent reads hash changes via a `useEffect`). Initial implementation can defer the drawer to a later refinement; the iframe alone delivers the lifecycle UX.
   - Sidebar "Alerts" item enabled and routed to this screen.
5. **kthxbye verification**
   - Acking an alert in Karma creates a silence in AM. While the alert keeps firing, kthxbye extends the silence so re-pages don't happen.
   - Once the alert resolves, kthxbye stops extending; the silence times out naturally.
6. **Tests**
   - Integration: trigger a known alert; from a programmatic Karma-like API call (Karma exposes its alert state at `/alerts.json`), ack the alert; AM shows a silence; alert keeps firing; silence is auto-extended by kthxbye after a short window.
   - Playwright: log into our dashboard, open Alerts, see at least one alert in the iframe (the test pre-trips one), click ack via Karma's UI, see the alert switch to "silenced" status.

## Out of scope for this stage

## Decisions in this stage

- **Iframe vs full Karma rebuild** — iframe. Karma is feature-rich and battle-tested; rebuilding it would be wasted effort. The iframe approach satisfies the spec's lifecycle requirements out of the box.
- **Karma reverse-proxied through our auth** — yes. Defense in depth and consistent UX.
- **kthxbye config defaults** — extend silence by 5m at a time, cap at 24h.
