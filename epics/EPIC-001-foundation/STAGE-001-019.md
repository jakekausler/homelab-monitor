# STAGE-001-019: Karma + kthxbye

## Status: Complete

## Overview

Add Karma (alert lifecycle UI) and kthxbye (auto-extends silences while alerts fire). Embed Karma into the dashboard's Alerts screen via iframe. Wire the Alerts sidebar item live. After this stage, the user can ack and snooze alerts from the dashboard with the same ergonomics as standalone Karma — and acks behave correctly (silence stays in place until the underlying alert resolves, courtesy of kthxbye).

## Concrete deliverables

1. **Karma sidecar** in compose
   - Image pinned (e.g., `ghcr.io/prymitive/karma:v0.x.x`).
   - Config at `deploy/karma/karma.yml`:
     - `alertmanager.servers: [- name: default, uri: http://alertmanager:9093]`
     - `authentication.basicAuth.users` is NOT used; we put Karma behind our own dashboard's auth via reverse proxy on the monitor (see Embedding below).
     - `ui.refresh: 10s`, default columns, group_by labels.
2. **kthxbye sidecar**
   - Image pinned. Config: AM URL, default silence duration to extend by, max-extend cap. Runs as a long-lived process polling AM.
3. **Embedding Karma in our UI**
   - Karma is hosted on the docker network only; not exposed to the LAN directly.
   - The monitor's FastAPI proxies `/karma/*` → Karma's `/...` after authenticating the request with our cookie session (so Karma is only reachable through our auth).
   - The Alerts screen renders an `<iframe src="/karma/" />` with appropriate sandbox attributes.
4. **Alerts screen** at `apps/ui/src/routes/alerts.tsx`
   - Layout: Karma iframe occupies the main pane.
   - Right-side drawer (collapsible) shows "Selected alert detail" populated from our enriched data — but the iframe-to-parent communication is via `postMessage` from a small script we inject at proxy time (or, simpler: clicking an alert in Karma updates the URL hash; the parent reads hash changes via a `useEffect`). Initial implementation can defer the drawer to a later refinement; the iframe alone delivers the lifecycle UX.
   - Sidebar "Alerts" item enabled and routed to this screen.
5. **kthxbye verification**
   - Acking an alert in Karma creates a silence in AM. While the alert keeps firing, kthxbye extends the silence so re-pages don't happen.
   - Once the alert resolves, kthxbye stops extending; the silence times out naturally.
6. **Tests**
   - Integration: trigger a known alert; from a programmatic Karma-like API call (Karma exposes its alert state at `/alerts.json`), ack the alert; AM shows a silence; alert keeps firing; silence is auto-extended by kthxbye after a short window.
   - Playwright: log into our dashboard, open Alerts, see at least one alert in the iframe (the test pre-trips one), click ack via Karma's UI, see the alert switch to "silenced" status.

## Decisions in this stage

- **Iframe vs full Karma rebuild** — iframe. Karma is feature-rich and battle-tested; rebuilding it would be wasted effort. The iframe approach satisfies the spec's lifecycle requirements out of the box.
- **Karma reverse-proxied through our auth** — yes. Defense in depth and consistent UX.
- **kthxbye config defaults** — extend silence by 5m at a time, cap at 24h.

## Design Phase

- **Approaches to present:**
  - (a) iframe + reverse proxy (recommended)
  - (b) consume Karma's API and rebuild the UI ourselves (waste)
  - (c) drop Karma; build a simple ack/snooze ourselves (loses dedup & maintenance feature parity)
- **UI Options Presented:**
  - Two iframe layouts: full-bleed; iframe + right drawer (the drawer can be empty initially).
- **User Choice:**
  - **(A) Proxy implementation:** FastAPI passthrough router (catch-all path with httpx.AsyncClient + StreamingResponse, all methods, header allow-list). Stays in existing auth/middleware stack.
  - **(B) Auth model:** `require_session()` + CSRF-exempt on the proxy route + Origin/Referer same-origin assertion. Karma's iframe POSTs (ack/silence) cannot carry our CSRF token; bypass is bounded by same-origin enforcement and private-network threat model.
  - **(C) iframe sandbox:** `sandbox="allow-scripts allow-same-origin allow-forms"` — minimum needed for Karma's same-origin React SPA + XHR-with-cookies + ack/silence form submissions.
  - **(D) CSP scope:** Minimal — add `Content-Security-Policy: frame-ancestors 'self'` on monitor responses (modern X-Frame-Options replacement). Defer comprehensive CSP to a future hardening epic.
  - **(E) kthxbye version:** Accept v0.16 (pinned tag `ghcr.io/prymitive/kthxbye:v0.16`). Last release 2023-01-13 — small Go daemon, plausibly "done" rather than abandoned.
  - **(F) Test strategy:** Pytest unit tests for proxy (auth gating, path validation, header passthrough, Origin check, error mapping, CSP injection) + ONE `@pytest.mark.slow` integration test for ack→kthxbye-extends flow. User does manual desktop + mobile UI inspection in Refinement (binding for frontend stages). NO Playwright in this stage.
  - **(G) Karma port:** Override Karma's default 8080 → **8081** via `listen.port: 8081` (and `listen.prefix: /api/karma/` for sub-path hosting). Avoids conflict with pihole-unbound on host:8080. Internal-only on `homelab-monitor-net` (no host-port binding).
- **Seed Data Agreed:** existing test rig fixtures
- **Session Notes:** Brainstormer (Opus) presented options for 6 architectural decisions; user selected the recommended package across the board. Mid-design, user flagged that Karma's default port 8080 conflicts with the host's pihole-unbound (network_mode: host) — adjusted to listen.port: 8081. Existing FastAPI proxy patterns (metrics.py, logs.py) are JSON-shaped single-endpoint proxies; Karma proxy is a NEW pattern (transparent HTTP relay for HTML+JS+CSS+fonts+XHR). No WebSockets needed (Karma uses HTTP polling at ui.refresh: 10s). The "Alerts" sidebar item already exists at apps/ui/src/components/SidebarNav.tsx as a disabled placeholder with disabledNote='Coming soon — STAGE-001-019' — conversion is trivial. App has NO CSP middleware today; we add the first one (frame-ancestors only). Per-alert deep-link / postMessage iframe→parent communication is NOT a Karma feature — confirmed via upstream docs review; the right-side drawer is correctly deferred to a later UI stage.

### Refinement-phase commits (exception per CLAUDE.md: external-system tests need ground-truth fixes during Refinement)

1. **kthxbye flag-validation crash** in both `deploy/compose/docker-compose.yml` and `docker-compose.test.yml`: `extend-by` must be greater than `extend-if-expiring-in`. Prod values were 5m/6m (< invalid); test values were 10s/15s (< invalid). Swapped to 10m/6m (prod) and 30s/15s (test). e2e-tester caught this immediately when bringing up the test rig.
2. **Karma healthcheck** removed from both compose files — Karma's image is distroless (no `wget`/`curl`/`sh`), so the `CMD-SHELL wget ...` healthcheck stuck the container at "starting" forever. No service `depends_on` Karma's health, so removing the healthcheck is safe; Karma's readiness is observed via the monitor's reverse-proxy returning 502 if Karma is briefly down.
3. **Origin/Referer same-origin check made reverse-proxy aware** in `apps/monitor/homelab_monitor/kernel/api/routers/karma.py::_verify_origin`. Old: compared Origin to `f"{request.url.scheme}://{request.url.netloc}"`. New: uses `Host` header (forwarded verbatim by HTTP/1.1 proxies) and `X-Forwarded-Proto` (set by nginx, traefik, etc.). Falls back to `request.url.{scheme,netloc}` only when those headers are absent. This fixes both the dev-time vite-proxy issue AND the production-time nginx issue (the latter was a real STAGE-019 design defect not noticed in Build).
4. **Vite dev proxy `changeOrigin: false`** in `apps/ui/vite.config.ts` — preserves browser's original Origin/Host so the backend's same-origin check sees the dev-server origin rather than the proxy target. In production the monitor serves the UI itself; this dev setup imitates that.
5. **Alerts page React error #31 fix**: added `errorComponent: RootErrorComponent` to `apps/ui/src/routes/__root.tsx` AND `defaultErrorComponent: DefaultErrorComponent` to the router in `apps/ui/src/router.tsx`. Both safely coerce unknown errors to a string so a raw HttpProblem object can never reach React as a child.
6. **`apps/monitor/tests/test_scheduler.py::test_process_run_kind`**: bumped `asyncio.sleep(0.6)` → `asyncio.sleep(1.5)` because forkserver subprocess startup was non-deterministic on this host. Pre-existing failure (failed on baseline `cfebf8a` too); fixed during STAGE-019 Refinement per CLAUDE.md "fix pre-existing failures that block your verification".
7. **`.dockerignore`** added `!apps/monitor/alembic/` to re-include list — surfaced when attempting prod compose for Refinement testing; the prod monitor image was missing alembic and aborting on lifespan migrations. Pre-existing pre-STAGE-019 bug; deferred actual prod-stack validation to STAGE-021's `make dev-prod` deliverable but the dockerignore fix lands here so the rest of the world's "build the image" stops failing.
8. **8 new UI tests** for `RootErrorComponent` + `DefaultErrorComponent` (`__root.test.tsx` new file + extended `router.test.tsx`) — restored UI branch coverage above 75% threshold after the new error-component branches dropped it to 74.23%.
