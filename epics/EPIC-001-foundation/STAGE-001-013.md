# STAGE-001-013: Alert ingestor + first `inproc-dashboard` channel

## Status: Design (Complete)

## Overview

Build the alert ingestion pipeline. Accept Alertmanager v2 webhook payloads at `/api/alerts/ingest`, store alerts in SQLite (with deduplication by fingerprint), record outcomes, and dispatch them through a per-channel adapter framework. Implement the first dispatcher channel: `inproc-dashboard`, which fans out to the SSE pub/sub broker so the dashboard can show live alerts.

This stage also wires the **scheduler quarantine alert** TODO from STAGE-001-008 — when a collector is quarantined, the scheduler now emits a synthetic alert via the new ingestor.

After this stage, posting an Alertmanager-shaped webhook to the API produces a row, dedupes correctly, and the in-process channel emits the alert on SSE.

**Spec references:** §3.1 (alert ingestor + alert dispatcher), §4.4 (alert lifecycle flow), §6.1 (alerts and alert_outcomes tables), §8 (notifications + severity).

## Concrete deliverables

1. **Alert ingestor endpoint** at `apps/monitor/homelab_monitor/kernel/api/alerts.py`
   - `POST /api/alerts/ingest` — accepts Alertmanager v2 webhook payload (`{version, groupKey, status, receiver, alerts: [...]}`).
   - Authentication: API token with scope `alerts:ingest:write`. Cookie sessions ALSO accepted for dashboard-driven test posts.
   - For each alert in the payload:
     - Compute `fingerprint` (use Alertmanager's if present, else SHA-256 of sorted-labels JSON).
     - `source_tool` derived from labels (e.g., `vmalert-metrics`, `vmalert-logs`, `karma-manual`); fallback to receiver name.
     - Severity from `labels.severity`; default `warning` if missing.
     - On `status=firing`: insert or update existing row (set `opened_at` if first time, `last_seen_at` on every poke); dispatch.
     - On `status=resolved`: update existing row's `resolved_at`, dispatch a resolved notification.
   - Returns 202 Accepted with the count of alerts processed.
2. **Dispatcher** at `apps/monitor/homelab_monitor/kernel/dispatch/`
   - `dispatcher.py` — accepts a normalized `Alert` and routes it through `routing_rules` to one or more channels. For this stage, no routing rules logic — every alert goes to `inproc-dashboard` regardless of severity. Routing rules logic lands later (EPIC-012).
   - `channels/base.py` — `Channel` Protocol: `async def deliver(alert: Alert) -> DeliveryResult`.
   - `channels/inproc_dashboard.py` — implements `Channel`; publishes to the SSE broker on a topic `alerts.firing` / `alerts.resolved`.
3. **Outcome tracking**
   - `alert_outcomes` rows are written when:
     - dashboard ack (a later stage's UI; for this stage, `POST /api/alerts/{id}/ack` exists and writes `outcome=acked`)
     - dashboard dismiss (`POST /api/alerts/{id}/dismiss`, `outcome=dismissed`)
     - auto-fix run (later epic, but the column shape is here)
     - escalation (currently undefined; the column accepts the value)
   - These per-stage rows feed the tool-effectiveness analyzer in EPIC-010.
4. **Listing endpoints**
   - `GET /api/alerts` — paginated list with filters: status, severity, source_tool, fingerprint, age. Returns `{items: [...], total, next_cursor}`.
   - `GET /api/alerts/{id}` — single alert detail with outcome history and full payload.
5. **Quarantine integration**
   - Replace the TODO from STAGE-001-008. When a collector enters quarantine, the scheduler now constructs an `Alert` with: `alertname=collector_quarantined`, `labels={collector_name, reason}`, `severity=warning`, `source_tool=scheduler` and dispatches via the ingestor's internal API (NOT a self-HTTP call).
6. **Dashboard SSE event extension**
   - Add `alert.firing` and `alert.resolved` event kinds to the SSE broker. The dashboard subscribes in STAGE-001-014.
7. **Tests**
   - POST a synthetic Alertmanager payload; row created; SSE event delivered.
   - POST same payload again; deduplicates (single row, `last_seen_at` updated, `opened_at` unchanged).
   - POST resolved version; row updated, resolved event delivered.
   - Quarantine flow: induce a collector to fail repeatedly; alert appears in `/api/alerts`.
   - Ack endpoint writes outcome row.
   - Dismiss endpoint writes outcome row.
   - Logs at info level never include the full alert payload (assertion).

## Decisions in this stage

- **Fingerprint format** — prefer Alertmanager's own fingerprint when present in the payload; fall back to a SHA-256 of sorted-label JSON.
- **Severity default** — `warning` when missing, with a logged warning so misconfigured rules surface.
- **Authenticated by default** — anonymous webhook posting is rejected. Alertmanager will use an API token with `alerts:ingest:write` scope.

### Design phase decisions (locked 2026-05-07)

**D1 — SSE broker for alert events: `BaseEvent` Protocol with single broker.**

Define `BaseEvent` Protocol in `kernel/events.py` with `kind: str` and `model_dump() -> dict[str, Any]`. The existing `SchedulerTickEvent` and new `AlertFiringEvent`/`AlertResolvedEvent` all conform. `SseBroker.publish(event: BaseEvent)` widens to accept any conforming event. Topic filtering happens via `event.kind` in the SSE consumer (STAGE-014 dashboard). Single broker mandatory because STAGE-014 expects ONE `/api/events` connection that filters by topic.

**D2 — Scheduler → dispatcher integration: Constructor injection.**

`FailureBudget(repo, log, dispatcher: AlertDispatcher | None = None)`. Mirrors the existing SseBroker injection pattern in scheduler. `None` default keeps existing tests passing during migration. The `record_failure` method's quarantine TODO at `failure_budget.py:262-264` is replaced with `if self._dispatcher: await self._dispatcher.dispatch(quarantine_alert)`. Tests pass a `FakeAlertDispatcher` and assert on captured calls.

**D3 — Dispatcher framework shape: Static channel list at boot.**

`AlertDispatcher(channels=[InprocDashboardChannel(broker)])` constructed in app lifespan startup. `Channel` Protocol defines `kind: str` and `async def deliver(event: AlertFiringEvent | AlertResolvedEvent) -> None`. Per-channel delivery failures: log at WARNING with `channel.kind`, increment a counter, do NOT raise (one bad channel must not break others). DLQ/retry deferred to later epic. Future channels (HA push, Discord, SMTP) wired via app startup, not via DB-driven `channels` table (also deferred).

**D4 — Alert dedup semantics: Strict fingerprint while firing; new row on re-fire after resolve.**

While `alerts.resolved_at IS NULL`: matching fingerprint UPDATEs `last_seen_at` only (`opened_at` unchanged). On resolution payload: set `resolved_at`. On NEXT firing of the same fingerprint with the prior row already resolved: INSERT a new row. Each fire→resolve→fire cycle is operationally a new incident. Outcome history (acks, dismissals) attaches to its corresponding row, not aggregated across cycles.

Migration 0005 must add `last_seen_at TEXT NOT NULL` to the `alerts` table.

**D5 — Quarantine alert dedup: Same fingerprint scheme as D4.**

Quarantine alert fingerprint = `sha256("collector_quarantined:{collector_name}:{reason}")`. Re-quarantine while firing bumps `last_seen_at`. Operator clearing the collector emits a `resolved` dispatch (sub-task of this stage — added in `Scheduler.clear_quarantine()` or wherever quarantine state is cleared). Next quarantine after clear creates a new row via D4's resolve-then-fire path. D4 and D5 share one dedup story.

### Other locked specifics (from stage spec — confirmed in Design)

- **Auth posture:** `Depends(require_user_or_token({Scope.ALERTS_INGEST_WRITE}))`. Cookie path enforces CSRF (state-changing POST). Token path checks scope membership. `Scope.ALERTS_INGEST_WRITE` already exists in `kernel/auth/scopes.py`.
- **Status codes:** 202 Accepted on POST `/api/alerts/ingest`; 200 OK on GET `/api/alerts` and `/api/alerts/{id}`; 401 Unauthenticated; 403 InsufficientScope or CSRF mismatch; 422 on malformed Alertmanager payload.
- **Migration 0005:** additive. Adds to `alerts`: `source_tool TEXT, severity TEXT, status TEXT, opened_at TEXT, last_seen_at TEXT, resolved_at TEXT, ack_at TEXT, ack_by INTEGER, runbook_id TEXT, payload_json TEXT`. Adds to `alert_outcomes`: `outcome TEXT, decided_at TEXT, decided_by INTEGER`. Existing tables are empty stubs from STAGE-001-004; no backfill needed.
- **Quarantine wiring location:** `apps/monitor/homelab_monitor/kernel/scheduler/failure_budget.py:262-264` (the SCAFFOLDING comment). Plus `Scheduler.clear_quarantine()` for the resolve-on-clear path.
- **Logging discipline:** info-level log lines MUST NOT contain the full alert payload. Test assertion required (caplog).
- **Fingerprint computation:** prefer Alertmanager's `fingerprint` field when present in the payload; fall back to `sha256(json.dumps(sorted_labels))`.
- **Severity default:** `warning` when missing; emit a structured WARNING log at the same time noting the misconfigured rule.

### Test plan (Build phase)

- **Ingestor unit:** synthetic Alertmanager v2 payload → row created + SSE event emitted with `kind="alert.firing"`; same payload again → single row, `last_seen_at` bumped, `opened_at` unchanged; resolved variant → `resolved_at` set + SSE `kind="alert.resolved"`; resolved-then-refire → NEW row inserted.
- **Auth adversarial (per STAGE-011 lesson):** anonymous → 401; cookie auth without CSRF → 403 (CsrfMismatchProblem); token lacking `alerts:ingest:write` → 403 (InsufficientScopeProblem); valid token → 202; valid cookie + CSRF → 202.
- **Quarantine flow:** drive a collector through 3 consecutive failures → quarantine → alert appears via `GET /api/alerts`; clear quarantine via `Scheduler.clear_quarantine()` → SSE resolved event; re-quarantine after clear → NEW row in `alerts`.
- **Listing endpoint:** filters by status/severity/source_tool/fingerprint work; pagination cursor stable; total count accurate.
- **Privacy assertion:** caplog checks `record.getMessage()` for any structured field containing the full payload at INFO level — fail the test if found.
- **Outcome tracking:** POST ack/dismiss endpoints → `alert_outcomes` row created; outcome history surfaces in GET `/api/alerts/{id}`.
- **Channel hardening:** patch `InprocDashboardChannel.deliver` to raise → assert dispatcher logs WARNING, increments failure counter, does not propagate.
