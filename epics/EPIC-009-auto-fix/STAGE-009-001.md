# STAGE-009-001: Runbook schema & config-file contract

## Status: Not Started

## Overview

Lay the data foundation for the auto-fix subsystem: an additive alembic migration that fills in the behavioral columns on the stub `runbooks` and `runbook_runs` tables (which EPIC-001 created as `id`/`path`/`created_at` and `id`/`runbook_id`/`created_at` placeholders), plus a pydantic model for the **runbook-folder config file** (the machine-read manifest co-located with the runbook's markdown). Per LOCKED Decision 2, a runbook is a folder of markdown + a config file, and the **config file is authoritative** â€” the DB row is a registration/enablement record plus a content hash (for audit non-negotiable #4).

**Vertical slice:** after this stage, the schema and the config-file contract exist and validate; a runbook folder's config can be parsed into a typed model; the migration applies up/down cleanly. No orchestration yet â€” this is the contract everything else reads/writes.

## Cross-cutting standards (from EPIC-009 cross-stage acceptance criteria)

- `pyright --strict`, 100% branch coverage on the new schema + config-model code. No bare `Any`, no `# type: ignore`.
- The config model must encode the seven non-negotiables' inputs: match patterns (#1 allow-list), `risk_tag` defaulting to `risky` (#5 dry-run-by-default), `rate_limit_per_hour` + `cooldown_seconds` (#6), the scoped-capabilities block (#2 scope).
- Open-source-safe defaults: a new runbook config defaults to `risk_tag: risky`, `dry_run_required: true`, `enabled: false`, `auto_trigger: false` (Decision 4).

## Concrete deliverables

### 1. Additive migration

Add to `runbooks`: `alert_match_patterns` (JSON), `risk_tag` (`safe`/`risky`, default `risky`), `dry_run_required` (bool, default true), `rate_limit_per_hour` (int), `cooldown_seconds` (int), `enabled` (bool, default false), `auto_trigger` (bool, default false), `content_hash` (text). Add to `runbook_runs`: `alert_id`, `mode` (`dry`/`real`), `prompt`, `transcript_path`, `exit_code`, `started_at`, `ended_at`, `fixer_user`, `host`, `runbook_hash`. Migration applies up/down; tests cover both directions.

### 2. Runbook-folder config pydantic model

A pydantic model for the config file (YAML/TOML) declaring: `match_patterns` (regex vs alertname/labels), `risk_tag`, `dry_run_required`, `rate_limit_per_hour`, `cooldown_seconds`, and the **`scoped_capabilities`** block â€” `docker` (container + allowed actions), an optional `ssh` (`target_id` reference to an existing EPIC-017 target, READ-only), and an `egress` allow-list. Validation rejects malformed configs with a clear error. Decide + document the file-authoritative registry model (config = truth, DB = registration + hash).

### 3. Tests

Migration up/down; config-model validation (valid config parses; risky/disabled defaults applied; malformed config rejected; scoped-capabilities block parses docker + ssh-read + egress shapes).
