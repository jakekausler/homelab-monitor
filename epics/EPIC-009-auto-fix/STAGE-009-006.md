# STAGE-009-006: Dry-run mode + approval flow

## Status: Complete

## Overview

Implement the dry-run gate (non-negotiable #5): when a runbook's `risk_tag == risky` (the default for every new runbook), the orchestrator first runs `claude` in **plan-only** mode — a prompt asking for a plan with NO execution — stores the resulting plan, surfaces it, and requires **explicit user approval** before any real run is permitted. A risky runbook can NEVER bypass dry-run. This builds directly on the STAGE-009-005 orchestrator.

**Vertical slice:** after this stage, a risky runbook triggered (manually or by auto-trigger) produces a stored, surfaced plan and halts; only an explicit user approval transitions it to a real run; a non-risky (`safe`) runbook skips the plan gate. The "risky can never bypass dry-run" assertion is the headline test.

**Spec references:** §4.5 (the dry-run branch of the auto-fix flow: risky → plan → explicit ack → real run), §7.4 (#5 dry-run), §6.1 (`runbook_runs.mode` = `dry`/`real`). EPIC-009 LOCKED Decision 4 (risky-by-default). §7.2 (confirm-on-destructive on a real runbook run).

## Concrete deliverables

### 1. Plan-only invocation

When `risk_tag == risky` (or `dry_run_required`), the orchestrator runs `claude` with a plan-only prompt (no execution). Captures the plan as a `runbook_runs` row with `mode = dry`.

### 2. Approval flow

Store the plan; surface it (API; UI in STAGE-009-010/011); require an explicit user approval (audited, confirm-on-destructive) before a real run. The approval references the specific dry run + runbook hash (so an approved plan is invalidated if the runbook changed).

### 3. Safe-path skip

A `safe` runbook (explicitly downgraded by the author) skips the plan gate and runs real directly (still subject to all other gates).

### 4. Tests (fake claude)

Risky → plan-only → stored → halts; explicit approval → real run; risky NEVER reaches real without approval; safe → direct real run; approval invalidated if runbook hash changed.

## Design Notes

**Decisions selected by the implementing session (autonomous epic-loop), 2026-07-01. Grounded in the verified STAGE-009-005 orchestrator code + the §7.4/#5 safety contract.**

**A — Plan-only (dry) claude invocation differs by COMMAND.** Dry-run execs `claude -p <runbook.path> --permission-mode plan` (claude's built-in plan mode, which cannot execute tools) and deliberately OMITS `--dangerously-skip-permissions`. Real runs keep `claude -p <runbook.path> --dangerously-skip-permissions`. The command difference is the testable contract: the fake claude asserts `--permission-mode plan` is present AND `--dangerously-skip-permissions` is ABSENT for dry runs, and emits a recorded plan transcript. Claude's real plan mode is the runtime guarantee; the observable arg difference is what unit tests assert (no trust of external-binary internals required). Rejected A1 (plan flag alone — untestable without trusting the binary) and A3 (separate plan.md per runbook — authoring scope creep).

**B — Approval state lives in a NEW sibling `runbook_run_approvals` table (migration 0047).** Keeps `runbook_runs` immutable (aligns with STAGE-009-012 making runbook_runs non-API-deletable), makes drift-invalidation a single-field comparison, and gives rejection/re-approval a first-class home. Columns: `id` (uuid7 PK), `dry_run_id` (FK -> runbook_runs.id, the stored plan), `runbook_id`, `alert_id` (nullable, originating alert), `pinned_runbook_hash` (content_hash snapshotted when the plan was produced — the drift key), `status` (plain Text, no CHECK per the 0045 convention: 'pending' | 'approved' | 'rejected'), `approved_by` (principal_label at decision time; null while pending), `decided_at` (ISO8601 UTC; null while pending), `real_run_id` (FK -> runbook_runs.id, set when the real run fires; null otherwise), `created_at` (ISO8601 UTC). Indexes on (status), (dry_run_id), (runbook_id). The pending approval row is created in the SAME txn that stores the dry run. Rejected B1/B3 (columns/status on runbook_runs — conflates run-record with approval-lifecycle and fights the 012 immutability direction).

**C — Real run is triggered via a dedicated `execute_approved(approval_id, *, confirm_phrase, principal, ip)` orchestrator method.** It loads the approval + dry run + CURRENT RunbookRecord, RE-VALIDATES drift (current content_hash == pinned_runbook_hash), RE-RUNS the operational gate sequence (kill-switch -> allow-list -> rate-limit -> cooldown — a kill switch may have flipped or rate/cooldown may now block since the plan was produced), then reuses the EXISTING `_claim_and_exec` real-exec/persist helper (mode=RunMode.REAL) — ZERO divergent exec code. On success it sets `real_run_id` on the approval. Refactor: extract gate steps 1-4 into a `_check_operational_gates` helper that both `_check_gates` (which then appends the step-5 dry branch) and `execute_approved` call. Rejected C1 (re-enter handle_alert — re-runs matching + would re-deny at step 5, invasive) and C3 (exec inline in the router — leaks safety-critical exec/persist logic out of the orchestrator).

**D — API surface (BACKEND-only stage; approval UI is STAGE-009-010/011).** Three endpoints, all `Depends(require_session())` (auth 401 + CSRF on mutating methods), mirroring the STAGE-009-004 runbooks-router idiom (thread who_principal=user + ip=_client_ip(request) to the repo; audit in the SAME txn as the write):
- `GET /approvals?status=pending` — list pending approvals (approval_id, runbook_id, dry_run_id, created_at, pinned_runbook_hash, `drift_detected` computed live = current content_hash != pinned).
- `GET /approvals/{approval_id}/plan` — the stored plan transcript (reads the dry run's transcript_path) + metadata. Read-only.
- `POST /approvals/{approval_id}/approve` — body `{ confirm_phrase: str }`. Enforces confirm-on-destructive (§7.2) by requiring `confirm_phrase` to equal the fixed phrase (mirror the docker router's module-level confirm-phrase idiom; 400 on mismatch); enforces drift-invalidation (current content_hash != pinned_runbook_hash -> 409 `runbook_changed_since_plan`, mark approval rejected/superseded, do NOT execute); on success audits `autofix.approved` in-txn, sets status=approved, then calls `execute_approved(...)`.
- `POST /approvals/{approval_id}/reject` — status=rejected + audit `autofix.rejected` (cheap; completes the audit story).
Confirm mechanism = typed `confirm_phrase` body field (the proven docker-router pattern). Session-PIN is DEFERRED to the UI stages 010/011 (no session-PIN infra exists in the codebase yet).

**E — handle_alert halts after storing the dry plan.** New `RunOutcome.DRY_RUN_STORED` variant + a new `approval_id: str | None = None` field on `RunResult`. When `record.dry_run_required` is true, the pipeline branches (where 005 currently denies `risky_blocked`) to a claim-and-store-dry path: claim under the same per-runbook lock + `_in_lock_gate` re-check, `insert_started(..., mode=RunMode.DRY_RUN)`, exec claude in plan mode (Decision A), persist transcript + `mark_completed`, create the PENDING `runbook_run_approvals` row in the same txn, audit `what="autofix.dry_run_stored"` (who="system:autofix"), and return `RunResult(ran=True, outcome=DRY_RUN_STORED, runbook_id, run_id=<dry_run_id>, exit_code=<plan exec exit>, denial_reason=None, approval_id=<new>)`. No real exec fires. `ran=True` is honest (a plan exec genuinely ran); `outcome` is the discriminator callers key on. The `DenialReason.RISKY_BLOCKED` placeholder deny is REPLACED by this branch.

**Enum spelling:** keep the `RunMode` enum values (`"dry_run"`/`"real"`) as the single source of truth; correct the stale migration-0045 docstring / spec §6.1 note that say `'dry'`/`'real'` (the `mode` column is unconstrained Text; zero existing `dry_run` rows; changing a live enum value is riskier than fixing docs).

**Deferrals (tracked — no silent drops):**
- Markdown/whole-folder-drift detection -> **STAGE-009-012** (006's drift check is CONFIG-hash-only via `content_hash`; a runbook whose CLAUDE.md changed but whose config did not will NOT be flagged as drifted in 006). 012 already owns whole-folder/runbook_hash enrichment.
- Session-PIN confirm-on-destructive -> **STAGE-009-010/011** (006 ships the typed `confirm_phrase` body field only).
- Approval UI surface -> **STAGE-009-010/011** (006 is API-only per the card).
