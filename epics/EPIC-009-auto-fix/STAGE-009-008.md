# STAGE-009-008: Per-runbook scoped-capability granting + fixer-runner egress control

## Status: Not Started

## Overview

Make the scoped capabilities the orchestrator grants actually real at exec time (LOCKED Decision 2). A runbook's config file declares its capabilities; this stage implements how those become concrete grants on the `fixer-runner` at the moment of a `docker exec`: docker-socket access (for local container-action runbooks like the pihole example), optional bind-mounts, an egress allow-list, and — for READ access — citing an existing EPIC-017 SSH `target_id`. Also enforces the network posture: inbound to the fixer is denied; outbound is allow-listed (the Anthropic API is always allowed; any other endpoint only if the runbook declares it).

**Vertical slice:** after this stage, a runbook declaring `docker: {container: pihole-unbound, actions: [restart]}` causes the fixer to be granted exactly that docker capability and nothing broader; a runbook declaring no docker capability cannot reach the socket; egress beyond the allow-list is denied. Per non-negotiable #2, capabilities are exactly what's declared — never broad.

**Spec references:** §7.4 (network model: egress allowed for Anthropic API, inbound denied; #2 never broad paths), §7.5 (network model — private docker net), §10.2 (volume mounts). EPIC-009 LOCKED Decision 2 (per-runbook scoped capabilities; docker primary; SSH target_id read-only; verified EPIC-007/008 observe-only + EPIC-017 forced-command read-only).

## Concrete deliverables

### 1. Capability granting at exec time

Translate a runbook's declared `scoped_capabilities` into concrete grants on the `fixer-runner` for that exec: docker-socket exposure (scoped to the declared container + actions where feasible), bind-mounts, the egress allow-list. A capability not declared is not granted.

### 2. Egress control

Inbound denied; outbound allow-listed (Anthropic API always; runbook-declared endpoints only). Document how this composes with the private docker network (§7.5).

### 3. SSH target_id read reference

A runbook may cite an existing EPIC-017 SSH `target_id` for READ access (the forced-command is server-pinned, read-only). Wire the reference; do NOT create new write-capable targets.

### 4. Tests

Declared docker capability → granted; undeclared docker → denied (cannot reach socket); egress beyond allow-list → denied; Anthropic egress → allowed; SSH target_id read-ref resolves to an existing EPIC-017 target.
