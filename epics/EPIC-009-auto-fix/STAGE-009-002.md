## Design Phase

**Status**: [x] Complete

### Design Notes

**Selected approach (2026-06-29): Option B — host bind-mount + POSIX default ACLs, with the A-borrow (env-var UID/GID contract).**

**Decision 1 — Transcript volume mechanism: host bind-mount under the monitor data dir, NOT a named docker volume.** Path: `${HOMELAB_MONITOR_DATA_DIR}/runbook-transcripts` (host), mounted into the monitor container at `/data/runbook-transcripts`. Rationale: spec §10.2 requires transcripts be "always backed up"; §10.4 backs up by rsyncing HOST paths, so a host bind-mount is backup-able with zero new plumbing (a named volume would force a `--volumes-from` backup shim — premature indirection). Survives container rebuilds.

**Decision 2 — Cross-container UID/GID interop: POSIX default ACLs on the transcript directory, applied by host-setup.sh.** The fixer-runner container (STAGE-009-003) writes transcripts; the monitor container must READ them (transcript viewer in STAGE-009-011 + audit). The monitor runs at a host-specific runtime UID/GID (e.g. 995 vs build-time 1000), so a same-UID assumption is fragile. POSIX **default** ACLs (`setfacl -d -m u:<monitor-uid>:rX -m u:<fixer-uid>:rwX <dir>`) make every file the runner creates inherit monitor-readable + fixer-writable perms regardless of the writer's UID/umask. This reuses the proven setfacl idiom already in `scripts/host-setup.sh` (which ACLs `/etc/crontab` and `/etc/cron.d`), with the same idempotent `--check` support. **Fallback:** if `setfacl` is unavailable on the host, host-setup WARN-degrades to a shared supplementary GID + setgid directory (`chmod 2770`) — the same group-sharing idiom as the existing `amconfig` GID 2000 / docker GID wiring.

**Decision 3 — env-var UID/GID contract (the A-borrow):** introduce `HM_FIXER_UID` / `HM_FIXER_GID` (safe defaults in `.env.example`), so STAGE-009-003's fixer-runner adopts the exact numeric identity that 002's ACLs grant. This is the seam that makes the cross-stage deferral clean — 002 provisions the host-side grant + the directory; 003 builds the container whose `homelab-fixer` user adopts that UID/GID.

**Decision 4 — public vs overrides split (§11):** PUBLIC repo ships the open-source-safe surface — the `host-setup.sh` helper additions (idempotent, `--check`-aware), the compose volume/mount declaration, the `HM_FIXER_UID`/`HM_FIXER_GID` env vars with safe defaults, and the public provisioning docs (recommend `docs/deploy/`). The private `homelab-monitor-overrides` repo supplies only the real host-specific numeric UID/GID VALUES + any host-specific path — no host-mutating logic that isn't already in the public helper.

**Non-negotiable enforcement (#3 identity, #4 audit):**
- ACL grants the monitor **read-only** (`rX`) and the runner **write** — NEVER the reverse. The monitor must not be able to mutate an in-progress audit transcript (protects #4 audit integrity).
- `homelab-fixer` gets NO docker-group membership and NO host sudoers from this stage. 002 establishes only the read/write file boundary; the narrow sudoers (§7.4) is a separate later concern.
- Docker socket access for orchestrator `docker exec` is ALREADY present (monitor mounts `/var/run/docker.sock` RW + `group_add` docker GID, from EPIC-003 / STAGE-003-010). 002 VERIFIES this path; it does not re-provision it.

**Explicit deferrals (no silent deferral — each names its owning stage):**
- fixer-runner service block in `docker-compose.yml` (the container, its `USER homelab-fixer`, `group_add`, transcript-dir RW mount, egress-only network) → **STAGE-009-003**.
- In-container `homelab-fixer` OS-user creation (in the runner image Dockerfile) → **STAGE-009-003**.
- Orchestrator `docker exec` invocation + populating `runbook_runs.fixer_user`/`runbook_runs.host` columns → later EPIC-009 orchestrator-wiring stage (STAGE-009-005 owns the exec; the populating-write stage owns the column writes).
- Real host-specific UID/GID VALUES for `homelab-fixer` → the private `homelab-monitor-overrides` repo env (not the public repo).

**Host-mutation pause judgment:** 002 generates the script + docs + compose wiring (the bulk of the deliverable) WITHOUT mutating the host. Whether a real host mutation (creating the transcript dir + applying ACLs on THIS host) is needed for instance-A validation is judged during Refinement — ACLs key on numeric UID and can be applied before the in-container user exists, so the directory + ACL grant can be validated by dropping a file as an arbitrary UID and confirming the live monitor reads it. If a real host mutation is required to validate, that is the first legitimate host-action prompt (per the session charter).
