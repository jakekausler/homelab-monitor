## Overview

Build the heart of the auto-fix subsystem: the runbook orchestrator. Given an ingested alert, it matches the alert to a registered runbook, runs the gate sequence (kill switch â†’ allow-list/enabled â†’ rate-limit â†’ cooldown), claims the runbook, grants its declared scoped capabilities to the `fixer-runner` container, `docker exec`s `claude` non-interactively, captures the full transcript + stdout/stderr + exit code, and writes `runbook_runs` + an `alert_outcomes('auto_fixed')` row + an `audit_log` row. This is the keystone stage â€” it cannot be built before the schema (001), the fixer-runner (003), and the registry (004) exist. **Built and tested entirely against the FAKE `claude` binary** (a shell script that records args + emits a recorded transcript); the real Claude API is NEVER called in CI.

**Vertical slice:** after this stage, an alert that matches an enabled+auto-trigger runbook flows through the full gate sequence and (in real mode) execs the fake claude in the fixer-runner, capturing a complete audited run record. Rate-limit + cooldown are enforced. The dry-run gate (STAGE-009-006) and kill switch (STAGE-009-007) layer on top.

## Design Notes

**Design completed 2026-06-29.** Approach selected after brainstormer (Opus) presented options Aâ€“H; all recommendations adopted (conservative, safety-first). Design verified against REAL kernel code, not just the card prose.

### Selected approach (Aâ€“H)

**A. Structure.** A single async service class `AutoFixOrchestrator` (deps via `__init__`: RunbookRepo, AlertRepository, AppSettingsRepository, AsyncSecretsRepository, DockerSocketClient, SqliteRepository, structlog logger). Single entrypoint `async handle_alert(alert: Alert) -> RunResult | None` with private phase methods (`_match`, `_check_gates`, `_claim_and_exec`, `_persist`). Tests await `handle_alert` directly. Ingestion wire-up (`kernel/api/routers/alerts.py::ingest_alerts`, firing branch) dispatches the orchestrator as a **background task** (`asyncio.create_task`-style), NOT a blocking inline call â€” Alertmanager webhook retry + multi-minute exec make a blocking call unsafe; the 202 must not wait on the fix.

**B. Match resolution (#1).** Reject-ambiguous. Iterate all runbooks, collect every matcher hit. 0 matches â†’ return `None`, record NOTHING (a non-match is not a denial). Exactly 1 â†’ proceed. â‰Ą2 â†’ DENY with recorded reason `ambiguous_match`. Matcher reconstructed via `AlertMatcher.model_validate(dict)` from `RunbookRecord.alert_match_patterns` (already `list[dict]`); matches when matcher `alertname` equals `alert.labels["alertname"]` (if set) AND all matcher `labels` are a subset of `alert.labels`. (Alert has no top-level `alertname`.) Rationale: a wrong auto-fix is worse than no auto-fix.

**C. Claim / concurrency (#prevent-dupes).** Durable DB claim = an open-ended `runbook_runs` row (`ended_at IS NULL` = in flight). Check "no existing row for this runbook with `ended_at IS NULL`" then INSERT the started row, both inside one `repo.transaction()`, guarded by a short in-process per-runbook `asyncio.Lock` around the check-and-insert critical section. On exec completion (success/failure/timeout) set `ended_at` + `exit_code`. Reuses migration-0045 columns; survives restart; audit-visible. Stale-claim reconciliation noted as a STAGE-009-007 concern.

**D. exec_capture extension.** Extend the existing `DockerSocketClient.exec_capture` with optional keyword-only params: `user: str | None = None` (sets exec `User`), `env: Mapping[str, str] | None = None` (sets exec `Env` as `KEY=value` list), and stdin-EOF behavior matching the 003 `< /dev/null` contract. Defaults preserve current behavior byte-for-byte (no existing caller changes). Each new branch covered with the fake claude (api-key-present exercises `Env`, a `User` assertion exercises that branch, stdin drain exercises EOF).

**E. Transcript capture (#4).** `runbook_runs.transcript_path` points at the claude-session transcript file the fixer writes into `/data/runbook-transcripts/` (STAGE-009-002 ACL dir; pin the exact filename contract from the fake-claude script during Build). Exec stdout/stderr from `ExecResult` is written by the monitor to a sibling `<run_id>.exec.log` in the same ACL'd dir and its path referenced in the audit `after` payload. NO migration this stage (`runbook_runs` has one `transcript_path` and no stdout/stderr column; sibling-file + audit-payload satisfies #4 without a schema change).

**F. Gate-denial recording (#4).** A gate denial writes ONE `audit_log` row only â€” `insert_audit(conn, who="system:autofix", what="autofix.denied", after={"runbook_id", "alert_id", "gate": "<kill_switch|allow_list|rate_limit|cooldown|ambiguous_match|risky_blocked>", "detail": ...})`. NO `runbook_runs` row and NO `alert_outcome` for a denial (keeps `runbook_runs` meaning "an exec happened" and keeps the claim/rate-limit predicates clean). A no-MATCH writes nothing at all. The `gate` value doubles as a future metric label.

**G. EPIC-012 maintenance-window hook seam.** A private no-op `@asynccontextmanager` `_maintenance_window(runbook, alert)` wraps the actual exec (`async with self._maintenance_window(...):`). Today it `yield`s once with no side effect, documented with an EPIC-012 reference. EPIC-012 fills the body (open window before yield, close after) with zero refactor to the orchestrator control flow. NOT built here.

**H. Dry-run-for-risky (#5) â€” enforced as a DENY gate here.** After kill-switch/allow-list/rate-limit/cooldown, 005 adds a final pre-exec gate: if `runbook.dry_run_required` is True, DENY with recorded reason `risky_blocked` and do NOT exec. 005's real-exec path therefore fires ONLY for an allow-listed (`enabled AND auto_trigger`), gate-passing, NON-risky (`dry_run_required=False`) runbook. This keeps non-negotiable #5 enforced in code from the keystone stage forward (no live safety hole between 005 and 006). **STAGE-009-006 is the explicit owner** of the dry-run â†’ plan â†’ approval â†’ real-run path that will later intercept `dry_run_required` runbooks ahead of this `risky_blocked` deny.

### Gate sequence (final order)

kill-switch (reads `AppSettingsRepository.get("autofix_enabled")`; STAGE-009-007 owns writing it) â†’ allow-list (`enabled AND auto_trigger`) â†’ rate-limit (N/hour global, sliding window over `runbook_runs.started_at`) â†’ per-runbook cooldown (last run `ended_at` + `cooldown_seconds`) â†’ dry-run/risky gate (`dry_run_required` â†’ `risky_blocked`). Any failed gate DENIES with a recorded reason (choice F). Kill-switch is checked FIRST.

### Secret injection (#3)

`ANTHROPIC_API_KEY` fetched at exec time via `AsyncSecretsRepository.get("ANTHROPIC_API_KEY")`, injected into the exec `Env`, NEVER logged. Exercised in CI against the fake claude (which reports `anthropic_api_key_present`), never a live API call. Exec runs as `homelab-fixer` (`-u`), persisted as `runbook_runs.fixer_user`.

## Concrete deliverables

### 1. Alert â†’ runbook match

Match an ingested alert against registered runbooks' `match_patterns` (regex vs alertname/labels). Resolve to at most one runbook (or none â†’ no auto-fix; manual button only).

### 2. Gate sequence

In order: kill switch (STAGE-009-007 provides the check; wire the call), allow-list/enabled + auto-trigger, rate-limit (max N/hour globally), cooldown (per-runbook). A failed gate denies the run with a recorded reason.

### 3. Claim + exec

Claim the runbook (prevent concurrent duplicate runs), grant its declared scoped capabilities to the fixer-runner (STAGE-009-008 provides the granting mechanism; wire it), `docker exec` the non-interactive `claude` (STAGE-009-003 contract). Capture transcript/stdout/stderr/exit/timestamps.

### 4. Persist

Write `runbook_runs` (mode, prompt, transcript_path, exit_code, timestamps, runbook_hash, fixer_user, host), `alert_outcomes('auto_fixed')`, `audit_log`.

### 5. Tests (fake claude)

End-to-end with the fake claude: match â†’ gates â†’ exec â†’ capture â†’ persist; rate-limit denial; cooldown denial; full audit record written; `auto_fixed` outcome written; no-match â†’ no run.
