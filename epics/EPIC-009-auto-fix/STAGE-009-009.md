# STAGE-009-009: Claude→user improvement-feedback channel

## Status: Not Started

## Overview

Build the improvement-feedback channel (LOCKED Decision 3): when the fixer hits a limitation and exhausts its options, OR finishes but had to work around something inefficiently, it emits **structured feedback** ("this would've gone better with capability X / config change Y") that the orchestrator captures, persists, and the UI surfaces for the user to review. The user can then decide whether to adjust the runbook's configuration. This is a NEW first-class concept with no downstream consumer yet (purely additive — it does not change any existing contract).

**Vertical slice:** after this stage, a fixer run that emits structured feedback (via the agreed output contract) produces persisted feedback rows linked to the run, and the user can see them in the Auto-fix UI. A run with no feedback produces none.

**Spec references:** EPIC-009 LOCKED Decision 3. §3.1 (runbook orchestrator captures run output — extended here to parse a structured feedback marker). §6.1 (a new sibling table alongside `runbook_runs`). §9.2 (Auto-fix history screen — the review surface).

## Concrete deliverables

### 1. Structured feedback output contract

Define how the fixer signals feedback: a sentinel feedback file in its transcript dir (e.g. `feedback.json`) OR a structured stdout marker the orchestrator parses. Document the schema (what fields: a free-text suggestion, an optional structured "wanted capability / config change" hint, a severity/kind).

### 2. Persistence (`runbook_run_feedback` sibling table)

A new migration: `runbook_run_feedback` (id, runbook_run_id FK, kind, suggestion_text, structured_hint JSON, created_at). Multiple feedback items per run. Linked to the run + (transitively) the runbook.

### 3. UI review surface

Surface feedback items in the Auto-fix UI (the run-detail / history view). The user reads them and can decide to adjust the runbook config. (If Design folds this into STAGE-009-011, document that.)

### 4. Tests

Fake claude emits a feedback marker → parsed → persisted → linked to the run; multiple items per run; a run with no feedback → no rows; malformed feedback marker → recorded as an error, not a crash.
