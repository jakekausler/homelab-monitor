# STAGE-009-004: Runbook registry — folder loader + DB registration + allow-list management API

## Status: Not Started

## Overview

Build the runbook registry: load runbook folders (markdown + config) from disk, validate each config against the STAGE-009-001 pydantic model, compute a content hash, and register them in the DB — keeping the registry **file-authoritative** (config = truth; DB row = registration/enablement + hash). Expose the allow-list management API: register a runbook, enable/disable it, and opt it into auto-trigger. Per non-negotiable #1, these are **three separate explicit gates** (registered ≠ enabled ≠ auto-trigger) so the default posture is "present but inert."

**Vertical slice:** after this stage, runbook folders are discovered + validated + registered; the API lists/registers/enables/disables runbooks and toggles auto-trigger; a disabled or non-auto-trigger runbook is correctly inert. No execution yet (the orchestrator is STAGE-009-005).

**Spec references:** §3.1 (runbook orchestrator looks up the runbook for an alert — this stage provides the registry it looks up in), §6.1 (`runbooks` table), §7.4 (#1 allow-list per alert type; default manual). EPIC-009 LOCKED Decisions 2 (file-authoritative folder = markdown + config) + 4 (three gates, inert defaults).

- `pyright --strict`, 100% branch coverage. No bare `Any`, no `# type: ignore`.

- Non-negotiable #4: the content hash is computed + stored at registration (so a later run can detect the runbook changed).
- File-authoritative: re-loading a changed config file updates the registration + hash; the DB never silently diverges from the file.

## Concrete deliverables

### 1. Folder loader

Discover runbook folders, read each markdown + config file, validate the config (STAGE-009-001 model), compute the content hash (over the folder's authoritative files). Surface validation errors clearly without crashing the loader (a malformed runbook is reported, not fatal).

### 2. DB registration / enablement

Register/upsert the runbook row (path, hash, parsed config fields). Three independent flags: `enabled`, `auto_trigger`, plus `risk_tag` (default risky). Registration ≠ enablement ≠ auto-trigger.

### 3. Allow-list management API

Typed FastAPI routes: list runbooks, register/refresh from folder, enable/disable, set auto-trigger. Auth required; audited.

### 4. Tests

Loader validates valid/invalid configs; registration upserts + hashes; the three gates are independent; unauth call → 401; changed config → updated hash.
