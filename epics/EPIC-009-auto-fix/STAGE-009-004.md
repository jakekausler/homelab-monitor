# STAGE-009-004: Runbook registry — folder loader + DB registration + allow-list management API

## Status: Complete

## Overview

Build the runbook registry: load runbook folders (markdown + config) from disk, validate each config against the STAGE-009-001 pydantic model, compute a content hash, and register them in the DB — keeping the registry **file-authoritative** (config = truth; DB row = registration/enablement + hash). Expose the allow-list management API: register a runbook, enable/disable it, and opt it into auto-trigger. Per non-negotiable #1, these are **three separate explicit gates** (registered ≠ enabled ≠ auto-trigger) so the default posture is "present but inert."

**Vertical slice:** after this stage, runbook folders are discovered + validated + registered; the API lists/registers/enables/disables runbooks and toggles auto-trigger; a disabled or non-auto-trigger runbook is correctly inert. No execution yet (the orchestrator is STAGE-009-005).

**Spec references:** §3.1 (runbook orchestrator looks up the runbook for an alert — this stage provides the registry it looks up in), §6.1 (`runbooks` table), §7.4 (#1 allow-list per alert type; default manual). EPIC-009 LOCKED Decisions 2 (file-authoritative folder = markdown + config) + 4 (three gates, inert defaults).

## Concrete deliverables

### 1. Folder loader

Discover runbook folders, read each markdown + config file, validate the config (STAGE-009-001 model), compute the content hash (over the folder's authoritative files). Surface validation errors clearly without crashing the loader (a malformed runbook is reported, not fatal).

### 2. DB registration / enablement

Register/upsert the runbook row (path, hash, parsed config fields). Three independent flags: `enabled`, `auto_trigger`, plus `risk_tag` (default risky). Registration ≠ enablement ≠ auto-trigger.

### 3. Allow-list management API

Typed FastAPI routes: list runbooks, register/refresh from folder, enable/disable, set auto-trigger. Auth required; audited.

### 4. Tests

Loader validates valid/invalid configs; registration upserts + hashes; the three gates are independent; unauth call → 401; changed config → updated hash.

## Design Phase

**Status**: [x] Complete

### Selected Approach

1. **Loader trigger:** EXPLICIT REFRESH ONLY via `POST /api/runbooks/refresh` — no startup/lazy auto-scan. DB mutates only on deliberate operator action.
2. **Content-hash scope:** REUSE STAGE-009-001's config-only `compute_runbook_content_hash` verbatim. NOT a silent scope drop — whole-folder/markdown-drift hash coverage is by-design deferred to and OWNED BY STAGE-009-012.
3. **Refresh reconciliation:** new folder -> INSERT (+ audit `runbook_registered`); changed folder -> explicit column-list UPDATE of (path, alert_match_patterns, risk_tag, dry_run_required, rate_limit_per_hour, cooldown_seconds, content_hash) (+ audit `runbook_refreshed`). The operator gates `enabled`/`auto_trigger` are NEVER in any UPDATE set — clobber-safety is structural.
4. **Validation-error surfacing:** loader returns a structured result `{registered, skipped, errors:[{path, message}]}`; the refresh endpoint echoes it. One malformed folder is REPORTED, not fatal (does not abort the scan). Catches the path-tagged ValueError that `RunbookConfig.load_from_path` already raises.
5. **Markdown handling:** require a fixed-name `CLAUDE.md` per runbook folder, validated as existing + non-empty, but NOT parsed this stage. Absence -> per-folder validation error (flows through #4). Consumed by the STAGE-009-005 orchestrator as the Claude prompt body (CLAUDE.md is what `claude --dangerously-skip-permissions` auto-loads).
6. **API surface** (matches the crons-router convention GET-list + PATCH-mutate): `GET /api/runbooks` (list registered runbooks with gates + file-authoritative fields) + `POST /api/runbooks/refresh` (scan/register/reconcile, returns the #4 result shape) + `PATCH /api/runbooks/{id}` (partial body `{enabled?, auto_trigger?}`, operator gate writes only). All `require_session()`-authed (401 on unauth). PATCH + refresh both audited via `insert_audit` in the SAME transaction as the data write. NO risk_tag mutation endpoint (risk_tag is file-authoritative).

**Cross-cutting:** the loader SKIPS any folder whose name starts with `_` (excludes `runbooks/_examples/`, self-documenting; the pihole/example exemplar must NOT be auto-registered). Audit `what` values: `runbook_registered`, `runbook_refreshed`, `runbook_gates_changed` (single PATCH event recording before/after the changed gate).

**Non-negotiables mapping:** #1 Trigger (three orthogonal gates registered≠enabled≠auto-trigger, all default OFF/inert; unauth registry-management call -> 401) and #4 Audit (content_hash computed + stored at registration so a later run detects drift; all management calls authed + audited in-transaction) are the two load-bearing ones and are genuinely satisfied by this design.
