# STAGE-003-013: Container drill-down unification + integration-page simplification

## Status

Status: Design Complete

## Goal

Unify all container-level information and actions into a single per-container drill-down page with tab navigation. Slim the Docker integration page to a compose-grouped list with badges and resource metrics. All container actions (Pull & Restart, probe management, logs viewing, image update review) move out of the integration page proper into the per-container drill-down.

This is a structural refactor — no new backend endpoints required. All existing routes survive; a new parent route is added to host shared layout + tab navigation.

## Success Criteria

1. Integration page (`/integrations/docker`) renders a compose-grouped list. Each compose file's containers appear under a non-collapsible header showing the compose file's parent directory basename. Containers WITHOUT a compose file appear under an "Ungrouped" header. Within each group, containers sort alphabetically by name.
2. Each list row shows: container name, image (subtitle), 4 status badges (status, image-update, probes, healthcheck) + restart-count chip (24h, only if >0) + CPU% + RAM MiB + a small "→" affordance indicating the row is clickable.
3. Clicking a row navigates to `/integrations/docker/containers/$name` (the new parent route).
4. New parent route renders a shared sticky header (name + status badge + image as subtitle + back-link to integration page) and a horizontal scrollable tab strip with 4 tabs: **Overview**, **Probes**, **Logs**, **Recent Actions**.
5. Default child route (when user visits `/$name` with no specific tab) redirects to `/$name/overview`.
6. **Overview tab** contains: image update detail (current/latest digest, source, timestamps, error reasons) + Pull & Restart button (visible only when `update_available === true`) + container metadata block (image, status, exposed ports, healthcheck command, compose info) + restart/exit history block + CPU & RAM block with 15-min sparklines.
7. **Probes tab** contains the existing `ContainerProbesCard` content (active + suggested + add new), styled to fit the tab's layout.
8. **Logs tab** contains the existing `DockerContainerLogsViewer` content.
9. **Recent Actions tab** contains the existing `RecentActionsPanel` content.
10. All four tabs share the page header + tab strip; switching tabs is fast (route-level navigation; React Query caches persist across tab switches).
11. Mobile (< sm breakpoint): horizontal scrollable tab strip; sticky header collapses to name + status badge; tab content is full-width.
12. All existing routes `/probes`, `/image-update`, `/logs` continue to work (children of the new parent). `/image-update` is RENAMED to `/overview` per the new structure (the image update detail moves into Overview). All internal links to `/image-update` get updated.
13. Old `DockerIntegrationPage` sections (the desktop grid, the mobile cards, the ProbesPanel section) are DELETED. Old `ContainerGridRow.tsx` + `ContainerGridCard.tsx` + `ContainerGrid.tsx` are DELETED.
14. The ProbesPanel (just shipped in STAGE-003-012) is REMOVED from the integration page and its content (`ContainerProbesCard`) becomes the Probes tab on the unified per-container page.
15. `make verify` passes green with 100% backend kernel coverage maintained and UI coverage gates met.

## Locked Design Decisions

### D-NESTED-ROUTES
Use nested TanStack Router routes for the unified per-container page. Parent route `/integrations/docker/containers/$name` provides shared header + tab strip + `<Outlet>`. Child routes `/overview`, `/probes`, `/logs`, `/actions` render in the Outlet. Browser back/forward + deep links work naturally. URLs are stable.

### D-COMPOSE-GROUP-NON-COLLAPSIBLE
Integration list groups containers by compose file basename. Each group has a non-collapsible header. Standalone containers (no compose) go under an "Ungrouped" group. Within group: alphabetical by container name. Between groups: alphabetical by group name. No localStorage state, no expand/collapse interaction this stage (could be added in a future iteration if user requests).

### D-FULL-ROW-LINK-WITH-CHEVRON
Each integration list row is a single `<Link>` element wrapping name + image + badges + CPU/RAM + a trailing `→` chevron icon. No inline action buttons in the list. All actions live on the unified per-container page.

### D-RESPONSIVE-LIST-ROW
ContainerListRow renders the same on desktop and mobile. Layout: name + image stacked vertically on the left; badges + CPU/RAM stacked on the right on narrow viewports, inline on wide viewports. NO separate desktop-grid vs mobile-card components.

### D-TAB-ORDER-LOCKED
4 tabs in fixed order: **Overview → Probes → Logs → Recent Actions**. Image update info lives on Overview (not its own tab). Pull & Restart button lives on Overview (only when `update_available === true`).

### D-OVERVIEW-CONTENTS
Overview tab shows (all four blocks):
1. **Image update block** — full image-update detail (source, current/latest digest, timestamps, error reasons). Pull & Restart button at the bottom of this block when `update_available === true`.
2. **Container metadata block** — image, status, exposed ports (from inspect if available), healthcheck command, compose info (project, service, file path) if applicable.
3. **Restart/exit history block** — restart_count_24h, cumulative restart_count, last exit_code, last_recreated_at timestamp.
4. **Resource usage block** — current CPU% + RAM MiB with 15-min sparklines (see D-OVERVIEW-SPARKLINES).

### D-OVERVIEW-SPARKLINES
CPU and RAM on the Overview tab include 15-minute sparklines from VictoriaMetrics. Series: `homelab_container_cpu_percent` and `homelab_container_memory_mib` (or whatever the actual metric names are — confirm at Build start). Reuse the existing `<Sparkline>` component from `apps/ui/src/components/tiles/Sparkline.tsx`. If the metrics-range query endpoint doesn't exist, Build phase falls back to current-values-only and adds a follow-up TODO for sparklines.

### D-STICKY-HEADER
Per-container page header (sticky on all tabs):
- Container name as h1
- Status badge inline next to name
- Image as muted subtitle (truncated if long)
- "← Back to Docker integration" link
- Tab strip below the header (also sticky)

Restart-count chip + healthcheck chip in header → DEFERRED to a future iteration. The integration list row already shows these badges; once user is on the per-container page, they're focused on one container's tabs.

### D-MOBILE-TAB-STRIP-HORIZONTAL-SCROLL
Tab strip is horizontally scrollable on narrow viewports. No collapse-to-dropdown. Same navigation pattern across breakpoints.

### D-OVERVIEW-IS-DEFAULT-TAB
Visiting `/integrations/docker/containers/$name` with no sub-path → redirect to `/$name/overview`. Bookmarkable; consistent landing.

### D-NO-BACKEND-CHANGES
This is a pure structural refactor. No new endpoints. No new Pydantic models. No new repo methods. All data already available via existing hooks.

### D-DELETION-LIST-LOCKED
Deletions are scoped exclusively to the integration-page surface that's being replaced:
- `ContainerGridRow.tsx`, `ContainerGridCard.tsx`, `ContainerGrid.tsx` (replaced by ContainerList + ContainerListRow)
- `ProbesPanel.tsx` (the section disappears from integration page; ContainerProbesCard moves to Probes tab)
- `ContainerProbesPage.tsx`, `ContainerImageUpdatePage.tsx` (replaced by tab components under the new parent route)
- Their test files

Everything else stays. `ContainerProbesCard`, `AddProbeModal`, `EditProbeModal`, `RecentActionsPanel`, `DockerContainerLogsViewer` (refactored to drop its own page wrapper), `ImageUpdateBadge`, `ProbesBadge`, `StatusBadge`, `HealthcheckBadge`, `Sparkline` all survive.

### D-DEFER-LIST-INLINE-ACTIONS
No inline Pull & Restart button in the integration list rows (originally an option from the brainstorm). All actions live on the per-container page. The badge tells the user "update available"; they click the row to drill in and act on it.

## Out of Scope

- New backend endpoints (the stage is a pure UI refactor)
- Persistent state for compose-group expand/collapse (groups are non-collapsible this stage)
- Container metadata that isn't already in `useListContainers()`'s response (no new fields)
- Inspect-driven data on Overview (e.g., exposed ports from docker inspect) UNLESS the existing endpoint already returns it; do NOT add a new inspect call for the metadata block. If a piece of metadata isn't available, just don't show it.
- Bulk operations (Pull & Restart all in a compose group, etc.)
- Reordering containers within a group
- Custom column visibility / per-user grid preferences
- Per-container favorites / pinning
- Container creation / removal (Docker-side operations beyond Pull & Restart)
- Settings / configuration tab on the unified page
- Notifications tab
- Live tail in the Logs tab (still deferred to EPIC-004)

## Known Gotchas (carry from prior stages)

- **Stale dist bug**: `pnpm --filter ui run build` BEFORE `docker compose build monitor`. The Dockerfile copies pre-built dist/.
- **`as never` not `as any`** in vitest mock returns (ESLint enforces).
- **noUncheckedIndexedAccess**: optional chaining `arr[i]?.field` on any array indexing.
- **TanStack Router params**: typed access via `useParams({ from: ... })` or `useParams({ strict: false })`. Match existing patterns in DockerContainerLogsViewer.tsx.
- **Tests that render components with `<Link>`** need `RouterProvider` wrapper + async `findBy*` queries (NOT sync `getBy*`).
- **Coverage gates**: backend 100% kernel (no backend changes this stage; gate trivially met). Frontend functions 80% / statements 80% / branches 75%.
- **ApiError constructor signature**: `new ApiError({status, code, message, retryAfterSeconds, details})`. NOT positional args.
