# STAGE-003-006: Label-based probe auto-config

## Status: Complete

## Overview

Containers carrying `homelab-monitor.<kind>.<name>=<value>` labels get probes created automatically — the label IS the user's consent, no suggestion required. Supported kinds: `http`, `tcp`, `exec`, `metrics`. Multiple probes per container supported (e.g., HA gets `http.api` + `tcp.mqtt` + `exec.healthz` simultaneously). A new `probe_targets` table holds the resolved probe configs; a new `ProbeRunner` collector executes them on schedule.

**Vertical slice:** a container labeled `homelab-monitor.http.api=http://localhost:8123/api/` gets an HTTP probe created on the next discoverer tick (STAGE-003-004's output is the trigger). The probe runs every 30s; failures emit `homelab_probe_up{container,kind,name} 0` and fire a vmalert rule. The drill-down grid shows a "Probes: 1 active" indicator.

**Spec references:** §2 Q13 (label-based config, A3 — labels primary); §5.2 (Collector contract — probes are collectors); §5.4 (concurrency groups — `docker.<container>` group prevents per-container DDOS); §6.1 (`targets` table — probe-targets are a kind).

**Scope note:** This stage is at the upper bound of a single-session stage. The Design phase should consider splitting into 005a (label parsing + probe_targets table + resolver + discoverer integration) and 005b (ProbeRunner execution + vmalert rule + UI badges/drawer + API endpoints) if Design phase finds the scope too large. If the stage is kept as one, Build phase should plan for it to potentially span two sessions.

## Concrete deliverables

### 1. Label parsing + collision detection

New module `apps/monitor/homelab_monitor/kernel/docker/label_parser.py`:

- Parses `homelab-monitor.<kind>.<name>=<value>` labels into structured probe descriptors.
- Supported kinds: `http`, `tcp`, `exec`, `metrics`.
- `<name>` defaults to `default` if the label is shaped as `homelab-monitor.<kind>=<value>` (no second dot segment).
- Returns `list[ProbeDescriptor]` + `list[LabelCollision]`. Caller decides what to do with each.
- Collision = two labels resolving to the same `(kind, name)` tuple. Cross-stage criterion #11: collisions become suggestions, NOT probes.

### 2. `ProbeDescriptor` + `probe_targets` table

New module `apps/monitor/homelab_monitor/kernel/db/repositories/probe_targets_repository.py`:

- `probe_targets` table — id, container_name, kind (http/tcp/exec/metrics), name (probe instance name), target_value (the raw label value or file-override value), config_source (`label`|`file_override`|`auto_default`|`discovered_accepted`), enabled, interval_seconds, timeout_seconds, last_run_at, last_status, last_error, created_at, hidden_at (soft-delete).
- Indexed on `(container_name, kind, name)` — uniqueness enforced.
- Upserted by the discoverer-callback flow when labels are parsed.

### 3. Probe-target resolver — integrates with the discoverer

Extends STAGE-003-005's discoverer hook. When the discoverer sees a container with `homelab-monitor.*` labels:

1. Parse labels into `ProbeDescriptor[]` + `LabelCollision[]`.
2. For each `LabelCollision` → create a `docker_label_collision` suggestion (STAGE-003-005 path).
3. For each `ProbeDescriptor` → upsert a `probe_targets` row. Container name + kind + name is the upsert key. No suggestion needed (label = consent).
4. For containers in `profiles: ["disabled"]` → SKIP probe upsert per cross-stage criterion #9 (still emit the disabled-profile suggestion from STAGE-003-005).
5. For containers without ANY `homelab-monitor.*` labels → no probe-target created (STAGE-003-005 surfaces as plain suggestion; user accepts via STAGE-003-012 + EPIC-011).

### 4. Probe target resolution — host-IP vs container-IP

`apps/monitor/homelab_monitor/kernel/docker/probe_resolver.py`:

- Given a `ProbeDescriptor` + the container's metadata (network_mode, ports, IP from the socket collector's data), produce a concrete probe target (URL, host:port, or exec command).
- Host-network containers (`network_mode: "host"`) → resolve to the host IP (per criterion #8). Local probes use `127.0.0.1` or the configured `docker.host_ip` config value.
- `host.docker.internal:host-gateway` aliases are preserved (per criterion #14) — if the label value contains `host.docker.internal`, resolve through the docker-managed alias (the docker daemon handles this when the probe is run from inside the monitor container).
- For `tcp.<name>=port:host` label values, parse the `port:host` syntax (where `host` is `host` literal meaning "the container host's IP" or `container` meaning the container's docker-network IP).
- For `exec.<name>=<cmd>`, the resolver emits `docker exec <container> <cmd>`. NOTE: docker exec is a write operation on the Docker API — the socket :ro bind-mount flag does NOT restrict API operations (it only prevents the mount-point file itself from being modified). The security boundary for exec probes is the docker.probes.exec_enabled config flag (default false), NOT the socket mount mode. Only opt-in users get exec probes.

### 5. `ProbeRunner` — collector that executes probes

New module `apps/monitor/homelab_monitor/kernel/metrics/probe_runner.py`:

- A `Collector` per `concurrency_group = "docker.<container_name>"` — one runner per container so probes for the same container serialize (per criterion #12 — never DDOS one service with parallel probes).
- Tick interval per probe = its `interval_seconds` (default 30s); the runner respects per-probe schedule.
- Executes each enabled `probe_targets` row for its container:
  - `http`: GET (or HEAD if config says so), check 2xx/3xx; bounded timeout.
  - `tcp`: open a socket, no payload, check connectivity; bounded timeout.
  - `exec`: `docker exec <container> <cmd>`; check exit code = 0; bounded timeout.
  - `metrics`: GET the metric endpoint (assume Prometheus exposition format); validate parse; no further action this stage (downstream alerts evaluate the metrics themselves).
- Emits `homelab_probe_up{container,kind,name}` 1/0 per run + `homelab_probe_duration_seconds{...}` + `homelab_probe_last_error_age_seconds{...}`.
- Per spec §16, the runner emits self-metrics (`homelab_collector_run_*`).
- Failures NEVER trigger a container restart (criterion #7). A failing probe emits a metric + the dispatcher routes via the normal alert channels.

### 6. vmalert rule — probe failure

`deploy/vmalert/metrics/docker_probes.yml` (new rule file):

```yaml
- alert: DockerProbeFailing
  expr: homelab_probe_up == 0
  for: 2m
  labels:
    severity: warning
    integration: docker
  annotations:
    summary: "Probe {{ $labels.kind }}.{{ $labels.name }} on {{ $labels.container }} has been failing for >2m"
```

## Decisions in this stage

## Locked Design Decisions

D-LABEL-NAMESPACE — `homelab-monitor.<kind>.<name>=<value>`. Kind ∈ `{http, tcp, exec, metrics}`. `<name>` defaults to `default` if absent. Multiple probes per container supported. (Spec Q13 + EPIC-003 brainstorm.)

D-LABEL-IS-CONSENT — a container with `homelab-monitor.*` labels gets probes auto-created. No suggestion needed (the label IS the consent). Containers WITHOUT labels go through the suggestion flow (STAGE-003-004 + STAGE-003-010 + EPIC-011).

D-EXEC-OPT-IN — `exec` probes are NEVER auto-created from labels by default. Config flag `docker.probes.exec_enabled` (default `false`) gates exec probes. Even with the flag enabled, the user must opt in per cron-style consent (TBD in Design phase — could be a per-container additional label like `homelab-monitor.exec_authorized=true`).

D-PER-CONTAINER-CONCURRENCY — every probe runs in `concurrency_group = "docker.<container_name>"` so probes for the same container serialize. Prevents per-service DDOS. (Cross-stage criterion #12.)

D-PROBE-FAIL-NO-RESTART — a failing probe NEVER triggers a container restart. Alert via normal channels; user decides. (Cross-stage criterion #7.)

D-HOST-NETWORK-IP-RESOLVE — host-network containers probe via host IP (configurable `docker.host_ip`, default `127.0.0.1`). Container-network containers probe via container IP from the socket data. (Cross-stage criterion #8.)

D-COLLISION-NO-PROBE — label collisions emit a `docker_label_collision` suggestion (from STAGE-003-004) and BLOCK probe creation. No probe is created until the user resolves the collision (by editing labels). (Cross-stage criterion #11.)

D-EXEC-PER-CONTAINER-LABEL — exec probes require BOTH the global `docker.probes.exec_enabled=true` flag AND a per-container `homelab-monitor.exec_authorized=true` label. The authorize label is a parser carve-out (flag-style, not parsed as a `ProbeDescriptor`). Locked STAGE-003-006 Design phase.

D-PROBE-RUNNER-HYBRID — probe execution uses a hybrid supervisor + per-container asyncio.Task model (mirrors `DockerDiscoverer.start_events_loop`). One registered `ProbeSupervisor` Collector handles bookkeeping + tick-reconciliation; per-container tasks are spawned in lifespan and cancelled on shutdown. No scheduler changes. Locked STAGE-003-006 Design phase.

D-PROBE-DETAIL-ROUTE — per-container probe drill-down lives at the route `/integrations/docker/containers/$name/probes`. Symmetric with STAGE-003-011's upcoming `/integrations/docker/containers/$name/logs` route. Locked STAGE-003-006 Design phase.

D-TCP-LABEL-URL-DSL — `tcp.<name>` label values use URL-like syntax `tcp://host:8080`, `tcp://container:8080`, or `tcp://<explicit-ip>:8080`. Symmetric with `http.<name>=http://...` and `metrics.<name>=http://...`. Sentinel tokens `host` and `container` are intercepted by the resolver before DNS. Malformed values emit `docker_label_malformed` suggestions. Locked STAGE-003-006 Design phase.

## Design Phase

- **Approaches Considered:**
  - **Exec opt-in mechanism:** (A) config flag alone; (B) config flag + per-container `homelab-monitor.exec_authorized=true` label; (C) config flag + config-level allowlist of container names.
  - **Probe-runner instantiation model:** (A) dynamic per-container collectors registered into Scheduler; (B) single ProbeRunner Collector with internal per-container loop; (C) hybrid — supervisor Collector + per-container asyncio.Task spawned in lifespan (mirroring `DockerDiscoverer.start_events_loop`).
  - **UI surface for per-container probe state:** (A) side drawer/sheet on badge click; (B) inline expandable row; (C) dedicated route `/integrations/docker/containers/$name/probes`.
  - **`tcp.<name>` label-value DSL:** (A) strict `<port>:<scope>` (e.g. `8080:container`); (B) smart defaults inferring scope from `network_mode`; (C) URL-like `tcp://host:8080` / `tcp://container:8080`.

- **User Choice:**
  - **Q1 Exec opt-in → Option B (per-container label + global flag).** Both `docker.probes.exec_enabled=true` AND `homelab-monitor.exec_authorized=true` on the container AND the `exec.<name>=<cmd>` label(s) must be present. Consent lives in the compose file (auditable in git). Defense-in-depth: enabling globally doesn't activate every container with a leftover exec label. The `exec_authorized` label is a parser carve-out from the `homelab-monitor.<kind>.<name>` shape — recognized as a flag-style label, not parsed into a `ProbeDescriptor`.
  - **Q2 Runner model → Option C (hybrid supervisor + per-container asyncio.Task).** Scheduler has no `add_collector` / `remove_collector` public API and `_threshold_for` explicitly notes "no reload mechanism exists" — Option A would require a scheduler refactor outside this stage's scope. Option B's single failure-budget envelope risks one container's chronic failures quarantining probes for all others. Option C mirrors the established `DockerDiscoverer` pattern (registered periodic Collector + lifespan-managed long-lived task) that the team has already shipped, debugged, and tested. The supervisor's periodic `run()` reconciles `probe_targets` table state against per-container `asyncio.Task` set, spawning/cancelling as needed. Lifespan shutdown cancels all per-container tasks (mirroring `stop_events_loop`).
  - **Q3 UI surface → Option C (dedicated route `/integrations/docker/containers/$name/probes`).** Symmetric with STAGE-003-011's upcoming `/integrations/docker/containers/$name/logs` route — gives operator a consistent "drill into container X, here are its facets" mental model. Trivially mobile-correct (route navigation is identical on desktop and mobile; no responsive layout). No new UI primitive needed (codebase only has Radix Dialog; no Sheet/Drawer exists). The grid badge ("2 active, 1 failing") continues to answer the at-a-glance question; the route is for investigation + enable/disable actions.
  - **Q4 `tcp.<name>` DSL → Option C (URL-like `tcp://host:8080` / `tcp://container:8080`).** Symmetric with `http.<name>=http://...` and `metrics.<name>=http://...` — one parser path (urlparse + sentinel-token check for `host` / `container`). Self-documenting. Explicit IPs supported as `tcp://192.168.1.5:8080`. Malformed labels emit `docker_label_malformed` suggestions via the existing suggestion infrastructure.

- **Seed Data Agreed:** acceptance test — a labeled container (e.g. `homelab-monitor.http.health=http://container:8080/healthz`) produces a `probe_targets` row auto-created on the next discoverer tick; stopping the labeled HTTP service inside its container → `homelab_probe_up == 0` → `DockerProbeFailing` alert fires within 2m; the drill-down route shows the failing probe + last_error.

- **Cross-cutting consequences of selections:**
  - **No scheduler changes in 003-006** (Q2.C). Per-container failure isolation is implemented inside the supervisor (bespoke, not via Scheduler.FailureBudget). A future epic can lift this into the scheduler if desired.
  - **`/integrations/docker/containers/$name/...` URL pattern established by 003-006.** STAGE-003-011 must use the same URL shape for the logs route (`/integrations/docker/containers/$name/logs`).
  - **`docker_label_malformed` suggestion kind is new** (Q4.C). Reuses `insert_or_update_docker_suggestion_conn` pattern from STAGE-003-005; dedup_key is the logical-key string; the malformed-label detail captures which label was malformed.
  - **`homelab-monitor.exec_authorized` is a flag-style label** (Q1.B), parsed as a special case alongside the `homelab-monitor.<kind>.<name>` shape. Document this carve-out clearly in `label_parser.py`.

- **Session Notes:**
  - Brainstormer verified scheduler has no dynamic register/deregister API — rules out Q2.A.
  - Brainstormer verified UI has no Sheet/Drawer primitive; only Radix Dialog — adds cost to Q3.A.
  - All 7 pre-locked design decisions (D-LABEL-NAMESPACE, D-LABEL-IS-CONSENT, D-EXEC-OPT-IN global, D-PER-CONTAINER-CONCURRENCY, D-PROBE-FAIL-NO-RESTART, D-HOST-NETWORK-IP-RESOLVE, D-COLLISION-NO-PROBE) remain in force; the four user-choice questions above resolve the remaining open design space.
