# STAGE-003-003: Docker socket collector — container inventory + status + restart_count + exit_code + healthcheck

## Status: Not Started

## Overview

Build the Docker socket collector that gathers signals cadvisor cannot see: container inventory (running / exited / restarting / paused / dead state), restart counts, last-exit codes, healthcheck status (when defined), image references (with RepoDigest for STAGE-003-007), and per-container labels (the `homelab-monitor.*` namespace consumed by STAGE-003-005).

Reads `/var/run/docker.sock:ro` (mount widens to RW only at STAGE-003-009). Writes to the SQLite `targets` table (Docker containers as a `kind`) and emits `homelab_container_*` metrics to VictoriaMetrics. Fills the container grid in the drill-down UI (STAGE-003-002 skeleton).

**Vertical slice:** the drill-down's container grid populates with all running + exited containers on the host; each row shows live status, restart count, last exit code, and healthcheck status. Grafana shows the same data via `homelab_container_*` metrics.

**Spec references:** §3.4 (Docker containers as discovered targets); §5.2 (`Collector` protocol); §5.4 (concurrency groups — the socket collector runs in `docker.socket` group); §6.1 (`targets` table contract); Q13.

## Concrete deliverables

### 1. `DockerSocketClient` — reusable bounded socket client

New module `apps/monitor/homelab_monitor/kernel/docker/socket_client.py`:

- Connects to `/var/run/docker.sock` (path configurable via `HOMELAB_MONITOR_DOCKER_SOCKET`, default `/var/run/docker.sock`).
- Uses the `docker` Python SDK (or `aiodocker` — final choice in Design) for typed responses.
- Read-only operations only this stage: `containers.list(all=True)`, `containers.get(id).attrs`, `containers.get(id).stats(stream=False)`.
- Bounded: a single tick MUST complete within the collector timeout. The list call is bounded by Docker; per-container detail calls are gated by concurrency limits (per spec §5.4).
- Failure handling: socket unreachable → raise typed `DockerSocketError`; collector logs + skips the tick; emits `homelab_collector_run_success{collector="docker_socket"} 0`.

### 2. `DockerSocketCollector` — scheduler-registered collector

New module `apps/monitor/homelab_monitor/kernel/metrics/docker_socket_collector.py`:

- Tick interval: 30s (configurable via `HOMELAB_MONITOR_DOCKER_SOCKET_INTERVAL_SECONDS`).
- `concurrency_group = "docker.socket"`.
- Registered in `lifespan.py` alongside `HeartbeatStateCollector` and `CronRunReconciler`.
- Each tick:
  1. List all containers (`all=True` so exited containers are seen).
  2. For each container, extract: id, name (first item from `Names`), image (ref + RepoDigest from `Image` + `ImageID`), state (running / exited / restarting / paused / dead), restart_count (from `RestartCount`), last_exit_code (from `State.ExitCode`), healthcheck status (from `State.Health.Status` if `Health` is present, else `null`), labels (full label dict; downstream stages filter for `homelab-monitor.*`), network_mode (so probe-target resolution in STAGE-003-005 can route via host IP when `host`).
  3. Upsert into `targets` table — `kind="docker_container"`, `name=<container_name>`, `labels=<json>`, `status=<state>`, `first_seen`, `last_seen`.
  4. Emit `homelab_container_status{name,state}` (1 if in that state, 0 otherwise), `homelab_container_restart_count{name}`, `homelab_container_last_exit_code{name}`, `homelab_container_healthcheck{name,status}` (1 if healthy, 0 if unhealthy / starting / none).

### 3. `targets` table — Docker container rows

`apps/monitor/homelab_monitor/kernel/db/repositories/targets_repository.py` (new repository OR extension of existing `targets` repo from EPIC-001):

- `upsert_docker_container(...)` — insert-or-update by `(kind="docker_container", name=container_name)`.
- `list_docker_containers()` — for the API endpoint.
- `mark_docker_container_missing(name)` — when a container disappears between ticks (e.g., user removed it), update `last_seen` but DON'T delete the row (keeps history). Set `status="missing"`.
- Soft-delete via `hidden_at` (pattern from STAGE-002-003) — user can hide containers they don't want in the grid.

### 4. API endpoint — `GET /api/integrations/docker/containers`

`apps/monitor/homelab_monitor/kernel/api/routers/docker.py` — new router:

- `GET /api/integrations/docker/containers` — returns the list of containers from the `targets` table, joined with the most recent `homelab_container_*` metric values from VM (so the grid has live status without two API calls).
- Response shape feeds `ContainerRow[]` defined in STAGE-003-002. Adds the fields scaffolded as optional: `status`, `restart_count`, `exit_code`, `healthcheck`, `image`, and the new `network_mode` field (for STAGE-003-005 probe routing).
- Auth: requires session (read-only — no scope guard needed beyond authenticated).

## Decisions in this stage

The Design phase confirms the socket client library (`docker` SDK vs `aiodocker`), the tick interval, the per-container metric label cardinality (do we keep one series per container or aggregate?), and the missing-container handling rule.

## Locked Design Decisions

D-SOCKET-READ-ONLY — the docker socket mount is `:ro` in this stage and remains so until STAGE-003-009. The collector never invokes write operations on the socket.

D-COLLECTOR-NOT-DISCOVERER — this stage's collector ONLY refreshes the `targets` table for ALREADY-KNOWN containers. New-container detection (and the resulting suggestion) is STAGE-003-004's discoverer. Separation of responsibilities.

D-MISSING-NOT-DELETED — a container that disappears between ticks is marked `status="missing"`, NOT deleted from `targets`. History is preserved. User can hide via `hidden_at` (the soft-delete pattern from STAGE-002-003).

D-CADVISOR-VS-SOCKET — resource metrics (CPU / RAM / IO / network) come from cadvisor (STAGE-003-001); the socket collector handles status / restart / exit code / healthcheck / image / labels / network_mode. No overlap.
