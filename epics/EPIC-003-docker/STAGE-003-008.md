# STAGE-003-008: Image-update detection (registry digest)

## Status: Complete

## Overview

Detect when a newer image is available upstream for any container whose compose entry uses `image:` (pulled from a registry). Periodically compare the local image's `RepoDigest` against the digest the registry serves for the container's tag. When they differ, emit `homelab_image_update_available{container,image,current_digest,latest_digest} 1`. A vmalert rule fires an `info`-severity alert; the drill-down grid shows an "Update available" badge.

This is the diun-style mechanism from spec Q13. It is READ-ONLY — no `docker pull` happens here (that's STAGE-003-009 "Pull & Restart"). Until then, only the *knowledge that an update exists* is recorded.

**Vertical slice:** when a container's upstream tag publishes a new image (e.g., Pi-hole 2025.05 → 2025.06), within one tick the drill-down grid shows an "Update available" badge on the Pi-hole row, vmalert emits an info-severity alert, and the dispatcher routes per the user's info-channel rules.

**Spec references:** §2 Q13 (diun-style update notifications); §3.4 (container image registries as a discovered target type); §6.2 (`homelab_image_update_available{container}` metric); §7.5 (network model — outbound to Docker Hub / GHCR for digest checks).

## Concrete deliverables

**Image-pull event handling (inherited scope from STAGE-003-005).** The Docker socket's `image pull` event stream is NOT subscribed in STAGE-003-005 (where the discoverer was built) — handling of image-pull events lives HERE, alongside registry digest comparison. STAGE-003-008 extends `DockerSocketClient.events()` (added in 005) with `?filters={"type":["image"]}` or subscribes a second filtered stream. On `image pull` for an image currently used by a running container, evaluate whether the new digest differs from the cached digest; if so, surface an image-update suggestion. Coordinate with the periodic registry-digest scan from this stage's main deliverable to avoid double-suggesting.

### 1. Registry client — `RegistryDigestClient`

New module `apps/monitor/homelab_monitor/kernel/docker/registry_digest_client.py`:

- Given an image reference (`ghcr.io/blakeblackshear/frigate:0.14.0`, `pihole/pihole:2025.05`, etc.), parse into `(registry, repo, tag)`.
- Authenticate against the registry's token endpoint (anonymous for Docker Hub public images; bearer-token flow for GHCR public; configurable auth for private — but private-registry creds are deferred to a follow-up; this stage targets public images only).
- HEAD the manifest endpoint: `https://<registry>/v2/<repo>/manifests/<tag>` with `Accept: application/vnd.docker.distribution.manifest.v2+json, application/vnd.oci.image.manifest.v1+json` (both formats supported).
- Read the `Docker-Content-Digest` response header → that's the "latest digest" for the tag.
- Compare against the local image's `RepoDigest` (from the socket collector's data, STAGE-003-004).
- Bounded: HTTP timeout (default 10s), retry with exponential backoff capped at 60s on transient errors (5xx, network), no retry on auth failures (4xx).
- Rate-limit aware: Docker Hub anonymous = 100 pulls / 6h per source IP. The client tracks calls per registry and emits `homelab_registry_rate_limit_remaining{registry}` metrics. When the remaining count is low (<10), the collector logs + slows scrape interval.

### 2. `ImageUpdateCollector` — scheduler-registered collector

New module `apps/monitor/homelab_monitor/kernel/metrics/image_update_collector.py`:

- Tick interval: 6 hours (configurable via `HOMELAB_MONITOR_IMAGE_UPDATE_INTERVAL_SECONDS`). Rationale: Docker Hub rate limits + most images update at most daily.
- For each running container with a `RepoDigest` (from STAGE-003-003's socket collector data) AND an image reference that contains a `:tag` (not just `:latest` — Design phase considers whether to support `:latest`):
  - Call `RegistryDigestClient.get_latest_digest(image_ref)`.
  - If the latest digest != local digest → emit `homelab_image_update_available{container,image,current_digest,latest_digest} 1`.
  - Else → emit `homelab_image_update_available{...} 0`.
  - Always emit `homelab_image_update_last_checked_seconds{container} <epoch>` so vmalert can detect stale checks.
- Skips:
  - Containers whose image reference is a local image (no registry).
  - Containers using `build:` (no `image:` tag — covered by STAGE-003-008).
  - Containers with `RepoDigest=None` (image was loaded from a tar, never pulled from a registry).
- Per spec §16, the collector emits self-metrics.
- Failure handling: a single image's digest check failing does NOT abort the whole tick (per-image try/except, mirroring the lesson from STAGE-002-013 reconciler).

### 3. vmalert rule

`deploy/vmalert/metrics/docker_image_updates.yml`:

```yaml
- alert: DockerImageUpdateAvailable
  expr: homelab_image_update_available == 1
  for: 30m
  labels:
    severity: info
    integration: docker
  annotations:
    summary: "Update available for {{ $labels.container }} ({{ $labels.image }})"
    description: "Current: {{ $labels.current_digest }}; latest: {{ $labels.latest_digest }}"
```

`for: 30m` prevents alert flap during rapid digest changes (e.g., a maintainer pushing two images in a few minutes).

## Locked Design Decisions

D-REGISTRY-READ-ONLY — image-update detection is PURELY READ. No `docker pull`, no container changes, no disk writes beyond metric + per-image last-checked state. The PULL is STAGE-003-009.

D-DIGEST-COMPARISON — comparison is by `RepoDigest` (the cryptographic content hash), NOT by tag string. Two different tags with the same digest are "equivalent" (no update needed). A tag's digest changing IS the signal.

D-PUBLIC-IMAGES-V1 — v1 targets public registries (Docker Hub, GHCR, ghcr.io, quay.io, registry.k8s.io). Private-registry credentials are a follow-up not addressed by this stage.

D-PER-IMAGE-FAILURE-ISOLATION — a single image's digest check failing does NOT abort the whole tick. Per-image try/except, lesson from STAGE-002-013.

D-RATE-LIMIT-AWARE — the client tracks Docker Hub pull-count remaining (`RateLimit-Limit` / `RateLimit-Remaining` headers) and slows the collector tick when low. Metric `homelab_registry_rate_limit_remaining{registry}` surfaces this in Grafana.

D-INFO-SEVERITY — image-update alerts are `info` severity. The user opted into routing rules per spec §8.3; info defaults to dashboard-only (Karma-only). Not paging.

## Additional Locked Decisions (Design phase, 2026-05-23)

D-STATE-SIDECAR-TABLE — per-container image-update check state lives in a NEW sidecar table `image_update_state` (migration 0026), keyed by `container_name TEXT PRIMARY KEY` with columns:
  - `last_local_digest TEXT NULL` (RepoDigest cached from socket inspect)
  - `last_registry_digest TEXT NULL` (most recent HEAD result)
  - `last_image_ref TEXT NOT NULL` (the parsed ref the loader queried)
  - `last_checked_at TEXT NULL`
  - `check_failed_at TEXT NULL`
  - `check_error_reason TEXT NULL` (one of: `parse_failed`, `network_error`, `auth_failed`, `rate_limited`, `not_found`)
  - `update_available INTEGER NOT NULL DEFAULT 0` (denormalized cache; metric source of truth is the gauge)
Mirrors the `docker_override_ownership` precedent from STAGE-003-007. Different lifecycle from `targets_docker` (6h check writes vs. discovery-time target writes). Survives target re-keys cleanly (keyed by container_name).

D-LATEST-ALWAYS-CHECK — `:latest`, `:main`, `:edge`, `:stable`, and similar rolling tags ARE checked. Most homelab containers (linuxserver/*, foundry:release, plex:latest, etc.) use these; skipping defeats the feature. Info-severity (D-INFO-SEVERITY) bounds notification noise to Karma-only. Operator opt-out via override file is deferred (YAGNI; no evidence of need). Tags that are clearly unparseable (`<none>`, raw `sha256:...` digest pins with no tag) ARE skipped because there's nothing to query.

D-RATE-LIMIT-HARD-CAP-PLUS-BANNER — when Docker Hub `RateLimit-Remaining` drops below 10 for the next-tick window, the collector SKIPS remaining checks for that registry and:
  1. Emits gauge `homelab_registry_rate_limit_remaining{registry}` (machine-queryable, vmalert-rule-able)
  2. Emits counter `homelab_image_update_check_skipped{reason="rate_limit"}`
  3. The UI summary endpoint surfaces a `rate_limit_skipped_count: int` field; `ContainerGrid` renders a non-dismissible yellow banner ("X containers skipped due to Docker Hub rate limit; will retry next tick"). Auto-clears on next successful tick.
Tick interval stays at 6h default. No exponential backoff (silent degradation). Defense-in-depth: defensive + operator-visible + machine-queryable.

D-BADGE-TEXT-PLUS-BLUE-PILL — `ImageUpdateBadge` renders as a blue-pill link `<Link className="rounded bg-blue-50 px-2 py-0.5 text-xs text-blue-800 hover:underline">Update available</Link>`. Click-through to `/integrations/docker/containers/$name/image-update` (sibling of probes drill-down). Extends STAGE-003-007's severity-pill vocabulary: red=Config error, blue=Update available. Renders in BOTH `ContainerGridRow.tsx` (desktop) AND `ContainerGridCard.tsx` (mobile) per STAGE-003-005 mobile-parity rule. The "Image Update" column scaffolding slot in those files (`SCAFFOLDING: STAGE-003-008/009`) is filled by THIS stage; STAGE-003-009 will extend for locally-built images.

D-EVENTS-FILTERS-KWARG — `DockerSocketClient.events()` gains `filters: dict[str, list[str]] | None = None` kwarg. Default `None` preserves STAGE-003-005's existing single-stream container-events subscription. The new image-update collector subscribes a SECOND background task in lifespan with `filters={"type": ["image"]}` — on `image pull` events, it invalidates the cached digest for affected containers and triggers an out-of-band check on the next tick. Smallest surface change; backward compatible.

D-LABEL-KEY-NAME — image-update metrics use `name=` label key for container (NOT `container=`). Matches STAGE-003-004's existing convention (`homelab_container_status{name="..."}`). Operator can `name`-JOIN metrics across domains in PromQL without `label_replace()`. Stage spec's `{container,...}` label set is updated to `{name, image, current_digest, latest_digest}` accordingly.

D-SUMMARY-SIBLING-ENDPOINT — NEW endpoint `GET /api/integrations/docker/image-updates/summary` returning:
```
{
  "summaries": [
    { "container_name": str, "available": bool, "current_digest": str | null,
      "latest_digest": str | null, "last_checked_at": str | null,
      "check_error_reason": str | null }
  ],
  "rate_limit_skipped_count": int,
  "rate_limit_remaining_by_registry": { "docker.io": int, ... }
}
```
Sibling of `/probes/summary`. Does NOT extend `/probes/summary` (keeping summary endpoints single-purpose). React Query parallelizes the two grid summary fetches.

D-PER-CONTAINER-ENDPOINT — NEW endpoint `GET /api/integrations/docker/containers/{name}/image-update` returns the full row for one container's image-update state (consumed by the drill-down route at `/integrations/docker/containers/$name/image-update`).

D-IMAGE-REF-PARSER-INHOUSE — image-reference parsing (split into registry, repo, tag, digest) is implemented in-house in `apps/monitor/homelab_monitor/kernel/docker/image_ref_parser.py` (~30 LoC). No new dependency. Handles: bare images (`postgres:16` → docker.io/library/postgres:16), namespaced (`pihole/pihole:2025.05`), full refs (`ghcr.io/foo/bar:v1.0@sha256:...`), digest-pinned. Test coverage 100%.

D-COLLECTOR-CADENCE-6H — `ImageUpdateCollector` tick interval default 6h via `HOMELAB_MONITOR_IMAGE_UPDATE_INTERVAL_SECONDS` env var. Rationale: Docker Hub rate-limit window is 6h; most images update at most daily. Operator can lower for testing; spec discourages going below 1h.

### Lessons learned (this stage)

- **RTK output filter silently swallowed `make verify` output** mid-stage, producing 0-byte log files that looked like indefinite hangs. The verifier subagent retried for 45 minutes against blank logs before the loop was caught. Root cause: `rtk hook claude` in the user's `~/.claude/settings.json` wrapping every Bash call through the RTK filter; the project-local filter passed sha256 trust but the output transformation munged everything to empty. Resolution: removed the hook entry globally (backup at `settings.json.bak`). Lesson: when verify logs are 0 bytes, the FIRST hypothesis must be "the shell wrapper is eating output", NOT "the tests are still running".
- **`_image_events_loop` busy-looped on mock streams**: a mocked Docker events stream that yields no events and exits cleanly fell through `async for` immediately and re-entered the `while True` loop with no backoff. In tests this presented as the entire pytest suite hanging at 14 minutes (the loop kept re-subscribing). Fixed with the same exponential backoff used on the exception path. Lesson: any `while True: async for x in stream():` must treat clean-stream-exit AS WELL AS exception-exit with backoff — otherwise mock or empty streams produce a CPU-bound busy loop. Future event-loop background tasks must have this pattern by default.
- **`isinstance` over `is not None` when the runtime type is `AsyncMock`**: `cooldown_until_for(registry)` returns `float | None` in production. In tests with `AsyncMock`, `is not None` is `True` because `MagicMock` is not `None` — leading to `TypeError: '<' not supported between 'float' and 'MagicMock'`. The fix is `isinstance(cooldown_until, (int, float))`, which both type-narrows AND defends against the mock case. Same pattern for `Retry-After` (httpx returns `str | None` but AsyncMock returns coroutines). Lesson: code that gets exercised through `AsyncMock` in tests should narrow on the EXPECTED type, not on `is not None`.
- **Docker Hub API host ≠ canonical name**: `parse_image_ref("pihole/pihole:latest")` correctly returns `registry=docker.io`, but the manifest API endpoint is `https://registry-1.docker.io/v2/...`, NOT `https://docker.io/v2/...` (the latter 302-redirects to the marketing site, returning 403). Caught only by 3a e2e. Lesson: when adopting a registry, distinguish "canonical name (for image-ref user-facing display)" from "API host (for the HTTP client)" — the registry config table now encodes both.
- **Verifier subagent off-task twice**: modified code without permission during verification. Future verifier prompts must STRICTLY enumerate forbidden tools (Edit / Write / NotebookEdit). For this stage we caught it via `git diff` review; in a worse session it would have been a silent regression. Carried lesson from STAGE-003-007.
- **Shared `formatDigest()` helper** (apps/ui/src/lib/digest.ts) was extracted ONCE during Refinement and consumed in BOTH the drill-down AND the main grid `container.image` cell. Resists the dual-codepath drift that would otherwise show different truncation rules in different views. Lesson: when a formatting helper would otherwise be inlined in two places, extract on first use, NOT on second.
