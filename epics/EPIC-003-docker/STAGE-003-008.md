# STAGE-003-008: Image-update detection (locally-built images) — build-context source hash

## Status: Not Started

## Overview

For containers whose compose entry uses `build:` (locally built; no `image:` tag from a registry), digest-comparison is meaningless. Instead, hash the build-context source tree and compare against the stored hash from the last build. When the source changes but no rebuild has happened, emit the same `homelab_image_update_available` metric so the same alert + UI badge mechanics from STAGE-003-007 cover both cases uniformly.

Per `reference_docker_inventory.md`, several user containers use `build:` (e.g., `nginx-configuator`, custom services). This stage handles them.

**Vertical slice:** editing a file under a locally-built container's `build.context` directory (e.g., `nginx-configuator/Dockerfile`) without rebuilding the image causes the drill-down to show "Source changed — rebuild needed" within one tick. The alert reuses the STAGE-003-007 vmalert rule (same metric family).

**Spec references:** §2 Q13 (Docker monitoring decision); `reference_docker_inventory.md` (locally-built images list); EPIC-003.md note in `## Notes` on the build-context source-hash strategy.

## Concrete deliverables

### 1. Build-context source hasher

New module `apps/monitor/homelab_monitor/kernel/docker/source_hash.py`:

- Given a `build.context` path (absolute path on host, mounted into the monitor at `/host-compose/<relative-path>`), recursively hash all files under it.
- Respects `.dockerignore` semantics (same exclusion rules `docker build` uses — `dockerignore` library or custom parser).
- Returns a single content hash (SHA-256 of concatenated file-path + file-content hashes, sorted by path for determinism).
- Bounded:
  - Max file size: 10 MB per file (configurable via `HOMELAB_MONITOR_BUILD_HASH_MAX_FILE_BYTES`). Larger files skipped + logged + counted in metrics.
  - Max total bytes hashed: 1 GB per build-context. Exceeded → log + emit a special "context too large" hash that always-differs (forces user attention) + a self-metric.
  - Max file count: 50,000. Exceeded → same handling.
  - Max recursion depth: 30.
- Per-file failure (permission denied, symlink to /dev/random, etc.) → skipped + logged + included in self-metric.

### 2. `LocalBuildUpdateCollector` — scheduler-registered

New module `apps/monitor/homelab_monitor/kernel/metrics/local_build_update_collector.py`:

- Tick interval: 30 minutes (configurable). Faster than registry-digest because hashing is cheap (no network, no rate limit) and source changes happen more often than registry pushes.
- For each container with `build:` in its compose entry (parsed from the read-only compose mount — requires reading the compose file, which is the FIRST stage to do this; previous stages used socket-only data):
  - Resolve the `build.context` to a host path. Account for `${COMPOSE_DIR}` and relative paths.
  - Hash the source tree via `source_hash.compute(...)`.
  - Compare against the stored hash for this container from the last container build time (the local image's `Config.Labels["org.opencontainers.image.created"]` from the socket data tells us when the image was built; we store the source-hash at that point if known, else mark "unknown previous hash").
  - If hash differs → emit `homelab_image_update_available{container,source="local_build",current_hash,latest_hash} 1`. (Same metric family as STAGE-003-007 but with `source="local_build"` label so the vmalert rule can disambiguate if needed.)
  - Else → emit 0.
- Per spec §16, emit self-metrics.

### 3. Source-hash storage

Need somewhere to store the "last known hash" per container. Two options (Design phase chooses):

- **In SQLite** — new `docker_build_hashes` table or extend `targets`.
- **In a label on the image itself** — `homelab-monitor.source-hash` label added by the build process. Requires the user's build to set this label.

The second option is more elegant but requires the user to add the label to their builds. The first is more robust and the recommended starting point. The user's host-overrides repo can later add a build hook that writes the label, allowing the second approach if desired.

### 4. Compose file reading — first time in EPIC-003

`apps/monitor/homelab_monitor/kernel/docker/compose_reader.py`:

- Reads the compose file at `docker.compose_file_path` (default unset; the user's deployment sets `/storage/docker/compose/docker-compose.yml` via host-overrides config).
- Parses with `pyyaml`.
- Extracts per-service: `image`, `build.context`, `build.dockerfile`, `profiles`, `network_mode`, `labels`.
- Read-only — never writes (per EPIC-003 cross-stage criterion #6).
- Failure handling: file not present → emit `homelab_docker_compose_readable{} 0` + log; the collector emits no per-container hashes (graceful degradation).

This module will be reused by STAGE-003-009 for Pull & Restart action targets.

### 5. UI — build-source variant

`apps/ui/src/components/docker/badges.tsx` — extend `ImageUpdateBadge` to render a different copy for `source="local_build"` ("Source changed — rebuild needed") vs `source="registry"` ("Update available — `docker compose pull`").

`apps/ui/src/components/docker/ContainerGridRow.tsx` — the hover-tooltip in the Image Update column shows the appropriate detail per source.

### 6. Tests

- `test_source_hash.py` — determinism (same tree → same hash; different tree → different hash); `.dockerignore` exclusion correctness; max-file-size handling; max-total-bytes handling; max-file-count handling; symlink handling; permission-denied handling.
- `test_local_build_update_collector.py` — hash matches → no metric; hash differs → metric emitted; missing `build.context` directory → graceful failure; per-container isolation.
- `test_compose_reader.py` — parse a known-good compose file; missing file → graceful; malformed YAML → graceful.
- UI tests for the build-source badge variant.

## Out of scope for this stage

- Pull & Restart action (STAGE-003-009) — this stage detects "source changed"; the rebuild action happens later.
- Private registry creds for STAGE-003-007 — still deferred.
- Automatic rebuild on source change — never. The user decides if + when to rebuild.
- Source-tree DIFF rendering (e.g., "which files changed since the last build") — not in v1; metric just flags the difference.

## Decisions in this stage

The Design phase confirms the source-hash storage location (SQLite vs image-label), the .dockerignore parser library, the bounded hashing limits, and the compose-reader's path-resolution rules (especially `${COMPOSE_DIR}` and home-dir expansion).

## Locked Design Decisions

D-SHARED-METRIC-FAMILY — `homelab_image_update_available` is the SAME metric for registry-pulled and locally-built images, differentiated by the `source` label (`registry` | `local_build`). Same vmalert rule covers both. UI badge varies copy.

D-COMPOSE-READER-READ-ONLY — the new compose reader is STRICTLY read-only. Per cross-stage criterion #6, this is non-negotiable.

D-HASHING-BOUNDED — every layer of the source-hasher has explicit bounds. A pathological build-context (1 TB of files, infinite-depth recursion) must NOT crash the collector. Limits are explicit and configurable.

D-SQLITE-FIRST-HASH-STORE — v1 stores last-known-hash in SQLite (not image labels). The image-label approach is preserved as a future option but defers to a follow-up.
