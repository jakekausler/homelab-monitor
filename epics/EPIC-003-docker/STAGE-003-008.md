# STAGE-003-008: Image-update detection (registry digest)

## Status: Not Started

## Overview

Detect when a newer image is available upstream for any container whose compose entry uses `image:` (pulled from a registry). Periodically compare the local image's `RepoDigest` against the digest the registry serves for the container's tag. When they differ, emit `homelab_image_update_available{container,image,current_digest,latest_digest} 1`. A vmalert rule fires an `info`-severity alert; the drill-down grid shows an "Update available" badge.

This is the diun-style mechanism from spec Q13. It is READ-ONLY — no `docker pull` happens here (that's STAGE-003-009 "Pull & Restart"). Until then, only the *knowledge that an update exists* is recorded.

**Vertical slice:** when a container's upstream tag publishes a new image (e.g., Pi-hole 2025.05 → 2025.06), within one tick the drill-down grid shows an "Update available" badge on the Pi-hole row, vmalert emits an info-severity alert, and the dispatcher routes per the user's info-channel rules.

**Spec references:** §2 Q13 (diun-style update notifications); §3.4 (container image registries as a discovered target type); §6.2 (`homelab_image_update_available{container}` metric); §7.5 (network model — outbound to Docker Hub / GHCR for digest checks).

## Concrete deliverables

### 1. Registry client — `RegistryDigestClient`

New module `apps/monitor/homelab_monitor/kernel/docker/registry_digest_client.py`:

- Given an image reference (`ghcr.io/blakeblackshear/frigate:0.14.0`, `pihole/pihole:2025.05`, etc.), parse into `(registry, repo, tag)`.
- Authenticate against the registry's token endpoint (anonymous for Docker Hub public images; bearer-token flow for GHCR public; configurable auth for private — but private-registry creds are deferred to a follow-up; this stage targets public images only).
- HEAD the manifest endpoint: `https://<registry>/v2/<repo>/manifests/<tag>` with `Accept: application/vnd.docker.distribution.manifest.v2+json, application/vnd.oci.image.manifest.v1+json` (both formats supported).
- Read the `Docker-Content-Digest` response header → that's the "latest digest" for the tag.
- Compare against the local image's `RepoDigest` (from the socket collector's data, STAGE-003-004).
- Bounded: HTTP timeout (default 10s), retry with exponential backoff capped at 60s on transient errors (5xx, network), no retry on auth failures (4xx).
- Rate-limit aware: Docker Hub anonymous = 100 pulls / 6h per source IP. The client tracks calls per registry and emits `homelab_registry_rate_limit_remaining{registry}` metrics. When the remaining count is low (<10), the collector logs + slows scrape interval.

### 2. `ImageUpdateCollector` — scheduler-registered collector

New module `apps/monitor/homelab_monitor/kernel/metrics/image_update_collector.py`:

- Tick interval: 6 hours (configurable via `HOMELAB_MONITOR_IMAGE_UPDATE_INTERVAL_SECONDS`). Rationale: Docker Hub rate limits + most images update at most daily.
- For each running container with a `RepoDigest` (from STAGE-003-003's socket collector data) AND an image reference that contains a `:tag` (not just `:latest` — Design phase considers whether to support `:latest`):
  - Call `RegistryDigestClient.get_latest_digest(image_ref)`.
  - If the latest digest != local digest → emit `homelab_image_update_available{container,image,current_digest,latest_digest} 1`.
  - Else → emit `homelab_image_update_available{...} 0`.
  - Always emit `homelab_image_update_last_checked_seconds{container} <epoch>` so vmalert can detect stale checks.
- Skips:
  - Containers whose image reference is a local image (no registry).
  - Containers using `build:` (no `image:` tag — covered by STAGE-003-008).
  - Containers with `RepoDigest=None` (image was loaded from a tar, never pulled from a registry).
- Per spec §16, the collector emits self-metrics.
- Failure handling: a single image's digest check failing does NOT abort the whole tick (per-image try/except, mirroring the lesson from STAGE-002-013 reconciler).

### 3. vmalert rule

`deploy/vmalert/metrics/docker_image_updates.yml`:

```yaml
- alert: DockerImageUpdateAvailable
  expr: homelab_image_update_available == 1
  for: 30m
  labels:
    severity: info
    integration: docker
  annotations:
    summary: "Update available for {{ $labels.container }} ({{ $labels.image }})"
    description: "Current: {{ $labels.current_digest }}; latest: {{ $labels.latest_digest }}"
```

`for: 30m` prevents alert flap during rapid digest changes (e.g., a maintainer pushing two images in a few minutes).

## Out of scope for this stage

- Locally-built image update detection (STAGE-003-009).
- Pull & Restart action (STAGE-003-010).
- Private-registry credentials. This stage handles public images only. Private registries (paid Docker Hub plans, private GHCR, self-hosted) are a follow-up — STAGE-003-009 may revisit if needed.
- Subscribing to registry webhooks instead of polling — registries' webhook offerings are uneven and unreliable; polling at 6h is the spec's default.
- Update-recommendation logic (e.g., "this is a minor version bump, this is a major one"). Just availability detection in v1.

## Decisions in this stage

The Design phase confirms the registry-call cadence, the `:latest` tag handling policy, the rate-limit-handling strategy, and the auth bootstrap for GHCR (anonymous works for public; private creds deferred).

## Locked Design Decisions

D-REGISTRY-READ-ONLY — image-update detection is PURELY READ. No `docker pull`, no container changes, no disk writes beyond metric + per-image last-checked state. The PULL is STAGE-003-009.

D-DIGEST-COMPARISON — comparison is by `RepoDigest` (the cryptographic content hash), NOT by tag string. Two different tags with the same digest are "equivalent" (no update needed). A tag's digest changing IS the signal.

D-PUBLIC-IMAGES-V1 — v1 targets public registries (Docker Hub, GHCR, ghcr.io, quay.io, registry.k8s.io). Private-registry credentials are a follow-up not addressed by this stage.

D-PER-IMAGE-FAILURE-ISOLATION — a single image's digest check failing does NOT abort the whole tick. Per-image try/except, lesson from STAGE-002-013.

D-RATE-LIMIT-AWARE — the client tracks Docker Hub pull-count remaining (`RateLimit-Limit` / `RateLimit-Remaining` headers) and slows the collector tick when low. Metric `homelab_registry_rate_limit_remaining{registry}` surfaces this in Grafana.

D-INFO-SEVERITY — image-update alerts are `info` severity. The user opted into routing rules per spec §8.3; info defaults to dashboard-only (Karma-only). Not paging.
