### 3. New API endpoints

`apps/monitor/homelab_monitor/kernel/api/routers/docker.py`:

- `POST /api/integrations/docker/suggestions/{id}/accept` — accept with default probe behavior. Returns the created `probe_targets` row(s).
- `POST /api/integrations/docker/suggestions/{id}/customize` — accept with user-specified probes. Body: `{"probes": [ProbeDescriptor[]]}`. Returns the created `probe_targets` row(s).
- `POST /api/integrations/docker/suggestions/{id}/ignore` — archive the suggestion.

All three are auth-required, audit-logged, and idempotent (re-accepting a suggestion is a no-op + 200; double-action doesn't crash).

## Design Decisions (Locked 2026-05-26)

These extend the spec's previously-locked decisions (D-IN-EPIC-PANEL-TEMPORARY, D-ACCEPT-DEFAULT-CONSERVATIVE, D-IGNORE-NOT-DELETE, D-CUSTOMIZE-PROBE-LIST, D-CARRY-FORWARD-EXPLICIT) with the remaining presentation-layer and API-shape choices locked at Design phase.

### D-ACCEPT-INVENTORY-ONLY-WITH-CONSERVATIVE-PROBES
Accept (without customizing) transitions the suggestion to `state="accepted"` and inserts probe_targets ONLY when BOTH of these are true: the container has at least one exposed port (`HostConfig.PortBindings` or `NetworkSettings.Ports` non-empty in docker inspect) AND the container has a healthcheck defined. In that case, insert a minimal conservative set: one `tcp` probe per exposed port + one `exec` probe wrapping the healthcheck command. Otherwise, Accept ONLY transitions suggestion state — no probe rows created. User can add probes later via Customize. This implements D-ACCEPT-DEFAULT-CONSERVATIVE.

### D-CUSTOMIZE-FORM-ROWS
Customize modal renders a list of probe rows. User clicks "Add probe" to add another row; clicks "Remove" on a row to delete it. Each row contains: kind dropdown (`http | tcp | exec | metrics`), name (text input, required, unique within container), target_value (text input, required — URL for http, host:port for tcp, command for exec, scrape URL for metrics), interval_seconds (number input, default 60), timeout_seconds (number input, default 10). Per-row client-side validation (required fields + unique name). Form submits the full list to `POST /api/integrations/docker/suggestions/{id}/customize`. Backend upserts probe_targets row per item. No JSON editor; no auto-fill from container metadata in this stage (could be added in EPIC-011 customize flow).

### D-EPIC-011-CODE-COMMENTS-ONLY
EPIC-011 cross-references live in: (1) code comments at the panel + hook + endpoint sites, (2) `epics/EPIC-003-docker/regression.md` under a "Cross-epic carry-forward to EPIC-011" section, (3) augmented `epics/EPIC-003-docker/EPIC-003.md` carry-forward section, (4) NEW carry-forward note in `epics/EPIC-011-discovery-suggestions/EPIC-011.md` so the inbound side knows about the deferred decisions. NO user-visible UI banner about the temporary stub — keeps the UI clean. Matches the pattern used in STAGE-003-011 (EPIC-004 cross-references).

### D-IDEMPOTENT-200
All three POST endpoints (`/suggestions/{id}/accept`, `/customize`, `/ignore`) are idempotent at the REST level: repeated calls on a suggestion already in the target state return 200 OK with the current state. No 409 Conflict for state-already-matches. Specifically:
- Accept on already-accepted → 200, no duplicate probe_target rows (relies on `upsert_probe_target_conn` UNIQUE constraint on `(container_name, kind, name)`).
- Ignore on already-ignored → 200 no-op.
- Customize on already-accepted → 200, probe_targets upserted with new config (intentionally allows the user to update probe config via re-running Customize).
- Customize on a `state="ignored"` or `state="container_gone"` suggestion → 409 Conflict (can't customize a non-pending non-accepted suggestion); user must un-ignore first. (Un-ignore is out of scope this stage — see Out of Scope below.)

### D-CUSTOMIZE-MODAL-MOBILE-FALLBACK
Customize modal is a full-screen sheet on mobile viewports (< sm breakpoint) and a centered modal on desktop. Uses the same components as the existing Pull-and-Restart confirmation modal in `ContainerGridRow.tsx`'s ActionsCell. No new modal infrastructure.

### D-CUSTOMIZE-RESERVED-NAMES
Probe names must match `^[a-zA-Z0-9_-]{1,64}$` (server-validated). Client also rejects empty / whitespace-only. Duplicate-name-within-form rejected client-side; backend defense-in-depth checks the UNIQUE constraint on `(container_name, kind, name)` in probe_targets.

### D-IGNORE-FROM-ANY-STATE
Ignore endpoint accepts a suggestion in ANY state (pending, accepted, container_gone, even already-ignored). It transitions to `state="ignored"`. If the suggestion was previously accepted, the existing `probe_targets` rows are NOT automatically deleted (D-IGNORE-NOT-DELETE generalized: the probe_targets cleanup is a separate concern, owned by the user via the probe-disable UI from STAGE-003-007). Documented in code so reviewers don't expect cascading delete.

### D-NO-PROBE-MISSING-ON-IGNORE
Per D-IGNORE-FROM-ANY-STATE, ignoring an accepted suggestion does NOT touch probe_targets. Future enhancement (EPIC-011): a "remove probes too?" confirmation on Ignore-after-Accept could be added. Out of scope this stage.
