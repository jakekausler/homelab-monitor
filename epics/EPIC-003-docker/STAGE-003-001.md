# STAGE-003-001: cadvisor sidecar + scrape config + first container metrics in VM

## Status: Complete

## Overview

Deploy cadvisor as a sidecar in prod compose, dev compose, and the integration test rig. Wire it into the vmagent scrape config so cadvisor's per-container resource metrics (CPU / RAM / network / disk / filesystem) land in VictoriaMetrics. Add a dev-only seed CLI that emits synthetic `container_*` metrics so the UI drill-down can be exercised during Refinement without spinning up real workloads.

This is the foundation stage of EPIC-003. No `apps/monitor` business logic yet — pure ops + a small CLI. The Grafana embed in the existing `/metrics` screen will be the only user-visible surface this stage.

**Vertical slice:** after `make dev`, opening Grafana shows live container CPU/RAM/IO metrics for every container running on the host. Dev-mode users can also run `hm dev seed-container-metrics` to inject synthetic metrics for the UI drill-down planned in STAGE-003-003.

## Concrete deliverables

### 1. cadvisor sidecar in prod compose

`deploy/compose/docker-compose.yml` — add a `cadvisor` service:

- Image: `gcr.io/cadvisor/cadvisor:v0.49.1` (or current stable; pinned in Design phase).
- Mounts: `/:/rootfs:ro`, `/var/run:/var/run:ro`, `/sys:/sys:ro`, `/var/lib/docker/:/var/lib/docker:ro`, `/dev/disk/:/dev/disk:ro`, `/var/run/docker.sock:/var/run/docker.sock:ro`.
- Container-internal port `8080` (cadvisor default). NOT exposed on host in prod — the monitor proxies via `/api/cadvisor/` if needed (no UI surface yet this stage).
- Privileged: false; cap_add: `SYS_ADMIN` if required by cadvisor on this kernel (verified in Build).
- On the `homelab-monitor-net` docker network.

### 2. cadvisor sidecar in dev compose

`deploy/dev/docker-compose.dev.yml` — same cadvisor service. Published on dev port `18081` (per dev `1xxxx` invariant; final port confirmed in Design).

### 3. cadvisor sidecar in integration test rig

`deploy/compose/docker-compose.test.yml` — cadvisor service scraping the test rig's toy containers. Integration tests can assert specific `container_*` metric shapes against deterministic toy workloads.

### 4. vmagent scrape config update

`deploy/vmagent/scrape.yml` (or the equivalent monitor-rendered scrape config — see §3.2 spec) — add a scrape target for cadvisor with appropriate `relabel_configs`:

- Drop noisy series (per-container per-filesystem metrics for filesystems we don't care about — `overlay`, `tmpfs` shims).
- Keep `container_cpu_*`, `container_memory_*`, `container_network_*`, `container_fs_*`, `container_last_seen`.
- Label `container_label_homelab_monitor_*` extracted as `homelab_monitor_*` for downstream label-driven probe wiring (consumed by STAGE-003-005).

### 5. Dev seed CLI — `hm dev seed-container-metrics`

`apps/monitor/homelab_monitor/cli/dev.py` — extend the existing `dev` CLI group (added in STAGE-002-015):

- New subcommand `seed-container-metrics [--containers N] [--clear]`.
- Local-only (hostname gate, same as `seed-cron-runs`).
- Writes synthetic `container_*` time-series directly to VictoriaMetrics via the existing `MetricsWriter` (the same primitive STAGE-001-015 uses).
- Generates N fake containers (default 5) with realistic-shape data: CPU sawtooth, memory step, network bursts. Deterministic seeds so runs are reproducible.
- `--clear` drops the synthetic series (uses VM's series-delete API).
- Tested with the same fixture pattern as `test_cli_dev_seed_cron_runs.py`.

### 6. Tests

- `apps/monitor/tests/test_cli_dev_seed_container_metrics.py` — unit tests for the seed CLI: hostname gate, generated series shape, idempotent re-run, `--clear` behavior.
- `apps/monitor/tests/integration/test_cadvisor_scrape.py` (or equivalent under the existing integration test pattern) — asserts that the cadvisor sidecar in the test rig produces at least one expected `container_*` metric in VM within a bounded scrape interval. Uses the toy containers from `docker-compose.test.yml`.
- 100% kernel coverage maintained.

## Locked Design Decisions

D-CADVISOR-FIRST — cadvisor lands BEFORE the socket collector (STAGE-003-004). Resource metrics (CPU/RAM/IO/network) come from cadvisor; the socket collector fills the gaps cadvisor can't see (status, restart count, exit code, healthcheck status, image labels). No throwaway code.

D-CADVISOR-SIDECAR-EVERYWHERE — cadvisor sidecar lands in prod compose, dev compose, AND the integration test rig. Dev published on `1xxxx` port; prod container-internal only; test rig container-internal scraped by test-rig vmagent.

D-DEV-SYNTH-METRICS — dev rig has a `hm dev seed-container-metrics` CLI so frontend Refinement works without real cadvisor data. Pattern mirrors `seed-cron-runs` from STAGE-002-015. Local-only (hostname gate).

D-PROBE-RW-DEFERRED — the docker socket mount is `:ro` in this stage and remains so until STAGE-003-010 (where Pull & Restart needs RW).

D-CADVISOR-VERSION — Pin to `gcr.io/cadvisor/cadvisor:v0.49.1` (tag pin). Build phase MAY promote to a digest pin if an audit of other sidecars in `deploy/compose/docker-compose.yml` reveals digest-pinning is the established convention; otherwise tag pin is canonical for cadvisor.

D-SCRAPE-HYBRID — Hybrid scrape config: a static base `deploy/vmagent/scrape.yml` covers the cadvisor sidecar (plus existing sidecars) AND declares a `file_sd_configs: targets.d/*.yml` extension point. The `targets.d/` directory is empty in this stage; STAGE-003-005+ writes monitor-rendered dynamic targets into it. Honors spec §3.2's "scrape config generated by the monitor" without paying for the rendering machinery yet.

D-CADVISOR-RELABEL-CURATED — Scrape config keeps a curated allowlist of ~13 metric families (`container_cpu_usage_seconds_total`, `container_cpu_cfs_throttled_seconds_total`, `container_memory_working_set_bytes`, `container_memory_usage_bytes`, `container_memory_rss`, `container_network_receive_bytes_total`, `container_network_transmit_bytes_total`, `container_network_receive_errors_total`, `container_network_transmit_errors_total`, `container_fs_reads_bytes_total`, `container_fs_writes_bytes_total`, `container_last_seen`, `container_start_time_seconds`). PLUS a relabel rule drops `container_fs_*` series for filesystems matching `device=~"(overlay|tmpfs|shm|/dev/loop.*)"`. The allowlist is documented inline in `scrape.yml` so future stages know where to extend.

D-SEED-CLI-DETERMINISTIC — `hm dev seed-container-metrics [--containers N] [--clear]`. Default `N=5`. Each fake container gets a distinct realistic shape (CPU sawtooth, memory step, network bursts, idle, spiky) driven by a seeded RNG keyed on the run inputs — same inputs produce same series for reproducibility. Series are labeled `homelab_synthetic="true"` so `--clear` (via VM `/api/v1/admin/tsdb/delete_series`) targets ONLY synthetic series and never touches real cadvisor data. Mirrors the `seed-cron-runs` pattern from STAGE-002-015.

D-CADVISOR-LEAST-PRIV — cadvisor sidecar starts with `privileged: false` and NO extra capabilities. Build phase may promote to `cap_add: [SYS_ADMIN]` ONLY if specific metrics are verifiably missing and adding SYS_ADMIN resolves the gap. Never `privileged: true`.

D-CADVISOR-FLAGS — cadvisor's command line includes: `--docker_only=true` (skip non-Docker cgroups + kernel threads), `--store_container_labels=false`, `--whitelisted_container_labels=homelab-monitor.*,org.label-schema.*,com.docker.compose.*`. The whitelisted label list MUST include `homelab-monitor.*` so STAGE-003-005's label-based probe discovery can use them as Prometheus labels.

D-TEST-RIG-TOYS — `deploy/compose/docker-compose.test.yml` includes a minimal `nginx:alpine` + `busybox sleep infinity` workload pair, each pinned by digest for deterministic CI. If suitable toy containers already exist from EPIC-001/002 test rig setup, reuse those; Build phase confirms this in step 1.

D-DEV-PORT-18081 — cadvisor dev-rig host port is `18081` (matches the `1xxxx` invariant; 18080 is taken by Karma). Bound to `127.0.0.1` per the localhost-only invariant.
