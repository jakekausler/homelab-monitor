# STAGE-003-005: Docker discoverer + suggestions data ‚Äî periodic + socket-event-driven

## Status: Complete

## Overview

Build the Docker discoverer plugin. Watches the Docker socket for events (`container create`, `container destroy`) and emits suggestions for newly-seen containers to the `suggestions` table. Also runs a periodic full-scan tick that catches containers the event stream missed (kernel restart, missed events). This is the FIRST live exercise of the discoverer plugin contract from STAGE-001-006 ‚Äî small refinements to that contract may surface and feed back into EPIC-001 regression notes.

(Image-pull event handling is deferred to STAGE-003-008 / STAGE-003-009 ‚Äî see Drift #1 in Locked Drift Resolutions below.)

The discoverer is the entry point for label-driven probe creation (STAGE-003-006 consumes the discoverer's output to create probe-target rows when `homelab-monitor.*` labels are present). For containers WITHOUT labels, the discoverer emits a suggestion the user must explicitly accept ‚Äî the suggestions data lands in the `suggestions` table this stage, and the in-epic suggestions panel from STAGE-003-003 starts populating. The Accept/Customize/Ignore wiring is STAGE-003-012 (and ultimately EPIC-011).

**Vertical slice:** `docker compose up -d <new-container>` produces a suggestion within seconds (socket event); the suggestion appears in the "Pending suggestions" panel of the Docker drill-down; `docker compose down <container>` does NOT remove the suggestion (the row goes `state="container_gone"` instead so the user knows what happened).

## Concrete deliverables

### 1. `DockerDiscoverer` ‚Äî event-driven + periodic plugin

New module `apps/monitor/homelab_monitor/plugins/discoverers/docker_discoverer.py`:

- Implements the `Discoverer` protocol from STAGE-001-006.
- TWO modes running concurrently:
  - **Event stream:** background task subscribes to `docker events --filter "type=container"` (or `socket_client.events()` if using `aiodocker`). On `create` event ‚Üí check if `targets` has this container with `kind="docker_container"`; if not, emit suggestion. On `destroy` event ‚Üí if there's a pending suggestion for this container, mark it `state="container_gone"`.
  - **Periodic full-scan:** every 5 minutes (configurable via `HOMELAB_MONITOR_DOCKER_DISCOVERER_SCAN_INTERVAL_SECONDS`), list all containers, check for any missing from the suggestion + targets state (catches missed events).
- Concurrency group: `docker.discoverer` (separate from `docker.socket` so the discoverer and the socket collector don't share locks).
- Event subscription is bounded: if the event stream drops, the discoverer logs + reconnects (with exponential backoff capped at 60s).

### 2. Suggestion payload shape

The `suggestions` table stores a JSON `payload` per row. For Docker discoveries, the shape is:

```json
{
  "kind": "docker_container_discovered",
  "container_id": "abc123",
  "container_name": "frigate",
  "image": "ghcr.io/blakeblackshear/frigate:0.14.0",
  "labels": { "homelab-monitor.http.api": "...", "...": "..." },
  "ports": [{"private": 5000, "public": 5000, "protocol": "tcp"}],
  "network_mode": "host",
  "profiles": ["disabled"],
  "first_seen_at": "2026-05-21T10:00:00+00:00",
  "has_homelab_monitor_labels": true,
  "is_disabled_profile": true
}
```

Per EPIC-003 cross-stage criterion #9, containers in `profiles: ["disabled"]` produce suggestions but with `is_disabled_profile=true` ‚Äî STAGE-003-005's label-config path SKIPS these. The user can still see them in the suggestions panel for context.

Per cross-stage criterion #11, label collisions (two `homelab-monitor.<kind>.<name>=...` labels resolving to the same `(kind, name)` for the same container) ‚Üí produce a suggestion with `kind="docker_label_collision"` + the collision details. No probe is created until the collision is resolved.

## Locked Design Decisions

**D-EVENT-STREAM-MECHANISM** ‚Äî Extend `DockerSocketClient.events()` with httpx streaming (`client.stream("GET", "/events?filters=...")`). Pure httpx-over-UDS pattern, no new deps. Read-timeout overridden to `None` for the long-lived stream. ~10 LoC; logic-heavy dispatch lives in `DockerDiscoverer` to keep `socket_client.py` thin and the discoverer fully unit-testable. T-DOCKER-SDK (rejected SDKs) stands.

**D-PERIODIC-EVENT-COORDINATION** ‚Äî Single `DockerDiscoverer` plugin instance spawns two concurrent `asyncio.Task`s in `start()`: `events_task` (long-lived `/events` subscription with exponential-backoff reconnect capped at 60s) and `periodic_task` (5-minute full-scan, interval configurable via `HOMELAB_MONITOR_DOCKER_DISCOVERER_SCAN_INTERVAL_SECONDS`). Both call a shared `_upsert_suggestion()` method protected by an `asyncio.Lock`. Periodic task is the unconditional reconciliation safety net. Single plugin ‚Üí single set of metrics, one lifecycle, one shutdown signal. `stop()` cancels both tasks and awaits gather with `return_exceptions=True`.

**D-SUGGESTIONS-SCHEMA-MIGRATION** ‚Äî Sidecar table pattern mirroring `targets`/`targets_docker`. New alembic migration 0019 creates:
  - Generic `suggestions` table: `id TEXT PRIMARY KEY, kind TEXT NOT NULL, deduplication_key TEXT NOT NULL, state TEXT NOT NULL DEFAULT 'pending', created_at TEXT NOT NULL, updated_at TEXT NOT NULL, UNIQUE (kind, deduplication_key)`. The existing 3-column stub from migration 0001 is migrated forward (additive or rebuild-via-temp-table as needed ‚Äî Build phase resolves the SQLite ALTER NOT NULL constraint mechanically).
  - Sidecar `suggestions_docker`: `suggestion_id TEXT PRIMARY KEY REFERENCES suggestions(id) ON DELETE CASCADE, container_id TEXT NOT NULL, container_name TEXT NOT NULL, image_ref TEXT NOT NULL, labels_json TEXT NOT NULL DEFAULT '{}', compose_project TEXT, compose_service TEXT, detection_reason TEXT NOT NULL`.
  - Repositories: `SuggestionsRepository` (anchor) and `SuggestionsDockerRepository` (sidecar) ‚Äî both follow the `TargetsRepository` shape (instance methods for queries, static `_conn`-suffixed helpers for transaction-safe writes inside `SqliteRepository.write_tx`).

**D-UI-PANEL-LAYOUT** ‚Äî Card list, vertical stack, identical desktop + mobile layout. Per-card content: container name (heading), image_ref (subtle), label badges, detection-reason pill ("No homelab-monitor.* label" or "Disabled profile" or "Label collision"), three scaffolded-disabled buttons in row (Accept / Customize / Ignore) with explicit comment `// SCAFFOLDING: STAGE-003-010 wires Accept/Customize/Ignore. EPIC-011 will subsume with global Suggestions inbox.`. Panel header reads "Pending suggestions (N)". Cursor pagination via a "Load more" button at the bottom. Edited in place at `apps/ui/src/routes/integrations/PendingSuggestionsPanel.tsx`.

**D-TEST-EVENT-STREAM-STRATEGY** ‚Äî Constructor injection: `DockerDiscoverer.__init__(self, socket_client: DockerSocketClient, ...)`. Unit tests inject a fake `DockerSocketClient` whose `events()` yields a controllable `AsyncIterator[dict]` and whose `list_containers()` returns a controllable list. No `pytest-httpx` mocking of the long-lived events stream. Lifespan-tick contamination of OTHER tests still requires the `is_optional=True` VM regex mock pattern (project memory: STAGE-003-004 lesson 2). Real `events()` HTTP path is validated by Refinement 3b against the host docker socket, not by unit tests. Keep `DockerSocketClient.events()` ‚â§ ~10 LoC to minimize uncovered code.
