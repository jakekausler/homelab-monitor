# STAGE-003-005: Docker discoverer + suggestions data — periodic + socket-event-driven

## Status: Not Started

## Overview

Build the Docker discoverer plugin. Watches the Docker socket for events (`container create`, `container destroy`, `image pull`) and emits suggestions for newly-seen containers to the `suggestions` table. Also runs a periodic full-scan tick that catches containers the event stream missed (kernel restart, missed events). This is the FIRST live exercise of the discoverer plugin contract from STAGE-001-006 — small refinements to that contract may surface and feed back into EPIC-001 regression notes.

The discoverer is the entry point for label-driven probe creation (STAGE-003-006 consumes the discoverer's output to create probe-target rows when `homelab-monitor.*` labels are present). For containers WITHOUT labels, the discoverer emits a suggestion the user must explicitly accept — the suggestions data lands in the `suggestions` table this stage, and the in-epic suggestions panel from STAGE-003-003 starts populating. The Accept/Customize/Ignore wiring is STAGE-003-012 (and ultimately EPIC-011).

**Vertical slice:** `docker compose up -d <new-container>` produces a suggestion within seconds (socket event); the suggestion appears in the "Pending suggestions" panel of the Docker drill-down; `docker compose down <container>` does NOT remove the suggestion (the row goes `state="container_gone"` instead so the user knows what happened).

## Concrete deliverables

### 1. `DockerDiscoverer` — event-driven + periodic plugin

New module `apps/monitor/homelab_monitor/plugins/discoverers/docker_discoverer.py`:

- Implements the `Discoverer` protocol from STAGE-001-006.
- TWO modes running concurrently:
  - **Event stream:** background task subscribes to `docker events --filter "type=container"` (or `socket_client.events()` if using `aiodocker`). On `create` event → check if `targets` has this container with `kind="docker_container"`; if not, emit suggestion. On `destroy` event → if there's a pending suggestion for this container, mark it `state="container_gone"`.
  - **Periodic full-scan:** every 5 minutes (configurable via `HOMELAB_MONITOR_DOCKER_DISCOVERER_SCAN_INTERVAL_SECONDS`), list all containers, check for any missing from the suggestion + targets state (catches missed events).
- Concurrency group: `docker.discoverer` (separate from `docker.socket` so the discoverer and the socket collector don't share locks).
- Event subscription is bounded: if the event stream drops, the discoverer logs + reconnects (with exponential backoff capped at 60s).

### 2. Suggestion payload shape

The `suggestions` table stores a JSON `payload` per row. For Docker discoveries, the shape is:

```json
{
  "kind": "docker_container_discovered",
  "container_id": "abc123",
  "container_name": "frigate",
  "image": "ghcr.io/blakeblackshear/frigate:0.14.0",
  "labels": { "homelab-monitor.http.api": "...", "...": "..." },
  "ports": [{"private": 5000, "public": 5000, "protocol": "tcp"}],
  "network_mode": "host",
  "profiles": ["disabled"],
  "first_seen_at": "2026-05-21T10:00:00+00:00",
  "has_homelab_monitor_labels": true,
  "is_disabled_profile": true
}
```

Per EPIC-003 cross-stage criterion #9, containers in `profiles: ["disabled"]` produce suggestions but with `is_disabled_profile=true` — STAGE-003-005's label-config path SKIPS these. The user can still see them in the suggestions panel for context.

Per cross-stage criterion #11, label collisions (two `homelab-monitor.<kind>.<name>=...` labels resolving to the same `(kind, name)` for the same container) → produce a suggestion with `kind="docker_label_collision"` + the collision details. No probe is created until the collision is resolved.

### 3. Suggestions repository

`apps/monitor/homelab_monitor/kernel/db/repositories/suggestions_repository.py` (new OR extension if it exists from EPIC-001's stub):

- `insert_or_update(...)` — upsert by `(kind, deduplication_key)` where `deduplication_key = container_id` for Docker suggestions. Re-seeing the same container does NOT spam suggestions.
- `list_pending_docker_suggestions()` — for the API endpoint.
- `mark_container_gone(container_id)` — sets `state="container_gone"` for pending suggestions of a destroyed container.

The `suggestions` table schema may need to grow this stage (`deduplication_key` column if not present). If migration is needed, follow the alembic pattern from STAGE-002-003.

## Locked Design Decisions

D-DISCOVERER-FIRST-LIVE — this is the FIRST live exercise of the discoverer plugin contract from STAGE-001-006. Refinements to that contract surface here and are captured as EPIC-001 regression items, NOT contract changes (the contract is stable).

D-EVENT-PLUS-SCAN — the discoverer uses BOTH event-driven AND periodic-scan paths. Event-driven is fast; periodic-scan is the safety net for missed events. Neither alone is sufficient.

D-NO-DUPE-SUGGESTIONS — re-seeing the same container does NOT create duplicate suggestions. Dedup key = `container_id`. A container removed and recreated with a new `container_id` IS a new suggestion (this matches the user's intuition).

D-DISABLED-PROFILE-SURFACED — containers in `profiles: ["disabled"]` are surfaced as suggestions BUT marked `is_disabled_profile=true`. STAGE-003-005's auto-config path skips them. (Per EPIC-003 cross-stage criterion #9.)

D-COLLISION-AS-SUGGESTION — label collisions produce a `docker_label_collision` suggestion, NOT a silent error. The user resolves by editing labels. No probe is created until resolution. (Per cross-stage criterion #11.)

D-EPIC-011-XREF-PANEL — the in-epic Pending Suggestions panel is TEMPORARY. EPIC-011 will subsume with the global Suggestions inbox. Code comment in `PendingSuggestionsPanel.tsx` makes this explicit.
