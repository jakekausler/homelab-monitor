# STAGE-003-009: Image-update detection (locally-built images) — build-context source hash

## Status: Complete

## Overview

For containers whose compose entry uses `build:` (locally built; no `image:` tag from a registry), digest-comparison is meaningless. Instead, hash the build-context source tree and compare against the stored hash from the last build. When the source changes but no rebuild has happened, emit the same `homelab_image_update_available` metric so the same alert + UI badge mechanics from STAGE-003-007 cover both cases uniformly.

Per `reference_docker_inventory.md`, several user containers use `build:` (e.g., `nginx-configuator`, custom services). This stage handles them.

**Vertical slice:** editing a file under a locally-built container's `build.context` directory (e.g., `nginx-configuator/Dockerfile`) without rebuilding the image causes the drill-down to show "Source changed — rebuild needed" within one tick. The alert reuses the STAGE-003-007 vmalert rule (same metric family).

**Spec references:** §2 Q13 (Docker monitoring decision); `reference_docker_inventory.md` (locally-built images list); EPIC-003.md note in `## Notes` on the build-context source-hash strategy.

## Concrete deliverables

**Local-build image events (inherited scope from STAGE-003-005).** If STAGE-003-008's image-pull event subscription does not cover local docker builds (`docker build` produces an image without a pull event), STAGE-003-009's local-build hash detection MUST account for image-update signals via local-build event sources. STAGE-003-005 deferred all image-event handling to 008/009; this stage closes the local-build half of that deferral.

### 1. Build-context source hasher

New module `apps/monitor/homelab_monitor/kernel/docker/source_hash.py`:

- Given a `build.context` path (absolute path on host, mounted into the monitor at `/host-compose/<relative-path>`), recursively hash all files under it.
- Respects `.dockerignore` semantics (same exclusion rules `docker build` uses — `dockerignore` library or custom parser).
- Returns a single content hash (SHA-256 of concatenated file-path + file-content hashes, sorted by path for determinism).
- Bounded:
  - Max file size: 10 MB per file (configurable via `HOMELAB_MONITOR_BUILD_HASH_MAX_FILE_BYTES`). Larger files skipped + logged + counted in metrics.
  - Max total bytes hashed: 1 GB per build-context. Exceeded → log + emit a special "context too large" hash that always-differs (forces user attention) + a self-metric.
  - Max file count: 50,000. Exceeded → same handling.
  - Max recursion depth: 30.
- Per-file failure (permission denied, symlink to /dev/random, etc.) → skipped + logged + included in self-metric.

### 2. `LocalBuildUpdateCollector` — scheduler-registered

New module `apps/monitor/homelab_monitor/kernel/metrics/local_build_update_collector.py`:

- Tick interval: 30 minutes (configurable). Faster than registry-digest because hashing is cheap (no network, no rate limit) and source changes happen more often than registry pushes.
- For each container with `build:` in its compose entry (parsed from the read-only compose mount — requires reading the compose file, which is the FIRST stage to do this; previous stages used socket-only data):
  - Resolve the `build.context` to a host path. Account for `${COMPOSE_DIR}` and relative paths.
  - Hash the source tree via `source_hash.compute(...)`.
  - Compare against the stored hash for this container from the last container build time (the local image's `Config.Labels["org.opencontainers.image.created"]` from the socket data tells us when the image was built; we store the source-hash at that point if known, else mark "unknown previous hash").
  - If hash differs → emit `homelab_image_update_available{container,source="local_build",current_hash,latest_hash} 1`. (Same metric family as STAGE-003-007 but with `source="local_build"` label so the vmalert rule can disambiguate if needed.)
  - Else → emit 0.
- Per spec §16, emit self-metrics.

### 3. Source-hash storage

Need somewhere to store the "last known hash" per container. Two options (Design phase chooses):

- **In SQLite** — new `docker_build_hashes` table or extend `targets`.
- **In a label on the image itself** — `homelab-monitor.source-hash` label added by the build process. Requires the user's build to set this label.

The second option is more elegant but requires the user to add the label to their builds. The first is more robust and the recommended starting point. The user's host-overrides repo can later add a build hook that writes the label, allowing the second approach if desired.

### 4. Compose file reading — first time in EPIC-003

`apps/monitor/homelab_monitor/kernel/docker/compose_reader.py`:

- Reads the compose file at `docker.compose_file_path` (default unset; the user's deployment sets `/storage/docker/compose/docker-compose.yml` via host-overrides config).
- Parses with `pyyaml`.
- Extracts per-service: `image`, `build.context`, `build.dockerfile`, `profiles`, `network_mode`, `labels`.
- Read-only — never writes (per EPIC-003 cross-stage criterion #6).
- Failure handling: file not present → emit `homelab_docker_compose_readable{} 0` + log; the collector emits no per-container hashes (graceful degradation).

This module will be reused by STAGE-003-010 for Pull & Restart action targets.

### 5. UI — build-source variant

`apps/ui/src/components/docker/badges.tsx` — extend `ImageUpdateBadge` to render a different copy for `source="local_build"` ("Source changed — rebuild needed") vs `source="registry"` ("Update available — `docker compose pull`").

`apps/ui/src/components/docker/ContainerGridRow.tsx` — the hover-tooltip in the Image Update column shows the appropriate detail per source.

## Decisions in this stage

The Design phase confirms the source-hash storage location (SQLite vs image-label), the .dockerignore parser library, the bounded hashing limits, and the compose-reader's path-resolution rules (especially `${COMPOSE_DIR}` and home-dir expansion).

## Locked Design Decisions

D-SHARED-METRIC-FAMILY — `homelab_image_update_available` is the SAME metric for registry-pulled and locally-built images, differentiated by the `source` label (`registry` | `local_build`). Same vmalert rule covers both. UI badge varies copy.

D-COMPOSE-READER-READ-ONLY — the new compose reader is STRICTLY read-only. Per cross-stage criterion #6, this is non-negotiable.

D-HASHING-BOUNDED — every layer of the source-hasher has explicit bounds. A pathological build-context (1 TB of files, infinite-depth recursion) must NOT crash the collector. Limits are explicit and configurable.

D-SQLITE-FIRST-HASH-STORE — v1 stores last-known-hash in SQLite (not image labels). The image-label approach is preserved as a future option but defers to a follow-up.

## Additional Locked Decisions (Design phase, 2026-05-24)

D-NEW-BUILD-HASHES-TABLE — store last-known source hashes in a NEW sidecar table `docker_build_hashes` (migration 0027), keyed by `container_name TEXT PRIMARY KEY`. Mirrors the `image_update_state` (STAGE-003-008) + `docker_override_ownership` (STAGE-003-007) precedent. Columns: `container_name PK`, `compose_service TEXT NOT NULL`, `build_context_path TEXT NOT NULL`, `last_source_hash TEXT NULL`, `last_checked_at TEXT NULL`, `check_failed_at TEXT NULL`, `check_error_reason TEXT NULL` (one of: `compose_unreadable`, `context_missing`, `context_too_large`, `permission_denied`, `unknown`), `update_available INTEGER NOT NULL DEFAULT 0`. Rejected: extending `image_update_state` with a `source` discriminator + nullable columns (riskier rollback; nullable fields always-null for the other source; violates single-responsibility).

D-PATHSPEC-LIBRARY — `.dockerignore` parsing uses the `pathspec>=0.12` library (added to `apps/monitor/pyproject.toml` deps). Rejected: in-house ~50 LoC parser (`.dockerignore` semantics — negation precedence, `**` vs `*`, BuildKit differences — are subtle enough that a naive parser will produce false-positive "source changed" alerts; this is exactly the kind of correctness gap that defeats the feature). Rejected: no `.dockerignore` support (hashing `.git/`, `node_modules/`, `dist/` would churn on every commit; effectively unusable for `udo-viewer` / `kingdom-rules`).

D-HASHING-LIMITS-STAGEDOC-PLUS-ABORT — adopt stage-doc limits verbatim: max 10 MB / file, max 1 GB total, max 50,000 files, max recursion depth 30. ALL configurable via env vars (`HOMELAB_MONITOR_BUILD_HASH_MAX_FILE_BYTES`, `_MAX_TOTAL_BYTES`, `_MAX_FILE_COUNT`, `_MAX_DEPTH`). On exceed: abort-on-exceed model — emit `check_error_reason="context_too_large"` row + special sentinel hash (`"OVERSIZED:<reason>"`) that always-differs (drives the badge to "needs operator attention") + counter metric `homelab_build_source_hash_skipped{reason="context_too_large" | "context_missing" | "permission_denied"}`. Rejected: partial-hash mode (silent detection holes — operator can't know whether a critical changed file was inside or outside the partial set). Rejected: tighter defaults (homelab build contexts are typically small; stage-doc limits are generous-but-safe).

D-FIRST-CHECK-BASELINE — first-ever check for a container computes the hash, stores it, and emits `homelab_image_update_available{source="local_build"} 0`. NO day-1 alerts. Subsequent ticks compare. Rationale: "is the collector working?" is answered by the existing self-metric (`homelab_collector_run_local_build_update_checker{phase="tick", result="ok"}`) and the `/api/integrations/docker/image-updates/summary` endpoint (which will list every tracked container, baseline or not). Rejected: alert-immediately (creates N simultaneous alerts on rollout — 5 local builds = 5 alerts — and trains operators to auto-dismiss). Rejected: `org.opencontainers.image.created` label comparison (best signal in theory, but none of the host's locally-built containers set that label today; mtime-based fallback is unreliable across `git clone`).

D-COMPOSE-DIR-WHOLE-MOUNT — mount the entire compose directory as a single read-only volume on the monitor service. Dev rig: `HOMELAB_MONITOR_COMPOSE_DIR=/tmp/hm-dev/compose` (configurable via `dev.env`). Prod: env var `HOMELAB_MONITOR_COMPOSE_DIR=/storage/docker/compose` (mounted as `${HOMELAB_MONITOR_COMPOSE_DIR}:/host-compose:ro` in `deploy/compose/docker-compose.yml`). Compose reader resolves `build.context` relative to the compose file's directory; the relative paths resolve under `/host-compose/<service>/...` inside the container. Operator-doc caveat: "the monitor will read every file under this path; do not store plaintext secrets here." Precedent: existing `/host/etc:ro` and `/config:ro` mounts. Rejected: separate-build-contexts-root (forces operator restructure; fights typical homelab colocation pattern). Rejected: per-build-context dynamic mounts (high friction; defeats "set-and-forget" UX).

D-SEED-UDO-VIEWER — acceptance test seed = `udo-viewer` (local build, port 14572, custom webapp, isolated, no external dependencies). End-to-end success criterion: within one `LocalBuildUpdateCollector` tick (60s in dev / 30 min in prod) after touching a file inside the udo-viewer build context (e.g., a comment line in its Dockerfile), the container grid shows a blue "Source changed — rebuild needed" pill on the `udo-viewer` row, the drill-down at `/integrations/docker/containers/udo-viewer/image-update` shows current source hash != stored hash, and `sqlite3 .../homelab.db "SELECT update_available, check_error_reason FROM docker_build_hashes WHERE container_name='udo-viewer'"` returns `1|`. Rejected: pihole-unbound (DNS for whole network; HIGH blast); plex (host-network; large media; HIGH); frigate (USB passthrough; HIGH); podcast-feed (depends on `/rackstation/*` NFS; HIGH); kingdom-rules (custom; criticality unknown; MED); grocy-homeassistant (HA integration data; MED-HIGH).

### Lessons learned (this stage)

- **"X changed since Y" needs SEPARATE persistence for X and Y.** The naive `update_available = current_hash != prior_hash` self-erases the signal on next tick because `prior` gets overwritten with `current` after the comparison. Solution: store BASELINE separately, update baseline only on a different event (here: image_id change = real rebuild). Caught only by 3a multi-tick e2e — unit tests with mocked DB happily wrote+read the same hash. Future "detection" features should add a multi-tick regression test.
- **`await_immediate_run` Future-leak in early-return paths.** Adding "register Future, set result on completion" to a method with N early-returns leaks the Future on every return that doesn't explicitly resolve. Pytest hung at ~88% with 30s timeout * N tests = 15min cumulative silence. Solution: helper called before EVERY return path. Lesson: when adding awaitable async events to producer code, audit EVERY return.
- **Don't gate prod behavior behind test env vars; fix test isolation instead.** First instinct on startup-tick test breakage was `HOMELAB_MONITOR_DISABLE_X_ON_STARTUP=1` in conftest. User correctly rejected ("fix this correctly, not via gate"). Real fix: `await_immediate_run` + StubRegistryDigestClient (eventually removed as dead code — pytest-httpx provides the isolation). Result: tests exercise SAME prod path; no test/prod gap.
- **Mid-Refinement scope expansion was necessary, not optional.** STAGE-003-009 was supposed to be source-hash detection; ended up shipping a generic config layer + path resolver + scheduler async pattern + 4 host-side dockerignore files. Deferring would have shipped a feature that didn't work on the operator's real layout. When 3a/3b surface architectural issues, treat scope expansion as load-bearing.
- **`# pragma: no branch` vs `# pragma: no cover` distinction.** `no branch` only marks the conditional itself; the BODY of an unreachable-in-tests branch still counts toward line coverage. For env-var-set or "validated via dev rig" branches, use `# pragma: no cover` on the if-line; the body is then excluded too.
- **`.dockerignore` self-exclusion.** Docker's daemon never includes `.dockerignore` itself in the build context, but most `.dockerignore` files don't list themselves. The hasher must `if fname == _DOCKERIGNORE_FILENAME and dirpath == build_context: continue` explicitly. Found in source_hash test.
- **Set iteration order is non-deterministic across Python runs.** `for i, name in enumerate(container_names_set)` builds non-deterministic SQL parameter names. Always `sorted()` if the iteration order matters or could appear in test assertions later.
- **Recovery rule violation recurred.** Made the chained `cd && make` recovery mistake AGAIN despite knowing the rule. Caught instantly but it's an awareness gap. When recovering from a wrong cwd, the rule is: ONE bash call to `cd /storage/programs/homelab-monitor`, then a SEPARATE bash call for the next command. Slow down on recovery steps even when broader work is going fast.
- **Docker Hub rate-limit differentiation.** Direct `docker pull` hits a different rate-limit counter than BuildKit's `--frontend=dockerfile:1.7`. When BuildKit is 429'd on `docker/dockerfile:1.7`, a plain `docker pull docker/dockerfile:1.7` may still succeed and cache the image locally; subsequent buildkit invocations then use the cached image.
- **Fixer subagents truncate mid-summary periodically.** The fixer's text output sometimes cuts off after a long sequence of edits, even when all edits landed correctly. Always verify via standalone `make uv ARGS="--directory apps/monitor ruff check ..."` + `pyright` after a long fixer dispatch.
