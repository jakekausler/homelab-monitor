# STAGE-003-009: "Pull & Restart" action — compose-aware + audit + confirm

## Status: Not Started

## Overview

The first kernel-driven Docker WRITE action in the project. A confirm-gated per-container button executes `docker compose pull <service> && docker compose up -d <service>`, captures stdout/stderr, records the action in a new `compose_actions` audit table, and refreshes the UI. The docker socket mount widens from `:ro` to RW for this stage; the compose file is mounted `:ro`.

This stage is small in code surface but high in blast radius. Confirm-on-destructive per spec §7.2. Audit-logged per spec §16. The pattern established here is the precursor to EPIC-009 (auto-fix subsystem) — the same confirm + audit + capture-transcript machinery will be reused for Claude-driven runbook real-runs.

**Vertical slice:** clicking "Pull & Restart" on a row in the drill-down grid (where an image-update badge is showing) opens a confirm modal; typing a confirm phrase triggers the action; a "Pulling…" status appears on the row; on completion the row badge clears (no update available), the Recent Actions panel shows the new action with stdout/stderr toggleable.

**Spec references:** §2 Q13 (Pull & Restart dashboard action, auto-update OFF); §7.2 (confirm-on-destructive); §7.4 (auto-fix isolation patterns inform but do NOT block — this is a structured action, not a Claude run); §16 (audit_log).

## Concrete deliverables

### 1. Dockerfile updates

`apps/monitor/Dockerfile`:

- Install `docker` CLI + `docker-compose` plugin (or `docker compose` v2 plugin specifically — pinned in Design).
- Add `/usr/local/bin/docker` to the image. ~50 MB image growth acceptable.

### 2. Compose mounts widen — socket RW + compose file RO

`deploy/compose/docker-compose.yml`:

- The monitor's docker socket mount changes from `:ro` to RW: `/var/run/docker.sock:/var/run/docker.sock`. (Stages 001-008 use `:ro`; this stage widens.)
- New mount: `/storage/docker/compose:/host-compose:ro` (compose dir mounted RO; the host path is configurable via `docker.compose_dir_path`).
- The same widening applies to `deploy/dev/docker-compose.dev.yml` for parity in dev rig.

The security boundary is at the docker socket level — anyone with socket access has root-equivalent on the host. This is documented and accepted per EPIC-003 cross-stage criterion #18. STAGE-003-009 commit message includes the explicit acknowledgment.

### 3. `ComposeActionRunner`

New module `apps/monitor/homelab_monitor/kernel/docker/compose_action_runner.py`:

- Single public method: `run_pull_and_restart(container_name: str, who: User) -> ComposeActionResult`.
- Looks up the compose service name for the given container name (via `compose_reader` from STAGE-003-008 — re-used here).
- Validates: the container exists in the compose file; the user has authenticated + passed the confirm gate (handled at the API layer).
- Constructs the command: `docker compose -f /host-compose/docker-compose.yml pull <service> && docker compose -f /host-compose/docker-compose.yml up -d <service>`.
- Executes via `asyncio.create_subprocess_exec` with bounded timeout (default 5 minutes, configurable).
- Captures stdout + stderr (separately).
- On timeout: SIGTERM then SIGKILL; recorded as `state="timeout"` in the audit row.
- Returns `ComposeActionResult(exit_code, stdout, stderr, started_at, ended_at, state)`.

### 4. `compose_actions` table

`apps/monitor/homelab_monitor/kernel/db/repositories/compose_actions_repository.py`:

```
compose_actions:
  id INTEGER PRIMARY KEY AUTOINCREMENT
  action TEXT NOT NULL CHECK (action IN ('pull_and_restart'))   -- extensible enum
  container_name TEXT NOT NULL
  compose_service TEXT NOT NULL
  before_image TEXT                   -- image ref captured before the action
  before_digest TEXT                  -- digest captured before the action
  after_image TEXT                    -- image ref after (post-up -d)
  after_digest TEXT                   -- digest after
  command TEXT NOT NULL               -- the literal command executed
  stdout TEXT                         -- captured (truncated to 1 MB)
  stderr TEXT                         -- captured (truncated to 1 MB)
  exit_code INTEGER
  state TEXT NOT NULL CHECK (state IN ('running', 'success', 'failed', 'timeout', 'killed'))
  started_at TEXT NOT NULL
  ended_at TEXT
  duration_seconds REAL
  who TEXT NOT NULL                   -- username from session
  client_ip TEXT
  audit_log_id INTEGER                -- FK to audit_log
```

Indexed on `(container_name, started_at DESC)`.

A row is written when the action starts (`state="running"`). Updated to terminal state on completion.

### 5. API endpoint — confirm-gated POST

`apps/monitor/homelab_monitor/kernel/api/routers/docker.py`:

- `POST /api/integrations/docker/containers/{name}/pull-and-restart` — kicks off the action.
- Request body: `{"confirm_phrase": "<typed-phrase>"}` (or session-PIN; final UX decided in Design).
- Returns immediately with `{"action_id": <id>, "state": "running"}`.
- The action runs in a background task. UI polls `GET /api/integrations/docker/compose-actions/{id}` for state.
- Per spec §7.2 — typed-phrase confirm OR session-lifetime PIN. Confirm enforcement is API-side (not just UI-side) — the endpoint rejects 400 without a valid confirm.
- Auth: requires session + a new scope `docker:write` on the session (or a different gating mechanism — Design decides). The Pull & Restart action is HIGH blast radius and should not be available to a stolen session by default.
- Audit log row written via `audit_log` (the existing table from EPIC-001 stage tracking — per spec §16 cross-cutting requirement).

- `GET /api/integrations/docker/compose-actions/{id}` — fetch one action's full record (stdout/stderr included).
- `GET /api/integrations/docker/compose-actions?container={name}&limit=N` — list recent actions for a container.

### 6. UI — Action button + confirm modal + Recent Actions panel

`apps/ui/src/components/docker/ContainerGridRow.tsx`:

- The "Actions" column gets a "Pull & Restart" button. The button is only enabled when `image_update_available=true`.
- Clicking opens a confirm modal: shows current vs latest digest, the compose command to be run, and a typed-phrase input ("type 'pull' to confirm" — the exact phrase decided in Design).

`apps/ui/src/components/docker/PullRestartModal.tsx` (new) — the confirm modal. Submits → POST `/api/.../pull-and-restart`; on success the modal closes and the row shows a "Pulling…" pulsing indicator. The UI polls the action status endpoint every 2s while `state="running"`.

`apps/ui/src/components/docker/RecentActionsPanel.tsx` — the empty-state placeholder from STAGE-003-002 fills in. Lists recent actions; each row shows container, action, state, duration, started_at, "who", and an expand toggle showing stdout/stderr.

`apps/ui/src/api/docker.ts` — `useStartPullAndRestart`, `useGetComposeAction`, `useListComposeActions` hooks.

### 7. vmalert rule — failed Pull & Restart

`deploy/vmalert/metrics/docker_compose_actions.yml`:

```yaml
- alert: DockerComposeActionFailed
  expr: increase(homelab_compose_action_failed_total[10m]) > 0
  for: 0m
  labels:
    severity: warning
    integration: docker
  annotations:
    summary: "Pull & Restart action failed for {{ $labels.container }}"
```

The runner emits `homelab_compose_action_failed_total{container,action,reason}` on every failed action (non-zero exit, timeout, killed) and `homelab_compose_action_success_total{container,action}` on every success.

## Out of scope for this stage

- Auto-pull (never — auto-update is OFF per spec Q13; this is always manual).
- Pull WITHOUT restart (a "Pull only" action). Could be added in a follow-up; not needed in v1.
- Restart WITHOUT pull (a "Restart" action). Same — follow-up.
- "Pull all updates" (bulk action). Same — follow-up.
- Rollback (revert to previous image). Follow-up — would require image-digest history per container.
- Private-registry credentials for the pull (deferred from STAGE-003-007).

## Decisions in this stage

The Design phase confirms the confirm-gate UX (typed phrase vs PIN), the timeout default, the `docker:write` scope semantics, and the relationship to EPIC-009's runbook orchestrator (table merge or keep separate).

## Locked Design Decisions

D-COMPOSE-AWARE — the action uses `docker compose -f <file> pull <service> && up -d <service>`, NOT `docker pull` + `docker stop/start`. Compose-aware preserves the user's env, volumes, networks, dependencies.

D-INSIDE-MONITOR — the action runs INSIDE the monitor container. NOT via a separate fixer-runner pattern (that's EPIC-009 for Claude-driven runbooks). The monitor needs the `docker` CLI installed and the docker socket RW + compose file RO.

D-CONFIRM-API-SIDE — confirm gate is enforced API-side, not just UI-side. Even a hand-crafted curl must pass the confirm. (Spec §7.2.)

D-CONFIRM-EXTENDS-SPEC-7-2 — Spec §7.2 enumerates confirm-on-destructive actions but does NOT explicitly list Pull & Restart. This stage EXTENDS §7.2's list because Pull & Restart restarts a container (high blast radius). Documented here rather than added to the spec because the spec's list was illustrative, not exhaustive. (Spec §7.2 + EPIC-003 brainstorm decision.)

D-AUDIT-EVERY-ATTEMPT — an audit_log row + a compose_actions row are written for EVERY attempt, regardless of outcome (success, failure, timeout, killed). (Spec §16.)

D-COMPOSE-ACTIONS-OWN-TABLE — `compose_actions` is its own table, NOT a reuse of `runbook_runs` (which is a Claude-runs-only table). EPIC-009's auto-fix subsystem will revisit table layout — this stage establishes the audit pattern. (Cross-epic carry-forward → EPIC-009 noted in EPIC-003.md.)

D-SOCKET-RW-WIDEN — this stage widens the docker socket mount from `:ro` to RW. The security boundary is at the docker socket level and this is documented + accepted. (Cross-stage criterion #18.)

D-AUTO-UPDATE-OFF — auto-update is permanently OFF. Pull & Restart is always manual. (Spec Q13.)
