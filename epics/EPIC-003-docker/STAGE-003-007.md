# STAGE-003-007: Per-service config-file override — YAML supersedes labels

## Status: Complete

## Overview

Add a higher-precedence probe-config source: per-service YAML files under `/config/plugins/docker/<container_name>.yaml`. The file override supersedes labels for any `(container, kind, name)` tuple it defines. Per spec Q13 (A3): labels are primary; the config file is the override. The user's host-overrides repo (`homelab-monitor-overrides`) is where these files typically live, mounted into the monitor at `/config/`.

This stage's surface is small: one new resolver path, one new precedence rule, two new fields in `probe_targets` (`config_source` already exists from STAGE-003-005; this stage adds use of the `file_override` value), and a UI badge showing which source supplied each probe's config.

**Vertical slice:** placing a `/config/plugins/docker/homeassistant.yaml` file defining custom probes for the `homeassistant` container causes the next probe-resolver tick to replace the label-derived probes with the file-derived ones. The drill-down grid shows "config-file" badge instead of "label" on those probes.

**Spec references:** §2 Q13 (label primary, config-file override = A3); §5.6 (configuration — per-plugin YAML under `/config/plugins/<kind>/<name>.yaml`); §6.5 (application configuration).

## Concrete deliverables

### 1. File-override schema

`apps/monitor/homelab_monitor/kernel/docker/override_schema.py` — pydantic model:

```python
class ProbeOverride(BaseModel):
    kind: Literal["http", "tcp", "exec", "metrics"]
    name: str = "default"
    target: str  # URL or "port:host" or "command"
    enabled: bool = True
    interval_seconds: int = 30
    timeout_seconds: int = 10

class DockerContainerOverride(BaseModel):
    container: str  # required — must match a real container name
    probes: list[ProbeOverride]
    disabled: bool = False  # disable ALL probes for this container (overrides labels)
```

### 2. File loader + hot-reload

`apps/monitor/homelab_monitor/kernel/docker/override_loader.py`:

- Scans `/config/plugins/docker/*.yaml` on startup AND on file-change (using `inotify` via `watchdog` library, or a simple periodic re-scan every 30s — Design phase chooses).
- Parses each file with `DockerContainerOverride`. Validation failures log + skip (don't crash); the failed file is reported in self-metrics and visible in the drill-down (badge: "Config file error").
- Returns `dict[container_name, DockerContainerOverride]`.

### 3. Probe-config resolver — three-source precedence

Extends STAGE-003-005's resolver. The precedence order for a container's probes (highest wins):

1. **File override** (`/config/plugins/docker/<container>.yaml`) — if present and not `disabled: true`.
2. **Labels** (`homelab-monitor.<kind>.<name>=...` on the container).
3. **Auto-default** (none in v1 — the spec defers default probe inference; this stage leaves the slot for EPIC-018 or a follow-up).

Algorithm:

```
for each container c:
  if file_override[c].disabled:
    delete all probes for c
    skip
  if file_override[c] exists:
    upsert probes from file_override[c] with config_source="file_override"
    delete any probes for c with config_source IN ("label", "auto_default") that are not in file_override[c]
  else:
    # fall through to label-based behavior from STAGE-003-005
    upsert probes from labels with config_source="label"
```

## Decisions in this stage

The Design phase confirms the hot-reload mechanism (inotify vs periodic rescan), the file-naming convention (`<container_name>.yaml` vs `docker.yaml` listing many containers), and the validation-error UX (banner vs inline indicator on the affected rows).

## Locked Design Decisions

D-PRECEDENCE-FILE-OVER-LABEL — config-file overrides win over labels. The user's intent expressed in a YAML file is more specific than container labels and easier to update without restarting containers. (Spec Q13 — A3.)

D-FILE-PER-CONTAINER — one YAML file per container (`/config/plugins/docker/<container_name>.yaml`). NOT a single `docker.yaml` listing everything — easier to manage in the host-overrides repo as separate files, easier to diff, easier to hot-reload one at a time.

D-VALIDATION-NO-CRASH — file validation failure logs + reports in self-metrics + surfaces a UI banner. Does NOT crash the monitor or block probes for other containers. (Spec §6.5 — "refuses to start with a clear error" applies to the top-level config; per-plugin configs are softer.)

D-NO-FILE-WRITE-FROM-UI — the UI READS the config state but does NOT write config files in this stage. Editing is done on the host (by the user or by the host-overrides repo's deploy step). EPIC-011 + EPIC-018 may later add UI-side editing.

D-NO-AUTO-DEFAULT-PROBES-V1 — auto-default probe inference (e.g., "container with exposed port 8080 + healthcheck → auto-create HTTP probe") is deferred. The resolver structurally supports a third precedence source but emits nothing in v1.

## Additional Locked Decisions (Design phase, 2026-05-22)

D-HOTRELOAD-PERIODIC-30S — file scanning is a 30s periodic asyncio loop in `override_loader.py`, using the same pattern as `TtlCachingSecretsResolver.refresh_loop` (asyncio.sleep + try/except + cancellable). Rejected: inotify via `watchdog` (new dep + bind-mount + observer-thread complexity); reload-on-discoverer-tick (couples Docker API polling with file reading; worse lag).

D-ROUTING-INDEPENDENT-LOADER — file-override path lives in a NEW module `apps/monitor/homelab_monitor/kernel/docker/override_loader.py` with its own periodic loop and its own upsert site. It does NOT extend `DockerDiscoverer._extract_detection_reason`. Rationale: the only routing option that handles "override file exists before container starts" (operator writes YAML, then starts container, file-derived probes should appear as soon as the container exists — without requiring labels). Clean separation: loader owns file path, discoverer owns label path.

D-OWNERSHIP-TOTAL-PER-CONTAINER — when a container has an override file, the loader takes TOTAL ownership of its probe rows for that tick. The discoverer skips the label-upsert for owned containers. Means `mark_missing_except_conn` works unchanged: loader passes file-derived kept_keys; discoverer passes label-derived kept_keys for unowned containers only. Loader tracks the previous-tick owned set; on file-delete it calls `mark_missing_except_conn(container, set(), now)` to release ownership, then the discoverer picks up labels on its next tick (1-tick gap).

D-OWNERSHIP-COORDINATION-VIA-SQLITE — cross-task coordination of "loader's owned-containers set" lives in a new tiny SQLite table `docker_override_ownership(container_name TEXT PRIMARY KEY, claimed_at TEXT NOT NULL)`. Discoverer reads this table cheaply at the start of each tick to filter. Avoids in-process shared state + asyncio.Lock; testable; visible to debug queries. New migration `0024_docker_override_ownership.py`.

D-EXEC-DUAL-GATE-FILE-OVERRIDE — file-override `kind: exec` requires BOTH the existing global env `HOMELAB_MONITOR_DOCKER_PROBES_EXEC_ENABLED=true` AND a new container-level field `exec_authorized: bool = False` on `DockerContainerOverride`. Matches D-EXEC-OPT-IN + D-EXEC-PER-CONTAINER-LABEL from STAGE-003-006 (defense-in-depth per spec §7.4). When either gate fails, exec probes are dropped from the upsert and a `docker_file_override_malformed` suggestion is emitted with reason `exec_not_authorized`.

D-ERROR-UX-BADGE-PLUS-SUGGESTION — validation errors surface in TWO complementary places:
  1. **Row badge** — extend the existing `GET /api/integrations/docker/probes/summary` endpoint to include per-container `config_errors: list[str] | None`. The container grid row renders a red "Config error" badge when populated. Locality of information.
  2. **Suggestion (orphan only)** — when a `/config/plugins/docker/<name>.yaml` exists but no matching container is running, emit a `docker_file_override_malformed` suggestion (kind matches existing precedent `docker_label_malformed`). Surfaces in the existing `PendingSuggestionsPanel`.

No new endpoint, no banner component, no dismiss-state machine. Single source of truth: validation runs in the loader; errors flow to whichever surface fits.

D-OVERRIDE-FILE-NAMING — files are matched on the canonical container name (post `/`-strip + post `<12hex>_` rename-prefix strip; see `docker_discoverer.py:296-299`). The file basename must equal the container name. `.yaml` and `.yml` both accepted. Other extensions ignored. Files in subdirs of `/config/plugins/docker/` are ignored.

D-OVERRIDE-FILE-SCHEMA — locked schema is exactly the one in this stage's "Concrete deliverables" §1, EXTENDED with `exec_authorized: bool = False` on `DockerContainerOverride`:

```python
class ProbeOverride(BaseModel):
    model_config = ConfigDict(extra="forbid")
    kind: Literal["http", "tcp", "exec", "metrics"]
    name: str = "default"
    target: str
    enabled: bool = True
    interval_seconds: int = 30
    timeout_seconds: int = 10

class DockerContainerOverride(BaseModel):
    model_config = ConfigDict(extra="forbid")
    container: str
    exec_authorized: bool = False  # NEW (D-EXEC-DUAL-GATE-FILE-OVERRIDE)
    disabled: bool = False
    probes: list[ProbeOverride]
```

`extra="forbid"` so typos in the YAML produce a validation error (surfaced via D-ERROR-UX-BADGE-PLUS-SUGGESTION). The `container` field MUST equal the file basename or validation fails (defends against operator confusion).

D-MOUNT-PATH-PROD — `deploy/compose/docker-compose.yml` adds the first `/config` bind-mount to the monitor service:
`- ${HOMELAB_MONITOR_OVERRIDES_DIR:-/var/lib/homelab-monitor/overrides}:/config:ro`
Read-only inside the container. Host default `/var/lib/homelab-monitor/overrides` matches existing host-side state dirs (`crontab-snapshot`, `cron-apply`). Empty by default = no-op (loader logs once, never fails).

D-MOUNT-PATH-DEV — dev rig (hybrid `make dev`) does NOT need a compose change because the monitor runs on the host. A new env var `HOMELAB_MONITOR_DOCKER_OVERRIDES_DIR` is added to `deploy/dev/dev.env.example` defaulting to `/tmp/hm-dev/overrides` so dev-rig overrides are isolated from prod. The override_loader reads from `HOMELAB_MONITOR_DOCKER_OVERRIDES_DIR` (no fallback to `/config/plugins/docker/` in host mode).

D-SUMMARY-ENDPOINT-EXTENSION — `GET /api/integrations/docker/probes/summary` payload extended (not a new endpoint):
```
per-container row gains:
  - source_breakdown: { label: int, file_override: int, auto_default: int, discovered_accepted: int }
  - config_errors: list[str] | None  # validation error messages for this container's override file, if any
```

Backward compatible: clients that ignore the new fields still work. The badge work in this stage consumes both fields.

## Design Phase

- **Approaches Presented:** hot-reload mechanism (30s asyncio loop vs inotify-watchdog vs reload-on-discoverer-tick), routing path (independent loader vs extend discoverer detection-reason vs hybrid), exec-gating (dual gate vs global-only vs separate flag), error UX (row badge + suggestion vs suggestion-only vs banner + endpoint vs both).
- **UI Options Presented:** config_source badge in drill-down (text label per probe row); per-container row badge for config errors (red, replaces probes badge when populated); no banner component this stage.
- **User Choice:**
  - Hot-reload: 30s periodic asyncio loop (D-HOTRELOAD-PERIODIC-30S)
  - Routing: independent loader (D-ROUTING-INDEPENDENT-LOADER)
  - Ownership: total-per-container via SQLite ownership table (D-OWNERSHIP-TOTAL-PER-CONTAINER, D-OWNERSHIP-COORDINATION-VIA-SQLITE)
  - Exec gate: dual gate (env + per-container exec_authorized in YAML) (D-EXEC-DUAL-GATE-FILE-OVERRIDE)
  - Error UX: row badge (extant containers) + suggestion (orphan files) (D-ERROR-UX-BADGE-PLUS-SUGGESTION)
