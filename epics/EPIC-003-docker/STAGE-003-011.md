# STAGE-003-011: Per-container log viewer — `/integrations/docker/containers/$name/logs`

## Status: Not Started

## Overview

Build the per-container log viewer route + API endpoint. Each container in the drill-down grid gets a "View logs →" link that opens a route showing recent log lines from VictoriaLogs (most recent N lines + a manual Refresh button for now; live-tail is deferred to EPIC-004). Reuses `VictoriaLogsClient` from STAGE-002-013 (the bounded VL query primitive built for cron run logs).

This stage depends on STAGE-003-002 (vector container-log ingestion fix) — without that, no container logs flow into VL, so the viewer would show nothing. It also depends on STAGE-003-003 (UI skeleton route + Logs column scaffold) and STAGE-003-004 (container inventory — so we know what containers exist).

EPIC-004 will layer log-anomaly + pattern detection ON TOP of this viewer (e.g., highlighting error spikes, signature clustering, time-of-day anomalies). This stage delivers the raw viewer.

**Vertical slice:** clicking "View logs →" on the `homeassistant` row in the Docker drill-down opens `/integrations/docker/containers/homeassistant/logs`. The route shows recent log lines (default last 500 or last 1 hour, whichever is smaller) with timestamps, scrollable monospace body, a sticky metadata header (container name, status, last-log-at), and a Refresh button. Container that doesn't exist → 404 surface. Container exists but has no log lines yet → "no log lines (yet)" empty state.

**Spec references:** §4.3 (Logs data flow); §9.2 (Integrations sub-page — drill-down); STAGE-002-013 (VictoriaLogsClient pattern); STAGE-002-015 (run-log viewer pattern — mirror its UX); EPIC-004 carry-forward (this viewer is the foundation EPIC-004 enhances).

## Concrete deliverables

### 1. API endpoint — `GET /api/integrations/docker/containers/{name}/logs`

`apps/monitor/homelab_monitor/kernel/api/routers/docker.py` — extends the router from STAGE-003-004:

- `GET /api/integrations/docker/containers/{name}/logs?limit=500&since=1h`
  - `limit` (query param, optional, default 500, max 5000) — max log lines returned.
  - `since` (query param, optional, default `1h`) — time-window upper bound. Accepts `Xs`, `Xm`, `Xh`, `Xd` (e.g., `30m`, `2h`). Capped at `7d` per spec retention.
- Response shape:
  ```json
  {
    "container_name": "homeassistant",
    "log_status": "available" | "no_lines" | "container_unknown" | "vl_unavailable",
    "lines": [
      {"timestamp": "2026-05-21T14:30:00.123456Z", "line": "INFO ..."},
      ...
    ],
    "truncated": false,
    "window_start": "2026-05-21T13:30:00Z",
    "window_end": "2026-05-21T14:30:00Z"
  }
  ```
- `log_status` semantics:
  - `available` — at least one line returned.
  - `no_lines` — container known to inventory (STAGE-003-004 `targets` table) but VL has no lines in the queried window.
  - `container_unknown` — container name not in `targets` table → 404 status code.
  - `vl_unavailable` — VL unreachable or 5xx → 503 status code, no `lines` array.
- LogsQL query shape: vector ingests container logs with a `service` (or similar) label that equals the container name. The exact label is determined by vector's `docker_logs` source config — Build phase verifies what label vector emits and adjusts the LogsQL query accordingly. Likely `_msg:* AND service:"<container_name>"` or `container_name:"<container_name>"`.
- Auth: requires session.
- Reuses `VictoriaLogsClient` from STAGE-002-013 — bounded query, max-lines cap, max-bytes cap, HTTP timeout.

### 2. UI route — `/integrations/docker/containers/$name/logs`

The route was SCAFFOLDED in STAGE-003-003 with a placeholder component (`ContainerLogsPlaceholder.tsx`). This stage swaps the placeholder for the real component:

`apps/ui/src/routes/integrations/DockerContainerLogsViewer.tsx`:
- Sticky metadata header: container name, status badge (from inventory), last-log-at timestamp (relative + absolute tooltip), Refresh button (manual refetch).
- Body: scrollable monospace log lines. Each line shows timestamp (compact) + log text. Long lines wrap. Empty state shows "No log lines in the last X" message.
- "since" picker (dropdown: last 5m / 15m / 1h / 6h / 24h / 7d). Default 1h.
- "limit" hard-capped at 5000 client-side; user can't request more.
- Handles all four `log_status` values:
  - `available` → renders the lines.
  - `no_lines` → empty state with "no lines in window — try expanding the time window" + the picker.
  - `container_unknown` → 404 page with back link.
  - `vl_unavailable` → "logs temporarily unavailable" banner (NOT a hard error), Refresh button still present.
- `truncated: true` indicator at the top of the body when the cap was hit (e.g., "showing first 5000 of N+ lines — narrow the time window for more").

`apps/ui/src/components/docker/ContainerLogsPlaceholder.tsx` — DELETED (its job is done; replaced by the real viewer).

`apps/ui/src/api/docker.ts` — new `useContainerLogs(containerName, { limit, since })` hook.

## Locked Design Decisions

D-VIEWER-PER-CONTAINER — the log viewer is per-container (`/integrations/docker/containers/$name/logs`), NOT a global multi-container explorer. Cross-container log work is EPIC-004's generic logs explorer. (EPIC-003 brainstorm 2026-05-21 decision Q2: "Per container.")

D-MANUAL-REFRESH-V1 — no live-tail in this stage. Manual Refresh button. Live-tail (SSE/WebSocket) is deferred to EPIC-004 to keep this stage focused. (Spec EPIC-003.md Docker-log requirements callout.)

D-VL-CLIENT-REUSE — the endpoint reuses `VictoriaLogsClient` (STAGE-002-013) — bounded queries, max-lines cap, max-bytes cap, HTTP timeout, graceful failure. NO new VL query primitive.

D-FOUR-STATES — the log viewer explicitly handles four `log_status` shapes: `available`, `no_lines`, `container_unknown` (404), `vl_unavailable` (503). UI never crashes on any of them. Mirrors the run-log viewer pattern from STAGE-002-015.

D-SCAFFOLDING-FROM-003 — the route + placeholder component was scaffolded in STAGE-003-003. This stage swaps the placeholder for the real component. The router change is zero — only the imported component changes.

D-FULL-LOGS-IN-EPIC-003 — full per-container log functionality (recent history + state handling + truncate + manual refresh) lands HERE in EPIC-003. EPIC-004 adds anomaly detection, pattern analysis, error highlighting, and live-tail features ON TOP of this base. (User decision EPIC-003 brainstorm 2026-05-21: "We should do full logs now. EPIC-004 will add extra features like anomaly detection, etc.")
