# STAGE-003-011: Per-container log viewer ā€” `/integrations/docker/containers/$name/logs`

## Status: Complete

## Overview

Build the per-container log viewer route + API endpoint. Each container in the drill-down grid gets a "View logs ā†’" link that opens a route showing recent log lines from VictoriaLogs (most recent N lines + a manual Refresh button for now; live-tail is deferred to EPIC-004). Reuses `VictoriaLogsClient` from STAGE-002-013 (the bounded VL query primitive built for cron run logs).

This stage depends on STAGE-003-002 (vector container-log ingestion fix) ā€” without that, no container logs flow into VL, so the viewer would show nothing. It also depends on STAGE-003-003 (UI skeleton route + Logs column scaffold) and STAGE-003-004 (container inventory ā€” so we know what containers exist).

EPIC-004 will layer log-anomaly + pattern detection ON TOP of this viewer (e.g., highlighting error spikes, signature clustering, time-of-day anomalies). This stage delivers the raw viewer.

**Vertical slice:** clicking "View logs ā†’" on the `homeassistant` row in the Docker drill-down opens `/integrations/docker/containers/homeassistant/logs`. The route shows recent log lines (default last 500 or last 1 hour, whichever is smaller) with timestamps, scrollable monospace body, a sticky metadata header (container name, status, last-log-at), and a Refresh button. Container that doesn't exist ā†’ 404 surface. Container exists but has no log lines yet ā†’ "no log lines (yet)" empty state.

## Concrete deliverables

### 1. API endpoint ā€” `GET /api/integrations/docker/containers/{name}/logs`

`apps/monitor/homelab_monitor/kernel/api/routers/docker.py` ā€” extends the router from STAGE-003-004:

- `GET /api/integrations/docker/containers/{name}/logs?limit=500&since=1h`
  - `limit` (query param, optional, default 500, max 5000) ā€” max log lines returned.
  - `since` (query param, optional, default `1h`) ā€” time-window upper bound. Accepts `Xs`, `Xm`, `Xh`, `Xd` (e.g., `30m`, `2h`). Capped at `7d` per spec retention.
- Response shape:
  ```json
  {
    "container_name": "homeassistant",
    "log_status": "available" | "no_lines" | "container_unknown" | "vl_unavailable",
    "lines": [
      {"timestamp": "2026-05-21T14:30:00.123456Z", "line": "INFO ..."},
      ...
    ],
    "truncated": false,
    "window_start": "2026-05-21T13:30:00Z",
    "window_end": "2026-05-21T14:30:00Z"
  }
  ```
- `log_status` semantics:
  - `available` ā€” at least one line returned.
  - `no_lines` ā€” container known to inventory (STAGE-003-004 `targets` table) but VL has no lines in the queried window.
  - `container_unknown` ā€” container name not in `targets` table ā†’ 404 status code.
  - `vl_unavailable` ā€” VL unreachable or 5xx ā†’ 503 status code, no `lines` array.
- LogsQL query shape: vector ingests container logs with a `service` (or similar) label that equals the container name. The exact label is determined by vector's `docker_logs` source config ā€” Build phase verifies what label vector emits and adjusts the LogsQL query accordingly. Likely `_msg:* AND service:"<container_name>"` or `container_name:"<container_name>"`.
- Auth: requires session.
- Reuses `VictoriaLogsClient` from STAGE-002-013 ā€” bounded query, max-lines cap, max-bytes cap, HTTP timeout.

### D-LOG-LABEL-SERVICE
LogsQL filter uses `service:"<container_name>"`. Vector's `add_labels` transform guarantees `.service = .container_name` for the `docker_logs` source, so a single label vocabulary works across container logs and (future) journald logs. Endpoint MUST use `logsql_quote_phrase(container_name)` for safe escaping.

### D-DEFAULT-WINDOW-15M
Default `since` window when the viewer first opens is `15m`. Tighter default than the spec's draft `1h`; matches "actively investigating an issue" as the primary use case. User can widen via the picker.

### D-WINDOW-PRESETS-6
The `since` picker offers six fixed presets: `5m / 15m / 1h / 6h / 24h / 7d`. Hard-capped at 7d. NO custom datetime range input in this stage ā€” explicitly deferred to EPIC-004 (see D-DEFER-CUSTOM-RANGE).

### D-TRUNCATION-BANNER-500
Hard limit: 500 lines per request (clamped server-side). When `truncated: true`, render a sticky amber banner at the top of the log body: "Showing first 500 lines. Narrow the time window to see all entries." Banner stays visible while user scrolls. Matches the existing `CronRunLogViewer` truncation pattern. NO pagination in this stage ā€” explicitly deferred to EPIC-004 (see D-DEFER-PAGINATION).

### D-DEFER-PAGINATION
Cursor-based "Load older lines" pagination is intentionally deferred to EPIC-004 STAGE-004-005 (Logs explorer UI). Rationale: the 500-line cap + sticky banner is sufficient for normal debugging; pagination adds API + UI surface that belongs in the full logs-explorer redesign. Forward-reference added in:
- `epics/EPIC-004-logs-pipeline/EPIC-004.md` ā€” explicit acceptance criterion that pagination must be added to all log-viewer surfaces (per-container logs AND cron run logs)
- `epics/EPIC-002-foundation/STAGE-002-015.md` (cron run-log viewer) ā€” cross-reference note that pagination is co-deferred so both surfaces redesign together
- Code comment in `DockerContainerLogsViewer.tsx` and `CronRunLogViewer.tsx` ā€” `// TODO: EPIC-004 STAGE-004-005 ā€” cursor-based pagination`

### D-DEFER-CUSTOM-RANGE
Custom datetime range picker is intentionally deferred to EPIC-004 STAGE-004-005. Rationale: the 6 presets cover common debugging scenarios; custom range adds picker UX (validation, future dates, timezone) that belongs in the full logs-explorer redesign. Forward-reference added in same three places as D-DEFER-PAGINATION (EPIC-004 acceptance criterion, STAGE-002-015 cross-ref, code TODO).

### D-FILE-NAMING
Component replacing the placeholder: `apps/ui/src/routes/integrations/DockerContainerLogsViewer.tsx`. Old `ContainerLogsPlaceholder.tsx` is DELETED (not left as a stub). Router updated to point at new component.

### D-API-CONTRACT
Endpoint: `GET /api/integrations/docker/containers/{name}/logs?since=<duration>&limit=<int>`
- `since`: optional duration string `Xs|Xm|Xh|Xd`, default `15m`, hard-capped at `7d`. Invalid values ā†’ 422.
- `limit`: optional int, default 500, hard-capped at 500 (clamped server-side, not 5000 as in original spec).
- Auth: session required (no API-token scope required for read).
- Response (200): `{container_name, log_status, lines, truncated, window_start, window_end}` per spec's response shape.
- 404 when container name not in `targets` table (`log_status: "container_unknown"` body still included).
- 503 when VL unreachable (`log_status: "vl_unavailable"` body still included, no `lines`).

### D-LOGS-COLUMN-WIRING
The Logs column in `ContainerGridRow.tsx` (desktop) and `ContainerGridCard.tsx` (mobile) is wired to render `<Link to="/integrations/docker/containers/$name/logs">View logs ā†’</Link>`. The `logs_url?` field on `ContainerRow` is NOT populated by the backend ā€” link is constructed client-side from the container `name`. Stage spec implied `logs_url` from backend; this decision keeps the URL construction client-side (simpler, no backend change for a deterministic URL).

Frontend-only stage. Refinement = 3a (dev rig) only. No 3b (prod-rig real-data delta) needed ā€” the VL query and label vocabulary are the same in dev and prod. Desktop AND Mobile viewport approval still required.
