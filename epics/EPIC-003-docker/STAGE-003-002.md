# STAGE-003-002: Vector container-log ingestion fix + opt-out exclude_containers wiring

## Status: Not Started

## Overview

Fix the long-standing gap from STAGE-001-016: the vector `docker_logs` source hardcodes `include_containers = []` which means **zero containers are tailed**. Per-container stdout/stderr is NOT actually flowing into VictoriaLogs despite STAGE-001-016 claiming the deliverable was complete. This stage fixes that — vector tails ALL containers on the docker socket by default. An opt-out `exclude_containers` env var is provided for noise reduction.

Container LOG INGESTION is the prerequisite for STAGE-003-011 (per-container log viewer) and for EPIC-004's anomaly-detection work. EPIC-003.md cross-stage criterion `Docker log requirements` (added 2026-05-21) #1 lives here.

**Vertical slice:** after `make dev-prod` (or equivalent prod restart), querying VictoriaLogs for any running container's logs returns recent stdout/stderr lines. E.g.: `_stream_id:* AND service:"homeassistant"` (or whatever stream/service labels vector assigns) returns log entries. Specifically — for every container currently running on the host, VL contains at least one line within ~30s of stack restart.

**Spec references:** §3.2 (sidecars — vector); §4.3 (Logs data flow); §6.3 (VictoriaLogs streams); STAGE-001-016 (the original vector setup; its docker_logs source). EPIC-004.md Docker log requirements §1.

## Concrete deliverables

### 1. Vector template fix — `deploy/vector/vector.toml.template`

The current template has:
```toml
[sources.docker_logs]
type = "docker_logs"
include_containers = []
exclude_containers = []
auto_partial_merge = true
```

`include_containers = []` is interpreted by vector as "include zero containers". Remove the `include_containers` line entirely (or set it to `null`, depending on vector's TOML semantics — Build phase verifies). With `include_containers` omitted, vector tails ALL containers visible via the docker socket — which is what we want for auto-discovery.

Replace `exclude_containers = []` with a template placeholder that the render-on-boot path substitutes:
```toml
[sources.docker_logs]
type = "docker_logs"
# include_containers omitted: tail ALL containers visible via the docker socket.
exclude_containers = ${VECTOR_DOCKER_EXCLUDE}
auto_partial_merge = true
```

The placeholder substitution renders to a TOML array (e.g., `[]` for the default, or `["noisy-container-1", "another"]` if the user has set the env var).

### 2. Render-on-boot substitution

Extend `apps/monitor/homelab_monitor/kernel/cron/render.py` (or rename to a more general `kernel/render/vector.py` if appropriate — Build phase decides) to handle the new placeholder.

Current behavior: single placeholder `${CRON_EVENTS_INGEST_TOKEN}` substituted at render-on-boot time.

New behavior: also substitute `${VECTOR_DOCKER_EXCLUDE}`. Source:
- Read env var `VECTOR_DOCKER_EXCLUDE` (CSV string, default empty).
- Parse into a list (e.g., `"foo,bar"` → `["foo", "bar"]`).
- Render as TOML array literal: `["foo", "bar"]` or `[]` if empty.

If multiple template placeholders are needed in this stage and future ones, refactor the render function to a dict-based substitution (`{"CRON_EVENTS_INGEST_TOKEN": ..., "VECTOR_DOCKER_EXCLUDE": ...}`) — cleaner than chained `.replace()` calls.

### 3. Environment variable wiring

Add to `deploy/dev/dev.env.example`:
```
# Comma-separated list of container names to EXCLUDE from log ingestion.
# Default: empty (tail every container). Use to silence noisy/sensitive containers.
VECTOR_DOCKER_EXCLUDE=
```

Add to `deploy/compose/.env.example` (if it exists; otherwise document in CLAUDE.md):
```
VECTOR_DOCKER_EXCLUDE=
```

The compose file (`deploy/compose/docker-compose.yml`) MUST pass this env var to the monitor container so render-on-boot can read it. Verify the existing `environment:` block on the monitor service includes `VECTOR_DOCKER_EXCLUDE=${VECTOR_DOCKER_EXCLUDE:-}`.

### 4. Tests

`apps/monitor/tests/test_vector_render.py` (new OR extend existing test):
- Empty VECTOR_DOCKER_EXCLUDE → `exclude_containers = []` in rendered TOML.
- `VECTOR_DOCKER_EXCLUDE=foo` → `exclude_containers = ["foo"]`.
- `VECTOR_DOCKER_EXCLUDE=foo,bar` → `exclude_containers = ["foo", "bar"]`.
- Malformed input (extra whitespace, trailing commas) handled gracefully (strip + filter empty).
- Atomic-write pattern preserved (tempfile + os.replace).
- Group-own + mode 0o640 preserved.

Integration test (extend `apps/monitor/tests/integration/`):
- After bringing up the test rig + vector + a known fixture container, query VL for that container's logs within a bounded wait — confirm log lines arrive.

### 5. Verify the gap is fixed

Document an acceptance probe in the stage's Build phase:
1. Bring up prod stack: `make dev-prod`.
2. Wait ~30s.
3. Query VL: `docker compose exec victorialogs wget -qO- 'http://127.0.0.1:9428/select/logsql/field_values?query=*&field=service&limit=50'`.
4. Confirm container names appear (not just journald `service` values like `docker.service`, `cron.service`).

If you set `VECTOR_DOCKER_EXCLUDE=homelab-cadvisor`, the cadvisor container's logs should NOT appear in VL after restart. (Cadvisor's logs are noisy; excluding it from log ingestion is a likely default in the host-overrides repo.)

## Decisions in this stage

The Design phase confirms the exact env var name, the parsing rules (CSV vs JSON vs other), whether to support glob patterns, whether to refactor the render function to dict-based substitution, and whether the env var is monitor-side OR vector-side (i.e., does the monitor render the config, or does vector read the env directly?).

## Locked Design Decisions

D-OPT-OUT-MODEL — vector tails ALL containers by default (include_containers omitted); `VECTOR_DOCKER_EXCLUDE` is the opt-out mechanism. NOT an opt-in (include) model. User decision: "we can just rely on exclude." (EPIC-003 brainstorm 2026-05-21.)

D-RENDER-AT-BOOT — `VECTOR_DOCKER_EXCLUDE` is substituted at render-on-boot time, NOT dynamically. Containers come/go without re-rendering; vector's docker_logs source watches the docker events stream and auto-attaches to new containers, auto-detaches when containers die. Re-rendering only needed when the EXCLUDE list itself changes.

D-NO-INCLUDE-PATTERNS-V1 — wildcard patterns in include_containers (e.g., `homelab-*`) are NOT supported in v1. Either a container is excluded by exact name or it's included. Glob support deferred.

D-NO-HOT-RELOAD — changing `VECTOR_DOCKER_EXCLUDE` requires a stack restart (monitor process re-renders the config, vector restarts to pick it up). Documented; no inotify watcher in v1.

D-FIXES-STAGE-001-016 — this stage fixes the gap from STAGE-001-016 where the deliverable "tails Docker container stdout/stderr" was declared complete but actually shipped non-functional (empty include_containers). The fix is small but is BLOCKING for STAGE-003-011 (log viewer) and EPIC-004 (log anomaly detection).
