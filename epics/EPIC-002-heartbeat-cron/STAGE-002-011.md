# STAGE-002-011: `cron_runs` table + per-run history schema + heartbeat `run_id` threading

## Status: Not Started

**(New stage. EPIC-002 re-opened on 2026-05-19 to add cron run history & run-log viewing — see `docs/superpowers/specs/2026-05-19-cron-run-logs-design.md`. First of five appended stages STAGE-002-011 … STAGE-002-015.)**

## Overview

Lay the backend foundation for cron run history: a new `cron_runs` table that records **one row per cron invocation**, plus the heartbeat-receiver changes that let A-mode (wrapper-installed) crons create and close run rows synchronously.

Today `heartbeats_state` keeps only the *last* run's timestamp/duration/exit-code plus aggregate counters — individual run history is discarded. This stage adds the per-invocation table and wires the A-mode synchronous-write path into it. The B-mode (log-scrape) write path and the asynchronous VL-enrichment reconciler land in later stages; this stage delivers the table, the repository, and the heartbeat-receiver `run_id` threading only.

**Vertical slice:** with the new schema in place, a `POST /api/hb/{fingerprint}/start?run_id=<uuid>` followed by `POST /api/hb/{fingerprint}/ok?run_id=<uuid>&duration=<s>` produces exactly one `cron_runs` row for that cron, transitioning `running` → `ok`, with `started_at`, `ended_at`, `duration_seconds`, and `vl_window_*` populated. A `/ok` with a `run_id` that never had a `/start` (a normal case — heartbeat POSTs are best-effort and lossy) still produces a single closed row via UPSERT.

**Spec references:** `docs/superpowers/specs/2026-05-19-cron-run-logs-design.md` §3 (architecture), §4 (data model — `cron_runs`), §4 (`run_id` on heartbeat events), D1, D2, D3, D9, D13, D14.

## Concrete deliverables

### 1. `cron_runs` table + alembic migration

New migration `apps/monitor/alembic/versions/0015_cron_runs.py` (the head at spec time is `0014`; confirm the head and renumber if it has advanced). Creates the `cron_runs` table:

| Column | Type | Null | Notes |
| --- | --- | --- | --- |
| `run_id` | TEXT | NOT NULL | **PRIMARY KEY.** UUID. A-mode: generated by the wrapper (STAGE-002-012). B-mode: generated server-side (STAGE-002-013). |
| `cron_fingerprint` | TEXT | NOT NULL | References `crons.fingerprint`. Indexed. |
| `source` | TEXT | NOT NULL | `wrapper` (A-mode) or `logscrape` (B-mode). This stage only writes `wrapper`. |
| `state` | TEXT | NOT NULL | `running` / `ok` / `fail` / `unknown`. |
| `started_at` | TEXT | NOT NULL | UTC ISO-8601. |
| `ended_at` | TEXT | NULL | UTC ISO-8601. NULL while `state=running`. |
| `duration_seconds` | REAL | NULL | Set at close. |
| `exit_code` | INTEGER | NULL | A-mode: from the `/ok` (exit 0) or `/fail` call — the wrapper sends `exit_code` on both. B-mode: from the `exit=` syslog tag if present (STAGE-002-013). |
| `vl_window_start` | TEXT | NULL | UTC ISO-8601 — lower bound for the VL log query. |
| `vl_window_end` | TEXT | NULL | UTC ISO-8601 — upper bound. NULL while running. |
| `overlapping` | INTEGER | NOT NULL DEFAULT 0 | B-mode only (STAGE-002-013). Always `0` for A-mode rows. |
| `enriched_at` | TEXT | NULL | UTC ISO-8601 — set by the reconciler (STAGE-002-013). NULL = not yet enriched. |
| `line_count` | INTEGER | NULL | Set by the reconciler (STAGE-002-013). |
| `byte_count` | INTEGER | NULL | Set by the reconciler (STAGE-002-013). |
| `content_digest` | TEXT | NULL | Set by the reconciler (STAGE-002-013). |
| `anomaly_flags` | TEXT | NOT NULL DEFAULT '' | Comma-separated heuristic flags (STAGE-002-014). Empty string = none. |

**Indexes (all created in this migration):**
- `(cron_fingerprint, started_at DESC)` — serves the run-history list query (STAGE-002-014).
- Partial index on `enriched_at IS NULL AND state != 'running'` — serves the reconciler's enrichment work-queue (STAGE-002-013).
- `(cron_fingerprint, state)` — serves the reconciler's window-finalize scan of open runs (STAGE-002-013).

The migration must be idempotent-guarded (`batch_alter_table` for SQLite) and have a working `down_revision` chaining from the prior head, with a round-trip (up/down) test.

> **Note on later-stage columns:** the columns `overlapping`, `enriched_at`, `line_count`, `byte_count`, `content_digest`, `anomaly_flags` are CREATED in this stage's migration (so there is exactly one `cron_runs` migration) but are only WRITTEN by STAGE-002-013 / STAGE-002-014. This stage leaves them at their defaults / NULL.

### 2. `CronRunRepository`

New module `apps/monitor/homelab_monitor/kernel/cron/run_repository.py` (mirrors the existing `cron/repository.py` style — Core SQL, not ORM):

- `insert_run(...)` — INSERT a `cron_runs` row in `state=running`. INSERT-OR-IGNORE on `run_id` conflict (idempotent — a duplicate `run_id` on `/start`, e.g. a wrapper retry, is a no-op, not an error).
- `close_run(run_id, state, ended_at, duration_seconds, exit_code, vl_window_end)` — UPSERT (INSERT-or-UPDATE): if the row exists (the normal `/start`-then-`/ok` path), UPDATE it; if it does not exist (a lost `/start` — a normal case, see Locked Decision D-LOST below), INSERT a closed row. When inserting on the lost-`/start` path, `started_at` is derived best-effort as `ended_at - duration_seconds`, and `vl_window_start` is derived alongside it to the same value (`ended_at - duration_seconds`) so the run-log query in STAGE-002-014 has a lower bound.
- `get_run(run_id)` — fetch one run record.
- `list_runs(cron_fingerprint, limit, cursor, state_filter)` — paginated history list (consumed by STAGE-002-014's API; the repository method is defined here so the table has a complete read surface, even though the endpoint ships in 014).
- A `CronRunRecord` dataclass mirroring the table columns.

### 3. Heartbeat receiver `run_id` threading

`apps/monitor/homelab_monitor/kernel/heartbeat/` — the receiver's `/start`, `/ok`, `/fail` handlers:

- Add an **optional** `run_id` query parameter to all three heartbeat query-schema classes: `HeartbeatStartQuery`, `HeartbeatOkQuery`, `HeartbeatFailQuery` (in `kernel/heartbeat/schemas.py`). `HeartbeatStartQuery` is currently intentionally empty — it is the model designed for exactly this kind of additive param.
- Additionally add an **optional** `exit_code` query parameter to `HeartbeatOkQuery` (in addition to `run_id`) so a successful A-mode run can carry its exit code — the wrapper knows it is `0`. `HeartbeatFailQuery` already conceptually carries exit info; confirm it has an `exit_code` param too. This stage therefore adds `run_id` to all three query classes AND `exit_code` to `HeartbeatOkQuery`.
- `run_id` is a free-text string (validated for reasonable length / character set — it becomes a PRIMARY KEY; reject absurd values, but do NOT require strict UUID format, since the value is client-supplied).
- **When `run_id` is present** (A-mode, upgraded wrapper from STAGE-002-012):
  - `/start` → `CronRunRepository.insert_run(...)` (`state=running`, `started_at=now`, `source=wrapper`, `vl_window_start=now`).
  - `/ok` / `/fail` → `CronRunRepository.close_run(...)` (`state=ok|fail`, `ended_at=now`, `duration_seconds` from the query, `exit_code` from the query — `/ok` stores `exit_code=0` (a successful run's exit code, which the wrapper knows and sends), `/fail` stores the actual non-zero code, `vl_window_end=now`).
- **When `run_id` is absent** (legacy wrapper, or the `/register` path) → behavior is **exactly as today**: no `cron_runs` row is created. This keeps legacy wrappers fully working until their one-time re-install in STAGE-002-012.
- The `cron_runs` write happens **in addition to** the existing `heartbeats_state` update — it does not replace or alter any existing heartbeat-receiver behavior. The `cron_runs` row and the `heartbeats_state` update for the same A-mode invocation share the `run_id`.
- `openapi.json` regenerates as a result of the new query params (run `bash scripts/export-openapi.sh`).

## Locked Design Decisions

D-TABLE — `cron_runs` is a NEW per-invocation table; `heartbeats_state` is unchanged (it remains the derived last-run/aggregate view that `HeartbeatStateCollector` reads for vmalert metrics). No existing `heartbeats_state` data is migrated. (Spec D3.)

D-MIGRATION — exactly ONE `cron_runs` migration (`0015_cron_runs.py`), creating the table and ALL its columns and ALL three indexes — even the columns/index that only later stages write. This avoids a migration-per-stage sprawl.

D-RUNID — `run_id` is the TEXT PRIMARY KEY. A-mode value is wrapper-generated and arrives over the network on the heartbeat query string. `/start` uses INSERT-OR-IGNORE (a duplicate `run_id` is an idempotent no-op). `/ok` and `/fail` use UPSERT (INSERT-or-UPDATE) so a lost `/start` — a normal case, because the wrapper's heartbeat POSTs are best-effort `curl --max-time 5 ... || true` — still yields exactly one closed row. (Spec §4 "`run_id` on heartbeat events", D9.)

D-LOST — a `/ok` or `/fail` carrying a `run_id` with no prior `/start` row is EXPECTED, not an error. The UPSERT inserts a closed row; `started_at` is derived as `ended_at - duration_seconds`, and `vl_window_start` is derived alongside it to the same value (`ended_at - duration_seconds`) — so STAGE-002-014's run-log query has a lower bound on the lost-`/start` path.

D-ABSENT — `run_id` absent on a heartbeat call → no `cron_runs` row, existing heartbeat behavior byte-for-byte unchanged. Legacy wrappers keep working until their STAGE-002-012 re-install.

D-ADDITIVE — the `cron_runs` write is additive: it happens alongside, never instead of, the existing `heartbeats_state` update.

D-OKEXIT — a successful A-mode run records `exit_code=0` on its `cron_runs` row; the wrapper sends `exit_code` on `/ok` as well as `/fail`, and `HeartbeatOkQuery` carries an optional `exit_code` param. Rationale: STAGE-002-014's `exit_code_changed` anomaly rule needs the success-path exit code or it can only ever compare failure codes.
