# STAGE-002-008: B-mode log-scrape — REWORKED to match logs by fingerprint heuristic, runs always (no observe-mode gating)

## Status: Not Started

**(Was STAGE-002-004. Renumbered after the cron-derived-state-redesign brainstorm on 2026-05-11.)**

## Overview

Ship the log-scrape pipeline: vector + journald + a `cron-log-collector` plugin that watches VictoriaLogs for cron output and updates `heartbeats_state` rows. After this stage, every cron in the registry — regardless of whether the user installed a wrapper — gets `last_seen_state` updates from log evidence whenever the host runs the cron.

**Reworked from the original STAGE-002-004 plan in two ways:**

1. **No `integration_mode` gating.** The original plan skipped log-scrape for crons in `heartbeat`-only mode. The derived-state model has no `integration_mode` field; log-scrape ALWAYS runs for any registered cron. If the cron also has heartbeat evidence, the timestamps tie-break naturally (most recent wins; cross-check happens in STAGE-002-010 wrapper-health alert).
2. **Match logs to crons by fingerprint heuristic.** The original plan matched by `(host, command)` index. New plan: parse the log line into `(host, command_extracted)`, search the `crons` table for any fingerprint where `crons.host = host AND crons.command = command_extracted`. If multiple match (unlikely; same host + same command but different schedules), update all of them — they're all valid candidates for the same execution event.

Vertical slice: with the rig running, plant a syslog line `Apr 30 04:10:01 host CRON[12345]: (root) CMD (/storage/scripts/cron/backup.sh) exit=0` → wait up to 60s → assert `heartbeats_state.last_ok_at` updated for the matching fingerprint row.

## Concrete deliverables

1. **Migration** at `apps/monitor/alembic/versions/XXXX_stage_002_008_logscrape_watermark.py`
   - ADD COLUMN `heartbeats_state.last_log_watermark TEXT` (NULL until first scrape; serialized as `{timestamp}|{line_hash}`).
   - ADD COLUMN `heartbeats_state.observed_runs_total INTEGER NOT NULL DEFAULT 0` (lifetime count from B-mode evidence).

2. **Vector configuration** at `deploy/vector/cron-sources.toml`
   - Source `cron_journald` (filter on `_SYSTEMD_UNIT in ["cron.service", "crond.service"]`).
   - Source `cron_syslog` (file: `/var/log/syslog`; filter `CRON[`).
   - Transform `cron_parsed` (VRL): extract `pid, user, command, exit_code`.
   - Sink `cron_to_vl` (VL with `_stream=cron`).
   - Test fixture: noisy-logger `POST /cron-event` plants log lines for integration tests.

3. **Cron log collector plugin** at `apps/monitor/homelab_monitor/plugins/collectors/cron_log_collector.py`
   - In-process Python plugin; collector kind ASYNC; interval 30s; concurrency_group `logs`.
   - On each run:
     - Query VL for `{service="cron"} AND _time > {oldest_watermark - 5m}`.
     - For each line, parse: `host`, `command`, `user`, `exit_code`, `timestamp`.
     - Search `crons` for any fingerprint with matching `(host, command)`. If 0 matches → log + skip. If 1+ matches → process each.
     - Compute line hash `sha256(f"{host}|{timestamp_iso}|{command}|{exit_code}").hexdigest()[:16]`.
     - Skip if `(timestamp, line_hash) <= heartbeats_state.last_log_watermark` for that fingerprint.
     - Otherwise: call `HeartbeatRepo.record_ok` (exit_code == 0) or `record_fail` (non-zero); advance watermark; increment `observed_runs_total`; audit entry with `source=log-scrape`.
   - **No mode gating.** Process all crons regardless of wrapper-installed status. (The wrapper-health alert in STAGE-002-010 uses log-scrape evidence as one of its two inputs.)
   - Emits `homelab_collector_run_*` and `homelab_cron_logscrape_matches_total{host}` counter.

4. **Log line parser** at `apps/monitor/homelab_monitor/kernel/cron/log_parser.py`
   - `parse_cron_log_line(line: str) -> CronLogEvent | None` returns `(host, command, user, ts, exit_code | None)`.
   - Handles Debian/Ubuntu syslog, systemd journald, cronic-wrapped exit codes.
   - Returns None for non-cron lines.

5. **B-mode + heartbeat coexistence** in `HeartbeatRepo`
   - Both data sources update `heartbeats_state` independently. Tie-breaker: most-recent-timestamp-wins for `current_state`.
   - `current_streak` increments only if the new event is genuinely a new run (heuristic: > 60s gap from last event of the same outcome class).
   - No mode gating (the original plan's `integration_mode == 'heartbeat'` skip is removed).

6. **Tests**
   - `apps/monitor/tests/kernel/cron/test_log_parser.py`: format coverage.
   - `apps/monitor/tests/kernel/plugins/collectors/test_cron_log_collector.py`:
     - Single ok line → `current_state=ok`, watermark advanced.
     - Replay → idempotent skip.
     - Unknown command → no match → no row updated.

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **Exit-code source: vanilla cron syslog vs require `cronic` / wrapper?**
   - **Recommended: best-effort from vanilla cron + document the limitation.** If no exit code parseable, default to `ok`. Users wanting hard exit-code tracking opt-in to the wrapper (STAGE-002-009).
   - Alternative: require cronic. Stronger guarantees but breaks the "works on unmodified crons" promise.

2. **Vector source: journald vs syslog vs both?**
   - **Recommended: both, with journald preferred.** journald is structured. Syslog is the fallback for non-systemd hosts.

3. **Idempotency mechanism: per-cron `(timestamp, content_hash)` watermark vs separate events table?**
   - **Recommended: per-cron watermark (single column on heartbeats_state).** Cheap; no separate table.

4. **Multiple-match policy: when 1 log line matches 2+ fingerprints with same `(host, command)`?**
   - **Recommended: update all matching fingerprints.** Each is a valid candidate for the same execution event. Same audit row count → audit_log doesn't bloat.
   - Alternative: pick the one with the most-recent activity (last `last_seen_at`). More precise but heuristic.
