## Overview

Ship the log-scrape pipeline: vector + journald + a `cron-log-collector` plugin that watches VictoriaLogs for cron output and updates `heartbeats_state` rows. After this stage, every cron in the registry â€” regardless of whether the user installed a wrapper â€” gets `last_seen_state` updates from log evidence whenever the host runs the cron.

**Reworked from the original STAGE-002-004 plan in two ways:**

1. **No `integration_mode` gating.** The original plan skipped log-scrape for crons in `heartbeat`-only mode. The derived-state model has no `integration_mode` field; log-scrape ALWAYS runs for any registered cron. If the cron also has heartbeat evidence, the timestamps tie-break naturally (most recent wins; cross-check happens in STAGE-002-010 wrapper-health alert).
2. **Match logs to crons by fingerprint heuristic.** The original plan matched by `(host, command)` index. New plan: parse the log line into `(host, command_extracted)`, search the `crons` table for any fingerprint where `crons.host = host AND crons.command = command_extracted`. If multiple match (unlikely; same host + same command but different schedules), update all of them â€” they're all valid candidates for the same execution event.

Vertical slice: with the rig running, plant a syslog line `Apr 30 04:10:01 host CRON[12345]: (root) CMD (/storage/scripts/cron/backup.sh) exit=0` â†’ wait up to 60s â†’ assert `heartbeats_state.last_ok_at` updated for the matching fingerprint row.

## Locked Design Decisions (2026-05-15)

**CRITICAL OVERRIDE NOTE:** The decisions below OVERRIDE the pre-documented recommendations in the Â§Decisions section and Â§Concrete deliverables. These are locked pending Build phase implementation.

### Architecture: Option D â€” Vector push (NOT poll-VictoriaLogs)

The original spec's "in-process collector polls VictoriaLogs on a lookback window + content-hash dedup" is REPLACED. Instead:

- Vector parses cron log lines via a VRL transform into structured events and pushes them to a NEW internal endpoint `POST /api/internal/cron-events` on the monitor app, with `acknowledgements.enabled: true` and a disk buffer (`buffer.type: disk`) on the HTTP sink.
- The monitor app's ingest endpoint does an idempotent `INSERT ... ON CONFLICT IGNORE` keyed on the journald `__CURSOR` field (a globally-stable per-entry ID; Vector emits it via `emit_cursor: true`). This replaces the content-hash watermark/recent-hash-set entirely â€” there is NO polling loop and NO `recent_log_hashes`/`last_log_watermark` column.
- **Rationale:** VictoriaLogs has no cursor/stable-record-ID API (researched + confirmed), so poll-and-dedup is fragile. Vector's journald source already checkpoints its read position and only advances after the app ACKs â€” standard at-least-once + idempotent-write pattern. Vector still ALSO sinks raw logs to VictoriaLogs for the general logs UI (unchanged); this adds a parallel structured-cron-event sink.

### D1 â€” Exit-code source / what log evidence asserts

**OVERRIDES the pre-doc'd "default to ok".** Vanilla cron logs only a dispatch line (`CRON[pid]: (user) CMD (command)`) with NO exit code and no completion line â€” it proves "cron fired the job", not "the job succeeded". So a bare CMD line records a NEUTRAL "observed run": a new `HeartbeatRepo.record_observed_run()` increments `observed_runs_total` and sets `last_observed_run_at`, and does NOT touch `current_state`/`last_ok_at`/`last_fail_at`/`current_streak`. ONLY a wrapper-tagged line carrying a real `exit=N` calls `record_ok` (exit 0) / `record_fail` (non-zero). The acceptance test must be rewritten: plant a VANILLA line (no `exit=`) and assert `observed_runs_total` incremented + `last_observed_run_at` set + `current_state` UNCHANGED; a second test plants a wrapper-tagged `exit=0` line and asserts `current_state=ok`.

### D2 â€” Vector source

**OVERRIDES the pre-doc'd "both journald + syslog".** journald-only by DEFAULT (intelnuc is systemd; journald is the complete source). A syslog file source is a MUTUALLY-EXCLUSIVE fallback (config flag) for non-systemd hosts â€” never both on one host (both would double-ingest every cron run). Document the flag in `docs/architecture/cron-logscrape.md`.

### D3 â€” Idempotency

**RESOLVED BY the Option D architecture.** Idempotency = `INSERT ... ON CONFLICT IGNORE` on the journald `__CURSOR` at the ingest endpoint. The pre-doc'd `last_log_watermark` / recent-hash-set column is NOT created.

### D4 â€” Multiple-match policy

**OVERRIDES the pre-doc'd "update all".** When one cron event matches 2+ cron fingerprints on `(host, log_match_key)`, SKIP the state write, emit a `homelab_cron_logscrape_ambiguous_total` metric, and structured-log a warning listing the candidate fingerprints. No fan-out (only one physical process ran; updating all manufactures phantom evidence).

### D5 â€” Command matching (the make-or-break correctness fix)

The original "exact `crons.command == log_command` match" is REPLACED. `crons.command` is stored SCRUBBED (secret-redacted via `kernel/cron/secrets.py`); the log line has the RAW command â€” exact match fails 100% for any secret-bearing cron.

**Fix:** a shared function `canonical_log_key(command)` in `kernel/cron/` = `scrub_secrets(command)` then whitespace-collapse then strip one wrapping `(...)` layer (`scrub_secrets` is idempotent so both sides converge). Add a new INDEXED column `crons.log_match_key` populated by the `cron_discoverer` at discovery time; the migration backfills it for existing rows. The ingest endpoint applies the IDENTICAL `canonical_log_key()` to the raw log command and matches on `(host, log_match_key)`.

**Known limitation (document in `cron-logscrape.md`):** cron's `%`-substitution makes a `%`-using cron's disk vs logged command fundamentally differ â€” the canonicalizer cannot fix that; such crons won't match. Accepted as a documented B-mode limitation.

### Schema changes

- New column `crons.log_match_key TEXT` (indexed `(host, log_match_key)`; backfilled).
- New columns on `heartbeats_state`: `observed_runs_total INTEGER NOT NULL DEFAULT 0`, `last_observed_run_at TEXT NULL`.
- NO `last_log_watermark` / `recent_log_hashes` column (Option D removed the need).

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **Exit-code source: vanilla cron syslog vs require `cronic` / wrapper?**
   - **Recommended: best-effort from vanilla cron + document the limitation.** If no exit code parseable, default to `ok`. Users wanting hard exit-code tracking opt-in to the wrapper (STAGE-002-009).
   - Alternative: require cronic. Stronger guarantees but breaks the "works on unmodified crons" promise.

2. **Vector source: journald vs syslog vs both?**
   - **Recommended: both, with journald preferred.** journald is structured. Syslog is the fallback for non-systemd hosts.

3. **Idempotency mechanism: per-cron `(timestamp, content_hash)` watermark vs separate events table?**
   - **Recommended: per-cron watermark (single column on heartbeats_state).** Cheap; no separate table.

4. **Multiple-match policy: when 1 log line matches 2+ fingerprints with same `(host, command)`?**
   - **Recommended: update all matching fingerprints.** Each is a valid candidate for the same execution event. Same audit row count â†’ audit_log doesn't bloat.
   - Alternative: pick the one with the most-recent activity (last `last_seen_at`). More precise but heuristic.
