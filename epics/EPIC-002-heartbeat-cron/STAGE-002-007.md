# STAGE-002-007: Cron auto-discovery — REWORKED to emit fingerprints, populate registry directly, populate disk-source fields

## Status: Complete

**(Was STAGE-002-003. Renumbered after the cron-derived-state-redesign brainstorm on 2026-05-11.)**

## Overview

Stand up a `cron-discoverer` plugin that scans the host's crontab files and writes directly into the `crons` registry — fingerprint-keyed, populating `host`, `source_path`, `schedule`, `command`, `cadence_seconds`, `schedule_canonical`, and `last_discovered_at`. **No suggestion-queue interaction**: per the derived-state redesign, cron discovery bypasses suggestions entirely. Crons appear in the registry as soon as discovery sees them.

Sources scanned: `/etc/crontab`, `/etc/cron.d/*`, `/etc/cron.{daily,hourly,weekly,monthly}/*`, root + per-user crontabs (via `crontab -l -u <user>`).

After this stage, the user's known crons (`/storage/scripts/startup/startup.sh @reboot`, `/storage/scripts/rtlamr-watchdog.sh 17 * * * *`, `/etc/cron.d/certbot 0 12 * * *`, `/storage/scripts/cron/backup.sh 10 4 * * *`) appear in the Crons tab within one collector cycle, each with the correct fingerprint and source_path.

**Spec references:** `docs/superpowers/specs/2026-05-11-cron-derived-state-redesign.md` §Identity, §EPIC-002 stage sequence (007).

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **Discovery cadence: periodic poll (every 5min) vs event-driven via inotify on `/etc/cron.d/`?**
   - **Recommended: periodic poll at 300s.** Simpler; cron file changes are infrequent; inotify in a container requires `--privileged` or capabilities tweaks.
   - Alternative: inotify watcher daemon. Faster reaction but invasive container config.

2. **`@reboot` job tracking: track via "should have been seen since last boot" or skip?**
   - **Recommended: track. The `@reboot` literal is part of the fingerprint; the cron appears in the registry. The lateness rule for `@reboot` crons is "did this fingerprint heartbeat after the host's last boot?"** Compares `heartbeats_state.last_ok_at` to a self-monitor `last_boot_at` (from EPIC-014).
   - Alternative: skip — `@reboot` crons can't be reliably heartbeated without a heartbeat early in the boot sequence.

3. **Per-user crontab access: bind-mount `/var/spool/cron:ro` vs host-side daemon vs require nsenter?**
   - **Recommended: bind-mount `/var/spool/cron/crontabs:ro` (Debian/Ubuntu) and `/var/spool/cron:ro` (RHEL/CentOS) under `/host/var/spool/cron`.** Container reads files directly. World-readable concerns addressed in Design.
   - Alternative: host-side daemon over unix socket. Overkill for v1.

4. **Stale row threshold: 2× discovery interval (10min) vs 12hr vs configurable?**
   - **Recommended: 2× interval (10min) for the "no longer discovered" badge.** Catches "cron was deleted from disk" within one full cycle of certainty. User-configurable via env var if anyone complains.
   - Alternative: 12hr — gentler, allows for transient mount issues; risks staleness going unnoticed.

## Locked Design Decisions (2026-05-13)

### D1 — Container perms strategy
**Locked: Dedicated host user, one-time sudo setup script.**
- Add `scripts/host-setup.sh` (idempotent, runs as sudo). Creates host user `homelab-monitor` (or similar — Build phase picks the name), adds it to `crontab` group, sets up read ACLs on `/var/spool/cron/crontabs/*` so the user can read all per-user crontabs.
- Add `HM_CRON_HOST_UID` and `HM_CRON_HOST_GID` to `deploy/dev/dev.env.example` (and document for production env).
- In `deploy/compose/docker-compose.yml`, add `user: "${HM_CRON_HOST_UID}:${HM_CRON_HOST_GID}"` to the monitor service — overrides the Dockerfile's `USER homelab` directive at runtime.
- Bind-mounts: `/etc:/host/etc:ro` + `/var/spool/cron/crontabs:/host/var/spool/cron/crontabs:ro` (path is Debian/Ubuntu-specific; document the override for other distros).
- The host-setup script will extend across future epics/stages as additional perms are needed (e.g., systemd, NAS mount perms). This stage establishes the pattern.
- **Why:** Keeps container non-root (aligns with auto-fix safety model's low-privilege spirit). One-time host setup is acceptable for single-operator homelab.
- **Alternative rejected:** Container-as-root (B) — increases blast radius for marginal convenience. `group_add` alone won't work because individual spool files are `0600`, not group-readable by default.

### D2 — Scan-set semantics for 007A reconciliation
**Locked: Best-effort / partial scan.**
- Scanner returns `CronScanResult(found_fingerprints: frozenset[str], partial: bool, errors: list[CronScanError])`.
- `partial=True` iff at least one source file errored (perm denied, parse failure, missing file mid-scan).
- New fingerprints from successfully-read files: INSERT regardless of `partial`.
- `last_discovered_at` bumps: applied regardless of `partial`.
- STAGE-002-007A's soft-delete reconciliation gates on `partial=False` (skips soft-delete pass if partial).
- **Why:** A single bad user crontab should not freeze discovery+reconciliation indefinitely. New crons in readable files still get registered.
- **Cross-stage edit needed:** Update STAGE-002-007A.md to clarify "scan must complete without an unhandled exception that prevented running the scan at all" — per-file errors do NOT count as scan failure for 007A purposes.

### D3 — Discovery cadence
**Locked: Periodic 300s poll, configurable.**
- `BaseCollector` plugin with default `interval=timedelta(seconds=300)`, `concurrency_group="discovery"`.
- Configurable via env var `HM_CRON_DISCOVERY_INTERVAL_SECONDS` (plumbed through `CollectorConfig.interval_seconds`).
- E2e test sets short interval (e.g., 5s) via env-override.
- **Why:** Uses existing scheduler protocol verbatim (free self-metrics, free quarantine, free failure-budget). 5-min lag from `crontab -e` to UI is acceptable for homelab.
- **Alternative rejected:** Hybrid (poll + inotify) — adds 2x test surface; inotify task doesn't fit Collector Protocol. Layerable later as non-breaking addition if lag becomes painful.

### D4 — Audit volume policy
**Locked: `crons.discover` on first-insert + `crons.discover.update` on non-bump field change; no audit on bump-only.**
- New fingerprint inserted → `crons.discover` audit row, `before=null`, `after=full_row_dict`, `who="system"`.
- Existing fingerprint, non-bump field changed (anything except `last_discovered_at`, `updated_at`, `wrapper_last_seen_at`, `last_seen_state`) → `crons.discover.update`, `before=old, after=new`. Rare in practice.
- Existing fingerprint, ONLY `last_discovered_at` bumped → NO audit row.
- **Why:** `last_discovered_at` is heartbeat-like (mechanical, idempotent). Audit log is for state changes. Volume in steady state: ~0 rows/day from discovery.
- **Deliberate divergence from STAGE-002-005 D10:** D10's "audit every refresh" pattern is right for operator-initiated wrapper refresh; wrong for mechanical periodic discovery.

### D5 — `@reboot` handling
**Locked: Track row with `cadence_seconds=0`; defer lateness to STAGE-002-010.**
- Scanner writes `schedule="@reboot"`, `schedule_canonical="@reboot"` (no croniter canonicalization), `cadence_seconds=0`.
- Existing code path `kernel/heartbeat/repository.py:132` already guards `if cadence_seconds <= 0: return None` — sentinel `0` is safe.
- Lateness rule for @reboot crons ("did this fingerprint heartbeat since the host's last boot?") becomes STAGE-002-010's responsibility.
- **Cross-stage edit needed:** STAGE-002-010.md acceptance criteria gets a new line: "@reboot crons (cadence_seconds=0) MUST have a vmalert rule that fires lateness when `last_heartbeat_at < host_boot_time`. Boot time source: `/host/proc/stat` btime line."

### D6 — Trigger surfaces (force-scan)
**Locked: API endpoint + CLI + frontend button.**
- `POST /api/crons/discover-now`: admin-only (session-auth), in-memory min 10s throttle between calls (returns 429 if throttled with `Retry-After` header). On success: returns 202 Accepted + scan-result summary `{found_count, partial, errors}`.
- CLI: `hm cron discover` (calls service code directly; bypasses HTTP — Build phase decides whether to call internal function or hit the endpoint).
- Frontend: "Discover now" button on `/inventory/crons` page. Toast feedback (success: "Discovery scan triggered." / throttle: "Discovery scan already triggered recently. Try again in N seconds.").
- E2e test: drives scans via the endpoint (cleaner than env-override hack).
- **Why:** Real operational value; convenient e2e testing. Replaces YAGNI position.

### D7 — Schedule normalization at fingerprint time
**Locked: Raw schedule in fingerprint; `schedule_canonical` populated for display.**
- `compute_fingerprint(host, source_path, schedule_raw, command)` — unchanged from existing contract.
- Discovery writes both `schedule` (raw, from crontab file) and `schedule_canonical` (croniter-normalized).
- `@hourly` ↔ `0 * * * *` edit → fingerprint changes → new row appears, old row goes stale → 007A soft-deletes old within 5 min. Acceptable churn.
- **Why:** STAGE-002-006 wrapper installer commits to raw-text fingerprinting. Discovery must match that contract. Changing to canonical-text fingerprinting would break wrapper convergence catastrophically.

### D8 — Cron line parser
**Locked: Hand-roll + croniter for schedule validation.**

Edge-case behavior:
| Input | Action |
|---|---|
| Blank line | Skip |
| `#`-prefixed line | Skip (cron does NOT support inline comments) |
| `KEY=VALUE` env var line (regex `^[A-Z_][A-Z0-9_]*\s*=.*$`) | Skip |
| `@reboot CMD` | Track; `schedule="@reboot"`; `cadence_seconds=0` |
| `@hourly`, `@daily`, `@weekly`, `@monthly`, `@yearly`, `@annually`, `@midnight` | Track; validate via croniter; raw goes to fingerprint, canonical computed via `canonicalize_schedule` |
| 5-field expr + cmd (user crontab) | Track; user = filename (`source_path = "crontab:<user>"`) |
| 5-field + user + cmd (`/etc/cron.d/*`, `/etc/crontab`) | Track; user = explicit field; `source_path = "/host/etc/cron.d/<filename>"` or `"/host/etc/crontab"` |
| `/etc/cron.{hourly,daily,weekly,monthly}/*` | Skip the directory entirely (these are scripts run by run-parts, not crontab entries) |
| `/etc/anacrontab` | Skip (different format) |
| Malformed schedule | Log error, scan continues, contributes `partial=True` |

## Approach Summary

`cron-discoverer` plugin under `apps/monitor/homelab_monitor/plugins/discoverers/` (new sibling directory to `collectors/`). Uses existing `Collector` Protocol. Registered in `builtin_registry.py`. Runs every 300s via scheduler. Scans `/host/etc/crontab`, `/host/etc/cron.d/*`, `/host/var/spool/cron/crontabs/*`. Parses each line, validates schedule via croniter, computes fingerprint via `compute_fingerprint`, upserts row with `last_discovered_at` bump. Returns `CronScanResult` with `partial` flag for 007A reconciliation.

New API endpoint `POST /api/crons/discover-now` triggers a scan on-demand (admin-only, 10s throttle). CLI command `hm cron discover` does the same. Frontend "Discover now" button on `/inventory/crons` calls the endpoint.

DB migration adds `last_discovered_at TEXT NULL` to `crons` table.

UI changes:
- New "Discover now" button on `/inventory/crons` page (with throttle-aware toast feedback).
- "Last discovered: <ts>" field added to CronDetail Disk source panel (cross-stage carryover from STAGE-002-006).

Audit verbs:
- `crons.discover` — first-insert of a new fingerprint by discovery.
- `crons.discover.update` — existing fingerprint, non-bump field changed.

## Risks / Things to Watch in Build

1. **Container user identity:** Dockerfile currently has `USER homelab` (UID 1000). Compose `user:` directive overrides at runtime. Verify the override actually wins.
2. **Bind-mount paths:** Debian/Ubuntu uses `/var/spool/cron/crontabs/`; document for other distros. Make path env-driven.
3. **`source_path` convention for user crontabs:** Use `"crontab:<user>"` (matches existing fingerprint module docstring). Pre-fix path for system crons.
4. **E2E test interval:** Use the new `/api/crons/discover-now` endpoint, not interval override hack.
5. **CLI vs API for `hm cron discover`:** Build phase picks. Simpler: CLI calls service code directly (skips HTTP/auth). Risk: code duplication if not factored cleanly.
