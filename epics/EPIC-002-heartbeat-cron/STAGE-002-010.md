# STAGE-002-010: vmalert rules — REWORKED to include wrapper-health alert via separate monitoring-health channel

## Status: Complete

**(Was STAGE-002-006. Renumbered after the cron-derived-state-redesign brainstorm on 2026-05-11.)**

## Overview

Close out EPIC-002 with (1) a heartbeat metric collector that reads from `heartbeats_state` and emits per-cron freshness metrics to the kernel's Prometheus registry, (2) vmalert rules for stale heartbeats + lateness ladder + flap detector, and (3) **a new wrapper-health rule** that fires `WrapperPossiblyStale` when a wrapper-installed cron's log-scrape evidence outpaces its heartbeat evidence by 3+ runs. The wrapper-health alert routes through a separate `monitoring-health` Alertmanager channel — distinct from the cron-health channel — so users can mute "your monitoring is broken" alerts independently from "your cron is broken" alerts.

Vertical slice 1: stop sending heartbeats for a known cron with cadence 60s and grace 60s; within 3 minutes a `HeartbeatStale` alert with severity `warning` appears in `/api/alerts` with `source_tool=vmalert-metrics`. Resume heartbeats; alert resolves within 60s.

Vertical slice 2: for a wrapper-installed cron, plant 3 log-scrape entries WITHOUT corresponding heartbeats; within 5 minutes a `WrapperPossiblyStale` alert appears in `/api/alerts` with `routing_channel=monitoring-health`.

**Reworked from the original STAGE-002-006 plan in three ways:**

1. **No `integration_mode` filter on metric emission.** The original plan emitted zero metrics for observe-mode crons; the new plan emits metrics for ALL registered crons (integration_mode is gone).
2. **New `WrapperPossiblyStale` rule** + Alertmanager route to a dedicated `monitoring-health` channel.
3. **`cadence_seconds` label exposed on all heartbeat metrics** (so future per-cron-cadence-scaled severity ladder can be implemented — currently using fixed thresholds per Decision 2 from the original stage).

**Spec references:** `docs/superpowers/specs/2026-05-11-cron-derived-state-redesign.md` §Wrapper health monitoring; §Grace and lateness; §3.1 (metrics).

## Concrete deliverables

1. **Heartbeat metric collector** at `apps/monitor/homelab_monitor/kernel/metrics/heartbeat_collector.py`
   - Class `HeartbeatStateCollector` invoked at each `/metrics` scrape.
   - For each `crons` row (joined with `heartbeats_state`):
     - `homelab_heartbeat_seconds_since_last_ok{fingerprint, host, name, cadence_seconds, wrapper_installed}` (gauge)
     - `homelab_heartbeat_seconds_since_last_fail{fingerprint, host, name, cadence_seconds, wrapper_installed}`
     - `homelab_heartbeat_current_streak{fingerprint, host, name, cadence_seconds, wrapper_installed}` (signed int)
     - `homelab_heartbeat_expected_next_seconds{fingerprint, host, name, cadence_seconds, wrapper_installed}` (negative = overdue)
     - `homelab_heartbeat_last_duration_seconds{fingerprint, host, name, cadence_seconds, wrapper_installed}`
     - `homelab_heartbeat_logscrape_count_since_last_heartbeat{fingerprint, host, name, wrapper_installed}` (NEW — drives wrapper-health rule)

   - **Emits metrics for ALL registered crons**, regardless of wrapper-installed status (no mode gating; mode field is gone).
   - Skips rows where `hidden_at IS NOT NULL` (suppress alerts on hidden crons).

2. **vmalert rule file** at `deploy/vmalert/metrics/heartbeats.yaml`
   - Rule group `homelab.heartbeats`:
     - `HeartbeatStale_Warning`: `homelab_heartbeat_expected_next_seconds < -60` for 60s. Severity `warning`. Labels: `routing_channel=cron-health`.
     - `HeartbeatStale_Error`: `homelab_heartbeat_expected_next_seconds < -300` for 60s. Severity `error`. `routing_channel=cron-health`.
     - `HeartbeatStale_Critical`: `homelab_heartbeat_expected_next_seconds < -3600` for 60s. Severity `critical`. `routing_channel=cron-health`.
     - `HeartbeatFailed`: `homelab_heartbeat_current_streak <= -1 AND homelab_heartbeat_seconds_since_last_fail < 600`. Severity `error`. `routing_channel=cron-health`.
     - `HeartbeatFlapping`: `changes(homelab_heartbeat_current_streak[10m]) > 6`. Severity `warning`. `routing_channel=cron-health`.
     - **NEW `WrapperPossiblyStale`**: `wrapper_installed="yes" AND homelab_heartbeat_logscrape_count_since_last_heartbeat >= 3` for 60s. Severity `warning`. Labels: `routing_channel=monitoring-health`, `source_tool=vmalert-metrics`. Annotation: `summary="Wrapper for {{ $labels.name }} on {{ $labels.host }} may have stopped sending heartbeats"`, `description="Log-scrape sees {{ $value }} runs since last heartbeat; check wrapper script."`

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **`logscrape_count_since_last_heartbeat` derivation: maintain a separate counter on `heartbeats_state` (incremented by STAGE-002-008 log collector, reset by STAGE-002-001 heartbeat receiver `/ok` and `/fail`) vs compute from existing `observed_runs_total` minus heartbeat run-count?**
   - **Recommended: separate counter on `heartbeats_state.logscrape_runs_since_heartbeat INTEGER NOT NULL DEFAULT 0`.** Incremented atomically by log collector; reset to 0 on heartbeat receiver state-transition. Simple, fast at metric-emission time.
   - Alternative: compute from existing counters. Requires a heartbeat-run-count column (doesn't exist; we have streak which can wrap signs). More machinery.

2. **`WrapperPossiblyStale` threshold: 3 log-scrapes without heartbeat vs 5 vs N×cadence-multiple?**
   - **Recommended: 3 runs (hardcoded).** Means the wrapper has missed 3 consecutive cron executions while log-scrape still sees them. Catches the failure mode fast without flap-firing on a single network blip.
   - Alternative: 5 runs. Less noisy; misses the early signal.

3. **Routing channel naming: `monitoring-health` vs `meta` vs `internal`?**
   - **Recommended: `monitoring-health`.** Self-explanatory; describes the alert subject. Distinct from `cron-health` (the default for cron-related alerts).
   - Alternative: `meta`. Cryptic.

4. **Recording rule `homelab:wrapper_stale_count` exposed in Grafana Overview tile vs only the heartbeats dashboard?**
   - **Recommended: both. Overview tile shows the count as a small single-stat alongside `homelab:heartbeat_overdue_count`.** Highly-actionable for the user.
   - Alternative: only in the heartbeats dashboard. Less prominent; user has to look for it.

## Locked Design Decisions

D1 — logscrape counter: add a new column `heartbeats_state.logscrape_runs_since_heartbeat INTEGER NOT NULL DEFAULT 0`. It is incremented (+1) by `record_observed_run`, and RESET to 0 by `record_ok`, `record_fail`, and `record_start` (any of the three means the wrapper executed and is alive). The WrapperPossiblyStale rule reads a metric derived from this counter. Rationale: the rule needs a COUNT; observed_runs_total is all-time and cannot supply a since-last-heartbeat count.

D2 — WrapperPossiblyStale threshold: hardcoded N=3 log-scraped runs with no fresh heartbeat. cadence_seconds IS exposed as a metric label (future-proofing a cadence-scaled lateness ladder) but the WrapperPossiblyStale rule does NOT scale on it. Rationale: a network blip cannot reach 3; the wrapper-not-phoning-home failure is cadence-independent.

D3 — monitoring-health channel: the routing_channel label value is `monitoring-health` (parallel to the existing `cron-health`). Mechanism: edit deploy/alertmanager/alertmanager.yml.template — add a child `route` under the top-level route with `matchers: ['routing_channel="monitoring-health"']` → a new receiver `monitoring-health-channel`. The public default points monitoring-health-channel at the SAME ingest webhook as monitor-webhook (dev-safe — the alert still lands in /api/alerts, just tagged); the separate homelab-monitor-overrides repo swaps in a distinct destination. render_on_boot already substitutes ${ALERTMANAGER_INGEST_TOKEN} — no code change needed for rendering.

D4 — @reboot lateness + host boot time: add a `/proc:/host/proc:ro` bind-mount to deploy/compose/docker-compose.yml (the WHOLE /proc read-only — Docker cannot reliably bind-mount an individual synthetic /proc file; whole-/proc-ro is the standard host-monitoring pattern, and /proc/stat btime is world-readable). The HeartbeatStateCollector reads host `btime` from /host/proc/stat once per scrape. The @reboot lateness logic lives IN THE COLLECTOR, not a standalone vmalert rule: for crons with cadence_seconds=0 (@reboot), the collector emits homelab_heartbeat_expected_next_seconds as a NEGATIVE value when last_ok_at < host_boot_time (overdue), and 0/positive otherwise — so the existing HeartbeatStale_* ladder fires on @reboot crons naturally with no dedicated rule. Build-phase guard: a @reboot cron that never logged an last_ok_at and was discovered after boot should not alert until it has had a chance to run (guard with last_ok_at IS NOT NULL OR last_discovered_at < boot_time — exact guard settled in Build). ALSO fix the PRE-EXISTING bug: the host collector (plugins/collectors/builtin/host.py) currently calls psutil.boot_time() which in a container reads the CONTAINER's /proc — homelab_host_uptime_seconds is currently wrong (container uptime, not host). Fix host.py to read host btime from /host/proc/stat (the same source), documented in the commit message as a pre-existing-bug fix.

D5 — wrapper-health badge: a backend-computed `wrapper_health` enum field (values: "ok" | "stale" | "unknown") on the cron-detail API response, computed from the logscrape_runs_since_heartbeat counter (>=3 → "stale", consistent with the WrapperPossiblyStale rule so the badge and the alert never disagree). The badge renders as Row 3 of the ActionsPanel in CronDetail.tsx, ONLY when wrapper_installed is true (hidden entirely otherwise). Use the existing Badge component.

D6 — UI label: rename the CronDetail.tsx "Next due" row label to "Overdue after" (expected_next_at = last_run + cadence + grace is the late-threshold/deadline; "Overdue after" states exactly that and aligns with the alert vocabulary). One-line change + its test.

D7 — metric emission filter: the HeartbeatStateCollector emits per-cron metrics ONLY for crons where `hidden_at IS NULL AND soft_deleted_at IS NULL`. A soft-deleted cron (no longer found on disk, STAGE-007A) must not generate lateness/stale alerts. The proposed separate `homelab_cron_hidden` label/metric (from an old stage-file addendum) is DROPPED — filtering at the source (no series emitted) is simpler than emitting series + a hidden label + PromQL gating.

STALE STAGE-FILE WORDING (Build must IGNORE these — the real facts are): the stage file says `wrapper_installed_at` — the real column is `wrapper_installed` (bool); the wrapper_installed metric label = "yes" when wrapper_installed is true. The stage file says `alertmanager.yml.tmpl` — the real file is `alertmanager.yml.template`. An old "2026-05-12" addendum proposing a `homelab_cron_hidden` label + Alertmanager route inhibition is SUPERSEDED by D7. A test-section note saying logscrape count is "derived from observed_runs_total minus heartbeat counter" is SUPERSEDED by D1 (separate counter). `wrapper_installed` and `wrapper_last_seen_at` are SEPARATE columns — it was never a rename of one to the other.
