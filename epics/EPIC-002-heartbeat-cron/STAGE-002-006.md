# STAGE-002-006: UI redesign — 4-panel detail page, hidden replaces archive, remote-banner, wrapper-installed display

## Status: Complete

## Overview

Redesign the Cron detail page to reflect the derived-state model. After this stage, the cron detail page is split into four clearly-labeled panels: Disk source (read-only), Monitoring policy (editable), Heartbeat state (read-only with diagnostics), and Actions (Hide/Unhide, Install heartbeat — disabled in EPIC-002, enabled in STAGE-002-009 for local host only). The Crons toolbar replaces "include_archived" with "include_hidden"; replaces the `integration_mode` filter with a `wrapper_installed` filter. List view shows a "Remote (no disk source)" badge for rows where `source_path IS NULL`.

This stage closes the visual loop on the redesign — after it ships, the UI matches the derived-state mental model and downstream stages (007 discovery, 009 wrapper helpers) extend it without further structural changes.

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **4-panel grid layout: 2x2 vs 1x4 stacked vs sidebar-style?**
   - **Recommended: 2x2 on desktop (1280+px), 1x4 stacked on mobile (<768px).** Each panel gets its own card with a clear heading. Mobile collapses to vertical stack.
   - Alternative: tabbed (one tab per panel). Hides information; user can't see policy + state at the same time.
   - Alternative: sidebar (panels 1, 3, 4 in a sticky sidebar; panel 2 as the main content). Asymmetric; better for "policy is the action" model but doesn't reflect the equal weight of disk source + state.

2. **"Install heartbeat" button placement: in Actions panel vs as a separate prominent CTA on the detail page?**
   - **Recommended: in the Actions panel.** Consistent with the disabled-tooltip pattern; doesn't compete with Hide/Unhide for attention; once enabled in STAGE-002-009 it's still in the right place.
   - Alternative: prominent CTA in the header. Feels louder; harder to ignore.

3. **Wrapper-installed display granularity: just timestamp vs timestamp + install method?**
   - **Recommended: timestamp + install method ("Wrapper installed via UI on 2026-05-12 14:30 UTC" or "Wrapper installed via CLI on 2026-05-12 14:30 UTC" or "No wrapper installed").** Install method comes from a future column we don't have yet — STAGE-002-009 will add `wrapper_install_method ENUM('ui', 'cli', 'manual') NULL`. For now, just show "Wrapper installed on <ts>" or "No wrapper installed."
   - Alternative: just "Wrapper: yes/no". Loses the timestamp; user can't see when wrapper was installed/refreshed.

4. **Hide button confirmation: same type-name confirm pattern as current "Archive" vs simpler "Are you sure?" modal?**
   - **Recommended: keep the type-name confirmation.** "Hide" is reversible (Unhide button), but type-name confirmation matches existing UX for destructive-feeling actions and prevents accidental hides via misclick.
   - Alternative: drop to simple confirm. Faster; matches the "Hide is reversible, no big deal" semantic.
