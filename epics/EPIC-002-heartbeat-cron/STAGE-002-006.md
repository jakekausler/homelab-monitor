# STAGE-002-006: vmalert rules for stale heartbeats + lateness ladder + flap detector

## Status: Not Started

## Overview

Close out EPIC-002 by (1) adding a heartbeat metric collector that reads from `heartbeats_state` and emits per-cron freshness metrics to the kernel's Prometheus registry, and (2) writing the vmalert (metrics) rules that fire when a cron heartbeat is overdue, when a streak breaks unexpectedly, or when a cron's state flaps. The rules query the metrics emitted by this stage (`homelab_heartbeat_seconds_since_last_ok`, `homelab_heartbeat_expected_next_seconds`, `homelab_heartbeat_current_streak`) and route alerts through Alertmanager → the ingestor → channels (the path proven by STAGE-001-017 + STAGE-001-021's canonical e2e test).

Vertical slice: stop sending heartbeats for a known cron with cadence 60s and grace 60s; wait up to 3 minutes; assert a `HeartbeatStale` alert with severity `warning` appears in `/api/alerts` with `source_tool=vmalert-metrics`. Resume heartbeats; assert resolution within 60s.

**Spec references:** §3.1 (heartbeat receiver — exposes lateness metrics to VM), §3.2 (vmalert metrics — rules under `deploy/vmalert/metrics/`), §10.3 (alert lifecycle — opened/resolved/acked/dismissed).

## Concrete deliverables

1. **Heartbeat metric collector and Prometheus registry integration** (deferred from STAGE-002-001)
   - New file at `apps/monitor/homelab_monitor/kernel/metrics/heartbeat_collector.py`
   - Class `HeartbeatStateCollector` that:
     - At each scrape (invoked by the `/metrics` endpoint), reads all rows from `heartbeats_state` table
     - For each row, computes `seconds_since_last_ok = now_utc() - last_ok_at` (or large value if NULL)
     - Exposes five gauges via the kernel's `prom_registry` (see `apps/monitor/homelab_monitor/kernel/metrics/prometheus_writer.py` for the registry pattern):
       - `homelab_heartbeat_seconds_since_last_ok{cron_id, host, name}` (gauge; large value = stale)
       - `homelab_heartbeat_seconds_since_last_fail{cron_id, host, name}`
       - `homelab_heartbeat_current_streak{cron_id, host, name}` (signed integer)
       - `homelab_heartbeat_expected_next_seconds{cron_id, host, name}` (negative = overdue)
       - `homelab_heartbeat_last_duration_seconds{cron_id, host, name}`
   - Register the collector in the kernel's metric initialization (likely `apps/monitor/homelab_monitor/kernel/metrics/__init__.py` or kernel app initialization)
   - Emits zero metrics for `integration_mode=observe` crons (avoids polluting the metrics namespace before the user opts in)

3. **vmalert rule file** at `deploy/vmalert/metrics/heartbeats.yaml`
   - Rule group `homelab.heartbeats`:
     - **`HeartbeatStale_Warning`**: `homelab_heartbeat_expected_next_seconds < -60` for 60s. Severity `warning`. Annotation: `summary="Cron {{ $labels.name }} on {{ $labels.host }} is 1×grace late"`, `description="Expected next run was {{ $value }}s ago; cadence {{ $labels.cadence_display }}"`.
     - **`HeartbeatStale_Error`**: `homelab_heartbeat_expected_next_seconds < -300` for 60s (after 3×grace OR 5min, whichever is smaller — Decision 2). Severity `error`. Same annotation pattern with sharper wording.
     - **`HeartbeatStale_Critical`**: `homelab_heartbeat_expected_next_seconds < -3600` for 60s (1h late on ANY cron). Severity `critical`. Pages immediately.
     - **`HeartbeatFailed`**: `homelab_heartbeat_current_streak <= -1` AND `homelab_heartbeat_seconds_since_last_fail < 600`. Severity `error`. Fires within 10min of any fail; auto-resolves when streak goes positive.
     - **`HeartbeatFlapping`**: `changes(homelab_heartbeat_current_streak[10m]) > 6` — streak changed sign or jumped 6+ times in 10 minutes. Severity `warning`. Indicates an unstable cron (intermittent failures).
   - All rules include labels `source_tool=vmalert-metrics`, `cron_id={{ $labels.cron_id }}`, `host={{ $labels.host }}`, plus a `runbook_url` annotation pointing at the dashboard `/inventory/crons/{id}` page

4. **Recording rules** (also in `deploy/vmalert/metrics/heartbeats.yaml`)
   - `homelab:heartbeat_overdue_count` = `count(homelab_heartbeat_expected_next_seconds < -60)` — useful for Overview tile
   - `homelab:heartbeat_fail_count` = `count(homelab_heartbeat_current_streak <= -1)` — Overview tile
   - `homelab:heartbeat_runtime_p95_seconds` = `quantile(0.95, homelab_heartbeat_last_duration_seconds)` — digest input

5. **vmalert rules unit tests** at `deploy/vmalert/metrics/__tests__/heartbeats.tests.yaml`
   - Uses vmalert's `unittest` mode (per VictoriaMetrics docs)
   - Cases:
     - Cron with cadence 60s, grace 60s, last_ok=120s ago → `HeartbeatStale_Warning` fires
     - Cron with cadence 60s, grace 60s, last_ok=400s ago → `HeartbeatStale_Warning` AND `HeartbeatStale_Error` fire
     - Cron with cadence 60s, grace 60s, last_ok=4000s ago → all three stale levels fire
     - Cron with streak -1 → `HeartbeatFailed` fires
     - Cron with streak alternating +1 / -1 / +1 / -1 / +1 / -1 / +1 over 10min → `HeartbeatFlapping` fires
     - Cron with streak +1 (stable) → no alerts
     - Cron in `observe` mode (no metrics emitted) → no alerts (because nothing to query)

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **Rule file location: `deploy/vmalert/metrics/heartbeats.yaml` (one file) vs per-cron rule files?**
   - **Recommended: single file with per-cron labels.** Per-cron rule files would be auto-generated (e.g., from the `crons` table on monitor boot) which is doable but adds machinery (rule-file generator + vmalert reload trigger). One file with labels works fine — the per-cron context comes from the metrics' labels (`cron_id`, `host`, `name`), not the rule's identity.
   - Alternative: generated per-cron rule files. Cleaner per-cron lifecycle (delete cron → delete rule file) but requires a rule-render step on cron create/update/delete + vmalert reload.

2. **Lateness severity ladder: warning at 1×grace, error at 3×grace OR 5min (smaller of), critical at 1h?**
   - **Recommended: warning 1×grace (60s past), error 5min OR 3×grace whichever is smaller, critical 1h.** Hardcoded thresholds keep rules simple. Per-cron severity tuning is a power-user feature deferred to later.
   - Alternative: severity scales with each cron's own cadence (warning at 1×cadence, error at 5×cadence, critical at 50×cadence). Cleaner for crons with vastly different cadences (1min cron vs daily cron — a 5min delay means very different things), but requires the metric to expose `cadence_seconds` as a label. We'd need to add that to STAGE-002-001's metrics emitter.
   - Decision matters for: STAGE-002-001's metrics emitter — should it expose `cadence_seconds` as a label?

3. **Flap detector window + threshold: changes > 6 in 10 minutes?**
   - **Recommended: `changes(streak[10m]) > 6`.** 6 sign-changes in 10 minutes = on/off every ~100 seconds. A cron with `* * * * *` (every minute) that's failing intermittently would trip this. Less noisy than 4-in-5min; less lax than 10-in-30min.
   - Alternative: `rate(changes(streak[5m])) > 0.02` (rate-based; > 2% per second). Smoother but harder to reason about.

4. **Resolved-too-quickly / "self-healing" detection: alert if heartbeat was OK then immediately FAIL within 1min — flap warning?**
   - **Recommended: covered by `HeartbeatFlapping`.** Don't add a separate rule. The flap detector catches rapid state changes; adding a "self-healing" alert risks noise. If users want it later, easy to add.
   - Alternative: explicit `HeartbeatSelfHealed` info-level alert. Useful as an audit signal but doesn't drive action; the digest summary section already surfaces this in narrative form.
