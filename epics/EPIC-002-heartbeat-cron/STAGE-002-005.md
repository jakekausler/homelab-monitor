# STAGE-002-005: C-mode wrapper helpers + dashboard "Install heartbeat" action

## Status: Not Started

## Overview

Make A-mode (active heartbeat) easy to adopt without modifying the user's crontab silently. This stage ships: (1) a POSIX-bash wrapper script `cron-with-heartbeat.sh` that any cron can be replaced with — it calls `/hb/{id}/start`, runs the real command, captures exit code + runtime, and calls `/hb/{id}/ok` or `/hb/{id}/fail`; (2) a CLI subcommand `hm cron install-wrapper <cron_id>` that prints the exact crontab line the user should swap in; (3) a "Install heartbeat" dashboard action on the cron detail page that produces the same copy-paste output.

Per cross-stage criterion #6, this stage does NOT silently edit any user crontab. Output is always copy-paste. A future stage (or the host-overrides repo) can add an SSH-push variant.

Vertical slice: from the dashboard, click on a cron row → detail page → "Install heartbeat" button → modal opens with copy-paste-able shell snippet (the wrapper invocation), a "Download wrapper script" link, and a checklist: (1) save the wrapper script to `/usr/local/bin/cron-with-heartbeat.sh`, (2) `chmod +x`, (3) update the crontab line. The cron is auto-promoted from `observe` to `heartbeat` mode on the row when the user clicks "Mark as installed."

**Spec references:** §2 Q5 (C-mode bootstrap helper), §3.1 (heartbeat receiver — clients), §11 (`/storage/scripts/cron/backup.sh` integration; this is the model for what the public release ships and the overrides repo applies).

## Concrete deliverables

1. **Wrapper script** at `apps/monitor/homelab_monitor/data/cron-with-heartbeat.sh`
   - Shell: POSIX-bash (no bashisms; runs under `dash` for Debian `/bin/sh` compatibility on Synology DSM)
   - Invocation: `cron-with-heartbeat.sh <cron_id> <token> <heartbeat_url_base> -- <real-command> [args...]`
     - Example: `cron-with-heartbeat.sh 7 hb_tok_abc123 http://homelab-monitor:9090 -- /storage/scripts/cron/backup.sh`
   - Behavior:
     - Records start time
     - `curl --max-time 5 -X POST -H "Authorization: Bearer $token" "$heartbeat_url_base/hb/$cron_id/start"` (best-effort; failures do NOT block the real command)
     - Exec `$real_command "$@"`; capture exit code + duration
     - On success: `curl --max-time 5 -X POST -H "Authorization: Bearer $token" "$heartbeat_url_base/hb/$cron_id/ok?duration=$duration"`
     - On fail: `curl --max-time 5 -X POST -H "Authorization: Bearer $token" "$heartbeat_url_base/hb/$cron_id/fail?duration=$duration"` (passes exit code in body for future use)
     - Always `exit $real_exit_code` so cron's own logging is preserved
     - Logs heartbeat-side errors to stderr (so syslog captures them) but never propagates them to the real command's exit code
   - Token can also be sourced from an env file: `cron-with-heartbeat.sh --token-file /etc/homelab-monitor/heartbeat.token <cron_id> <heartbeat_url_base> -- <command>` (recommended pattern; avoids token-in-crontab)
   - Header: shebang + license + auto-generated banner ("DO NOT EDIT — version controlled at apps/monitor/homelab_monitor/data/cron-with-heartbeat.sh")

2. **CLI subcommand** at `apps/monitor/homelab_monitor/cli/cron.py`
   - `hm cron install-wrapper <cron_id>` — prints:
     - The crontab line the user should swap in (with the wrapper invocation)
     - A copy-paste curl command to test the heartbeat path
     - A "next steps" checklist
   - `hm cron get-wrapper-script` — prints the full wrapper script to stdout (so the user can pipe it: `hm cron get-wrapper-script | sudo tee /usr/local/bin/cron-with-heartbeat.sh && sudo chmod +x ...`)
   - Both subcommands work in containerless mode via `python -m homelab_monitor.cli`

3. **API endpoint** at `apps/monitor/homelab_monitor/kernel/api/routers/crons.py` (extending STAGE-002-002's file)
   - `POST /api/crons/{cron_id}/install-snippet` → body validates `InstallSnippetRequest { token_id: int }`; returns `InstallSnippetResponse { crontab_line: str, wrapper_path_suggestion: str, test_curl: str, dashboard_callback_url: str }`
     - `crontab_line`: the literal line to put in crontab, with the wrapper invocation + real command preserved
     - `wrapper_path_suggestion`: `/usr/local/bin/cron-with-heartbeat.sh` by default
     - `test_curl`: a curl command the user can run from the target host to verify the receiver is reachable + token works
     - `dashboard_callback_url`: a callback the wrapper-install confirmation page can POST to after the user confirms they've installed it (auto-promotes cron mode `observe` → `heartbeat`)
   - `POST /api/crons/{cron_id}/mark-installed` → flips `integration_mode` from `observe` to `heartbeat` (or to `both` if user prefers); audits the change; returns updated `CronOut`. CSRF required.

4. **UI: Install heartbeat modal** at `apps/ui/src/pages/inventory/InstallHeartbeatModal.tsx`
   - Triggered by "Install heartbeat" button on the cron detail page (STAGE-002-002)
   - Step 1: Token selection — dropdown of existing `heartbeat:write` tokens; "Create new token" link → side flow into existing settings
   - Step 2: Mode choice — radio buttons: "heartbeat only (replace observation)" vs "both (heartbeat + keep log-scrape)"
   - Step 3: Display payload from `POST /api/crons/{id}/install-snippet`:
     - Crontab line (copy-button)
     - Wrapper script (download link → serves `cron-with-heartbeat.sh`; copy-button serves inline copy)
     - Test curl (copy-button)
   - Step 4: Confirmation checkbox: "I have installed the wrapper and updated my crontab" → enables "Mark as installed" button
   - Step 5: On click → POST `/api/crons/{id}/mark-installed` → modal closes → cron row updates mode badge

5. **Static asset endpoint** at `apps/monitor/homelab_monitor/kernel/api/routers/assets.py` (new file if not existing)
   - `GET /api/assets/cron-with-heartbeat.sh` → serves the wrapper script with `Content-Type: text/x-shellscript; charset=utf-8` + `Content-Disposition: attachment; filename=cron-with-heartbeat.sh`
   - No auth (the script content is not sensitive; it's also git-versioned in the public repo)
   - Cache headers: `Cache-Control: public, max-age=300, must-revalidate`

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **Wrapper script language: POSIX bash vs bash with heredoc + declared deps vs Python?**
   - **Recommended: POSIX bash (no bashisms).** Runs everywhere — Debian/Ubuntu, RHEL, Synology DSM, OpenWRT, Alpine. Single file, no install. Python requires interpreter present; Bash-with-bashisms breaks on `dash` (Synology DSM). The cost: weaker error handling and less expressive code — manageable since the script is small (~80 lines).
   - Alternative: Python with `#!/usr/bin/env python3`. Cleaner code, more testable, but adds an interpreter dependency.

2. **Dashboard "Install" action UX: copy-paste snippet vs dashboard SSH-pushes the wrapper to the host?**
   - **Recommended: copy-paste snippet only in this stage.** SSH push requires the SSH probe infrastructure (per-target dedicated low-priv users with key-restricted forced commands per spec Q29) which doesn't exist yet. Copy-paste is the safe public-release default. Per cross-stage criterion #6 — no modifications without confirmation, and copy-paste is explicit confirmation.
   - Alternative: ship both, with SSH push behind an opt-in feature flag. Doubles scope; defers cleanly to a follow-up.

3. **Token provisioning for the wrapper: re-use a global heartbeat token vs mint per-cron token?**
   - **Recommended: re-use a global `heartbeat:write` token (one token per host, optional).** Matches STAGE-002-001's Decision 3. Per-cron tokens are a hardening stage; not required for v1. Document the security trade-off: a global token, if leaked from one cron, could spoof heartbeats for every cron on that host — but cron commands typically run as root or a privileged user that can already write logs, so the marginal risk is small.
   - Alternative: mint a token per cron on install. Stronger isolation; more management overhead; more tokens to rotate.

4. **Wrapper-side timeout: how long to wait on heartbeat HTTP calls before giving up?**
   - **Recommended: `--max-time 5` (5 seconds).** The receiver should respond in <100ms in normal operation. A 5-second cap means at most 10 extra seconds of latency per cron run (start + ok = 2× max-time) in worst-case network failure. Long enough to absorb LAN hiccups; short enough not to delay critical crons noticeably.
   - Alternative: tighter (2s) — fails faster on real outages. Looser (15s) — more forgiving but interferes with quick crons.
