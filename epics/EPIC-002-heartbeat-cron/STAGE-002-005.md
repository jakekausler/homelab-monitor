# STAGE-002-005: `/api/hb/{fingerprint}/register` endpoint + heartbeat receiver behavior change

## Status: Not Started

## Overview

Add the `/register` endpoint that creates `crons` rows from heartbeat-side metadata. After this stage, a wrapper script (or any heartbeat client) can POST `{host, source_path, schedule, command, wrapper: bool}` to `/api/hb/{fingerprint}/register` and the server creates the matching cron row idempotently. The heartbeat receiver continues to 404 on unknown fingerprints for `/start`, `/ok`, `/fail` — `/register` is the only creation path on the heartbeat side.

This stage formalizes the "heartbeat-first" creation path that was previously rejected in STAGE-002-001 D1 (404 on unknown). The reversal is justified by the new derived-state model where the heartbeat wrapper is a legitimate authority on what it's wrapping.

## Concrete deliverables

1. **New API endpoint** at `apps/monitor/homelab_monitor/kernel/api/routers/heartbeat.py`
   - `POST /api/hb/{fingerprint}/register`
   - Auth: `Scope.HEARTBEAT_WRITE` (token-only).
   - Path param: `fingerprint` — 64-char SHA256 hex (regex-validated).
   - Body (Pydantic `RegisterCronBody`):
     ```python
     class RegisterCronBody(BaseModel):
         host: str = Field(min_length=1, max_length=255)
         source_path: str | None = Field(default=None, max_length=500)
         schedule: str = Field(min_length=1, max_length=200)
         command: str = Field(min_length=1, max_length=2000)
         wrapper: bool = False
         model_config = ConfigDict(extra='forbid')
     ```
   - Behavior:
     1. Compute `expected_fingerprint = SHA256(f"{body.host}|{body.source_path or ''}|{body.schedule}|{body.command}").hexdigest()`.
     2. If `expected_fingerprint != path_fingerprint` → 422 with detail `{"fingerprint_mismatch": True}`.
     3. Validate `body.schedule` via croniter; reject 422 on invalid.
     4. Compute `cadence_seconds` and `schedule_canonical` from schedule.
     5. Default `name` to command basename.
     6. Try INSERT. If row exists with same fingerprint:
        - Compare ALL stored metadata fields to body. If identical → 200 idempotent (no audit row, return existing CronOut).
        - If any field differs → 409 with detail `{"fingerprint_collision": True, "expected": <stored fields>, "received": <body>}`. (Should never happen by construction; defensive check.)
     7. If new row created: set `wrapper_installed_at = utc_now_iso()` if `body.wrapper`. Audit verb `crons.register` with full row in `after`.
   - Returns 201 (created) or 200 (idempotent existing).

2. **Heartbeat receiver behavior unchanged**
   - `POST /api/hb/{fingerprint}/{start,ok,fail}` still returns 404 for unknown fingerprints. No change to the existing logic from STAGE-002-001 (now reading via fingerprint per STAGE-002-003).

3. **CronRepo extension** at `apps/monitor/homelab_monitor/kernel/cron/repository.py`
   - New method `async def register_cron(self, body: RegisterCronBody, *, who: str, ip: str | None) -> tuple[CronRecord, bool]` returning `(record, created)`. The `created` boolean indicates 201 (true) vs 200 (false-idempotent).
   - The audit verb is `crons.register`. The `who` field in audit is the token name (per the existing pattern from STAGE-002-001).
   - Same atomic-transaction discipline as `update_cron`: INSERT + audit_log INSERT in one transaction.

4. **Pydantic schemas** at `apps/monitor/homelab_monitor/kernel/cron/schemas.py`
   - Add `RegisterCronBody` (above).
   - The response from `/register` reuses `CronOut` (no new response schema).

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **Idempotent semantics: 200 OK + same body vs 200 OK + `{idempotent: true}` flag vs 304 Not Modified?**
   - **Recommended: 200 OK + same CronOut body.** Cleanest REST semantics; client doesn't need special-case handling.
   - Alternative: 304 Not Modified. Cache-friendly but weird for a POST; clients expect 304 only for GET.
   - Alternative: 200 OK + `{idempotent: true}` envelope. Useful if clients want to log "actually created vs already-existed" separately, but most clients don't care.

2. **`wrapper: true` semantics if `wrapper_installed_at` was already set: refresh the timestamp or no-op?**
   - **Recommended: refresh the timestamp on every successful /register with `wrapper: true`.** A re-install (e.g., wrapper script updated) should bump the install time.
   - Alternative: only set on first registration; ignore subsequent. Preserves "when was this first wrapper installed" history but loses "when was it last refreshed."

3. **Token attribution in audit `who` field: token name vs token id vs both?**
   - **Recommended: token name (current pattern from STAGE-002-001).** Human-readable in audit log; matches existing audit rows.
   - Alternative: token id (UUID). More precise but unreadable; requires JOIN to make sense.

4. **Rate limiting on `/register`: same per-cron bucket as `/start//ok//fail` (60/min) vs separate / stricter?**
   - **Recommended: same bucket per fingerprint.** /register is rare in practice (once per cron install + occasional refresh); the existing limit is more than sufficient and avoids two different rate-limit configs.
   - Alternative: stricter (e.g., 5/min per fingerprint). Defensive against /register being abused for table-spam, but the body validation + 422 already filters most spam.
