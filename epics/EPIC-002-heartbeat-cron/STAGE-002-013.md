# STAGE-002-013: VictoriaLogsClient + CronRunReconciler + B-mode log-scrape run rows

## Status: Complete

**(New stage. EPIC-002 re-opened on 2026-05-19 — see `docs/superpowers/specs/2026-05-19-cron-run-logs-design.md`. Third of five appended stages STAGE-002-011 … STAGE-002-015.)**

## Overview

Deliver the asynchronous half of cron run history: a reusable `VictoriaLogsClient`, a scheduler-registered `CronRunReconciler` that closes B-mode run windows and enriches every closed run with VictoriaLogs-derived fields, and the B-mode (log-scrape) path that creates `cron_runs` rows for unwrapped crons.

STAGE-002-011 + STAGE-002-012 give A-mode crons synchronously-written, UUID-tagged run rows. This stage adds: (1) B-mode crons get run rows too, created as an extension of the existing cron-events log-scrape ingest; (2) every closed run (A- and B-mode) gets `line_count`, `byte_count`, and `content_digest` computed from VictoriaLogs by a background reconciler; (3) retention pruning of `cron_runs`.

**Vertical slice:** an unwrapped cron firing produces a `cron_runs` row (`source=logscrape`) from the cron-daemon `CMD` syslog line; the reconciler closes it on the next `CMD` line (or a timeout cap) and, ~15s later, enriches it with line/byte counts and a content digest read from VictoriaLogs; a wrapper-installed cron's run (from STAGE-002-012's tagged lines) is enriched the same way via the UUID-exact query.

**Spec references:** `docs/superpowers/specs/2026-05-19-cron-run-logs-design.md` §6 (VictoriaLogsClient + CronRunReconciler), §6.3 (B-mode event correlation), §3.2 (B-mode data flow), §4 (retention), D2, D8, D9, D10.

## Concrete deliverables

### 1. `VictoriaLogsClient` — reusable bounded VL query module

New module `apps/monitor/homelab_monitor/kernel/logs/victorialogs_client.py` (the genuinely reusable piece; EPIC-004's generic `/api/logs` proxy will later be built on top of it):

- Connects to VictoriaLogs (`http://victorialogs:9428` in prod; configurable; dev publishes `19428` per the project port map).
- Executes a LogsQL query over a **bounded time window**.
- Parses the JSON-lines response into structured log lines (timestamp, message, journald fields).
- **Hard limits (mandatory):**
  - Every query bounded by an explicit time range.
  - A max-lines cap — env `HM_VL_QUERY_MAX_LINES`, default `10000`.
  - A max-bytes cap — env `HM_VL_QUERY_MAX_BYTES`, default `5_000_000` (5 MB).
  - An HTTP timeout.
  - A query exceeding the cap returns the capped lines + a `truncated: true` flag — never an unbounded fetch.
- **Two query shapes:**
  - A-mode: `SYSLOG_IDENTIFIER:hmrun AND run_id:<run_id>` over `[vl_window_start, vl_window_end]` — UUID-exact (relies on STAGE-002-012's Vector transform). This query depends on STAGE-002-012's `D-VECTORFIELD` decision — `run_id` is a regular VictoriaLogs field (not a stream field); the query filters on it as a regular field, and the per-run time-window bound keeps it performant.
  - B-mode: the cron-fingerprint heuristic (reuse STAGE-002-008's `match_by_log_key` fingerprint-match logic) over the run window.
- **Failure handling:** VL unreachable / timeout / non-200 → raise a typed error. Callers degrade gracefully and never crash.

### 2. `CronRunReconciler` — scheduler-registered background task

New module (e.g. `apps/monitor/homelab_monitor/kernel/metrics/cron_run_reconciler.py`, consistent with the existing `kernel/metrics/heartbeat_collector.py` location). Registered in `lifespan.py` exactly like `HeartbeatStateCollector` (STAGE-002-010). Runs every ~30s (env-configurable). Each tick, three phases:

1. **Window-finalize** — for B-mode runs still `state=running`: if a newer `CMD` line exists for the same cron, close the prior run (`ended_at` = the next run's `started_at`; `state` from the `exit=` tag if one was correlated, else `unknown`); if a run has been open past a fixed max-duration cap, close it by timeout (`state=unknown`). When closing a B-mode run (by next-`CMD` or by timeout), window-finalize sets `vl_window_end` (to the same value as `ended_at`) in addition to `ended_at` — so STAGE-002-014's run-log endpoint has a populated upper bound for B-mode runs. Set `overlapping=1` on any B-mode runs whose windows intersect another run of the same cron. **A-mode runs are closed synchronously by the `/ok|/fail` heartbeat call (STAGE-002-011) — the reconciler does NOT change A-mode run state.**
2. **Enrich** — for each run where `state != 'running' AND enriched_at IS NULL AND ended_at is older than a short grace delay (~15s, env-configurable, to let VL ingest trailing lines)`: query `VictoriaLogsClient` for the run's window, compute `line_count`, `byte_count`, and `content_digest` (a hash of NORMALIZED output — strip timestamps/PIDs so the digest reflects content shape; the exact normalization regex is a Build-phase implementation detail), set `enriched_at`. (The anomaly evaluator is wired in here by STAGE-002-014 — this stage leaves `anomaly_flags` empty.)
3. **Prune** — delete `cron_runs` rows beyond the retention bound: runs within the last `HM_CRON_RUN_RETENTION_DAYS` (default 30) AND at most `HM_CRON_RUN_MAX_ROWS_PER_CRON` (default 50000) rows per cron — whichever bound is hit first prunes the oldest rows.

**Properties:**
- Idempotent and stateless — a missed or re-run tick re-derives.
- Two distinct idempotency mechanisms, not to be conflated: B-mode `cron_runs` **row-creation** idempotency is provided by the existing cron-events cursor claim (`try_claim_cursor` — a replayed event is rejected before row creation); the **reconciler's** idempotency is re-derivation of windows + enrichment.
- If VL is down: the enrich phase is skipped this tick (rows stay `enriched_at IS NULL`, retried next tick); window-finalize and prune still proceed (they do not need VL).
- Emits scheduler-task run self-metrics (the same self-observation pattern as `HeartbeatStateCollector`).

### 3. B-mode `cron_runs` row create/close — extension of the existing cron-events ingest

This is an **extension of the existing `cron_events._process_one` ingest handler** (`apps/monitor/homelab_monitor/kernel/api/routers/cron_events.py`) — NOT a new endpoint and NOT a parallel ingest path:

- On the `OBSERVED_RUN` disposition (a cron-daemon `CMD` line): create a `cron_runs` row IN ADDITION TO the existing `record_observed_run` behavior — `run_id` generated server-side (UUID), `source=logscrape`, `state=running`, `started_at` from the event timestamp, `vl_window_start` from the event timestamp.
- On the `STATE_OK` / `STATE_FAIL` disposition (a later `... exit=N` line): correlate to "the most recent open (`state=running`) `cron_runs` row for that fingerprint whose window contains the exit line's timestamp" and close it (`state`, `ended_at`, `exit_code`, `duration_seconds`). Whenever a B-mode run is closed — whether here by the `exit=` line or by the reconciler's window-finalize (next-`CMD` or timeout) — `vl_window_end` is set to the same value as `ended_at`, so STAGE-002-014's run-log endpoint has a populated upper bound for B-mode runs.
- Row-creation idempotency = the existing `try_claim_cursor` cursor claim (a replayed event is rejected before row creation). The `cron_runs` write reuses the existing `match_by_log_key` fingerprint resolution.
- If the `exit=` line is dropped by the at-most-once cursor gap, or the cron simply never emits one, the run is later closed by the reconciler's window-finalize with `state=unknown` (links to D8).

## Locked Design Decisions

D-VLCLIENT — a reusable internal `VictoriaLogsClient` is built now, with MANDATORY bounded-query limits (explicit time range, max-lines cap default 10000, max-bytes cap, HTTP timeout, `truncated` flag). The generic `/api/logs` proxy stays in EPIC-004, built later on top of this client. (Spec D10.)

D-RECONCILER — run records are written SYNCHRONOUSLY by the heartbeat receiver (A-mode, STAGE-002-011) and the log-scrape ingest (B-mode, this stage); a background `CronRunReconciler` ASYNCHRONOUSLY enriches each closed run with VL-derived fields. Event-sourced — the reconciler is a projection that re-derives. (Spec D9, Approach A.)

D-BMODE-EXT — B-mode `cron_runs` row creation/closing is an EXTENSION of the existing `cron_events._process_one` ingest handler, on its `OBSERVED_RUN` / `STATE_OK` / `STATE_FAIL` dispositions, reusing `match_by_log_key`. It happens IN ADDITION TO `record_observed_run`. NOT a new endpoint, NOT a parallel ingest. (Spec §6.3.)

D-BMODE-CORR — the B-mode `exit=` line is correlated to the most-recent open run of that fingerprint whose window contains the exit line's timestamp. A dropped/absent `exit=` line → the reconciler closes the run `unknown`. Row-creation idempotency = the existing cursor claim, NOT the reconciler. (Spec §6.3, D8.)

D-BMODE-WINDOW — a B-mode run window ends at the next `CMD` line for the same cron, OR a fixed max-duration timeout cap. The next-`CMD` rule STRUCTURALLY ASSUMES non-overlap, so the `overlapping` flag primarily catches the timeout-closed case — true overlap of unwrapped crons is not reliably detectable. A-mode overlap is exact (per-line UUID). (Spec D8, §6.2, §14.)

D-RETENTION — `cron_runs` is pruned (in the reconciler tick) to runs within the last 30 days (env `HM_CRON_RUN_RETENTION_DAYS`) AND at most 50000 rows per cron (env `HM_CRON_RUN_MAX_ROWS_PER_CRON`); whichever bound hits first prunes oldest. The hard cap is a non-negotiable safety valve against a runaway high-frequency cron. (Spec D4.)

D-DIGEST — `content_digest` is computed from NORMALIZED output (timestamps/PIDs stripped). It is computed in v1 but consumed by NO v1 anomaly rule — it is a forward investment so EPIC-004's content-clustering work has historical digests to compare against. The exact normalization regex is a Build-phase detail. (Spec D2 note, §7.)
