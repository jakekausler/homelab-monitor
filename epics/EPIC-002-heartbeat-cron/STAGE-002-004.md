# STAGE-002-004: B-mode log-scrape (vector + journald + collector parsing)

## Status: Not Started

## Overview

Ship the public-safe default for cron monitoring: log-scrape via vector + journald + a `cron-log-collector` plugin that watches VictoriaLogs for cron output and updates `heartbeats_state` rows for any cron in `observe`, `heartbeat`, or `both` mode. Works on unmodified crons (per spec Q5 B-mode). After this stage, every cron in the registry — even one in `observe` mode — gets a `last_seen_state` of `ok` or `fail` from log evidence, without the user touching their crontab.

The collector watches three signal sources in VictoriaLogs: (1) `cron.service` journald entries with the `(CRON)` tag and exit code, (2) script-emitted stdout/stderr captured via systemd's `StandardOutput=journal`, (3) `/var/log/syslog` entries with `cron` prefix (for hosts that don't use systemd journal). Idempotency via a per-cron high-water mark plus a per-line content hash so re-processing the same logs never double-counts.

Vertical slice: with the rig running, plant a syslog line `Apr 30 04:10:01 host CRON[12345]: (root) CMD (/storage/scripts/cron/backup.sh) exit=0` → wait up to 60s → assert `heartbeats_state.last_ok_at` updated for the matching cron row.

**Spec references:** §2 Q5 (B-mode is log-scrape), §3.1 (collector framework), §6.1 (`heartbeats_state` table), §3.2 (vector + VictoriaLogs).

2. **Vector configuration update** at `deploy/vector/cron-sources.toml` (or extending the existing vector config from STAGE-001-016)
   - **Source** `cron_journald`:
     - Type: `journald`
     - Filter on `_SYSTEMD_UNIT in ["cron.service", "crond.service"]` (Debian + RHEL naming)
     - Includes hosts via the existing host-discovery pattern
   - **Source** `cron_syslog`:
     - Type: `file`
     - Path: `/var/log/syslog` (host bind-mount: `/var/log/syslog:/host/var/log/syslog:ro`)
     - Filter: lines containing `CRON[` (PID pattern)
   - **Transform** `cron_parsed`:
     - Type: `remap`
     - VRL script: extracts schedule-runner-style fields (`pid`, `user`, `command`, `exit_code`) from the message; outputs structured JSON
     - Tags log lines with `service=cron` for VL query convenience
   - **Sink** `cron_to_vl`:
     - Writes to VictoriaLogs (existing `vl_sink` re-used or a dedicated sink with `_stream=cron`)
   - Test fixture: a `noisy-logger`-style fixture (from STAGE-001-021) gains a `POST /cron-event` endpoint that emits a planted cron-style log line for the integration tests

3. **Cron log collector plugin** at `apps/monitor/homelab_monitor/plugins/collectors/cron_log_collector.py`
   - In-process Python plugin registered as `cron-log-collector`; collector kind ASYNC; interval 30s; concurrency_group `logs`
   - On each run:
     - Query VL for `{service="cron"} AND _time > {oldest_watermark - 5m}` (5min slop for clock skew + late-arriving logs)
     - For each line, parse: `host`, `command`, `user`, `exit_code`, `timestamp`
     - Match line to a `crons` row by (host, command) — uses the index `crons(host, command)` from spec §6.1
     - Compute line hash `sha256(f"{host}|{timestamp_iso}|{command}|{exit_code}".encode()).hexdigest()[:16]`
     - Skip line if `(timestamp, line_hash)` is <= the cron's existing watermark
     - Otherwise: call `HeartbeatRepo.record_ok` (exit_code == 0) or `record_fail` (non-zero); advance watermark; increment `observed_runs_total`; audit entry with `source=log-scrape`
   - Emits `homelab_collector_run_*` per scan
   - Emits `homelab_cron_logscrape_matches_total{host}` counter (new VM metric for self-observability of the collector)
   - Skips crons in `integration_mode=heartbeat` only (no logscrape ambiguity); processes `observe` and `both` modes

4. **Log line parser** at `apps/monitor/homelab_monitor/kernel/cron/log_parser.py`
   - `parse_cron_log_line(line: str) -> CronLogEvent | None`
   - Handles formats:
     - Debian/Ubuntu syslog: `Apr 30 04:10:01 host CRON[12345]: (user) CMD (command...)` (no exit code in this line — pair with the postfix `(user) MAIL ...` line for exit info, or rely on systemd-cron / cronic patterns)
     - systemd journald: structured JSON with `MESSAGE`, `_SYSTEMD_UNIT`, `_PID`, `_COMM=cron`
     - cron exit-code logging via `cronic` wrapper (if present): captures exit code explicitly
   - Returns `None` if line doesn't match any known cron format (log + skip)
   - Comment: "We can't always get an exit code from vanilla cron — see Decision 1."

5. **B-mode + heartbeat coexistence logic** in `apps/monitor/homelab_monitor/kernel/heartbeat/repo.py`
   - For crons in `both` mode: heartbeat (POST /hb/) and log-scrape (this stage's collector) BOTH update `heartbeats_state`. Use timestamps as the tie-breaker (most recent wins for `current_state`).
   - For crons in `observe` mode: only log-scrape updates state.
   - For crons in `heartbeat` mode: only the receiver updates state; the log collector skips them (avoids spurious "stale" if the log scrape misses one).
   - Conflict resolution between near-simultaneous heartbeat-receive and log-scrape: `record_ok` updates `last_ok_at = max(existing, new_ts)`; `current_streak` only increments if the new event is genuinely a new run (heuristic: > 60s gap from last event).

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **Exit-code source: vanilla cron syslog vs require `cronic` / wrapper?**
   - **Recommended: best-effort from vanilla cron + document the limitation.** Debian/Ubuntu syslog cron lines don't include exit codes for successful runs; only failures + mail-sent emit a second line (`(user) MAIL ...`). systemd-cron does include exit codes via the journal. We parse what we can: if no exit code is parseable, default to `ok` (since cron didn't log a failure). Users who want hard exit-code tracking should either install `cronic` or opt-in to A-mode (heartbeat).
   - Alternative: require `cronic` wrapper; refuse to scrape without it. Stronger guarantees but breaks the "works on unmodified crons" promise.
   - Decision matters for: STAGE-002-005's wrapper script (does it just heartbeat, or also wrap-and-log?), and STAGE-002-006's vmalert rules (can they trust "fail" verdicts?).

2. **Vector source for cron logs: journald (systemd) vs `/var/log/syslog` tail vs both?**
   - **Recommended: both, with journald preferred.** journald is more structured (we get `_SYSTEMD_UNIT`, exit_code via `EXIT_STATUS=`, host as `_HOSTNAME`). syslog is the fallback for non-systemd hosts (rare, but the user's Synology box is non-systemd). Vector config has both sources; the parser tries journald format first, falls back to syslog.
   - Alternative: pick one (journald). Cleaner config, but breaks discovery on Synology / OpenWRT in future stages.

3. **Idempotency mechanism: high-water mark per cron vs global high-water mark vs per-line hash table?**
   - **Recommended: per-cron `(timestamp, content_hash)` watermark (single column on `heartbeats_state`).** Scoped per cron, so adding a new cron starts fresh; cheap to update; no separate table.
   - Alternative: a separate `cron_log_events` table with UNIQUE INDEX on the hash. Stronger guarantees (can detect out-of-order logs) but doubles writes and bloats DB.

4. **Partial-success detection: cron exits 0 but stderr has WARN — flag or ignore?**
   - **Recommended: ignore for v1.** Exit code is the source of truth. stderr text scanning is heuristic-heavy and surface-area-large. A future stage can add an "anomaly score" based on log content if useful.
   - Alternative: parse stderr lines, tag `partial_success` state. Useful but speculative; defer.
