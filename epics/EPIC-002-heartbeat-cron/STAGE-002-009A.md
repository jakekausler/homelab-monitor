# STAGE-002-009A: Wrapper removal helpers — uninstall heartbeat wrapper, restore the original crontab line

## Status: Complete

**(Inserted 2026-05-17 between STAGE-002-009 and STAGE-002-010. Wrapper removal is the inverse of STAGE-002-009's install-only flow.)**

## Overview

Ship the heartbeat wrapper REMOVAL infrastructure. STAGE-002-009 ships wrapper INSTALL (install-only); this stage delivers the inverse operation: reverse a wrapped crontab line back to its original form and clean up the installed artifacts.

A wrapped crontab line has the shape `<schedule> <wrapper-path> -- <original-command>`. Removal reverses it back to `<schedule> <original-command>`, with the original command restored **byte-exact** via the existing `unwrap_command()` (the same function the discoverer already uses to compute the original fingerprint from a wrapped line). Three removal paths parallel STAGE-009's three install paths:

1. **UI button (local host only).** STAGE-009 ships a static "Install heartbeat wrapper" row in CronDetail's Actions panel. This stage makes that row **state-aware**: when a wrapper is installed (`wrapper_last_seen_at` is set) AND the cron is local, the row becomes "Remove heartbeat wrapper". The Install↔Remove toggle is delivered HERE — STAGE-009 ships the Install button only and does NOT change based on wrapper state.

2. **CLI `hm cron uninstall-wrapper <fingerprint>`** invoked on the monitor host. Equivalent to the UI button; for users who prefer CLI. Available for crons on the monitor's host only.

3. **Standalone remote-host CLI gains an uninstall mode.** The `install_wrapper_remote.py` standalone script (shipped in STAGE-009) gains an uninstall mode that runs on a remote host, scans local crontabs for wrapped lines, lets the user pick one, and reverses it. Binary distribution itself remains part of EPIC-019; this stage ships the source.

**Fingerprint convergence.** After uninstall the crontab line is back to the original `<schedule> <command>`. The discoverer already computes the original fingerprint from that form (and also unwraps wrapped lines via `unwrap_command()`), so the fingerprint is stable across the full install → uninstall round-trip. A test must verify this round-trip: install → uninstall → fingerprint unchanged.

**Spec references:** `docs/superpowers/specs/2026-05-11-cron-derived-state-redesign.md` §Layered install paths, §Wrapper script knows its fingerprint via install-time bake (uninstall is the inverse).

## Scope / Design questions

These are open questions the Design phase + brainstormer must resolve. They are **not** locked decisions — pre-positions are noted where one exists, but Design decides.

1. **Wrapper script file cleanup.** On removal, should `/usr/local/bin/cron-with-heartbeat.sh` be deleted? The wrapper script may be **shared** by multiple wrapped crons — one wrapper script file, many crontab lines reference it. Removing one cron's wrapper must NOT delete a script still referenced by other wrapped crons. Design must decide between: (a) reference-count remaining wrapped crons and delete the script only when the last one is removed; (b) leave the script in place permanently (harmless if unreferenced); (c) delete only when no wrapped crons remain across all known crontabs.

2. **Token file.** `/etc/homelab-monitor/heartbeat.token` is a **single shared token** across all wrapped crons. Removing one wrapper must NOT delete the shared token (it would break every other wrapped cron). Likely position: never delete the token on a per-cron uninstall. Design confirms.

3. **`wrapper_last_seen_at` on successful uninstall.** Should the column be cleared (set NULL) after a successful uninstall? It is "last seen" telemetry — the timestamp the wrapper was last observed by discovery/heartbeat. Design decides whether uninstall nulls it (signalling "no wrapper now") or leaves it as historical record.

4. **Rollback.** Like install, uninstall must snapshot the crontab before any change and fully roll back on failure (restore the wrapped crontab line; do not leave a half-reverted file).

5. **Local-host gate.** Same `cron.host == resolve_hostname()` gate as install (resolved with the discoverer's exact fallback logic so the two always agree). Remote uninstall via SSH push is deferred to EPIC-017.

6. **Endpoint shape.** Likely `POST /api/crons/{fingerprint}/uninstall-wrapper` (or `DELETE`), session-auth + CSRF, with the dry-run + confirm pattern mirroring the install endpoint (`confirm: false` returns the un-wrap diff preview; `confirm: true` performs the change). Audit verb `crons.wrapper_uninstalled`.

7. **UI.** CronDetail's Actions panel: when `wrapper_last_seen_at` is set AND `is_local`, show "Remove heartbeat wrapper" instead of "Install heartbeat wrapper". A RemoveHeartbeatModal — or a parameterized/reused InstallHeartbeatModal — with a dry-run preview (the un-wrap diff) → confirmation checkbox → confirm.

## Concrete deliverables

1. **Local-host uninstall flow** in `apps/monitor/homelab_monitor/kernel/cron/installer.py`
   - `async def uninstall_wrapper_local(fingerprint: str, *, who: str, ip: str | None) -> UninstallResult` (or similar):
     1. Verify `host == local hostname`; raise `RemoteHostUnsupported` otherwise.
     2. Fetch the cron row; verify a wrapper is installed (`wrapper_last_seen_at` set).
     3. Snapshot the crontab file at `source_path`.
     4. Parse the source file, find the wrapped line whose unwrapped fingerprint matches, reverse `<schedule> <wrapper> -- <command>` to `<schedule> <command>` via `unwrap_command()` — inner command byte-exact.
     5. Atomic write+rename.
     6. Per Design decision #1/#2: handle wrapper-script + token-file cleanup (likely leave the token; conditionally handle the script).
     7. Per Design decision #3: clear or retain `wrapper_last_seen_at`.
     8. Audit verb `crons.wrapper_uninstalled`.
   - Errors: full rollback — restore the original (wrapped) crontab file.

2. **API endpoint** in `apps/monitor/homelab_monitor/kernel/api/routers/crons.py`
   - `POST /api/crons/{fingerprint}/uninstall-wrapper` (final verb/method confirmed in Design):
     - Body: `UninstallWrapperRequest { confirm: bool }`. `confirm: true` required to actually modify the crontab.
     - For `confirm: false`: returns a **dry-run preview** with the un-wrap diff of the crontab change. No file modifications.
     - For `confirm: true`: calls `uninstall_wrapper_local`. Returns updated `CronOut`.
   - Session-auth + CSRF.
   - Returns 400 if the cron's host is not the local host, or if no wrapper is installed.

3. **CLI subcommand** in `apps/monitor/homelab_monitor/cli/cron.py`
   - `hm cron uninstall-wrapper <fingerprint>` — equivalent to the UI button. Same dry-run-then-confirm pattern as `install-wrapper`.

## Locked Design Decisions

**D1 — Wrapper script cleanup:** NEVER delete the shared /usr/local/bin/cron-with-heartbeat.sh on a per-cron uninstall. Uninstall is a pure crontab-line edit. Rationale: the script is ~1.5KB and harmless when unreferenced; reference-counting would force a multi-file crontab scan into the executor with a delete-while-still-referenced TOCTOU race.

**D2 — Shared token:** NEVER touch /etc/homelab-monitor/heartbeat.token on a per-cron uninstall. It is one shared token used by every wrapped cron. The uninstall IPC request is a single unwrap-crontab operation — no token op.

**D3 — wrapper_last_seen_at:** CLEAR it to NULL on successful uninstall, via a new repository method record_wrapper_uninstalled that writes the audit row AND nulls the column in one transaction. Rationale: the column is a live "wrapper installed & functioning" signal (List filter, UI badge, STAGE-010 wrapper_installed metric + WrapperPossiblyStale alert); retaining it after uninstall would make STAGE-010's WrapperPossiblyStale alert false-fire permanently. Note the asymmetry: record_wrapper_installed does NOT set the column (a heartbeat does), but uninstall MUST clear it directly because no heartbeat will ever arrive to clear it naturally.

**D4 — Endpoint:** POST /api/crons/{fingerprint}/uninstall-wrapper with body {confirm: bool}, mirroring the install endpoint exactly (dry-run preview when confirm=false → confirm when confirm=true). New schemas UninstallWrapperRequest / UninstallWrapperPreview / UninstallWrapperResult mirroring the Install* schemas. Rationale: codebase consistency + the mandatory un-wrap diff preview both point to POST-with-confirm; DELETE has no body-and-preview idiom.

**D5 — UI:** build a NEW RemoveHeartbeatModal component (NOT a parameterized InstallHeartbeatModal). Extract a shared CrontabLineDiff presentational sub-component used by both the install and remove modals (the old->new crontab line diff renderer). The CronDetail ActionsPanel row becomes an Install/Remove TOGGLE: show "Install heartbeat wrapper" when wrapper_last_seen_at is null, "Remove heartbeat wrapper" when it is set; both local-host only. (STAGE-009 shipped only the static Install button; this toggle is delivered in 009A.) **NOTE (Refinement): During 3a refinement, the toggle signal was changed from wrapper_last_seen_at to the new wrapper_installed boolean column (alembic 0013) to handle the case where a wrapper is installed but has not yet fired a heartbeat; the Design decision's original choice of wrapper_last_seen_at was superseded.**

**D6 — Uninstall gate:** gate on the discovered crontab line's is_wrapped() (the crontab line is ground truth), NOT on wrapper_last_seen_at (which is null for an installed-but-never-fired wrapper). Add a new typed error NotWrappedError mapped to HTTP 409, the exact mirror of AlreadyWrappedError. wrapper_last_seen_at is used ONLY by the UI to choose which toggle row to show.

**D7 — Executor op:** new UnwrapCrontabOp(target_crontab, old_line, new_line) dataclass in cron_apply_ipc.py — NO `command` field (a wrapped line fully contains its original; unwrap derives everything from old_line). new_line is the defense-in-depth cross-check only. New apply_unwrap_crontab function in scripts/hm-cron-apply.sh mirrors apply_wrap_crontab: resolve target via resolve_target (same allow-list), REFUSE if old_line is NOT wrapped (inverse of wrap's already_wrapped guard — reject with error_code "not_wrapped"), re-derive new_line by stripping the wrapper prefix (executor authoritative, byte-identical to unwrap_command), cross-check the supplied new_line, snapshot + rollback via the existing shared machinery. New unwrap-crontab arm in the op-dispatch case statement; new OP constant in cron_apply_constants.py.

Additional: fingerprint round-trip (install -> uninstall -> fingerprint unchanged) is already correct by construction — the cron-discoverer unwraps before fingerprinting — so this needs only a REGRESSION TEST, no code change. Rollback is inherited free from the shared executor snapshot/rollback machinery.

Three uninstall paths mirror the three install paths: the UI Remove button, the CLI `hm cron uninstall-wrapper`, and the standalone remote CLI gaining an uninstall mode. Local-host gate (cron.host == HM_HOST_HOSTNAME); remote uninstall deferred to EPIC-017.

## Refinement Results (3a + 3b)

Refinement was a mixed-stage validation: 3a (dev rig, synthetic) + 3b (prod rig, real host data) + frontend viewport sign-off.

**3a — dev rig:** Backend e2e 8/8 scenarios passed (discovery with unwrapped command + is_local; dry-run uninstall preview; confirm uninstall reverts the crontab byte-exact; D1 wrapper script survives; D2 token survives; D6 not-wrapped → 409; fingerprint round-trip unchanged; wrapper_last_seen_at cleared). Frontend: RemoveHeartbeatModal + CrontabLineDiff + the Install/Remove toggle approved at desktop and mobile viewports.

**3a found TWO real bugs (both fixed):**
- **Bug 1** — the Install/Remove toggle keyed on wrapper_last_seen_at (a wrapper-HEALTH signal that is NULL for an installed-but-never-fired wrapper), so after install the button did not flip to "Remove" and a re-install hit AlreadyWrappedError. Fix: added a stored `wrapper_installed` boolean column (alembic migration 0013), populated by the discoverer from the crontab line's actual wrapped state and by install/uninstall; exposed on CronOut; the toggle now keys on it. This SUPERSEDES locked Design decision D5's choice of wrapper_last_seen_at as the toggle signal.
- **Bug 2** (found in 3a, root-caused as a real PRODUCTION bug) — the install/uninstall dry-run gate reads the crontab snapshot while the executor rewrites the spool; the snapshot was only reconciled by a 300s timer (the .path watcher misses the executor's rename-in-place writes), so toggling the wrapper twice within 5 minutes failed. Fix: the executor (hm-cron-apply.sh) now refreshes the snapshot INLINE after every successful wrap/unwrap (a new refresh_user_snapshot helper copies the spool to the snapshot dir).

**3b — prod rig (real host, the live wrapped rtlamr-watchdog.sh cron):** the monitor container was rebuilt with STAGE-009A code; host-setup.sh re-run to install the updated executor. First 3b run passed functionally (real crontab uninstalled→reinstalled byte-exact, fingerprint stable, vixie-cron healthy, no INSECURE MODE) BUT found a THIRD real bug: the executor's inline snapshot-refresh failed with "mktemp failed" because homelab-monitor-cron-apply.service's ProtectSystem=strict + ReadWritePaths did NOT include the crontab-snapshot dir — the inline refresh was silently dead, masked by the .path-triggered snapshot service. Fix: added /var/lib/homelab-monitor/crontab-snapshot to the cron-apply.service ReadWritePaths, plus a static contract test. 3b RE-validation after the fix: 4/4 scenarios passed — the executor journal now shows "snapshot refreshed: jakekausler" with zero mktemp warnings; the full uninstall→reinstall round-trip works byte-exact on the real host; the host was left with rtlamr-watchdog wrapped (as found).

## Refinement follow-up (prod-instance frontend 3b + a fourth fix)

The initial Refinement-complete marking was premature — the frontend 3b validation against the real PROD instance had been skipped (3b had covered backend/API validation + 3a covered dev-rig frontend testing, but the production UI was never user-tested against real host data). This was corrected:

- A FOURTH issue surfaced: after migration 0013 added the wrapper_installed column with a default of false, every pre-existing wrapped cron's row shows wrapper_installed=false until the next discovery tick (up to 300s) — so on a real upgrade the Install/Remove toggle briefly shows the wrong state. Fix: the monitor now requests a one-shot cron-discovery run on startup (lifespan.py calls scheduler.request_immediate_run for the cron-discoverer collector, guarded on successful registration, best-effort, fire-and-forget). This converges wrapper_installed within seconds of any restart/upgrade instead of up to 5 minutes. Verified on the rebuilt prod container: startup discovery fired at T+0.05s and converged the real rtlamr-watchdog cron to wrapper_installed=true.

- A pre-existing flaky test was found and fixed: test_lifespan_warns_when_https_only_cookies_disabled in test_lifespan_e2e.py was order-dependent (configure_logging() replaces root handlers behind a process-global flag, wiping pytest's caplog handler when the test runs first on its xdist worker). Fixed to use structlog.testing.capture_logs, which is immune to handler replacement. This was a latent make-verify flake, now removed.
