## Overview

Surface cron run history and run-log viewing in the dashboard UI: a "Recent runs" teaser panel on the cron detail page, a dedicated run-history list route, and a dedicated run-log viewer route.

STAGE-002-014 exposes `GET /api/crons/{fingerprint}/runs` and `GET /api/crons/{fingerprint}/runs/{run_id}/log`. This stage builds the three frontend surfaces that consume them. It is a frontend-only stage.

**Vertical slice:** on a cron's detail page, a "Recent runs" panel shows the last 3–5 runs with exit-code chips and anomaly badges and a "View all runs →" link; the link opens `/crons/{fingerprint}/runs`, a paginated run-history table; clicking a run opens `/crons/{fingerprint}/runs/{run_id}`, a run-log viewer showing that run's stdout/stderr.

**Spec references:** `docs/superpowers/specs/2026-05-19-cron-run-logs-design.md` §9 (UI), §8 (the API it consumes), D12.

## Concrete deliverables

### 1. "Recent runs" teaser panel on `CronDetail.tsx`

`apps/ui/src/components/crons/CronDetail.tsx` — a new (5th) panel:

- Shows the **last 3–5 runs** of the cron — for each: the run timestamp, duration, an exit-code chip, and an anomaly badge when the run has non-empty `anomaly_flags`.
- A "View all runs →" link to the dedicated run-history list route.
- Consumes `GET /api/crons/{fingerprint}/runs?limit=5`.
- Keeps the detail page light — the full history lives on its own page.
- Empty state — a cron with no runs yet shows a clear "no runs recorded yet" message.

### 2. `/crons/{fingerprint}/runs` — run-history list route

A new dedicated route + page component:

- A paginated run-history table — for each run: started-at time, duration, an exit-code chip, `source` (wrapper / logscrape), anomaly badges, and an `overlapping` indicator for B-mode runs flagged `overlapping`.
- A filter by `state` (`running`/`ok`/`fail`/`unknown`).
- Pagination via the API's `cursor`.
- Each row links to the run-log viewer route.
- Consumes `GET /api/crons/{fingerprint}/runs`.
- Deep-linkable.

### 3. `/crons/{fingerprint}/runs/{run_id}` — run-log viewer route

A new dedicated route + page component:

- A run-metadata header — state, duration, exit code, the anomaly flags (if any).
- The log text — monospace, scrollable.
- Consumes `GET /api/crons/{fingerprint}/runs/{run_id}/log`.
- **Behaviors for the API's response shapes:**
  - In-flight run (`state: running`) → shows output-so-far + a **manual refresh** button (NO live-tail this stage).
  - Aged-out logs (`log_status: expired`) → an explicit "log text no longer available (past VictoriaLogs retention)" notice + the run metadata.
  - VL unreachable (`503`) → a "logs temporarily unavailable" message — NOT a hard error page.
  - `truncated: true` → a clear "output truncated at N lines" indicator.
- Deep-linkable so a specific failed run can be bookmarked/shared.

### 4. Routing + API client

- Register the two new routes in the UI router alongside the existing cron routes.
- Add the API-client functions for the two new endpoints (the generated `apps/ui/src/api/schema.ts` already reflects them after STAGE-002-014 regenerated `openapi.json`).

### 5. Tests

`apps/ui/src/components/crons/__tests__/` and route tests (`vitest`):

- The "Recent runs" teaser panel — renders runs, exit-code chips, anomaly badges, the empty state, the "View all runs" link.
- The run-history list route — table rendering, the `state` filter, pagination, the `overlapping` indicator, row → log-viewer navigation.
- The run-log viewer route — all four response states (available, in-flight + manual refresh, expired, `503` unavailable) and the `truncated` indicator.
- 75%+ UI branch coverage maintained; eslint warnings stay at the 35 baseline.

## Locked Design Decisions

D-UI-SURFACES — the UI is exactly three surfaces: a "Recent runs" teaser panel on `CronDetail` (last 3–5 runs) + a dedicated `/crons/{fingerprint}/runs` list route + a dedicated `/crons/{fingerprint}/runs/{run_id}` log-viewer route. The list and viewer are deep-linkable. (Spec D12.)

D-UI-REFRESH — in-flight runs get a MANUAL REFRESH button, NOT live-tail. Live-tail is explicitly EPIC-004. (Spec D12.)

D-UI-STATES — the run-log viewer explicitly handles all four API response shapes: available, in-flight (manual refresh), expired ("logs no longer available" notice), and `503` ("temporarily unavailable" — not a hard error). Plus a `truncated` indicator. (Spec §8.2, §9.3.)

D-UI-TEASER — the cron detail page gets only a light teaser panel; the full history lives on its own route. (Spec §9.1, D12.)

**Design Decisions (resolved 2026-05-19):**

| Decision | Resolution |
| --- | --- |
| `CronDetail` 5th-panel layout | Span-full row at the bottom (`lg:col-span-2` on the existing 2-col grid). Detail page extends downward; mobile stays linear. Zero disruption to existing 4-panel screenshot baselines. |
| Run-history list mobile | Table on desktop, card list on mobile (below `md` breakpoint). Each card carries timestamp + state chip + duration in the header row and exit_code · source + anomaly badges + overlapping indicator in the secondary row. Triage-critical badges (anomaly, overlapping) are always visible without horizontal scroll. |
| Log viewer layout | Sticky metadata header (state, duration, exit, anomaly flags, Refresh button for in-flight) + scrolling log body. Header pinned via `position: sticky`; log body inherits page scroll. Matches familiar log-viewer ergonomics (kubectl, GitHub Actions). |
| Run-state chip | New `RunStateBadge` component (separate from the existing `StateBadge`). Maps `ok → variant=ok`, `fail → variant=critical`, `running → variant=default`, `unknown → variant=muted`. Clean semantic separation from cron-aggregate states; no regression risk on existing badge tests. |
| Refinement seed data | A dev-only `hm dev seed-cron-runs <fingerprint>` CLI subcommand (or pytest fixture helper) inserts ~60 deterministic `cron_runs` rows for a chosen cron: mixed states (`ok` / `fail` / `unknown`), several with non-empty `anomaly_flags`, two with `overlapping=1`, both sources (`wrapper` and `logscrape`). VL-side response shapes (`running` / `expired` / 503 / `truncated`) handled by MSW mocks in vitest unit tests + manual `docker compose stop victorialogs` for the 503 demo. |
