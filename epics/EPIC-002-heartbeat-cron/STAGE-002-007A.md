## Overview

Discovered crons can disappear from their source files when the operator edits a crontab, removes a cron job, or deletes a `/etc/cron.d/` file. Currently, the registry accumulates orphan rows indefinitely — rows whose underlying cron no longer exists on disk but which have no way to signal their absence. The only available operator action is manual hide, which requires noticing the stale row in the first place.

This stage introduces **automatic soft-delete** driven by discovery scan results. When the `cron-discoverer` plugin completes a successful scan of a host+file combination and the fingerprint of a previously-registered cron is no longer present in that scan's results, the cron row is marked soft-deleted (`soft_deleted_at = utc_now_iso()`). This is the system's signal that "the source file was reachable and parsed, but this cron is gone." If a later scan finds the fingerprint again (e.g., the operator re-adds the cron), the row is auto-restored (`soft_deleted_at = NULL`).

The reachability gate is the critical safety invariant: if a discovery scan fails to reach the host or read the file (network error, permission denied, parse error, or any other exception), the soft-delete logic is **not run** for that scan. "We couldn't see the file" is not the same as "the cron is gone." Only a successful end-to-end scan can trigger soft-delete. This check is in-memory at scan time — no persistent reachability state is stored.

This stage is **distinct from operator-driven hide** (`hidden_at`, from STAGE-002-003). `hidden_at` is a presentation and notification suppression mechanism that the operator controls manually. `soft_deleted_at` is an automated, presence-driven signal set and cleared by the discovery scanner. Both columns can coexist on the same row. Heartbeats and direct-fetch APIs are unaffected by either column — data capture always continues.

## Concrete deliverables

1. **Alembic migration** (new revision after whatever migration STAGE-002-007 ships — likely `0012_soft_deleted_at.py` or the appropriate next number): `ADD COLUMN soft_deleted_at TEXT NULL` on the `crons` table (UTC ISO-8601 string; NULL means active; non-NULL means auto-soft-deleted).

2. **Discovery scanner extension** (in `apps/monitor/homelab_monitor/plugins/discoverers/cron_discoverer.py` introduced by STAGE-002-007): after a successful scan of a host+source_path combination completes (file reachable AND fully parsed), apply the following reconciliation logic:
   - Compute the set of fingerprints present in the current scan for this host+source_path.
   - Query the registry for all cron rows whose `host` + `source_path` matches the scanned file.
   - For each such row:
     - Fingerprint **IS** in the scan set AND `soft_deleted_at IS NOT NULL` → SET `soft_deleted_at = NULL`. Audit verb: `crons.restore` (auto). Log: `"auto-restored cron previously soft-deleted"`.
     - Fingerprint **IS NOT** in the scan set AND `soft_deleted_at IS NULL` → SET `soft_deleted_at = utc_now_iso()`. Audit verb: `crons.soft_delete` (auto). Log: `"auto-soft-deleted cron absent from scan"`.
     - All other combinations (present + already active; absent + already soft-deleted) → no-op.
   - This reconciliation runs only after a scan completes without exception; any exception short-circuits and skips reconciliation for that scan.

3. **In-memory reachability gate**: if the discovery scan raises any exception (host unreachable, file unreadable, permission denied, parser error, etc.), catch it, log the error, and return without running the soft-delete reconciliation. The gate is enforced at the scan-function level — a single try/except around the scan + reconciliation block, with reconciliation inside the happy path only.

4. **API filter** (`GET /api/crons`): default query adds `AND soft_deleted_at IS NULL` to the existing filter (alongside the existing `hidden_at IS NULL` default). New optional query param `include_soft_deleted: bool | None = None`. When `True`, the `soft_deleted_at IS NULL` clause is omitted. Follows the same pattern as `include_hidden`.

5. **`GET /api/crons/{fingerprint}` direct fetch**: returns the row regardless of `soft_deleted_at` (same behavior as `include_hidden` on direct-fetch — direct access is always unfiltered).

6. **`CronOut` schema extension**: add `soft_deleted_at: str | None` field to the Pydantic response schema (and regenerate `apps/ui/src/api/schema.ts` via the `generate-ui-types` hook).

7. **`/register` soft-delete auto-restore**: if a fingerprint receives a `POST /api/hb/{fingerprint}/register` call and `soft_deleted_at IS NOT NULL`, clear it (`soft_deleted_at = NULL`) as part of the register transaction. Audit verb: `crons.restore` (from register). This handles the case where a cron was soft-deleted and then the heartbeat wrapper is re-installed for it.

8. **UI list view filter**: add a "Show soft-deleted" toggle in `CronsToolbar` (mirrors the "Show hidden" toggle). When ON, appends `include_soft_deleted=true` to the list query. Toggle is off by default.

9. **UI table indicator**: small "Soft-deleted" badge in the cron row when `soft_deleted_at` is set. Parallel to the "Hidden" badge from STAGE-002-006. Badge color: muted red/amber to signal automated removal, distinct from the grey "Hidden" badge.

## Scan-set partial-success semantics (locked 2026-05-13 in STAGE-002-007)

**STAGE-002-007 returns `CronScanResult(found_fingerprints, partial, errors)`**. 007A's auto-soft-delete reconciliation runs only when `partial=False`. Per-file errors (perm denied, parse error) set `partial=True` and skip the soft-delete pass that cycle, but new INSERTs and `last_discovered_at` bumps still apply. An unhandled exception that prevented running the scan at all (e.g., host completely unreachable) short-circuits both scan and reconciliation.

| Decision | Recommendation |
|---|---|
| Auto-restore on re-discovery | Yes — if scan finds fingerprint again, clear `soft_deleted_at`. Emit `crons.restore` audit. |
| Auto-restore on `/register` | Yes — if a `/register` call arrives for a soft-deleted fingerprint, clear `soft_deleted_at` as part of the register transaction. |
| Soft-delete audit verb | `crons.soft_delete` (auto-set by scan); `crons.restore` (auto-cleared by scan or by `/register`). |
| Reachability check granularity | In-memory at scan time. Any exception during the scan short-circuits soft-delete logic for that scan invocation. No persistent reachability state. |
| Column storage type | `TEXT NULL` (UTC ISO-8601 string), consistent with all other timestamp columns in the `crons` table. |
| Manual operator undelete | Operator can manually clear `soft_deleted_at` via a future PATCH endpoint. Out of scope for this stage (uncommon use case). |
| Operator hide + soft-delete coexistence | Both columns can be set simultaneously. A cron can be both hidden (operator-controlled) and soft-deleted (scanner-controlled). The list API filters on both independently. |
| Hard delete | OUT OF SCOPE. Soft-deleted rows persist indefinitely. Hard-delete automation (e.g., purge rows older than 30d) is a future stage. |
| Cross-host / remote-only crons | Crons with `source_path IS NULL` (remote-only, no source file on this host) are NEVER auto-soft-deleted. The scanner only reconciles rows whose `host` + `source_path` match the scanned file. |
| Heartbeat behavior on soft-deleted crons | `/start`, `/ok`, `/fail` on a soft-deleted fingerprint continue to work normally. Data capture is unaffected by `soft_deleted_at`. |
| Stale badge interaction | The STAGE-002-007 "Stale (no longer discovered)" badge (based on `last_discovered_at` age) and the STAGE-002-007A "Soft-deleted" badge serve different purposes: stale = "we haven't seen it recently but don't know why"; soft-deleted = "a successful scan confirmed it's gone." When `soft_deleted_at` is set, prefer the "Soft-deleted" badge; suppress or deprioritize the "Stale" badge for that row. |
| `include_soft_deleted` default | `False` (omitted from default list). Matches operator mental model: soft-deleted rows are gone and should be invisible by default. |

## Locked Design Decisions (2026-05-14)

### D1 — Reconciliation gate: per-source-file (OVERRIDES pre-doc'd global `partial=False` lock)
**Locked: Per-source-file gating. CronScanResult extended.**

- Brainstormer pressure-test (2026-05-14) revealed that a global `partial=False` gate creates a long-tail failure mode: any chronically-broken file (perm denied, persistent parse error) pins `partial=True` permanently and soft-delete reconciliation NEVER runs on that host.
- **Fix:** extend `CronScanResult` to carry per-source-file state:
  - `clean_source_paths: frozenset[str]` — paths cleanly inspected this scan, AND absent-but-known paths where the parent directory was cleanly iterated (e.g., `/etc/cron.d/foo` if `iterdir(/etc/cron.d/)` succeeded but `foo` wasn't returned)
  - `unreachable_source_path_prefixes: frozenset[str]` — parent dir prefixes where `iterdir` itself failed (e.g., `/etc/cron.d/` if its iterdir threw OSError)
  - `found_by_source_path: dict[str, frozenset[str]]` — fingerprints grouped by source_path
- **Reconciliation logic:** for each path in `clean_source_paths`, compare DB rows for `host=local_host AND source_path=path` against `found_by_source_path[path]`. Rows present in DB but absent from scan → soft-delete. Rows present in both with `soft_deleted_at != NULL` → restore.
- **Per-host filter:** reconciliation MUST include `WHERE host = <local_hostname>` to prevent touching rows registered by remote hosts.
- **Why:** A broken user crontab no longer blocks reconciliation for /etc/crontab + /etc/cron.d/*. Real-world hosts have persistent partial-scan conditions; this design accommodates them.
- **Cost:** ~20 LoC dataclass extension + ~60 LoC reconcile method. Brainstormer's revised Build estimate: 8-12 verify cycles.

### D2 — Reconciliation placement: new `CronRepo.reconcile_soft_deletes()` method
**Locked: New first-class repo method.**

- Signature: `async def reconcile_soft_deletes(self, *, host: str, clean_paths: frozenset[str], unreachable_prefixes: frozenset[str], found_by_path: Mapping[str, frozenset[str]], now: str) -> tuple[int, int]` returns `(soft_deleted_count, restored_count)`.
- Called from BOTH `CronDiscoverer.run()` (after `scan()`) AND `POST /api/crons/discover-now` endpoint (after `scan()`).
- Keeps `scan()` as a parse-and-upsert layer; reconciliation is a separate repo operation. Easier to test in isolation.
- **Why:** Matches existing repo-method pattern (`upsert_discovered`, `register_cron`, `soft_delete_cron`). Both auto-trigger paths exercise the same code path.

### D3 — Absent-but-known files count as "clean" for reconciliation
**Locked: Yes.**

- If DB has rows with `source_path = /etc/cron.d/foo` but the next clean scan doesn't find that file on disk (admin did `rm /etc/cron.d/foo`), those rows are soft-deleted.
- Implementation: `_scan_one_file` is NOT called for the absent file (it's not in the iterdir output), so `found_by_source_path` won't have a key for it. BUT `clean_source_paths` MUST include known-absent paths under cleanly-iterated parents. The discoverer builds this set explicitly: any path the scanner WOULD have inspected (`/etc/crontab`, `/etc/cron.d/*`, `/var/spool/cron/crontabs/*`) that wasn't blocked by an unreachable parent prefix.
- Rows with source_paths the scanner doesn't recognize (e.g., custom path inserted by some future plugin) are NEVER touched by reconciliation — they're not in `clean_source_paths`.
- **Why:** Without this, removing a whole crontab file leaves orphan rows forever. The spec's whole point is automated cleanup.

### D4 — Audit row content: minimal diff
**Locked: One-field transitions only.**

- `crons.soft_delete`: `before = {"fingerprint": "<fp>", "soft_deleted_at": null}`, `after = {"fingerprint": "<fp>", "soft_deleted_at": "<ts>"}`
- `crons.restore`: `before = {"fingerprint": "<fp>", "soft_deleted_at": "<ts>"}`, `after = {"fingerprint": "<fp>", "soft_deleted_at": null}`
- Mirrors `crons.hide`/`crons.unhide` precedent from STAGE-002-005.
- `who = "system"` for discovery-triggered transitions; `who = <user.username>` for `/register`-triggered restores.
- **Why:** Soft-delete is a one-bit transition. Full-row dumps are wasteful; the row itself is the source of truth (hard-delete is out of scope per locked spec). Bug-B-style full-row diff was for the `crons.discover.update` verb where MULTIPLE fields could change; here only `soft_deleted_at` changes.

### D5 — `/register` auto-restore: two separate audit rows in one transaction
**Locked: `crons.restore` followed by `crons.register`.**

- When `/register` lands on a fingerprint with `soft_deleted_at IS NOT NULL`:
  1. Clear `soft_deleted_at` (set to NULL).
  2. Write `crons.restore` audit row, `who = <authenticated user.username>`, before/after = the soft_deleted_at transition (per D4 minimal diff).
  3. Proceed with the normal `crons.register` flow (handles `wrapper_last_seen_at` update + writes its own `crons.register` audit row).
- Both writes in the SAME transaction; if any step fails, all roll back.
- **Why:** The `crons.restore` verb has one consistent semantic regardless of trigger (discovery scan OR /register). Querying audit log for "all restore events" returns a clean list. Two rows are cheap; the alternative (folding restore into crons.register's before/after) overloads the register audit verb's diff shape.

### D6 — UI: parallel to "Show hidden" pattern
**Locked: Toggle + badge + opacity + detail page header + Disk source panel field.**

- **Toolbar toggle**: "Show soft-deleted" checkbox in CronsToolbar, parallel to existing "Show hidden". When ON, list refreshes with `?include_soft_deleted=true`. OFF by default.
- **List row badge**: "Soft-deleted" badge near the State pill, muted red/amber color (distinct from operational badges). Tooltip shows soft-delete timestamp + relative time.
- **List row opacity**: rows with `soft_deleted_at` set get `opacity-60` (or similar) to indicate inactive state. Mirrors how hidden rows visually de-emphasize.
- **CronDetail header**: "Soft-deleted" badge alongside Hidden, Remote badges (which already exist post-STAGE-002-006).
- **CronDetail Disk source panel**: New "Soft-deleted" row showing the timestamp + relative-time tooltip. Parallels "Last discovered" pattern added in STAGE-002-007.
- **No new action button**: restore is automatic via `/register` or re-discovery. No manual UI control.
- **Why:** "Show hidden" pattern is well-established; building a parallel is mostly mechanical. Per binding rule, Refinement requires desktop+mobile sign-off — but the surface is small.

## Approach Summary

Adds soft-delete reconciliation to the cron discovery pipeline. After every scan (scheduled tick OR `/api/crons/discover-now`), `CronRepo.reconcile_soft_deletes()` runs with the scan's per-source-file state. For each cleanly-inspected source path, DB rows for that host+path are compared against the scan's fingerprints for that path. Mismatches drive soft-delete or restore transitions, emitting `crons.soft_delete` or `crons.restore` audit rows with minimal field diffs.

`/register` calls additionally auto-restore the row if it was soft-deleted, writing two audit rows in one transaction (`crons.restore` then `crons.register`).

API surface adds `?include_soft_deleted` query param to `GET /api/crons` (default false). Direct fetch via `GET /api/crons/{fingerprint}` is always unfiltered. `CronOut` gains `soft_deleted_at: str | None`.

UI adds "Show soft-deleted" toolbar toggle + "Soft-deleted" badge + opacity-dimmed soft-deleted rows + detail-page header badge + Disk source panel field. Mirrors existing "Show hidden" pattern.

Migration 0011 adds `soft_deleted_at TEXT NULL` to crons table with idempotent inspector pattern (mirrors 0010).

## Risks/Things to Watch in Build

1. **CronScanResult field additions are NOT backward-compatible.** Consumers of `CronScanResult` (in `discoverer.run()`, the POST endpoint, tests) all need to accommodate the new shape. Default values via `field(default_factory=...)` keep dataclass defaults clean.

2. **`/register` audit ordering**: `crons.restore` writes BEFORE `crons.register`. Both in same transaction. Test must verify the row ordering in audit_log.

3. **Reconciliation must not affect `last_discovered_at`**: it's a separate column with its own semantics. Bumping `last_discovered_at` is the upsert path's job. `reconcile_soft_deletes` only touches `soft_deleted_at`.

4. **Test the SEQUENCE**: upsert phase (which bumps last_discovered_at and inserts new rows) runs FIRST. Then reconcile_soft_deletes runs SECOND. If reconcile ran before upsert, newly-inserted rows would be soft-deleted instantly because they're "not in DB before upsert."

5. **The clean_source_paths set must distinguish "scanner KNOWS this path was empty" from "scanner has never heard of this path."** Custom source_paths from future plugins must NOT be auto-soft-deleted just because the local scanner doesn't recognize them.

## Cross-stage notes

- Depends on STAGE-002-007 shipping the `cron-discoverer` plugin and the `last_discovered_at` column. The `soft_deleted_at` column added here coexists with `last_discovered_at` — they serve different purposes: `last_discovered_at` is "when did we last see this cron"; `soft_deleted_at` is "when did a successful scan confirm it's gone."
- STAGE-002-008 (B-mode log-scrape) should filter its cron lookup to exclude soft-deleted rows by default, consistent with how the API list works. If a soft-deleted cron's log lines arrive, they can be silently dropped or attributed to an archived row — Design for 008 should account for this.
- STAGE-002-009 (wrapper helpers) "Install heartbeat" UI button: the button should be disabled or hidden for soft-deleted rows in the detail page (no point installing a wrapper on a cron that no longer exists on disk). This is a Design decision for 009 to handle.
- STAGE-002-010 (vmalert rules): soft-deleted crons should be excluded from lateness/failure alerting, similar to hidden crons. The metric filter `soft_deleted_at IS NULL` must flow through to the metrics export layer. Design for 010 to confirm.
