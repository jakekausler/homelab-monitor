# STAGE-002-001: Heartbeat receiver + `crons`/`heartbeats_state` schema + audit

## Status: Not Started

## Overview

Replace the auth-only heartbeat router stub from STAGE-001-021 with a real receiver that persists per-cron state. Land the migration that adds the behavioral columns to `crons` (cadence, grace, integration_mode) and `heartbeats_state` (current_state, last_ok_at, current_streak, expected_next_at), wire `/hb/{id}/start`, `/hb/{id}/ok`, and `/hb/{id}/fail` to update those tables, audit every state transition, and emit per-cron freshness metrics to VictoriaMetrics so STAGE-002-006 can write rules against them.

This stage delivers the foundation the rest of the epic depends on. Vertical slice: a registered cron's first ping creates a `heartbeats_state` row, subsequent pings update it, the row is queryable via a new `/api/crons/{id}/heartbeat` endpoint, and `homelab_heartbeat_seconds_since_last_ok{cron_id="..."}` appears in VM within 30 seconds of the ping.

2. **Heartbeat receiver router replacement** at `apps/monitor/homelab_monitor/kernel/api/routers/heartbeat.py`
   - Replace the stub with three endpoints:
     - `POST /hb/{cron_id}/start` â€” records `last_start_at = now_utc()`; sets `current_state` to `running` if currently `ok`/`unknown`, leaves `fail` alone; returns 204
     - `POST /hb/{cron_id}/ok` â€” accepts optional `?duration=<seconds>` query param; records `last_ok_at`, increments `current_streak` (or resets from negative to +1), recomputes `expected_next_at`, sets `current_state = ok`, mirrors to `crons.last_seen_state`; returns 204
     - `POST /hb/{cron_id}/fail` â€” records `last_fail_at`, decrements `current_streak`, sets `current_state = fail`, mirrors to `crons.last_seen_state`; returns 204
   - All three: 404 if `cron_id` not found in `crons` (per cross-stage criterion #9 â€” no auto-create); 401 if no token (existing); 403 if token lacks `heartbeat:write` (existing).
   - All three: write to `audit_log` (who = token name, what = `heartbeat.{start|ok|fail}`, before/after = state JSON).
   - All three: structured log entry at `info` level with `cron_id`, `state_transition`, `token_name`.

3. **Heartbeat repository** at `apps/monitor/homelab_monitor/kernel/db/heartbeat_repo.py`

     - `async def upsert_state(cron_id: int, **fields) -> HeartbeatsState` (atomic upsert)
     - `async def record_start(cron_id: int, ts: datetime) -> HeartbeatsState`
     - `async def record_ok(cron_id: int, ts: datetime, duration: float | None) -> HeartbeatsState`
     - `async def record_fail(cron_id: int, ts: datetime) -> HeartbeatsState`
   - All methods take a `connection` parameter; the router opens a transaction so a single heartbeat updates `heartbeats_state` + `crons.last_seen_state` + `audit_log` atomically.

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **`/hb/{id}/start` on unknown id: 404 vs auto-create the cron row?**
   - **Recommended: 404 (force explicit registration).** Auto-create is a footgun â€” typos in cron commands become orphan registry entries, and "discovered through ping" loses the rich context (host, command, schedule) that auto-discovery provides. The trade-off: users have to add the cron to the registry first before pinging. That's fine for opt-in heartbeats since the user is already touching the dashboard.
   - Alternative: auto-create with `host="unknown"`, `command="unknown"`, `integration_mode="heartbeat"`. Recovers from "I added the curl ping before adding the cron in the dashboard" UX hiccups but creates registry pollution.
   - Decision matters for: STAGE-002-005 (wrapper helper UX â€” does the dashboard create the cron row when the user clicks "Install heartbeat", or does the wrapper handle it?).

2. **`/ok?duration=Ns` query param: accept or always 200 OK with no body?**
   - **Recommended: accept `?duration=<seconds>`** as an optional query param. Healthchecks.io supports it (via POST body), and runtime tracking is genuinely useful for the digest's "cron heartbeat report" section (per spec Â§11). Capping at 86400s (24h) prevents pathological values.
   - Alternative: ignore duration entirely; rely on `last_ok_at - last_start_at` math when both arrive. Slightly cleaner data model, but requires the cron to call `/start` AND `/ok` (most users will skip `/start`).

3. **Token scoping: single global `heartbeat:write` vs per-cron token?**
   - **Recommended: single global `heartbeat:write` token, with per-cron rate-limiting.** Avoids token-explosion at the start. Per-cron tokens become useful when crons run on different hosts (compromise of one host shouldn't let an attacker spoof every cron's heartbeat) â€” defer to a later stage.
   - Alternative: per-cron token field on the `crons` table; receiver checks `token.id == cron.token_id`. Stronger blast-radius isolation, more management overhead.

4. **`last_seen_state` mirror on `crons` table: keep or drop?**
   - **Recommended: keep the mirror.** Spec Â§6.1 explicitly lists `last_seen_state` on the `crons` table. Denormalization makes the Inventory list query a single SELECT vs JOIN. Update both tables in the same transaction.
   - Alternative: drop the mirror; always JOIN `heartbeats_state`. Cleaner schema, slightly slower list endpoint. Not worth the schema deviation from spec.
