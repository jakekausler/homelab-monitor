# STAGE-002-001: Heartbeat receiver + `crons`/`heartbeats_state` schema + audit

## Status: Design (Complete)

## Overview

Replace the auth-only heartbeat router stub from STAGE-001-021 with a real receiver that persists per-cron state. Land the migration that adds the behavioral columns to `crons` (cadence, grace, integration_mode) and `heartbeats_state` (current_state, last_ok_at, current_streak, expected_next_at), wire `/hb/{id}/start`, `/hb/{id}/ok`, and `/hb/{id}/fail` to update those tables, audit every state transition, and emit per-cron freshness metrics to VictoriaMetrics so STAGE-002-006 can write rules against them.

This stage delivers the foundation the rest of the epic depends on. Vertical slice: a registered cron's first ping creates a `heartbeats_state` row, subsequent pings update it, the row is queryable via a new `/api/crons/{id}/heartbeat` endpoint, and `homelab_heartbeat_seconds_since_last_ok{cron_id="..."}` appears in VM within 30 seconds of the ping.

**Spec references:** ¬ß3.1 (heartbeat receiver), ¬ß6.1 (`crons`, `heartbeats_state` tables), ¬ß11.4 (`api_tokens` table ‚Äî already exists), ¬ß10.4 (audit log).

## Concrete deliverables

1. **Alembic migration** at `apps/monitor/alembic/versions/XXXX_stage_002_001_heartbeat_receiver.py`
   - ALTER TABLE `crons` to add (or confirm) columns:
     - `cadence_seconds INTEGER NOT NULL DEFAULT 0` (0 = "use schedule expression")
     - `expected_grace_seconds INTEGER NOT NULL DEFAULT 300`
     - `integration_mode TEXT NOT NULL DEFAULT 'observe'` (CHECK constraint: `observe`/`heartbeat`/`both`)
     - `enabled BOOLEAN NOT NULL DEFAULT 1`
     - `last_seen_state TEXT` (`ok`/`fail`/`stale`/`unknown` ‚Äî NULL until first observation)
   - ALTER TABLE `heartbeats_state` to add (or confirm) columns:
     - `cron_id INTEGER NOT NULL REFERENCES crons(id) ON DELETE CASCADE`
     - `current_state TEXT NOT NULL DEFAULT 'unknown'` (matches `last_seen_state` enum)
     - `last_ok_at TIMESTAMP` (UTC; NULL until first OK)
     - `last_fail_at TIMESTAMP` (UTC; NULL until first fail)
     - `last_start_at TIMESTAMP` (UTC; NULL until first start)
     - `current_streak INTEGER NOT NULL DEFAULT 0` (positive = OK streak; negative = fail streak)
     - `expected_next_at TIMESTAMP` (UTC; computed from cadence_seconds OR croniter from schedule)
     - `last_duration_seconds REAL` (NULL until a /ok arrives with duration; populated for runtime tracking)
   - UNIQUE INDEX on `heartbeats_state(cron_id)` (one row per cron).
   - Index on `crons(integration_mode, enabled)` for fast scan of "which crons should B-mode collector watch?"
   - Defensive: if STAGE-001-004's stubs already created any of these columns, the migration uses `op.add_column` only when absent (introspect via `inspect(bind).get_columns(...)`).

2. **Heartbeat receiver router replacement** at `apps/monitor/homelab_monitor/kernel/api/routers/heartbeat.py`
   - Replace the stub with three endpoints:
     - `POST /hb/{cron_id}/start` ‚Äî records `last_start_at = now_utc()`; sets `current_state` to `running` if currently `ok`/`unknown`, leaves `fail` alone; returns 204
     - `POST /hb/{cron_id}/ok` ‚Äî accepts optional `?duration=<seconds>` query param; records `last_ok_at`, increments `current_streak` (or resets from negative to +1), recomputes `expected_next_at`, sets `current_state = ok`, mirrors to `crons.last_seen_state`; returns 204
     - `POST /hb/{cron_id}/fail` ‚Äî records `last_fail_at`, decrements `current_streak`, sets `current_state = fail`, mirrors to `crons.last_seen_state`; returns 204
   - All three: 404 if `cron_id` not found in `crons` (per cross-stage criterion #9 ‚Äî no auto-create); 401 if no token (existing); 403 if token lacks `heartbeat:write` (existing).
   - All three: write to `audit_log` (who = token name, what = `heartbeat.{start|ok|fail}`, before/after = state JSON).
   - All three: structured log entry at `info` level with `cron_id`, `state_transition`, `token_name`.

3. **Heartbeat repository** at `apps/monitor/homelab_monitor/kernel/db/heartbeat_repo.py`
   - `class HeartbeatRepo` with methods:
     - `async def get_cron(cron_id: int) -> Cron | None`
     - `async def upsert_state(cron_id: int, **fields) -> HeartbeatsState` (atomic upsert)
     - `async def record_start(cron_id: int, ts: datetime) -> HeartbeatsState`
     - `async def record_ok(cron_id: int, ts: datetime, duration: float | None) -> HeartbeatsState`
     - `async def record_fail(cron_id: int, ts: datetime) -> HeartbeatsState`
   - All methods take a `connection` parameter; the router opens a transaction so a single heartbeat updates `heartbeats_state` + `crons.last_seen_state` + `audit_log` atomically.
   - `_compute_expected_next_at(cron, ok_at) -> datetime | None` helper; uses `cadence_seconds` if > 0, else parses `schedule` via croniter (in host-local TZ, returns UTC), else returns None.

4. **Heartbeat metrics emitter** at `apps/monitor/homelab_monitor/kernel/heartbeat/metrics.py`
   - Periodic task (registered with the scheduler from STAGE-001-007) that scans all enabled crons every 15s and writes:
     - `homelab_heartbeat_seconds_since_last_ok{cron_id, host, name}` (gauge; large value = stale)
     - `homelab_heartbeat_seconds_since_last_fail{cron_id, host, name}`
     - `homelab_heartbeat_current_streak{cron_id, host, name}` (signed integer)
     - `homelab_heartbeat_expected_next_seconds{cron_id, host, name}` (negative = overdue)
     - `homelab_heartbeat_last_duration_seconds{cron_id, host, name}`
   - Writes via the kernel's `vm-write` client (same path collectors use).
   - Emits zero metrics for `integration_mode=observe` crons (avoids polluting the metrics namespace before the user opts in).

5. **Pydantic schemas** at `apps/monitor/homelab_monitor/kernel/schemas/heartbeat.py`
   - `HeartbeatStateOut` (id, cron_id, current_state, last_ok_at, last_fail_at, last_start_at, current_streak, expected_next_at, last_duration_seconds)
   - `HeartbeatPostQuery` (optional `duration: float = Field(ge=0, le=86400)` ‚Äî 24h cap to reject typos like `?duration=999999999`)

6. **API endpoint to read heartbeat state** at `apps/monitor/homelab_monitor/kernel/api/routers/crons.py` (new file; STAGE-002-002 will expand it)
   - `GET /api/crons/{cron_id}/heartbeat` ‚Üí `HeartbeatStateOut` (session-auth, not token-auth ‚Äî this is dashboard-facing)
   - Returns 404 if cron doesn't exist.

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **`/hb/{id}/start` on unknown id: 404 vs auto-create the cron row?**
   - **Recommended: 404 (force explicit registration).** Auto-create is a footgun ‚Äî typos in cron commands become orphan registry entries, and "discovered through ping" loses the rich context (host, command, schedule) that auto-discovery provides. The trade-off: users have to add the cron to the registry first before pinging. That's fine for opt-in heartbeats since the user is already touching the dashboard.
   - Alternative: auto-create with `host="unknown"`, `command="unknown"`, `integration_mode="heartbeat"`. Recovers from "I added the curl ping before adding the cron in the dashboard" UX hiccups but creates registry pollution.
   - Decision matters for: STAGE-002-005 (wrapper helper UX ‚Äî does the dashboard create the cron row when the user clicks "Install heartbeat", or does the wrapper handle it?).

2. **`/ok?duration=Ns` query param: accept or always 200 OK with no body?**
   - **Recommended: accept `?duration=<seconds>`** as an optional query param. Healthchecks.io supports it (via POST body), and runtime tracking is genuinely useful for the digest's "cron heartbeat report" section (per spec ¬ß11). Capping at 86400s (24h) prevents pathological values.
   - Alternative: ignore duration entirely; rely on `last_ok_at - last_start_at` math when both arrive. Slightly cleaner data model, but requires the cron to call `/start` AND `/ok` (most users will skip `/start`).

3. **Token scoping: single global `heartbeat:write` vs per-cron token?**
   - **Recommended: single global `heartbeat:write` token, with per-cron rate-limiting.** Avoids token-explosion at the start. Per-cron tokens become useful when crons run on different hosts (compromise of one host shouldn't let an attacker spoof every cron's heartbeat) ‚Äî defer to a later stage.
   - Alternative: per-cron token field on the `crons` table; receiver checks `token.id == cron.token_id`. Stronger blast-radius isolation, more management overhead.

4. **`last_seen_state` mirror on `crons` table: keep or drop?**
   - **Recommended: keep the mirror.** Spec ¬ß6.1 explicitly lists `last_seen_state` on the `crons` table. Denormalization makes the Inventory list query a single SELECT vs JOIN. Update both tables in the same transaction.
   - Alternative: drop the mirror; always JOIN `heartbeats_state`. Cleaner schema, slightly slower list endpoint. Not worth the schema deviation from spec.

#### Cross-decision interactions noted
- **STAGE-002-005 wrapper UX:** D1=404 + D3=global-token means `hm cron install` must (a) register the cron via the STAGE-002-002 API, (b) write a crontab line that uses `$HM_HEARTBEAT_TOKEN` env var.
- **Audit log volume:** D1=404 means 404 responses do NOT write audit (no state change). Only successful state transitions audit. Combined with D3a (rate-limit) bounds spoofing rate if global token leaks.
- **Inventory query (STAGE-002-002):** D4=mirror means single SELECT on `crons` for the list view, no JOIN required.
- **Migration scope:** D2a (`last_exit_code`) + D4 (`last_seen_state`) bundle into one Alembic migration alongside the other behavioral columns enumerated in the stage spec.
