# STAGE-002-001: Heartbeat receiver + `crons`/`heartbeats_state` schema + audit

## Status: Complete

## Overview

Replace the auth-only heartbeat router stub from STAGE-001-021 with a real receiver that persists per-cron state. Land the migration that adds the behavioral columns to `crons` (cadence, grace, integration_mode) and `heartbeats_state` (current_state, last_ok_at, current_streak, expected_next_at), wire `/hb/{id}/start`, `/hb/{id}/ok`, and `/hb/{id}/fail` to update those tables, audit every state transition, and emit per-cron freshness metrics to VictoriaMetrics so STAGE-002-006 can write rules against them.

This stage delivers the foundation the rest of the epic depends on. Vertical slice: a registered cron's first ping creates a `heartbeats_state` row, subsequent pings update it, the row is queryable via a new `/api/crons/{id}/heartbeat` endpoint. Metric emission deferred to STAGE-002-006 (see Deferred decisions below).

**Spec references:** §3.1 (heartbeat receiver), §6.1 (`crons`, `heartbeats_state` tables), §11.4 (`api_tokens` table — already exists), §10.4 (audit log).

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **`/hb/{id}/start` on unknown id: 404 vs auto-create the cron row?**
   - **Recommended: 404 (force explicit registration).** Auto-create is a footgun — typos in cron commands become orphan registry entries, and "discovered through ping" loses the rich context (host, command, schedule) that auto-discovery provides. The trade-off: users have to add the cron to the registry first before pinging. That's fine for opt-in heartbeats since the user is already touching the dashboard.
   - Alternative: auto-create with `host="unknown"`, `command="unknown"`, `integration_mode="heartbeat"`. Recovers from "I added the curl ping before adding the cron in the dashboard" UX hiccups but creates registry pollution.
   - Decision matters for: STAGE-002-005 (wrapper helper UX — does the dashboard create the cron row when the user clicks "Install heartbeat", or does the wrapper handle it?).

2. **`/ok?duration=Ns` query param: accept or always 200 OK with no body?**
   - **Recommended: accept `?duration=<seconds>`** as an optional query param. Healthchecks.io supports it (via POST body), and runtime tracking is genuinely useful for the digest's "cron heartbeat report" section (per spec §11). Capping at 86400s (24h) prevents pathological values.
   - Alternative: ignore duration entirely; rely on `last_ok_at - last_start_at` math when both arrive. Slightly cleaner data model, but requires the cron to call `/start` AND `/ok` (most users will skip `/start`).

3. **Token scoping: single global `heartbeat:write` vs per-cron token?**
   - **Recommended: single global `heartbeat:write` token, with per-cron rate-limiting.** Avoids token-explosion at the start. Per-cron tokens become useful when crons run on different hosts (compromise of one host shouldn't let an attacker spoof every cron's heartbeat) — defer to a later stage.
   - Alternative: per-cron token field on the `crons` table; receiver checks `token.id == cron.token_id`. Stronger blast-radius isolation, more management overhead.

4. **`last_seen_state` mirror on `crons` table: keep or drop?**
   - **Recommended: keep the mirror.** Spec §6.1 explicitly lists `last_seen_state` on the `crons` table. Denormalization makes the Inventory list query a single SELECT vs JOIN. Update both tables in the same transaction.
   - Alternative: drop the mirror; always JOIN `heartbeats_state`. Cleaner schema, slightly slower list endpoint. Not worth the schema deviation from spec.

**Approach:** Backend-only stage. Per binding rule, no manual user prompt for desktop/mobile testing. e2e-tester ran the 4-scenario backend API validation template against a live FastAPI backend on port 19090 with a real SQLite DB.

**Bug found and fixed during Refinement:**
- **`extra='forbid'` not enforced for FastAPI query params via `Depends()`.** First e2e pass found T7 (`POST /api/hb/<id>/ok?foo=bar`) returned 204 instead of 422. FastAPI's `Depends()` on a Pydantic BaseModel injects individual `Query()` parameters bypassing `model_validate()`, so Pydantic's `extra='forbid'` semantics never fire.
- **Fix:** Added a custom `query_model()` dependency factory in `kernel/heartbeat/schemas.py` that calls `Model.model_validate(dict(request.query_params))` and re-raises `ValidationError` as `HTTPException(422)`. Router `/start`, `/ok`, `/fail` switched to `Depends(query_model(X))`. Added 3 unit tests (one per verb) that POST `?foo=bar` and assert 422.
- **Result:** Re-verified live (T7, T10, T11 all PASS post-fix). 1184 backend tests now (+3 vs Build's 1181). 100% coverage maintained.

**No frontend testing required** (no UI changes).

**No regressions introduced.** Pre-existing: none. New: none.

#### Notable findings during Refinement
- E2E uncovered an integration bug that unit tests missed: Pydantic `extra='forbid'` is silently inert when used via FastAPI's `Annotated[Model, Depends()]` on query params. The unit tests only validated specific value-range failures (negative duration, oversized exit_code) — they did not test the case "an unknown query param is sent". The 100% coverage gate didn't catch it because all the `extra='forbid'`-enforcing code paths were technically untested (the model lib's forbid mechanism never fired). Fix added explicit unit + e2e coverage.
