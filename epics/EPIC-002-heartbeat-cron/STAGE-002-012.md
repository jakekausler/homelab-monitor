# STAGE-002-012: wrapper rewrite — generic shared script, run-UUID capture, per-line tagging, Vector transform

## Status: Not Started

**(New stage. EPIC-002 re-opened on 2026-05-19 — see `docs/superpowers/specs/2026-05-19-cron-run-logs-design.md`. Second of five appended stages STAGE-002-011 … STAGE-002-015.)**

## Overview

Rewrite the cron heartbeat wrapper so that A-mode (wrapper-installed) crons capture their stdout/stderr with exact per-run attribution, and make the wrapper a **generic shared script** so all future wrapper upgrades are a single file replacement.

Today the wrapper `cron-with-heartbeat.sh.tmpl` bakes the cron's fingerprint into the script body at install time, so the shared file `/usr/local/bin/cron-with-heartbeat.sh` is per-cron-specific in practice and cannot be upgraded in place. This stage makes the script **byte-identical for every cron** (fingerprint passed as argument `$1`, URL base read from a config file), adds run-UUID generation, per-line UUID prefixing of captured output, explicit start/end boundary markers, and `run_id` on the heartbeat calls — and adds the Vector transform that parses the per-line marker into a `run_id` field in VictoriaLogs.

It also delivers the one-time migration path for the single existing wrapped cron via a `wrapper_format_version` marker and the existing wrapper-health badge.

**Vertical slice:** install the upgraded wrapper on a test cron, let it run; journald shows `HM_RUN_START` / `HM_RUN_END` markers and the command's output lines, all under `SYSLOG_IDENTIFIER=hmrun`, each output line prefixed `HM_RUN=<uuid> `; Vector parses the prefix into a `run_id` field; VictoriaLogs is queryable by `SYSLOG_IDENTIFIER:hmrun AND run_id:<uuid>`; the heartbeat receiver receives `run_id` on `/start` and `/ok|/fail` and (via STAGE-002-011) writes a `cron_runs` row.

**Spec references:** `docs/superpowers/specs/2026-05-19-cron-run-logs-design.md` §5 (wrapper rewrite), §3.2 (A-mode data flow), §3.3 (components), D5, D6, D7.

## Concrete deliverables

### 1. Generic wrapper template rewrite

`apps/monitor/homelab_monitor/data/cron-with-heartbeat.sh.tmpl` reworked:

- **Calling convention:** the installed crontab line becomes `cron-with-heartbeat.sh <fingerprint> -- <original command>` — fingerprint is positional argument `$1`, NOT baked into the script body.
- **Generic body:** the script body no longer substitutes `{{FINGERPRINT}}`. The shared script `/usr/local/bin/cron-with-heartbeat.sh` is **byte-identical for every cron**.
- **URL base from config:** the heartbeat URL base moves out of the script into a config file `/etc/homelab-monitor/wrapper.env` (alongside the existing `heartbeat.token`), containing `HEARTBEAT_URL_BASE=...`. The script reads it at runtime. The token file path remains a fixed constant as today.
- **`WRAPPER_FORMAT_VERSION`** — a version constant embedded in the template.
- **Per-run behavior:**
  1. Generate `RUN_ID` — `cat /proc/sys/kernel/random/uuid`, with `uuidgen` as a fallback (use absolute paths — cron has a minimal PATH).
  2. `logger --tag hmrun "HM_RUN_START fp=<fp> run=<RUN_ID>"` — a journald start boundary marker. The `--tag` is the **stable constant `hmrun`**, never a per-run value.
  3. `POST /api/hb/<fp>/start?run_id=<RUN_ID>`.
  4. Run the real command. Its combined stdout+stderr is processed two ways: (a) the **original, un-prefixed** output is echoed to the wrapper's own stdout/stderr (so a human running the cron by hand, or an existing `>> logfile` redirect, still sees normal output); (b) a copy where **each line is prefixed with `HM_RUN=<RUN_ID> `** is piped through `logger --tag hmrun` (stable identifier) → journald → Vector → VictoriaLogs.
  5. Capture the real command's exit code and the run duration.
  6. `logger --tag hmrun "HM_RUN_END fp=<fp> run=<RUN_ID> exit=<n> duration=<s>"` — a journald end boundary marker.
  7. `POST /api/hb/<fp>/ok|fail?run_id=<RUN_ID>&duration=<s>&exit_code=<n>` — `exit_code` is sent on **both** the `/ok` call (`0`, the successful run's exit code) and the `/fail` call (the actual non-zero code), not just `/fail`.
- **Heartbeat POSTs remain best-effort and bounded** (`curl --max-time 5`, output discarded, `|| true`) — a network failure NEVER blocks or alters the real command. Unchanged principle from the current wrapper.

### 2. Wrapper edge-case handling (mandatory)

- **Exit-code preservation** — the original command's exit code is ALWAYS preserved and returned as the wrapper's exit code. The `logger` pipe must NOT mask it: use a pipe-status-safe shell construct (capture the command's own status explicitly — e.g. via a named pipe / process-substitution-free POSIX technique, or `{ cmd; echo $? >fd; } | logger`). This is a known shell footgun and a specific test target (see Tests).
- **`logger` missing or failing** — if `logger` is unavailable or errors, the original output still reaches the wrapper's stdout/stderr and the run still completes. Capture degrades gracefully; the cron never breaks.
- **UUID generation failure** — if both UUID sources fail, the wrapper still runs the command (capture degrades; no `run_id` sent).

### 3. New Vector transform — `hmrun` parsing

`deploy/vector/vector.toml.template` — a new transform branch:

- Match journald lines with `SYSLOG_IDENTIFIER == "hmrun"`.
- Parse the `HM_RUN=<uuid>` prefix out of the message body into a structured field named `run_id`. Strip the `HM_RUN=<uuid> ` prefix from the message body so the stored line shows the original command output.
- `run_id` is a **regular VictoriaLogs field — NOT a stream field** (a stream field per unique run would explode stream cardinality; VictoriaLogs explicitly warns against this).
- The `HM_RUN_START` / `HM_RUN_END` boundary-marker lines also carry `SYSLOG_IDENTIFIER=hmrun` and flow through normally (they are just messages — they do not need special parsing beyond being attributed to the run window).
- Confirm the new branch does not collide with the existing cron-events branch (which filters `SYSLOG_IDENTIFIER == "CRON" || "crond"`).

### 4. Install-executor / `hm-cron-apply.sh` changes

- `_rewrite_line` emits the new `<fingerprint> --` prefix shape.
- `_build_wrapper_content` stops substituting `{{FINGERPRINT}}` and `{{HEARTBEAT_URL_BASE}}` into the script body.
- A new host operation writes `/etc/homelab-monitor/wrapper.env` with `HEARTBEAT_URL_BASE=...` (written once by the install / host-setup path). The file is written with mode `644` — it contains only a non-secret URL base, unlike the `heartbeat.token` file (`600`).
- The unwrap logic (from STAGE-002-009A) updates to strip the new `<fingerprint> --` prefix shape.
- STAGE-002-009A decisions hold: the shared wrapper script is never removed on unwrap; the shared token file is never touched.

### 5. `wrapper_format_version` column + format-outdated badge

- A new `crons` table column `wrapper_format_version`. STAGE-002-012 has no other migration, so this column gets its own alembic migration — `0016_crons_wrapper_format_version.py` (the next free number after `0015_cron_runs.py` from STAGE-002-011; confirm the head at implementation time). Set at install time to the wrapper's `WRAPPER_FORMAT_VERSION`.
- Extend the `WrapperHealth` Literal type in `apps/monitor/homelab_monitor/kernel/cron/schemas.py` (currently `Literal["ok", "stale", "unknown"]`) with a new member `"format_outdated"`.
- The `_compute_wrapper_health` logic in `kernel/api/routers/crons.py` returns `"format_outdated"` when the cron is wrapper-installed but its `wrapper_format_version` predates the run-log-capable format.
- The CronDetail wrapper-health badge (from STAGE-002-010) renders the new state: **"wrapper format outdated — re-install to enable run logs."**
- `openapi.json` regenerates (the `WrapperHealth` Literal change flows into `CronWithStateOut`).
- **No silent crontab edits** — the user re-installs each outdated cron via the existing one-click install UI. (Only one wrapped cron exists on the prod host today.)

## Locked Design Decisions

D-GENERIC — the wrapper becomes a generic shared script: fingerprint is positional argument `$1`; the URL base is read at runtime from `/etc/homelab-monitor/wrapper.env`. `/usr/local/bin/cron-with-heartbeat.sh` is byte-identical for every cron, so every future wrapper-logic upgrade is a single shared-file replacement with no per-cron re-install and no crontab edit. (Spec D5.)

D-CAPTURE — the wrapper generates a `RUN_ID` UUID per invocation, prefixes every captured output line with `HM_RUN=<RUN_ID> `, pipes the prefixed copy through `logger --tag hmrun` (a STABLE identifier, never per-run), and emits `HM_RUN_START` / `HM_RUN_END` boundary markers. The original un-prefixed output is still echoed to the wrapper's own stdout/stderr. (Spec D6.)

D-VECTORFIELD — a new Vector transform parses the `HM_RUN=<uuid>` per-line prefix into a **regular** VictoriaLogs field `run_id` (NOT a stream field — avoids stream-cardinality explosion). The A-mode run-log query is `SYSLOG_IDENTIFIER:hmrun AND run_id:<run_id>`. (Spec D6, §6.1, §14.)

D-EXITCODE — the original command's exit code is always preserved and returned through the `logger` pipe via a pipe-status-safe shell construct. `logger` missing/failing degrades capture but never breaks the cron. (Spec §5.3.)

D-RUNID-API — the wrapper threads `run_id` onto the `/start|/ok|/fail` heartbeat calls, and sends `exit_code` on **both** `/ok` (exit `0`) and `/fail` (the non-zero code) — not just `/fail` (the receiver side ships in STAGE-002-011; this stage makes the wrapper actually send it).

D-FORMATVER — legacy wrappers are migrated via a `wrapper_format_version` column on `crons` + a new `"format_outdated"` member on the `WrapperHealth` Literal + the existing wrapper-health badge. The user re-installs per cron via the one-click UI. NO silent crontab edits — consistent with the EPIC-002 "no cron modified without dashboard confirmation" policy. (Spec D7.)
