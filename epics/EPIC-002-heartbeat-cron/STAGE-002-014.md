# STAGE-002-014: run-history API + narrow run-log endpoint + anomaly heuristics v1

## Status: Complete

**(New stage. EPIC-002 re-opened on 2026-05-19 — see `docs/superpowers/specs/2026-05-19-cron-run-logs-design.md`. Fourth of five appended stages STAGE-002-011 … STAGE-002-015.)**

## Overview

Expose cron run history and run-log text over the API, and add anomaly detection v1.

STAGE-002-011 … 013 produce a fully-populated `cron_runs` table (run metadata + VL-derived line/byte counts + content digest). This stage adds: (1) a run-history list endpoint; (2) a narrow run-log endpoint that fetches a single run's stdout/stderr text from VictoriaLogs on demand; (3) a rule-based anomaly heuristic evaluator, wired into the reconciler's enrich phase, that flags runs deviating from a cron's norm.

**Vertical slice:** `GET /api/crons/{fingerprint}/runs` returns the cron's recent run records (no log text); `GET /api/crons/{fingerprint}/runs/{run_id}/log` returns that one run's stdout/stderr lines fetched live from VictoriaLogs; a run that took 4× its rolling-p95 duration carries `duration_outlier` in `anomaly_flags`.

**Spec references:** `docs/superpowers/specs/2026-05-19-cron-run-logs-design.md` §7 (anomaly detection v1), §8 (API), §6.2 (reconciler enrich phase), D11, D13.

## Concrete deliverables

### 1. `GET /api/crons/{fingerprint}/runs` — run-history list

Extends `apps/monitor/homelab_monitor/kernel/api/routers/crons.py`. Behind the **normal session auth** (alongside the existing `GET /api/crons/{fingerprint}`):

- **Query params:** `limit` (default 50, capped at a max), `cursor` (pagination), optional `state` filter (`running`/`ok`/`fail`/`unknown`).
- **Returns:** a list of run records — `run_id`, `state`, `started_at`, `ended_at`, `duration_seconds`, `exit_code`, `source`, `overlapping`, `line_count`, `byte_count`, `anomaly_flags`, and an `enriched` boolean. **No log text.**
- Pure SQLite read — never touches VictoriaLogs. Uses `CronRunRepository.list_runs` (defined in STAGE-002-011) over the `(cron_fingerprint, started_at DESC)` index.
- **Pagination cursor (carried-forward from STAGE-002-011 code review):** STAGE-002-011's `CronRunRepository.list_runs` ships with a single-column `started_at` cursor. When this stage wires `GET /runs` over HTTP, switch `list_runs` to a COMPOUND keyset cursor `(started_at, run_id)` so the tie-break column participates in the cursor predicate — otherwise two runs with an identical `started_at` straddling a page boundary can be dropped. A-mode `started_at` is microsecond-precision so collisions are rare, but the lost-`/start`-derived path can produce equal values.

### 2. `GET /api/crons/{fingerprint}/runs/{run_id}/log` — narrow run-log endpoint

The only VL-backed API this work exposes. Behind the **normal session auth**:

- The monitor loads the `cron_runs` row, builds the LogsQL query from the stored fields (A-mode: `SYSLOG_IDENTIFIER:hmrun AND run_id:<run_id>`; B-mode: the fingerprint heuristic) over `[vl_window_start, vl_window_end]`, calls `VictoriaLogsClient` server-side, and returns that run's lines + a `truncated` flag.
- The frontend NEVER sees LogsQL and NEVER talks to VictoriaLogs directly.
- **Four response shapes:**
  - Run completed, VL data present → `200` — log lines + `truncated` flag + `log_status: available`.
  - Run still `state=running` → `200` — output-so-far + `state: running`. For an in-flight run `vl_window_end` is NULL, so the endpoint substitutes `now()` as the query's upper time bound.
  - VL data aged out (the run's window is older than VictoriaLogs retention) → `200` — `log_status: expired` + the run metadata, no log text. **This is not an error.**
  - VL unreachable / timeout → `503` — a typed error the UI renders as "logs temporarily unavailable".

### 3. Anomaly heuristic evaluator (v1)

New module `apps/monitor/homelab_monitor/kernel/cron/run_anomaly.py`. Rule-based, computed entirely from `cron_runs` history for the same cron — no log-content analysis, no ML. Invoked by the `CronRunReconciler` enrich phase (STAGE-002-013): after the reconciler computes `line_count`/`byte_count`/`content_digest` for a run, it calls the evaluator, which returns the run's `anomaly_flags` (a comma-separated string written to the `cron_runs` row).

Rules — each tripped rule appends a flag:

| Flag | Rule |
| --- | --- |
| `duration_outlier` | `duration_seconds` exceeds `k ×` the rolling p95 of the last `N` completed runs of this cron (`k`, `N` env-configurable). |
| `exit_code_changed` | `exit_code` differs from this cron's recent dominant exit code. |
| `output_size_spike` | `line_count` (or `byte_count`) exceeds the rolling-median band for this cron. |
| `output_size_drop` | `line_count` (or `byte_count`) falls below the rolling-median band for this cron. |
| `unexpected_empty` | `line_count == 0` for a cron that normally produces output. |
| `new_failure` | `state == fail` when this cron's recent runs were all `ok`. |

**Min-history gate:** every rule needs baseline history to be meaningful — a rule does not fire until at least `min_history` completed runs exist for the cron (env-configurable). Anomaly flags therefore begin producing signal only once `cron_runs` has accumulated history. All flags are rule-derived and human-explainable; no black-box scoring.

`content_digest` is NOT consumed by any v1 rule — it remains a forward investment for EPIC-004 content clustering.

## Locked Design Decisions

D-AUTH — the run-history and run-log READ endpoints use the NORMAL session auth (alongside `GET /api/crons/{fingerprint}`). The wrapper's heartbeat `/start|/ok|/fail` calls (with `run_id`) stay on the `heartbeat:write` token path — that split is owned by STAGE-002-011 / 012. (Spec D13.)

D-NARROW — only a NARROW, run-specific run-log endpoint is exposed (`GET /runs/{run_id}/log`). It builds the LogsQL query server-side from the run's stored window; the frontend never sees LogsQL or talks to VL. The generic `/api/logs` proxy stays in EPIC-004. (Spec D10.)

D-LOGSTATUS — the run-log endpoint has four explicit response shapes: `available`, `running` (with `now()` substituted as the VL window upper bound), `expired` (aged-out logs — a `200`, NOT an error), and `503` (VL unreachable). (Spec §8.2, §10.)

D-ANOMALY — anomaly detection v1 is RULE-BASED HEURISTICS over `cron_runs` metadata only (`duration_outlier`, `exit_code_changed`, `output_size_spike`, `output_size_drop`, `unexpected_empty`, `new_failure`), each gated by a `min_history` threshold. No log-content analysis, no ML. Error-keyword scanning and Drain content clustering are explicitly EPIC-004 and must be backfilled to cron logs there. (Spec D11, D15.)

D-EVALUATOR-WIRING — the evaluator is invoked by the `CronRunReconciler` enrich phase; it writes `anomaly_flags` on the run row. The evaluator itself is independently unit-testable with synthetic history (so this stage can be developed/tested with synthetic data independent of STAGE-002-013's live reconciler).

| Decision | Resolution |

| `GET /api/crons/{fingerprint}/runs/{run_id}/log` response shape | A SINGLE `RunLogResponse` Pydantic model with a `log_status` discriminator field taking values `"available"` / `"running"` / `"expired"`.
