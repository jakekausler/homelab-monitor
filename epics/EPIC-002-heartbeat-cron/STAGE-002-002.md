# STAGE-002-002: Cron registry CRUD API + Inventory "Crons" tab UI

## Status: Build (Complete)

## Overview

Build the user-facing surface for the cron registry: a typed CRUD API for `crons` rows (list, get, create, update, soft-delete) plus a "Crons" tab in the Inventory screen that shows every known cron with last-seen state, expected next run, current streak, and integration mode. Lets the user manually add a cron the auto-discovery missed (e.g., a cron on a remote host that the local discoverer can't see), edit cadence/grace, flip integration mode between `observe`/`heartbeat`/`both`, and soft-delete crons that have been retired.

Vertical slice: from a logged-in browser, navigate to Inventory → Crons, see all rows from STAGE-002-001's seed data, click a row to open detail, edit the grace period to 600s, save, and see the updated row in the list. The cron's `last_seen_state` continues to update as heartbeats arrive (proves the STAGE-001 path still works through the new UI surface).

**Spec references:** §3.1 (UI screens — Inventory tab), §6.1 (`crons` table contract), §10.4 (audit log on edits), §10.5 (CSRF on mutating endpoints), §16 (timezone handling — display in local TZ).

## Concrete deliverables

1. **Expand the `crons` router** at `apps/monitor/homelab_monitor/kernel/api/routers/crons.py` (created in STAGE-002-001):
   - `GET /api/crons` → paginated list (default 100, max 500); query filters: `host`, `integration_mode`, `enabled`, `state` (filters on `last_seen_state`), `q` (substring on `command` or `name`)
   - `GET /api/crons/{cron_id}` → full row + joined `heartbeats_state`
   - `POST /api/crons` → create; body validates against `CronCreate` schema (host, command, schedule, name, cadence_seconds OR schedule, expected_grace_seconds, integration_mode); 409 if `(host, command)` already exists; CSRF required
   - `PATCH /api/crons/{cron_id}` → partial update; body validates against `CronUpdate`; 404 if missing; CSRF required; emits audit log entry with `before`/`after` JSON of changed fields only
   - `DELETE /api/crons/{cron_id}` → soft-delete (sets `deleted_at`); CSRF required; cascades to disable `heartbeats_state` row (does NOT delete it; preserves history)
   - All endpoints session-auth (not token-auth — this is dashboard-facing)

Validator: exactly one of `schedule` OR `cadence_seconds` must be set.

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **Cron schedule storage format: cron expression string vs structured (minute/hour/day/month/dow) parsed columns?**
   - **Recommended: cron expression string.** Matches what the user types in `crontab -e`. Structured columns are easier to query but lose round-trip fidelity (`@hourly`, `*/5`, `MON-FRI` all canonicalize lossy-ly). croniter parses the string at use-time; we don't query schedule fields directly anyway (filtering is on host/mode/state).
   - Alternative: parse to structured columns, render back to canonical string. Defensible if we expect a "show me all crons that run on weekends" query later, but no such query is in scope for EPIC-002 or near-term.

2. **Soft-delete vs hard-delete?**
   - **Recommended: soft-delete (`deleted_at` column).** Audit-log shows the delete; `heartbeats_state` row preserved for history; admin can un-delete via `?include_deleted=1` + PATCH. Only downside: query writers must remember the filter (added by default in the repo layer, opt-out via `include_deleted=True`).
   - Alternative: hard-delete with audit-log capture of the deleted row. Simpler queries, harder recovery.

3. **Audit entry granularity on PATCH: full-row diff vs changed-fields only?**
   - **Recommended: changed-fields only.** Reduces audit_log row size; the `before`/`after` JSONs only include the keys that changed. Reconstructing full state from the audit log requires walking back through entries, but that's acceptable — audit-log is for "what happened?" not "what was the state at time T?".
   - Alternative: full-row snapshots. Easier point-in-time reconstruction, but bloats audit_log fast (rows are wide).

4. **Detail view: drawer (slide-in from right) vs row expansion vs separate route page?**
   - **Recommended: separate route page at `/inventory/crons/{id}`.** Bookmarkable, deep-linkable from alerts ("alert on cron X" → click → land on detail page). Drawer is faster but breaks deep-linking. Row expansion looks cool but limits the detail UI to ~400px height.
   - Alternative: drawer with URL state preserved. More modern feel, more JS.
