# STAGE-002-009: Wrapper helpers — REWORKED to embed fingerprint at install time, call /register first, "Install heartbeat" UI button local-host only

## Status: Complete

**(Was STAGE-002-005. Renumbered after the cron-derived-state-redesign brainstorm on 2026-05-11.)**

## Overview

Ship the heartbeat wrapper infrastructure. After this stage, the user has three install paths:

1. **UI button "Install heartbeat" on the cron detail page (local host only).** The monitor writes the wrapper script to `/usr/local/bin/cron-with-heartbeat.sh`, rewrites the matching crontab line to invoke the wrapper, calls `/register` with `wrapper: true`. One click. **Disabled for any cron with `host != <monitor's own host>`** with a tooltip pointing at EPIC-017 (SSH probe framework) for the remote-host install variant.

2. **CLI `hm cron install-wrapper <fingerprint>` invoked on the monitor host.** Equivalent to the UI button; for users who prefer CLI. Available for crons on the monitor's host only.

3. **CLI `hm cron install-wrapper` distributed as a standalone binary** that runs on a remote host (Synology, NAS, etc.). The binary scans the local crontabs, lets the user pick one, computes the fingerprint, writes the wrapper locally, rewrites the crontab line, calls `/register` against the (configured) monitor URL with `wrapper: true`. The binary needs a `heartbeat:write` token + monitor URL (env var or CLI flag). Binary distribution itself is part of EPIC-019 release pipeline; this stage ships the source.

**Reworked from the original STAGE-002-005 plan in three ways:**

1. **Wrapper embeds fingerprint at install time** (was: cron_id from the registry). The install flow computes `_compute_fingerprint(host, source_path, schedule, command)` from the user's current crontab line and bakes it into the wrapper script.
2. **Install flow calls `/register` first**, then heartbeats work. Old flow assumed the row already existed; new flow creates it.
3. **UI "Install heartbeat" button is LOCAL-HOST ONLY.** For any cron with a different host, the button is disabled with a tooltip pointing at EPIC-017.

**Spec references:** `docs/superpowers/specs/2026-05-11-cron-derived-state-redesign.md` §Layered install paths, §Wrapper script knows its fingerprint via install-time bake.

## Concrete deliverables

1. **Wrapper script template** at `apps/monitor/homelab_monitor/data/cron-with-heartbeat.sh.tmpl`
   - POSIX-bash (no bashisms; runs under `dash`).
   - Template variables: `{{FINGERPRINT}}`, `{{HEARTBEAT_URL_BASE}}`, `{{TOKEN_FILE_PATH}}`.
   - Behavior:
     1. Record start time.
     2. `curl --max-time 5 -X POST -H "Authorization: Bearer $(cat $TOKEN_FILE)" "$HEARTBEAT_URL_BASE/api/hb/$FINGERPRINT/start"` (best-effort).
     3. Exec the real command (passed via `--` separator).
     4. On success: POST `/api/hb/$FINGERPRINT/ok?duration=N&exit_code=0`.
     5. On fail: POST `/api/hb/$FINGERPRINT/fail?duration=N&exit_code=$EC`.
     6. Always `exit $real_exit_code`.
   - Header comment: "DO NOT EDIT — generated by `hm cron install-wrapper` on YYYY-MM-DD; fingerprint=<fp>".

2. **Local-host install flow** at `apps/monitor/homelab_monitor/kernel/cron/installer.py`
   - `async def install_wrapper_local(fingerprint: str, *, token_id: int, who: str, ip: str | None) -> InstallResult`:
     1. Verify `host == local hostname`; raise `RemoteHostUnsupported` otherwise.
     2. Fetch the cron row from `crons` table.
     3. Compose the wrapper script from the template with `{{FINGERPRINT}}`, `{{HEARTBEAT_URL_BASE}} = config.public_url`, `{{TOKEN_FILE_PATH}} = /etc/homelab-monitor/heartbeat.token`.
     4. Write wrapper to `/usr/local/bin/cron-with-heartbeat.sh` (chmod 0755).
     5. Write token file (chmod 0600).
     6. Rewrite the crontab line: parse the source file at `source_path`, find the line whose hash matches the fingerprint, replace `<schedule> <command>` with `<schedule> /usr/local/bin/cron-with-heartbeat.sh -- <command>`.
     7. POST `/register` with `wrapper: true` (server-side; same fingerprint matches → idempotent upsert sets `wrapper_installed_at`).
     8. Audit verb `crons.wrapper_installed`.
   - Errors: rollback if any step fails — restore the original crontab line + delete the wrapper file.

3. **API endpoint** at `apps/monitor/homelab_monitor/kernel/api/routers/crons.py`
   - `POST /api/crons/{fingerprint}/install-wrapper`:
     - Body: `InstallWrapperRequest { token_id: int, confirm: bool }`. `confirm: true` required to actually modify the crontab.
     - For `confirm: false`: returns a **dry-run preview** with the diff of the crontab change + the wrapper script content. No file modifications.
     - For `confirm: true`: calls `install_wrapper_local`. Returns updated `CronOut` (with new `wrapper_installed_at`).
   - Session-auth + CSRF.
   - Returns 400 if the cron's host is not the local host.

4. **CLI subcommand** at `apps/monitor/homelab_monitor/cli/cron.py`
   - `hm cron install-wrapper <fingerprint>` — equivalent to the UI button. Same dry-run-then-confirm pattern.
   - `hm cron get-wrapper-template` — prints the wrapper template to stdout (useful for inspecting).

5. **Standalone remote-host CLI** at `apps/monitor/homelab_monitor/cli/install_wrapper_remote.py`
   - Runs on any Linux host with python3. Distributed as a single-file Python script (or a PyInstaller binary in EPIC-019).

   - Configuration via env: `HM_MONITOR_URL`, `HM_HEARTBEAT_TOKEN`. CLI flags override.
   - This is the install path for hosts the monitor cannot reach via UI button (Synology, NAS, foreign containers).

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **Wrapper template variables: shell expansion at install time vs at runtime?**
   - **Recommended: install-time expansion via Python templating.** Final wrapper script has hardcoded fingerprint + URL; no env-var dependency at runtime. Simpler runtime, no `set -u` surprises.
   - Alternative: runtime expansion via env vars. More flexible (one wrapper file shared across all crons), but introduces "what if env var missing" failure modes.

2. **Crontab rewrite strategy: in-place edit with `crontab -e` vs read + parse + replace + `crontab -` (replace whole file)?**
   - **Recommended: read + parse + replace + `crontab -`.** Predictable; can show exact diff in dry-run; no editor interaction.
   - Alternative: shell-out to `sed -i` — line-noise-prone and dangerous.

3. **Token file location: `/etc/homelab-monitor/heartbeat.token` vs `$HOME/.config/homelab-monitor/token`?**
   - **Recommended: `/etc/homelab-monitor/heartbeat.token` (chmod 0600, owned by root).** System-wide convention; all users' crons share the same token.
   - Alternative: per-user `$HOME/.config/...`. Useful for multi-tenant systems but homelab is single-user.

4. **Rollback on install failure: full restore vs partial-state-recorded-and-flagged?**
   - **Recommended: full rollback on any step failure.** Restore crontab to original; delete wrapper file; do NOT mark `wrapper_installed_at`. User sees the install failed cleanly.
   - Alternative: leave the partial state + alert user "wrapper installed but crontab rewrite failed." More flexible but confusing.

### Locked Design Decisions (2026-05-16)

**Brainstormer pressure-test overturned the stage file's core premise.** The stage file assumed a server-side installer running inside the monitor container would rewrite host crontabs. Investigation found the container has READ-ONLY bind-mounts of `/etc` and `/var/spool/cron/crontabs`, runs as a non-root UID, has no `crontab` binary, and cannot reach the host cron daemon (PID/mount/IPC namespace isolation). `crontab -e` / `crontab -` from the container is impossible regardless of permissions.

**Resolution — Option F (direct file writes via extended host-setup.sh):**
- The container writes crontab files DIRECTLY (Python atomic write+rename — no `crontab` binary). On Debian/Ubuntu the host cron daemon auto-reloads on inotify mtime change, so a direct file write is picked up automatically.
- Scope: user crontabs (`/var/spool/cron/crontabs/<user>`) AND system crontabs (`/etc/crontab`, `/etc/cron.d/*`). User decision: "this is a system monitor that will have access to root files."
- `scripts/host-setup.sh` extended to grant WRITE ACLs (`rw`) on `/var/spool/cron/crontabs/` (dir + per-file + default ACL) and on `/etc/crontab` + `/etc/cron.d/`. It currently grants read-only ACLs and does not touch `/etc` at all. Re-run host-setup.sh on the host (idempotent). The existing systemd `.path` watcher re-applies ACLs to newly created files.
- `deploy/compose/docker-compose.yml`: the `/etc` and crontab-spool mounts change from `:ro` to `:rw`.
- Net effect: the UI "Install heartbeat" button is genuinely one-click; the endpoint writes wrapper + token + rewrites crontab server-side. STAGE-007A's read-only ACL design becomes a read-write ACL design.
- Security note: a compromised monitor could inject cron entries into crontabs it has write ACL on — an accepted escalation; the monitor is already an auto-fix-capable service.

**Fingerprint convergence — Option D (discoverer unwraps):**
- Wrapping changes a crontab line's command text to `<wrapper-path> -- <original>`. Without mitigation, the discoverer recomputes a different fingerprint every 5 min, treats the wrapped cron as new, and STAGE-007A auto-soft-deletes the original.
- Fix: teach the cron discoverer/parser to detect the known wrapper prefix and compute the fingerprint from the INNER original command. Single source of truth (the crontab on disk), survives operator hand-edits.
- The wrapper template and the unwrap regex are COUPLED — tested as a wrap → discover → fingerprint-equality round-trip.
- Requires the inner command to be preserved byte-exact during crontab rewrite (no requoting / whitespace normalization).

**The four pre-documented decisions (brainstormer-validated):**
- D1 — Template expansion: INSTALL-TIME. Wrapper bakes in fingerprint + heartbeat URL + token-file path as constants.
- D2 — Crontab rewrite: read + parse + fingerprint-match + replace `<schedule> <command>` with `<schedule> <wrapper> -- <command>`, inner command byte-exact; atomic write+rename for all three file kinds (no `crontab -`).
- D3 — Token file: `/etc/homelab-monitor/heartbeat.token`, mode 0600, owned by the user whose crontab is wrapped (so a non-root user cron can read it).
- D4 — Rollback on install failure: FULL rollback — snapshot the original crontab before any change; on failure restore crontab + delete wrapper + delete token.

**Token model — auto-managed dual-row token (no UI token selection):**
- The heartbeat endpoints (`/api/hb/{fingerprint}/{start,ok,fail,register}`) require an API token with the `HEARTBEAT_WRITE` scope. API tokens are hashed-at-rest — plaintext is not recoverable from `api_tokens`.
- The install flow uses a single system-managed `HEARTBEAT_WRITE` token via a new `ensure_heartbeat_wrapper_token()` helper, mirroring STAGE-008's `ensure_cron_events_token()` dual-row pattern EXACTLY: `make_api_token()` → hash stored in `api_tokens` via `create_api_token`, plaintext stored encrypted (AES-256-GCM) in the `secrets` table via `secrets_repo.set()`. Idempotent.
- The installer reads the plaintext via `secrets_repo.get()` and writes it into the token file.
- One shared token reused across all wrapped crons — NOT per-cron token minting (still out of scope).
- The stage file's success criterion #6 "Step 1 (token selection)" is DROPPED. The install modal flow is: dry-run preview → confirmation checkbox → confirm.

**Local-host detection:**

- Expose a computed `is_local` boolean on `CronOut`. The current UI's `isRemote = source_path === null` test is a latent bug — fix it in this stage.

**Public URL (Build-phase addendum):** The wrapper script runs on the HOST (as a cron job), so it must POST heartbeats to a host-reachable monitor URL — the container-internal URL is unreachable from a host cron. There is no existing host-reachable-URL config. Decision (user-approved): introduce a NEW env var `HOMELAB_MONITOR_PUBLIC_URL` (e.g. `http://127.0.0.1:29090`), with NO silent default — wrapper install fails with a clear typed error (`PublicUrlNotConfiguredError`) if it is unset. The wrapper template's `{{HEARTBEAT_URL_BASE}}` is populated from it. Chosen over deriving the URL from the existing bind-host/port because it prepares cleanly for EPIC-017 remote wrapping (where the URL must be LAN-reachable, not `127.0.0.1`). `deploy/compose/.env` + `docker-compose.yml` gain the var; `docs/cron/install-heartbeat.md` documents it as a prerequisite.

**3b prod-rig redesign (2026-05-17) — split-by-source-kind install:** Refinement 3b against the real host surfaced TWO load-bearing flaws in the original Option F ("the container writes host crontabs directly"). (1) vixie-cron requires USER crontab files (`/var/spool/cron/crontabs/<user>`) to be EXACTLY mode `0600` with no ACL; the STAGE-009 `host-setup.sh` write-ACL on those files makes cron see `INSECURE MODE` and refuse to run the user's ENTIRE crontab. A non-root container (uid 995) fundamentally cannot write a `0600` file owned by another user without an ACL (→ breaks cron) or being root (→ violates the non-root container invariant from the auto-fix safety model). (2) the heartbeat token was written `0600 homelab-monitor`-owned, unreadable by the cron's actual run-as user. Decision (user-approved): adopt a **split-by-source-kind** install model.
- **System crontabs** (`/etc/crontab`, `/etc/cron.d/*`) — NO strict-0600 rule; the container writes them directly. The UI "Install" button is one-click / fully automated for these.
- **User crontabs** (`/var/spool/cron/crontabs/*`) — the install CANNOT be container-automated. The UI/endpoint GENERATES an install kit (wrapper script + the exact crontab-line diff + a copy-pasteable host command); the operator applies it on the host as the cron's owner via `hm cron install-wrapper <fingerprint> --confirm`, which uses the `crontab` mechanism to write a proper `0600 <user>` file. One operator step — this is forced by the OS security model, not a UX choice. The standalone `install_wrapper_remote.py` CLI already does exactly this operator-side write.
- **Token:** written `0644` (world-readable). It is a single narrow `HEARTBEAT_WRITE`-scoped credential on a single-user homelab host; `0644` lets any cron run-as user read it and removes the fragile in-container `fchown` (which fails because the container's `/etc/passwd` is not the host's). The per-cron `fchown` dance is dropped.
- **`host-setup.sh`:** the `rw` ACL grants on `/var/spool/cron/crontabs` (dir + per-file + default) are REMOVED — they are what broke the user's crontab on the prod rig. That path reverts to read-only ACLs (discovery still needs read). The `/etc/crontab` + `/etc/cron.d` write ACLs stay (system-crontab install remains container-driven). The systemd `.path` watcher reverts to maintaining `r` not `rw` on user crontabs.
- **`docker-compose.yml`:** the `/var/spool/cron/crontabs` mount reverts to `:ro`. `/etc:rw` and `/usr/local/bin:rw` stay (system-crontab + wrapper-script writes).
- **`install.py`:** `install_wrapper_local` serves only system-crontab sources with the full container write path; `crontab:<user>` sources are served by `build_install_kit` (the kit) only and raise a typed error if a container write is attempted. Token write switches to `0644`, the `fchown` is dropped.
- **The API endpoint:** for user-crontab crons, `confirm:true` returns the kit + the host command rather than performing a write; for system-crontab crons, `confirm:true` writes as before.
- **The UI `InstallHeartbeatModal`:** three-way state — system-local one-click install / user-local operator-kit (shows the wrapper + crontab diff + the copy-paste host command) / remote (disabled, EPIC-017).
- **STAGE-009A (wrapper removal)** is affected symmetrically: user-crontab uninstall must also be operator-applied. Fold the split-by-source-kind model into STAGE-009A's Design.
This is real-rig evidence overturning the originally-locked Option F for the user-crontab case; the container-writes path is retained only where it is provably safe (system crontabs).

**Host-side cron-apply executor (2026-05-17, supersedes the split-by-source-kind operator-applied decision above):** After exploring whether user-crontab install could be fully automated, the design is revised again. The "operator runs a host command for user crontabs" decision is REPLACED by a fully-automated host-side executor (architecture 2a). User-approved decisions:
- **The container stays READ-ONLY toward all crontabs.** The monitor container (uid 995) only reads crontabs (discovery) and GENERATES the install request. It never writes a crontab. This is a hard privilege boundary — a compromised monitor cannot write crontabs directly.
- **A host-side executor performs all crontab writes.** New components, installed on the host by `scripts/host-setup.sh`: (1) a small audited apply script `hm-cron-apply` at `/usr/local/sbin/`; (2) a systemd `.path` unit watching a request directory; (3) a systemd `oneshot` `.service` that runs the apply script. This mirrors the EXISTING `homelab-monitor-crontab-acl.path`/`.service` pattern already in the repo.
- **Executor identity = a hardened root systemd oneshot service (decision A).** No separate user account. The `cron-apply.service` runs as root but is locked down with systemd hardening directives (`NoNewPrivileges`, `ProtectSystem`, `ReadWritePaths=` limited to the crontab spool + `/etc/cron.d` + the request/result dirs, etc.). The hardened unit IS the confined identity.
- **Request/result IPC over a bind-mounted directory.** A host dir (e.g. `/var/lib/homelab-monitor/cron-apply/` with `requests/` + `results/` subdirs) is bind-mounted into the container. The install endpoint writes a request JSON (`{id, target_crontab, old_line, new_line/derivation, wrapper_script_content, token_content}`); the executor processes it and writes a result JSON (`{id, status, message}`); the endpoint polls the result with a timeout so the install stays synchronous from the UI's perspective. The container has write on `requests/` + read on `results/`; the executor the reverse.
- **The apply script's hard constraint (the load-bearing security control):** the executor will ONLY wrap an already-present crontab line — it prepends the known wrapper invocation prefix to an existing line and refuses anything else. It MUST: only touch `/var/spool/cron/crontabs/*` and `/etc/cron.d/*` / `/etc/crontab`; verify `old_line` actually exists in the target crontab; re-derive the wrapped line itself from `old_line` rather than trusting a caller-supplied `new_line`; reject any request that is not "wrap this existing line". Result: even a fully-compromised monitor can at most cause an already-discovered cron to become wrapped — it cannot inject new crons or run arbitrary commands.
- **Uniform routing (decision):** ALL crontab writes — user crontabs AND system crontabs (`/etc/crontab`, `/etc/cron.d/*`) — route through the host-side executor. One code path, one threat model. The container no longer writes ANY crontab directly. (The wrapper-script write to `/usr/local/bin` and the token write to `/etc/homelab-monitor` — those are NOT crontab writes; decide during planning whether they also move to the executor or stay as direct container writes via the existing `:rw` mounts. The crontab WRITE specifically is what must go through the executor.)
- **vixie-cron 0600 correctness:** because the executor runs as root and uses the proper `crontab` mechanism (or writes the file as the correct owner + exactly 0600, no ACL), user crontabs stay valid — no INSECURE MODE. The broken `host-setup.sh` `rw` ACLs on `/var/spool/cron/crontabs` are REMOVED (that path reverts to read-only for the monitor; discovery still reads it).
- **Token:** `0644` (world-readable), as decided in the prior addendum — unchanged.
- **UI:** stays one-click for ALL local crons (system and user). The async request/poll round-trip is invisible to the operator. Remote crons remain disabled (EPIC-017).
- **Scope:** this host-side executor is built WITHIN STAGE-009 (not a separate stage). STAGE-009 absorbs it.
- **STAGE-009A (wrapper removal)** uses the same executor channel — the apply script also handles the unwrap operation.

This supersedes the "operator-applied for user crontabs" part of the previous addendum: user-crontab install IS now fully automated, via the host-side executor. The split-by-source-kind framing collapses — ALL crontab writes go through the one executor path.

**Full uniform routing — container has zero host-write (2026-05-17, refines the executor decision above):** The host-side executor handles not just crontab writes but ALL privileged host writes. The wrapper-script write (`/usr/local/bin/cron-with-heartbeat.sh`) and the token write (`/etc/homelab-monitor/heartbeat.token`) ALSO route through the executor — NOT direct container writes. User-approved decisions:
- The `hm-cron-apply` apply script handles THREE write operation types (plus unwrap for STAGE-009A): `wrap-crontab` (rewrite a crontab line — re-derived from old_line, the load-bearing-validated operation), `write-wrapper-script` (write `/usr/local/bin/cron-with-heartbeat.sh` 0755), `write-token` (write `/etc/homelab-monitor/heartbeat.token` 0644). The request JSON carries an `operation` field selecting one. The wrapper-script and token operations write to FIXED destination paths only — the apply script validates the destination is exactly the known wrapper path / token path and refuses any other.
- The monitor container has **ZERO host-write capability**. Its ONLY write into the host is the cron-apply IPC `requests/` directory (a request file). Every actual privileged host write — crontab, wrapper script, token — is performed by the hardened root executor service. A compromised monitor can at most drop request files, all of which the executor hard-validates.
- `docker-compose.yml`: the `/etc` mount reverts to `:ro` (discovery reads only — the container never writes under `/etc`). The `/usr/local/bin:rw` mount is REMOVED entirely (the container no longer writes the wrapper script directly). The crontab spool mount stays `:ro`. The ONLY `:rw` mount toward the host is the cron-apply IPC directory.
- Future note: if a genuine direct `/etc` write need ever arises, the `:rw` mount must be scoped to the specific subdirectory needed — never all of `/etc`.
- This supersedes the earlier "keep wrapper-script + token as direct container writes" planning default (spec open question O1) — the user chose full uniform routing for a single, consistent threat model: one audited executor performs every privileged host write; the container is read-only toward the host except for the IPC request channel.

**Spec open questions resolved (2026-05-17):** O1 → full uniform routing (above — token + wrapper-script go through the executor). O2 → moot (the `/etc` mount is now `:ro`; no `/etc` write at all). O3 → accepted: the e2e/integration test drives the apply script in a poll loop (no systemd in CI); real systemd `.path`/`.service` wiring is validated in 3b prod-rig. O4 → accepted: `jq` is a required host dependency for the apply script; `host-setup.sh` checks for it and fails loud if absent. O5 → accepted: CRLF crontab line endings need no mitigation (Debian host, `crontab -e` always writes `\n`).

**Cron-discovery user-crontab read fix — Option B crontab snapshot (2026-05-17):** STAGE-009's 3b prod-rig testing surfaced a defect that PREDATES STAGE-009 — it is a STAGE-007 (cron discovery) flaw, found only because 3b was the first time anyone read vixie-cron's own logs. Root conflict: the monitor container (non-root, uid 995) must READ host user crontab files (`/var/spool/cron/crontabs/<user>`) for discovery; `host-setup.sh` granted a per-file read ACL to allow that; but vixie-cron rejects ANY user crontab carrying an ACL (the `+` flag) as `INSECURE MODE` and refuses to run that user's entire crontab. A `0600` file owned by another user is, by Unix design, unreadable by a non-root process — "cron-valid" and "container-readable via ACL" are mutually exclusive. STAGE-007A's `refresh-crontab-acl.sh` + `homelab-monitor-crontab-acl.{path,service}` watcher actively RE-APPLIED the breaking ACL after every `crontab -e`. Net effect: user crontabs on any host where `host-setup.sh` ran have been silently rejected by cron since STAGE-007 (commit cfa9d14, 2026-05-14).

User-approved fix — **Option B (crontab snapshot), implemented WITHIN STAGE-009**:
- A root-side "crontab snapshot" mechanism: a script runs `crontab -l -u <user>` for each user (the cron-sanctioned read path — never touches file modes) and writes the output into a container-readable snapshot directory (root-owned, world-readable; crontab schedules are not secrets, and discovery already secret-scrubs commands).
- The cron discoverer's user-crontab branch reads the SNAPSHOT directory instead of the real `0600` spool files. The real `/var/spool/cron/crontabs/<user>` files keep their `0600` no-ACL mode — cron stays healthy permanently, regardless of future edits.
- The snapshot is regenerated by BOTH (a) a systemd `.timer` (periodic, ~300s — aligned to the discovery poll) AND (b) a systemd `.path` unit watching `/var/spool/cron/crontabs` — so a `crontab -e` regenerates the snapshot near-instantly. Decision B3: timer + path-watcher. The manual "Discover now" / refresh button is UNCHANGED — it triggers discovery as today; discovery reads the (always-fresh) snapshot; no IPC round-trip on the refresh path.
- **Retired:** the existing `homelab-monitor-crontab-acl.{path,service}` units and `scripts/refresh-crontab-acl.sh` become obsolete (their only job was re-applying the now-eliminated ACL) and are DELETED. The new `homelab-monitor-crontab-snapshot.{path,service,timer}` units take over watching `/var/spool/cron/crontabs` (a directory can have only one `.path`→`.service` mapping — clean swap). `host-setup.sh`: REMOVE the per-file user-crontab read ACL grants (sections 3b/3c) and the `crontab-acl` unit install; ADD the snapshot script + the snapshot units. The directory traversal ACL is no longer needed (the root snapshotter lists users).
- `/etc/crontab` + `/etc/cron.d/*` discovery is UNAFFECTED — those are `0644` (root-owned, world-readable); the container reads them directly without any ACL. The snapshot mechanism is ONLY for the `0600` user-crontab spool.
- New: a host-side snapshot script, the 3 systemd units, the discoverer change (read snapshot dir for user crontabs — likely a new `HM_CRON_SNAPSHOT_DIR` env var), a `docker-compose.yml` bind-mount for the snapshot dir, host-setup.sh changes, tests.
- This is a STAGE-007-attributed defect but, per user decision, fixed within STAGE-009 (not a separate STAGE-007B) — STAGE-009's 3b cannot honestly complete on a host where discovery breaks crons, so the fix is done in-flight.
