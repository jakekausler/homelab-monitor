# STAGE-002-003: Cron auto-discovery + suggestion queue integration

## Status: Not Started

## Overview

Stand up a `cron-discoverer` plugin that scans the host's crontab files and emits "found new cron" suggestions into the suggestion queue. Sources scanned: `/etc/crontab`, `/etc/cron.d/*`, `/etc/cron.{daily,hourly,weekly,monthly}/*`, root + per-user crontabs (via `crontab -l -u <user>`). Each discovered cron becomes a `suggestion` row that the user accepts (creates a `crons` row with `integration_mode=observe`), customizes (opens the add-cron modal pre-filled), or ignores (archived; not re-suggested).

After this stage, the user's four known crons (`/storage/scripts/startup/startup.sh @reboot`, `/storage/scripts/rtlamr-watchdog.sh 17 * * * *`, `/etc/cron.d/certbot 0 12 * * *`, `/storage/scripts/cron/backup.sh 10 4 * * *`) appear as pending suggestions in the dashboard within one collector cycle of stage deployment. Accept all four → registry populated with all real crons.

**Spec references:** §3.1 (discovery engine), §3.1 (suggestion engine), §6.1 (`suggestions` table), §10.1 (Q27 — discovery sources include cron files).

## Concrete deliverables

1. **Migration** at `apps/monitor/alembic/versions/XXXX_stage_002_003_discovery_fingerprint.py`
   - Confirm `suggestions` table exists (created by STAGE-001-004 stub).
   - ADD COLUMN `suggestions.fingerprint TEXT NOT NULL` with UNIQUE INDEX on `(kind, fingerprint)` (so re-scans don't duplicate suggestions; idempotency).

2. **Cron discoverer plugin** at `apps/monitor/homelab_monitor/plugins/discoverers/cron_discoverer.py`
   - In-process Python plugin registered as `cron-discoverer` (collector kind: ASYNC; interval: 300s; concurrency_group: `discovery`).
   - Scan logic:
     - Read `/etc/crontab` (host fileystem; mounted in container at `/host/etc/crontab` — bind-mount config in `docker-compose.yml`)
     - Read all files under `/host/etc/cron.d/`, `/host/etc/cron.{daily,hourly,weekly,monthly}/`
     - Read root + per-user crontabs by execing `crontab -l -u <user>` for every user with UID >= 1000 (and root). Since the monitor runs as uid 1000 in a container, this requires either bind-mounting `/var/spool/cron/crontabs/` read-only OR a host-side daemon — pick during Design.
   - Parsing logic:
     - Identifies lines that match `<schedule> <command>` pattern (cron format) or `<schedule> <user> <command>` (system crontab format) — handles both
     - Extracts: schedule, command, user (if present), source file path
     - Skips lines that are comments or empty
     - Skips lines whose parsed schedule fails `croniter.croniter.is_valid` (logs a warning; doesn't suggest)
   - Fingerprint:
     - `sha256(f"{host}|{source_path}|{schedule}|{command}".encode()).hexdigest()[:16]`
     - Same fingerprint → existing suggestion is touched (`last_seen_at` bumped); no duplicate insert
     - On accept/ignore, the suggestion's status is updated but the fingerprint is preserved so the discoverer doesn't re-suggest
   - Output:
     - Emits one `homelab_collector_run_*` per scan
     - Emits `homelab_discovery_crons_found{host}` gauge (current count of pending cron suggestions)
   - The discoverer does NOT directly create `crons` rows — that's an explicit user action via the suggestions UI.

3. **Suggestion handlers** at `apps/monitor/homelab_monitor/kernel/api/routers/suggestions.py` (new file)
   - `GET /api/suggestions` → paginated list; filters by `kind` (e.g., `cron`), `status` (`pending`/`accepted`/`ignored`/`customized`)
   - `POST /api/suggestions/{id}/accept` → for `kind=cron`, creates the corresponding `crons` row with default `integration_mode=observe`; sets suggestion status `accepted`; CSRF required; returns the created `CronOut`
   - `POST /api/suggestions/{id}/ignore` → sets status `ignored`; CSRF required
   - `POST /api/suggestions/{id}/customize` → returns the suggested payload (`{host, command, schedule, user, source_path}`) for the UI to populate the add-cron modal; does NOT mutate the suggestion (final state set when the user submits the modal — which calls accept with overrides)

## Decisions in this stage

These need brainstormer-Opus input during Design phase. Pre-recommended position in **bold**.

1. **Discovery cadence: periodic poll (every 5min) vs event-driven via inotify on `/etc/cron.d/`?**
   - **Recommended: periodic poll at 300s.** Simpler; cron file changes are infrequent; inotify in a container requires `--privileged` or capabilities tweaks. The 5-minute lag is acceptable for "I added a new cron, when does it show up?" UX.
   - Alternative: inotify watcher daemon. Faster reaction but invasive container config. Defer until users complain.

2. **`@reboot` job tracking: track via "should have been seen since last boot" or skip entirely?**
   - **Recommended: track, via a self-monitor `last_boot_at` UTC timestamp + grace.** When a `@reboot` cron is in heartbeat mode, the heartbeat-receiver's "stale" check compares `heartbeats_state.last_ok_at < last_boot_at` and considers that stale. When in observe mode, just record the suggestion and never re-suggest.
   - Alternative: skip — `@reboot` crons can't be reliably heartbeated without a heartbeat early in the boot sequence. Pragmatic, leaves a gap in coverage.
   - Decision matters for: STAGE-002-006 (vmalert rule for "@reboot stale" requires the `last_boot_at` source — comes from where?).

3. **systemd timer support: include in this stage or defer?**
   - **Recommended: defer.** Different parsing path (no cron syntax; uses systemd OnCalendar syntax); separate "list-timers" exec; different "discovered_by" plugin. Adding it here doubles the scope. The user has no known systemd timers; pure speculation.
   - Alternative: ship together since they're "the same concept conceptually." Doubles test surface; delays the cron-only path.

4. **Per-user crontab access: bind-mount `/var/spool/cron:ro` vs host-side daemon vs require nsenter?**
   - **Recommended: bind-mount `/var/spool/cron/crontabs:ro` (Debian/Ubuntu layout) and `/var/spool/cron:ro` (RHEL/CentOS) under `/host/var/spool/cron`.** Container reads files directly. Requires `docker-compose.yml` change + `docs/deploy/host-prerequisites.md` note. Compatible with non-root container user since the files are world-readable for the owning user, but we may need to relax permissions OR mount as a tmpfs-overlay with relaxed perms — Design phase nails this.
   - Alternative: host-side daemon that emits parsed crontabs over a unix socket. Overkill for v1.
   - Alternative: exec `crontab -l -u <user>` via `nsenter` from inside container. Requires privileged container; pass.
