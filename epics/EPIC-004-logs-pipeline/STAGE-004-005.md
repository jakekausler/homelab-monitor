# STAGE-004-005: Cron fingerprint enrichment (vector hmrun transform + parsed cron_fingerprint field)

## Status: Complete

## Overview

Today's vector `hmrun` source parses `HM_RUN=<uuid>` into a `.run_id` field but does NOT extract or carry the cron job's stable `fingerprint`. Without fingerprint at the per-line level, Drain clustering (STAGE-004-025+) would either:

- Lump ALL cron jobs into one Drain model under `service="hmrun"` (losing per-job signature differentiation), OR
- Need a separate post-fetch lookup step at Drain-time (more complex)

This stage enriches the hmrun pipeline so each cron output line carries `cron_fingerprint` as a top-level VL field. Drain's model-key override hook (built in STAGE-004-025) will use it to key per-fingerprint models.

The fingerprint is already known at run start (it's how we identify the cron job; computed from the cron line text per STAGE-002-*). We need to thread it through `hmrun` into the systemd journal so vector's existing journald-filtered source picks it up.

**Vertical slice:** every line emitted by `hmrun` carries `cron_fingerprint:<hash>` in VL. Queries like `cron_fingerprint:<X>` return only that cron job's lines. Drain (when built) can key models per fingerprint.

**Spec references:** Master spec §6.3 (cron capture).

## Concrete deliverables

### 1. `hmrun` wrapper enriches log output

`apps/monitor/homelab_monitor/cli/hmrun.py` (or wherever the wrapper lives) — when it invokes the user's cron command and pipes stdout/stderr, prefix lines with structured fields including:

- `HM_RUN=<uuid>` (already today)
- `HM_FP=<fingerprint>` (new — the stable fingerprint of this cron job)

OR, alternatively, emit the fingerprint once at run-start via a `logger -p user.info "HM_RUN=<uuid> HM_FP=<fp> START"` line and rely on vector's `add_labels` to remember it for subsequent lines (more complex; Design phase chooses).

Recommendation: per-line prefix is simpler; vector's parser extracts both fields with a single regex.

### 2. Vector hmrun source parser update

`deploy/vector/vector.toml.template` — update the `cron_parsed` (or equivalent) transform to extract `HM_FP=<value>` into `.cron_fingerprint`:

```toml
[transforms.hmrun_parsed]
type = "remap"
inputs = ["hmrun_journald_filter"]
source = '''
parsed = parse_regex(.message, r'^HM_RUN=(?P<run_id>[a-f0-9-]+) HM_FP=(?P<fingerprint>[a-f0-9]+) (?P<message>.*)$') ?? {}
.run_id = parsed.run_id
.cron_fingerprint = parsed.fingerprint
.message = if exists(parsed.message) { parsed.message } else { .message }
'''
```

(Final VRL syntax confirmed in Design phase.)

### 3. Existing cron run log endpoint adapts to new field

`apps/monitor/homelab_monitor/kernel/api/routers/crons.py` — the per-run log endpoint already queries VL by `run_id`; no change needed unless we want to validate that fingerprint matches the expected one. Design phase decides.

### 4. Integration test

`apps/monitor/tests/integration/test_hmrun_fingerprint_enrichment.py` — spawn `hmrun` against a known fingerprint, emit lines, query VL, assert lines carry both `run_id` AND `cron_fingerprint`.

### 5. Backward-compatibility check

Existing in-flight cron runs (those started before this stage deploys) won't have `cron_fingerprint` field. Code paths that consume the field MUST handle the absent case (treat as `unknown` or fall back to `service="hmrun"`).

## Decisions in this stage

### Design phase will confirm

- **Per-line prefix vs single start-of-run emission** — recommendation: per-line prefix; vector parses each line independently. Simpler, robust to systemd line boundaries.
- **Whether to also include `cron_name` or other human-readable fields** in the prefix — recommendation: no, fingerprint is enough; UI looks up name from cron registry. YAGNI for additional fields.

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-CRON-FINGERPRINT-PRE-DRAIN** — fingerprint enrichment lands BEFORE Drain (this stage is a precursor to STAGE-004-025).
- **D-FINGERPRINT-PER-LINE** — each cron output line carries `cron_fingerprint` independently (not just one start-of-run emission).
- **D-DRAIN-KEY-PER-CRON-FINGERPRINT** — Drain model key for `service="hmrun"` is `cron:<fingerprint>` (override hook applied by STAGE-004-025).
- **D-ENRICHMENT-IS-ADDITIVE** — existing run_id-based lookups continue to work unchanged.

## Design Decision (locked 2026-05-29)

### Selected approach: Option B — journald structured field via `logger --journald`

**Decision:** Thread the already-known fingerprint into each hmrun log line as a journald STRUCTURED FIELD (not a message-text prefix), so Vector extracts it regex-free and the human-readable message stays clean at every layer.

**Wrapper change** (`apps/monitor/homelab_monitor/data/cron-with-heartbeat.sh.tmpl`):
- The per-line emission currently runs `printf 'HM_RUN=%s %s\n' "$RUN_ID" "$_line" | logger --tag hmrun`.
- Change to emit journald fields per line via `logger --journald`, supplying:
  - `SYSLOG_IDENTIFIER=hmrun` (REQUIRED — `--journald` ignores `--tag`, and `hmrun_filter` matches on `.SYSLOG_IDENTIFIER == "hmrun"`)
  - `PRIORITY=<n>` (REQUIRED — `--journald` ignores `-p`/`--priority`; preserve current severity behavior)
  - `HM_RUN=<uuid>`
  - `HM_FP=<fingerprint>` (the full 64-char SHA256 hex from `$FINGERPRINT`)
  - `MESSAGE=<clean original line>` (no prefix)
- Preserve the wrapper's safety invariant: logger failures must NEVER break the cron (`|| true` best-effort), exactly as today.

**Vector change** (`deploy/vector/vector.toml.template`, `hmrun_shaped` transform):
- Set `.cron_fingerprint = .HM_FP` and `.run_id = .HM_RUN` directly from the surfaced journald fields, each `exists()`-guarded. No regex needed for the structured-field path.
- **D-ENRICHMENT-IS-ADDITIVE:** keep `run_id` extraction independent of fingerprint extraction (a missing `HM_FP` must NOT prevent `run_id` assignment, and vice versa).
- **Transitional fallback:** RETAIN the existing `HM_RUN=<uuid> `-prefix regex branch so in-flight / legacy journald lines emitted before this change still parse `.run_id` (and `.message`) correctly during rollout.

**Secondary decisions (locked):**
- **D-FINGERPRINT-FULL-64CHAR** — emit the FULL 64-char fingerprint on the wire; do NOT truncate. The full value is the Drain model key (`cron:<fingerprint>`), the deep-link filter value (STAGE-004-021), and the `cron_runs.cron_fingerprint` column — truncation would force reconstruction-by-lookup everywhere, defeating D-FINGERPRINT-PER-LINE.
- **D-FINGERPRINT-FIELD-ONLY** — this stage adds ONLY `cron_fingerprint` (plus the already-present `run_id`). No command/name snippet per line; the command is recoverable via the fp→cron mapping in SQLite, and a per-line snippet would re-introduce byte cost and variable-length field bloat. Human-readable labels, if wanted later, are derived at display time.

**Rejected:** Option A (per-line text prefix `HM_RUN=... HM_FP=<64hex> ...`) — pollutes the raw journald MESSAGE with a 64-char prefix and makes run_id extraction contingent on the fp regex. Kept as the documented FALLBACK if 3a/3b Refinement surfaces a `--journald` wart (e.g. multi-line MESSAGE handling). Option C (Vector-side stateful carry-forward via `reduce` from the boundary marker) — rejected as fragile under reordered/dropped/interleaved journald streams.

**Look-Ahead confirmation:** `cron_fingerprint` MUST be a queryable VL field. Confirmed consumers: STAGE-004-025 (Drain model key `cron:<fingerprint>`), STAGE-004-016 (Field Inspector display), STAGE-004-021 (deep-link filter `cron_fingerprint:"<fp>"`), STAGE-004-034 (cron failure log correlation filter).

**Refinement plan:** 3a (dev rig, synthetic cron trigger → query `cron_fingerprint:* AND service:hmrun` in dev VL, confirm fp matches cron_runs) + 3b (prod rig, real cron trigger → confirm cross-correlation against real cron_runs row). Both required — host-integration stage.

### Refinement finding + fix: boundary markers missed the fingerprint (2026-05-29)

**Symptom (caught during Refinement against real rtlamr-watchdog cron):** Some real hmrun lines reached VictoriaLogs with EMPTY `cron_fingerprint`. Initial hypothesis (a syslog Vector source) was WRONG — there is no syslog source; every VL hmrun record comes from `[sources.journald]`. The true discriminator was the journald `_TRANSPORT` field: the regular output-capture lines (emitted via `logger --journald`) had `_TRANSPORT=journal` with the `HM_FP` structured field and got `cron_fingerprint` populated, but the **HM_RUN_START / HM_RUN_END boundary markers** were still emitted via `logger --tag hmrun` (plain text → `_TRANSPORT=syslog`, no structured fields) so they reached VL with empty `cron_fingerprint`. This violated D-FINGERPRINT-PER-LINE (boundary lines are hmrun lines too).

**Root cause:** Build left the wrapper's `log_marker()` helper on the old `logger --tag hmrun` path (the Build's Design Decision had said "boundary markers left unchanged — additive"; that was wrong against D-FINGERPRINT-PER-LINE).

**Fix (repo-config only, no host journald.conf change):**
- `apps/monitor/homelab_monitor/data/cron-with-heartbeat.sh.tmpl` — `log_marker()` now emits structured journald fields via `logger --journald` (`SYSLOG_IDENTIFIER=hmrun`, `PRIORITY=$PRIO`, `HM_RUN=$RUN_ID`, `HM_FP=$FINGERPRINT`, `MESSAGE=$1`), same as the output-capture path. `$PRIO`/`$RUN_ID`/`$FINGERPRINT` are all in scope at both call sites (HM_RUN_START line ~103, HM_RUN_END line ~163). Best-effort `|| true` safety preserved. Header comment updated.
- `apps/monitor/homelab_monitor/kernel/cron/wrapper_constants.py` — `WRAPPER_FORMAT_VERSION` bumped `1.0.0` → `1.1.0` (boundary-marker emission format changed).
- Version-coupled tests updated to reference the `WRAPPER_FORMAT_VERSION` constant instead of hardcoding `"1.0.0"` (so future bumps don't break them): `test_wrapper_constants.py::test_format_version_is_semver` (dropped the `== "1.0.0"` equality, kept semver-shape check), `test_cron_install.py::test_build_install_kit_produces_correct_diff`, `test_api_crons.py` (`test_wrapper_health_ok_when_version_current` seed + `test_wrapper_format_is_current_equal_is_true` + stale docstring + added module-scope import, removed redundant function-local import), `test_cron_repo.py::test_set_wrapper_format_version`. The two `_parse_semver("1.0.0")` parser unit tests were intentionally left as literals (they test parsing, not the current version).
- `test_cron_wrapper_script.py::test_hm_run_markers_emitted` assertions flipped to the structured form (`MESSAGE=HM_RUN_START fp=...`).
