# STAGE-004-005: Cron fingerprint enrichment (vector hmrun transform + parsed cron_fingerprint field)

## Status: Not Started

## Overview

Today's vector `hmrun` source parses `HM_RUN=<uuid>` into a `.run_id` field but does NOT extract or carry the cron job's stable `fingerprint`. Without fingerprint at the per-line level, Drain clustering (STAGE-004-025+) would either:

- Lump ALL cron jobs into one Drain model under `service="hmrun"` (losing per-job signature differentiation), OR
- Need a separate post-fetch lookup step at Drain-time (more complex)

This stage enriches the hmrun pipeline so each cron output line carries `cron_fingerprint` as a top-level VL field. Drain's model-key override hook (built in STAGE-004-025) will use it to key per-fingerprint models.

The fingerprint is already known at run start (it's how we identify the cron job; computed from the cron line text per STAGE-002-*). We need to thread it through `hmrun` into the systemd journal so vector's existing journald-filtered source picks it up.

**Vertical slice:** every line emitted by `hmrun` carries `cron_fingerprint:<hash>` in VL. Queries like `cron_fingerprint:<X>` return only that cron job's lines. Drain (when built) can key models per fingerprint.

**Spec references:** Master spec §6.3 (cron capture).

## Concrete deliverables

### 1. `hmrun` wrapper enriches log output

`apps/monitor/homelab_monitor/cli/hmrun.py` (or wherever the wrapper lives) — when it invokes the user's cron command and pipes stdout/stderr, prefix lines with structured fields including:

- `HM_RUN=<uuid>` (already today)
- `HM_FP=<fingerprint>` (new — the stable fingerprint of this cron job)

OR, alternatively, emit the fingerprint once at run-start via a `logger -p user.info "HM_RUN=<uuid> HM_FP=<fp> START"` line and rely on vector's `add_labels` to remember it for subsequent lines (more complex; Design phase chooses).

Recommendation: per-line prefix is simpler; vector's parser extracts both fields with a single regex.

### 2. Vector hmrun source parser update

`deploy/vector/vector.toml.template` — update the `cron_parsed` (or equivalent) transform to extract `HM_FP=<value>` into `.cron_fingerprint`:

```toml
[transforms.hmrun_parsed]
type = "remap"
inputs = ["hmrun_journald_filter"]
source = '''
parsed = parse_regex(.message, r'^HM_RUN=(?P<run_id>[a-f0-9-]+) HM_FP=(?P<fingerprint>[a-f0-9]+) (?P<message>.*)$') ?? {}
.run_id = parsed.run_id
.cron_fingerprint = parsed.fingerprint
.message = if exists(parsed.message) { parsed.message } else { .message }
'''
```

(Final VRL syntax confirmed in Design phase.)

### 3. Existing cron run log endpoint adapts to new field

`apps/monitor/homelab_monitor/kernel/api/routers/crons.py` — the per-run log endpoint already queries VL by `run_id`; no change needed unless we want to validate that fingerprint matches the expected one. Design phase decides.

### 4. Integration test

`apps/monitor/tests/integration/test_hmrun_fingerprint_enrichment.py` — spawn `hmrun` against a known fingerprint, emit lines, query VL, assert lines carry both `run_id` AND `cron_fingerprint`.

### 5. Backward-compatibility check

Existing in-flight cron runs (those started before this stage deploys) won't have `cron_fingerprint` field. Code paths that consume the field MUST handle the absent case (treat as `unknown` or fall back to `service="hmrun"`).

## Out of scope for this stage

- Drain clustering itself — STAGE-004-025+
- Backfill of historical cron lines without fingerprint — accept the gap
- Per-fingerprint UI — comes naturally when Drain ships
- UI changes — pure backend/ingest

## Decisions in this stage

### Design phase will confirm

- **Per-line prefix vs single start-of-run emission** — recommendation: per-line prefix; vector parses each line independently. Simpler, robust to systemd line boundaries.
- **Whether to also include `cron_name` or other human-readable fields** in the prefix — recommendation: no, fingerprint is enough; UI looks up name from cron registry. YAGNI for additional fields.

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-CRON-FINGERPRINT-PRE-DRAIN** — fingerprint enrichment lands BEFORE Drain (this stage is a precursor to STAGE-004-025).
- **D-FINGERPRINT-PER-LINE** — each cron output line carries `cron_fingerprint` independently (not just one start-of-run emission).
- **D-DRAIN-KEY-PER-CRON-FINGERPRINT** — Drain model key for `service="hmrun"` is `cron:<fingerprint>` (override hook applied by STAGE-004-025).
- **D-ENRICHMENT-IS-ADDITIVE** — existing run_id-based lookups continue to work unchanged.
