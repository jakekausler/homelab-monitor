# STAGE-004-040: Throttle/budget alerts (L2) — stream budget threshold warnings

## Status: Complete

## Overview

The `log_stream_budget` collector (from STAGE-001-018) already emits per-stream byte/line metrics. This stage adds vmalert rules that alert when a stream is:

- Approaching its daily byte budget (e.g., 80% consumed)
- Being throttled by vector's rate limit (50 lines/sec/service today)
- Showing an unusually large jump in per-second line rate (operational anomaly)

Categorization: `anomaly_kind: log_budget_warning`.

**Vertical slice:** when a service's daily VL byte allocation hits 80%, a `LogStreamApproachingBudget` alert fires. When vector starts throttling a service, `LogStreamThrottling` fires. Both annotations link to the Explorer pre-filtered to that service so the user can investigate.

**Spec references:** master spec §6.3 (per-stream caps).

### 2. vmalert rules

`deploy/vmalert/logs/budget_threshold.yml` (new):

```yaml
groups:
  - name: log_budget
    interval: 60s
    rules:
      - alert: LogStreamApproachingBudget
        expr: homelab_log_stream_bytes_today / homelab_log_stream_bytes_budget > 0.80
        for: 5m
        labels:
          severity: warning
          source_tool: homelab-monitor
          category: log-anomaly
          anomaly_kind: log_budget_warning
        annotations:
          summary: "Stream {{ $labels.service }} at {{ $value | humanizePercentage }} of daily budget"
          description: |
            Stream {{ $labels.service }} on {{ $labels.host }} has consumed {{ $value | humanizePercentage }} of its daily byte budget.
            URL: /logs?logsql=service:"{{ $labels.service }}"

      - alert: LogStreamThrottling
        expr: rate(vector_events_throttled_total[5m]) > 0
        for: 5m
        labels:
          severity: warning
          source_tool: homelab-monitor
          category: log-anomaly
          anomaly_kind: log_budget_warning
        annotations:
          summary: "Vector is throttling stream {{ $labels.component_id }}"
          description: |
            Vector is dropping lines from {{ $labels.component_id }} due to throttle rules.
            Recent rate of throttled events: {{ $value }}/sec.
            URL: /logs?logsql=service:"{{ $labels.component_id }}"

      - alert: LogStreamUnusualLineRate
        # Per-second rate is N× the service's 1-hour baseline
        expr: |
          rate(homelab_log_stream_lines_per_sec[5m])
          > 5 * avg_over_time(rate(homelab_log_stream_lines_per_sec[5m])[1h:5m])
        for: 5m
        labels:
          severity: info
          source_tool: homelab-monitor
          category: log-anomaly
          anomaly_kind: log_budget_warning
        annotations:
          summary: "Unusual line rate from {{ $labels.service }}"
          description: |
            Stream {{ $labels.service }} is at {{ $value }}/sec, 5× its 1-hour baseline.
            URL: /logs?logsql=service:"{{ $labels.service }}"
```

## Decisions in this stage

### Design phase will confirm

- **Warning threshold** — 80% recommended; second tier (95%) → critical?
- **Unusual rate baseline window** — 1 hour recommended (shorter than spike rules because budget anomalies are operational, not security)
- **Whether to alert on vector backpressure events** — yes if available

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-BUDGET-WARN-AT-80** — first-tier warning at 80% of daily budget.
- **D-THROTTLE-VISIBILITY** — alert when vector is actively throttling.
- **D-RATE-SPIKE-AS-OPS-SIGNAL** — unusual rate vs 1-hour baseline = `anomaly_kind: log_budget_warning`, severity info.

## Design Phase

**Design phase complete: 2026-06-08.**
**Status**: [x] Design (Complete)

### Design Notes

**SELECTED APPROACH:** A1 + B3 + C (with B2 fallback for throttle).

**THREE RULES**, in `deploy/vmalert/metrics/budget_threshold.yaml` (NOT logs/, NOT .yml — same spec correction as 039: MetricsQL rules → metrics vmalert instance, .yaml extension):

- **LogStreamApproachingBudget:** `homelab_log_stream_bytes_today / homelab_log_stream_bytes_budget > 0.80`, for: 5m, severity warning.
- **LogStreamThrottling:** `increase(vector_component_discarded_events_total{component_id="throttle"}[5m]) > 0`, for: 2m, severity warning.
- **LogStreamUnusualLineRate:** `homelab_log_stream_lines_per_sec > clamp_min(5 * avg_over_time(homelab_log_stream_lines_per_sec[1h]), 1)`, for: 5m, severity info.

All three labels: `source_tool=vmalert-metrics`, `category=log-anomaly`, `anomaly_kind=log_budget_warning`, `target_kind=log_stream`. Annotations with Explorer deep-links where service-attributable.

**A1 (budget denominator):** The `log_stream_budget` collector (`apps/monitor/homelab_monitor/plugins/collectors/builtin/log_stream_budget.py`) currently emits ONLY `homelab_log_stream_bytes_today{host,service}` and `homelab_log_stream_lines_per_sec{host,service}`. The budget DENOMINATOR `homelab_log_stream_bytes_budget` does NOT exist. **FIX:** extend the collector to emit `homelab_log_stream_bytes_budget{host,service}` as a constant gauge = `LogStreamBudgetConfig.bytes_per_day_per_stream` (default 500 MiB, already loaded via `load_log_stream_budget_config()`). Single source of truth in config; rule is dumb MetricsQL. Rejected A2 (bake 524288000 magic number into YAML = config drift).

**B3 (throttle):** `vector_events_throttled_total` does NOT exist — vector's throttle transform doesn't export internal counters; only the redaction `log_to_metric` sink exports to prometheus_exporter on :9598 (which vmagent already scrapes). **FIX:** add `[sources.internal_metrics]` to `deploy/vector/vector.toml.template` and route it into the existing `[sinks.redaction_metrics]` prometheus_exporter inputs → surfaces vector's native `vector_component_discarded_events_total{component_id="throttle"}` (a REAL dropped-event counter). Rejected B1 (the spec's literal "incoming>50 ∧ stored<50" fallback) as INFEASIBLE: the collector queries VL post-storage only and cannot see the pre-throttle incoming rate. B2 (infer from stored rate pinned at ceiling) is the FALLBACK if B3's metric proves unusable — to be verified on the dev rig during Build/3a; even a GLOBAL "vector throttling" signal satisfies the locked D-THROTTLE-VISIBILITY decision.

**C (unusual rate):** `clamp_min` (element-wise, NOT max which aggregates labels to {} — documented MetricsQL trap), floor 1 line/sec to avoid tiny-baseline noise (tunable in Refinement), gauge re-emitted 60s so `avg_over_time[1h]` is staleness-safe.

**HOST-INTEGRATION:** B3 adds a vector config change → this stage REQUIRES both Refinement 3a (dev rig — verify the throttle metric's actual labels) AND 3b (prod rig — real vector throttling behavior). Note this explicitly.

**COVERAGE:** Only A1 is new Python (the budget-gauge emission + 1 test); B3 is vector config, C is YAML — both off the kernel coverage gate. 100% kernel coverage maintained.

**DEFERRED to STAGE-042:** per-service budget + threshold overrides. No parsed-but-unused schema needed now (A1's emitted gauge means 042 changes the emitted VALUE, not the rule). Add a DEFERRED comment in the rule file mirroring `error_rate_spike.yaml`.

**REDUNDANCY NOTE:** existing `deploy/vmalert/metrics/log_stream_budget.yaml` has `LogStreamBudgetOverrun` (`homelab_log_stream_lines_per_sec > 45`, for 5m) — ORTHOGONAL to the 3 new rules (it's the near-throttle line-rate warning; the new ones are byte-budget, active-throttle, rate-spike). Do NOT extend it; new file `budget_threshold.yaml`.

**TESTS:** `promtool *.tests.yaml` fixture for `budget_threshold.yaml` (fire/no-fire per rule), mirroring `error_rate_spike.tests.yaml`. Collector test for the new budget gauge. (Rule fixtures are static forward-compatible, not CI-wired — same as siblings.)

**SPEC CORRECTIONS:** (1) file path `logs/budget_threshold.yml` → `metrics/budget_threshold.yaml`; (2) `homelab_log_stream_bytes_budget` must be BUILT (didn't exist); (3) `vector_events_throttled_total` doesn't exist → use `vector_component_discarded_events_total` via internal_metrics.
