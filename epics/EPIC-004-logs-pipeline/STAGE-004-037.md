# STAGE-004-037: Anomaly Type C — "Service-wide error rate spike" rules + `homelab_container_error_rate` metric

## Status: Not Started

## Overview

Independent of signatures: per-service count of lines matching ERROR/FATAL/panic/traceback/Exception in the last 5 min exceeds either (a) a static threshold OR (b) N× the service's rolling baseline. This also satisfies EPIC-004 docker requirement #4 (per-container error-rate spike detection).

Categorization: `anomaly_kind: error_rate_spike`.

A new metric `homelab_container_error_rate{name, service, severity_class}` is emitted by a small collector that periodically queries VL for severity=error/fatal counts per service.

**Vertical slice:** per-service error-rate metric is available in VM. A vmalert rule fires when any service's 5-min error rate exceeds 5× its 7-day baseline OR 10 errors/min (whichever is more restrictive — the EPIC plan's documented threshold). Alert payload includes the spiking service, the error rate, the baseline, and a deep-link to the Explorer pre-filtered to errors from that service.

**Spec references:** EPIC-004 plan #4.

## Concrete deliverables

### 1. Error-rate collector

`apps/monitor/homelab_monitor/plugins/collectors/builtin/log_error_rate.py` (new):

- Runs every 60s
- For each distinct `service` in VL (cached list refreshed every 5 min):
  - Query VL: `service:"<svc>" AND severity:error` over last 60s → count
  - Same for severity:critical, fatal — combined into a single rate
  - Plus pattern-matching for "panic", "traceback", "exception" via LogsQL contains for languages whose `severity` isn't normalized to error (e.g., Python's logging output)
- Emit `homelab_container_error_rate{name=<svc>, severity_class="error"}` (lines/min)
- Emit `homelab_container_error_rate_pattern{name=<svc>, pattern_kind="panic"}` etc. for the pattern-matched signals

### 2. Pattern-based error matchers

Configurable via `homelab-monitor.yaml`:

```yaml
logs:
  error_patterns:
    - kind: panic
      regex: '\bpanic\b'
    - kind: traceback
      regex: '\bTraceback\b'
    - kind: exception
      regex: '\bException\b'
    - kind: http_5xx
      regex: 'status[:=]\s*5\d{2}\b'
```

Render into the collector's matching logic. Yaml-configurable so user can extend without code changes.

### 3. vmalert rule

`deploy/vmalert/logs/error_rate_spike.yml` (new):

```yaml
groups:
  - name: error_rate_spike
    interval: 60s
    rules:
      - alert: ServiceErrorRateSpike
        # Fire if 5-min error rate exceeds 5× 7-day baseline OR 10/min, whichever is more restrictive
        expr: |
          (
            rate(homelab_container_error_rate[5m])
            >
            max(
              5 * avg_over_time(rate(homelab_container_error_rate[5m])[7d:5m]),
              10  # static floor: 10 errors/min minimum
            )
          )
        for: 2m
        labels:
          severity: warning
          source_tool: homelab-monitor
          category: log-anomaly
          anomaly_kind: error_rate_spike
        annotations:
          summary: "Error-rate spike in {{ $labels.name }}"
          description: |
            Service {{ $labels.name }} is emitting errors at {{ $value }}/min,
            exceeding both its 7-day baseline and the 10/min floor.
            URL: /logs?logsql=service:"{{ $labels.name }}" AND severity:error
```

### 4. Per-container configurable threshold

`homelab-monitor.yaml` extension:

```yaml
logs:
  error_rate_overrides:
    - service: "noisy-service-by-design"
      static_floor: 100  # this service is allowed to be noisy
    - service: "critical-service"
      multiplier: 2  # tighter threshold
```

Rendered into per-service rule overrides at boot.

### 5. Cold-start fallback

Same as STAGE-004-036: if a service has < 7 days of error history, use 1-hour baseline instead. Adjust the rule's `avg_over_time` window accordingly.

### 6. Tests

- Collector unit tests (mocked VL responses)

## Out of scope for this stage

- Per-service auto-baseline learning (with manual override) — config-driven for v1
- Cross-service correlated error spikes — speculative
- Auto-suppress noisy services — user can use error_rate_overrides instead

## Decisions in this stage

### Design phase will confirm

- **Default static floor** — 10 errors/min recommended
- **Default multiplier** — 5× recommended
- **Pattern-match scope** — recommended kinds in pattern list above; tunable
- **Whether `service` and `name` labels are aliased** — recommend yes (since docker `service` was set from `container_name` originally)

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-ERROR-RATE-METRIC-FIRST** — emit the metric; rule then references it. Decouples collector from rules.
- **D-STATIC-FLOOR-AND-BASELINE** — both checks AND'd: must exceed BOTH static floor AND baseline-multiplier.
- **D-PATTERNS-YAML-CONFIGURABLE** — error patterns in `homelab-monitor.yaml` for extensibility.
