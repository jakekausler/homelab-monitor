# STAGE-004-037: Anomaly Type C — "Service-wide error rate spike" rules + `homelab_container_error_rate` metric

## Status: Complete

## Overview

Independent of signatures: per-service count of lines matching ERROR/FATAL/panic/traceback/Exception in the last 5 min exceeds either (a) a static threshold OR (b) N× the service's rolling baseline. This also satisfies EPIC-004 docker requirement #4 (per-container error-rate spike detection).

Categorization: `anomaly_kind: error_rate_spike`.

A new metric `homelab_container_error_rate{name, service, severity_class}` is emitted by a small collector that periodically queries VL for severity=error/fatal counts per service.

**Vertical slice:** per-service error-rate metric is available in VM. A vmalert rule fires when any service's 5-min error rate exceeds 5× its 7-day baseline OR 10 errors/min (whichever is more restrictive — the EPIC plan's documented threshold). Alert payload includes the spiking service, the error rate, the baseline, and a deep-link to the Explorer pre-filtered to errors from that service.

**Spec references:** EPIC-004 plan #4.

## Concrete deliverables

### 1. Error-rate collector

`apps/monitor/homelab_monitor/plugins/collectors/builtin/log_error_rate.py` (new):

- Runs every 60s
- For each distinct `service` in VL (cached list refreshed every 5 min):
  - Query VL: `service:"<svc>" AND severity:error` over last 60s → count
  - Same for severity:critical, fatal — combined into a single rate
  - Plus pattern-matching for "panic", "traceback", "exception" via LogsQL contains for languages whose `severity` isn't normalized to error (e.g., Python's logging output)
- Emit `homelab_container_error_rate{name=<svc>, severity_class="error"}` (lines/min)
- Emit `homelab_container_error_rate_pattern{name=<svc>, pattern_kind="panic"}` etc. for the pattern-matched signals

### 2. Pattern-based error matchers

Configurable via `homelab-monitor.yaml`:

```yaml
logs:
  error_patterns:
    - kind: panic
      regex: '\bpanic\b'
    - kind: traceback
      regex: '\bTraceback\b'
    - kind: exception
      regex: '\bException\b'
    - kind: http_5xx
      regex: 'status[:=]\s*5\d{2}\b'
```

Render into the collector's matching logic. Yaml-configurable so user can extend without code changes.

### 3. vmalert rule

`deploy/vmalert/logs/error_rate_spike.yml` (new):

```yaml
groups:
  - name: error_rate_spike
    interval: 60s
    rules:
      - alert: ServiceErrorRateSpike
        # Fire if 5-min error rate exceeds 5× 7-day baseline OR 10/min, whichever is more restrictive
        expr: |
          (
            rate(homelab_container_error_rate[5m])
            >
            max(
              5 * avg_over_time(rate(homelab_container_error_rate[5m])[7d:5m]),
              10  # static floor: 10 errors/min minimum
            )
          )
        for: 2m
        labels:
          severity: warning
          source_tool: homelab-monitor
          category: log-anomaly
          anomaly_kind: error_rate_spike
        annotations:
          summary: "Error-rate spike in {{ $labels.name }}"
          description: |
            Service {{ $labels.name }} is emitting errors at {{ $value }}/min,
            exceeding both its 7-day baseline and the 10/min floor.
            URL: /logs?logsql=service:"{{ $labels.name }}" AND severity:error
```

### 4. Per-container configurable threshold

`homelab-monitor.yaml` extension:

```yaml
logs:
  error_rate_overrides:
    - service: "noisy-service-by-design"
      static_floor: 100  # this service is allowed to be noisy
    - service: "critical-service"
      multiplier: 2  # tighter threshold
```

Rendered into per-service rule overrides at boot.

### 5. Cold-start fallback

Same as STAGE-004-036: if a service has < 7 days of error history, use 1-hour baseline instead. Adjust the rule's `avg_over_time` window accordingly.

### 6. Tests

- Collector unit tests (mocked VL responses)
- vmalert rule test (plant metric history + spike)
- Pattern matching tests with realistic log line samples

## Out of scope for this stage

- Per-service auto-baseline learning (with manual override) — config-driven for v1
- Cross-service correlated error spikes — speculative
- Auto-suppress noisy services — user can use error_rate_overrides instead

## Decisions in this stage

### Design phase will confirm

- **Default static floor** — 10 errors/min recommended
- **Default multiplier** — 5× recommended
- **Pattern-match scope** — recommended kinds in pattern list above; tunable
- **Whether `service` and `name` labels are aliased** — recommend yes (since docker `service` was set from `container_name` originally)

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-ERROR-RATE-METRIC-FIRST** — emit the metric; rule then references it. Decouples collector from rules.
- **D-STATIC-FLOOR-AND-BASELINE** — both checks AND'd: must exceed BOTH static floor AND baseline-multiplier.
- **D-PATTERNS-YAML-CONFIGURABLE** — error patterns in `homelab-monitor.yaml` for extensibility.

### Design Decision — chosen approach (locked 2026-06-07)

**CHOSEN: Stateless per-window GAUGE + direct-comparison metrics rule (Option B).**

**A. Counter vs Gauge — GAUGE.** The collector emits `homelab_container_error_rate{name}` as a per-60s-window error COUNT gauge (a "rate" only because the window is pinned to the 60s collector interval). REJECTED a monotonic counter (`homelab_container_errors_total` + `rate()` rule): a counter needs an in-memory per-service accumulator that is RESTART-FRAGILE (on restart the accumulator resets to 0 while VM holds the old high value → `rate()` sees a counter reset → a real spike straddling the restart is MASKED) and needs stale-service eviction. The gauge is a PURE FUNCTION of the current VL response → stateless, restart-safe, trivially 100%-coverable. It also kills the card's `rate(homelab_container_error_rate[5m])` BUG (rate() is counter-only; the metric is emitted via `ctx.vm.write_gauge`, same gauge-vs-counter bug class as STAGE-035/036).

**The rule (DUMB, direct gauge comparison — NO rate()):**
```
homelab_container_error_rate > max(5 * avg_over_time(homelab_container_error_rate[7d:5m]), 10)
```
This maps 1:1 onto D-STATIC-FLOOR-AND-BASELINE: `> max(5*baseline, 10)` fires only when the value exceeds the HIGHER of the two thresholds = BOTH the 5× baseline AND the 10/min floor satisfied. **The locked 10/min static floor DOUBLES as the gap-robustness floor**: the gauge is gappy (a service with no errors in a window emits no sample → avg_over_time averages only windows-with-errors → slightly inflated baseline), but a quiet/gappy service can NEVER false-fire because it must also clear 10 absolute errors/min. So unlike STAGE-036, NO separate `min_baseline` parameter is needed — the locked floor provides it for free.

**Metric name DECISION: keep `homelab_container_error_rate{name}`** (matches the card title + EPIC table + docker req #4 wording). REJECTED renaming to `homelab_container_errors_per_min` (would diverge from the card/EPIC). Document in BOTH the collector and the rule that it's a per-60s-window count gauge ("rate" because the window == the 60s interval); pin the interval to 60s and note the coupling. Label set: `{name}` (the service). (Card was inconsistent — title said `{name, service, severity_class}`, deliverable said `{name, severity_class}`; v1 folds all severities into one count so there is NO `severity_class` split → label is just `{name}`.)

**B. VL query mechanism — inject a `VictoriaLogsClient`.** `ctx.vl` is WRITE-ONLY (a `LogsWriter`, only `ingest()`) — it CANNOT query VL (card's implication was wrong). The collector is injected a `VictoriaLogsClient` (constructor arg, defaulting to a lazily-built client from `ctx.http` + `os.environ["HOMELAB_MONITOR_VL_URL"]` (default `http://victorialogs:9428`) + `load_vl_query_limits()`), mirroring `LogStreamBudgetCollector`'s optional-injection pattern. Reuses the tested bounded/parsed primitive (`kernel/logs/services.py` already uses `client.query()` with `stats by`). A SINGLE `stats by (service) count() as count` query over `[now-60s, now]` returns EVERY service's error count in one shot (confirmed — same as `fetch_services`); no separate service-list query needed. Tests inject a FAKE client returning a canned `VlQueryResult` (no HTTP mocking).

**C. Error definition + YAML patterns (D-PATTERNS-YAML-CONFIGURABLE) — fold ALL error-like lines into the ONE gauge; defer the per-pattern metric.** One LogsQL query unions severity + pattern matches:
```
(severity:error OR severity:critical OR severity:fatal OR _msg:~"<p1>|<p2>|...") | stats by (service) count() as count
```
Config shape mirrors the existing `RedactPattern` / `DEFAULT_REDACT_PATTERNS` / `load_redact_patterns()` precedent in `kernel/config.py` (a `logs:`-section list-of-frozen-dataclass + a `load_logs_config()`): `ErrorPattern(kind: str, regex: str)` with defaults `panic` (`panic`), `traceback` (`[Tt]raceback`), `exception` (`[Ee]xception`); the collector ORs `severity:...` with `_msg:~"<regexes joined by |>"`. Keep `kind` in the dataclass so the deferred per-pattern breakdown is a non-breaking add. DEFER the separate `homelab_container_error_rate_pattern{name, pattern_kind}` metric — v1's locked scope is "metric + rule" and one folded count satisfies it.

**D. Per-service overrides + cold-start — single global rule; BOTH DEFERRED.** ONE global vmalert-metrics rule with the default 10/min floor + 5× 7d baseline for all services. DEFER:
- `logs.error_rate_overrides` (per-service static_floor/multiplier) → needs STAGE-042's render-at-boot infra (does not exist). Define the config key NOW so 042 can consume it; document the deferral.
- Per-service 1h cold-start fallback → STAGE-036's `or`-of-two-age-guarded-branches trick requires a per-service `first_seen_ts` metric, which does NOT exist for error-rate (would re-introduce restart-fragile persistent state — Option C's cost). The 10/min ABSOLUTE floor already prevents young-service false-fires (a new service can't fire on a low 7d baseline unless it also clears 10/min), so the cold-start branch is LOW-VALUE for v1. Defer to 042/044.
This keeps 037 strictly "metric + single global rule" and avoids building 042's per-service-rule-rendering early (consistent with how STAGE-036 was scoped).

### Refinement Results (2026-06-07, prod rig)

Backend-only stage; validated LIVE on the prod rig. Caught + fixed TWO real Build bugs that all synthetic gates (100% unit coverage, vmalert fixture, full verify) missed:

- **BUG 1 (rule file extension):** `error_rate_spike.yml` was silently NOT loaded — vmalert-metrics globs `-rule=/etc/vmalert/metrics/*.yaml` (`.yaml` ONLY). Renamed to `error_rate_spike.yaml`. (NOTE: the pre-existing `container_crash.yml` + `container_healthcheck.yml` from STAGE-032/033 have the SAME latent bug — they also do NOT load in prod. Out of THIS stage's scope; flagged for a follow-up.) (FIXED 2026-06-08: both renamed to `.yaml` in yaml-glob bugfix.)

- **BUG 2 (MetricsQL label collapse):** expr `> max(5*avg_over_time(...),10)` NEVER fired — `max(vector, scalar)` is an AGGREGATION collapsing all series to empty labels `{}`, so the `>` against the `{name}`-labeled gauge matched nothing AND aggregated baselines across services. Fixed to `> clamp_min(5*avg_over_time(...),10)`: element-wise, per-series, preserves `{name}`. Proven against live VM (original returned `[]` with gauge=716; clamp_min returns 6 per-`{name}` series).

Final e2e (6/6 PASS): rule loads with `clamp_min` + health=ok; the collector emits `homelab_container_error_rate{name}` per service (even from REAL homelab error logs — homeassistant/homelab-monitor); the corrected expr matches per-service (label preserved); a fresh service planted with 40→300 errors FIRED `ServiceErrorRateSpike` (state firing, value 300, labels category=log-anomaly/anomaly_kind=error_rate_spike/name/severity=warning/target_kind=container); reached Alertmanager (state active); a below-floor service (homeassistant) did NOT fire.

Behavioral note (NOT a bug, by design): the `[7d:5m]` baseline self-elevates as a burst is sustained (5×40=200 after one minute), so the rule fires on SPIKES relative to recent history, not sustained absolute counts — the intended D-STATIC-FLOOR-AND-BASELINE spike-detection semantics.
