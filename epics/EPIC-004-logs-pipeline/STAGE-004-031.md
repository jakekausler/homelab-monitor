# STAGE-004-031: `LogWindowFetcher` shared service module (used by crash / healthcheck / cron correlation)

## Status: Complete

## Overview

A small backend service that fetches the last N seconds of log lines around a timestamp, scoped to a `service` (or arbitrary LogsQL filter). Used by:

- Container crash correlation (STAGE-004-032)
- Healthcheck-failure log enrichment (STAGE-004-033)
- Cron run failure log correlation (STAGE-004-034)

Splitting this shared module out as its own precursor stage avoids duplicating the windowed-fetch logic across 3 stages.

**Vertical slice:** `LogWindowFetcher(...).fetch(filter, ts, window_before_s, window_after_s)` returns the relevant `LogLine[]` from VL with caching.

**Spec references:** master spec §6.3.

```python
class LogWindowFetcher:
    def __init__(self, vl_client: VictoriaLogsClient, *, cache_ttl_s: int = 300):
        ...

    async def fetch(
        self,
        logs_ql: str,  # filter, e.g., 'service:"home-assistant"'
        anchor_ts: datetime,
        window_before_s: int = 60,
        window_after_s: int = 60,
        limit: int = 200,
    ) -> LogWindowResult:
        ...
```

`LogWindowResult`:

```python
class LogWindowResult(BaseModel):
    lines: list[LogLine]
    truncated: bool
    window_start: datetime
    window_end: datetime
    queried_at: datetime  # when this fetch happened (cache-hit-aware)
```

### 2. Caching

- Cache key: `(logs_ql, anchor_ts.isoformat(), window_before, window_after, limit)`
- TTL default 5 min (the same call is often repeated by multiple consumers shortly after an event)
- In-process LRU cache, max 1000 entries

### 3. Defensive limits

- Window total (before + after) capped at 1 hour
- `limit` capped at 1000 lines
- VL transport errors return an empty result with a logged warning (don't raise — downstream consumers should degrade gracefully when log context is unavailable)

### 4. Tests

- Unit: mocked VL client, asserts query window math is correct
- Cache: same call within TTL is a cache hit
- VL error: returns empty result, doesn't raise
- Limit + window cap: enforced

## Decisions in this stage

### Design phase will confirm

- **Cache TTL** — 5 min recommended
- **LRU max entries** — 1000 recommended
- **Window cap** — 1 hour total recommended
- **Limit cap** — 1000 recommended

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-LOG-WINDOW-SHARED** — single module used by all log-enrichment stages.
- **D-LOG-WINDOW-DEGRADE-GRACEFULLY** — VL transport errors return empty result, don't raise.
- **D-LOG-WINDOW-CACHED** — bounded LRU cache prevents thundering herd.

## Design Phase

### Design Decisions (chosen 2026-06-06)

#### Selected approach

**D1 — VL fetch via stream_query (honors limit + truncated).**
- Use `vl_client.stream_query(expr, start, end, limit=effective_limit + 1)`, collect into a list; `truncated = len(collected) > effective_limit`; return `collected[:effective_limit]` mapped to LogLine.
- Rationale: `query()` has NO limit param (caps at 10k internally) → wasteful for a ~200-line window AND conflates VL's cap with our limit in `truncated`. stream_query honors the per-call limit at the source (O(1) memory) and gives `truncated` its only meaningful definition: "more lines existed in the window than we returned."
- Off-by-one is load-bearing: pass effective_limit+1, slice to effective_limit. Pin with an exactly-(limit+1)-lines boundary test.

**D2 — Bounded LRU + TTL cache, TWO injected clocks.**
- `OrderedDict`-based LRU, max 1000 entries. get: expired→delete+None, else move_to_end+return. put: set+move_to_end, if len>max → popitem(last=False). Lazy-expire on get (no background sweeper; mirrors ServicesCache/CycleStatusStore).
- TWO injected clocks: `clock: Callable[[], float] = time.monotonic` (TTL expiry only — robust to wall-clock jumps) AND `wall_clock: Callable[[], datetime] = lambda: datetime.now(UTC)` (produces queried_at + window math). Both injectable for deterministic tests.
- Cache key: `(logs_ql, anchor_utc_iso, window_before_s, window_after_s, limit)` using RAW args (clamping is deterministic from them) and the UTC-normalized anchor ISO.
- `queried_at` is the ORIGINAL VL-fetch wall-clock time, STORED in the cached LogWindowResult — a cache HIT returns the original queried_at unchanged (this IS "cache-hit-aware": consumers see when data was actually read from VL, up to ttl ago).
- Both LRU(1000) AND TTL are load-bearing (NOT redundant): every distinct anchor_ts is a new key, so a burst of crash events explodes keys — LRU caps absolute count, TTL prevents stale slots persisting.

**D3 — Window math + caps.**
- Normalize anchor_ts to UTC (naive→assume UTC via replace(tzinfo=UTC); aware→astimezone(UTC)), mirroring parse_iso_to_ms convention.
- window_start = anchor - window_before_s; window_end = anchor + window_after_s.
- Total-window cap 3600s: CLAMP proportionally (preserve before/after ratio): if before+after>3600: scale=3600/total; before=int(before*scale); after=3600-before. (Defaults 60/60 never trigger.) Clamp (not raise) — degrade-gracefully library on non-interactive paths.
- limit cap: min(limit, 1000), applied before the +1 stream call.
- Result window_start/window_end reflect the CLAMPED, UTC-normalized window actually queried (datetime); rendered to VL via .isoformat().

**D4 — Mapping + graceful degradation.**
- Map each collected VlLogLine via `from_victorialogs_line(...) -> LogLine` (kernel/logs/models.py). LogWindowResult.lines is list[LogLine] (converged model).
- On VictoriaLogsClientError: return LogWindowResult(lines=[], truncated=False, degraded=True, window_start=<clamped>, window_end=<clamped>, queried_at=wall_clock()) + structlog WARNING (e.g. log.warning("log_window_fetcher.degraded", error=str(exc))). The empty result still carries window bounds + queried_at so consumers always get a well-formed result.
- DO NOT cache error results — only cache.put on the success branch (a VL blip must not blind enrichment for the TTL).

**D-EXTRA — ADDED `degraded: bool = False` field to LogWindowResult (deliberate, justified extension of the locked contract).**
- The card's LogWindowResult fields are marked locked, but `degraded` is added as a STRICTLY ADDITIVE optional field (default False) — does NOT change the locked method signature or any existing field.
- Rationale: D-LOG-WINDOW-DEGRADE-GRACEFULLY is only useful if consumers (STAGE-032/033/034) can DISTINGUISH "VL fetch failed / log context unavailable" (degraded=True, lines=[]) from "window genuinely had zero logs" (degraded=False, lines=[]). Without the flag both are identical empty results, defeating half the degradation goal (the consumer can't communicate the degradation in its alert enrichment UI). Also makes the error-degrade path cleanly testable without log-capture gymnastics.
- The lock fixes the SHAPE of present fields, not a prohibition on additive optional fields. Decision made autonomously per user's "go with your recommendation" instruction.

#### Final LogWindowResult shape (Pydantic)
```
class LogWindowResult(BaseModel):
    lines: list[LogLine]
    truncated: bool
    degraded: bool = False   # ADDED (D-EXTRA)
    window_start: datetime
    window_end: datetime
    queried_at: datetime
```

#### Risks to carry into Build
1. Off-by-one in truncated: pass effective_limit+1 to stream_query, slice to effective_limit; truncated = collected>effective_limit. Boundary test with exactly limit+1 lines.
2. High-cardinality anchor keys: LRU(1000) AND TTL both required; test eviction with 1001 distinct anchors.
3. Keep TTL clock (monotonic float) and wall_clock (datetime) as SEPARATE injected params; do NOT derive queried_at from the monotonic clock.
4. queried_at stored in cached value, never refreshed on hit.
5. Never cache on error (single cache.put inside the success branch only).
6. 100% kernel coverage (EPIC-004 standard) — every branch (clamp, cap, cache hit/miss/evict/expire, error-degrade, naive-vs-aware tz) needs a test.
