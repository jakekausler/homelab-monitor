# STAGE-004-018: Filter-scope-aware field discovery (only show fields present in current query scope)

## Status: Complete

## Overview

After STAGE-004-017 lands, every JSONL line gets nested fields flattened into `json.*` paths. But across the whole VL store there could be HUNDREDS of distinct field paths (HA, Pi-hole, monitor itself, etc., all with their own structures). Showing the user a flat list of every field ever seen is overwhelming.

This stage adds a discovery mechanism: when the user has a filter set (e.g., `service:"home-assistant"`), the Explorer only surfaces fields that actually appear in that subset. If they filter to last 7 days, only fields present in that window. If they remove all filters, the list grows back to everything.

Surfaces affected:
- Field Inspector (STAGE-004-016) — only show fields that exist on the inspected line PLUS sibling lines in the current scope (so the "Add to filter" can suggest siblings the user might want)
- A new "Available fields" panel/sidebar tab that lists fields-in-current-scope with example values and one-click filter injection

**Vertical slice:** filter to `service:"home-assistant"` over last 1h → field discovery panel lists `level, attributes.entity_id, attributes.state, json.context.user_id, ...` only — NOT `pihole.upstream_dns_response_code` etc. Click a field name → it appears in the Field Inspector with sample values.

## Concrete deliverables

### 1. Backend endpoint: discover fields in scope

`apps/monitor/homelab_monitor/kernel/api/routers/logs.py` — new endpoint:

```
GET /api/logs/fields?expr=<LogsQL>&start=<ISO>&end=<ISO>&sample=200
```

Response:

```python
class LogsFieldsResponse(BaseModel):
    fields: list[FieldDescriptor]
    sampled_lines: int  # how many lines the field set came from
    truncated: bool  # were there more lines than the sample?

class FieldDescriptor(BaseModel):
    name: str  # dotted path, e.g., "json.context.user_id"
    sample_values: list[str]  # up to 5 example values seen in the sample
    coverage: float  # 0.0-1.0; fraction of sampled lines that had this field
    type_hint: str  # 'string' | 'number' | 'bool' | 'object' | 'array' | 'mixed'
```

Implementation:
1. Query VL with `<expr> | head N=200 | fields *` (or similar; LogsQL exact syntax confirmed in Design)
2. Iterate sampled lines, collect all field paths seen and their coverage / sample values / type hints
3. Cache the result for (expr_hash, range_hash, sample_size) for 30s
4. Truncate sample_values to first 5 distinct values per field

Sample size default: 200 lines. Larger samples are more accurate but slower; 200 is a reasonable balance.

## Decisions in this stage

### Design phase will confirm

- **Sample size** — recommend 200; tunable per Design
- **Cache TTL** — 30s recommended
- **Coverage threshold for "always show" vs "rare field" UI distinction** — recommendation: show all with coverage as a percentage
- **LogsQL syntax for field enumeration** — confirm against current VL version

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-SCOPE-AWARE-DISCOVERY** — fields shown reflect the CURRENT filter scope, not the global schema.
- **D-SAMPLE-BASED** — sample-based per-query, not pre-computed schema. Simpler, always-fresh, modest VL cost via caching.
- **D-COVERAGE-PERCENT-VISIBLE** — UI shows coverage % so the user knows which fields are rare vs ubiquitous.

## Design Phase

**Design phase complete:** 2026-06-03

**Selected Approach: Option C — Hybrid field discovery**

Hybrid field discovery (`field_names` for names+coverage + a bounded `query` sample for values+type-hints).

Backend `GET /api/logs/fields?expr=&start=&end=&services=&sample_n=200` (auth `Depends(require_session())`):
- Reuse `_compose_services_expr(expr, services)` + `parse_and_validate_window(start, end)` (same scope-composition + window validation as `/api/logs/query`). Map `VictoriaLogsClientError` → 502 `upstream_unavailable`.
- **Call 1 — `/select/logsql/field_names`** (NEW `VictoriaLogsClient.field_names(expr, start, end) -> list[tuple[str,int]]`, parses `{"values":[{"value","hits"}]}`): authoritative complete field-name list + per-field hit counts. Coverage% = `field.hits / (_msg entry hits)` — EXACT full-population coverage (the `_msg` field is on every line, so its hits == total matching lines).
- **Call 2 — existing `client.query(expr, start, end)` with HTTP `limit=sample_n` (N=200, most-recent lines)**: derive sample values (up to K distinct per field) + type hints from the sampled lines' field bags, mapped ONTO the authoritative field_names list.
- **Total: 2 HTTP calls, FIXED regardless of field count** (no per-field fan-out). Use HTTP `limit=N` (recent) NOT `| limit N` (random) — sample's only job is representative values, and "recent" is the more useful bias + reuses the existing query() primitive.
- Type hint inferred from sampled string values: `numeric | bool | string | object | array | mixed | unknown` (unknown when a field has accurate coverage but no sampled value).
- **30s in-memory cache** in a new `kernel/logs/fields.py` module (mirrors `kernel/logs/services.py`'s `ServicesCache`): `FieldsCache` keyed by `(sha256(effective_expr), start, end, sample_n)`, injectable monotonic clock. Hash the expr (composed exprs can be ~4KB).

Response models (new in `kernel/api/schemas.py`):
```
FieldDescriptor: name: str; sample_values: list[str]; coverage: float (0..1, exact); type_hint: str
LogsFieldsResponse: fields: list[FieldDescriptor] (sorted DESC by coverage, tie-break name ASC); sampled_lines: int; truncated: bool
```

Frontend: `<FieldsDiscoveryPanel>` as a 4th sidebar tab ("Available fields") in `LogsExplorerBody.tsx`; `useLogsFieldsQuery` react-query hook (key `['logs','fields',expr,start,end,services,sample]`, `staleTime: 30_000`, enabled when start/end set) mirroring `useLogsServicesQuery`; fields sorted by coverage desc, show coverage% per field; click a sample value → reuse the existing `appendFieldFilter(field, value)` one-click injection; click a field → cross-link into the existing `<FieldInspectorPanel>`. Field Inspector gains an "N more fields in this scope" affordance linking to the panel.

**Accepted trade-off (v1):** a low-coverage field appears in `field_names` (so shows accurate coverage%) but may be absent from the 200 most-recent sampled lines → it shows empty `sample_values` + `type_hint:"unknown"`. Self-documenting: coverage% tells the user it's rare. Acceptable per card scope.

**Rejected alternatives:**
- **Option A (pure sample, 1 `/query?limit=200` call, derive everything client-side):** rejected — a field present in <1/200 lines is silently MISSED entirely (never appears), a real correctness gap for the 018A autocomplete consumer; coverage is only sample-approximate. Cheaper by 1 call but structurally can't guarantee rare fields surface.
- **Option B (pure native: `/field_names` + one `/field_values?limit=K` PER field):** rejected — 1+N HTTP calls = 50-200 VL round-trips per uncached load after `json.*` flattening (fan-out explosion, violates bounded-query discipline); worst latency/testability/failure-handling; and `field_values` with `limit` ZEROES the hit counts so it doesn't even buy real value frequency for the cost.

**Verified VL v0.30.0 facts (backend-contract verification):**
- VL pinned `victoriametrics/victoria-logs:v0.30.0-victorialogs` (deploy/compose/*.yml). Field-discovery endpoints exist since v0.8.0.
- `/select/logsql/field_names?query=&start=&end=` returns `{"values":[{"value","hits"}]}` with per-field hit counts → accurate coverage in ONE call.
- `/select/logsql/field_values?...&limit=K` zeroes `hits` when limited (top values only, no frequency).
- LogsQL: `| head` INVALID; use HTTP `limit=N` (recent) or `| limit N` (random). `| fields *` INVALID (query returns all fields by default).
- Existing `VictoriaLogsClient` wraps ONLY `/select/logsql/query` today — `field_names` is net-new. It has `logsql_quote_phrase()`, `_parse_ndjson`, `with_limits()`.

**Forward-compat:** 018A (autocomplete) gets the COMPLETE field-name list + accurate coverage (to rank completions) + sample_values; 018B (columns) gets name + type_hint + sample_values. Option C never drops rare fields, so both consumers see the full scope.
