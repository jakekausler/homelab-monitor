# STAGE-004-012A: Service source_type field + grouped/collapsible stream picker

## Status: Complete

## Overview

Builds on STAGE-004-012 (the flat service-only stream picker sidebar). Adds a real `source_type` dimension to the log pipeline so services can be GROUPED by what KIND they are (docker container, systemd unit, cron, etc. — extensible as more kinds appear), and refactors the stream-picker sidebar into collapsible sections (one per source_type) with select-all / select-none per section.

Motivated by user feedback during STAGE-004-012 Refinement: the flat service list mixes crons and docker (and more kinds eventually) with no way to group by type. Investigation confirmed NO existing field stores service type — type is only implicitly encoded across scattered fields (container_name/image/compose labels = docker; `_SYSTEMD_UNIT` / `.service` suffix = systemd; `SYSLOG_IDENTIFIER` CRON/crond + `service=="hmrun"` = cron).

**Vertical slice:** the `/logs` stream-picker sidebar groups services into collapsible sections by kind — a "docker" section, a "systemd" section, a "cron" section (and "unknown" fallback) — each with a header showing the type + aggregate count + collapse/expand toggle + a select-all / select-none control. Selected services still flow into 012's `selectedServices[]` model and `&services=` URL param unchanged.

**Two-part scope (note dependency direction):**

1. **BACKEND / INGEST (the dependency — must land first):** add an explicit `source_type` label at Vector ingest, then surface it on the services-list source so the UI can group.
2. **FRONTEND (depends on part 1):** refactor `StreamPickerSidebar` into collapsible sections, one per `source_type`, with per-section select-all/none.

The internal ordering MUST be: ingest field → endpoint shape → UI grouping.

**Spec references:** master spec §9.2 (stream picker). Builds directly on STAGE-004-012.

## Concrete deliverables

### 1. Backend / ingest: `source_type` label at Vector ingest

`deploy/vector/vector.toml.template` — set an explicit `source_type` label in/after the existing transforms (`transforms.add_labels` / `docker_enrich` / `hmrun_shaped`), derived from the presence of existing markers:

- docker: has `container_name` / docker compose labels / `.image` → `source_type="docker"`
- systemd unit: has `_SYSTEMD_UNIT` (and not docker) → `source_type="systemd"`
- cron: `SYSLOG_IDENTIFIER` in (`CRON`, `crond`) OR `service=="hmrun"` → `source_type="cron"`
- fallback → `source_type="unknown"`

Make `source_type` a VictoriaLogs field. Decide in Design whether it also becomes a **stream field** alongside `host`/`service`/`severity`, or a regular field — note the trade-off: stream-field cardinality is low here (~4 values) so it's a reasonable stream field, but confirm in Design.

This is a HOST-INTEGRATION change (Vector config) → Refinement is 3a + 3b (dev rig synthetic + prod rig REAL host data: real cron, real docker, real journal). Vector VRL gotchas apply (no `emit_metric`; `r'...'` regex; `vector validate` gate; image `timberio/vector:0.41.1-debian`) — reference STAGE-004-017 notes.

### 2. Backend: surface `source_type` on the services-list source

The services-list source (STAGE-004-012's `GET /api/logs/services` and/or a new grouped variant) must return `source_type` per service so the UI can group. Decide in Design:

- **Option A:** extend `/api/logs/services` to include `source_type` on each `ServiceCount` (→ `ServiceCount` gains a `source_type` field).
- **Option B:** a new grouped endpoint.

If the response shape changes → OpenAPI regen required (`make openapi-export` then `bash scripts/generate-ui-types.sh` BEFORE frontend verify).

### 3. Frontend: collapsible grouped sidebar

Refactor `apps/ui/src/routes/logs/StreamPickerSidebar.tsx` (from 012) into COLLAPSIBLE SECTIONS, one per `source_type`. Each section header shows:

- the type label
- the aggregate count for that section
- a collapse/expand toggle
- a select-all / select-none control for that section

Selected services still flow into the same `selectedServices[]` model from 012 — do NOT break 012's URL/state contract (`&services=`). Preserve desktop-inline / mobile-drawer responsive behavior (the existing `useMediaQuery` / AppShell drawer pattern from 012).

### 4. Tests

- Backend: `source_type` derivation (docker / systemd / cron / unknown) at ingest; services-list response carries `source_type`.
- Frontend: sidebar groups by `source_type` into collapsible sections; collapse/expand toggle; per-section select-all/none; selected services still flow into `selectedServices[]`; URL `&services=` contract unchanged; mobile drawer preserved.

## Out of scope for this stage

- Grouping by fields other than `source_type` (host/severity) — still the future "Streams tab".
- Per-service-kind custom icons/styling beyond basic section headers (YAGNI unless trivial).

### Locked Design Decisions (inherited from STAGE-004-012 Refinement feedback 2026-06-01)

- **D-SOURCE-TYPE-EXPLICIT** — service type is made an explicit `source_type` label at ingest; no existing field stores it (type was only implicitly encoded across container_name/image/compose labels, `_SYSTEMD_UNIT`/`.service`, `SYSLOG_IDENTIFIER` CRON/crond + `service=="hmrun"`).
- **D-SOURCE-TYPE-EXTENSIBLE** — the set of source_types is open-ended (docker / systemd / cron / unknown today; more kinds as new sources land).
- **D-GROUPED-PICKER-PRESERVES-012-CONTRACT** — the grouped/collapsible sidebar reuses 012's `selectedServices[]` state and `&services=` URL param unchanged.
- **D-012A-DEPENDS-ON-012** — 012A is a DEPENDENT of 012 (needs the picker to exist); its own frontend half depends on its backend `source_type` field landing first (ingest field → endpoint shape → UI grouping).

## Design phase decisions (locked 2026-06-01)

**D-012A-A1 — source_type derived via a dedicated Vector transform (selected over folding into add_labels).**
- Add a NEW `remap` transform `source_type_classify` on the MAIN VL-sink path, placed AFTER `docker_severity_extract` and BEFORE `redact_main` (so docker `.label`/`.image`/`compose_*` markers are fully settled, matching the docker_severity_extract precedent).
- Marker precedence (ORDER MATTERS — cron BEFORE systemd, per user decision Q1=Both):
  1. `exists(.label) || exists(.image)` → `source_type = "docker"`
  2. `.SYSLOG_IDENTIFIER == "CRON" || .SYSLOG_IDENTIFIER == "crond" || includes(["cron.service","crond.service"], to_string(._SYSTEMD_UNIT) ?? "")` → `source_type = "cron"` (CHECKED BEFORE systemd so raw system-cron journald lines, which carry `_SYSTEMD_UNIT=cron.service`, land under "cron" not "systemd")
  3. `exists(._SYSTEMD_UNIT)` → `source_type = "systemd"`
  4. else → `source_type = "unknown"`
- ALSO set `.source_type = "cron"` inside the existing `hmrun_shaped` transform (the A-mode hmrun branch sets service="hmrun" and feeds sinks.vl on a separate path; it must carry source_type too).
- Rationale (user decision Q1): the "cron" section must include BOTH raw system cron/crond journald lines AND the A-mode hmrun wrapper output. Confirmed from config: journald fans out — raw CRON lines DO reach the main VL sink (not siphoned to cron_events), so the cron-before-systemd ordering routes them to the cron section.
- VRL gotchas apply: no lookarounds, `r'...'` regex if any, no emit_metric, `vector validate` is the gate (image timberio/vector:0.41.1-debian), config rendered at boot from template.

**D-012A-B1 — source_type is a VictoriaLogs STREAM field (selected over regular field).**
- Cardinality ~4 (docker/cron/systemd/unknown) → safe as a stream field; VL infers stream fields from the ingested JSON automatically, so NO sink/config change is required beyond setting the field. Add an explanatory comment in the template (mirroring the existing run_id/cron_fingerprint "deliberately NOT stream" comment) noting source_type IS an intended stream field.

**D-012A-C (PER-IDENTITY) — endpoint returns one ServiceCount per (service, source_type) identity; NO collapse (selected per user decision Q2: "always filter per identity").**
- `GET /api/logs/services` query changes from `* | stats by (service) count()` to `* | stats by (service, source_type) count()`.
- Response: `ServiceCount` gains a `source_type: str` field. The endpoint returns ONE ServiceCount row PER DISTINCT (service, source_type) PAIR — a service name that exists under two source_types appears as TWO rows (one per identity), each with its own count. (Do NOT collapse-to-dominant — the user explicitly chose per-identity.)
- `truncated` semantics: top-N now applies to identity rows (decide limit application in Build — likely top-`limit` identities by count).
- 30s cache key `(start,end,limit)` unchanged.
- OpenAPI regen REQUIRED (ServiceCount shape change).

**D-012A-C-FILTER (PER-IDENTITY filtering on /api/logs/query) — supersedes 012's bare-name services contract.**
- The `services` filter must target identities, not bare names. Selecting docker-`nginx` filters to `(service:"nginx" AND source_type:"docker")`, NOT just `service:"nginx"`. Multiple selections are OR'd: `((service:"a" AND source_type:"docker") OR (service:"b" AND source_type:"cron")) AND (<expr>)`.
- Backend composition still uses the canonical `logsql_quote_phrase` for BOTH the service value AND the source_type value. The no-services path stays byte-identical.
- This EXTENDS/SUPERSEDES STAGE-004-012's bare-name `services` CSV. The new param encoding is identity-qualified.

**D-012A-URL (identity-qualified URL encoding) — selected: `type:service` comma-joined.**
- URL/param format: `&services=docker:nginx,cron:hmrun,systemd:sshd.service` (each entry is `<source_type>:<service>`, comma-joined; omit when empty; omit-absent-keys).
- Assumption: service names contain no `:` or `,` (container names / `*.service` units / "hmrun" — clean in this homelab). The backend still passes the service value (the part after the first `:`) through logsql_quote_phrase so even an odd name filters safely. Parsing splits each comma entry on the FIRST `:` into (source_type, service).
- `selectedServices` state becomes a list of `{service, source_type}` identity objects (NOT bare strings). Must be added to BOTH router.tsx validateSearch AND the test harness inline validateSearch.
- NOTE: this CHANGES 012's `&services=a,b` bare-name format → 012's regression test for `&services=` will need updating to the identity-qualified format; the existing flat-name behavior is superseded. Document this in the regression/changelog at Finalize.

**D-012A-D (PER-IDENTITY grouped collapsible sidebar) — selected.**
- The StreamPickerSidebar refactors into ONE COLLAPSIBLE SECTION PER source_type, STACKED vertically (order: docker, cron, systemd, unknown-LAST; within-section by count desc, consistent with 012).
- Each section header: source_type label + aggregate count for the section + collapse/expand toggle + a select-all / select-none control for that section's listed identities.
- A service name appearing under two source_types shows as a row in EACH relevant section (per-identity, distinct counts).
- Clicking a row toggles that IDENTITY `{service, source_type}` into the selection (filters per identity per D-012A-C-FILTER).
- select-all: idempotent union of that section's CURRENTLY-LISTED identities into the selection; select-none: remove only that section's listed identities. Section checkbox shows all/some(indeterminate)/none.
- Collapse state: EPHEMERAL component state for v1 (STAGE-015 owns explorer-state persistence — note the future hook, do NOT entangle/persist now).
- Truncation: select-all selects only listed (visible) identities; surface the truncated indicator per 012's convention.
- Preserve desktop-inline / mobile-drawer (Radix Dialog) responsive behavior from 012.
