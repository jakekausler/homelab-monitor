# STAGE-004-026: Periodic batch consumer service (queries VL, feeds Drain, persists state)

## Status: Complete

## Overview

A new monitor in-process service that runs every 5 minutes (configurable via `HOMELAB_MONITOR_DRAIN_INTERVAL_S=300`), queries VL for log lines newer than its last-processed cursor, feeds them through `DrainEngine` from STAGE-004-025, and persists model state. This is the periodic batch model decided in brainstorming.

No metrics emission yet — that's STAGE-004-027.

**Vertical slice:** the monitor process runs a periodic Drain consumer. Every 5 min, it picks up new VL lines and updates Drain models. State persists across monitor restarts (resumes from cursor in SQLite).

**Spec references:** master spec §6.3 (periodic Drain clustering job).

## Concrete deliverables

### 1. `DrainConsumer` service

`apps/monitor/homelab_monitor/kernel/logs/drain_consumer.py` (new):

```python
class DrainConsumer:
    def __init__(self, vl_client: VictoriaLogsClient, engine: DrainEngine, *, interval_s: int = 300, batch_max_lines: int = 50_000):
        ...

    async def run_forever(self) -> None:
        """Periodic loop. Cancellable."""

    async def run_once(self) -> DrainCycleResult:
        """One cycle: query VL since last cursor, feed through engine, snapshot. Returns metrics + new cursor."""
```

`DrainCycleResult`:

```python
class DrainCycleResult(BaseModel):
    started_at: datetime
    finished_at: datetime
    lines_processed: int
    new_templates: int  # how many first-seen templates this cycle
    models_touched: int  # how many distinct model_keys this cycle
    cycle_status: Literal['ok', 'partial', 'failed']
    error: str | None
```

### 2. Cycle algorithm

1. Read `last_processed_ts` per model from `drain_models` table (max across all models = effective global cursor)
2. Query VL: `* | filter _time >= <cursor> | filter _time < now() | head <batch_max_lines>`
3. For each line:
   - Convert to `LogLine` via the mapper from STAGE-004-002
   - Feed through `engine.add_line(line)` — returns `SignatureEvent`
4. After all lines processed: `engine.snapshot()` persists all touched models
5. Update each touched model's `last_processed_ts` to the max timestamp seen in that model
6. Return `DrainCycleResult` for observability

### 3. Batch size cap

`batch_max_lines: int = 50_000` — if a cycle would exceed this (catching up after monitor downtime), process the first N lines and mark the cycle as `'partial'`. Next cycle picks up where this one left off.

### 4. Cancellation + shutdown

The service handles `asyncio.CancelledError` cleanly: ensures `engine.snapshot()` is called before exit so no partial-cycle progress is lost.

### 5. Service registration

Wire into the monitor's existing periodic-services framework (whatever pattern `log_stream_budget` uses). Service is named `drain_consumer`. Starts at monitor boot if drain feature is enabled (env-var-gated for safe rollout: `HOMELAB_MONITOR_DRAIN_ENABLED=1`).

### 6. Tests

- `apps/monitor/tests/test_drain_consumer.py` — mocked VL client + in-memory `DrainEngine`; cycle picks up lines, advances cursor, persists snapshots
- Resume-after-restart test: cycle 1 processes lines, snapshot; cycle 2 starts from cycle 1's cursor
- Partial-cycle test: more lines than `batch_max_lines` → status `'partial'`, next cycle continues
- Error handling: VL transport error mid-cycle → status `'failed'`, no progress lost
- Cancellation: `asyncio.CancelledError` mid-cycle → snapshot called, service exits

## Design — Chosen Approach

### Verified codebase facts driving the design
- **VL response order is NOT guaranteed and cannot be sorted server-side** (tail_service.py:91 + pagination.py:198 both sort defensively in-process; LogsQL `sort offset limit` OOMs VL per issues #129/#8127). DECISIVE for Q2: any "newest processed line" must be computed as MAX(_time) over the streamed batch, NEVER "the last yielded line."
- VL client `stream_query(*, expr, start, end, limit) -> AsyncIterator[VlLogLine]` is O(1) memory — use it for up to 50k lines/cycle (not query() which buffers). ISO-8601 UTC string windows. LogsQL time filter `_time:[start, end]`. `VictoriaLogsClientError(.status_code)`.
- The VL client does NOT enforce future-skew (only the /api/logs/query route does); the consumer anchors its own window end.
- Continuous-service template = OverrideLoader (start_task idempotent / async stop_task cancel+suppress / run_forever while-True try/except+sleep). Lifespan: env-gate → app.state → shutdown cancel+suppress.
- Engine (025): async add_line(line)->SignatureEvent (sets model.last_processed_ts per line), async snapshot() (upserts ALL loaded models incl. per-model last_processed_ts + first_seen_map). SqlitePersistence.load_state_for/persist exist; NO cursor-reader method yet.
- AppSettingsRepository.get/set(key, value:str) is a generic kv store (no migration needed for a watermark).

### Q1 — Cursor strategy: CHOSEN = single GLOBAL watermark in app_settings; per-model cursors still maintained by engine
A single cycle watermark bounds the ONE mixed-service VL query per cycle: `_time:[watermark, query_end]`. Feed all mixed lines; the engine advances each model's last_processed_ts as before (D-DRAIN-CURSOR-PER-MODEL intent preserved — every model still tracks its own last-seen for first-seen/metrics). The watermark is the QUERY BOUND; per-model cursors are bookkeeping. Stored in `app_settings` under key `drain.cycle_watermark_ms` (value = str(unix_ms)). Cold-start seed: if key absent, seed from MAX(last_processed_ts) across drain_models (resume after restart) if rows exist, else `now - ingest_lag_grace` (don't replay all history — templates bootstrap from snapshots).
- Rejected (A) global MAX(last_processed_ts) as query start — SILENTLY SKIPS lines for a lagging/quiet model (lines between its cursor and the max are never re-queried = data loss).
- Rejected (B) per-model separate VL queries — cron:<sha256(canonical_log_key(command))[:16]> keys are IRREVERSIBLE, so a per-service LogsQL filter cannot be built; also N queries/cycle.
- Reconciliation with D-DRAIN-CURSOR-PER-MODEL: per-model cursors still EXIST and advance; the global watermark is an ADDITIVE query-bound mechanism, not a replacement. Documented divergence (a quiet model's cursor lags the watermark) is CORRECT, not a bug.

### Q2 — Partial-cycle mechanics + 50k cap + ingestion lag: CHOSEN = count-based partial + lag margin
- Track `count` (lines streamed) and `max_ts_seen` = MAX(_parse_iso_ms(line.timestamp)) over EVERY streamed line (MAX over batch — NOT last yielded, because VL order is arbitrary).
- `count == batch_max_lines` → PARTIAL: set next watermark = max_ts_seen (resume from newest processed line; the unprocessed tail in (max_ts_seen, query_end] is re-queried next cycle).
- `count < batch_max_lines` → COMPLETE: set next watermark = query_end.
- `count == 0` → complete; advance to query_end.
- Ingestion-lag safety margin: `query_end = now_utc - INGEST_LAG_GRACE` (default 30s, env HOMELAB_MONITOR_DRAIN_INGEST_LAG_GRACE_S, clamp max(x,0)). Never query/advance into the lag zone, so late-arriving lines (VL/vector ingestion lag) with _time < now are not skipped. Mirrors the route's FUTURE_SKEW_GRACE philosophy (larger, since ingestion lag > clock skew).
- Rejected (B) always watermark=max_ts_seen — a complete cycle would re-query from the newest line's ms every cycle forever (perpetual boundary re-processing + line_count inflation).
- Rejected overlap-re-query lag handling — per-cycle line_count inflation.

### Q3 — DrainCycleResult + run_once algorithm + error boundary: CHOSEN
```python
@dataclass(frozen=True, slots=True)
class DrainCycleResult:
    started_at: int          # unix-ms
    finished_at: int         # unix-ms
    lines_processed: int
    new_templates: int       # count of is_new events this cycle
    models_touched: int      # distinct model_keys seen
    cycle_status: Literal["ok", "partial", "failed"]
    error: str | None
```
run_once(): started_at=now_ms → read watermark (seed if absent) → query_end_ms = now_ms - lag_grace_ms; if query_end_ms <= watermark → early-return ok/0-lines → build [start_iso,end_iso] + match-all expr ("*") → try stream_query: per line from_victorialogs_line + engine.add_line, tally count/max_ts_seen/new_templates(+=is_new)/models_seen.add(model_key) → except VictoriaLogsClientError: engine.snapshot() (persist lines fed before failure), DO NOT advance watermark, return cycle_status="failed"+error → else engine.snapshot(); partial = count==batch_max_lines; next_wm = max_ts_seen if partial else query_end_ms (query_end_ms if count==0); write watermark; return "partial"/"ok".
- Error boundary: run_once is SELF-CONTAINED for VL/transport errors (catch VictoriaLogsClientError, snapshot, leave watermark unadvanced so next cycle retries the same window — drain idempotent). run_forever's `except Exception` is a last-resort backstop only (pragma-covered like OverrideLoader, OR a real test via run_once side_effect [Exception, CancelledError] — Build decides; a real test is preferred to avoid the pragma).

### Q4 — Cancellation → snapshot guarantee: CHOSEN = per-cycle snapshot only, NO snapshot-on-cancel
A mid-cycle cancel skips both snapshot() and the watermark write. Verified safe: add_line mutates ONLY in-memory _Model state; nothing durable until snapshot(); the watermark advances STRICTLY AFTER snapshot(). So a cancel mid-cycle leaves durable state at the last completed cycle — next process start re-queries the identical window, drain re-clusters idempotently, first_seen restored from last good snapshot, per-model cursors restored from drain_models. NO DATA LOSS, only recomputation. No first_seen drift (first_seen restored before replay; add_line only writes first_seen[hash] when absent). run_forever re-raises CancelledError naturally (propagates from asyncio.sleep / await run_once — no explicit catch). stop_task mirrors OverrideLoader (cancel + contextlib.suppress(CancelledError)).
- Rejected (A) catch CancelledError + snapshot before re-raise — async DB work in a cancel handler is fragile (loop tearing down, DB connection closing, second cancel can interrupt the snapshot), hard to test, and provides ZERO real benefit given the watermark-not-advanced replay safety.

### Q5 — Testing the infinite run_forever loop at 100% coverage: CHOSEN
- Test run_once() DIRECTLY for every scenario: cold-start seed, resume-after-restart, partial (count==limit→watermark=max_ts_seen), complete (count<limit→watermark=query_end), empty (count==0), VL-failure (cycle_status="failed", watermark unchanged), early-return (query_end<=watermark). Fake DrainPersistence/AppSettingsRepository + fake VL client (async generator yielding canned VlLogLines or raising VictoriaLogsClientError).
- Test run_forever via `monkeypatch.setattr(asyncio, "sleep", AsyncMock(side_effect=[None, CancelledError()]))` then `pytest.raises(CancelledError)` + assert run_once called twice (covers try, sleep, cancel-propagation).
- Backstop except-Exception branch: run_once AsyncMock(side_effect=[RuntimeError, CancelledError]) → assert loop survived the exception (2nd call happened) + logged. (Or mirror OverrideLoader's `# pragma: no cover -- defensive` on the backstop; a real test is preferred.)
- Rejected (B) inject test-only max-iterations/stop-event into run_forever — pollutes production signature, diverges from OverrideLoader.
