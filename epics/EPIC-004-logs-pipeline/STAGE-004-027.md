# STAGE-004-027: Metrics emission + manual refresh API

## Status: Complete

## Overview

Emit the Drain signature metrics into VictoriaMetrics so vmalert rules (later stages) can act on them. Also expose `POST /api/logs/signatures/refresh` for users to trigger an out-of-cycle Drain consumer run from the UI (used by the Signature catalog UI from STAGE-004-028).

**Vertical slice:** after each Drain cycle, the following metrics land in VM:

- `homelab_log_signature_count{service_key, template_hash, severity}` — count of lines per template per cycle
- `homelab_log_signature_first_seen_ts{service_key, template_hash}` — gauge = nanosecond timestamp of first emission
- `homelab_log_signature_total{service_key, template_hash}` — cumulative count across all cycles
- `homelab_drain_cycle_duration_seconds` — histogram of cycle durations
- `homelab_drain_cycle_lines_total` — counter of all lines processed
- `homelab_drain_cycle_new_templates_total` — counter of first-seen templates across all cycles

The user can hit `POST /api/logs/signatures/refresh` to force an immediate cycle.

**Spec references:** master spec §6.3.

## Concrete deliverables

### 1. Metric emission in `DrainConsumer`

`apps/monitor/homelab_monitor/kernel/logs/drain_consumer.py` — extend the cycle to emit metrics:

After each cycle (in `run_once`):
- For each `(service_key, template_hash, severity)` tuple seen in the batch: emit `homelab_log_signature_count` with the count
- For each first-seen template: emit `homelab_log_signature_first_seen_ts` gauge with the timestamp
- Update `homelab_log_signature_total{service_key, template_hash}` (cumulative)
- Emit `homelab_drain_cycle_duration_seconds` (histogram observation)
- Emit `homelab_drain_cycle_lines_total` += `lines_processed`
- Emit `homelab_drain_cycle_new_templates_total` += `new_templates`

Uses the project's existing `MetricsWriter` (from STAGE-001-015).

### 2. Manual refresh endpoint

`apps/monitor/homelab_monitor/kernel/api/routers/logs.py`:

```
POST /api/logs/signatures/refresh
Response 202 Accepted: {cycle_id: str, queued: true}
Response 409 Conflict: {error: "cycle_in_progress", cycle_started_at: ...}
```

Implementation:
- Signal the consumer service to run a cycle (don't block the HTTP request on completion — return 202 with a cycle_id and let the cycle run asynchronously)
- If a cycle is already running, return 409 with details
- CSRF required (POST endpoint)

### 3. Cycle status endpoint

`GET /api/logs/signatures/refresh/{cycle_id}` — returns the cycle's result once finished (uses the `DrainCycleResult` shape from STAGE-004-026). Polled by the UI's manual-refresh button to show progress.

Stored ephemerally in-process (or a small SQLite table if we want post-restart query — Design decides; recommendation: in-process map with 1h TTL).

### 4. Cardinality safeguards

`template_hash` cardinality is bounded by `max_clusters` per model × number of models. Worst case 5000 × 50 = 250k unique series, which is fine. But to avoid runaway cases:

- Emit a warning metric `homelab_log_signature_cardinality_warn` when total active signatures exceeds 100k
- Optional: drop low-coverage templates from emission if total exceeds a hard cap (configurable, default off; Design decides)

### 5. Tests

- Unit: metric emission per cycle (mocked MetricsWriter)
- Integration: spawn the consumer in a test rig; query VM for emitted metrics; assert shape + values
- Endpoint tests: 202 / 409 / status polling

## Out of scope for this stage

- Anomaly rules using these metrics — STAGE-004-035+
- Signature catalog UI — STAGE-004-028, 029
- Per-signature retention — locked-defer

## Decisions in this stage

### Design phase will confirm

- **Cycle status persistence** — in-process map (recommended) vs SQLite
- **TTL on cycle status** — 1h recommended
- **Cardinality hard-cap behavior** — drop low-coverage templates? Recommendation: off in v1, alert only
- **Emit `severity` label or not** — recommended yes; it's already on the line

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-METRICS-EMISSION-AFTER-EACH-CYCLE** — emitted at cycle boundary, not per-line.
- **D-MANUAL-REFRESH-202-ASYNC** — refresh API returns 202 immediately; cycle runs in background; status polled separately.
- **D-FIRST-SEEN-METRIC** — `homelab_log_signature_first_seen_ts` enables "new signature detected" anomaly rule (STAGE-004-035).

## Design — Chosen Approach

### Verified facts driving the design
- MetricsWriter (kernel/plugins/io.py): write_gauge → absolute .set(); write_counter → additive .inc(); write_summary → histogram-ish. ctx.vm (collectors) and app.state.metrics_writer are the SAME MultiplexMetricsWriter. GAUGE CLOBBER happens ONLY when the SAME name+labels is written by TWO sources — different label-VALUES are distinct series. So the consumer (a SINGLE source) writing gauges with unique (service_key, template_hash) labels does NOT clobber → it can emit gauges DIRECTLY inline, no separate collector. InMemoryMetricsWriter exists as a test double.
- DrainConsumer.run_once() loop already has BOTH `line` (→ line.severity, normalized lowercase or None) AND `event: SignatureEvent` (→ cluster_size [drain3 CUMULATIVE across cycles via persisted snapshot], is_new, template_hash, model_key, first_seen_ts) in scope per line. This is the single source for ALL per-signature metrics — no engine iterator needed.
- DrainConsumer.__init__ has no metrics_writer (027 adds metrics_writer: MetricsWriter). run_once ends: snapshot() → partial/next_watermark → settings.set → return DrainCycleResult. Metrics emit after snapshot, before return.
- run_once has NO re-entrancy guard; periodic loop + manual refresh both call it → concurrent cycles corrupt the watermark. 027 MUST add a guard.
- API: ConflictProblem(409) exists; require_session() auto-enforces CSRF on POST; _require_state pattern for get_drain_consumer (503 if disabled); Pydantic response models in kernel/api/schemas.py (ConfigDict(extra="forbid")); DrainCycleResult is a frozen dataclass → needs a Pydantic mirror. asyncio.create_task for background cycle. NO existing async-job/status-poll pattern → 027 builds it.

### Q1 — Re-entrancy guard: CHOSEN = internal asyncio.Lock inside run_once + endpoint pre-check + typed CycleInProgressError
Add `self._cycle_lock = asyncio.Lock()` (held for the cycle DURATION). run_once: `if self._cycle_lock.locked(): raise CycleInProgressError(started_at=...)` then `async with self._cycle_lock: <body>`. RACE-FREE because there is NO `await` between the .locked() check and the `async with` acquire — asyncio's single-threaded loop yields only at await, so check-then-acquire is atomic. The periodic run_forever calls the same run_once; if a manual cycle is mid-flight it raises CycleInProgressError → run_forever catches it (explicitly, log at debug not via the generic exception backstop) → that periodic tick is skipped (desired: no overlap). The endpoint also calls a synchronous `consumer.is_cycle_running()` (returns self._cycle_lock.locked()) BEFORE create_task to return 409 synchronously; the internal guard is the authoritative race-closer (belt-and-suspenders: if the pre-check passes but a periodic cycle starts same-tick, the backgrounded run_once raises → recorded as failed in the status store).
- Rejected (A) serialize-all Lock inside run_once where manual QUEUES behind periodic — can't return 409, violates the card's 409-if-already-running contract (it queues instead of rejecting).
- Rejected (C) separate run_once_locked wrapper + endpoint checks lock.locked() then create_task — real TOCTOU window (check and create_task separated; create_task acquires the lock LATER, two POSTs both see unlocked → second blocks instead of 409).

### Q2 — Metrics emission location: CHOSEN = inline in run_once after snapshot()
The consumer holds metrics_writer and emits all 6 metrics + the cardinality-warn gauge inline after snapshot(), before return.
- Rejected: a separate DrainMetricsCollector polling on its own interval — DECOUPLES from the cycle boundary (violates D-METRICS-EMISSION-AFTER-EACH-CYCLE) and per-cycle counters can't be .inc()'d by a polling collector without double-counting.
Emission map (after snapshot):
- write_counter("homelab_drain_cycle_lines_total", count, {}) — .inc() by this cycle's lines.
- write_counter("homelab_drain_cycle_new_templates_total", new_templates, {}) — .inc().
- write_summary("homelab_drain_cycle_duration_seconds", (finished_at-started_at)/1000.0, {}).
- write_gauge("homelab_log_signature_count", cycle_count, {service_key, template_hash, severity}) — per-cycle count.
- write_gauge("homelab_log_signature_total", cluster_size, {service_key, template_hash}) — cumulative (= event.cluster_size).
- write_gauge("homelab_log_signature_first_seen_ts", first_seen_ts*1e6, {service_key, template_hash}) — ns (ms→ns).
- write_gauge("homelab_log_signature_cardinality_warn", 0|1, {}).
Emit on ok/partial/failed (failed: emit what was processed before the VL error, consistent with the existing snapshot-on-failure). Empty/early-return cycle: emit cycle counters/duration (0 lines), skip per-signature gauges.

### Q3 — severity label + signature_count value: CHOSEN = per-line severity counted in the consumer loop
The consumer loop already sees line.severity per line alongside event.template_hash. Maintain `cycle_counts: dict[(model_key, template_hash, severity), int]` (severity = line.severity or "unknown"), increment per line, emit one signature_count gauge per tuple after snapshot. NO engine change, NO new persistence. A mixed-severity template naturally produces multiple series (correct — that's the label's purpose). Value = per-cycle line count (resets each cycle = card's "lines per template per cycle").
- Rejected (A) drop the severity label — loses the locked dimension; severity is already on the line.
- Rejected (B) persist per-template severity histograms in the engine — real new engine state + persistence + churn; over-engineered (YAGNI); catalog (028) doesn't need it.

### Q4 — signature_count vs signature_total semantics: CHOSEN = both from SignatureEvent, no new engine state, emit only templates touched this cycle
- signature_total{service_key,template_hash} = event.cluster_size (drain3's cluster size ALREADY accumulates across cycles via the persisted snapshot — inherently cumulative). .set() as a gauge.
- signature_count{...,severity} = per-cycle DELTA (lines matched THIS cycle) from the Q3 cycle_counts accumulator. Distinct from total; both useful (count = current activity/rate; total = lifetime prevalence).
- first_seen_ts = event.first_seen_ts (* 1e6 → ns). Already on SignatureEvent (durably backed by drain_models first_seen_map from 025).
- Emit gauges ONLY for templates TOUCHED this cycle (sourced from the event stream). This (1) avoids needing an all-models iterator entirely (NO all_templates() method — supersedes the handoff's assumption), (2) bounds emitted cardinality to active signatures, (3) VM staleness handling keeps an untouched signature's last gauge value until it goes stale = the correct "signature went quiet" signal for STAGE-035. Loop maintains two maps: cycle_counts[(model,hash,severity)]→count and sig_state[(model,hash)]→(cluster_size, first_seen_ts).
- Rejected: adding an all_templates() engine iterator + full periodic re-emit of quiet signatures — unnecessary now (YAGNI); add when a later stage needs it.

### Q5 — cycle-status store + TTL + cardinality safeguard: CHOSEN
Store: a `CycleStatusStore` class on app.state (wired in lifespan, injected via a get_cycle_status_store dependency using _require_state). NOT a module-level dict (global state, hard to reset in tests). Shape: CycleStatusEntry(status: Literal["running","done","failed"], created_ms, result: DrainCycleResult|None, error: str|None); methods begin(cycle_id)/complete(cycle_id, result)/fail(cycle_id, error)/get(cycle_id) (prune-on-access). TTL=1h, injected clock (deterministic, no background sweeper).
- POST /api/logs/signatures/refresh: if consumer.is_cycle_running() → 409 ConflictProblem {cycle_in_progress, cycle_started_at}. Else: cycle_id=uuid4().hex, store.begin(cycle_id), asyncio.create_task(_run_and_record), return 202 {cycle_id}. The background wrapper _run_and_record calls store.complete/fail (and translates a CycleInProgressError from the internal guard → store.fail(cycle_id,"cycle_in_progress") so the rare TOCTOU loser is observable).
- GET /api/logs/signatures/refresh/{cycle_id}: unknown/expired → 404 NotFoundProblem; running → 200 {status:"running"}; done → 200 {status:"done", result: DrainCycleResultResponse}; failed → 200 {status:"failed", error}. (200+body-flag for in-flight, NOT 202 — 202 is the POST-accepted semantic; the poll is a normal GET.)
- Pydantic models in schemas.py: RefreshCycleResponse{cycle_id}, RefreshStatusResponse{status, result?: DrainCycleResultResponse, error?}, DrainCycleResultResponse (mirror of the DrainCycleResult dataclass).
- Cardinality: emit homelab_log_signature_cardinality_warn gauge = 1 when distinct signatures this cycle > threshold (DrainConfig field, default 100_000), else 0 (self-clearing), + log.warning on the rising edge. NO hard cap / no template dropping in v1 (card says optional/Design-decides → defer, YAGNI).
