# STAGE-004-009: Local-time timestamp rendering with UTC toggle in `<LogViewer>`

## Status: Finalize

## Overview

VL stores timestamps in UTC. Today the log viewers render them as raw UTC strings. Per master spec ("UTC internal, local rendering for America/New_York, configurable"), and consistent with the rest of the UI's timestamp behavior, log timestamps should render in the configured local timezone by default with a UTC toggle.

Tooltip on every timestamp shows the OTHER-format equivalent (hover a local-time stamp → see the UTC equivalent). Toggle is persisted in user preferences (or in component state — Design decides which scope).

This applies once across `<LogViewer>` and propagates automatically to docker viewer, cron viewer, and (once it lands) the Logs Explorer.

**Vertical slice:** open any log viewer, see local-time timestamps. Click the UTC toggle in the viewer header → all timestamps re-render in UTC. Toggle persists across navigation.

**Spec references:** master spec timestamp policy.

## Critical Implementation Note

**UTC→local timezone conversion MUST cover BOTH surfaces:**

1. **Each log ROW's timestamp** — the per-line timestamp column in `<LogViewer>` / `LogLineList` (the `<span>{ts}</span>` part of each rendered line).
2. **The header timestamp displays** — the "last: <timestamp>" summary in `DockerContainerLogsViewerBody.tsx`, cron viewer header timestamp(s), and any equivalent timestamps in future log-display surfaces.

**Context:** As of STAGE-004-003, both surfaces render a FRIENDLY but still-UTC timestamp (nanoseconds trimmed, `T`/`Z` noise removed, suffixed " UTC"). This stage's job is to add the actual timezone SHIFT (UTC→America/New_York, configurable) on top of that friendly formatting for BOTH per-row timestamps AND header timestamps. The friendly-UTC formatting helper introduced in STAGE-004-003 (likely `formatLogTimestamp` in `apps/ui/src/lib/` or similar) is the integration point; 004-009 layers the zone conversion into/around it.

**Future implementer checkpoint:** If you touch log timestamps but header timestamps still show UTC (or vice versa), this stage is incomplete.

## Concrete deliverables

### 1. `<LogViewer>` accepts `timezone` prop (already in STAGE-004-003 spec)

Refine the prop to: `timezone: 'local' | 'utc'` (default `'local'`).

### 2. Toggle control in viewer header

A small `<LocalUtcToggle>` icon-button in the `<LogViewer>` header (or in the caller's `headerSlot` — Design phase decides; recommendation: bake it into the component so every consumer gets it for free).

### 3. Timestamp rendering helper

`apps/ui/src/lib/timestampFormat.ts` (new or extend existing):

```typescript
function formatLogTimestamp(
  ts: Date,
  timezone: 'local' | 'utc',
  configuredLocalTz: string = 'America/New_York'
): { display: string; tooltip: string } {
  // display: the visible string in the chosen timezone
  // tooltip: the equivalent in the other timezone, for hover reveal
}
```

Display format: `YYYY-MM-DD HH:mm:ss.SSS` (with timezone suffix when UTC: `Z`; when local: TZ abbreviation or offset). Design phase locks the exact format.

### 4. Persistence scope

Two options:
- **Option A: per-viewer-instance** — toggle state lives in the component; navigating away resets to default
- **Option B: persistent user preference** — store in localStorage (or a future user-prefs server endpoint); applies across viewers

Recommendation: **Option B** with localStorage. Cheap, low-risk, sticky behavior matches user expectation.

### 5. Tests

- `apps/ui/src/lib/__tests__/timestampFormat.test.ts` — formatting in both timezones, DST boundary, midnight rollover edge cases, with a fake clock
- `apps/ui/src/components/logs/__tests__/LogViewer.test.tsx` — timezone toggle changes rendered timestamps; tooltip shows the other format

## Out of scope for this stage

- Server-side timezone configuration UI — uses `America/New_York` as a constant for v1; configurable env var if needed
- Per-line timezone override (some log lines might carry their own TZ data) — YAGNI
- Time format customization (12h vs 24h) — fixed at 24h ISO-ish

## Decisions in this stage

### Design phase will confirm

- **Persistence scope (A vs B)** — recommendation B
- **Display format details** — `YYYY-MM-DD HH:mm:ss.SSS TZ`? With or without millisecond precision? Recommendation: include ms because logs cluster densely
- **Toggle location** — bake into `<LogViewer>` (recommended) vs leave to caller via headerSlot
- **TZ abbreviation source** — Intl.DateTimeFormat with browser locale; tests mock `Intl` for deterministic output

### Locked Design Decisions (2026-05-30)

**Recommended approach confirmed: extend existing shared formatter — no parallel utility.**

- **D009-1 (Extend the EXISTING shared formatter — no parallel utility):** Extend `formatLogTimestamp` in `apps/ui/src/lib/relativeTime.ts` to `formatLogTimestamp(raw: string | null | undefined, opts?: { timezone?: 'local' | 'utc'; tz?: string }): string`. Back-compatible: no opts OR `timezone:'utc'` → the existing regex fast-path, returns the current `YYYY-MM-DD HH:MM:SS UTC` text UNCHANGED (honors STAGE-004-003's "keep signature and return shape stable" — existing bare `formatLogTimestamp(raw)` call sites keep working). `timezone:'local'` → `Date.parse(raw)` + `Intl.DateTimeFormat('en-CA', { timeZone: tz ?? DEFAULT_DISPLAY_TZ, hour12: false, year/month/day/hour/minute/second })`, re-append `.SSS` (millisecond, captured from the regex; nanoseconds beyond ms dropped for display — in scope per card) and a short zone label (e.g. EDT/EST via `timeZoneName:'short'`). On `Date.parse` NaN / non-match → return `raw` unchanged (same as today). Optionally add a SAME-FILE thin wrapper `formatLogTimestampParts(raw, opts) -> { display: string; tooltip: string }` that calls the formatter twice (once for the active timezone = display, once with the FLIPPED timezone = tooltip's other-format). This is NOT a parallel utility — it composes the one shared formatter.

- **D009-2 (Shared persistence hook):** New `useTimezonePreference()` hook in `apps/ui/src/lib/` — backed by localStorage key `homelab-monitor:timezone`, default `'local'`, SSR-safe lazy initializer (typeof window guard, mirroring AppShell's `readInitialTheme`) + useEffect write-back. Returns a tuple `['local' | 'utc', toggleFn]`. This is the shared source of truth so BOTH the per-row timestamps AND the caller-rendered header timestamp read the same preference.

- **D009-3 (TimezoneToggle in headerSlot, not baked into LogViewer):** New `TimezoneToggle` primitive (clone `WrapToggle.tsx`'s native-checkbox styling; label "UTC", checked = utc). Each of the 3 viewers (docker, cron, future Explorer) calls `useTimezonePreference()`, renders `<TimezoneToggle>` in its `headerSlot` next to the existing `<WrapToggle>`, and passes the `timezone` value into `<LogViewer>`. RATIONALE for not baking the toggle+state inside `<LogViewer>` (which the card literally suggested): the header "Last: <ts>" timestamp is rendered by the CALLER inside `headerSlot`, OUTSIDE LogViewer's component tree — so LogViewer-internal state could not convert the header timestamp. The shared-hook + per-viewer-toggle pattern mirrors the existing `wrap` prop wiring and keeps header + rows in lockstep. This satisfies D-PROPAGATE-VIA-LOGVIEWER in spirit (one toggle primitive + one hook propagate to all surfaces).

- **D009-4 (Propagation + tooltip):** `LogViewer` forwards its existing (currently-inert) `timezone?: 'local'|'utc'` prop (types.ts:40) → `LogLineList` → the per-row render, which calls `formatLogTimestamp(line.timestamp, { timezone })` for the displayed text and renders the FLIPPED-timezone result in a native `title=` attribute on the timestamp span (per-row tooltip = the other format). Native `title=` chosen over a shadcn Tooltip component for per-row PERFORMANCE (log lists render hundreds of rows; per-row Tooltip portals/listeners are a real DOM/perf cost). The header "Last:" timestamp (DockerContainerLogsViewerBody.tsx:150) calls the same formatter with the same `timezone` from the hook → header and rows stay consistent. Both per-row AND header surfaces convert (satisfies the locked CRITICAL note).

- **D009-5 (Configurable-zone seam):** `export const DEFAULT_DISPLAY_TZ = 'America/New_York'` in relativeTime.ts with a `// configurable: future env var` comment. The formatter's `tz` param defaults to it. YAGNI — do NOT wire a Vite env var now; the single constant is the clean seam (swap the literal for an env read later with no signature change).

- **D009-6 ('local' = configured zone, NOT browser zone):** `'local'` MUST mean the CONFIGURED zone (America/New_York), achieved via the explicit `Intl.DateTimeFormat(..., { timeZone: DEFAULT_DISPLAY_TZ })` option — NOT bare `toLocaleString()` / `new Date().toLocaleString()` which use the BROWSER's zone (a California viewer of this homelab must still see Eastern, per master spec "local rendering for America/New_York"). Flag this in a code comment so it isn't "simplified" away.

**Render sites to update** (both surfaces, per the locked CRITICAL note): per-row `apps/ui/src/components/logs/LogLineList.tsx` (currently `formatLogTimestamp(line.timestamp)` ~line 43) AND header `apps/ui/src/routes/integrations/DockerContainerLogsViewerBody.tsx` ("Last: …" ~line 150). Cron viewer (`CronRunLogViewer.tsx`) gets the toggle + passes `timezone` to LogViewer (its rows convert); it has no separate header timestamp to convert.

**Display zone:** hardcoded `America/New_York` for v1 (env-var configurable later — NOT a user UI setting this stage, NOT a backend field).

**Out of scope:** per-line tz override; 12h/24h customization (fixed 24h); server-side tz config UI.

**Refinement:** 3a only (frontend; no host-integration) → needs USER manual Desktop + Mobile approval.

**Downstream:** STAGE-004-010 (Logs Explorer) consumes `<LogViewer>` and expects this toggle already integrated — no extra tz work in 010.

**Acceptance test:** `apps/ui/src/components/logs/__tests__/LogViewer.test.tsx` — toggle changes rendered timestamps; tooltip shows the other format. Plus a `formatLogTimestamp` unit test covering both timezones, DST, and midnight edges.
