# STAGE-004-043A: User Alert Rules management UI + prod vmalert dry-run validation

## Status: Complete

## Overview

Two deferred concerns from STAGE-004-043, bundled into one stage:

1. **User-rules management UI** — a surface to LIST, view, enable/disable, edit, and DELETE the user-authored alert rules created via the CreateAlertModal (STAGE-043). Today the rules are write-only from the UI: the create modal exists but there is NO way to see or manage created rules. The backend (STAGE-042) already has full CRUD (`GET/POST/PATCH/DELETE /api/logs/user-rules` + enable/disable). This stage adds the frontend management surface — naturally a new tab in the Logs area (`/logs/user-rules` or similar), mirroring the existing `SilenceAllowlistTab` pattern (list + inline enable/disable/edit/delete). "Edit" should reuse the STAGE-043 CreateAlertModal (it already accepts initialValues + initialMode). Also surface each rule's vmalert load HEALTH (ok/err + lastError) so a user can see if a rule failed to load — this is where the dry-run validation (below) and the rule-health read belong.

2. **Prod vmalert -dryRun exact-parser validation** — STAGE-043 added a HEURISTIC backend expr validator (`expr_validate.py`) that catches the common vmalert-reject patterns (reserved-word filter field, missing stats pipe, unbalanced quotes/parens). It is env-portable (pure Python) but INCOMPLETE — a novel invalid expr can slip through the heuristic. This stage adds the EXACT-parser layer: validate the expr against real vmalert via `vmalert -dryRun -rule.validateExpressions` as a best-effort gate when docker + the vmalert image are available (prod), gracefully skipped in dev/test/CI (where the heuristic remains the floor). The monitor container has the docker socket + docker CLI (verified). This belongs with the rule-health surfacing (both answer "what does vmalert actually think of this rule").

**Spec references:** master spec §9.2 ("user rule management"); EPIC-004 plan #7 (deferred from 043 Refinement).

## Concrete deliverables

### 1. User Alert Rules management tab

`apps/ui/src/routes/logs/UserRulesTab.tsx` (new) — new route in the Logs area alongside Explorer, Signatures, etc. Displays:
- List of all user rules (rule_name, expr_kind, severity, enabled toggle, vmalert health)
- Per-rule actions: enable/disable toggle, edit (reuse CreateAlertModal with `initialValues` + `initialMode`), delete (with confirm)
- Empty state when no rules exist
- Inline vmalert health per rule: green "ok", red "err <lastError>" with a tooltip

### 2. Rule-health aggregation endpoint (backend)

`GET /api/logs/user-rules-health` (new) — returns rule-health for all user rules by querying vmalert's `/api/v1/rules` endpoint (proxied via the monitor's vmalert proxy, or direct http call if vmalert is reachable):

```typescript
RuleHealthResponse = {
  rules: {
    [rule_name: string]: {
      health: "ok" | "err"
      lastError?: string
      datasourceType?: string  // "vlogs" for logsql, "prometheus" for metricsql
    }
  }
}
```

Graceful fallback: if vmalert is unreachable, return `{rules: {}}` (empty health data; UI shows "unknown").

### 3. vmalert `-dryRun -rule.validateExpressions` exact-parser validation (backend)

Add to the POST/PATCH user-rules endpoints: when docker + vmalert image are available, invoke `vmalert -dryRun -rule.validateExpressions -rule=<temp-rule-file>` to validate the expr against the exact parser. On failure (nonzero exit or `error` in stderr), reject with 400 `invalid_expr` + the vmalert error message.

**Graceful fallback:** If docker is unreachable or vmalert image is missing, skip the dry-run (rely on the heuristic validator from STAGE-043). The create/update path MUST NOT break dev/test/CI/host backend.

### 4. Reuse CreateAlertModal for edit

Modify CreateAlertModal (`apps/ui/src/components/logs/CreateAlertModal.tsx`) to accept `initialMode` (create vs edit) + `initialValues` with a pre-existing rule ID, and support PATCH instead of POST on save.

### 5. Tests

- Frontend: UserRulesTab list rendering, enable/disable toggle, edit relaunch, delete confirm, empty state
- Frontend: rule-health display with ok/err states
- Backend: rule-health endpoint with realistic vmalert responses
- Backend: dry-run validator with docker available (happy path + error cases)
- Backend: dry-run validator with docker unavailable (graceful skip, heuristic-only)
- Integration: create rule → see in list → enable/disable → edit → delete (full flow)

## Out of scope for this stage

- Rule history / audit log of rule changes
- Bulk rule import/export
- Auto-generated rules (still user-curated only per the epic)
- Advanced rule-health metrics (execution time, fire count, etc.)

- ⚠️ **CARD CLAIM CORRECTED — D-MGMT-TAB-REUSES-MODAL premise was FALSE.** `CreateAlertModal` currently supports CREATE/POST ONLY. `initialMode`/`initialValues` merely prefill the form + pick the starting tab — there is NO `ruleId` prop, NO PATCH branch, NO edit-mode self-exclusion on the duplicate-name check. Edit support must be BUILT (this expands scope; see decision 3). The stage remains coherent.

**Chosen approaches (decided in Design per standing instructions — no user prompt):**

**Decision 1 — Rule-health surface: dedicated endpoint.**
- Chosen: a new `GET /api/logs/user-rules-health` returning `{rule_name: {health, lastError}}`, fanned out to BOTH vmalert instances (logs + metrics), filtered to the `user-rules-logs`/`user-rules-metrics` groups. Degraded-200 contract: when a vmalert instance is unreachable, return that rule's health as `"unknown"` (or omit) — NEVER 500. vmalert URLs from env (`HOMELAB_MONITOR_VMALERT_LOGS_URL` default `http://vmalert-logs:8880`, `HOMELAB_MONITOR_VMALERT_METRICS_URL` default `http://vmalert-metrics:8880`), mirroring the karma `_karma_url()` env pattern + the `logs_window`/`signature_samples` degraded-200 precedent.
- UI: new `useUserRulesHealth()` react-query hook with `refetchInterval` ~20s PLUS invalidation after any rule mutation; the management tab joins health to rules by `rule_name`.
- Rejected: (b) folding health into `GET /api/logs/user-rules` — would couple list latency (incl. the modal's own `useUserRules()` uniqueness check + every post-mutation invalidation) to vmalert health; violates "list = pure DB read". (c) thin generic vmalert proxy + client-side join — leaks sidecar topology + JSON shape + both-instance fan-out into the FE.

**Decision 2 — vmalert -dryRun exact-parser validation: repo seam + DI runner, env-gated.**
- Chosen: run the dry-run INSIDE the repo at the `_validate_and_render_check` seam, ordered AFTER `render_yaml` (the heuristic `validate_expr` stays the cheap always-on floor before it). Inject a `dryrun_runner: DryRunRunner | None = None` callable; the router constructs the real runner and passes it down, unit tests pass a fake or omit it. The real runner writes the rendered rule YAML (with `type: vlogs` for logsql) to a tempfile and runs `docker run --rm victoriametrics/vmalert:v1.107.0 -dryRun -rule.validateExpressions -rule=<tmpfile>` (exit 0 = valid, exit 1 + stderr = invalid). The runner does availability detection (`shutil.which("docker")` + image-present probe + socket reachable) and returns a "skipped" sentinel on ANY miss / `TimeoutExpired` / `OSError` — NEVER raises. Only a clean exit-1 with stderr maps to `ExprValidationError(check="dryrun")` → existing 400 `invalid_expr` (stderr in `details`). Whole feature gated behind `HOMELAB_MONITOR_VMALERT_DRYRUN_ENABLED` (default OFF in CI/tests, ON in prod) — the operational kill switch; the DI runner is the primary test seam.
- This is the only seam with the fully-merged candidate for BOTH POST and PATCH (the repo owns the PATCH merge; that's why rule_name/expr_kind are immutable).
- LOAD-BEARING: create/update path + ALL unit tests pass with NO docker and NO reachable vmalert. DI lets us unit-test both "fake runner exit-1 → 400 invalid_expr" and "runner skipped → persist succeeds" with zero docker.
- Rejected: (b) validating in the router — must re-derive the PATCH merge and wire into both endpoints. (c) env-flag-only test seam — can't assert the failure-mapping without a fake runner.

**Decision 3 — CreateAlertModal edit-mode: additive `editRuleId?` prop.**
- Chosen: add `editRuleId?: number` prop (seed via the existing `initialValues`/`initialMode` open-time reset effect). Branch submit to a new `usePatchUserRule(ruleId)` hook when `editRuleId` is set, sending only the 6 mutable fields (expr/severity/summary/description/for_duration/enabled). Render `rule_name` + `expr_kind` read-only/disabled in edit mode (backend treats them immutable). The duplicate-name `existingNames` set excludes the rule's own row (`rules.filter(r => r.id !== editRuleId)`). Dialog title "Edit alert rule" / submit "Save changes" vs "Create alert", keyed on `editRuleId`. Create path stays byte-stable (044 reuses it).
- Rejected: (b) separate `EditAlertModal` — requires extracting ~300 lines of shared form body first, high regression surface on the create path 044 depends on. (c) discriminated `{mode:'create'} | {mode:'edit',...}` union — forces churn at every call site + collides with the existing `initialMode`/`initialValues` props.

**New hooks required in `apps/ui/src/api/userRules.ts`** (currently only `useUserRules` + `useCreateUserRule`): `usePatchUserRule`, `useDeleteUserRule`, `useEnableUserRule`, `useDisableUserRule`, `useUserRulesHealth` — mirror `apps/ui/src/api/silenceAllowlist.ts`.
