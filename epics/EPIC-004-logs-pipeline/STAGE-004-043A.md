# STAGE-004-043A: User Alert Rules management UI + prod vmalert dry-run validation

## Status: Not Started

## Overview

Two deferred concerns from STAGE-004-043, bundled into one stage:

1. **User-rules management UI** — a surface to LIST, view, enable/disable, edit, and DELETE the user-authored alert rules created via the CreateAlertModal (STAGE-043). Today the rules are write-only from the UI: the create modal exists but there is NO way to see or manage created rules. The backend (STAGE-042) already has full CRUD (`GET/POST/PATCH/DELETE /api/logs/user-rules` + enable/disable). This stage adds the frontend management surface — naturally a new tab in the Logs area (`/logs/user-rules` or similar), mirroring the existing `SilenceAllowlistTab` pattern (list + inline enable/disable/edit/delete). "Edit" should reuse the STAGE-043 CreateAlertModal (it already accepts initialValues + initialMode). Also surface each rule's vmalert load HEALTH (ok/err + lastError) so a user can see if a rule failed to load — this is where the dry-run validation (below) and the rule-health read belong.

2. **Prod vmalert -dryRun exact-parser validation** — STAGE-043 added a HEURISTIC backend expr validator (`expr_validate.py`) that catches the common vmalert-reject patterns (reserved-word filter field, missing stats pipe, unbalanced quotes/parens). It is env-portable (pure Python) but INCOMPLETE — a novel invalid expr can slip through the heuristic. This stage adds the EXACT-parser layer: validate the expr against real vmalert via `vmalert -dryRun -rule.validateExpressions` as a best-effort gate when docker + the vmalert image are available (prod), gracefully skipped in dev/test/CI (where the heuristic remains the floor). The monitor container has the docker socket + docker CLI (verified). This belongs with the rule-health surfacing (both answer "what does vmalert actually think of this rule").

**Spec references:** master spec §9.2 ("user rule management"); EPIC-004 plan #7 (deferred from 043 Refinement).

## Concrete deliverables

### 1. User Alert Rules management tab

`apps/ui/src/routes/logs/UserRulesTab.tsx` (new) — new route in the Logs area alongside Explorer, Signatures, etc. Displays:
- List of all user rules (rule_name, expr_kind, severity, enabled toggle, vmalert health)
- Per-rule actions: enable/disable toggle, edit (reuse CreateAlertModal with `initialValues` + `initialMode`), delete (with confirm)
- Empty state when no rules exist
- Inline vmalert health per rule: green "ok", red "err <lastError>" with a tooltip

### 2. Rule-health aggregation endpoint (backend)

`GET /api/logs/user-rules-health` (new) — returns rule-health for all user rules by querying vmalert's `/api/v1/rules` endpoint (proxied via the monitor's vmalert proxy, or direct http call if vmalert is reachable):

```typescript
RuleHealthResponse = {
  rules: {
    [rule_name: string]: {
      health: "ok" | "err"
      lastError?: string
      datasourceType?: string  // "vlogs" for logsql, "prometheus" for metricsql
    }
  }
}
```

Graceful fallback: if vmalert is unreachable, return `{rules: {}}` (empty health data; UI shows "unknown").

### 3. vmalert `-dryRun -rule.validateExpressions` exact-parser validation (backend)

Add to the POST/PATCH user-rules endpoints: when docker + vmalert image are available, invoke `vmalert -dryRun -rule.validateExpressions -rule=<temp-rule-file>` to validate the expr against the exact parser. On failure (nonzero exit or `error` in stderr), reject with 400 `invalid_expr` + the vmalert error message.

**Graceful fallback:** If docker is unreachable or vmalert image is missing, skip the dry-run (rely on the heuristic validator from STAGE-043). The create/update path MUST NOT break dev/test/CI/host backend.

### 4. Reuse CreateAlertModal for edit

Modify CreateAlertModal (`apps/ui/src/components/logs/CreateAlertModal.tsx`) to accept `initialMode` (create vs edit) + `initialValues` with a pre-existing rule ID, and support PATCH instead of POST on save.

### 5. Tests

- Frontend: UserRulesTab list rendering, enable/disable toggle, edit relaunch, delete confirm, empty state
- Frontend: rule-health display with ok/err states
- Backend: rule-health endpoint with realistic vmalert responses
- Backend: dry-run validator with docker available (happy path + error cases)
- Backend: dry-run validator with docker unavailable (graceful skip, heuristic-only)
- Integration: create rule → see in list → enable/disable → edit → delete (full flow)

## Out of scope for this stage

- Rule history / audit log of rule changes
- Bulk rule import/export
- Auto-generated rules (still user-curated only per the epic)
- Advanced rule-health metrics (execution time, fire count, etc.)

## Decisions in this stage

### Design phase will confirm

- **Rule-health endpoint location** — dedicated `GET /api/logs/user-rules-health` vs folded into the list response? Recommend dedicated for efficiency (health can be polled separately if UI updates realtime).
- **vmalert dry-run image** — which vmalert image to validate against? Recommend the one in the prod compose stack (same version + config as what actually runs).
- **Edit UX** — modal with pre-filled form vs in-line edit fields? Recommend modal (reuse CreateAlertModal).
- **Health polling strategy** — on-demand (fetch when tab opens) vs periodic (poll every N seconds) vs websocket? Recommend on-demand + refresh button for now.

### Locked Design Decisions (inherited / from 043 Refinement)

- **D-MGMT-TAB-REUSES-MODAL** — edit reuses STAGE-043's CreateAlertModal (initialValues + initialMode props already support this).
- **D-DRYRUN-BEST-EFFORT** — the vmalert dry-run is an ADDITIVE gate on top of the heuristic; it is SKIPPED gracefully when docker/vmalert is unavailable (dev/test/CI/host backend). The heuristic (043) is the always-on floor. Never make the create path depend on a reachable vmalert.
- **D-MGMT-MIRRORS-SILENCE-ALLOWLIST** — the management tab follows the existing SilenceAllowlistTab structure (rhf list + inline actions).
- **D-HEALTH-AGGREGATION** — rule health is read-only from vmalert's `/api/v1/rules`; no mutation of vmalert state in this stage.
