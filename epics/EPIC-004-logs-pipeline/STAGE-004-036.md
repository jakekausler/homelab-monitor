# STAGE-004-036: Anomaly Type B — "Signature count spike vs baseline" rules (7d rolling baseline + 1h cold-start fallback)

## Status: Complete

## Overview

A known signature suddenly fires N× its rolling baseline → alert. The baseline is a 7-day rolling median (or avg; Design picks) per-signature. During cold-start (signature has < 7 days of history), fall back to a static 1-hour-window threshold so the rule isn't silently inactive.

Categorization: `anomaly_kind: signature_spike`.

User-curated: this stage ships the rule *template* but does NOT auto-create rule instances for every signature. The Create-alert-from-signature flow (STAGE-004-044) generates per-signature instances on user demand.

**Vertical slice:** the rule template exists in `deploy/vmalert/logs/`. The signature catalog has a per-signature "Create spike alert" action that instantiates the rule for that template with user-chosen parameters (multiplier, evaluation window).

**Spec references:** master spec §6.3; EPIC-004 docker req #4 (per-container error-rate baseline) — covered partially here, fully in STAGE-004-037.

## Concrete deliverables

### 1. Rule template

`deploy/vmalert/logs/signature_spike_template.yml.tmpl` (new) — a Jinja-ish template that the alert-authoring flow renders with user parameters:

```yaml
# Rendered per-signature; user supplies multiplier (default 5) and window (default 5m)
- alert: SignatureSpike_{{ template_hash }}_{{ service_key | safe }}
  expr: |
    (
      # Spike against 7-day rolling baseline
      increase(homelab_log_signature_count{template_hash="{{ template_hash }}", service_key="{{ service_key }}"}[{{ window }}])
      >
      {{ multiplier }} * avg_over_time(increase(homelab_log_signature_count{template_hash="{{ template_hash }}", service_key="{{ service_key }}"}[{{ window }}])[7d:5m])
    )
    or
    (
      # Cold-start fallback: signature has < 7 days history; use 1h baseline instead
      (time() - homelab_log_signature_first_seen_ts{template_hash="{{ template_hash }}", service_key="{{ service_key }}"} < 604800)
      and
      (
        increase(homelab_log_signature_count{template_hash="{{ template_hash }}", service_key="{{ service_key }}"}[{{ window }}])
        >
        {{ multiplier }} * avg_over_time(increase(homelab_log_signature_count{template_hash="{{ template_hash }}", service_key="{{ service_key }}"}[{{ window }}])[1h:5m])
      )
    )
  for: 1m
  labels:
    severity: warning
    source_tool: homelab-monitor
    category: log-anomaly
    anomaly_kind: signature_spike
    template_hash: "{{ template_hash }}"
    service_key: "{{ service_key }}"
  annotations:
    summary: "Spike in signature for {{ '{{ $labels.service_key }}' }}"
    description: |
      Signature {{ template_hash }} fired {{ '{{ $value }}' }} times in window {{ window }},
      exceeding {{ multiplier }}× its baseline.
      URL: /logs/signatures#{{ template_hash }}/{{ service_key }}
```

(Final templating engine + escaping policy in Design phase. Note the nested `{{ }}` — Jinja and Go template need careful escaping for vmalert's `$value` / `$labels` substitutions.)

### 2. Persistence model

Per-signature rule instances are persisted via STAGE-004-042's rule persistence model and rendered into `deploy/vmalert/logs/signature_spike/*.yml` at boot. This stage SHIPS THE TEMPLATE; the per-signature instantiation depends on STAGE-004-042 + STAGE-004-044. If 042+044 are not yet built, this stage can include 1-2 hardcoded test rules so the template is validated end-to-end.

### 3. Suppression integration

The `unless on (template_hash, service_key) (homelab_log_signature_suppressed == 1)` clause is added so suppressed signatures don't alert. Same as STAGE-004-035.

### 4. Baseline computation considerations

- `avg_over_time(... [7d:5m])` looks back 7 days at 5-min steps; reasonable cardinality
- Cold-start gate (`time() - first_seen_ts < 604800`) switches to 1-hour baseline when signature is younger than 7 days
- Both branches OR'd together so the rule fires correctly in both regimes

### 5. Rule test fixtures

`apps/monitor/tests/test_signature_spike_rule.py` — plant metric history (steady baseline then spike), assert the rule fires. Plant cold-start scenario (first_seen recent + 1h baseline + spike), assert fires. Plant suppressed signature (suppressed gauge = 1), assert does NOT fire.

### 6. Karma / Alertmanager rendering

Default Alertmanager template handles `anomaly_kind: signature_spike` alerts with a sensible default rendering (template hash, multiplier observed, baseline value, deep-link to catalog). Customized in Alertmanager template config.

## Decisions in this stage

### Design phase will confirm

- **Baseline statistic** — `avg_over_time` vs `quantile_over_time(0.5, ...)` (median). Recommendation: `avg_over_time` (simpler, easier to reason about; median requires more PromQL plumbing)
- **Cold-start cutoff** — 7 days recommended (`604800` seconds)
- **Default multiplier** — 5×
- **Default window** — 5 min
- **for: duration** — 1 min recommended (smooths brief spikes)

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-ROLLING-7D-BASELINE** — 7-day rolling.
- **D-COLD-START-1H-FALLBACK** — 1-hour static-baseline fallback when signature is younger than 7 days.
- **D-USER-CURATED-RULE-INSTANCES** — rule template lives in this stage; per-signature instantiation goes through user-curated alert-authoring (STAGE-004-044).

## Design Decision — chosen approach (locked 2026-06-07)

**Stage ships a TEMPLATE (not live instances).** Per D-USER-CURATED-RULE-INSTANCES, 036 ships a parameterized vmalert rule template + a reusable pure render function + tests. Per-signature INSTANCES are created later (STAGE-042 persistence renders them; STAGE-044 UI instantiates them on user demand). The template CANNOT fire today (no instances exist) — that is expected and correct.

**A. Spike PromQL (corrected against source — the card's expr had bugs):**
- Current volume over the window uses `sum_over_time(homelab_log_signature_count{...}[__WINDOW__])` — NOT `increase()` (the card used `increase()`, which is meaningless on a GAUGE; `_count` is a per-cycle line-count gauge, not a counter).
- 7-day baseline: `avg_over_time(sum_over_time(homelab_log_signature_count{...}[__WINDOW__])[7d:__WINDOW__])`.
- GAP-ROBUSTNESS: `homelab_log_signature_count` is GAPPY (a signature with zero lines in a cycle emits NO sample, so avg_over_time averages only over present windows). To prevent a rare/bursty signature with a tiny noisy baseline from hair-triggering, apply a minimum-baseline FLOOR: `__MULTIPLIER__ * max(avg_over_time(...), __MIN_BASELINE__)` (default min_baseline 10). Fire when current > multiplier * max(baseline, floor).
- COLD-START (D-COLD-START-1H-FALLBACK): encoded as an `or` of two age-guarded branches — a mature branch gated by `(time() - homelab_log_signature_first_seen_ts{...}/1e9 >= 604800)` using the 7d baseline, and a cold-start branch gated by `(... < 604800)` using a `[1h:__WINDOW__]` baseline. NOTE `/1e9`: `first_seen_ts` is in NANOSECONDS (the card omitted the conversion — same bug class as STAGE-035). Each branch wrapped in parens; `and on(service_key, template_hash)` for 1:1 vector match (each instance is one concrete signature).
- Chose `_count` over `_total`: `_total` (cumulative cluster_size, no severity label) is also gappy and `delta()` over it extrapolates across gaps + drain re-cluster fluctuations unreliably; `sum_over_time(_count)` does no cross-gap extrapolation (just sums present samples) and keeps the severity label.

**B. Suppression: NO suppression clause in 036's template.** The card proposed `unless on(...) (homelab_log_signature_suppressed == 1)` — but that metric DOES NOT EXIST (STAGE-035 folded suppression into a collector decision; there is no standalone suppressed gauge). D-SUPPRESSED-SIGNATURES-NEVER-ALERT is satisfied STRUCTURALLY: a suppressed signature has no rendered instance (STAGE-042/044 own not-rendering/deleting instances for suppressed signatures), so no rule exists to fire. Adding a suppressed gauge just for this would duplicate logic that belongs in the curate/render layer and build 042/044's job into 036 — rejected. The template header comment documents this invariant so 042 knows it owns suppression-driven instance teardown.

**C. Templating mechanism: `__UPPER_SNAKE__` sentinel tokens + Python `str.replace()`** (matches the repo's existing `kernel/cron/render.py` `${TOKEN}` precedent). REJECTED `string.Template` (`$placeholder`) because it COLLIDES with vmalert's own Go-template `{{ $labels.x }}` / `{{ $value }}` (`$labels`/`$value` use `$`). REJECTED custom-delimiter Jinja (no Jinja2 dependency; foot-gun). Token set: `__ALERT_SLUG__`, `__SERVICE_KEY__`, `__TEMPLATE_HASH__`, `__MULTIPLIER__` (default 5), `__WINDOW__` (default 5m), `__MIN_BASELINE__` (default 10). `for:` is a static `1m` in the template (locked default). The render function validates/rejects label values containing `"` or `\` (mirrors `cron/render.py`'s quote-rejection).

**D. Deliverables (the CORRECTED file list):**
1. `deploy/vmalert/metrics/signature_spike.yml.tmpl` — NOTE the **metrics/** dir, NOT logs/. The card said `deploy/vmalert/logs/signature_spike_template.yml.tmpl` — WRONG: the rule queries the PromQL metric `homelab_log_signature_count`, so it's a METRICS (PromQL) rule (`source_tool: vmalert-metrics`, no `type: vlogs`). The `.tmpl` extension keeps it out of vmalert's live `*.yml`/`*.yaml` rule glob (it can't fire until rendered to a real instance by 042).
2. A pure kernel render function `render_signature_spike_rule(template_hash, service_key, multiplier, window, min_baseline) -> str` (e.g. in `kernel/logs/signature_spike_render.py`) — string-in/string-out, no DB/IO, fully unit-testable, REUSABLE by STAGE-042. (Chose this over a throwaway test-local renderer so the logic is covered + typed + reused, not re-derived in 042. 036 does NOT enumerate signatures / persist instances / wire a collector — that's 042/044's job.)
3. `apps/monitor/tests/test_signature_spike_rule.py` (the binding gate): render+`yaml.safe_load` parse; structural asserts (expr contains the substituted sum_over_time + correct labels category=log-anomaly / anomaly_kind=signature_spike / source_tool=vmalert-metrics / target_kind=log_signature, `for: 1m`); COLLISION regression (assert `{{ $labels.` and `{{ $value }}` survive UN-substituted in annotations); cold-start branch present (both 604800 guards + `/1e9`); min-baseline floor present (`max(..., 10)`); validation paths (rejects `"`/`\` in label values). 100% coverage of the render module.
4. A forward-compatible `deploy/vmalert/metrics/__tests__/signature_spike.tests.yaml` static fixture (mirrors `heartbeats.tests.yaml`) with planted `homelab_log_signature_count` + `homelab_log_signature_first_seen_ts` series demonstrating spike-fires / steady-doesn't / cold-start-1h — documentation + ready for a future vmalert test harness (no make target runs `.tests.yaml` today; the pytest render test is the binding gate).

**Build verification flag:** confirm vmalert does NOT load the `.tmpl` (it globs `*.yml`/`*.yaml`, not `*.tmpl`); confirm `and on(...)`/`or` precedence by parenthesizing each branch; the `first_seen_ts` series is gappy too — if a signature is quiet at eval the age guard has no series → `and` yields empty → no alert (acceptable: no current `_count` either).

**Stage type:** BACKEND-ONLY. Alert (once instances exist) surfaces in Karma automatically. No new frontend → no Desktop+Mobile Refinement prompt.

- **Scenario 4 (KEY):** A manually-rendered instance was injected into LIVE vmalert-metrics on the prod rig: it LOADED with `health: ok`, `lastError: ""` (PromQL syntactically valid + evaluable — vmalert rejects malformed PromQL at load). Planted a baseline (count=2/cycle) + a spike (count=200) with a mature-enough first_seen_ts; `sum_over_time([5m])`=402 > threshold `5*max(avg,10)=50` → alert FIRED with labels category=log-anomaly, anomaly_kind=signature_spike, target_kind=log_signature, severity=warning + exported_severity=error. Steady-state (sum=4 < 50 floor) did NOT fire.

**DOCUMENTED LIMITATION:** The e2e exercised the COLD-START branch (`< 604800`, `[1h:5m]`) by firing; planting a full 7-day baseline to fire the MATURE branch (`>= 604800`, `[7d:5m]`) was impractical. The mature branch uses identical logic with a longer lookback and its PromQL was ACCEPTED by vmalert (loaded health=ok), so it is syntactically valid/evaluable — just not fired live. Acceptable for a template-only forward-dependency stage; full mature-branch live-fire will be naturally exercised once STAGE-042/044 render real instances against real 7d history.
