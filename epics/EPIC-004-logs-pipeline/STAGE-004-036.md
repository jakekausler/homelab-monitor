# STAGE-004-036: Anomaly Type B — "Signature count spike vs baseline" rules (7d rolling baseline + 1h cold-start fallback)

## Status: Not Started

## Overview

A known signature suddenly fires N× its rolling baseline → alert. The baseline is a 7-day rolling median (or avg; Design picks) per-signature. During cold-start (signature has < 7 days of history), fall back to a static 1-hour-window threshold so the rule isn't silently inactive.

Categorization: `anomaly_kind: signature_spike`.

User-curated: this stage ships the rule *template* but does NOT auto-create rule instances for every signature. The Create-alert-from-signature flow (STAGE-004-044) generates per-signature instances on user demand.

**Vertical slice:** the rule template exists in `deploy/vmalert/logs/`. The signature catalog has a per-signature "Create spike alert" action that instantiates the rule for that template with user-chosen parameters (multiplier, evaluation window).

**Spec references:** master spec §6.3; EPIC-004 docker req #4 (per-container error-rate baseline) — covered partially here, fully in STAGE-004-037.

## Concrete deliverables

### 1. Rule template

`deploy/vmalert/logs/signature_spike_template.yml.tmpl` (new) — a Jinja-ish template that the alert-authoring flow renders with user parameters:

```yaml
# Rendered per-signature; user supplies multiplier (default 5) and window (default 5m)
- alert: SignatureSpike_{{ template_hash }}_{{ service_key | safe }}
  expr: |
    (
      # Spike against 7-day rolling baseline
      increase(homelab_log_signature_count{template_hash="{{ template_hash }}", service_key="{{ service_key }}"}[{{ window }}])
      >
      {{ multiplier }} * avg_over_time(increase(homelab_log_signature_count{template_hash="{{ template_hash }}", service_key="{{ service_key }}"}[{{ window }}])[7d:5m])
    )
    or
    (
      # Cold-start fallback: signature has < 7 days history; use 1h baseline instead
      (time() - homelab_log_signature_first_seen_ts{template_hash="{{ template_hash }}", service_key="{{ service_key }}"} < 604800)
      and
      (
        increase(homelab_log_signature_count{template_hash="{{ template_hash }}", service_key="{{ service_key }}"}[{{ window }}])
        >
        {{ multiplier }} * avg_over_time(increase(homelab_log_signature_count{template_hash="{{ template_hash }}", service_key="{{ service_key }}"}[{{ window }}])[1h:5m])
      )
    )
  for: 1m
  labels:
    severity: warning
    source_tool: homelab-monitor
    category: log-anomaly
    anomaly_kind: signature_spike
    template_hash: "{{ template_hash }}"
    service_key: "{{ service_key }}"
  annotations:
    summary: "Spike in signature for {{ '{{ $labels.service_key }}' }}"
    description: |
      Signature {{ template_hash }} fired {{ '{{ $value }}' }} times in window {{ window }},
      exceeding {{ multiplier }}× its baseline.
      URL: /logs/signatures#{{ template_hash }}/{{ service_key }}
```

(Final templating engine + escaping policy in Design phase. Note the nested `{{ }}` — Jinja and Go template need careful escaping for vmalert's `$value` / `$labels` substitutions.)

### 2. Persistence model

Per-signature rule instances are persisted via STAGE-004-042's rule persistence model and rendered into `deploy/vmalert/logs/signature_spike/*.yml` at boot. This stage SHIPS THE TEMPLATE; the per-signature instantiation depends on STAGE-004-042 + STAGE-004-044. If 042+044 are not yet built, this stage can include 1-2 hardcoded test rules so the template is validated end-to-end.

### 3. Suppression integration

The `unless on (template_hash, service_key) (homelab_log_signature_suppressed == 1)` clause is added so suppressed signatures don't alert. Same as STAGE-004-035.

### 4. Baseline computation considerations

- `avg_over_time(... [7d:5m])` looks back 7 days at 5-min steps; reasonable cardinality
- Cold-start gate (`time() - first_seen_ts < 604800`) switches to 1-hour baseline when signature is younger than 7 days
- Both branches OR'd together so the rule fires correctly in both regimes

### 5. Rule test fixtures

`apps/monitor/tests/test_signature_spike_rule.py` — plant metric history (steady baseline then spike), assert the rule fires. Plant cold-start scenario (first_seen recent + 1h baseline + spike), assert fires. Plant suppressed signature (suppressed gauge = 1), assert does NOT fire.

### 6. Karma / Alertmanager rendering

Default Alertmanager template handles `anomaly_kind: signature_spike` alerts with a sensible default rendering (template hash, multiplier observed, baseline value, deep-link to catalog). Customized in Alertmanager template config.

## Out of scope for this stage

- Per-signature rule instances at scale — STAGE-004-044 creates them
- Drop-to-baseline detection (opposite direction) — speculative
- Service-wide error rate (Type C) — STAGE-004-037
- Signature went silent (Type D) — STAGE-004-038

## Decisions in this stage

### Design phase will confirm

- **Baseline statistic** — `avg_over_time` vs `quantile_over_time(0.5, ...)` (median). Recommendation: `avg_over_time` (simpler, easier to reason about; median requires more PromQL plumbing)
- **Cold-start cutoff** — 7 days recommended (`604800` seconds)
- **Default multiplier** — 5×
- **Default window** — 5 min
- **for: duration** — 1 min recommended (smooths brief spikes)

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-ROLLING-7D-BASELINE** — 7-day rolling.
- **D-COLD-START-1H-FALLBACK** — 1-hour static-baseline fallback when signature is younger than 7 days.
- **D-USER-CURATED-RULE-INSTANCES** — rule template lives in this stage; per-signature instantiation goes through user-curated alert-authoring (STAGE-004-044).
