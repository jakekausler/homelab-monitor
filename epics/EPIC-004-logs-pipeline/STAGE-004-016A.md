# STAGE-004-016A: LogsQL structured field filters (Add-to-filter uses structured operators for non-message fields)

## Status: Not Started

## Overview

STAGE-004-016 (Field inspector) currently maps the "Add to filter" action to `_msg:"<value>"` substring for ALL fields except `service` (which uses the stream-picker identity separate-state). This stage upgrades non-`message` fields (host, severity, stream, and `fields`-bag entries) to add STRUCTURED LogsQL filters (e.g. `host:"value"`, `severity:"error"`, or the appropriate LogsQL field operator) instead of the imprecise `_msg` substring. `message` stays as `_msg:"<substring>"` (substring is the correct semantic for free-text message). This was the REJECTED "Option B" deferred during STAGE-004-016 Design — deferred because it needs (a) backend-contract verification of which fields are actually indexed/queryable in VictoriaLogs, and (b) a new separate-state "field filters" channel (or an equivalent compose-into-LogsQL mechanism) threaded through writeUrl + URL serialization + STAGE-015 persistence.

**Why it's its own stage:** Requires backend-contract verification (are `fields`-bag keys indexed in VictoriaLogs? does `key:"value"` LogsQL return results for them, or silently nothing?) and a new filter-state channel — too large to fold into 016.

## Concrete deliverables

### 1. Backend-contract verification (Design phase)

Verify which LogLine fields are queryable via structured LogsQL operators:
- Top-level indexed fields: `host`, `severity`, `stream`, `service` (already handled via stream picker in 016)
- `fields`-bag entries: which are indexed? Which support `key:"value"` LogsQL syntax? Which should fall back to `_msg` substring?
- Read the VictoriaLogs client code and ingest-time field configuration to confirm queryability.
- Document findings in Design phase section.

### 2. New filter-state channel OR composition helper

Choose between:
- **(Option A)** New separate-state channel `fieldFilters: { [fieldName]: string[] }` (parallel to `selectedIdentities`), threaded through writeUrl + URL serialization + STAGE-015 persistence. Render as removable chip rows (consistent with service-identity UX).
- **(Option B)** Compose structured operators directly into LogsQL plain-text search; handle mode-coupling carefully (plain-text mode wraps entire text in `_msg:"..."`, so structured operators only work in advanced mode).
- **(Option C)** Hybrid: removable field-filter chips for top-level fields, plain-text substring for `fields`-bag.

Design phase resolves which approach, then implement accordingly.

### 3. Updated FieldInspectorPanel Add-to-filter mapping

Modify the field→filter mapping from STAGE-004-016:

| Field | Maps to | Mechanism |
|---|---|---|
| service | selectedIdentities separate state | handleToggleIdentity (unchanged from 016) |
| host | fieldFilters OR LogsQL text | structured `host:"value"` OR composed into query |
| severity | fieldFilters OR LogsQL text | structured `severity:"error"` OR composed into query |
| stream | fieldFilters OR LogsQL text | structured `stream:"value"` OR composed into query |
| message | search text | `_msg:"<substring>"` (unchanged) |
| fields[k] bag entry | fieldFilters OR LogsQL text | structured `key:"value"` IF indexed, else `_msg` substring OR composed into query |
| timestamp | — | Copy-only (unchanged) |

### 4. Removable filter chips (if Option A)

If Design chooses Option A (new separate state), render field-filter chips in the Explorer header or filter sidebar, alongside the service-identity chips. Chips are removable. Consistent styling + interaction with 016's service chips.

### 5. Tests

- Backend-contract verification confirmed in code comments.
- Field-filter state compose/decompose round-trips through writeUrl + URL + persistence (if Option A).
- Add-to-filter on host/severity/stream/fields populates the correct filter state or LogsQL text.
- Removable chips (if Option A) clear their filter when clicked.
- Mode coupling behavior tested (if Option B; plain text wrapping vs advanced mode composing).

## Out of scope for this stage

- Filtering by `message` stays `_msg` substring (correct for free-text).
- Nested `fields`-bag tree UI (that is STAGE-004-016B or later stages).
- Auto-suggestions on structured operators (deferred to STAGE-004-018A).
