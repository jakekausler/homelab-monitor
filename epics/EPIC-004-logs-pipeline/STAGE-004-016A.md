# STAGE-004-016A: LogsQL structured field filters (Add-to-filter uses structured operators for non-message fields)

## Status: Complete

## Overview

STAGE-004-016 (Field inspector) currently maps the "Add to filter" action to `_msg:"<value>"` substring for ALL fields except `service` (which uses the stream-picker identity separate-state). This stage upgrades non-`message` fields (host, severity, stream, and `fields`-bag entries) to add STRUCTURED LogsQL filters (e.g. `host:"value"`, `severity:"error"`, or the appropriate LogsQL field operator) instead of the imprecise `_msg` substring. `message` stays as `_msg:"<substring>"` (substring is the correct semantic for free-text message). This was the REJECTED "Option B" deferred during STAGE-004-016 Design — deferred because it needs (a) backend-contract verification of which fields are actually indexed/queryable in VictoriaLogs, and (b) a new separate-state "field filters" channel (or an equivalent compose-into-LogsQL mechanism) threaded through writeUrl + URL serialization + STAGE-015 persistence.

**Why it's its own stage:** Requires backend-contract verification (are `fields`-bag keys indexed in VictoriaLogs? does `key:"value"` LogsQL return results for them, or silently nothing?) and a new filter-state channel — too large to fold into 016.

### Backend contract verification (done in Design — DECISIVE)

- ALL LogLine fields are queryable via structured LogsQL `field:"value"` syntax. Vector sends every field as a flat JSON field to VictoriaLogs (/insert/jsonline); the backend (apps/monitor/homelab_monitor/kernel/api/routers/logs.py) passes the frontend `expr` VERBATIM to VL (only additively composing the services CSV at _compose_services_expr).
- The earlier Option-B rejection worry ("fields-bag keys may not be indexed → silently return nothing") is FALSE — all JSON fields are queryable.
- Confirmed queryable field names: severity, host, service, source_type, compose_service, compose_project, container_name, image_name/image_tag/image_digest, run_id, cron_fingerprint, SYSLOG_IDENTIFIER, and arbitrary fields-bag keys. Syntax: `fieldname:"value"` (colon + quoted phrase, AND-joined).
- Raw LogsQL field filters ALREADY work end-to-end in ADVANCED mode today (advanced mode passes committedLogsQl verbatim). Working precedent: the services CSV composes to `service:"x" AND source_type:"y"` (test_logs_services.py).
- Evidence files: apps/monitor/homelab_monitor/kernel/api/routers/logs.py (~74-104, 142-149), victorialogs_client.py (124-150), deploy/vector/vector.toml.template (72-82, 101-205), apps/monitor/tests/test_logs_services.py (32-44), test_api_logs.py (790-806).

### CHOSEN: Option B — compose structured `field:"value"` clauses into the LogsQL expr (mirror appendMsgFilter)

- Add `appendFieldFilter(fieldName, value)` to LogsExplorerPage that composes a discrete `fieldName:"value"` clause into committedLogsQl, switches to advanced mode, and routes through the existing writeUrl(true, committedPlainText, nextLogsQl, range, selectedIdentities) choke-point — EXACTLY the shipped appendMsgFilter pattern (016). If plain-text search is active, translate it to `_msg:"..."` first, then AND-append the new clause (LogsQL AND-joins space-separated clauses).
- New pure helper `fieldFilterClause(field, value)` in logsQlTranslate.ts, reusing the existing escapeLogsQlPhrase escaper (sibling to msgFilterClause).
- FieldInspectorPanel Add-to-filter mapping after 016A:
  - service → existing identity-chip filter (onAddIdentity) — UNCHANGED
  - host → appendFieldFilter("host", value)
  - severity → appendFieldFilter("severity", value)
  - stream → appendFieldFilter (BUILD CAVEAT below — verify real queryable field name first; fall back to _msg if not directly queryable)
  - fields[k] bag entry → appendFieldFilter(k, value) (the bag key IS the real VL field name)
  - message → onAddMsgFilter (_msg:"<substring>") — UNCHANGED (substring is correct for free-text)
  - timestamp → Copy-only — UNCHANGED

### Rationale

- Option A's separate-state-channel template (selectedIdentities) exists for a backend reason that does NOT apply here: service needs out-of-band composition into the backend `services` CSV (service:"x" AND source_type:"y"). Field filters are already valid LogsQL clauses passed verbatim — no backend reason for a separate channel. Option A = full ~8-call-site writeUrl threading + URL codec + persistence schema for ZERO backend benefit, then composes into the expr frontend-side anyway.
- Option B is minimal (one pure helper + one handler + prop threading), uniform with the existing add-to-filter UX (raw-text-in-LogsQL, same as message filters since 016), and keeps the LogsQL expr as a SINGLE SOURCE OF TRUTH that upcoming 018A (autocomplete) and 016B (JSON drill-down) build ON rather than fragment against.
- Trade-off accepted: field filters appear as raw text in the LogsQL editor (advanced mode), removed by editing the query text (no removable chip). This is already the accepted model for message filters since 016. Option C (chips for common fields) can layer on later WITHOUT rework if usage proves removal is a pain point.

### REJECTED

- Option A (new fieldFilters separate-state channel + removable chips): large surface (new writeUrl param ~8 call sites, new URL key + codec, new persistence schema field, chip UI) for zero backend benefit; field filters compose into the expr frontend-side regardless. Chips would also live OUTSIDE the expr, invisible to 018A autocomplete / 016B — fragmenting the query source of truth.
- Option C (hybrid chips + text): highest surface, introduces a THIRD filter model, per-field inconsistency (identical-looking buttons behaving differently), worst fragmentation for 018A/016B. Premature without usage data.

### BUILD CAVEAT (must resolve in Build)

- `stream`: the Field Inspector shows `line.stream` as a core row, but `stream` may NOT be a literal queryable VL field name (host/severity ARE real fields; bag keys carry their own real keys). Build must verify the real queryable VL field for stream before composing `stream:"value"` — a wrong field name silently matches nothing. If stream is not directly queryable, fall back to `_msg:"<value>"` for the stream row (or omit its add-to-filter). Resolve via backend/VL field inspection in Build.

### Refinement — Additional In-Session Fix: LogsQL Single-Line Enforcement

During Refinement, user reported an unrelated editor bug and requested it be fixed immediately:

**LogsQL editor single-line enforcement.** The advanced-mode CodeMirror LogsQL editor inserted a newline on Enter (in addition to submitting the query). Fixed after THREE debugging iterations:

1. **First attempt:** Wrapped the submit keymap in `Prec.high` — insufficient (newline persisted; precedence only governs the keyboard-dispatch path, not text insertion).
2. **Second attempt:** EditorState.transactionFilter that rewrote newline insertions to spaces — fixed the newline but introduced a stray SPACE on Enter, then OSCILLATION (repeated Enter: space appears → deletes → nothing, repeating indefinitely).
3. **Root-cause fix:** The transactionFilter returned an inconsistent `(changes, selection)` pair — it rewrote `changes` to drop the newline but passed `selection: tr.selection` verbatim (which assumed the newline had been inserted; cursor at +1). CodeMirror's contract requires all selection offsets to refer to the doc AFTER the transaction, so the mismatch caused reconciliation oscillation. **Atomic fix:** the filter now REJECTS multi-line transactions entirely via `tr.docChanged && tr.newDoc.lines > 1 ? [] : tr` (returns `[]` — rejects changes AND selection together, structurally oscillation-proof).

**Trade-off accepted by user:** Multi-line PASTE into the LogsQL editor is now BLOCKED (does nothing) rather than collapsed to one line — acceptable for a single-line query box.
