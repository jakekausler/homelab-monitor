# STAGE-004-004: Container label enrichment (compose_project, compose_service, image labels as top-level VL fields)

## Status: Complete

## Overview

Today's vector docker_logs source passes through raw container labels (e.g., `label.com.docker.compose.project`, `label.com.docker.compose.service`) and image metadata but does NOT promote them to top-level VL fields. That means a useful LogsQL query like "all errors in the homelab-monitor compose stack" requires enumerating container names instead of writing `compose_project:homelab-monitor`.

This stage adds a vector remap transform that extracts the standard compose + image labels into top-level fields on every docker-sourced log line. The fields become queryable in LogsQL, appear in the Field Inspector (STAGE-004-016), and surface in the filter-scope-aware discovery (STAGE-004-018).

**Vertical slice:** after this stage, `compose_project:homelab-monitor` is a valid LogsQL filter across all container logs. The `<LogViewer>` Field Inspector (when built) will show these as parsed fields per line.

**Spec references:** Master spec §3.2 (vector sidecar), §6.3 (VL streams).

## Concrete deliverables

### 1. New `[transforms.docker_enrich]` block in vector.toml.template

`deploy/vector/vector.toml.template` — after `add_labels` (or after `drop_noise`), insert a `remap` transform that promotes well-known docker labels to top-level fields:

```toml
[transforms.docker_enrich]
type = "remap"
inputs = ["add_labels"]  # or wherever in the pipeline makes sense
source = '''
# Compose labels
.compose_project = .label."com.docker.compose.project"
.compose_service = .label."com.docker.compose.service"
.compose_config_files = .label."com.docker.compose.project.config_files"
.compose_working_dir = .label."com.docker.compose.project.working_dir"
.compose_container_number = .label."com.docker.compose.container-number"
# Image info
.image_name = .image_name  # vector already provides this; normalize field name
.image_tag = .image_tag
.image_digest = .label."org.opencontainers.image.revision"
# Container info
.container_id = .container_id
.container_name = .container_name
# Strip the raw label bag from the output to keep VL records small (optional, decided in Design)
# del(.label)
'''
```

(Final VRL syntax confirmed in Design phase against vector's current language.)

## Decisions in this stage

### Design phase will confirm

- **Whether to strip the raw `.label` bag** after extracting top-level fields — trade-off: smaller VL records vs. raw label availability for future ad-hoc queries. Recommendation: KEEP the raw `.label` bag for forensic access; storage cost is negligible.
- **Which compose labels to promote** — list above is the seed; Design phase locks the final set.
- **Field naming convention** — `compose_project` vs `compose.project` vs `compose_label_project`. Recommendation: flat snake_case (e.g., `compose_project`).

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-ENRICH-DOCKER-LABELS** — compose project, compose service, image name/tag/digest, container id/name are top-level VL fields, queryable in LogsQL.
- **D-ENRICH-VECTOR-LAYER** — enrichment happens in vector at ingest, not at query time. One-time cost per line; queries are fast.
- **D-ENRICH-FLAT-SNAKE-CASE** — top-level field names are flat snake_case (compose_project, image_name, etc.).
- **D-KEEP-RAW-LABEL-BAG** — keep the raw `.label.*` bag in addition to promoting select fields. Forensic access for future ad-hoc queries.

## Design Phase

**Design phase complete: 2026-05-29.** All four locked D-* decisions stand (D-ENRICH-DOCKER-LABELS, D-ENRICH-VECTOR-LAYER, D-ENRICH-FLAT-SNAKE-CASE, D-KEEP-RAW-LABEL-BAG). The "Design will confirm" items resolved:

Pipeline topology: NEW dedicated `[transforms.docker_enrich]` remap (single-responsibility; matches existing cron_parsed/hmrun_shaped pattern). `inputs=["parse_json"]`, placed AFTER `parse_json` so trusted Vector container metadata wins over arbitrary app-JSON fields (cannot be spoofed by log payload). Rewire `drop_noise.inputs=["docker_enrich"]`. `[sinks.vl].inputs` UNCHANGED (["throttle","hmrun_shaped"]).

Promoted field set (6 new, flat snake_case):
- compose_project ← .label."com.docker.compose.project"
- compose_service ← .label."com.docker.compose.service"
- image_name ← parsed from .image
- image_tag ← parsed from .image (defaults "latest" when absent)
- image_digest ← @sha256: portion of .image (nullable; usually null)
- image_revision ← .label."org.opencontainers.image.revision" (git SHA — honest name)
Left as-is: container_name/container_id (already top-level from docker_logs source — no re-copy). Left in raw .label bag only (NOT promoted, YAGNI): config_files, working_dir, container-number.

Image parsing: split-based VRL (split off digest on "@"; disambiguate tag vs registry-port by the LAST "/" segment) — NO regex, NO lookarounds (honors Rust-regex constraint). Handles bare name / name:tag / registry:port/name:tag / @sha256 forms.

journald no-op safety: whole-block gating on `exists(.label)` and `exists(.image)` (NOT per-field fallbacks) so systemd/journald lines never carry null compose_*/image_* fields.

Raw bag: KEPT (no `del(.label)`); the seed's commented `# del(.label)` is explicitly dropped from the implementation.

TWO SEED CORRECTIONS folded in (the stage seed was wrong): (1) Vector's docker_logs emits only `.image` (a string), NOT pre-split `.image_name`/`.image_tag` — must parse. (2) `org.opencontainers.image.revision` is a git REVISION, not an image digest — promote it as `image_revision`, and derive `image_digest` from the `@sha256:` portion of `.image`.

Testability: 10 template-parse assertions in `apps/monitor/tests/test_vector_template.py` (docker_enrich present; inputs==["parse_json"]; drop_noise consumes docker_enrich; compose_* + image_* field names present; snake_case enforced; journald guards present; raw .label bag kept i.e. no del(.label); no-lookarounds on the block; sinks.vl.inputs unchanged).
