# STAGE-004-034: Cron run failure log correlation (L4)

## Status: Complete

## Overview

When a cron run fails (non-zero exit), enrich the cron-failure alert with the last N lines of `hmrun` output for that run. Parallel pattern to container crash correlation (STAGE-004-032) but for cron rather than docker.

Uses the shared `LogWindowFetcher` from STAGE-004-031.

**Vertical slice:** when a cron run fails, the resulting Karma/Telegram/email notification contains the last ~50 lines of the run's output, AND the alert detail page in the UI renders it inline.

**Spec references:** master spec §6.3 (cron capture); EPIC-004 brainstorm L4.

## Design Decisions (selected 2026-06-07)

### Selected approach (cron differs from the 032/033 docker correlators: cron already has a run lifecycle + an existing CronRunReconciler whose _enrich phase already fetches run logs)

- **C1 — Persistence:** NEW table `cron_run_failure_enrichments` (per locked D-CRON-RETAIN-30D). Columns: failure_id (TEXT PK, uuid4 hex), cron_fingerprint, run_id, exit_code, started_at, ended_at, lines_json (JSON blob of LogLine dicts), line_count, truncated, degraded, window_start, window_end, created_at. UNIQUE index ux_cron_failure_enrich_fp_run on (cron_fingerprint, run_id) = the dedup. Independent 30d retention (env HOMELAB_MONITOR_CRON_ENRICHMENT_RETENTION_DAYS=30) + per-fingerprint cap, one transaction (mirror CrashEnrichmentsRepository.prune). NEW migration 0038 (down_revision "0037"), bump EXPECTED_TABLES 35→36 in tests/test_db_migrations.py. Rationale: decoupled retention is load-bearing — cron_runs' own prune (age + 50k-per-cron cap) can delete a failed run's row inside 30d for frequent crons, losing the forensic record; the separate table outlives it.
  - Rejected: reuse cron_runs (add lines_json columns) — contradicts D-CRON-RETAIN-30D (same prune schedule as the lifecycle row) + fattens the hot 30s-scanned table.
- **C2 — Detection trigger:** Hook a failure-enrich STEP into the EXISTING CronRunReconciler._enrich per-run loop (NOT a new reconciler). When a closed run has state=="fail", after the existing enrich query, do ONE dedicated capped VL fetch + persist to the new table via INSERT OR IGNORE on (cron_fingerprint, run_id) (idempotent; no "already-enriched" column needed). Rationale: cron already has a reconciler iterating closed runs + a VictoriaLogsClient in hand; a second 30s scanner of cron_runs would duplicate work. (Diverges from 032/033's standalone reconciler because those were greenfield Docker with no existing scanner.)
  - Rejected: new standalone reconciler — duplicates the existing per-run scan + a second grace/timing window.
- **C3 — Alert + deep-link:** Emit COUNTER `homelab_cron_run_failure_total{cron_fingerprint, run_id, name, host}` once per NEWLY-failure-enriched run (gated on the INSERT OR IGNORE returning a new row, like 032/033's counters). ADD a NEW vmalert rule `CronRunFailed` to deploy/vmalert/metrics/heartbeats.yaml (NOT a new cron.yml — that file does not exist; cron alerts live in heartbeats.yaml): expr `increase(homelab_cron_run_failure_total[5m]) > 0`, for 0m, labels severity:error, source_tool:vmalert-metrics, category:cron (EPIC invariant), target_kind:cron, routing_channel:cron-health; annotation summary uses {{ $labels.name }}/{{ $labels.run_id }} + description deep-links to /inventory/crons/{{ $labels.cron_fingerprint }}/runs/{{ $labels.run_id }} (the EXISTING CronRunLogViewer route). Rationale: the existing HeartbeatFailed alert fires on cron STATE with name/host labels only (NO run_id/fingerprint) so it CANNOT deep-link to a specific failed run; a per-run counter+rule is the only path to the card's stated intent (notification links to the enrichment) and matches the 032/033 counter+rule+category pattern. CronRunFailed coexists with HeartbeatFailed (semantically distinct: per-run failure-with-logs vs sustained streak).
- **C4 — Window + which lines:** A CAPPED RUN-SCOPED query (the last ~50 lines of THIS run) — NOT a time-window-around-anchor. run_id is an EXACT VL selector (build_amode_query(run_id) / build_bmode_query(command) over the run's existing vl_window_start/vl_window_end with the wrapper-slack widening), newest-first, limit 50. A dedicated capped fetch reusing the reconciler's VictoriaLogsClient (one extra VL call per newly-failed run only). Rationale: unlike the Docker correlators (fuzzy name+time selector needing an anchor window), cron's run_id is exact — failure forensics = "the last N lines this run emitted." This CORRECTS the card's copy-pasted Docker-style window spec (started_at→ended_at+5s).
- **C5 — UI:** BACKEND-ONLY, no new UI. The alert deep-link targets the EXISTING CronRunLogViewer run page (/inventory/crons/{fingerprint}/runs/{run_id}) which already renders the run's full logs; a persisted-snapshot viewer would duplicate it (the card itself says "CronRunLogViewer already shows the full log; nothing to add there"). CONSEQUENCE: this is a BACKEND-ONLY stage → Refinement is backend e2e (3a dev rig synthetic failed run + 3b prod rig real failed cron) with NO Desktop+Mobile FE prompt.
- **Endpoint:** NEW `GET /api/crons/{fingerprint}/runs/{run_id}/failure-enrichment` in the crons router (returns CronRunFailureEnrichment: failure_id, cron_fingerprint, run_id, exit_code, started_at, ended_at, line_count, truncated, degraded, window_start, window_end, created_at, lines: list[LogLine]). 404 if the run/enrichment is unknown; session auth. openapi regen + generate-types after (the FE consumes the schema type even though no UI renders it — actually, since BACKEND-ONLY, the generate-types is still run to keep schema.ts consistent, but no new FE hook/component is added).
- **Config:** NEW CronEnrichmentConfig (or fold into existing cron config) + loader reading HOMELAB_MONITOR_CRON_ENRICHMENT_RETENTION_DAYS (default 30) + a fixed line_limit=50 + per-fingerprint cap. Clamp >=1.

## Concrete deliverables

### 1. Cron-failure detector

`apps/monitor/homelab_monitor/kernel/logs/cron_failure_correlator.py` (new):

- Subscribes to cron run completion events (already emitted per EPIC-002)
- For each completed run with `exit_code != 0`:
  - Fetch log window via `LogWindowFetcher`:
    - Filter: `cron_fingerprint:"<fp>" AND run_id:"<run_id>"` (enabled by STAGE-004-005's enrichment)
    - Anchor: run's finished_at
    - Window: from run's `started_at` to `finished_at + 5s` (entire run + small grace)
    - Limit: 50 lines (configurable; cron output is usually small)
- Store enrichment in SQLite `cron_run_failure_enrichments`

### 2. SQLite schema

```sql
CREATE TABLE cron_run_failure_enrichments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    fingerprint TEXT NOT NULL,
    run_id TEXT NOT NULL,
    finished_at INTEGER NOT NULL,
    exit_code INTEGER NOT NULL,
    lines_json TEXT NOT NULL,
    line_count INTEGER NOT NULL,
    truncated INTEGER NOT NULL DEFAULT 0,
    created_at INTEGER NOT NULL,
    UNIQUE (fingerprint, run_id)
);
CREATE INDEX idx_cron_fail_enrich_fp ON cron_run_failure_enrichments(fingerprint, finished_at);
```

Retention: 30 days (cron failures are higher-signal and longer-lived; configurable via `HOMELAB_MONITOR_CRON_ENRICHMENT_RETENTION_DAYS=30`).

### 3. Enrichment endpoint

```
GET /api/crons/{fingerprint}/runs/{run_id}/failure-enrichment -> CronFailureEnrichment
```

(The existing per-run-log endpoint from STAGE-002 already returns the full run log; this enrichment is the same content surfaced in a different shape for alert integration.)

### 4. vmalert rule integration

Cron-run-failed alerts already exist (from EPIC-002). This stage UPDATES the existing rule annotation template (in `deploy/vmalert/metrics/cron.yml` or wherever) to:

- Reference the new enrichment endpoint
- Provide a deep-link to the cron run log viewer (STAGE-002 already produced this) for full context
- Update notification templates to render the enrichment lines

### 5. Alertmanager template enhancement

Update the cron-failed alert template to include the last 10 lines of output (truncated; full available via UI link).

### 6. UI: render enrichment on alert detail + cron run page

- `apps/ui/src/routes/alerts/AlertDetailPage.tsx` — for cron-failed alerts, fetch + render the enrichment lines
- The existing cron run log viewer (`CronRunLogViewer.tsx`) already shows the full log; nothing to add there

### 7. Tests

- Backend: detector unit tests, SQLite repo, endpoint
- Integration: simulate a failed cron run, assert enrichment row + retrievable
- Frontend: AlertDetailPage renders cron-failure enrichment

## Out of scope for this stage

- Cross-run failure pattern detection (e.g., "this cron has failed N times this week") — defer to a future epic
- Auto-restart cron — EPIC-009 (auto-fix); just emit alert
- Cron-run-skipped detection — EPIC-002 territory

## Decisions in this stage

### Design phase will confirm

- **Line limit** — 50 recommended for cron (output usually small)
- **Retention** — 30 days recommended (cron events more important to keep long-term than container crashes)
- **Whether to also enrich timeout failures** (cron killed by hmrun timeout) — recommendation: yes; same path

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-CRON-FAILURE-ENRICHMENT** — separate stage, not bundled with crash correlation; parallel architecture.
- **D-CRON-USES-SHARED-FETCHER** — `LogWindowFetcher` from STAGE-004-031.
- **D-CRON-RETAIN-30D** — longer than container crash because cron failures often investigated days later.
