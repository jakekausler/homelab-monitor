# STAGE-004-019: Histogram of line counts (bar chart above results showing density over time)

## Status: Build

## Overview

Above the line list in the Logs Explorer, render a bar chart showing the line count per time-bucket across the current range. Buckets size scale with range: 1-min buckets for a 15-min range, 1-hour buckets for a 24-hour range, etc.

The histogram is the fastest way to spot bursts ("oh, error volume tripled around 14:30"). Clicking a bar narrows the time range to that bucket's window.

**Vertical slice:** above results in the Explorer, a small bar chart shows counts per bucket. Click a bar → time range narrows to that bucket; results refresh accordingly. Severity coloring on the bars (red portion = error+ lines, yellow = warn, gray = info) gives at-a-glance density-by-severity.

## Concrete deliverables

### 1. Backend endpoint: bucketed line counts

`apps/monitor/homelab_monitor/kernel/api/routers/logs.py` — new endpoint:

```
GET /api/logs/histogram?expr=<LogsQL>&start=<ISO>&end=<ISO>&buckets=60
```

Response:

```python
class LogsHistogramResponse(BaseModel):
    buckets: list[HistogramBucket]
    bucket_duration_ms: int  # so the UI can render axis labels

class HistogramBucket(BaseModel):
    start_ts: datetime  # bucket start (UTC)
    counts_by_severity: dict[str, int]  # {"error": 12, "warn": 4, "info": 200, ...}
    total: int
```

Implementation: LogsQL `<expr> | stats by (_time:bucket_ms, severity) count() | head <buckets>` (or equivalent VL syntax). Default 60 buckets across the requested range.

### 2. `<HistogramChart>` component

`apps/ui/src/routes/logs/HistogramChart.tsx` (new) — small stacked-bar chart:

- Each bar = one bucket; height = total count
- Stacked by severity (error red bottom, warn yellow middle, info gray top), so the visible color tells the story at a glance
- Hover any bar → tooltip with bucket time + counts by severity
- Click a bar → emit `(bucketStart, bucketEnd)` to parent; parent narrows the range

Library: reuse the project's existing chart primitive if one exists (e.g., the `<Sparkline>` from `apps/ui/src/components/tiles/Sparkline.tsx` — likely needs extension to support stacked bars and click handlers). If extension is impractical, evaluate a small lib like Recharts (lazy-loaded).

### 3. Explorer integration

`LogsExplorerPage.tsx` — render `<HistogramChart>` between the search controls and the `<LogViewer>`. Bar click narrows range and triggers a re-query.

### 4. Performance considerations

- 60 buckets max — don't overload the chart
- Backend caches `(expr_hash, range_hash)` for 30s — same as fields endpoint
- Frontend skips fetching the histogram when results are empty (no point)

### 5. Tests

- Backend: `apps/monitor/tests/test_api_logs_histogram.py` — response shape, bucket alignment, severity grouping
- Frontend: `__tests__/HistogramChart.test.tsx` — renders bars, click emits range, tooltip content

## Design Phase

### Selected Approach: Option A — Native VictoriaLogs `/select/logsql/hits` with `field=severity` (exact per-bucket per-severity counts in ONE call)

Backend `GET /api/logs/histogram?expr=&start=&end=&buckets=60&services=` (auth `Depends(require_session())`):
- Reuse `_compose_services_expr(expr, services)` + `parse_and_validate_window(start, end)`. Map `VictoriaLogsClientError` → 502 `upstream_unavailable`.
- **NEW `VictoriaLogsClient.hits(expr, start, end, step, field="severity") -> list[HitsSeries]`** → GET `/select/logsql/hits`, parses `{"hits":[{"fields":{"severity":"<raw>"},"timestamps":[...ISO...],"values":[...counts...],"total":N}, ...]}` into `HitsSeries(field_value, timestamps, counts)`. Mirrors `field_names()`: small JSON (no NDJSON cap), tolerant parse (skip malformed rows, `[]` on unparseable body), wraps transport/non-200 in `VictoriaLogsClientError`. `field=severity` returns ONE series per distinct raw severity token.
- **NEW `kernel/logs/histogram.py`** (mirrors `kernel/logs/fields.py`):
  - `coarse_bucket(raw_severity) -> "error"|"warn"|"info"`: runs raw token through the existing `_normalize_severity()` (single source of truth) then a coarse map (error/critical/alert/emergency→error; warn→warn; info/notice/debug/None→info). PURE, unit-testable.
  - `compute_buckets(start_ms, end_ms, buckets) -> (step_ms, count)`: `step = ceil(span/buckets)`, START-aligned (bucket i = `[start+i*step, start+(i+1)*step)`), guarantees ≤buckets entries.
  - `assign_bucket(ts_ms, start_ms, step_ms, n)`: `floor((ts-start)/step)` clamped [0,n-1] — RE-BINS VL's epoch/step-aligned timestamps onto start-aligned buckets (VL aligns to epoch, not `start`).
  - `HistogramCache` (mirrors FieldsCache): key `(sha256(effective_expr), start, end, buckets)`, 30s TTL, injectable clock.
  - `fetch_histogram(client, expr, start, end, buckets) -> LogsHistogramResponse`: compute step, call `client.hits(field="severity")`, re-bin each raw-severity series onto start-aligned buckets via `assign_bucket`, coarse-map each raw token, sum per (bucket, coarse-severity). Emit EXACTLY `buckets` entries, time-ascending, each with all 3 coarse keys present (zeros included) so the stacked chart has no gaps. Empty range → `buckets` zero-filled entries.
- `GET /api/logs/histogram` route: module-scoped `_histogram_cache`; cache check → miss → fetch_histogram → store. `buckets` param default 60 (clamp/validate sensibly).

Response models (new in `kernel/api/schemas.py`, `extra="forbid"`):
```
HistogramBucket: start_ts: str (ISO-8601 UTC bucket start; the click target); counts_by_severity: dict[str,int] ({"error":N,"warn":N,"info":N}); total: int
LogsHistogramResponse: buckets: list[HistogramBucket] (exactly `buckets`, time-ascending); bucket_duration_ms: int (step; frontend computes [start, start+dur))
```

Frontend: NEW `apps/ui/src/routes/logs/HistogramChart.tsx` — recharts stacked `<BarChart>` (recharts ^3.8.1 already a dep, no existing usage). Bars stacked by coarse severity, colors reused from `apps/ui/src/.../severity.ts` (red=error+ #ef4444, yellow=warn #eab308, gray=info/other). Hover tooltip; click a bar → range becomes `[bucket.start_ts, bucket.start_ts + bucket_duration_ms)` committed via the page's `writeUrl`/range path. Mounted in `LogsExplorerBody.tsx` BETWEEN the control bar and `<LogViewer>`. `useLogsHistogramQuery` hook mirrors `useLogsFieldsQuery` (key `['logs','histogram',expr,start,end,buckets,services]`, staleTime 30_000, enabled when expr/start/end set). jsdom test: recharts ResponsiveContainer needs a mocked/fixed parent width (no existing chart-test precedent — establish one).

**Verified VL v0.30.0 facts (backend-contract verification):**
- `/select/logsql/hits` exists since v0.8.0; `field=`-grouping on the HTTP endpoint landed v0.25.0 (2024-06-28); pinned v0.30.0 (2024-09-27) POSTDATES it → `field=severity` grouping WORKS. (The v1.37.0+ "group by field" changelog entries are Web-UI only — red herrings.)
- Params: `query` (req), `start`/`end` (optional; **`end` INCLUSIVE `[start,end]` in v0.30.0** — exclusive is v1.36.0+, account for in bucket math), `step` (req; duration string s/m/h/d, ms-granularity), `field` (groups by distinct values; multiple allowed), `fields_limit`, `offset`.
- Response: one entry per distinct field value, parallel `timestamps[]` (bucket-START times, ISO-8601 UTC, **aligned to step/epoch NOT to `start`** → re-bin in Python) + `values[]` (per-bucket counts) + `total`. Without `field`: single `fields:{}` entry = total-per-bucket.
- `severity` is a queryable low-cardinality VL string field — grouping by it is the intended use.
- `VictoriaLogsClient` does NOT yet wrap `/hits` — net-new method, mirrors the `field_names()` GET+tolerant-JSON-parse pattern.

**Rejected alternatives:**
- **Option B (native `/hits` WITHOUT grouping, N per-coarse-severity filtered calls):** rejected as primary, KEPT as documented fallback ONLY if field-grouping were unavailable. The v0.30.0 check CONFIRMED grouping works → fallback not triggered. B is exact but 3-4 HTTP calls/request AND duplicates the raw→coarse severity table inside LogsQL filters (a second place severity logic lives; a forgotten raw token = silent miscount).
- **Option C (client-side: bounded `/query` + Python bucketing):** REJECTED OUTRIGHT. Cap-approximate — a busy range exceeding the line cap yields a WRONG-but-plausible histogram. A histogram's entire value is ACCURATE density, so cap-induced approximation is a correctness defect, not an acceptable v1 trade-off (contrast STAGE-018, where sampling was acceptable for example VALUES because coverage% self-documents rarity; density has no such escape hatch). Also transfers many lines to do work VL does server-side.

**Forward-compat:** independent of 020 (export) / 021 (open-in-Explorer). No new surface they depend on.

**Refinement:** frontend 3a (Desktop+Mobile) + a MANDATORY integration test confirming v0.30.0 `/hits` field-grouping returns per-severity series on the real rig (the one behavior mocked unit tests can't validate).
