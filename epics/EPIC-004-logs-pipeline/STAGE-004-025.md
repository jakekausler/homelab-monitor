# STAGE-004-025: drain3 wrapper + SQLite persistence layer

## Status: Complete

## Overview

Introduce the Drain log clustering infrastructure as a standalone, well-tested layer with NO ingest loop yet. This stage delivers:

- A `DrainEngine` Python module wrapping `drain3` with project-conventional types
- A `SqlitePersistence` backend (custom drain3 persistence implementation) that serializes models to a `drain_models` SQLite table
- A pluggable model-key function with the locked cron special-case (`service="hmrun"` → `cron:<fingerprint>`)
- 100% unit-tested against fixture log lines

The periodic batch consumer that USES this layer is STAGE-004-026; the metric emission + manual refresh is STAGE-004-027. Splitting the layer first lets us bottom-out the data model + persistence semantics without entangling them with the consumer's polling logic.

**Vertical slice:** Python code can `engine = DrainEngine(persistence=SqlitePersistence(...)); engine.add_line(line); engine.snapshot()` and replay later with the model state intact. Tests against fixture lines from HA, Pi-hole, hmrun.

**Spec references:** master spec §6.3 (Drain log clustering).

### 3. `DrainEngine` wrapper

`apps/monitor/homelab_monitor/kernel/logs/drain_engine.py` (new):

```python
class DrainEngine:
    def __init__(self, persistence: DrainPersistence, model_key_fn: Callable[[LogLine], str] = default_model_key):
        ...

    def add_line(self, line: LogLine) -> SignatureEvent:
        """Feed one line through the appropriate model. Returns the matched/new template ID + metadata."""

    def snapshot(self) -> None:
        """Persist all dirty models to storage."""

    def get_model(self, key: str) -> drain3.TemplateMiner:
        """Lazy-load (and cache in memory) the model for this key."""

    def templates(self, key: str) -> list[SignatureTemplate]:
        """Return all templates currently in this model."""
```

`SignatureEvent`:

```python
class SignatureEvent(BaseModel):
    service_key: str  # model key
    template_id: int  # drain3's template ID within this model
    template_hash: str  # stable hash of the template string (cross-restart stable)
    template_str: str  # the canonical template (with wildcards)
    is_new: bool  # was this template first-seen this call?
    first_seen_ts: datetime  # timestamp the template was first emitted
    last_seen_ts: datetime  # this event's timestamp
```

`SignatureTemplate`:

```python
class SignatureTemplate(BaseModel):
    template_id: int
    template_hash: str
    template_str: str
    line_count: int  # how many lines have matched this template
    first_seen_ts: datetime
    last_seen_ts: datetime
```

### 4. Model-key function (with override hook)

```python
def default_model_key(line: LogLine) -> str:
    """Default: key by `service`. Special-case hmrun → `cron:<fingerprint>`."""
    if line.service == "hmrun":
        fp = line.fields.get("cron_fingerprint") or "unknown"
        return f"cron:{fp}"
    return line.service or "unknown"
```

The constructor accepts a custom `model_key_fn` so future sources (Synology DSM, UDM, etc.) can override if needed. Default works for the v1 sources.

### 5. `SqlitePersistence` backend

`apps/monitor/homelab_monitor/kernel/logs/drain_persistence.py` (new):

```python
class DrainPersistence(Protocol):
    def load(self, key: str) -> bytes | None: ...
    def save(self, key: str, snapshot: bytes, line_count: int, template_count: int, last_processed_ts: int | None) -> None: ...
    def all_keys(self) -> list[str]: ...

class SqlitePersistence(DrainPersistence):
    def __init__(self, conn: sqlite3.Connection): ...
```

drain3's persistence interface (`drain3.persistence_handler.PersistenceHandler`) is wrapped to delegate to this protocol; project tests verify drain3 wires through correctly.

### 6. Template hash stability

`template_hash` MUST be stable across restarts: a SHA-256 (or similar) of the canonical template string, NOT drain3's internal template_id (which is per-instance). This is critical for downstream metrics + signature catalog rows (later stages) to remain consistent.

## Decisions in this stage

### Design phase will confirm

- **drain3 version pin** — current stable (`0.9.11` as of 2026; confirm latest)
- **Template hash function** — SHA-256 of canonical template string (recommended)
- **`max_clusters` per model** — drain3 default is unlimited; set to 5000 per model with overflow eviction policy. Tune later if needed.
- **Snapshot frequency** — every batch cycle (consumer-driven); this stage's `snapshot()` is the primitive

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-DRAIN-LIB-DRAIN3** — use `drain3` (IBM, MIT-licensed) rather than hand-rolled or alternatives.
- **D-DRAIN-SQLITE-PERSISTENCE** — single `drain_models` table; one row per model. Matches project's SQLite-everything convention.
- **D-PER-SERVICE-MODELS** — default model key is `service`; cron special-case to `cron:<fingerprint>`; override hook for future sources.
- **D-TEMPLATE-HASH-STABLE** — hash the canonical template string for cross-restart stability.

### Verified drain3 real API (front-loaded risk, confirmed against source)

- Library: `drain3` v0.9.11 (current 0.9.x line). PEP-621 dependency string: `drain3>=0.9.11` (NO caret — pyproject uses `>=` constraints).
- `TemplateMiner(persistence_handler=None, config=None)` — auto-loads saved state in `__init__` when a persistence_handler is supplied.
- `add_log_message(message: str) -> dict` returns keys: `change_type` ("none"|"cluster_created"|"cluster_modified"|...), `cluster_id` (int, per-instance counter), `cluster_size` (int), `template_mined` (str), `cluster_count` (int).
- `PersistenceHandler` ABC: `save_state(self, state: bytes) -> None`, `load_state(self) -> Optional[bytes]`. State is BYTES (jsonpickle of the Drain object, optionally zlib+base64 when snapshot_compress_state=True).
- drain3 AUTO-CALLS save_state inside add_log_message on `change_type != "none"` or when snapshot_interval_minutes elapses — a control concern handled below.
- `TemplateMinerConfig()` configured by direct attribute assignment (drain_sim_th, drain_depth, drain_max_clusters, snapshot_interval_minutes, snapshot_compress_state). No .ini file needed.
- Iterate templates: `for cluster in miner.drain.clusters: cluster.cluster_id / cluster.get_template() / cluster.size`.
- CONFIRMED: drain3 `cluster_id` is a per-instance incrementing counter, UNSTABLE across restarts/instances — validating D-TEMPLATE-HASH-STABLE (we hash the canonical template string ourselves).

### Q1 — Sync-drain3 ↔ async-SQLite boundary: CHOSEN = in-memory buffer + explicit async snapshot()

drain3's `save_state` only stores bytes in an in-memory `dict[model_key, bytes]` (synchronous, non-blocking). The engine exposes `async def snapshot()` that drains the buffer and writes each model row via the normal async `SqliteRepository`. `load_state` returns bytes pre-loaded asynchronously (via the repo) before the TemplateMiner is constructed.

- Rejected (A) dedicated synchronous stdlib sqlite3 connection — violates the "all DB I/O goes through SqliteRepository" convention; second connection to same file. (Keep as fallback if async pre-load sequencing proves hard in Build.)
- Rejected (C) asyncio.run_coroutine_threadsafe from sync context — brittle, deadlock-prone, untestable.

### Q2 — Explicit snapshot() vs drain3 auto-save: CHOSEN = neuter auto-save + buffer

Set `config.snapshot_interval_minutes = 1440` (24h) and `config.snapshot_compress_state = True`. drain3 may still call save_state on change, but save_state is a dict write (no I/O). Real persistence happens ONLY in the engine's `async snapshot()`.

- Rejected: write-through save_state — re-introduces the sync/async bridge, blocking I/O in add_line hot path, non-deterministic snapshot timing.

### Q3 — Value types + first_seen source: CHOSEN = engine-owned first_seen map persisted in an added column

Frozen dataclasses (`@dataclass(frozen=True, slots=True)`), timestamps as unix-ms INT (matches the card's INT columns for drain_models):

SignatureEvent: model_key:str, template_id:int (drain3 cluster_id; in-run correlation only), template_hash:str (sha256 hex of template_str), template_str:str, is_new:bool, cluster_size:int, first_seen_ts:int (unix ms), last_seen_ts:int (unix ms).

SignatureTemplate: model_key:str, template_id:int, template_hash:str, template_str:str, size:int, first_seen_ts:int, last_seen_ts:int.

first_seen tracking: the engine maintains a per-model `dict[template_hash, first_seen_ms]`, seeded on the first line producing that hash, and persists it. CHOSEN persistence: one ADDITIVE column on drain_models — `first_seen_map TEXT` (JSON map) — beyond the card's originally-suggested columns.

- WHY the added column: STAGE-004-027 emits a `homelab_log_signature_first_seen_ts` gauge that needs durable first-seen backing data; deferring first_seen to 028 would leave that gauge with no backing data for a full stage. Still ONE table, ONE row per model — consistent with D-DRAIN-SQLITE-PERSISTENCE.
- Rejected (b) derive first_seen purely from line.timestamp — wrong for any line after the first. (c) defer to 028 — leaves 027's gauge unbacked.

### Q4 — template_hash stability & is_new: CHOSEN = recompute hash each line; is_new = hash-not-yet-seen-in-model

`template_hash = sha256_hex(cluster.get_template())` recomputed every add_line. `is_new = template_hash not in <model's first_seen_map>`.

- Precise meaning of D-TEMPLATE-HASH-STABLE: "stable" = DETERMINISTIC ACROSS RESTARTS for an unchanged template string (the same string always hashes the same, unlike drain3's per-instance cluster_id). It does NOT mean a cluster keeps one hash forever — drain3 generalizes templates in place ("user 5 logged in" → "user <NUM> logged in"), and a generalized template legitimately becomes a NEW signature hash. This is the intended catalog behavior.
- Rejected (B) pin cluster_id→first-hash — cluster_id is unstable across restart, defeats the stability goal. (C) is_new = change_type=="cluster_created" — decouples is_new from hash-novelty, contradicts the card's "is_new = first time this hash seen" definition.
- Document this semantic explicitly in the drain_engine.py module docstring.

### Q5 — default_model_key: CHOSEN = service bucket; hmrun→cron:<canonical-command-hash>; None→"_unknown"

```
def default_model_key(line: LogLine) -> str:
    svc = line.service
    if svc is None:
        return "_unknown"
    if svc == "hmrun":
        cmd = line.fields.get("command")
        if isinstance(cmd, str) and cmd:
            fp = sha256(canonical_log_key(cmd).encode()).hexdigest()[:16]
        else:
            fp = "unknown"
        return f"cron:{fp}"
    return svc
```

- Cron fingerprint = sha256(canonical_log_key(command))[:16], REUSING the existing `kernel/cron/log_match.py::canonical_log_key` (per-job stable, NOT per-run). Documented fallback `cron:unknown` when the command field is absent.
- None service → "_unknown" catch-all bucket (never raise / never silently drop inside the clustering ingest path).
- Rejected: run_id fingerprint (per-run model explosion), message-hash fingerprint (too volatile), raise/skip on None (fragile/lossy).
