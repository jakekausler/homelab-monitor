# STAGE-004-025: drain3 wrapper + SQLite persistence layer

## Status: Not Started

## Overview

Introduce the Drain log clustering infrastructure as a standalone, well-tested layer with NO ingest loop yet. This stage delivers:

- A `DrainEngine` Python module wrapping `drain3` with project-conventional types
- A `SqlitePersistence` backend (custom drain3 persistence implementation) that serializes models to a `drain_models` SQLite table
- A pluggable model-key function with the locked cron special-case (`service="hmrun"` → `cron:<fingerprint>`)
- 100% unit-tested against fixture log lines

The periodic batch consumer that USES this layer is STAGE-004-026; the metric emission + manual refresh is STAGE-004-027. Splitting the layer first lets us bottom-out the data model + persistence semantics without entangling them with the consumer's polling logic.

**Vertical slice:** Python code can `engine = DrainEngine(persistence=SqlitePersistence(...)); engine.add_line(line); engine.snapshot()` and replay later with the model state intact. Tests against fixture lines from HA, Pi-hole, hmrun.

**Spec references:** master spec §6.3 (Drain log clustering).

### 2. SQLite migration

`apps/monitor/homelab_monitor/kernel/db/migrations/` — new alembic migration:

```sql
CREATE TABLE drain_models (
    service_key TEXT PRIMARY KEY,  -- e.g., "home-assistant", "cron:<fingerprint>"
    snapshot BLOB NOT NULL,  -- drain3 tree serialization
    line_count INTEGER NOT NULL DEFAULT 0,  -- lines processed by this model
    template_count INTEGER NOT NULL DEFAULT 0,  -- templates currently in the tree
    last_processed_ts INTEGER,  -- nanoseconds; cursor into VL for the periodic consumer
    updated_at INTEGER NOT NULL
);
CREATE INDEX idx_drain_models_updated ON drain_models(updated_at);
```

### 3. `DrainEngine` wrapper

`apps/monitor/homelab_monitor/kernel/logs/drain_engine.py` (new):

```python
class DrainEngine:
    def __init__(self, persistence: DrainPersistence, model_key_fn: Callable[[LogLine], str] = default_model_key):
        ...

    def add_line(self, line: LogLine) -> SignatureEvent:
        """Feed one line through the appropriate model. Returns the matched/new template ID + metadata."""

    def snapshot(self) -> None:
        """Persist all dirty models to storage."""

    def get_model(self, key: str) -> drain3.TemplateMiner:
        """Lazy-load (and cache in memory) the model for this key."""

    def templates(self, key: str) -> list[SignatureTemplate]:
        """Return all templates currently in this model."""
```

`SignatureEvent`:

```python
class SignatureEvent(BaseModel):
    service_key: str  # model key
    template_id: int  # drain3's template ID within this model
    template_hash: str  # stable hash of the template string (cross-restart stable)
    template_str: str  # the canonical template (with wildcards)
    is_new: bool  # was this template first-seen this call?
    first_seen_ts: datetime  # timestamp the template was first emitted
    last_seen_ts: datetime  # this event's timestamp
```

`SignatureTemplate`:

```python
class SignatureTemplate(BaseModel):
    template_id: int
    template_hash: str
    template_str: str
    line_count: int  # how many lines have matched this template
    first_seen_ts: datetime
    last_seen_ts: datetime
```

### 4. Model-key function (with override hook)

```python
def default_model_key(line: LogLine) -> str:
    """Default: key by `service`. Special-case hmrun → `cron:<fingerprint>`."""
    if line.service == "hmrun":
        fp = line.fields.get("cron_fingerprint") or "unknown"
        return f"cron:{fp}"
    return line.service or "unknown"
```

The constructor accepts a custom `model_key_fn` so future sources (Synology DSM, UDM, etc.) can override if needed. Default works for the v1 sources.

### 5. `SqlitePersistence` backend

`apps/monitor/homelab_monitor/kernel/logs/drain_persistence.py` (new):

```python
class DrainPersistence(Protocol):
    def load(self, key: str) -> bytes | None: ...
    def save(self, key: str, snapshot: bytes, line_count: int, template_count: int, last_processed_ts: int | None) -> None: ...
    def all_keys(self) -> list[str]: ...

class SqlitePersistence(DrainPersistence):
    def __init__(self, conn: sqlite3.Connection): ...
```

drain3's persistence interface (`drain3.persistence_handler.PersistenceHandler`) is wrapped to delegate to this protocol; project tests verify drain3 wires through correctly.

### 6. Template hash stability

`template_hash` MUST be stable across restarts: a SHA-256 (or similar) of the canonical template string, NOT drain3's internal template_id (which is per-instance). This is critical for downstream metrics + signature catalog rows (later stages) to remain consistent.

## Decisions in this stage

### Design phase will confirm

- **drain3 version pin** — current stable (`0.9.11` as of 2026; confirm latest)
- **Template hash function** — SHA-256 of canonical template string (recommended)
- **`max_clusters` per model** — drain3 default is unlimited; set to 5000 per model with overflow eviction policy. Tune later if needed.
- **Snapshot frequency** — every batch cycle (consumer-driven); this stage's `snapshot()` is the primitive

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-DRAIN-LIB-DRAIN3** — use `drain3` (IBM, MIT-licensed) rather than hand-rolled or alternatives.
- **D-DRAIN-SQLITE-PERSISTENCE** — single `drain_models` table; one row per model. Matches project's SQLite-everything convention.
- **D-PER-SERVICE-MODELS** — default model key is `service`; cron special-case to `cron:<fingerprint>`; override hook for future sources.
- **D-TEMPLATE-HASH-STABLE** — hash the canonical template string for cross-restart stability.
