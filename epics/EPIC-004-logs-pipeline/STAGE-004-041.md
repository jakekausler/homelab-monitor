# STAGE-004-041: VL backend health alerts (L3) — VL down, write latency, disk thresholds

## Status: Complete

## Overview

A set of vmalert rules + supporting metrics to detect VictoriaLogs backend problems before users notice silently empty log viewers. Includes the VL-specific disk threshold alerting (the locked decision from brainstorming 13c — each store has independent thresholds; this stage owns VL's).

Categorization: `anomaly_kind: log_backend_health` and `category: ops`.

**Vertical slice:** alerts fire for: VL down (no response to health probe), VL write latency exceeding threshold, VL data-dir usage exceeding warn/crit thresholds. Each alert links to the homelab-monitor UI's settings/logs page for diagnostics.

**Spec references:** master spec §3.2 (sidecars) + §6.4 (disk budget).

## Concrete deliverables

### 1. VL health probe collector

`apps/monitor/homelab_monitor/plugins/collectors/builtin/vl_health.py` (new):

- Runs every 30s
- HTTP GET `<vl_url>/health` (transport timeout 5s)
- Emit `homelab_vl_up{}` (gauge, 1 = healthy, 0 = down)
- Emit `homelab_vl_response_time_seconds` (histogram of probe response times)
- Emit `homelab_vl_last_check_age_seconds` (gauge of seconds since last successful check)

### 2. VL disk usage collector

Extend or create:

- Read `/storage/.../victorialogs/` directory size (via `os.statvfs` for mount-aware percentage)
- Emit `homelab_vl_disk_used_bytes{}` (gauge)
- Emit `homelab_vl_disk_used_pct{}` (gauge)

### 3. Configurable thresholds

Env vars:
- `HOMELAB_MONITOR_VL_DISK_WARN_PCT=70`
- `HOMELAB_MONITOR_VL_DISK_CRIT_PCT=85`
- `HOMELAB_MONITOR_VL_HEALTH_TIMEOUT_S=5`

These are surfaced in the Settings → Logs UI (STAGE-004-022).

### 4. vmalert rules

`deploy/vmalert/metrics/vl_health.yml` (new):

```yaml
groups:
  - name: vl_health
    interval: 30s
    rules:
      - alert: VictoriaLogsDown
        expr: homelab_vl_up == 0
        for: 1m
        labels:
          severity: critical
          source_tool: homelab-monitor
          category: ops
          anomaly_kind: log_backend_health
        annotations:
          summary: "VictoriaLogs is down"
          description: |
            VL health probe has failed for 1+ minute.
            URL: /settings/logs

      - alert: VictoriaLogsHighLatency
        expr: histogram_quantile(0.95, rate(homelab_vl_response_time_seconds_bucket[5m])) > 2.0
        for: 5m
        labels:
          severity: warning
          source_tool: homelab-monitor
          category: ops
          anomaly_kind: log_backend_health
        annotations:
          summary: "VictoriaLogs slow"
          description: |
            VL p95 response time is {{ $value }}s over the last 5 minutes.
            URL: /settings/logs

      - alert: VictoriaLogsDiskWarning
        expr: homelab_vl_disk_used_pct > 70
        for: 10m
        labels:
          severity: warning
          source_tool: homelab-monitor
          category: ops
          anomaly_kind: log_backend_health
        annotations:
          summary: "VictoriaLogs disk usage at {{ $value }}%"
          description: |
            VL data directory at {{ $value }}% capacity. Consider shortening retention.
            URL: /settings/logs

      - alert: VictoriaLogsDiskCritical
        expr: homelab_vl_disk_used_pct > 85
        for: 5m
        labels:
          severity: critical
          source_tool: homelab-monitor
          category: ops
          anomaly_kind: log_backend_health
        annotations:
          summary: "VictoriaLogs disk usage at {{ $value }}% — critical"
          description: |
            VL data directory critically full. Logs may be dropped if not addressed.
            URL: /settings/logs
```

## Design Phase

**Design phase complete: 2026-06-08.**
**Status**: [x] Design (Complete)

### Design Notes

**SELECTED APPROACH: Minimal new surface (YAGNI)** — build ONLY the VL health-probe collector + 4 rules; reuse self_disk's existing per-slot bytes for the disk pct via a recording rule; reuse the existing VlDiskWarningConfig + STAGE-022 /settings/logs UI.

**METRICS (all GAUGES — the MetricsWriter has NO histogram support, only write_gauge/counter/summary):**
- `homelab_vl_up` (gauge 1.0 on HTTP 200, 0.0 on non-200/timeout/transport-error)
- `homelab_vl_response_time_seconds` (GAUGE = last-probe latency, always emitted incl. on failure — NOT a histogram as the spec assumed)
- `homelab_collector_run_vl_health` (standard self-metric, free from BaseCollector)
- Recording rule `homelab:vl_disk_used_pct = 100 * homelab_self_disk_used_bytes{slot="vl"} / homelab_self_disk_budget_bytes{slot="vl"}` (reuses self_disk's existing slot="vl" bytes — NO new disk collector)

**COLLECTOR (the only genuinely-new Python):** `VlHealthCollector` at `apps/monitor/homelab_monitor/plugins/collectors/builtin/vl_health.py` — BaseCollector, interval 30s, GET `{vl_url}/health` with timeout `HOMELAB_MONITOR_VL_HEALTH_TIMEOUT_S` (default 5s), mirrors `log_stream_budget`'s vl_url+http_client post-construction injection in lifespan.py. Emits `homelab_vl_up` + `homelab_vl_response_time_seconds`. DROPPED `homelab_vl_last_check_age_seconds` (YAGNI — `absent(homelab_vl_up)` covers collector-death; an age gauge re-emitted every 30s reading ~0 adds no signal the Down rule lacks).

**RULES** (`deploy/vmalert/metrics/vl_health.yaml` — .yaml NOT .yml; group vl_health, interval 30s):
- record: `homelab:vl_disk_used_pct` (the recording rule above)
- `VictoriaLogsDown`: `homelab_vl_up == 0 or absent(homelab_vl_up)`, for:1m, severity critical (the `absent()` arm catches collector-death; `==0` catches VL-down-collector-alive — both = silently-empty log viewer)
- `VictoriaLogsHighLatency`: `quantile_over_time(0.95, homelab_vl_response_time_seconds[5m]) > 2`, for:5m, severity warning (quantile_over_time on the gauge — ~10 samples/5m at 30s interval; valid since no histogram)
- `VictoriaLogsDiskWarning`: `homelab:vl_disk_used_pct > 70`, for:10m, severity warning
- `VictoriaLogsDiskCritical`: `homelab:vl_disk_used_pct > 85`, for:5m, severity critical
- Thresholds 70/85 BAKED into the rules (mirror self_disk.yaml which bakes 70/85/95); VlDiskWarningConfig remains the source of truth for the 022 UI DISPLAY. Add a YAML comment noting the coupling.
- All labels: severity, source_tool: vmalert-metrics, category: ops, anomaly_kind: log_backend_health. Deep-link annotation /settings/logs.

**CONFIG:** add `HOMELAB_MONITOR_VL_HEALTH_TIMEOUT_S` (default 5.0s) loader in kernel/config.py (small, mirror existing env-loader pattern). VlDiskWarningConfig (70/85) already exists — no change.

**FORWARD-REF (doc-only deliverable):** add a note to `epics/EPIC-014-self-monitor/EPIC-014.md` that VM-health + SQLite-health mirror this exact pattern (singleton _up gauge + last-probe-latency gauge + recording-rule disk pct from self_disk{slot=...} + 4-rule YAML).

**COVERAGE:** only vl_health.py + the tiny config loader are new kernel Python → 100% coverage (mirror log_stream_budget's tests: success/non-200/timeout/transport → up=0 + latency-on-failure). Disk rule + recording rule + config-already-exists → no coverage delta.

**SPEC CORRECTIONS (6, all confirmed by code-reading):**
1. histogram → gauge (no write_histogram in MetricsWriter); HighLatency uses quantile_over_time.
2. .yml → .yaml (vmalert globs *.yaml).
3. NO new VL-disk collector — recording rule over self_disk's slot="vl" bytes (avoids a redundant /var/vl-data walk; self_disk already walks it).
4. `homelab_vl_disk_used_pct` never existed and STAGE-022's /settings/logs UI calls `compute_vl_disk_usage()` DIRECTLY (reads no VM metric) — so no metric-name contract to honor; the new series is the recording-rule-namespaced `homelab:vl_disk_used_pct`.
5. source_tool: vmalert-metrics (NOT the spec's "homelab-monitor") — every existing metrics rule uses vmalert-metrics; an orphan value would break Karma/Alertmanager routing consistency. DEVIATION from spec, made deliberately to conform to the codebase + routing; noted for audit.
6. DROPPED `homelab_vl_last_check_age_seconds` + the floated ingest-rate metric (YAGNI).

**HOST-INTEGRATION (3a+3b):** 3a — confirm `homelab_vl_up=1` normally; stop the VL sidecar → `homelab_vl_up=0` + VictoriaLogsDown fires. 3b — real /var/vl-data drives `homelab:vl_disk_used_pct`; confirm latency p95 against real VL /health; confirm the recording rule + disk alerts evaluate on real self_disk bytes.

**REJECTED:** A-iii (new/extended VL-disk collector — redundant /var/vl-data walk, the 040 redundant-collector anti-pattern). A-ii (inline division in each alert expr — works but less readable than the recording rule).
