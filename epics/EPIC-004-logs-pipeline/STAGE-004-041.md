# STAGE-004-041: VL backend health alerts (L3) — VL down, write latency, disk thresholds

## Status: Not Started

## Overview

A set of vmalert rules + supporting metrics to detect VictoriaLogs backend problems before users notice silently empty log viewers. Includes the VL-specific disk threshold alerting (the locked decision from brainstorming 13c — each store has independent thresholds; this stage owns VL's).

Categorization: `anomaly_kind: log_backend_health` and `category: ops`.

**Vertical slice:** alerts fire for: VL down (no response to health probe), VL write latency exceeding threshold, VL data-dir usage exceeding warn/crit thresholds. Each alert links to the homelab-monitor UI's settings/logs page for diagnostics.

**Spec references:** master spec §3.2 (sidecars) + §6.4 (disk budget).

## Concrete deliverables

### 1. VL health probe collector

`apps/monitor/homelab_monitor/plugins/collectors/builtin/vl_health.py` (new):

- Runs every 30s
- HTTP GET `<vl_url>/health` (transport timeout 5s)
- Emit `homelab_vl_up{}` (gauge, 1 = healthy, 0 = down)
- Emit `homelab_vl_response_time_seconds` (histogram of probe response times)
- Emit `homelab_vl_last_check_age_seconds` (gauge of seconds since last successful check)

### 2. VL disk usage collector

Extend or create:

- Read `/storage/.../victorialogs/` directory size (via `os.statvfs` for mount-aware percentage)
- Emit `homelab_vl_disk_used_bytes{}` (gauge)
- Emit `homelab_vl_disk_used_pct{}` (gauge)

### 3. Configurable thresholds

Env vars:
- `HOMELAB_MONITOR_VL_DISK_WARN_PCT=70`
- `HOMELAB_MONITOR_VL_DISK_CRIT_PCT=85`
- `HOMELAB_MONITOR_VL_HEALTH_TIMEOUT_S=5`

These are surfaced in the Settings → Logs UI (STAGE-004-022).

### 4. vmalert rules

`deploy/vmalert/metrics/vl_health.yml` (new):

```yaml
groups:
  - name: vl_health
    interval: 30s
    rules:
      - alert: VictoriaLogsDown
        expr: homelab_vl_up == 0
        for: 1m
        labels:
          severity: critical
          source_tool: homelab-monitor
          category: ops
          anomaly_kind: log_backend_health
        annotations:
          summary: "VictoriaLogs is down"
          description: |
            VL health probe has failed for 1+ minute.
            URL: /settings/logs

      - alert: VictoriaLogsHighLatency
        expr: histogram_quantile(0.95, rate(homelab_vl_response_time_seconds_bucket[5m])) > 2.0
        for: 5m
        labels:
          severity: warning
          source_tool: homelab-monitor
          category: ops
          anomaly_kind: log_backend_health
        annotations:
          summary: "VictoriaLogs slow"
          description: |
            VL p95 response time is {{ $value }}s over the last 5 minutes.
            URL: /settings/logs

      - alert: VictoriaLogsDiskWarning
        expr: homelab_vl_disk_used_pct > 70
        for: 10m
        labels:
          severity: warning
          source_tool: homelab-monitor
          category: ops
          anomaly_kind: log_backend_health
        annotations:
          summary: "VictoriaLogs disk usage at {{ $value }}%"
          description: |
            VL data directory at {{ $value }}% capacity. Consider shortening retention.
            URL: /settings/logs

      - alert: VictoriaLogsDiskCritical
        expr: homelab_vl_disk_used_pct > 85
        for: 5m
        labels:
          severity: critical
          source_tool: homelab-monitor
          category: ops
          anomaly_kind: log_backend_health
        annotations:
          summary: "VictoriaLogs disk usage at {{ $value }}% — critical"
          description: |
            VL data directory critically full. Logs may be dropped if not addressed.
            URL: /settings/logs
```

## Decisions in this stage

### Design phase will confirm

- **Warn/crit thresholds (70 / 85)** — recommended; consistent with master spec §6.4
- **Latency p95 threshold (2s)** — recommended; tune in 3b based on real-world cadence
- **Health probe interval (30s)** — recommended
- **Whether to include cumulative VL ingest rate as a related metric** — recommendation: yes, useful for understanding context

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-PER-STORE-THRESHOLDS** — VL has its own thresholds; VM and SQLite have their own independently (forward-ref).
- **D-VL-HEALTH-ALERTABLE** — down, latency, disk-warn, disk-crit all alertable.
- **D-NO-COORDINATOR** — no cross-store auto-shrink (locked).
