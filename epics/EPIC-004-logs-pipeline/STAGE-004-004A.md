# STAGE-004-004A: Docker log severity-level extraction

## Status: Not Started

## Overview

Docker container logs currently arrive with `severity` defaulting to `"info"` (Vector's `add_labels` transform sets `.severity` from `.level`/`.PRIORITY`, neither of which docker_logs provides), so genuinely error/warn lines render untinted in `<LogViewer>`. This stage adds severity-level extraction for docker logs so the structured `LogLine.severity` field reflects the real level — enabling the severity tint (added in STAGE-004-003) to work for docker logs.

**Vertical slice:** after this stage, docker logs with severity tokens (e.g., `ERROR`, `WARN`, `CRITICAL`) render with appropriate severity tinting in `<LogViewer>`, matching the visual treatment of other log sources.

**Spec references:** Master spec §3.2 (vector sidecar + log processing).

## Concrete deliverables

### 1. Severity extraction location (Design-phase decision)

Choose ONE approach:

- **Option A (ingest-time):** Vector remap transform in `deploy/vector/vector.toml.template` parses message text for common severity tokens before lines hit VictoriaLogs. One-time CPU cost per line; queries fast. Requires vector config validation.
- **Option B (read-time):** Backend mapping function in `apps/monitor/homelab_monitor/kernel/logs/models.py` `from_victorialogs_line()` (or docker-specific adapter) parses severity on query. Simpler vector config; higher per-query CPU on large result sets. Risk: log lines stored with wrong severity for historical queries.

**Recommendation:** Option A (ingest-time) — matches the pattern from STAGE-004-004 (label enrichment), avoids future misclassification, and aligns with the redaction-pipeline pattern (STAGE-004-006).

### 2. Severity token parsing

Extract severity from message text matching these patterns (decide in Design which are high-confidence enough):

- Leading-token anchors: `^ERROR`, `^WARN(ING)?`, `^CRITICAL`, `^DEBUG`, `^INFO` (case-insensitive)
- ANSI color hint patterns: `\x1b\[31m` (red, maps to error) `\x1b\[33m` (yellow, maps to warn) — if message body carries ANSI codes
- Structured log patterns: if message contains JSON, check for `.level` / `.severity` / `.type` fields (decide scope in Design)

Map to canonical severity enum: `debug` / `info` / `notice` / `warn` / `error` / `critical` / `alert` / `emergency`.

### 3. False-positive avoidance

Prevent misclassification of lines that merely mention "error" in prose:

- **Anchored patterns only** — match tokens at line start or after standard log prefixes (timestamp, service name), not mid-sentence
- **Whitelist high-confidence keywords** — e.g., only `ERROR`, `CRITICAL`, `FATAL` are error-tier; avoid generic words like "issue" or "problem"
- **Design decision:** Define the final set of trusted keywords; document exclusions

Example false positives to avoid:
- `This operation encountered an error but recovered.` — should NOT escalate severity
- `ERROR is not the same as exception.` — prose mention, should NOT match

### 4. Implementation (ingest option)

Vector VRL remap in `deploy/vector/vector.toml.template`:

```toml
[transforms.docker_severity_extract]
type = "remap"
inputs = ["docker_enrich"]  # after label enrichment
source = '''
# If .severity is already set (non-docker source), leave it
# For docker, .severity defaults to "info"; extract from message
if is_null(.severity) || .severity == "info" {
  # Anchored pattern match: line starts with severity token
  if match!(.message, r"^(ERROR|CRITICAL|FATAL|ALERT|EMERGENCY)", "i") {
    .severity = "error"
  } else if match!(.message, r"^(WARN|WARNING)", "i") {
    .severity = "warn"
  } else if match!(.message, r"^(DEBUG)", "i") {
    .severity = "debug"
  } else if match!(.message, r"^(NOTICE)", "i") {
    .severity = "notice"
  }
  # Else leave at default "info"
}
'''
```

(Final VRL syntax confirmed in Design phase against vector's current language; may require guard checks for null .message.)

## Out of scope for this stage

- Extraction for journald / syslog / cron sources — docker-specific; future stages or source-specific stages handle others
- Per-line override UI (e.g., "this line's severity is wrong, let me manually correct it") — not in spec
- Severity recalibration based on downstream alert history — YAGNI
- Configurable severity keyword set (hard-coded in code or env var, not UI) — v1 is fixed; future if demand emerges

## Decisions in this stage

### Design phase will confirm

- **Extraction location** — Option A (ingest, vector transform) vs Option B (read-time, backend mapping). Recommendation: A.
- **Trusted severity keywords** — final anchored-pattern set and case-sensitivity rules.
- **ANSI color detection** — include or exclude? Recommendation: include if docker logs in test rig carry ANSI; skip if not observable.
- **Structured log parsing** — if message is JSON, should we check `.level` field? Recommendation: Design to skip in v1; add in follow-up if real docker logs carry JSON structure.

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-SEVERITY-EXTRACTION-DOCKER** — docker logs are eligible for severity extraction; rendering severity tint depends on this stage.
- **D-ANCHORED-PATTERNS-ONLY** — false-positive avoidance via leading-token anchor rules, not mid-sentence matching.
- **D-CANONICAL-SEVERITY-ENUM** — maps to `LogLine.severity` enum (debug/info/notice/warn/error/critical/alert/emergency).
