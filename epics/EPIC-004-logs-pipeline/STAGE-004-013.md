# STAGE-004-013: Saved queries (SQLite table + UI to save / name / load / delete)

## Status: Complete

## Overview

User can save the current Explorer state (LogsQL, selected services, time range or relative since-preset) as a named query and recall it later. Saved queries persist in SQLite so they survive sessions.

This stage adds the simplest possible save/load/delete UX. It does NOT integrate with alert-from-query (STAGE-004-044 does that).

**Vertical slice:** in the Explorer, click "Save query…" → modal asks for a name → query is persisted. The user can later open a "Saved queries" panel (sidebar tab or popover) and click an entry to load it.

**Spec references:** master spec §9.2 ("saved queries").

## Concrete deliverables

### 1. SQLite schema

`apps/monitor/homelab_monitor/kernel/db/migrations/` — new alembic migration:

```sql
CREATE TABLE log_saved_queries (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE,
    logs_ql TEXT NOT NULL,
    selected_services TEXT NOT NULL,  -- JSON array
    since_preset TEXT,  -- '15m', '1h', etc.; NULL if custom range
    range_start_iso TEXT,  -- ISO-8601; NULL if preset
    range_end_iso TEXT,  -- ISO-8601; NULL if preset
    advanced_mode INTEGER NOT NULL DEFAULT 0,  -- 0/1
    created_at INTEGER NOT NULL,  -- unix ms
    updated_at INTEGER NOT NULL
);
CREATE INDEX idx_log_saved_queries_name ON log_saved_queries(name);
```

## Decisions in this stage

### Selected approach (Design phase, decided 2026-06-02)

- **List location: Sidebar tab** — saved-queries list lives as a tab/section next to the existing StreamPickerSidebar (012A), reusing the collapsible-section pattern. Persistent and visible while browsing. (Rejected: popover under a "Saved" button — hides the list, fights existing sidebar layout.)
- **Mode mismatch on load: Force mode switch** — loading a query forces Explorer into the saved query's mode (advanced LogsQL vs plain-text). Guarantees D-SAVE-FULL-STATE reproducibility; loading a LogsQL query into plain-text mode would silently break it.
- **Max saved queries: No hard cap** — list sorted by name, sidebar scrolls. Single-user homelab tool; realistic counts are dozens. Search/folders are out of scope (future stage if >~500).
- **Name collision (D-UNIQUE-NAMES): Server 409 → inline modal error** — duplicate name returns HTTP 409; SaveQueryModal surfaces "name already exists" inline so user renames and retries. No silent overwrite.

### Inherited locked decisions (unchanged)

- D-SAVED-QUERIES-SQLITE: persist in SQLite (survives sessions)
- D-SAVE-FULL-STATE: restore LogsQL + selected services + range + mode
- D-UNIQUE-NAMES: UNIQUE constraint on name

### Seed data (confirmed)

3 sample saved queries for Refinement manual testing:
1. Plain-text mode + preset range (e.g. nginx errors, since=1h)
2. Advanced LogsQL mode + custom start/end date range
3. Plain-text mode + multi-service selection

### Build-phase note

Existing repo convention (suggestions_repository.py / compose_actions_repository.py) stores timestamps as strings set in Python via `self._repo.transaction()` + `text(sql)`. Align `created_at`/`updated_at` to that convention rather than the card's "[unix ms]" phrasing; pick one consistent representation in Build.
