# STAGE-004-013: Saved queries (SQLite table + UI to save / name / load / delete)

## Status: Not Started

## Overview

User can save the current Explorer state (LogsQL, selected services, time range or relative since-preset) as a named query and recall it later. Saved queries persist in SQLite so they survive sessions.

This stage adds the simplest possible save/load/delete UX. It does NOT integrate with alert-from-query (STAGE-004-044 does that).

**Vertical slice:** in the Explorer, click "Save query…" → modal asks for a name → query is persisted. The user can later open a "Saved queries" panel (sidebar tab or popover) and click an entry to load it.

**Spec references:** master spec §9.2 ("saved queries").

## Concrete deliverables

### 1. SQLite schema

`apps/monitor/homelab_monitor/kernel/db/migrations/` — new alembic migration:

```sql
CREATE TABLE log_saved_queries (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL UNIQUE,
    logs_ql TEXT NOT NULL,
    selected_services TEXT NOT NULL,  -- JSON array
    since_preset TEXT,  -- '15m', '1h', etc.; NULL if custom range
    range_start_iso TEXT,  -- ISO-8601; NULL if preset
    range_end_iso TEXT,  -- ISO-8601; NULL if preset
    advanced_mode INTEGER NOT NULL DEFAULT 0,  -- 0/1
    created_at INTEGER NOT NULL,  -- unix ms
    updated_at INTEGER NOT NULL
);
CREATE INDEX idx_log_saved_queries_name ON log_saved_queries(name);
```

### 2. Repo layer

`apps/monitor/homelab_monitor/kernel/logs/saved_queries_repo.py` (new):

- `create(name, logs_ql, selected_services, since_preset, range_start, range_end, advanced_mode) -> SavedQuery`
- `list() -> list[SavedQuery]` (sorted by name)
- `get(id) -> SavedQuery | None`
- `delete(id) -> None`
- `rename(id, new_name) -> SavedQuery`
- 100% test coverage

### 3. API endpoints

`apps/monitor/homelab_monitor/kernel/api/routers/logs.py`:

- `GET /api/logs/saved-queries` → list all
- `POST /api/logs/saved-queries` → create (body: SavedQueryCreate)
- `PATCH /api/logs/saved-queries/{id}` → rename
- `DELETE /api/logs/saved-queries/{id}` → delete

CSRF on POST/PATCH/DELETE (project convention).

### 4. Frontend hook + components

`apps/ui/src/api/savedLogQueries.ts` — React Query hooks for list/create/patch/delete with appropriate cache invalidation.

`apps/ui/src/routes/logs/SavedQueriesPanel.tsx` (new) — list of saved queries with click-to-load + rename + delete actions. Could be a sidebar tab OR a popover under a "Saved" button next to the search box. Design phase picks.

`apps/ui/src/routes/logs/SaveQueryModal.tsx` (new) — modal for naming a query. Validates unique-name client-side (with server-side enforcement via UNIQUE constraint).

### 5. Explorer integration

`LogsExplorerPage.tsx`:

- "Save query…" button near the search box → opens modal
- "Saved queries" entry point → opens panel/popover
- Loading a saved query restores: LogsQL, selected services, range (preset or custom), mode (plain/advanced)
- URL updates so the loaded state is deep-linkable

### 6. Tests

- Backend repo tests
- Backend endpoint tests (including CSRF, name uniqueness, validation)
- Frontend hook + component tests

## Decisions in this stage

### Design phase will confirm

- **Sidebar tab vs popover for the saved queries list** — recommendation: sidebar tab (sits next to the stream picker)
- **What happens if a saved query was created in advanced mode and the user is currently in plain mode** — recommendation: loading forces a mode switch
- **Max number of saved queries** — recommendation: no hard cap; just sort by name and add a UI scroll. If it ever becomes a problem (>500) we add search

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-SAVED-QUERIES-SQLITE** — persisted via SQLite so they survive sessions and migrations.
- **D-SAVE-FULL-STATE** — saved query restores LogsQL + selected services + range + mode (everything that makes the query reproducible).
- **D-UNIQUE-NAMES** — name is unique (UNIQUE constraint); UI prompts for rename on collision.
