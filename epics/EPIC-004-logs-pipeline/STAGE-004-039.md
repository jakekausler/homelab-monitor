# STAGE-004-039: Severity escalation rules (L1) тАФ any line at severity=critical triggers paging

## Status: Complete

## Overview

A simple safety net: if ANY log line at `severity=critical` (or `emergency`/`alert` per syslog severity levels) appears from ANY service in the last 5 min, fire a high-priority alert. Today there's nothing watching the severity field at this level.

Categorization: `anomaly_kind: severity_escalation`.

**Vertical slice:** any critical-severity log line lands in VL тЖТ within 5 min, vmalert fires a `CriticalLogLine` alert with the line content + service + a deep-link to the Explorer.

**Spec references:** master spec ┬з6.3.

### 2. vmalert rule

`deploy/vmalert/logs/severity_escalation.yml` (new):

```yaml
groups:
  - name: severity_escalation
    interval: 30s
    rules:
      - alert: CriticalLogLine
        expr: increase(homelab_log_lines_total{severity=~"critical|emergency|alert"}[5m]) > 0
        for: 0m
        labels:
          severity: critical
          source_tool: homelab-monitor
          category: log-anomaly
          anomaly_kind: severity_escalation
        annotations:
          summary: "Critical log line from {{ $labels.service }}"
          description: |
            Service {{ $labels.service }} emitted {{ $value }} lines at severity={{ $labels.severity }} in the last 5 minutes.
            URL: /logs?logsql=service:"{{ $labels.service }}" AND severity:critical
```

### Design Notes

**SELECTED APPROACH:** Option A тАФ Static LogsQL rule mirroring KernelOOM; per-service overrides deferred to STAGE-042.

**Rationale:**
1. Satisfies the EPIC-004 criterion "rule pattern matches existing KernelOOM rule shape" тАФ KernelOOM is a LogsQL `type: vlogs` rule.
2. Follows the precedent set by STAGE-037 (error_rate_spike) and STAGE-038 (signature_silent), which shipped a static v1 rule and deferred per-service overrides to STAGE-042 with a parsed-but-unused config schema.
3. Querying VictoriaLogs directly avoids the counter/increase() VM-staleness class of bug.
4. Smallest correct change, fully reversible.

**Rejected Approaches:**
- **Option B** (build per-severity metric + MetricsQL rule) тАФ Contradicts the KernelOOM-shape criterion, reintroduces increase() staleness, heaviest coverage burden, YAGNI.
- **Option C** (render overrides from config now) тАФ Front-runs STAGE-042's mandate, adds a render layer KernelOOM lacks.

### KEY CORRECTNESS FINDING

The VictoriaLogs `severity` field is NOT uniform. Docker-origin lines carry WORD severities (critical/emergency/alert); journald/systemd-origin lines carry NUMERIC syslog PRIORITY (0=emergency, 1=alert, 2=critical) because vector's docker_severity_extract transform only runs for docker lines. **The rule MUST match BOTH representations or it silently misses every journald-origin critical line.**

Severity match set: `severity:(critical OR emergency OR alert OR "0" OR "1" OR "2")`

### DESIGN DECISIONS

- **Rule file:** `deploy/vmalert/logs/severity_escalation.yaml` тАФ `type: vlogs` group, alert `CriticalLogLine`, expr LogsQL `_time:5m severity:(critical OR emergency OR alert OR "0" OR "1" OR "2") | stats by (service) count() as crit_count | filter crit_count:>0`, for: 0m.
- **Fast-fire twin:** `deploy/vmalert/logs-test/severity_escalation.yaml` (shorter window/interval for integration fast-fire testing, mirroring `deploy/vmalert/logs-test/system.yaml` convention).
- **Labels (modern block, NOT KernelOOM's stale labels):** severity=critical, source_tool=vmalert-logs, category=log-anomaly, anomaly_kind=severity_escalation, target_kind=container (or service тАФ match sibling rules).
- **Annotations:** summary referencing `{{ $labels.service }}` + description + Explorer deep-link (pattern from error_rate_spike.yaml: `/logs?logsql=...`).
- **Config schema (parsed-but-unused in v1, used in STAGE-042):** add to `LogsConfig` in `kernel/config.py` тАФ `severity_escalation_excluded_services: tuple[str, ...]` and `severity_escalation_floors: tuple[SeverityFloor, ...]` (new frozen dataclass `SeverityFloor(service: str, floor: str)`), mirroring the existing `ErrorRateOverride` parsed-but-unused pattern + its loader. Loader parses `logs.severity_escalation.excluded_services` and `logs.severity_escalation.severity_floors`. Include an explicit "reserved for STAGE-042; parsed but UNUSED in v1" comment.
- **Tests:** vmalert rule test (planted critical line fires; non-critical does not; numeric-severity line fires) following the `*.tests.yaml` unit-test format used by `signature_silent.tests.yaml`; config loader tests for the new parsed-but-unused fields (in `test_config.py`). 100% kernel coverage maintained (config loader is the only new Python; rule + test fixtures are YAML).

### SPEC CORRECTIONS

1. The assumed metric `homelab_log_lines_total{service,severity}` does NOT exist тАФ approach uses direct LogsQL, not MetricsQL.
2. Filename `.yml` тЖТ `.yaml` (vmalert globs `*.yaml`, `.yml` silently skipped).
3. Directory `deploy/vmalert/logs/` not `deploy/vmalert/metrics/`.
4. Rule is `type: vlogs`.
