# STAGE-004-039: Severity escalation rules (L1) — any line at severity=critical triggers paging

## Status: Not Started

## Overview

A simple safety net: if ANY log line at `severity=critical` (or `emergency`/`alert` per syslog severity levels) appears from ANY service in the last 5 min, fire a high-priority alert. Today there's nothing watching the severity field at this level.

Categorization: `anomaly_kind: severity_escalation`.

**Vertical slice:** any critical-severity log line lands in VL → within 5 min, vmalert fires a `CriticalLogLine` alert with the line content + service + a deep-link to the Explorer.

**Spec references:** master spec §6.3.

## Concrete deliverables

### 1. Per-severity line count metric (if not present)

Verify that the existing log-stream-budget collector already emits per-severity counts. If not, extend:

`homelab_log_lines_total{service, severity}` — counter of lines per service per severity. Emitted every 60s by a collector via VL `stats by (service, severity) count()`.

(May already exist from STAGE-001-018.)

### 2. vmalert rule

`deploy/vmalert/logs/severity_escalation.yml` (new):

```yaml
groups:
  - name: severity_escalation
    interval: 30s
    rules:
      - alert: CriticalLogLine
        expr: increase(homelab_log_lines_total{severity=~"critical|emergency|alert"}[5m]) > 0
        for: 0m
        labels:
          severity: critical
          source_tool: homelab-monitor
          category: log-anomaly
          anomaly_kind: severity_escalation
        annotations:
          summary: "Critical log line from {{ $labels.service }}"
          description: |
            Service {{ $labels.service }} emitted {{ $value }} lines at severity={{ $labels.severity }} in the last 5 minutes.
            URL: /logs?logsql=service:"{{ $labels.service }}" AND severity:critical
```

### 3. Per-service overrides

`homelab-monitor.yaml`:

```yaml
logs:
  severity_escalation:
    excluded_services:  # services that legitimately emit critical lines without action needed
      - "noisy-test-container"
    severity_floors:  # alternative: per-service severity floor
      "noisy-test-container": "emergency"  # only emergency, not critical
```

Render into rule overrides at boot.

### 4. Tests

- vmalert rule test: planted critical line fires; excluded service does not
- Per-service override test

## Out of scope for this stage

- Auto-page-on-call routing — Alertmanager owns that
- Per-line content matching — covered by Drain anomalies
- Severity normalization at ingest — depends on vector's transforms; if a service uses non-standard severity strings (e.g., "FATAL"), Drain consumer normalizes; this rule expects normalized values

## Decisions in this stage

### Design phase will confirm

- **Severity set** — `critical|emergency|alert` recommended (covers syslog severities 0-2)
- **Window** — 5 min recommended
- **Whether to include `error` severity** — recommend no for this rule (error rate spike covers it via STAGE-004-037; this rule is for catastrophic-only)

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-CRITICAL-ALWAYS-ALERTS** — `critical` severity is unconditionally alertable unless service is explicitly excluded.
- **D-PER-SERVICE-OVERRIDES** — yaml-configurable excluded services + severity floors.
