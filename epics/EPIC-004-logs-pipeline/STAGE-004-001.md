# STAGE-004-001: Multi-line log handling (Python tracebacks, Java stack traces stitched into single events)

## Status: Refinement (Complete)

## Overview

Today the vector `docker_logs` and `journald` sources emit one VL record per physical line. Python tracebacks, Java stack traces, and any other multi-line log emission produce one VL record per line — each with a different Drain template, fragmenting both queries and (especially) future signature clustering.

This stage enables vector's `multiline` codec on the relevant sources so that consecutive continuation lines are stitched into a single logical event before the line ever reaches the throttle / VL sink. This lands FIRST in EPIC-004 because every subsequent stage — `<LogViewer>` extraction, Drain clustering, anomaly rules — produces dramatically cleaner output when multi-line events are coherent.

**Vertical slice:** after this stage, a Python traceback emitted by any homelab-monitor or HA container appears in `/api/logs/query` as a single multi-line `_msg` (with embedded `\n`), not 12 separate records. The `<LogViewer>` (built in STAGE-004-003) will render it as a single expandable block.

**Spec references:** Master spec §3.2 (vector sidecar), §6.3 (VictoriaLogs streams). EPIC-004 §"Out-of-the-box log noise reduction".

## Concrete deliverables

### 1. Multi-line codec on docker_logs source

`deploy/vector/vector.toml.template` — add `multiline` block to `[sources.docker_logs]`:

- `start_pattern` matches the canonical line start (typically a timestamp prefix or non-whitespace at column 0)
- `condition_pattern` matches continuation lines (whitespace-indented OR matches known traceback markers like `File "`, `    at `, `Caused by:`, `\tat `, etc.)
- `mode = "continue_through"` so continuation lines accumulate into the start line
- `timeout_ms` bounded (default 1000ms) so unfinished multi-lines don't stall forever

Final regex set agreed in Design phase based on the actual log corpus in VL today.

### 2. Multi-line codec on journald source

`deploy/vector/vector.toml.template` — add `multiline` block to `[sources.journald]` covering the same continuation patterns plus systemd-specific cases (kernel oops blocks, sshd auth failure stanzas, etc.).

### 3. Multi-line handling for the hmrun journald-filtered source

The `hmrun` source (`SYSLOG_IDENTIFIER == "hmrun"`) — confirm whether cron job output ever produces multi-line emissions; if yes, add a matching multiline block; if no, document why no config needed.

### 4. Render-on-boot validation

`apps/monitor/homelab_monitor/kernel/cron/render.py` — extend the render-time test fixture set (`tests/test_render_vector_toml.py` or sibling) to cover the new multiline blocks. Ensure that when env vars are unset the rendered TOML still parses and the multiline regexes are syntactically valid.

### 5. Fixture-driven regression tests

`apps/monitor/tests/integration/test_vector_multiline.py` — fixtures containing planted multi-line emissions (Python traceback, Java stack trace, sshd auth-failure stanza, kernel oops):

- Spawn a test rig vector + VL pair (existing integration test pattern)
- Plant each fixture via `logger -p user.info` or a test container's stdout
- Assert each multi-line event appears in VL as a SINGLE record with `_msg` containing the joined text (with `\n` line breaks)
- Assert no leftover fragments (each fragment is part of exactly one record)

### 6. Documentation

`docs/dev/local-environment.md` (or sibling) — short paragraph explaining what the multiline codec does and how to add new continuation patterns when a future source emits multi-line text.

## Decisions in this stage

### Design phase will confirm

- **Exact regex set** for `start_pattern` and `condition_pattern` — Design phase reviews actual VL log corpus and locks the regex list per source. Initial seed: timestamp-prefix start, whitespace/tab continuation.
- **Timeout** for `multiline.timeout_ms` — default 1000ms; Design phase validates against worst-case real-world traceback emission cadence.
- **Whether to gate the multiline codec behind a feature flag** for the first deploy — Design phase decides based on rollback risk.

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-MULTILINE-FIRST** — multi-line handling lands FIRST in EPIC-004 because all downstream stages (`<LogViewer>` extraction, Drain, anomalies) produce dramatically cleaner output once multi-line events are coherent.
- **D-MULTILINE-GLOBAL-PATTERNS** — patterns are global, not per-service. Per-service overrides are YAGNI; revisit only if a real source needs it.
- **D-MULTILINE-CONTINUE-THROUGH** — vector's `continue_through` mode (continuation lines accumulate into the start) — matches the natural shape of tracebacks and stack traces.

## Design Phase

- **Approaches presented:** narrow-safe vs broad-aggressive vs timestamp-anchored regex sets
- **UI Options Presented:** N/A — backend only
- **User Choice (regex set):** Narrow-safe — strict whitespace-indented continuation OR explicit traceback markers
- **User Choice (timeout_ms):** 1000ms (spec default)
- **User Choice (feature flag):** No flag — vector multiline is well-tested; rollback via revert + redeploy

  - `timeout_ms = 1000`
- **Seed Data Agreed:** noisy-logger `/log_lines` endpoint; test fixtures = one Python traceback (HA-style, 4 fragments), one Java stack trace (`at ` prefix), one indented kernel-oops-style block, plus one negative control of two consecutive unrelated single lines that must NOT stitch

**Refinement design evolution (3 iterations, all driven by Refinement-phase reality):**
1. Original Build used negative lookaheads in start_pattern. Vector's Rust `regex` crate rejects lookarounds → entire multiline block silently disabled. Unit tests with Python `re` passed; Vector rejected at runtime.
2. Switched to `halt_before` with `condition_pattern == start_pattern`. But halt_before cannot distinguish a LEADING exception line (Java `RuntimeException:` line 1) from a TRAILING one (Python `ValueError:` line 4) — the same token means "new event" vs "continuation" depending on language. Java/Ruby/Node/.NET/PHP shattered.
3. FINAL: `continue_through` mode. start_pattern = first-line markers (timestamps, Traceback, panic:, exception-class lines, PHP Fatal, ...); condition_pattern = continuation markers (^\s, Caused by:, #N frames, goroutine, 0x..., thrown in, trailing exception terminators). Verified live against Vector 0.41.1 for all 9 languages + negative controls.
