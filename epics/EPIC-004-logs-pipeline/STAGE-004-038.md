# STAGE-004-038: Anomaly Type D — "Signature went silent" rules + expected-silence allowlist

## Status: Not Started

## Overview

A signature that has been emitting regularly suddenly stops. Often indicates a service stopped logging at all (= service crashed or hung), OR a signature has been legitimately silenced (e.g., the underlying cron job stopped running for a known reason).

Higher false-positive risk than the other types because legitimate stops are common. Mitigated by an **expected-silence allowlist** — user marks specific signatures (or services) as "expected to be silent during X period" so the rule doesn't fire.

Categorization: `anomaly_kind: signature_silent`.

**Vertical slice:** rule fires when a previously-active signature hasn't appeared in 15 min (configurable per-signature). Allowlist UI lets the user mark a signature as "expected to be silent overnight" or "expected to disappear after Feb 2026 when feature X is removed".

**Spec references:** EPIC-004 plan #6 (pattern abnormality detection — sequence/silence anomalies).

## Concrete deliverables

### 1. SQLite schema: expected-silence allowlist

`apps/monitor/homelab_monitor/kernel/db/migrations/` — new alembic migration:

```sql
CREATE TABLE log_signature_silence_allowlist (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    template_hash TEXT,  -- if NULL, applies to all signatures of the service
    service_key TEXT NOT NULL,
    schedule_kind TEXT NOT NULL,  -- 'always' | 'cron' | 'window'
    schedule_value TEXT,  -- cron expression for 'cron'; ISO range for 'window'; NULL for 'always'
    reason TEXT,  -- user-supplied
    created_at INTEGER NOT NULL,
    expires_at INTEGER  -- NULL = no expiry
);
CREATE INDEX idx_silence_service ON log_signature_silence_allowlist(service_key);
```

### 2. Silence-allowed metric collector

A small periodic collector (60s) evaluates each allowlist entry against current time and emits:

`homelab_log_signature_silence_allowed{template_hash, service_key} = 1` if currently in an allow-window, else 0.

vmalert rule uses this as a `unless` clause.

### 4. vmalert rule

`deploy/vmalert/logs/signature_silent.yml` (new):

```yaml
groups:
  - name: signature_silent
    interval: 60s
    rules:
      - alert: SignatureWentSilent
        # Signature exists (we have first_seen + last_seen tracking) but hasn't fired in 15m
        # AND was active in the last hour
        # UNLESS it's in the expected-silence allowlist
        expr: |
          (
            (time() - homelab_log_signature_last_seen_ts > 900)  # > 15 min silent
            and
            (time() - homelab_log_signature_last_seen_ts < 3600)  # active in last hour
          )
          unless on (template_hash, service_key) (homelab_log_signature_silence_allowed == 1)
          unless on (template_hash, service_key) (homelab_log_signature_suppressed == 1)
        for: 0m
        labels:
          severity: warning
          source_tool: homelab-monitor
          category: log-anomaly
          anomaly_kind: signature_silent
        annotations:
          summary: "Signature went silent: {{ $labels.service_key }} / {{ $labels.template_hash }}"
          description: |
            Template {{ $labels.template_hash }} from {{ $labels.service_key }} hasn't been emitted
            in 15 minutes despite being active in the last hour.
            URL: /logs/signatures#{{ $labels.template_hash }}/{{ $labels.service_key }}

            To suppress this alert when it's expected, add to silence allowlist.
```

(`homelab_log_signature_last_seen_ts` may need to be a new emitted metric — verify against what STAGE-004-027 emits and add if missing.)

## Decisions in this stage

### Design phase will confirm

- **Silence threshold** — 15 min recommended for "this was active and now isn't"
- **Active-window check** — 1 hour recommended (prevents firing on signatures that have always been rare)
- **Allowlist schedule UI** — recommendation: drill-in panel "Allow silence" button + dedicated tab for global list
- **Expected silence kinds (always / cron / window)** — all three supported in v1

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-SILENT-RULE-IN-V1** — Type D is in EPIC-004 (per user direction).
- **D-EXPECTED-SILENCE-ALLOWLIST** — prevents false positives on known-quiet signatures.
- **D-PER-(HASH,SERVICE)-ENTRIES** — allowlist entries can be per-signature OR per-service (template_hash NULL means "all signatures for this service").
