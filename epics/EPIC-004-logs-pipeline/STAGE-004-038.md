# STAGE-004-038: Anomaly Type D — "Signature went silent" rules + expected-silence allowlist

## Status: Complete

## Overview

A signature that has been emitting regularly suddenly stops. Often indicates a service stopped logging at all (= service crashed or hung), OR a signature has been legitimately silenced (e.g., the underlying cron job stopped running for a known reason).

Higher false-positive risk than the other types because legitimate stops are common. Mitigated by an **expected-silence allowlist** — user marks specific signatures (or services) as "expected to be silent during X period" so the rule doesn't fire.

Categorization: `anomaly_kind: signature_silent`.

**Vertical slice:** rule fires when a previously-active signature hasn't appeared in 15 min (configurable per-signature). Allowlist UI lets the user mark a signature as "expected to be silent overnight" or "expected to disappear after Feb 2026 when feature X is removed".

**Spec references:** EPIC-004 plan #6 (pattern abnormality detection — sequence/silence anomalies).

CREATE TABLE log_signature_silence_allowlist (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    template_hash TEXT,  -- if NULL, applies to all signatures of the service
    service_key TEXT NOT NULL,
    schedule_kind TEXT NOT NULL,  -- 'always' | 'cron' | 'window'

### 2. Silence-allowed metric collector

A small periodic collector (60s) evaluates each allowlist entry against current time and emits:

`homelab_log_signature_silence_allowed{template_hash, service_key} = 1` if currently in an allow-window, else 0.

vmalert rule uses this as a `unless` clause.

### 4. vmalert rule

`deploy/vmalert/logs/signature_silent.yml` (new):

```yaml
groups:
  - name: signature_silent
    interval: 60s
    rules:
      - alert: SignatureWentSilent
        # Signature exists (we have first_seen + last_seen tracking) but hasn't fired in 15m
        # AND was active in the last hour
        # UNLESS it's in the expected-silence allowlist
        expr: |
          (
            (time() - homelab_log_signature_last_seen_ts > 900)  # > 15 min silent
            and
            (time() - homelab_log_signature_last_seen_ts < 3600)  # active in last hour
          )
          unless on (template_hash, service_key) (homelab_log_signature_silence_allowed == 1)
          unless on (template_hash, service_key) (homelab_log_signature_suppressed == 1)
        for: 0m
        labels:
          severity: warning
          source_tool: homelab-monitor
          category: log-anomaly
          anomaly_kind: signature_silent
        annotations:
          summary: "Signature went silent: {{ $labels.service_key }} / {{ $labels.template_hash }}"
          description: |
            Template {{ $labels.template_hash }} from {{ $labels.service_key }} hasn't been emitted
            in 15 minutes despite being active in the last hour.
            URL: /logs/signatures#{{ $labels.template_hash }}/{{ $labels.service_key }}

            To suppress this alert when it's expected, add to silence allowlist.
```

(`homelab_log_signature_last_seen_ts` may need to be a new emitted metric — verify against what STAGE-004-027 emits and add if missing.)

## Design Decision — chosen approach (locked 2026-06-07)

**CHOSEN: Option C — Hybrid `SilenceDetectionCollector` (fold the whole decision into Python; DUMB vmalert rule).** Structurally identical to STAGE-035's `NewSignatureCollector`.

**A. Silence-detection signal — Hybrid collector (NOT a last_seen_ts metric, NOT PromQL gap-detection).**
A new `SilenceDetectionCollector` (a DB-reader `BaseCollector`, mirror `kernel/metrics/new_signature_collector.py`) runs every 60s, reads `log_signatures` (which already has `last_seen_at` unix-ms + `status`) cross-checked against the `log_signature_silence_allowlist` table, and emits `homelab_log_signature_silent{service_key, template_hash}=1` via `replace_family` ONLY for genuinely-silent-and-alertable signatures. Per-row decision:
```
silent_for_ms = now_ms - last_seen_at
if not (900_000 <= silent_for_ms <= 3_600_000): continue   # >15m silent AND active <1h ago
if status == "suppressed": continue                          # fold suppression (035 parity)
if allowlist_allows(service_key, template_hash, now): continue  # fold allowlist
emit (1.0, {service_key, template_hash})
```
The vmalert rule is a DUMB `homelab_log_signature_silent == 1`.

REJECTED **Option A (emit `homelab_log_signature_last_seen_ts` + metric-math rule):** FATAL VM-staleness bug — a gauge not re-emitted goes STALE/absent after VM's default 5-min staleness window, so a silent signature's series VANISHES at ~5min — BEFORE the 15-min threshold can ever be crossed → `time() - last_seen_ts > 900` evaluates over an absent series and NEVER fires. Fixing it (re-emit every cycle) collapses into Option C anyway, but with a hairier rule. (The card's `homelab_log_signature_last_seen_ts` metric DOES NOT EXIST — only the `last_seen_at` TABLE column does; the card flagged "may need a new metric — verify"; verified: a metric is the wrong shape.)

REJECTED **Option B (PromQL gap-detection on `_count`/`_total`):** `_count` is per-severity + gappy; "silent 15m AND active 1h" is a brittle 2-metric/2-window join; and CRUCIALLY suppression-exclusion + allowlist-schedule-exclusion are IMPOSSIBLE in PromQL (no suppressed metric exists; the allowlist lives in SQLite with cron/window schedules). Not 100%-coverage-testable.

**FOLD THE ALLOWLIST INTO THE SILENT GAUGE (new decision D-FOLD-ALLOWLIST-INTO-SILENT-GAUGE):** The card's deliverable #2 names a SEPARATE `homelab_log_signature_silence_allowed` gauge for an `unless` clause. SUPERSEDED — the allowlist decision is folded into the single `homelab_log_signature_silent` Python decision (a candidate currently in an allow-window is simply not emitted). Reasons: the VM-staleness/dumb-rule win comes from ONE Python decision driving ONE gauge; an `unless`-join between two collector gauges reintroduces a non-trivial rule + a second staleness surface; 035 set the precedent of folding suppression rather than emitting a `_suppressed` gauge — the allowlist is the direct analog. Deliverable #2's INTENT (allowlist prevents false positives) is fully satisfied; it just lives in the collector decision, not a standalone series.

**Why VM-staleness-proof:** the gauge is RE-EMITTED every tick for every alertable signature, so it never goes stale while the alert should be live. When the signature recovers / ages past 1h / gets suppressed / enters an allow-window, it stops being emitted → `replace_family` drops it → the alert self-resolves (035's resolution mechanic).

**B. Schedule evaluator — pure `is_silence_allowed(schedule_kind, schedule_value, now) -> bool`** in a new module `kernel/logs/silence_schedule.py` (no I/O, 100% unit-tested):
- `always` → True (permanent exemption).
- `window` → `schedule_value` is an ISO range `"<start-iso>/<end-iso>"`; allowed iff `start <= now <= end`. (maintenance/seasonal "known quiet" windows.)
- `cron` → reuse `kernel/cron/schedule.py` croniter (`canonicalize_schedule`, croniter). **D-CRON-SILENCE-SEMANTIC-V1:** a `cron` allowlist entry means "this signature only emits around its scheduled runs; silence between runs is EXPECTED." So `is_silence_allowed` returns True EXCEPT within a `grace_seconds` (default 900, config-tunable) post-fire ACTIVE window: `seconds_since_prev_cron_fire(now) > grace_s` → allowed. This suppresses the inevitable between-runs silence of scheduled-job loggers, while still catching a hung scheduled job (silent DURING its own post-fire active window → not allowed → emits the silent gauge).
- Cross-cutting (handled in the collector's allowlist MATCHER, not the pure evaluator): `expires_at` (nullable ISO) — an entry past its expiry is ignored entirely (filtered before evaluation); per-service entries (`template_hash IS NULL`) apply to ALL signatures of `service_key` (matcher checks per-(hash,service) first, then per-service; ANY matching non-expired currently-allowing entry suppresses the candidate). `canonicalize_schedule` applied at WRITE time (API field_validator) so stored cron values are canonical.

**C. Frontend — standalone `SilenceAllowlistTab` ONLY (form + list + delete). DEFER the SignatureDetailPage drill-in.**
The tab fully satisfies D-EXPECTED-SILENCE-ALLOWLIST self-contained, doesn't touch `SignatureDetailPage.tsx`, is independently testable. The drill-in "Allow silence" prefill button is convenience on an already-huge full-stack stage → DEFERRED (fast follow). Wiring: add `{ path: '/logs/silence-allowlist', label: 'Silence Allowlist' }` to `LogsLayout.tsx` TABS + a route in `router.tsx`; new `routes/logs/SilenceAllowlistTab.tsx`; new `api/silenceAllowlist.ts` react-query hooks (list/create/delete) modeled on `api/signatures.ts` (CSRF auto-attached by `client.ts` middleware).
