# STAGE-004-016B: JSON message drill-down in field inspector (recursive collapsible tree)

## Status: Not Started

## Overview

When a log line's `message` is valid JSON (e.g. structured collector logs like `{"collector":"docker_socket","event":"...","level":"info","tick_id":"...","trigger_kind":"scheduled","ts":"..."}`), the field inspector should render the message as a RECURSIVE collapsible tree — letting the user drill into nested objects and arrays — instead of showing it as a flat string. **Decided behavior (locked with user 2026-06-03):** the JSON tree REPLACES the flat-key behavior — when message parses as JSON, STOP flattening the JSON's top-level keys into sibling field rows (currently collector/event/level/etc. each show as their own row in addition to the message row). The message field instead becomes an expandable tree node; non-JSON messages render as today (plain string).

## Concrete deliverables

### 1. JSON detection and tree rendering

Detect when a `message` field is valid JSON (parse attempt; ignore primitives/numbers/booleans — only object/array are candidates for a tree view):
- If JSON object or array: render as a recursive collapsible tree component.
- If JSON primitive or non-JSON string: render as plain text (today's behavior).

Create `apps/ui/src/components/logs/JsonMessageTree.tsx` (or similar name) — a recursive tree view for JSON objects and arrays:
- Each node shows its key (for objects) or index (for arrays) and type hint.
- Objects render nested keys as child rows.
- Arrays render elements as numbered child rows.
- Expand/collapse toggle per node (using existing Radix Accordion or custom toggle).
- Depth limit / truncation handling for large/deeply-nested objects (Design decides; initial cap recommended: depth ≤ 10, size ≤ 5000 keys).
- Leaf nodes render typed values (string, number, boolean, null).

### 2. Integration into FieldInspectorPanel

Modify FieldInspectorPanel (from STAGE-004-016) to:
- Detect JSON-parseable `message` field on render.
- If JSON object/array: render JsonMessageTree component instead of flat string.
- If non-JSON or primitive: fall back to flat string render.

### 3. Suppress duplicate flat sibling rows for JSON-parsed keys

**Clarify during Design:** does the backend currently populate the `fields` bag with the parsed JSON's top-level keys (e.g. `fields.collector`, `fields.event`, etc.) when a structured collector logs a JSON message? If so:
- When message is JSON, the inspector should NOT show duplicate rows for those parsed keys in the `fields` bag (the tree shows them already).
- Implement a "suppress fields keys that are already in the JSON tree" filter during Design phase (i.e., if the tree contains key `collector`, don't show `fields.collector` as a separate row).
- If the backend does NOT populate these keys, no action needed — no duplicates to suppress.

### 4. Copy affordances (if applicable)

Per-leaf copy-to-clipboard on tree nodes (hover affordance)? Design decides scope:
- **Option A** (minimal): no copy on tree leaves; copy only the entire JSON via the parent field's copy button.
- **Option B** (moderate): copy-to-clipboard available per-leaf.
- **Option C** (rich): copy per-leaf + "copy subtree" on parent nodes.

Recommended initial scope: Option A (entire tree copy only). Leaves can be added in a follow-up if demand surfaces.

### 5. Add-to-filter on tree nodes (out of scope, may revisit)

Defer add-to-filter on deep leaves (could be added in a follow-up stage). Initial scope: copy only.

### 6. Tests

- **Tree rendering**: JSON object and array parsing, recursive rendering.
- **Expand/collapse**: state per node, children show/hide correctly.
- **Non-JSON fallback**: non-JSON strings and primitives render as plain text (no tree).
- **Depth/size handling**: large objects truncate or handle gracefully without crashes.
- **No duplicate rows**: when message is JSON, verify `fields` bag keys don't duplicate tree keys.
- **Copy**: entire tree copy to clipboard works.

## Out of scope for this stage

- Add-to-filter on deep leaves (copy only; may be added in a follow-up).
- JSON tree for arbitrary `fields`-bag values (this stage targets the `message` field specifically).
- Custom JSON formatters or syntax highlighting (plain structured tree only).
- Editing JSON in-place (view-only).
