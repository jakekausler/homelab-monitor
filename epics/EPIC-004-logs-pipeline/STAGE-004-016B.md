# STAGE-004-016B: JSON message drill-down in field inspector (recursive collapsible tree)

## Status: Complete

## Overview

When a log line's `message` is valid JSON (e.g. structured collector logs like `{"collector":"docker_socket","event":"...","level":"info","tick_id":"...","trigger_kind":"scheduled","ts":"..."}`), the field inspector should render the message as a RECURSIVE collapsible tree — letting the user drill into nested objects and arrays — instead of showing it as a flat string. **Decided behavior (locked with user 2026-06-03):** the JSON tree REPLACES the flat-key behavior — when message parses as JSON, STOP flattening the JSON's top-level keys into sibling field rows (currently collector/event/level/etc. each show as their own row in addition to the message row). The message field instead becomes an expandable tree node; non-JSON messages render as today (plain string).

## Concrete deliverables

### 1. JSON detection and tree rendering

Detect when a `message` field is valid JSON (parse attempt; ignore primitives/numbers/booleans — only object/array are candidates for a tree view):
- If JSON object or array: render as a recursive collapsible tree component.
- If JSON primitive or non-JSON string: render as plain text (today's behavior).

Create `apps/ui/src/components/logs/JsonMessageTree.tsx` (or similar name) — a recursive tree view for JSON objects and arrays:
- Each node shows its key (for objects) or index (for arrays) and type hint.
- Objects render nested keys as child rows.
- Arrays render elements as numbered child rows.
- Expand/collapse toggle per node (using existing Radix Accordion or custom toggle).
- Depth limit / truncation handling for large/deeply-nested objects (Design decides; initial cap recommended: depth ≤ 10, size ≤ 5000 keys).
- Leaf nodes render typed values (string, number, boolean, null).

### 2. Integration into FieldInspectorPanel

Modify FieldInspectorPanel (from STAGE-004-016) to:
- Detect JSON-parseable `message` field on render.
- If JSON object/array: render JsonMessageTree component instead of flat string.
- If non-JSON or primitive: fall back to flat string render.

### 3. Suppress duplicate flat sibling rows for JSON-parsed keys

**Clarify during Design:** does the backend currently populate the `fields` bag with the parsed JSON's top-level keys (e.g. `fields.collector`, `fields.event`, etc.) when a structured collector logs a JSON message? If so:
- When message is JSON, the inspector should NOT show duplicate rows for those parsed keys in the `fields` bag (the tree shows them already).
- Implement a "suppress fields keys that are already in the JSON tree" filter during Design phase (i.e., if the tree contains key `collector`, don't show `fields.collector` as a separate row).
- If the backend does NOT populate these keys, no action needed — no duplicates to suppress.

### 4. Copy affordances (if applicable)

- **Option B** (moderate): copy-to-clipboard available per-leaf.

Recommended initial scope: Option A (entire tree copy only). Leaves can be added in a follow-up if demand surfaces.

### 5. Add-to-filter on tree nodes (out of scope, may revisit)

Defer add-to-filter on deep leaves (could be added in a follow-up stage). Initial scope: copy only.

## Design Phase — Chosen Approach (2026-06-03)

**Design phase complete: 2026-06-03.**

### Backend contract verification (DECISIVE)

Vector's `parse_json` transform (deploy/vector/vector.toml.template ~85-99) parses the message: `parsed = parse_json(.message)`; if it's a JSON OBJECT, `. = merge(., parsed)` flattens the message's TOP-LEVEL keys into the event as fields. host/service/severity are saved before merge and RESTORED after, so they are NOT duplicated. The raw `message` string is KEPT unchanged.

So for a JSON message like `{"collector":"docker_socket","event":"...","level":"info","tick_id":"...","trigger_kind":"scheduled","ts":"..."}`: `LogLine.message` = raw JSON string; `LogLine.fields` contains the flattened TOP-LEVEL keys (collector, event, level, tick_id, trigger_kind, ts). THIS is the source of the user-observed duplication (message row + sibling rows).

Only TOP-LEVEL keys are flattened currently. STAGE-004-017 (future, NOT this stage) will add recursive `json.*` dotted-path flattening — out of scope. 016B is a VIEW concern (tree the message + suppress its top-level dupes); 017 is an INGEST concern.

**Evidence**: deploy/vector/vector.toml.template:85-99 (parse_json + merge), apps/monitor/homelab_monitor/kernel/logs/victorialogs_client.py:211 (fields = all non-builtin VL fields), models.py from_victorialogs_line.

### Chosen approach (8 decisions, all per brainstormer recommendation)

**1. JSON detection:** new helper `detectJsonMessage(message: string): { kind: 'tree'; value: unknown } | { kind: 'text' }` — try/catch JSON.parse; `kind:'tree'` iff `typeof parsed === 'object' && parsed !== null` (objects AND arrays, including empty {} and []); else `kind:'text'`. Bare primitives (numbers, quoted strings, booleans, null) → text. Returns a discriminated result so the panel branches once and passes `value: unknown` to the recursive renderer with safe narrowing.

**2. Tree component:** CUSTOM recursive `JsonMessageTree.tsx` (apps/ui/src/components/logs/, new). NO library. Rationale: no JSON-view lib is installed; codebase builds its own primitives (sheet, badge, etc.); view-only + copy scope; 100% coverage gate needs testid control; React 19 + TS-strict + exactOptionalPropertyTypes compatibility risk on third-party JSON viewers. Renders `unknown` via Array.isArray / typeof narrowing; no `any`.
- REJECTED: react-json-view / react-json-tree library (new dep, bundle weight, theming/Tailwind reconciliation, React-19/TS-strict risk, harder jsdom assertions).

**3. Interaction:** per-node `useState<boolean>` + lucide ChevronRight/ChevronDown toggle with aria-expanded; TOP LEVEL expanded by default, deeper levels collapsed. Stable per-node toggle testids (e.g. json-node-toggle-<path>) for deterministic jsdom tests.
- REJECTED: Radix Collapsible/Accordion (not installed; accordion semantics ill-suited to arbitrary-depth tree). REJECTED: native <details>/<summary> (jsdom doesn't reliably simulate open/close toggling → threatens the coverage gate).

**4. Duplicate-row suppression:** when (and ONLY when) message renders as a tree AND parsed is an OBJECT, build a Set of its top-level keys and filter BAG rows with `!suppress.has(key)`. Never touch coreRows (curated; host/service/severity are restored by Vector so not dupes; a JSON key like timestamp/stream landing in the bag IS a true dupe and gets suppressed). Arrays → empty suppress set (no-op). If message is text → suppress nothing.
- REJECTED: dotted-path/recursive matching (premature — recursive flattening is STAGE-004-017, out of scope).

**5. Copy:** single tree-level Copy button → `JSON.stringify(parsed, null, 2)` (pretty-printed), reusing the existing useCopyToClipboard hook with label "message" (toast reads "message copied"). Leaf/subtree copy DEFERRED (out of scope per card).

**6. Depth + size caps:** max depth 10; per-container child cap 1000 (render first 1000 + "… (M more)"); global node budget 5000; over-limit subtrees collapse to a raw-string leaf (NOT a crash, NOT full text fallback). Top-level JSON.parse in try/catch → parse failure → text fallback. Caps also bound jsdom test runtime.

**7. Mobile:** same tree on mobile and desktop (single code path; inspector is already a scrollable panel / right SheetContent on mobile). Constraint: fixed small per-level indent (padding, not compounding margin) + leaf values WRAP (break-all/break-words) to avoid horizontal overflow on narrow viewports. Collapse-past-depth-0 default mitigates width blow-up.
- REJECTED: collapse-to-raw-text on narrow (behavioral divergence hard to test in jsdom; hides the feature where small screens benefit).

**8. Placement:** special-case the `message` row inside the existing FieldInspectorPanel rows map — when message.kind === 'tree', render a full-width JsonMessageTree in the value column instead of the flat name/value text row. KEEP the "message" label column (w-24) for consistency (DECIDED: label retained, not full-width-edge-to-edge — cosmetic call made in Design). Preserve `data-testid="field-row-message"` on the wrapper so existing tests resolve; add tree-specific testids (json-node-*, json-message-copy). Non-JSON message → today's exact flat-text row (no regression).
- REJECTED: dedicated tree section above/below rows (breaks single-list mental model + canonical order).
