# STAGE-004-006: Redaction pipeline (vector VRL + audit metric + yaml-driven patterns)

## Status: Complete

## Overview

Strip well-known sensitive patterns (bearer tokens, JWTs, passwords-in-URLs, AWS access keys, generic API keys) from log lines BEFORE they reach VictoriaLogs. Redaction happens in vector at ingest, in-process, so no monitor latency is added and no secrets land in VL ever.

Pattern source is `homelab-monitor.yaml` under `logs.redact:`; rendered into the vector.toml VRL transform at boot. Per-pattern-type counts are emitted as `vector_redactions_total{pattern_type}` to VictoriaMetrics; the spec requires an audit trail of counts (never the redacted values themselves).

This lands BEFORE the Logs Explorer + any new UI work so we never expose pre-redaction secrets in any later surface.

**Vertical slice:** after this stage, planting a bearer token / JWT / AWS key in a log line at the source results in the VL record containing `[REDACTED]` instead. A metric `vector_redactions_total{pattern_type="bearer_token"}` increments per redaction. The audit log on the monitor side surfaces the counts but never the values.

**Spec references:** Master spec §3.1 (audit log: counts only, never values).

## Concrete deliverables

### 1. `homelab-monitor.yaml` schema extension

`apps/monitor/homelab_monitor/kernel/config/schema.py` (or wherever the yaml schema lives) — add:

```yaml
logs:
  redact:
    - name: bearer_token
      pattern: '(?i)bearer\s+[A-Za-z0-9._-]+'
      replacement: 'bearer [REDACTED]'
    - name: jwt
      pattern: 'eyJ[A-Za-z0-9_-]+\.[A-Za-z0-9_-]+\.[A-Za-z0-9_-]+'
      replacement: '[REDACTED_JWT]'
    - name: password_in_url
      pattern: '://[^:@/\s]+:[^@/\s]+@'
      replacement: '://[REDACTED]:[REDACTED]@'
    - name: aws_access_key
      pattern: 'AKIA[0-9A-Z]{16}'
      replacement: '[REDACTED_AWS_KEY]'
    - name: api_key_generic
      pattern: '(?i)(api[-_]?key|api[-_]?token|access[-_]?token|secret[-_]?key)["'']?\s*[:=]\s*["'']?[A-Za-z0-9._-]{16,}'
      replacement: '$1=[REDACTED]'
```

Loader validates each entry has `name`, `pattern`, `replacement`. Invalid entries fail at config load (don't ship a partially-broken redaction policy).

### 2. Render-on-boot expansion into vector.toml

`apps/monitor/homelab_monitor/kernel/cron/render.py` (or wherever the render-on-boot logic lives) — read `logs.redact:` from `homelab-monitor.yaml`, generate a `[transforms.redact]` VRL block that runs BEFORE the throttle / sink:

```toml
[transforms.redact]
type = "remap"
inputs = ["docker_enrich", "journald_enriched", "hmrun_parsed"]  # after enrichment, before throttle
source = '''
# Per pattern from yaml:
.message, redacted_bearer = replace(.message, r'(?i)bearer\s+[A-Za-z0-9._-]+', "bearer [REDACTED]")
if redacted_bearer > 0 {
  emit_metric("vector_redactions_total", {"pattern_type": "bearer_token"}, redacted_bearer)
}
# ... repeated per pattern, generated from yaml
'''
```

(Final VRL syntax confirmed in Design phase. Vector's `emit_metric` may have a different name; substitute the actual function.)

### 3. Metric: `vector_redactions_total{pattern_type}`

Vector's prometheus_exporter sink (or equivalent metric pathway) exposes the counter; vmagent scrapes it. The metric is queryable in MetricsQL.

## Design Decision (locked 2026-05-29)

### Selected approach: Option B — per-path redact remaps + per-pattern boolean markers + multi-entry log_to_metric

**CRITICAL API CORRECTION (stage card was wrong — verified against Vector 0.41 docs):**
- VRL has NO `emit_metric()` function; a `remap` transform cannot emit a metric. The metric path is a **`log_to_metric` transform → `prometheus_exporter` sink → vmagent scrape**.
- VRL `replace()` returns ONLY the new string (no count). Detect a hit with `match()` (INFALLIBLE — never `match!`) BEFORE replacing. Counter semantics = LINES-with-pattern (the only thing the API supports), not occurrences.
- `replace()` DOES support capture-group backrefs `$1`/`${1}` (Rust regex); use the brace form `${1}` where the backref is adjacent to literal replacement text (matters for `password_in_url` and `api_key_generic`).

**Config (D-REDACT-YAML-CONFIG):** add a `logs.redact:` section to `apps/monitor/homelab_monitor/kernel/config.py` as PLAIN dataclasses (mirror `DiskBudgetConfig`/`LogStreamBudgetConfig`, NOT Pydantic): `RedactPattern{name, pattern, replacement}` + a loader returning a list, with manual validation that raises `ValueError` on a missing field, duplicate name, or a pattern containing a regex lookaround. Default v1 set = 5 patterns (bearer_token, jwt, password_in_url, aws_access_key, api_key_generic). Absent section → empty list (no redaction).

**Render (D-REDACT-VECTOR-LAYER):** extend `apps/monitor/homelab_monitor/kernel/cron/render.py::render_config()` (+ `VectorRenderContext`, + `render_on_boot` calling the redact loader) to generate the redact VRL body from the loaded patterns and substitute it into the template. The generated per-pattern VRL idiom (API-correct): `if match(.message, r'<pattern>') { .message = replace(.message, r'<pattern>', "<replacement>"); .rdt_<name> = true }`.

**DAG topology (Option B — preserves existing throttle/hmrun behavior):**
- `[transforms.redact_main]` inserted between `drop_noise` and `throttle` → `throttle.inputs = ["redact_main"]`.
- `[transforms.redact_hmrun]` inserted between `hmrun_shaped` and the VL sink → `sinks.vl.inputs = ["throttle", "redact_hmrun"]`.
- Both remaps run the SAME generated VRL body (drift-guard test asserts byte-identical, mirroring the existing multiline drift-guard precedent).
- Every line reaching `sinks.vl` (docker, journald, hmrun) passes through a redact transform — NO un-redacted bypass. (cron_events branch carries no free-text message → out of scope.)
- Marker fields `.rdt_<name>` are `del()`'d before the VL-bound path so they never ship to VL.

**Metric path (D-REDACT-AUDIT-COUNTS-ONLY, structurally enforced):** a `[transforms.redaction_metric]` `log_to_metric` taps a copy of both redact outputs; one counter entry per pattern: `type=counter`, `name=vector_redactions_total`, `field=rdt_<name>` (null-skips when absent → non-redacted lines emit nothing), `tags.pattern_type="<name>"` as a STATIC LITERAL (never matched text — counts-only is structurally impossible to violate). → `[sinks.redaction_metrics]` `prometheus_exporter` on a dedicated port; add a scrape job to `deploy/compose/vmagent/scrape.yaml` and expose the port on the `vector` compose service. (Check first whether an existing internal_metrics/prometheus sink + scrape job can be reused.)

**Audit task:** a scheduler-registered `BaseCollector` (mirror `CronRunReconciler`/`HeartbeatStateCollector` registration in `lifespan.py` with `interval_seconds: 300` = 5 min). Queries VM instant `/api/v1/query?query=vector_redactions_total` (reuse the httpx-to-`get_vm_url()` idiom from `routers/metrics.py`), diffs cumulative-per-`pattern_type` against last-seen, writes ONE `audit_log` row via `audit_write(who="system", what="logs.redaction_counts", after={pattern_type: {delta, cumulative}})`. NEVER writes matched values; NEVER logs to VL (avoids recursion).

**Test corpus (CRITICAL acceptance — no un-redacted sensitive line leaves the pipeline):**
- Per-pattern positives (1 each): bearer_token, jwt, password_in_url (assert URL prefix kept via `${1}`, password gone), aws_access_key, api_key_generic (assert key-name token preserved via `$1` if replacement keeps it).
- Edge cases: pattern at line START; pattern mid-line; multiple SAME-pattern matches on one line (all redacted, counter +1 = lines-with-pattern); multiple DIFFERENT patterns on one line (all redacted, each counter +1 independently); no-match line (unchanged, NO metric emitted); near-miss line containing the word "bearer"/"password" but not the pattern (NOT over-redacted).
- Planted-credential corpus injected on all three source paths (docker, journald, hmrun); assert no line containing any of the 5 patterns reaches `sinks.vl` un-redacted; assert `rdt_*` marker fields are absent from VL records; assert metric label set is exactly `{pattern_type=<one of 5 literals>}`.
- Authoritative VRL compile gate: plain `vector validate` (the slow pytest test) + docker `vector validate` (timberio/vector:0.41.1-debian) on the rendered template at Build. Extend `_assert_no_lookarounds` to the rendered redact block.

**Rejected:** Option A (single unified remap + marker-array + explode) — changes hmrun-bypasses-throttle topology and adds array/explode VRL risk. Option C (single remap + `route` per pattern) — adds route + 5 taps for no v1 capability beyond B's null-field-skip.

## Refinement Notes (2026-05-29)

**Refinement complete:** 2026-05-29. Validated against the REAL prod compose stack (redaction deployed: monitor rebuilt + restarted → vector.toml re-rendered with redact transforms → vector restarted; vmagent scrapes vector:9598). Planted-credential corpus injected via docker stdout (docker_logs tails all containers) + journald. On this host the dev/prod rigs share one VL/vector stack, so 3a (real ingest from source) and 3b (real prod stack) collapse onto a single real-stack validation covering both.

**Two bugs found against the real rig (Build gates incl. docker `vector validate` passed — these are RUNTIME-behavior bugs that compilation can't catch):**

**Bug A — `log_to_metric` dropped ALL events (metric never reached VictoriaMetrics).** The Design assumption that `log_to_metric` SKIPS emission when its target `field` is absent is FALSE for Vector 0.41.1: it ERRORS (`Field does not exist`) and DROPS the event per-entry for every line lacking that field. Since `redact_main`/`redact_hmrun` set `.rdt_<name>=true` only on matched lines, almost all lines lacked the fields → continuous ERROR storm + empty prometheus_exporter body + `vector_redactions_total` absent. (`.message` redaction itself was unaffected — only the metric tap.) FIX (Opus-debugger runtime-proven against the real binary): redact VRL now ALWAYS sets integer markers `.rdt_<name> = 0` (default) and `= 1` on match — so `field=rdt_<name>` always exists — and `log_to_metric` uses `increment_by_value = true` (adds 0 for non-match, 1 per match → total == lines-with-pattern). Booleans don't work (log_to_metric float-parses the field → ParseFloatError); integers do. A pre-filter alone is insufficient (per-entry field-absence still errors).

**Bug B — bearer_token over-redaction.** `(?i)bearer\s+[A-Za-z0-9._-]+` matched `Bearer password` (any 1+ char token value), over-redacting benign text. FIX (security-first reasoning — over-redaction is cosmetic, UNDER-redaction leaks a secret, so the pattern must err toward over-redaction but a defensible floor exists): raise the token-value floor to `{20,}` → `(?i)bearer\s+[A-Za-z0-9._-]{20,}`. Real bearer tokens are ≥20 chars (OAuth opaque ≥22, JWTs 100+ and double-caught by the jwt pattern, PATs 40); English words top out ~14 chars. Excludes word false-positives without risking real-token misses. Other 4 patterns assessed low-risk (jwt/aws structurally distinctive; api_key_generic has a keyword anchor + 16-char floor).
