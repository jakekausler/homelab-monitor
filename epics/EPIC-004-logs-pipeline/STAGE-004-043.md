# STAGE-004-043: Create-alert-from-query UX (guided form, YAML preview, persistence via STAGE-004-042)

## Status: Refinement

## Overview

UI surface that lets the user author a vmalert rule from the current Logs Explorer query. Opens a guided form with sensible defaults (name, severity, `for` duration, threshold expression scaffolding), renders the resulting YAML in a preview pane, requires explicit confirmation before persisting via STAGE-004-042's API.

**Vertical slice:** in the Explorer, click "Create alert from this query" → modal/drawer opens with form fields (rule name, severity, `for`, threshold, summary, description, extra labels). The form's right pane shows the rendered YAML in real-time. Click "Save" → rule persists + renders + vmalert picks it up. Click "Cancel" → no change.

**Spec references:** master spec §9.2 ("create alert from this query"); EPIC-004 plan #7.

### 3. Form fields

- **Rule name**: free text, validated `^[a-zA-Z_][a-zA-Z0-9_]*$` client-side, name-uniqueness checked on save
- **Severity**: dropdown (info / warning / critical)
- **For duration**: text input (e.g., `0m`, `5m`, `1h`), validated against duration regex
- **Threshold expression**: pre-filled with `count(<current LogsQL>) > 10` — user can edit; full LogsQL/MetricsQL allowed
- **Summary template**: free-text; supports vmalert template syntax (`{{ $labels.foo }}`, `{{ $value }}`)
- **Description template**: same
- **Extra labels**: key/value rows (add/remove)
- **Extra annotations**: same

### 4. YAML preview (right pane)

Renders the full rule YAML in real-time as form changes (debounced ~300ms). Read-only; uses CodeMirror (or `<pre>` with syntax highlighting).

### 5. Validation feedback

- Name uniqueness checked via debounced API call to `GET /api/logs/user-rules?name=<...>`
- Expression syntax validated client-side via a minimal LogsQL/MetricsQL grammar check (best-effort; vmalert's full parser is the source of truth)
- Inline error messages near each field

### 6. Save action

- POST to `/api/logs/user-rules` (from STAGE-004-042)
- On 201: close modal + toast "Rule '<name>' created"
- On 409 (name collision): inline error
- On 4xx other: inline error
- On 5xx: toast error + don't close modal so user can retry

- `__tests__/CreateAlertModal.test.tsx` — form interactions, YAML preview updates, validation messages, save flow, error handling

## Design Phase

**Design phase complete: 2026-06-08**

### Design Notes

**Decision Summary:** STAGE-004-043 implements a reusable `<CreateAlertModal>` launched from the Logs Explorer with a two-pane form (react-hook-form + zod left, real-time client-side YAML preview right). The form maps 1:1 to the actual POST `/api/logs/user-rules` API contract discovered during backend verification. Frontend-only (Refinement 3a only). Designed for STAGE-044 reuse via `initialValues` + `sourceKind`/`sourceRef` props contract.

**Key Backend-Contract Verification Finding:** The initial stage card assumed a `GET /api/logs/user-rules?name=<...>` uniqueness endpoint — **this endpoint does NOT exist**. The actual API contract for POST is:

```typescript
LogUserRuleCreateRequest = {
  rule_name: string        // 1-200 chars, regex ^[a-zA-Z_][a-zA-Z0-9_]*$
  expr: string             // 1-8192 chars, LogsQL or MetricsQL
  expr_kind: "logsql" | "metricsql"
  severity: "info" | "warning" | "critical"
  summary: string          // 1-1000 chars
  description?: string     // ≤4000 chars, default ""
  for_duration: string     // ≤16 chars, default "0s"
}
```

Response: 201 on success; 409 `{error:{code:"conflict",message:"rule_name already exists: <name>"}}` on duplicate; 400 `invalid_rule` on validation failure. CSRF auto-injected by apiClient. The API has **no labels/annotations field** — severity/source_tool/category/summary/description are hard-coded by the backend renderer (STAGE-004-042).

| ID | Decision | Choice | Justification |
|----|----------|--------|---------------|
| D1 | "Extra labels/annotations key/value rows" (criterion 3) vs API has no labels field | **DESCOPE the key/value rows.** Form fields = rule_name, severity, for_duration, expr, summary, description only. Criterion-3's "extra labels/annotations" sub-item is DEFERRED. | The 042 API/renderer hard-codes labels (severity, source_tool: user, category: user-rule) + annotations (summary/description) and has NO field to store arbitrary labels. Adding them = backend change to a Complete stage = sequential-dependency violation. Storing custom labels should be a separate backend stage if real need arises. Also avoids the useFieldArray NEW-pattern risk. |
| D2 | Threshold expression pre-fill | Scaffold `<currentQuery> \| stats count() as count \| filter count:>10`, expr_kind=logsql (prop, default logsql for 044's metricsql reuse). Reuse the CodeMirror LogsQL editor for the expr field. Threshold + window editable. Guard against double-wrap when query already ends in `\| stats` pipe. | The card's literal `count(<LogsQL>) > 10` is verified-WRONG — 042's rig Refinement proved vmalert vlogs ALERTING exprs require a `\| stats ... \| filter ...` pipe (bare filters/metricsql-shaped exprs → health=err). The scaffold produces a loadable logsql rule matching the built-in logs-rule shape. |
| D3 | Name-uniqueness validation (no `?name=` endpoint exists) | Client-side uniqueness check against the cached GET-list (react-query, debounced), with the POST 409 as the authoritative backstop (inline error on 409); consistent messaging between the two paths. | The assumed `?name=` endpoint does NOT exist. List is tiny (single-user homelab) so fetching all is cheap; gives instant inline feedback; 409 covers the stale-cache race. No backend change. |
| D4 | Modal reusability shape for STAGE-044 | Single `<CreateAlertModal open onOpenChange initialValues?={Partial<FormValues>} sourceKind={"query"\|"saved_query"\|"signature"} sourceRef?={string} />`. Launch site computes the scaffolded expr + passes it via initialValues. 043 sends sourceKind="query". | Makes 044 a trivial "open with different initialValues + sourceKind". source_kind/source_ref map to 042's free-string source-tracking fields. Headless-hook refactor deferred unless 044 reveals real chrome divergence. |
| D5 | YAML preview rendering source | Client-side TS preview that MIRRORS 042's exact render_yaml template (group user-rules-logs, `type: vlogs` for logsql, hard-coded labels severity/source_tool:user/category:user-rule, summary/description as block scalars). Read-only CodeMirror (editable.of(false) + @codemirror/lang-yaml). Debounced ~300ms. Labeled "Preview". Centralized in ONE module + fidelity test pinned to 042's known output. | No round-trip; satisfies the real-time preview criterion. Drift risk mitigated by mirroring the exact template + a test that breaks loudly if 042's renderer changes. Server-side dry-run render endpoint deferred to a future hardening stage. |

**Cross-cutting Guards (carry into Build):**

- **Criterion-3 descope is FIRM:** Build must NOT implement dynamic key/value label rows. Form fields are exactly: rule_name (regex `^[a-zA-Z_][a-zA-Z0-9_]*$`), severity (dropdown info|warning|critical), for_duration (validated `^\d+[smhd]$` or 0s), expr (LogsQL CodeMirror, scaffolded), summary, description.
- `@codemirror/lang-yaml` likely NOT yet a dep in apps/ui — verify in Build, add if absent.
- The client-side YAML preview module must faithfully mirror 042's `kernel/logs/user_rules_render.py` render_yaml output (group name, type: vlogs for logsql, the three labels, block scalars). Add a fidelity/snapshot test. Build MUST READ user_rules_render.py to mirror it exactly.
- Two-pane = `grid grid-cols-1 gap-4 md:grid-cols-2` (stacks on mobile) — confirm modal scroll + preview max-height during Mobile refinement (3a).
- Patterns to mirror: SaveQueryModal (sibling launch + 409-inline), SilenceAllowlistTab (rhf+zod), api/silenceAllowlist.ts (react-query hook: query-key namespace, useQuery list, useMutation POST + invalidate), Radix Dialog, sonner toasts.
- **NEW pattern this stage:** read-only CodeMirror YAML pane (no existing wrapper). useFieldArray is AVOIDED (D1 descope).

**Design-Phase Confirmations (from the stage card's "Design will confirm" list):**

- Default threshold expr: scaffolded `| stats count() as count | filter count:>10` (NOT `count(<LogsQL>)>10` — corrected per D2).
- Default severity: warning. Default for_duration: 5m. (Both editable.)
- YAML preview editor: CodeMirror 6 read-only (+ @codemirror/lang-yaml).
- Modal vs drawer vs full-page: modal (Radix Dialog), two-pane.

## Refinement Notes

**Refinement type:** Frontend (3a). User manually tested on the prod rig (UI built into the monitor image). Desktop + Mobile both approved by the user.

**MAJOR mid-Refinement scope expansion (user-directed):** What started as the create-alert modal grew substantially because rig testing exposed real defects:
- The original auto-scaffold/raw-expr-only UX was not user-friendly and produced INVALID LogsQL. User directed adding a Simple mode (guided form → safe LogsQL) + an Advanced mode (raw expr + LogsQL doc link). Implemented.
- **BUG 1 (fixed):** the scaffold/builder used the reserved LogsQL keyword `count` as the stats alias (`| stats count() as count | filter count:>N`) — vmalert rejects `| filter count:` because `count` is reserved. Renamed to `match_count`. Affected every generated rule (incl. future signature/model queries). Rig-verified: rules now load health=ok.
- **BUG 2 (fixed):** one invalid user rule disabled ALL user rules (vmalert rejects the entire `-rule` glob if any file fails to parse). Per-rule-file rendering (one `<rule_name>.yaml` per rule) was added but does NOT fully isolate vmalert's glob-reload. The real fix: **validate-before-persist** — a heuristic backend expr validator (`expr_validate.py`) rejects vmalert-unloadable exprs at POST/PATCH (400 `invalid_expr`), so an invalid rule never reaches the glob. Checks: logsql requires `| stats` pipe; `| filter <reserved-word>` rejected; unbalanced quotes/parens rejected. Plus client-side advisory warnings + inline expr-error UX.
- **Backend escaping bug (fixed):** STAGE-042's expr-field validator wrongly forbade `"`/`\` (used `_FORBIDDEN_INLINE`); expr is a YAML block scalar where they're safe + REQUIRED for LogsQL. Changed to `allow_newlines=True` (block pattern). One-line fix in user_rules_repo.py + test.

**Deferred to STAGE-004-043A (created this session):** (a) user-rules MANAGEMENT UI (list/enable/disable/edit/delete — the API has full CRUD but no UI surface; created rules are currently write-only from the UI); (b) the prod-only vmalert `-dryRun` exact-parser validation (exact-parser fidelity beyond the heuristic; belongs with 043A's rule-health surfacing).

**Rig validation:** Final e2e — user created Test3 (Advanced, the real Home Assistant model-signature query that originally broke) and Test4 (Simple, special chars in contains: `error""spec*alC|haracters\n`). BOTH persist, render to per-rule files, and load in vmalert with health=ok. Invalid exprs (missing stats / reserved word / unbalanced quotes) are rejected with 400 `invalid_expr` and never persisted. `make verify` GREEN (backend 100% coverage, frontend tsc/eslint/prettier/vitest/build).

**Out-of-band fix (separate concern, committed separately):** Models tab (`ModelsDebugPage.tsx`) detail pane lacked `min-w-0` → wide content clipped instead of scrolling; fixed. Also pinned `react-hooks/incompatible-library` eslint rule to `warn` (its severity floated across the plugin's ^7.x range, intermittently blocking commits).
