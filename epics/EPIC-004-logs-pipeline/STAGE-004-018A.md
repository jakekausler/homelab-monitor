# STAGE-004-018A: LogsQL editor autocomplete + field suggestions

## Status: Complete

## Overview

Add autocomplete/suggestions to the advanced-mode LogsQL editor (CodeMirror 6, from STAGE-004-011). As the user types, the editor offers completions for: LogsQL keywords/pipe-functions (AND/OR/NOT/stats/count/by/sort/head/limit/fields/filter/as), known field names, and — sourced from the scope-aware field-discovery endpoint (STAGE-004-018) — actual field names present in the current query scope plus their sample values. This makes the advanced LogsQL editor usable without memorizing the schema.

**Vertical slice:** in advanced mode, typing `serv` offers `service:` from the discovered fields; after `service:` the editor suggests sample values seen in scope; typing `| ` offers pipe functions. Completion accepts on Enter/Tab without submitting the query.

**Depends on:** STAGE-004-011 (the CodeMirror editor + StreamLanguage) and STAGE-004-018 (`GET /api/logs/fields` scope-aware field list + sample values). This is why it is inserted AFTER 018.

**Spec references:** master spec §9.2 (LogsQL explorer). Note: STAGE-004-011 explicitly deferred autocomplete ("YAGNI for v1, defer to a future stage if anyone asks") — this stage is that follow-up, added at user request.

## Concrete deliverables

### 1. CodeMirror autocomplete extension

Add `@codemirror/autocomplete` to the lazy editor (`LogsQlEditorImpl.tsx` / `logsQlLanguage.ts`). A completion source offering: (a) static LogsQL keywords/pipe-functions, (b) field names, (c) context-aware values after a `field:` token.

### 2. Field/value source from STAGE-004-018

Wire the completion source to the field-discovery data (`GET /api/logs/fields?expr&start&end&sample`) so suggestions reflect fields actually present in the current scope, with sample values. Debounce/caching reuses 018's 30s cache. Degrade gracefully if the endpoint is unavailable (fall back to static keyword/field completions).

### 3. Completion UX

Accept on Enter/Tab WITHOUT submitting the query (must not conflict with STAGE-004-011's Enter-to-submit — autocomplete popup open → Enter accepts completion; popup closed → Enter submits). Escape closes the popup. Mobile (textarea fallback) has no autocomplete (acceptable; document).

### 4. Tests

- Completion source unit tests (keyword completions, field completions from a mocked field list, value completions after `field:`).
- Editor integration: popup opens on type, accept-on-Tab inserts, Enter-with-popup-open does not submit.

## Out of scope for this stage

- Full LogsQL semantic validation / error squigglies — separate concern, still deferred.
- A visual drag-and-drop query builder (a separate possible future stage; this delivers in-editor autocomplete only).
- Autocomplete on the plain-text (non-advanced) search box.

### Design Notes (STAGE-004-018A — recorded at Design completion)

**Architecturally non-trivial — brainstormer (Opus) consulted on 3 decisions; main agent made the final calls.**

**D1 — Enter/Tab vs submit keymap ordering (CHOSEN: separate Prec.highest accept-keymap).**
Add a NEW keymap at `Prec.highest` in `LogsQlEditorImpl.tsx` binding **Enter and Tab** to `acceptCompletion` (from `@codemirror/autocomplete`). `acceptCompletion` returns `false` when no completion popup is active → CM6 falls through to the existing untouched `Prec.high(submitKeymap)` (lines 49–64/88), which submits. Popup OPEN → Enter/Tab accept; popup CLOSED → Enter submits. Escape + arrows left to the library's default completion keymap (`autocompletion()`). Tab-to-accept is safe (Tab is currently inert in this single-line editor). Keep submitKeymap at `Prec.high`; do NOT demote it.
- Rejected (Option A) mutate the single Enter `run` to call acceptCompletion first then fall through to submit — acceptable but entangles accept logic into the submit handler; chosen B keeps both handlers individually pure and the feature removable without touching submit.
- Rejected (Option C) lift the whole library `completionKeymap` at raised precedence — lifts more bindings than needed.

**D2 — Completion-source architecture + data flow (CHOSEN: lift data in Body, pass prop, feed source via useEffect-updated ref).**
`LogsExplorerBody.tsx` already resolves the COMMITTED `expr` (~line 211), `startIso`/`endIso` (~226–240), `servicesCsv` (~246) and calls `useLogsFieldsQuery` for `<FieldsDiscoveryPanel>` (lines 1132–1138; React Query dedupes the shared query by identical queryKey). Lift/share that field data, map it to a completion-ready structure, and pass it as a NEW OPTIONAL prop (e.g. `completionData`) through `LogsQlEditor` (currently `{value,onChange,onSubmit,placeholder?,ariaLabel,className?}`, no scope props) → `LogsQlEditorImpl`. In the impl, hold the latest data in a `useRef`, update it via a `useEffect`; the `CompletionSource` closure (created ONCE in the extensions array) reads `ref.current` at call time, so the `EditorView` is NEVER re-created when field data changes. Editor stays presentational.
- Use the COMMITTED expr (the same scope the Fields panel uses), NOT the live per-keystroke `liveLogsQl` — avoids queryKey thrash.
- Rejected (Option B/C) fetch inside `LogsQlEditorImpl` — forces per-keystroke queryKey thrash on live expr, or requires plumbing a confusing second expr-like prop (live `value` vs committed scope) and makes the editor non-presentational.
- Export the currently-PRIVATE `KEYWORDS` and `FIELD_NAMES` consts from `logsQlLanguage.ts` so the completion source and the StreamLanguage tokenizer share one source of truth (do not duplicate the lists).

**D3 — Testability (CHOSEN: pure completion-source function + thin wiring smoke).**
Extract a PURE `(context, fields) => CompletionResult | null` (likely new `logsQlCompletion.ts` beside `logsQlLanguage.ts`). Unit-test with a hand-rolled mock `CompletionContext` surface `{ pos, explicit, matchBefore }` across the three cursor contexts: (a) keyword position, (b) field-name position, (c) sample-values after a `field:` token (from a mocked field list). If the `field:` backward-scan needs more than one `matchBefore`, promote ONLY that case to a DOM-free real `EditorState` (CM6 `EditorState` works without a DOM; only the *view* needs jsdom). One thin assertion that the editor wires the source/`autocompletion()` in. NO jsdom popup-driving tests (CM6 view is jsdom-hostile; existing `LogsQlEditor.test.tsx` already mocks `@/lib/useMediaQuery` to force the textarea path). Keymap accept-vs-submit interaction (D1) validated in manual Refinement.

**Graceful degradation:** when field-discovery data is absent/loading/errored, the ref holds `[]`/stale and the source returns static keywords + base field names only (success criterion 2).

**New dependency:** `@codemirror/autocomplete@^6` (siblings: state ^6.6.0, view ^6.43.0, commands ^6.10.3, language ^6.12.3).

**Verified anchors:** all confirmed this session; two corrections vs handoff — (1) `KEYWORDS`/`FIELD_NAMES` are PRIVATE consts (must export), (2) `LogsQlEditor` receives NO scope props today (new optional prop required). `logsQlExtensions()` is tokenizer+highlight only (no completion injection point — autocomplete is a separate extension).

### Code Review Disposition (Opus, pre-commit)

**APPROVED.** Both load-bearing mechanisms confirmed correct:

1. **CompletionSource ref-closure design:** The `CompletionSource` closure is created ONCE in the extensions array with `fieldsRef` captured. At call time, the closure reads `fieldsRef.current` lazily. The editor mount effect has dependency array `[]` (no restabilization on field data change), so the `EditorView` is created once and never recreated. No stale-capture risk, confirmed correct.

2. **Enter/Tab keymap precedence:** A `Prec.highest` accept keymap (Enter+Tab → `acceptCompletion`) layers above the untouched `Prec.high` submit keymap. When the completion popup is OPEN, `acceptCompletion` returns `true` and the Enter/Tab is consumed (no fallthrough). When the popup is CLOSED, `acceptCompletion` returns `false` and CM6 falls through to the `Prec.high` submit keymap, which triggers the query submit. Popup-open accept vs popup-closed submit behavior is correct.

**No Critical or Important findings.** Completion tests exercise all 3 contexts (keywords, field names, sample values after `field:`) plus graceful degradation when field data is unavailable.

### Reasoned No-Action (1 Minor, reviewer-classified optional)

**Minor:** The blanket 350ms timer-draining `afterEach` hook added to `CreateAlertModal.test.tsx` (lines 2–7) runs for all 41 tests in that file (~14s added runtime), even tests that never mount the modal. The reviewer recommended documenting no-action: (1) the fix is sound and does NOT mask assertion failures (assertions execute before the `afterEach` cleanup), (2) a per-test-gated version would couple the environmental hygiene fix to per-test state and degrade simplicity and robustness. Accepted as-is per the optional-finding exception (reviewer-classified optional; fix would degrade consistency).
