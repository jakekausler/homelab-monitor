# STAGE-004-008: Custom datetime range picker across all 3 log-viewer surfaces

## Status: Complete

## Overview

Today's log viewers offer 6 preset since-windows (5m / 15m / 1h / 6h / 24h / 7d) but no custom range. This stage adds a "Custom range" option to the time-range control wherever it appears: docker per-container viewer, cron per-run viewer, and the Logs Explorer (once it lands in STAGE-004-010).

Custom range = two datetime pickers (start, end) with validation:
- `start < end`
- Neither in the future
- Total range ≤ 30 days (VL retention boundary)
- Pickers default to "current visible range" when opened

Fixes the deferred STAGE-003-011 design decision D-DEFER-CUSTOM-RANGE.

**Vertical slice:** clicking "Custom range" in the time-range control opens a popover with start + end datetime inputs. Clicking Apply re-queries the current view with the custom range. The choice persists in the URL query string (so deep-links work).

**Spec references:** master spec §9.2; EPIC-004 STAGE-004-005 acceptance criteria.

## Concrete deliverables

`apps/ui/src/components/logs/TimeRangeControl.tsx` (new) — composable control with:

### 2. Validation rules

In a shared utility (`apps/ui/src/lib/timeRange.ts`):

```typescript
function validateRange(start: Date, end: Date): { ok: boolean; error?: string } {
  if (!(start instanceof Date) || isNaN(start.getTime())) return { ok: false, error: 'Invalid start' }
  if (!(end instanceof Date) || isNaN(end.getTime())) return { ok: false, error: 'Invalid end' }
  if (start >= end) return { ok: false, error: 'Start must be before end' }
  const now = Date.now()
  if (start.getTime() > now || end.getTime() > now) return { ok: false, error: 'Cannot select future' }
  const MAX_RANGE_MS = 30 * 24 * 60 * 60 * 1000
  if (end.getTime() - start.getTime() > MAX_RANGE_MS) return { ok: false, error: 'Range exceeds 30 days (VL retention)' }
  return { ok: true }
}
```

100% test coverage on this utility.

### 3. URL state encoding

Each viewer route accepts `?start=<ISO>&end=<ISO>` (overrides `since=<preset>` when present). The Explorer also persists this in URL state so deep-links to a specific log window work.

### 4. Backend endpoints accept `start` + `end` params

All 3 endpoints already accept time bounds in some form (per the recon: `since: str` for docker + cron, `start`/`end` for the generic logs query). Normalize so all 3 accept BOTH:
- `since=<preset>` (compatibility with existing callers)
- `start=<ISO>&end=<ISO>` (new; preferred when present)

Server-side: validate the same rules (range ≤ 30d, neither in future). Return 422 on violation.

## Design Phase (locked 2026-05-30)

**Design Decision (locked 2026-05-30):**

**D008-1 (Docker endpoint — extend with start/end):** Extend `GET /api/integrations/docker/containers/{name}/logs` with optional `start`/`end` ISO-8601 params, mutually exclusive with `since` (if both supplied → 422). Reuse the exact validation block from `GET /api/logs/query` (ISO parse, UTC-normalize, start<end, ≤30d span, no-future → 422 on violation). This makes all 3 log endpoints feed `paginate_older(window_start, window_end)` identically. `/api/logs/query` already accepts/validates start/end (no change). Cron run-log endpoint keeps its fixed DB window (no time params added server-side). OpenAPI regen required after the docker change.

**D008-2 (Picker primitive — native datetime-local, no new dep):** Two native `<input type="datetime-local">` (start, end) inside the control's popover. Use `min`/`max` attributes as first-line UX guards (no-future, 30d span). Authoritative re-validation on Apply uses zod (react-hook-form + zod are already installed in apps/ui — no new dependency). NO calendar library added.

**D008-3 (Shared controlled `<TimeRangeControl>`):** ONE shared, CONTROLLED component `<TimeRangeControl value onChange>` emitting a `TimeRangeValue` discriminated union: `{ kind: "preset"; token: "5m"|"15m"|"1h"|"6h"|"24h"|"7d" }` OR `{ kind: "custom"; start: Date; end: Date }` (absolute instants). URL query string is the single source of truth; each viewer maps the emitted value onto its own hook + URL params (custom → `?start=<ISO>&end=<ISO>`; preset → `?since=<token>`, per D-URL-STATE-PERSISTS). Props include `mode?: "full"|"bounded"`, `min?`/`max?` Dates (for cron clamp), and `presets?` list. Emitting absolute Date instants keeps STAGE-004-009's display-only timezone toggle from fighting the value — they layer cleanly. Explorer (STAGE-004-010) reuses this control as-is.

**D008-4 (Cron run-log viewer — bounded mode, client-side narrow):** The cron per-run viewer gets the SAME `<TimeRangeControl>` in `mode="bounded"`, clamped to `min=run.start`, `max=run.end`. It applies a CLIENT-SIDE filter on already-fetched run-log lines (zoom into a long run). NO new cron backend param. This honors the locked D-CUSTOM-RANGE-EVERYWHERE decision (user explicitly confirmed: "everywhere", clamped solution). NOT scoped out.

**D008-5 (Responsive container):** Popover/dropdown on desktop; full-screen `dialog` (existing radix primitive) below the `sm` breakpoint, giving the two datetime inputs + validation message + Apply/Cancel full-width touch room on mobile.

**Validation rules** (locked, both client zod AND server): start < end; neither start nor end in the future; total span ≤ 30 days.

**Shared validation utility location:** `apps/ui/src/lib/timeRange.ts` with `validateRange(start, end)` (100% test coverage target).

**Out of scope** (explicitly): relative-time keywords ("yesterday 14:00"); saved custom ranges (STAGE-004-013); the local/UTC display toggle (STAGE-004-009 — picker stays browser-local).

**Refinement:** 3a only (frontend-only, no host-integration). Frontend half needs USER manual Desktop+Mobile approval; backend param-validation can be e2e'd autonomously.
