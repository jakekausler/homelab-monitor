# STAGE-004-007: Cursor pagination across all 3 endpoints + `<LogViewer>`

## Status: Complete

## Overview

Today's log viewers fetch a single batch of lines and render them. Beyond the batch size cap (500 or so), older lines are silently dropped with only a `truncated: true` banner. The user can't go back further than that window.

This stage adds cursor-based pagination ‚Äî "load older lines" ‚Äî to:

- `GET /api/integrations/docker/containers/{name}/logs`
- `GET /api/crons/{fp}/runs/{run_id}/log`
- `GET /api/logs/query`

‚Ä¶and to the `<LogViewer>` component (STAGE-004-003), which gains an "Older lines" button at the top of the line list (or auto-loads on scroll, Design decides).

This fixes the deferred STAGE-003-011 design decision D-DEFER-PAGINATION.

**Cursor scheme:** `(timestamp_ns, line_seq)` where `line_seq` is a deterministic position counter within the VL result set. Forward + backward pagination both supported. Cursor is opaque to the client (base64-encoded JSON or similar).

**Vertical slice:** in any log viewer, the user can click "Load older" and see lines older than the current window. The cursor survives navigation away + back (when state persistence lands in STAGE-004-015).

**Spec references:** master spec ¬ß9.2; EPIC-004 STAGE-004-005 acceptance criteria (the original plan grouped pagination + custom range under one stage; we've split them).

## Concrete deliverables

### 1. Cursor type definition

`apps/monitor/homelab_monitor/kernel/api/models/logs.py` ‚Äî add:

```python
class LogCursor(BaseModel):
    """Opaque cursor for log pagination. Base64-encoded JSON over the wire."""
    timestamp_ns: int  # nanosecond precision (VL native)
    line_seq: int  # tie-breaker for same-timestamp lines

class LogsQueryResponse(BaseModel):
    lines: list[LogLine]
    next_cursor: str | None  # opaque, encode/decode helpers in line_mapping.py
    has_more: bool
    # existing fields...
```

`encode_cursor(cursor: LogCursor) -> str` and `decode_cursor(s: str) -> LogCursor` helpers in `line_mapping.py`. Base64-encoded JSON works fine (it's not security-sensitive, just opaque to UI).

### 2. Endpoint params + behavior

All three endpoints accept `cursor: str | None = None` query param:

- If `cursor` is `None`: return most recent N lines + a cursor to the "older" boundary
- If `cursor` is present: return N older lines starting from the cursor + a new cursor (or `None` if exhausted)
- `has_more: bool` tells the UI whether the "Load older" button should remain enabled

VL query: use VL's native cursor or `_time_ns` ordering with strict less-than against the cursor's timestamp_ns. Tie-breaking via `line_seq` (sequence within the VL response or another deterministic ordering ‚Äî Design decides).

### 3. Page size

Each endpoint accepts `limit: int = 500` (or whatever the current default is). Max enforced server-side. Frontend default: 500.

### 4. Backward + forward pagination semantics

For now: backward (older) only. Forward (newer) auto-handled by re-fetching the latest window. If a use case for "show me lines newer than cursor X" emerges, add `direction: "older" | "newer"` param.

### 5. `<LogViewer>` component update

`apps/ui/src/components/logs/LogViewer.tsx` ‚Äî add:

- "Load older lines" button at top of line list (or auto-load on scroll-to-top ‚Äî Design picks)
- Append older lines to the existing list (don't replace)
- Disable button when `has_more === false`
- Loading spinner during pagination fetch
- Error state on failed pagination (retry preserves current state)

Caller-provided `useLogs` hook gets a new optional `useOlder` method:

useLogs: () => {

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-CURSOR-PAGINATION-ALL-VIEWERS** ‚Äî pagination is in `<LogViewer>` and on all 3 endpoints, not viewer-specific.
- **D-CURSOR-OPAQUE** ‚Äî UI does not interpret the cursor; just passes it back.

## Design Decision (locked 2026-05-29)

### Selected approach: A1 synthetic collision-safe cursor + pagination on ALL viewers (incl. cron) when content exceeds page size

**User directive (2026-05-29):** "pagination on everything longer than a certain number of lines." ‚Üí ALL three log endpoints get cursor pagination (the literal D-CURSOR-PAGINATION-ALL-VIEWERS reading, B2+C2). Pagination only SURFACES when the result exceeds the page size ‚Äî `next_cursor` is null and the "Load older" button is hidden whenever the full result fits in one page. So a short run/window never shows the affordance; a long one does.

**The cursor (A1 ‚Äî collision-safe, VL-OOM-safe):**
- Cursor = `base64(JSON {t: <oldest returned line's _time in ns>, n: <count of returned lines sharing that exact ns at the boundary>})`. Opaque to the UI (D-CURSOR-OPAQUE).
- "Load older" page = re-query the SAME expr with an INCLUSIVE upper time bound `[start, t]` over VictoriaLogs' existing `limit=N`-latest path, fetch `limit = page_size + n` latest, then DROP the `n` boundary lines already shown. This is provably gap-free AND dup-free under `_time` collisions.
- **Burst subtlety (load-bearing for Build):** if consecutive pages share the same boundary ns `t` (a burst of >page_size lines at one timestamp), `n` ACCUMULATES across those pages (`n2 = n + lines-at-t-this-page` when the new oldest ns == previous cursor's `t`) so paging through a same-timestamp burst stays correct.
- **Exhaustion / `has_more`:** when VL returns fewer than `page_size` strictly-older lines after the skip ‚Üí `next_cursor = None`, `has_more = false`. Cursor time ‚â§ window start ‚Üí empty page + null cursor.
- **CRITICAL VL constraint (verified):** do NOT use VL's `sort offset limit` pagination ‚Äî it OOMs even on tiny result sets and makes the `limit=` query-arg behave inconsistently (VL issues #129/#8127). Use the time-window-narrowing + `limit=N`-latest path (the existing `VictoriaLogsClient.query` mechanism + an `end`-override + `limit`-override). NO `sort` pipe.

**Backend endpoints (all 3 get the cursor):**
- `GET /api/logs/query` ‚Äî `LogsQueryResponse` already has `next_cursor: str|None`; add `has_more` + accept `cursor: str|None` query param. Page size default 500 (cap 5000 retained).
- `GET /api/integrations/docker/containers/{name}/logs` ‚Äî add `next_cursor` + `has_more` to `ContainerLogsResponse`; accept `cursor`. Page size 500 (existing clamp).
- `GET /api/crons/{fp}/runs/{run_id}/log` ‚Äî add `next_cursor` + `has_more` to `RunLogResponse`; accept `cursor`. WITHIN-run pagination over `[vl_window_start, vl_window_end]`: when a run emits >page_size lines, "Load older" pages within the run's window; null cursor when the run fits in one page.
- Shared cursor encode/decode helpers (base64-JSON) + a `LogCursor` model; malformed cursor ‚Üí 400 `invalid_cursor`. Put in a shared module (e.g. `kernel/api/models/logs.py` or `kernel/logs/`).

**Frontend (C2 ‚Äî shared affordance, cron viewer migrated):**
- Extend `UseLogsResult` (apps/ui/src/components/logs/types.ts) with OPTIONAL `hasMore?`, `isLoadingOlder?`, `loadOlder?` ‚Äî all optional so existing callers are unaffected.
- `<LogViewer>` renders an explicit "Load older" button at the TOP of the line list, shown only when `loadOlder != null && hasMore`. Append older pages to the existing list (don't replace); disable/hide when `hasMore===false`; spinner via `isLoadingOlder`; error preserves already-loaded lines.
- Hooks use React Query `useInfiniteQuery` mirroring the existing `useListDockerSuggestions` precedent (`getNextPageParam: lastPage.next_cursor ?? undefined`; flatten `data.pages.flatMap(p => p.lines)`): a new `/api/logs/query` hook, updated `useContainerLogs`, updated `useCronRunLog`.
- **CronRunLogViewer refactor (C2):** migrate the bespoke cron JSX (sticky metadata header, run-state badge, anomaly flags, duration, running-refresh, expired notice) to render through `<LogViewer>` (via `headerSlot` for the metadata) so the shared "Load older" affordance applies. Remove the `// TODO: STAGE-004-005 cursor pagination` placeholder.

**Affordance / opacity / page size (locked):**
- Cursor opacity: **base64-JSON** (URL-safe, opaque-by-appearance).
- Affordance: **explicit "Load older" button** at top of list (NOT auto-load-on-scroll ‚Äî avoids scroll-anchor jank + surprise fetches; deterministic to test).
- Page size: **500 default**, per-viewer override prop available (unused v1).
- Cursor source: **synthetic** `(timestamp_ns, n)` ‚Äî VL has no native cursor.

**Honors locked decisions:** D-CURSOR-OPAQUE (UI passes the cursor straight back, never decodes), D-CURSOR-BACKWARD-ONLY-V1 (only tightens `end` toward the past; no forward/newer path), D-CURSOR-PAGINATION-ALL-VIEWERS (all 3 endpoints + shared `<LogViewer>` ‚Äî literal reading per user directive).

**Test strategy:** cursor base64-JSON round-trip + malformed‚Üí400; BOUNDARY-COLLISION correctness (plant ‚â•3 lines at same ns straddling a page boundary ‚Üí page1 ‚à™ page2 == full set, no dup/no gap, asserted via unique per-line marker in message); burst-at-one-timestamp (page_size + k lines at one ns ‚Üí accumulated n, eventual exhaustion); has_more exhaustion + window-floor; existing-viewer-still-works (docker test unchanged when hook omits new fields); cron viewer renders via LogViewer with pagination; frontend useInfiniteQuery append + button gating + error-preserves-lines.

**Stage flags:** frontend + backend, NO host-integration ‚Üí Refinement **3a only** (dev rig). Frontend ‚Üí Refinement requires manual Desktop + Mobile testing (user prompt). Acceptance: user clicks "Load older" in a viewer, sees older lines appear, button disables when exhausted.

**Look-Ahead (consumers of the cursor contract ‚Äî keep stable):** STAGE-004-010 (Explorer `useLogsExplorerQuery` returns the pagination shape), 004-015 (persists `cursor?: string` across navigation), 004-023 (live-tail `TailConnection.last_cursor: LogCursor`), 004-026 (Drain consumer `last_processed_ts` cursor). The `LogCursor` shape + base64 encoding must stay backward-compatible for these.

## Refinement Notes (2026-05-30)

**Refinement complete:** 2026-05-30. 3a only (no host-integration). Backend e2e against real VictoriaLogs + frontend manual Desktop+Mobile (user approved both viewports).

**Backend e2e (real VL, planted-corpus) ‚Äî confirmed + found a real bug:**
- **[A-END-INCLUSIVE] CONFIRMED:** VL's `end` time bound is INCLUSIVE at nanosecond precision (queried with `end` == an exact collision ns ‚Üí all same-ns lines returned). So `_ns_to_iso(decoded.t)` is correct; no `+1` needed.
- **DATA-LOSS BUG FOUND + FIXED (the value of real-data Refinement):** VictoriaLogs' `limit=N` TRUNCATES MID same-ns group ‚Äî when the cap lands inside a set of lines sharing the exact `_time` (ns), VL returns an ARBITRARY SUBSET of that group, not all of it. The A1 algorithm assumed `fetch_limit = page_size + skip_n` always contained the full boundary group; it didn't ‚Üí pagination terminated early (`has_more=False`) and SILENTLY DROPPED lines (e.g. page_size=3 over a 7-line corpus with a 3-line collision returned only 3 lines, dropping 4). Unit tests (which mocked VL returning the full group) could not catch this. FIX: a `[GROUP-COMPLETE]` branch ‚Äî when the fetch is capped at a same-ns boundary, re-query the COMPLETE group at exactly that ns (`start==end`) and include the whole group in the page (a page never splits a same-ns group), guaranteeing gap-free pagination + cursor advance even for unbounded single-ns bursts. Plus `has_more` corrected to use `result.truncated` (VL's authoritative more-in-window signal), NOT `len(fetched)==fetch_limit` (which over-fired). New regression test `test_paginate_mid_group_truncation_no_data_loss` with a truncating-VL mock reproduces it (fails-before/passes-after).
- malformed cursor ‚Üí 400; short content (page_size > corpus) ‚Üí single page, no cursor; exhaustion ‚Üí null cursor: all PASS.

**Frontend manual (Desktop + Mobile) ‚Äî found + fixed an order bug:**
- **BUG: "Load older" appended older lines at the BOTTOM (newest position) instead of prepending at the TOP** (lists render oldest‚Üínewest). Root cause: `useInfiniteQuery` accumulates `data.pages` newest-first (pages[0]=newest window, later pages=older), and the viewers flattened `pages.flatMap(p => p.lines)` ‚Äî putting older pages after newer ones. FIX: flatten in reverse PAGE order `pages.slice().reverse().flatMap(p => p.lines)` in both DockerContainerLogsViewerBody.tsx and CronRunLogViewer.tsx (status/truncated/header still derive from `pages[0]`; hasMore from `hasNextPage` ‚Äî unaffected). Added a multi-page order test per viewer (`renders older pages above newer pages in multi-page load`). `useLogsQuery` docstring corrected for the next consumer.
