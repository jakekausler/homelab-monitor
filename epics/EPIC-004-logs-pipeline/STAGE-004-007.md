# STAGE-004-007: Cursor pagination across all 3 endpoints + `<LogViewer>`

## Status: Not Started

## Overview

Today's log viewers fetch a single batch of lines and render them. Beyond the batch size cap (500 or so), older lines are silently dropped with only a `truncated: true` banner. The user can't go back further than that window.

This stage adds cursor-based pagination — "load older lines" — to:

- `GET /api/integrations/docker/containers/{name}/logs`
- `GET /api/crons/{fp}/runs/{run_id}/log`
- `GET /api/logs/query`

…and to the `<LogViewer>` component (STAGE-004-003), which gains an "Older lines" button at the top of the line list (or auto-loads on scroll, Design decides).

This fixes the deferred STAGE-003-011 design decision D-DEFER-PAGINATION.

**Cursor scheme:** `(timestamp_ns, line_seq)` where `line_seq` is a deterministic position counter within the VL result set. Forward + backward pagination both supported. Cursor is opaque to the client (base64-encoded JSON or similar).

**Vertical slice:** in any log viewer, the user can click "Load older" and see lines older than the current window. The cursor survives navigation away + back (when state persistence lands in STAGE-004-015).

**Spec references:** master spec §9.2; EPIC-004 STAGE-004-005 acceptance criteria (the original plan grouped pagination + custom range under one stage; we've split them).

## Concrete deliverables

### 1. Cursor type definition

`apps/monitor/homelab_monitor/kernel/api/models/logs.py` — add:

```python
class LogCursor(BaseModel):
    """Opaque cursor for log pagination. Base64-encoded JSON over the wire."""
    timestamp_ns: int  # nanosecond precision (VL native)
    line_seq: int  # tie-breaker for same-timestamp lines

class LogsQueryResponse(BaseModel):
    lines: list[LogLine]
    next_cursor: str | None  # opaque, encode/decode helpers in line_mapping.py
    has_more: bool
    # existing fields...
```

`encode_cursor(cursor: LogCursor) -> str` and `decode_cursor(s: str) -> LogCursor` helpers in `line_mapping.py`. Base64-encoded JSON works fine (it's not security-sensitive, just opaque to UI).

### 2. Endpoint params + behavior

All three endpoints accept `cursor: str | None = None` query param:

- If `cursor` is `None`: return most recent N lines + a cursor to the "older" boundary
- If `cursor` is present: return N older lines starting from the cursor + a new cursor (or `None` if exhausted)
- `has_more: bool` tells the UI whether the "Load older" button should remain enabled

VL query: use VL's native cursor or `_time_ns` ordering with strict less-than against the cursor's timestamp_ns. Tie-breaking via `line_seq` (sequence within the VL response or another deterministic ordering — Design decides).

### 3. Page size

Each endpoint accepts `limit: int = 500` (or whatever the current default is). Max enforced server-side. Frontend default: 500.

### 4. Backward + forward pagination semantics

For now: backward (older) only. Forward (newer) auto-handled by re-fetching the latest window. If a use case for "show me lines newer than cursor X" emerges, add `direction: "older" | "newer"` param.

### 5. `<LogViewer>` component update

`apps/ui/src/components/logs/LogViewer.tsx` — add:

- "Load older lines" button at top of line list (or auto-load on scroll-to-top — Design picks)
- Append older lines to the existing list (don't replace)
- Disable button when `has_more === false`
- Loading spinner during pagination fetch
- Error state on failed pagination (retry preserves current state)

Caller-provided `useLogs` hook gets a new optional `useOlder` method:

```typescript
useLogs: () => {
  lines: LogLine[]
  hasMore: boolean
  loadOlder: () => Promise<void>  // new
  isLoadingOlder: boolean
  // ... existing fields
}
```

### 6. Hook implementations

`apps/ui/src/api/docker.ts` — `useContainerLogs(name, since)` returns the pagination-capable hook.
`apps/ui/src/api/crons.ts` — `useCronRunLog(fp, runId)` returns the pagination-capable hook.
`apps/ui/src/api/logs.ts` — `useLogsQuery(expr, range)` returns the pagination-capable hook.

Each uses React Query's `useInfiniteQuery` (the right primitive for cursor pagination).

## Decisions in this stage

### Design phase will confirm

- **Cursor opacity** — base64-JSON vs raw JSON; recommendation: base64-JSON (opaque looking, no client guessing)
- **Auto-load-on-scroll vs explicit "Load older" button** — recommendation: explicit button for v1
- **Page size** — 500 default, configurable per-viewer
- **VL native cursor vs synthetic timestamp+seq cursor** — recommendation: synthetic (timestamp_ns + line_seq) because it works across all 3 endpoints regardless of VL version

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-CURSOR-PAGINATION-ALL-VIEWERS** — pagination is in `<LogViewer>` and on all 3 endpoints, not viewer-specific.
- **D-CURSOR-OPAQUE** — UI does not interpret the cursor; just passes it back.
- **D-CURSOR-BACKWARD-ONLY-V1** — "Load older" only in v1. Forward / `newer` pagination defers until needed.
