# STAGE-004-023: Backend SSE endpoint for live tail (server-side streaming from VL)

## Status: Not Started

## Overview

Add a Server-Sent Events (SSE) endpoint that streams new log lines matching a LogsQL filter to the client in real time. Replaces today's "click refresh to see new lines" pattern with a `tail -f`-like experience.

This is the BACKEND half of live tail. The FRONTEND consumer (Explorer tail mode + button-toggle) lands in STAGE-004-024.

**Vertical slice:** `GET /api/logs/tail?expr=<LogsQL>` returns `text/event-stream`. New lines matching the filter are pushed as `data: {...LogLine JSON...}` events with ~1s latency. Connection survives idle periods via keepalive comments.

**Spec references:** master spec §9.2 (live tail).

## Concrete deliverables

### 1. SSE endpoint

`apps/monitor/homelab_monitor/kernel/api/routers/logs.py`:

```
GET /api/logs/tail?expr=<LogsQL>&services=<csv>
Content-Type: text/event-stream

event: line
data: {"timestamp":"...", "severity":"info", ...}

event: line
data: {...}

: keepalive 1717000000

event: error
data: {"code":"vl_unavailable", "message":"..."}
```

- Auth: cookie session (same as other endpoints)
- CSRF: SSE GET requests don't need CSRF tokens (per project policy)
- Events: `line` (new log line as JSON), `error` (VL unavailable, query parse error), keepalive comments every 30s

### 2. SSE service implementation

`apps/monitor/homelab_monitor/kernel/logs/tail_service.py` (new):

- Polls VL every ~1s (configurable via env: `HOMELAB_MONITOR_TAIL_POLL_MS=1000`)
- For each poll: queries VL for lines newer than the last-seen cursor (since the previous poll)
- Streams new lines to the connected client
- Tracks cursor per connection
- On client disconnect: closes the polling loop cleanly

### 3. Per-connection state

```python
class TailConnection:
    expr: str  # LogsQL
    services: list[str]  # selected services filter
    last_cursor: LogCursor  # tracking position
    started_at: datetime
    client_addr: str  # for logging
    queue: asyncio.Queue[bytes]  # outbound events
```

### 4. Connection limits

Configurable via env vars:

- `HOMELAB_MONITOR_TAIL_MAX_CONNECTIONS=5` (max concurrent tail connections globally)
- `HOMELAB_MONITOR_TAIL_MAX_LINES_PER_SEC=200` (per-connection backpressure: if VL returns more than this, drop oldest from queue and inject a "lines_dropped" event)
- `HOMELAB_MONITOR_TAIL_MAX_DURATION_S=3600` (per-connection max session; closed after expiry)

When max-connections is exceeded, the new request returns 503 with `Retry-After: 60`.

### 5. Error handling

- VL transport error: emit `error` event with `code: "vl_unavailable"`, retry every 10s; if 5 consecutive failures, close the connection
- Bad LogsQL: 422 on initial request; never reaches the SSE stream
- Client disconnect: terminate the polling loop within ~2s

### 6. Tests

- `apps/monitor/tests/test_api_logs_tail.py` — SSE event format, keepalive, error event on VL down, max-connections enforcement, client-disconnect cleanup
- Uses `httpx` async streaming client to consume the SSE response in tests
- Mock VL with `pytest-httpx` to simulate new lines arriving over time

### 7. Metrics

Emit gauges/counters for observability:

- `homelab_log_tail_active_connections` (gauge)
- `homelab_log_tail_lines_streamed_total` (counter)
- `homelab_log_tail_lines_dropped_total` (counter; backpressure)
- `homelab_log_tail_errors_total{kind}` (counter)

## Out of scope for this stage

- Frontend consumer — STAGE-004-024
- WebSocket alternative (SSE is simpler and unidirectional, sufficient here)
- Server-side filtering beyond LogsQL — VL does it
- Replay / catch-up of historical lines after reconnect — SSE re-connect starts from "now"; user can use regular query for history
- Multi-client fanout from a single VL poll — premature optimization; each connection gets its own VL query

## Decisions in this stage

### Design phase will confirm

- **VL poll interval** — 1s default; tune lower if VL latency is fine
- **Cursor delivery to client** — include cursor as a field on the event? Recommendation: yes, so reconnects can use it
- **Connection limit defaults** — 5/200/3600s seed values; tune in 3b
- **Whether to expose `homelab_log_tail_*` metrics already** — yes, no reason not to

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-LIVE-TAIL-SSE** — Server-Sent Events, not WebSocket. Unidirectional, simpler, well-supported.
- **D-LIVE-TAIL-POLL-VL** — backend polls VL; doesn't subscribe to a real-time VL feed. Avoids coupling to VL version specifics.
- **D-LIVE-TAIL-BACKEND-FIRST** — backend ships standalone; frontend integration is STAGE-004-024.
- **D-LIVE-TAIL-CONNECTION-CAPS** — enforce max-connections + per-connection backpressure to bound resource use.
