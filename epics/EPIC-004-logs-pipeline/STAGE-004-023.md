# STAGE-004-023: Backend SSE endpoint for live tail (server-side streaming from VL)

## Status: Finalize

## Overview

Add a Server-Sent Events (SSE) endpoint that streams new log lines matching a LogsQL filter to the client in real time. Replaces today's "click refresh to see new lines" pattern with a `tail -f`-like experience.

This is the BACKEND half of live tail. The FRONTEND consumer (Explorer tail mode + button-toggle) lands in STAGE-004-024.

**Vertical slice:** `GET /api/logs/tail?expr=<LogsQL>` returns `text/event-stream`. New lines matching the filter are pushed as `data: {...LogLine JSON...}` events with ~1s latency. Connection survives idle periods via keepalive comments.

## Concrete deliverables

### 1. SSE endpoint

`apps/monitor/homelab_monitor/kernel/api/routers/logs.py`:

```
GET /api/logs/tail?expr=<LogsQL>&services=<csv>
Content-Type: text/event-stream

event: line
data: {"timestamp":"...", "severity":"info", ...}

event: line
data: {...}

: keepalive 1717000000

event: error
data: {"code":"vl_unavailable", "message":"..."}
```

- Auth: cookie session (same as other endpoints)
- CSRF: SSE GET requests don't need CSRF tokens (per project policy)
- Events: `line` (new log line as JSON), `error` (VL unavailable, query parse error), keepalive comments every 30s

### 2. SSE service implementation

`apps/monitor/homelab_monitor/kernel/logs/tail_service.py` (new):

- Polls VL every ~1s (configurable via env: `HOMELAB_MONITOR_TAIL_POLL_MS=1000`)
- For each poll: queries VL for lines newer than the last-seen cursor (since the previous poll)
- Streams new lines to the connected client
- Tracks cursor per connection
- On client disconnect: closes the polling loop cleanly

### 3. Per-connection state

```python
class TailConnection:
    expr: str  # LogsQL
    services: list[str]  # selected services filter
    last_cursor: LogCursor  # tracking position
    started_at: datetime
    client_addr: str  # for logging
    queue: asyncio.Queue[bytes]  # outbound events
```

### 4. Connection limits

Configurable via env vars:

- `HOMELAB_MONITOR_TAIL_MAX_CONNECTIONS=5` (max concurrent tail connections globally)
- `HOMELAB_MONITOR_TAIL_MAX_LINES_PER_SEC=200` (per-connection backpressure: if VL returns more than this, drop oldest from queue and inject a "lines_dropped" event)
- `HOMELAB_MONITOR_TAIL_MAX_DURATION_S=3600` (per-connection max session; closed after expiry)

When max-connections is exceeded, the new request returns 503 with `Retry-After: 60`.

### 5. Error handling

- VL transport error: emit `error` event with `code: "vl_unavailable"`, retry every 10s; if 5 consecutive failures, close the connection
- Bad LogsQL: VictoriaLogs (v0.30.0) returns HTTP 200 + empty results for arbitrary phrase-garbage (it treats unrecognized text as a phrase filter), and HTTP 4xx ONLY for STRUCTURAL errors (unparseable start/end, dangling pipe like `|limit 10`). The pre-flight probe maps VL's structural 4xx -> 422 `invalid_logsql` on the initial request (never reaches the SSE stream); arbitrary phrase-garbage is VALID LogsQL and correctly opens a 200 stream that emits zero lines. This matches VL's real contract and how /logs/query and /logs/export already behave (no client-side LogsQL validation anywhere in the repo). [Confirmed against real VL v0.30.0 during Refinement.]
- Client disconnect: terminate the polling loop within ~2s

### 6. Tests

- `apps/monitor/tests/test_api_logs_tail.py` â€” SSE event format, keepalive, error event on VL down, max-connections enforcement, client-disconnect cleanup
- Uses `httpx` async streaming client to consume the SSE response in tests
- Mock VL with `pytest-httpx` to simulate new lines arriving over time

### 7. Metrics

Emit gauges/counters for observability:

- `homelab_log_tail_active_connections` (gauge)
- `homelab_log_tail_lines_streamed_total` (counter)
- `homelab_log_tail_lines_dropped_total` (counter; backpressure)
- `homelab_log_tail_errors_total{kind}` (counter)

## Out of scope for this stage

- Frontend consumer â€” STAGE-004-024

## Design Decision (Selected Approach)

**Selected: Option B â€” `TailService` class owns the poll loop; thin handler; counters written direct, `active_connections` gauge published by a lightweight collector reading a registry.**

**New file `kernel/logs/tail_service.py`:**
- `TailConfig` (frozen dataclass, slots) + `load_tail_config()` reading `HOMELAB_MONITOR_TAIL_POLL_MS=1000`, `HOMELAB_MONITOR_TAIL_MAX_CONNECTIONS=5`, `HOMELAB_MONITOR_TAIL_MAX_LINES_PER_SEC=200`, `HOMELAB_MONITOR_TAIL_MAX_DURATION_S=3600` (mirrors `load_vl_query_limits`).
- `ForwardCursor` â€” `{last_seen_ns:int, n_at_boundary:int}`; `advance(lines) -> (emit_lines, new_cursor)` does same-ns boundary dedup (inverse of `paginate_older`). First poll anchors `last_seen_ns = now_ns`, `n=0` (tail starts empty â†’ shows only NEW lines). Subsequent poll window = `[last_seen_ns, now]`.
- `TailRegistry` (held on `app.state.tail_registry`, initialized in `lifespan.py`): `active_count:int`, `try_acquire() -> bool` (False at cap), `release()`. Single source of truth for the gauge.
- `TailEvent` tagged union: `LineEvent(LogLine)`, `DroppedEvent(count:int)`, `ErrorEvent(code,message)`, plus a keepalive sentinel.
- `TailSession` (injected `clock` + `vl_client` + `metrics_writer`): `async def events() -> AsyncIterator[TailEvent]` owns 1s poll cadence, window bounding, cursor dedup, per-second drop-surplus backpressure, duration cap (3600s), and VL-error retry (emit `error`, sleep ~10s, close after 5 consecutive failures). Writes counters directly: `homelab_log_tail_lines_streamed_total`, `homelab_log_tail_lines_dropped_total`, `homelab_log_tail_errors_total{kind}`.

**VL access per poll:** use bounded `query(expr, [last_seen, now])` (NOT `stream_query`) â€” simpler for a repeated 1s poll, already capped by `VlQueryLimits.max_lines`, no per-second stream-context churn.

**Backpressure:** per-second window counter â€” cap emission at 200 lines/sec; surplus dropped, `lines_dropped_total += surplus`, and ONE `event: dropped\ndata: {"count":n}` SSE event emitted (its own event type so STAGE-004-024 can render a "N dropped" chip without parsing line payloads).

**Metrics path:** counters written directly from the request/session path via `app.state.metrics_writer` (`write_counter` does a true `.inc()`, correct under concurrency). The `active_connections` gauge is published by a NEW lightweight collector (`TailMetricsCollector`, registered in `lifespan.py` like `LogStreamBudgetCollector`) that reads `tail_registry.active_count` each tick and calls `ctx.vm.write_gauge("homelab_log_tail_active_connections", count)`. (Gauge via collector â€” NOT request-path `.set()` â€” because `.set()` from N concurrent connections clobbers, and collector-published gauges are the established repo convention.)

**Handler `GET /api/logs/tail?expr=<LogsQL>&services=<csv>` (in `logs.py`):**
1. Validate `len(expr) <= 4096` (400 `invalid_expr`); `effective_expr = _compose_services_expr(expr, services)`.
2. `registry.try_acquire()` â†’ if False, `HttpProblem(503, "tail_capacity", Retry-After: 60)`.
3. Probe: one bounded `query` over a tiny `[now-1s, now]` window. Map VL **4xx â†’ 422 `invalid_logsql`**, VL **5xx/transport â†’ 502 `upstream_unavailable`**. **On ANY probe failure, `registry.release()` before raising** (prevents slot leak). This requires the VL client to expose the HTTP status (add `status_code` to `VictoriaLogsClientError`, or a probe method raising a distinct `LogsQLInvalidError` on 4xx) â€” today it swallows 4xx vs 5xx.

   NOTE (Refinement, confirmed against real VL v0.30.0): VL returns 4xx ONLY for STRUCTURAL errors (bad start/end params, dangling pipe). Arbitrary phrase-garbage returns 200 + empty (valid phrase filter). So the 422 path fires for structural errors, NOT for every "looks wrong" string â€” consistent with /logs/query and /logs/export, which have no LogsQL validation at all. No code change; the probe's mapping is correct as-is.
4. Build `TailSession`, return `StreamingResponse(gen(), media_type="text/event-stream", headers={Cache-Control:no-cache, X-Accel-Buffering:no, Content-Type:text/event-stream; charset=utf-8})`. `gen()` formats `TailEvent`s â†’ SSE bytes (`event: line` / `event: dropped` / `event: error` + `: keepalive` comment ~30s, driven by an idle timeout INSIDE `session.events()` â€” NOT by wrapping `__anext__` in `wait_for`, per the events.py bug note). `finally:` `registry.release()` (guarantees release on disconnect/cancel/duration-close).

**Strict ordering:** cap-check (503) â†’ LogsQL probe (422 bad / 502 VL-down) â†’ 200 stream. Acquire BEFORE probe (so bad-query floods can't run unbounded probes); release-on-probe-failure.

**Event contract for STAGE-004-024:** `event: line` (LogLine JSON), `event: dropped` ({count}), `event: error` ({code,message}), `: keepalive` comments. No cursor field in events (reconnect starts from "now").

**Files to touch:** `kernel/logs/tail_service.py` (new), `kernel/api/routers/logs.py` (handler), `kernel/api/dependencies.py` (`get_tail_registry`), `kernel/api/lifespan.py` (registry + TailMetricsCollector), `kernel/config.py` (`TailConfig`+`load_tail_config`), `kernel/logs/victorialogs_client.py` (4xx/5xx distinction for the probe), possibly `kernel/api/errors.py` (Retry-After header on HttpProblem). Tests in `apps/monitor/tests/test_api_logs_tail.py` + unit tests for `ForwardCursor`/`TailSession`/`TailRegistry`.

**Rejected alternatives:**
- **Option A (inline generator in the handler)** â€” rejected: cramming the poll loop + cursor + backpressure + retry state machine + duration cap into one ~150-line inline `gen()` makes the 100%-kernel-coverage gate painful (those branches reachable only through the ASGI/StreamingResponse path). Option B extracts them into pure, directly-unit-testable objects.
- **Option C (Option B's structure but the gauge written directly from the request path via `TailRegistry` calling `metrics_writer.write_gauge` on acquire/release)** â€” rejected despite being the brainstormer's nominal recommendation: request-path gauge writes are NOT an established pattern in this repo (every existing gauge is collector-published), it would predictably draw a Finalize code-review flag, and the only benefits (no ~5s staleness, one fewer file) are negligible for a 5-connection-max homelab gauge. Option B matches the "plugins observe themselves" / collector-published-gauge convention at trivial cost.

**Open items confirmed for Build:** (1) does `HttpProblem` support a `Retry-After` response header? â€” verify in `errors.py`, add if absent. (2) Tail start anchor = `now` (empty-then-live). (3) VL client 4xx-vs-5xx distinction is a hard prerequisite for the 422 requirement and the most likely scope surprise.
