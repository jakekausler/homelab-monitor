# STAGE-004-002: Backend `LogLine` shape convergence across all 3 log-fetching endpoints

## Status: Not Started

## Overview

Today three endpoints return log data with three different response shapes:

- `GET /api/integrations/docker/containers/{name}/logs` → `ContainerLogsResponse { lines: LogLine[], log_status, truncated, window_start/end }`
- `GET /api/crons/{fp}/runs/{run_id}/log` → `RunLogResponse { entries: LogEntry[], state, duration_seconds, line_count, anomaly_flags, log_status, truncated }`
- `GET /api/logs/query` → `LogsQueryResponse { lines: VictoriaLogsLine[] }` (raw VL pass-through)

The inner per-line types diverge (`LogLine` vs `LogEntry` vs `VictoriaLogsLine`) in field names, optionality, and field set. This blocks the shared `<LogViewer>` component (STAGE-004-003) because the component would need three rendering paths.

This stage converges the inner line type to ONE Pydantic model (`LogLine`) used by all three endpoints, while preserving each endpoint's outer-envelope metadata (container status, cron state + anomaly flags, etc.). UI doesn't change yet — `<LogViewer>` extraction is the next stage.

**Vertical slice:** the three endpoints all return the same line shape. Existing UI continues to work (the converged shape is a superset of what each viewer reads today). Tests for all three endpoints assert the converged shape.

**Spec references:** Master spec §6.3 (VL streams keyed by host/service/severity), §9.2 (Logs explorer screen).

## Concrete deliverables

### 1. The converged `LogLine` model

`apps/monitor/homelab_monitor/kernel/api/models/logs.py` (new file or extend existing) — single Pydantic model used by all 3 endpoints:

```python
class LogLine(BaseModel):
    timestamp: datetime  # UTC, ISO-8601
    severity: str | None = None  # info / warn / error / debug / etc., normalized to lowercase
    service: str | None = None  # service label as set by vector add_labels transform
    host: str | None = None  # host label
    message: str  # the log message (post-multiline-stitch; may contain \n)
    fields: dict[str, Any] = Field(default_factory=dict)  # extra structured fields (compose_*, run_id, etc.)
```

- `fields` is the bag for everything not in the typed core (compose_project, compose_service, container_id, run_id, cron_fingerprint, image labels — STAGE-004-004 will start populating these)
- All optional fields are typed `... | None` not `Optional[...]` per project pyright style
- Tests cover serialization (Pydantic → JSON) + deserialization with realistic VL payload shapes

## Decisions in this stage

### Design phase will confirm

- **Whether to rename `RunLogResponse.entries` → `lines`** — semantic consistency but frontend churn. Recommendation: keep `entries` name, change inner type only. Design phase locks.
- **Whether `LogLine.fields: dict[str, Any]`** stays untyped or becomes a typed sub-model — recommendation: dict[str, Any] for v1 to absorb the heterogeneous bag. Typing it later is non-breaking.
- **Severity normalization scheme** — what's "warn" vs "warning" vs "WARN" vs syslog severity numeric? Recommendation: normalize to lowercase canonical set (`debug, info, notice, warn, error, critical, alert, emergency`) at ingest time.

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-CONVERGED-LINE-SHAPE** — single `LogLine` model used by all log-fetching endpoints. Outer envelope (status, container metadata, cron run state) stays endpoint-specific.
- **D-SHAPE-CONVERGENCE-PRECEDES-COMPONENT** — backend shape convergence lands BEFORE the shared `<LogViewer>` extraction (STAGE-004-003) so the component is built against the final shape.
- **D-FIELDS-BAG-UNTYPED-V1** — `fields: dict[str, Any]` for v1; typed sub-models can come later without breaking the wire.
- **D-NO-FRONTEND-REFACTOR** — existing UI continues to work; only TypeScript types regenerate.
