# STAGE-004-044: Create-alert-from-signature + saved-query shortcuts (L8 merged in)

## Status: Complete

## Overview

Add quick entry points from the Signature Catalog (STAGE-004-028) and the Saved Queries (STAGE-004-013) to the alert-authoring modal from STAGE-004-043. Both shortcuts pre-fill the modal with sensible per-source defaults.

For signatures: pre-fill threshold using the spike-vs-baseline template from STAGE-004-036.
For saved queries: pre-fill threshold using a simple `count() > N` over the saved query.

This is L8 from brainstorming — "saved query → alert rule" — merged with create-alert-from-signature into a single stage because they use the same underlying flow.

**Vertical slice:** in the Signature drill-in panel, "Create alert for this signature" button opens the modal with the spike-vs-baseline rule template pre-filled (multiplier 5, window 5m, severity warn, name `SignatureSpike_<hash>_<service>`). In the Saved Queries panel, "Create alert from this saved query" opens the modal with `count(<saved query LogsQL>) > 10` pre-filled.

**Spec references:** master spec §9.2 (saved queries + create alert).

## Design Phase

**Design phase complete: 2026-06-09.**

**Backend-contract verification (Design phase) — card claims corrected:**
- Signature component is `apps/ui/src/routes/logs/SignatureDetailPage.tsx` (card said `SignatureDetailPanel.tsx` — WRONG). Saved-query component `apps/ui/src/routes/logs/SavedQueriesPanel.tsx` ✅ confirmed.
- Modal form type is `CreateAlertFormValues` (card said `CreateAlertFormState` — WRONG).
- `SignatureResponse` carries template_hash, service_key, template_str; `templateToLogsQl(template_str)` (from `@/lib/logsQlTranslate`) is already used by SignatureDetailPage to convert a signature to LogsQL. `SavedQuery` carries id, name, logs_ql, advanced_mode.
- ⚠️ **CONTRACT GAP — source tracking + extra labels are NOT client-settable.** `LogUserRuleCreateRequest` has `extra="forbid"` and NO source_kind/source_ref/labels fields; the modal's create POST explicitly omits source_kind/source_ref (its own prop comments say "FUTURE use / 044, NOT sent in POST"). The 042 *table* has source_kind/source_ref columns (server-defaulted) but the *create API* does not accept them. So the card's "pre-fill source_kind/source_ref + extra labels (template_hash/service_key/anomaly_kind)" is NOT achievable without a 042 schema migration → DESCOPED (see below).
- ⚠️ **CONTRACT GAP — the STAGE-036 spike-vs-baseline template is NOT frontend-reachable.** `render_signature_spike_rule(...)` is backend-only Python (apps/monitor/.../kernel/logs/signature_spike_render.py), produces a METRICSQL spike rule, has NO API endpoint and NO frontend binding. The modal's Simple/scaffold path is logsql-only. → the signature shortcut ships a logsql count-threshold instead (see Decision 1); the spike variant is DESCOPED to a future backend stage.

**Chosen approaches (decided in Design per standing instructions — no user prompt):**

**Decision 1 — Signature alert expr: HYBRID (ship logsql count-threshold now, defer the metricsql spike variant).**
- The signature shortcut (a "Create alert" / "Create spike alert" button on `SignatureDetailPage.tsx`) opens the shared `CreateAlertModal` in ADVANCED mode with `initialValues = { expr: scaffoldLogsqlExpr(templateToLogsQl(sig.template_str)), expr_kind: 'logsql', rule_name: <deterministic safe name> }`, `sourceKind='signature'` (cosmetic, not posted). This mirrors 043's create-from-query advanced path 1:1 and reuses `templateToLogsQl` which SignatureDetailPage already calls. The scaffold guarantees the `| stats count() as match_count | filter match_count:>N` pipe (satisfies the 400-invalid_expr + reserved-`count` regression gates).
- DESCOPED + deferred: the true spike-vs-baseline metricsql rule (036 `render_signature_spike_rule`) — requires a new backend endpoint to expose it to the FE, which violates this stage's frontend-only/3a-only scope and adds 3b host-integration. Recommend a future stage (backend endpoint + FE binding) if the spike semantic is wanted.
- Rejected: (b) building the backend endpoint now (out of frontend-only scope, highest risk on the epic's last stage).

**Decision 2 — Saved-query alert expr: MIRROR 043's mode-detection.**
- If `savedQuery.advanced_mode` → ADVANCED, `initialValues = { expr: scaffoldLogsqlExpr(savedQuery.logs_ql), expr_kind:'logsql', rule_name: <safe> }`. Else → SIMPLE, `initialValues = { simple_contains: savedQuery.logs_ql, expr_kind:'logsql', rule_name: <safe> }`. `sourceKind='saved_query'` (cosmetic).
- DECISIVE correctness fact: a plain-mode saved query stores RAW PLAIN TEXT in `logs_ql` (not compiled LogsQL — confirmed via LogsExplorerPage `logsQl = advanced ? lql : plain`). So "always advanced" (rejected option a) would feed unescaped raw text as LogsQL → 400. Mode-detection routes plain text through the Simple builder's escaping — exactly the proven 043 contract.

**Decision 3 — Duplicate/disambiguation: lean on the modal's existing dup-name backstop via deterministic rule_name.**
- A shared `deriveRuleName(prefix, idPart, namePart)` sanitizer produces a DETERMINISTIC safe rule_name (`^[a-zA-Z_][a-zA-Z0-9_]*$`): collapse non `[a-zA-Z0-9_]` to `_`, guarantee a leading letter/underscore (prefix `r_` if needed). Because the same signature/saved-query always yields the SAME default rule_name, re-launching the shortcut collides with the existing rule → the modal's EXISTING `existingNames` client check surfaces the `create-alert-name-conflict` warning and the backend 409 is the hard backstop. This satisfies the SPIRIT of card deliverable #4 ("don't silently create a duplicate") with ZERO new dedupe code.
- The literal card deliverable ("dedupe by source_kind+source_ref") is NOT satisfiable because those fields aren't persisted (see contract gap). Approximated via deterministic naming. The rule_name stays user-editable so a collision is recoverable.

**New file:** a small pure helper module (e.g. `apps/ui/src/routes/logs/alertPrefill.ts`) exporting `prefillFromSignature(sig, opts?)`, `prefillFromSavedQuery(sq, opts?)`, and `deriveRuleName(...)` — returns `{ initialMode, initialValues, sourceKind, sourceRef }`. 100% test coverage on this pure logic (the card asks for it). The two panels import + use it to launch the shared modal.

**User manual test (both shortcuts): APPROVED Desktop + Mobile.**

Two issues found + fixed during Refinement:
1. **Saved-query service-scope bug (user-reported).** "Create alert from saved query" IGNORED the saved query's `selected_services` (service scope is stored SEPARATELY from logs_ql and the Explorer ANDs it on at query time). Alerts from service-scoped saved queries (e.g. "nginx errors" → docker:nginx; "infra services overview" → grafana/victorialogs/alertmanager) fired on ALL services. FIX (alertPrefill.ts): new pure `composeServicesExpr(baseExpr, services)` replicating the backend `_compose_services_expr` pattern EXACTLY — `service:"<svc>" AND source_type:"<src>"` per identity (NO `=`), OR'd then AND'd with base: single → `(clause) AND (base)`, multiple → `((c1) OR (c2) ...) AND (base)`, cap 1000, skip malformed, empty-base → just the services clause (no trailing ` AND ()`). `prefillFromSavedQuery` now FORCES Advanced mode when `selected_services` is non-empty, expr = `scaffoldLogsqlExpr(composeServicesExpr(logs_ql, selected_services))` (service scope visible/editable in the Advanced expr). Empty selected_services → byte-stable prior behavior. Simple-mode multi-select picker was DEFERRED (broader modal change; the Advanced expr carries the multi-service filter correctly — the "fix it correctly" path the user chose).
2. **Latent 409-name-conflict race bug (found during testing).** Under CPU contention (only manifests in `make verify`'s concurrent backend-pytest + UI-vitest run — NOT isolated vitest, which false-greened 5 fix attempts), the modal's 300ms client-side uniqueness DEBOUNCE timer could fire AFTER the 409 catch set `nameError` and CLOBBER it back to null → the conflict warning rendered then vanished. ROOT CAUSE (Opus debugger, instrumented): two independent writers of nameError racing. FIX (CreateAlertModal.tsx): track the uniqueness debounce timer in a ref; a `setAuthoritativeNameError` helper cancels the pending debounce before setting the submit-path error (client-side dup backstop + 409 catch). Verified 10/10 under nproc-core CPU load (was 7/8 fail) + clean full `make verify` GREEN.

Also fixed two PRE-EXISTING flaky tests blocking verification: two time-dependent cron-event tests (hardcoded stale 2026-05-19 timestamps closed by the reconciler → made now-relative); scheduler e2e load-flakes (test_scheduler*.py — confirmed environmental, pass in isolation, @pytest.mark.slow, no source change — not modified). The CreateAlertModal post-mutation test cluster (409/400) was hardened to `waitFor` (bounded retry) + the component fix.

Verification: clean `make verify` GREEN — pyright 0, pytest 4390 + 100% kernel coverage, tsc 0, eslint 0 errors, vitest 1372 pass (incl. the now-deterministic 409/400 tests) ZERO unhandled errors, UI build OK.
