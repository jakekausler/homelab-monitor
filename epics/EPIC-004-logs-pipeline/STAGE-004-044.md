# STAGE-004-044: Create-alert-from-signature + saved-query shortcuts (L8 merged in)

## Status: Not Started

## Overview

Add quick entry points from the Signature Catalog (STAGE-004-028) and the Saved Queries (STAGE-004-013) to the alert-authoring modal from STAGE-004-043. Both shortcuts pre-fill the modal with sensible per-source defaults.

For signatures: pre-fill threshold using the spike-vs-baseline template from STAGE-004-036.
For saved queries: pre-fill threshold using a simple `count() > N` over the saved query.

This is L8 from brainstorming — "saved query → alert rule" — merged with create-alert-from-signature into a single stage because they use the same underlying flow.

**Vertical slice:** in the Signature drill-in panel, "Create alert for this signature" button opens the modal with the spike-vs-baseline rule template pre-filled (multiplier 5, window 5m, severity warn, name `SignatureSpike_<hash>_<service>`). In the Saved Queries panel, "Create alert from this saved query" opens the modal with `count(<saved query LogsQL>) > 10` pre-filled.

**Spec references:** master spec §9.2 (saved queries + create alert).

## Concrete deliverables

### 1. Signature shortcut

`apps/ui/src/routes/logs/SignatureDetailPanel.tsx` (from STAGE-004-028) — add button:

```
[ Create spike alert ]
```

Click → opens `<CreateAlertModal>` (from STAGE-004-043) with pre-filled:

- **Rule name**: `SignatureSpike_<template_hash>_<service_key_safe>` (auto-generated; user can edit)
- **Severity**: warning
- **For**: `1m`
- **Threshold expression**: rendered from STAGE-004-036's template with default multiplier (5×) and window (5m), per-signature instantiated
- **Summary**: "Spike in {{ template_hash }} from {{ service_key }}"
- **Description**: detailed template with deep-link to catalog
- **Extra labels**: `template_hash`, `service_key`, `anomaly_kind: signature_spike`
- **source_kind**: `signature`
- **source_ref**: `<template_hash>:<service_key>`

User reviews + adjusts + saves.

### 2. Saved-query shortcut

`apps/ui/src/routes/logs/SavedQueriesPanel.tsx` (from STAGE-004-013) — add per-row action "Create alert" or a hover button.

Click → opens `<CreateAlertModal>` with pre-filled:

- **Rule name**: `SavedQuery_<saved_query_name_safe>` (auto-generated; editable)
- **Severity**: warning
- **For**: `5m`
- **Threshold expression**: `count(<saved query's LogsQL>) > 10` over a 5-min window
- **Summary**: "Saved query '{{ saved_query_name }}' matched > 10 lines"
- **Description**: includes the saved query name + LogsQL
- **Extra labels**: `saved_query_id`, `anomaly_kind: saved_query_alert`
- **source_kind**: `saved_query`
- **source_ref**: `<saved_query_id>`

User reviews + adjusts + saves.

### 3. Pre-fill helper

`apps/ui/src/routes/logs/alertPrefill.ts` (new):

```typescript
function prefillFromSignature(sig: Signature, opts?: { multiplier?: number, window?: string }): CreateAlertFormState
function prefillFromSavedQuery(sq: SavedQuery, opts?: { threshold?: number, window?: string }): CreateAlertFormState
```

100% test coverage on pre-fill logic.

### 4. Disambiguation

If the user already has an alert rule for the same source_kind + source_ref, surface this via the modal title bar or a banner ("An alert already exists for this signature; this will create a second one. Use the rules manager to edit the existing rule.").

- Pre-fill helper tests with realistic inputs
- Signature drill-in panel → click button → modal opens with right fields

## Decisions in this stage

### Design phase will confirm

- **Auto-generated rule name policy** — recommended above; user can override
- **Default multiplier / threshold values** — match the rule template defaults (multiplier 5, window 5m)
- **Duplicate-rule policy** — warn + allow (user can intentionally create variants); recommendation: warn only, don't block

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-SHORTCUTS-SHARE-MODAL** — both shortcuts go through the same `<CreateAlertModal>` for consistency.
- **D-PRE-FILL-WITH-DEFAULTS** — modal opens with sensible defaults; user reviews + adjusts + saves.
- **D-SOURCE-TRACKING** — `source_kind` + `source_ref` columns in `log_user_rules` track origin for future linking.
