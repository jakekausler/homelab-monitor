# STAGE-004-030: Drain models dump endpoint + UI viewer for debugging

## Status: Not Started

## Overview

Add a debugging/inspection surface for the raw Drain model state: a backend endpoint that dumps all models (or one specific model) as JSON, plus a small UI viewer page that renders it. Useful when investigating why a particular log line clustered to an unexpected template, when tuning drain3 parameters, or when verifying SQLite persistence is intact.

**Vertical slice:** at `/logs/models-debug` (or a similar admin-y route), a UI page lists all model keys with their template counts; clicking a key shows that model's complete tree (templates, parameters, cluster sizes). Backend endpoint `GET /api/logs/signatures/models/export` returns the JSON dump.

**Spec references:** master spec §6.3 (Drain clustering); not explicit in spec — this is operator UX added per user request 2026-05-28.

## Concrete deliverables

### 1. Models export endpoint

`apps/monitor/homelab_monitor/kernel/api/routers/logs.py`:

```
GET /api/logs/signatures/models/export?service_key=<optional>
```

Response (when `service_key` omitted): `{models: [ModelSummary, ...]}`
Response (when `service_key` provided): `{model: ModelDetail}`

```python
class ModelSummary(BaseModel):
    service_key: str
    template_count: int
    line_count: int  # cumulative
    last_processed_ts: datetime | None
    updated_at: datetime

class ModelDetail(BaseModel):
    service_key: str
    template_count: int
    line_count: int
    last_processed_ts: datetime | None
    updated_at: datetime
    templates: list[TemplateDump]

class TemplateDump(BaseModel):
    template_id: int  # drain3's internal id (informational)
    template_hash: str  # stable cross-restart hash
    template_str: str  # canonical template
    line_count: int  # lines matched
    first_seen_ts: datetime
    last_seen_ts: datetime
    sample_recent_lines: list[str]  # up to 3 recent matching lines from VL
```

Auth: session required. CSRF: not required (GET).

Implementation: read from `drain_models` table + `log_signatures` table + best-effort VL sample lookup.

### 2. Frontend: models debug page

`apps/ui/src/routes/logs/ModelsDebugPage.tsx` (new) — route `/logs/models-debug`:

- Left: list of all models with template_count + last_processed_ts + size badge
- Right: when a model is selected, render its full template tree as a sortable / searchable table

This is an "advanced / admin" surface — not flashy, focused on diagnostics. Render the JSON-ish content in a structured table, not raw JSON.

### 3. Navigation

A small "Models (debug)" link in the Signatures tab footer OR a top-level admin nav entry (Design picks). Recommendation: link inside the Signatures tab, low-visibility, near a "Last cycle stats" mini-panel.

### 4. Last cycle stats mini-panel

In addition to the dump endpoint, the Signatures tab gains a small footer panel showing:

- Last cycle finished at: `<timestamp>`
- Lines processed: `<N>`
- New templates: `<N>`
- Cycle duration: `<X>ms`
- Manual refresh button (calls STAGE-004-027's endpoint)

This is super useful for the user to know "did the catalog just update or am I looking at stale data."

### 5. Tests

- Backend: dump endpoint response shapes, both list + detail modes
- Frontend: models debug page renders list, click selects model

## Out of scope for this stage

- Edit / mutate model state from the UI — read-only diagnostics
- Deletion of a model (force fresh cluster) — could add as an "Advanced" action; YAGNI for v1
- Re-cluster all data from scratch — YAGNI; manual refresh is sufficient
- Diff between two snapshots — speculative

## Decisions in this stage

### Design phase will confirm

- **Route location** — `/logs/models-debug` recommended (sub-route under `/logs`)
- **Sample lines per template** — 3 recommended
- **Table virtualization** — only needed if templates can exceed ~1000; YAGNI initially

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-DEBUG-DUMP-ENDPOINT** — JSON dump for inspection; separate from operator-facing catalog UI.
- **D-DEBUG-UI-VIEWER** — UI viewer renders the dump; matches user request "API endpoint as well as a corresponding UI viewer".
- **D-CYCLE-STATS-VISIBLE** — Signatures tab shows last cycle status + manual refresh.
