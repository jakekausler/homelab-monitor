# STAGE-004-030: Drain models dump endpoint + UI viewer for debugging

## Status: Complete

## Overview

Add a debugging/inspection surface for the raw Drain model state: a backend endpoint that dumps all models (or one specific model) as JSON, plus a small UI viewer page that renders it. Useful when investigating why a particular log line clustered to an unexpected template, when tuning drain3 parameters, or when verifying SQLite persistence is intact.

**Vertical slice:** at `/logs/models-debug` (or a similar admin-y route), a UI page lists all model keys with their template counts; clicking a key shows that model's complete tree (templates, parameters, cluster sizes). Backend endpoint `GET /api/logs/signatures/models/export` returns the JSON dump.

**Spec references:** master spec §6.3 (Drain clustering); not explicit in spec — this is operator UX added per user request 2026-05-28.

## Concrete deliverables

### 1. Models export endpoint

`apps/monitor/homelab_monitor/kernel/api/routers/logs.py`:

```
GET /api/logs/signatures/models/export?service_key=<optional>
```

Response (when `service_key` omitted): `{models: [ModelSummary, ...]}`
Response (when `service_key` provided): `{model: ModelDetail}`

```python
class ModelSummary(BaseModel):
    service_key: str
    template_count: int
    line_count: int  # cumulative
    last_processed_ts: datetime | None
    updated_at: datetime

class ModelDetail(BaseModel):
    service_key: str
    template_count: int
    line_count: int
    last_processed_ts: datetime | None
    updated_at: datetime
    templates: list[TemplateDump]

class TemplateDump(BaseModel):
    template_id: int  # drain3's internal id (informational)
    template_hash: str  # stable cross-restart hash
    template_str: str  # canonical template
    line_count: int  # lines matched
    first_seen_ts: datetime
    last_seen_ts: datetime
    sample_recent_lines: list[str]  # up to 3 recent matching lines from VL
```

Auth: session required. CSRF: not required (GET).

Implementation: read from `drain_models` table + `log_signatures` table + best-effort VL sample lookup.

### 2. Frontend: models debug page

`apps/ui/src/routes/logs/ModelsDebugPage.tsx` (new) — route `/logs/models-debug`:

- Left: list of all models with template_count + last_processed_ts + size badge
- Right: when a model is selected, render its full template tree as a sortable / searchable table

This is an "advanced / admin" surface — not flashy, focused on diagnostics. Render the JSON-ish content in a structured table, not raw JSON.

### 3. Navigation

A small "Models (debug)" link in the Signatures tab footer OR a top-level admin nav entry (Design picks). Recommendation: link inside the Signatures tab, low-visibility, near a "Last cycle stats" mini-panel.

### 4. Last cycle stats mini-panel

In addition to the dump endpoint, the Signatures tab gains a small footer panel showing:

- Last cycle finished at: `<timestamp>`
- Lines processed: `<N>`
- New templates: `<N>`
- Cycle duration: `<X>ms`
- Manual refresh button (calls STAGE-004-027's endpoint)

This is super useful for the user to know "did the catalog just update or am I looking at stale data."

### 5. Tests

- Backend: dump endpoint response shapes, both list + detail modes
- Frontend: models debug page renders list, click selects model

## Out of scope for this stage

- Edit / mutate model state from the UI — read-only diagnostics
- Deletion of a model (force fresh cluster) — could add as an "Advanced" action; YAGNI for v1
- Re-cluster all data from scratch — YAGNI; manual refresh is sufficient
- Diff between two snapshots — speculative

## Design Phase

### Design Decisions (chosen 2026-06-06)

**D1 — Models-dump data source: HYBRID (list from drain_models columns; detail via the live engine).**
- LIST view sourced from the `drain_models` table columns directly (model_key, line_count, template_count, last_processed_ts, updated_at) — cheap (no blob deserialization), and COMPLETE (every persisted model, works on a fresh process with no cycle run yet).
- DETAIL view (one model's full template list) sourced via `engine.get_model(key)` then `engine.templates(key)` — reuses the SINGLE existing template-extraction code path (which already has corrupt-snapshot fallback at drain_engine.py:194-198), avoiding a second drain3-deserialize implementation.
- Rejected: (a) live-engine-only `_models` (incomplete — empty on fresh start; only this-process models); (c) log_signatures catalog (cycle-gated/partial, lacks template_id+cluster size, and is the OPERATOR catalog this debug view is explicitly SEPARATE from).
- New accessors required: `SqlitePersistence.list_all_summaries()` (+ add to the `DrainPersistence` Protocol); `DrainConsumer.get_engine() -> DrainEngine` (the engine lives inside the consumer, NOT on app.state — expose via the consumer, do NOT add a parallel app.state.drain_engine).

**D2 — Last-cycle stats source: `last_result` singleton on DrainConsumer.**
- Add `self._last_result: DrainCycleResult | None`, assigned after EVERY `run_once()` (covers BOTH periodic background cycles AND manual /refresh — both go through run_once). Expose `consumer.last_result` property + a new `GET /logs/signatures/cycle/last` endpoint. Footer panel shows finished_at, lines_processed, new_templates, duration (= finished_at - started_at).
- Rejected: reusing CycleStatusStore with a well-known key (TTL-pruned → panel goes blank after 1h; periodic path doesn't write to it); deriving from drain_models aggregate (cannot reconstruct per-cycle deltas: lines_processed/new_templates/duration).
- Note: in-memory only → null until the first cycle completes after a restart (FE shows "No cycle completed yet").

**D3 — Endpoint shape: TWO routes (not one overloaded query-param route).**
- `GET /logs/signatures/models` → ModelListResponse (array of ModelSummary{model_key, template_count, line_count, last_processed_ts, updated_at}).
- `GET /logs/signatures/models/{model_key}` → ModelDetailResponse{model_key, summary, templates: TemplateEntry[] with template_id, template_hash, template_str, size, first_seen_ts, last_seen_ts}.
- `GET /logs/signatures/cycle/last` → last-cycle stats (per D2).
- CRITICAL route-ordering: register the literal `models` + `cycle` routes BEFORE the existing `/logs/signatures/{template_hash}/{service_key}` param route, else `/models/foo` is swallowed as template_hash="models",service_key="foo" (FastAPI matches in registration order).
- Rejected: one route `.../models/export?service_key=<optional>` returning a union (list|detail) — union response fights pyright --strict + openapi-typescript FE codegen; two clean routes match the existing list_signatures/get_signature split.
- All three endpoints 503 cleanly via the existing `get_drain_consumer` dependency when drain is disabled (HOMELAB_MONITOR_DRAIN_ENABLED off → no app.state.drain_consumer).

**D4 — Frontend: master-detail page, two queries.**
- New `apps/ui/src/api/models.ts` mirroring `apps/ui/src/api/signatures.ts` (a `modelKeys` query-key factory; `useModelsList()`; `useModelDetail(modelKey, enabled)` firing only on selection via `enabled`; `useLastCycle()` for the footer; staleTime 30_000, retry:false; ApiError from ./client).
- `ModelsDebugPage.tsx` at `/logs/models-debug`: desktop = sidebar model-list + right-panel template table; mobile = model-key dropdown/select + stacked cards (reuse SignaturesTab's table↔card breakpoint; meet the no-horizontal-scroll-on-mobile bar — debug page, lower mobile polish but still no h-scroll). Client-side sort (template_id, size, first_seen_ts) + client-side search over template_str (pure client over the fetched detail array — no new endpoints).
- Footer "Last cycle stats" mini-panel on the Signatures tab (NOT only the debug page) per D-CYCLE-STATS-VISIBLE: finished_at (relative), lines_processed, new_templates, duration, + manual refresh button (reuses existing POST /logs/signatures/refresh + poll flow; on poll completion also refetch useLastCycle).
- Nav: "Models (debug)" link in the Signatures tab area.

### Risks to carry into Build
1. Drain-disabled (HOMELAB_MONITOR_DRAIN_ENABLED off): all 3 endpoints 503 via get_drain_consumer; FE must render a friendly "drain disabled" empty state on 503, NOT a React error (footer panel + nav link tolerate 503).
2. Corrupt snapshot: get_model() degrades a corrupt blob to an empty fresh model → detail returns templates:[] (not 500). The LIST column template_count may then disagree with the DETAIL live count — surface BOTH (stored count from the column + live count from extraction) so a mismatch is visible (a debug tool should reveal corruption).
3. get_model() warms the engine `_models` cache + shares `_load_lock` with live ingestion — benign (brief lock; consumer would load the same model anyway); add a one-line comment so a future reader knows the debug detail path mutates engine cache.
4. model_key contains a colon for cron keys ("cron:<hex>") — FE must encodeURIComponent the key; backend must not split on ":" (proven safe via existing {service_key} routes).
5. last_result resets to null on process restart until the first cycle completes — FE shows "No cycle completed yet", do NOT persist to DB.
