# STAGE-004-033: Healthcheck-failure log enrichment (60s log window attached to healthcheck unhealthy alerts)

## Status: Complete

## Overview

When a container's healthcheck transitions to `unhealthy`, fetch the surrounding 60s log window (before + after) from VL and attach the result as enrichment to the resulting alert. Parallel pattern to the crash correlation in STAGE-004-032, but triggered by healthcheck state changes rather than container exits.

Built on top of the shared `LogWindowFetcher` from STAGE-004-031.

**Vertical slice:** when a container healthcheck transitions unhealthy, the user sees the surrounding log window in both Karma's notification AND inline on the alert detail page.

**Spec references:** master spec §6.3; EPIC-004 plan #5.

## Design Decisions (selected 2026-06-07)

### Selected approach (parallels STAGE-004-032 crash enrichment; deltas noted)

- **H1 — Transition detection + anchor:** COLLECTOR-side. Extend docker_socket_collector's existing prior-row SELECT (the one that already fetches prior_container_id for recreated_at) to also fetch `prior_healthcheck`. Add TWO new columns to `targets_docker`: `healthcheck_changed_at TEXT NULL` (ISO UTC) and `previous_healthcheck TEXT NULL`. On a transition INTO unhealthy, the upsert stamps `healthcheck_changed_at = now` + `previous_healthcheck = prior` (using the same `CASE WHEN ... ELSE <existing col> END` preserve idiom already used for recreated_at, so non-transition ticks preserve the prior stamp). The new `ContainerHealthcheckReconciler` (mirrors ContainerCrashReconciler) reads `list_docker_containers` rows where `healthcheck == "unhealthy"`, uses `healthcheck_changed_at` as BOTH the log-window anchor AND the dedup key. Restart-surviving (SQLite, not in-memory), deterministic dedup (no per-tick alert storm — an unhealthy container stays unhealthy across many ticks but the stamp is fixed).
  - Rejected (b) reconciler-side in-memory diff: does NOT survive restart (alert storm on every restart); breaks the stateless-per-tick model.
  - Rejected (c) previous_healthcheck column only + reconciler stamps now(): non-deterministic dedup key → re-inserts + re-alerts every 30s (the 032 lesson).

- **H2 — Transition predicate + dedup:** edge = `prior != "unhealthy" AND new == "unhealthy"` (covers BOTH healthy→unhealthy AND starting→unhealthy, i.e. boot/start-period failures; first-sight-already-unhealthy with NULL prior also stamps once). Dedup UNIQUE = `(logical_key, healthcheck_changed_at)`. Re-arm: unhealthy→healthy→unhealthy stamps a fresh healthcheck_changed_at → new episode → new row + new alert. One enrichment per EPISODE, not per tick.

- **H3 — Window split:** 30s before / 30s after (symmetric), limit 100, retention 7d. `HOMELAB_MONITOR_HEALTHCHECK_LOG_WINDOW_S` (default 60 total) drives `window_before_s` (the env-configurable half); `window_after_s = total - before` (mirrors CrashLogConfig where window_before_s is the configurable one). Caveat documented: the anchor is poll-stamped (up to ~30s late); the symmetric window absorbs this because the lag pushes the true failure instant into the "before" half. LogsQL is the SAME as 032: `container_name:"<name>" AND source_type:docker` (the container's normal stdout/stderr — NOTE the healthcheck PROBE output, State.Health.Log[].Output, is NOT shipped to VL, so the window is the container's logs around the transition, not the probe output).

- **H4 — Metric:** COUNTER `homelab_container_healthcheck_unhealthy_total{container_name, compose_project, compose_service}` incremented 1.0 once per NEW episode (inside the inserted==True branch, like 032's homelab_container_crash_total) + gauge `homelab_container_healthcheck_with_log_context{container_name}` = line count. vmalert rule `deploy/vmalert/metrics/container_healthcheck.yml` alert `ContainerUnhealthy` fires on `increase(homelab_container_healthcheck_unhealthy_total[5m]) > 0`, labels severity:warning, source_tool:vmalert-metrics, category:container, integration:docker; annotation deep-links to /integrations/docker/containers/{{ $labels.container_name }}/overview.

- **H5 — Persistence + API + UI (parallel to 032):**
  - NEW table `container_healthcheck_enrichments` (migration 0037, down_revision "0036"; bump EXPECTED_TABLES 34→35 in tests/test_db_migrations.py). Columns mirror container_crash_enrichments but healthcheck-specific: incident_id TEXT PK, logical_key, container_name, container_id, previous_healthcheck (the from-state), new_state (= "unhealthy"), healthcheck_changed_at (the anchor), image_name, compose_project, compose_service, lines_json, line_count, truncated, degraded, window_start, window_end, created_at. UNIQUE index on (logical_key, healthcheck_changed_at) + index on (container_name, healthcheck_changed_at). The 0037 migration ALSO adds the two targets_docker columns (healthcheck_changed_at, previous_healthcheck) with the same guarded-ALTER pattern; downgrade rebuilds targets_docker to pre-0037 shape (drops the two columns) AND drops the enrichment table (mirror 0036's downgrade table-rebuild — CRITICAL so cross-migration round-trips stay clean).
  - NEW repo `HealthcheckEnrichmentsRepository` (mirrors crash_enrichments_repo: insert INSERT OR IGNORE, list_for_container, get, prune age+per-key-cap).
  - NEW reconciler `ContainerHealthcheckReconciler` registered in lifespan.py (+ no test-lifespan wiring needed; tests build a bare CollectorContext).
  - NEW config `HealthcheckLogConfig` + `load_healthcheck_log_config` (env HOMELAB_MONITOR_HEALTHCHECK_LOG_WINDOW_S, HOMELAB_MONITOR_HEALTHCHECK_ENRICHMENT_RETENTION_DAYS; clamp >=1; window_after_s + max_rows_per_container fixed constants with the defensive clamp).
  - NEW endpoints GET /api/integrations/docker/containers/{name}/healthcheck-incidents[/{id}] + response models ContainerHealthcheckIncidentSummary / ContainerHealthcheckIncidentsResponse / ContainerHealthcheckIncidentDetail in docker.py (name→logical_key resolution, 404, cross-container guard, require_session). openapi regen + generate-types after.
  - UI: NEW `RecentHealthcheckIncidentsSection` in ContainerOverviewTab (mirrors RecentCrashesSection) — each incident shows the `previous_healthcheck → unhealthy` transition string + healthcheck_changed_at + the <LogViewer> window + OpenInExplorer deep-link. NEW hooks useContainerHealthcheckIncidents / useContainerHealthcheckIncidentDetail + dockerHealthcheckQueryKeys in api/docker.ts.

## Concrete deliverables

### 1. Healthcheck transition detector

`apps/monitor/homelab_monitor/kernel/logs/healthcheck_correlator.py` (new):

- Subscribes to docker socket collector's healthcheck state-change events (already emitted per EPIC-003 work)
- For each transition `healthy → unhealthy`:
  - Fetch log window via `LogWindowFetcher` with filter `service:"<container_name>"`
  - Anchor timestamp: the transition time
  - Window: `window_before_s` (default 30s before), `window_after_s` (default 30s after)
  - Limit: 100 lines (configurable)
- Emit metric: `homelab_container_healthcheck_unhealthy{name, image, compose_project, compose_service}`
- Emit metric: `homelab_container_healthcheck_with_log_context{name, lines_count}` so vmalert can know enrichment is available
- Store enrichment in SQLite `container_healthcheck_enrichments` keyed by event ID

### 2. SQLite schema

```sql
CREATE TABLE container_healthcheck_enrichments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    container_name TEXT NOT NULL,
    transitioned_at INTEGER NOT NULL,  -- unix ms; healthy→unhealthy moment
    new_state TEXT NOT NULL DEFAULT 'unhealthy',  -- could later be other states; for now unhealthy
    image_name TEXT,
    compose_project TEXT,
    compose_service TEXT,
    lines_json TEXT NOT NULL,
    line_count INTEGER NOT NULL,
    truncated INTEGER NOT NULL DEFAULT 0,
    created_at INTEGER NOT NULL
);
CREATE INDEX idx_hc_enrich_container ON container_healthcheck_enrichments(container_name, transitioned_at);
```

Retention: 7 days, configurable via `HOMELAB_MONITOR_HEALTHCHECK_ENRICHMENT_RETENTION_DAYS=7`.

### 3. Window size env var

`HOMELAB_MONITOR_HEALTHCHECK_LOG_WINDOW_S=60` (total; split half-before / half-after; Design phase locks the asymmetry — 30/30 by default).

### 4. Enrichment endpoint

```
GET /api/integrations/docker/containers/{name}/healthcheck-incidents?limit=20
GET /api/integrations/docker/containers/{name}/healthcheck-incidents/{id}
```

### 5. vmalert rule

`deploy/vmalert/metrics/container_healthcheck.yml` (new):

```yaml
groups:
  - name: container_healthcheck
    interval: 30s
    rules:
      - alert: ContainerUnhealthy
        expr: increase(homelab_container_healthcheck_unhealthy[5m]) > 0
        for: 0m
        labels:
          severity: warning
          source_tool: homelab-monitor
          category: docker
        annotations:
          summary: "Container {{ $labels.name }} became unhealthy"
          description: |
            Container {{ $labels.name }} (image: {{ $labels.image_name }}) transitioned
            to unhealthy. Healthcheck context: see homelab-monitor UI.
            URL: /integrations/docker/containers/{{ $labels.name }}/overview
```

### 6. UI: render enrichment on ContainerOverviewTab + Alert detail

- `ContainerOverviewTab.tsx` (from EPIC-003) — when a recent healthcheck transition has enrichment, render a collapsible "Last healthcheck incident log context" section
- `AlertDetailPage.tsx` — for `ContainerUnhealthy` alerts, fetch + render enrichment inline

### 7. Tests

- Backend: detector unit tests, SQLite repo, endpoints
- Integration: synthetic healthcheck transition, assert metric + enrichment row + retrievable
- vmalert rule: planted metric fires
- Frontend: ContainerOverviewTab renders incident context; AlertDetailPage renders enrichment
