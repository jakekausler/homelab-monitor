# STAGE-004-035: Anomaly Type A — "New signature detected" rules + first_seen metric refinement

## Status: Complete

## Overview

The first kind of signature anomaly: alert when a NEW signature template (one that has never been seen before, or hasn't been seen in the last X days) suddenly appears in the log stream. First-time appearance often indicates new code paths, new failure modes, new attack patterns.

Categorization: this is the **"NEW"** anomaly kind, surfaced in alert metadata as `category: log-anomaly` + `anomaly_kind: new_signature` so downstream consumers (Karma, Telegram, the homelab-monitor UI, future Claude integration) can render it appropriately.

**Vertical slice:** a previously-unseen template appears in any service. Within 5 min, an alert fires with the template, the originating service, the first sample line, and a deep-link to the signature catalog + Explorer.

**Spec references:** master spec §6.3; EPIC-004 docker req #6 (new error patterns).

## Concrete deliverables

### 1. First-seen metric refinement

`homelab_log_signature_first_seen_ts` was emitted in STAGE-004-027 as a gauge with the timestamp value. Verify and confirm:

- Once-emitted: doesn't update on subsequent appearances of the same template
- Includes `severity` label (so the rule can scope to error/warn templates only)
- Includes `service_key` so per-service rules work

This stage refines if needed.

### 2. vmalert rule

`deploy/vmalert/logs/new_signature.yml` (new):

```yaml
groups:
  - name: new_signature
    interval: 60s
    rules:
      - alert: NewLogSignature
        # Template that has fired at least once in last 5m AND was first-seen within the last 5m
        # Excludes suppressed signatures (status = 'suppressed' in log_signatures)
        expr: |
          (increase(homelab_log_signature_count[5m]) > 0)
          and
          (homelab_log_signature_first_seen_ts >= time() - 300)
          unless on (template_hash, service_key) (homelab_log_signature_suppressed == 1)
        for: 0m
        labels:
          severity: info  # New signatures are informational by default; user can promote via label
          source_tool: homelab-monitor
          category: log-anomaly
          anomaly_kind: new_signature
        annotations:
          summary: "New log signature in {{ $labels.service_key }}"
          description: |
            A previously-unseen template appeared:

            Service: {{ $labels.service_key }}
            Template hash: {{ $labels.template_hash }}
            Severity: {{ $labels.severity }}

            See the signature catalog for the template + sample lines.
            URL: /logs/signatures#{{ $labels.template_hash }}/{{ $labels.service_key }}
```

### 3. Suppression metric

Add `homelab_log_signature_suppressed{template_hash, service_key}` (gauge, value 1 when suppressed, 0 otherwise) — emitted by a small collector that reads `log_signatures.status` from SQLite every minute and produces the metric. Lets vmalert rules exclude suppressed signatures.

### 4. Severity scoping

The rule fires only on signatures with severity in {error, critical, warning} by default. Configurable via vmalert rule label matchers (env-var override in Design phase if needed). Info/debug signatures still emit metrics but don't trigger new-signature alerts unless explicitly enabled.

### 5. Alert payload enrichment

Annotations include:
- A link to the signature catalog entry (anchor link to the specific signature)
- A link to the Logs Explorer with a pre-filled filter to show recent matching lines

### 6. Tests

- vmalert rule test (planted metrics fire the rule; suppressed signature does not fire)
- Suppression metric emission test (signature status change reflected within 1 cycle)
- Karma rendering smoke test (alert appears with `anomaly_kind` label)

## Decisions in this stage

### Design phase will confirm

- **First-seen window** — 5 min recommended (catches up if Drain cycle was delayed)
- **Severity scoping rule** — error/critical/warning by default; info/debug muted
- **Alert severity label** — `info` for "new signature" because it's informational; user can change via signature catalog later

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-ANOMALY-KIND-LABEL** — every signature-anomaly rule carries `anomaly_kind` label so downstream consumers can route/render differently.
- **D-CATEGORIZED-ALERTS** — separate stages = separate rules; alert payload tells the user which kind of anomaly this is.
- **D-USER-CURATED-V1** — no auto-suppression of NewLogSignature; user labels/suppresses via catalog.
- **D-SUPPRESSED-SIGNATURES-NEVER-ALERT** — the suppression flag from the catalog wins.

## Design Decision — chosen approach (locked 2026-06-07)

**CHOSEN: Hybrid-Collector.** A new backend collector (`NewSignatureCollector`, a `BaseCollector`) makes the entire "new signature" determination in Python and emits a single purpose-built gauge `homelab_log_signature_new{service_key, template_hash, severity}` = 1 for signatures that are (first-seen within the window) AND (not suppressed) AND (severity in the configured in-scope set). The vmalert-metrics rule is then a DUMB `homelab_log_signature_new == 1`.

**Build shape:**
1. Migration `0036_*` adds a `first_seen_severity` column to `log_signatures` (persisted on INSERT only, preserved on UPDATE — mirrors `first_seen_at`). This lets the collector decide purely from seeded DB rows (testable, self-contained). FALLBACK if sync lacks severity at INSERT time: derive severity-at-first-seen from the drain consumer's last-cycle `cycle_counts` (keyed `(service_key, template_hash, severity)`) — decide in Build after reading `signature_sync.py`.
2. `signature_sync.py` persists `first_seen_severity` on INSERT.
3. `NewSignatureCollector(BaseCollector)` modeled on `kernel/metrics/local_build_update_collector.py`: reads `log_signatures` each tick, emits `homelab_log_signature_new` via `MemoryRetainingMetricsWriter.replace_family` (so aged-out/suppressed signatures DISAPPEAR from the family — natural self-resolution), plus the mandatory `homelab_collector_run_*` self-metric.
4. Config: `new_signature_window_seconds` (default 300) + `new_signature_severities` (default `{error, critical, warning}`), read via the frozen-dataclass `load_*()` env pattern.
5. ONE dumb vmalert-metrics rule `deploy/vmalert/metrics/log_anomaly.yaml` (NOT a vlogs rule): `expr: homelab_log_signature_new == 1`, `for: 0m`, labels `severity: warning`, `source_tool: vmalert-metrics`, `category: log-anomaly`, `anomaly_kind: new_signature`, `target_kind: log_signature`. Annotations: catalog deep-link `/logs/signatures#{{ $labels.template_hash }}/{{ $labels.service_key }}` + Explorer pre-fill link (pattern from `apps/ui/src/lib/explorerLink.ts` `buildExplorerUrl`).

**Why Hybrid over the card's original proposal:** correctness + 100% kernel coverage + atomic suppression + a dumb rule. The card's originally-proposed PromQL expr had three confirmed bugs (see "Corrected facts" below).

**REJECTED — Metric-PromQL (the card's original proposal):** a smart PromQL rule on existing metrics + a separate `homelab_log_signature_suppressed` gauge. Rejected because: (a) the hard logic lives in YAML that the 100%-kernel-coverage gate does NOT cover; (b) `_count` is sparse (per-cycle, absent when no lines that cycle) → the severity `group_left` join flaps; (c) a signature can carry multiple severities across cycles → `group_left` "multiple matches" ambiguity; (d) a separate suppression gauge re-introduces a two-series RACE that D-SUPPRESSED-SIGNATURES-NEVER-ALERT exists to prevent (status flips between catalog write and next gauge emit can leak one alert). Hybrid's single-decision-per-tick is atomic by construction.

**REJECTED — Logs-LogsQL (`type: vlogs` rule):** RULED OUT. The `template_hash` is computed in-process by the drain engine and is NEVER written back to VictoriaLogs, so a LogsQL rule literally cannot identify which template_hash is "new". Would require feeding drain output back into VL — out of scope.

**Corrected facts (the original card deliverables had STALE/WRONG claims — source-verified 2026-06-07):**
- `homelab_log_signature_first_seen_ts` carries labels `{service_key, template_hash}` ONLY — it does NOT carry `severity` (deliverable #1's "includes severity label" was wrong; severity lives only on `homelab_log_signature_count`).
- `homelab_log_signature_first_seen_ts` value is in NANOSECONDS (`first_seen_ms * 1_000_000`), so the card's proposed `first_seen_ts >= time() - 300` (seconds) NEVER held — confirmed bug, now moot under Hybrid (window compared in Python against `first_seen_at` ms).
- `homelab_log_signature_count` is a GAUGE (per-cycle line count), not a counter — the card's `increase(...count[5m])` was meaningless. Moot under Hybrid.
- The `log_signatures` table has NO severity column today → Build adds `first_seen_severity` (migration 0036).
- `homelab_log_signature_first_seen_ts` IS already once-set/stable across cycles (deliverable #1 "once-emitted" is already satisfied — verify, don't rework).

**Stage type:** BACKEND-ONLY. Alert surfaces in Karma automatically; suppression status is already viewable/editable in the existing Signature Catalog (`/logs/signatures`). No new frontend → no Desktop+Mobile Refinement prompt.

**Stale file refs cut:** `apps/ui/src/routes/alerts/AlertDetailPage.tsx` (does NOT exist — alert UI is the Karma iframe), `deploy/alertmanager/template.tmpl` (does NOT exist — pure webhook to `/api/alerts/ingest`; actual file is `alertmanager.yml.template`).

### Build Notes (2026-06-07)

- **Migration number corrected: 0039, not 0036.** The card/Design referenced `0036`, but the actual alembic head was `0038` (down_revision "0037"). The new migration is `apps/monitor/alembic/versions/0039_log_signature_first_seen_severity.py` (revision "0039", down_revision "0038"). It ADDS a `first_seen_severity TEXT` column to `log_signatures` (column-add → table-REBUILD downgrade, full index recreation, mirrors the 0036 `targets_docker` pattern). EXPECTED_TABLES in `test_db_migrations.py` is UNCHANGED (column-add, not table-add).
- **Severity source resolved:** `signature_sync.py` `sync_cycle` receives `cycle_counts` keyed `(model_key, template_hash, severity)`, so per-line severity IS in scope at INSERT. The collector reads a persisted `first_seen_severity` (highest-severity-wins rank: critical > error > warning > notice > info > debug > unknown; ties broken alphabetically), persisted on INSERT only, preserved on UPDATE (mirrors `first_seen_at`). Collector stays a pure DB reader (no drain-engine coupling) → fully testable from seeded rows.
- **No OpenAPI change:** `Signature` (internal dataclass) is serialized to the API via a separate `SignatureResponse` Pydantic model that does NOT include `first_seen_severity`; `_signature_to_response` does not forward it. So `make openapi-export` is NOT needed for this stage. (Confirmed by grep during Build.)
- **Collector self-resolution:** `NewSignatureCollector` emits the `homelab_log_signature_new{service_key, template_hash, severity}` family via duck-typed `getattr(ctx.vm, "replace_family", None)` (production `ctx.vm` is a `MultiplexMetricsWriter`, not the concrete `MemoryRetainingMetricsWriter` — an isinstance check would silently drop every series). Self-metric: `homelab_collector_run_new_signature{phase, result}`. Collector unit test at 100% coverage (12 tests).
- **Registration:** two-step pattern in `lifespan.py` — `loader.register(NewSignatureCollector, {...})` (class) + a post-registration `isinstance` wiring block that injects `load_new_signature_config()` into the instance's `_config`. The collector returns an error result + `result="dependencies_unwired"` self-metric if `_config` is None (unwired safety).
- **⚠️ FLAG FOR FINALIZE CODE REVIEW (annotation/label clobber):** the vmalert rule `deploy/vmalert/metrics/log_anomaly.yaml` sets `severity: warning` as a RULE label, which OVERRIDES the gauge series' `severity` label (which carries the LINE severity: error/critical/warning) in the firing alert. Therefore the rule's annotations `summary: "... ({{ $labels.severity }})"` and `description: "First-seen severity: {{ $labels.severity }}"` render `warning` (the alert severity), NOT the originating log-line severity as intended. DECISION DEFERRED to Finalize code review: either rename the gauge's line-severity label (e.g. `line_severity`) so the annotation can show the true severity without colliding with the alert's `severity` label, OR accept that the annotation shows the alert severity and reword it. Do NOT silently ship the misleading annotation — resolve in Finalize.
- **vmalert rule fixtures are NOT CI-exercised:** `deploy/vmalert/metrics/__tests__/log_anomaly.tests.yaml` is a forward-compatible static fixture matching the schema of the existing `heartbeats.tests.yaml`, but no `make` target or pytest harness runs `.tests.yaml` fixtures today (same status as all existing rule fixtures). The rule will be LIVE-validated in Refinement (confirm vmalert LOADS it on the prod rig).
