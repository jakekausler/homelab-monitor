# STAGE-004-035: Anomaly Type A — "New signature detected" rules + first_seen metric refinement

## Status: Not Started

## Overview

The first kind of signature anomaly: alert when a NEW signature template (one that has never been seen before, or hasn't been seen in the last X days) suddenly appears in the log stream. First-time appearance often indicates new code paths, new failure modes, new attack patterns.

Categorization: this is the **"NEW"** anomaly kind, surfaced in alert metadata as `category: log-anomaly` + `anomaly_kind: new_signature` so downstream consumers (Karma, Telegram, the homelab-monitor UI, future Claude integration) can render it appropriately.

**Vertical slice:** a previously-unseen template appears in any service. Within 5 min, an alert fires with the template, the originating service, the first sample line, and a deep-link to the signature catalog + Explorer.

**Spec references:** master spec §6.3; EPIC-004 docker req #6 (new error patterns).

## Concrete deliverables

### 1. First-seen metric refinement

`homelab_log_signature_first_seen_ts` was emitted in STAGE-004-027 as a gauge with the timestamp value. Verify and confirm:

- Once-emitted: doesn't update on subsequent appearances of the same template
- Includes `severity` label (so the rule can scope to error/warn templates only)
- Includes `service_key` so per-service rules work

This stage refines if needed.

### 2. vmalert rule

`deploy/vmalert/logs/new_signature.yml` (new):

```yaml
groups:
  - name: new_signature
    interval: 60s
    rules:
      - alert: NewLogSignature
        # Template that has fired at least once in last 5m AND was first-seen within the last 5m
        # Excludes suppressed signatures (status = 'suppressed' in log_signatures)
        expr: |
          (increase(homelab_log_signature_count[5m]) > 0)
          and
          (homelab_log_signature_first_seen_ts >= time() - 300)
          unless on (template_hash, service_key) (homelab_log_signature_suppressed == 1)
        for: 0m
        labels:
          severity: info  # New signatures are informational by default; user can promote via label
          source_tool: homelab-monitor
          category: log-anomaly
          anomaly_kind: new_signature
        annotations:
          summary: "New log signature in {{ $labels.service_key }}"
          description: |
            A previously-unseen template appeared:

            Service: {{ $labels.service_key }}
            Template hash: {{ $labels.template_hash }}
            Severity: {{ $labels.severity }}

            See the signature catalog for the template + sample lines.
            URL: /logs/signatures#{{ $labels.template_hash }}/{{ $labels.service_key }}
```

### 3. Suppression metric

Add `homelab_log_signature_suppressed{template_hash, service_key}` (gauge, value 1 when suppressed, 0 otherwise) — emitted by a small collector that reads `log_signatures.status` from SQLite every minute and produces the metric. Lets vmalert rules exclude suppressed signatures.

### 4. Severity scoping

The rule fires only on signatures with severity in {error, critical, warning} by default. Configurable via vmalert rule label matchers (env-var override in Design phase if needed). Info/debug signatures still emit metrics but don't trigger new-signature alerts unless explicitly enabled.

### 5. Alert payload enrichment

Annotations include:
- A link to the signature catalog entry (anchor link to the specific signature)
- A link to the Logs Explorer with a pre-filled filter to show recent matching lines

### 6. Tests

- vmalert rule test (planted metrics fire the rule; suppressed signature does not fire)
- Suppression metric emission test (signature status change reflected within 1 cycle)
- Karma rendering smoke test (alert appears with `anomaly_kind` label)

## Decisions in this stage

### Design phase will confirm

- **First-seen window** — 5 min recommended (catches up if Drain cycle was delayed)
- **Severity scoping rule** — error/critical/warning by default; info/debug muted
- **Alert severity label** — `info` for "new signature" because it's informational; user can change via signature catalog later

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-ANOMALY-KIND-LABEL** — every signature-anomaly rule carries `anomaly_kind` label so downstream consumers can route/render differently.
- **D-CATEGORIZED-ALERTS** — separate stages = separate rules; alert payload tells the user which kind of anomaly this is.
- **D-USER-CURATED-V1** — no auto-suppression of NewLogSignature; user labels/suppresses via catalog.
- **D-SUPPRESSED-SIGNATURES-NEVER-ALERT** — the suppression flag from the catalog wins.
