# STAGE-004-031A: "Show surrounding logs" in the Explorer

## Status: Complete

## Overview

Add a "Show surrounding logs" affordance to the Logs Explorer so an operator can, from a single
log line, pull the ~100 lines before and after that line's timestamp to reconstruct the context
around an incident. The user picks the scope: **all services** (everything in the time window) or
**only this line's service** (filtered to the same service). This is the third step in the
end-to-end debugging flow the user described:

  1. Choose a model to debug (Models page — STAGE-030)
  2. Open in Explorer (Models page "Open in Explorer" — STAGE-030)
  3. **Show surrounding logs** to track down the issue (THIS stage)

**Vertical slice:** In the Logs Explorer, each log line's inspector (or a per-line action) gains a
"Show surrounding logs" control. Activating it offers a scope choice (All services / Only `<service>`)
and renders a window of the N lines before + N after the selected line's timestamp, using the
`<LogViewer>` renderer (the epic's sole log-line renderer).

## Dependency

**Depends on STAGE-004-031 (`LogWindowFetcher`).** That stage delivers the shared, cached, capped,
gracefully-degrading windowed-fetch backend:

```python
async def fetch(
    self,
    logs_ql: str,            # filter, e.g. 'service:"home-assistant"' or '' for all services
    anchor_ts: datetime,
    window_before_s: int = 60,
    window_after_s: int = 60,
    limit: int = 200,
) -> LogWindowResult  # {lines, truncated, window_start, window_end, queried_at}
```

LogWindowFetcher already enforces: in-process LRU (max 1000, 5-min TTL), total window ≤ 1h, limit ≤ 1000,
and VL transport errors → empty result (no raise). THIS stage reuses it rather than re-implementing
windowed fetch.

> Note: STAGE-031 fetches by *time window in seconds* around an anchor timestamp. "100 lines before/after"
> (a count-based window) may need either (a) a generous time window + client-side trim to N lines each
> side of the anchor, or (b) a small extension to LogWindowFetcher / a new endpoint for count-based
> windows. Design phase must decide and, if an extension is needed, scope it here (or as a small 031
> follow-up) — do NOT silently assume the time-based fetch yields exactly N lines.

## Concrete deliverables (Design will refine)

1. **Backend endpoint** (or reuse) to fetch a surrounding window: given an anchor timestamp + an optional
   service filter + before/after counts (or seconds), return the windowed lines via LogWindowFetcher.
   (Design decides: new endpoint under `/api/logs/...` vs extending an existing Explorer query path.)
2. **Frontend: "Show surrounding logs" control** on a log line in the Explorer (inspector panel or a
   per-line action), with the **All services / Only `<service>`** scope choice.
3. **Frontend rendering** of the returned window via `<LogViewer>`, visually anchored on the selected
   line (e.g. highlight the anchor; "N before / N after").
4. **Truncation/empty handling**: surface `truncated` and gracefully handle VL-down/empty (friendly state,
   no React error) — inheriting LogWindowFetcher's degrade-gracefully contract.
5. **Tests**: backend window endpoint (mocked + rig); FE control + scope toggle + render + empty/truncated states.

## Design Decisions (locked 2026-06-06)

### D1 — Window semantics: COUNT-based, two-sided fetch (CHOSEN: Option 1C)

The window is COUNT-based ("~100 lines before AND 100 after"), NOT raw time-based. Implemented by calling STAGE-031's LogWindowFetcher TWICE per request — once for the before-side, once for the after-side — each with `limit=N` (default N=100). This structurally defeats the lopsided-density failure mode (a burst before the anchor cannot starve the after-side, because each side has its own independent limit). The locked LogWindowFetcher is REUSED UNMODIFIED — only called twice from the endpoint; the split/merge logic lives in the new endpoint (server-side), keeping the frontend rendering-only.

- before-side fetch: `fetch(logs_ql, anchor, window_before_s=1800, window_after_s=0, limit=N)` → nearest N before the anchor.
- after-side fetch: `fetch(logs_ql, anchor, window_before_s=0, window_after_s=1800, limit=N)` → nearest N after the anchor.
- Endpoint merges, dedupes the anchor line, sorts by timestamp, returns `lines` + per-side `truncated_before`/`truncated_after` + `degraded` (OR of both sides) + an `anchor_index` (position of the anchor line in the returned `lines`).
- Adaptive window-expansion (refetch with a wider window if a side under-fills) is DEFERRED as YAGNI — fixed 1800s generous windows are sufficient for incident reconstruction; the per-side `truncated` flags surface "more exists". Revisit only if real data shows chronic under-fill.

REJECTED:
- 1A (time-window-only, label by time): silently abandons the "~100 each side" requirement; lopsided results.
- 1B (one generous time-window fetch + trim N each side server-side): reuses the fetcher with one call, but CANNOT GUARANTEE 100-after when density is high before the anchor — the single global `limit`/`truncated` can't express per-side exhaustion.

### D2 — Backend endpoint: new GET /api/logs/window + LogWindowFetcher singleton on app.state (CHOSEN: Option 2C)

A NEW endpoint `GET /api/logs/window` (anchor-centered, dual-truncation shape), keeping the stable cursor-paginated `/api/logs/query` untouched. Params: `anchor_ts` (ISO), optional `service` + `source_type` (for scope), optional base `expr` (default `*`), `before`/`after` (counts, default 100), `window_s` (per-side generous window seconds, default 1800). Scope ("all" vs "only this service") maps to LogsQL by reusing the existing `_compose_services_expr(expr, f"{source_type}:{service}")` helper — omit the service clause for "all services".

LogWindowFetcher is constructed ONCE at lifespan startup (next to the section-7g VL wiring, reusing the shared `http_client` + a `VictoriaLogsClient`) and stored on `app.state.log_window_fetcher`, exposed via a `get_log_window_fetcher` DI dependency. This is a deliberate departure from the per-request `VictoriaLogsClient` construction used by other logs routers: a per-request fetcher would throw away the TTL+LRU cache that is the locked module's entire reason for existing.

Response includes an explicit `anchor_index` (integer position of the anchor line in the returned `lines[]`) for unambiguous FE highlighting (sidesteps timestamp-collision ambiguity from `${timestamp}-${index}` row keys).

REJECTED:
- 2A (new endpoint, per-request fetcher): silently defeats the fetcher cache — wrong given the module's design.
- 2B (extend /api/logs/query with an anchor mode): overloads a cursor-paginated infinite endpoint with a fundamentally different anchor-centered/dual-truncation shape; conflicting response models; regression risk on the Explorer's main query path; uses a different code path (paginate_older) that doesn't reuse the fetcher naturally.

### D3 — Frontend UX: trigger in FieldInspectorPanel, render 2nd <LogViewer> in modal/Sheet (CHOSEN: Option 3A)

The "Show surrounding logs" control + scope toggle live in the EXISTING FieldInspectorPanel for the selected line (reuses LogsExplorerBody's `selection {key, line}` state — no new per-row UI, no list-row clutter). Activating it opens a modal (desktop) / right Sheet (mobile) hosting a SECOND `<LogViewer>` (the epic's sole renderer — locked) driven by a new `useSurroundingLogs(anchor, scope)` react-query hook. The anchor row is highlighted via `selectedKey` (= the returned `anchor_index`'s row key). Scope toggle is a two-segment "All services / Only `<service>`" control defaulting to "Only this service" (tighter, more relevant for incident reconstruction); toggling re-fetches via a scope-inclusive query key. truncated_before/after → existing truncated banner; degraded → existing unavailable banner; empty → existing no_lines empty state (all inherited from the useLogs()/<LogViewer> contract). On open, scroll the modal viewer to center the anchor row.

REJECTED:
- 3B (per-line hover button + inline expansion below the row): fights LogLineList's flat `${timestamp}-${index}` keying + the scroll-persistence/windowed-buffer machinery in LogsExplorerBody (high integration risk against existing STAGE-015/024 scroll code); adds row-level UI to the shared renderer.
- 3C (replace the main list temporarily / in-place context mode): destroys/restores main query state (scroll, pagination buffer in useWindowedLogs); error-prone mode-switching of a single viewer between two data sources; operator loses their place.

## Refinement Design Change (2026-06-06 — D3 revised: 3A → 3C in-place mode)

During Refinement (prod-rig manual test), the user rejected the modal/Sheet UX (Option 3A) and the empty-window behavior, and locked a different design:

**D3-REVISED — In-place surrounding-logs MODE in the Explorer (supersedes the original 3A "modal/Sheet" decision; this is the previously-rejected Option 3C, now chosen by user directive).**

Requirements (user-locked):
1. "Show surrounding logs" (triggered from the Field Inspector on a selected line, as before) switches the MAIN Logs Explorer view INTO a "surrounding-logs mode" — it does NOT open a separate modal/Sheet. The main list is replaced in-place by the surrounding-logs window.
2. The window shows ~100 lines before + the selected line + ~100 after, with the SELECTED (anchor) line highlighted.
3. A button to EXIT surrounding-logs mode and return to the normal query view (restoring the prior query/range/scroll state).
4. While in surrounding-logs mode, the ONLY available actions are:
   a. Load PREVIOUS (older) — infinite scroll upward: fetch the next batch of older lines before the current earliest line, ACCUMULATING (keep what's shown), keeping the selected line highlighted.
   b. Load NEXT (newer) — infinite scroll downward: fetch the next batch of newer lines after the current latest line, ACCUMULATING, keeping the selected line highlighted.
   c. Open a log line (Field Inspector) — which may in turn trigger "Show surrounding logs" on a DIFFERENT line, RE-ANCHORING the mode to that new line.
   (Normal-query affordances — search box, time range, service filter, tail — are disabled/hidden while in surrounding-logs mode.)
5. Bidirectional infinite scroll ACCUMULATES (does not re-center); the anchor/selected line stays highlighted as the window grows in both directions.

SUPERSEDES: the original D3 (Option 3A — second `<LogViewer>` in a Dialog/Sheet) and the SurroundingLogsModal component built in Build. The in-place mode reuses the Explorer's existing list rendering + scroll/pagination infrastructure instead.

Also note Build-output BUG found in Refinement: the modal showed ONLY the selected line, not surrounding lines (empty window against real data) — to be diagnosed + fixed as part of the rework (the in-place mode uses the same windowed fetch, so the root cause must be resolved).

Backend impact: the existing GET /api/logs/window (two-sided LogWindowFetcher fetch) is largely reused, but bidirectional "load more / accumulate" requires fetching additional batches relative to the current earliest/latest shown line (anchor stays fixed for highlight; pagination cursors move). Design/Build of the rework will determine whether /api/logs/window needs added params (e.g. fetch older-than / newer-than a cursor timestamp) or whether repeated calls with shifted anchors + client-side accumulation suffice.

## Refinement Outcome (2026-06-07)

User-tested on the prod rig (Desktop + Mobile). Approved both layouts after the following Refinement-driven fixes (all FE-only except the first; no openapi/type changes):

1. **D3 revised 3A→3C in-place mode** (already recorded above): replaced the modal/Sheet with an in-place surrounding-logs mode (enter from Field Inspector, exit button, bidirectional accumulating load-older/newer, re-anchor).
2. **Backend empty-window fix** (`LogWindowFetcher.fetch`): VictoriaLogs returns lines newest-first and `stream_query` truncated to the first `limit`; for a one-sided window the anchor-adjacent lines were dropped. Fixed by oversampling + sorting ascending + keeping the N lines NEAREST the anchor (side-aware), preserving the symmetric path for existing tests. +2 unit tests, +1 integration regression.
3. **Scope source_type fix**: the FE sent `source_type=unknown` for only-this-service scope when the line lacked a usable source_type → `service:"x" AND source_type:"unknown"` matched nothing. Fixed to only scope when a real source_type AND a raw `fields.service` exist; otherwise fall back to all-services.
4. **Default scope → All services**: only-this-service was the default, but `line.service` is a derived field (promoted from `SYSLOG_IDENTIFIER`) that often has no raw VL `service` field, so the scoped query collapsed the window to the anchor. Default is now All services (the user can still narrow via the toggle).
5. **Surrounding window ignores the Explorer's active filter**: `useSurroundingLogs` + the incremental load-older/newer fetches now use `expr='*'` (a `SURROUNDING_EXPR` constant) instead of inheriting the committed query. Opening the Explorer from a model left a narrow `expr` active that collapsed the window; surrounding logs must show full context regardless of the active filter.
6. **Template "Open in Explorer" no-results fix** (ModelsDebugPage + SignatureDetailPage): the old translation AND-chained every inter-wildcard template segment; a trailing segment carrying a non-printable ANSI-reset byte (`\x1b[0m`) matched nothing, zeroing the conjunction. Fixed by anchoring on the single LONGEST literal run (`templateToLogsQl` helper in logsQlTranslate.ts). +5 unit tests.
7. **Anchor highlight in all-services** (`surroundingSelectedKey`): the FE re-found the anchor by an exact (timestamp,stream,message) triple, but `stream`/`_time`-precision diverge between the normal query and the window query, so the highlight failed in all-services. Fixed to use the backend-authoritative `anchor_index` (captured into `surrAnchorKey` state at seed time, tracked by composite key across load-more), building the row key from the windowed line's own timestamp. +1 regression test.
8. **Full-width highlight** (`LogLineList`): added `min-w-max` to the log row so the selected/hover background + ring span the entire line, not just the viewport width, when a line overflows the horizontally-scrollable container. (Also fixes normal-mode selection — shared renderer.)

All fixes verified end-to-end against the live prod rig (real VictoriaLogs) via reproduced API calls + the user's manual Desktop+Mobile test. `make verify` re-run after the final fix wave.
