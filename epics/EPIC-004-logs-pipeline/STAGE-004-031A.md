# STAGE-004-031A: "Show surrounding logs" in the Explorer

## Status: Not Started

## Overview

Add a "Show surrounding logs" affordance to the Logs Explorer so an operator can, from a single
log line, pull the ~100 lines before and after that line's timestamp to reconstruct the context
around an incident. The user picks the scope: **all services** (everything in the time window) or
**only this line's service** (filtered to the same service). This is the third step in the
end-to-end debugging flow the user described:

  1. Choose a model to debug (Models page — STAGE-030)
  2. Open in Explorer (Models page "Open in Explorer" — STAGE-030)
  3. **Show surrounding logs** to track down the issue (THIS stage)

**Vertical slice:** In the Logs Explorer, each log line's inspector (or a per-line action) gains a
"Show surrounding logs" control. Activating it offers a scope choice (All services / Only `<service>`)
and renders a window of the N lines before + N after the selected line's timestamp, using the
`<LogViewer>` renderer (the epic's sole log-line renderer).

- Has Input Forms: [ ] No (scope toggle only, no free-text form)

## Dependency

**Depends on STAGE-004-031 (`LogWindowFetcher`).** That stage delivers the shared, cached, capped,
gracefully-degrading windowed-fetch backend:

```python
async def fetch(
    self,
    logs_ql: str,            # filter, e.g. 'service:"home-assistant"' or '' for all services
    anchor_ts: datetime,
    window_before_s: int = 60,
    window_after_s: int = 60,
    limit: int = 200,
) -> LogWindowResult  # {lines, truncated, window_start, window_end, queried_at}
```

LogWindowFetcher already enforces: in-process LRU (max 1000, 5-min TTL), total window ≤ 1h, limit ≤ 1000,
and VL transport errors → empty result (no raise). THIS stage reuses it rather than re-implementing
windowed fetch.

> Note: STAGE-031 fetches by *time window in seconds* around an anchor timestamp. "100 lines before/after"
> (a count-based window) may need either (a) a generous time window + client-side trim to N lines each
> side of the anchor, or (b) a small extension to LogWindowFetcher / a new endpoint for count-based
> windows. Design phase must decide and, if an extension is needed, scope it here (or as a small 031
> follow-up) — do NOT silently assume the time-based fetch yields exactly N lines.

## Cross-cutting standards

- `<LogViewer>` / `LogLineList` is the ONLY log-line renderer (epic invariant) — render the surrounding
  window with it.
- No pre-redaction raw content leaves un-redacted (STAGE-006 planted-credential assertion must hold).
- 100% backend kernel coverage; pyright --strict; TS strict (exactOptionalPropertyTypes;
  no react-hooks/set-state-in-effect).
- The scope toggle is a discrete control; the "anchor line" is identified by its timestamp (and, for
  the single-service scope, its `service`).

## Concrete deliverables (Design will refine)

1. **Backend endpoint** (or reuse) to fetch a surrounding window: given an anchor timestamp + an optional
   service filter + before/after counts (or seconds), return the windowed lines via LogWindowFetcher.
   (Design decides: new endpoint under `/api/logs/...` vs extending an existing Explorer query path.)
2. **Frontend: "Show surrounding logs" control** on a log line in the Explorer (inspector panel or a
   per-line action), with the **All services / Only `<service>`** scope choice.
3. **Frontend rendering** of the returned window via `<LogViewer>`, visually anchored on the selected
   line (e.g. highlight the anchor; "N before / N after").
4. **Truncation/empty handling**: surface `truncated` and gracefully handle VL-down/empty (friendly state,
   no React error) — inheriting LogWindowFetcher's degrade-gracefully contract.
5. **Tests**: backend window endpoint (mocked + rig); FE control + scope toggle + render + empty/truncated states.

## Out of scope for this stage

- The Models page "Open in Explorer" jump (delivered in STAGE-030) — this stage assumes the user is
  already in the Explorer on a line of interest.
- Saved/persisted "incident context" snapshots.
- Cross-window diffing or correlation across multiple anchors.
- Any change to LogWindowFetcher's caching/limit semantics beyond a possible count-based-window option
  (decided in Design).

## Decisions in this stage

### Design phase will confirm
- Count-based ("100 lines each side") vs time-based window, and whether LogWindowFetcher needs a small
  extension or a client-side trim suffices.
- Where the control lives (per-line inspector vs a line action menu) — match the Explorer's existing
  line-interaction pattern.
- The exact scope-toggle UX (All services vs Only `<service>`), defaulting to the safer/most-useful default.
- Endpoint shape vs reusing the existing Explorer query mechanism.

### Locked Design Decisions (inherited from this Refinement conversation)
- D-SURROUNDING-LOGS-SCOPE-CHOICE — the feature MUST offer both "all services" and "only this service".
- D-SURROUNDING-LOGS-REUSE-FETCHER — reuse STAGE-031's LogWindowFetcher; do not duplicate windowed-fetch.
- D-SURROUNDING-LOGS-LOGVIEWER — render the window with `<LogViewer>` (epic's sole renderer).
