# STAGE-004-021: "Open in Explorer" deep-link from Docker + Cron viewers

## Status: Complete

## Overview

Add a small "Open in Explorer" button to the Docker container log viewer + the Cron run log viewer. Clicking it navigates the user to `/logs?logsql=<pre-filled>&start=<...>&end=<...>` with filters pre-populated from the source viewer's context.

For the docker viewer: pre-fills `service:"<container_name>"` (or `compose_service:"<svc>"` if a richer match makes sense; Design picks).
For the cron viewer: pre-fills `cron_fingerprint:"<fp>"` AND `run_id:"<run_id>"` to scope to the exact run.

**Vertical slice:** open the docker container logs viewer for `home-assistant` → click "Open in Explorer" → land in `/logs` with the query pre-filled to `service:"home-assistant"` and the same time range. User can refine from there.

## Decisions in this stage

### Design phase will confirm

- **Docker filter shape** — `service:"<name>"` vs `container_name:"<name>"` (after STAGE-004-004 enrichment lands, both work; pick the canonical one)

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-OPEN-IN-EXPLORER** — every per-context log viewer gets a button to jump to the Explorer with filters pre-filled.

- **D-FUTURE-PAGES-INHERIT** — helper documented so HA / Pi-hole / Synology / etc. detail pages can add the button cheaply when they land.

## Design Phase

**Design phase complete:** 2026-06-04

**Selected Approach: Option A — pure `buildExplorerUrl(): string` helper + shared `<OpenInExplorerButton>` using TanStack `<Link to={url}>`**

A free-standing pure function `buildExplorerUrl(opts): string` (in `apps/ui/src/lib/explorerLink.ts`) constructs a `/logs?...` URL string using `URLSearchParams` and the EXACT search-param keys the Explorer route reads. A shared `<OpenInExplorerButton>` component calls the helper and renders `<Button asChild><Link to={url}>` (SPA navigation — no full page reload). Both viewers drop the button into their existing header actions `<div className="flex items-center gap-2">`. Chosen because it satisfies the explicit `buildExplorerUrl(...): string` deliverable, is trivially 100%-coverable as a pure function, and best serves D-FUTURE-PAGES-INHERIT (a portable helper + a drop-in button that future detail pages — HA/Pi-hole/Synology/Unifi in EPIC-005/006/007/008 — reuse in one line).

**Verified Explorer URL contract (frontend-contract verification — the param names MUST match):**
- The `/logs` route (TanStack Router `validateSearch`, router.tsx) reads: `logsql` (advanced LogsQL) OR `q` (plain) — mutually exclusive; `since` (preset token ∈ `5m|15m|1h|6h|24h|7d`); `start` + `end` (ISO-8601 Z custom range); `services` (CSV `source_type:service,...`). `writeUrl()` in LogsExplorerPage.tsx writes these EXACT keys. The card's assumed `logsql` param is CONFIRMED correct. Mode is inferred: `logsql` present → advanced; `q`/neither → plain.
- Canonical LogsQL filter fields (verified vs vector config + backend): Docker container → `service:"<containerName>"` (vector `.service = .container_name`; docker.py:582 builds `service:<quoted>`). Cron run → `run_id:"<runId>"` + `cron_fingerprint:"<fp>"` (vector `.run_id = .HM_RUN`, `.cron_fingerprint = .HM_FP`).
- No backend changes — navigates to the EXISTING `/logs` route.

**Resolved design questions:**
- **Q1 (string vs router object):** STRING helper (the explicit deliverable; trivially 100%-coverable). Mitigations: type the helper's `sincePreset` param against the route's preset union (`5m|15m|1h|6h|24h|7d`); add a guard unit test asserting the emitted param keys are a subset of the route's accepted `validateSearch` keys (fails if the route contract drifts).
- **Q2 Docker time range — faithful inheritance:** carry the viewer's current range (preset → `sincePreset`; custom → `rangeStart`/`rangeEnd`) so the Explorer opens on the same window.
- **Q2 Cron time range — precise rule:** (1) if the user narrowed via `search.start`/`search.end` (both present) → use those as custom `start`/`end`; (2) else if `runMin` AND `runMax` defined → custom `start`/`end` = `runMin`..`runMax` padded ±1s (avoid clipping boundary lines); (3) else (loading/empty) → `1h` preset. Running run (no `runMax`) → `start = runMin − 1s`, NO `end` (open-ended → Explorer applies default upper bound = now); if `runMin` also undefined → `1h` preset.
- **Q3 Cron filter composition:** emit `cron_fingerprint:"<fp>" AND run_id:"<runId>"` (run_id alone suffices but the conjunction honors locked D-PRE-FILLED-FILTERS and reads clearly).
- **Q4 button:** shared `<OpenInExplorerButton>` (prevents icon/label/aria drift across the 4 anticipated future pages); lucide `SquareArrowOutUpRight` icon; label "Open in Explorer"; `<Button asChild><Link to={url}>` for SPA navigation (NOT `<a href>` which would full-reload).

**Deliverables (confirmed):**
1. `apps/ui/src/lib/explorerLink.ts` — pure `buildExplorerUrl(opts: {logsQl?, plainText?, selectedServices?, sincePreset?, rangeStart?, rangeEnd?}): string` (100% coverage; `explorerLink.test.ts`).
2. Shared `apps/ui/src/components/logs/OpenInExplorerButton.tsx` (or co-located) — drop-in button.
3. Docker viewer (`DockerContainerLogsViewerBody.tsx`) — button with `logsQl: service:"<containerName>"` + faithful range.
4. Cron viewer (`CronRunLogViewer.tsx`) — button with `logsQl: cron_fingerprint:"<fp>" AND run_id:"<runId>"` + run-window range per the rule above.
5. `apps/ui/src/components/logs/README.md` — "Deep-linking to the Explorer" section documenting the helper + button for future detail pages (D-FUTURE-PAGES-INHERIT).
6. Tests: `explorerLink.test.ts` (URL composition, all branches, key-subset guard) + viewer tests asserting the button/link is present with the correct `to`/href.

**Rejected alternatives:**
- **Option B (typed router search-object helper `buildExplorerSearch(): LogsSearch` + `<Link to="/logs" search={obj}>`):** Maximum compile-time type-safety, BUT contradicts the explicit `buildExplorerUrl(...): string` deliverable, couples future pages to the router's route/type graph, and the "100% helper coverage" guarantee is weaker (the router owns serialization, so unit tests cover less of the user-visible URL). Rejected: deliverable says string; type-safety recovered cheaply via Option A's mitigations.
- **Option C (string helper + inline `<Link>` per viewer, no shared button component):** Slightly less code, BUT D-FUTURE-PAGES-INHERIT becomes "copy the JSX snippet" → icon/label/aria drift across the 4 anticipated future detail pages. Rejected: a shared button prevents drift for negligible extra code.

**Out of scope (confirmed):** "Open in Explorer" on alert detail pages (per-epic work), "back to viewer" link from Explorer (YAGNI), bidirectional state sync (deep-link only).

**Forward-compat:** Establishes `buildExplorerUrl` + `<OpenInExplorerButton>` as the reusable primitives for EPIC-005/006/007/008 detail pages (D-FUTURE-PAGES-INHERIT); documented in the logs README. No forward dependency WITHIN EPIC-004 (022/023/024 don't use it).

**Refinement:** Frontend-only → 3a, Desktop + Mobile (manual test of the button on both viewers + that clicking lands in the Explorer with the right pre-filled query + range). The helper is a pure function (100%-unit-tested) and the nav is DOM-state, so Refinement is largely a formality, but the actual cross-navigation + pre-fill must be eyeballed on the prod rig.
