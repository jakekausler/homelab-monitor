# STAGE-004-042: Rule persistence model (SQLite schema + render-on-boot to `deploy/vmalert/logs/`)

## Status: Not Started

## Overview

Foundation for the user-authored alert flow. User-authored vmalert rules are persisted in SQLite; on monitor boot (and on any rule create/update/delete via the API), the rules are rendered into YAML files under `deploy/vmalert/logs/user-rules/` so vmalert picks them up.

This stage delivers the persistence + render mechanism, NOT the create-alert-from-query UX (STAGE-004-043) or the saved-query / signature shortcuts (STAGE-004-044).

**Vertical slice:** API can persist a rule via SQLite; render-on-boot produces a valid `*.yml` file under `deploy/vmalert/logs/user-rules/`; vmalert reloads and picks up the rule. Round-trip works across monitor restarts.

**Spec references:** master spec §3.2 (vmalert rules under `deploy/vmalert/logs/`); EPIC-004 plan #7.

## Concrete deliverables

### 1. SQLite schema

`apps/monitor/homelab_monitor/kernel/db/migrations/` — new alembic migration:

```sql
CREATE TABLE log_user_rules (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    rule_name TEXT NOT NULL UNIQUE,  -- vmalert rule name (unique across all user rules)
    rule_group TEXT NOT NULL DEFAULT 'user_rules',  -- the vmalert group name
    expr TEXT NOT NULL,  -- LogsQL or MetricsQL expression
    expr_kind TEXT NOT NULL,  -- 'logsql' | 'metricsql' — determines which vmalert instance owns the rule
    for_duration TEXT NOT NULL DEFAULT '0m',
    severity TEXT NOT NULL DEFAULT 'warning',
    summary TEXT,  -- annotation template
    description TEXT,  -- annotation template
    extra_labels TEXT,  -- JSON object of additional labels
    extra_annotations TEXT,  -- JSON object of additional annotations
    source_kind TEXT NOT NULL,  -- 'query' | 'signature' | 'saved_query' — how the user authored it
    source_ref TEXT,  -- e.g., saved_query_id or signature template_hash + service_key
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL,
    enabled INTEGER NOT NULL DEFAULT 1  -- 0/1; disabled rules aren't rendered
);
CREATE INDEX idx_log_user_rules_kind ON log_user_rules(expr_kind);
CREATE INDEX idx_log_user_rules_source ON log_user_rules(source_kind, source_ref);
```

### 2. Repo layer

`apps/monitor/homelab_monitor/kernel/logs/user_rules_repo.py` (new):

- CRUD operations with name-uniqueness enforcement
- `list_enabled(expr_kind)` for the renderer
- 100% test coverage

### 3. Render-on-boot

`apps/monitor/homelab_monitor/kernel/logs/user_rules_renderer.py` (new):

- On boot AND on rule CRUD: read all enabled rules, group by `expr_kind`, render YAML files:
  - `deploy/vmalert/logs/user-rules/logsql_rules.yml` (for logsql expressions)
  - `deploy/vmalert/metrics/user-rules/metricsql_rules.yml` (for metricsql expressions)
- File naming: deterministic per group; one file per group
- Validation: invalid YAML (Jinja errors, missing fields) is caught BEFORE writing; failures logged + raised; existing files untouched on failure (atomic rename pattern)
- Trigger vmalert reload (HTTP API or signal — Design picks)

### 4. API surface (just the CRUD; UI is next stage)

```
GET /api/logs/user-rules
POST /api/logs/user-rules
PATCH /api/logs/user-rules/{id}
DELETE /api/logs/user-rules/{id}
POST /api/logs/user-rules/{id}/enable
POST /api/logs/user-rules/{id}/disable
```

CSRF on mutations.

### 5. Render validation

Before persisting, validate that:
- `rule_name` matches `^[a-zA-Z_][a-zA-Z0-9_]*$` (vmalert constraint)
- `expr` is non-empty
- `expr_kind` in {`logsql`, `metricsql`}
- Render produces parseable YAML
- vmalert dry-run validates the rule (optional — Design phase decides)

### 6. Tests

- Repo CRUD tests
- Render tests with realistic rules (logsql + metricsql)
- Validation rejection tests (bad name, bad expr)
- Round-trip test: persist → render → re-read file → matches semantics
- Disable test: disabled rule is not rendered

### 7. Tooling: pre-commit + CI verification

A new `make` target `verify-rules` that re-renders all user rules and confirms vmalert (in test rig) accepts the result. Catches regressions in the renderer.

## Decisions in this stage

### Design phase will confirm

- **vmalert reload trigger** — HTTP reload API vs SIGHUP — depends on vmalert deployment specifics
- **Validation depth** — name + expr non-empty + YAML render at minimum; full vmalert dry-run optional
- **File naming** — one file per group recommended
- **Whether to allow user-set `rule_group`** — recommend yes for organization; default `user_rules`

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-RULES-SQLITE-RENDERED** — persist in SQLite, render on boot + CRUD. NOT direct git commits.
- **D-RULE-KIND-SPLIT** — logsql vs metricsql kind determines which vmalert instance owns the rule.
- **D-VALIDATION-BEFORE-PERSIST** — invalid rules can't be saved.
- **D-DISABLED-RULES-NOT-RENDERED** — temporary disable doesn't require deletion.
