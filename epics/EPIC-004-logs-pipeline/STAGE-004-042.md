# STAGE-004-042: Rule persistence model (SQLite schema + render-on-boot to `deploy/vmalert/logs/`)

## Status: Complete

## Overview

Foundation for the user-authored alert flow. User-authored vmalert rules are persisted in SQLite; on monitor boot (and on any rule create/update/delete via the API), the rules are rendered into YAML files under `deploy/vmalert/logs/user-rules/` so vmalert picks them up.

This stage delivers the persistence + render mechanism, NOT the create-alert-from-query UX (STAGE-004-043) or the saved-query / signature shortcuts (STAGE-004-044).

**Vertical slice:** API can persist a rule via SQLite; render-on-boot produces a valid `*.yml` file under `deploy/vmalert/logs/user-rules/`; vmalert reloads and picks up the rule. Round-trip works across monitor restarts.

**Spec references:** master spec §3.2 (vmalert rules under `deploy/vmalert/logs/`); EPIC-004 plan #7.

## Concrete deliverables

### 1. SQLite schema

`apps/monitor/homelab_monitor/kernel/db/migrations/` — new alembic migration:

```sql
CREATE TABLE log_user_rules (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    rule_name TEXT NOT NULL UNIQUE,  -- vmalert rule name (unique across all user rules)
    rule_group TEXT NOT NULL DEFAULT 'user_rules',  -- the vmalert group name
    expr TEXT NOT NULL,  -- LogsQL or MetricsQL expression
    expr_kind TEXT NOT NULL,  -- 'logsql' | 'metricsql' — determines which vmalert instance owns the rule
    for_duration TEXT NOT NULL DEFAULT '0m',
    severity TEXT NOT NULL DEFAULT 'warning',
    summary TEXT,  -- annotation template
    description TEXT,  -- annotation template
    extra_labels TEXT,  -- JSON object of additional labels
    extra_annotations TEXT,  -- JSON object of additional annotations
    source_kind TEXT NOT NULL,  -- 'query' | 'signature' | 'saved_query' — how the user authored it
    source_ref TEXT,  -- e.g., saved_query_id or signature template_hash + service_key
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL,
    enabled INTEGER NOT NULL DEFAULT 1  -- 0/1; disabled rules aren't rendered
);
CREATE INDEX idx_log_user_rules_kind ON log_user_rules(expr_kind);
CREATE INDEX idx_log_user_rules_source ON log_user_rules(source_kind, source_ref);
```

### 3. Render-on-boot

`apps/monitor/homelab_monitor/kernel/logs/user_rules_renderer.py` (new):

- On boot AND on rule CRUD: read all enabled rules, group by `expr_kind`, render YAML files:
  - `deploy/vmalert/logs/user-rules/logsql_rules.yml` (for logsql expressions)
  - `deploy/vmalert/metrics/user-rules/metricsql_rules.yml` (for metricsql expressions)
- File naming: deterministic per group; one file per group
- Validation: invalid YAML (Jinja errors, missing fields) is caught BEFORE writing; failures logged + raised; existing files untouched on failure (atomic rename pattern)
- Trigger vmalert reload (HTTP API or signal — Design picks)

### 5. Render validation

Before persisting, validate that:
- `rule_name` matches `^[a-zA-Z_][a-zA-Z0-9_]*$` (vmalert constraint)
- `expr` is non-empty
- `expr_kind` in {`logsql`, `metricsql`}
- Render produces parseable YAML
- vmalert dry-run validates the rule (optional — Design phase decides)

### 6. Tests

- Validation rejection tests (bad name, bad expr)
- Round-trip test: persist → render → re-read file → matches semantics
- Disable test: disabled rule is not rendered

### 7. Tooling: pre-commit + CI verification

A new `make` target `verify-rules` that re-renders all user rules and confirms vmalert (in test rig) accepts the result. Catches regressions in the renderer.

### Design phase will confirm

- **vmalert reload trigger** — HTTP reload API vs SIGHUP — depends on vmalert deployment specifics

- **File naming** — one file per group recommended
- **Whether to allow user-set `rule_group`** — recommend yes for organization; default `user_rules`

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-RULES-SQLITE-RENDERED** — persist in SQLite, render on boot + CRUD. NOT direct git commits.
- **D-RULE-KIND-SPLIT** — logsql vs metricsql kind determines which vmalert instance owns the rule.
- **D-VALIDATION-BEFORE-PERSIST** — invalid rules can't be saved.
- **D-DISABLED-RULES-NOT-RENDERED** — temporary disable doesn't require deletion.

## Design Phase

**Design phase complete: 2026-06-08.**

### Design Notes

**Design summary:** STAGE-004-042 persists user-authored vmalert rules in SQLite (`log_user_rules`) and renders them to aggregate YAML files on a new shared writable volume, picked up by vmalert via its 30s config poll. Backend-only vertical slice; UI is STAGE-043/044.

**Decisions table:**

| ID | Decision | Choice | Justification |
|----|----------|--------|---------------|
| D1 | Scope: user-rules only vs also wire deferred per-service overrides | User-rules vertical slice ONLY | Success criteria + Out-of-Scope are exclusively `log_user_rules`. The `ErrorRateOverride`/`SeverityFloor` overrides template EXISTING operator rules — a distinct render path with no shared code. Their "reserved for STAGE-042" comments in config.py get re-pointed to a successor stage. |
| D2 | Rendered file layout & vmalert glob | One aggregate YAML file per `expr_kind` (`logs.yaml`, `metrics.yaml`) on a sibling-path mount; add `-rule=/etc/vmalert/{logs,metrics}-user/*.yaml` as a SECOND glob flag | Full-rewrite of exactly two files makes orphan cleanup (delete/disable/rename) structurally impossible to get wrong → satisfies criteria 3 & 6 with least moving parts. Per-rule isolation is low value because D-VALIDATION-BEFORE-PERSIST guarantees every persisted rule renders. |
| D3 | Render-on-CRUD reload semantics | Rely on the 30s `configCheckInterval` poll (no explicit reload) | vmalert already polls — this is the vector precedent (render, no reload), NOT alertmanager. No `/-/reload` endpoint is wired for vmalert. ≤30s activation delay is acceptable for single-user alert authoring; surface "active within ~30s" in the API response. |
| D4 | Render invocation architecture | Single `user_rules_render.py` with `render_all(repo, logs_path, metrics_path) -> None`, full reconcile, called identically on boot (new lifespan block alongside 8b/8c) AND after every CRUD/enable/disable mutation. Inner pure `render_yaml(rules) -> str` for filesystem-free validation. | Pure-function-of-state → idempotent, trivial round-trip test, automatic orphan cleanup. Pure inner renderer satisfies criterion 5 ("YAML render succeeds before persist") without touching the filesystem. Reuses the `kernel/cron/render.py` atomic-write pattern (mkstemp→os.replace, chmod 0o640, group-own). |
| D-INFRA | Compose volume gap (BLOCKER verified) | Add named volumes `data_vmalert_user_logs` and `data_vmalert_user_metrics`, mounted rw on the monitor, ro on the respective vmalert sidecar; add the second `-rule=` glob to BOTH `deploy/compose/docker-compose.yml` and `deploy/compose/docker-compose.test.yml`. | VERIFIED: the monitor container currently has NO mount into the vmalert rule dirs (vmalert uses `../vmalert/{logs,metrics}:ro` host binds). Mirrors the alertmanager/vector shared-named-volume precedent (monitor rw + sidecar ro). Operator-authored static rules stay on their existing `:ro` host bind — never made writable. |

**Cross-cutting guards (carry into Build):**
- Rendered files MUST be `.yaml`, NOT `.yml` (vmalert globs `*.yaml`; assert the extension in the renderer).
- Render user rules into a dedicated group (`user-rules-logs` / `user-rules-metrics`) AND set a `source_tool: user` label, to avoid alert-name collisions with operator rules and keep Alertmanager/Karma routing distinguishable.
- Render + validate BOTH YAML strings fully in memory, THEN do the two `os.replace()` calls back-to-back, minimizing the partial-state window.
- The second `-rule=` flag must land in the TEST compose too, or integration tests won't see user-rules.
- DB: new migration `0041_log_user_rules`; bump `EXPECTED_TABLES` count from 41 to 42 and add `"log_user_rules"`; add `test_migration_0041_round_trip`.

**Inherited locked decisions (unchanged):** D-RULES-SQLITE-RENDERED, D-RULE-KIND-SPLIT, D-VALIDATION-BEFORE-PERSIST, D-DISABLED-RULES-NOT-RENDERED.

**Design-phase confirmations (from the card's "Design Phase Will Confirm" list):**
- vmalert reload trigger: 30s config poll (no SIGHUP/HTTP reload).
- Validation depth: name regex + non-empty expr + pure YAML-render success (no full vmalert dry-run in v1).
- File naming: one aggregate file per expr_kind (see D2).
- User-settable `rule_group`: deferred — v1 uses fixed group names `user-rules-logs`/`user-rules-metrics` for collision safety (was a "recommend yes" but collision isolation wins for v1; note this as a v1 simplification).

### Refinement Notes

**Refinement type:** Backend host-integration stage — 3a+3b validated on the full prod rig (`docker compose -f deploy/compose/docker-compose.yml up -d --build`). No user prompt (backend-only). Validated via e2e-tester against REAL containers + REAL vmalert.

**Two real bugs caught on the rig that 100%-coverage unit tests could not (mock-vs-real):**

- **Volume permission denied.** The new `data_vmalert_user_logs`/`data_vmalert_user_metrics` named volumes defaulted to `root:root 0755`; the monitor (uid 995) could not write them, so render silently failed (logged + swallowed by design). FIX: extended `config-init` (docker-compose.yml) AND `shared-init` (docker-compose.test.yml) to `chown 0:2000 && chmod 3777` the two new volumes, mirroring the existing vector/alertmanager config-volume prep; also added the dirs to the monitor Dockerfile's mkdir/chown/chmod block (defense-in-depth for fresh volumes). Verified: volumes became `drwxrwsrwt 0 2000` and boot render logged `user_rules.render.success`.

- **Missing `type: vlogs` group directive.** The rendered logsql group defaulted to `type: prometheus`, so vmalert-logs queried VictoriaLogs's unsupported `/api/v1/query` endpoint → `health=err` for all user logsql rules. FIX: renderer now emits a group-level `type: vlogs` line for the logsql group ONLY (metricsql correctly omits `type:`, defaulting to prometheus), matching the built-in logs rules. Added unit-test assertions (logs group has `type==vlogs`; metrics group has no `type` key). Verified: valid logsql rule loads `health: ok`, `datasourceType: vlogs`.

**One rig observation dispositioned as OUT OF SCOPE (no code change):** vmalert rejects bare-filter logsql exprs (e.g. `_msg:error`) at parse time — valid vmalert vlogs ALERTING exprs require a `| stats ...` pipe (confirmed: all 3 built-in logs rules use one). The card's locked validation depth is "name regex + non-empty expr + pure YAML-render success (no full vmalert dry-run in v1)" — validating that the expr is a loadable vmalert query is explicitly deferred beyond 042. A valid logsql alerting expr loads end-to-end with health=ok. This was a bad e2e test INPUT, not a feature defect.

**Final e2e result:** 7/7 scenarios pass with valid inputs — auth, logsql create→render(type:vlogs)→shared-volume→vmalert load health=ok, metricsql create→render→vmalert load health=ok, disable omits from render, delete full-reconcile (no orphans), restart round-trip (SQLite persistence + boot render writes successfully), validation (400 bad name / 409 duplicate). `make verify` GREEN throughout (100% kernel coverage maintained after both fixes). Rig torn down (volumes preserved).
