# STAGE-004-017: Generic nested-field extraction at ingest (full nested JSONL → flat fields)

## Status: Complete

## Overview

Today's vector `parse_json` transform parses a JSON envelope and exposes the top-level keys but does not flatten nested objects. A log line like:

```json
{"level": "error", "context": {"user_id": 42, "request": {"path": "/api/foo", "latency_ms": 1234}}, "message": "..."}
```

…lands in VL with `.context` as a stringified-or-opaque sub-object, NOT with `context.user_id`, `context.request.path`, etc. as queryable top-level fields. This blocks expressive LogsQL queries on nested data.

This stage adds full nested-path flattening at ingest time. For every JSON line, vector emits all leaves as dotted-path fields:

- `level: "error"`
- `context.user_id: 42`
- `context.request.path: "/api/foo"`
- `context.request.latency_ms: 1234`
- `message: "..."`

LogsQL queries can then filter on `context.request.latency_ms:>1000` and similar. This is similar to what New Relic does with its log ingestion.

Filter-scope-aware field discovery (showing only the fields present in the current query scope, not the full schema) is the NEXT stage (STAGE-004-018).

**Vertical slice:** any service emitting JSONL logs gets its nested fields flat-extracted automatically at ingest. Query `context.request.latency_ms:>1000` works. Field Inspector (STAGE-004-016) shows the nested paths as flat field rows.

**Spec references:** master spec §6.3 (VL streams + pattern extraction).

## Concrete deliverables

### 1. Enhanced JSON parser transform

`deploy/vector/vector.toml.template` — replace or extend the existing `parse_json` transform with a `[transforms.parse_json_nested]` block that:

1. Detects JSON in `.message` (heuristic: starts with `{` or `[`)
2. If JSON, parses it and FLATTENS nested objects into dotted-path top-level fields
3. If not JSON, passes through unchanged

VRL pseudocode:

```toml
[transforms.parse_json_nested]
type = "remap"
inputs = ["docker_enrich"]  # after enrichment
source = '''
parsed, err = parse_json(.message)
if err == null && is_object(parsed) {
  # Flatten the parsed object into dotted-path fields on the root event
  for_each(parsed) -> |key, value| {
    flat_key = "json." + key  # prefix to avoid collisions with core fields
    .[flat_key] = value  # vector should handle the nested object collapse natively; if not, recurse
  }
  # OR use vector's `flatten` function if it exists
}
'''
```

Vector's native `flatten` function (if available) handles nested objects with a configurable separator. Design phase confirms exact VRL syntax and whether `flatten` exists in the pinned vector version.

### 2. Naming policy: prefix vs no-prefix

To avoid collision with existing core fields (`severity`, `service`, `host`, etc.):

- **Option A**: prefix extracted fields with `json.` so `json.level`, `json.context.user_id`, etc.
- **Option B**: no prefix; if collision detected, log a warning and skip extraction for that key

Recommendation: **Option A (prefix `json.`)** — explicit, no surprises, easy to query.

This means LogsQL queries look like `json.context.request.latency_ms:>1000`, NOT `context.request.latency_ms:>1000`. Slightly verbose but predictable.

### 3. Depth + size limits

Avoid unbounded recursion / memory explosion:

- Max depth: 8 (config via env var `HOMELAB_MONITOR_LOG_JSON_MAX_DEPTH=8`)
- Max number of leaf fields per line: 100 (config via env var `HOMELAB_MONITOR_LOG_JSON_MAX_FIELDS=100`)
- Lines exceeding these limits are processed up to the limit; remainder ignored with a marker field `json._truncated:true`

### 4. Render-on-boot test coverage

`apps/monitor/tests/test_render_vector_toml.py` extension — assert the new transform renders correctly with default limits + custom env-var overrides.

### 5. Integration tests

`apps/monitor/tests/integration/test_vector_nested_json_extraction.py` — plant a JSONL line with known nested structure, query VL, assert:

- Flat-path fields are queryable (`json.context.user_id:42` matches)
- Non-JSON lines pass through unchanged
- Depth-cap engages on deeply-nested input
- Field-count cap engages on wide input
- Collision avoidance: a JSON line with `{"severity":"error"}` doesn't overwrite the vector-set `severity` field

### 6. Field Inspector update

`<FieldInspectorPanel>` from STAGE-004-016 — automatically renders the `json.*` fields as ordinary flat rows. No tree view needed (deferred; see Out of scope).

## Design Phase

**Design phase complete:** 2026-06-03

### Selected Approach: Option A — Native VRL `flatten()` + single-pass post-flatten key filtering

A new `[transforms.json_flatten]` remap transform (type = "remap"), `inputs = ["docker_enrich"]`, inserted between `docker_enrich` and `docker_severity_extract` (repoint `docker_severity_extract.inputs = ["json_flatten"]`). It:

- Re-parses `.message` as JSON (or reuses the already-merged object), builds a `candidate` object EXCLUDING core fields (host, service, severity, timestamp, message, source_type, and any other promoted core keys).
- Calls native `flatten(candidate, separator: ".")` → single-level dotted-path map (arbitrary depth, `.` separator confirmed default in Vector 0.41.1 / VRL 0.18.0).
- Single `for_each` pass over the flattened map applies: the `json.` prefix (final keys `json.context.request.path`), the DEPTH cap (drop keys whose dot-segment count exceeds MAX_DEPTH), and the FIELDS cap (running counter; stop adding past MAX_FIELDS). When the field cap trips, set `.json._truncated = true`.
- Core-field safety is STRUCTURAL: core keys are filtered out of `candidate` before flatten, and the mandatory `json.` prefix guarantees no collision with core fields.

Caps are configurable via env vars `HOMELAB_MONITOR_LOG_JSON_MAX_DEPTH` (default 8) and `HOMELAB_MONITOR_LOG_JSON_MAX_FIELDS` (default 100), threaded through `render.py`'s existing `str.replace()` substitution: add fields to `VectorRenderContext`, read env in `render_on_boot()` (defaults 8/100), add `${...}` placeholders to the substitutions dict, embed as integer literals in the VRL body.

### Rejected Alternatives

**Option B (pre-trim then flatten):** Rejected — requires Python to GENERATE hand-unrolled N-level `for_each` VRL (str.replace can't generate nested loops). Categorically more complex than any existing render code; carries code-gen burden without B's independence from `flatten()`.

**Option C (pure Python-generated unrolled VRL, no `flatten()`):** Rejected as primary, KEPT as documented fallback ONLY if `flatten()` were unavailable. Since the Vector 0.41.1 check CONFIRMED `flatten()` exists with `.` default separator, the fallback is not triggered. Most code, highest risk of a nested-closure `vector validate` surprise at prod-rig (3b).

### Key Verified Facts (Backend-Contract / Vector-version verification)

- Vector pinned at `timberio/vector:0.41.1-debian` (VRL 0.18.0).
- VRL `flatten(value, [separator])` exists; default separator `.`; NO `except` param (that's VRL 0.32.0+); NO depth cap; NO field cap.
- VRL has NO user-defined recursion / while loops → a TRUE per-level depth cap is infeasible in pure VRL. Depth cap is therefore APPROXIMATE: enforced post-flatten by counting `.` separators in each flattened key and dropping keys deeper than MAX_DEPTH. This is acceptable for a single-user homelab tailing its own container logs.
- **Array handling caveat (note for Build):** native `flatten()` does NOT emit numeric-indexed keys like `a.0`/`a.1`; it collapses nested ARRAYS into a single flat array and leaves an array value under its parent key (`{"a":{"b":[1,2]}}` → `{"a.b":[1,2]}`). So array elements are NOT individually queryable as scalar fields. This is acceptable for v1 (out of scope to index array elements). Build/tests should assert object-nesting flattening, not array-index flattening.
- No downstream changes needed: `victorialogs_client.py` (~line 211) already stringifies all non-builtin fields into the `fields` bag, so `json.*` dotted fields auto-appear and are queryable via verbatim LogsQL.
