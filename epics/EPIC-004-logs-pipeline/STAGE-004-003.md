# STAGE-004-003: `<LogViewer>` component extraction + cron/docker viewers refactor + embedding contract documented

## Status: Not Started

## Overview

Extract a shared `<LogViewer>` component from the two existing per-context log viewers (`DockerContainerLogsViewerBody`, `CronRunLogViewerPage`) and refactor both to use it. The component takes a caller-provided React Query hook (or fetcher) so it can render logs from any source: docker per-container, cron per-run, generic LogsQL, or future syslog sources (Synology, UDM, HA, Pi-hole).

This stage explicitly documents the **embedding contract** — `<LogViewer>` must support drop-in use on future detail pages (probe detail, alert detail, target detail) with a caller-provided filter + time range. Per master spec §9.2 ("inventory detail pages include related logs (LogsQL pre-filtered)").

The Logs Explorer page (STAGE-004-010) will consume the same component.

**Vertical slice:** docker container log viewer + cron run log viewer both render through `<LogViewer>`. No user-visible change. The component is now reusable for the Explorer + future detail-page embedding.

**Spec references:** Master spec §9.2 (logs screen + related logs on detail pages).

## Concrete deliverables

### 1. `<LogViewer>` component

`apps/ui/src/components/logs/LogViewer.tsx` (new) — shared component with these props:

```typescript
interface LogViewerProps {
  // Data hook the caller controls — returns lines + status + error + loading
  useLogs: () => {
    lines: LogLine[] | undefined
    isLoading: boolean
    isError: boolean
    error: ApiError | undefined
    logStatus?: 'available' | 'no_lines' | 'unavailable' | 'unknown' | 'expired' | 'running'
    truncated?: boolean
  }
  // Optional slot above the line list (e.g., since-picker + refresh button for docker viewer)
  headerSlot?: ReactNode
  // Optional slot for empty-state copy
  emptyStateCopy?: string
  // Optional slot for unavailable-state copy
  unavailableCopy?: string
  // Render timestamps in local time by default (UTC toggle added in STAGE-004-009)
  timezone?: 'local' | 'utc'
}
```

- Renders the line list with monospace formatting matching the existing docker viewer
- Handles all log_status states: `available`, `no_lines`, `unavailable` (503), `unknown` (404), `expired`, `running`
- Renders truncated banner when applicable
- Renders error states from `ApiError`
- Renders empty state with caller-supplied copy
- NO pagination yet (STAGE-004-007), NO custom range yet (STAGE-004-008), NO UTC toggle yet (STAGE-004-009), NO search yet
- Mobile-responsive: line list scrolls horizontally on narrow viewports OR wraps (Design decides)

## Decisions in this stage

### Design phase will confirm

- **Mobile line rendering:** horizontal scroll vs wrap. Recommendation: horizontal scroll with sticky timestamp + service columns (matches existing docker viewer behavior).
- **Line color coding by severity:** which severities get which colors (or none). Recommendation: subtle text-color tinting (`text-red-500` for error/critical, `text-yellow-500` for warn) with NO background colors so the line list stays readable in bulk.
- **`useLogs` hook shape:** literal shape locked above; Design phase confirms or adjusts based on first consumer needs.

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-LOGVIEWER-COMPONENT** — single shared `<LogViewer>` used by all log-display surfaces in the app.
- **D-CALLER-PROVIDES-HOOK** — `<LogViewer>` is data-source agnostic; caller supplies `useLogs()`. This is the embedding contract for future detail pages.
- **D-EMBEDDING-DOCUMENTED** — explicit deliverable: written contract + example in `apps/ui/src/components/logs/README.md` so future epics (HA / Pi-hole / Unifi / Synology detail pages) drop in `<LogViewer>` without rebuilding infrastructure.
- **D-NO-BREAKING-CHANGES-TO-EXISTING-VIEWERS** — docker + cron viewers render identically after refactor. Tests assert this.
