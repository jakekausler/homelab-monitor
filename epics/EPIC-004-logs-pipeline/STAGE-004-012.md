# STAGE-004-012: Stream picker sidebar (distinct service values with line counts; click-to-filter)

## Status: Complete

## Overview

Add a left sidebar to the Logs Explorer that lists every distinct `service` value present in the current time window, with line counts. Clicking a service injects `service:"<name>"` (AND'd with existing filters) into the query. Provides discoverability for users who don't know LogsQL.

This is the biggest "wow" moment for first-time Explorer users: they see the menu of available log sources without having to remember any field names.

**Vertical slice:** `/logs` page now has a left sidebar showing "home-assistant (1,234), pi-hole (892), homelab-monitor (445), docker.service (102), sshd.service (12), ..." for the current time window. Click "home-assistant" → query updates to include `service:"home-assistant"`. Results refresh.

**Spec references:** master spec §9.2 (stream picker).

## Concrete deliverables

### 1. Backend endpoint: distinct services in window

`apps/monitor/homelab_monitor/kernel/api/routers/logs.py` — new endpoint:

```
GET /api/logs/services?start=<ISO>&end=<ISO>&limit=100
```

Response:

```python
class LogsServicesResponse(BaseModel):
    services: list[ServiceCount]  # sorted desc by count
    truncated: bool  # true if more services exist beyond limit

class ServiceCount(BaseModel):
    service: str
    count: int  # line count in window
```

Implementation: uses VL's `/select/logsql/stats` (or equivalent) with `stats by (service) count()` LogsQL. Caches result per (start, end, limit) for 30s to avoid hammering VL.

### 4. Recommended policy (Design confirms)

**Hybrid approach:**
- Stream-picker selection is a **separate filter state** (`selectedServices: string[]`) NOT directly part of the LogsQL string
- Backend combines `selectedServices` AND `expr` into the actual VL query: `(service:"a" OR service:"b") AND (<expr>)`
- UI shows selected services as chips above the search box: `[home-assistant ×] [pi-hole ×]`
- Click a service in sidebar → adds to `selectedServices`; click chip × → removes
- URL state: `&services=home-assistant,pi-hole`
- In LogsQL advanced mode, the user CAN write their own `service:` filter in the expr; the sidebar's `selectedServices` AND's with whatever they write

This is cleaner than mutating the LogsQL string and works in both plain-text and advanced modes uniformly.

## Decisions in this stage

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-SIDEBAR-LEFT** — left sidebar on the Explorer; lists distinct services with counts.
- **D-CLICK-INJECTS-FILTER** — clicking a service adds it to the query (via separate state per recommended policy).
- **D-SIDEBAR-FOR-DISCOVERABILITY** — biggest UX win for users without LogsQL knowledge.

### Design phase decisions (locked 2026-06-01)

**D-012-A1 — Backend composes the services filter (selected over frontend-composition A2).**
- The main log query AND's selected services server-side. Add a new `services` query param to `GET /api/logs/query` (CSV string, e.g. `services=home-assistant,pi-hole`; omit/empty = no service filter).
- Backend builds the effective LogsQL as `(service:"a" OR service:"b") AND (<expr>)` using the canonical `logsql_quote_phrase` quoter for each service value. The user's `expr` is passed through verbatim — never parsed/mutated.
- Rationale: the card's locked text mandates backend composition; CLAUDE.md mandates ALL LogsQL-embedded values go through the single canonical `logsql_quote_phrase` (no duplicate TS quoter — honors reuse-over-fork and avoids a second injection-sensitive quoter). OpenAPI regen is already forced this stage by the new services endpoint, so the `/api/logs/query` contract change adds marginal cost.

**D-012-B1 — New dedicated services-list endpoint (selected over extending /api/logs/streams).**
- `GET /api/logs/services?start=<ISO>&end=<ISO>&limit=100` → `LogsServicesResponse{ services: list[ServiceCount] sorted desc by count, truncated: bool }`; `ServiceCount{ service: str, count: int }`.
- Implemented via VL `stats by (service) count()` over the window. 30s server-side cache keyed on `(start, end, limit)`. `truncated=true` when more than `limit` distinct services exist.
- Add a forward-compat code comment: STAGE-004-018's future `/api/logs/fields` will generalize distinct-value+count discovery and may absorb/replace this endpoint; do not couple new callers to this path beyond the stream picker. (012 must NOT depend on 018 — correct dependency direction preserved.)
- Leave the existing `/api/logs/streams` endpoint untouched.

**D-012-C1 — Single sidebar component, responsive via existing breakpoint (selected over dropdown/chip-row).**
- One `StreamPickerSidebar` component rendered inline on desktop and inside an overlay drawer on mobile, gated by the existing `useMediaQuery('(max-width:767px)')` / AppShell drawer pattern (reuse, not fork).
- Selected-state visibility on mobile is carried by the always-visible selection chips above the search box.

**D-012-D1 — Window-scoped counts; refetch on range change only (selected over expr-aware counts).**
- The services list/counts reflect the time window only (independent of the user's expr and of selectedServices). Refetch triggers: time-range change only. Expr edits and service toggles do NOT refetch the sidebar. Cache key `(start, end, limit)`.

**Additional locked UI/state details (from card):**
- `selectedServices: string[]` is SEPARATE state (not baked into expr). URL param `&services=a,b` (comma-joined; omit when empty; omit-absent-keys per exactOptionalPropertyTypes). MUST be added to BOTH `router.tsx` validateSearch AND the test harness's inline validateSearch.
- Chips above the search box show selected services with × to remove; click a sidebar row to toggle; currently-selected rows highlighted.
- Default limit 100 with a "Show more"/truncated banner when `truncated=true`.
- Hover highlights rows; counts displayed (e.g. "1,204").
- Works in BOTH plain and advanced modes (the services AND wraps whatever expr the active mode produces).

**Out of scope (YAGNI, from card):** other fields (host/severity), service grouping, search-within-sidebar.

**Stage flags:** Frontend-dominant + small backend (new endpoint + `/api/logs/query` param). Refinement scope 3a only (no host-integration concerns). Backend half (new `/api/logs/services` + the `services` param) gets an autonomous real-VL e2e in Refinement; the UI sidebar gets USER manual Desktop+Mobile testing.

**OpenAPI regen required this stage:** YES (new `LogsServicesResponse`/`ServiceCount` schemas + `/api/logs/services` endpoint + new `services` param on `/api/logs/query`). Sequence: `make openapi-export` then `bash scripts/generate-ui-types.sh` BEFORE frontend verify.
