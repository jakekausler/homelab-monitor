# STAGE-004-011: LogsQL advanced mode + syntax highlighting

## Status: Complete

## Overview

Add an "Advanced (LogsQL)" toggle to the Logs Explorer search box (STAGE-004-010). When on, the plain-text search box swaps for a LogsQL textarea with **syntax highlighting**. When off, the plain-text search continues to translate to `_msg:"..."`.

Both modes use the same backend `/api/logs/query` endpoint with the same paginated response — the only difference is what the UI sends as the `expr` param.

**Vertical slice:** user clicks "Advanced" checkbox → search box swaps to LogsQL editor with syntax highlighting for keywords, strings, operators, time literals. User can write `service:home-assistant AND severity:error | stats count()`. Toggle back to plain text preserves the last-used plain text input.

**Spec references:** master spec §9.2 (LogsQL explorer).

## Concrete deliverables

### 1. Toggle UI

`apps/ui/src/routes/logs/LogsExplorerPage.tsx` — add a small "Advanced (LogsQL)" checkbox or toggle button near the search input. When toggled on, the input area switches to a LogsQL editor component.

### 2. `<LogsQlEditor>` component

`apps/ui/src/components/logs/LogsQlEditor.tsx` (new) — wraps a code editor library (CodeMirror 6 or Monaco — Design decides):

- LogsQL syntax highlighting via custom language definition
- Tokens to highlight: keywords (`AND`, `OR`, `NOT`, `|`, `stats`, `head`, `sort`), strings (single + double quoted), comparison operators, field names (`_msg`, `_stream`, `_time`, `service`, `host`, `severity`), time literals
- Monospace font, dark/light theme matching app palette
- Multi-line editing (Tab indents, Enter creates newline, Ctrl+Enter executes)
- Placeholder text when empty
- Error highlighting on bad parse (best-effort — Design decides; bare textarea fallback is acceptable for v1 if a clean implementation is hard)

### 3. Mode state

State lives in `LogsExplorerPage`:
- `advancedMode: boolean`
- `plainText: string` (last plain-text input)
- `logsQl: string` (last LogsQL input)
- Switching modes preserves the other mode's content

The URL state carries either `q=<plain>` OR `logsql=<expr>` — not both. Deep-links open the right mode.

### 4. Library selection guidance

Design phase compares:
- **CodeMirror 6**: smaller bundle (~50KB minified), good extension API, slightly less polished UX
- **Monaco**: bigger bundle (~600KB lazy-load), VSCode-grade UX, possibly overkill

Recommendation: **CodeMirror 6** because bundle weight matters. Lazy-load the editor (and its language definition) so users who never toggle advanced don't pay the cost.

### 5. LogsQL language definition

`apps/ui/src/components/logs/logsQlLanguage.ts` — token rules for CodeMirror's `@codemirror/language`:

- Reserve keywords list
- String literals (single + double quote)
- Field accessor syntax (`field:value`)
- Pipe operator
- Comments (if LogsQL has them)
- Numbers + time literals

Initial scope is **basic tokens** only. NO autocomplete, NO error squigglies, NO LogsQL semantic validation (defer to a future stage if anyone asks).

### 6. Tests

- `__tests__/LogsQlEditor.test.tsx` — renders, accepts input, produces correct value events
- `__tests__/LogsExplorerPage.test.tsx` — mode toggle preserves prior input; URL state matches mode

## Decisions in this stage

### Design phase will confirm

- **Library: CodeMirror 6 vs Monaco vs bare textarea** — recommendation CodeMirror 6
- **Lazy-loading strategy** — code-split so non-advanced users don't load the editor
- **Initial highlighting scope** — basic tokens; explicit non-goals listed above
- **Mobile editor behavior** — on narrow viewports, simplify to a textarea (no syntax highlighting) for tap-friendliness? Recommendation: yes — Monaco/CodeMirror are not great on mobile

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-ADVANCED-TOGGLE** — explicit "Advanced (LogsQL)" toggle; plain-text mode is default.
- **D-SYNTAX-HIGHLIGHTING-V1** — basic tokens only (keywords, strings, operators, field accessors, pipe). No autocomplete or error analysis in v1.
- **D-MODE-CONTENT-PRESERVED** — toggling does NOT clear the other mode's content; user can flip back and forth.

## Design Phase

### Selected approach: Option A — CodeMirror 6 + StreamLanguage, lazy-loaded (shared `<LogsQlEditor>`)

Per user instruction "use your recommendations; make the editor a shared utility; reuse one if it already exists." Recon CONFIRMED no code editor exists in the frontend (no codemirror/monaco/prism/shiki/lezer in any package.json; no editor component anywhere) → a new dependency is added, but as a SHARED reusable component. Matches the card's recommendation and the brainstormer's reasoned pick.

### Confirmed deferred-decision answers

| Decision | Choice | Rationale |
| --- | --- | --- |
| Library | CodeMirror 6 (`@codemirror/state`, `/view`, `/language`, `/commands` + `@lezer/highlight` tags). NO `@uiw/react-codemirror` wrapper — hand-wire the EditorView. | Minimal dependency surface; StreamLanguage avoids authoring a Lezer grammar |
| Highlighting impl | `StreamLanguage.define(...)` regex-style line tokenizer + `HighlightStyle` in `logsQlLanguage.ts`. Basic tokens ONLY. | D-SYNTAX-HIGHLIGHTING-V1; the "is CM6 overkill?" worry is really about Lezer grammars — StreamLanguage sidesteps it |
| Lazy-load | `React.lazy(() => import('./LogsQlEditorImpl'))` + `<Suspense>`; CM6 chunk loads only when advanced mode is first enabled on desktop. Introduces the app's FIRST lazy/Suspense pattern. | Most sessions never pay the ~45-55KB gzipped editor chunk |
| Mobile behavior | Narrow viewport → plain `<textarea>` (no syntax highlighting, no CM6 load) via `useMediaQuery`. The same textarea serves as Suspense fallback, mobile editor, AND plain-mode-consistent input — one fallback reused three ways. | CM6 is not tap-friendly; mobile never pays the bundle cost |

### Shared component API (the user's "shared utility" requirement)

`apps/ui/src/components/logs/LogsQlEditor.tsx` — props:
```ts
interface LogsQlEditorProps {
  value: string
  onChange: (next: string) => void
  onSubmit: () => void        // Enter runs the query
  placeholder?: string
  ariaLabel: string
  className?: string
}
```
Same value/onChange/onSubmit contract as the plain search input, so the Body renders exactly ONE of `<LogsQlEditor>` (advanced) / plain input (plain) based on `advancedMode`. Swappable seam preserved for any future query box.

### File layout
- `apps/ui/src/components/logs/LogsQlEditor.tsx` (shell: viewport check → plain textarea on narrow, else `<Suspense>`-wrapped lazy CM6)
- `apps/ui/src/components/logs/LogsQlEditorImpl.tsx` (the CM6 EditorView; lazy-import target — keeps CM6 imports out of the eager bundle)
- `apps/ui/src/components/logs/logsQlLanguage.ts` (`StreamLanguage` tokenizer + `HighlightStyle`)
- Advanced toggle: follow the existing raw-`<input type=checkbox>` pattern from `WrapToggle.tsx`/`TimezoneToggle.tsx` (no Switch primitive exists).

### State + URL (extends the STAGE-004-010 seam)
- LogsExplorerPage gains THREE INDEPENDENT state values: `advancedMode: boolean`, `plainText: string`, `logsQl: string` (do NOT collapse to a union — STAGE-004-015 persists exactly {advancedMode, plainText, logsQl, ...}).
- `expr = advancedMode ? logsQl : translateSearchToLogsQl(plainText)`.
- URL: `logsql=<expr>` (advanced) XOR `q=<plain>` (plain), mutually exclusive; write via existing `useNavigate` OMITTING the absent key (exactOptionalPropertyTypes). Deep-links open the correct mode.
- The existing `committedSearchText`/`liveSearchText` plain-text wiring is refactored into the `plainText` value; the Body's seam comment location is where the conditional editor/input renders.

### LogsQL tokens to highlight (grounded in repo usage)
- Keywords: `AND OR NOT stats count count_uniq sort head limit fields filter by as`
- Field names: `_msg _time _stream _stream_id service host severity compose_service compose_project container_name cron_fingerprint image`
- Operators: `| : :> :< :>= :<= = -`
- Literals: double/single-quoted strings, numbers, duration literals (`5m/1h/24h/7d`). No comment syntax in this project.

### Forward-compat seams (look-ahead — leave clean, don't build)
- STAGE-004-012 stream picker keeps `selectedServices` SEPARATE and AND's it with expr SERVER-SIDE — do NOT bake stream filtering into the editor or expr. The editor owns only the `logsQl` text.
- STAGE-004-015 persists {advancedMode, plainText, logsQl, selectedServices, range, scrollPosition, cursor, lastVisitedAt} to localStorage (URL-wins) — keep the three mode values independent + serializable.

### Out of scope (unchanged)
Autocomplete, error squigglies, semantic validation, prettifier, cheatsheet, vim/emacs keybindings.

### Locked decisions inherited (unchanged)
D-ADVANCED-TOGGLE, D-SYNTAX-HIGHLIGHTING-V1, D-MODE-CONTENT-PRESERVED.

### User-testing findings + resolutions

1. **`range_in_future` error loading logs** (clock-skew bug — FIXED). The Explorer sends an absolute `end == browser-now`; the backend's strict `end > now` check rejected it under browser/container clock skew. Fix: added a 5-second clock-skew grace (`FUTURE_SKEW_GRACE`) to `parse_and_validate_window` in `apps/monitor/.../kernel/logs/time_window.py` (end ≤ now+5s accepted; genuinely-future ranges still rejected). 4 boundary tests added to `tests/test_time_window.py`. Verified live on prod: end=now → 200, end=now+3s → 200, end=now+1h → 400 range_in_future. PRE-EXISTING latent issue in shared validation; surfaced by the Explorer (only viewer sending absolute end==now).
2. **Pre-existing flaky SSE test fixed (discovered during verify).** `tests/test_api_events_sse.py::test_sse_http_endpoint_smoke` errored with "uvicorn did not start within 20s" because its uvicorn fixture booted the real lifespan against a LIVE Docker socket (the prod rig was running), so startup collectors did real Docker+registry I/O. Fix: `tests/_uvicorn_fixture.py` now overrides `HOMELAB_MONITOR_DOCKER_SOCKET` to a nonexistent path (mirrors conftest.py), so startup collectors short-circuit. Unrelated to this stage's feature; fixed because it blocked verification.
3. **CodeMirror cursor/click bug** (UX — FIXED). No visible caret + clicks past the text didn't position the cursor. Root cause: the editor host `<div>` carried the input-box styling but CM6's editable area didn't fill it. Fix: moved box styling into the CM6 `EditorView.theme` (`.cm-content` min-height 2.25rem + full width + `caretColor` + focused `.cm-cursor` color); host div reduced to `w-full max-w-md`. Caret now visible; clicks anywhere in the box position the caret.
