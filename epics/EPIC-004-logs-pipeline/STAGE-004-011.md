# STAGE-004-011: LogsQL advanced mode + syntax highlighting

## Status: Not Started

## Overview

Add an "Advanced (LogsQL)" toggle to the Logs Explorer search box (STAGE-004-010). When on, the plain-text search box swaps for a LogsQL textarea with **syntax highlighting**. When off, the plain-text search continues to translate to `_msg:"..."`.

Both modes use the same backend `/api/logs/query` endpoint with the same paginated response — the only difference is what the UI sends as the `expr` param.

**Vertical slice:** user clicks "Advanced" checkbox → search box swaps to LogsQL editor with syntax highlighting for keywords, strings, operators, time literals. User can write `service:home-assistant AND severity:error | stats count()`. Toggle back to plain text preserves the last-used plain text input.

**Spec references:** master spec §9.2 (LogsQL explorer).

## Concrete deliverables

### 1. Toggle UI

`apps/ui/src/routes/logs/LogsExplorerPage.tsx` — add a small "Advanced (LogsQL)" checkbox or toggle button near the search input. When toggled on, the input area switches to a LogsQL editor component.

### 2. `<LogsQlEditor>` component

`apps/ui/src/components/logs/LogsQlEditor.tsx` (new) — wraps a code editor library (CodeMirror 6 or Monaco — Design decides):

- LogsQL syntax highlighting via custom language definition
- Tokens to highlight: keywords (`AND`, `OR`, `NOT`, `|`, `stats`, `head`, `sort`), strings (single + double quoted), comparison operators, field names (`_msg`, `_stream`, `_time`, `service`, `host`, `severity`), time literals
- Monospace font, dark/light theme matching app palette
- Multi-line editing (Tab indents, Enter creates newline, Ctrl+Enter executes)
- Placeholder text when empty
- Error highlighting on bad parse (best-effort — Design decides; bare textarea fallback is acceptable for v1 if a clean implementation is hard)

### 3. Mode state

State lives in `LogsExplorerPage`:
- `advancedMode: boolean`
- `plainText: string` (last plain-text input)
- `logsQl: string` (last LogsQL input)
- Switching modes preserves the other mode's content

The URL state carries either `q=<plain>` OR `logsql=<expr>` — not both. Deep-links open the right mode.

### 4. Library selection guidance

Design phase compares:
- **CodeMirror 6**: smaller bundle (~50KB minified), good extension API, slightly less polished UX
- **Monaco**: bigger bundle (~600KB lazy-load), VSCode-grade UX, possibly overkill

Recommendation: **CodeMirror 6** because bundle weight matters. Lazy-load the editor (and its language definition) so users who never toggle advanced don't pay the cost.

### 5. LogsQL language definition

`apps/ui/src/components/logs/logsQlLanguage.ts` — token rules for CodeMirror's `@codemirror/language`:

- Reserve keywords list
- String literals (single + double quote)
- Field accessor syntax (`field:value`)
- Pipe operator
- Comments (if LogsQL has them)
- Numbers + time literals

Initial scope is **basic tokens** only. NO autocomplete, NO error squigglies, NO LogsQL semantic validation (defer to a future stage if anyone asks).

### 6. Tests

- `__tests__/LogsQlEditor.test.tsx` — renders, accepts input, produces correct value events
- `__tests__/LogsExplorerPage.test.tsx` — mode toggle preserves prior input; URL state matches mode

## Out of scope for this stage

- LogsQL autocomplete — YAGNI for v1
- LogsQL syntax error highlighting (squigglies) — defer
- "Format LogsQL" prettifier — YAGNI
- LogsQL examples / cheatsheet inside the editor — could add as a small follow-up if requested
- Vim/Emacs keybindings — YAGNI

## Decisions in this stage

### Design phase will confirm

- **Library: CodeMirror 6 vs Monaco vs bare textarea** — recommendation CodeMirror 6
- **Lazy-loading strategy** — code-split so non-advanced users don't load the editor
- **Initial highlighting scope** — basic tokens; explicit non-goals listed above
- **Mobile editor behavior** — on narrow viewports, simplify to a textarea (no syntax highlighting) for tap-friendliness? Recommendation: yes — Monaco/CodeMirror are not great on mobile

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-ADVANCED-TOGGLE** — explicit "Advanced (LogsQL)" toggle; plain-text mode is default.
- **D-SYNTAX-HIGHLIGHTING-V1** — basic tokens only (keywords, strings, operators, field accessors, pipe). No autocomplete or error analysis in v1.
- **D-MODE-CONTENT-PRESERVED** — toggling does NOT clear the other mode's content; user can flip back and forth.
