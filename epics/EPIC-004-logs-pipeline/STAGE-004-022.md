# STAGE-004-022: Global retention settings UI (VL retention + disk thresholds; per-store, no cross-store coordinator)

## Status: Complete

## Overview

A small Settings → Logs page that displays VL's current retention setting, disk usage, and configured warning/critical thresholds — with a UI control to change retention. **No cross-store orchestration coordinator** (locked decision); each of VM, VL, SQLite has independent thresholds.

For this stage we ship the VL surface. Future epics or polish stages add the mirroring UI for VM and SQLite thresholds.

**Vertical slice:** navigate to `/settings/logs` → see current VL retention (default 30 days), current disk usage, warn/crit thresholds. Change retention → save → backend updates the runtime config + persists the change.

**Spec references:** master spec §6.4 (disk budget, per-stream caps).

## Concrete deliverables

### 1. Backend endpoints

`apps/monitor/homelab_monitor/kernel/api/routers/settings_logs.py` (new):

```
GET /api/settings/logs/retention -> {retention_days: int, disk_used_pct: float, warn_pct: int, crit_pct: int, disk_used_gb: float, retention_source: str}
PATCH /api/settings/logs/retention {retention_days: int}
```

- `retention_source`: where the current value comes from — `"env"` (from `HOMELAB_MONITOR_VL_RETENTION_DAYS`), `"runtime"` (changed via UI), or `"default"` (30d fallback)
- `PATCH` writes the new value to a runtime config (SQLite kv table) AND signals VL to apply it (VL retention is restart-required in some versions; Design phase confirms whether the change is live-applied or queued for next restart)

### 2. VL retention application

`apps/monitor/homelab_monitor/kernel/logs/vl_retention.py` (new) — orchestrates the change:

4. Surface "restart required" state to the UI if applicable

## Design Phase

**Design phase complete:** 2026-06-04

**Selected Approach: Option A — persist-to-SQLite retention override + explicit effective-vs-pending contract (no auto-apply)**

A new `/settings/logs` page surfaces VL retention, disk usage, and warn/crit thresholds. The backend honors the hard reality that **VL retention is startup-only** (set via the `-retentionPeriod=${HOMELAB_MONITOR_VL_RETENTION_DAYS}d` compose flag; VL v0.30.0 has NO runtime retention API), so a PATCH cannot live-apply — it persists the desired value to SQLite and signals **restart-required**. GET reconciles the EFFECTIVE retention (what VL is actually running, from the env var via `load_vl_retention_days()`, or VL's built-in 30d default) against a PENDING runtime override (the SQLite value, if set + differing), exposing `retention_source` (`env`|`runtime`|`default`) and an explicit `restart_required: bool`. The UI shows current-effective vs pending with a restart-required banner that persists until they reconcile. This is the only option honoring ALL three locked decisions: D-VL-RETENTION-UI (genuine persisting PATCH), D-RESTART-REQUIRED-IF-NEEDED (explicit signal, NOT a forced restart), D-NO-COORDINATOR (single VL store only).

**Verified backend-contract facts (file:line confirmed):**
- VL retention = startup-only CLI flag `-retentionPeriod=${HOMELAB_MONITOR_VL_RETENTION_DAYS:-30}d` (deploy/compose/docker-compose.yml:277). No runtime HTTP API in VL v0.30.0. Changing it = edit env + `docker compose up -d --force-recreate victorialogs`.
- Disk usage ALREADY collected: `plugins/collectors/builtin/self_disk.py` `SelfDiskCollector` emits `homelab_self_disk_used_bytes{slot="vl"}` / `homelab_self_disk_budget_bytes{slot="vl"}` / `homelab_self_disk_used_pct` every 60s, reading RO-mounted `/var/vl-data` + `load_disk_budget_config()` (default 50GB total, vl_ratio 0.30).
- No settings/kv table exists → new alembic migration (0032) + repo (pattern: `kernel/db/repositories/override_ownership_repository.py`). DB at apps/monitor/alembic/versions/ (0031 = log_saved_queries is latest).
- Config: `kernel/config.py` dataclass+loader pattern; `load_vl_retention_days()` (HOMELAB_MONITOR_VL_RETENTION_DAYS, default 30) + `load_disk_budget_config()` already exist; add `load_vl_disk_warning_config()` (HOMELAB_MONITOR_VL_DISK_WARN_PCT=70, _CRIT_PCT=85).
- Routers mounted `app.include_router(x.router, prefix="/api")` (kernel/api/app.py). `admin.py` uses `require_user_or_token({Scope...})` for writes; logs routers use `require_session()` for reads. No `/api/settings/*` router exists yet.
- Frontend: NO `/settings` route exists (this stage introduces it — mirror the `inventoryLayoutRoute` parent+child pattern in router.tsx). SidebarNav.tsx has Settings as a DISABLED "Coming soon" entry (line 52) → enable it. `Card`/`CardHeader`/`CardContent` exist; NO Alert/Banner component (use a styled div). API hooks mirror `savedLogQueries.ts` (useQuery GET + useMutation PATCH + invalidate).
- NOTE: real backend path is `apps/monitor/homelab_monitor/...` (NO `src/` prefix).

**Resolved design questions:**
- **Q1 (persistence model):** Option A — PATCH validates 1≤days≤365, writes a runtime override to a NEW settings/kv table (migration 0032 + repo). GET returns effective + (pending if differing) + `retention_source` + `restart_required`. NO auto-apply (rejected C: editing host compose .env / docker-socket recreate violates least-privilege + the auto-fix safety model; nullifies D-RESTART-REQUIRED's "signal not perform").
- **Q2 (disk-usage denominator):** reuse the existing `self_disk` BUDGET semantics — `disk_used_pct` and warn/crit (70/85) both mean "% of the VL disk BUDGET" (`used_bytes{slot=vl}` / `budget_bytes{slot=vl}`), `disk_used_gb` = used_bytes/1e9. One denominator across all three. (Rejected `shutil.disk_usage` device-level — dominated by unrelated disk tenants, makes 70/85 meaningless; rejected VL `/metrics` — not currently wired.) Build confirms how the endpoint reads the latest collector sample.
- **Q3 (router + auth):** new `kernel/api/routers/settings_logs.py`, prefix `/api` → `/api/settings/logs/retention`. GET = `require_session()`. PATCH = reuse an admin write Scope if a suitable one exists, else `require_session()` (single-user homelab — do NOT invent a scope). Build confirms which.
- **Q4 (restart-required):** explicit `restart_required: bool` field (true iff a runtime override exists + differs from effective) drives the UI banner. Explicit > implicit (`pending != effective`).
- **Q5 (frontend):** new `/settings` parent route (Outlet) + `/settings/logs` child; enable SidebarNav Settings entry; `SettingsLogsPage.tsx` Card-based — Retention card (current effective + source chip + pending + numeric input + Save + restart banner) + Disk-usage card (used GB + % vs budget, warn/crit coloring) + per-stream-caps info card. exactOptionalPropertyTypes-safe hooks.

**Endpoint contract:**
- `GET /api/settings/logs/retention -> {retention_days: int (effective), disk_used_pct: float, warn_pct: int, crit_pct: int, disk_used_gb: float, retention_source: "env"|"runtime"|"default", restart_required: bool, pending_retention_days: int|null}` (pending field added for the effective-vs-pending model).
- `PATCH /api/settings/logs/retention {retention_days: int}` (1≤≤365, 422 out of range) → persists runtime override, returns the updated GET shape with restart_required=true.

**Rejected alternatives:**
- **Option B (read-only display + guided manual change, NO persistence):** simplest + most truthful to the constraint, BUT VIOLATES locked D-VL-RETENTION-UI ("allow a runtime change WITH persistence") — the card explicitly wants a persisting PATCH. Rejected: violates a locked decision.
- **Option C (persist + auto-rewrite compose .env + force-recreate VL via docker socket):** only option making the change truly live, BUT requires docker-socket + host-file-write privileges for the monitor — a large least-privilege violation against the auto-fix safety model; recreating VL mid-operation degrades the monitor's own log store; nullifies D-RESTART-REQUIRED's "signal not perform"; poor testability. Rejected: out of scope + against the security model.

**Out of scope (confirmed):** per-stream retention overrides, per-signature retention, VM threshold UI, SQLite threshold UI, cross-store coordinator, retention preview/simulation, auto-apply. (The new kv table is created minimal — one VL retention key — though shaped to be reusable for future single-store threshold persistence.)

**Forward-compat:** Establishes the `/settings` route shell + the settings/kv persistence table + the `/api/settings/*` router pattern. EPIC-014 (self-monitor) or a future polish epic adds the VM + SQLite threshold UIs reusing this surface. STAGE-004-041 owns the vmalert disk-threshold rules (this stage only surfaces the values).

**Refinement:** Backend + frontend. Frontend → 3a, Desktop + Mobile (manual test of the /settings/logs page + retention change → restart-required banner). Backend smoke via e2e-tester (GET/PATCH against the prod rig: retention_source states, restart_required toggle, disk usage values, 422 validation). The effective-vs-pending UX must be eyeballed on the real rig.
