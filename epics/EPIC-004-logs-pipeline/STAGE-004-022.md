# STAGE-004-022: Global retention settings UI (VL retention + disk thresholds; per-store, no cross-store coordinator)

## Status: Not Started

## Overview

A small Settings → Logs page that displays VL's current retention setting, disk usage, and configured warning/critical thresholds — with a UI control to change retention. **No cross-store orchestration coordinator** (locked decision); each of VM, VL, SQLite has independent thresholds.

For this stage we ship the VL surface. Future epics or polish stages add the mirroring UI for VM and SQLite thresholds.

**Vertical slice:** navigate to `/settings/logs` → see current VL retention (default 30 days), current disk usage, warn/crit thresholds. Change retention → save → backend updates the runtime config + persists the change.

**Spec references:** master spec §6.4 (disk budget, per-stream caps).

## Concrete deliverables

### 1. Backend endpoints

`apps/monitor/homelab_monitor/kernel/api/routers/settings_logs.py` (new):

```
GET /api/settings/logs/retention -> {retention_days: int, disk_used_pct: float, warn_pct: int, crit_pct: int, disk_used_gb: float, retention_source: str}
PATCH /api/settings/logs/retention {retention_days: int}
```

- `retention_source`: where the current value comes from — `"env"` (from `HOMELAB_MONITOR_VL_RETENTION_DAYS`), `"runtime"` (changed via UI), or `"default"` (30d fallback)
- `PATCH` writes the new value to a runtime config (SQLite kv table) AND signals VL to apply it (VL retention is restart-required in some versions; Design phase confirms whether the change is live-applied or queued for next restart)

### 2. VL retention application

`apps/monitor/homelab_monitor/kernel/logs/vl_retention.py` (new) — orchestrates the change:

1. Validate the new value (positive integer, ≤ some sane max like 365)
2. Write to runtime config
3. Attempt to apply to VL (HTTP call or restart-required flag)
4. Surface "restart required" state to the UI if applicable

### 3. Threshold metrics + alerts (mirror of STAGE-004-041's VL backend health)

Add env-var-configurable thresholds:
- `HOMELAB_MONITOR_VL_DISK_WARN_PCT=70`
- `HOMELAB_MONITOR_VL_DISK_CRIT_PCT=85`

(Note: STAGE-004-041 owns the actual vmalert rules + alert payload for VL backend health; this stage only surfaces the values in the settings UI.)

### 4. Frontend: Settings → Logs page

`apps/ui/src/routes/settings/SettingsLogsPage.tsx` (new) — route `/settings/logs`:

```
+---------------------------------------------+
| Logs settings                               |
+---------------------------------------------+
| Retention                                   |
| Current: 30 days (source: env)              |
| New: [____30 days____] [Save]               |
| ⚠ Restart required for change to apply      |
+---------------------------------------------+
| Disk usage                                  |
| Used: 14.2 GB (47% of available)            |
| ⚠ Warning at 70% / 🔴 Critical at 85%       |
+---------------------------------------------+
| Per-stream caps                             |
| Configured in homelab-monitor.yaml          |
| Edit file + restart to change.              |
+---------------------------------------------+
```

Top-level Settings page (if not present) adds a "Logs" link.

## Decisions in this stage

### Design phase will confirm

- **Live-apply vs restart-required** for retention changes — depends on VL version; Design phase confirms
- **Max retention input** — recommend cap at 365 days
- **Settings nav structure** — does `/settings/*` already exist or does this stage introduce it?

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-NO-COORDINATOR** — no cross-store auto-shrink; each store has its own thresholds independently.
- **D-VL-RETENTION-UI** — surface VL retention in `/settings/logs`; allow runtime change with persistence.
- **D-RESTART-REQUIRED-IF-NEEDED** — gracefully signal restart requirement when applicable.
