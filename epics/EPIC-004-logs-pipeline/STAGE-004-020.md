# STAGE-004-020: Log-line export (download matching lines as .txt or .json)

## Status: Not Started

## Overview

Add an "Export" button to the Logs Explorer that downloads the matching log lines for the current query + range in the user's chosen format (`.txt` plain text or `.json` array of `LogLine` objects). Useful for sharing with team, opening in external tools, or archiving a forensic slice.

**Vertical slice:** in the Explorer, click "Export" → format picker → download dialog. The browser receives a file containing the current query's matching lines (up to a configurable cap).

**Spec references:** master spec §9.2 (logs explorer features).

## Stage Flags

- Has Input Forms: [x] Yes — format picker dialog
- Touches secrets / production data: [x] Yes (sensitive log content) — redaction (STAGE-004-006) already strips secrets at ingest, so exports are safe

## Concrete deliverables

### 1. Backend endpoint: streamed export

`apps/monitor/homelab_monitor/kernel/api/routers/logs.py` — new endpoint:

```
GET /api/logs/export?expr=<LogsQL>&start=<ISO>&end=<ISO>&format=txt|json&max=10000
```

Response: streamed (`text/plain` or `application/json`) with `Content-Disposition: attachment; filename="logs_<timestamp>.<ext>"`.

Implementation:
- Query VL with the same LogsQL + range (no pagination — use a higher batch size, paginate internally)
- Stream the output as it's fetched (avoid loading the whole result into memory)
- Cap at `max` lines (default 10,000, hard max 100,000)
- For `txt`: format `<timestamp> [<severity>] <service>: <message>` per line
- For `json`: open with `[`, comma-separated `LogLine` objects, close with `]`

### 2. Frontend: Export button + dialog

`apps/ui/src/routes/logs/ExportButton.tsx` (new):

- Button: "Export"
- Click → modal with format radio (txt / json) + max-lines input (default 10,000, capped at 100,000) + "Download" / "Cancel"
- Download triggers a navigation to the export endpoint with current query params (uses standard browser download flow; not React-Query-cached)

### 3. Filename convention

`logs_YYYY-MM-DD_HHmmss.{txt,json}` — UTC timestamp at the moment of export.

### 4. Tests

- Backend: `apps/monitor/tests/test_api_logs_export.py` — content-type, content-disposition header, streamed body format, cap enforcement
- Frontend: `__tests__/ExportButton.test.tsx` — modal renders, navigation URL has correct query params

## Out of scope for this stage

- CSV format — could add later if asked
- Direct copy-to-clipboard for small result sets — YAGNI; download covers it
- Email export — YAGNI
- Per-line redaction at export time — redaction happens at ingest already; nothing to do here
- Compression (gzip) — YAGNI

## Decisions in this stage

### Design phase will confirm

- **Max line cap** — 10,000 default, 100,000 hard max — adjust based on real-world use
- **TXT format details** — exact field order + separators
- **Whether to expose a "Filter to errors only" toggle in the dialog** — recommendation no; user filters in the Explorer before exporting

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-EXPORT-TXT-AND-JSON** — both formats in v1.
- **D-EXPORT-CAPS-ENFORCED** — line cap prevents accidental million-line downloads.
- **D-EXPORT-STREAMED** — backend streams; doesn't buffer the whole result in memory.
