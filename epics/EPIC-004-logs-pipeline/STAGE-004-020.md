# STAGE-004-020: Log-line export (download matching lines as .txt or .json)

## Status: Complete

## Overview

Add an "Export" button to the Logs Explorer that downloads the matching log lines for the current query + range in the user's chosen format (`.txt` plain text or `.json` array of `LogLine` objects). Useful for sharing with team, opening in external tools, or archiving a forensic slice.

**Vertical slice:** in the Explorer, click "Export" → format picker → download dialog. The browser receives a file containing the current query's matching lines (up to a configurable cap).

**Spec references:** master spec §9.2 (logs explorer features).

## Concrete deliverables

### 1. Backend endpoint: streamed export

`apps/monitor/homelab_monitor/kernel/api/routers/logs.py` — new endpoint:

```
GET /api/logs/export?expr=<LogsQL>&start=<ISO>&end=<ISO>&format=txt|json&max=10000
```

Response: streamed (`text/plain` or `application/json`) with `Content-Disposition: attachment; filename="logs_<timestamp>.<ext>"`.

Implementation:
- Query VL with the same LogsQL + range (no pagination — use a higher batch size, paginate internally)
- Stream the output as it's fetched (avoid loading the whole result into memory)
- Cap at `max` lines (default 10,000, hard max 100,000)
- For `txt`: format `<timestamp> [<severity>] <service>: <message>` per line
- For `json`: open with `[`, comma-separated `LogLine` objects, close with `]`

### 2. Frontend: Export button + dialog

`apps/ui/src/routes/logs/ExportButton.tsx` (new):

- Button: "Export"
- Click → modal with format radio (txt / json) + max-lines input (default 10,000, capped at 100,000) + "Download" / "Cancel"
- Download triggers a navigation to the export endpoint with current query params (uses standard browser download flow; not React-Query-cached)

### 3. Filename convention

`logs_YYYY-MM-DD_HHmmss.{txt,json}` — UTC timestamp at the moment of export.

### 4. Tests

- Backend: `apps/monitor/tests/test_api_logs_export.py` — content-type, content-disposition header, streamed body format, cap enforcement
- Frontend: `__tests__/ExportButton.test.tsx` — modal renders, navigation URL has correct query params

## Design Phase

**Design phase complete:** 2026-06-04

**Selected Approach: Option A — True streaming via httpx `client.stream()` + `StreamingResponse`**

A new backend endpoint `GET /api/logs/export` streams matching log lines directly to the browser as a download, bounded to O(1) memory (one line at a time). A new low-level VL access path opens `http_client.stream("GET", <vl_url>/select/logsql/query, params={query, start, end, limit=max})` and async-iterates `aiter_lines()`, parsing each NDJSON line via the existing `_parse_one` logic, mapping to `LogLine` via `from_victorialogs_line`, formatting per `format` (txt/json), and yielding bytes into a `StreamingResponse` — stopping at `max` lines. This is the only option that honors the locked **D-EXPORT-STREAMED** decision literally (never materializes the whole result), mirroring the established `StreamingResponse` async-generator precedent in `kernel/api/routers/events.py`.

**De-risking refinements (adopted from brainstorm):**
- **Pre-flight upstream-error probe:** before returning the `StreamingResponse`, open the VL stream and pull the FIRST line inside the route handler so connection failures / VL non-200 surface as `HttpProblem(502, "upstream_unavailable")` with the correct status code and headers (you cannot change status after the first byte flows). Genuine mid-stream failures (rare, after VL already returned 200 + began the body) truncate the download — accepted limitation of streamed downloads, documented.
- **Empty-result correctness:** JSON formatter emits `[]` (not an empty body) when zero lines match; TXT emits an empty body. Both covered explicitly for the 100%-coverage gate.

**Endpoint contract:**
`GET /api/logs/export?expr=<LogsQL>&start=<ISO>&end=<ISO>&format=txt|json&max=10000&services=<csv>`
- `format`: `txt` (default) | `json` — validated; out-of-set → 422.
- `max`: `Query(10000, ge=1, le=100000)` → 422 out of range. Default 10,000, hard max 100,000.
- Reuses `parse_and_validate_window(start, end)` (ISO / ±5s future grace / ≤30d) and `_compose_services_expr(expr, services)`.
- Auth: `Depends(require_session())`. DI: `get_vl_url`, `get_http_client`.
- `text/plain; charset=utf-8` (txt) or `application/json` (json); `Content-Disposition: attachment; filename="logs_YYYY-MM-DD_HHmmss.{txt,json}"` (UTC at moment of export).
- **TXT line format:** `<timestamp> [<severity>] <service>: <message>` (one per line). `None` severity → `[unknown]`/`[-]` placeholder; `None` service → empty. (Exact placeholder finalized in Build.)
- **JSON:** `[`, comma-joined `LogLine` JSON objects, `]` (incremental framing: first-object-no-leading-comma state machine).
- No caching (one-shot download — the 30s-cache pattern does NOT apply).

**Frontend:**
- New `apps/ui/src/routes/logs/ExportButton.tsx` — a button in the Explorer control bar (Row C, after Save Query) opening a Radix `Dialog` (mirror `SaveQueryModal.tsx`).
- Dialog: format radio (txt/json — native `<input type="radio">` + `Label`, since no `RadioGroup` component exists) + max-lines numeric `Input` (default 10000, clamped 1–100000) + Download / Cancel.
- On Download: build the export URL from the current `expr`/`startIso`/`endIso`/`servicesCsv` + chosen `format`/`max`, trigger a browser download via a hidden `<a download href=...>` click (cookie-auth GET → direct navigation works). NO react-query hook (it is a one-shot download, not a fetch-into-state).

**Verified backend-contract facts (VL v0.30.0 / current code):**
- `VictoriaLogsClient.query()` is NOT streaming — it materializes the full `VlQueryResult`. Export therefore needs a NEW streaming access path (does not reuse `query()`); it reuses the lower-level NDJSON line parsing (`_parse_one`) + `from_victorialogs_line` + `normalize_severity`.
- VL `/select/logsql/query` returns NDJSON (one JSON object per line) and accepts a `limit` param that caps returned lines. VL pinned at `v0.30.0`.
- `httpx.AsyncClient` supports streaming via `client.stream("GET", ...)` + `response.aiter_lines()`.
- Streaming precedent: `kernel/api/routers/events.py` uses `starlette.responses.StreamingResponse(async_gen(), media_type=...)`.

**Rejected alternatives:**
- **Option B (reuse existing `query()` → `StreamingResponse` over the in-memory list):** Simplest + max reuse + cleanest error handling (errors map to 502 pre-stream), BUT VIOLATES the spirit of D-EXPORT-STREAMED — it buffers up to 100k `LogLine` objects (~tens of MB) before the first byte. Rejected because D-EXPORT-STREAMED is locked and explicitly says "does NOT buffer the whole result in memory." (Fallback only if that decision were reinterpreted as "stream the HTTP response.")
- **Option C (hybrid: internal batched pagination via repeated bounded queries + `StreamingResponse`):** O(batch) memory, honors D-EXPORT-STREAMED in spirit, BUT dominated by A — concedes the SAME mid-stream-error limitation as A while adding cursor/`_time`-boundary correctness risk (duplicate/dropped lines at batch boundaries) and the hardest test surface (multiple sequential `pytest_httpx` mocks). Rejected: strictly more code + more risk than A's O(1) for no compensating benefit.

**Out of scope (confirmed):** CSV, copy-to-clipboard, email export, export-time redaction (redacted at ingest), gzip, "errors-only" toggle in the dialog (user filters in Explorer before exporting).

**Forward-compat:** Establishes the streamed-download pattern; CSV could be added as a third `format` value later. The new low-level VL streaming method is reusable by any future bulk-read feature.

**Refinement:** Frontend stage → 3a only (no host-integration delta). Manual Desktop+Mobile test of the dialog + a real download against the prod rig; backend streaming smoke via e2e-tester (curl: headers, both formats, cap, empty result). No `make integration` needed unless an integration test is added.
