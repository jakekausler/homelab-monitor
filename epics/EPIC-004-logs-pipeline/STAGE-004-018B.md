# STAGE-004-018B: Configurable visible columns in logs results

## Status: Not Started

## Overview

Let the user choose which fields appear as COLUMNS in the Logs Explorer results, beyond the default timestamp + message. Selectable columns include promoted fields (`service`, `host`, `severity`, `compose_service`, `container_name`, etc.) and any field surfaced by scope-aware field discovery (STAGE-004-018). This lets a user separate log-line types in the unified `/logs` view (e.g. add a `service` column to tell home-assistant lines from pi-hole lines at a glance).

**Vertical slice:** in the Logs Explorer, a "Columns" control lets the user toggle `service`/`host`/`severity` (and other discovered fields) on/off; the results render those as additional columns alongside time + message; the selection persists (URL and/or the STAGE-004-015 persistence layer).

**Depends on:** STAGE-004-018 (scope-aware field discovery — the list of fields available to add as columns) and the shared `<LogViewer>`/`LogLineList` (STAGE-004-003) which renders the rows. Inserted after 018 (and 018A) for this reason.

**Spec references:** master spec §9.2 (Logs explorer screen).

## Concrete deliverables

### 1. Column picker control

A "Columns" dropdown/popover in the Logs Explorer header listing toggleable fields: the default-promoted fields + fields from scope-aware discovery (STAGE-004-018). Add/remove (and ideally reorder) visible columns.

### 2. LogViewer / LogLineList column rendering

Extend `<LogViewer>` / `LogLineList` (STAGE-004-003) to optionally render extra columns from each line's fields, in addition to timestamp + message. Must be backward-compatible: existing consumers (docker/cron viewers) that pass no column config render exactly as today (timestamp + message only). New optional prop, e.g. `columns?: ColumnDescriptor[]`.

### 3. State + persistence

The selected columns are explorer state. Reflect in the URL and/or hand off to the STAGE-004-015 persistence layer (per-user remembered column set). Keep the shape forward-compatible with 015's persisted ExplorerState.

### 4. Empty/missing-field handling

A line missing a configured column's field renders an empty cell (not a crash). Severity column may reuse existing severity tinting.

### 5. Tests

- LogLineList renders configured columns; lines missing a field show empty cells.
- Column picker toggles add/remove columns; backward-compat: no column config → timestamp+message only (existing viewers unchanged).
- State round-trips (URL/persistence).

## Out of scope for this stage

- Column resizing / drag-to-reorder polish (basic reorder optional; fancy DnD is YAGNI for v1).
- Changing columns on the docker/cron context viewers (this targets the Explorer; other viewers opt in later if desired).
- Server-side column projection (the backend already returns full LogLine + fields; this is a display concern).

## Decisions in this stage

### Design phase will confirm

- The column-set data model + the `<LogViewer>`/`LogLineList` API for optional columns (must stay backward-compatible for docker/cron).
- Where selected columns live (URL param vs STAGE-004-015 persistence vs both) — coordinate with 015's ExplorerState shape.
- Table vs flex-row layout for multi-column rendering on desktop, and mobile behavior (recommended: collapse extra columns or show a compact subset on narrow viewports).
- Which fields are offered by default vs discovered dynamically.

### Locked Design Decisions (inherited from brainstorming 2026-05-28 + user request 2026-06-01)

- **D-COLUMNS-OPT-IN** — configurable columns are an Explorer feature; the shared `<LogViewer>` gains an OPTIONAL columns prop so docker/cron viewers are unaffected by default.
- **D-COLUMNS-FROM-DISCOVERY** — the selectable field list comes from scope-aware field discovery (STAGE-004-018) plus the known promoted fields; hence sequenced after 018.
