# STAGE-004-018B: Configurable visible columns in logs results

## Status: Complete

## Overview

Let the user choose which fields appear as COLUMNS in the Logs Explorer results, beyond the default timestamp + message. Selectable columns include promoted fields (`service`, `host`, `severity`, `compose_service`, `container_name`, etc.) and any field surfaced by scope-aware field discovery (STAGE-004-018). This lets a user separate log-line types in the unified `/logs` view (e.g. add a `service` column to tell home-assistant lines from pi-hole lines at a glance).

**Vertical slice:** in the Logs Explorer, a "Columns" control lets the user toggle `service`/`host`/`severity` (and other discovered fields) on/off; the results render those as additional columns alongside time + message; the selection persists (URL and/or the STAGE-004-015 persistence layer).

**Depends on:** STAGE-004-018 (scope-aware field discovery — the list of fields available to add as columns) and the shared `<LogViewer>`/`LogLineList` (STAGE-004-003) which renders the rows. Inserted after 018 (and 018A) for this reason.

**Spec references:** master spec §9.2 (Logs explorer screen).

## Concrete deliverables

### 1. Column picker control

A "Columns" dropdown/popover in the Logs Explorer header listing toggleable fields: the default-promoted fields + fields from scope-aware discovery (STAGE-004-018). Add/remove (and ideally reorder) visible columns.

### 2. LogViewer / LogLineList column rendering

Extend `<LogViewer>` / `LogLineList` (STAGE-004-003) to optionally render extra columns from each line's fields, in addition to timestamp + message. Must be backward-compatible: existing consumers (docker/cron viewers) that pass no column config render exactly as today (timestamp + message only). New optional prop, e.g. `columns?: ColumnDescriptor[]`.

### 3. State + persistence

The selected columns are explorer state. Reflect in the URL and/or hand off to the STAGE-004-015 persistence layer (per-user remembered column set). Keep the shape forward-compatible with 015's persisted ExplorerState.

### 4. Empty/missing-field handling

A line missing a configured column's field renders an empty cell (not a crash). Severity column may reuse existing severity tinting.

### 5. Tests

- LogLineList renders configured columns; lines missing a field show empty cells.
- Column picker toggles add/remove columns; backward-compat: no column config → timestamp+message only (existing viewers unchanged).
- State round-trips (URL/persistence).

## Out of scope for this stage

- Column resizing / drag-to-reorder polish (basic reorder optional; fancy DnD is YAGNI for v1).
- Changing columns on the docker/cron context viewers (this targets the Explorer; other viewers opt in later if desired).
- Server-side column projection (the backend already returns full LogLine + fields; this is a display concern).

## Decisions in this stage

### Design phase will confirm

- The column-set data model + the `<LogViewer>`/`LogLineList` API for optional columns (must stay backward-compatible for docker/cron).
- Where selected columns live (URL param vs STAGE-004-015 persistence vs both) — coordinate with 015's ExplorerState shape.
- Table vs flex-row layout for multi-column rendering on desktop, and mobile behavior (recommended: collapse extra columns or show a compact subset on narrow viewports).
- Which fields are offered by default vs discovered dynamically.

### Locked Design Decisions (inherited from brainstorming 2026-05-28 + user request 2026-06-01)

- **D-COLUMNS-OPT-IN** — configurable columns are an Explorer feature; the shared `<LogViewer>` gains an OPTIONAL columns prop so docker/cron viewers are unaffected by default.
- **D-COLUMNS-FROM-DISCOVERY** — the selectable field list comes from scope-aware field discovery (STAGE-004-018) plus the known promoted fields; hence sequenced after 018.

## Design Phase

**Design phase complete: 2026-06-09.**
**Status**: [x] Design (Complete)

### Design Notes (STAGE-004-018B — recorded at Design completion)

**Architecturally non-trivial — brainstormer (Opus) consulted on 6 decisions; main agent made the final calls. Overarching theme: backward-compat guard (6 production LogViewer/LogLineList callers must render identically) + YAGNI above all.**

**D1 — Column model + value resolution (CHOSEN: `string[]` + pure resolver).**
Columns = an ordered `string[]` of field names. A PURE `getColumnValue(line, field): string` resolves a cell: check promoted top-level fields (`severity`/`host`/`service`) FIRST, then `line.fields[field]`, return `''` on missing (empty cell, no crash — success criterion 4). `auto` column widths (no width hint). The `severity` column reuses the EXISTING severity tinting already in LogLineList.tsx. The `string[]` model equals the URL serialization shape (collapses a serialization layer).
- Rejected `ColumnDescriptor { field; label?; width? }[]` — speculative label/width (YAGNI), and a structured descriptor fights the CSV URL shape.

**D2 — Multi-column LAYOUT (CHOSEN: separate `LogRowWithColumns` component; original `LogRow` byte-for-byte UNTOUCHED).**
This is the highest-leverage decision. `LogLineList` picks the renderer ONCE based on `columns?.length`: no columns → the EXISTING `LogRow` (unchanged code path → structural back-compat guarantee for the 6 callers); columns present → a new sibling `LogRowWithColumns`. The new component uses inline `style={{ gridTemplateColumns }}` = `auto` + `repeat(N,auto)` + `1fr` (timestamp | configured columns | message takes 1fr). The template is derived from `columns` (NOT per-line) so columns align across rows. Shared cell logic (timestamp cell, message cell with ANSI parse, severity-tint) extracted into shared helpers used by both row components. NO header/label row (YAGNI — the picker is the affordance; documented as a future enhancement). Wrap/`min-w-max` behavior unchanged → extra auto columns trigger horizontal scroll when !wrap (this IS the desired mobile behavior, D5).
- Rejected (a) dynamic grid-template on the single existing LogRow and (c) unified always-N renderer (N=0) — both make the back-compat DOM an EMERGENT runtime property (a future refactor of the conditional silently regresses the 6 callers). The separate component makes it STRUCTURAL.

**D3 — Column picker UX `<ColumnsControl>` (CHOSEN: shadcn Popover + grouped checkbox list).**
An icon Button (WrapIconToggle pattern: `<Tooltip><Button size="sm" variant={...?'secondary':'ghost'} className="h-8 w-8 p-0" aria-pressed aria-label>`) in Row C of LogsExplorerBody, inserted AFTER the Wrap toggle (~line 731). Popover body = a scrollable checkbox list: a fixed "common" group (service/host/severity) ALWAYS shown + discovered fields from the EXISTING `useLogsFieldsQuery(expr,startIso,endIso,servicesCsv)` call (line 254, added by 018A — reuse it, React Query dedupes). CRITICAL correctness: render the UNION of (discovered fields ∪ currently-configured columns) so a column whose field left the discovered set for the current scope STILL appears as a checked toggle and can be removed. Toggling calls onChange with the field added/removed. Column order = insertion/picker order.
- DESCOPE reorder for v1 (card says "ideally"; DnD already out of scope; skip up/down arrows too — real a11y/array-move code for a nice-to-have). Documented as future.
- Rejected DropdownMenu (awkward for multi-toggle, closes per item) and overloading FieldsDiscoveryPanel (conflates filtering/inspection with presentation; sidebar may be closed).

**D4 — Persistence (CHOSEN: URL-only).**
A `columns` CSV param threaded through the STAGE-004-015 URL pattern: add `parseColumnsParam(raw): string[]` (split on `,`, trim, drop empties) and `columnsToCsv(cols): string` beside `parseServicesParam`/`identitiesToServicesCsv` in `api/logs.ts`/`router.tsx`; parse in `router.tsx` `/logs/query` validateSearch (~line 170-187); serialize in `LogsExplorerPage.tsx` `writeUrl()` (~line 112-148) with `if (columns.length > 0) next.columns = columnsToCsv(columns)`. Matches q/logsql/since/services (consistent Explorer state). Empty/absent param → `[]` → triggers the no-columns back-compat render path. CSV is safe (log field names are identifiers; no commas). Shape stays forward-compatible with 015's ExplorerState (a plain string[] is what 015 would store).
- Rejected URL + 015 dual-persistence — premature; two sources of truth → precedence rules + extra tests. 015 wiring is a clean future add.

**D5 — Mobile (CHOSEN: horizontal scroll, no breakpoint logic).**
Rely on the existing `!wrap` + `min-w-max` behavior — the same mechanism wide log lines already use to scroll horizontally. All configured columns stay accessible on mobile. NO useMediaQuery gate.
- Rejected hide-columns-below-768px (silently disappearing columns = confusing UX; new stateful branch + tests) and vertical stacking (substantial new layout). Both are cheap additive follow-ups if Refinement shows mobile scroll is annoying — not built speculatively.

**D6 — Testability (CHOSEN: back-compat structural guard first).**
THE key test: render `<LogLineList lines={[...]} />` with NO `columns` prop and assert the multi-column path is ABSENT (e.g. `LogRowWithColumns` testid not in DOM) + the existing `grid-cols-[auto_1fr]` timestamp+message structure is present — a STRUCTURAL guarantee the 6 callers' path is unchanged. One parallel guard at `<LogViewer>` level. Pure-function tests for `getColumnValue` (promoted / fields-bag / missing→'') and the CSV round-trip (`parseColumnsParam`/`columnsToCsv`, empty→[]). Render tests: columns render expected values; severity cell carries the tint class; a line missing a configured field → empty cell, no crash; inline gridTemplateColumns reflects auto+repeat(N)+1fr order. Picker tests: promoted toggles always present; discovered fields listed (mock useLogsFieldsQuery); toggle add/remove calls onChange; stale configured column still removable. Extend the EXISTING `line(message, severity)` factory in LogLineList.test.tsx to accept a `fields` arg (it already sets `fields: {}`). All jsdom-fine (NOT CodeMirror).

**Verified anchors (all CONFIRMED this session, no corrections):** LogLine has top-level severity/host/service + `fields: dict[str,Any]` bag → frontend-only (no backend change). LogLineList LogRow grid is `grid grid-cols-[auto_1fr] items-start gap-x-2` (line 95). 6 back-compat call sites: Explorer (gets columns), CronRunLogViewer, DockerContainerLogsViewerBody, RecentCrashesSection, RecentHealthcheckIncidentsSection, SignatureDetailPage (renders LogLineList directly). `useLogsFieldsQuery` already in Body (line 254, 018A). URL helpers parseServicesParam/identitiesToServicesCsv/writeUrl confirmed.

**No new npm dependency. No backend work. Frontend-only → Refinement 3a only (Desktop + Mobile).**
