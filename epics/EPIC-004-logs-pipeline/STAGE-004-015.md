# STAGE-004-015: Explorer state persistence (last query / range / scroll position across navigation)

## Status: Refinement

## Overview

The Explorer remembers its last state — current query, mode (plain vs advanced), selected services, time range, and scroll position — when the user navigates away and returns. URL state from STAGE-004-010 covers query + range (deep-linkable), but doesn't cover scroll position or transient UI state. This stage closes the gap.

**Vertical slice:** user runs a search, scrolls down to line 200, clicks elsewhere in the app, navigates back to `/logs` — sees the same query, results, and scroll position. URL deep-link to `/logs?q=foo&since=1h` still works.

## Concrete deliverables

### 1. Persistence layer

`apps/ui/src/lib/explorerState.ts` (new) — localStorage-backed state:

```typescript
interface ExplorerState {
  logsQl: string
  plainText: string
  advancedMode: boolean
  selectedServices: string[]
  rangeKind: 'preset' | 'custom'
  sincePreset?: string
  rangeStart?: string  // ISO
  rangeEnd?: string  // ISO
  scrollPosition: number  // pixels from top
  cursor?: string  // current pagination cursor
  lastVisitedAt: number  // unix ms
}

function saveExplorerState(state: ExplorerState): void
function loadExplorerState(): ExplorerState | null
```

### 2. URL takes precedence over persisted state

When `/logs` mounts:
1. If URL has query params → URL wins, persisted state is overwritten
2. Else if persisted state exists and is < 7 days old → restore from persisted
3. Else → default empty state

`lastVisitedAt` older than 7 days is ignored (stale; user has clearly moved on).

### 3. Save triggers

State is persisted on:
- Every executed query
- Every scroll event (debounced 200ms)
- Every selectedServices change
- Every mode toggle

### 4. Scroll restoration

When state loads from persistence, the Explorer's results pane scrolls to `scrollPosition` after the initial results render. Uses React's `useLayoutEffect` to avoid flicker.

If the persisted cursor + query + range still produce the same results, scroll position is meaningful. If not (e.g., new lines have arrived), scroll position falls back to top.

### 5. Tests

- `apps/ui/src/lib/__tests__/explorerState.test.ts` — save / load round-trip, stale-state ignored
- Component test: navigate away, navigate back, verify state restored

## Out of scope for this stage

- Cross-device sync — Option B from STAGE-004-014 if it becomes a thing; YAGNI now
- State snapshots beyond the most-recent (history covers it)
- Restoring exact scroll position when results have changed — gracefully fall back to top

## Decisions in this stage

### Design phase will confirm

- **Stale-state TTL** — 7 days recommended
- **Scroll debounce ms** — 200ms recommended
- **Whether to persist cursor too** — yes if we want exact scroll restore; recommended yes

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-STATE-LOCALSTORAGE-V1** — localStorage; cross-device sync deferred.
- **D-URL-PRECEDENCE** — URL params override persisted state when both present.

## Design Phase

**Design phase complete: 2026-06-02**

**Selected approach (Design phase, decided 2026-06-02 — Option A: pure lib + inline Page seeding + scroll wiring colocated in Body):**

- **Pure core module `apps/ui/src/lib/explorerState.ts`**: ExplorerState interface (per the card), `saveExplorerState(state)`, `loadExplorerState(): ExplorerState | null` (with 7-day TTL check inside), `resolveInitialExplorerState(urlParams, persisted)` precedence resolver, and `patchExplorerState(partial)` (read-modify-write merge). Mirror queryHistory.ts/useTimezonePreference.ts patterns: SSR guard (`typeof window === 'undefined'`), try/catch on JSON.parse, silent setItem failure on quota. STORAGE_KEY `homelab-monitor:logs-explorer-state`. PURE + unit-testable (satisfies explorerState.test.ts).

- **URL precedence — ALL-OR-NOTHING**: `resolveInitialExplorerState`: if the URL has ANY of q/logsql/since/start/end/services → use URL entirely (ignore persisted); else if persisted exists AND `Date.now() - lastVisitedAt <= 7 days` → restore persisted; else default empty. Rationale: per-field merge produces incoherent hybrids and contradicts deep-link intent (a shared link must reproduce exactly what it encodes). The Page already reads all six params in one place, so "any present" is one boolean.

- **TTL in loadExplorerState**: lastVisitedAt unix ms; on load if expired (>7d) → return null (resolver treats null as "no persisted"). Keeps TTL in one spot, unit-tested.

- **Page seeding (LogsExplorerPage.tsx)**: change the existing useState initializers to consume one `resolveInitialExplorerState(urlParams, loadExplorerState())` computed once at mount (destructured into the initializers). Add a write-back useEffect keyed on [advancedMode, committedPlainText, committedLogsQl, range, selectedIdentities] that calls patchExplorerState({...query fields}). Reuse 014's HistoryEntry-style range serialization (preset OR start/end ISO); the existing reconstructFromPayload encodes the inverse range logic.

- **Scroll capture + restore — colocated in LogsExplorerBody** (NOT the Page), because flatLines/logs.isLoading and the scroll `<pre>` live there (the decisive data-locality reason Option A beats a Page-level hook):
  - Capture: a scroll ref on the `<pre>` (threaded LogsExplorerBody → LogViewer → LogLineList, 2 hops, via a scrollRef/onScroll prop pair) + a 200ms-debounced scroll handler that patchExplorerState({ scrollPosition }).
  - Restore (the crux — restore-TIMING): naive useLayoutEffect on mount restores to 0 because useInfiniteQuery hasn't populated the `<pre>` yet. SOLUTION: a `pendingRestoreRef` (init true if persisted scrollPosition > 0) + a useLayoutEffect keyed on flatLines.length; when `flatLines.length > 0 && pendingRestoreRef.current` → scrollTo({top: scrollPosition}) ONCE, then set pendingRestoreRef.current=false. After that, scroll handlers only SAVE, never re-restore (don't fight the user).
  - Staleness/mismatch: **naive scrollTop for v1** — if results changed between visits, scrollTop may be off; accept it (the card explicitly scopes graceful-degradation-on-changed-results OUT and accepts "falls back to top"). Structure the restore as an isolated strategy so a future line-anchor restore (Option C) is a swappable upgrade.

- **"cursor": INCLUDE the `cursor?: string` field in the persisted shape (per the card's stated ExplorerState interface) for forward-compat, but v1 does NOT resume pagination from it.** useInfiniteQuery refetches initialPageParam on mount and can't resume mid-pagination without bespoke initialData (out of scope). v1 scroll restore is pixel-based within page 1 of results. Mirrors 014's reserved `resultCount` field.

- **Save model — single object, read-modify-write `patchExplorerState(partial)`**: load() ?? {} → {...prev, ...partial, lastVisitedAt: Date.now()} → save(). Query-field writer patches the query fields; scroll writer patches scrollPosition. Read-modify-write prevents the debounced scroll-save from clobbering a query change that landed in the 200ms window (and vice versa).

- **Save triggers**: write-back useEffect on query state (executed query / services change / mode toggle, via the keyed effect); debounced (200ms) scroll handler for scrollPosition.

**Inherited locked decisions (unchanged):**
- D-STATE-LOCALSTORAGE-V1 (localStorage; cross-device deferred)
- D-URL-PRECEDENCE (URL overrides persisted when present)

**ExplorerState shape (per card):** { logsQl, plainText, advancedMode, selectedServices (or selected_services as {service,source_type}[] — match the serialization used; the card lists selectedServices:string[] but the codebase uses ServiceIdentity[] — use the ServiceIdentity {service,source_type}[] shape consistent with 013/014 serialization, document the deviation from the card's string[] as a deliberate alignment with the existing identity model), rangeKind, sincePreset?, rangeStart?, rangeEnd? (ISO), scrollPosition (px), cursor? (reserved v1), lastVisitedAt (unix ms) }.

**Look-ahead caveats (record; do NOT implement now):**
- STAGE-004-018B (configurable visible columns → variable row heights invalidate pixel scrollTop → motivates the Option C line-anchor restore upgrade).
- STAGE-004-024 (live-tail auto-scroll conflicts with persisted scroll → those stages must add a precedence rule, e.g. live-tail suppresses restore). The pendingRestoreRef restore-once semantics already prevent persisted scroll from fighting ordinary user scrolling.

**Seed data:** none (auto-populated as the user interacts; manual testing in Refinement exercises it).

**Cross-cutting:** frontend-only (no backend/migration). >=80% UI functions coverage. Tests: explorerState.test.ts (save/load/TTL-expiry/precedence resolver pure functions) + a component nav round-trip test (navigate away → back → state + scroll restored; URL-precedence; first-visit default).
