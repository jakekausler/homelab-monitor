# STAGE-004-029: Signature annotations (timestamped notes)

## Status: Complete

## Overview

Add free-form annotations to each signature: a chronological list of `(timestamp, author, note)` entries so the user can leave investigation notes like "2026-04-15 — investigated; root cause was HA config typo". Surface in the signature drill-in panel from STAGE-004-028.

**Vertical slice:** in the Signature drill-in panel, an "Annotations" section lists prior notes + an input to add a new one. Annotations persist in SQLite, ordered chronologically.

**Spec references:** master spec §9.2 (signature catalog detail).

## Concrete deliverables

### 1. SQLite schema

`apps/monitor/homelab_monitor/kernel/db/migrations/` — new alembic migration:

```sql
CREATE TABLE log_signature_annotations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    template_hash TEXT NOT NULL,
    service_key TEXT NOT NULL,
    note TEXT NOT NULL,
    author TEXT NOT NULL,  -- the username (single-user system → always 'admin' or similar)
    created_at INTEGER NOT NULL,  -- unix ms
    FOREIGN KEY (template_hash, service_key) REFERENCES log_signatures(template_hash, service_key) ON DELETE CASCADE
);
CREATE INDEX idx_signature_annotations_sig ON log_signature_annotations(template_hash, service_key);
CREATE INDEX idx_signature_annotations_created ON log_signature_annotations(created_at);
```

### Design Decisions (resolved 2026-06-06)

Four open decisions resolved via brainstormer (Opus), grounded by reading source. The three D-ANNOTATIONS-* decisions remain locked.

**STALE-REFERENCE CORRECTION:** The card's deliverable references `SignatureDetailPanel.tsx`, but STAGE-028 renamed that to **`SignatureDetailPage.tsx`** (a deep-linked full-page detail route, not a side panel). The annotations section is added to **`SignatureDetailPage.tsx`** after the "Recent Samples" section.

**DECISION A — Author storage: denormalize `author TEXT`** *(CHOSEN)*
- Store the author as a denormalized `author TEXT` column, captured from `_user.username` (the session User) at creation time. NO `created_by` FK to users.
- Rationale: the frontend has NO id→username lookup (only `/api/auth/me` for the logged-in user), so a `created_by` int FK would force a backend join or a new users endpoint — overkill for a single-user tool. Denormalized username gives correct point-in-time audit semantics and renders directly. No users-table coupling.
- *Rejected A1 (created_by FK + ON DELETE SET NULL):* no frontend id→username path exists; forces a join/new endpoint. *Rejected A3 (both):* extra columns, no benefit single-user.

**DECISION B — Route shape: NESTED for all three** *(CHOSEN)*
- `GET  /api/logs/signatures/{template_hash}/{service_key}/annotations` (list, no CSRF)
- `POST /api/logs/signatures/{template_hash}/{service_key}/annotations` {note} (CSRF; author from session)
- `DELETE /api/logs/signatures/{template_hash}/{service_key}/annotations/{annotation_id}` (CSRF; handler scopes delete by composite key + id for defense-in-depth)
- Rationale: the card's suggested FLAT `DELETE /signatures/annotations/{id}` is positionally identical (`/signatures/<seg>/<seg>`) to the existing `/signatures/{template_hash}/{service_key}` GET — a latent route ambiguity (literal `annotations` masquerading as a `{template_hash}` slot). The 4-segment nested form is unambiguous, consistent with GET/POST, and the composite key is already in frontend scope. Declared AFTER the existing `/samples` route.
- *Rejected B1 (flat delete):* structural ambiguity with the signature GET; inconsistent with nested GET/POST; fragile for future routes.

**DECISION C — UX: invalidate-on-success + window.confirm** *(CHOSEN)*
- Mutations use invalidate-on-success (mirrors every existing mutation: signatures.ts, savedLogQueries.ts). POST add → invalidate annotations-list query → refetch shows the new note (sub-100ms on a local tool; feels instant). DELETE → `window.confirm(...)` (precedent: SavedQueriesPanel.tsx:68) → DELETE → invalidate. Add-button disabled while the note is empty/whitespace OR the mutation is pending.
- *Rejected C2 (true optimistic cache write + rollback):* temp-id/rollback surface for no perceptible gain on a local single-user tool; diverges from every existing mutation pattern. (The card's "optimistic insert" is satisfied by C1's fast refetch.)

**DECISION D — Validation: 1–2000 chars, whitespace-stripped** *(CHOSEN)*
- Backend authoritative: Pydantic `note: str = Field(min_length=1, max_length=2000)` PLUS a field_validator that `.strip()`s and rejects whitespace-only (store the stripped value).
- Frontend (UX only): `<textarea maxLength={2000}>` + submit-button disabled when `value.trim().length === 0`.

**Schema (migration 0035_log_signature_annotations, down_revision 0034):**
`log_signature_annotations(id INTEGER PRIMARY KEY AUTOINCREMENT, template_hash TEXT NOT NULL, service_key TEXT NOT NULL, note TEXT NOT NULL, author TEXT NOT NULL, created_at TEXT NOT NULL)` + composite FK `(template_hash, service_key) REFERENCES log_signatures(template_hash, service_key) ON DELETE CASCADE` (auto-cascades; PRAGMA foreign_keys=ON is enabled per-connection) + indexes on `(template_hash, service_key)` and `created_at`. created_at is ISO-8601 UTC TEXT via `utc_now_iso()` (matches saved_queries_repo, NOT the unix-ms drain *_at columns). **ADD `log_signature_annotations` to EXPECTED_TABLES in tests/test_db_migrations.py (32 → 33).**
