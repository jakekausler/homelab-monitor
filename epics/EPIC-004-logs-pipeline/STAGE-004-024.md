# STAGE-004-024: Frontend tail mode (Explorer consumes SSE; toggle button)

## Status: Complete

## Overview

Add a "Live tail" toggle to the Logs Explorer. When on, the page subscribes to the SSE endpoint from STAGE-004-023 and renders new lines as they arrive (appended to the line list). Pagination (Load older) is disabled while tailing; resuming pagination requires turning tail off.

UX details: visible "Live" indicator, line buffer cap to avoid runaway memory, pause/resume button, auto-scroll-to-bottom when new lines arrive (with a "pause auto-scroll" affordance when the user manually scrolls up).

**Vertical slice:** in the Explorer, click "Live tail" → results pane switches to streaming mode; new lines appear at the bottom; auto-scroll keeps newest in view; status indicator shows "Live · 142 lines/min".

**Spec references:** master spec §9.2 (live tail).

## Concrete deliverables

### 1. SSE consumer hook

`apps/ui/src/api/logsTail.ts` (new) — `useLogsTail(expr, services)` hook:

- Opens an EventSource to `/api/logs/tail?expr=...&services=...`
- Pushes received `line` events into an in-memory buffer
- Caps buffer at 5,000 lines (configurable); drops oldest
- Exposes `{lines, isLive, error, linesPerMin, pause(), resume()}` shape

Reconnect logic: on transport drop, retry with exponential backoff (max 30s).

### 2. Tail mode toggle on Explorer

`LogsExplorerPage.tsx`:

- "Live tail" button next to Search button
- Click → switches the `useLogs` source from `useLogsExplorerQuery` (paginated) to `useLogsTail` (streaming)
- Pagination controls hidden / disabled while tailing
- Range picker disabled while tailing (live = "from now"; previous range becomes meaningless)
- LogsQL + services filter still apply to the stream

### 3. Auto-scroll behavior

- New lines auto-append to the bottom of `<LogViewer>`
- Auto-scroll to bottom when new lines arrive AND user is currently within 100px of bottom (the "live tail" sticky behavior)
- If user scrolls up beyond that threshold, auto-scroll disengages and a "Resume auto-scroll" button appears

### 4. Tail status indicator

Small bar above the line list while tailing:

```
🟢 Live · 142 lines/min · 1,243 buffered · [Pause] [Stop tail]
```

- Pause: stops appending (but stream continues — buffer fills until cap, then drops oldest)
- Stop tail: closes the SSE connection; switches back to paginated mode (last buffered lines stay visible)

### 5. Error handling

- VL unavailable / SSE error event: show banner above line list; auto-retry after 10s
- Transport drop: auto-reconnect with backoff
- Connection limit (503): show explanatory message, suggest closing other tails or trying later

### 6. Tests

- `apps/ui/src/api/__tests__/logsTail.test.ts` — EventSource interaction (mocked), reconnect logic, buffer cap
- `__tests__/LogsExplorerPage.test.tsx` — toggle tail mode, pagination disabled during tail, auto-scroll behavior

## Out of scope for this stage

- Tail mode in docker / cron per-context viewers — could add via `<LogViewer>` opt-in prop; YAGNI for v1 (user can Open in Explorer + tail)
- Tail history persistence — buffer is in-memory, lost on navigate-away
- Multiple simultaneous tails (e.g., side-by-side) — single tail per Explorer
- Sound notification on new lines — YAGNI

## Design Decision (Selected Approach)

**Selected: A2 + B1 + C1 + ephemeral state + dedicated status bar.** Purpose-built `useLogsTail` EventSource hook (mirroring the existing `sse.ts` lifecycle) + a `useLogs`-shaped adapter feeding the EXISTING LogViewer + an auto-scroll-sticky effect in LogsExplorerBody + ephemeral `isTailing` state + a dedicated `<LiveTailStatusBar>`.

**New files:**
- `apps/ui/src/lib/logsTail.ts` — `useLogsTail(expr, services, opts?)`. Mirrors `apps/ui/src/lib/sse.ts`'s lifecycle (injectable `factory` for EventSource, `sourceRef`/`timerRef` guards, exponential backoff 500ms→30s, unmount cleanup, manual-reconnect flag), but with a tail-specific data model:
  - THREE event listeners: `line` (parse LogLine JSON → push to buffer), `dropped` (parse `{count}` → accumulate `droppedCount`), `error` (parse `{code,message}` → structured error, distinct from EventSource transport `onerror`).
  - Accumulating capped buffer in a `useRef<LogLine[]>` (cap 5000, drop-oldest). Incoming `line` events push to the ref synchronously (NO burst-drops), flushed to React state via an rAF/throttled batch (≤ ~once/frame) to avoid 200-setState/sec thrash.
  - `pause()`/`resume()` via a `pausedRef`: paused = keep EventSource open + keep filling the ref buffer, but STOP flushing to state. resume re-enables flush.
  - `linesPerMin` = rolling 60s window of arrival timestamps. `buffered` = current buffer length. `droppedCount` = server-side dropped lines.
  - **Pre-flight `fetch` (injectable `fetchImpl`) BEFORE opening EventSource** to resolve the "EventSource onerror has no HTTP status" problem: GET the tail URL, read status — 503 → `{code:'over_cap', retryAfter: Retry-After header}` (do NOT open EventSource), 422 → `{code:'invalid_logsql'}`, 502 → `{code:'upstream_unavailable'}`, 200 → proceed to open the real EventSource. (BUILD CAVEAT: the 023 tail endpoint is the SAME streaming GET — a 200 "probe" would itself open a stream + consume a connection slot. The 023 handler does cap-check BEFORE streaming, so a 503 returns fast WITHOUT consuming a slot, but a 200 probe acquires+opens. Resolve in Build: either (i) the pre-flight uses a HEAD or a dedicated lightweight status, (ii) the pre-flight's 200 response IS the stream we keep (don't double-open — reuse the connection), or (iii) accept that detecting 503/422/502 needs the request and reuse-or-immediately-reopen carefully. Simplest correct: open ONE EventSource; on its first `error` with no prior `open`, do a single status-probing `fetch` to classify why, then surface 503/422/502 — avoids the double-open entirely.)
  - Honors an `enabled` flag (only opens when tailing) to respect rules-of-hooks at the call site.
  - Return: `{ lines: LogLine[]; isLive: boolean; status: 'connecting'|'open'|'error'; error: {code,message,retryAfter?}|null; linesPerMin: number; droppedCount: number; buffered: number; paused: boolean; pause(); resume(); reconnect() }`.
- `apps/ui/src/lib/logsTail.test.ts` — DIRECT hook test (renderHook + a `FakeEventSource` mirroring the `sse.test.ts` idiom + a fake `fetchImpl`). MUST exercise every branch for the functions≥80/lines80/branches75 global coverage gate (the hook is NOT excluded): line accumulation + cap eviction, dropped accumulation, structured error event, pre-flight/error-classification 503/422/502/200 branches, pause/resume, reconnect backoff. (A page test that MOCKS useLogsTail leaves the real hook at 0% — the direct test is mandatory.)
- `apps/ui/src/components/logs/LiveTailStatusBar.tsx` (+ `.test.tsx`) — presentational status bar: `🟢 Live · N lines/min · M buffered · [K dropped] · [Pause/Resume] [Stop tail]`; variants for Paused (⏸), Reconnecting (🔴 + failureCount), and 503 over-cap (🟠 + retryAfter). Props-in/DOM-out; assert each status variant + button callbacks.

**Changes to `apps/ui/src/routes/logs/LogsExplorerBody.tsx`:**
- Add ephemeral `const [isTailing, setIsTailing] = useState(false)` (NOT URL-persisted — live tail is transient; a `?tail=1` would auto-open a stream + burn a backend slot on every reload).
- Call `useLogsTail(expr, servicesCsv, { enabled: isTailing })`.
- In the `useLogs` adapter: when `isTailing`, return `{ lines: tail.lines.slice(-1000), isLoading: tail.status==='connecting', isError:false, error:undefined, logStatus: tail.lines.length ? 'available':'no_lines', hasMore:false }` (no `loadOlder` → "Load older" auto-hidden → satisfies D-PAGINATION-DISABLED-WHILE-TAILING structurally). RENDER-cap at last 1000 lines (retain 5000 in the hook) to bound the DOM — the app has NO virtualization (LogLineList maps every line to a DOM row), so 5000 rows would jank.
- Add a "Live tail" toggle button in control-bar Row C beside TimeRangeControl; dim/disable TimeRangeControl + Refresh while tailing (live = "from now", range is meaningless → reinforces live-OR-historical).
- **GATE the scroll-restore `useLayoutEffect`** (the one keyed on `[flatLines.length, restoreScrollTarget]` that scrolls to a saved `scroll_position`): early-return when `isTailing`; set `pendingRestoreRef.current = false` on tail entry. Also SKIP the debounced scroll-SAVE while tailing (so the historical scroll position survives a tail session).
- Add **sticky auto-scroll**: extend the existing `handleScroll` to compute `nearBottom = scrollHeight - scrollTop - clientHeight < 100` → set a `sticky` state (scroll-up disengages, scroll-to-bottom re-engages); add a `useLayoutEffect` keyed on `tail.lines.length` that `el.scrollTo({top: el.scrollHeight})` when `isTailing && sticky`. Show a floating "Resume auto-scroll" button when `isTailing && !sticky` (click → set sticky + scroll to bottom). The `!isTailing` gate on scroll-restore GUARANTEES the two layout effects are mutually exclusive — do NOT weaken it.
- On Stop tail: `setIsTailing(false)` (hook unmounts → closes EventSource) + bump `refreshNonce` (already documented as live-tail groundwork) so the historical query re-resolves to fresh "now" data; the `useLogs` source reverts to `useLogsQuery` (Load-older re-enabled). Buffered lines discarded on revert (v1; retention is a follow-up).
- Render `<LiveTailStatusBar>` between the header/histogram and `<LogViewer>` when `isTailing`.

**Auth/transport:** EventSource `withCredentials:true` sends the session cookie (same-origin via vite proxy in dev; monitor serves UI in prod). Mirrors `sse.ts`. Events are hand-typed (SSE isn't in the openapi-typed client).

**Status-bar spec:** `🟢 Live · N lines/min · M buffered · [Pause] [Stop tail]`. Pause = stop appending, keep buffer filling. Stop = close SSE, revert to paginated. Dropped chip when droppedCount>0. Reconnect/503 surfaced from the hook's status/error.

**Rejected alternatives:**
- **A1 (reuse `useSSE` + accumulate in a wrapper)** — rejected: `useSSE` overwrites a single latest `value`; a burst of line events between renders collapses to one → silent line loss. Also single-`topic` (tail needs 3 event types) and no pause.
- **A3 (extend `useSSE` to a generic accumulating mode)** — rejected: balloons a stable primitive's API + union return for one consumer, forces re-verifying the 2 existing useSSE consumers; tail-specific pause/dropped/rate/pre-flight semantics would pollute it. Violates single-responsibility + reversibility.
- **B2 (separate `<LiveTailViewer>` component)** — rejected: would duplicate LogLineList's ANSI/severity/wrap/timezone/inspector feature set. The adapter reuses LogViewer with zero changes.

**Residual risks (for Build):** (1) LogViewer perf — no virtualization exists; mitigate with the 1000-line render cap (+ throttle the flush cadence to ≤250ms if 1000 still janks). True windowing is out of scope. (2) The pre-flight/error-classification 503-detection must NOT double-open a stream or burn two connection slots (see BUILD CAVEAT above — simplest: open ONE EventSource, classify on first error via a single status fetch). (3) Keep the `!isTailing` gate intact for layout-effect mutual exclusion.

## Refinement Redesign (user feedback, 2026-06-05)

**User tested the first tail implementation in the prod rig and requested a significant UX redesign that expands scope into a general bidirectional windowed log pager. Design revised via brainstormer; going with the recommendation. Re-implementation in progress under Refinement.**

**Requested changes (authoritative):**
1. Tail KEEPS already-loaded historical logs; live lines append BELOW them (don't clear the view).
2. REMOVE the entire `LiveTailStatusBar` (lines/min, buffered, pause, separate stop button) — gone. The "Live tail" TOGGLE BUTTON turns GREEN when active (only live indicator); click again to stop.
3. Auto-scroll sticky, NO pause.
4. On STOP: FREEZE on-screen lines (historical + tailed), set the Explorer CUSTOM END TIME to the stop-moment, do NOT re-query/discard.
5. NEW general bidirectional windowed pager (applies to historical browsing too, not just tail — explicitly in-scope per user): 1000-line render cap as a FIFO. "Load older" trims from the END (newest) + turns OFF tailing. NEW "Load newer" button at bottom trims from the FRONT (oldest), hidden while tailing. Turning ON tail trims the FRONT for headroom. When the cap trims an end, show a message at that end ("Older lines removed — Load older" at top / "Newer lines removed — Load newer" at bottom).

**This SUPERSEDES parts of the original design:** the `LiveTailStatusBar` is removed; D-PAGINATION-DISABLED-WHILE-TAILING is partially superseded (tail keeps historical + "Load newer" hidden while tailing, but "Load older" is allowed and turns tail off).

**Selected redesign:**
- **New `apps/ui/src/lib/useWindowedLogs.ts`** — a `useReducer` hook owning the single rendered line list + 1000-cap FIFO. State `{lines: LogLine[], trimmedOlder: boolean, trimmedNewer: boolean}`. Ops: `reset(lines)` (wholesale replace, clear flags), `prependOlder(older)` (Load-older; over cap → trim END, set trimmedNewer), `appendNewer(newer)` (Load-newer + tail appends; over cap → trim FRONT, set trimmedOlder; dedup on composite key `timestamp|message|severity|host|service`), `trimFrontTo(n)` (tail-start headroom; sets trimmedOlder if dropped). An op adding to an end clears that end's flag; trimming an end sets it. CAP=1000.
- **"Load newer" needs NO backend change** (KEY de-risk): `/api/logs/query` returns the LATEST N lines in any `[start,end]` window, so Load-newer = a window-shift query `[newestShownTimestamp, endIso]` (no cursor) + dedup. Add `fetchNewerLogs(expr,start,end,services)` to `apps/ui/src/api/logs.ts`. Known limitation (documented, accepted for a homelab pager): if >500 (backend default page) new lines accumulated since newestShown, the oldest of the gap are skipped — converges via repeat Load-newer or tail; NOT a crash, NOT a backend dependency.
- **On-Stop freeze:** add a `frozen` state flag in LogsExplorerBody that gates the historical-query→buffer dispatchers (NOT the query `enabled` — that would kill sibling histogram/services/fields queries). On stop: setIsTailing(false), capture stopMoment, setFrozen(true), set custom range end = stopMoment (writes URL). Un-freeze on the next genuine user query action (submit/range-change/refresh/load-older/services-change). The frozen on-screen lines are NOT replaced by the background refetch.
- **Slim `useLogsTail`:** drop UI-facing pause/linesPerMin/buffered/droppedCount/paused/resume. New surface `{status, error, reconnect()}` + an `onLines?: (batch: LogLine[]) => void` rAF-batched callback the body wires to `dispatch(appendNewer(batch))`. Keep the backoff + failure-classification (503/422/502) logic verbatim; drop the 5000-line retention buffer (the windowed buffer is now source of truth). Keep consuming `dropped` events (no UI surface).
- **Minimal error surface (status bar gone):** green toggle = the live indicator (aria-pressed + emerald button variant); tail errors (503 over-cap/422/502/transient) surface in a single inline `<p role="alert">` chip below the control rows (reuse the existing generic-API-error slot pattern). No toast infra.
- **Trimmed indicators:** mount inline as the FIRST child (top, when trimmedOlder) and LAST child (bottom, when trimmedNewer) of the scroll region — reuse `LogBanner`. Scrolling to that end naturally reveals them (no scroll listener needed).
- **Extend `UseLogsResult`** (types.ts) with `trimmedOlder?/trimmedNewer?/hasNewer?/isLoadingNewer?/loadNewer?`. **`LogViewer`** renders the top trimmed-older banner + existing Load-older at top; after the line list a `LoadNewerButton` (ChevronDown, `data-testid="load-newer"`, gated `loadNewer != null && hasNewer`) + the bottom trimmed-newer banner. `hasNewer = !isTailing` (simple; empty load-newer is a no-op).
- **Auto-scroll coexistence:** re-key the sticky effect on `windowed.lines.length` (tail no longer exposes lines); front-trim doesn't move the bottom scroll target → auto-scroll-to-bottom stays correct; keep the `!isTailing` restore gate; restore gates on `windowed.lines.length > 0`. "Load older" while tailing → setIsTailing(false) first.

**Files:** CREATE `lib/useWindowedLogs.ts` (+ test). MODIFY `lib/logsTail.ts` (slim + onLines), `api/logs.ts` (fetchNewerLogs), `routes/logs/LogsExplorerBody.tsx` (adopt windowed buffer, frozen flag, green toggle, inline error, load-newer, remove status bar render), `components/logs/types.ts` (extend UseLogsResult), `components/logs/LogViewer.tsx` (load-newer + trimmed banners). DELETE `components/logs/LiveTailStatusBar.tsx` + its test + the import.

**Residual risks:** (1) Load-newer >500-line gap (documented limitation). (2) No server line ID → heuristic composite-key dedup (consistent with existing index-based render keys). (3) LogViewer perf at 1000 non-virtualized rows (same cap as before; eyeball in Refinement; virtualization is a separate stage). (4) frozen-flag user-action disambiguation (picker is pointer-events-none during tail).
