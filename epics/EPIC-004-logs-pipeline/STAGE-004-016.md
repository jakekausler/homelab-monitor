# STAGE-004-016: Field inspector (click a line → side panel with parsed fields)

## Status: Not Started

## Overview

Click any log line in `<LogViewer>` → side panel opens showing all parsed fields of that line (typed core fields + the `fields: dict` bag). Each field row shows name / value with a copy-to-clipboard button and an "Add to filter" button that injects `field:"value"` into the current query.

Works in any `<LogViewer>` consumer (docker viewer, cron viewer, Explorer). Most valuable in the Explorer but a quality-of-life win everywhere.

**Vertical slice:** click a log line → side panel slides in from the right with a table of `timestamp / severity / service / host / compose_project / compose_service / run_id / cron_fingerprint / [any other fields the line carries]`. Click "Add to filter" on `compose_service` → query updates to filter on that value.

**Spec references:** master spec §9.2 (logs explorer).

## Concrete deliverables

### 1. `<FieldInspectorPanel>` component

`apps/ui/src/components/logs/FieldInspectorPanel.tsx` (new) — right-side slide-in panel:

```
+--------------------------------+
| Line at 2026-05-28 12:34:56  X |
+--------------------------------+
| Field         | Value          |
+--------------------------------+
| timestamp     | 2026-05-28...  |
| severity      | error          |
| service       | home-assistant |
| host          | hassio         |
| compose_proj  | homelab-monitor|
| compose_serv  | monitor        |
| message       | (full message) |
| ... more ...  |                |
+--------------------------------+
[Copy field] [Add to filter]
```

Each row has hover affordances for Copy + Add-to-filter actions.

### 2. `<LogViewer>` integration

`<LogViewer>` becomes click-aware:
- Click anywhere on a log-line row → opens `<FieldInspectorPanel>` for that line
- Currently-inspected line is highlighted
- Click again on the same line OR click the panel × → closes
- Click a DIFFERENT line → swaps panel contents (no close/reopen flicker)

Caller opts in via a new `<LogViewer>` prop:
```typescript
fieldInspectorEnabled?: boolean  // default false; Explorer opts in
```

(Docker + cron viewers can opt in later if desired.)

### 3. "Add to filter" semantics

When clicked, the field's value is added to the current Explorer query as `field:"value"`. Uses the separate-state filter composition from STAGE-004-012 (stream picker pattern) so it works uniformly in plain + advanced mode.

For fields that aren't queryable (e.g., the `message` itself), "Add to filter" injects `_msg:"<substring>"` instead — Design phase confirms.

### 4. Copy-to-clipboard

Uses the project's existing clipboard utility (if one exists; otherwise add a small helper using the Clipboard API with a fallback toast). 100% test coverage.

### 5. Tests

- `__tests__/FieldInspectorPanel.test.tsx` — renders fields, copy button works (mocked clipboard), add-to-filter callback fires
- `__tests__/LogViewer.test.tsx` — click-to-inspect interaction, swap behavior

## Out of scope for this stage

- Nested field display (e.g., `fields.compose.project` rendered as a tree) — basic flat view; nested view comes with STAGE-004-017 / STAGE-004-018
- Field statistics (e.g., "this field has 12 distinct values in current query window") — out of scope; could be a follow-up
- JSON pretty-printing for fields whose values are JSON objects — could be added inline; YAGNI for v1

## Decisions in this stage

### Design phase will confirm

- **Panel position** — right slide-in (recommended), bottom drawer, or modal? Recommend right slide-in (matches Karma's alert detail pattern)
- **Mobile behavior** — full-screen overlay on narrow viewports
- **Field ordering** — fixed canonical order for typed core fields, then alphabetical for `fields` bag

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-INSPECTOR-CLICK-TO-OPEN** — click any line, panel opens for that line.
- **D-INSPECTOR-OPT-IN** — `<LogViewer>` prop controls whether inspector is enabled; Explorer opts in; docker/cron viewers can opt in later.
- **D-ADD-TO-FILTER-SEPARATE-STATE** — composes via separate filter state, consistent with stream picker pattern.
