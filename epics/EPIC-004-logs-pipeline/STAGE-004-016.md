# STAGE-004-016: Field inspector (click a line → side panel with parsed fields)

## Status: Complete

## Overview

Click any log line in `<LogViewer>` → side panel opens showing all parsed fields of that line (typed core fields + the `fields: dict` bag). Each field row shows name / value with a copy-to-clipboard button and an "Add to filter" button that injects `field:"value"` into the current query.

Works in any `<LogViewer>` consumer (docker viewer, cron viewer, Explorer). Most valuable in the Explorer but a quality-of-life win everywhere.

**Vertical slice:** click a log line → side panel slides in from the right with a table of `timestamp / severity / service / host / compose_project / compose_service / run_id / cron_fingerprint / [any other fields the line carries]`. Click "Add to filter" on `compose_service` → query updates to filter on that value.

**Spec references:** master spec §9.2 (logs explorer).

### 2. `<LogViewer>` integration

`<LogViewer>` becomes click-aware:
- Click anywhere on a log-line row → opens `<FieldInspectorPanel>` for that line
- Currently-inspected line is highlighted
- Click again on the same line OR click the panel × → closes
- Click a DIFFERENT line → swaps panel contents (no close/reopen flicker)

Caller opts in via a new `<LogViewer>` prop:
```typescript
fieldInspectorEnabled?: boolean  // default false; Explorer opts in
```

(Docker + cron viewers can opt in later if desired.)

### 3. "Add to filter" semantics

When clicked, the field's value is added to the current Explorer query as `field:"value"`. Uses the separate-state filter composition from STAGE-004-012 (stream picker pattern) so it works uniformly in plain + advanced mode.

For fields that aren't queryable (e.g., the `message` itself), "Add to filter" injects `_msg:"<substring>"` instead — Design phase confirms.

### 4. Copy-to-clipboard

Uses the project's existing clipboard utility (if one exists; otherwise add a small helper using the Clipboard API with a fallback toast). 100% test coverage.

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-INSPECTOR-CLICK-TO-OPEN** — click any line, panel opens for that line.

## Design Phase — Chosen Approach (2026-06-03)

**Design phase complete: 2026-06-03 (Brainstormer ran Opus; 4 key decisions locked below)**

### Backend contract verification (confirmed in Design)

- `LogLine` response carries: `timestamp, message, stream, severity, host, service, fields: dict`. There is NO top-level `source_type` field.
- `source_type` IS recoverable per-line from `LogLine.fields["source_type"]` — Vector tags every line (docker/cron/systemd/unknown) at ingest (deploy/vector/vector.toml.template). Defaults to "unknown" when absent.
- `fields` bag typical keys:
  - **docker** → label.*, image, container_id, container_name, compose_project, compose_service, source_type
  - **cron** → SYSLOG_IDENTIFIER, run_id, cron_fingerprint, source_type
  - **systemd** → _SYSTEMD_UNIT, _HOSTNAME, source_type
  - Plus severity_raw when present.

### Decision 1: Panel rendering mechanism

**CHOSEN = inline right `<aside>` push (desktop) + right `Sheet` overlay (mobile).**

- Add `side?: 'left' | 'right'` prop to apps/ui/src/components/ui/sheet.tsx (default 'left' to preserve the existing left Filter sidebar). Inspector uses the right side.
- Desktop (>767px): a second inline `<aside>` on the right of the LogsExplorerBody flex row, mirroring the existing left Filter `<aside className="w-56 shrink-0">` push pattern.
- Mobile (≤767px): right-side `Sheet` overlay.
- **Rationale:** a field inspector exists to read the line AND its fields together; a desktop overlay would dim/cover the logs. Inline push mirrors the existing Filter sidebar → symmetric left-filter/right-inspector layout, maximum consistency. STAGE-015 scroll-restore targets AppShell's `<main data-app-scroll-container>`, NOT the results column, so the column reflow is structurally safe — but flag a manual scroll-restore check in Refinement.
- **REJECTED Option A** (right Sheet overlay on BOTH desktop+mobile): overlay dims/covers logs on desktop, fighting the inspector's core read-together purpose.
- **REJECTED Option C** (bespoke portal/fixed panel): reinvents Radix Dialog focus-trap/Esc/scroll-lock that sheet.tsx already provides; no codebase precedent; highest test surface.

### Decision 2: Add-to-filter field→filter mapping

**CHOSEN = service→identities, everything else→`_msg` substring.**

| Field | Maps to | Mechanism |
|---|---|---|
| service | selectedIdentities separate state | handleToggleIdentity({ service, source_type: fields["source_type"] ?? "unknown" }) via the existing onToggleIdentity callback |
| host | search text | append `_msg:"<host>"` |
| severity | search text | append `_msg:"<severity>"` |
| stream | search text | append `_msg:"<stream>"` |
| message | search text | `_msg:"<substring>"` |
| fields[k] bag entry | search text | `_msg:"<value>"` |
| timestamp | — | Copy-only, NO add-to-filter button (no sensible substring filter) |

- A new Explorer-level `appendMsgFilter(value)` helper sets committedPlainText/livePlainText and routes through the existing `writeUrl(false, …)` choke-point so it remains consistent with STAGE-015 persistence + history.
- **Rationale:** only `service` maps cleanly to the existing stream-picker separate state (honoring D-ADD-TO-FILTER-SEPARATE-STATE). `_msg` substring is universally-valid LogsQL with no backend changes.
- **REJECTED Option B** (new structured `fieldFilters` separate-state channel with LogsQL `key:"value"`): large surface (new writeUrl param threaded through ~8 sites, new URL key, new persistence field, likely backend composition work); `fields` bag keys may not be indexed in VictoriaLogs so `key:"value"` could silently return nothing — needs backend-contract verification. Deferred to a future stage.
- **REJECTED Option C** (hybrid: structured LogsQL text for host/severity/stream): plain mode wraps text in `_msg:"…"` via translateSearchToLogsQl, so structured operators only work in advanced mode — introduces awkward mode-coupling.

### Decision 3: Click-handler & highlight wiring

**CHOSEN = LogViewer owns selection, emits selected line upward.**

- LogViewer owns `selectedKey` + selected `LogLine`, gated by a new `fieldInspectorEnabled?: boolean` prop (default false; Explorer opts in).
- LogViewer passes `onLineClick(line, key)` + `isSelected` (or `selectedKey`) down through LogLineList to the private LogRow. When `fieldInspectorEnabled` is false (Docker, Cron), LogRow receives no onClick/isSelected → renders identically to today (zero behavior change for non-opting viewers).
- LogViewer emits the selected `LogLine` upward (callback/render prop) so LogsExplorerBody renders the inspector `<aside>`/`Sheet` as a flex-sibling of the results column (required by Decision 1's desktop push layout).
- Line identity = existing composite key `${line.timestamp}-${index}` (pure timestamp+message is not unique for duplicate lines; index disambiguates within the current render). Selection may drift/clear when "Load older" prepends lines — acceptable for a transient selection.
- The Explorer-only add-to-filter callback is an optional prop into LogViewer; Docker/Cron don't supply it.
- **REJECTED Option B** (handle entirely inside LogLineList): LogLineList is the inner `<pre>` body shared by 3 viewers, not the layout shell — embedding a side panel there fights the Body-level flex layout.
- **REJECTED Option C** (selection state in each viewer's Body): leaks the opt-in feature into Docker/Cron Bodies that don't use it.

### Decision 4: Clipboard utility shape

**CHOSEN = pure lib/clipboard.ts + useCopyToClipboard hook.**

- apps/ui/src/lib/clipboard.ts: pure `copyToClipboard(text: string): Promise<boolean>` (navigator.clipboard.writeText + document.execCommand fallback). Trivially 100%-coverage unit-testable (mock the API, assert return).
- apps/ui/src/lib/useCopyToClipboard.ts (or hooks dir): thin `useCopyToClipboard` hook wrapping the pure fn, firing a `sonner` toast on success/failure (matches the established sonner pattern used across ~15 files).
- **REJECTED Option B** (pure fn only, caller toasts): risks inconsistent feedback across call sites.
- **REJECTED Option C** (inline writeText at each button): no fallback, duplicated logic, harder to unit-test, worst for the coverage gate.

### Open items confirmed/decided in Design (no user prompt needed — delegated authority)

- **source_type per-line:** CONFIRMED recoverable via fields["source_type"] (defaults "unknown").
- **timestamp add-to-filter:** DECIDED Copy-only.
- **Selection drift on "Load older":** ACCEPTED as transient (may drift/clear).

### Refinement Summary (2026-06-03)

Desktop + Mobile approved by user on prod rig. Refinement scope expansions:

1. **Hide null/undefined/empty-string/whitespace-only field rows entirely** (no "—" placeholder).
   - Root cause: backend coerces all VictoriaLogs field values to strings (victorialogs_client.py `str(v)`), so absent journald fields arrive as `""` not null.
   - Predicate: omit iff `raw == null || (typeof raw === 'string' && raw.trim() === '')`.
   - 0/false/non-empty values remain visible.

2. **Independent scroll containers + static panel headers** (cross-cutting layout fix, done in-stage at user request).
   - Filter / Logs / Inspector panels each scroll independently to viewport bottom.
   - App header + each panel header (filter tabs, logs control bar, inspector header) stay static.
   - STAGE-015 scroll persistence re-pointed from AppShell `<main data-app-scroll-container>` to new inner `data-log-scroll-container`.
   - Implemented via opt-in `fillHeight` prop on LogViewer (default false) so Docker/Cron viewers unaffected.

3. **Follow-up stages created from Refinement feedback**:
   - **STAGE-004-016A** (LogsQL structured field filters): deferred Option B from Design.
   - **STAGE-004-016B** (JSON message drill-down in field inspector): deferred nested display.
   - Current stage keeps `_msg` interim filter mapping + flat field rows.
