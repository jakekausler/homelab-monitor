# STAGE-004-014: Query history (last 20 executed queries; separate from saved queries)

## Status: Complete

## Overview

Automatically track the last 20 queries the user has executed in the Explorer. Surfaced as a "Recent" panel or dropdown that lets the user re-run a recent query without saving it. Different from Saved Queries — history is automatic and disposable; saved queries are explicit and named.

**Vertical slice:** every Explorer search appends to history. A "Recent" entry point shows the last 20 with their timestamp, click to re-run. Items roll off as new ones come in.

**Spec references:** master spec §9.2 (logs explorer).

## Concrete deliverables

### 1. Persistence scope (Design phase decides)

Two options:

- **Option A: localStorage** — client-side only. Survives navigation + reload. Lost on browser/profile change. Simpler.
- **Option B: SQLite (`log_query_history` table)** — server-side. Cross-device. Slightly more work.

Recommendation: **Option A (localStorage)** for v1. Promote to Option B later if cross-device matters.

### 2. History entry shape

```typescript
interface HistoryEntry {
  timestamp: number  // unix ms when run
  logsQl: string
  plainText?: string  // if plain mode
  selectedServices: string[]
  rangeKind: 'preset' | 'custom'
  sincePreset?: string
  rangeStart?: string  // ISO
  rangeEnd?: string  // ISO
  advancedMode: boolean
  resultCount?: number  // optional; for "this query returned 142 lines" annotation
}
```

### 3. History recorder

`apps/ui/src/lib/queryHistory.ts` (new) — append + cap to 20 + persist. De-duplicate consecutive identical entries (don't grow when user re-runs same query repeatedly).

### 4. UI surface

`apps/ui/src/routes/logs/QueryHistoryPanel.tsx` (new) — list of last 20 with:

- Relative timestamp ("2 min ago")
- Truncated query preview
- Result count (if known)
- Click to load + re-run

Entry point: a sidebar tab OR a popover under a "Recent" button. Design picks (recommend sidebar tab alongside Saved Queries).

### 5. Tests

- `apps/ui/src/lib/__tests__/queryHistory.test.ts` — append, cap, dedupe consecutive
- `__tests__/QueryHistoryPanel.test.tsx` — renders list, click reloads

## Out of scope for this stage

- History beyond 20 entries — YAGNI
- Searching within history — YAGNI
- Auto-promote frequently-rerun query to saved queries — speculative
- History sync across devices — Option B promotion if needed

## Decisions in this stage

### Design phase will confirm

- **Persistence scope (A vs B)** — recommendation A
- **Cap (20 vs configurable)** — recommendation fixed 20 in v1
- **Dedupe policy** — consecutive identical only, or rolling dedupe of any duplicate within window? Recommendation: consecutive only (simpler, matches browser history behavior)

### Locked Design Decisions (inherited from brainstorming 2026-05-28)

- **D-HISTORY-AUTO** — every executed query is recorded automatically.
- **D-HISTORY-CAP-20** — last 20 entries; older entries roll off.
- **D-HISTORY-SEPARATE-FROM-SAVED** — history is disposable; saved queries are explicit.

## Design Phase

**Design phase complete: 2026-06-02**
**Status**: [x] Design (Complete)

**Selected approach (Design phase, decided 2026-06-02) — Option A: Choke-point recorder + pure core + thin hook:**

- **Persistence: localStorage** (locked). Mirror the existing `apps/ui/src/lib/useTimezonePreference.ts` pattern (STORAGE_KEY const + readInitial() with SSR guard + useState(readInitial) + useEffect write-back). STORAGE_KEY like `homelab-monitor:logs-query-history`.
- **Where to record: the writeUrl() choke-point** in LogsExplorerPage.tsx (~lines 84-116). Every committed query (handleSubmitSearch, handleRangeChange, handleLoadSavedQuery, identity toggles) flows through writeUrl, so a single record() call there auto-captures all executed queries (satisfies D-HISTORY-AUTO, can't-forget, future-proof). Extract record() as a single line.
- **Dedupe: consecutive-identical** — collapse a new entry only if it equals the immediately-previous entry. Preserves chronology (a true "Recent" event log, distinct from Saved per D-HISTORY-SEPARATE). Equality key = advanced_mode (bool) + active expression (committedLogsQl if advanced else committedPlainText) + sorted selected identities (`source_type:service`) + range (preset token OR resolved/serialized start+end ISO). Exclude timestamp + resultCount. Reuse buildSavePayload's canonical serialization to derive the key.
- **Module design: pure core + thin hook.** Pure `apps/ui/src/lib/queryHistory.ts` (append/dedupe/cap-20/serialize/equalityKey — fully unit-testable, satisfies queryHistory.test.ts) + thin `useQueryHistory()` hook (reactive read + localStorage write-back, mirrors useTimezonePreference; exposes entries, record(entry), clear()).
- **UI: 3rd "Recent" sidebar tab** in LogsExplorerBody.tsx (extend sidebarTab 'services'|'saved' → add 'history'; add 3rd tab button + branch rendering QueryHistoryPanel). Each entry: relative timestamp (formatRelative from lib/relativeTime.ts) + query preview + click-to-load (re-run via the existing handleLoadSavedQuery deserialize logic — a HistoryEntry is payload-shaped, like a SavedQuery minus name/id plus timestamp). Include a **"Clear history" button**. Empty state mirrors saved-queries-empty ("No recent queries yet. Run a search to populate.").
- **resultCount: OMIT for v1** — at writeUrl/submit time the fetch hasn't resolved, so capturing it synchronously couples recording to fetch completion. Keep `resultCount?: number` as an optional field in HistoryEntry for non-breaking future lazy-fill.
- **Churn note:** writeUrl fires on granular identity toggles too, so heavy interactive filtering can churn the 20-cap. SHIP the literal choke-point first; if churn is annoying in Refinement 3a, add a one-line predicate to record only when expression OR range changes (skip pure identity toggles) — NOT an architecture change.

**Inherited locked decisions (unchanged):**
- D-HISTORY-AUTO (auto-track every executed query), D-HISTORY-CAP-20 (cap 20, oldest rolls off), D-HISTORY-SEPARATE-FROM-SAVED (separate storage + concept from saved queries).

**HistoryEntry shape:** { timestamp: number (unix ms); advanced_mode: boolean; logs_ql: string (active-mode expression, same convention as buildSavePayload); selected_services: {service, source_type}[]; since_preset?: string|null; range_start_iso?: string|null; range_end_iso?: string|null; resultCount?: number (optional, omitted v1) }. Exactly one of since_preset OR (range_start_iso AND range_end_iso), same invariant as saved queries.

**Reuse from STAGE-004-013:** buildSavePayload (serialize current state) + handleLoadSavedQuery (deserialize → Explorer + writeUrl) in LogsExplorerPage.tsx; the sidebar two-tab pattern in LogsExplorerBody.tsx; SavedQueriesPanel.tsx structure/empty-state as the model for QueryHistoryPanel.tsx.

**Seed data:** none (auto-generated as the user runs queries; manual testing in Refinement populates it).

**Cross-cutting:** frontend-only (no backend, no migration). >=80% UI functions coverage. queryHistory.test.ts (pure core: append/cap/dedupe/equalityKey) + QueryHistoryPanel.test.tsx (renders, empty state, click-reloads, clear).
