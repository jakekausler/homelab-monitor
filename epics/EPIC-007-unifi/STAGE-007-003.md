# STAGE-007-003: Persistent `unifi_clients` registry table (MAC-keyed, time-stamped IP‚ÜîMAC observations)

## Status: Complete

## Overview

Create the persistent client registry ‚Äî the canonical client inventory that EPIC-007 OWNS and EPIC-006
ENHANCES. A new SQLite table (Alembic migration) keyed by **MAC** (the stable identity), holding the current
identity + connection facts, **plus time-stamped IP‚ÜîMAC observations** so EPIC-006's join is time-windowed
(never a static IP‚Üídevice map ‚Äî DHCP IPs rotate). The host `192.168.2.148` is a first-class row.

**Vertical slice:** the migration creates the `unifi_clients` table (+ observation history); a repository
facade can upsert a client and append an IP‚ÜîMAC observation; migrate up/down is tested; a seeded host row
(`192.168.2.148`) is present.

**Spec references:** spec ¬ß6.1 (registry/`targets` "single source of truth" principle; SQLAlchemy Core +
narrow repository facade), ¬ß6 (Alembic migrations). EPIC-007 "Persistent client registry" + "Client-join
contract". This is the data foundation the identity collector (007), the EPIC-006 enrichment, the digest
"what changed", and `UnifiNewClient` all read.

## Cross-cutting standards

- SQLAlchemy Core (NOT ORM) behind a narrow repository facade; pydantic/dataclass row shapes.
- Alembic migration; `HOMELAB_MONITOR_AUTO_MIGRATE` honored; migrate up/down both tested.
- All internal timestamps UTC.
- Cardinality: the registry is bounded by real device count (not metric cardinality), but the observation
  history needs a retention/cap policy (Design decides ‚Äî e.g., keep last N observations per client or a
  time window) so it doesn't grow unbounded.

## Concrete deliverables

### 1. Migration + tables

`unifi_clients` keyed by MAC: current `ip`, `hostname`, `name`, `oui`/device-type, `network`/SSID,
`ap_mac`, `sw_mac`, `sw_port`, `fixed_ip`/`use_fixedip`, `online`, `first_seen`, `last_seen`,
`lease_expiry` (nullable ‚Äî populated only by the opt-in lease collector 012). A satellite
`unifi_client_observations` table (or a bounded JSON history) recording **time-stamped IP‚ÜîMAC observations**
for the time-windowed join. Indexes for IP-at-time lookups (the EPIC-006 join) and `last_seen`.

### 2. Repository facade

`UnifiClientRepo` with upsert-by-MAC, append-observation, query-by-IP-at-time-window (the contract EPIC-006
calls), and a host-row guarantee (the configured `unifi_host_lan_ip`=`192.168.2.148` is always present as a
first-class row).

### 3. Host seed

Seed/ensure the host `192.168.2.148` row exists as a first-class device, marked appropriately so views never
hide it.

## Design Phase

### Selected Approach (Design)

SQLite via SQLAlchemy Core (raw `text()` DDL in the migration; no central MetaData). New Alembic migration `0044_unifi_clients.py` (`down_revision="0043"`, the current HEAD). All timestamps UTC ISO-8601 TEXT via `utc_now_iso()`. Four locked decisions.

**D1 ‚Äî Observation history: satellite table, per-(mac,ip) SPAN rows (Option A).**
A second table `unifi_client_observations(mac TEXT, ip TEXT, first_seen TEXT, last_seen TEXT)` keyed/uniqued on `(mac, ip)` ‚Äî repeated sightings of the same mac‚Üîip pairing COLLAPSE into one span (append UPSERTs `last_seen`), not one row per observation. Indexes: `(ip, first_seen)` for the EPIC-006 time-window join, `(mac, last_seen)` for per-client history. Rejected: per-OBSERVATION rows (grows on every collector tick ‚Üí aggressive prune needed); JSON-history column (SQLite can't index `json_each` for the `WHERE ip=:ip AND first_seen<=:t` lookup ‚Üí full scans; the EPIC-006 join is the PRIMARY consumer and must be indexed; read-modify-write JSON is race-prone + hard to cover 100%). The span model gives bounded growth + natural retention on a home /24.

**D2 ‚Äî Retention: 90-day time-window prune, INLINE in `append_observation_conn` (Option A).**
After each observation append, `DELETE FROM unifi_client_observations WHERE last_seen < :cutoff`. The cutoff is `now - retention_days`; the retention default is a new `UnifiConfig` field `observation_retention_days = 90` (env `HOMELAB_MONITOR_UNIFI_OBSERVATION_RETENTION_DAYS`), added to `kernel/config.py` (mirrors the STAGE-007-002 config-extension precedent). The repo method takes the CUTOFF timestamp as a parameter (caller computes it from config) ‚Äî keeps the repo pure + testable, decoupled from config. Inline prune needs no collector (003 has none) and is trivially coverage-tested (insert old span ‚Üí append ‚Üí assert pruned). Rejected: keep-last-N-per-mac (correlated-subquery SQL, more branch surface, and N doesn't map to EPIC-006's time-based lookback need); both (YAGNI at home scale). **Trade-off:** a device offline >90d loses its IP-span history, so a very stale DNS log line can't be MAC-resolved ‚Äî acceptable (DNS enrichment is best-effort; 90d covers realistic lookback).

**D3 ‚Äî `unifi_clients` columns + types (INTEGER 0/1 booleans; dedicated `is_host`).**
PRIMARY KEY `mac` (lowercase colon-separated). Booleans as INTEGER 0/1 (SQLite-native, project norm). Dedicated `is_host INTEGER NOT NULL DEFAULT 0` marks the host row first-class (explicit + queryable + survives the host IP changing ‚Äî beats inferring from IP at query time). `lease_expiry` ships NULLABLE NOW ‚Üí **STAGE-007-012 needs NO migration**, only writes the existing column (CONFIRMED). DDL column list:

| Column | Type | Null | Default | Notes |
|---|---|---|---|---|
| `mac` | TEXT | NOT NULL | ‚Äî | PRIMARY KEY; lowercase colon-separated |
| `ip` | TEXT | NULL | ‚Äî | current IP; dotted-decimal |
| `hostname` | TEXT | NULL | ‚Äî | DHCP/reported hostname |
| `name` | TEXT | NULL | ‚Äî | user-assigned alias |
| `oui` | TEXT | NULL | ‚Äî | vendor/device-type; not always available |
| `network` | TEXT | NULL | ‚Äî | network/SSID |
| `ap_mac` | TEXT | NULL | ‚Äî | uplink AP mac |
| `sw_mac` | TEXT | NULL | ‚Äî | uplink switch mac |
| `sw_port` | INTEGER | NULL | ‚Äî | switch port |
| `use_fixedip` | INTEGER | NOT NULL | `0` | 0/1 boolean |
| `fixed_ip` | TEXT | NULL | ‚Äî | configured fixed IP (distinct from current `ip`) |
| `online` | INTEGER | NOT NULL | `0` | 0/1 boolean |
| `is_host` | INTEGER | NOT NULL | `0` | 0/1; set for the host row |

Indexes on `unifi_clients`: `ix_unifi_clients_last_seen(last_seen)` (recency/staleness lists) and `ix_unifi_clients_ip(ip)` (current-IP lookups by the 022 UI). (No separate unique index on `mac` ‚Äî it's the PK.) The time-window join's indexes live on the observations table.

**D4 ‚Äî Repo facade `UnifiClientRepo` + host seed + row shape.**
Write helpers are STATIC + transaction-composable (so STAGE-007-004's identity-upsert can call upsert+append atomically in ONE `repo.transaction()`):
- `upsert_client_conn(conn, *, mac, ip, hostname, name, oui, network, ap_mac, sw_mac, sw_port, use_fixedip, fixed_ip, online, last_seen) -> None` ‚Äî `INSERT ... ON CONFLICT(mac) DO UPDATE SET ...` updating mutable fields + `last_seen`, PRESERVING `first_seen` (set only on INSERT), never touching `is_host`/`lease_expiry`.
- `append_observation_conn(conn, *, mac, ip, observed_at, cutoff) -> None` ‚Äî `INSERT ... ON CONFLICT(mac, ip) DO UPDATE SET last_seen=excluded.last_seen` (span), then inline retention `DELETE ... WHERE last_seen < :cutoff`.
Instance read/seed methods (own `self._repo`, call `fetch_*`):
- `find_mac_by_ip_at(ip, at) -> str | None` ‚Äî the EPIC-006 contract: `WHERE ip=:ip AND first_seen <= :at ORDER BY last_seen DESC LIMIT 1`.
- `get_client(mac) -> UnifiClientRow | None`, `list_clients() -> list[UnifiClientRow]`.
- `ensure_host_row(host_ip, host_mac=None) -> None` ‚Äî idempotent host-row guarantee.
Row shape: a frozen `@dataclass UnifiClientRow` mapping all columns (0/1 INTEGER ‚Üí `bool` at the map boundary; nullable cols `str | None`). Build VERIFIES sibling-repo row-shape convention (dataclass vs pydantic) and matches it.

**Host seed (config-driven, NOT a migration seed):** `ensure_host_row()` is called at lifespan startup (after migrations) reading the configured `unifi_host_lan_ip` (=192.168.2.148, the STAGE-007-002 config field). A migration can't read runtime config, so the host IP is NOT baked into the migration. **MAC-key wrinkle:** the registry is MAC-keyed but the host is known by IP (real MAC discovered later by the collector). `ensure_host_row` inserts a SENTINEL row keyed `host:<host_ip>` (e.g. `host:192.168.2.148`) with `is_host=1`, `ip=host_ip`, `online=0`, `first_seen=last_seen=utc_now_iso()`, ONLY if no `is_host=1` row already exists (idempotent).

**Tracked deferrals (no silent scope-drop ‚Äî each owned by a named stage):**
- **Sentinel-host-row ‚Üí real-MAC reconciliation** ‚Üí **STAGE-007-004** (identity-upsert) / **STAGE-007-007** (active-client collector). When the collector first sees the host's REAL MAC, it must merge the `host:<ip>` sentinel into the real-MAC row (carry `is_host=1`, merge `first_seen`, drop the sentinel) so the host isn't double-listed. This is collector-side identity logic, NOT data-layer. 003 only guarantees a first-class host row exists.
- **`lease_expiry` writes (NO migration)** ‚Üí **STAGE-007-012** (writes the existing nullable column via UPDATE/extended upsert).
- **90d retention vs. actual EPIC-006 lookback** ‚Üí **EPIC-006 STAGE-006-027** (verify the 90d default covers 006's DNS-enrichment lookback; bump if 006 looks back further).
- **`oui`/device-type nullability vs. live Unifi shape** ‚Üí **STAGE-007-004** (identity-upsert decides whether `stat/sta`/`stat/alluser` always provide it).

**Scope boundary:** 003 ships the TABLES + migration + `UnifiClientRepo` facade + host seed + retention. STAGE-007-004 owns the `stat/sta`+`stat/alluser` API-shape‚Üíregistry MAPPING (it calls this repo with already-extracted typed fields) + cardinality caps. Do NOT build stat/sta parsing in 003.

## Refinement Phase

### Refinement Results

**3a (backend DB lifecycle, REAL migrated on-disk SQLite through the actual `UnifiClientRepo`):** PASS. A standalone async script built a fresh migrated DB (mirroring the repo test fixture; `0044` ran ‚Äî both `unifi_clients` + `unifi_client_observations` tables confirmed present) and drove the full lifecycle:
- **Host row:** `ensure_host_row("192.168.2.148")` inserts exactly one sentinel `host:192.168.2.148` row with `is_host=True`; a second call is a no-op (still exactly one is_host row) ‚Äî idempotent.
- **Upsert + first_seen preservation:** first upsert sets `first_seen==last_seen==now`, bool mapping correct (`online=True`); a second upsert of the same mac with a changed IP + later timestamp updates `ip`/`last_seen` but PRESERVES `first_seen`.
- **Observation span + 90-day prune:** an old span (100d ago) is DELETED by the inline prune at cutoff `now-90d`; the fresh span survives (verified by a raw SELECT ‚Äî one row remained).
- **Time-window query `find_mac_by_ip_at` (the EPIC-006 contract):** with two macs reusing one IP at different times, returns the mac whose span covers `at` with the latest `last_seen`; returns None for an `at` before any span's `first_seen`; returns None for an IP never seen.

This validated the span model, inline retention, the time-windowed join, and the config-driven host seed against real DB reality (not isolated unit fixtures). All timestamps relative (no time-rot).
