# STAGE-007-001: Unifi API client (key auth; classic + v1) + `unifi_api_key` secret + lifespan wiring + smoke

## Status: Complete

## Overview

The first foundation stage of the Unifi integration bundle. Build a real Unifi API client that
authenticates with the **single read-only API key** and reaches BOTH the official `v1` Integrations API
and the classic reverse-engineered API (the live recon confirmed the same `X-API-KEY` works on both on this
firmware). Wire it into the collector context so every Wave-B/C collector can reach the UDM. Establish the
single-credential model and the self-signed-TLS handling.

**Vertical slice:** after this stage a smoke test (and the collector status API) shows the monitor reaching
the UDM with the `unifi_api_key` secret and successfully calling `GET /proxy/network/integrations/v1/sites`
(returns the site id) and a classic endpoint (e.g. `GET /proxy/network/api/s/default/stat/sysinfo`).

**Spec references:** master spec §2 (integration_bundle; Q11 AMENDED — direct-API, no unpoller), §5.2
(`CollectorContext` client injection — `ctx.ha` precedent), §6.2 (`homelab_unifi_*`). EPIC-007 "Verified
deployment reality" + "Credential / transport model". EPIC-005 STAGE-005-001 is the structural precedent
(an integration's API client constructed, credentialed via the secret store, injected into context).
EPIC-006 STAGE-006-001 is a close sibling (session client + secret + lifespan + smoke).

## Cross-cutting standards (from EPIC-007 cross-stage acceptance criteria)

- The API key is NEVER logged (assert in tests).
- **Read-only / observe-only** — the client only ever issues GETs; no write helpers exist.
- Self-signed cert (`CN=unifi.local`, no CA chain) → `verify=False` (or pin the cert); document the choice.
- UDM 5xx/timeout never propagates as our 5xx — the client surfaces a typed error the collector turns into
  a failed run (failure budget takes over).
- All collectors join the `unifi` concurrency group (never DDoS the controller).
- All internal timestamps UTC.

## Concrete deliverables

### 1. Real Unifi API client

A new `UnifiClient` (Design picks the exact module path within the bundle area; keep any Protocol in `io.py`,
impl separate; mirror how the HA/Pi-hole clients were structured). Uses the shared `httpx.AsyncClient` from
the context. Responsibilities:
- **Auth:** attach `X-API-KEY: <unifi_api_key>` to every request. No login/CSRF/cookie dance (the key is
  stateless). `verify=False` for the self-signed cert.
- **Base URL + paths:** configurable `unifi_url` (default `https://192.168.2.1`). Helpers for BOTH surfaces:
  - official v1: `v1_sites()`, `v1_devices()`, `v1_device(id)`, `v1_device_stats(id)`, `v1_clients()`.
  - classic: `stat_device()`, `stat_sta()`, `stat_alluser()`, `stat_health()`, `stat_dpi()`,
    `rest_networkconf()`, `rest_alarm()`, `stat_sysinfo()`. (Not all consumed in THIS stage — this
    establishes the client surface; collectors land in Wave B/C.)
- **`took`/latency capture:** time each request; expose duration so the API-latency collector (013) reads it.
- **Site-id resolution:** resolve and cache the site id (`default`) once at startup from `v1/sites`.
- Typed error mapping (auth failure/401, 429, 5xx, timeout, JSON-shape mismatch).

### 2. Secret + config wiring

- Secret-store key `unifi_api_key` (single, read-only). Document the exact UDM UI steps to generate it
  (Settings → Control Plane → Integrations → create API key) + the secret name, in the integration README.
  State that it is read-only. Do NOT reveal in any committed doc that a passwordless-root SSH trust to the
  router exists.
- `unifi_url` config (default `https://192.168.2.1`), read via the established `getattr(ctx.config, ...)`
  pattern. Inject a live `UnifiClient` into the collector context (the `ctx.ha=None`→real precedent).

## Design Phase

### Selected Approach (Design)

The Unifi client is **kernel infrastructure** mirroring the Home Assistant client (`kernel/ha/`), NOT part of the `integrations/unifi/` collector bundle (that's STAGE-007-002). Four locked decisions:

**D1 — Self-signed TLS: dedicated verify-disabled shared httpx client in lifespan (Option B).**
Lifespan creates a SECOND shared `httpx.AsyncClient` with `verify=False`, parallel to the existing vanilla `http_client`, scoped to the UDM, and injects it into the `UnifiRestClient` (the client borrows it, lifespan owns construction + teardown — exactly the HA "client receives a shared httpx" pattern). Blast radius of `verify=False` is exactly one target (the UDM, self-signed `CN=unifi.local`). Cert-pinning (Option C) rejected: rotation on firmware update silently breaks the integration; over-engineered for a read-only LAN GET against the user's own gateway. Document the `verify=False` choice + threat model in the integration README.

**D2 — `took`/latency exposure: `UnifiResponse` wrapper (Option A).**
Each client GET returns a small frozen `UnifiResponse(payload, took_seconds, endpoint)` (timed via `time.monotonic()` in the private GET helper). Keeps the client a PURE RPC layer (no shared mutable state, no metrics coupling) — easiest path to 100% branch coverage. The future API-latency collector (STAGE-007-013) reads `.took_seconds`/`.endpoint` and emits `homelab_unifi_api_took_seconds{endpoint}` itself, mirroring HA's "collector owns emission" precedent. Rejected: internal duration store (async-race + lossy), client-emits-metric (couples client to metrics, harder to test to 100%).

**D3 — Surface shape + site-id: one flat `UnifiRestClient`, eager non-fatal site-id (Option A).**
A single flat `UnifiRestClient` exposing typed v1 helpers (`v1_sites`, `v1_devices`, `v1_device(id)`, `v1_device_stats(id)`, `v1_clients`) and classic helpers (`stat_device`, `stat_sta`, `stat_alluser`, `stat_health`, `stat_dpi`, `rest_networkconf`, `rest_alarm`, `stat_sysinfo`) — mirrors the HA single-class precedent and gives Wave-B/C collectors a discoverable typed surface. Two private URL-builders `_get_v1(path)` / `_get_classic(ep)` centralize `/proxy/network/...` path construction, key-header attachment, TLS-client selection, timing, and error mapping. Site-id (`default`) resolved EAGERLY at startup from `v1/sites` and cached — but **non-fatal**: if `v1/sites` is unreachable at construction the client still constructs (consistent with HA "always construct, never None"); `site_id` stays unresolved and classic calls return a typed error / re-attempt resolution until it resolves (a startup UDM/key failure must NOT crash the app). Rejected: two sub-clients + facade (YAGNI, diverges from HA single-class), thin get-only client (pushes path strings into every collector, loses the typed surface the card requires).

**Refinement correction:** the single `site_id` was split into `site_name` (classic NAME "default") + `v1_site_id` (v1 UUID) — the two Unifi APIs use different site identifiers. See Refinement Results.

**D4 — Location: kernel mirror (Option A).**
`kernel/unifi/client.py` (impl `UnifiRestClient`) + `kernel/unifi/errors.py` (typed `UnifiError` discriminated-union dataclass: `reason: Literal[...]`, `message` NEVER containing the key, `status: int | None`) + Protocol `UnifiClient` in `kernel/plugins/io.py` + `UnifiConfig`/`load_unifi_config()` in `kernel/config.py` (default `unifi_url = "https://192.168.2.1"`, env `HOMELAB_MONITOR_UNIFI_URL`). Injected non-None into `CollectorContext.unifi` via the `ctx_factory` in `kernel/api/lifespan.py`. The `unifi_api_key` secret is read via the TTL-cached resolver (`ttl_resolver.current().get("unifi_api_key")`) using the per-request `key_provider` lambda pattern (mirrors HA's `token_provider`); the key is attached as the `X-API-KEY` header and NEVER logged or placed in any `UnifiError.message`.

**Error mapping (UnifiError reasons):** `unreachable` (ConnectError/ConnectTimeout), `timeout` (ReadTimeout/TimeoutException), `auth` (401/403 — also returned WITHOUT a network call when `key_provider()` is None), `rate_limited` (429), `http_error` (other non-2xx incl. 5xx — UDM 5xx surfaces as typed error, never propagates as our 5xx), `bad_response` (2xx but unparseable/shape-mismatch JSON).

**Smoke:** the v1/sites call is exercised by the eager site-id resolution at startup; a dedicated controller-up/smoke COLLECTOR (mirroring `HaUpCollector`) is established in Wave B/C, NOT in the client constructor.

## Refinement Phase

### Refinement Results

**3a (unit, mocked transport):** 36 tests pass, 100% branch coverage on `kernel/unifi/` — X-API-KEY header, v1 + classic surfaces, `took` capture, site-id resolution, and all typed-error branches (401/403/429/5xx/timeout/bad-response) covered.

**3b (LIVE UDM Pro, read-only, real key, `verify=False`):** all four probes PASS against `https://192.168.2.1`:
- `v1_sites()` → 200, real site UUID, ~0.07s; self-signed TLS via `verify=False` confirmed (no SSL error).
- `resolve_site_id()` → None; cached v1 UUID into `v1_site_id`; classic `site_name` preserved as `"default"`.
- `stat_sysinfo()` (classic) → 200 with rich sysinfo payload.
- `v1_devices()` (v1 site-scoped) → 200, 8 devices.

**Refinement-discovered design correction (TWO site identifiers):** the initial live probe revealed `stat_sysinfo()` returned HTTP 401 while `v1_sites()` succeeded with the SAME key. Root cause (confirmed via read-only live probes): the classic API path `/proxy/network/api/s/{site}/` requires the SHORT site NAME (`"default"`), while the v1 Integrations site-scoped paths `/sites/{id}/` require the site UUID. The client originally conflated both into one `site_id` field, and `resolve_site_id()` overwrote it with the UUID → classic calls built `/api/s/{UUID}/...` → 401. FIX: split into `site_name` (classic, default `"default"`, never overwritten) and `v1_site_id` (UUID, resolved from v1/sites). The constructor keyword stays `site_id` (seeds both). The `UnifiClient` Protocol now declares both `site_name` and `v1_site_id`. A regression-guard unit test (`test_classic_url_uses_site_name_not_resolved_v1_uuid`) asserts the classic URL stays `/api/s/default/...` after a UUID is resolved. The EPIC-007 recon claim ("same X-API-KEY works on both APIs") is CONFIRMED correct — both return 200 with the right path. **Downstream Wave B/C collectors must thread `v1_site_id` into v1 site-scoped helpers and `site_name` into classic helpers.**
