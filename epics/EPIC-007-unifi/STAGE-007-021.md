### Design Notes

**Selected approach** (brainstormer-backed; recommendation selected per pre-authorized Design rule ‚Äî no user prompt):

1. **Structure** ‚Äî Promote `NetworkPage` from a flat route to a tabbed `NetworkLayout` (header + `<Link>` tab strip + `<Outlet/>`, mirroring the STAGE-020 `UnifiLayout`). Tabs: **Overview** (filled this stage) + a **Clients** placeholder tab rendering an honest "coming in STAGE-022" empty state. Confirmed from BOTH the 021 card ("tabbed shell", "Clients tab placeholder filled in 022") and the 022 card ("The Clients tab (under Network)") that the Clients tab lives under the Network page.

2. **DNS-posture indicator ‚Üí A1 (honest list + empty state, NO ‚úì/drift badge this stage).** The `/network/dns-posture` endpoint returns only `{network, dns}` (no expected value, no drift status) and is legitimately EMPTY for this UDM. The expected value `192.168.2.148` lives ONLY in the `UnifiDnsSteeringDrift` vmalert rule (`deploy/vmalert/metrics/unifi.yaml`), which already flags that constant as duplication-prone "versioned config". Frontend-hardcoding it (option A2) would create a THIRD copy of a sync-fragile host-specific constant, embed it in the public OSS frontend, and ship ‚úì/drift logic unexercised on this empty-data UDM ‚Äî three honest-data / single-source-of-truth violations. So 021 renders the raw per-network handout list + an honest "No per-network DNS overrides configured ‚Äî clients use default DHCP DNS steering" empty state.

   **DEFERRAL (tracked, NOT silent):** the card's "green ‚úì / red drift" DNS sub-deliverable is deferred to the NEW **STAGE-007-026** (backend adds `expected_dns`/derived `drift` to the DNS-posture contract from a single source, then the badge UI consumes it honestly). Sequential Dependency Ordering: 026 is higher-numbered because the badge depends on this stage's DNS widget existing; within 026 the backend field lands before the badge UI. See STAGE-007-026 card.

3. **WAN/speedtest history ‚Üí B1.** Factor a small reusable `<UnifiRangeChart>` (recharts LineChart + ResponsiveContainer, fixed height) consuming the existing `useMetricsRange` proxy hook: download/upload Mbps as two lines + a separate compact latency chart. Fixed once-per-mount lookback window (the `HostCpuTile` `useState(() => ...)` pattern) for a stable query key; ~24h window, coarse step (speedtests run periodically). Explicit empty/sparse-series state (speedtest data is often null on this UDM ‚Äî handled honestly, not fabricated).

4. **Overview tab layout ‚Üí C1.** Full-width lead `PanelSection` for WAN/ISP current stats + the speedtest history chart; below, a `grid-cols-1 sm:grid-cols-2` grid for DHCP (pool/occupancy/reservations ‚Äî honest "DHCP not enabled / occupancy unavailable" when `dhcp_enabled=false`/`occupancy=null`), WiFi-experience (poor_signal/poor_satisfaction/high_retries + by_band/by_link), SSID distribution, and DNS-posture. Extract the `QueryState` pending/502/error helper currently inside `UnifiOverviewTab.tsx` to a shared module so both Overview tabs use one copy. Single-column on mobile.

**Backend contract verification (DONE):** the four `/network/{wan,dhcp,wifi,dns-posture}` endpoints in `kernel/api/routers/integrations_unifi.py` return REAL VM-derived data (not stubs), typed in `apps/ui/src/api/schema.ts` (UnifiWanCurrent, UnifiNetworkDhcpResponse, UnifiWifiResponse, UnifiDnsPostureResponse). The DHCP lease-count-vs-approximation degradation is SERVER-SIDE (frontend renders what it gets). `/api/metrics/range` proxy + `MetricsRangeResponse` confirmed for history. recharts v3.8.1 already a dep.

**Patterns to mirror:** extend `apps/ui/src/api/unifi.ts` with `useUnifiWan/Dhcp/Wifi/DnsPosture`; reuse `unifiFormat.ts` formatters (formatBitrate/formatPct/formatBytes) + `formatUptime`; clone `HistogramChart` recharts shape for `<UnifiRangeChart>`; mirror STAGE-020 widget empty/loading/502 states.
