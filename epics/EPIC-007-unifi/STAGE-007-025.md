### Design Notes

**Selected approach: Option B panels (barchart distributions, timeseries for anomaly + speedtest) with ALL ROWS EXPANDED to match the `homelab-unifi.json` (STAGE-024) sibling convention.** (Brainstormer Opus consulted; main agent selected per pre-authorization — the brainstormer's "hybrid" of A's all-expanded + B's barcharts.) Authors `deploy/grafana/dashboards/network.json` (uid `network` per the file==uid convention, title "Network", tags `["homelab-monitor","network"]`, schemaVersion 40, datasource `{type:prometheus,uid:victoriametrics}` on every panel + target, bind-mount provisioned, editable:false, refresh 30s). Plus a `/metrics/network` Grafana embed tab.

**Why all-expanded (divergence from the brainstorm's collapse-default):** collapsing DHCP/WiFi would hide the DHCP-occupancy gauge (a card deliverable) on first paint AND make this dashboard inconsistent with its adjacent sibling `homelab-unifi.json` (all-expanded). Consistent collapse behavior across the two Metrics tabs beats optimizing one's scroll length. Panel TYPES follow Option B (barcharts for raw low-cardinality distributions beat pie; timeseries preserve the anomaly story + honest 0-speedtest line).

**Rows (5: 4 network rows + optional DNS table — network-as-concept; gear/device detail is STAGE-024's `homelab-unifi.json`, NOT here):**
1. WAN / ISP + speedtest: stat group (`wan_up`/`wan_latency_seconds`/`wan_uptime_seconds`/`wan_failover_active`+`_capable`) + WAN throughput timeseries (`wan_xput_down/up_bytes_per_sec * 8 / 1e6`, unit Mbits/sec) + speedtest timeseries (download/upload over time, shows the 0 line honestly) + a "Last speedtest" stat from `speedtest_lastrun` (epoch→date).
2. DHCP occupancy: a gauge/bargauge using the BOTH-PATHS expr (lease-on AND lease-off) + pool-size/reservation stats.
3. Clients: stats (`active_client_count`/`known_client_count`/`offline_client_count`) + active-over-time timeseries (count-anomaly = visible spike/drop, there's no explicit anomaly metric) + distribution BARCHARTS by band/link/ssid.
4. WiFi experience: per-client signal distribution (histogram of `client_signal_dbm{mac}`, 54 series) + poor-* summary stats (`clients_poor_signal{threshold="-70"}`/`clients_poor_satisfaction{threshold="50"}`/`clients_high_retries{threshold="10"}`) + per-AP airtime timeseries (`radio_cu_total{device,radio}`) + `device_satisfaction{device}` bargauge (NOT `radio_satisfaction` which is the -1 sentinel).
5. DNS-posture (INCLUDED — card says optional; cheap + honest + completes the epic capstone): a plain table of `dhcp_dns_primary{network,dns}` (network→dns handout). NO ✓/drift badge, NO hardcoded expected-DNS — the drift indicator is owned by STAGE-026; do not step on it.

**DHCP both-paths occupancy expr (card requirement: reads sensibly lease-ON and lease-OFF):** `homelab_unifi_dhcp_lease_count` IS implemented (STAGE-012 `ssh_lease.py`) but OPT-IN/default-OFF (gated on `HOMELAB_MONITOR_UNIFI_SSH_LEASE_ENABLED`), so it currently emits no series. The occupancy panel uses a `scalar()`-wrapped `or` fallback so it works BOTH ways without code changes:
`(scalar(homelab_unifi_dhcp_lease_count) or scalar(homelab_unifi_client_count_by_network{network="Default"})) / scalar(homelab_unifi_dhcp_pool_size{network="Default"}) * 100`
— `scalar()` strips labels on both numerator candidates (lease_count has NO labels; client_count_by_network has `network`) so the `or` substitutes cleanly and the division has no label-match constraint. Lease-ON → lease path; lease-OFF (now) → active-client approximation (≈22%). VERIFY empirically in Refinement that it reads ≈22% now. NO deferral — the lease metric exists; the panel degrades gracefully.

**Honest 0-speedtest (verified live):** `speedtest_lastrun` is recent (~16h ago) but `speedtest_download/upload/ping` are ALL 0 — the UDM recorded a 0-result run (not "never run", not a collector bug). The result panels WILL have series reading 0. Make 0 legible: the download/upload timeseries shows the 0 line, an adjacent "Last speedtest" lastrun stat dates it, and a panel description notes "0 = the last UDM speedtest returned 0, not missing data."

**Data-shape notes baked into the design:** single-WAN (WAN metrics scalar, no per-WAN label); single network "Default" (DHCP/by-network panels show one series). `dhcp_pool_start`/`pool_end` are uint32-encoded IPv4 — do NOT render as raw ints (omit them or note encoding). Distribution labels are RAW: band `2.4ghz`/`5ghz`/`6ghz`, link `wired`/`wireless`, ssid human-readable ("Basically Magic Particles"), network "Default" — barchart legends use them as-is. `client_count_by_ap` and `ap_client_count` are REDUNDANT (identical, ap_mac only) → use ONE (`client_count_by_ap`); ap has no friendly name (mac only). WAN throughput is BYTES/sec → ×8/1e6 for Mbps.

**Frontend embed:** new `apps/ui/src/routes/metrics/MetricsNetworkTab.tsx` (mirrors MetricsUnifiTab/MetricsSystemTab: `<GrafanaDashboardEmbed src="/api/grafana/d/network/network?kiosk" title="Network metrics (Grafana)" />`) + `MetricsNetworkTab.test.tsx` (asserts the embed src) + a `{path:'/metrics/network', label:'Network'}` entry in `MetricsLayout.tsx` TABS + a `metricsNetworkRoute` child under `metricsLayoutRoute` in `router.tsx`.

Refinement = 3a (jq valid, provisions, real metric names, no gridPos overlap, DHCP panel reads sensibly both lease-on/off) + 3b (renders against live data + interactive readability pass) → then PROMPT the user to eyeball.

Added two persistent info gauges so Grafana can resolve names: `homelab_unifi_client_info{mac, name}` (client_stats.py, 7th capped family, name = name→hostname→mac fallback) and `homelab_unifi_device_info{mac, device, kind, model}` (device.py, from the device's own mac). **Clients-by-X barchart category-label fix (3 failed attempts — see lessons-learned):** the 4 distribution barcharts initially rendered wrong category labels. Root cause (Grafana 12.3.3): the barchart takes its category (x) axis from the FIRST STRING FIELD of the data frame, NOT from `legendFormat`. A time_series frame has no string field → it fell back to the Time field → every bar labeled "2026" (the timestamp); a raw table frame → the metric-name column was used. FIX: target `format:"table"` + `instant:true` + an `organize` transformation excluding Time/__name__/instance/job/service (and `ap_mac` for the AP panel) so the label column (band/link/ssid/device) is the first string field, + explicit `options.xField` = that label column. This matches the repo's existing working table panels. `label_replace` in the band/link exprs prettifies the raw labels (2.4ghz→"2.4 GHz", wired→"Wired"). Result (verified live): band 2.4 GHz/5 GHz/6 GHz, link Wired/Wireless, ssid human-readable, AP friendly device names. The earlier `range:true` (fan-out → ~54 bars) and the metric-name/timestamp mislabels are all resolved.

**Other readability:** the "Clients by AP" panel is implicitly wireless-only (wired clients aren't AP-attached; bars sum to the wireless count, not total active) — acceptable/correct, noted. Single-SSID environment → the SSID panel shows one bar (kept for future-proofing).
