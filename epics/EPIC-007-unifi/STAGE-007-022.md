# STAGE-007-022: Clients tab (registry table) + Client page (per-client drill-down + DNS-enrichment extension point)

## Status: Complete

## Overview

The **Clients tab** (under Network) + the per-client **Client page** — the surfaces EPIC-007 builds heavily
and EPIC-006 enhances. The Clients tab renders the `unifi_clients` registry as a sortable table; clicking a
row opens the rich Client page. The Client page exposes a documented **DNS-enrichment extension point** (a
stable contract slot) that EPIC-006 plugs Pi-hole DNS-behavior into — EPIC-006 does NOT build its own client
view.

**Vertical slice:** the Clients tab lists the live roster (host `192.168.2.148` first-class); clicking a
client opens its page showing identity + connection (AP+signal / switch+port) + uptime + bandwidth +
per-client DPI + DHCP-lease field + online/offline history; the DNS-enrichment slot renders as an empty/
"provided by Pi-hole integration" placeholder until EPIC-006 fills it.

**Spec references:** spec §9.2 (inventory drill-downs). EPIC-007 "UI structure" (surface 3 + the enrichment
extension point) + "Build order + client-identity ownership" (007 owns the view, 006 enhances). EPIC-006
STAGE-006-027 (the enhancement that plugs in here). STAGE-007-019 (endpoints + extension slot),
STAGE-007-012 (lease field).

## Concrete deliverables

### 1. Clients tab (registry table)

Sortable table of the registry: name/hostname, IP, MAC, connection (AP+signal / switch+port), uptime,
bandwidth, device type, online/offline, optional lease-expiry column. Row click → Client page.

### 2. Client page (per-client drill-down)

Identity + connection + uptime + bandwidth + per-client DPI + DHCP-lease field (expiry / static-vs-dynamic)
+ online/offline history. Host is a first-class client page.

### 3. DNS-enrichment extension point

A documented slot/contract on the Client page (and a column hook on the Clients tab) that EPIC-006 populates
with Pi-hole DNS behavior (query volume / block rate / top domains / recent blocks / DNSSEC). Renders an
empty/"provided by Pi-hole integration" state until 006 lands.

## Design Phase

### Design Notes

**Selected approach** (brainstormer-backed; recommendation selected per pre-authorized Design rule — no user prompt):

1. **Clients tab (table) — A3: fetch-all + client-side sort + search.** One `useUnifiClients(limit=500, offset=0)` call (covers this homelab's ~86 real clients in one request); full client-side clickable-header sort across the whole set; a debounced search box filtering by name/hostname/ip/mac (pure Array.filter). No pager. Wrapped in the shared `QueryState`. Fills the STAGE-021 `NetworkClientsTab` placeholder.

2. **Client page — route `/integrations/network/clients/$mac`** (NEW child of `networkLayoutRoute`, mirroring STAGE-020's `unifiDeviceRoute`/`UnifiDevicePage`; MAC param passed RAW — TanStack Router handles encoding). Back-link to the Clients tab; data-driven `PanelSection`s: Identity (incl. "Host" badge when is_host), Connection (ap_mac / sw_mac+sw_port), Bandwidth (tx/rx via formatBitrate), Signal (new formatSignal, dBm), Uptime (first_seen→last_seen), per-client DPI table (formatBytes), Lease, and the `<ClientDnsSection>` slot.

3. **DNS-enrichment extension point — B1: dedicated `<ClientDnsSection>` + documented contract.** A real component taking `dns: DnsEnrichment | null`: renders an HONEST placeholder when null ("DNS insights provided by Pi-hole — available in a future update"; NEVER fakes 0-counts or domains), and the FULL populated branch (top_domains, blocked_count, last_query_at) written NOW against the typed contract so EPIC-006 supplies data with ZERO UI rework. Documented as a stable contract in a new `apps/ui/src/components/clients/README.md` mirroring the `components/logs/README.md` (LogViewer) precedent, cross-referencing EPIC-006 + the `DnsEnrichment` type. The Clients-tab "DNS column hook" is a DOCUMENTED COMMENT placeholder only (no always-empty column — per-row DNS belongs on the detail page).

4. **Host first-class — C1: pin the is_host row to the top** of the table with a "Host" badge, EXEMPT from the search filter (never hidden, regardless of sort/search); the host links to its Client page like any other row; a "Host" badge also appears on the detail page identity section. For this homelab is_host marks 192.168.2.148.

5. **QueryState extension:** extend the shared `QueryState` with an optional `notFoundLabel` (for the Client page's real 404 contract) rather than hand-rolling the 404 branch — reusable + testable.

**Backend-contract verification (DONE) + a scope clarification:** the `/clients` (paginated `UnifiClientsResponse {clients, total, limit, offset}`) and `/clients/{mac}` (`UnifiClientDetail` + `series` + `dpi[]` + `dns`) endpoints return REAL VM/registry data, typed in schema.ts. **IMPORTANT granularity finding:** the LIST row model `UnifiClientRowModel` carries ONLY mac/ip/hostname/name/network/online/use_fixedip/ap_mac/last_seen/lease_expiry/is_host. The card's "uptime / bandwidth / signal / device-type / switch-port" TABLE columns are NOT in the row payload — they live only on `UnifiClientDetail` (series.signal_dbm/tx_rate_bps/rx_rate_bps, oui, sw_mac, sw_port, first_seen). This is a frontend/backend granularity reality, NOT a drop: those rich fields are RELOCATED to the Client DETAIL page (where the data genuinely exists). The Clients TABLE shows only row-backed columns: name (Link), IP, MAC, network, connection (wireless AP when ap_mac present, else "Wired/Offline" — switch-port detail is detail-page-only), last_seen ("X ago"), online/offline badge, lease-expiry (optional, hidden below a mobile breakpoint), Host badge. NO data is lost — it all renders, just on the detail page for the per-client-detail-only fields.

**New util:** `formatSignal(dbm: number|null)` in unifiFormat.ts ("-58 dBm" / "—"). Reuse formatBitrate (rates), formatBytes (DPI), formatRelative/formatAbsolute/formatUptime (timestamps).

**Documented ceiling (tracked, not silent):** the fetch-all `limit=500` covers this homelab (~86 clients) but is a ceiling; a code comment at the `useUnifiClients(500,0)` call site + this note record that a server-side sort/filter param would be the follow-up IF client count ever approaches 500. (No new stage created now — this homelab is far under the ceiling; revisit only if it grows.)

**Mobile:** wide table uses `overflow-x-auto`; name column primary; lease-expiry + connection-detail columns hidden below a breakpoint; search box stacks full-width.

**Honest-data integrity:** host first-class never hidden; DNS placeholder never fakes Pi-hole data; wired clients show "—" for null signal/rates; 404 client renders an honest "not found" state.

## Refinement Phase

### Refinement Notes

**SCOPE EXPANSION (user-directed, folded into this stage):** During Refinement the user asked about empty per-client DPI. Investigation (read-only, live UDM) found the DPI data IS present in the controller (the user confirmed seeing per-client traffic/activity in the UniFi UI), but the classic `stat/stadpi` endpoint the collector polls returns empty on UniFi Network 10.4.57 — Ubiquiti moved per-app traffic to the v2 traffic API (matches the documented unpoller regression #834 → PR #985). User directed: fix it all in THIS stage, correctly, no half-measures, plus audit other endpoints for staleness.

This expands STAGE-022 from frontend-only to **frontend + BACKEND**. Gate is now BOTH `make verify` (100% branch coverage, pyright strict — backend) AND `make ui-verify` (frontend); the `openapi-export` hook will regenerate `packages/shared-types/openapi.json` (the `/clients/{mac}` DPI/contract may change). Backend changes get live-UDM Refinement validation (not just unit tests).

**DPI work folded in (all done in this stage):**
1. **Bug — per-client DPI query label mismatch:** `/clients/{mac}` filtered `homelab_unifi_client_dpi_bytes{mac="..."}` but the collector emits the MAC under the `client` label → per-client DPI panel always empty even with data. Fix: `{client="..."}` (`integrations_unifi.py:1334`).
2. **Switch the DPI collector to the v2 traffic API** (the fix that makes data FLOW): new v2 client helper + `v2_traffic(start_ms,end_ms)` in `kernel/unifi/client.py` (base `/proxy/network/v2/api/site/default/`, header `X-API-KEY`); rewrite `client_dpi.py` to consume `client_usage_by_app[].usage_by_app[]` (fields: application, category, bytes_received, bytes_transmitted, total_bytes, activity_seconds — all int; client.mac). CRITICAL: `start`/`end` are epoch-MILLISECONDS (epoch-seconds silently returns empty-200). Emit `homelab_unifi_client_dpi_bytes{client,app,cat}` (+ optionally the site-wide `total_usage_by_app` for the global `/dpi`).
3. **Bug — app/cat shown as numeric IDs, not names:** resolve UniFi application/category numeric IDs → human names (compound key `(category<<16)+application` for apps; category keyed on category id). Bundle a catalog; fall back to the raw ID gracefully when an ID isn't in the catalog (firmware may add IDs the bundled catalog lacks — never crash, never blank).
4. **DPI empty state:** the existing "No DPI data for this client." empty state is KEPT AS-IS (user confirmed it's fine).
5. **Orphaned `stat/dpi`:** the classic `stat_dpi()` client method + `stat/dpi` is empty-sentinel on 10.x and has NO collector consumer (dead scaffolding) — remove it OR repoint the global `/dpi` endpoint to the v2 `total_usage_by_app` (decide in Build; do not leave a known-dead endpoint).

**Endpoint-staleness audit (user-requested, DONE — read-only live probe of every Unifi collector endpoint on 10.4.57):** ONLY the DPI endpoints are genuinely outdated. `stat/device`, `stat/sta`, `stat/alluser`, `stat/health`, `rest/networkconf`, `stat/sysinfo`, the v1 endpoints — ALL populated + flowing to VM (verified). `rest/alarm` (threats/IPS) returns empty but for a LEGITIMATE reason (IPS is `disabled` in the controller — `get/setting/ips` ips_mode="disabled"), NOT a deprecated endpoint; it will populate if the user ever enables IPS. The SSH lease collector (`ssh_lease.py`) is env-gated off by default (expected empty). So the audit scope beyond DPI is: nothing else to migrate.

**Live-validation outcome (user approved BOTH Desktop + Mobile):** Validated on the rebuilt PROD stack (`make dev-prod`, :29090) with REAL live-UDM data. Clients tab: ~86 clients, host (intelnuc/192.168.2.148, is_host) pinned + filter-exempt. Client page: identity/connection/bandwidth/lease/DNS-placeholder render; **DPI now flows via the v2 traffic API** — `homelab_unifi_client_dpi_bytes` went from 0 → 100 series; per-client `dpi[]` populates (Bug 1 fix validated on real data); global `/dpi` returns rows. **App/cat name resolution validated on live data: ~88.6% app coverage (31/35 distinct apps resolved to names — Amazon Instant Video, Facebook, Instagram, LinkedIn, GitHub, etc.), 100% category coverage (10/10).** The remaining 4 distinct app IDs (15/195/199/33) are not in the 2,279-app vendored catalog and fall back to raw numeric IDs honestly (their categories still resolve). Both gates green: `make verify` (backend 100% branch coverage, pyright strict) + `make ui-verify`.

**Catalog provenance + refresh:** the DPI app/cat catalog (`dpi_catalog_data.py`, 2,279 apps / 22 cats) is a VENDORED snapshot from ubntwiki cat_app.json (cross-validated against unpoller dpi.go), compound-keyed `(category<<16)+application` for apps, category-id for cats; refresh ~yearly. Unknown IDs raw-ID-fall-back (never blank/crash). Chasing the last ~4 uncatalogued app IDs would require extracting the controller's per-build JS bundle — out of scope; the honest fallback covers them.
