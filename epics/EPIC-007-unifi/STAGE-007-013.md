# STAGE-007-013: Controller-up composite + API-latency collector

## Status: Complete

## Overview

The "is the integration itself healthy" collector — emits the `homelab_unifi_up` composite (can we reach +
authenticate to the controller) and `homelab_unifi_api_took_seconds{endpoint}` (per-endpoint latency, from
the `took`/timing the client captures). This is the universal health pattern every bundle has; it drives the
`UnifiControllerDown` (critical) and `UnifiApiSlow` (info) alerts.

**Vertical slice:** the collector emits `homelab_unifi_up` (1 when reachable+authed) and per-endpoint
`api_took_seconds` against the live UDM; visible in VM; feeds the controller-down alert.

**Spec references:** spec §6.2 (`homelab_target_up` universal up/down). EPIC-007 "Metric families"
(controller-up row) + alert catalog (`UnifiControllerDown`, `UnifiApiSlow`). Mirrors Pi-hole's `up` +
`api_took` pattern (EPIC-006).

## Design Phase

**Status**: [x] Design (Complete)

### Design Notes

**Selected approach (coordinator-selected per epic-stage workflow; brainstormer consulted on both design choices):**

**Choice 1 — `homelab_unifi_up` compositing = Dedicated cheap probe (Option A).**
The collector calls `ctx.unifi.stat_sysinfo()` itself. `homelab_unifi_up = 1` iff HTTP 200 + `meta.rc == "ok"` + non-empty `data`; `0` otherwise. Self-contained: no cross-collector shared state, no reads of other collectors' last-success. Clean "unreachable vs auth" separation straight from `UnifiError.reason`. Independent 30s health cadence. Matches the card's "composite from stat/sysinfo" spec ref and the EPIC-006 Pi-hole `up`+`api_took` pattern. Rejected Option B (aggregate last-successful-fetch across collectors) — requires shared state the project standard discourages, makes the failure reason ambiguous, and explodes branch-coverage combinatorics. Rejected Option C (hybrid) — same coupling for marginal benefit; per-collector freshness is better expressed later as PromQL over `homelab_collector_run_*`.

**Choice 2 — API-latency deliverable = add only the probe's own latency; NO new aggregate metric (Option A).**
The 7 existing HTTP collectors already emit `homelab_unifi_api_took_seconds{endpoint}` per-endpoint. 013 adds exactly ONE new member to that family: `homelab_unifi_api_took_seconds{endpoint="stat/sysinfo"}`. The "composite/normalized view" the card describes is the full family across the `{endpoint}` label. p95 aggregation belongs in PromQL, owned by the STAGE-007-015 `UnifiApiSlow` rule (`quantile`/`histogram_quantile` over `{endpoint}`), NOT pre-aggregated in the collector. This avoids double-emission on a shared series, avoids cross-collector shared state, and preserves per-endpoint granularity (a real p95 needs raw per-endpoint samples). Rejected Option B (new aggregate gauge) — needs shared state and a pre-aggregated gauge cannot yield a true p95.

**Metric series emitted by STAGE-007-013 (and only 013):**
- `homelab_unifi_up` (gauge, value `1`/`0`) — NO `reason` label, so `homelab_unifi_up == 0` stays a single stable series for the critical `UnifiControllerDown` alert (no label churn when the failure mode changes).
- `homelab_unifi_up_reason{reason=...}` (info-gauge, value always `1.0`) — `reason` in {`ok`, `unreachable`, `timeout`, `auth`, `rate_limited`, `http_error`, `bad_response`, `empty_data`, `not_configured`}. This surfaces unreachable-vs-auth as a queryable/alertable dimension; the 015 alert annotation joins it for human-readable context ("UDM unreachable" vs "auth failed -> check API key"). Chosen over a label on `up` (avoids cardinality flap + keeps the alert key stable) and over annotation-only (reason stays graphable).
- `homelab_unifi_api_took_seconds{endpoint="stat/sysinfo"}` (gauge) — joins the existing family. No new aggregate metric.
- `homelab_collector_run_*` — kernel-provided automatically.

**Per-outcome emission rules:**
- `ctx.unifi is None` -> `ok=False`; emit `up=0` + `up_reason{reason="not_configured"}`. Do NOT emit `api_took_seconds` (no request happened). NOTE: this is a deliberate divergence from the other 7 collectors (which emit nothing on null-ctx) — 013 is the HEALTH collector, so it must still publish `up=0` when unconfigured so `UnifiControllerDown` can fire. Confirm the exact null-ctx idiom against the existing collectors during Build.
- `UnifiError(reason=R)` -> `up=0` + `up_reason{reason=R}`. Do NOT emit `api_took_seconds` (no successful timing).
- `UnifiResponse` but `meta.rc != "ok"` OR empty `data` -> `up=0` + `up_reason{reason="bad_response"|"empty_data"}`. DO emit `api_took_seconds{endpoint="stat/sysinfo"}` (HTTP layer succeeded; latency is valid signal).
- Success (200 + `rc=="ok"` + non-empty data) -> `up=1` + `up_reason{reason="ok"}` + `api_took_seconds{endpoint="stat/sysinfo"}=took_seconds`.

**No new infrastructure:** No new client helper (`stat_sysinfo()` already exists). No shared state. No migration. No new cardinality cap (bounded single-series).

**Live recon (read-only, this Design):** `stat/sysinfo` valid key -> HTTP 200, `meta.rc=="ok"`, `data` = 1-entry list with `version`(str)/`uptime`(int)/`ubnt_device_type`(str). Bad key -> HTTP 401 JSON `{"error":{"code":401}}` (client maps to `reason="auth"`). Unreachable -> connection error (`reason="unreachable"|"timeout"`). Confirms the probe + the unreachable/auth distinction are both reliably observable.

**run() branches for 100% branch coverage:** null-ctx; `isinstance(result, UnifiError)` true (reason carried table-driven; parametrize all reasons in tests) / false; `rc != "ok"`; empty `data`; success. Design reason-mapping table-driven (not per-reason `if`) to keep branch count low; tests parametrize every reason value to prove the `up_reason` label.
