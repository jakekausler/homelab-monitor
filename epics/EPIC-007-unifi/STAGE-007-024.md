## Design Phase

**Status**: [x] Design (Complete)

### Design Notes

**Selected approach: Option B — fleet overview TABLE + kind-scoped grouped timeseries, no template-repeat.** (Brainstormer Opus consulted; main agent selected per pre-authorization.) Authors `deploy/grafana/dashboards/unifi.json` (uid `homelab-unifi`, title "UniFi Network", tags `["homelab-monitor","unifi"]`, schemaVersion 40, datasource `{type:prometheus, uid:victoriametrics}` on every panel AND target), bind-mount provisioned (30s poll, no rebuild for JSON edits) into the `Homelab` folder via `deploy/grafana/provisioning/dashboards/default.yaml`. Plus a `/metrics/unifi` tab embedding it.

**Why Option B over A/C:** the fleet table is the best at-a-glance gear-health artifact; static flat-`panels[]` + fixed gridPos matches the existing dashboards (host-overview.json) and is easiest to author correctly + verify before load; it avoids the empty "looks-broken" panels that `$device` template-repeat (Option A) would stamp across the wrong device kinds (ports on the AP/PDU, radios on the gateway/switch). Option C (8 static per-device sections) doesn't scale to 8 heterogeneous devices and buries cross-device comparison.

**Kind-scoping simplification (adjustment to the brainstorm):** ports metrics only exist for switches+gateway and radio metrics only exist for APs in live data, so plain `homelab_unifi_port_*` / `homelab_unifi_radio_*` queries are ALREADY kind-scoped — NO `group_left` join needed for the Ports/Radios rows (verify this still holds in Build). The `device_up` join (`* on (device) group_left(kind,model) device_up`) is used ONLY where the fleet table needs to ANNOTATE rows with kind/model/firmware (those labels live only on `device_up`/`device_state`).

**Rows (6, gear-centric — network concepts WAN/DHCP/clients are STAGE-025's `network.json`, NOT here):**
1. Device health (expanded) — fleet overview table (device/up/cpu/mem/temp/uptime/firmware/update via the device_up join) + cpu/mem grouped timeseries by device.
2. Controller & API (expanded, compact, high) — `homelab_unifi_up`, `up_reason{reason}`, `api_took_seconds{endpoint}` (8 endpoints incl. `v2/traffic`); the "is the collector working" signal up top.
3. Switch ports & PoE (collapsed) — per-port up/speed/poe_power/errors/satisfaction (down ports report 0; non-PoE poe 0).
4. AP radios (collapsed) — cu_total/num_sta/tx_power/tx_retries_pct/channel/bandwidth by device,radio (wifi0=2.4/wifi1=5/wifi2=6).
5. DPI top-apps (collapsed) — see spike-clamp below.
6. Threats (collapsed) — honest minimal (see below).

**DPI per-counter spike-clamp (card §2.5 — relocated from STAGE-009/015; this dashboard OWNS the guard):** the DPI top-apps panel uses `topk(10, clamp_max(sum by (app)(rate(homelab_unifi_client_dpi_bytes[$__rate_interval])), <sane Bps ceiling>))` — `rate()` over `$__rate_interval` handles counter resets natively; `clamp_max` neutralizes Ubiquiti's per-counter cumulative-spike bug so a spurious jump can't render as a false traffic spike. Render as a horizontal bar chart (ranking). The collector emits RAW cumulative bytes by design; the clamp lives here in the consumer. Live DPI: `client_dpi_bytes{client,app,cat}`, 101 series / 34 apps / 10 cats; one series has an unresolved numeric `app="15"` (UniFi emits an un-resolved app id) — leave it as-is, add a panel description noting it's a source-side unresolved id (honest, no fabrication). Also show `dpi_client_records` + `dpi_enabled` stats.

**Threats honesty (load-bearing — verified live):** ONLY `homelab_unifi_threat_count` exists (single scalar, value 0, NO type/device labels). `homelab_unifi_threat{type=...}` does NOT exist (IPS disabled). So the Threats row = one `stat` on `threat_count` (currently 0) + a `text`/markdown panel stating IDS/IPS is disabled and real threats flow via the structured-alarm path, not per-type metric series. Do NOT author any `threat{type}` panel — that would imply non-existent data. (See project memory `project_udm_threat_data_reality.md`.)

**Sentinel / partial-data handling:** `radio_satisfaction = -1` on ALL radios (no real data) → filter `!= -1` at the query level so it never renders as 0% or skews aggregates. `device_temperature_celsius` exists ONLY for the UDM Pro (labels device/name/type) → the temp panel is titled/described as gateway-only (single device), not "broken".

**Collapse defaults (conscious divergence — existing dashboards expand all rows):** Device health + Controller/API expanded (the glance + collector-up signal); Switch ports / AP radios / DPI / Threats collapsed (investigate-on-demand detail). Keeps the default page short for a homelab glance; documented here so it's a deliberate choice, not drift.

**Frontend embed:** new `apps/ui/src/routes/metrics/MetricsUnifiTab.tsx` (mirrors `MetricsSystemTab.tsx`: `<GrafanaDashboardEmbed src="/api/grafana/d/homelab-unifi/homelab-unifi?kiosk" title="Unifi metrics (Grafana)" />`) + a `metricsUnifiRoute` child (`path:'unifi'`) under `metricsLayoutRoute` in `router.tsx` + a `{path:'/metrics/unifi', label:'Unifi'}` entry in `MetricsLayout.tsx` TABS + a tiny tab test asserting the embed src (mirrors `MetricsSystemTab.test.tsx`) for the coverage gate.

**Gates:** dashboard JSON → `jq -e .` valid + no gridPos overlap + provisions + panels reference real metric names; the frontend embed tab → `make ui-verify` (it edits .tsx + router). Refinement = 3a (jq valid, provisions, real metric names, no gridPos overlap) + 3b (renders against live data on the prod rig + interactive readability pass) → then PROMPT the user to eyeball the dashboard.

**No deferrals:** the DPI spike-clamp (relocated from STAGE-009/015) is implemented in THIS stage; nothing dropped from the card.

### Refinement Notes

3a verified in Build (jq valid, 37→38 panels, no gridPos overlap, datasource on every target, real metric names). Dashboard bind-mount auto-provisioned into Grafana cleanly (uid `homelab-unifi`, no provisioning errors). Prod rig rebuilt + UI bundle sha256-proven for the `/metrics/unifi` tab; 3b = interactive readability pass against LIVE data.

**Readability-pass fixes (user-driven, iterated live):**
- Fleet table was BROKEN (devices repeated 3×, "Value #A/B/C" columns) — root cause: instant-table targets with differing extra labels (kind/model on device_up, version on firmware_info, name/type on temperature) fractured the Grafana `merge`. Fix: every fleet-table target now `max by (device)` (strips extra labels → one frame per device) + merge + organize-rename; firmware split to its own table panel. Now one row per device, named columns.
- CPU % + Memory % → timeseries; Uptime → table; PDU relay state → timeseries; Port satisfaction → timeseries; Channel & bandwidth → table (panel-type changes per user readability preference).
- Removed the per-radio satisfaction panel (every radio reports the `-1` "no data" sentinel on the U7-Pro-Wall hardware — genuinely unavailable; an always-empty panel is dishonest).

**AP satisfaction recon + new metric (user asked "can we get satisfaction from the APs?"):** read-only UniFi recon found the per-radio `radio_table_stats[].satisfaction` is `-1` (unavailable on this hardware) BUT the device-level `satisfaction` field in the same `stat/device` payload is REAL. Added collector metric `homelab_unifi_device_satisfaction{device}` (device.py `_emit_device_level`, guarded `isinstance(int|float) and not bool and >= 0` to skip the -1 sentinel + bool edge case) + tests covering all guard branches (AP=98 emit / switch=-1 skip / absent skip / bool skip) — `make verify` 100% branch coverage. Live: 5 devices report real satisfaction (both APs=99, switches=100, PDU=91; UDM gateway absent → -1 skipped). Panel titled "Device satisfaction" (not "AP satisfaction") since it honestly covers all reporting devices, replacing the removed empty per-radio panel.

**DPI top-apps rendered 0 (user-reported bug):** root cause — DPI collector polls every 300s (5min) but vmagent scrapes /metrics every 15s, so `$__rate_interval` (~60s) was shorter than the DPI sample spacing → `rate()` saw a flat counter between polls → 0. Fix: converted the DPI panel barchart→timeseries with target `interval:"5m"` (so `$__rate_interval` ≥ 5min ≥ the DPI cadence), removed `instant:true`, unit Bps, kept the `clamp_max(...,125000000)` spike-guard. Verified: range query (5m step) returns 26 app series non-zero over time.

**DPI SSL/TLS exclusion (user request):** excluded the broad `SSL/TLS` catch-all (`{app!="SSL/TLS"}`) which dominated and obscured real apps; kept specific SSL-named apps (Google APIs(SSL) etc.). Panel retitled "Top apps by DPI throughput (excl. SSL/TLS)" + description notes the exclusion.

**Scope note:** the AP-satisfaction collector metric is a BACKEND addition beyond the original card (dashboard-only) scope — it emerged from a Refinement request; covered by `make verify` (100% branch) and the Finalize code review. No API/openapi/migration change (Prometheus metric only).
