# STAGE-007-007: Active-client identity collector → registry upsert

## Status: Complete

## Overview

The identity collector — polls `stat/sta` (active) and `stat/alluser` (known) and drives the
identity-upsert helper (004) to populate the persistent `unifi_clients` registry: MAC↔IP↔hostname↔
AP/switch-port↔fixed-IP, online/offline, with time-stamped IP↔MAC observations. This is the authoritative
client-identity layer EPIC-007 owns and EPIC-006 enhances.

**Vertical slice:** the collector runs on its interval, upserts the live client roster into the registry
(56 active clients on the live UDM), appends observations, attributes the host `192.168.2.148`, and emits
roster-count self-metrics; the registry is queryable.

**Spec references:** spec §6.1 (registry). EPIC-007 "Persistent client registry" + "Client-join contract".
STAGE-007-003 (table), STAGE-007-004 (upsert helper). Feeds `UnifiNewClient`, `UnifiClientCountAnomaly`,
the EPIC-006 enrichment, and the digest "what changed".

## Cross-cutting standards

- `unifi` group; 60s cadence. Cardinality cap (004) applied to any per-client emitted series.
- Host `192.168.2.148` is first-class; never dropped.
- New-client detection: a never-before-seen MAC sets the signal for `UnifiNewClient` (info) — EPIC-011 later
  upgrades this into a discovery/suggestion.
- All timestamps UTC; observations time-stamped for the time-windowed join.

## Concrete deliverables

### 1. Identity collector

Polls `stat/sta` + `stat/alluser`; calls the 004 upsert helper; updates `unifi_clients` + observations.
Emits roster rollups (active count, by-network/SSID/AP/band counts → `UnifiClientCountAnomaly` /
`UnifiSsidClientCountAnomaly`) and a `new_client` signal (→ `UnifiNewClient`). Marks online/offline
transitions. Emits `homelab_collector_run_*`.

## Design Phase

**Status**: [x] Design (Complete)

### Design Notes & Selected Approach

**Selected approach:** A new `UnifiActiveClientCollector(BaseCollector)` mirroring the `unifi/wan.py` + `device.py` collectors. It fetches `stat/sta` (active) + `stat/alluser` (known), then drives the EXISTING STAGE-007-004 helper `upsert_identity(conn, *, stat_sta, stat_alluser, host_lan_ip, observation_cutoff, now) -> UpsertResult` INSIDE a caller-owned transaction to populate the `unifi_clients` registry + observations, and emits roster rollups + a new-client signal + the UpsertResult self-metrics. Brainstormer WAS used (4 real design decisions, locked below).

**Collector ClassVars:** `name="unifi_active_client"`, `interval=timedelta(seconds=60)`, `timeout=timedelta(seconds=30)` (TWO API calls), `concurrency_group="unifi"`. Bundle wiring: add `from .active_client import UnifiActiveClientCollector` + append to `_UNIFI_COLLECTORS` in `plugins/collectors/integrations/unifi/__init__.py` (device + wan stay).

**Invocation contract (load-bearing):** `repo = UnifiClientRepo(ctx.db)` (constructed from `ctx.db: SqliteRepository`). The collector COMPUTES `now` (ISO-8601 UTC) and `observation_cutoff` (= now - `UnifiConfig.observation_retention_days` [90] days, ISO-8601 UTC) — these are INJECTED into the helper (the helper makes NO time calls, for determinism). Use the project's UTC-now util (check how other code gets UTC now — e.g. `kernel` utc_now/`datetime.now(UTC)`; mirror the existing convention; do NOT use a naive datetime). `host_lan_ip` from `UnifiConfig.host_lan_ip` ("192.168.2.148"). Pattern:
```
repo = UnifiClientRepo(ctx.db)
async with ctx.db.transaction() as conn:
    prior_macs = {row.mac for row in await repo.list_clients()}   # READ INSIDE the txn (atomic with the write) -- for new-client detection
    result = await upsert_identity(conn, stat_sta=sta_data, stat_alluser=alluser_data, host_lan_ip=host_ip, observation_cutoff=cutoff_iso, now=now_iso)
```
NOTE: `list_clients()` is an INSTANCE method on UnifiClientRepo (a separate read) — confirm at Build whether it can run on the SAME `conn` or is a separate connection; if it must be a separate read, read `prior_macs` via `repo.list_clients()` immediately BEFORE opening the write txn (acceptable: a client newly appearing between the read and the upsert is at worst flagged new one tick late). Planner decides the exact atomicity mechanism; prefer reading prior_macs as close to the upsert as practical.

**LOCKED DESIGN DECISIONS (brainstormer options -> coordinator selections):**

- **A. New-client detection = A1 (pre-upsert MAC-set snapshot diff).** Snapshot `prior_macs = {mac for each row in repo.list_clients()}` BEFORE the upsert; after the upsert, `new_macs = current_macs - prior_macs` where `current_macs` = the set of macs successfully extracted from the live stat/sta+stat/alluser parse (use the SAME extraction the helper uses, or derive from the parsed records' `mac` field with the same canonicalization). EXCLUDE the host sentinel mac `host:<ip>` from both sets (it's seeded, not a real association). **Why NOT A2 (`first_seen==now` query): the helper sets a new MAC's `first_seen` from the record's epoch `first_seen_iso`, only falling back to `now` if absent — so `first_seen==now` MISSES most genuinely-new clients. A2 is incorrect against the real helper.** Emit BOTH metric shapes:
  - `homelab_unifi_new_client_total` — a COUNTER incremented by `len(new_macs)` each tick (Alertmanager `increase(...) > 0` fires reliably across scrapes; a one-tick gauge can be missed). Use `ctx.vm.write_counter(...)`.
  - `homelab_unifi_new_client{mac,hostname,network}` — a per-MAC info GAUGE set to 1.0 for each new mac this tick (for the UnifiNewClient alert annotation + EPIC-011 discovery). New-macs/tick is ~0 normally -> trivial bounded cardinality. (hostname/network from the parsed record; use "" or omit-label if absent — pick a consistent convention, e.g. label value "" when absent.)
- **B. Roster rollups = B2, ALL computed from the LIVE `stat/sta` parse** (the registry `unifi_clients` table does NOT store `essid`/`radio`/`is_wired`, so per-SSID/per-band/wired-split MUST come from the live parse). Emit (gauges):
  - `homelab_unifi_active_client_count` (no labels) = count of stat/sta records (online). Drives `UnifiClientCountAnomaly`.
  - `homelab_unifi_known_client_count` (no labels) = count of distinct macs across stat/sta + stat/alluser.
  - `homelab_unifi_offline_client_count` (no labels) = known - active (alluser macs not in sta).
  - `homelab_unifi_ssid_client_count{ssid}` = per-`essid` online count (wireless only; `essid` absent on wired). Drives `UnifiSsidClientCountAnomaly`.
  - `homelab_unifi_client_count_by_network{network}` = per-`network` online count.
  - `homelab_unifi_client_count_by_ap{ap_mac}` = per-`ap_mac` online count (wireless).
  - `homelab_unifi_client_count_by_band{band}` = per-`radio` online count; map radio "ng"->"2.4ghz","na"->"5ghz","6e"->"6ghz" for the label (or keep raw "ng"/"na"/"6e" — pick ONE; recommend mapping to human bands). Wireless only.
  - `homelab_unifi_client_count_by_link{link}` = online count split `wired`/`wireless` via `is_wired`.
  All dimensions are BOUNDED handfuls (SSIDs/APs/networks/bands/link) -> no cardinality cap needed (per-client STATS capping is STAGE-007-008's job; these are aggregate COUNTS). Verify no rollup label is per-MAC/per-IP (would be unbounded) — none of the above are.
- **C. API latency = C1.** Emit `homelab_unifi_api_took_seconds` TWICE: `{endpoint="stat/sta"}` from the sta response's `took_seconds`, and `{endpoint="stat/alluser"}` from the alluser response's `took_seconds`. Emit each ONLY for an endpoint that actually returned a UnifiResponse (skip the latency emit for a failed call). Each successful emit increments the emitted counter.
- **D. Failure semantics = D2 (asymmetric degrade).** If `stat_sta()` returns UnifiError -> hard fail the run (`ok=False`, `errors=[sta_err.message]`, no upsert — the online roster is irreplaceable). If `stat_alluser()` returns UnifiError -> PROCEED: call the helper with `stat_alluser=[]` (the helper's PASS 2 no-ops on []), still do all rollups/new-client/transitions from stat/sta, emit `homelab_unifi_alluser_degraded`=1.0 (else 0.0 on success), include the alluser error in the result `errors` list, log a warning, and return `ok=True` (the run did its primary job). None-client guard (`ctx.unifi is None`) -> `ok=False, errors=["unifi client not configured"]`.

**LIVE-RECON FIELD MAP (from the real UDM stat/sta + stat/alluser — confirms the 004 helper reads the right fields + grounds the rollups):**
- Wrapper `{"meta":{"rc":"ok"},"data":[...]}`; no `meta.took` on either; records list is `payload["data"]`. Live counts: 59 active (stat/sta) [14 wired, 45 wireless], 85 known (stat/alluser), 27 offline-only.
- **stat/sta identity fields** (all present unless noted): `mac`(str, all), `ip`(str, all — the CURRENT ip), `hostname`(str, ~51/59 — KEY ABSENT when unset, NOT ""), `name`(str, ~41/59 — key absent when unset), `oui`(str, all), `network`(str, all), `ap_mac`(str, wireless-only 45/59), `sw_mac`+`sw_port`(int, wired-only 14/59), `use_fixedip`(bool, 15/59 — spelling `use_fixedip` no underscore), `fixed_ip`(str, co-present), `first_seen`/`last_seen`/`assoc_time`/`latest_assoc_time`(epoch-second INTEGERS, all), `is_wired`(bool, all — authoritative wired/wireless gate). NO string-encoded numerics on these endpoints (the poe-style gotcha does NOT appear here).
- **stat/sta rollup fields** (wireless-only, gated by `is_wired==False`): `essid`(str), `radio`(str "ng"/"na"/"6e"), `channel`(int), `bssid`(str). Wired clients have ZERO of these.
- **stat/alluser differences** (the helper relies on these): NO `ip` (use `last_ip`(str, all)); SPARSE (~43 keys); NO connection fields (no ap_mac/essid/sw_port/assoc_time/network); has `mac`(all), `last_ip`(all), `hostname`(~74/85), `name`(~52/85), `oui`(all), `is_wired`(all), `first_seen`/`last_seen`(epoch int, all), `use_fixedip`/`fixed_ip`(~24/85). `network` is ABSENT in alluser.
- **Host reconciliation:** host `192.168.2.148` IS present in stat/sta (wired, `network="Default"`, has `sw_mac`) with `ip==192.168.2.148`, so the helper's `promote_to_host_conn` fires. The collector passes `host_lan_ip="192.168.2.148"`.
- **Timestamps:** all epoch-second ints; the 004 helper converts to ISO internally (the collector does NOT convert — it just forwards the raw record dicts to the helper).

**Self-metrics from UpsertResult:** emit `homelab_unifi_identity_clients_upserted`, `_observations_appended`, `_hosts_reconciled`, `_skipped` (gauges of the per-tick UpsertResult counts). Plus the kernel-provided `homelab_collector_run_*`.

**Online/offline transitions:** the card mentions "marks online/offline transitions". The helper already sets `online=True` for stat/sta clients and `online=False` for alluser-only clients (PASS 2). The collector does NOT need extra transition logic beyond what the helper does + the rollup counts (active vs offline). If a dedicated transition COUNTER is desired, derive from prior_macs online-state vs current — but keep it SIMPLE: the rollup `active_client_count` + `offline_client_count` already expose the aggregate; a per-transition signal is OUT OF SCOPE for 007 (the digest "what changed" / EPIC-006 consume the registry directly). Do NOT add per-client transition events this stage.

**Parse helper:** the collector must turn `payload["data"]` (a list of records, each `dict[str,object]`) into `list[dict[str, object]]` to pass to `upsert_identity(stat_sta=..., stat_alluser=...)`. Narrow defensively: `isinstance(payload, dict)` -> `cast("dict[str,object]", ...)`; `data = payload.get("data")`; `isinstance(data, list)`; build `[cast("dict[str,object]", r) for r in data if isinstance(r, dict)]`. Non-dict entries skipped. For the rollups, iterate the SAME parsed sta records reading `essid`/`radio`/`network`/`ap_mac`/`is_wired`/`mac` with isinstance guards (use shared `as_bool` for `is_wired`; string fields via isinstance str). Reuse shared `_parsing.py` (`as_float`/`as_bool`/`emit_numeric`) where applicable.
