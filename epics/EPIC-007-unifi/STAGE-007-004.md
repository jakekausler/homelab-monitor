## Overview

The shared helper layer the Wave-B collectors build on: reuse the EPIC-005-004 cardinality-cap mechanism for
per-client/per-device/per-DPI series, and build the **identity-upsert helper** that maps `stat/sta` (active)
and `stat/alluser` (known) records into the `unifi_clients` registry (upsert by MAC, append a time-stamped

collector (007) and any later consumer reuse it.

**Vertical slice:** given fixture `stat/sta` + `stat/alluser` payloads, the helper upserts the registry,
appends observations, attributes the host correctly, and respects the cardinality cap; unit-tested against
fixtures.

## Design Phase

### Selected Approach (Design)

004 ships (a) the cardinality-cap CONFIG reuse + (b) a PURE identity-upsert HELPER mapping live `stat/sta`+`stat/alluser` JSON â†’ the STAGE-007-003 `unifi_clients` registry. It is NOT a registered collector (STAGE-007-007 builds that, calling this helper). Live shape was recon'd against the real UDM (read-only) to ground the design.

**Live shape (verified â€” design to this):** `stat/sta` = currently-online clients (full fields, current `ip`, ~120 keys); `stat/alluser` = ALL known incl. offline (sparse, 43 keys, a strict SUBSET of sta keys; NO current `ip` â€” only `last_ip`; NO connection fields). No explicit online flag â€” **presence in `stat/sta` = online**. `is_wired: bool` distinguishes wired (`sw_mac`/`sw_port`) vs wireless (`ap_mac`/`essid`/`signal`) in BOTH endpoints. Fixed-IP fields: `use_fixedip` (no underscore) + `fixed_ip`. Display name: `name or hostname or mac`. **`first_seen`/`last_seen` are epoch INTEGER seconds â†’ the helper MUST convert to ISO-8601 UTC TEXT** (the registry stores ISO).

**D1 â€” Two-pass merge + open-conn signature (Option A).**
Signature: `async def upsert_identity(conn: AsyncConnection, *, stat_sta: list[dict[str, object]], stat_alluser: list[dict[str, object]], host_lan_ip: str, observation_cutoff: str, now: str) -> UpsertResult`. The COLLECTOR (007) opens `repo.transaction()` and passes the conn (so it can atomically bundle upserts + its self-metrics; composes with the 003 `*_conn` statics). `now` + `observation_cutoff` are INJECTED (the helper NEVER calls utc_now â€” time-rot rule + test determinism). Merge model: **PASS 1** iterate `stat/sta` (online) â†’ `upsert_client_conn(online=True, current ip, connection fields)` + `append_observation_conn(mac, ip, observed_at)`; track macs in a `seen` set. **PASS 2** iterate `stat/alluser` â†’ for macs NOT in `seen`, `upsert_client_conn(online=False, ip=last_ip, no connection fields)` and NO observation append (`last_ip` is not a current sighting). Record type is raw `dict[str, object]`; a `_extract(record, *, is_online, now) -> ExtractedClient | None` guard isinstance-checks every field and returns None (â†’ skip) when `mac` is absent/not-a-str (the card's "graceful handling of missing fields"). `ExtractedClient` is a frozen dataclass with epochâ†’ISO already applied. Rejected: sta-only (loses offline clients â€” card lists alluser as input); union-into-intermediate (doesn't escape the two-source bookkeeping, more allocation); repo-owning-transaction signature (fights the caller-owns-transaction `*_conn` design, blocks 007 from extending the txn).

**D2 â€” Cardinality cap = CONFIG only; helper caps NOTHING (Option A).**
004 extends the central `CardinalityCapsConfig.families` with the per-family caps for the FUTURE per-client/DPI METRIC families (e.g. `unifi_client_stats`, `unifi_dpi`) + documents them in the bundle config. The helper takes NO `cap` param and applies NO cap. Rationale: the registry is the COMPLETE canonical inventory (a home /24 is ~85 clients, far below any cap; dropping registry rows = losing devices = wrong). The 005-004 cap (`CappedEmitter`) protects per-client/DPI METRIC-series cardinality in Prometheus â€” a real risk in the metric-emitting stages, NOT in the bounded registry or the per-(mac,ip)-bounded observations. Rejected: capping registry rows (contradicts single-source-of-truth); capping inside the helper (004 emits no metrics â€” it's a pure DB helper).

**D3 â€” Sentinel-host reconciliation in the helper, via a new repo method (Option A).**
Add `@staticmethod async promote_to_host_conn(conn, *, real_mac: str, host_ip: str) -> bool` to `UnifiClientRepo`: look up the sentinel `mac=f"host:{host_ip}"`; if absent â†’ return False (no-op); if present â†’ `UPDATE unifi_clients SET is_host=1, first_seen=MIN(real.first_seen, sentinel.first_seen) WHERE mac=:real_mac` then `DELETE` the sentinel row; return True. In the helper PASS 1, after upserting a record whose `ip == host_lan_ip`, call `promote_to_host_conn(conn, real_mac=mac, host_ip=host_lan_ip)` and increment `hosts_reconciled` when it returns True. **Order: upsert real-mac row FIRST (so it exists), THEN promote** (the MIN/UPDATE targets an existing row). Idempotent (sentinel deleted after first reconcile â†’ later runs no-op). Keeping `is_host` mutation in the repo layer is consistent with STAGE-007-003 deliberately keeping `is_host` OUT of the generic `upsert_client_conn`. **Reconciliation is sta-pass-only:** if the host is OFFLINE at run time it appears only in alluser (with `last_ip`), the sentinel persists (still correctly represents the host as a first-class `is_host=1` row), and reconciliation happens on the host's next ONLINE appearance. This is a complete, correct design decision (the host is never un-represented) â€” NOT a deferral.

**D4 â€” Epochâ†’ISO conversion, missing-field tolerance, UpsertResult.**
Timestamps: the record's `last_seen` (epochâ†’ISO via `datetime.fromtimestamp(v, tz=UTC).isoformat()`, int-guarded) feeds BOTH the registry `last_seen` AND the observation `observed_at` (it's the truest "client was at this ip as-of" time for the time-window join); the record's `first_seen` (epochâ†’ISO) feeds the registry `first_seen` (preserved on conflict â€” INSERT-only). Injected `now` is the deterministic FALLBACK when a record's timestamp is missing/non-int. `_extract` skips (returns None, counts `skipped`) iff `mac` absent/not-a-str; every optional field via a uniform typed getter (strâ†’str|None, intâ†’int|None, boolâ†’bool default False). `UpsertResult` (frozen dataclass, 007 emits as self-metrics): `clients_upserted: int`, `observations_appended: int`, `hosts_reconciled: int`, `skipped: int`.

**Module location:** the helper + `ExtractedClient` + `UpsertResult` live in a new module under the unifi integration area (Build picks the exact path â€” likely `plugins/collectors/integrations/unifi/identity.py` or a `kernel/unifi/identity.py`; mirror where 007's collector will import it from). The `promote_to_host_conn` repo method goes in the existing `kernel/db/repositories/unifi_clients_repository.py`.

**Tracked deferrals (no silent scope-drop â€” each owned by a named stage):**
- **Per-client/DPI metric cap APPLICATION** (consuming the config 004 ships) â†’ **STAGE-007-008** (per-client stats: `cap_for("unifi_client_stats")`) + **STAGE-007-009** (DPI: `cap_for("unifi_dpi")`); **STAGE-007-007** applies it to any roster-rollup metric series. 004 ships ONLY the cap config + the documented `CappedEmitter` reuse path â€” no live emit (004 has no metrics; it's a pure DB helper).
- **Live-data exercise of the helper** â†’ **STAGE-007-007** (the active-client collector fetches real `stat/sta`/`stat/alluser` and calls this helper; 004's Refinement is fixtures-only per the card â€” "no 3b delta").

**Scope boundary:** 004 = the cap CONFIG + the pure `upsert_identity` helper + `promote_to_host_conn` + fixtures/tests. STAGE-007-007 = the registered COLLECTOR (fetches the API on a schedule, calls this helper, emits self-metrics + new-client signal). Do NOT register a collector in 004.
