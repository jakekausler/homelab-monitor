## Overview

Author the Unifi log alert rules (`deploy/vmalert/logs/unifi.yaml`) over the multi-format UDM syslog stream
(CEF audit + iptables firewall + systemd) that STAGE-007-016 established. **This stage comes AFTER 016** so
the stream exists and the rules validate against real lines.

block, login event) fires the matching rule → Alertmanager.

catalog" (logs table) + "Events pipeline" (two-paths-one-alert: `UnifiThreatLog` is forensics-only, NO

**Log alert rules (REVISED per STAGE-007-016 live-stream findings — see note below).**
The real UDM stream is multi-format (CEF audit + iptables firewall + systemd), NOT
CEF-everything as originally assumed. The rules below are grounded in events actually
OBSERVED live (audit login, config-change, WAN firewall blocks, earlyoom pressure) plus
the originally-planned device/firmware/threat alerts which were NOT observed yet and are
authored against the documented format with deferred live validation.

Logs-rule schema reminder: these are vmalert-LOGS rules → `type: vlogs`, LogsQL ending in
a `... | stats by (<label>) count() as <n> | filter <n>:>T` pipe, `source_tool: vmalert-logs`,
group/interval mirroring `deploy/vmalert/logs/system.yaml`. NEW file: `deploy/vmalert/logs/unifi.yaml`.

CONFIRMED-real (grounded in STAGE-016 live capture — build + live-validate these):
- **UnifiAdminLoginLog** (info; warning if `src` non-LAN) — RETARGETED from generic "admin
  login lines" to the real CEF audit event: `service:=udm-audit cef_name:="Network Accessed"`
  (sig 544). Surface `udm_admin` + `src`; escalate to warning when `src` is outside the LAN.
- **UnifiConfigChangeLog** (info) — NEW. Real CEF audit config-change: `service:=udm-audit
  cef_name:="Config Modified"` (sig 546). Surface `udm_admin` + `udm_settings_section` +
  `udm_settings_entry`. (Captured live: the operator's Pi-Hole-DNS firewall-group rename.)
- **UnifiWanBlockSpikeLog** (warning) — NEW. External IPs hitting WAN block rules =
  recon/scan/DDoS — the dominant real security signal. `service:=udm-firewall` +
  `fw_chain` matching the WAN block chain (`WAN_LOCAL-D*`) → `stats by (src) count()` over a
  window → `filter count:>T` (spike threshold, NOT per-line, to avoid noise). Surface `src`
  + count + `fw_dpt`.
- **UnifiOomPressureLog** (warning) — NEW. UDM memory pressure: `service:=udm-system
  process:=earlyoom`. A real device-health signal we observed; also a precursor to the UDM
  dropping syslog (which would spike the STAGE-016 parse-failed alert).

PLANNED-but-UNOBSERVED (author against documented format + a fixture; live-validation DEFERRED
until the events actually appear — mirror STAGE-012's operator-gated waiver; do NOT block 017
completion on events that may not fire on this deployment):
- **UnifiPortFlapLog** (warning) — port up/down flap lines. Not observed in the STAGE-016
  capture (needs device-event categories / actual flaps). Author + fixture-validate.
- **UnifiDeviceDisconnectLog** (warning) — AP/gateway disconnect lines. Not observed. Author +
  fixture-validate.
- **UnifiFirmwareEventLog** (info) — firmware update/apply lines. Not observed (no update

FORENSICS-ONLY (NO alert — unchanged):
- **UnifiThreatLog** — IDS/IPS detection lines = a saved query / LogViewer filter for forensics
  ONLY. The threat ALERT fires off the structured alarm path (STAGE-007-010), so no

### Design Notes

**Selected approach (coordinator-selected from brainstormer options; user pre-authorized to self-select).**

**File layout.** New `deploy/vmalert/logs/unifi.yaml`, single group `name: unifi_logs`, `type: vlogs` (REQUIRED — dispatches LogsQL), `interval: 30s`. Plus a TWIN `deploy/vmalert/logs-test/unifi.yaml` (identical group/labels/annotations; `interval: 5s`, `_time:30s`, LOWERED thresholds so a small plant fires) — mirrors the repo convention (every logs/*.yaml has a logs-test twin glob-mounted into vmalert-logs-test). Glob mount already covers both — NO compose edit.

**Labels (every rule):** `severity` (info|warning) + `source_tool: vmalert-logs` + `category: network` + `integration: unifi` (matches metrics/unifi.yaml so Karma groups UDM logs + metrics alerts together; NOT `target_kind` — UDM is network gear, not a host/container).

**Testing (NO promtool for LogsQL — fundamentally different from STAGE-015 metrics rules):**
- (1) Syntax/eval gate for ALL rules: load the prod file into the running vmalert-logs, confirm every rule `health:"ok"` (no `lastError`) at `/api/v1/rules`.
- (2) Firing gate for the 4 confirmed-real rules: plant/observe real UDM lines on the prod rig → confirm fire → Alertmanager with `integration: unifi` labels.
- (3) Regression artifact: the `logs-test/unifi.yaml` twin + a NEW integration test `apps/monitor/tests/integration/test_vmalert_unifi_logs_pipeline.py` (`@integration @slow`, `require_rig_components("victorialogs","vmalert-logs","alertmanager")`) that plants lines + asserts fire→webhook. Auto-skips in `make verify`/`make test-fast`; runs on `make integration`.

**The 8 rules (prod exprs, `_time:5m`):**
1. `UnifiAdminLoginLog` (info, for:0m) — `service:="udm-audit" cef_signature_id:="544"` → `stats by (host, udm_admin, src) count() | filter >0`. (sig-id over cef_name = firmware-stable.)
2. `UnifiAdminLoginExternalLog` (warning, for:0m) — same + `NOT src:ipv4_range(...)` excluding all RFC1918 (192.168/16, 10/8, 172.16/12) → fires when admin logs in from off-network. (`ipv4_range` has NO repo precedent — verify health:ok at Refinement; glob-negation `NOT src:192.168.* NOT src:10.* NOT src:172.16.*…` is the fallback.)
3. `UnifiConfigChangeLog` (info, for:0m) — `service:="udm-audit" cef_signature_id:="546"` → `stats by (host, udm_admin, udm_settings_section) count() | filter >0`. (NOT group-by udm_settings_entry — high cardinality; surface it in the annotation.)
4. `UnifiWanBlockSpikeLog` (warning, for:5m) — `service:="udm-firewall" fw_chain:WAN_LOCAL-D*` → `stats by (host, src) count() as block_count | filter block_count:>20`. (Group by `src` to catch a single scanner; do NOT group by `fw_dpt` — would fragment a port-scan below threshold. >20/5m avoids firing on stray multicast drops.)
5. `UnifiOomPressureLog` (warning, for:0m) — `service:="udm-system" process:="earlyoom" "sending SIGTERM"` → `stats by (host) count() | filter >0`. **FOOTGUN:** earlyoom logs a periodic `mem avail:` heartbeat at info even when NOT killing — a naive `process:="earlyoom"` rule fires constantly. The `"sending SIGTERM"` substring restricts to actual kills. BUT that exact wording is UNVALIDATED (only the heartbeat line is in the 016 fixture) → regression item to confirm the kill-line wording on a real OOM event.
6. `UnifiPortFlapLog` (warning, for:5m) — UNVALIDATED — `service:="udm-system" ("link down" OR "link up" OR "Link is Down")` → `filter >4`.
7. `UnifiDeviceDisconnectLog` (warning, for:0m) — UNVALIDATED — `service:="udm-system" ("disconnected" OR "device" "offline" OR "adopted")` → `filter >0`.
8. `UnifiFirmwareEventLog` (info, for:0m) — UNVALIDATED — `service:="udm-system" ("firmware" OR "Updating firmware" OR "upgrade" "complete")` → `filter >0`.

`UnifiThreatLog` stays FORENSICS-ONLY (no alert). `UnifiUdmLogParseFailed` already ships in metrics/unifi.yaml — NOT duplicated here.

**Annotations:** `summary` (static one-liner) + `description` (folded `>-`, `{{ $labels.* }}` + `{{ $value }}`); optionally an `explorer:` deep-link annotation (mirror home-assistant.yaml).

**UNVALIDATED rules stay IN this stage (authored + health-validated), with live-firing validation DEFERRED via regression items on THIS card — NOT dropped, NOT moved to another stage.** Regression items to file at Refinement: (a) refine UnifiPortFlapLog/DeviceDisconnectLog/FirmwareEventLog substrings when a real event is observed; (b) confirm the earlyoom kill-line wording; (c) verify `ipv4_range` LogsQL syntax + external-login firing.

**Build files:** `deploy/vmalert/logs/unifi.yaml` (new), `deploy/vmalert/logs-test/unifi.yaml` (new twin), `apps/monitor/tests/integration/test_vmalert_unifi_logs_pipeline.py` (new integration test). No compose edit (glob-mounted). No metrics-rule change (parse-fail alert already shipped).
