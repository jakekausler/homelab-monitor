# STAGE-007-008: Per-client stats + WiFi-experience rollups collector

## Status: Complete

## Overview

The per-client quality collector — from the `stat/sta` payload (already fetched for identity), emit
per-client signal/rate/uptime/bytes (capped) and the WiFi-experience rollups (distributions: counts of
clients below signal thresholds, low satisfaction, high retries). This is the "how well are clients
connected" layer, distinct from identity.

**Vertical slice:** the collector emits `homelab_unifi_client_*` per-client gauges (capped) + experience
rollup metrics against the live UDM; visible in VM and the WiFi-experience widget.

**Spec references:** spec §6.2. EPIC-007 "Metric families" (per-client stats + WiFi-experience row). EPIC-005
cardinality-cap precedent (005-004). Feeds `UnifiClientPoorExperience`, the WiFi-experience widget, and
`network.json`.

## Concrete deliverables

### 1. Per-client stats + rollups collector

From `stat/sta`: `client_signal_dbm`, `client_tx/rx_rate_bps`, `client_uptime`, bytes per client (capped
top-N). Experience rollups: count of clients with signal < threshold, count with satisfaction < threshold,
count with high `tx_retries_pct`, per-AP airtime summary. Feeds `UnifiClientPoorExperience` + the WiFi
experience surfaces. Emits `homelab_collector_run_*`.

## Design Phase

**Status**: [x] Design (Complete)

### Design Notes & Selected Approach

**Selected approach:** A new `UnifiClientStatsCollector(BaseCollector)` mirroring `unifi/active_client.py`/`wan.py`. It fetches `stat/sta` (a SEPARATE fetch from the identity collector — both 60s in the `unifi` group, serialized; do NOT share one fetch) and emits per-client QUALITY metrics (signal/rate/uptime/bytes) — CARDINALITY-CAPPED via `CappedEmitter` — plus bounded WiFi-experience rollup counts + a per-AP airtime summary. NO registry writes (007 owns identity). Brainstormer WAS used (4 real decisions, locked below).

**Collector ClassVars:** `name="unifi_client_stats"`, `interval=timedelta(seconds=60)`, `timeout=timedelta(seconds=15)`, `concurrency_group="unifi"`. Bundle wiring: add `from .client_stats import UnifiClientStatsCollector` + append to `_UNIFI_COLLECTORS` (device, wan, active_client stay).

**CAP PATTERN (the defining deliverable — mirror `homeassistant/ha_sensor_value.py`):** in run(): `caps = load_cardinality_caps_config()`; `cap = caps.cap_for("unifi_client_stats")` (=200); build per-family `observations: list[tuple[dict[str,str], float]]`; `events: list[CollectorEvent] = []`; `emitter = CappedEmitter(writer=ctx.vm, events=events)`; for EACH per-client family `survivors = emitter.emit_family(family_name, cap, observations)` and accumulate `emitted += survivors + 1` (CappedEmitter ALWAYS writes the `homelab_metric_family_dropped_series{family}` gauge -> +1 per emit_family call). Return `CollectorResult(ok=..., metrics_emitted=emitted, errors=..., events=events, ...)` — the events list carries the per-family warning SuggestionEvent(s) when dropped>0. Error handling identical to active_client: None-client -> ok=False errors=["unifi client not configured"]; `stat_sta()` UnifiError -> ok=False errors=[msg]; malformed payload -> empty records -> rollups/per-client emit nothing but still emit the api latency, ok=True. Emit `homelab_unifi_api_took_seconds{endpoint="stat/sta"}` on success.

**LOCKED DESIGN DECISIONS (brainstormer options -> coordinator selections):**

- **A. Cap granularity = A2 (per-family emit_family, same cap=200, same `{mac}`-only label set).** Each per-client family is emitted via its OWN `emit_family(name, 200, obs)` call, but ALL per-client families use the IDENTICAL `{mac}`-only label set. Because `CardinalityCapper.apply` sorts by `tuple(sorted(labels.items()))` then slices `[:cap]`, every family with the same `{mac}` candidate set yields the PROVABLY IDENTICAL survivor-MAC set — so a survivor client appears in signal AND rate AND uptime consistently, with NO shared-state machinery (CappedEmitter has none). Meaning of the cap: "~200 distinct CLIENTS per family", NOT 200 total series (config comment: "~85 home clients x a few stat series each -> 200 headroom"). At ~85 real clients the cap never bites (fixture-tested for the >cap path). DO NOT use A1-misread (200-per-family-as-independent-budget) or A3 (one `{mac,stat}` blob — breaks naming + over-prunes to ~33 clients). Build MUST: (1) use `{mac}`-only labels on every capped per-client family, (2) add a test asserting the survivor MAC set is IDENTICAL across two families when >cap clients are fed, (3) document "200 = ~200 clients".

- **B. Experience-rollup thresholds = B1 + `threshold` label (hardcoded constants, count computed over the FULL pre-cap client list).** Module-level documented constants: `SIGNAL_POOR_DBM = -70`, `SATISFACTION_POOR = 50`, `RETRIES_HIGH_PCT = 10`. Emit bounded AGGREGATE count gauges (NOT capped) with a `threshold` label so each series is self-describing and STAGE-007-015's vmalert rule reads the label as its single source of truth:
  - `homelab_unifi_clients_poor_signal{threshold="-70"}` = count of WIRELESS clients with `signal < -70` (wireless-only; signal absent on wired).
  - `homelab_unifi_clients_poor_satisfaction{threshold="50"}` = count of clients with `satisfaction < 50` (satisfaction present on all; wired always 100 so they never count).
  - `homelab_unifi_clients_high_retries{threshold="10"}` = count of WIRELESS clients whose retry ratio `100 * tx_retries / wifi_tx_attempts` > 10 (see retries note below; guard divide-by-zero: skip clients with `wifi_tx_attempts==0`).
  Compute these counts over the FULL parsed client list (BEFORE the per-client cap) so they're accurate even when per-client series are capped. (B1 chosen over B2-config: 3 numbers a single user sets once don't justify a new `load_*_config()`; promoting to config later is a mechanical refactor. B3-defer-to-PromQL REJECTED: it descopes a named card deliverable AND undercounts against the cap.)

- **C. Rate unit = C1 (kbps -> bps, x1000).** Live recon CONFIRMED `tx_rate`/`rx_rate` are KBPS (sample 1,921,600 = 1921.6 Mbps for a WiFi-6E client; 866,700 = 866.7 Mbps WiFi-5). The collector converts: `client_tx_rate_bps = as_float(tx_rate) * 1000.0` (skip if None), same for rx. Keep the `_bps` metric name; document the kbps source + x1000 (mirror device.py's `port_speed_bps = speed * 1_000_000` Mbps->bps comment). NEVER emit raw kbps under a `_bps` name.

- **D. Per-client label set = D1 (`{mac}` only for capped series).** All capped per-client families carry `{mac}` ONLY (stable tick-to-tick survivors; enables A2's cross-family survivor identity; minimal cardinality; hostname/AP enrichment is the registry-join's job per EPIC-006, NOT duplicated here). The per-AP airtime rollup (an AGGREGATE, not capped) uses `{ap_mac}` (optionally `{band}`); the experience counts use `{threshold}`. Do NOT add volatile labels (hostname/ap_mac/band) to the capped per-client series — they'd jitter the survivor set on roam + break A2.

**LIVE-RECON STAT FIELD MAP (from the real UDM stat/sta, 60 clients [46 wireless, 14 wired] — authoritative; NO string-encoded numerics; HYPHENATED keys need bracket access `rec.get("tx_bytes-r")`):**
- **Signal/RF (WIRELESS-ONLY, absent on wired, gate on `is_wired==False`):** `signal` (int, true dBm, negative, e.g. -54) -> `client_signal_dbm`. (`rssi` is a relative 0-95 value, NOT dBm — DO NOT use. `noise` dBm + `ccq` 0-871 exist but are NOT in scope.)
- **Rates (WIRELESS-ONLY):** `tx_rate`/`rx_rate` (int, KBPS) -> `client_tx_rate_bps`/`client_rx_rate_bps` (x1000). (`tx_mcs` = -1 universally, skip.)
- **Satisfaction (ALL clients; wired always 100):** `satisfaction` (int, 0-100; -1 may mean not-computed -> if you emit a per-client `client_satisfaction` series, skip -1). Used in the satisfaction rollup.
- **Retries (counts, cumulative):** `tx_retries` (int, cumulative; wireless, 0 on wired) + `wifi_tx_attempts` (int, cumulative, ALL clients, 0 on wired). NO `tx_retries_pct` field exists. The high-retries rollup computes `ratio = 100 * tx_retries / wifi_tx_attempts` (guard `wifi_tx_attempts > 0`, else skip the client). NOTE: these are LIFETIME cumulative counters, so the ratio is a lifetime-average retry %, a coarse "historically retries a lot" signal — NOT a current rate (a true rate needs cross-tick deltas, OUT OF SCOPE this stage; a PromQL rate on the raw counters is the future-precise path). Emit per-client raw `client_tx_retries` + `client_wifi_tx_attempts`? OPTIONAL — keep the per-client capped families to the §6.2 set (signal/rate/uptime/bytes); the retry COUNT is only the rollup. (Planner: do NOT over-emit; stick to the §6.2 per-client families + the 3 rollups + per-AP airtime.)
- **Uptime (ALL):** `uptime` (int, seconds) -> `client_uptime`.
- **Bytes (NAMESPACE SPLIT by is_wired):** WIRELESS: `tx_bytes`/`rx_bytes` (int, cumulative bytes); WIRED: `wired-tx_bytes`/`wired-rx_bytes` (int, cumulative). Per-client capped `client_tx_bytes`/`client_rx_bytes`: read the wireless OR wired field by `is_wired` (so every client contributes bytes). Rate fields `tx_bytes-r`/`rx_bytes-r` (float bytes/sec, can be None — guard) + `wired-tx_bytes-r`/`wired-rx_bytes-r` exist but are NOT required by §6.2 (cumulative bytes is the deliverable); skip the `-r` rate fields unless trivially clean.
- **Per-AP airtime rollup:** the per-client records do NOT carry AP airtime directly (airtime is in `stat/device` radio_table_stats, already emitted by STAGE-007-005's `homelab_unifi_radio_cu_*`). So the "per-AP airtime SUMMARY" here is best expressed as a per-AP CLIENT-COUNT or per-AP aggregate of the per-client signal — concretely emit `homelab_unifi_ap_client_count{ap_mac}` = count of wireless clients per `ap_mac` (bounded, aggregate). (True channel-airtime is already on the device collector; do NOT re-derive it. If the planner finds a clean per-client airtime field in the recon, may add it, but the radio airtime is owned by 005 — avoid duplication. Keep this rollup simple: per-AP wireless client count.)
- **Wired graceful degrade:** wired clients have NO signal/rate/retries -> they contribute to satisfaction (=100, never poor), uptime, and bytes (via wired-* fields) rollups/series, but NOT signal/rate per-client families nor the signal/retries experience counts. Gate wireless-only emits on `is_wired==False` (use shared `as_bool`).

**Per-client metric families (capped, `{mac}` label, prefix `homelab_unifi_`):** `client_signal_dbm` (wireless), `client_tx_rate_bps` (wireless, x1000), `client_rx_rate_bps` (wireless, x1000), `client_uptime` (all), `client_tx_bytes` (all, wireless tx_bytes / wired wired-tx_bytes), `client_rx_bytes` (all). Each via its OWN `emit_family(...)` with cap=200, `{mac}`-only labels. (That's 6 capped families -> 6 `homelab_metric_family_dropped_series` gauges, one per family — acceptable per A2.)
**Bounded rollups (NOT capped):** `clients_poor_signal{threshold="-70"}`, `clients_poor_satisfaction{threshold="50"}`, `clients_high_retries{threshold="10"}`, `ap_client_count{ap_mac}` (+ `homelab_unifi_api_took_seconds{endpoint="stat/sta"}`).

**emitted accounting:** seed `emitted=[0]`; +1 for the latency gauge; per `emit_family` call += (survivors + 1); += 1 per rollup/aggregate `write_gauge`. metrics_emitted == total writes (gauge + the drop gauges). Mirror active_client's pattern.

**Tracked deferrals:** none. (Per-client retry/airtime PRECISE rate via cross-tick deltas or PromQL -> the raw counters + STAGE-007-015 vmalert rules already own the precise-rate path; the per-AP channel airtime is already emitted by STAGE-007-005's `homelab_unifi_radio_cu_*` — not re-derived here. DPI per-client = STAGE-007-009. Threshold tunability via config = a future mechanical B1->B2 refactor, not a tracked deferral.)

**Tests (Build authors):** fixture-based unit tests mirroring `test_unifi_active_client_collector.py` (fake-client + in-memory writer + `_gauges`/`_gauge_value`/`_counter_value`) AND `test_ha_sensor_value_collector.py` (the CAP test pattern: feed >cap clients via a YAML config file setting a tiny cap through `HOMELAB_MONITOR_CONFIG` + `monkeypatch.setenv`, assert survivors==cap, the `homelab_metric_family_dropped_series{family}` gauge == dropped count incl. 0.0 when under cap, exactly ONE warning SuggestionEvent in `result.events` when dropped>0, none when under).
