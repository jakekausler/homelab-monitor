# STAGE-007-009: Per-client DPI collector (capped top-N×top-N + clamp)

## Status: Complete

## Overview

The DPI collector — `stat/dpi` per-client-per-application byte breakdown ("what app is each client using"),
hard-capped (top-N clients × top-N apps) so cardinality can't explode, with a clamp-rule for Ubiquiti's
known per-counter spike bug. The user explicitly chose per-client DPI (not site-level only).

**Vertical slice:** the collector emits `homelab_unifi_client_dpi_bytes{client,app,cat}` (capped) against the
live UDM; visible in VM and the DPI widget.

**Spec references:** spec §6.2. EPIC-007 "Metric families" (per-client DPI row — user chose per-client) +
"Scope-outs" (PDU power excluded, DPI capped + clamped). EPIC-005 cardinality-cap precedent.

## Cross-cutting standards

- `unifi` group; 5m cadence (DPI is the expensive, high-cardinality fetch — slower interval).
- **Hard cardinality cap: top-N clients × top-N apps** (the expensive part — must be bounded; reuse 005-004).
- **Clamp-rule** for the known Ubiquiti counter-spike bug (values that jump to nonsense a few times/day).
- Graceful degrade: if DPI is disabled on the controller, emit nothing + a `dpi_enabled=0` indicator rather
  than erroring.
- Emits `homelab_collector_run_*`.

## Concrete deliverables

### 1. Per-client DPI collector

`stat/dpi` (+ per-client DPI where available) → `client_dpi_bytes{client,app,cat}`, top-N×top-N capped, with
the spike clamp applied. Emits `unifi_dpi_enabled` indicator. Feeds the DPI widget + `unifi.json`. Document
the cap sizes + clamp in the bundle config.

## Design Phase

### Design Notes (locked)

**Live recon (READ-ONLY, today):** All DPI endpoints exist (200/rc=ok) and DPI fingerprinting is enabled (`get/setting/dpi` → enabled:true), but the live UDM currently has ZERO accumulated DPI data. `stat/stadpi` (per-client) returns `data: []`; `stat/dpi` / `stat/sitedpi` (site-level) return `data: [{}]` (single empty-object sentinel). So Refinement can validate only the graceful-empty + indicator path against live data; the populated parse + by-volume-cap path is validated via a recorded/synthetic `by_app` fixture in unit tests.

**Endpoint (Decision 1 = 1A):** Add a new one-line `stat_stadpi()` helper to the `UnifiClient` Protocol (`kernel/plugins/io.py`) and `UnifiRestClient` (`kernel/unifi/client.py`), mirroring the existing `stat_dpi`/`stat_sta` one-liners: `return await self._get_classic("stat/stadpi")`. The PRE-EXISTING `stat_dpi()` helper hits `stat/dpi` which is SITE-LEVEL (`[{}]` sentinel) and does NOT carry a client dimension — it is NOT used by this collector. Per-client `stat/stadpi` is required to honor the card's `{client,app,cat}` label contract. Per-client entry shape: `{ "mac": <client mac>, "by_app": [ {app:<int>, cat:<int>, rx_bytes:<int>, tx_bytes:<int>}, ... ] }`. `app`/`cat` are NUMERIC integer IDs (stringified into labels); byte counters are cumulative native ints.

**Metric (Decision 4 = 4A):** Emit `homelab_unifi_client_dpi_bytes{client,app,cat}` as a GAUGE of the COMBINED cumulative `rx_bytes + tx_bytes` per `(client, app, cat)` — single series, NO `dir` label (matches the card's exact label set; stretches the tight cap). Cumulative-as-gauge mirrors the other unifi byte families; PromQL `increase()`/`rate()` computes deltas downstream. Also ALWAYS emit `homelab_unifi_dpi_enabled` (1.0 when DPI on per `get/setting/dpi`, else 0.0) and the `homelab_unifi_api_took_seconds{endpoint="stat/stadpi"}` latency gauge — even on the empty path (graceful-degrade indicators).

**Cardinality cap (Decision 2 = 2B, by-volume):** Cap = `load_cardinality_caps_config().cap_for("unifi_dpi")` (=100). The cap is BY VOLUME (top-N by combined bytes), NOT lexical. **Deliberate, documented deviation from "mirror client_stats CappedEmitter usage exactly":** CappedEmitter.emit_family sorts LEXICALLY by `tuple(sorted(labels.items()))` and slices — it keeps the 100 lexically-smallest label tuples, NOT the 100 biggest consumers. The card explicitly requires "top-N clients x top-N apps" = by bytes. Therefore this collector does its OWN top-N-by-bytes selection: sort all `(client,app,cat)`->combined-bytes observations by value descending with a stable secondary sort on the sorted-label-tuple (deterministic ties), keep the top `cap` survivors, emit them, then emit the true `homelab_metric_family_dropped_series{family="homelab_unifi_client_dpi_bytes"}` gauge = `max(0, total_candidates - cap)` and append ONE warning SuggestionEvent when dropped > 0 (preserving emit_family's drop-accounting honesty). Reuse the `M_FAMILY_DROPPED_SERIES` const + the SuggestionEvent shape from `kernel/metrics/cardinality.py`; do not pre-truncate-then-call-emit_family (that would silently report dropped=0). Code review (Finalize): the drop-gauge `family` label uses the METRIC NAME (`homelab_unifi_client_dpi_bytes`), consistent with the `emit_family` convention + the `MetricFamilyDroppedSeries` vmalert alert; `unifi_dpi` remains only the cap-config lookup key. Negative configured caps are clamped to 0 (max(0, cap_for(...))).

**Clamp (Decision 3 = 3A, DEFERRED to STAGE-007-015):** The collector emits RAW cumulative bytes. The "known Ubiquiti per-counter spike bug" clamp is a PromQL clamp that requires delta/temporal context a stateless cumulative-gauge collector does not have — it belongs in the Wave-D vmalert rules. **Deferred to STAGE-007-015** (which already owns the clamp per the epic notes). Mark this with a `# TODO(STAGE-007-015): per-counter spike clamp lives in vmalert/PromQL, not here` comment in the collector.

**Empty/graceful-degrade:** Treat BOTH `data == []` AND `data` consisting only of empty objects (e.g. `[{}]`) as "no DPI data" — emit nothing data-level, still emit `homelab_unifi_dpi_enabled` + latency, return ok=True. `ctx.unifi is None` -> ok=False. `stat_stadpi()` UnifiError -> ok=False. Malformed (non-dict payload / non-list data) -> records=[] -> ok=True.

**ClassVars:** `name="unifi_client_dpi"`, `interval=timedelta(seconds=300)` (5m), `timeout=timedelta(seconds=15)`, `concurrency_group="unifi"`. Register in the unifi bundle `_UNIFI_COLLECTORS` list. (`concurrency_group` is a ClassVar, NOT a CollectorConfig field — the collector test's `_ctx`/`CollectorConfig` must NOT pass it.)

**Brainstormer skipped?** No — brainstormer (Opus) ran; coordinator selected recommendations 1A / 2B / 3A / 4A + combined-rx+tx per pre-authorization.
