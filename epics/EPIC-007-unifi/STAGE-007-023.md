## Overview

Wire the embedded `<LogViewer>` into the Unifi panel, scoped to the UDM syslog stream (`service:"udm-*"`,
with device/category filtering), plus a **threat-forensics** view (the raw IDS/IPS CEF lines — forensics

drill-down pre-filters to that device's lines; a threat-forensics view surfaces IDS/IPS detection lines for
incident review.

## Design Phase

**Status**: [x] Design (Complete)

### Design Notes

**Selected approach: Option B — two Unifi panel tabs ("Logs" + "Threats") sharing a `useUdmLogs` hooks module, reusing `<LogViewer>` untouched.** (Brainstormer Opus consulted; main agent selected the recommendation per pre-authorization.)

**Structure:**
- New `UnifiLogsTab.tsx` (tab `Logs` at `/integrations/unifi/logs`) and `UnifiThreatsTab.tsx` (tab `Threats` at `/integrations/unifi/threats`), both registered in `UnifiLayout.tsx` `TABS` and `router.tsx` under `unifiLayoutRoute`. Mirrors the existing `HomeAssistantLogsTab.tsx` caller pattern (build expr/services → `useLogsQuery` → adapt to `UseLogsResult` in a `useLogs()` callback → render `<LogViewer headerSlot={...}>`).
- `<LogViewer>` and `apps/ui/src/api/logs.ts` (`useLogsQuery`) are reused AS-IS — no log infrastructure is built.

**UDM coverage mechanism:** use a `service:udm-*` wildcard in the LogsQL `expr` (leave the `services` CSV empty), NOT a hardcoded 4-service CSV. Reason: the backend `services` CSV composes EXACT-match `(service:"X" AND source_type:"Y")` clauses (no wildcard), so a CSV would require listing all 4 udm services and would drift if a new `udm-*` category appears. The wildcard composes cleanly with the category presets (a preset swaps the `service:` clause in `expr`). The backend AND-joins the user `expr` after the services clause, so `expr=source_type:udm service:udm-firewall` is well-defined.

**Logs tab UX:** `headerSlot` exposes category preset chips — All (`service:udm-*`), Firewall (`service:udm-firewall`), Audit (`service:udm-audit`), System (`service:udm-system`) — plus the existing time-range picker/refresh, and an optional client-IP filter (matches `src`/`dst`) for honest "device" scoping.

**Threats tab UX (honest, IPS-disabled reality):** pinned to the security-category audit lines (`source_type:udm service:udm-audit udm_category:threat`). An ALWAYS-VISIBLE honest banner in `headerSlot` states the reality: "IPS/IDS may be disabled on the controller — this view shows security-category audit lines for forensic review; the alert fires off the structured alarm path, not from here." This makes the empty-by-default state structural and prevents misreading "no lines" as false reassurance ("no threats happened"). Forensics ONLY — no alert fires from this surface. No fabricated threat rows.

**Device drill-down (honest):** `UnifiDevicePage` embeds the Logs LogViewer pre-filtered by the device's IP via `src`/`dst`, with a one-line note that UDM syslog is GATEWAY-sourced — "device" filtering means traffic to/from this device's IP, not per-AP/per-switch logs (APs/switches don't emit to this stream). This honors the card's "device-filtered instance" deliverable honestly given the data shape.

**Backend contract verified (read-only stage):** `GET /api/logs/query` (`apps/monitor/.../routers/logs.py`) exists, is read-only, returns REAL VictoriaLogs data, accepts `expr/start/end/limit/cursor/services`. UDM stream fields confirmed: `source_type:"udm"`; `service` ∈ {udm-audit, udm-firewall, udm-system, udm-other}; CEF-extracted `udm_category`/`cef_signature_id`/`cef_name`/`udm_admin`/`src`; firewall `fw_chain`/`fw_proto`/`fw_spt`/`fw_dpt`/`src`/`dst`; system `process`. No new backend endpoint or model change — `openapi-export` will NOT fire.

**Coverage gate note:** the new `useUdmLogs` hooks module gets its OWN vitest file (clone `apps/ui/src/api/queries.test.ts` harness) in the SAME Build phase to clear the 80/80/80/75 gate. The two tab components + the device-page embed get component tests; the Radix Tooltip mock pattern applies if tooltip assertions are needed.

**Refinement plan:** FRONTEND stage → force-rebuild prod dist + `make dev-prod` + sha256-prove served==built, then Desktop + Mobile sign-off at `http://127.0.0.1:29090` for: (1) panel Logs tab + category chips, (2) device-page filtered LogViewer, (3) Threats tab honest empty/banner state. Viewport-waiver carve-out applies to the threat-POPULATED branch (cannot be reached without enabling IPS on the live UDM) → automated render test for the populated branch + live sign-off for the empty/banner state; document the waiver.

**No deferrals:** all three card deliverables (panel LogViewer, device-filtered embed, threat-forensics view) are in scope for this stage.

**Live-data finding → DESIGN CHANGE (Threats tab reframed):** Live VictoriaLogs recon revealed the UDM syslog stream has NO reliable threat discriminator — `udm_category` (extracted from CEF `UNIFIcategory`) is never set to `threat` (20,533 lines empty, 0 threat over the sample window); IDS/IPS detections are delivered via the controller's structured-alarm path (`rest/alarm`), not syslog, and IPS is disabled anyway.
