## Design Phase

### Selected Approach (Design)

Mirror the EPIC-005-003 HA integration-bundle skeleton/registration EXACTLY. Two locked decisions + tracked deferrals.

**D1 â€” Placeholder collector: a genuine no-op (Option A).**
A `UnifiPlaceholderCollector` in `plugins/collectors/integrations/unifi/placeholder.py`: emits ONE static gauge `homelab_unifi_bundle_loaded = 1` (so status-API visibility is observable end-to-end), NEVER calls `ctx.unifi`, `concurrency_group = "default"` (NOT `"unifi"` â€” it makes no controller calls, so it must not consume the unifi group's serialization budget; real Wave-B collectors join `"unifi"`), `interval = 60s`, `timeout = 5s`, `run()` always returns `CollectorResult(ok=True, metrics_emitted=1, errors=[], events=[], duration_seconds=...)` mirroring `ha_up`'s always-ok=True convention. It is throwaway SCAFFOLDING â€” a comment at the top of the file AND on its `_UNIFI_COLLECTORS` list entry names STAGE-007-005 as the remover. Rejected: a real `unifi_up` health collector (Option B) â€” that calls the live UDM (the card says skeleton = no live UDM calls) AND collides with STAGE-007-013's named "Controller-up composite + API-latency collector" deliverable (silent scope-drift); a stubbed-to-become-unifi_up placeholder (Option C) muddies the 013 boundary.

**D2 â€” Bundle config: minimal, extend the existing `UnifiConfig` (Option A).**
Add ONLY two new fields to the existing `UnifiConfig` frozen dataclass in `kernel/config.py` (where `unifi_url` already lives via `load_unifi_config()`, mirroring `HaConfig`): `host_lan_ip: str` (default `"192.168.2.148"`, env `HOMELAB_MONITOR_UNIFI_HOST_LAN_IP`, for host attribution) and `ssh_lease_enabled: bool` (default `False`, env `HOMELAB_MONITOR_UNIFI_SSH_LEASE_ENABLED`, the opt-in gate STAGE-007-012's SSH DHCP-lease collector reads). Do NOT redefine `unifi_url` (already single-sourced in `UnifiConfig`). Do NOT define cardinality-cap fields here. `load_unifi_config()` is extended to read the two new env vars with rstrip/bool-parse as appropriate. Rejected: a full config schema now (Option B) â€” cardinality-cap fields would be dead config until STAGE-007-004 (YAGNI + pure 100%-coverage cost) and would pre-empt 004's reuse-of-005-004 cap design; duplicating `unifi_url` diverges from the single source. Rejected: a separate bundle-local pydantic model â€” fragments where 012 reads the flag; the kernel/config.py frozen-dataclass + load_*() + env-var pattern is the established convention.

**Bundle layout (mirror HA):** `plugins/collectors/integrations/unifi/__init__.py` defining `_UNIFI_COLLECTORS: list[type[BaseCollector]] = [UnifiPlaceholderCollector]` and `register_all(loader)` with the per-collector try/except isolation (mirroring `register_ha_collectors`); `placeholder.py` (the no-op collector); a `_shared.py` only if a genuine shared util is needed (otherwise omit â€” YAGNI). Wired in `kernel/api/lifespan.py` via an inline `from ...integrations.unifi import register_all as register_unifi_collectors  # noqa: PLC0415` then `register_unifi_collectors(loader)`, placed alongside the existing `register_ha_collectors`/`register_ssh_collectors` calls.

**Tracked deferrals (no silent scope-drop â€” each lands in an EXISTING named stage that already owns it):**
- Cardinality-cap config fields + machinery â†’ **STAGE-007-004** ("Cardinality caps (reuse 005-004) + identity-upsert helper"). NOT defined in this skeleton.
- Per-collector intervals â†’ **Wave B (STAGE-007-005..011)** as `interval`/`timeout` ClassVars on each collector (the established convention, not config fields).
