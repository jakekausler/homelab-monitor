# STAGE-007-026: DNS-posture drift badge (backend expected_dns/drift on the DNS-posture contract + ✓/drift UI)

## Status: Not Started

## Overview

This stage completes the DNS-posture "green ✓ / red drift" indicator that STAGE-007-021 deferred (021 shipped only the honest raw-handout list + empty state). Two parts: (1) BACKEND — add an `expected_dns` (and optionally a derived `drift: bool`) field to the `/network/dns-posture` contract (`UnifiDnsHandout`/`UnifiDnsPostureResponse`), sourcing the expected DNS steering target from a SINGLE source of truth (the same value the `UnifiDnsSteeringDrift` vmalert rule uses, `192.168.2.148` for this homelab — avoid a third hardcoded copy; consider a shared config/setting). (2) FRONTEND — the Network → Overview DNS-posture widget renders a green ✓ when handout==expected and a red "drift" indicator otherwise, consuming the new contract field honestly. Note the data is currently EMPTY for the user's UDM (no per-network DNS overrides), so the badge will be unexercised on real data until per-network DNS overrides exist — Refinement should use the viewport-waiver carve-out (automated render test) if it can't be reached with real data.

**Vertical slice:** navigating to Integrations → Network → Overview renders the DNS-posture widget with a green ✓ (expected==actual, or both empty) or a red "drift" badge when the per-network handout diverges from the configured expected DNS steering target (the single-sourced constant).

**Spec references:** spec §9.2; EPIC-007 "Verified deployment reality" (DNS steering = DHCP handout); STAGE-007-021 (the deferring stage); STAGE-007-011 (the networkconf collector emitting `homelab_unifi_dhcp_dns_primary{network,dns}`); `deploy/vmalert/metrics/unifi.yaml` `UnifiDnsSteeringDrift` rule.
