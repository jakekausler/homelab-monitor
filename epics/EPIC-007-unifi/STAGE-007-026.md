## Design Phase

**Status**: [x] Design (Complete)

### Design Notes

**Design decision (recorded; brainstormer SKIPPED — single obvious implementation after context-gathering, decision space small and settled by repo-convention investigation):**

**Backend contract** (`apps/monitor/homelab_monitor/kernel/api/routers/integrations_unifi.py`):
- `UnifiDnsHandout` gains two fields:
  - `expected_dns: str | None` — the configured expected DNS steering IP (None when unset/empty).
  - `drift: bool` — computed `True` when `expected_dns` is set AND `dns != expected_dns`; `False` otherwise (including the expected-unset case).
- The `/network/dns-posture` handler reads the new config value, sets `expected_dns` on each handout, and derives `drift`. Three branches to cover for 100% branch coverage: drift-true, drift-false (match), expected-unset.

**Config** (`apps/monitor/homelab_monitor/kernel/config.py` `UnifiConfig`):
- New field `expected_dns_steering_ip: str = "192.168.2.148"`.
- Rationale: matches the EXISTING repo convention of host-specific defaults baked into public config (`host_lan_ip="192.168.2.148"`, `base_url="https://192.168.2.1"`, `HaConfig.base_url="http://192.168.2.148:8123"` all do this). The "open-source-safe defaults" principle governs BEHAVIOR safety (observe-only vs aggressive), not hiding IPs. NOTE: this intentionally diverges from the session-handoff's suggestion of a generic/None public default, because that suggestion conflicts with the actual established repo precedent; baking the host default also means the live prod badge shows ✓ with NO extra prod env wiring (default already equals the live UDM handout `192.168.2.148`).
- Env loader: `HOMELAB_MONITOR_UNIFI_EXPECTED_DNS_STEERING_IP`, following the existing `HOMELAB_MONITOR_UNIFI_*` pattern. Empty-string env is treated as None/unset so the badge degrades to an honest "not configured" state rather than a false ✓.

**Frontend** (`apps/ui/src/routes/integrations/NetworkDnsPostureWidget.tsx`):
- Per network row: green ✓ Badge (`variant="ok"`) when `!h.drift && h.expected_dns`; red "drift" Badge (`variant="critical"`) when `h.drift`; neutral/no-badge ("not configured") when `expected_dns` is null. Empty-networks honest empty state unchanged.
- Mirror house-style badges from `apps/ui/src/routes/integrations/badges.tsx` (`variant="ok"`/`variant="critical"`, lucide icon). Render test covers all three badge branches; the drift-red branch is the viewport-waiver branch (unreachable on live data — the live UDM hands out the expected `192.168.2.148`).

**Single-source-of-truth + vmalert duplication:** The expected DNS is single-sourced server-side from the new `UnifiConfig.expected_dns_steering_ip` field for the API/badge (no 3rd hardcode in the frontend). The `192.168.2.148` hardcode in `deploy/vmalert/metrics/unifi.yaml` (`UnifiDnsSteeringDrift` rule) REMAINS — it is static YAML not templated from app config at runtime, so full single-sourcing across app-config + vmalert-yaml needs templating infra beyond this stage's scope. This is an ACKNOWLEDGED documented duplicate → a regression.md item (marked post-epic / future) will be filed during Refinement so it is not silently dropped.

**No migration** (config-derived + computed). **openapi.json + apps/ui/src/api/schema.ts WILL regenerate** on commit (FastAPI response_model change → openapi-export + generate-ui-types pre-commit hooks fire; both regenerated files staged with the impl commit).

**Refinement plan:** Backend e2e/live-recon validates `expected_dns` populated + `drift:false` on the live ✓ case against the prod `/api/integrations/unifi/network/dns-posture`. Frontend: force-rebuild dist + `make dev-prod` + sha256-prove served==built, then Desktop+Mobile sign-off for the ✓ state at `http://127.0.0.1:29090`; viewport-waiver + vitest render test for the unreachable drift-red state.
