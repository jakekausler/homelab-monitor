# STAGE-007-014: VPN/Teleport health collector

## Status: Complete

## Overview

The VPN/Teleport collector — monitors WiFiman **Teleport** remote-access health (the user accesses the LAN
from their phone via Teleport). Emits Teleport session/connection state from the `stat/health` vpn subsystem
+ any Teleport-specific fields. The exact API field shape was NOT drilled in recon, so **confirming the live
field names is an explicit Design-phase step** for this stage (don't guess metric names ahead of Build).

**Vertical slice:** the collector emits `homelab_unifi_teleport_*` (sessions, up/state) against the live UDM;
visible in VM; feeds `UnifiTeleportDown`.

**Spec references:** spec §6.2, §2 Q11 (Unifi signals — VPN in scope). EPIC-007 "Metric families" (VPN/
Teleport row — "confirm field shape live at Design") + alert catalog (`UnifiTeleportDown`).

## Design Phase

**Status**: [x] Design (Complete)

### Design Notes

**Live recon (read-only, this Design) CHANGED the assumed metric shape.** The card assumed `teleport_up` + `teleport_sessions` from the `stat/health` vpn subsystem. Live recon proved that shape does NOT exist on this firmware:
- `stat/health` vpn subsystem entry = ONLY `{subsystem: "vpn", status: "unknown"}`. `status` is literally `"unknown"` even though Teleport IS configured and enabled (other subsystems read `"ok"`). It is NOT a usable health signal -> DROPPED entirely.
- Dedicated endpoints `stat/teleport`, `stat/vpn`, `stat/remoteuserconnection`, `rest/vpnserver`, `rest/remoteuserconf` all return `rc: error` / empty.
- Teleport CONFIG lives in `rest/setting` (a list of setting sections) -> section `{key: "teleport", enabled: true, subnet_cidr: "192.168.3.1/24", ...}`. Also a `magic_site_to_site_vpn` section (site-to-site WireGuard).
- Teleport RUNTIME signal: the UDM gateway's `stat/device` entry has `teleport_version: "1"` (confirms Teleport initialized on the gateway).
- NO `teleport_sessions` / session-count / peer-list field exists when Teleport is IDLE, via the Network API. Whether an active-session field appears when a phone connects is UNKNOWN (Refinement will probe by connecting from the phone, per the card).

**Selected approach (coordinator-selected; brainstormer consulted on all three decisions):**

**Decision 1 — `homelab_unifi_teleport_up` source = `stat/device` `teleport_version` (Option B).** `teleport_up = 1` when the UDM gateway's `stat/device` entry has a non-empty `teleport_version`; `0` otherwise (or any error). DROP `stat/health` vpn.status (reports "unknown" unconditionally -> meaningless series; emitting it would invite a downstream alert on noise). Rejected Option A (`rest/setting enabled` AND `stat/device version`) and the config-`enabled` signal -> that is config INTENT, not runtime; the gauge should key off runtime, and Option A costs a new client helper + doubled error-branch matrix for no alert value. HONEST MEANING (documented in metric HELP + collector docstring): `teleport_up=1` means "the UDM reports Teleport is initialized on the gateway" -- it does NOT prove a client can establish a session. `UnifiTeleportDown` (warning, STAGE-007-015) is up/down-only on this signal.

**Decision 2 — NO new `rest_setting()` client helper.** It would buy only the config `enabled` toggle (which Decision 1 does not use) + `subnet_cidr` (info label, not alert-relevant), at the cost of a new Protocol method + updating the 3 per-file `_FakeUnifiBase` test fakes + a doubled coverage branch matrix. Single endpoint (`stat_device()`, already exists), no Protocol churn.

**Decision 3 — `teleport_sessions` DEFERRED with a concrete Refinement probe (tracked on THIS card).** No idle session-count field exists; building speculative parsing for an unknown field would be untestable and can't honestly hit 100% coverage. RESOLUTION (no silent drop):
- Refinement active-session probe (the card already mandates this): with the rig up, connect from the phone over Teleport to create an active session, then re-pull `stat/device` + re-probe `stat/health` vpn + the dedicated endpoints that were empty when idle, and DIFF against the idle captures to find any session/peer/connected-user field + its endpoint.
- If a field IS found -> a NEW tracked stage will be created at that point (carrying the discovered endpoint + field shape) and the 014 Refinement results will name it. The session-count metric is NOT bolted onto 014's Build.
- If NO field is found even when active -> recorded on this card as "no session-count field exists on this firmware (idle or active)"; `teleport_sessions` is out of scope until firmware changes.
- Forward note to STAGE-007-015: the `UnifiTeleportDown` alert keys on `homelab_unifi_teleport_up` (up/down), NOT a session count -- session-count is not a usable metric as of this stage.

**Site-to-site VPN (`magic_site_to_site_vpn`) — OUT OF SCOPE (discovery, not a deferred card deliverable).** It is a DIFFERENT subsystem (gateway-to-gateway WireGuard, not remote-user Teleport access); the card's subject is Teleport and §6.2 lists only `teleport_*`. It lives in `rest/setting` (not being added). It was never a 014 deliverable, so this is newly-discovered adjacent scope, not a silent drop of existing scope. NOT building it (YAGNI) and NOT inventing a stage for it; flagged here so a future session/the user can decide whether site-to-site VPN health is wanted (would be its own `homelab_unifi_vpn_site2site_*` collector stage if so). No existing EPIC-007 card covers it (confirmed).

**Metric series emitted by STAGE-007-014 (all from `stat/device`):**
- `homelab_unifi_teleport_up` (gauge, NO labels) -- always emitted; 1.0 when gateway `teleport_version` present+non-empty, else 0.0. HELP states it means "gateway reports Teleport initialized", not "session establishable".
- `homelab_unifi_teleport_reason{reason}` (info-gauge always 1.0; exactly ONE series per run) -- reasons: `ok`, `not_initialized` (reachable, gateway found, no version), `device_not_found` (reachable, no gateway entry in data), `unreachable`, `timeout`, `auth`, `rate_limited`, `http_error`, `bad_response`, `not_configured` (ctx.unifi None).
- `homelab_unifi_teleport_version{version}` (info-gauge 1.0) -- ONLY when version present.
- `homelab_unifi_api_took_seconds{endpoint="stat/device"}` -- on HTTP-response paths only (UnifiResponse + UnifiError-carrying-took); NOT on null-ctx.

**Health-collector behavior (mirrors STAGE-007-013):** emit `teleport_up=0` + reason on EVERY failure path (null-ctx, UnifiError, malformed, gateway-not-found, not-initialized) so `UnifiTeleportDown` has a stable series. Graceful degrade: idle/disabled Teleport -> `not_initialized`, ok=False, NO error raised.

**Gateway identification:** the UDM gateway entry in `stat/device` `data` -- identify by the entry whose type indicates the gateway / that carries `teleport_version` (mirror wan.py's `_find_subsystem` idiom as a `_find_gateway(data)` returning the entry or None). Confirm the exact gateway-identifying key at Build against a real `stat/device` record.

**No new infrastructure:** no migration, no cardinality cap, no Protocol change, no shared state, no new client helper.

**run() branches for 100% coverage:** null-ctx (`not_configured`); `isinstance(result, UnifiError)` per reason (parametrize 6 reasons) / not; malformed payload (`bad_response`); gateway entry not found (`device_not_found`); gateway found + version absent (`not_initialized`); gateway found + version present (`ok` + version gauge). Design reason-mapping table-driven; tests parametrize each reason.
