# STAGE-007-016A: vector config auto-reload on re-render (no manual restart)

## Status: Complete

## Overview

Vector loads its config ONLY at container start and has no hot-reload wired up. The monitor
re-renders `vector.toml` into the shared config volume on every boot (`render_vector_on_boot` in the
FastAPI lifespan), but nothing signals the long-lived `homelab-vector` container to pick up the new
config. Result: after ANY vector template change (e.g. STAGE-007-016's parser), the running vector
keeps executing its STALE in-memory config until someone manually restarts the container. This caused
a multi-cycle false-failure during STAGE-007-016 Refinement (the fixed config was on disk + valid, but
the running process ran the old VRL).

This stage closes that gap so config changes propagate automatically, eliminating a silent
"deployed-but-not-applied" failure mode for the whole vector pipeline (not just Unifi). EPIC-004's
regression notes already DOCUMENT the manual "restart monitor then restart vector" step; this stage
AUTOMATES it.

**Vertical slice:** a vector template/config change re-rendered by the monitor is picked up by the
running vector WITHOUT a manual `docker restart homelab-vector`, validated by changing the rendered
config and observing vector reload it.

**Spec references:** EPIC-004 (vector→VL pipeline; render-on-boot mechanism). STAGE-007-016 Refinement
Findings (the deploy-sequencing false-failure that motivated this).

## Cross-cutting standards

- The reload mechanism must handle the monitor's ATOMIC-REPLACE write pattern (render.py writes a temp
  file then renames) — naive inotify on the path may miss a rename; verify the chosen mechanism
  actually fires on the real write pattern.
- Must NOT introduce a restart loop or drop in-flight events unnecessarily (prefer a graceful reload
  signal over a hard restart where possible).
- Applies to the WHOLE vector config (all sources/transforms/sinks), not just the Unifi branch.

## Concrete deliverables

Pick ONE approach at Design (both are viable — decide via the Design phase):

1. **`--watch-config`** — add `--watch-config` (and, given the atomic-rename write, likely
   `--watch-config-method poll`) to the prod vector `command:` in `deploy/compose/docker-compose.yml`
   (+ the dev override if applicable). Vector then auto-reloads when the rendered file changes.
   Verify poll-vs-inotify behavior against render.py's atomic rename.

2. **SIGHUP-on-render** — have the monitor's render step send `SIGHUP` (or `docker kill --signal=HUP
   homelab-vector`) to the vector container after rewriting the config (the monitor already has the
   docker socket). Vector reloads config on SIGHUP.

Plus: a note/update in EPIC-004's vector-config regression documentation that the manual restart step
is now automated (or kept as a documented fallback).

## Design Phase

**Status**: [x] Design (Complete)

### Design Notes

**Selected approach: Approach 1 — `--watch-config` + `--watch-config-method poll`** (coordinator selection per stage-resume pre-authorization; brainstormer consulted; no user prompt).

Add `--watch-config` and `--watch-config-method poll` to the prod vector service `command:` in `deploy/compose/docker-compose.yml`. Vector polls the rendered config file on an interval and gracefully hot-reloads the whole config when it changes — no manual `docker restart homelab-vector`.

**Why Approach 1 over Approach 2 (SIGHUP-on-render):**
- **Atomic-rename safe (the decisive constraint):** render.py writes via `os.replace()` (atomic rename, swaps inode). Vector's default inotify watcher (`recommended`) watches the old inode/path and an atomic rename commonly breaks it (silent no-reload — the exact multi-cycle false-failure this stage exists to kill). `--watch-config-method poll` re-stats the path each interval and is rename-safe. `poll` is NON-NEGOTIABLE for this write pattern.
- **Approach 2 has a fatal structural flaw:** render runs ONLY at monitor boot (`lifespan.py`), and compose declares `vector depends_on monitor: condition: service_healthy` (docker-compose.yml line ~341). So vector starts AFTER the monitor is healthy — i.e. AFTER the render. A SIGHUP sent right after render at monitor boot would hit a vector container that does not exist yet, then vector starts and reads the fresh config at start anyway. On the exact path the card targets, SIGHUP is redundant with vector's start-time read while adding a write-capable docker-kill endpoint (`POST /containers/{id}/kill?signal=SIGHUP`, not implemented today) requiring 100% branch coverage for no live benefit.
- **Graceful, whole-config, easy to prove:** vector's poll reload is graceful (diffs topology, drains affected paths — no hard restart, no event loss), reloads the entire config (all sources/transforms/sinks, not just Unifi), and is trivially provable on the live rig via the reload log line + unchanged container uptime.
- **Lowest complexity:** ~2 compose lines; no Python; NOT covered by `make verify` (compose-only) so the live e2e reload test + a render→`vector validate` check are the real gate.

**Build scope:**
- Add `--watch-config` + `--watch-config-method poll` to the vector `command:` array in `deploy/compose/docker-compose.yml`. Decide explicit poll interval (modest, e.g. 15-30s) vs vector default at Build — document the choice.
- Check the dev compose override (`deploy/dev/`) and apply the same flags there if a dev vector service exists, so dev parity holds.
- Verify the rendered config still passes real `vector validate` (timberio/vector:0.41.1-debian) — no `$`-in-comment regression — and that vector actually reloads on a file change (don't trust the flag; prove it). The `make verify` slow `vector validate` test SKIPS in CI (no binary).
- Update EPIC-004 regression doc: the manual `docker restart homelab-vector` step is now automated (keep as documented fallback).

**Build-phase correction (2026-06-20): flag is `--watch-config` ALONE, not `poll`.** `--watch-config-method poll` does NOT exist in vector 0.41.1 — it was added in vector 0.43.0 (verified via `docker exec homelab-vector vector --help`). In 0.41.1 only `--watch-config` (boolean, env `VECTOR_WATCH_CONFIG`) is available, using the default `recommended` (inotify) watcher. The original concern that motivated `poll` (inotify breaking on `os.replace` atomic rename) does NOT apply to 0.41.1: verified against the vector v0.41.1 source `src/config/watcher.rs`, vector re-arms its inode watch via `add_paths()` BEFORE sending `ReloadFromDisk` on a config-change event — it is explicitly designed for the inode-swap/atomic-rename case. `--watch-config-method poll` (0.43.0+) targets a DIFFERENT failure (filesystems with no working inotify: NFS/EFS, ConfigMap symlink swaps), not atomic-rename. The `data_vector_config` named volume is on host-local fs (inotify works). Implementation: `command: ["--config", "/etc/vector/vector.toml", "--watch-config"]`. Residual risk (no-inotify filesystem) is resolved by the Refinement live reload test (prove 2-3 successive re-render reloads).

### Refinement Notes

**Live prod-rig validation (e2e-tester, no user prompt — infra stage): PASS.**

Recreated ONLY the `homelab-vector` sidecar with the new compose command (`docker compose up -d --no-deps --force-recreate vector`) so `--watch-config` became active — the change is not testable without it. Confirmed running command is now `[--config /etc/vector/vector.toml --watch-config]`; vector started cleanly (`Creating configuration file watcher.` / `Watching configuration files.`), no crash loop.

Proved auto-reload across 3 successive atomic-rename swaps (a one-shot `busybox` container rewrote `/etc/vector/vector.toml` in the `homelab-monitor_data_vector_config` volume via `cp + append-comment + mv` = same atomic rename as the monitor's `os.replace`):
- Each swap produced: `Configuration file changed.` → `Reloading running topology with new configuration.` → `New configuration loaded successfully.` → `Vector has reloaded.`
- Vector PID (2820786) and StartedAt (2026-06-20T13:12:39Z) UNCHANGED across all 3 swaps → in-process GRACEFUL reload, NOT a restart. No restart loop, no event loss, no config error. Final status: `Up`, healthy.

This confirms the Design's source-backed claim: vector 0.41.1's default `--watch-config` (inotify) watcher catches the `os.replace` atomic-rename and reloads in-process — `--watch-config-method poll` (0.43.0+) is NOT needed on a local-fs named volume.

**Anomaly (benign, not a failure):** reload latency ~12-21s (inotify on an overlayfs-backed Docker named volume, not sub-second). Reloads fire reliably; latency is well within acceptable bounds for config propagation. The probe comment lines left in the rendered config are benign TOML comments that the monitor's next boot-render (`os.replace`) overwrites entirely.
