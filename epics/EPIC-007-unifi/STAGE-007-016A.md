# STAGE-007-016A: vector config auto-reload on re-render (no manual restart)

## Status: Not Started

## Overview

Vector loads its config ONLY at container start and has no hot-reload wired up. The monitor
re-renders `vector.toml` into the shared config volume on every boot (`render_vector_on_boot` in the
FastAPI lifespan), but nothing signals the long-lived `homelab-vector` container to pick up the new
config. Result: after ANY vector template change (e.g. STAGE-007-016's parser), the running vector
keeps executing its STALE in-memory config until someone manually restarts the container. This caused
a multi-cycle false-failure during STAGE-007-016 Refinement (the fixed config was on disk + valid, but
the running process ran the old VRL).

This stage closes that gap so config changes propagate automatically, eliminating a silent
"deployed-but-not-applied" failure mode for the whole vector pipeline (not just Unifi). EPIC-004's
regression notes already DOCUMENT the manual "restart monitor then restart vector" step; this stage
AUTOMATES it.

**Vertical slice:** a vector template/config change re-rendered by the monitor is picked up by the
running vector WITHOUT a manual `docker restart homelab-vector`, validated by changing the rendered
config and observing vector reload it.

**Spec references:** EPIC-004 (vector→VL pipeline; render-on-boot mechanism). STAGE-007-016 Refinement
Findings (the deploy-sequencing false-failure that motivated this).

## Cross-cutting standards

- The reload mechanism must handle the monitor's ATOMIC-REPLACE write pattern (render.py writes a temp
  file then renames) — naive inotify on the path may miss a rename; verify the chosen mechanism
  actually fires on the real write pattern.
- Must NOT introduce a restart loop or drop in-flight events unnecessarily (prefer a graceful reload
  signal over a hard restart where possible).
- Applies to the WHOLE vector config (all sources/transforms/sinks), not just the Unifi branch.

## Concrete deliverables

Pick ONE approach at Design (both are viable — decide via the Design phase):

1. **`--watch-config`** — add `--watch-config` (and, given the atomic-rename write, likely
   `--watch-config-method poll`) to the prod vector `command:` in `deploy/compose/docker-compose.yml`
   (+ the dev override if applicable). Vector then auto-reloads when the rendered file changes.
   Verify poll-vs-inotify behavior against render.py's atomic rename.

2. **SIGHUP-on-render** — have the monitor's render step send `SIGHUP` (or `docker kill --signal=HUP
   homelab-vector`) to the vector container after rewriting the config (the monitor already has the
   docker socket). Vector reloads config on SIGHUP.

Plus: a note/update in EPIC-004's vector-config regression documentation that the manual restart step
is now automated (or kept as a documented fallback).
