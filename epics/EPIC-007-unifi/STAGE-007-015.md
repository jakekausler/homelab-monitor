# STAGE-007-015: vmalert-METRICS rules (the ~24 metric alerts)

## Status: Not Started

## Overview

Author the Unifi metric alert rules (`deploy/vmalert/metrics/unifi.yaml`) — the ~24 alerts in the EPIC-007
catalog (5 critical / 16 warning / 5 info). Needs ONLY the Wave-B/C collectors (all present by now); it does
NOT depend on the syslog stream (those are the separate log rules in 017).

**Vertical slice:** the rules file provisions into vmalert-metrics; each rule is promtool-validated where
possible; a representative critical rule (`UnifiControllerDown`) and an anomaly rule are demonstrated firing
against the live/test data path → Alertmanager.

**Spec references:** spec §3.1 (vmalert metrics), §4.1 (metric flow). EPIC-007 "Alert catalog" (the metric
table). EPIC-005-005 (user-authored-rule machinery — these default rules coexist with user-tunable ones).
EPIC-006 STAGE-006-016 (sibling metric-rules stage; promtool idioms; the §4.1 cross-metric empty-join trap).

## Cross-cutting standards

- Severity vocab `info|warning|critical` ONLY.
- Anomaly rules use the rolling-baseline `clamp_min(K*avg_over_time(metric[window]), floor)` idiom with
  warm-up; absolute-threshold rules (down/probe) carry the load immediately.
- Avoid the §4.1 cross-metric empty-join trap (a binary op between two different metrics returning empty =
  dead rule). promtool can't eval `[w:s]` subqueries / `label_value()` — those validate via live-replay.
- Annotations carry the device/client name so an alert says WHICH device.
