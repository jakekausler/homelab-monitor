# STAGE-007-006: WAN/ISP + speedtest + failover collector

## Status: Complete

## Overview

The WAN/ISP collector — reads the `www` subsystem from `stat/health` for WAN up/down, latency, packet
drops, throughput, the UDM's last speedtest result, and multi-WAN failover state. This is the *primary* ISP
outage detector (EPIC-016 adds independent corroboration later).

**Vertical slice:** the collector emits `homelab_unifi_wan_*` + `_speedtest_*` metrics against the live UDM;
visible in VM.

**Spec references:** spec §2 Q21 (ISP/WAN; reuse UDM speedtest results), §6.2. EPIC-007 "Metric families"
(WAN row). EPIC-016 (shares the ISP-outage detection; this is the primary detector).

## Cross-cutting standards

- `unifi` group; 30s cadence (WAN state is the highest-priority signal).
- Read-only: reads the LAST speedtest result; NEVER triggers a speedtest.
- Graceful degrade: single-WAN setups have no failover peer — `wan_failover_active` reports accordingly.
- Emits `homelab_collector_run_*`.

## Concrete deliverables

### 1. WAN/ISP collector

Reads `stat/health` (`www` block): `wan_up`, `wan_latency_seconds`, `wan_drops`, `wan_xput_down/up`,
`speedtest_download/upload/ping`, `speedtest_lastrun`, `gw_mac`, and per-WAN failover state
(`wan_failover_active`). Feeds Wave-D alerts: UnifiWanDown, UnifiWanLatencyHigh, UnifiWanPacketLoss,
UnifiSpeedtestDegraded. Surfaces in the Network page WAN widget + `network.json`.

## Design Phase

**Status**: [x] Design (Complete)

### Design Notes & Selected Approach

**Selected approach:** Single `UnifiWanCollector(BaseCollector)` mirroring the STAGE-007-005 device collector EXACTLY (`plugins/collectors/integrations/unifi/device.py`): same `_as_float`/`_as_bool`/`_emit_numeric`/`emitted=[1]`-counter helpers, same None-client + UnifiError return-not-raise handling, same `homelab_unifi_api_took_seconds{endpoint=...}` latency emit, same `cast("dict[str, object]", ...)` narrowing idiom. One `await ctx.unifi.stat_health()` per 30s tick; parse the classic `{"meta":{"rc":"ok"},"data":[...]}` wrapper where `data` is a LIST of subsystem entries (NOT a single block). Emit `homelab_unifi_wan_*` + `homelab_unifi_speedtest_*` gauges. Brainstormer skipped: single obvious implementation, no architectural choice, device collector is the locked precedent; all unit/field decisions are determined by the live recon below.

**Collector ClassVars:** `name="unifi_wan"`, `interval=timedelta(seconds=30)` (WAN is the highest-priority signal — faster than device's 60s), `timeout=timedelta(seconds=15)`, `concurrency_group="unifi"`.

**Bundle wiring:** add `from .wan import UnifiWanCollector` and append `UnifiWanCollector` to `_UNIFI_COLLECTORS` in `plugins/collectors/integrations/unifi/__init__.py` (one-line edit; the device collector stays).

**Error handling:** identical to device collector. `ctx.unifi is None` -> `ok=False, errors=["unifi client not configured"]`. `UnifiError` from `stat_health()` -> `ok=False, errors=[result.message]`. Neither raises. On success emit `homelab_unifi_api_took_seconds{endpoint="stat/health"}` from `result.took_seconds` (the classic endpoint has NO `meta.took`).

**LIVE-RECON FIELD MAP (from the real UDM `stat/health` — AUTHORITATIVE; corrects the card's aspirational names):**

- **Wrapper:** `{"meta":{"rc":"ok"},"data":[...]}`. No `meta.took`. `data` is a LIST of 5 subsystem entries; each entry has a `subsystem` string field. The WAN/ISP data is split across TWO entries: `subsystem=="www"` (health: latency/drops/throughput/speedtest) and `subsystem=="wan"` (gw identity + failover via `uptime_stats`). Find each by scanning `data` for the matching `subsystem` value (guard each entry `isinstance(entry, dict)`; the lookup is `next(e for e in data if isinstance(e,dict) and e.get("subsystem")=="www")` style, but defensively — if absent, skip that family, ok=True).

- **From the `www` entry:**
  - **`homelab_unifi_wan_up`** (no labels) <- `1.0 if www["status"]=="ok" else 0.0`. `status` is a STRING enum (`"ok"`/`"warning"`/`"error"`) — there is NO `wan_up` boolean field. If `status` absent/non-str, skip wan_up (graceful).
  - **`homelab_unifi_wan_latency_seconds`** <- `www["latency"]` is an INTEGER in MILLISECONDS; CONVERT to seconds: `latency_ms / 1000.0`. (Sample live value 10ms -> 0.01s.) Use `_as_float` then divide; skip if None.
  - **`homelab_unifi_wan_drops`** <- `www["drops"]` (int raw packet-drop count, NOT a percentage). Emit as-is via `_as_float`.
  - **`homelab_unifi_wan_xput_down_bytes_per_sec`** <- `www["rx_bytes-r"]` (int bytes/sec, LIVE rate). **`homelab_unifi_wan_xput_up_bytes_per_sec`** <- `www["tx_bytes-r"]` (int bytes/sec). NOTE: the field name has a HYPHEN (`rx_bytes-r`) — use `www.get("rx_bytes-r")`. Keep raw bytes/sec (no lossy Mbps conversion); the metric NAME encodes the unit. (These are the live throughput; distinct from the speedtest xput below.)
  - **Speedtest:** the UDM exposes `xput_down`/`xput_up` (FLOAT Mbps — these ARE the last speedtest down/up; there are NO `speedtest_download`/`speedtest_upload` fields), `speedtest_ping` (int ms), `speedtest_lastrun` (int epoch seconds), `speedtest_status` (str e.g. "Idle"). **STALE/NEVER-RUN HANDLING (critical):** if `speedtest_lastrun` is absent or `== 0`, the speedtest has NEVER been run (this UDM's current state) and `xput_down`/`xput_up`/`speedtest_ping` are all 0 — do NOT emit them as real values. Rule: ALWAYS emit `homelab_unifi_speedtest_lastrun` <- `speedtest_lastrun` (even when 0 — it signals "never run"); ONLY emit `homelab_unifi_speedtest_download_mbps` <- `xput_down`, `homelab_unifi_speedtest_upload_mbps` <- `xput_up`, `homelab_unifi_speedtest_ping_seconds` <- `speedtest_ping/1000.0` WHEN `speedtest_lastrun` parses to a value > 0. (speedtest_ping is ms -> convert to seconds like wan latency.)

- **From the `wan` entry (failover / multi-WAN):**
  - Failover state lives in `wan["uptime_stats"]` — a dict keyed by WAN name (`"WAN"`, `"WAN2"`, ...). Guard `isinstance(uptime_stats, dict)`.
  - **`homelab_unifi_wan_failover_active`** (no labels) <- `1.0` ONLY if a SECONDARY WAN is currently carrying traffic (active failover), else `0.0`. Detection: the primary is `WAN`; a secondary key (e.g. `WAN2`) that has a live `uptime`/`availability>0` (currently up) AND the primary is down would mean active failover. In THIS UDM's reality WAN2 is present but fully DOWN (`uptime_stats.WAN2.downtime>0`, no top-level `uptime`/`availability`), so `wan_failover_active=0.0`. Practical rule for the collector: `wan_failover_active = 1.0 if (a secondary WAN key exists AND that secondary has availability>0 / a positive uptime AND primary WAN status is not ok) else 0.0`. Keep it simple and defensive — single-WAN (no secondary key) -> 0.0. Because the precise "active" semantics are only observable during a real failover (not available on this rig), emit `0.0` for the not-failed-over case and DOCUMENT that the active-failover branch is exercised by the fixture test, not live (the live rig has no active failover to observe — viewport-waiver-equivalent for a collector: the active path is fixture-tested).
  - **`homelab_unifi_wan_failover_capable`** (no labels) <- `1.0 if uptime_stats has MORE THAN ONE WAN key (a secondary is configured) else 0.0`. (THIS UDM: WAN+WAN2 -> capable=1.0 even though WAN2 is down. This distinguishes "single-WAN, no failover possible" from "multi-WAN, secondary currently down" — the card's "single-WAN setups have no failover peer; wan_failover_active reports accordingly".)
  - Optionally emit `homelab_unifi_wan_uptime_seconds` <- the primary `uptime_stats.WAN.uptime` if present (int seconds). Low-priority; include if clean.

- **String-encoded gotcha:** the `wan` entry's `gw_system-stats` sub-dict has STRING cpu/mem/uptime — but THIS collector does NOT emit gateway cpu/mem (the device collector already covers the UDM via stat/device). Do NOT emit gw cpu/mem/uptime here (avoid double-emitting; the UDM device row owns it). Only failover + wan_up/latency/drops/throughput/speedtest belong to this collector.

- **Timestamps:** `speedtest_lastrun` is epoch seconds (int) — emit as-is (a gauge of the epoch value; alerts compute staleness as `now - lastrun`). No ISO conversion (no registry writes this stage).

**Metric names (final, prefix `homelab_unifi_`):** `wan_up`, `wan_latency_seconds`, `wan_drops`, `wan_xput_down_bytes_per_sec`, `wan_xput_up_bytes_per_sec`, `wan_failover_active`, `wan_failover_capable`, `wan_uptime_seconds` (optional), `speedtest_download_mbps`, `speedtest_upload_mbps`, `speedtest_ping_seconds`, `speedtest_lastrun`, plus `api_took_seconds{endpoint="stat/health"}`. (Wave-D alerts UnifiWanDown/UnifiWanLatencyHigh/UnifiWanPacketLoss/UnifiSpeedtestDegraded bind these — no alert currently binds a specific name, so these names are authoritative.)

**Graceful-degrade rule:** every field isinstance-guarded via the shared helpers; absent `www` or `wan` subsystem entry -> skip that family, ok=True (never error). The `data` list scan must not crash on a non-dict entry or a missing `subsystem` key. STRING numerics float-parsed in guards. Exclude `bool` before `int`/`float` in `_as_float` (already handled by the shared helper).

**Helper reuse decision:** `_as_float`/`_as_bool`/`_emit_numeric` are currently PRIVATE module helpers in `device.py`. For Build: the planner should decide between (A) importing them from device.py, or (B) duplicating the tiny helpers in wan.py. RECOMMENDATION: extract `_as_float`/`_as_bool`/`_emit_numeric` into a shared `plugins/collectors/integrations/unifi/_helpers.py` (or `kernel/unifi/parsing.py`) module that BOTH device.py and wan.py import — this avoids duplication across the 9 Wave-B/C collectors that will all need them. If extraction is too invasive for this stage, duplicate them locally in wan.py and note a deferral to refactor (cite a tracking owner). PLANNER DECIDES at Build; prefer extraction.

**Tracked deferrals:** none required. (Active-failover live observation is NOT a deferral — it's fixture-tested; the live rig has no active failover, which is documented above. Independent ISP corroboration is EPIC-016, already tracked.)

**Tests (Build authors):** fixture-based unit test mirroring `test_unifi_device_collector.py`. Fixtures: (1) a healthy single-active-WAN payload with a populated `www` (status ok, latency 10, drops 3, rx/tx bytes-r) + a `wan` entry with `uptime_stats={WAN:{uptime,availability:100}, WAN2:{downtime>0}}` (capable=1, failover_active=0, lastrun=0 -> NO speedtest down/up/ping emitted, lastrun=0 emitted); (2) a payload WITH a completed speedtest (`speedtest_lastrun`>0, `xput_down`/`xput_up`/`speedtest_ping`>0 -> speedtest metrics ARE emitted, ms->s converted); (3) a single-WAN payload (`uptime_stats={WAN:{...}}` only -> capable=0, failover_active=0); (4) an ACTIVE-FAILOVER payload (`www.status` not ok + secondary WAN up -> failover_active=1) to exercise that branch; (5) graceful-degrade: `data` missing the `www` and/or `wan` entry, a non-dict entry in `data`, absent `uptime_stats`, status non-string -> no crash, ok=True; (6) error path (UnifiError) + None-client path. Watch the lint conventions: `# noqa: PLR2004` on magic-number asserts; `# pyright: ignore[reportPrivateUsage]` on private-helper imports; inline dict literals passed to a `dict[str,object]`-typed helper must be bound to a `dict[str, object]`-annotated local FIRST (the dict value-type-invariance gotcha from STAGE-007-005); top-level imports only; ASCII-only comments. Run the GLOBAL `--cov=homelab_monitor` 100% branch gate — exercise EVERY branch (each emit, each skip, the lastrun==0 vs >0 split, the failover capable/active permutations, the bool-exclusion, the str-unparseable path, both error paths, the None-client path).
