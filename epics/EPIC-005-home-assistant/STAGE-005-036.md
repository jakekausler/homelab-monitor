## Design Phase

### Design Notes

**Chosen approach: Option B â€” `line-clamp-2` description + `Learn more â†—` link on its own secondary line.**

Two-line+ `space-y-0.5` row container, mirroring the just-shipped STAGE-005-035 `HaUpdatesDrill.tsx` pattern:
- Primary line unchanged: `issue_id` (font-medium) + `domain` (muted) on the left, severity `Badge` on the right.
- Conditional secondary region (`showSecondary = description present OR learn_more_url present`) below the primary line:
  - `description` rendered as a plain-text muted `<p>` with `line-clamp-2` (plain `{row.description}` node â€” NOT `dangerouslySetInnerHTML`; HA repair descriptions are plain text).
  - `Learn more â†—` external link on its OWN line below the prose (NOT inline-trailing), so the 2-line clamp can never hide it.

**D-REPAIRS-UI-DESCRIPTION-RENDER** â†’ `line-clamp-2` (Tailwind). Two lines is enough prose to identify the repair while bounding row height in the compact scrolling list; `learn_more_url` carries the full story. Plain-text node, never HTML.

**D-REPAIRS-UI-LAYOUT** â†’ description as a muted secondary `<p>` below the primary line; `Learn more â†—` on its OWN secondary line below the prose. Guard the prose with `row.description != null && row.description.trim() !== ''` (mirror 035's empty-string guard); guard the link with `row.learn_more_url != null`; gate the whole secondary region on either being present.

**D-REPAIRS-UI-LINK-AFFORDANCE** â†’ `Learn more â†—` (U+2197 glyph), matching the 035 affordance exactly: `target="_blank" rel="noopener noreferrer"`, `hover:text-foreground`, `shrink-0`.

**D-REPAIRS-UI-PRIVACY** â†’ render-only; no client-side logging of `description`. Trivially satisfies the STAGE-005-031 never-logged posture (display-only to the single authenticated owner).

**Rejected:**
- Option A (description `line-clamp-2` with the link INLINE-trailing the prose) â€” when prose fills both clamped lines, the clamp ellipsis truncates the prose tail AND the trailing link, hiding `Learn more â†—` entirely. - Option C (single-line `truncate` prose + pinned trailing link) â€” most compact, but one line of prose is too little to identify a repair from "a sentence or several," and it reads poorly on Mobile.

**Contract verified (Design):** `HaRepairRow` carries `description` / `learn_more_url` as `str | None`, populated from a live HA `repairs/list_issues` WS snapshot via `build_repairs_index` (real data, independent null-degradation, HaError swallowed silently / never logged) â€” confirmed in `integrations_home_assistant.py` + `kernel/ha/enrichment.py`.
