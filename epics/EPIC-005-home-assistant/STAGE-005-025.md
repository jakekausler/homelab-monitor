# STAGE-005-025: Embedded LogViewer (scoped service:"homeassistant")

## Status: Complete

## Overview

Embed the shared EPIC-004 `<LogViewer>` in the HA panel, scoped to the HA log stream
(`service:"homeassistant"`), so the user sees recent HA logs (including automation/script errors)
inline in the integration panel. This is the use case EPIC-004 STAGE-004-003 explicitly designed the
LogViewer embedding contract for. Scenario #20 (inline).

**Vertical slice:** after this stage, the HA panel has a logs section showing live HA logs filtered to
`service:"homeassistant"` via a caller-provided `useLogs` hook — recent automation failures visible
without leaving the panel.

**Spec references:** master spec §9.2 (per-integration panel + related logs). EPIC-005 §"Stage
decomposition" Wave F. Embedding contract: `apps/ui/src/components/logs/README.md` (`UseLogsResult` /
`LogViewerProps`); worked examples: `DockerContainerLogsViewerBody.tsx`, `CronRunLogViewer.tsx`;
hook: `useLogsQuery(expr,start,end,services)` in `apps/ui/src/api/logs.ts`.

## Concrete deliverables

### 1. HA logs `useLogs` hook

A caller-provided `useLogs(): UseLogsResult` hook (in `apps/ui/src/routes/integrations/` alongside the
panel) that wraps `useLogsQuery(expr, start, end, services)` scoped to the HA stream. The `services`
argument is the `useLogsQuery` services CSV of `{source_type}:{service}` identities (the same format
`identitiesToServicesCsv` produces) — here a single identity `{source_type}:homeassistant` (the
vector mapping sets `service=homeassistant`; confirm the HA docker container's `source_type` label
during Design — likely `docker`). Adapts the query result into `UseLogsResult` exactly like
`DockerContainerLogsViewerBody.tsx` does (lines/isLoading/isError/logStatus/pagination).

### 2. Embed `<LogViewer>` in the panel logs section

Render `<LogViewer useLogs={haUseLogs} headerSlot={...} emptyStateCopy="No recent Home Assistant
logs." timezone wrap fillHeight />` in the HA panel's logs section (005-022 placeholder). Default the
query to recent errors/all HA logs over a sensible window. Do NOT modify the shared LogViewer
(backward-compat — it has many other callers).

### 3. Tests

Render the embed with a mocked `useLogsQuery` → assert lines render, the HA service scope is applied
(the hook passes `homeassistant`), empty/loading/error states; assert LogViewer is used as-is (no
fork). Desktop + Mobile.

## Out of scope for this stage

- Changing the shared LogViewer (it's EPIC-004's; reuse only).
- Other widgets (023/024) and the Grafana dashboard (026).
- A new logs API (reuses `useLogsQuery`).

## Decisions in this stage

### Design phase will confirm

- **D-HALOGS-SELECTOR** — the exact `source_type:service` CSV for HA (confirm the HA container's
  `source_type` label alongside `service=homeassistant`).
- **D-HALOGS-DEFAULT-QUERY** — default expr/window for the embedded view (all HA logs vs errors-first)
  and which LogViewer controls to expose in `headerSlot`.

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-LOGVIEWER-EMBED-CONTRACT** — use the EPIC-004 LogViewer embedding contract (caller-provided
  `useLogs`) exactly as `DockerContainerLogsViewerBody`/`CronRunLogViewer` do; this is the use case
  STAGE-004-003 designed for.
- **D-REAL-SERVICE-LABEL** — scope to `service:"homeassistant"` (no hyphen — verified vector label).

### Design Decisions (resolved 2026-06-13)

**D-HALOGS-SELECTOR — RESOLVED.** Scope the embedded view to `services = identitiesToServicesCsv([{ source_type: 'docker', service: 'homeassistant' }])` (CSV `"docker:homeassistant"`). Verified against the LIVE prod rig: VictoriaLogs holds 139,472 HA Core log lines under `service:homeassistant` (no hyphen), zero under the hyphenated `home-assistant`. The HA container's `container_name` is literally `homeassistant` (image `ghcr.io/home-assistant/home-assistant:stable`, v2026.2.1), and vector sets `.service = .container_name` (`deploy/vector/vector.toml.template`). CAVEAT: use the EXACT value `homeassistant` — a `*homeassistant*` wildcard would also catch the unrelated `grocy-homeassistant` (Grocy) container. `source_type` is `docker`.

**D-HALOGS-DEFAULT-QUERY — RESOLVED (Option B, errors-focused).** Default `expr = 'severity:error OR severity:warn'` over a `1h` window. `headerSlot` (lean) carries: a time-range control, a refresh control, an "Errors only ↔ All lines" toggle that swaps `expr` between `'severity:error OR severity:warn'` and `'*'`, and an "Open in Explorer" deep-link. Render a plain `<LogViewer>` (NO `fillHeight`) to stay faithful to the `DockerContainerLogsViewerBody.tsx` worked example. Verified against the LIVE prod rig that the structured `severity` field IS populated on HA lines (values `error`/`warn`/`info`/`debug`/`critical`; `severity:error OR severity:warn` ≈ 196k hits) — so the errors-only default returns abundant real data, NOT an empty panel. Use the structured `severity` field, NOT message substring and NOT `level` (which is empty on HA lines).

**D-REAL-SERVICE-LABEL — CONFIRMED.** `homeassistant` (no hyphen) — the card's locked decision was correct, now verified against live-rig data.

**Build mechanics (grounded in real files):** mirror `apps/ui/src/routes/integrations/DockerContainerLogsViewerBody.tsx` for the `useLogs` adapter shape (lines/isLoading/isError/logStatus/pagination mapping) and the window memoization on `[token, refreshNonce]`. Build start/end ISO strings via `resolvePreset(token, new Date())` + `toIsoZ` (`apps/ui/src/lib/timeRange.ts`). Match-all expr is `'*'` (the codebase convention). `useLogsQuery(expr, start, end, services)` is at `apps/ui/src/api/logs.ts`; `identitiesToServicesCsv` is in the same file. NOTE: no `CronRunLogViewer.tsx` exists — the only `<LogViewer>` embed example in `routes/integrations/` is the Docker pair; mirror that.

**Rejected — Option A (mirror Docker exactly: match-all `'*'` default, full header).** Faithful to the worked example but a 15m/1h window of UNFILTERED HA logs is noisy (HA is chatty — 139K lines), burying the panel's stated purpose ("recent automation/script failures inline") under INFO chatter. Rejected: does not serve the failure-surface intent.

**Rejected — Option C (match-all default + `fillHeight` internal scroll, minimal controls).** Cleaner embedded scrolling but `fillHeight` diverges from the Docker mirror and pulls in a parent-height layout-contract verification that is out of this tight FE stage's scope; still noisy (match-all). Deferred `fillHeight` polish unless the panel shell visibly needs it.
