# STAGE-005-027: HA entity + battery detail endpoints (drill-down backend)

## Status: Design (Not Started)

## Overview

Backend detail endpoints that 028's drill-down widgets consume. STAGE-005-021 shipped a counts-only
`/summary` (per D-PANEL-ENDPOINT-SPLIT, detail deferred to "023/024 if widgets need them");
STAGE-005-023 then shipped counts-only entity-health + battery widgets and DEFERRED the drill-lists
because no detail endpoint exists. This stage builds those detail endpoints.

**Vertical slice:** after this stage, `GET /api/integrations/home-assistant/entities` returns the
per-entity rows for the unavailable/stale drill-list (entity_id, domain, last-changed age, available
flag) and `GET /api/integrations/home-assistant/batteries` returns per-device battery rows
(entity_id/device, level), so 028 can render "which 984 entities are unavailable" and "which battery
is critical".

**Scope extended (2026-06-13)** to also cover updates / config-entry / repairs detail endpoints deferred from STAGE-005-024.

**Spec references:** EPIC-005 §Stage decomposition; the deferred work recorded in STAGE-005-023
Design Notes ("DEFERRED FOLLOW-UP WORK"). Data source: VictoriaMetrics per-entity gauges
(`homelab_ha_entity_available{entity_id,domain}`, `homelab_ha_entity_last_changed_seconds{entity_id,domain}`,
`homelab_ha_battery_level{entity_id,domain}`) — extend the `vm_query` read path from per-aggregate
`vm_count` to per-series row extraction. Mirror the 021 router
(`kernel/api/routers/integrations_home_assistant.py`).

## Concrete deliverables

### 1. Data models

Pydantic models in `apps/monitor/homelab_monitor/kernel/api/models/home_assistant.py`:
- `HaEntityRow` (entity_id: str, domain: str, available: bool, last_changed_age_seconds: float)
- `HaEntityRowsResponse` (entities: list[HaEntityRow], filtered_to: str | None)
- `HaBatteryRow` (entity_id: str, device: str | None, domain: str, level: float)
- `HaBatteryRowsResponse` (batteries: list[HaBatteryRow], filtered_to: str | None)

### 2. Router endpoints

In `apps/monitor/homelab_monitor/kernel/api/routers/integrations_home_assistant.py`:
- `GET /api/integrations/home-assistant/entities?filter=unavailable` → `HaEntityRowsResponse`
  (list of unavailable entities sourced from VM per-series queries). `Depends(require_session())`.
- `GET /api/integrations/home-assistant/batteries?filter=low_or_critical` → `HaBatteryRowsResponse`
  (list of low/critical battery devices sourced from VM per-series). `Depends(require_session())`.

### 3. VM query adapter

Extend `apps/monitor/homelab_monitor/kernel/services/victoria_metrics.py` (or mirror in a helper):
- Per-series row extraction function(s) (`query_entities_by_availability()`, `query_batteries_by_level()`)
  hitting VictoriaMetrics range-query + parsing per-series labels (entity_id, domain, level) into rows.
- Handle VM-down (502), empty results (empty list), and per-metric schema contracts (entity_id + domain
  labels, level as float).

### 4. Tests

In `apps/monitor/tests/unit/` or `tests/integration/`:
- httpx + pytest_httpx mocking VM per-series responses → correct row shape, unavailable-only filter,
  low/critical filter.
- Session-auth (401 on missing session).
- 502 on VM-down.
- Empty result (no unavailable entities) returns `{entities: []}`.
- Per-series label extraction (entity_id, domain correctly pulled from metric labels).

### 5. `GET /api/integrations/home-assistant/updates` — `HaUpdateRowsResponse` (list of `HaUpdateRow`: entity_id, title, installed_version, latest_version, release_url). Sourced from VM per-series (`homelab_ha_update_available{entity_id,title,...}`). (Deferred from STAGE-005-024.)

### 6. `GET /api/integrations/home-assistant/config-entries?filter=error` — `HaConfigEntryRowsResponse` (list: domain, title, state — setup_error vs setup_retry distinguished). Sourced from VM per-series (`homelab_ha_config_entry_setup_error{domain,title}` etc.). (Deferred from STAGE-005-024.)

### 7. `GET /api/integrations/home-assistant/repairs` — `HaRepairRowsResponse` (list: domain, issue_id, severity, summary). Sourced from VM per-series (`homelab_ha_repair_issue{domain,issue_id,severity}`). (Deferred from STAGE-005-024.)

### 8. OpenAPI export + TS codegen

- Run `make openapi-export` to regenerate OpenAPI spec.
- Run `pnpm run generate-types` in `apps/ui` to generate TS client types.
- Mirror TS re-exports in `apps/ui/src/routes/integrations/types.ts` if needed.

## Out of scope for this stage

- The drill-list UI widgets (028 — this is the data contract only).
- Pagination beyond a reasonable cap (Design decides if a cap/top-N is needed given ~1906 entities).
- Live HA re-query for detail (prefer VM per-series; Design confirms).

## Decisions in this stage

### Design phase will confirm

- **D-DETAIL-DATA-SOURCE** — VM per-series query vs live HA `get_states()` re-query for the
  per-entity rows (lean VM per-series, consistent with 021; confirm the VM read path supports
  per-series row extraction).
- **D-DETAIL-CARDINALITY** — cap/top-N/pagination given ~1906 entities (the unavailable subset is
  ~984; a cap or "top N + count" may be needed).
- **D-DETAIL-ENDPOINT-SPLIT** — separate `/entities` + `/batteries` vs one nested detail endpoint.
- **D-DETAIL-VM-LABELS-PRESENT** — VERIFY at Design that the per-series VM labels actually carry the needed detail fields (update title/versions/release_url; config-entry domain/state; repair domain/issue_id/severity/summary). Any field NOT present as a VM label (e.g. a repair summary string, or update release_url) cannot be sourced from VM per-series and must drop to a live-HA re-query (its own concern / stage) rather than being assumed available here.

### Locked Design Decisions (inherited from STAGE-005-023)

- **D-DETAIL-CONSUMES-VM** — detail rows come from the same VM metrics 021's counts come from
  (the per-series gauges), NOT a new SQLite cache and (preferably) NOT a live HA re-query.
- **D-DETAIL-BACKEND-FIRST** — backend detail endpoints land before 028's drill-down widgets
  (verified contract).
