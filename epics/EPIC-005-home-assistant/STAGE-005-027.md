# STAGE-005-027: HA entity + battery detail endpoints (drill-down backend)

## Status: Complete

## Overview

Backend detail endpoints that 028's drill-down widgets consume. STAGE-005-021 shipped a counts-only
`/summary` (per D-PANEL-ENDPOINT-SPLIT, detail deferred to "023/024 if widgets need them");
STAGE-005-023 then shipped counts-only entity-health + battery widgets and DEFERRED the drill-lists
because no detail endpoint exists. This stage builds those detail endpoints.

**Vertical slice:** after this stage, `GET /api/integrations/home-assistant/entities` returns the
per-entity rows for the unavailable/stale drill-list (entity_id, domain, last-changed age, available
flag) and `GET /api/integrations/home-assistant/batteries` returns per-device battery rows
(entity_id/device, level), so 028 can render "which 984 entities are unavailable" and "which battery
is critical".

**Scope extended (2026-06-13)** to also cover updates / config-entry / repairs detail endpoints deferred from STAGE-005-024.

**Spec references:** EPIC-005 §Stage decomposition; the deferred work recorded in STAGE-005-023
Design Notes ("DEFERRED FOLLOW-UP WORK"). Data source: VictoriaMetrics per-entity gauges
(`homelab_ha_entity_available{entity_id,domain}`, `homelab_ha_entity_last_changed_seconds{entity_id,domain}`,
`homelab_ha_battery_level{entity_id,domain}`) — extend the `vm_query` read path from per-aggregate
`vm_count` to per-series row extraction. Mirror the 021 router
(`kernel/api/routers/integrations_home_assistant.py`).

## Decisions in this stage

### Design phase will confirm

- **D-DETAIL-DATA-SOURCE** — VM per-series query vs live HA `get_states()` re-query for the
  per-entity rows (lean VM per-series, consistent with 021; confirm the VM read path supports
  per-series row extraction).
- **D-DETAIL-CARDINALITY** — cap/top-N/pagination given ~1906 entities (the unavailable subset is
  ~984; a cap or "top N + count" may be needed).
- **D-DETAIL-ENDPOINT-SPLIT** — separate `/entities` + `/batteries` vs one nested detail endpoint.
- **D-DETAIL-VM-LABELS-PRESENT** — VERIFY at Design that the per-series VM labels actually carry the needed detail fields (update title/versions/release_url; config-entry domain/state; repair domain/issue_id/severity/summary). Any field NOT present as a VM label (e.g. a repair summary string, or update release_url) cannot be sourced from VM per-series and must drop to a live-HA re-query (its own concern / stage) rather than being assumed available here.

### Locked Design Decisions (inherited from STAGE-005-023)

- **D-DETAIL-CONSUMES-VM** — detail rows come from the same VM metrics 021's counts come from
  (the per-series gauges), NOT a new SQLite cache and (preferably) NOT a live HA re-query.
- **D-DETAIL-BACKEND-FIRST** — backend detail endpoints land before 028's drill-down widgets
  (verified contract).

### Design Decisions (resolved 2026-06-13)

**D-DETAIL-VM-LABELS-PRESENT — RESOLVED (the load-bearing check). VERIFIED against the collectors: NONE of the five endpoints can be fully VM-sourced.** Per-series VM labels actually emitted:
- `homelab_ha_entity_available{entity_id,domain}` + `homelab_ha_entity_last_changed_seconds{entity_id,domain}` → `/entities` is FULLY VM-sourceable (entity_id, domain, available, last_changed_age all present).
- `homelab_ha_battery_level{entity_id,domain}` → MISSING `device`/friendly_name.
- `homelab_ha_update_available{entity_id,title}` → MISSING installed_version / latest_version / release_url (collector D-UPDATE-VERSION-CARDINALITY deliberately omits versions).
- `homelab_ha_config_entry_setup_error{domain,title}` → the gauge=1 collapses setup_error|setup_retry|migration_error|failed_unload; the SPECIFIC state is NOT a VM label.
- `homelab_ha_repair_issue{domain,issue_id,severity}` → MISSING `summary` (collector deliberately never emits free-text).

**Resolution — Option C (VM-only 027; defer the missing fields to NAMED collector/enrichment stages).** Chosen over Option B (live-HA enrichment inside 027) because the LOCKED D-DETAIL-CONSUMES-VM ("detail rows come from the same VM metrics ... NOT a live HA re-query") outranks this card's to-confirm D-DETAIL-VM-LABELS-PRESENT, and B would override the locked decision + pull 029's privacy-review machinery forward. C keeps 027 VM-only/single-source/single-failure-mode (mirrors 021 exactly), and fixes the config-entry gap at the collector layer (root cause) rather than masking it with a live query.

**What 027 ships (VM-only):**
- `/entities?filter=unavailable` — FULL (entity_id, domain, available, last_changed_age_seconds) + cardinality cap (see below).
- `/batteries?filter=low_or_critical` — `HaBatteryRow {entity_id, domain, level}` — `device` DROPPED → STAGE-005-031.
- `/updates` — `HaUpdateRow {entity_id, title}` — installed_version/latest_version/release_url DROPPED → STAGE-005-031.
- `/config-entries?filter=error` — `HaConfigEntryRow {domain, title, state}` where `state` is the coarse literal `"error"` only — precise setup_error-vs-setup_retry DEFERRED → STAGE-005-032 (collector change).
- `/repairs` — `HaRepairRow {domain, issue_id, severity}` — `summary` DROPPED → STAGE-005-031.

**No silent deferral (rule satisfied):** dropped fields are tracked in two NEW cards created this Design — **STAGE-005-031** (HA detail live-HA enrichment: battery device + update versions/release_url + repair summary, mirroring 029's live-HA machinery) and **STAGE-005-032** (config-entry state-distinction collector change). Epic denominator bumped 30 → 32.

**D-DETAIL-CARDINALITY — RESOLVED.** `/entities?filter=unavailable` can return ~984 rows. Apply a server-side cap via VM `topk()`/`sort_desc()` on `homelab_ha_entity_last_changed_seconds` (stalest first), default top-N = 100. Response envelope carries `total` + `returned` (so 028 can render "100 of 984, oldest first" — matches 028's D-DRILL-CAP). The other endpoints (`/batteries` ~tens, `/updates`, `/config-entries`, `/repairs`) are naturally bounded; a cap is cheap insurance, not load-bearing. Add `total`/`returned` to every `*RowsResponse` model.

**D-DETAIL-DATA-SOURCE — RESOLVED: VM per-series.** Generalize the per-series read pattern from `docker_socket_collector._query_prometheus` (parses `/api/v1/query` → `data.result` vector → `{metric: labels, value: [ts, str]}`) and reuse the 021 router's `vm_instant_query` (`apps/monitor/homelab_monitor/kernel/api/vm_query.py`) + `get_vm_url`/`get_http_client` deps + `HttpProblem(502, "upstream_unavailable")` error shape + pyright-safe Unknown-JSON discipline from `metrics.py`. Extend from per-aggregate `vm_count` to per-series ROW extraction.

**D-DETAIL-ENDPOINT-SPLIT — RESOLVED: separate endpoints** (`/entities`, `/batteries`, `/updates`, `/config-entries`, `/repairs`) on the 021 router (`integrations_home_assistant.py`), each `Depends(require_session())`, each its own `*RowsResponse` Pydantic model with `ConfigDict(extra="ignore")`. OpenAPI regen REQUIRED (new models) — `make openapi-export` then `pnpm run generate-types`.

**Build mechanics (grounded):** mirror the 021 router structure + `test_api_ha_summary.py` test pattern (pytest_httpx callback planting per-series vector responses; `authenticated_client` fixture; GET needs session, no CSRF; `is_reusable=True` on the VM mock). VM-down → 502; empty vector → empty list; the router is already mounted so new endpoints auto-appear in OpenAPI.

**Rejected — Option A** (drop fields + sprawl into 2-3 trailing stages): C consolidates the enrichment into ONE stage (031) + one collector stage (032), cleaner deferral trail.
**Rejected — Option B** (live-HA enrichment inside 027): overrides the locked D-DETAIL-CONSUMES-VM, adds a second data source + partial-degradation test matrix + an early privacy review the card scoped to 029. Would require the user to explicitly relock D-DETAIL-CONSUMES-VM — not assumed.
