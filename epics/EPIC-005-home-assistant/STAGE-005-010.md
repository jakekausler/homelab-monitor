# STAGE-005-010: Config-entry state collector (websocket)

## Status: Complete

## Overview

The first websocket-consuming collector: subscribe to HA's config-entry state so the monitor knows,
authoritatively, when an integration is `loaded` vs `setup_error` vs `setup_retry` vs `not_loaded`.
This is the precise version of "integration setup failure" (scenario #10). The REST API does NOT
expose this (`/api/config/config_entries` → 404) and the `.storage/core.config_entries` file holds
only static config (no live state) — so the websocket is required. A coarse log-pattern approximation
also exists (005-014), but this gives structured, per-integration state.

**Vertical slice:** after this stage, `homelab_ha_config_entry_state{domain,title,state}` (or a
loaded/error gauge per integration) reflects each integration's live load state, updating when an
integration drops or recovers.

**Spec references:** master spec §2 Q10 (integration health), EPIC-005 §"Signal-source matrix"
(config-entry state via websocket). Scenario #10.

## Concrete deliverables

### 1. Config-entry state subscriber

Using the STAGE-005-002 websocket client, subscribe to the config-entries state source (the WS command
that lists config entries with live `state`/`reason` + subscribe to changes — Design confirms the
exact WS command, e.g. `config_entries/get` + `config_entries/subscribe` or `subscribe_events` on the
relevant event). Maintain current per-integration state and emit a metric:
`homelab_ha_config_entry_state{domain,title}` encoded so `setup_error`/`setup_retry`/`not_loaded` are
distinguishable from `loaded` (Design: a numeric state code, or a `..._loaded` 1/0 + a
`..._setup_error` 1/0). Include the failure `reason` where available (label or panel-only).

### 2. Bridge websocket → metric (not per-tick)

Because this is event-driven (websocket) rather than poll-per-tick, Design decides how the
subscriber's state becomes metrics: a lightweight periodic emit of the current snapshot (a thin
collector tick that reads the subscriber's cached state) vs direct emission on each WS update. Keep it
consistent with the metrics path + cap.

### 3. Tests

Fake WS events: integration goes `loaded` → `setup_retry` → `loaded` → assert metric transitions;
reconnect mid-stream → state recovers; disconnect → state marked stale. 3b: observe a real
integration's loaded state.

## Design Phase

### Design Notes (resolved 2026-06-11)

Brainstormer (Opus) generated options; coordinator selected. First WS-consuming collector — establishes
the pattern 011/012 inherit. Grounded in verified WS-client + collector-framework anchors.

**D-WS-TO-METRIC-BRIDGE → CHOSEN: Option A1 — inject the WS client as a private field; per tick call
`send_command("config_entries/get")` (snapshot RPC); NO subscription.**
- The WS client is ALREADY instantiated + start_task()-ed in the lifespan (`app.state.ha_ws_client`).
  010 injects it into the collector as a private field via lifespan post-construction, mirroring the
  `DockerSocketCollector._client` precedent (`# pyright: ignore[reportPrivateUsage]`). Do NOT add a `ws`
  field to CollectorContext (cross-cutting change for one consumer; promote later only if many WS
  collectors proliferate).
- Each tick: if `not ws.connected` → skip (stale, don't crash); else
  `result = await ws.send_command("config_entries/get")`; `isinstance(result, HaError)` → skip/stale
  (emit NOTHING — don't emit stale 1s that would mask a real outage; the collector-run metric records
  the failure). Else parse entries, emit the 1/0 gauge families.
- WHY NOT the push subscription (Option A2/B): a 30-60s tick CANNOT deliver freshness that survives the
  tick/scrape boundary; building full state from subscribe-deltas needs an initial `config_entries/get`
  snapshot ANYWAY (so subscribe is additive, not substitutive); the bounded queue (256, oldest-dropped)
  introduces a silent desync failure mode that only a periodic re-list (= A1) repairs. A1 is the YAGNI,
  reconnect-trivial, cap+`homelab_collector_run_*`-compliant choice that gives 011/012 a 3-line recipe
  (inject ws, RPC-list each tick, emit 1/0 gauges) with ZERO asyncio-queue machinery. REJECTED Option C
  (direct emission from subscriber) — forfeits the cap budget + collector self-metrics the card mandates.
- REVERSIBILITY: if a future stage ever needs sub-tick reaction, re-add `config_entries/subscribe`
  alongside the existing `get`. A1 doesn't paint us into a corner.

**D-CONFIGENTRY-METRIC-SHAPE → CHOSEN: Option C — two 1/0 gauges (mirrors 006/008 convention).**
- `homelab_ha_config_entry_loaded{domain,title}` = 1.0 iff state == "loaded", else 0.0.
- `homelab_ha_config_entry_setup_error{domain,title}` = 1.0 iff state ∈ {setup_error, setup_retry,
  migration_error, failed_unload}, else 0.0.
- `not_loaded` / `setup_in_progress` (and any unknown future state) → BOTH gauges 0.0. This cleanly
  separates "broken" (setup_error=1) from "intentionally off / transient" (both 0) — which a numeric
  code gauge cannot express to a rule without a magic-number lookup.
- Define the state→(loaded, setup_error) mapping as ONE explicit dict in the collector — the single
  place the HA state enum is interpreted; unknown states default to (0,0) safely.
- 015's #10 integration-down rule consumes `homelab_ha_config_entry_loaded == 0` (a value comparison,
  self-documenting). REJECTED Option A (state-as-label) — Prometheus enum/state-set anti-pattern;
  emit-only-active churns series identity on every transition (stale-series window), emit-all-states =
  7x cardinality. REJECTED Option B (numeric code) — opaque magic-number contract, diverges from the
  1/0 convention.

**D-CONFIGENTRY-WS-COMMAND → CHOSEN: `config_entries/get` only (no subscribe).**
- Single snapshot RPC per tick via `send_command`. HARDENING: the exact response wrapping is NOT
  verifiable from code (no codebase precedent for the command). `send_command` returns `dict | HaError`;
  HA may return the entries list under a key OR the result may be the list itself. Build implements a
  DEFENSIVE parse handling both shapes (bare list vs wrapped-under-key), with tests for both; 3b
  confirms which is real against live HA (ideally observing a real setup_retry).

**`reason` → PANEL-ONLY, NOT a metric label.** `reason` is high-churn free text (e.g. "Failed to
connect to host: timeout") — exactly the case the 008/009 metric-hygiene discipline excludes from
metrics (would churn series identity + pollute cardinality). The metric carries only stable
{domain,title}. `reason` surfaces in the UI panel / detail view (queries HA directly), like 008/009.
`title` stability noted: moves on integration rename (low-frequency, acceptable — same risk class as any
human-readable label).

**Bridge / lifecycle:** the collector is a normal tick-based BaseCollector (name `ha_config_entry`,
concurrency_group "homeassistant", interval ~60s — config-entry state changes are rare, 60s latency to
detect setup_retry is fine). WS reconnect/backoff is the WS client's job (005-002); the collector just
checks `ws.connected` each tick. Cardinality: tens of config entries → global default cap 500 ample,
no per-family override.

**Out of scope (confirmed):** the integration-down RULE (015 #10 / 014 log path); repairs (011) +
notifications (012) sibling WS collectors; the coarse log-pattern setup-failure rule (014, complementary).

**Verified anchors (2026-06-11):** WS client `HomeAssistantWebsocketClient` instantiated + started in
lifespan @ `app.state.ha_ws_client`; `send_command(type_, **fields) -> dict | HaError`; `connected`
property; `Subscription` is async-iterable (NOT used — A1 chosen); collectors STRICTLY tick-based (no
lifecycle hook beyond run()); long-lived deps injected as private fields via lifespan (DockerSocketCollector
precedent); CollectorContext has NO ws field; `config_entries/get` is the documented HA WS list command.

**3b found a BLOCKING WS-client bug — fixed during Refinement.** This was the load-bearing real-data
check this stage was designed around.
- The WS client's `_result_payload` (kernel/ha/websocket.py) collapsed any non-dict `result` to `{}`.
  HA's `config_entries/get` returns a top-level JSON ARRAY (189 entries on this install). So
  `send_command("config_entries/get")` returned `{}` → the collector emitted ZERO series. The defensive
  parse handled it without crashing, but "doesn't crash" ≠ "works".
- FIX (kernel change, applied + re-verified during Refinement): widened `_result_payload` /
  `send_command` / the pending+subscribe Futures to `dict[str,object] | list[object] | HaError`, and
  `_extract_entries`'s param to `dict | list`. Added a WS unit test `test_send_command_success_list_
  result_returns_list` (list passthrough). 100% coverage on websocket.py maintained. Downstream type
  fixes: removed a now-redundant cast in `_extract_entries`; widened one test-local Future annotation
  (invariant Future type-param match).
- POST-FIX 3b RE-VALIDATION (real HA, rebuilt+recreated monitor image): `send_command` now returns a
  `list` of 189; collector `ok=True`, metrics_emitted=380 (189 loaded + 189 setup_error + 2 drop
  gauges); **158 loaded / 31 not-loaded; 3 real integrations in setup_error (blink, frigate, shelly)**;
  drop gauges 0.0 (189 << 500 cap); labels {domain,title} only (no reason); BOTH families confirmed

**3a (wiring):** `HaConfigEntryCollector` confirmed in `_HA_COLLECTORS`; lifespan injects
