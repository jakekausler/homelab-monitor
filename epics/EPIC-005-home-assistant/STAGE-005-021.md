# STAGE-005-021: Backend HA panel data endpoint(s)

## Status: Complete

## Overview

The backend data feed for the HA integration UI panel: typed endpoint(s) under
`/api/integrations/home-assistant/` returning the summary rows the panel widgets render. Built before
the frontend (so the UI builds on a verified contract), mirroring the Docker integration's
`/api/integrations/docker/containers` → `ContainerRow[]` pattern.

**Vertical slice:** after this stage, `GET /api/integrations/home-assistant/summary` returns typed JSON
(entity-health counts, battery summary, update count, recent automation-failure count,
integration-issue/repairs/notification counts, HA up + last-seen) — consumable by the frontend.

**Spec references:** master spec §9.2 (Integrations per-integration panels), §3.1 (API layer + OpenAPI
→ frontend types). EPIC-005 §"Stage decomposition" Wave F. Docker pattern:
`apps/monitor/.../api/routers/docker.py` + `apps/ui/src/routes/integrations/types.ts`.

## Concrete deliverables

### 1. Router `/api/integrations/home-assistant/`

A new router (`kernel/api/routers/home_assistant.py`, `APIRouter(prefix="/integrations/home-assistant")`,
registered in `app.py` like docker). Endpoints (Design finalizes the split):

- `GET /summary` — `HASummaryResponse`: entity counts (total / available / unavailable / stale),
  battery summary (count low/critical), update count (available / total), recent automation-failure
  count (window), config-entry issue count (loaded / error / retry), repairs count, notification count,
  `ha_up`, last-seen.
- Optionally `GET /entities` (paged entity-health rows) and/or `/updates`, `/issues` if widgets need
  detail beyond the summary (Design decides whether to split or return nested in `/summary`).

### 2. Typed models + UI types

Pydantic response models; add the mirrored TS types to `apps/ui/src/routes/integrations/types.ts`
(pulled from `@/api/schema`), and run `make openapi-export` + `pnpm run generate-types` so the
frontend (Wave F) consumes them. Data sourced from the collectors' metrics/state (Design: query VM via
the existing read path, or read collector-cached state, or the DB — whichever the codebase already
uses for integration panels).

### 3. Tests

Route tests (httpx AsyncClient, in-memory): summary returns correct aggregated shape from planted
collector data; session-auth required (401 without); HA-down reflected in `ha_up`. Confirm OpenAPI
regen produces the TS types.

### Design Notes (resolved 2026-06-12)

Brainstormed via brainstormer (Opus); decisions made by the agent per workflow §0.4 (no user pause). Grounded in verified codebase facts.

**D-PANEL-DATA-SOURCE — CHOSEN: VM instant queries via a reusable API-layer instant-query helper (batched).**
- Summary count aggregates come from VictoriaMetrics instant queries (`/api/v1/query`), NOT a SQLite cache (Docker pattern) and NOT live HA re-query.
- The public VM read path (`kernel/api/routers/metrics.py::metrics_range`) is RANGE-only (`/api/v1/query_range`). No public INSTANT-query helper exists in the API layer. The only instant-query precedent is private: `docker_socket_collector._query_prometheus`. So this stage builds a NEW reusable instant-query helper (sibling to the metrics read path), takes `(http_client, vm_url, query)`, hits `/api/v1/query`, returns the parsed instant vector, raises a typed error on VM failure.
- Counts batched into ONE `or`-joined instant query keyed by a synthetic label (e.g. `label_replace(count(homelab_ha_entity_available == 1), "k","entity_available","","") or ...`) to collapse ~8 round-trips into 1.
- REJECTED (b) collector-cached SQLite aggregates: unjustified new table/migration/collector; second drift-prone source vs Grafana/metrics. REJECTED (c) live HA re-query for counts: duplicates collector aggregation logic; ~1906-entity payload per card refresh.

**D-PANEL-ENDPOINT-SPLIT — CHOSEN: single `GET /summary` only this stage.**
- All summary counts share one mechanism (VM). Detail endpoints (entity lists, notification bodies, config-entry retry split) need a different (HA-live) source + unproven shape — deferred to 023/024 where the consuming widget pins the contract.

**D-PANEL-NOTIF-EXPOSURE — CHOSEN: count only, no bodies.**
- `notifications: int` from `homelab_ha_persistent_notification` (VM gauge, label `notification_id` only — privacy-safe). Bodies were deliberately kept out of metrics (stage 012); no HA-live call on the summary hot path. Bodies deferred to a future 024 detail endpoint if a widget needs them.

**Sub-decisions (forced by ground-truth data constraints):**
- **stale count — OMITTED.** The codebase deliberately avoids a blanket stale threshold (~50% of ~1906 entities are chronically unavailable; staleness is opt-in per-entity user rules only). A blanket stale count would invent the threshold the design rejected. `/summary` ships `entities: {total, available, unavailable}` — NO stale field.
- **config-entry retry — loaded/error only (2-way).** The collector folds `setup_retry` into `setup_error` (only two gauges: `homelab_ha_config_entry_loaded`, `homelab_ha_config_entry_setup_error`). `/summary` ships `config_entries: {loaded, error}`. A 3-way retry split would require an HA-live call.
- **automation-failure count — DEFERRED (excluded from /summary).** This count's source is VictoriaLogs (LogsQL: `_time:5m service:="homeassistant" ("Error while executing automation" OR "Error executing script")`), a DIFFERENT upstream than VM. Including it would double the endpoint's failure surface for one number. Stage 025's embedded HA-scoped LogViewer surfaces these inline. If a numeric badge is later wanted, add an isolated `GET /automation-failures` (VictoriaLogs-sourced) endpoint.

**Three card fields intentionally dropped vs original card text:** `stale` count, config-entry `retry`, automation-failure count. Each forced by a deliberate prior-stage design decision (012/015/016); each can return as an isolated future endpoint if a consuming widget genuinely needs it.

**Battery thresholds (match vmalert `deploy/vmalert/metrics/home-assistant.yaml` exactly):** critical `< 10`, low `>= 10 and < 20`.

**Failure contract:**
- VM unreachable → endpoint returns 503 (NOT 200-with-zeros — zeros would be indistinguishable from a healthy-empty HA).
- HA down but VM up → 200 with `ha_up=false` + last-known counts (`homelab_ha_up == 0`).
- `last_seen` derived from the `homelab_ha_up` sample timestamp.

**Net response model** (`model_config = ConfigDict(extra="ignore")`):
`HaSummaryResponse { entities: {total, available, unavailable}, battery: {low, critical}, updates: {available, total}, config_entries: {loaded, error}, repairs: int, notifications: int, ha_up: bool, last_seen: datetime }`.

**Verified path note:** real module path is `apps/monitor/homelab_monitor/...` (NO `src/`); Build re-verifies exact paths/signatures.
