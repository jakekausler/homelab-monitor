# STAGE-005-031: HA detail live-HA enrichment (battery device + update versions + repair summary)

## Status: Complete

## Overview

STAGE-005-027 built the HA detail endpoints (`/entities`, `/batteries`, `/updates`, `/config-entries`, `/repairs`) sourced VM-only (honoring the locked D-DETAIL-CONSUMES-VM: detail rows come from the same per-series VM gauges 021's counts come from, NOT a live HA re-query). The load-bearing D-DETAIL-VM-LABELS-PRESENT check in 027 Design verified that several display fields the original response models named are NOT emitted as VM labels (deliberately — the collectors omit free-text / high-cardinality fields for privacy + cardinality reasons). 027 dropped those fields from its response models and tracked them here.

This stage adds a live-HA enrichment layer to the 027 detail endpoints, supplying the dropped display fields via a single `HomeAssistantRestClient.get_states()` (and, for repairs, a `repairs/list_issues` WS call) keyed on the VM-selected rows — mirroring the live-HA re-query machinery established by STAGE-005-029 (notification bodies). VM still drives row SELECTION + filtering (count-consistent, cardinality-bounded); live-HA only enriches.

Vertical slice: after this stage, `/batteries` rows carry `device` (friendly name), `/updates` rows carry `installed_version` / `latest_version` / `release_url`, and `/repairs` rows carry `summary` — so 028's drill widgets can show human-readable detail.

## Deferred from

STAGE-005-027 Design (2026-06-13), D-DETAIL-VM-LABELS-PRESENT. The fields dropped from 027's VM-only response models:
- `HaBatteryRow.device` (friendly name) — VM `battery_level` carries only {entity_id, domain}.
- `HaUpdateRow.installed_version` / `latest_version` / `release_url` — VM `update_available` carries only {entity_id, title} (collector D-UPDATE-VERSION-CARDINALITY deliberately omits versions).
- `HaRepairRow.summary` — VM `repair_issue` carries only {domain, issue_id, severity} (collector deliberately never emits free-text).
- (Optionally `HaEntityRow.friendly_name` if 028 needs it — confirm in this stage's Design.)

## Decisions in this stage

### Design phase will confirm

- D-ENRICH-SOURCE — `get_states()` vs per-entity fetch vs WS for each field.
- D-ENRICH-PRIVACY — whether any enriched field is sensitive enough to gate (mirror 029).
- D-ENRICH-DEGRADE — exact shape when HA is down (null fields vs omitted).

### Design Notes (2026-06-14) — decisions made

Wiring + data shapes VERIFIED against the live codebase. The REST client `HomeAssistantRestClient.get_states() -> list[HaState] | HaError` returns all ~1900 entities in ONE call (enrichment attributes live in `HaState.attributes`); the WS `repairs/list_issues` command (already used by `ha_repairs.py` with an `_extract_issues` helper) carries the free-text `summary`. The WS client is on `app.state.ha_ws_client` (`get_ha_ws_client` dep exists from 029); the REST client is built but NOT on app.state — 031 adds `app.state.ha_client` + a new `get_ha_client` dep.

- **D-ENRICH-SOURCE → shared `kernel/ha/enrichment.py` helper + per-endpoint bulk `get_states()` (CHOSEN).** New `kernel/ha/enrichment.py` with `build_states_index(states) -> dict[str, HaState]`, a kernel-local `extract_issues(result)` (defensive list/`{"issues":[...]}` shape, mirroring the collector's — kernel must NOT import from plugins), `build_repairs_index(issues) -> dict[tuple[str,str], str]`, and a typed `attr_str(state, key) -> str | None` coercion (the error-prone untyped-`attributes` extraction lives in ONE tested place). Each entity-keyed endpoint (batteries, updates, entities) calls `get_states()` ONCE (one bulk call; build index; look up VM-selected rows by entity_id); repairs calls `send_command("repairs/list_issues")` and indexes by `(domain, issue_id)`. *Rejected:* inline-per-endpoint (copy-pastes the fragile coercion 3×); per-entity/N+1 fetch (get_states is a single bulk call). VM still drives row SELECTION (LOCKED D-DETAIL-CONSUMES-VM honored — enrichment NEVER changes which rows appear). Add `get_ha_client` mirroring `get_ha_ws_client` (`_require_state(request, attr="ha_client", code="ha_client_unavailable", ...)`) + `app.state.ha_client = ha_client` in lifespan.
- **D-ENRICH-PRIVACY → full fields verbatim to the authenticated session, enforced never-logged/never-metric/never-error-leak (CHOSEN).** Mirrors 029: the single owner sees this content in HA itself; the bodies/free-text-out-of-METRICS posture (012/029/repairs-collector) does NOT mean out-of-the-authenticated-UI. Session-auth already enforced by the inherited 027 `Depends(require_session())`. Enriched fields (device, versions, release_url, summary, friendly_name) are display-only and MUST NOT become metric labels, MUST NOT be logged (only entity_id / (domain,issue_id) / counts loggable), MUST NOT appear in error messages (HA errors are swallowed → null, never raised). *Rejected:* redact/gate any field (security theater for a single-user panel; contradicts 029). MANDATORY sentinel test (029 pattern): inject a sentinel `friendly_name` + sentinel repair `summary`, assert each appears in the response body but in NO captured log event.
- **D-ENRICH-DEGRADE → enriched fields `str | None`; None for BOTH HA-down AND per-row-not-found; no degradation flag (CHOSEN).** On `isinstance(get_states()/send_command result, HaError)` the endpoint skips index-building and every row's enriched field is None; per-row index-miss also → None; found-but-attribute-missing/non-str → None (the two causes intentionally collapse, undistinguishable at field level — 028 renders null uniformly). VM rows + count + 200 ALWAYS preserved; HA-enrichment failure is NEVER a 502 (LOCKED graceful degradation — VM already supplied the rows). The response shape stays a bare row list (no `ha_up`/`enriched` wrapper) → ZERO frontend change required (028's widgets render the new fields present-or-absent defensively). Coercion rule: `attributes.get(key)` returned only if `isinstance(str)`, else None — never `str()`-coerce arbitrary objects. *Rejected:* empty-string sentinels (ambiguous vs legit-empty); a per-response `ha_up` flag (would wrap the bare-list shape = a frontend-visible contract change this stage — deferred as a future frontend enhancement).
- **D-ENRICH-ENTITY-FRIENDLY-NAME → add `friendly_name: str | None` to `HaEntityRow` now (backend only); DEFER the 028 widget rendering of it to STAGE-005-033 (CHOSEN).** Near-zero marginal cost (reuses the same `build_states_index`); 028's entities drill renders defensively so the present-but-unused field is harmless until a frontend tweak picks it up. The frontend work to DISPLAY friendly_name in the entities drill is tracked as the new STAGE-005-033 (created this Design; depends on 031's backend field → higher-numbered). *Rejected:* YAGNI/omit (a future stage would redo the same get_states wiring it gets free here).

**Models changed** (then `make openapi-export` → `pnpm generate-types`; all new fields `str | None`):
`HaBatteryRow.device`; `HaUpdateRow.installed_version` / `latest_version` / `release_url`; `HaRepairRow.description + HaRepairRow.learn_more_url`; `HaEntityRow.friendly_name`.

**Repair-field correction (Build-phase verification):** the Design working-name "repair `summary`" does NOT exist as a key on stock HA `repairs/list_issues` issue objects (verified against the real issue shape in `test_ha_repairs_collector.py`: issues carry `domain, issue_id, active, ignored, severity, translation_key, translation_placeholders, description, learn_more_url, breaks_in_ha_version, is_fixable` — no `summary`). To avoid shipping an always-null field, the repair enrichment reads the REAL fields: `description` (free-form prose, the human-readable text) + `learn_more_url` (documentation URL), via a `RepairEnrichment` frozen dataclass keyed `(domain, issue_id)`. Both degrade to None (HA-down / issue-absent / key missing / non-str / empty), per the locked graceful-degradation contract.

**LOCKED honored:** D-DETAIL-CONSUMES-VM (VM drives selection; live-HA only enriches) ✓; graceful degradation (HA-down → null fields + 200, NOT 502) ✓.

**Deferrals tracked:** (a) entities-drill rendering `friendly_name` → STAGE-005-033 (created this Design). (b) optional `ha_up`/`enriched` degradation badge → noted as a future frontend enhancement (not yet a stage; cheap, low-value on a single-user panel — to be created only if the user wants the badge). (c) collector/kernel `extract_issues` de-duplication (kernel gets its own copy this stage; collector keeps its own) → minor future tidy, no behavior impact.

**Mandatory tests:** backend enrichment-never-logged sentinel test; backend tests over mocked VM + mocked HA (get_states returns states / HaError → null enrichment + 200; repairs WS returns issues / HaError → null summary + 200; per-row not-found → null; both list & dict-wrapped repairs shapes); the enrichment helper's standalone unit tests (index build + attr_str coercion). OpenAPI regen.
