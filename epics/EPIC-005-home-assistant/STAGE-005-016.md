# STAGE-005-016: Built-in safety-sensor rules + HA threshold presets

## Status: Complete

## Overview

Cover the highest-value HA monitoring scenario (#3): dangerous-state / threshold alerting. Two halves:
(a) zero-config built-in vmalert rules for safety sensors — smoke/gas/CO/water binary sensors firing →
critical, plus door/window-left-open; and (b) HA numeric-threshold presets (e.g. "freezer temp")
seeded into the user-authored MetricsQL rule machinery built in STAGE-005-005, so the user gets
sensible starting thresholds they can customize.

**Vertical slice:** after this stage, out of the box, a `binary_sensor` with `device_class: smoke`
turning `on` produces a critical alert with no configuration; and the user finds pre-seeded,
editable HA threshold rules (freezer temp, etc.) in the metric-rules surface from 005-005.

**Spec references:** master spec §2 Q10 (HA signals), §8.2 (critical severity). EPIC-005 §"User-
customizable thresholds". Scenario #3 (both #3a built-in safety + #3b user thresholds).

## Concrete deliverables

### 1. Built-in safety-sensor rules (#3a)

vmalert-metrics rules (in `home-assistant.yaml` or a `home-assistant-safety.yaml`) firing on the
entity-availability/state metrics for `binary_sensor` entities by `device_class`:

- `device_class` in {`smoke`, `gas`, `carbon_monoxide`, `moisture` (water leak)} in the `on` state →
  **critical**, immediately.
- door/window (`device_class` in {`door`, `window`, `opening`}) `on` for > a configurable duration →
  warning ("left open"). Default duration conservative; user can tighten via 005-005.

These require the relevant binary_sensor state to be emitted as a metric — confirm 005-006 emits
enough (availability + state) or extend the emission minimally (flag if so). Zero user config for the
safety set.

### 2. HA numeric-threshold presets (#3b)

Seed a set of HA-specific numeric-threshold rule presets into the STAGE-005-005 user-authored
MetricsQL machinery: e.g. freezer/fridge temp bounds, indoor temp comfort bounds, humidity bounds —
as EDITABLE user rules (pre-populated defaults the user can change/disable), NOT hard-coded built-ins.
Design decides the preset list + how presets are seeded (a one-time seed migration / a "suggested
rules" surface in the 005-005 UI). The presets demonstrate the customizable-threshold capability with
real, useful HA examples.

### 3. Tests

Safety: plant a smoke binary_sensor `on` → critical alert fires immediately; water `on` → critical;
door open < duration → no alert, > duration → warning. Presets: the seed creates editable rules in the
metric-rules store; editing a preset's threshold changes the rendered vmalert rule.

## Out of scope for this stage

- The generic metric-rule machinery itself (005-005 — this stage only seeds presets into it).
- General entity unavailable/stale/battery/anomaly rules (005-015).
- Non-HA safety sensors (other integrations seed their own later).

### Design Decisions (RESOLVED)

**Scope note:** This stage is larger than a pure-rules stage — it adds TWO new HA collectors (Python) + safety vmalert-metrics rules + a preset-seeding migration. Both halves (#3a safety, #3b presets) require new metrics that do not exist today; this is real dependency work the card scopes, not scope creep. STAGE-005-006's D-STATE-METRIC explicitly deferred binary_sensor state emission to this stage.

**D-SAFETY-METRIC-SOURCE (#3a) → CHOSEN: new collector `HaSafetySensorsCollector` emitting `homelab_ha_binary_sensor_on{entity_id,domain,device_class}`.** Value 1.0 when state=="on", 0.0 when state=="off", SKIP (no emission) for unavailable/unknown/empty (an unavailable smoke detector must NOT read as "no fire" or "fire"). Filter: domain=="binary_sensor" AND device_class in the safety allow-set. Default allow-set (frozenset, overridable via env config in the established frozen-dataclass + load_*_config() pattern read inside run()): {smoke, gas, carbon_monoxide, moisture, door, window, opening}. Interval 60s (mirrors ha_battery; vmalert 60s eval is fast enough for the for:0s safety path). CappedEmitter at the default 500 cap (~≤100 safety series; NO _DEFAULT_CARDINALITY_FAMILIES entry). Register in homeassistant/__init__.py _HA_COLLECTORS. Matches 006's explicit FLAG verbatim (006 D-STATE-METRIC). Rejected: extending ha_entity_available (contradicts 006's "availability + last_changed, NO state metric" lock, muddies a clean collector); emitting for ALL ~357 binary_sensors (adds ~257 series of non-safety noise with no consumer).

**D-SAFETY-RULES-SHAPE (#3a) → CHOSEN: new file `deploy/vmalert/metrics/home-assistant-safety.yaml` (+ `__tests__/home-assistant-safety.tests.yaml`).** Two rules, each a single rule with a device_class regex (not one-per-class — identical severity/handling, the alert carries {{ $labels.device_class }} + {{ $labels.entity_id }} to self-identify):
- `HaSafetySensorTriggered`: `homelab_ha_binary_sensor_on{device_class=~"smoke|gas|carbon_monoxide|moisture"} == 1`, for: 0s, severity critical (a fire alarm must not wait).
- `HaContactSensorOpen`: `homelab_ha_binary_sensor_on{device_class=~"door|window|opening"} == 1`, for: 30m, severity warning (conservative default — a door open 10-15m is routine; user tightens per-entity via 005-005).
Labels match 015: severity / source_tool: vmalert-metrics / target_kind: container / integration: home_assistant; explorer annotation `/integrations/home-assistant`. NO dedicated routing label (YAGNI — Alertmanager already routes on severity: critical; no downstream stage consumes a safety route). Separate file (not 015's home-assistant.yaml) keeps life-safety rules isolated and avoids editing 015's verified-contract header; the card permits either. Test fixture (vmalert unit-test static-fixture format): smoke==1 → critical fires immediately; moisture==1 → critical; door==1 held 15m → no fire; door==1 held 35m → warning fires; all-zero baseline → no alerts.

**D-PRESET-METRIC-SOURCE (#3b) → CHOSEN: add a SCOPED collector `HaSensorValueCollector` emitting `homelab_ha_sensor_value{entity_id,device_class}` for temperature+humidity device_classes ONLY.** Raw float via parse_float_state (rejects non-finite, skips unavailable/unknown). Allow-set {temperature, humidity} (freezer/fridge/indoor temps all surface as device_class=temperature; humidity as device_class=humidity). ~50-150 series — fits the default 500 cap, NO _DEFAULT_CARDINALITY_FAMILIES entry. NO unit_of_measurement label (display-only, higher-churn; presets filter on device_class + entity_id, never unit). The controlling precedent is 015's HaEntityUnavailable lesson: do NOT ship a rule that can't work — preset rules need a real target metric. A SCOPED temp+humidity metric has a concrete in-stage consumer (the presets) and bounded cardinality, unlike a general all-~1075-sensors value metric (which has NO consumer — 021/023/024 panel re-queries HA live, not metric labels — and would need a 1500+ cap). Rejected: deferring #3b (ships a dead deliverable / re-scopes the card); general sensor-value metric (metric pollution, no consumer).

**D-PRESET-LIST (#3b) → CHOSEN: 6 presets, expr_kind=metricsql, source_kind='preset', enabled=0 (disabled), for: 15m, placeholder entity_ids the user edits.** A seeded preset cannot know the user's real entity_id (sensor.freezer_temp won't match sensor.garage_freezer_temperature); shipping ENABLED against a guessed id = a dead rule (matches nothing) or a noisy false rule (matches something unrelated). Disabled is the only non-noisy, non-dead option — a disabled rule is NOT rendered (render_all reads list_enabled()), so it sits as an editable template ("user finds pre-seeded editable rules out of the box") without firing. Each summary/description instructs: "PRESET — edit entity_id to your actual sensor, adjust threshold, then enable. Disabled by default because the placeholder entity does not exist on your system." Presets:
- PresetFreezerTooWarm: `homelab_ha_sensor_value{entity_id="sensor.freezer_temp"} > -15`, warning
- PresetFridgeTooWarm: `homelab_ha_sensor_value{entity_id="sensor.fridge_temp"} > 5`, warning
- PresetIndoorTempLow: `homelab_ha_sensor_value{entity_id="sensor.indoor_temp"} < 16`, warning
- PresetIndoorTempHigh: `homelab_ha_sensor_value{entity_id="sensor.indoor_temp"} > 28`, warning
- PresetHumidityHigh: `homelab_ha_sensor_value{entity_id="sensor.indoor_humidity"} > 65`, warning
- PresetHumidityLow: `homelab_ha_sensor_value{entity_id="sensor.indoor_humidity"} < 30`, info
Rejected: target-by-device_class-range (fires on EVERY temp sensor — outdoor/CPU/fridge — temperature thresholds are inherently per-entity).

**D-PRESET-SEEDING (#3b) → CHOSEN: non-gated, idempotent alembic migration `op.bulk_insert` of the 6 preset rows (NOT env-gated).** source_kind='preset', enabled=0, source_ref=NULL. Rejected: env-gated seed like 0008's HOMELAB_MONITOR_INCLUDE_DEMO_SEEDS (dev-rig-only; prod never sets it → presets would never appear in prod, defeating "out of the box"); a 005-005 suggested-rules UI (frontend, out of this backend stage's scope). The migration runs EXACTLY ONCE per DB (alembic version-stamped), so a preset the user later DELETES stays deleted — it is NOT resurrected on reboot (strictly better than a startup-seed step which would need explicit seeded-before bookkeeping). Presets are normal log_user_rules rows → fully editable/disable-able/deletable via existing CRUD (satisfies "EDITABLE, NOT hard-coded"). Build flag: op.bulk_insert BYPASSES the repo's create() _validate_and_render_check, so the migration author must ensure each preset expr renders cleanly; add a unit test asserting all 6 seeded rows render without error AND that editing a preset's threshold via the repo changes the rendered file (deliverable #3's acceptance test).

**Collector structure → TWO separate collectors** (HaSafetySensorsCollector for binary_sensor_on, HaSensorValueCollector for sensor_value) — matches the repo's one-collector-one-family norm and the z-score precedent of independent get_states() polls. Keeps each concern independently testable.

**Cross-cutting label note (do NOT "fix" later):** the user-rule renderer (kernel/logs/user_rules_render.py) HARDCODES source_tool: user / category: user-rule on every rendered rule and does NOT emit integration: home_assistant or source_kind. So the #3b presets render with USER-RULE labels, deliberately DIFFERENT from the #3a safety rules' source_tool: vmalert-metrics / integration: home_assistant. This is correct — presets ARE user rules. The two halves of this stage intentionally produce rules with different label families.

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-SAFETY-BUILTIN-ZERO-CONFIG** — safety-sensor (smoke/gas/CO/water) alerting is built-in, critical,
  zero-config out of the box.
- **D-THRESHOLDS-VIA-USER-RULES** — user-customizable numeric thresholds ride the 005-005 machinery
  (the user's explicit "extend the machinery" choice); this stage provides HA presets on top, it does
  NOT invent a separate HA threshold config.
