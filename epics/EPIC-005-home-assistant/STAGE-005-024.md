# STAGE-005-024: Updates + integration-status widgets

## Status: Design (Not Started)

## Overview

Fill the next two sections of the HA panel: an updates-available widget (list of HA entities with
updates pending, from 005-008) and an integration-status widget (config-entry health from 005-010,
repairs from 005-011, persistent notifications from 005-012). Surfaces all the "HA issue" signals the
user asked for in one place.

**Vertical slice:** after this stage, the HA panel shows "41 updates available" (with a list) and an
integration-status section listing failed/retrying integrations, active repairs, and current HA
notifications — so the user sees everything HA itself is flagging.

**Spec references:** master spec §9.2 (HA panel: integration status), §8.5 (update availability),
EPIC-005 §"Stage decomposition" Wave F. Data: 005-021 endpoint (fed by collectors 008/010/011/012).

## Stage Flags

- Has Input Forms: [ ] No (display widgets).
- Touches secrets / production data: [ ] No (notification bodies only as the 021 endpoint exposes them
  to the authenticated panel — do not render sensitive content unsafely).
- Host-integration concerns: [ ] No — frontend widgets. Refinement 3a (Desktop + Mobile).

## Cross-cutting standards (from EPIC-005 §Cross-stage acceptance criteria)

- Strict TS; typed query hook; calm empty/loading/error states; notification content handled
  carefully (privacy).

## Concrete deliverables

### 1. Updates widget

A count + list of `update.*` entities with updates available (title, installed → latest version,
release link if present). Grouped/collapsible since there can be many (41 on the live install). Reuses
the 005-021 summary/detail data.

### 2. Integration-status widget

Three sub-sections (or a unified status list): config-entry issues (integrations in
`setup_error`/`setup_retry`, from 005-010), active repairs (domain + summary, from 005-011), and
current persistent notifications (title + safe rendering of message, from 005-012). Empty states are
"all integrations healthy / no repairs / no notifications".

### 3. Tests

Render with mocked data covering updates present/absent, a failed integration, active repairs, and a
notification → assert each sub-section renders; empty states; loading/error. Desktop + Mobile.

## Out of scope for this stage

- Entity-health + battery widgets (005-023).
- Embedded LogViewer (005-025).
- Acting on updates/repairs/notifications (read-only display; no install/dismiss actions in v1).

## Decisions in this stage

### Design phase will confirm

- **D-UPDATES-WIDGET-GROUPING** — collapse/group the (many) updates; show count + expandable list.
- **D-INTEGRATION-STATUS-LAYOUT** — three sub-sections vs a unified status list; severity ordering.
- **D-NOTIF-RENDER-SAFETY** — how notification message text is rendered safely (no HTML injection;
  privacy).

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-ISSUES-SURFACED-IN-PANEL** — the user-requested HA issue surfaces (updates, integration setup
  failures, repairs, notifications) are surfaced in this panel widget.
- **D-WIDGETS-SPLIT-SMALL** — this stage is updates + integration-status only (entity/battery is 023,
  logs is 025).
