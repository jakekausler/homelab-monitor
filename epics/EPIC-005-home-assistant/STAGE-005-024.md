# STAGE-005-024: Updates + integration-status widgets

## Status: Complete

## Overview

Fill the next two sections of the HA panel: an updates-available widget (list of HA entities with
updates pending, from 005-008) and an integration-status widget (config-entry health from 005-010,
repairs from 005-011, persistent notifications from 005-012). Surfaces all the "HA issue" signals the
user asked for in one place.

**Vertical slice:** after this stage, the HA panel shows "41 updates available" (with a list) and an
integration-status section listing failed/retrying integrations, active repairs, and current HA
notifications — so the user sees everything HA itself is flagging.

**Spec references:** master spec §9.2 (HA panel: integration status), §8.5 (update availability),
EPIC-005 §"Stage decomposition" Wave F. Data: 005-021 endpoint (fed by collectors 008/010/011/012).

## Stage Flags

- Has Input Forms: [ ] No (display widgets).
- Touches secrets / production data: [ ] No (notification bodies only as the 021 endpoint exposes them
  to the authenticated panel — do not render sensitive content unsafely).
- Host-integration concerns: [ ] No — frontend widgets. Refinement 3a (Desktop + Mobile).

## Cross-cutting standards (from EPIC-005 §Cross-stage acceptance criteria)

- Strict TS; typed query hook; calm empty/loading/error states; notification content handled
  carefully (privacy).

## Concrete deliverables

### 1. Updates widget

A count + list of `update.*` entities with updates available (title, installed → latest version,
release link if present). Grouped/collapsible since there can be many (41 on the live install). Reuses
the 005-021 summary/detail data.

### 2. Integration-status widget

Three sub-sections (or a unified status list): config-entry issues (integrations in
`setup_error`/`setup_retry`, from 005-010), active repairs (domain + summary, from 005-011), and
current persistent notifications (title + safe rendering of message, from 005-012). Empty states are
"all integrations healthy / no repairs / no notifications".

### 3. Tests

Render with mocked data covering updates present/absent, a failed integration, active repairs, and a
notification → assert each sub-section renders; empty states; loading/error. Desktop + Mobile.

## Out of scope for this stage

- Entity-health + battery widgets (005-023).
- Embedded LogViewer (005-025).
- Acting on updates/repairs/notifications (read-only display; no install/dismiss actions in v1).

## Decisions in this stage

### Design phase will confirm

- **D-UPDATES-WIDGET-GROUPING** — collapse/group the (many) updates; show count + expandable list.
- **D-INTEGRATION-STATUS-LAYOUT** — three sub-sections vs a unified status list; severity ordering.
- **D-NOTIF-RENDER-SAFETY** — how notification message text is rendered safely (no HTML injection;
  privacy).

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-ISSUES-SURFACED-IN-PANEL** — the user-requested HA issue surfaces (updates, integration setup
  failures, repairs, notifications) are surfaced in this panel widget.
- **D-WIDGETS-SPLIT-SMALL** — this stage is updates + integration-status only (entity/battery is 023,
  logs is 025).

### Design Notes (resolved 2026-06-13)

Brainstormed via brainstormer (Opus); decisions made by the agent per workflow §0.4 (no user pause). FRONTEND-only, 3a-only (Desktop + Mobile). Reuses the 023 `useHomeAssistantSummary()` hook + the 023 widget/banner conventions.

**★ BACKEND-CONTRACT GAP (same as 023):** The card asked for LISTS — an updates list (title, versions, release link), a failed-integration list, an active-repairs list, and a persistent-notification list (title + message). The verified 021 `/summary` contract returns COUNTS ONLY: `updates: {available, total}`, `config_entries: {loaded, error}`, `repairs: int`, `notifications: int`. No detail lists; notification BODIES are stored NOWHERE (012 privacy). So the card's list language outran the contract.

**D-024-SCOPE — CHOSEN: COUNTS-ONLY this stage.** Updates widget = available/total count tiles; Integration-status widget = config-entry loaded/error + repairs + notifications count tiles. DROP the 4 lists. Vertical slice "41 updates available" = the count; "(with a list)" = deferred detail. REJECTED adding detail endpoints inside this frontend stage (mixes backend+frontend; notification bodies need a live HA re-query + privacy review).

**★ DEFERRED LISTS — TRACKED (2026-06-13), per the "deferred work MUST become a tracked stage" rule:** Split by cost:
  - **Tier 1 (VM-queryable, cheap — extends the existing detail stages):** the updates list, config-entry (error/retry) list, and repairs list are added to **STAGE-005-027** (backend detail endpoints — new `/updates`, `/config-entries?filter=error`, `/repairs`) and **STAGE-005-028** (frontend drill-list widgets). 027/028 scope was extended 2026-06-13. (027 must VERIFY at Design that the needed fields exist as VM labels — D-DETAIL-VM-LABELS-PRESENT; any that don't drop to live-requery.)
  - **Tier 2 (expensive, privacy-sensitive — new stage):** the persistent-notification BODY list is **STAGE-005-029** (HA persistent-notification detail — live HA re-query NOT VM, privacy review, safe text render). Created 2026-06-13. EPIC-005 stage count 28 → 29.

**D-024-UPDATES-WIDGET-SHAPE — CHOSEN: 2-cell `<dl>`** (Available / Total), mirroring HaEntityHealthWidget. Available tinted AMBER (`text-amber-700 dark:text-amber-300`) when > 0 (pending updates = mild attention); Total untinted (context). `<EmptyState>All up to date</EmptyState>` when `available === 0`.

**D-024-INTEGRATION-STATUS-WIDGET-SHAPE — CHOSEN: single 4-cell `<dl>` (2×2 grid)** + affirmative EmptyState gate. When `config_entries.error === 0 && repairs === 0 && notifications === 0` → `<EmptyState>All integrations healthy</EmptyState>` (gate on the ACTIONABLE signals only; `loaded > 0` is normal, doesn't suppress healthy). Otherwise a 4-cell `<dl className="grid grid-cols-2 gap-2 text-sm">`: Loaded (untinted context), Errors (RED `text-red-700 dark:text-red-300` when > 0 — config-entry setup failures are the hard-failure signal), Repairs (AMBER when > 0), Notifications (AMBER when > 0). Keep `grid-cols-2` (2×2 reads cleanly on mobile; grid-cols-4 would crush).

**D-024-STATES — CHOSEN: mirror `HomeAssistantHealthTab` EXACTLY.** The Status tab calls `useHomeAssistantSummary()` ONCE; renders the SAME tab-level connection banner ABOVE the grid (isPending → muted Loading; 502 → yellow "metrics temporarily unavailable"; `ha_up===false` → yellow "offline" banner with counts STILL shown below; other errors → ErrorDisplay; banner `role="status"`/`aria-live`). Passes `data.updates` / `data.config_entries` / `data.repairs` / `data.notifications` to the two widgets. Per-widget EmptyStates ("All up to date"; "All integrations healthy") live inside the widgets.

**D-NOTIF-RENDER-SAFETY — EVAPORATES this stage.** 024 renders ONLY the notification COUNT — no body text crosses the boundary, so ZERO HTML-injection surface and ZERO privacy surface this stage. The injection/privacy/safe-render concern is inherent to the notification BODY list, deferred to STAGE-005-029 (which owns the live re-query + privacy review + escaped text-only rendering).

**Files this stage touches:** NEW `apps/ui/src/routes/integrations/HaUpdatesWidget.tsx` + `HaIntegrationStatusWidget.tsx` (pure presentational); EDIT `apps/ui/src/routes/integrations/HomeAssistantStatusTab.tsx` (hook + banner + widgets, replace placeholders); tests (widget plain-props tests + a `HomeAssistantStatusTab.test.tsx` hook-mocked, and REMOVE the Status-tab placeholder case from `HomeAssistantTabs.test.tsx` — same move 023 did for Health). REUSES the existing `useHomeAssistantSummary()` hook (no new hook). NO backend change, NO OpenAPI regen.
