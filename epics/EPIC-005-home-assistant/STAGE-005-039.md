# STAGE-005-039: Activate availability/cadence vmalert rules over registry-filtered series

## Status: Complete

## Overview

`deploy/vmalert/metrics/home-assistant.yaml` carries four commented OPT-IN exemplars (lines 142-236): `HaEntityUnavailable` (#1), `HaEntityStale` (#2), `HaAutomationIdle` (#16), `HaAutomationDisabled` (#17). They were DEMOTED from active defaults in STAGE-005-015 Refinement 3b because, without registry filtering, `homelab_ha_entity_available == 0` fired on ~50% of 1906 entities (~924 instances) — a failed, noisy rule — and disabled automations made `HaAutomationIdle` perpetually noisy. With STAGE-005-037 (registry exclusion → `homelab_ha_entity_available` carries only registry-visible entities) and STAGE-005-034 (disabled automations excluded from `last_triggered_seconds`), the underlying series are now clean enough to reconsider activation. This stage decides, PER RULE, whether it becomes an active default (with a justified threshold + `for:` duration) or stays opt-in, GROUNDED in the now-filtered real data measured during this stage's Design/Refinement 3b. It also ensures the metric series/labels let rules filter to enabled+registry-visible entities only (this is mostly settled by 037's D-REGISTRY-EXCLUDE-VS-LABEL decision; this stage CONSUMES that decision — if 037 DROPS excluded series, the blanket `== 0` rule is directly viable; if 037 LABELS them, this stage's expr must filter on the label).

**Hard dependencies:** must be higher-numbered than STAGE-005-037 (B) and STAGE-005-034 (A). Per the ordering rules, B is the keystone and A is the idle-data dependency.

**Depends on:** STAGE-005-037 (registry filter), STAGE-005-034 (idle filter)

**Per-rule decisions (LOCKED):**
- **D-ALERT-ACTIVATE-UNAVAILABLE → KEEP-OPT-IN.** The central call. 197 is still chronic (permanently-unavailable) integration/cloud noise (Frigate, smart plugs, Plex players, cloud) — the SAME category 015 named, only smaller. Decisive insight: a `for:` filters TRANSIENT flaps, NOT CHRONIC state — all 197 permanently-unavailable entities satisfy any `for:` window, so no `for:` tuning converts 197 chronic instances into an actionable set. 197 remains a mass-fire = failed rule. Per-entity scoping belongs in the 005-005 user-rule machinery (`deploy/vmalert/metrics-user/`). 037 lowered the number (924→197) but did not change the KIND of noise.
- **D-ALERT-STALE-BLANKET → KEEP-OPT-IN.** Structural blocker unchanged by 037: there is no `device_class` label for a blanket threshold; a door closed all day is legitimately stale. 525–865 firing at every tested threshold confirms a blanket rule is a mass-fire. Registry filtering is orthogonal to the device-class problem.
- **D-ALERT-AUTOMATION-IDLE → KEEP-OPT-IN.** The only borderline case. Even the most favorable threshold (30d → 9) cannot separate "truly dead" from "legitimately seasonal" (a freeze-warning automation idles all summer, >90d). The threshold shrinks the count without improving signal-to-noise. Idle-ness is irreducibly a per-automation user judgment → belongs in user-rules. (Severity note, were it ever activated: `info`, not `warning` — an idle automation is informational, not a fault.)
- **D-ALERT-AUTOMATION-DISABLED → KEEP-OPT-IN.** Disabling is almost always intentional; 35 deliberate states = pure noise as a default. 034 does not change the enabled metric.
- **D-ALERT-FILTER-FORM → BARE EXPR (resolved by 037's KEYSTONE).** 037 chose DROP (not LABEL) — registry-excluded entities emit NO `homelab_ha_entity_available` series, so the active-form expr would be the bare `homelab_ha_entity_available == 0` with NO `{registry_visible="1"}` filter. (Documented here per 037's instruction; moot while the rule stays opt-in. If anyone ever flips 037 to LABEL, this expr must gain `{registry_visible="1"}`.)
- **D-ALERT-FORDURATION → N/A** (no rule activated; for the record, `for:` is near-useless for monotonic chronic state — only the threshold is a lever, and the threshold can't separate the populations).
