# STAGE-005-014: vmalert-LOGS rules (automation/script + integration-setup failures + error-rate)

## Status: Finalize (Not Started)

## Overview

Ship the default vmalert-LOGS rules over the HA log stream (`service:"homeassistant"`), which already
flows through EPIC-004's vector→VictoriaLogs pipeline (verified: HA's docker stdout contains the full
home-assistant.log including automation/script/setup errors — NO new mount needed). Three rules:
automation/script execution failures (#15 fail side), integration-setup failures (#10 log path,
complementing the structured websocket signal in 005-010), and a general HA error-rate spike (#20).

**Vertical slice:** after this stage, a real `Error while executing automation ...` (verified present
on the live install) produces an alert via vmalert-logs → Alertmanager → ingestor → dispatcher.

**Spec references:** master spec §6.3 (LogsQL streams, HA logs), §4.3 (logs → vmalert(logs) → AM),
§17 (vmalert rules per epic codified under `deploy/vmalert/logs/`). EPIC-005 §"Signal-source matrix".
Scenarios #15 (fail), #10 (log path), #20.

## Concrete deliverables

### 1. `deploy/vmalert/logs/home-assistant.yaml` (`type: vlogs`)

A rule group (LogsQL, `type: vlogs`, matching the existing `deploy/vmalert/logs/system.yaml`
convention) with rules scoped to `service:"homeassistant"`:

- **Automation/script failure** — match `"Error while executing automation"` and `"Error executing
  script"` (verified real line shapes on the live install). Severity `error`. Counts/groups by the
  automation/script logger where feasible.
- **Integration-setup failure** — match `"Error ... setup of component"` / `"ConfigEntryNotReady"`
  (verified in `/api/error_log` and the docker stream). Severity `error`/`warning`. Complements the
  structured config-entry collector (005-010) — both may exist; document the relationship.
- **General HA error-rate spike** — a rate rule: HA logging far more `ERROR` lines than its baseline
  over a window. Severity `warning`. (#20 — catches "HA is unhealthy / noisy" broadly.)

### 2. ANSI-robustness + test mirror

The HA docker stdout wraps ERROR lines in ANSI color codes (`\x1b[31m...\x1b[0m`) — confirm LogsQL
substring matching still matches (it does; the substring is intact). Add a shortened-window test
mirror under the logs-test directory and an integration test planting the real line shapes.

## Out of scope for this stage

- The structured config-entry collector (005-010) — this is the log-pattern complement.
- Routing these alerts to HA push (005-017/018).

### Design phase will confirm

- **D-SETUP-RULE-VS-COLLECTOR** — how the log setup-failure rule and the 005-010 structured signal

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-FAILURES-VIA-LOGS** — automation/script + integration-setup failures are vmalert-logs rules over

- **D-REAL-SERVICE-LABEL** — the stream label is `homeassistant` (no hyphen).

### Design Notes (resolved 2026-06-12)

Brainstormer (Opus) consulted for the open decisions. CONFIG/YAML stage (LogsQL vmalert-logs rules), NOT
Python. Exemplars to mirror: `deploy/vmalert/logs/system.yaml` (event rule shape) + `deploy/vmalert/logs/
severity_escalation.yaml` (for:/explorer/target_kind:container pattern); test mirror `deploy/vmalert/
logs-test/system.yaml`. Main agent made the final calls.

- **D-LOGRULE-GROUPING → Option A: flat count, `for: 0m`, for BOTH event rules.** HA docker stdout is
  free-text `_msg`; vector exposes NO per-automation/per-component field, so `stats by (automation)` is
  impossible and `stats by (_msg)` is unbounded-cardinality noise. These are discrete EVENTS that often
  do NOT recur in the next eval window, so a `for:` debounce would MISS the single-shot event the card
  wants. Mirror `KernelOOM`/`CriticalLogLine`: `| stats by (host) count() as <c> | filter <c>:>0`,
  `for: 0m`. Match the STABLE inner substring (ANSI color codes wrap the ERROR line; the substring
  survives intact — do NOT anchor to line start). `service:="homeassistant"` exact-match (no hyphen —
  verified vector label).
  - Automation/script-fail (severity error): `_time:5m service:="homeassistant" ("Error while executing
    automation" OR "Error executing script") | stats by (host) count() as fail_count | filter
    fail_count:>0`, `for: 0m`.
  - Setup-fail (severity error): `_time:5m service:="homeassistant" ("Error during setup of component" OR
    "ConfigEntryNotReady") | stats by (host) count() as setup_fail_count | filter setup_fail_count:>0`,
    `for: 0m`.

- **D-ERRORRATE-BASELINE → Option B: fixed absolute threshold tuned at 3b, WITH `for:` persistence.**
  LogsQL genuinely CANNOT express relative-to-baseline (no rate()/avg_over_time-across-windows; vmalert
  evaluates a vlogs expr over ONE `_time:` range per eval — verified). A TRUE relative baseline already
  exists METRICS-side: `deploy/vmalert/metrics/error_rate_spike.yaml` (STAGE-004-037) does
  `> clamp_min(5 * avg_over_time(homelab_container_error_rate[7d:5m]), 10)` for ALL container services incl
  HA. So 014's log error-rate rule is an HONEST baseline-free COARSE safety net (fires even if the
  metric pipeline is down) — NOT anomaly detection. Rule: `_time:5m service:="homeassistant"
  severity:error | stats by (host) count() as error_count | filter error_count:>20`, `for: 10m`,
  `severity: warning`. N=20 is a PLACEHOLDER tuned at 3b against the real HA error-log baseline. A YAML
  header comment must state this explicitly + cross-reference error_rate_spike.yaml as the real-baseline
  home for #20.

- **D-SETUP-RULE-VS-COLLECTOR → Hybrid-A: keep both, differentiate by fingerprint; EVENT vs STATE.** The
  014 log setup-fail rule (severity error, `source_tool: vmalert-logs`, distinct alertname) catches the
  setup-failure EVENT at the moment HA logs it — including transient/retry failures that clear and never
  become a stuck state. The 015 metric rule (over `homelab_ha_config_entry_setup_error`,
  `source_tool: vmalert-metrics`) catches the persistent STATE of an entry stuck in error. Distinct
  alertname + source_tool → distinct fingerprints → no Alertmanager dedup collision. Severity = `error`
  (a ConfigEntryNotReady IS a real failure event, matching the card's primary guidance + the
  automation-fail rule). Document the EVENT-vs-STATE relationship in the `home-assistant.yaml` header
  comment (and 015 cross-references back when it ships).

- **Labels (all three rules):** `severity` (error/error/warning), `source_tool: vmalert-logs`,
  `target_kind: container` (mirrors severity_escalation.yaml's service-scoped docker-log convention; HA is
  a container; AM groups by alertname + target_kind), `integration: home_assistant`. Annotations:
  `summary` + `description` (with `{{ $labels.host }}` / `{{ $value }}`) + optional `explorer:` link to the
  log viewer (mirror severity_escalation.yaml).

- **Files:** CREATE `deploy/vmalert/logs/home-assistant.yaml` (the 3-rule group, type: vlogs) + CREATE
  `deploy/vmalert/logs-test/home-assistant.yaml` (shortened-window mirror: `_time:30s`, `interval: 5s`;
  event rules keep `filter ...:>0` — the short window makes a single planted line fire; spike rule lowers
  threshold to `>2` and drops/shortens `for:` so the integration test observes firing fast). "keep BOTH
  files in sync" header comment. Mirror `logs-test/system.yaml`.

- **3b is the load-bearing check:** with the prod rig + real HA, confirm a REAL `Error while executing
  automation ...` line (ANSI-color-wrapped) fires the automation rule via vmalert-logs → Alertmanager;
  MEASURE the real HA error-log baseline volume over 5m to TUNE the spike threshold N (placeholder 20);
  confirm the real `Error during setup of component` / `ConfigEntryNotReady` line shapes match the
  substrings. 3a plants the line shapes in the dev rig.

**Refinement Notes (3a+3b, 2026-06-12):**
- 3a (planted lines, prod rig): all THREE rules loaded + active in vmalert-logs (`home_assistant_logs` group, type vlogs) with correct labels (severity error/error/warning, source_tool: vmalert-logs, target_kind: container, integration: home_assistant), no rule-load errors. Plain-text automation/setup/error-rate exprs all matched planted HA-shaped lines (automation fail_count=1, setup_fail_count=2, error_count>20).
- 3a found a BUG: ANSI-wrapped HA ERROR lines did NOT match. Real HA wraps ERROR lines as `\x1b[31mError ...\x1b[0m`; the glued ANSI prefix before "Error" breaks LogsQL contiguous-substring matching, so `"Error while executing automation"` / `"Error during setup of component"` would silently MISS real (ANSI-wrapped) failures. The Design assumption "the inner substring survives ANSI codes" was WRONG.
- FIX (Opus-diagnosed, vector-side strip): added a new `[transforms.strip_ansi]` transform to `deploy/vector/vector.toml.template` (inserted between docker_severity_extract and drop_noise; rewired drop_noise.inputs → strip_ansi) calling `strip_ansi_escape_codes(to_string(.message) ?? "")`, docker-only via `exists(.label)`, AFTER severity extraction (which already tolerates leading ANSI). Strips ANSI from docker `.message` before VL ingest so ALL substring rules match + the log viewer is cleaner. Updated `apps/monitor/tests/test_vector_template.py` (drop_noise inputs assertion + a new `test_strip_ansi_transform_present` lock test). Fixture `deploy/compose/test-fixtures/vector.toml` needed no edit (it has no docker_severity_extract block; the integration planter bypasses vector). A first VRL had an E651 (unnecessary `?? .message` coalescing on the non-fallible strip fn) caught by `vector validate` — corrected to drop the trailing coalesce.
- 3b (real HA, load-bearing): real HA logs confirmed flowing as `service:"homeassistant"` (no hyphen, ~374 lines/hr). Measured real HA ERROR baseline ~18-22 errors/5m (normal blink-integration retry chatter) — so the original `HaErrorRateSpike` N=20 would FALSE-FIRE constantly. TUNED N to 50 (≈2.5x headroom over the measured baseline) in the prod rule + comments (test mirror stays at >2). Real `ConfigEntryNotReady` setup failures exist and `HaIntegrationSetupFailure` was observed `state: firing` on real HA data; no real automation-execution errors were present in the 24h window (planted test covers the match logic).
- Re-validation after fix: `make verify` GREEN (4695 tests, 100% coverage; new strip_ansi test + updated drop_noise test pass); `vector validate` confirms the corrected strip_ansi VRL compiles; re-rendered running vector loaded the strip_ansi transform; transform proof: `\x1b[31mError while executing automation\x1b[0m ...` → clean `Error while executing automation ...` (ANSI fully stripped); real HA container has compose labels so `exists(.label)` fires for it. vmalert dry-run parses the N=50 rule cleanly.
- Files changed in Refinement: `deploy/vector/vector.toml.template`, `apps/monitor/tests/test_vector_template.py`, `deploy/vmalert/logs/home-assistant.yaml` (N=50 + comments). The two new rule YAMLs (logs/ + logs-test/) are the Build deliverable.
