# STAGE-005-001: HA REST client + secret + lifespan wiring

## Status: Complete

## Overview

Replace the empty `HomeAssistantClient` scaffold with a real REST client and wire it into the
collector context so every HA collector can reach Home Assistant. This is the first foundation stage
of the exemplar integration epic — it establishes how an integration's API client is constructed,
credentialed via the secret store, and injected into `CollectorContext`.

**Vertical slice:** after this stage, a smoke test (and the collector status API) shows the monitor
successfully reaching `http://192.168.2.148:8123/api/config` with the long-lived bearer token from
the secret store; `ctx.ha` is a live REST client instead of `None`.

**Spec references:** master spec §2 Q10 (HA connection: long-lived bearer over LAN), §6.2 (HA-derived
metrics from REST), §5.2 (`CollectorContext.ha`). EPIC-005 §"HA client = REST + websocket".

## Cross-cutting standards (from EPIC-005 §Cross-stage acceptance criteria)

- Connection failures handled gracefully — an HA 5xx/timeout never propagates as our 5xx; the client

## Concrete deliverables

### 1. Real `HomeAssistantClient` (REST)

Define the real client implementing the `HomeAssistantClient` Protocol that today is an EMPTY
scaffold at `apps/monitor/homelab_monitor/kernel/plugins/io.py` (verified: the Protocol is empty, so
`isinstance` is currently a tautology — Design must add real methods). REST methods needed by the
Wave-B collectors:

- `get_config()` → HA `/api/config` (version, timezone) — used by the smoke test.
- `get_states()` → HA `/api/states` (list of entity state objects) — used by entity-availability,
  battery, updates, automation-run-cadence.
- `get_error_log()` → HA `/api/error_log` (plain text) — available for log-based checks.
- `call_service(domain, service, data)` → HA `/api/services/<domain>/<service>` — used later by
  HAPushChannel (notify) and push-back-to-HA.

Implementation lives in a new module under the integration bundle area (Design picks the exact path;
likely `apps/monitor/homelab_monitor/kernel/ha/client.py` or within the integration bundle — keep the
Protocol in `io.py`, the impl separate). Uses the shared `httpx.AsyncClient` from the context. Bearer
auth header from the resolved secret. All HA error responses mapped to a typed client error.

### 2. Secret + config wiring

- A secret-store key (`ha_token`) holds the long-lived bearer; the HA base URL (`ha_url`,
  default `http://192.168.2.148:8123`) is config (per-plugin config / app config — Design decides,
  consistent with how other collectors read config today via `getattr(ctx.config, ...)`).
- Confirm the `hm secrets set ha_token` path works (or document the dashboard secret editor path).

### 3. Lifespan wiring (`ctx.ha`)

In `apps/monitor/homelab_monitor/kernel/api/lifespan.py`, the `ctx_factory` currently passes
`ha=None`. Construct the real client once at startup (resolving `ha_token` from the TTL secret
resolver + `ha_url` from config) and inject it as `ctx.ha`. If the token/URL is absent or the smoke
connect fails, log a clear warning and leave `ha=None` (HA collectors then no-op/skip rather than
crash the process) — graceful-degradation requirement.

### 4. Smoke connectivity test

A test (unit against a mocked HA + an integration/rig check against real HA in 3b) that asserts
`get_config()` returns the HA version and that a bad/missing token yields a typed auth error, not an
unhandled exception. Assert the token never appears in logs.

## Design Notes (2026-06-10)

Verified all cited anchors against current code (all hold): `HomeAssistantClient` Protocol empty at `kernel/plugins/io.py:116-123`; `CollectorContext.ha: HomeAssistantClient | None` with shared concrete `http: httpx.AsyncClient`; `ctx_factory` builds `ha=None` at `kernel/api/lifespan.py:621-635`; shared `httpx.AsyncClient` created once at `lifespan.py:577` (timeout 10.0, connect 5.0, 50-conn pool); `SyncSecretsResolver.get(name) -> str | None` is SYNC; TTL via `TtlCachingSecretsResolver(ttl_seconds=60)` with `ttl_resolver.current()` snapshot; config read via `getattr(ctx.config, "field", default)`; existing typed-error exemplar `FetchError` discriminated union in `kernel/docker/registry_digest_client.py`; `hm secrets set NAME --from-stdin`. Confirmed: NO app-level `BaseSettings`/`Settings` object exists — app-global config is module-level `os.environ.get("HOMELAB_MONITOR_*", default)` with typed dataclass loaders in `kernel/config.py` (e.g. `load_disk_budget_config`).

### D-HA-CLIENT-LOCATION → CHOSEN: new kernel package `kernel/ha/`

- **Chosen:** `kernel/ha/client.py` holds the concrete `HomeAssistantRestClient`; `kernel/ha/errors.py` holds the `HaError` union; the `HomeAssistantClient` Protocol stays in `kernel/plugins/io.py`. The websocket client (002) lands beside it as `kernel/ha/websocket.py`.
- **Rationale:** The client is shared infrastructure consumed by THREE lifespan-tier subsystems — per-collector ctx (`ctx.ha`), the background websocket task (002), and the `HAPushChannel` dispatch channel (017). Infrastructure shared across tiers belongs in the kernel beside `VictoriaLogsWriter`, mirroring `kernel/metrics/`, `kernel/logs/`, `kernel/secrets/`.
- **Rejected — put client in the integration bundle now:** the bundle dir doesn't exist until 003, and a kernel dispatch channel (017) / background ws task (002) importing from a plugin bundle would invert the dependency direction (kernel → plugin). The bundle (003) organizes only the COLLECTORS; collectors receive the client via `ctx.ha`, they don't own it.
- **Rejected — flat `kernel/ha_client.py`:** websocket + errors + config have no natural home; a package groups them cleanly.
- **Exemplar note for copiers:** shared client → kernel; collectors → bundle; wire via context. 003 reorganizes collectors only, NOT the client.

### D-HA-URL-CONFIG → CHOSEN: `HOMELAB_MONITOR_HA_URL` env var + typed `load_ha_config()` in `kernel/config.py`

- **Chosen:** Add `load_ha_config() -> HaConfig(base_url: str)` to `kernel/config.py` reading `HOMELAB_MONITOR_HA_URL` (default `http://192.168.2.148:8123`), matching the existing `load_disk_budget_config` idiom. Lifespan calls it once at startup. Secret name is `ha_token` (locked).
- **Token resolution at startup:** lifespan resolves the token via `ttl_resolver.current().get("ha_token")` — the UNFILTERED snapshot (NOT a per-collector `.filtered(...)` view). `ttl_resolver.refresh_now()` is already awaited earlier in lifespan, so the snapshot is populated by the construction site. The client holds a `token_provider` callable (`lambda: ttl_resolver.current().get("ha_token")`) rather than a baked-in token string — this is the seam for secret rotation + post-startup recovery and keeps the token out of any stored field.
- **Rejected — per-collector `CollectorConfig` field:** the client is constructed ONCE in lifespan (not in a collector) and is shared by non-collector consumers (ws task, dispatch channel); per-collector config has no construction-time context.
- **Rejected — YAML config block:** heavier than one URL needs; collector YAML subclass-loading is explicitly deferred elsewhere.

### D-HA-DEGRADE → CHOSEN: construct-when-token-present, no startup smoke-connect, per-tick re-resolution

- **No blocking startup smoke-connect.** HA being down at monitor boot must never block the monitor (it watches many things). Reachability is validated lazily by the first real call.
- **Token present → always construct** the REST client (deferred/lazy connect). Runtime failures (HA 5xx, timeout, auth) return a typed `HaError`; the collector converts to a failed run; the existing FailureBudget handles quarantine + recovery. No HA-specific degradation code beyond returning the typed error.
- **Token absent → `ha=None`.** HA collectors no-op quietly, emitting a `homelab_collector_run_*` self-metric carrying `reason="no_token"` and a single warn log (token value NEVER logged).
- **Post-startup recovery without restart:** because the client uses a `token_provider` callable re-reading `ttl_resolver.current()`, and `ctx_factory` runs per tick, a token added after startup is picked up within one TTL window (~60s) — no restart required.
- **Rejected — hard-fail startup if HA unreachable:** violates the locked "monitor must boot when HA is down" requirement.

### Client method contract (return-not-raise)

The `HomeAssistantClient` Protocol gains four async methods, each returning a success result OR a typed `HaError` (mirroring the `FetchError` return-not-raise pattern — guarantees HA 5xx/timeout never propagate as our 5xx):

- `get_config()` → HA `/api/config` (version, timezone) — used by the smoke test.
- `get_states()` → HA `/api/states` (list of entity state objects) — consumed by 006/007/008/009.
- `get_error_log()` → HA `/api/error_log` (plain text) — available, no early caller.
- `call_service(domain, service, data=None)` → HA `/api/services/<domain>/<service>` — consumed by HAPushChannel (017) and push-back (020).

`fire_event` is DEFERRED to STAGE-005-020 (push-back firer decides reuse-`call_service` vs new method) — NOT added now.

`HaError = dataclass(frozen=True, slots=True)` with `reason: Literal["unreachable","timeout","auth","http_error","bad_response"]`, `message: str`, `status: int | None = None`. The `message` field must never contain the token (asserted in tests).

### Smoke connectivity test (deliverable #4) clarification

Since there is no startup smoke-connect, the "smoke test" is: (1) a UNIT test against a mocked/faked HA asserting `get_config()` parses the HA version, that a bad/missing token yields `HaError(reason="auth")` (not an unhandled exception), and that the token never appears in logs/error messages; (2) a 3b RIG integration check hitting the real HA at `192.168.2.148:8123` with the real token (Refinement).

### Construction flow (lifespan, ~line 580 right after `http_client`)

```python
ha_config = load_ha_config()                          # HOMELAB_MONITOR_HA_URL + default
ha_token = ttl_resolver.current().get("ha_token")     # sync, UNFILTERED snapshot -> str | None
ha_client = (
    HomeAssistantRestClient(
        base_url=ha_config.base_url,
        http=http_client,                             # reuse the shared pool — do NOT create a 2nd
        token_provider=lambda: ttl_resolver.current().get("ha_token"),
    )
    if ha_token is not None else None
)
```
Then `ctx_factory` passes `ha=ha_client` (re-checking token presence per tick for no-restart recovery). The same `ha_client` instance is later handed to the 002 websocket builder and the 017 HAPushChannel.
