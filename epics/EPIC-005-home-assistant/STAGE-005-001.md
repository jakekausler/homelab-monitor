# STAGE-005-001: HA REST client + secret + lifespan wiring

## Status: Design (Not Started)

## Overview

Replace the empty `HomeAssistantClient` scaffold with a real REST client and wire it into the
collector context so every HA collector can reach Home Assistant. This is the first foundation stage
of the exemplar integration epic — it establishes how an integration's API client is constructed,
credentialed via the secret store, and injected into `CollectorContext`.

**Vertical slice:** after this stage, a smoke test (and the collector status API) shows the monitor
successfully reaching `http://192.168.2.148:8123/api/config` with the long-lived bearer token from
the secret store; `ctx.ha` is a live REST client instead of `None`.

**Spec references:** master spec §2 Q10 (HA connection: long-lived bearer over LAN), §6.2 (HA-derived
metrics from REST), §5.2 (`CollectorContext.ha`). EPIC-005 §"HA client = REST + websocket".

## Cross-cutting standards (from EPIC-005 §Cross-stage acceptance criteria)

- HA token never logged (assert in tests).
- Connection failures handled gracefully — an HA 5xx/timeout never propagates as our 5xx; the client
  surfaces a typed error the collector turns into a failed run (failure budget takes over).
- All internal timestamps UTC.

## Concrete deliverables

### 1. Real `HomeAssistantClient` (REST)

Define the real client implementing the `HomeAssistantClient` Protocol that today is an EMPTY
scaffold at `apps/monitor/homelab_monitor/kernel/plugins/io.py` (verified: the Protocol is empty, so
`isinstance` is currently a tautology — Design must add real methods). REST methods needed by the
Wave-B collectors:

- `get_config()` → HA `/api/config` (version, timezone) — used by the smoke test.
- `get_states()` → HA `/api/states` (list of entity state objects) — used by entity-availability,
  battery, updates, automation-run-cadence.
- `get_error_log()` → HA `/api/error_log` (plain text) — available for log-based checks.
- `call_service(domain, service, data)` → HA `/api/services/<domain>/<service>` — used later by
  HAPushChannel (notify) and push-back-to-HA.

Implementation lives in a new module under the integration bundle area (Design picks the exact path;
likely `apps/monitor/homelab_monitor/kernel/ha/client.py` or within the integration bundle — keep the
Protocol in `io.py`, the impl separate). Uses the shared `httpx.AsyncClient` from the context. Bearer
auth header from the resolved secret. All HA error responses mapped to a typed client error.

### 2. Secret + config wiring

- A secret-store key (`ha_token`) holds the long-lived bearer; the HA base URL (`ha_url`,
  default `http://192.168.2.148:8123`) is config (per-plugin config / app config — Design decides,
  consistent with how other collectors read config today via `getattr(ctx.config, ...)`).
- Confirm the `hm secrets set ha_token` path works (or document the dashboard secret editor path).

### 3. Lifespan wiring (`ctx.ha`)

In `apps/monitor/homelab_monitor/kernel/api/lifespan.py`, the `ctx_factory` currently passes
`ha=None`. Construct the real client once at startup (resolving `ha_token` from the TTL secret
resolver + `ha_url` from config) and inject it as `ctx.ha`. If the token/URL is absent or the smoke
connect fails, log a clear warning and leave `ha=None` (HA collectors then no-op/skip rather than
crash the process) — graceful-degradation requirement.

### 4. Smoke connectivity test

A test (unit against a mocked HA + an integration/rig check against real HA in 3b) that asserts
`get_config()` returns the HA version and that a bad/missing token yields a typed auth error, not an
unhandled exception. Assert the token never appears in logs.

## Out of scope for this stage

- The websocket client (STAGE-005-002).
- Any collector (Wave B).
- The integration bundle directory layout (STAGE-005-003) — this stage may place the client in the
  kernel; the bundle skeleton stage organizes the collectors.

## Decisions in this stage

### Design phase will confirm

- **D-HA-CLIENT-LOCATION** — exact module path for the client impl (kernel `ha/` vs integration
  bundle) while keeping the Protocol in `io.py`.
- **D-HA-URL-CONFIG** — where `ha_url` lives (per-plugin config vs app config) and the secret name for
  the token.
- **D-HA-DEGRADE** — exact behavior when token/URL missing or smoke-connect fails (leave `ha=None`;
  collectors skip; a self-metric/log records the reason).

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-HA-REST-FIRST** — the REST client is built first and separately from websocket (STAGE-005-002);
  REST covers entity/state/updates/automation-cadence/history; websocket covers only what REST can't
  reach.
- **D-HA-TOKEN-SECRET** — the long-lived bearer is stored in the secret store (`ha_token`), resolved
  via `ctx.secrets`/the TTL resolver, never in config files or logs.
