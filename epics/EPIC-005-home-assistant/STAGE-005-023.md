# STAGE-005-023: Entity-health + battery widgets

## Status: Complete

## Overview

Fill the first two sections of the HA panel shell (005-022) with live widgets: an entity-health grid
(total / available / unavailable / stale counts + a drill list) and a battery summary (devices low /
critical + a list). Consumes the `/api/integrations/home-assistant/summary` (+ detail) endpoint from
005-021.

**Vertical slice:** after this stage, the HA panel shows real entity-health counts and a battery
summary that updates from live data, letting the user see "12 entities unavailable, 2 batteries below
10%" at a glance.

**Spec references:** master spec §9.2 (HA panel shows entity health), EPIC-005 §"Stage decomposition"
Wave F. Data: 005-021 endpoint; UI pattern: the Docker integration list/widgets.

## Concrete deliverables

### 1. Data hook

A `useHomeAssistantSummary()` (and entity-detail hook if 021 split it out) in
`apps/ui/src/api/home_assistant.ts`, wrapping `GET /api/integrations/home-assistant/summary` via the
generated client (mirrors `apps/ui/src/api/docker.ts` hooks).

### 2. Entity-health grid widget

Counts (total / available / unavailable / stale) as stat tiles + a drill list of unavailable/stale
entities (entity_id, domain, last-changed age). Reuses the project's stat-tile / table primitives.

### 3. Battery summary widget

A summary (count low <20% / critical <10%) + a list of low/critical batteries (device, level). Reuses
severity tinting where sensible.

### 4. Tests

Render with mocked summary data → assert counts + lists render; empty state (no unavailable / no low
batteries) renders calmly; loading + error states. Desktop + Mobile layout.

### Design Notes (resolved 2026-06-12)

Brainstormed via brainstormer (Opus); decisions made by the agent per workflow §0.4 (no user pause). FRONTEND-only, 3a-only (Desktop + Mobile). Backend contract (021 `/summary`) verified.

**★ BACKEND-CONTRACT GAP discovered during Design verification (load-bearing):** The card asked for an entity `stale` count and drill-LISTS (unavailable-entity list, low/critical-battery list). The verified 021 `/summary` contract returns COUNTS ONLY: `entities: {total, available, unavailable}` (NO `stale`) and `battery: {low, critical}` — no entity-level or device-level detail, no list endpoint. 021's D-PANEL-ENDPOINT-SPLIT deliberately shipped a single `/summary` and deferred detail endpoints. `stale` has NO data source at any layer (no blanket stale threshold by design; ~50% of ~1906 entities are chronically unavailable). So the card's `stale` + drill-list language outran the 021 contract.

**D-023-SCOPE — CHOSEN: COUNTS-ONLY this stage.** Build the entity-health widget (total/available/unavailable count tiles) and the battery widget (low/critical count tiles) consuming `/summary`. DROP `stale` (no source) and the drill-lists (no detail endpoint). The vertical slice ("see N unavailable / M low at a glance") is fully satisfied by counts. REJECTED: adding a VM-backed or HA-re-query detail endpoint INSIDE this stage — that mixes backend+frontend, and the detail endpoint is a DEPENDENCY (per the sequential-dependency rule + keep-stages-small D-WIDGETS-SPLIT-SMALL), not a frontend bolt-on.

**★ DEFERRED FOLLOW-UP WORK (recorded, NOT abandoned):** The dropped drill-lists require NEW future work, to be decomposed as later stages after EPIC-005's current Wave F:
  1. A BACKEND stage: a paginated VM-backed detail endpoint (e.g. `GET /api/integrations/home-assistant/entities` returning unavailable entity_id/domain/last_changed_age, and a battery-device list device/level) — the deferred "021 split" that D-PANEL-ENDPOINT-SPLIT named for "023/024 if widgets need them" (→ STAGE-005-027).
  2. A FRONTEND stage: drill-list widgets consuming those detail endpoints (added to the Health tab below the count tiles) (→ STAGE-005-028).
  3. `stale` stays dropped until a per-entity staleness-rule surface exists (out of current EPIC-005 scope).
  **★ TRACKED (2026-06-13):** This deferred work is now captured as concrete stages — **STAGE-005-027** (HA entity + battery detail endpoints — the backend drill-down data, item 1) and **STAGE-005-028** (entity + battery drill-down widgets — the frontend lists, item 2). They are DEPENDENT (higher-numbered) stages appended to Wave F (epic count 26 → 28). `stale` (item 3) stays dropped until a per-entity staleness-rule surface exists (out of current EPIC-005 scope; not yet a stage).

**D-ENTITY-WIDGET-SHAPE — CHOSEN: 3-cell `<dl>` stat grid** (Total / Available / Unavailable), reusing the codebase's inline `<dl>` + `tabular-nums` + muted-uppercase-`<dt>` convention (ContainerOverviewTab). The Unavailable `<dd>` is tinted AMBER (`text-amber-700 dark:text-amber-300`) when > 0, neutral at 0. Amber not red — chronic unavailability would make red scream constantly; red is reserved for battery-critical.

**D-BATTERY-WIDGET-SHAPE — CHOSEN: 2-cell `<dl>`** (Low / Critical), mirroring the entity widget. Critical tinted RED (`text-red-700 dark:text-red-300`) when > 0; Low tinted amber when > 0; both neutral at 0. When `low === 0 && critical === 0`, render `<EmptyState>All batteries healthy</EmptyState>` (affirmative calm state).

**D-023-STATES — CHOSEN: tab-level connection banner + per-widget empty state.** Connection state is surfaced ONCE at the Health-tab level (above both PanelSections): `isPending` → muted "Loading…"; 502 VM-down → yellow `border-yellow-200 bg-yellow-50` banner "Home Assistant metrics temporarily unavailable" (matching the ContainerList 503-banner convention, NOT ErrorDisplay — VM-down is transient upstream); `ha_up === false` → yellow banner "Home Assistant offline" + `last_seen` age, with the count widgets STILL rendered below (counts are last-known VM samples); genuinely unexpected errors → `<ErrorDisplay>`. Banner gets `role="status"`/`aria-live="polite"`. Per-widget surfaces limited to the battery "all healthy" EmptyState.

**D-023-HOOK — CHOSEN: shared `useHomeAssistantSummary()`** in NEW `apps/ui/src/api/home_assistant.ts`, mirroring `docker.ts::useListContainers` (`useQuery` + `apiClient.GET('/api/integrations/home-assistant/summary', {})` + `unwrap<HaSummaryResponse>` + `refetchInterval: 30_000` + queryKey `['integrations','home-assistant','summary']`). The HEALTH TAB calls the hook ONCE and passes `entities`/`battery`/connection-state down to two PURE presentational widget components (so the banner renders once, not per-widget, and widgets are testable with plain props). Stage 024 (Status tab) REUSES this same hook (no new hook).

**Files this stage touches:** NEW `apps/ui/src/api/home_assistant.ts` (the hook); NEW widget components (EntityHealthWidget + BatteryWidget — presentational) under `apps/ui/src/routes/integrations/`; EDIT `apps/ui/src/routes/integrations/HomeAssistantHealthTab.tsx` (call the hook, render banner + widgets, replace placeholders); tests. NO backend change, NO OpenAPI regen.
