# STAGE-005-011: Repairs collector (websocket)

## Status: Design (Not Started)

## Overview

Surface HA's Repairs dashboard (the issue registry: deprecated config, reauth-needed, restart-required,
validation failures) via the websocket. Scenario #11. REST does not expose repairs
(`/api/repairs/*` → 404) and the `.storage/repairs.issue_registry` file is stale (only rewritten on
restart/config-change — 3 days old on inspection), so the websocket `repairs/list_issues` command is
the only live source.

**Vertical slice:** after this stage, `homelab_ha_repair_issue{domain,issue_id,severity}` reflects the
current active repair issues (43 on the live install at inspection time), updating as issues
appear/clear, feeding the integration-status widget (005-024).

**Spec references:** master spec §2 Q10 (integration health), EPIC-005 §"Signal-source matrix"
(repairs via websocket). Scenario #11.

## Concrete deliverables

### 1. Repairs subscriber

Using the websocket client, issue `repairs/list_issues` and subscribe to repair-registry updates
(Design confirms the exact WS command + whether a periodic re-list is needed). Maintain the current
set of active issues; emit `homelab_ha_repair_issue{domain,issue_id}` = 1 for each active issue (with
`severity` and `is_fixable`/`ignored` flags as labels or panel-only). Honor `dismissed`/`ignored` so
dismissed issues don't alert.

### 2. WS → metric bridge

Same event-driven-to-metric concern as 005-010: periodic snapshot-emit of current issues vs
emit-on-update. Apply the cap (a flood of repair issues shouldn't blow cardinality).

### 3. Tests

Fake WS repairs list with active + dismissed issues → assert only active emit; an issue clears →
metric drops; reconnect recovers state. 3b: real repairs list.

## Out of scope for this stage

- Auto-fixing repairs (out of scope; that's the far-future auto-fix epic, EPIC-009).
- The repair-issue ALERT RULE (if any) — Design of 005-015 decides whether active repairs warrant an
  alert or are panel-only (many repairs are low-severity/informational).
- Config-entry state (005-010) and notifications (005-012).

## Decisions in this stage

### Design phase will confirm

- **D-REPAIRS-ALERT-VS-PANEL** — whether repairs produce alerts (and at what severity) or are
  panel-only (likely panel-only + maybe a rule for `severity: critical` repairs).
- **D-REPAIRS-METRIC-SHAPE** — per-issue gauge + labels vs a count-by-domain.
