# STAGE-005-011: Repairs collector (websocket)

## Status: Complete

## Overview

Surface HA's Repairs dashboard (the issue registry: deprecated config, reauth-needed, restart-required,
validation failures) via the websocket. Scenario #11. REST does not expose repairs
(`/api/repairs/*` → 404) and the `.storage/repairs.issue_registry` file is stale (only rewritten on
restart/config-change — 3 days old on inspection), so the websocket `repairs/list_issues` command is
the only live source.

**Vertical slice:** after this stage, `homelab_ha_repair_issue{domain,issue_id,severity}` reflects the
current active repair issues (43 on the live install at inspection time), updating as issues
appear/clear, feeding the integration-status widget (005-024).

**Spec references:** master spec §2 Q10 (integration health), EPIC-005 §"Signal-source matrix"
(repairs via websocket). Scenario #11.

## Concrete deliverables

### 1. Repairs subscriber

Using the websocket client, issue `repairs/list_issues` and subscribe to repair-registry updates
(Design confirms the exact WS command + whether a periodic re-list is needed). Maintain the current
set of active issues; emit `homelab_ha_repair_issue{domain,issue_id}` = 1 for each active issue (with
`severity` and `is_fixable`/`ignored` flags as labels or panel-only). Honor `dismissed`/`ignored` so
dismissed issues don't alert.

### 2. WS → metric bridge

Same event-driven-to-metric concern as 005-010: periodic snapshot-emit of current issues vs
emit-on-update. Apply the cap (a flood of repair issues shouldn't blow cardinality).

### 3. Tests

Fake WS repairs list with active + dismissed issues → assert only active emit; an issue clears →
metric drops; reconnect recovers state. 3b: real repairs list.

### Design Notes (resolved 2026-06-12)

Brainstormer (Opus) consulted for D-REPAIRS-METRIC-SHAPE and D-REPAIRS-DISMISSED-HANDLING. Near-mirror of
STAGE-005-010 (config-entry WS collector). Main agent made the final calls.

- **D-REPAIRS-WS-COMMAND → A1 snapshot (inherited from 010).** Each tick: guard `_ws is None` / `not
  _ws.connected` → return ok=False (stale); else `await self._ws.send_command("repairs/list_issues")`.
  Defensive parse handles bare list / `{"issues":[...]}` dict / empty (an `_extract_issues` analogue of
  010's `_extract_entries`). NO subscription — a 30-60s tick can't deliver sub-tick freshness and a
  snapshot RPC is needed anyway. Re-list every tick. WS client's list-passthrough (`send_command` returns
  `dict | list | HaError`) is already fixed (010).

- **D-REPAIRS-METRIC-SHAPE → Option A (single per-issue 1/0 gauge).**
  `homelab_ha_repair_issue{domain,issue_id,severity}` = 1.0 for each active, non-ignored issue.
  - `severity` is HA's `IssueSeverity` enum (warning|error|critical — low-churn) → acceptable rule-friendly
    label; a future `severity="critical"` repairs rule (015's call) is free.
  - `issue_id` is HA's stable per-issue slug (low-churn) → stable identity label.
  - Panels derive count-by-domain via `count by(domain)` over the per-issue series — no aggregate family
    needed.
  - REJECTED Option B (separate `is_fixable` family): speculative; no verified consumer (015 panel-only,
    024 re-queries HA). REJECTED Option C (count-by-domain aggregate): breaks the 010 per-entity mirror and
    is derivable from A.
  - `is_fixable`, `translation_key`, `translation_placeholders`, descriptions, `learn_more_url`,
    `breaks_in_ha_version` → PANEL-ONLY (metric hygiene). Detail reaches the 021/024 panel via HA re-query,
    never via metric labels or logs.

- **D-REPAIRS-DISMISSED-HANDLING → Option A (exclude dismissed + inactive).** Emit a series ONLY when
  `active is True and ignored is not True`. Skip via a `None`-returning label helper (mirror 010's
  `_entry_labels` skip-via-None). Defensive defaults on a malformed/missing field: missing `active` →
  emit; missing `ignored` → treat as not-ignored. "Series exists ⇒ live actionable repair." Dismissed
  issues go absent → stale-series TTL cleans them up. Avoids the dismissed-alerting footgun of an
  `ignored="true"` label. Dismissed/detail counts for the panel come from 021's own HA re-query.

- **D-REPAIRS-ALERT-VS-PANEL → panel-only** for this stage. The repair-issue alert RULE itself is 015's
  scope (verified: 015 omits repairs from its metric-rule set; may add a future critical-repairs rule). The
  metric contract above supports that rule for free.

- **Cardinality:** global default cap (500) is ample (repairs are tens at most); NO per-family override in
  `_DEFAULT_CARDINALITY_FAMILIES`.

- **3b is the load-bearing check (WS collectors):** verify the REAL `repairs/list_issues` response SHAPE
  (bare list vs `{"issues":[...]}` — the WS client now passes lists through), the real issue count, the
  real `severity` enum values, and that `active`/`ignored` fields exist as assumed. Capture the raw frame
  shape, not just the metric count (010's green unit suite hid a real-payload bug that only 3b exposed).
