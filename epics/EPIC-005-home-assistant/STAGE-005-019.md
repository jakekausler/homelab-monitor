# STAGE-005-019: HA webhook ingester

## Status: Design (Not Started)

## Overview

The inbound half of bidirectional integration: an endpoint where HA automations can POST arbitrary
"something interesting happened" events into the monitor, authenticated by a scoped API token.
Scenario #23. Reuses the exact scoped-endpoint + pydantic-payload + audit-log pattern the alert
ingestor and heartbeat receiver already use.

**Vertical slice:** after this stage, an HA automation can `POST /api/integrations/ha/event` with a
token scoped `ha:event:write`, the payload is validated and recorded to `audit_log`, and a 202 is
returned — giving HA a push channel INTO the monitor.

**Spec references:** master spec §2 Q10 (custom HA-fired webhooks accepted as a push source), §3.1
(alert ingestor pattern), §7.2 (API token scopes, e.g. heartbeat `heartbeat:write`). EPIC-005
§"Bidirectional webhook". Scenario #23.

## Cross-cutting standards (from EPIC-005 §Cross-stage acceptance criteria)

- The endpoint follows the existing scoped-endpoint pattern (`require_user_or_token({Scope.X})` from
  `kernel/api/dependencies.py`); a token lacking the scope gets 403.
- Every accepted event writes an `audit_log` row; payload validated by a pydantic model (`extra` policy
  per the existing convention).

## Concrete deliverables

### 1. New scope `Scope.HA_EVENT_WRITE`

Add `HA_EVENT_WRITE = "ha:event:write"` to the `Scope` enum in `kernel/auth/scopes.py` (additive
change, same as the existing `heartbeat:write` / `alerts:ingest:write`).

### 2. Endpoint `POST /api/integrations/ha/event`

A new router (e.g. `kernel/api/routers/integrations_ha.py`, `APIRouter(prefix="/integrations/ha")`,
registered in `app.py` like `docker.router`). Guarded by
`Depends(require_user_or_token({Scope.HA_EVENT_WRITE}))`. Accepts a pydantic `HAEventPayload`
(event_type + arbitrary data + optional title/severity hint — Design defines the schema). Records the
event to `audit_log` (`who="home_assistant"`, `what="ha_event.<type>"`, `after=<payload>`). Returns 202.

### 3. Optional: route into the alert pipeline

Design decides whether HA-fired events are PURELY informational (audit_log only) or can optionally
become alerts (forwarded to the ingestor/dispatcher when the payload requests it, e.g. a severity
hint). Keep v1 minimal: audit_log by default; alert-forwarding only if a clear use case + a guarded
field. (The spec says "flows into the alert ingestor OR dedicated event log".)

### 4. Tests

Token with scope → 202 + audit row; token without scope → 403; malformed payload → 422; assert no
secret leakage. 3b: a real HA automation posts and the event lands in audit_log.

## Out of scope for this stage

- Push-BACK to HA (005-020 — the outbound direction).
- A UI for browsing HA-fired events (could be a future panel addition; not v1).
- Turning every HA event into an alert by default (informational by default).

## Decisions in this stage

### Design phase will confirm

- **D-HAEVENT-SCHEMA** — the `HAEventPayload` shape (required event_type, free-form data, optional
  severity/title) + `extra` policy.
- **D-HAEVENT-DESTINATION** — audit_log only vs optional alert-forwarding (and how the payload opts in).
- **D-HAEVENT-ROUTER-PLACEMENT** — confirm `integrations_ha.py` router + `app.py` registration mirrors
  the docker router.

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-HAEVENT-SCOPED** — guarded by a new `ha:event:write` token scope using the existing
  `require_user_or_token` pattern.
- **D-BIDIRECTIONAL** — bidirectional HA is the headline decision; the webhook ingester (in) pairs with
  the push-back firer (out, 005-020).
