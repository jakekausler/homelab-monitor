# STAGE-005-019: HA webhook ingester

## Status: Complete

## Overview

The inbound half of bidirectional integration: an endpoint where HA automations can POST arbitrary
"something interesting happened" events into the monitor, authenticated by a scoped API token.
Scenario #23. Reuses the exact scoped-endpoint + pydantic-payload + audit-log pattern the alert
ingestor and heartbeat receiver already use.

**Vertical slice:** after this stage, an HA automation can `POST /api/integrations/ha/event` with a
token scoped `ha:event:write`, the payload is validated and recorded to `audit_log`, and a 202 is
returned â€” giving HA a push channel INTO the monitor.

## Concrete deliverables

### 1. New scope `Scope.HA_EVENT_WRITE`

Add `HA_EVENT_WRITE = "ha:event:write"` to the `Scope` enum in `kernel/auth/scopes.py` (additive
change, same as the existing `heartbeat:write` / `alerts:ingest:write`).

### 2. Endpoint `POST /api/integrations/ha/event`

A new router (e.g. `kernel/api/routers/integrations_ha.py`, `APIRouter(prefix="/integrations/ha")`,
registered in `app.py` like `docker.router`). Guarded by
`Depends(require_user_or_token({Scope.HA_EVENT_WRITE}))`. Accepts a pydantic `HAEventPayload`
(event_type + arbitrary data + optional title/severity hint â€” Design defines the schema). Records the
event to `audit_log` (`who="home_assistant"`, `what="ha_event.<type>"`, `after=<payload>`). Returns 202.

### 3. Optional: route into the alert pipeline

Design decides whether HA-fired events are PURELY informational (audit_log only) or can optionally
become alerts (forwarded to the ingestor/dispatcher when the payload requests it, e.g. a severity
hint). Keep v1 minimal: audit_log by default; alert-forwarding only if a clear use case + a guarded
field. (The spec says "flows into the alert ingestor OR dedicated event log".)

### 4. Tests

Token with scope â†’ 202 + audit row; token without scope â†’ 403; malformed payload â†’ 422; assert no
secret leakage. 3b: a real HA automation posts and the event lands in audit_log.

## Design Phase

### Design Notes (decided 2026-06-12)

A new inbound POST endpoint mirroring the existing scoped-ingest pattern (alerts.py / cron_events.py):
`POST /api/integrations/ha/event`, guarded by `Depends(require_user_or_token({Scope.HA_EVENT_WRITE}))`,
validates a pydantic `HAEventPayload`, writes ONE `audit_log` row via `audit_write(repo, ...)`, returns 202.

**D-HAEVENT-SCHEMA â†’ CHOSEN: top-level `extra="forbid"`, free-form `data`, severity/title recorded-only.**
```
class HAEventPayload(BaseModel):
    model_config = ConfigDict(extra="forbid")
    event_type: str = Field(min_length=1, max_length=128, pattern=r"^[a-zA-Z0-9_.:-]+$")
    data: dict[str, Any] = Field(default_factory=dict)
    severity: Literal["info", "warning", "critical"] | None = None
    title: str | None = Field(default=None, max_length=256)
```
Top-level `forbid` (we own the contract; an HA automation author gets a loud 422 on a typo'd field);
`data` is the open free-form escape hatch. `event_type` is pattern-constrained â€” MANDATORY because it flows
into the audit `what` (see D-HAEVENT-WHAT). `severity` uses a `Literal["info","warning","critical"]` (NOT the
`Severity` enum) to avoid coupling this standalone ingest endpoint to the alert subsystem â€” identical 422
validation, zero cross-subsystem import; a later alert-forwarding stage maps literalâ†’enum at its seam.
`severity`/`title` are RECORDED-ONLY in v1 (land in `after_json`, not acted upon) â€” kept for forward-stability
so adding alert-forwarding later is not a breaking schema change.

**D-HAEVENT-DESTINATION â†’ CHOSEN: pure audit_log, NO alert-forwarding in v1.** The endpoint records to
`audit_log` and returns 202; it does NOT construct an AlertFiringEvent or touch the dispatcher. Rejected:
opt-in alert-forwarding â€” forwarding an HA-originated event into the dispatcher would round-trip BACK to HA
via the `ha_push` channel (005-017), and there is no origin-tag / loop-suppression mechanism. Loop-suppression
is real design + test surface that belongs in a future stage that treats it first-class, not smuggled into a
thin ingest endpoint. severity/title are recorded-only until then.

**D-HAEVENT-WHO â†’ CHOSEN: `who=_principal_label(principal)` (â†’ `api-token:<id>`).** Deliberate, documented
deviation from the card's literal `who="home_assistant"`: the principal label identifies WHICH token posted
(enables revocation-by-audit, matches the admin.py convention for every other audited action). The
"this is an HA event" semantics live in `what`, not `who`. `_principal_label` (currently in admin.py) is
hoisted to a shared auth/audit helper if it is private, rather than duplicated.

**D-HAEVENT-WHAT â†’ CHOSEN: `what=f"ha_event.{event_type}"`.** Per-type, greppable, indexed-column queryable
(`WHERE what LIKE 'ha_event.%'`). Safe ONLY because `event_type` is pattern-constrained `^[a-zA-Z0-9_.:-]+$`
+ max_length 128 (matched pair with the schema â€” no whitespace/newline can corrupt the audit `what`).
`event_type` ALSO lands in `after_json` (self-contained row).

**D-HAEVENT-ROUTER-PLACEMENT â†’ CHOSEN: new `kernel/api/routers/integrations_ha.py`,
`APIRouter(prefix="/integrations/ha", tags=["integrations"])`, registered in `app.py` via
`app.include_router(integrations_ha.router, prefix="/api")` â€” final path `/api/integrations/ha/event`,
mirroring the docker integration router exactly.**

**D-SECRET-LEAK-SCOPE â†’ confirmed framing.** The "HA token never logged" guarantee covers OUR inbound auth
token (the `ha:event:write` API token in the Authorization header) â€” auth-layer responsibility; the endpoint
never logs the request body at info level and never echoes the token. The free-form `data` dict is stored
VERBATIM in `after_json` BY DESIGN (audited user-supplied content; `audit_write` does `json.dumps` with no
redaction â€” callers must not pass secrets, and HA automation authors control what they send). Tests assert:
a token sentinel is absent from the audit row (who/what/after_json) AND from captured logs; no info-level body
logging; and a POSITIVE assertion that benign `data` persists verbatim in `after_json`.

**Recommended endpoint shape (for Build):**
```
@router.post("/event", response_model=HAEventResponse, status_code=202)
async def ingest_ha_event(
    payload: HAEventPayload,
    request: Request,
    principal: Annotated[User | ApiToken, Depends(require_user_or_token({Scope.HA_EVENT_WRITE}))],
    repo: Annotated[SqliteRepository, Depends(get_repo)],
) -> JSONResponse:
    await audit_write(
        repo,
        who=_principal_label(principal),
        what=f"ha_event.{payload.event_type}",
        after=payload.model_dump(mode="json"),
        ip=_client_ip(request),
    )
    return JSONResponse(status_code=202, content=HAEventResponse(status="accepted").model_dump(mode="json"))
```
`HAEventResponse` is a trivial `BaseModel` (`status: Literal["accepted"]`, `extra="forbid"`). `after=` uses
`model_dump(mode="json")` so the optional severity Literal serializes to its string value for `json.dumps`.

**Scope addition:** `HA_EVENT_WRITE = "ha:event:write"` added to the `Scope` StrEnum in `kernel/auth/scopes.py`
(purely additive; appended after `docker:write` in source order â€” the `Scope` StrEnum has no enforced ordering, so source position does not affect semantics).

**OpenAPI:** this stage adds API pydantic models (`HAEventPayload`, `HAEventResponse`) â†’ Build MUST regen
OpenAPI â†’ TS types (`make openapi-export` then `cd apps/ui && pnpm run generate-types`); only
`packages/shared-types/openapi.json` is tracked.

**020 hand-off:** 020 (push-back firer, outbound) is fully INDEPENDENT â€” no shared model/scope/router with 019.
Browsing HA-fired events in the UI is UNOWNED in EPIC-005 (future panel addition); 019 is write-only.
