# STAGE-005-008: Update-availability collector

## Status: Complete

## Overview

Emit "update available" metrics from HA's `update.*` entities — the `Settings > Updates` surface the
user explicitly asked for. On the live install there are 106 `update.*` entities with 41 currently
available, covering HA Core/OS/Supervisor, add-ons, HACS frontend cards, and integration firmware.
Scenario #13.

**Vertical slice:** after this stage, `homelab_ha_update_available{entity_id,title}` (1 when an update
is available, 0 when up to date) with `installed_version`/`latest_version` labels appears in VM,
feeding the update-roundup rule (005-015) and the updates widget (005-024).

**Spec references:** master spec §6.2 (HA metrics), §9.2 + §8.5 (update availability surfacing).

## Concrete deliverables

### 1. Update collector

A `BaseCollector` polling `/api/states` (may share the availability poll — Design). Filters
`entity_id` starting `update.`. Emits `homelab_ha_update_available{entity_id,title}` = 1 when
`state == "on"` (HA's convention: on = update available), 0 when off. Attaches `installed_version` and
`latest_version` as labels (or as info-metric companions — Design decides label-vs-info to control
cardinality, since versions are high-churn label values). Captures `auto_update`/`in_progress` if
useful.

### 2. Version-label cardinality care

Version strings as labels can inflate cardinality; Design decides whether `latest_version`/
`installed_version` are labels on the gauge, or a separate `homelab_ha_update_info{...}` info-style
series, or surfaced only via the panel data endpoint (005-021). The cap (005-004) still applies.

### 3. Tests

Mock `update.*` entities (on/off, with versions) → assert available=1/0, version handling, labels;
assert up-to-date entities emit 0 (not absent) so the roundup can count; UTC + cap.

## Out of scope for this stage

- The update-available roundup RULE (005-015) and the updates widget (005-024).
- Actually performing updates (the spec keeps auto-update OFF; this is detection only).
- HACS-critical repository alerts (out of scope — YAGNI).

## Design Phase

### Design Notes (resolved 2026-06-11)

Brainstormer (Opus) generated options; coordinator selected. Grounded in verified consumer
contracts (015/021/024) + the just-built 007 precedent.

**D-UPDATE-SHARE-POLL → CHOSEN: independent poll + `_shared.py` (per 007 precedent).**
- 008 does its own `await get_states_or_error(ctx)`; `concurrency_group="homeassistant"` serializes
  same-group runs so concurrent polls never hammer HA. Reuses 007's `_shared.py` helpers
  (`get_states_or_error`, `extract_domain`). No poll-sharing machinery, no TTL cache.

**D-UPDATE-VERSION-CARDINALITY → CHOSEN: Option C — metric carries `{entity_id,title}` only; versions are PANEL-ONLY (021's concern), NOT in any metric.**
- The 005-004 cap mechanism exists precisely because high-churn labels are the known failure mode.
  Version strings (`installed_version`/`latest_version`) are the canonical high-churn anti-pattern —
  every update bumps the label set, minting new series while old ones go stale → unbounded growth.
- VERIFIED consumer reality — NO consumer needs versions in a METRIC:
  - 015 (update-roundup rule, #13): consumes ONLY `homelab_ha_update_available == 1` (count/digest).
    Version-AGNOSTIC.
  - 024 (updates widget): displays title + installed→latest + release link, but explicitly "Reuses
    the 005-021 summary/detail data" — gets versions from the 021 ENDPOINT, not a metric.
  - 021 (panel endpoint): data source is INTENTIONALLY OPEN per its card ("query VM, OR collector-
    cached state, OR DB — whichever the codebase already uses"); the Docker-panel precedent reads
    detail from a SQLite cache / re-query, NOT metric labels.
- Option C is the only FORWARD-REVERSIBLE choice: adding version labels/info-metric LATER (if a real
  need emerges) is additive/non-breaking; Options A/B can't be walked back cleanly once dashboards
  bind to them.
- REJECTED Option A (versions as labels on the gauge): textbook high-churn-label anti-pattern;
  unbounded series growth; over-serves (no consumer needs it); breaking to remove later.
- REJECTED Option B (separate `homelab_ha_update_info` info-metric): would introduce the PROJECT'S
  FIRST info-metric pattern (no precedent in codebase) to serve ZERO consumers — premature
  abstraction; still has version-churn cardinality, just quarantined.
- **008 emits the boolean + title and NOTHING version-related. Versions are 021's problem, decided
  in 021's Design phase** (021 picks re-query-HA vs collector-cache then, against a concrete need).
  Do NOT have 008 pre-populate a version cache speculatively now.

**Metric semantics (locked for 015/021/024):**
- Filter: `entity_id` starts with `update.` (i.e. `extract_domain(entity_id) == "update"`).
- `homelab_ha_update_available{entity_id,title}` = 1.0 when `state == "on"` (HA convention: on =
  update available), 0.0 when `state == "off"` (up-to-date). EMIT THE 0s — gives 015 a stable
  denominator (total updatable entities) and disambiguates "up to date" from "no longer reported".
- `unavailable`/`unknown` states are SKIPPED (not emitted as 0 — an unavailable update entity is
  UNKNOWN, not up-to-date). Matches 006/007's parse-cleanly-or-skip discipline.
- `title` IS a label: titles are entity-stable / low-churn (human name of addon/integration, e.g.
  "Home Assistant Core") and give 015's digest + Grafana a human-readable label with no join.
- Always-emitted drop gauge `homelab_metric_family_dropped_series{family="homelab_ha_update_available"}`
  via CappedEmitter.

**Cardinality:** register `homelab_ha_update_available` in `_DEFAULT_CARDINALITY_FAMILIES` with an
explicit cap of **150** (106 real entities + headroom for new add-ons). An explicit tighter cap (vs
the loose global 500) DOCUMENTS the expected ceiling and catches a future label-churn regression
early. (This is the kernel↔integration coupling accepted per the `_DEFAULT_CARDINALITY_FAMILIES`
precedent — document in changelog.)

**Out of scope (confirmed):** the update-roundup RULE (015), the updates widget (024), performing
updates (detection only; auto-update stays OFF), HACS-critical repo alerts (YAGNI).

**Verified anchors (fresh from 007, 2026-06-11):** `_shared.py` helpers live; `ha_battery.py` is the
exemplar to mirror; `_HA_COLLECTORS` has 3 entries (add a 4th); `HaState.attributes: dict[str,object]`
carries `title`; entity `state` for `update.*` is "on"/"off"; no info-metric precedent exists.
