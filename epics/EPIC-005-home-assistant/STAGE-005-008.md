# STAGE-005-008: Update-availability collector

## Status: Design (Not Started)

## Overview

Emit "update available" metrics from HA's `update.*` entities — the `Settings > Updates` surface the
user explicitly asked for. On the live install there are 106 `update.*` entities with 41 currently
available, covering HA Core/OS/Supervisor, add-ons, HACS frontend cards, and integration firmware.
Scenario #13.

**Vertical slice:** after this stage, `homelab_ha_update_available{entity_id,title}` (1 when an update
is available, 0 when up to date) with `installed_version`/`latest_version` labels appears in VM,
feeding the update-roundup rule (005-015) and the updates widget (005-024).

**Spec references:** master spec §6.2 (HA metrics), §9.2 + §8.5 (update availability surfacing).
EPIC-005 §"Signal-source matrix". Scenario #13.

## Concrete deliverables

### 1. Update collector

A `BaseCollector` polling `/api/states` (may share the availability poll — Design). Filters
`entity_id` starting `update.`. Emits `homelab_ha_update_available{entity_id,title}` = 1 when
`state == "on"` (HA's convention: on = update available), 0 when off. Attaches `installed_version` and
`latest_version` as labels (or as info-metric companions — Design decides label-vs-info to control
cardinality, since versions are high-churn label values). Captures `auto_update`/`in_progress` if
useful.

### 2. Version-label cardinality care

Version strings as labels can inflate cardinality; Design decides whether `latest_version`/
`installed_version` are labels on the gauge, or a separate `homelab_ha_update_info{...}` info-style
series, or surfaced only via the panel data endpoint (005-021). The cap (005-004) still applies.

### 3. Tests

Mock `update.*` entities (on/off, with versions) → assert available=1/0, version handling, labels;
assert up-to-date entities emit 0 (not absent) so the roundup can count; UTC + cap.

## Out of scope for this stage

- The update-available roundup RULE (005-015) and the updates widget (005-024).
- Actually performing updates (the spec keeps auto-update OFF; this is detection only).
- HACS-critical repository alerts (out of scope — YAGNI).

## Decisions in this stage

### Design phase will confirm

- **D-UPDATE-VERSION-CARDINALITY** — versions as labels vs info-metric vs panel-only.
- **D-UPDATE-SHARE-POLL** — share the availability `/api/states` fetch vs separate poll.

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-UPDATES-VIA-REST** — updates come from `update.*` entities in `/api/states` (verified
  REST-reachable: 106 entities, 41 available).
- **D-UPDATES-COVER-ALL** — covers HA Core/OS/Supervisor + add-ons + HACS + integration firmware (all
  surface as `update.*`).
