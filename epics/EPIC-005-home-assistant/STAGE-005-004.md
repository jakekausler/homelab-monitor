# STAGE-005-004: Reusable cardinality cap

## Status: Complete

## Overview

Add a reusable per-metric-family series cap to the metrics-emission path. Home Assistant has hundreds
to thousands of entities (the live install has ~2000), so an HA entity-availability metric could
explode VictoriaMetrics cardinality. `MetricsWriter` has NO cap today. This stage builds a minimal,
configurable cap (default 500 series/family) plus an over-budget suggestion event — a cross-cutting
mechanism EPICs 006/007/008/021 inherit. It lands just before the first big-cardinality collector
(entity-availability, 005-006) needs it.

**Vertical slice:** after this stage, a collector emitting more than the configured cap for a metric
family has its excess series dropped (deterministically, with a recorded reason) and a suggestion
("metric X exceeded its 500-series budget") surfaces, rather than silently blowing up VM cardinality.

**Spec references:** master spec §5.4 (per-collector budgets), §6.2 (per-stream retention / cardinality),
EPIC-005 §"Cardinality cap" + §Cross-stage acceptance criteria ("Cardinality budget respected").

## Cross-cutting standards (from EPIC-005 §Cross-stage acceptance criteria)

- Cardinality budget respected — configurable per-family cap; over-budget surfaces a suggestion

## Concrete deliverables

### 1. Cap mechanism in the metrics path

A reusable cap layer over `MetricsWriter` (or a helper a collector wraps its emissions in — Design
decides whether it's enforced centrally in the writer or opt-in per collector). For a given metric
family (name), enforce a configurable max distinct label-set count. When exceeded: keep a
deterministic subset (Design picks the policy — e.g. first-N stable by sorted key, or
keep-most-recently-seen) and drop the rest, recording a count of dropped series. Must be deterministic
across ticks so the same series survive (no flapping).

### 2. Configuration

A default cap (500) configurable per metric family (and a global default). Consistent with how
collectors read config today (`getattr(ctx.config, ...)` / per-plugin config). The cap value and which
families it applies to are configurable so HA (and later integrations) can tune.

### 3. Over-budget suggestion event

When a family exceeds its cap, emit a `SuggestionEvent` (the existing collector event type) like
"metric `homelab_ha_entity_available` exceeded its 500-series budget (1,847 seen); raise the cap or
narrow the entity filter." Also emit an observability metric (e.g. `homelab_metric_family_dropped_series{family}`)
so the drop is itself monitorable.

### 4. Tests

Unit tests: under-cap emits all series; over-cap drops deterministically to the cap and records the
dropped count; the suggestion event fires once per over-budget family; the surviving set is stable
across repeated ticks.

## Out of scope for this stage

- VM-side cardinality limits / retention (that's VM config, EPIC-001/§6.2) — this is an
  emission-side cap.
- The entity-availability collector itself (005-006) — it's the first consumer.
- A UI for editing caps (the suggestion + config suffice for v1).

## Decisions in this stage

### Design phase will confirm

- **D-CAP-ENFORCEMENT-POINT** — central in `MetricsWriter` vs an opt-in wrapper a collector applies.
  (Central is more reusable but touches shared kernel code; opt-in is safer/smaller.)
- **D-CAP-EVICTION-POLICY** — which series survive when over budget (stable-sorted first-N vs
  most-recently-seen), chosen for determinism.
- **D-CAP-CONFIG-SHAPE** — per-family + global default config representation.

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-CARDINALITY-CAP-REUSABLE** — the cap is built as a reusable mechanism here (the exemplar epic),
  inherited by EPIC-006/007/008/021 — not an HA-only hack.
- **D-CAP-DEFAULT-500** — default 500 series/family, configurable.

### Design Decisions (resolved 2026-06-10)

Brainstormer (Opus) produced 3 options per decision; chosen + rejected recorded below. Resolved
autonomously per the session operating mode (no user prompt for backend D-* decisions).

**D-CAP-ENFORCEMENT-POINT → CHOSEN: Option C (opt-in capper helper + thin convenience), used inside a
collector's `run()`. NOT on the shared MetricsWriter hot path.**

- The central wrapping-writer approach (Option A) was REJECTED because it structurally cannot satisfy two
  hard requirements. `MultiplexMetricsWriter._fanout` (`kernel/metrics/multiplex.py:45-51`) receives a
  flat, undelimited, interleaved stream of `write_gauge(name, value, labels)` calls from ALL collectors
  (collectors run concurrently across concurrency groups). A writer on that path has (a) NO per-cycle
  boundary — it cannot know where one collector's `run()` ends, so "keep the same N survivors across
  ticks, reset per cycle" is undefined without fragile timer/heuristic state; and (b) NO return path for
  the `SuggestionEvent` — `write_gauge` returns `None` and the writer has no handle to the emitting
  collector's `CollectorResult.events`. Inside a collector's `run()`, BOTH problems vanish: the cycle
  boundary IS the function body, and the `CollectorResult` is in hand to append to.
- Pure opt-in (Option B) is the correct architecture; Option C adds a thin per-collector `CappedEmitter`
  convenience so wrapping is ~one line, making this the low-friction EXEMPLAR that EPIC-006/007/008/021
  inherit, while keeping a pure, independently-unit-testable `CardinalityCapper` core underneath.

**D-CAP-EVICTION-POLICY → CHOSEN: deterministic stable sort by canonical label-set key, take first-N.**

- Build a canonical key per candidate the way the codebase already canonicalizes label-sets
  (`frozenset(labels.items())` → sorted tuple, mirroring `MemoryRetainingMetricsWriter` at
  `kernel/plugins/io.py:244`), sort all candidates, keep the first `cap`.
- REJECTED most-recently-seen / LRU: it requires cross-tick state (capper no longer pure) and FLAPS when
  >cap series churn — directly violating the "same series survive across ticks, no flapping" requirement.
- Stable-sort-first-N is the only option that is simultaneously deterministic, stateless-per-tick, and
  non-flapping. Survivor set changes only when the underlying entity set changes (correct behavior).
  Over-budget is a misconfiguration signal (the SuggestionEvent tells the user to raise the cap / narrow
  the filter), not a steady state to optimize — so lexical-key survivor selection is acceptable.

**D-CAP-CONFIG-SHAPE → CHOSEN: dedicated `cardinality_caps` YAML section + `load_cardinality_caps_config()`
typed loader, exposing `cap_for(family: str) -> int`.**

```yaml
cardinality_caps:
  default: 500
  families:
    homelab_ha_entity_available: 2500   # example per-family override
```

- A frozen `CardinalityCapsConfig` dataclass + `load_cardinality_caps_config()` in `kernel/config.py`,
  mirroring the existing idiom (`load_disk_budget_config`, `kernel/config.py:53-104`): read YAML at
  `HOMELAB_MONITOR_CONFIG`, apply `HOMELAB_MONITOR_*` env override for the default, fall back to
  `default=500`. `cap_for(family)` returns the per-family override or the default.
- REJECTED per-collector config field (no app-global default → every inheriting epic re-declares it,
  caps drift) and REJECTED hard-coded constructor arg (not operator-tunable → fails "configurable").
- The collector resolves `cap_for(family)` once in `run()` and passes the resolved int into the capper,
  keeping the capper itself pure (no config dependency → fully unit-testable to the 100% branch gate).

**Build-scope note:** the first real consumer (entity-availability collector) is STAGE-005-006, NOT this
stage. STAGE-005-004 delivers the reusable mechanism (`CardinalityCapper` + `CappedEmitter`), the
`load_cardinality_caps_config()` loader, and the `homelab_metric_family_dropped_series{family}`
observability metric + `SuggestionEvent` plumbing — exercised by unit tests feeding synthetic
high-cardinality input. No collector wiring lands in this stage (that is 005-006's job).

**Spec-reference correction:** the card cites master-spec §5.4 (per-collector budgets) and §6.2
(per-stream retention/cardinality). On verification, §5.4 is about per-collector FAILURE budgets and §6.2
is about DISK/retention budgeting — neither is emission-side cardinality. This cap is genuinely new
machinery; the real driver is EPIC-005 §"Cardinality cap" + §Cross-stage acceptance criteria
("Cardinality budget respected"). Scope is unchanged.

**Refinement 3a summary (2026-06-10):** Backend-only, no rig, no prompt. Validated the pure mechanism
against a realistic ~2000-entity HA-style workload driven through the PRODUCTION writer stack
(`MultiplexMetricsWriter` → `MemoryRetainingMetricsWriter` + `PrometheusRegistryWriter`). All 6
real-world checks PASS:
1. Cap sits BEFORE the fan-out — with 2000 entities + cap 500, BOTH the snapshot sink and the
   Prometheus registry (VM scrape) held exactly 500 series; the dropped 1500 reached NEITHER sink.
   `homelab_metric_family_dropped_series` == 1500 in both sinks.
2. No flapping — 3 ticks with differently-shuffled input (seeds 1/2/3) produced the IDENTICAL 500
   survivor set, matching the deterministic first-N-by-sorted-label-key policy.
3. Exactly one warning SuggestionEvent; body: "metric homelab_ha_entity_available exceeded its
   500-series budget (2000 seen); raise the cap or narrow the entity filter."
4. Under-cap (300 entities, cap 500): all 300 survive both sinks, dropped gauge 0, zero suggestions.
5. Config loader: default 500; `HOMELAB_MONITOR_CARDINALITY_CAP_DEFAULT=750` override works;
   per-family YAML override (2500) works while unknown families fall back to default.
6. Performance: 2000-entity cap+emit through the full multiplex stack took ~9ms (negligible per tick).
