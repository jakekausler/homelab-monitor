# STAGE-005-003: integrations/ bundle skeleton + registration pattern

## Status: Complete

## Overview

Establish the `integrations/` collector-bundle directory and the registration pattern that EPIC-005's
collectors live in — and that EPICs 006/007/008/021 copy. Today `plugins/collectors/` has only
`builtin/`; there is no `integrations/` directory. This stage creates
`plugins/collectors/integrations/homeassistant/` and proves the loader registers a collector placed
there, so subsequent Wave-B stages just drop in collector modules.

**Vertical slice:** after this stage, a trivial HA "ping" collector placed under
`integrations/homeassistant/` is discovered/registered by the plugin loader, runs on schedule, and
appears in the collector status API — demonstrating the bundle layout end-to-end.

**Spec references:** master spec §5 (plugin layer, `integration_bundle` kind, directory layout
`plugins/collectors/integrations/<name>/`), §5.5 (plugin discovery & install). EPIC-005 establishes
the per-integration bundle pattern.

- Has Input Forms: [ ] No

  no new secret).
- Host-integration concerns: [ ] No — this is a code-structure + loader-wiring stage; the trivial

## Cross-cutting standards (from EPIC-005 §Cross-stage acceptance criteria)

- All plugins emit `homelab_collector_run_*` self-metrics (the kernel emits these for in-process
  plugins — confirm the bundle collector gets them).
- `concurrency_group="homeassistant"` is the shared group for all HA collectors (so HA is never
  hammered by parallel collectors).

## Concrete deliverables

### 1. `integrations/homeassistant/` bundle directory

Create `apps/monitor/homelab_monitor/plugins/collectors/integrations/__init__.py` and
`.../integrations/homeassistant/__init__.py`. Decide + document the intra-bundle layout the Wave-B
collectors follow (e.g. one module per collector: `entity_availability.py`, `battery.py`, … or a
package). This layout is the template 006/007/008/021 copy.

### 2. Registration pattern

Confirm and document exactly how a collector under `integrations/<name>/` gets registered with the
`PluginLoader` (today registration is programmatic via `loader.register(Cls, {...})` in lifespan /
test conftest; STAGE-001-009 mentioned a future filesystem scan). This stage picks the concrete
registration mechanism for integration-bundle collectors and writes it down so each Wave-B stage just
adds its collector + a registration line. Set the shared `concurrency_group = "homeassistant"`
convention.

### 3. Trivial HA ping collector (proof)

A minimal `BaseCollector` subclass under the bundle (`name="ha-ping"` or similar) that calls
`ctx.ha.get_config()` (or no-ops if `ctx.ha is None`) and emits a single `homelab_ha_up`-style gauge.
This proves the bundle + registration + `ctx.ha` injection all work end-to-end. (The real
`homelab_ha_up` reachability rule is STAGE-005-015; this is just the proof-of-wiring collector — it
may be the seed that 005-006/015 build on, Design decides whether to keep or replace it.)

### 4. Tests

Loader test: a collector under `integrations/homeassistant/` registers, runs, and appears in the
collector status API; emits `homelab_collector_run_*`. Confirm the `concurrency_group` serializes HA
collectors.

## Out of scope for this stage

- The real per-signal collectors (Wave B) — they consume this skeleton.
- Per-plugin YAML config loading (the spec notes this is a future kernel stage; collectors still read
  config via `getattr(ctx.config, ...)` for now — do not build YAML config here).
- Any UI.

## Design Notes (2026-06-10)

Verified the plugin loader/registration mechanism against current code: `PluginLoader.register(collector_cls, config_overrides: dict)` (`kernel/plugins/loader.py`) validates the dict against `CollectorConfig`, instantiates (zero-arg ctor), appends to `_loaded`; `load_all()` feeds the scheduler. Registration is PURELY programmatic — NO filesystem scan (the loader docstring defers auto-discovery to a future stage). Builtin collectors are registered as ~13 inline `try/except` + `loader.register(...)` blocks in `lifespan.py`. `BaseCollector` (`kernel/plugins/base.py`): required ClassVars `name`/`interval: timedelta`/`timeout: timedelta`; defaults `concurrency_group="default"`, `run_kind=ASYNC`, `trust_level=BUILTIN`; `async run(ctx) -> CollectorResult`. `ctx.ha` is ALWAYS a real `HomeAssistantRestClient` (never None — no-token handled inside). `homelab_collector_run_*` self-metrics are emitted by the SCHEDULER around `run()` (automatic). `concurrency_group` serializes same-group collectors via a per-group `asyncio.Lock()` in the scheduler. `GET /collectors` lists `loader.load_all()`. `integrations/` does NOT exist yet; `collectors/` has only `builtin/`.

### D-BUNDLE-LAYOUT → CHOSEN: flat module-per-collector (Option A)

- **Chosen:** `plugins/collectors/integrations/homeassistant/<collector>.py` — one `BaseCollector` subclass per module, mirroring the flat `builtin/` precedent. 003 ships `integrations/__init__.py` (docstring), `integrations/homeassistant/__init__.py` (the `register_all` + collector list), and `integrations/homeassistant/ha_up.py` (the real collector).
- **Rationale:** matches `builtin/` (zero new mental model for the 006/007/008/021 copies); one-file-per-collector = clean one-file Wave-B diffs, no merge contention as ~8 HA collectors land; lightest copyable exemplar.
- **No speculative `_shared.py`:** shared HA-collector parsing helpers (e.g. "extract battery entities from get_states()") get a bundle-private `_shared.py` ONLY when the first Wave-B stage needs a second consumer — do NOT create it in 003 (avoid a premature abstraction the exemplar would teach copies to cargo-cult). The leading underscore signals "bundle-private, not a collector module" for any future auto-discovery scan.
- **Rejected — package-per-collector (B):** premature; every HA collector is one class with one `run()`. **Rejected — single `collectors.py` (C):** all Wave-B stages edit the same file → merge contention; balloons for Unifi/Synology copies.

### D-BUNDLE-REGISTRATION → CHOSEN: per-bundle `register_all(loader)` + ClassVar-derived config helper (Option B)

- **Chosen:** `integrations/homeassistant/__init__.py` exposes `register_all(loader: PluginLoader) -> None` that loops a module-level `_HA_COLLECTORS: list[type[BaseCollector]]`, calling `loader.register(cls, config_from_classvars(cls))` per collector inside a `try/except` (per-collector failure isolation, mirroring the builtins). `lifespan.py` calls `homeassistant.register_all(loader)` ONCE (added in 003). After that, Wave-B HA stages NEVER touch lifespan — they add one collector module + two lines (`from .x import XCollector`; append `XCollector` to `_HA_COLLECTORS`).
- **Kernel helper (the ONLY kernel touch, generic, reused by all bundles — NOT copied per integration):** add `config_from_classvars(cls: type[BaseCollector], **overrides: object) -> dict[str, object]` to `kernel/plugins/loader.py` (or a small kernel helper module), deriving `{"name": cls.name, "interval_seconds": int(cls.interval.total_seconds()), "timeout_seconds": int(cls.timeout.total_seconds())}` + overrides. This removes the hand-built-config-dict boilerplate AND removes a drift surface (the collector's ClassVars are the source of truth for cadence, vs the builtins building the dict inline from the same ClassVars).
- **Rationale:** the only option satisfying "Wave-B touches only the bundle" while keeping lifespan thin (one line per integration) and preserving per-collector failure isolation. Cleanest thing 006/007/008/021 copy.
- **Rejected — inline-in-lifespan (A):** Wave-B would edit lifespan per collector; couples every integration's collector list to the kernel; bloats lifespan across 4 integrations × N collectors. **Rejected — declarative COLLECTORS constant iterated by lifespan (C):** lifespan still hardcodes which bundles to iterate (kernel edit per integration) or needs a bundle registry (more machinery than B); awkward per-collector try/except around a data loop.

### D-PING-COLLECTOR-FATE → CHOSEN: ship the REAL `homelab_ha_up` reachability collector (Option C)

- **Chosen:** the proof collector IS the real, permanent reachability collector — no throwaway. `HaUpCollector` (`integrations/homeassistant/ha_up.py`): `name="ha_up"`, `interval=timedelta(seconds=30)`, `timeout=timedelta(seconds=10)`, `concurrency_group="homeassistant"`. `run()` calls `await ctx.ha.get_config()`; emits gauge `homelab_ha_up` = `1` on success, `0` on `HaError` (emit 0, NOT absent, so STAGE-005-015's rule distinguishes "HA down" from "collector never ran" — `absent()` is a different alert condition).
- **Rationale:** 005-015 is the vmalert RULE for HA reachability (consumes a metric); the collector must EXIST for the rule. Reachability is a genuinely useful permanent first HA metric. Shipping the real collector (vs a throwaway proof) makes the exemplar demonstrate a COMPLETE shippable collector end-to-end — copies see what "done" looks like — and avoids churn (no later stage deletes+rewrites it).
- **Rejected — keep-as-ambiguous-seed (A):** "proof that happens to be permanent" invites later confusion. **Rejected — explicitly-temporary proof (B):** throwaway work; teaches copies to build disposable scaffolding.
- **BUILD-PHASE VERIFICATION (flagged, not a Design blocker):** before writing `ha_up.py`, READ a `builtin/` collector (e.g. `host.py`) to mirror the EXACT `CollectorResult` construction + metric-emission idiom (`ctx.vm.write_gauge(name, value, labels)`) and confirm the up/down gauge convention. The brainstormer's snippet used a placeholder metric API — use the real `ctx.vm.write_gauge(...)` + `CollectorResult(ok=..., metrics_emitted=..., ...)` idiom.

### Tests (acceptance)
Loader test: `register_all(loader)` registers `HaUpCollector`; it appears in `loader.load_all()` with `config name="ha_up"`, `interval_seconds=30`, `timeout_seconds=10` (proves `config_from_classvars`). Run-up: fake `ctx.ha.get_config()` success → `homelab_ha_up == 1`. Run-down: fake `get_config()` → `HaError` → `homelab_ha_up == 0`. Appears in `GET /collectors`. `homelab_collector_run_*` self-metrics emitted (scheduler-automatic; guards wiring). `concurrency_group == "homeassistant"` + (behavioral) two HA collectors sharing the group serialize via the scheduler lock (reuse the existing scheduler serialization test helper). `register_all` isolation: one collector raising in `register()` is logged and does not abort the rest.
