# STAGE-005-018: Minimal severity routing

## Status: Design (Not Started)

## Overview

Add a minimal severity-based routing layer so the HA push channel only receives `error`/`critical`
alerts — not every alert. Today the dispatcher fans EVERY alert to EVERY hardcoded channel (the
`routing_rules`/`channels` tables are scaffolding stubs with no filtering). Shipping HAPushChannel
without this would push every `info`/`warning` to the user's phone — bad UX. This is the smallest
routing slice; the full rule-builder UI + per-tag overrides are deferred to EPIC-012 (documented
there).

**Vertical slice:** after this stage, an `info`/`warning` alert appears in the dashboard but does NOT
push to the phone, while `error`/`critical` alerts do — driven by a minimal read of `routing_rules`
(severity gate), not a hardcoded `if`.

**Spec references:** master spec §8.3 (routing rules per severity), §6.1 (`routing_rules` table),
§4.4 (dispatch). EPIC-005 §"Routing". EPIC-012 §"Deferred from EPIC-005 (routing)".

## Cross-cutting standards (from EPIC-005 §Cross-stage acceptance criteria)

- Severity vocabulary `info|warning|error|critical`.
- State-changing config (if any default rule is seeded) writes `audit_log`.
- Must not break the existing inproc-dashboard channel (it still receives everything, or per the
  seeded default — Design confirms dashboard keeps getting all).

## Concrete deliverables

### 1. Minimal severity routing read

Introduce a minimal routing step in/around the `AlertDispatcher`: read `routing_rules` (a minimal use
of the existing stub table — Design decides whether to add the few columns needed now or read a
seeded-in-code default + a simple table read) to decide, per alert severity, which channels receive
it. Default policy: dashboard (inproc) gets all; HA push gets `error`/`critical` only. This replaces
the "all → all" fan-out ONLY to the extent of the severity gate — it is explicitly a minimal layer,
not the full engine.

### 2. Seed a sensible default routing

Seed a default routing config (in code or a minimal table seed): info/warning → dashboard;
error/critical → dashboard + HA push. Keep it overridable later by EPIC-012 without migration pain
(forward-compatible with the §6.1 `routing_rules` schema EPIC-012 expands).

### 3. Clear boundary with EPIC-012

Document (in code + the stage) that this is the MINIMAL gate and EPIC-012's routing engine
(STAGE-012-005) will REPLACE it with the full per-tag, priority-ordered, user-editable rules —
retiring this gate so there's one routing authority. EPIC-012 STAGE-012-005 installs the full rules
engine and retires this minimal gate in the SAME change; there is no transient period where the
severity gate is the only/permanent router — it is explicitly a stopgap until EPIC-012 lands. The
EPIC-012 file already carries the deferral note (its "Deferred from EPIC-005 (routing)" section);
cross-reference it.

### 4. Tests

Synthetic alerts at each severity → assert HA push receives only error/critical, dashboard receives
all; assert resolved events follow the same gate; assert no regression to the existing dispatcher
fan-out for the dashboard channel.

## Out of scope for this stage

- The routing rule-builder UI, per-tag overrides, drag-and-drop priority, dry-run preview — ALL
  EPIC-012.
- Expanding the `routing_rules`/`channels` schema beyond the minimum this gate needs — EPIC-012.
- Discord/SMTP routing — those channels are other epics.

## Decisions in this stage

### Design phase will confirm

- **D-ROUTING-MINIMAL-SHAPE** — read a seeded-in-code default + minimal table read vs add the few
  `routing_rules` columns now; chosen to be forward-compatible with EPIC-012 (no throwaway migration).
- **D-DASHBOARD-GETS-ALL** — confirm inproc-dashboard keeps receiving all severities (so the live feed
  is complete) while only HA push is severity-gated.
- **D-ROUTING-RETIREMENT** — exactly how EPIC-012 will retire this gate (documented hand-off).

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-MINIMAL-ROUTING-IN-005** — EPIC-005 ships HAPushChannel + a MINIMAL severity routing layer
  (error/critical → HA push) so the channel isn't noisy; the full routing system is EPIC-012 (the
  user's explicit "add channel + minimal routing" choice).
- **D-ROUTING-DEFERRAL-DOCUMENTED** — the full routing scope (rule-builder UI, per-tag overrides,
  schema expansion, dispatcher engine) is recorded in EPIC-012's "Deferred from EPIC-005" section.
