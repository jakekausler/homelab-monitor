# STAGE-005-018: Minimal severity routing

## Status: Complete

## Overview

Add a minimal severity-based routing layer so the HA push channel only receives `error`/`critical`
alerts — not every alert. Today the dispatcher fans EVERY alert to EVERY hardcoded channel (the
`routing_rules`/`channels` tables are scaffolding stubs with no filtering). Shipping HAPushChannel
without this would push every `info`/`warning` to the user's phone — bad UX. This is the smallest
routing slice; the full rule-builder UI + per-tag overrides are deferred to EPIC-012 (documented
there).

**Vertical slice:** after this stage, an `info`/`warning` alert appears in the dashboard but does NOT
push to the phone, while `error`/`critical` alerts do — driven by a minimal read of `routing_rules`
(severity gate), not a hardcoded `if`.

**Spec references:** master spec §8.3 (routing rules per severity), §6.1 (`routing_rules` table),
§4.4 (dispatch). EPIC-005 §"Routing". EPIC-012 §"Deferred from EPIC-005 (routing)".

## Concrete deliverables

### 1. Minimal severity routing read

Introduce a minimal routing step in/around the `AlertDispatcher`: read `routing_rules` (a minimal use
of the existing stub table — Design decides whether to add the few columns needed now or read a
seeded-in-code default + a simple table read) to decide, per alert severity, which channels receive
it. Default policy: dashboard (inproc) gets all; HA push gets `error`/`critical` only. This replaces
the "all → all" fan-out ONLY to the extent of the severity gate — it is explicitly a minimal layer,
not the full engine.

### 2. Seed a sensible default routing

Seed a default routing config (in code or a minimal table seed): info/warning → dashboard;
error/critical → dashboard + HA push. Keep it overridable later by EPIC-012 without migration pain
(forward-compatible with the §6.1 `routing_rules` schema EPIC-012 expands).

### 3. Clear boundary with EPIC-012

Document (in code + the stage) that this is the MINIMAL gate and EPIC-012's routing engine
(STAGE-012-005) will REPLACE it with the full per-tag, priority-ordered, user-editable rules —
retiring this gate so there's one routing authority. EPIC-012 STAGE-012-005 installs the full rules
engine and retires this minimal gate in the SAME change; there is no transient period where the
severity gate is the only/permanent router — it is explicitly a stopgap until EPIC-012 lands. The
EPIC-012 file already carries the deferral note (its "Deferred from EPIC-005 (routing)" section);
cross-reference it.

### 4. Tests

Synthetic alerts at each severity → assert HA push receives only error/critical, dashboard receives
all; assert resolved events follow the same gate; assert no regression to the existing dispatcher
fan-out for the dashboard channel.

## Design Phase

### Design Notes (decided 2026-06-12)

Chosen seam: **per-channel `accepts()` predicate** (brainstormer Option A). Add an optional
`accepts(self, event: AlertEvent) -> bool` method to the `Channel` Protocol (`dispatch/types.py`)
with a default-accept (returns `True`); the `AlertDispatcher` consults it before calling
`deliver()` and skips the channel (one `continue`) when it returns `False`. `HAPushChannel`
overrides `accepts` to admit only error/critical-class events; `InprocDashboardChannel` keeps the
default (accepts everything). No migration, no `lifespan.py` wiring change.

**D-ROUTING-MINIMAL-SHAPE → CHOSEN: code-default per-channel predicate (NOT a table read).**
The `routing_rules` table is a verified 3-column stub (`id`, `severity`, `created_at`) that maps
severities to NO channel and is read by nothing. Literally reading it adds zero value over a code
default and would require a throwaway migration (columns or seed rows) that EPIC-012 STAGE-012-005
must reverse. The card's deliverable permits "seeded-in-code default" — taking it. Rejected:
(B) a `RoutingPolicy` object passed to the dispatcher — a partial duplicate of EPIC-012's real
engine, more parts than a stopgap warrants; (C) reading/seeding the stub table — throwaway
migration + hot-path DB coupling for no benefit.

**D-ROUTING-SOURCE → CHOSEN: route on the RAW uncoerced label `event.labels.get("severity")`.**
Predicate: `(event.labels.get("severity") or "").strip().lower() in {"error", "critical"}`,
fail-closed on missing/blank. Load-bearing fact (flagged by 005-017): the `Severity` StrEnum is
only {info, warning, critical} — there is NO `error` — and ingest `_resolve_severity` coerces a raw
`severity: error` label to `Severity.WARNING`. So `event.severity` is never `error`; the raw string
survives only in `event.labels["severity"]`. Routing on coerced `event.severity in {critical}` would
MISS every error-class HA rule (the primary thing we want to push); routing on
`event.severity in {warning, critical}` would also push every genuine warning (noisy). The raw label
is the only source that admits the error tier precisely while excluding genuine warnings.

**D-DASHBOARD-GETS-ALL → CONFIRMED: dashboard stays unfiltered.** `InprocDashboardChannel` keeps the
default-accept `accepts` (receives every severity, including info and resolved events — it is the SSE
feed the UI consumes). Only `ha_push` is severity-gated.

**D-ROUTING-RETIREMENT → documented hand-off.** This is an explicit STOPGAP. EPIC-012 STAGE-012-005's
routing-rules engine replaces BOTH the EPIC-001 hard-coded fan-out AND this minimal `accepts` gate; at
that point the `accepts` override on `HAPushChannel` and the dispatcher's `accepts` branch are deleted
and channel selection is driven by `routing_rules`. EPIC-012's "Deferred from EPIC-005 (routing)"
section already carries this note; code comments in `ha_push.py` / `dispatcher.py` cross-reference
EPIC-012 STAGE-012-005. There is no transient period where this gate is the permanent router.

**020 hand-off note:** 020's per-alert push-back opt-in is "consistent with" this minimal routing but
018 chose a CHANNEL-LEVEL predicate, NOT a reusable routing-config object — so 020 Design should not
expect a shared config substrate here; 020 picks its own opt-in mechanism (routing config / tag /
label) per its own card.

**Resolved-event behavior:** `accepts()` is applied uniformly to firing AND resolved events (the
dispatcher does not distinguish), and resolved events carry the same `labels` — so a resolved
error/critical alert pushes (close-the-loop) and a resolved warning does not. Matches 005-017's
resolved-notify design.
