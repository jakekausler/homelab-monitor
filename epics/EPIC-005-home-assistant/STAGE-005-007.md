# STAGE-005-007: Battery-level collector

## Status: Design (Not Started)

## Overview

Emit battery-level metrics for HA battery entities — a thin sibling of the entity-availability
collector (same `/api/states` poll, different filter + metric). High user value: catches dead/dying
sensor batteries before the sensor goes silent. Scenario #5.

**Vertical slice:** after this stage, `homelab_ha_battery_level{entity_id,device}` (percent) appears
in VM for every battery entity, feeding the 20%/10% threshold rules in STAGE-005-015.

**Spec references:** master spec §6.2 (HA metrics), EPIC-005 file STAGE-005-003 example (battery
filter on `*_battery_level` / `%` + `device_class: battery`). Scenario #5.

## Concrete deliverables

### 1. Battery collector

A `BaseCollector` (or a thin extension/sharing of the entity-availability poll — Design decides
whether it reuses the same `/api/states` fetch to avoid double-polling). Filters entities with
`device_class: battery` (verified present on the live install, e.g.
`sensor.jake_s_android_battery_level` with `unit_of_measurement: "%"`, `device_class: battery`). Emits
`homelab_ha_battery_level{entity_id,device}` = the numeric percent. Skips non-numeric/unavailable
battery entities (no crash).

### 2. Device association

Where possible, label with the owning device (from the entity → device mapping) so the 20%/10% rule
and the UI battery summary can group by device. (The `core.device_registry` provides this mapping;
Design decides whether to read it or rely on entity naming.)

### 3. Tests

Mock states with battery/non-battery/unavailable-battery entities → assert only battery entities emit,
correct percent, correct labels; assert unavailable battery is skipped (not 0); assert UTC + cap.

## Out of scope for this stage

- The 20%/10% threshold RULES (STAGE-005-015).
- Battery-drain-RATE anomaly (could ride the z-score collector 005-013 if a Design there wants it; not
  this stage).
- The UI battery summary widget (STAGE-005-023).

## Decisions in this stage

### Design phase will confirm

- **D-BATTERY-SHARE-POLL** — share the entity-availability `/api/states` fetch (one poll, two
  collectors' worth of metrics) vs a separate poll. Sharing avoids hammering HA but couples the two.
- **D-BATTERY-FILTER** — `device_class: battery` only vs also `*_battery*` entity-id heuristics for
  devices that don't set device_class.
- **D-BATTERY-DEVICE-LABEL** — read `core.device_registry` for device association vs entity naming.

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-BATTERY-VIA-REST** — battery is the same `/api/states` REST data as availability.
- **D-BATTERY-THIN-SIBLING** — battery is a thin sibling of entity-availability, not a heavy new
  mechanism.
