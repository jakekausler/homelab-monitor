# STAGE-005-007: Battery-level collector

## Status: Complete

## Overview

Emit battery-level metrics for HA battery entities ‚Äî a thin sibling of the entity-availability
collector (same `/api/states` poll, different filter + metric). High user value: catches dead/dying
sensor batteries before the sensor goes silent. Scenario #5.

**Vertical slice:** after this stage, `homelab_ha_battery_level{entity_id,device}` (percent) appears
in VM for every battery entity, feeding the 20%/10% threshold rules in STAGE-005-015.

**Spec references:** master spec ¬ß6.2 (HA metrics), EPIC-005 file STAGE-005-003 example (battery
filter on `*_battery_level` / `%` + `device_class: battery`). Scenario #5.

## Concrete deliverables

### 1. Battery collector

A `BaseCollector` (or a thin extension/sharing of the entity-availability poll ‚Äî Design decides
whether it reuses the same `/api/states` fetch to avoid double-polling). Filters entities with
`device_class: battery` (verified present on the live install, e.g.
`sensor.jake_s_android_battery_level` with `unit_of_measurement: "%"`, `device_class: battery`). Emits
`homelab_ha_battery_level{entity_id,device}` = the numeric percent. Skips non-numeric/unavailable
battery entities (no crash).

### 2. Device association

Where possible, label with the owning device (from the entity ‚Üí device mapping) so the 20%/10% rule
and the UI battery summary can group by device. (The `core.device_registry` provides this mapping;
Design decides whether to read it or rely on entity naming.)

### 3. Tests

Mock states with battery/non-battery/unavailable-battery entities ‚Üí assert only battery entities emit,
correct percent, correct labels; assert unavailable battery is skipped (not 0); assert UTC + cap.

## Out of scope for this stage

- The 20%/10% threshold RULES (STAGE-005-015).
- Battery-drain-RATE anomaly (could ride the z-score collector 005-013 if a Design there wants it; not
  this stage).
- The UI battery summary widget (STAGE-005-023).

## Decisions in this stage

### Design phase will confirm

- **D-BATTERY-SHARE-POLL** ‚Äî share the entity-availability `/api/states` fetch (one poll, two
  collectors' worth of metrics) vs a separate poll. Sharing avoids hammering HA but couples the two.
- **D-BATTERY-FILTER** ‚Äî `device_class: battery` only vs also `*_battery*` entity-id heuristics for
  devices that don't set device_class.
- **D-BATTERY-DEVICE-LABEL** ‚Äî read `core.device_registry` for device association vs entity naming.

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-BATTERY-VIA-REST** ‚Äî battery is the same `/api/states` REST data as availability.
- **D-BATTERY-THIN-SIBLING** ‚Äî battery is a thin sibling of entity-availability, not a heavy new
  mechanism.

## Design Phase

### Design Notes (resolved 2026-06-11)

Brainstormer (Opus) generated options; coordinator selected. All decisions grounded in
verified code anchors (see "Verified facts" below).

**D-BATTERY-SHARE-POLL ‚Üí CHOSEN: A1 ‚Äî independent poll + extract `_shared.py`.**
- Each collector does its own `await ctx.ha.get_states()`. The scheduler runs collectors
  independently on their own intervals; `concurrency_group="homeassistant"` already serializes
  same-group runs, so two `/api/states` calls never hammer HA concurrently. "Sharing a poll" is a
  non-problem under the independent-collector model ‚Äî share CODE, not the poll.
- Extract `integrations/homeassistant/_shared.py` with: `get_states_or_error(ctx)` (the
  `ctx.ha is None` guard + `get_states()` + `isinstance(result, HaError)` handling),
  `extract_domain(entity_id)`, and a new `parse_float_state(state) -> float | None` helper
  (returns None for `unavailable`/`unknown`/empty/non-numeric). Refactor 006
  (`ha_entity_available.py`) to use the shared helpers; 006's tests must stay green.
- This discharges the debt 006 explicitly deferred to "the first device-needing stage" (007 is it).
- REJECTED A2 (keep inline): re-defers a deferral; copy-paste drift. REJECTED A3 (shared TTL cache):
  real machinery (TTL/invalidation/cross-collector mutable state) to save one HTTP call per 30s on a
  single-user homelab ‚Äî pure YAGNI failure, hardest to reverse.

**D-BATTERY-DEVICE-LABEL ‚Üí CHOSEN: B1-a ‚Äî DEFER `device`; emit `homelab_ha_battery_level{entity_id,domain}`.**
- VERIFIED: `device_id` is NOT in `/api/states` attributes (same as 006). The entity‚Üídevice map
  lives in HA's `core.device_registry`, reachable only via the WS client.
- VERIFIED: `CollectorContext` (`kernel/plugins/context.py`) exposes ONLY `ha: HomeAssistantClient | None`
  (the REST client) ‚Äî there is NO `ws`/`ha_ws` field. Sourcing the device label in 007 would require a
  `CollectorContext` change (WS plumbing into collectors) = out of scope for a thin-sibling stage.
- The `device` label is ADDITIVE with ZERO live consumers (015/021/023 are all unbuilt). The
  device-map's WS-down failure semantics would be designed against guesses. Per the add-vs-defer rule
  (resolve by whether the data is cheaply in the response you already fetch ‚Äî it is NOT), DEFER.
- Emit `{entity_id, domain}` (mirror 006's label set for consistency).
- **FLAG for 015 + 021 + 023:** battery metric ships `{entity_id, domain}` only ‚Äî NO `device` label.
  The entity‚Üídevice map is a CROSS-CUTTING FACILITY (needs `core.device_registry` via WS + a
  `CollectorContext` change), NOT a battery-collector responsibility. It should be built by/with the
  first stage GENUINELY BLOCKED on it (likely 023's UI device grouping, or a dedicated device-registry
  stage), designed against a concrete consumer ‚Äî not bolted onto a thin collector ahead of need. Adding
  `device` later is an ADDITIVE label change (no destructive migration; no live consumers to break).
- This is the SECOND deferral of `device` (006 was the first). Consistent precedent.
- REJECTED B1-b (build device now via WS): unknown-sized scope (CollectorContext change + supervised
  WS lifecycle + WS-down failure modes), building ahead of 3 unbuilt/undesigned consumers.

**D-BATTERY-FILTER ‚Üí CHOSEN: B2-a + unit guard.**
- Filter: `attributes.get("device_class") == "battery"` AND `attributes.get("unit_of_measurement") == "%"`.
  The unit guard closes the only real correctness hazard ‚Äî a battery-classed voltage sensor
  (`*_battery_voltage`) would otherwise emit volts (e.g. 3.7) into a percent `_level` family (silent
  data bug). VERIFIED: real battery entities carry `device_class: battery` + `unit_of_measurement: "%"`
  (e.g. `sensor.jake_s_android_battery_level`, `state="85"`).
- Value source: the entity `state` parsed via `parse_float_state` (e.g. "85" -> 85.0). NEVER an
  attribute. Non-numeric / `unavailable` / `unknown` battery entities are SKIPPED (not emitted as 0).
- REJECTED B2-b (`*battery*` entity-id heuristics): high false-positive risk
  (`binary_sensor.*_battery_charging`, `*_battery_state` strings, `*_battery_voltage` volts) ‚Äî the
  voltage case is a silent correctness bug.

**Cardinality:** battery entities are FEW (<100). The global default cap (500) is ample. NO per-family
entry added to `_DEFAULT_CARDINALITY_FAMILIES` for `homelab_ha_battery_level`. Still wire the
`CappedEmitter` for consistency. Confirm real battery cardinality in 3b.

**Metric contract (locked for 015/021/023):** `homelab_ha_battery_level{entity_id,domain}` = numeric
percent (0-100). Drop gauge `homelab_metric_family_dropped_series{family="homelab_ha_battery_level"}`
always emitted (via CappedEmitter). NO `device` label (deferred ‚Äî see FLAG above).

**Locked (inherited from brainstorming 2026-06-10):** D-BATTERY-VIA-REST (same `/api/states` REST data),
D-BATTERY-THIN-SIBLING (thin sibling of entity-availability).

**Verified facts (code anchors confirmed live 2026-06-11):**
- `HaState`: `entity_id, state, attributes: dict[str,object], last_changed, last_updated`. `attributes`
  carries `device_class`/`unit_of_measurement`; NOT `device_id`.
- 006 collector shape (the exemplar to mirror): `name`, `interval=30s`, `timeout=15s`,
  `concurrency_group="homeassistant"`; `ctx.ha is None` guard; `get_states()` -> `isinstance(_, HaError)`
  guard; `CappedEmitter(writer=ctx.vm, events).emit_family(name, caps.cap_for(name), obs)`;
  `caps = load_cardinality_caps_config()`.
- `__init__.py` `_HA_COLLECTORS = [HaUpCollector, HaEntityAvailableCollector]` ‚Äî add a new collector =
  one import + append one class.
- `CollectorContext` exposes ONLY `ha` (REST), no WS field.
