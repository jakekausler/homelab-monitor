# STAGE-005-009: Automation/script run-cadence collector

## Status: Design (Not Started)

## Overview

Emit run-cadence / last-triggered metrics for HA automations and scripts by polling their entity
`last_triggered` attribute. This is the "did it run?" half of automation health (scenario #15);
the "did it fail?" half is a vmalert-logs rule (STAGE-005-014). Per the locked no-websocket-for-
failures decision, failures are NOT collected here ‚Äî only run cadence.

**Vertical slice:** after this stage, `homelab_ha_automation_last_triggered_seconds{entity_id}` (and
the script equivalent) appear in VM, enabling the "automation idle longer than expected" rule
(005-015 #16) and the panel's recent-automation view.

**Spec references:** master spec ¬ß6.2 (HA metrics), EPIC-005 ¬ß"Signal-source matrix" (automation
run-cadence via REST; failures via logs). Scenario #15 (run side).

## Concrete deliverables

### 1. Run-cadence collector

A `BaseCollector` polling `/api/states` (may share the availability poll). For `automation.*` and
`script.*` entities, emit `homelab_ha_automation_last_triggered_seconds{entity_id}` =
now ‚àí `last_triggered` (and `homelab_ha_script_last_triggered_seconds`), plus the entity `state`
(`on`/`off` for automations) as `homelab_ha_automation_enabled{entity_id}` (feeds the
disabled-unexpectedly rule #17). `last_triggered` verified present on the live install
(`automation.turn_on_fish_tank_lights_at_sunrise` ‚Üí `last_triggered` ISO timestamp). Handle
never-triggered (null `last_triggered`) explicitly.

### 2. Optional expected-cadence config (for #16)

Scenario #16 ("automation should have run but didn't") needs a per-automation expected cadence to know
"too long." Design decides whether THIS collector carries an optional `expected_interval` config per
automation (so the metric + a 005-015 rule can flag idleness), or whether #16 is purely a rule reading
`last_triggered_seconds` against a configurable global/per-automation threshold. Keep it minimal ‚Äî a
config map of `entity_id ‚Üí max_idle_seconds` is sufficient; absent = no idle alert for that automation.

### 3. Tests

Mock automations/scripts with various `last_triggered` (recent, old, null) and states (on/off) ‚Üí
assert `last_triggered_seconds` (UTC), `enabled` gauge, null handling; cap.

## Out of scope for this stage

- Automation/script FAILURE detection ‚Äî that's the vmalert-logs rule (STAGE-005-014); do not subscribe
  to websocket failure events here (locked decision).
- The idle/disabled RULES themselves (STAGE-005-015 #16/#17) ‚Äî this stage emits the metrics they read.

## Decisions in this stage

### Design phase will confirm

- **D-CADENCE-EXPECTED-CONFIG** ‚Äî per-automation `max_idle_seconds` config map vs a pure-rule threshold
  for scenario #16.
- **D-CADENCE-SHARE-POLL** ‚Äî share the availability `/api/states` fetch vs separate.

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-AUTOMATION-RUN-VIA-REST** ‚Äî run cadence + last-triggered + enabled-state come from `/api/states`
  REST.
- **D-AUTOMATION-FAIL-VIA-LOGS** ‚Äî failures are a vmalert-logs rule (005-014), NOT websocket events
  (the no-websocket-for-failures choice). The HA docker log carries `Error while executing automation`
  / `Error executing script` (verified on the live install).
