# STAGE-005-009: Automation/script run-cadence collector

## Status: Complete

## Overview

Emit run-cadence / last-triggered metrics for HA automations and scripts by polling their entity
`last_triggered` attribute. This is the "did it run?" half of automation health (scenario #15);
the "did it fail?" half is a vmalert-logs rule (STAGE-005-014). Per the locked no-websocket-for-
failures decision, failures are NOT collected here — only run cadence.

**Vertical slice:** after this stage, `homelab_ha_automation_last_triggered_seconds{entity_id}` (and
the script equivalent) appear in VM, enabling the "automation idle longer than expected" rule
(005-015 #16) and the panel's recent-automation view.

**Spec references:** master spec §6.2 (HA metrics), EPIC-005 §"Signal-source matrix" (automation
run-cadence via REST; failures via logs). Scenario #15 (run side).

## Concrete deliverables

### 1. Run-cadence collector

A `BaseCollector` polling `/api/states` (may share the availability poll). For `automation.*` and
`script.*` entities, emit `homelab_ha_automation_last_triggered_seconds{entity_id}` =
now − `last_triggered` (and `homelab_ha_script_last_triggered_seconds`), plus the entity `state`
(`on`/`off` for automations) as `homelab_ha_automation_enabled{entity_id}` (feeds the
disabled-unexpectedly rule #17). `last_triggered` verified present on the live install
(`automation.turn_on_fish_tank_lights_at_sunrise` → `last_triggered` ISO timestamp). Handle
never-triggered (null `last_triggered`) explicitly.

### 2. Optional expected-cadence config (for #16)

Scenario #16 ("automation should have run but didn't") needs a per-automation expected cadence to know
"too long." Design decides whether THIS collector carries an optional `expected_interval` config per
automation (so the metric + a 005-015 rule can flag idleness), or whether #16 is purely a rule reading
`last_triggered_seconds` against a configurable global/per-automation threshold. Keep it minimal — a
config map of `entity_id → max_idle_seconds` is sufficient; absent = no idle alert for that automation.

### 3. Tests

Mock automations/scripts with various `last_triggered` (recent, old, null) and states (on/off) →
assert `last_triggered_seconds` (UTC), `enabled` gauge, null handling; cap.

## Out of scope for this stage

- Automation/script FAILURE detection — that's the vmalert-logs rule (STAGE-005-014); do not subscribe
  to websocket failure events here (locked decision).
- The idle/disabled RULES themselves (STAGE-005-015 #16/#17) — this stage emits the metrics they read.

## Decisions in this stage

### Design phase will confirm

- **D-CADENCE-EXPECTED-CONFIG** — per-automation `max_idle_seconds` config map vs a pure-rule threshold
  for scenario #16.
- **D-CADENCE-SHARE-POLL** — share the availability `/api/states` fetch vs separate.

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

## Design Phase

### Design Notes (resolved 2026-06-11)

Brainstormer (Opus) generated options; coordinator selected. Grounded in verified consumer contracts
(015 #16/#17, 021, 005-005) + 006/007/008 precedent.

**D-CADENCE-SHARE-POLL → CHOSEN: independent poll + `_shared.py` (per 007/008 precedent).**
- 009 does its own `await get_states_or_error(ctx)`; concurrency_group="homeassistant" serializes.

**D-CADENCE-EXPECTED-CONFIG → CHOSEN: Option B (+ C for overrides) — 009 emits ONLY raw metrics; the
per-automation idle THRESHOLD is NOT 009's concern.**
- 009 emits raw observations only: `last_triggered_seconds`, `script_last_triggered_seconds`,
  `automation_enabled`. NO threshold config, NO companion `expected_idle_seconds` gauge, NO threshold
  label.
- Emitting a static threshold as a metric value (Option A) is a config-as-data ANTI-PATTERN — pollutes
  the TSDB with a never-changing gauge, contradicts the cardinality discipline held across 006/007/008,
  and would invent TWO new patterns (dict-valued per-collector config + config-as-metric) to serve one
  downstream rule. REJECTED.
- The 015 #16 card phrasing "configured expectation (from 005-009)" describes WHICH METRIC the rule
  reads (`homelab_ha_automation_last_triggered_seconds`), NOT that 009 must carry the threshold. The
  threshold is ops/user POLICY that belongs where vmalert rules live:
  - Global default idle threshold → rendered into 015's rule YAML (`deploy/vmalert/metrics/*.yaml`) —
    matches the repo's vmalert pattern. 015's concern, decided in 015's Design.
  - Per-automation overrides → USER-authored via the 005-005 user-rule machinery (already shipped:
    `last_triggered_seconds{entity_id="x"} > N` via the Alerts UI). The capability already exists →
    YAGNI says don't pre-build a per-automation config map nobody has requested.
- VERIFIED no consumer needs the threshold from 009: 015 #17 reads `automation_enabled == 0` (no
  threshold); 021 reads only raw `last_triggered_seconds` + `enabled`; 005-005 is the user override path.
- **FLAG for 015:** the #16 idle threshold home is a 015 concern — default-in-rule-YAML + user-rule for
  per-automation overrides. 009 ships only the raw `last_triggered_seconds` metric the rule reads.

**D-CADENCE-PARSE-EXTRACT → CHOSEN: Option A (parse-only seam) — extract ISO parse to `_shared.py`,
refactor 006.**
- 009 is the 2nd consumer of the `last_changed`/`last_triggered` ISO→seconds pattern (006 has it local).
  Matches 007's established "extract shared helper at the 2nd real consumer" precedent.
- Extract ONLY the pure parse: `parse_iso_or_none(value: str | None) -> datetime | None` (fromisoformat
  + attach UTC if naive; return None on empty/missing/malformed) into `_shared.py`. Refactor 006's
  `_parse_last_changed` to use it (006's tests MUST stay green — byte-identical behavior).
- Keep LOCAL in each collector: the seconds computation (`max((now-ts).total_seconds(), 0.0)`) and the
  parse-error counting (006's counting is wired to ITS OWN parse-error gauge — collector-specific
  side-effect policy, not shared logic). Each caller decides what to do with a None return.
- Build planner must VERIFY 006's `_parse_last_changed` is a cleanly separable parse step (the anchor
  exploration confirmed it is — distinct function returning datetime, feeding the seconds block
  separately). If genuinely inseparable, fall back to local mirror + note deferred extraction.

**Metric contract (locked for 015/021):**
- `homelab_ha_automation_last_triggered_seconds{entity_id}` = `max((now - last_triggered).total_seconds(), 0.0)`
  for `automation.*` entities. `last_triggered` read from `attributes["last_triggered"]` (ISO ts).
- `homelab_ha_script_last_triggered_seconds{entity_id}` = same for `script.*` entities.
- `homelab_ha_automation_enabled{entity_id}` = 1.0 (state "on") / 0.0 (state "off") for `automation.*`;
  other states skipped.
- Null/missing `last_triggered` (never-triggered) → SKIP the last_triggered_seconds series for that
  entity (do NOT emit a fake huge number — absence is honest). Unparseable → count in a parse-error
  gauge (mirror 006's `homelab_ha_*_parse_errors` pattern, single series, value=count, only when >0).
- Always-emitted drop gauges per family via CappedEmitter.

**Cardinality:** automation/script counts are modest (tens, not thousands). Global default cap (500) is
ample. NO per-family override added to `_DEFAULT_CARDINALITY_FAMILIES`. Confirm real counts in 3b.

**Out of scope (confirmed):** automation/script FAILURE detection (005-014 vmalert-logs rule, NOT
websocket — locked); the idle/disabled RULES themselves (015 #16/#17); the expected-cadence threshold
config (015's concern per D-CADENCE-EXPECTED-CONFIG above).

**Verified anchors (fresh 2026-06-11):** `_shared.py` (get_states_or_error, extract_domain,
parse_float_state); `ha_update.py`/`ha_battery.py` exemplars; 006's `_parse_last_changed` (cleanly
separable parse step) + seconds-computation + parse-error gauge pattern; `HaState.attributes:
dict[str,object]` carries `last_triggered`; `_HA_COLLECTORS` has 4 entries (add a 5th); NO per-collector
dict-config precedent.
