# STAGE-005-005: User-authored MetricsQL alert-rule machinery

## Status: Complete

## Overview

Extend EPIC-004's user-authored-alert-rule machinery (which today authors **LogsQL** rules) to author
**MetricsQL** rules, so a user can define and customize numeric metric thresholds — e.g. "alert when
`sensor.freezer_temp > -10°C for 10m`." This is a GENERIC capability (works for any metric, not just
HA) built in EPIC-005's foundation because HA's threshold scenarios (#3b) consume it: STAGE-005-016
seeds HA threshold presets into this machinery, and EPIC-005A's built-in rules must stay coherent with
what users can author here. The user explicitly chose to build this machinery inside EPIC-005's
foundation rather than after.

**Vertical slice:** after this stage, a user can create a MetricsQL alert rule through the same
surface they use for log rules today (create + edit + enable/disable + validate), it persists, and a
matching metric crossing the threshold produces an alert through the normal vmalert→Alertmanager→
ingestor→dispatcher path.

**Spec references:** master spec §8 (notifications/alerting), §9.2 (Settings/alerting screens),
§17 (vmalert rules per epic). EPIC-005 §"User-customizable thresholds". Builds on EPIC-004's user-rule
surface.

## Cross-cutting standards (from EPIC-005 §Cross-stage acceptance criteria)

- Severity vocabulary `info|warning|error|critical` only.
- All state-changing user actions write an `audit_log` row.
- Authored rules must not silently conflict with built-in rules (document precedence).

## Concrete deliverables

### 1. Backend: `metric_user_rules` path (sibling of `log_user_rules`)

Mirror EPIC-004's `log_user_rules` table + CRUD API for metric rules: a `metric_user_rules` table
(rule name, MetricsQL `expr`, `for` duration, severity, labels/annotations, enabled), an API router
(e.g. `/api/metrics/user-rules`) reusing the EPIC-004 user-rule patterns (the same management
endpoints shape: list/create/edit/enable/disable/delete + a rule-health endpoint aggregating vmalert
`/api/v1/rules`). Rules are rendered into a vmalert-metrics rule group the metrics vmalert instance
loads (parallel to how user log-rules feed the logs vmalert).

### 2. Backend: validation (reuse the EPIC-004 layers)

Reuse EPIC-004's two-layer validation posture: an always-on heuristic/semantic floor for the
MetricsQL expr (the metrics analogue of `expr_validate.py`), plus the best-effort vmalert `-dryRun`
exact-parser validation (default-off + fail-open, as in EPIC-004 STAGE-004-043A). Reject `info`-only
no-op rules per the same rules-must-be-meaningful posture.

### 3. Frontend: authoring + management surface

Extend/parallel the EPIC-004 `CreateAlertModal` + the `/logs/user-rules` management tab to support
metric rules: a Simple mode (pick metric/label/comparison/threshold/duration/severity — the
"`sensor.freezer_temp > -10 for 10m`" UX) and an Advanced mode (raw MetricsQL) with a preview, plus a
management list (health/enable/disable/edit/delete). Decide whether this is a new tab or an extension
of the existing one (Design).

### 4. Tests

Backend: rule CRUD, rendering into a vmalert group, validation accept/reject, a planted metric
crossing threshold → rule fires (rig/integration). Frontend: authoring form (Simple + Advanced),
management actions, validation surfacing. Backward-compat: the existing LogsQL user-rule surface is
unchanged.

## Out of scope for this stage

- HA-specific threshold PRESETS (those are seeded in STAGE-005-016, consuming this machinery).
- Built-in default rules for 001-004 signals (EPIC-005A).
- Full routing of these alerts to channels beyond the existing dispatcher (routing is 005-017/018 +
  EPIC-012).
- Anomaly/baseline authoring (the z-score collector 005-013 emits a metric; a user could author a rule
  on it, but the baseline computation is the collector's job, not this stage's).

### Design Decisions (resolved 2026-06-10)

Brainstormer (Opus) ran after a code-level verification of the EPIC-004 user-rule stack. KEY FINDING:
**the card's "sibling `metric_user_rules` table" framing is SUPERSEDED by reality.** EPIC-004 already
built a UNIFIED `log_user_rules` table (migration 0041) with an `expr_kind ∈ {logsql, metricsql}`
discriminator; the CRUD API, renderer (writes metricsql rules to group `user-rules-metrics` →
`/var/vmalert-user-metrics/<rule>.yaml`), health aggregator (already fans out to BOTH vmalert
instances), and basic metricsql tests are ALL LIVE and working. The metricsql DATA/RENDER/HEALTH path
is done. Resolved autonomously per the session operating mode (FE stage — the user prompt happens at
Refinement viewport approval, not at Design).

**D-ARCH-EXTEND-VS-SIBLING → CHOSEN: EXTEND the unified `log_user_rules` system. NO new table, NO new
`/api/metrics/user-rules` router, NO migration.**
- The card's "sibling `metric_user_rules` table + `/api/metrics/user-rules` router" is **SUPERSEDED** —
  it predates EPIC-004's unification. Building a sibling would DUPLICATE a working CRUD/render/validate/
  health stack, fork the renderer (which already handles both kinds in one `render_all`), create two
  tables to keep coherent, and split the unified management UI + health aggregator. Zero forward-dep
  requires it: STAGE-005-016 presets seed through the SAME CRUD the frontend uses; EPIC-005A needs
  user/built-in rules in SEPARATE vmalert GROUPS, which already exist (`user-rules-metrics` vs
  `user-rules-logs`) independent of table count.
- The card's deliverable #3 ("author MetricsQL rules through the SAME surface as log rules") is the
  real goal and is delivered by EXTEND with far less code.

**D-METRIC-RULES-UI-PLACEMENT → CHOSEN: keep the existing `/logs/user-rules` surface; make `expr_kind`
user-selectable. Defer nav promotion.**
- This is the FIRST frontend stage of EPIC-005 — do not restructure nav here. The unified
  `UserRulesTab` + `CreateAlertModal` already render/list both kinds; only the CREATE flow is logs-only
  today (expr_kind hardcoded `'logsql'`). Making expr_kind selectable unlocks metricsql authoring in
  the existing surface. Promoting the surface to a top-level "Alert Rules" nav area is a clean, isolated
  FOLLOW-UP (Option C), not this stage. The existing logsql create/edit/list UX MUST remain unbroken
  (backward-compat deliverable).

**D-SIMPLE-MODE-METRICSQL-UX → CHOSEN: free-text metric name + comparison dropdown + threshold number +
duration → compiles to a safe `<metric> <op> <threshold>` with `for:`→`for_duration`. No new backend
endpoint.**
- Delivers the card's exact target UX ("`sensor.freezer_temp > -10 for 10m`") with least code. Safety:
  validate metric name against a Prometheus identifier regex (`[a-zA-Z_:][a-zA-Z0-9_:]*`) and REJECT
  non-matching (metric names need no special chars — reject, don't escape); operator from a fixed
  allow-list (`> < >= <= == !=`); threshold coerced to a number. Mirrors the logsql Simple mode's
  injection-proof constrained-shape posture (`buildSimpleExpr`), net-new `buildSimpleMetricsQLExpr`.
- REJECTED: metric-name autocomplete from a new VM `/api/v1/label/__name__/values` endpoint (scope creep
  for v1 — convenience not correctness; documented as a clean additive follow-up). REJECTED: label-matcher
  builder (over-built; user drops to Advanced for `{label="..."}` filters in v1).

**D-METRICSQL-VALIDATION-FLOOR → CHOSEN: hard-reject non-threshold metricsql (parity with logsql's
`| stats` hard-reject), conservative.**
- Net-new `_check_metricsql_has_threshold` called from `_validate_metricsql`. Accept if the expr (on a
  quote-stripped copy, so operators inside strings/label matchers don't count, and `=~`/`!~` label
  regex matchers are excluded) contains a top-level comparison (`> < >= <= == !=`) OR a recognized
  alerting construct (`absent(`, set-operators `unless`/`and`/`or`). Else reject with a message
  paralleling logsql's (a bare selector like `up` always fires → meaningless). Keep balanced
  quotes/parens checks; keep the env-gated fail-open vmalert `-dryRun` layer above it. Conservative:
  favor false-negatives (let borderline through, dryRun catches) over false-positives that block valid
  rules — matching the logsql floor philosophy.

**D-SEVERITY-ERROR → CHOSEN: add `error` to the severity vocab across the (unified) surface this stage.**
- Card + spec §8.2 require `info|warning|error|critical` (4); code has 3 `{info,warning,critical}`. The
  card lists the 4-value vocab as a cross-cutting standard FOR THIS STAGE. Mechanical, multi-file, NO DB
  migration (column is plain TEXT, no enum constraint). Touches: Pydantic Literals in `schemas.py`
  (create/patch/response), `VALID_SEVERITIES` in `user_rules_repo.py`, frontend zod enum + select
  options in `CreateAlertModal.tsx`, regenerated TS types. Because the surface is unified, this also
  corrects the logs rules — acceptable (documented standard on a shared surface, not scope creep).

**D-AUDIT → CHOSEN: wire `insert_audit` into all 5 mutating user-rule handlers.**
- The card's cross-cutting standard ("all state-changing user actions write an `audit_log` row") is
  currently UNMET — create/patch/delete/enable/disable write NO audit row (for logs OR metrics).
  `insert_audit`/`audit_write` already exist in `kernel/db/audit.py` and are used elsewhere. Wire them
  into `create_user_rule`/`patch_user_rule`/`delete_user_rule`/`enable_user_rule`/`disable_user_rule`
  in `kernel/api/routers/logs.py`. Fixes both logs and metrics (unified surface).

**D-RULE-PRECEDENCE → already SATISFIED (no code).** User rules render into group `user-rules-metrics`,
built-ins into their own named groups — separate vmalert groups, both can fire independently;
Alertmanager+ingestor dedupe by fingerprint. Document in stage notes; no separation code needed.

**SCOPE (Build) — EXTEND, not sibling:**
- Backend (no new table/router/migration): `schemas.py` (+`error` in 3 Literals); `user_rules_repo.py`
  (+`error` in VALID_SEVERITIES); `expr_validate.py` (net-new `_check_metricsql_has_threshold`);
  `routers/logs.py` (audit-write in 5 handlers).
- Frontend: `CreateAlertModal.tsx` (expr_kind selector; net-new `buildSimpleMetricsQLExpr` + metricsql
  Simple-fields branch; +`error` in zod+select); regenerate `schema.ts` (`make openapi-export` +
  `pnpm run generate-types`). `UserRulesTab.tsx`/`userRules.ts` UNCHANGED (already kind-agnostic).
  Optional kind-neutral tab-label tweak in `LogsLayout.tsx`.
- Tests: extend `test_expr_validate.py` (reject bare `up`; accept `up > 0`, `absent(up)`); extend repo/
  API tests for `error` severity + audit rows on each mutation; net-new FE test for
  `buildSimpleMetricsQLExpr` + kind selector.

**SUPERSEDED from card (document):** "sibling `metric_user_rules` table" and "`/api/metrics/user-rules`
router" — EPIC-004 unified storage via `expr_kind` on `log_user_rules`; 005-005 EXTENDS it. The
D-RULE-PRECEDENCE "separate group" is already satisfied at the render layer.
