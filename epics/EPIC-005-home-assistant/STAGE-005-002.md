# STAGE-005-002: HA websocket client

## Status: Design (Not Started)

## Overview

Add the persistent websocket half of the HA client. Three HA signals the user requires — config-entry
live state, repairs issues, and persistent notifications — are NOT reachable via REST (verified:
`/api/config/config_entries` → 404, repairs → 404, notifications absent from `/api/states`) and not
usefully recoverable from `.storage/` files (static/stale). The websocket (`/api/websocket`) is the
only authoritative source. This stage builds the reusable websocket client; the collectors that
consume it are STAGE-005-010/011/012.

**Vertical slice:** after this stage, the monitor maintains a persistent authenticated websocket to
HA that survives HA restarts (reconnect with backoff), exposes a subscribe API, and a smoke test
confirms it receives at least one event (e.g. `subscribe_events`) without wedging the event loop.

**Spec references:** master spec §2 Q10 (bidirectional HA, all listed signals), §6.2 (HA metrics via
websocket). EPIC-005 §"HA client = REST + websocket".

## Cross-cutting standards (from EPIC-005 §Cross-stage acceptance criteria)

- All websocket subscriptions reconnect with backoff and never wedge the event loop.
- HA token never logged (the auth-handshake message carries the token — assert it is not logged).
- Connection failures handled gracefully — a dropped websocket reconnects; collectors depending on it
  report "stale" rather than crashing.

## Concrete deliverables

### 1. Websocket client

A persistent `/api/websocket` client: the HA auth handshake (`auth_required` → send `auth` with the
bearer → `auth_ok`/`auth_invalid`), a `subscribe(...)`/`send_command(...)` API for the WS commands the
collectors need (`subscribe_events`, `config_entries/get` or its WS equivalent, `repairs/list_issues`,
persistent-notification subscription), and an async receive loop that dispatches messages to
subscribers. Runs as a supervised background task in the lifespan (not a per-tick collector). Lives in
the HA client area alongside the REST client (Design confirms the module split).

### 2. Reconnect + backoff + liveness

Exponential backoff reconnect on disconnect/auth-failure; a liveness signal (e.g. a
`homelab_ha_websocket_connected` gauge or a last-message-age) so a wedged/disconnected socket is
itself observable. The receive loop must never block the FastAPI event loop (fully async; bounded
queues).

### 3. Lifespan wiring + shutdown

Start the websocket task in `lifespan` after the REST client (STAGE-005-001) is up; clean shutdown
(close the socket, cancel the task) on app stop. If `ha_token`/`ha_url` absent, skip starting it (same
graceful-degrade posture as 005-001).

### 4. Smoke test

A test asserting: successful auth handshake against a fake WS server; reconnect after a forced drop;
the token is not logged; the receive loop yields a subscribed event. A 3b rig check confirms the real
handshake against `192.168.2.148:8123`.

## Out of scope for this stage

- The config-entry / repairs / notification COLLECTORS (005-010/011/012) — this stage only builds the
  transport + subscribe API; the collectors interpret the events into metrics/state.
- Automation structured-event subscription is NOT used for failures (failures are a logs rule per the
  locked decision) — but the WS `subscribe_events` capability built here MAY be reused by future
  precision upgrades; do not wire automation-failure handling here.

## Decisions in this stage

### Design phase will confirm

- **D-WS-SUPERVISION** — exactly how the WS background task is supervised (asyncio task + restart
  policy) and how subscribers register/deregister.
- **D-WS-LIVENESS-METRIC** — the shape of the connected/last-message-age liveness signal.
- **D-WS-LIBRARY** — raw `httpx`/`websockets` vs an existing dep; keep deps minimal.

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-HA-WEBSOCKET-REQUIRED** — the websocket is in scope (the user chose completeness): config-entry
  state, repairs, and persistent notifications require it and have no REST/file fallback.
- **D-HA-FAILURES-VIA-LOGS-NOT-WS** — automation/script + integration-setup FAILURES are detected via
  vmalert-logs rules over `service:"homeassistant"` (STAGE-005-014), NOT via websocket events. The
  websocket is for config-entry/repairs/notifications only.
