## Design Notes (2026-06-10)

Verified anchors against current code: `kernel/ha/` exists (client.py REST client `HomeAssistantRestClient(base_url, http, token_provider)`, errors.py `HaError`). NO websocket lib in `pyproject.toml` deps (httpx is HTTP-only; `websockets` is present TRANSITIVELY via `uvicorn[standard]` but not declared direct). Established background-task lifecycle: `BuildSourcesLoader`/`DrainConsumer` expose `start_task()` (idempotent `asyncio.create_task(self._loop(), name=...)`) + `async stop_task()` (`cancel()` + `contextlib.suppress(CancelledError)` + await), handle stored on `app.state`, stopped in the lifespan `finally`. Exponential-backoff reconnect exemplar: `_image_events_loop` (`_backoff=1.0; _MAX_BACKOFF=60.0; reset on success; await sleep(backoff) re-raising CancelledError; backoff=min(backoff*2, MAX)`). Background tasks receive `metrics_writer` (MetricsWriter: write_gauge/write_counter/write_summary(name,value,labels)) directly. `metrics_writer` is constructed at lifespan ~line 608, AFTER the REST client at 599-603 â€” so the WS client (which needs the MetricsWriter) must be constructed AFTER line 608. structlog `log.bind(component="ha_websocket")`.

### D-WS-LIBRARY â†’ CHOSEN: declare `websockets` as a DIRECT dependency

- **Chosen:** add `websockets` (pin floor to the currently-resolved version) to `apps/monitor/pyproject.toml` deps; use `websockets.asyncio.client.connect`.
- **Rationale:** already in the resolved env transitively via `uvicorn[standard]` â†’ ZERO install delta; PyPA-standard, async-native, clean JSON-text-frame send/recv matching HA's `/api/websocket` protocol; built-in ping/pong keepalive serves the wedged-socket-observability goal. Declaring it direct is honest (kernel depends on it, not just uvicorn's WS routing) and guards against a future uvicorn change dropping the transitive pin. New direct dep â†’ flag for code review (justified: WS locked in-scope, no stdlib WS client, httpx has no WS support).
- **Rejected â€” httpx-ws:** genuinely new dep (real install weight), smaller/less-tested; "pool reuse" benefit illusory for one long-lived socket.
- **Rejected â€” aiohttp:** adds a second HTTP stack, contradicts httpx-only posture.

### D-WS-SUPERVISION â†’ CHOSEN: HomeAssistantWebsocketClient with start/stop_task; Future-map + Queue-map demux; intent-replay auto-resubscribe

- **Class shape:** `HomeAssistantWebsocketClient` mirroring `BuildSourcesLoader` â€” `start_task()` (idempotent), `async stop_task()` (cancel+suppress+await). The supervised `_run_loop` is connectâ†’authâ†’receive; on disconnect backs off + reconnects.
- **HA protocol demux:** HA multiplexes `result` and `event` messages over ONE socket keyed by a client-assigned monotonic `id`. Two registries: `_pending: dict[int, asyncio.Future]` (one-shot commands; `send_command()` allocates id, registers Future, sends, awaits; receive loop pops on `{"type":"result","id":N}`) and `_subscriptions: dict[int, asyncio.Queue]` (streams; `subscribe()` registers a BOUNDED queue under the id; receive loop `put_nowait`s on `{"type":"event","id":N}`, drop-oldest+counter on full). Single receive loop demuxes. Queues (not callbacks) so a slow consumer can't block the demux loop.
- **Reconnect-resubscribe (the flagged trade-off) â†’ CHOSEN: intent-replay auto-resubscribe** (NOT fail-and-let-collectors-resubscribe). Keep a registry of subscription INTENTS (command + params) distinct from the per-connection idâ†’queue binding; on reconnect, replay each intent (fresh id), rebind the caller-facing handle/queue to the new id; push a lightweight reconnect-marker into the queue so consumers can re-snapshot. **Rationale:** this client is the exemplar 010/011/012 copy, and the locked design requires subscriptions surviving HA restarts; pushing reconnect-awareness onto each of 3 collectors is worse total complexity and a worse exemplar than containing it here. **Rejected â€” fail-and-let-collectors-resubscribe:** less client code but multiplies reconnect logic across collectors.
- **Pending futures on disconnect:** fail all immediately with an `HaError`-style exception (collectors treat as failed run, matching REST posture) â€” never leave a collector hung on a Future that won't resolve.
- **auth_invalid policy â†’ keep retrying with backoff**, log ONCE per state transition (entering auth_invalid), not per attempt; `token_provider` may return a rotated/fixed token later (mirrors REST no-restart recovery + locked graceful-degrade). Emit `error_total{reason="auth"}` per attempt so it's observable while logs stay quiet.
- **Graceful degrade:** in lifespan, start the task whenever `ha_url` is present; let token absence ride the auth-retry path (no-restart recovery). If `ha_url` absent, skip `start_task()`.

### D-WS-LIVENESS-METRIC â†’ CHOSEN: staleness-robust timestamp + connected + counters

- `homelab_ha_websocket_connected` â€” gauge 1.0/0.0, no labels, emit on connect/disconnect TRANSITION (a gauge set once stays put until overwritten â†’ transition-driven is correct, no periodic refresh needed).
- `homelab_ha_websocket_last_message_timestamp_seconds` â€” gauge epoch seconds, no labels, emit on EVERY received message. The ALERT rule computes `time() - <ts> > threshold` to detect a wedged-but-"connected" socket. Staleness-robust by construction (a timestamp, not a pre-computed age that goes stale between scrapes).
- `homelab_ha_websocket_reconnect_total` â€” counter, no labels.
- `homelab_ha_websocket_error_total` â€” counter, label `{reason}` reusing the `HaError` reason vocabulary (unreachable/timeout/auth/bad_response).
- The WS client holds the `MetricsWriter` (constructor-injected) and emits directly.
- **Rejected â€” pre-computed `last_message_age_seconds` gauge:** stale between emits in a pull-scrape model; would need a periodic re-emit timer and still be stale up to the interval.

### Error type â†’ REUSE `HaError` unchanged

No new reasons. `auth_invalid` maps to `reason="auth"`; recv/command timeout â†’ `timeout`; connect failure â†’ `unreachable`; malformed/non-JSON HA frame â†’ `bad_response`. Keeps collector error-handling uniform across REST + WS. Any WS-specific shutdown detail goes in `message`, not a new reason.

### Module layout + key surface

- New file: `kernel/ha/websocket.py` (path pre-declared in `client.py`'s module docstring).
- `HomeAssistantWebsocketClient(*, base_url, token_provider, metrics_writer, log, connect=websockets.connect)` â€” `connect` injectable for tests. Derive ws_url from base_url (httpâ†’ws / httpsâ†’wss, + `/api/websocket`). Public: `connected` property, `start_task()`, `stop_task()`, `async send_command(type_, **fields) -> dict | HaError`, `async subscribe(type_, **fields) -> Subscription | HaError`, `async unsubscribe(sub)`.
- Auth handshake: connect â†’ recv `auth_required` â†’ send `{"type":"auth","access_token":token}` (token from token_provider; if None â†’ HaError(auth, "no token configured"), no network spam) â†’ recv `auth_ok`/`auth_invalid`. NEVER log the auth frame or token.
- Lifespan wiring: construct the WS client AFTER `metrics_writer` (~line 608, NOT at 603); `if ha_config.base_url: ha_ws_client.start_task()`; store on `app.state.ha_ws_client`; `await ha_ws_client.stop_task()` in the finally block alongside other stop_task calls.

## Overview

Add the persistent websocket half of the HA client. Three HA signals the user requires â€” config-entry
live state, repairs issues, and persistent notifications â€” are NOT reachable via REST (verified:
`/api/config/config_entries` â†’ 404, repairs â†’ 404, notifications absent from `/api/states`) and not
usefully recoverable from `.storage/` files (static/stale). The websocket (`/api/websocket`) is the
only authoritative source. This stage builds the reusable websocket client; the collectors that
consume it are STAGE-005-010/011/012.

**Vertical slice:** after this stage, the monitor maintains a persistent authenticated websocket to
HA that survives HA restarts (reconnect with backoff), exposes a subscribe API, and a smoke test
confirms it receives at least one event (e.g. `subscribe_events`) without wedging the event loop.

## Concrete deliverables

### 1. Websocket client

A persistent `/api/websocket` client: the HA auth handshake (`auth_required` â†’ send `auth` with the
bearer â†’ `auth_ok`/`auth_invalid`), a `subscribe(...)`/`send_command(...)` API for the WS commands the
collectors need (`subscribe_events`, `config_entries/get` or its WS equivalent, `repairs/list_issues`,
persistent-notification subscription), and an async receive loop that dispatches messages to
subscribers. Runs as a supervised background task in the lifespan (not a per-tick collector). Lives in
the HA client area alongside the REST client (Design confirms the module split).

### 2. Reconnect + backoff + liveness

Exponential backoff reconnect on disconnect/auth-failure; a liveness signal (e.g. a
`homelab_ha_websocket_connected` gauge or a last-message-age) so a wedged/disconnected socket is
itself observable. The receive loop must never block the FastAPI event loop (fully async; bounded
queues).

### 3. Lifespan wiring + shutdown

Start the websocket task in `lifespan` after the REST client (STAGE-005-001) is up; clean shutdown
(close the socket, cancel the task) on app stop. If `ha_token`/`ha_url` absent, skip starting it (same
graceful-degrade posture as 005-001).

### 4. Smoke test

A test asserting: successful auth handshake against a fake WS server; reconnect after a forced drop;
the token is not logged; the receive loop yields a subscribed event. A 3b rig check confirms the real
handshake against `192.168.2.148:8123`.

## Out of scope for this stage

- The config-entry / repairs / notification COLLECTORS (005-010/011/012) â€” this stage only builds the
  transport + subscribe API; the collectors interpret the events into metrics/state.
- Automation structured-event subscription is NOT used for failures (failures are a logs rule per the
  locked decision) â€” but the WS `subscribe_events` capability built here MAY be reused by future
  precision upgrades; do not wire automation-failure handling here.
