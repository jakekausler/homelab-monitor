# STAGE-005-032: HA config-entry state-distinction metric (collector change)

## Status: Complete

## Overview

STAGE-005-027's `/config-entries?filter=error` endpoint can report that a config entry is "in error" but CANNOT distinguish setup_error from setup_retry (and migration_error / failed_unload). The `ha_config_entry` collector collapses all four non-loaded error states into a single `homelab_ha_config_entry_setup_error` gauge (value 1), so the specific state is not recoverable from VM per-series labels. 027 therefore ships `HaConfigEntryRow.state` as the coarse literal `"error"` and defers the precise distinction here.

This is a COLLECTOR-LAYER fix (not an API/live-HA fix): add a `state` label (or per-state gauges) to the `ha_config_entry` collector so VM CAN distinguish setup_error vs setup_retry vs migration_error vs failed_unload. The cardinality cost is negligible (config entries number in the tens). This keeps config-entry detail VM-sourced, consistent with the locked D-DETAIL-CONSUMES-VM.

Vertical slice: after this stage, the `ha_config_entry` collector emits the specific config-entry state as a VM label, the 027 `/config-entries` endpoint returns the precise `state`, and (if added) a vmalert rule / dashboard panel can distinguish transient retry from hard error.

## Decisions in this stage

### Design decisions (LOCKED 2026-06-14)

- **D-CFGSTATE-SHAPE [LOCKED]** — Add a `state` label to the EXISTING `homelab_ha_config_entry_setup_error` gauge (final shape `homelab_ha_config_entry_setup_error{domain,title,state}` 1/0). The `state` label carries the raw HA config-entry state string. The `homelab_ha_config_entry_loaded{domain,title}` gauge is UNCHANGED.
  - Rejected: per-state gauges (one metric name per error state) — would silently under-cover the existing `expr: ... == 1` vmalert rule until rewritten to an N-way OR; no `{{ $labels.state }}` for a generic summary; largest collector diff.
  - Rejected: new `homelab_ha_config_entry_state{...}=1` info-metric — clean model but forces a hard cutover (collector + rule + endpoint + loaded/error summary queries change atomically); overkill for "distinguish the state".
  - Why A: minimal diff (one collector line, one endpoint line, one contract comment); no metric rename / no consumer migration; the `state` label rides into the alert annotation for free.

- **D-CFGSTATE-LABEL-SCOPE [LOCKED]** — Emit the `state` label on ALL `homelab_ha_config_entry_setup_error` observations (both the value-1 error rows and the value-0 non-error rows), by giving `error_obs` its own label dict `{**labels, "state": state}` while `loaded_obs` keeps the bare `{domain,title}` dict. Keeps `_state_gauges` / the emit loop as a single uniform code path (no second branch on state). The resulting value-0 series churn on state transitions is invisible to both the endpoint query and the vmalert rule (both filter `== 1`) and is bounded by one VM staleness window; cardinality is negligible (tens of entries).
  - Rejected: conditional label (set `state` only when value==1, omit otherwise) — avoids cosmetic churn but reintroduces a second state-branch and yields heterogeneous label sets across the gauge's series; no consumer observes the churn benefit.

- **D-CFGSTATE-VMALERT [LOCKED]** — NO change to the `HaConfigEntryError` rule `expr: homelab_ha_config_entry_setup_error == 1`. Adding the `state` label does NOT break the unfiltered `== 1` match and does NOT double-count (an entry is in exactly one state at a time → exactly one value-1 series per entry). `{{ $labels.domain }}`/`{{ $labels.title }}` still interpolate. Two non-expr edits: (1) add `{{ $labels.state }}` to the alert `summary` annotation for operator value; (2) update the metric-contract comment to `homelab_ha_config_entry_setup_error{domain,title,state}  1/0`.

**OpenAPI:** No regen expected — `HaConfigEntryRow.state` is already `str` (the endpoint just populates it precisely instead of hardcoding "error"). If `make openapi-export` reports no drift in Build, skip `generate-types`.
