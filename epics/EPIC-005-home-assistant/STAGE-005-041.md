# STAGE-005-041: Grafana dashboard review & readability pass (System + Home Assistant) — interactive

## Status: Not Started

## Overview

A collaborative, interaction-driven stage to make sure EVERY metric we collect is surfaced on Grafana correctly, completely, and READABLY — for BOTH the System/host-overview dashboard AND the Home Assistant dashboard. Unlike the other stages, this stage's center of gravity is a back-and-forth working session WITH THE USER: the agent inventories what metrics exist vs what each dashboard panel shows, proposes panel additions/edits/reorganizations, applies them to the dashboard JSON, rebuilds/reloads the rig so the user can view the real rendered dashboards, and ITERATES on the user's feedback until the user is satisfied that everything is surfaced and easy to read.

This stage runs AFTER the audit-gap-closure stages (034–040) so the dashboards under review already contain the new panels/data those stages add (script cadence, registry-filtered availability, etc.). It is the final polish/sign-off on observability surfacing for EPIC-005's domain and the system dashboard.

**Depends on:** STAGE-005-034 through STAGE-005-040 (all gap-closure stages — the dashboards must contain the final metric set before the readability pass). Hard dependency: this is the LAST stage; it reviews the finished surface.

## Stage Flags

- Has Input Forms: No.
- Touches secrets / production data: No (Grafana dashboard JSON + queries over existing metrics; no new data emission).
- Host-integration concerns: Yes — panels are validated against REAL metric data on the prod rig; the user visually reviews the live-rendered dashboards. Refinement is the interactive review loop itself (real rig, real data).

## Interaction model (how this stage runs — READ THIS, it is non-standard)

This stage does NOT follow the usual "build to a fixed spec then verify" shape. Its Build and Refinement phases are a SINGLE interactive loop:

1. **Inventory pass (agent):** enumerate every `homelab_*` metric family relevant to (a) the System/host-overview dashboard and (b) the Home Assistant dashboard. For each, determine: is it on the dashboard? In a readable panel (right viz type, units, thresholds, legend)? Or missing / buried / mis-typed? Produce a gap-and-readability report PER DASHBOARD.
2. **Propose (agent → user):** present the proposed changes per dashboard (new panels, panel-type changes, unit/threshold/legend fixes, row reorganization, removal of redundant/empty panels) and DISCUSS with the user — the user steers priorities and taste.
3. **Apply (agent):** edit the dashboard JSON (`deploy/grafana/dashboards/*.json`), rebuild/reload the rig so the change is live.
4. **Review (user):** the user views the real dashboards (System tab + Home Assistant tab via the in-app Metrics page, and/or Grafana directly) and gives feedback.
5. **Iterate:** repeat 2–4 until the user signs off on BOTH dashboards.

Because this is user-driven, the agent must PROMPT the user throughout (this is an explicit, sanctioned interactive stage). Both dashboards must be signed off — this is the System+HA equivalent of the Desktop+Mobile dual sign-off.

## Concrete deliverables

1. A metric→panel coverage report for the System/host-overview dashboard: every relevant `homelab_*` system/host/container metric mapped to its panel (or flagged missing/unreadable).
2. A metric→panel coverage report for the Home Assistant dashboard: every `homelab_ha_*` metric mapped to its panel (or flagged missing/unreadable). Includes the families that were Grafana-only or unsurfaced before this slate (script cadence, z-score, registry-filtered availability).
3. Dashboard JSON edits to `deploy/grafana/dashboards/` (system dashboard + `home-assistant.json`) implementing the agreed additions/edits/reorganizations: correct panel types, units, thresholds, legends, row grouping, and removal of redundant/empty panels.
4. Each edited dashboard validated rendering REAL data on the prod rig and visually signed off by the user (both dashboards).
5. Regression notes capturing the agreed "every metric is surfaced and readable" invariant per dashboard.

## Decisions in this stage (Design phase will confirm)

- D-DASH-SCOPE — confirm exactly which dashboards are in scope (host-overview/system + home-assistant; any others the user wants, e.g. a dedicated logs or alerts dashboard).
- D-DASH-COMPLETENESS-BAR — what "everything surfaced" means: every metric family on a panel, vs a curated subset with the rest reachable via explore/ad-hoc (user's call on signal-vs-noise).
- D-DASH-READABILITY-RUBRIC — the readability criteria to apply (panel type fit, units, thresholds with sensible colors, legend clarity, row organization, mobile/narrow rendering of the embedded dashboards).
- D-DASH-INTERACTION-CADENCE — confirm the interactive loop format with the user (how changes are batched per review round).
