# STAGE-005-022: Panel shell + sidebar/router registration

## Status: Complete

## Overview

Register the "Home Assistant" integration panel in the UI: a sidebar nav entry, a route, and a page
shell with empty sections that the widget stages (023-025) fill. This establishes the per-integration
UI panel pattern (the first one after Docker) that EPICs 006/007/008/018/021 copy.

**Vertical slice:** after this stage, "Home Assistant" appears under Integrations in the sidebar;
clicking it navigates to `/integrations/home-assistant` and renders a titled page with labeled
(empty/placeholder) sections for entity health, battery, updates, integration status, and logs.

**Spec references:** master spec §9.1 (Integrations ▸ Home Assistant / …), §9.2 (per-integration
plugin-provided panel). EPIC-005 §"Stage decomposition" Wave F. Pattern:
`apps/ui/src/components/SidebarNav.tsx` `NAV_ITEMS`, `apps/ui/src/router.tsx`,
`apps/ui/src/routes/integrations/DockerIntegrationPage.tsx`.

## Concrete deliverables

### 1. Sidebar nav entry

Add a `NAV_ITEMS` entry in `apps/ui/src/components/SidebarNav.tsx`: `{ label: 'Home Assistant', to:
'/integrations/home-assistant', icon: <choose lucide icon, e.g. Home/HousePlug>, indent: true }` under
the existing `'Integrations'` section label (the Docker entry is the sibling pattern).

### 2. Route

Add the route in `apps/ui/src/router.tsx`: a `homeAssistantIntegrationRoute` (parallel to
`dockerIntegrationRoute`), `path: '/integrations/home-assistant'`, `component:
HomeAssistantIntegrationPage`, under the protected layout.

### 3. Page shell

Create `apps/ui/src/routes/integrations/HomeAssistantIntegrationPage.tsx` (mirroring
`DockerIntegrationPage.tsx`): title + description + a layout with clearly-labeled sections/placeholders
for the widgets that 023 (entity-health + battery), 024 (updates + integration-status), and 025
(embedded LogViewer) will fill. Each placeholder shows a calm empty/loading state.

### 4. Tests

Render test: the page mounts at the route; the nav entry appears and links correctly; the section
placeholders render. Desktop + Mobile layout sane.

### Design Notes (resolved 2026-06-12)

Brainstormed via brainstormer (Opus); decisions made by the agent per workflow §0.4 (no user pause). Grounded in the verified Docker panel pattern. This stage is FRONTEND-only, 3a-only (Desktop + Mobile). Backend contract (021 `/summary` endpoint + HA TS types in `apps/ui/src/routes/integrations/types.ts`) verified present; this shell stage does NOT consume the endpoint (data hook is 023).

**D-PANEL-ICON — CHOSEN: `HousePlug` (lucide-react).**
- Verified present in the installed lucide-react. Semantically exact for home automation; visually distinct from Docker's `Container` and any generic `Home`/`House`. Added to the existing lucide import block in `SidebarNav.tsx`.

**D-PANEL-LAYOUT — CHOSEN: Hybrid (count grid + full-width logs).**
- The downstream data is asymmetric: 023 (entity-health, battery) + 024 (updates, integration-status) are 4 small COUNT widgets; 025 is a WIDE LogViewer.
- Shell structure: page header (title + muted subtitle, mirroring `DockerIntegrationPage`) → a responsive grid `grid grid-cols-1 gap-4 sm:grid-cols-2 xl:grid-cols-4` holding 4 labeled count sections (Entity health, Battery, Updates, Integration status) → a full-width "Logs" section below (outside the grid). 1-up mobile → 2×2 tablet → 4-up wide desktop; logs always full width.
- REJECTED: uniform card grid (cramps the wide LogViewer into a half-width cell); stacked full-width sections (wastes horizontal space — a row of small count widgets each on its own full-width row looks sparse).

**Component seam — CHOSEN: page composes thin `<PanelSection title>` wrappers with INLINE placeholder bodies; NO named stub sub-components.**
- 023/024/025 just replace the inline placeholder body inside an already-labeled `<PanelSection>` (lowest coupling, nothing to delete). Named stub components (`<EntityHealthSection/>` etc.) would force later stages to gut a forwarding component — more churn, no benefit.

**D-PANEL-PATTERN — CHOSEN: extract exactly ONE thin primitive `<PanelSection title>`; do NOT extract a heavier `<IntegrationPanelLayout>`.**
- `<PanelSection>` = a presentational titled card (consistent padding/border + calm empty state), the one thing that provably repeats across all 7 future panels. The ARRANGEMENT does not repeat predictably (Docker is a single list; HA is count-grid+logs; future panels vary) → extracting a layout primitive now is premature (YAGNI) and contradicts keep-stages-small (D-PANEL-SPLIT-SMALL). Defer the layout primitive to the 2nd multi-widget panel (EPIC-006).
- `<PanelSection>` placement: `apps/ui/src/routes/integrations/PanelSection.tsx` (integration-scoped) — NOT `components/ui/` (which holds shadcn primitives; PanelSection is an app composition). Build re-confirms the exact convention.
- The reusable per-integration-panel pattern = page under `routes/integrations/` + nav entry + route + composed `<PanelSection>`s in a page-owned layout. Docker page left UNCHANGED this stage.

**Empty/placeholder state (spec §9.3 "not blank") — CHOSEN: one calm descriptive line per section.**
- Each section body is a single `<p className="text-sm text-muted-foreground">` line, e.g. "Entity health will appear here." Reads as deliberately-pending, not broken (vs a "—"/"No data" that reads as a failed fetch) and not loading (vs a skeleton shimmer that misleadingly implies a hung request, since nothing actually loads this stage). The widget stages cleanly REPLACE the `<p>`.

**Files this stage touches:** `apps/ui/src/components/SidebarNav.tsx` (nav entry), `apps/ui/src/router.tsx` (route), NEW `apps/ui/src/routes/integrations/HomeAssistantIntegrationPage.tsx` (page shell), NEW `apps/ui/src/routes/integrations/PanelSection.tsx` (thin primitive), plus a render test. NO OpenAPI regen (no backend model change).

**REFINEMENT DESIGN CHANGE (2026-06-12) — tabbed layout supersedes the single-grid D-PANEL-LAYOUT.**

During the Desktop+Mobile viewport review, the user requested: "Since this will contain many things per panel, change this to a tabbed layout like the logs page." This supersedes the prior single-page card-grid layout. Re-brainstormed (Opus) and grounded in the verified logs-page tab pattern.

- **D-PANEL-TABS-MECHANISM — CHOSEN: ROUTE-BASED tabs** (mirror `LogsLayout.tsx` + `logsLayoutRoute`). The user said "like the logs page," which is route-based, and it matches the app-wide tabbed-section convention (alerts/inventory/settings/logs/container-detail all use a layout route + `<Outlet/>` + TanStack Router `<Link activeProps={...}>` tab bar + child sub-routes per tab). REJECTED state-based (`useState`+ternary) — diverges from the main-page-tabs convention and loses deep-linking.
- **D-PANEL-TABSET — CHOSEN: 3 tabs — Health / Status / Logs.** Clean 1:1 map to the downstream stages: Health ← 023 (entity-health + battery widgets); Status ← 024 (updates + integration-status widgets); Logs ← 025 (embedded HA LogViewer). Honors "many things per panel" (2 widgets per tab, not sparse single-widget tabs). REJECTED 2-tab Overview+Logs (muddies the 023/024 boundary — both edit one page) and 5-tab one-per-region (sparse, crowds the mobile tab bar).
- **D-PANEL-SHELL-SCOPE — CHOSEN: build ALL 3 tab pages now**, each with calm `<PanelSection>` placeholders (a tab bar with dead/404 tabs would look broken in the viewport review). 023/024/025 fill placeholder bodies inside the already-existing tab pages — no router edits downstream.
- **Placeholders within tabs:** each region stays a `<PanelSection title>` card with one muted "… will appear here." line. Health tab = "Entity health" + "Battery" sections; Status tab = "Updates" + "Integration status" sections; Logs tab = a "Logs" placeholder (025 supplies the real LogViewer).
- **Routes:** `homeAssistantLayoutRoute` (path `/integrations/home-assistant`, component `HomeAssistantLayout` — the tab bar + `<Outlet/>`), child routes `health`/`status`/`logs` (components `HomeAssistantHealthTab`/`HomeAssistantStatusTab`/`HomeAssistantLogsTab`), plus a `homeAssistantIndexRoute` (path `/`) that `redirect`s to `/integrations/home-assistant/health` (mirrors `logsIndexRoute`). Parent = `protectedLayoutRoute`.
- **Nav UNCHANGED:** the existing nav entry `to: '/integrations/home-assistant'` is correct — it points at the parent layout route (mirroring the logs nav entry's `to: '/logs'`); clicking it lands on the index → redirects to the Health tab.
- The single-page `HomeAssistantIntegrationPage.tsx` from the first Build is REPLACED by the layout + tab pages. `<PanelSection>` primitive is RETAINED and reused inside the tabs.
