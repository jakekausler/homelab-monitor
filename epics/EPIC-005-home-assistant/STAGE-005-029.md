# STAGE-005-029: HA persistent-notification detail (live re-query + privacy + safe render)

## Status: Complete

## Overview

The notification-body detail list deferred from STAGE-005-024. STAGE-021's `/summary` exposes only a
notification COUNT; notification BODIES were deliberately kept out of metrics (STAGE-005-012 privacy
decision) so they are stored NOWHERE queryable. This stage adds a notification detail list (title +
safely-rendered message) to the HA panel's Integration-status section — which requires re-querying HA
live (NOT VM), a privacy review, and safe text rendering.

**Vertical slice:** after this stage, the Integration-status widget can show the actual current HA
persistent notifications (title + body), sourced by re-querying HA live, rendered safely (text-only,
escaped — no HTML injection).

- Host-integration concerns: [x] Yes — live HA re-query (NOT VM). Refinement 3a (dev rig) AND 3b (prod rig — real notification bodies).

## Cross-cutting standards (from EPIC-005 §Cross-stage acceptance criteria)

- Strict pyright + strict TS; session-auth; safe text rendering (no injection); privacy review.

## Concrete deliverables

### 1. Backend: persistent-notification endpoint

A `GET /api/integrations/home-assistant/notifications` endpoint that re-queries HA live (via the HA
REST/WS client — `get_states()` filtered to `persistent_notification.*`, or the notifications WS)
and returns notification rows (notification_id, title, message, created_at). `Depends(require_session())`.
NOT VM-sourced. Privacy: confirm what the authenticated single-user panel may expose.

### 2. Frontend: notification detail list widget

A notification detail list widget added to the Integration-status section (below the notification count),
rendering title + message SAFELY (text-only, escaped — NO dangerouslySetInnerHTML).

### 3. Privacy review

Confirm bodies may be shown to the authenticated user; no logging of bodies; no injection surface.

### 4. Tests

Backend: mocked HA get_states; frontend: mocked rows + a malicious-string render-safety test asserting
no HTML injection.

## Out of scope for this stage

- Entity/battery/updates/config/repairs detail (027/028).
- Grafana (026).
- Acting on notifications (dismiss).

## Decisions in this stage

### Design phase will confirm

- **D-NOTIF-SOURCE** — live HA `get_states()` filtered to persistent_notification vs the notifications WS subscription.
- **D-NOTIF-PRIVACY** — exactly what notification content the authenticated single-user panel may render; redaction if any.
- **D-NOTIF-SAFE-RENDER** — text-only escaped rendering; markdown handling (HA notifications can be markdown) WITHOUT enabling HTML injection.

### Locked Design Decisions (inherited)

- **D-NOTIF-LIVE-REQUERY** — bodies come from a live HA re-query (NOT VM, NOT a metric), because STAGE-012 deliberately kept bodies out of metrics for privacy.
- **D-NOTIF-DEFERRED-FROM-024** — this is the notification-body detail deferred from STAGE-005-024 (which shipped counts-only).

### Design Notes (2026-06-14) — decisions made

Backend wiring VERIFIED against the live codebase: the WS client `HomeAssistantWebsocketClient`
(`kernel/ha/websocket.py`, `send_command(type_, **fields) -> dict | list | HaError`) is ALREADY stored
on `app.state.ha_ws_client` and already used by the count collector; the REST client is built but NOT
stored on app.state. The UI has NO markdown/sanitizer deps and ZERO `dangerouslySetInnerHTML`.

- **D-NOTIF-SOURCE → WS `persistent_notification/get` (CHOSEN).** Endpoint calls
  `app.state.ha_ws_client.send_command("persistent_notification/get")` and parses via the existing
  defensive `_extract_notifications` helper (handles bare-list and `{"notifications":[...]}` shapes) —
  the helper is PROMOTED from the collector module to a shared `kernel/ha/` helper so the collector and
  the endpoint share one source of parsing truth. A new `get_ha_ws_client` dependency is added,
  mirroring `get_http_client` (`_require_state(request, attr="ha_ws_client", ...)`). *Rejected:* REST
  `get_states()` filtered to `persistent_notification.*` — requires NEW lifespan wiring (store the REST
  client on app.state) + a new `get_ha_client` dep + fetching ~1900 entities to filter client-side; WS
  is already wired, pre-filtered, and carries title/message/created_at directly.
- **Degradation contract → HaError → HTTP 502 (CHOSEN).** All HaError reasons (unreachable/timeout/
  http_error/bad_response AND `auth`) map to 502, matching the 027/028 detail/drill convention; the
  frontend `<DrillList>` shell already has a 502/error branch. `auth` is 502 (not 401) because a bad HA
  *token* is a server-side config problem, NOT a bad user *session*. *Rejected:* the 021 `/summary`
  `ha_up=false` 200-with-flag pattern — that fits a summary TILE; this is a drill-style LIST endpoint
  whose frontend shell expects the 502 branch (consistent with 027/028).
- **D-NOTIF-PRIVACY → full title + message + created_at to the authenticated session, with enforced
  no-log / no-metric / no-error-leak (CHOSEN).** The panel mirrors what the single owner already sees
  in HA itself; the STAGE-012 bodies-out decision was about keeping bodies out of METRICS (cardinality
  + persistence), NOT out of the authenticated UI. Session-auth (`Depends(require_session())`) required.
  *Rejected:* redact/truncate message — guts the feature (its whole purpose is showing bodies). Bodies-
  never-logged RULE: no log statement / `repr` / `str` in the endpoint path may take `message`, `title`,
  a row, or a raw notification dict; only `notification_id`s, counts, and `HaError.reason` are loggable;
  the 502 path never echoes the WS payload. ENFORCED by a sentinel test (a sentinel body string appears
  in the response JSON but in NO caplog record).
- **D-NOTIF-SAFE-RENDER → plain-text React auto-escape + `whitespace-pre-wrap break-words` (CHOSEN).**
  Matches the codebase convention (zero markdown/sanitizer deps, zero `dangerouslySetInnerHTML`;
  `SignatureAnnotations.tsx`/`LogLineList.tsx` precedent) and the acceptance criterion "text-only,
  escaped, NO HTML injection". *Rejected:* `react-markdown` + sanitizer — new deps + new XSS surface +
  directly contradicts the "no HTML injection" acceptance criterion. *Rejected:* URL-linkification —
  unrequested scope + a `javascript:`/`data:`-scheme surface. ENFORCED by a mandatory malicious-string
  RTL test (a message containing `<script>`/`<img onerror>`/`[x](javascript:...)`/`**bold**` renders as
  LITERAL text — assert no `<script>`/`<img>` DOM node, no handler fires, literal substrings present).
- **D-NOTIF-WIDGET-PLACEMENT → reuse the 028 `<DrillList<HaNotificationRow>>` shell with a rich
  `renderRow` (CHOSEN).** New "Notifications" `PanelSection` on the Status tab, below the Integration-
  status count tiles, alongside the 028 updates/config-entries/repairs drills. `renderRow` produces a
  notification card: `<h4>` title (fallback "(untitled)") + relative-time `<time>` (existing time util,
  no new date lib) + `<p className="whitespace-pre-wrap break-words …">{message}</p>`. New hook
  `useHomeAssistantNotifications` mirrors the 028 hooks (queryKey under `haQueryKeys`,
  `unwrap<HaNotificationsResponse>`, `refetchInterval: 30_000`). *Rejected:* a bespoke notification-list
  component — would re-implement the loading/empty/error/502 shell that 028 already provides;
  `<DrillList>` is generic over row type and `renderRow` returns arbitrary JSX, so a multi-line card is
  a valid renderRow.

**Backend response model** (mirrors the 027 envelope; `ConfigDict(extra="ignore")`):
`HaNotificationRow { notification_id: str; title: str | None; message: str; created_at: str | None }`
and `HaNotificationsResponse { rows: list[HaNotificationRow]; total: int; returned: int }`. OpenAPI
regen REQUIRED (new model) → `make openapi-export` → `pnpm generate-types`.

**LOCKED decisions honored:** D-NOTIF-LIVE-REQUERY (live WS, never VM) ✓; D-NOTIF-DEFERRED-FROM-024
(delivers the body rendering deferred from 024) ✓.

**Two mandatory tests:** (1) backend bodies-never-logged sentinel test; (2) frontend malicious-string
render-safety RTL test. Plus backend tests over mocked WS (`send_command` returns notifications /
HaError → 502 / both list & dict-wrapped shapes) and frontend hook-mocked render/empty/loading/502.
