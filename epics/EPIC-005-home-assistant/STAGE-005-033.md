# STAGE-005-033: Entities drill — render friendly_name

## Status: Complete

## Overview

STAGE-005-031 adds `friendly_name: str | None` to the `HaEntityRow` backend model (live-HA enrichment),
but does NOT wire the 028 entities drill widget to display it (031 is backend-only). This stage updates
the frontend entities drill-list (`HaEntitiesDrill` / its row render) to show the human-readable
`friendly_name` alongside or instead of the raw `entity_id`, so the "Unavailable entities" list reads
"Master Bedroom Motion" rather than `binary_sensor.master_bedroom_motion`.

**Deferred from:** STAGE-005-031 Design (2026-06-14), D-ENRICH-ENTITY-FRIENDLY-NAME — the backend field
was added in 031; this stage is the frontend follow-up that renders it.

## Decisions in this stage

### Design decisions (LOCKED 2026-06-14)

- **D-FRIENDLY-DISPLAY [LOCKED]** — Render BOTH, in a TWO-LINE row (Option B2), mirroring the established `HaNotificationsDrill` two-line primary/secondary precedent:
  - Line 1: `friendly_name` as the primary (`font-medium`) label on the left + the relative age (`formatAge(last_changed_age_seconds)`) on the right (unchanged right slot).
  - Line 2: `entity_id · domain` as a muted secondary line (`text-xs text-muted-foreground`), GUARDED by `hasName = Boolean(row.friendly_name?.trim())` — the secondary line renders ONLY when a friendly_name is present.
  - Fallback: when `friendly_name` is null / "" / whitespace-only, the primary shows `entity_id` (via `row.friendly_name?.trim() ? row.friendly_name : row.entity_id`) and the secondary line is SUPPRESSED (the `hasName` guard) so there is no redundant `entity_id / entity_id` double-render.
  - Domain is kept on the secondary line (`entity_id · domain`) to preserve parity with the current drill + sibling drills (batteries show domain); it costs nothing on the muted line. When friendly_name is null the domain is still readable from the entity_id prefix on the primary line.
  - Rejected Option A (friendly_name INSTEAD of entity_id): loses the raw entity_id when a friendly name is present — but the homelab owner debugging "unavailable entities" needs the actionable raw id (friendly names are ambiguous/non-unique).
  - Rejected Option B1 (HaUpdatesDrill single-line clone: label primary + entity_id muted on the right): HaUpdatesDrill has no age column, but THIS drill does — entity_id and age would contend for the right slot, and on a narrow mobile viewport the entity_id (the token the owner wants) is the first thing truncated. B2's two-line layout gives both identifiers a full-width line; the rows live in a scrolling PanelSection so the extra vertical space is nearly free.

**Frontend-only:** No backend change, no OpenAPI regen — `HaEntityRow.friendly_name: string | null` already exists in the generated `schema.ts` (added by STAGE-005-031) and is re-exported via `apps/ui/src/routes/integrations/types.ts`. Backend contract verified: the field is populated with real values on the rig (e.g. "Sliding Door").
