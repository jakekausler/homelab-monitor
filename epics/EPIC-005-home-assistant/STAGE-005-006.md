# STAGE-005-006: Entity-availability collector

## Status: Complete

## Overview

The first real HA collector: poll `/api/states` and emit per-entity availability + staleness metrics.
This is the foundational health signal (scenario #1) and the first consumer of the cardinality cap
(005-004), since HA exposes ~2000 entities.

**Vertical slice:** after this stage, `homelab_ha_entity_available{entity_id,domain}` (1/0) and
`homelab_ha_entity_last_changed_seconds{entity_id,domain}` appear in VictoriaMetrics for the current
HA scope, capped at the configured series budget.

**Spec references:** master spec §6.2 (HA-derived metrics), EPIC-005 §"Signal-source matrix" (entity
availability via REST `/api/states`). Scenario #1.

## Concrete deliverables

### 1. Entity-availability collector

A `BaseCollector` in `integrations/homeassistant/` (interval 30s per the locked default). Calls
`ctx.ha.get_states()`. For each entity emits `homelab_ha_entity_available{entity_id,domain}` = 1 when
state is a real value, 0 when state is `unavailable`/`unknown`. Also emits
`homelab_ha_entity_last_changed_seconds{entity_id,domain}` = now − `last_changed` (the staleness
signal that STAGE-005-015's stale/frozen-entity rule consumes). Applies the cardinality cap from
005-004.

### 2. Domain labelling + filtering

Derive `domain` from the `entity_id` prefix. Design decides whether to emit for ALL domains or a
configurable allow/deny (to manage cardinality alongside the cap). The cap is the safety net; a
sensible domain filter is the first-line control.

### 3. Tests

Mock `get_states()` with available/unavailable/unknown entities → assert metric values + labels;
assert `last_changed_seconds` computed in UTC; assert over-cap behavior; assert HA-unreachable yields a
failed `CollectorResult` (not a crash).

## Out of scope for this stage

- The unavailable/stale ALERT RULES (STAGE-005-015) — this stage emits metrics only.
- Battery / updates / automation specifics (their own stages).
- Device-level rollup (#7) — that's a rule in 005-015 grouping these per-device.

### Design Decisions (resolved 2026-06-11)

Brainstormer (Opus) resolved the open D-* plus 3 cross-stage tensions surfaced by the Look-Ahead.
Resolved autonomously per the session operating mode (backend stage — no user prompt).

**D-ENTITY-DOMAIN-FILTER → CHOSEN: allow-list (default real-device domains) + optional deny refinement;
keep the 500 cap default (NO hard-coded per-family override).**
- Filter is the FIRST-LINE cardinality control (per card); the 005-004 cap is the safety net. HA has
  ~2000 entities × 2 metrics (~4000 series) vs a 500 default — a domain filter must reduce the real set
  under the cap on a typical homelab.
- `DEFAULT_DOMAIN_ALLOW` (module constant): `sensor, binary_sensor, switch, light, climate, lock, cover,
  fan, device_tracker, media_player, vacuum, water_heater, humidifier, alarm_control_panel, camera,
  siren, valve`. Read via `getattr(ctx.config, "ha_domains_allow", DEFAULT_DOMAIN_ALLOW)` +
  `getattr(ctx.config, "ha_domains_deny", [])`; include iff `domain in allow and domain not in deny`.
- ALLOW-list over deny-list: HA's noise domains (automation/scene/script/input_*/zone/person/sun/update/
  tts/persistent_notification/...) grow per release; an allow-list fails CLOSED (new noise auto-excluded),
  a deny-list fails open. Correct default for a cardinality control.
- NO per-family cap override hard-coded in the collector — the user tunes via 005-004's
  `cardinality_caps.families.{homelab_ha_entity_available, homelab_ha_entity_last_changed_seconds}` YAML.
  Collector calls `cap_for(family)` per family (default 500 each). Avoids a magic number + respects the
  user's deliberate knob.

**D-ENTITY-UNAVAILABLE-SEMANTICS → CHOSEN:**
- Availability `homelab_ha_entity_available`: `state` (after `.strip()`) in `{"unavailable","unknown",""}`
  → 0.0; any other non-empty value → 1.0. CASE-SENSITIVE exact match (HA writes lowercase
  `STATE_UNAVAILABLE`/`STATE_UNKNOWN`); do NOT `.lower()` (a real sensor value "Unknown" must stay 1.0).
- No synthesized series for entities absent from `get_states()` (no "expected" list; VM `absent()`/015's
  `== 0 for Nm` handle that downstream).
- Staleness `homelab_ha_entity_last_changed_seconds` = `(now_utc - last_changed).total_seconds()` clamped
  ≥ 0 (clock-skew guard), emitted for ALL filtered entities INCLUDING unavailable/unknown ones (a stuck
  entity's staleness is exactly 015's stale/frozen signal). tzinfo guard: naive parse → assume UTC.
- Unparseable `last_changed` → emit availability for that entity but SKIP its staleness series; count via
  a single self-diagnostic gauge `homelab_ha_entity_last_changed_parse_errors{}` (no per-entity label —
  that would be unbounded cardinality), emitted only when >0; debug-log (not warn).

**D-DEVICE-LABEL → CHOSEN: DEFER. Emit `{entity_id, domain}` only for v1. NO `device` label, NO
device_registry fetch.**
- VERIFIED: `device_id` is NOT reliably present in `HaState.attributes` from `/api/states`; the
  entity→device map lives in HA's `core.device_registry` (a WS/registry call this stage did not budget).
  007 (battery) itself defers device-sourcing to its own design (`D-BATTERY-DEVICE-LABEL`).
- Adding `device` here = an unbudgeted second HA round-trip + join + new failure modes = scope creep the
  card explicitly fenced ("Device-level rollup #7 is a rule in 005-015"). Adding it later is an ADDITIVE
  label change (no destructive migration; 015 doesn't exist yet → no live consumers to break).
- **FLAG for 015 + 007:** these families ship `{entity_id, domain}` only. Scenario #7 device-rollup needs
  an entity→device map from `core.device_registry` (NOT from /api/states). 015 decides: (a) registry-backed
  enrichment adding a `device` label, or (b) a separate `homelab_ha_entity_device_info{entity_id,device}`
  info-metric + `* on(entity_id) group_left(device)`. Coordinate with 007's `D-BATTERY-DEVICE-LABEL`
  (same registry source — factor an entity→device helper once, at the first device-needing stage).

**D-STATE-METRIC → CHOSEN: DEFER to 016. Keep 006 to availability + last_changed (NO state metric).**
- A generic per-entity state metric is ill-defined ("home"/"heat"/"21.5" share no numeric encoding) and
  ~doubles this collector's cardinality. The USEFUL state signal for 016 is targeted binary_sensor on/off
  for safety device_classes — that targeting is 016's design work.
- **FLAG for 016:** 006 emits `homelab_ha_entity_available` (availability 1/0, NOT entity state). 016's
  safety-sensor rules need binary_sensor on/off — add a minimal targeted metric in 016 (e.g.
  `homelab_ha_binary_sensor_on{entity_id,domain,device_class}` for safety device_classes only, bounded
  cardinality), reusing 006's get_states() poll + extract_domain.

**D-SHARED-POLL-HELPER → CHOSEN: DEFER `_shared.py`; keep an inline `extract_domain` one-liner.**
- "No premature abstraction" (per STAGE-005-003): the right shared seam is only knowable once 007 is a
  second `/api/states` call-site. Keep poll/parse/filter/cap inline in `ha_entity_available.py`.
- Keep `def extract_domain(entity_id) -> str: return entity_id.partition(".")[0]` as a module-level
  one-liner (use `partition`, not `split(".")[0]`).
- **FLAG for 007:** when 007 lands (second poll consumer), extract
  `integrations/homeassistant/_shared.py` (get_states_or_error, extract_domain, domain-filter) against
  two real call-sites and refactor 006 to use it.

**Locked metric contract (for 015):** `homelab_ha_entity_available{entity_id,domain}` ∈ {0,1};
`homelab_ha_entity_last_changed_seconds{entity_id,domain}` ≥ 0. Plus capper auto-emits
`homelab_metric_family_dropped_series{family}` for both, and a self-diagnostic
`homelab_ha_entity_last_changed_parse_errors{}`.

**Build scope:** new collector module `plugins/collectors/integrations/homeassistant/ha_entity_available.py`
(`HaEntityAvailableCollector`, name `ha_entity_available`, interval 30s, timeout 10s,
concurrency_group `homeassistant`); append to `_HA_COLLECTORS` (2 lines in `__init__.py`); tests
`tests/test_ha_entity_available_collector.py` (mixed states, domain extraction+filter, UTC staleness,
all-entities-incl-unavailable, parse-error path, over-cap+suggestion, HaError→failed run, ctx.ha None →
failed run). 100% branch coverage.

**Refinement-driven design revision (D-ENTITY-DOMAIN-FILTER empirically corrected):** the Design assumed
the domain filter alone would land the real set under the 500 cap. Reality: 1906 filtered > 500, so the
cap fired and dropped 1406 series + emitted a suggestion on first run — a poor out-of-box experience
(the same UX the rejected "no filter" Option A was meant to avoid). FIX (config-layer, no collector
change): shipped a built-in per-family default cap of **2500** for `homelab_ha_entity_available` and
`homelab_ha_entity_last_changed_seconds` (`_DEFAULT_CARDINALITY_FAMILIES` in `kernel/config.py`; the
loader MERGES it under user YAML so user per-family overrides still win and the global default stays
500). RE-VALIDATED against real HA with the rebuilt image: `cap_for(...)` returns 2500; all **1906
entities emit, 0 dropped, 0 suggestions**, `result.ok=True`. The domain filter + 2500 default covers a
typical large HA out of the box; the cap remains a safety net for pathological installs (still fires +
suggests above 2500) and is fully user-tunable. ~5000 HA series is negligible for VictoriaMetrics.
