# STAGE-005-017: HAPushChannel

## Status: Complete

## Overview

Implement the Home Assistant push channel: a dispatcher channel that delivers alerts to HA's mobile
notify service (`notify.mobile_app_jake_s_android`), so alerts reach the user's phone. This is the
outbound half of the bidirectional integration and the first non-dashboard channel. Scenario #25.
This stage implements the channel itself; the MINIMAL severity routing in STAGE-005-018 determines
WHICH alerts reach it (error/critical only) — so the channel is not noisy. Do not read this stage as
"push every alert to the phone".

**Vertical slice:** after this stage, an alert routed to the HA push channel results in a mobile push
on the user's phone (via HA), delivered through the standard `Channel.deliver(AlertEvent)` contract.

**Spec references:** master spec §8.1 (HA push: POST `/api/services/notify/mobile_app_jake_s_android`
with `{message, title, data}`, bearer token; reuses the user's `claude_ready.sh` pattern), §3.1 (alert
dispatcher per-channel adapters), §4.4 (alert lifecycle → channels). EPIC-005 §"Routing". Scenario #25.

## Concrete deliverables

### 1. HAPushChannel implementing the `Channel` contract

A channel class with `kind = "ha_push"` implementing `async def deliver(self, event: AlertEvent) ->
None` (the real contract in `kernel/dispatch/types.py`; `InprocDashboardChannel` is the worked
example). On an `AlertFiringEvent`/`AlertResolvedEvent`, call `ctx.ha.call_service("notify",
"mobile_app_jake_s_android", {message, title, data})` (reusing the REST client from 005-001 — the
`call_service` method). Map alert fields → `{title, message, data}` per spec §8.1 (severity in the
title/data, a dashboard deep-link in `data`). Resolved alerts send a "resolved: <name>" notification.

### 2. Registration in the dispatcher

Register `HAPushChannel` in the `AlertDispatcher` channel list in `lifespan.py` (today only
`InprocDashboardChannel` is wired). The minimal severity routing (005-018) determines WHICH alerts
reach it; this stage makes the channel exist + deliver. The device id (`mobile_app_jake_s_android`) is
config (Design: per-channel config / app config).

### 3. Offline-phone handling

HA returns 200 even when the phone is offline (the push may not reach the device). Treat a 200 as
"delivered to HA" (the channel's job ends at HA); do not retry on 200. A non-200/timeout is a delivery
failure the dispatcher records (the dispatcher already catches per-channel failures and never
re-raises).

### 4. Tests

Mock `ctx.ha.call_service` → assert firing/resolved events produce the correct notify payload
(`{title, message, data}`); assert HA 5xx is caught + recorded (no crash); assert the token isn't
logged. 3b: a real alert delivers a real push.

## Decisions in this stage

### Design Decisions (RESOLVED)

**Codebase facts (verified):** Channel Protocol (kernel/dispatch/types.py) = `kind: ClassVar[str]` + `async def deliver(self, event: AlertEvent) -> None` (MAY raise; the dispatcher catches per-channel, increments `_delivery_failures[kind]`, records `DeliveryResult(ok=False, error=str(exc))`, NEVER re-raises). Worked example: `InprocDashboardChannel(broker)` (kind="inproc_dashboard"), constructor injects collaborators, built inline in lifespan.py ~653. The shared `HomeAssistantRestClient` is built in lifespan ~611-616 (BEFORE the dispatcher) and has `call_service(domain, service, data) -> HaServiceResult | HaError` (client.py:156; returns HaError, never raises, never logs token/payload). Alert events: `labels` carries the raw Alertmanager labels verbatim (alertname, severity, source_tool, integration, target_kind); `annotations` carries summary/description/explorer verbatim. `get_public_url()` (config.py:197, env HOMELAB_MONITOR_PUBLIC_URL) returns the external base URL or None. `/alerts/active` is the stable UI alerts route.

**D-HAPUSH-PAYLOAD → CHOSEN: annotation-first mapping with label fallbacks.**
- `alertname = event.labels.get("alertname") or "alert"`; `summary = event.annotations.get("summary") or alertname`; `desc = event.annotations.get("description") or summary`.
- `title = f"[{event.severity.value.upper()}] {summary}"`.
- `message = desc`.
- `data = {"tag": event.fingerprint, "severity": event.severity.value, "group": event.labels.get("integration") or event.source_tool}` plus `"url": <deep_link>` ONLY when a deep-link can be built. Deep-link: if `get_public_url()` is non-None AND `annotations.get("explorer")` present → `public_url.rstrip("/") + explorer`; else if `get_public_url()` non-None → `public_url.rstrip("/") + "/alerts/active"`; else OMIT `url` (never emit a bare relative path in a push).
- `tag = fingerprint` gives HA dedup/replace (a re-firing alert UPDATES the existing notification instead of stacking — anti-noise). Reuses the existing `annotations.explorer` deep-link convention (015/016) with zero new rule authoring. Degrades cleanly when annotations are empty (synthetic / failure-budget events). Rejected: severity+alertname-only (throws away description, no deep-link); single composed line with no `data` (no dedup tag → re-fires stack into spam).

**D-HAPUSH-DEVICE-CONFIG → CHOSEN: add `notify_service: str = ""` to HaConfig (env `HOMELAB_MONITOR_HA_NOTIFY_SERVICE`), empty default → NO-OP skip.**
- The channel reads the resolved `notify_service`; when EMPTY, `deliver()` returns immediately without calling HA — no error, no recorded delivery failure (an unconfigured optional channel is not a failure; raising would pollute `_delivery_failures["ha_push"]` and the WARNING log on every alert in a default install). The env var holds JUST the service name (e.g. `mobile_app_jake_s_android`); the channel calls `call_service("notify", notify_service, data)` — the `notify.` domain is implied (no prefix splitting; YAGNI).
- Open-source-safe default: ship `""` (NOT Jake's device id) so the public release is a silent no-op that won't push to a nonexistent device; the homelab override sets `HOMELAB_MONITOR_HA_NOTIFY_SERVICE=mobile_app_jake_s_android`. Matches the established HaConfig env-only pattern (the notify target is HA-instance-scoped deployment config, alongside base_url). Rejected: a separate HaPushConfig dataclass (heavier; one string doesn't earn its own loader — YAGNI); env-read inline in lifespan (scatters env-reading outside the centralized config loader).

**D-HAPUSH-RESOLVED → CHOSEN: resolved always notifies (for alerts that reach the channel), tag-replace.**
- On `AlertResolvedEvent`: `title = f"[RESOLVED] {summary or alertname}"`, `message = f"{summary or alertname} resolved"`, `data = {"tag": event.fingerprint, "severity": event.severity.value}`. Reusing `tag=fingerprint` makes the resolved push REPLACE the still-showing firing notification (clean close-the-loop; net-quieter than leaving a stale firing card). Noise control belongs at 018's routing gate (which alerts reach the channel at all), NOT at the resolve step — once an alert was important enough to push on firing, suppressing its resolve leaves an incomplete story + a stale card. Rejected: critical-only resolved (asymmetric; leaves non-critical firing cards lingering forever).

**D-HAPUSH-ERROR-DETAIL → CHOSEN: deliver raises a plain `RuntimeError` with a token-safe reason+status string.**
- `result = await client.call_service("notify", notify_service, data)`; `if isinstance(result, HaError): raise RuntimeError(f"ha_push delivery failed: {result.reason} status={result.status}")`. Uses only `reason` (enum) + `status` (int|None) — structurally incapable of carrying the token, so even a future HaError.message regression can't leak through. The dispatcher logs `str(exc)` at WARNING + records it in DeliveryResult; both see only the token-safe string. Plain RuntimeError (the dispatcher only consumes `str(exc)` + the `kind` counter; no caller branches on exception type — a typed exception is YAGNI).

**D-HAPUSH-SEVERITY-IN-PAYLOAD → CHOSEN: surface `event.severity` (canonical 3-value) only.** Do NOT duplicate the raw `labels["severity"]` in the push (showing a tier like "error" the system doesn't route on is confusing; the raw label remains in event.labels for the UI detail view). `data["severity"] = event.severity.value`.

**★ FLAG for STAGE-005-018 (severity-coercion contract) — NOT a 017 fix:** the `Severity` StrEnum is ONLY {info, warning, critical} (NO `error`). vmalert rules emit `severity: error` (e.g. HaConfigEntryError, HaBatteryCritical from 015/016), but at ingest `_resolve_severity` (alerts.py:111-118) COERCES `error → WARNING`. So by the time HAPushChannel (and any 018 routing filter) sees the event, `event.severity` is `warning`, NEVER `error`. 017 correctly consumes `event.severity` for display AND preserves the raw label in `event.labels`. **018 must decide its routing source:** routing on `event.severity in {critical}` would MISS every error-labeled rule (they arrive as warning); 018 should route on the RAW `event.labels.get("severity") in {"error","critical"}` (uncoerced) OR on `event.severity in {warning, critical}`. This is an 018 Design decision — flagged here so 018 handles it.

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-HAPUSH-STANDARD-CONTRACT** — HAPushChannel implements the standard `Channel.deliver(AlertEvent)`
  contract (so EPIC-012's future routing engine can select it without a rebuild).
- **D-HAPUSH-200-IS-DELIVERED** — HA returning 200 = delivered-to-HA (offline phone is HA's problem;
  don't retry).
- **D-HAPUSH-REUSES-REST-CLIENT** — uses the 005-001 REST client's `call_service`, not a separate HTTP
  path.
