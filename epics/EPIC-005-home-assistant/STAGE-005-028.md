# STAGE-005-028: Entity + battery drill-down widgets (frontend)

## Status: Design (Not Started)

## Overview

Add drill-down lists to the HA panel's Health tab, consuming the 027 detail endpoints.
STAGE-005-023 shipped counts-only entity-health + battery widgets (Total/Available/Unavailable counts;
Low/Critical counts) and DEFERRED the drill-lists. This stage adds, below the count tiles, a list of
the actual unavailable/stale entities (entity_id, domain, last-changed age) and the actual low/critical
battery devices (device, level), so the user can drill from "984 unavailable" into WHICH entities.

**Vertical slice:** after this stage, clicking/expanding the entity-health widget reveals the list of
unavailable entities; the battery widget reveals the low/critical devices — the user can answer "which
entities are down?" not just "how many".

**Scope extended (2026-06-13)** to also cover updates / config-entry / repairs drill-lists on the Status tab, deferred from STAGE-005-024 (notification bodies are STAGE-005-029).

**Spec references:** the deferred work from STAGE-005-023 Design Notes; the 027 detail endpoints;
UI pattern mirrors the Docker container list + the existing PanelSection/`<dl>` widgets from 023.

## Concrete deliverables

### 1. Data hooks

`useHomeAssistantEntities()` + `useHomeAssistantBatteries()` in `apps/ui/src/api/home_assistant.ts`
(mirroring `useHomeAssistantSummary`), hitting the 027 endpoints:
- `useHomeAssistantEntities(filter?: 'unavailable')` → `useQuery` + `apiClient.GET(...)` +
  `unwrap<HaEntityRowsResponse>` + `refetchInterval: 30_000`.
- `useHomeAssistantBatteries(filter?: 'low_or_critical')` → `useQuery` +
  `apiClient.GET(...)` + `unwrap<HaBatteryRowsResponse>` + `refetchInterval: 30_000`.

### 2. Entity drill-list widget

A presentational component `EntityDrillListWidget` (or nested in `EntityHealthWidget`) that renders a
list of unavailable/stale entities below the count tiles:
- Table/list of entity_id, domain, last_changed_age_seconds (formatted as age string like "2 hours ago").
- Empty state: "No unavailable entities" (affirmative, calm).
- Loading + error states (inherit from the tab-level banner pattern in 023).
- Desktop + Mobile reflow.

### 3. Battery drill-list widget

A presentational component `BatteryDrillListWidget` (or nested in `BatteryWidget`) that renders a list
of low/critical devices below the count tiles:
- Table/list of device, level (as percentage or bare number), severity tint (critical RED, low AMBER).
- Empty state: already satisfied by 023's "All batteries healthy" EmptyState (or extend if needed).
- Loading + error states (inherit from tab-level).
- Desktop + Mobile reflow.

### 4. EDIT HomeAssistantHealthTab

Update `apps/ui/src/routes/integrations/HomeAssistantHealthTab.tsx`:
- Call `useHomeAssistantEntities()` + `useHomeAssistantBatteries()` (or leverage a single summary hook
  + refetch detail in parallel, per Design).
- Render the drill-lists below the existing count-tile PanelSections.
- Connection state banner remains at tab level (no change from 023).

### 5. Updates drill-list widget — collapsible list of update.* entities with updates available (title, installed → latest version, release link), added below the Updates count tiles on the Status tab. Grouped/collapsible (can be many, ~41 on the live install). Consumes 027's `/updates`. (Deferred from STAGE-005-024.)

### 6. Config-entry drill-list widget — list of failed/retrying integrations (domain, state), added below the Integration-status count tiles. Consumes 027's `/config-entries?filter=error`. (Deferred from STAGE-005-024.)

### 7. Repairs drill-list widget — list of active repairs (domain, summary), added to the Integration-status section. Consumes 027's `/repairs`. (Deferred from STAGE-005-024.)

### 8. Tests

Render with mocked entity/battery/updates/config-entry/repairs detail data → assert rows render (entity_id/domain/age; device/level; update versions/link; config-entry domain/state; repair domain/summary).
Empty states. Loading + error states. Desktop + Mobile layout. Session-auth (401). VM-down (502).

## Out of scope for this stage

- The detail endpoints (027) and Grafana (026).
- A separate "all entities" browser/search page (this is the unavailable/low drill-list only).
- Pagination (unless Design adds it).

## Decisions in this stage

### Design phase will confirm

- **D-DRILL-EXPAND** — always-visible list vs expand/collapse under the count tile vs "view all" link to
  a fuller list.
- **D-DRILL-CAP** — top-N + "and X more" given the unavailable subset can be large (~984); how to
  present the cap.

### Locked Design Decisions (inherited from STAGE-005-023)

- **D-DRILL-CONSUMES-027** — the drill-lists consume the verified 027 detail-endpoint contract.
- **D-WIDGETS-SPLIT-SMALL** — drill-down is its own stage (kept separate from 023's counts to keep
  stages small).
