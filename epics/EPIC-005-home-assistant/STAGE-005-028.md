# STAGE-005-028: Entity + battery drill-down widgets (frontend)

## Status: Complete

## Overview

Add drill-down lists to the HA panel's Health tab, consuming the 027 detail endpoints.
STAGE-005-023 shipped counts-only entity-health + battery widgets (Total/Available/Unavailable counts;
Low/Critical counts) and DEFERRED the drill-lists. This stage adds, below the count tiles, a list of
the actual unavailable/stale entities (entity_id, domain, last-changed age) and the actual low/critical
battery devices (device, level), so the user can drill from "984 unavailable" into WHICH entities.

**Vertical slice:** after this stage, clicking/expanding the entity-health widget reveals the list of
unavailable entities; the battery widget reveals the low/critical devices — the user can answer "which
entities are down?" not just "how many".

**Scope extended (2026-06-13)** to also cover updates / config-entry / repairs drill-lists on the Status tab, deferred from STAGE-005-024 (notification bodies are STAGE-005-029).

**Spec references:** the deferred work from STAGE-005-023 Design Notes; the 027 detail endpoints;
UI pattern mirrors the Docker container list + the existing PanelSection/`<dl>` widgets from 023.

## Decisions in this stage

### Design phase will confirm

- **D-DRILL-EXPAND** — always-visible list vs expand/collapse under the count tile vs "view all" link to
  a fuller list.
- **D-DRILL-CAP** — top-N + "and X more" given the unavailable subset can be large (~984); how to
  present the cap.

### Locked Design Decisions (inherited from STAGE-005-023)

- **D-DRILL-CONSUMES-027** — the drill-lists consume the verified 027 detail-endpoint contract.
- **D-WIDGETS-SPLIT-SMALL** — drill-down is its own stage (kept separate from 023's counts to keep
  stages small).

### Design Notes (2026-06-13) — decisions made

Backend contract VERIFIED against the live 027 router + generated `schema.ts` (all 5 row/envelope
types present and current). Design grounded in the existing `Ha*Widget.tsx` + `PanelSection` siblings.

- **D-DRILL-EXPAND → always-visible (CHOSEN).** Each drill-list renders always-visible in the SAME
  PanelSection below its count tiles (preserves the at-a-glance "which entities are down?" goal; no
  click friction; single-user dashboard). The entities list — the only list that can hit the 100-row
  server cap — gets a `max-h-80 overflow-y-auto` scroll container so it never dominates the tab; the
  cap is applied generically (any list exceeding a row threshold), not special-cased to entities.
  *Rejected:* expand/collapse disclosure (adds a click before the data; pointless ceremony on the
  near-empty lists like repairs=0). *Rejected:* "view all" link to a separate page (no such page
  exists; building one is out of scope).
- **D-DRILL-CAP → conditional footnote caption (CHOSEN).** When `total > returned`, render a muted
  caption BELOW the list: `Showing {returned} of {total} — stalest first` (the ordering suffix is
  parameterized per list; only entities is "stalest first"). The four naturally-bounded lists
  (`total == returned`) show no caption. *Rejected:* silent truncation (misleads the user about the
  true scale of what's down — a correctness failure for a monitoring tool). *Rejected:* client-side
  "show more" pagination (infeasible — the 027 endpoint caps at 100 with no offset param).
- **D-DRILL-ROW-SHAPE → one generic `<DrillList<T>>` shell + per-list `renderRow` (CHOSEN).** A single
  presentational shell owns the list chrome (empty state, conditional cap caption, max-h scroll, row
  dividers); each list supplies a tiny `renderRow`. *Rejected:* 5 bespoke widget files mirroring the
  existing flat `<dl>` siblings (copy-pastes the empty/cap/scroll chrome 5× → drift risk; the existing
  4 count widgets are flat only because they had nothing to share — the 5 drill-lists genuinely share
  chrome). Row rule: categorical state/severity → `<Badge>` (config-entries `state`, repairs
  `severity`); numeric battery `level` → bare tinted span (critical `<10` red, low `10–<20` amber,
  matching the sibling Battery count widget's tint convention). Updates `title` falls back to
  `entity_id` when empty (HA update entities often carry no title). Entities age rendered from
  `last_changed_age_seconds` via a coarse single-unit `formatAge` helper with a day tier + " ago"
  suffix (e.g. "2h ago", "3d ago"). **Planner-verify:** confirm whether to extend the existing
  `formatDuration` (`apps/ui/src/lib/relativeTime.ts`) or add a new `formatAge` — `formatDuration`
  lacks a day tier and the " ago" suffix, so multi-day-stale entities would read as "72h 0m"; and
  confirm the `Badge` component variants (`critical`/`warn`) at `apps/ui/src/components/ui/badge.tsx`.
- **D-DRILL-DATA-WIRING → pure shell + thin per-list container that owns its hook (CHOSEN).** `<DrillList>`
  stays pure-presentational (props: items, loading, error, renderRow, empty label, total/returned) and
  fully unit-testable with no hook. A thin container per list (`Ha*Drill.tsx`) calls its own hook and
  maps the `UseQueryResult` into `<DrillList>` props, handling its OWN independent
  loading/empty/error/502 state. Tabs reduce to a clean grid of `<PanelSection><Ha*Drill/></PanelSection>`.
  *Rejected:* tab-owns-all-hooks (the existing summary convention) — that worked for ONE query with one
  shared banner, but this stage adds FIVE independent queries each with its own 502; threading 5 state
  triples through the tab would bloat and obscure both tabs.

**5 new hooks** in `apps/ui/src/api/home_assistant.ts` mirror `useHomeAssistantSummary` exactly (own
queryKey under `haQueryKeys`, `unwrap<...RowsResponse>`, `apiClient.GET(...)`, `refetchInterval: 30_000`):
`useHomeAssistantEntities`, `useHomeAssistantBatteries`, `useHomeAssistantUpdates`,
`useHomeAssistantConfigEntries`, `useHomeAssistantRepairs`. The 5 response/row types already exist in the
generated `schema.ts` (re-export in `routes/integrations/types.ts` per the existing pattern).

**Deferral-coordination (NO scope added here):** the thin fields argue for prioritizing the backend
enrichment stages, they do NOT justify a frontend live-HA call (would violate the VM-only architecture):
updates `title`/version → STAGE-005-031; battery `device` friendly-name → STAGE-005-031; repair `summary`
→ STAGE-005-031; precise config-entry `state` → STAGE-005-032. 028's widgets render usefully on the
AVAILABLE fields and must tolerate these fields being added later (defensive render).

**Consistency note:** the EPIC-005 table row for 028 originally described only Entity+Battery; the card's
extended scope (updates/config-entries/repairs drill-lists, deferred from STAGE-005-024 on 2026-06-13) is
documented in the card Overview. The epic row is updated in this same Design edit to reflect the full scope.
