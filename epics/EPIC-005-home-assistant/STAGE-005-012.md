# STAGE-005-012: Persistent-notifications collector (websocket)

## Status: Complete

## Overview

Surface HA's persistent notifications (the notification "bell" — HA itself flagging "X needs
attention") via the websocket. Scenario #12. These are NOT in `/api/states` and are NOT persisted to
disk (in-memory only — verified), so the websocket is the only source.

**Vertical slice:** after this stage, `homelab_ha_persistent_notification{notification_id}` reflects
current active HA notifications (title/message available to the panel), so the user sees in the
monitor whatever HA is surfacing in its own bell, feeding the integration-status widget (005-024).

**Spec references:** master spec §2 Q10 (HA health signals), EPIC-005 §"Signal-source matrix"
(notifications via websocket). Scenario #12.

## Concrete deliverables

### 1. Notification subscriber

Using the websocket client, subscribe to persistent-notification events (Design confirms the WS
command — e.g. subscribe to `persistent_notifications_updated` / the relevant event, plus an initial
list). Maintain the current active-notification set; emit
`homelab_ha_persistent_notification{notification_id}` = 1 per active notification (count + ids). Keep
title/message for the panel data endpoint (005-021) but NOT in metric labels or logs.

### 2. WS → metric bridge + privacy

Same snapshot-emit-vs-on-update bridge concern. Notification bodies are kept only in the in-memory
state surfaced to the authenticated panel endpoint — never emitted to VM labels (cardinality + privacy)
nor logged.

### 3. Tests

Fake WS notification create/dismiss → assert active set + count metric; dismiss clears it; reconnect
recovers; assert message bodies are NOT logged and NOT in metric labels.

## Out of scope for this stage

- Alerting on notifications (Design of 005-015/024 decides — likely surfaced in the panel + maybe a
  count rule, since HA notifications vary wildly in importance).
- Config-entry (005-010) and repairs (005-011).
- Creating/dismissing HA notifications from the monitor (read-only here).

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-NOTIF-VIA-WEBSOCKET** — persistent notifications require the websocket (not in `/api/states`, not

### Design Notes (resolved 2026-06-12)

Brainstormer (Opus) consulted for D-NOTIF-METRIC-SHAPE and (the key one) D-NOTIF-BODIES-STORAGE. Near-mirror
of STAGE-005-011 (repairs WS collector). Main agent made the final calls. The defining wrinkle is PRIVACY:
notification title/message must NEVER enter metric labels, event bodies, OR logs.

- **D-NOTIF-WS-COMMAND → A1 snapshot (inherited from 010/011).** Each tick: guard `_ws is None` /
  `not _ws.connected` → return ok=False (stale); else `await self._ws.send_command("persistent_notification/get")`.
  Defensive parse mirrors `_extract_issues`: handle bare list / `{"notifications":[...]}` dict / empty
  (`_extract_notifications`). NO subscription (notifications aren't sub-tick critical; snapshot RPC suffices).
  Exact response shape + any active/dismissed flag VERIFIED at 3b.

- **D-NOTIF-METRIC-SHAPE → Option A (single per-notification 1/0 gauge).**
  `homelab_ha_persistent_notification{notification_id}` = 1.0 for each active notification.
  - `notification_id` is HA's stable low-churn slug → the ONLY label. Count is derived downstream via
    `count(homelab_ha_persistent_notification)`.
  - REJECTED Option B (bare count gauge — discards per-notification identity 024 keys on). REJECTED Option C
    (both — the count is redundant, derivable from A).
  - title, message, created_at and ALL other fields → NOT labels (privacy + cardinality + hygiene).

- **D-NOTIF-BODIES-STORAGE → Option A: 012 STORES NOTHING (the key decision).** The metric is
  notification_id-only. 012 maintains NO body store — no in-memory dict, no SQLite cache, no disk
  persistence of title/message anywhere.
  - REJECTED Option C (SQLite cache like the docker panel): writing notification bodies to the monitor DB
    on disk DIRECTLY CONTRADICTS the card's "kept only in in-memory state ... never persisted to disk" and
    is a privacy regression (bodies in DB dumps/backups). The docker cache stores low-sensitivity
    operational fields (cpu/mem/status), NOT free-text — so it's not the same risk class.
  - REJECTED Option B (in-memory dict on the collector instance, read by 021 via app.state): speculative
    pre-work for an UNSTARTED stage (021) that owns its own exposure decision (D-PANEL-NOTIF-EXPOSURE).
    Anti-YAGNI; pushes a privacy contract into the wrong stage. The app.state precedent (image-update
    collector) exposes ints/counts, NOT sensitive free text.
  - **EXPLICIT DEFERRAL (recorded decision, not an omission):** body-sourcing for the 021/024 panel is
    ENTIRELY 021's problem. 021 can choose a live `persistent_notification/get` WS fetch in its request
    path, or an ephemeral store, per its D-PANEL-NOTIF-EXPOSURE. 012 emits the count/id metric and owns
    nothing about bodies.

- **D-NOTIF-SURFACE → panel-only** this stage; no alert rule (verified: 015 omits notification rules from
  its metric-rule set; a future count rule would be 015's call). The id-only metric supports a future
  `count()` alert for free.

- **D-NOTIF-PRIVACY-MECHANICS (leak-proofing — all mandatory):**
  1. ONLY `notification_id` becomes a label. A `_notification_labels(notification)` helper (mirroring
     `_issue_labels`) reads ONLY `notification_id` (returns `{"notification_id": <slug>}` or `None` to
     skip); it must NEVER read title/message/created_at into the label dict.
  2. The collector emits NO `SuggestionEvent`/`CollectorEvent` carrying body text. `run()` returns
     `events=[]` plus only whatever `CappedEmitter` appends (the cap drop-gauge event — metric name + drop
     count, never payload). This closes the `SuggestionEvent.body` leak path.
  3. `run()` logs NOTHING about parsed notifications — no debug dump of the WS result, no error string
     containing a body. Errors return generic strings ("ha websocket not configured" / "ha websocket not
     connected" / `HaError.message`, which is already body-free per the ha_websocket logging contract).
  4. `_extract_notifications(result)` defensive parse mirrors `_extract_issues` (list / dict-wrapped /
     degenerate `{}` → `[]`).

- **Required privacy test assertions (test-enforced, not prose):**
  - Label-content: run() with a fixture whose `message` is a sentinel (e.g. `"SECRET-BODY-XYZ"`) → assert
    NO recorded metric label value contains the sentinel; assert the only label key present is
    `notification_id`.
  - No-event-body: assert no `CollectorEvent` in the result carries the sentinel message/title.
  - No-log: assert the bound logger captures no record containing the sentinel (happy path AND HaError
    path).
  - Mirror-parity: None-`_ws` → ok=False; not-connected → ok=False; HaError result → ok=False;
    bare-list / dict-wrapped / empty parse → correct counts (reuse the 011 test shapes).

- **Cardinality:** global default cap (500) ample (notifications are tens at most); NO per-family override
  in `_DEFAULT_CARDINALITY_FAMILIES`.

- **3b is the load-bearing check (WS collectors):** verify the REAL `persistent_notification/get` response
  SHAPE (bare list vs `{"notifications":[...]}`), real `notification_id` slug stability, whether an
  active/dismissed flag exists, and (ideally) trigger a test notification in HA to observe ≥1 real one.
  Capture the raw frame shape, not just the metric count.
