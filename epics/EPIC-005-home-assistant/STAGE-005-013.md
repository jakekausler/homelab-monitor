# STAGE-005-013: History/anomaly z-score collector

## Status: Complete

## Overview

The heaviest collector and the clean optional cut point of the epic: query HA's Recorder/history API
for slow-moving numeric sensors (temperatures, humidities, power, CO₂), compute a rolling baseline,
and emit a z-score metric for anomaly rules. Scenario #4. This is the rolling-baseline pattern that
EPIC-005A's machine-metric anomaly rules mirror, so it establishes the lightweight (non-ML) anomaly
approach for the whole system.

**Vertical slice:** after this stage, `homelab_ha_entity_value_zscore{entity_id}` is emitted for the
configured slow-sensor set, so a vmalert rule (005-015) can fire on "this sensor is N σ off its recent
normal."

**Spec references:** master spec §6.2 (HA metrics; anomaly via rolling-baseline vmalert), §2 (anomaly
detection delegated to lightweight rolling-baseline rules, not ML). EPIC-005 §"Signal-source matrix".
Scenario #4. (EPIC-004 already shipped log-anomaly precedent.)

## Concrete deliverables

### 1. Recorder/history collector

A `BaseCollector` at the 5m interval. Calls the HA Recorder/history API
(`/api/history/period/...` — Design confirms the exact endpoint + params) for a configured set of
slow-moving numeric sensors (or auto-selected by `device_class` in {temperature, humidity, power,
carbon_dioxide, ...} and `state_class: measurement`). Maintains a rolling window/baseline (mean + std)
per sensor and emits `homelab_ha_entity_value_zscore{entity_id}` = (current − mean) / std. Guard the
zero-variance case (a frozen sensor has std≈0 → z-score undefined; emit absent/0 and let the
stale-entity rule (#2, 005-015) handle frozen sensors instead — document this division of labor).

### 2. Baseline state + windowing

Decide where the rolling baseline lives (in-memory per collector vs derived from VM via a query). Keep
it lightweight (this is NOT ML — explicitly the simple rolling z-score, distinct from EPIC-015 Netdata
anomaly). Configurable window length + min-samples-before-emitting (don't emit a z-score until the
baseline is meaningful).

### 3. Eligible-sensor selection + cardinality

Only slow-moving numeric sensors are eligible (bimodal/scheduled signals are noisy under z-score —
documented limitation). A config allow/deny + device_class heuristic picks the set; the cap (005-004)
is the safety net.

### 4. Tests

Mock history: stable sensor (z≈0), a spike (high z), a frozen sensor (zero-variance → no false z),
insufficient-samples (no emit). Assert UTC, window math, cap.

## Out of scope for this stage

- The anomaly ALERT RULE (005-015 emits the rule on this metric).
- ML/k-means behavioral anomaly — that's EPIC-015 (Netdata). This is the lightweight rolling-baseline
  pattern only.
- Stuck/frozen-sensor detection (#2) — z-score does NOT catch zero-variance; that's a separate
  staleness rule (005-015) on `last_changed_seconds` (005-006). Document the split.

### Design Notes (resolved 2026-06-12)

Brainstormer (Opus) consulted for the open decisions. The heaviest collector, but resolved to the SIMPLEST
correct design via YAGNI. Main agent made the final calls. NOTE: this is a REST collector (uses `ctx.ha`
via `get_states_or_error`), NOT a websocket collector — the exemplar to mirror is `ha_battery.py`, not the
WS collectors (010/011/012).

- **D-BASELINE-SOURCE (resolves D-HISTORY-ENDPOINT + D-BASELINE-STORAGE) → Option A: in-memory rolling
  window from `get_states()` snapshots. NO `/api/history`, NO new `get_history()` REST method.** Each 5m
  tick calls the EXISTING `ctx.ha.get_states()` (via `get_states_or_error`), extracts each eligible
  sensor's value with `parse_float_state`, appends to a per-entity `deque(maxlen=window_samples)` held as a
  collector INSTANCE VAR (verified: scheduler constructs each collector once and calls run() per tick, so
  in-memory state persists across ticks). Compute mean + population std over the window; emit z =
  (current − mean) / std once `len(window) >= min_samples`.
  - REJECTED Option B (seed from `/api/history/period` then maintain in-memory): would require writing +
    pyright-strict-typing + testing a NEW `get_history()` REST method against HA's heaviest data shape
    (list-of-lists-of-state-dicts) — permanent complexity on the stage explicitly designated the "clean
    optional cut point", paid only to remove a one-time, self-healing ~1h post-restart warmup. The binding
    contract (015) needs only the metric, not history provenance. The card's `/api/history` prose is
    aspirational, not binding.
  - REJECTED Option C (VM-query-derived baseline): requires first emitting every eligible sensor's raw
    value as a NEW metric family, then a per-tick VM query — the heaviest option, arguably belongs in
    vmalert (015), violates D-ANOMALY-LIGHTWEIGHT.
  - COLD-START: emits nothing for the first `min_samples` ticks after a restart (~1h at 12×5m). Acceptable
    for slow sensors with a `for Mm` alert window. A one-line code comment will note Option B (history-seed)
    is the deliberate future upgrade if warmup ever proves painful.

- **D-ELIGIBLE-SENSORS → Option C: device_class heuristic default + config allow/deny overrides**, via a
  SUBCLASSED `CollectorConfig` (Pydantic BaseModel, `extra="forbid"` — subclassing to add typed fields is
  the established pattern; battery uses none). The heuristic: a state is eligible iff
  `attributes.state_class == "measurement"` AND `attributes.device_class ∈ {temperature, humidity,
  pressure, power, carbon_dioxide, ...}` AND `parse_float_state(state)` is not None. Config fields:
  `extra_entity_ids: list[str] = []` (force-include) and `excluded_entity_ids: list[str] = []`
  (force-exclude). Empty overrides = pure heuristic → ships useful with zero config; the deny-list is the
  escape hatch for the noisy sensor the heuristic inevitably catches (matches the "observe by default,
  allow override" posture). `state_class == "measurement"` excludes `total`/`total_increasing` counters
  that would z-score nonsensically. The cap (500) is the cardinality safety net.

- **D-ZERO-VARIANCE → SKIP emission (no series) when `std < epsilon` (zero_variance_epsilon = 1e-9).**
  Emitting 0.0 would be a lie (a frozen sensor's value equals its mean → z = 0 reads "perfectly normal",
  actively MASKING the freeze) and risks inf/NaN poisoning the series. Skipping keeps the 015 z-score rule
  seeing only real finite z-scores, and correctly DEFERS frozen-sensor detection to the separate staleness
  rule (015 on `homelab_ha_entity_last_changed_seconds`, from 006). Mirrors battery's skip-rather-than-
  emit-a-misleading-0 precedent.

- **Min-samples / window / std (configurable defaults on the subclassed config):**
  - `min_samples: int = 12` (~1h warmup at 5m) — don't emit until the window is meaningful.
  - `window_samples: int = 48` (~4h at 5m) — `deque(maxlen=window_samples)`; balances responsiveness vs
    smoothing for slow sensors.
  - std = `statistics.pstdev` (POPULATION std), NOT sample stdev — sample std's n−1 correction inflates the
    denominator and destabilises z-scores near min_samples for small windows. Note the choice in a comment.
  - `zero_variance_epsilon: float = 1e-9`.

- **Metric:** `homelab_ha_entity_value_zscore{entity_id}` = (current − rolling_mean) / rolling_std;
  `entity_id` is the ONLY label (verified against 015's `abs(homelab_ha_entity_value_zscore) > N for Mm`
  rule contract). Value may be negative; 015 applies `abs()`.

- **Interval / concurrency / cardinality:** 5m interval (LOCKED D-ANOMALY-SLOW-INTERVAL);
  `concurrency_group="homeassistant"`; global default cap 500 ample (eligible set is small — NO per-family
  override in `_DEFAULT_CARDINALITY_FAMILIES`). Emits `homelab_collector_run_*` like all collectors.

- **Out of scope (unchanged):** the anomaly ALERT RULE (015); ML/k-means (EPIC-015); frozen-sensor
  detection (the staleness rule's job).

- **3b is the load-bearing check:** with real HA sensors at a 5m interval, the rolling window needs
  `min_samples` ticks (~1h) to warm up before ANY z-score emits — 3b must account for this (either wait
  ~1h of real ticks, or drive run() repeatedly with real get_states() snapshots to fill the window, OR
  validate the window math by injecting a sequence of real snapshots). 3b confirms: the eligible-sensor
  heuristic selects a sane real set (count, which device_classes), a stable sensor → z≈0, and that no
  frozen/zero-variance sensor emits a garbage z. Measure the real eligible-set cardinality vs the 500 cap.
