# STAGE-005-013: History/anomaly z-score collector

## Status: Design (Not Started)

## Overview

The heaviest collector and the clean optional cut point of the epic: query HA's Recorder/history API
for slow-moving numeric sensors (temperatures, humidities, power, CO₂), compute a rolling baseline,
and emit a z-score metric for anomaly rules. Scenario #4. This is the rolling-baseline pattern that
EPIC-005A's machine-metric anomaly rules mirror, so it establishes the lightweight (non-ML) anomaly
approach for the whole system.

**Vertical slice:** after this stage, `homelab_ha_entity_value_zscore{entity_id}` is emitted for the
configured slow-sensor set, so a vmalert rule (005-015) can fire on "this sensor is N σ off its recent
normal."

**Spec references:** master spec §6.2 (HA metrics; anomaly via rolling-baseline vmalert), §2 (anomaly
detection delegated to lightweight rolling-baseline rules, not ML). EPIC-005 §"Signal-source matrix".
Scenario #4. (EPIC-004 already shipped log-anomaly precedent.)

## Concrete deliverables

### 1. Recorder/history collector

A `BaseCollector` at the 5m interval. Calls the HA Recorder/history API
(`/api/history/period/...` — Design confirms the exact endpoint + params) for a configured set of
slow-moving numeric sensors (or auto-selected by `device_class` in {temperature, humidity, power,
carbon_dioxide, ...} and `state_class: measurement`). Maintains a rolling window/baseline (mean + std)
per sensor and emits `homelab_ha_entity_value_zscore{entity_id}` = (current − mean) / std. Guard the
zero-variance case (a frozen sensor has std≈0 → z-score undefined; emit absent/0 and let the
stale-entity rule (#2, 005-015) handle frozen sensors instead — document this division of labor).

### 2. Baseline state + windowing

Decide where the rolling baseline lives (in-memory per collector vs derived from VM via a query). Keep
it lightweight (this is NOT ML — explicitly the simple rolling z-score, distinct from EPIC-015 Netdata
anomaly). Configurable window length + min-samples-before-emitting (don't emit a z-score until the
baseline is meaningful).

### 3. Eligible-sensor selection + cardinality

Only slow-moving numeric sensors are eligible (bimodal/scheduled signals are noisy under z-score —
documented limitation). A config allow/deny + device_class heuristic picks the set; the cap (005-004)
is the safety net.

### 4. Tests

Mock history: stable sensor (z≈0), a spike (high z), a frozen sensor (zero-variance → no false z),
insufficient-samples (no emit). Assert UTC, window math, cap.

## Out of scope for this stage

- The anomaly ALERT RULE (005-015 emits the rule on this metric).
- ML/k-means behavioral anomaly — that's EPIC-015 (Netdata). This is the lightweight rolling-baseline
  pattern only.
- Stuck/frozen-sensor detection (#2) — z-score does NOT catch zero-variance; that's a separate
  staleness rule (005-015) on `last_changed_seconds` (005-006). Document the split.

## Decisions in this stage

### Design phase will confirm

- **D-HISTORY-ENDPOINT** — exact Recorder/history API endpoint + query shape; how much history per
  poll.
- **D-BASELINE-STORAGE** — in-memory rolling window vs VM-query-derived baseline.
- **D-ELIGIBLE-SENSORS** — config allow/deny + device_class heuristic for which sensors get a z-score.
- **D-ZERO-VARIANCE** — explicit handling so frozen sensors don't emit garbage z-scores (defer to the
  staleness rule).

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-ANOMALY-LIGHTWEIGHT** — rolling-baseline z-score (the EPIC-004 log-anomaly precedent), NOT ML;
  ML behavioral anomaly is EPIC-015.
- **D-ANOMALY-OPTIONAL-CUT** — this is the heaviest stage and the clean optional cut point if the epic
  must shrink; everything before it is independently useful.
- **D-ANOMALY-SLOW-INTERVAL** — history/anomaly polls at 5m (vs 30s for availability).
