# STAGE-005-040: Anomaly surface decision: minimal UI OR document Grafana-as-home

## Status: Not Started

## Overview

`homelab_ha_entity_value_zscore` (the z-score anomaly collector, STAGE-005-013) has NO API and NO UI — it is surfaced only on the Grafana "Top Anomaly z-scores" panel (dashboard line 860). This is an "implemented without a user-facing location" gap. This stage CLOSES the gap by DECISION: either (a) add a minimal anomaly detail endpoint + a small UI surface (a Status-tab "Top anomalies" widget mirroring the existing drill pattern), OR (b) formally DOCUMENT Grafana as the intended anomaly home (the panel already exists and Grafana is the right tool for time-series anomaly inspection) and record that decision so the gap is closed-by-decision rather than left ambiguous. This is the lowest-priority stage and is explicitly framed as a surface-decision stage; the likely outcome is (b) (document Grafana-as-home) given anomaly inspection is inherently time-series and Grafana already serves it — but Design makes the call. Soft dependency on STAGE-005-037 only because the registry-eligible entity set should be settled before deciding whether a dedicated anomaly UI is worth building.

**Depends on:** STAGE-005-037 (soft — registry-eligible anomaly set)

## Stage Flags

- Has Input Forms: No.
- Touches secrets / production data: No.
- Host-integration concerns: If UI chosen — yes (endpoint over real z-score metrics) → 3a + 3b for the endpoint, 3a Desktop+Mobile for the widget. If docs chosen — No (doc-only; no rig refinement).

## Concrete deliverables

**If D-ANOMALY-SURFACE → document Grafana-as-home (likely):**
1. Document in the HA integration panel / docs that Grafana's "Top Anomaly z-scores" panel is the intended anomaly home (and why: anomaly inspection is time-series-native). Add a link/affordance from the HA panel to the Grafana anomaly panel if cheap (the Metrics-page HA tab from 030 already embeds the dashboard — confirm the anomaly panel is reachable there).
2. Record the closed-by-decision rationale in the card + a short note in epic docs / CHANGELOG.

**If D-ANOMALY-SURFACE → minimal UI:**
1. Anomaly detail endpoint (e.g. `GET /api/integrations/home-assistant/anomalies`) returning top-N z-score rows (entity_id, zscore) from VM per-series (mirror `/entities` VM-selection + `_sample_float`). New response model → OpenAPI regen → schema.ts → drill test-mock cascade.
2. Status-tab "Top anomalies" widget + drill (mirror `HaEntitiesDrill`); new `useHomeAssistantAnomalies` hook + types.ts export + tests.

## Decisions in this stage

### Design phase will confirm

- D-ANOMALY-SURFACE — minimal anomaly UI vs document Grafana-as-home. THE stage decision. (Lean: document Grafana-as-home — lowest cost, correct tool; build UI only if the user wants in-app anomaly browsing.)
- D-ANOMALY-GRAFANA-LINK — if documenting, whether to add an in-app link/affordance from the HA panel to the Grafana anomaly panel (cheap nicety).
- D-ANOMALY-UI-SCOPE — if building UI, top-N count + threshold for "anomaly" display.
