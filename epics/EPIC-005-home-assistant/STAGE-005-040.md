# STAGE-005-040: Anomaly surface decision: minimal UI OR document Grafana-as-home

## Status: Complete

## Overview

`homelab_ha_entity_value_zscore` (the z-score anomaly collector, STAGE-005-013) has NO API and NO UI — it is surfaced only on the Grafana "Top Anomaly z-scores" panel (dashboard line 860). This is an "implemented without a user-facing location" gap. This stage CLOSES the gap by DECISION: either (a) add a minimal anomaly detail endpoint + a small UI surface (a Status-tab "Top anomalies" widget mirroring the existing drill pattern), OR (b) formally DOCUMENT Grafana as the intended anomaly home (the panel already exists and Grafana is the right tool for time-series anomaly inspection) and record that decision so the gap is closed-by-decision rather than left ambiguous. This is the lowest-priority stage and is explicitly framed as a surface-decision stage; the likely outcome is (b) (document Grafana-as-home) given anomaly inspection is inherently time-series and Grafana already serves it — but Design makes the call. Soft dependency on STAGE-005-037 only because the registry-eligible entity set should be settled before deciding whether a dedicated anomaly UI is worth building.

## Design Phase

**Status**: [x] Design (Complete)

### Design Notes

**D-ANOMALY-SURFACE → DOCUMENT GRAFANA-AS-HOME (locked).** The z-score anomaly metric `homelab_ha_entity_value_zscore` stays surfaced via the existing Grafana "Top Anomaly z-scores" panel; we do NOT build an in-app anomaly endpoint/widget. Rationale:
- **No real unreachable gap.** Verified this Design: the in-app Metrics page HA tab (`apps/ui/src/routes/metrics/MetricsHomeAssistantTab.tsx`, route `/metrics/home-assistant`) ALREADY embeds the FULL Grafana `home-assistant` dashboard in kiosk mode (`/api/grafana/d/home-assistant/home-assistant?kiosk`), so the anomaly panel (id=19) is ALREADY reachable in-app by scrolling. The only genuine gap is that it is not EXPLICITLY designated/linked as the anomaly home — which this stage closes.
- **Drill-pattern mismatch (decisive).** The other HA detail surfaces (entities/batteries/updates/repairs) are point-in-time STATUS lists — "what is true now." An anomaly is a TEMPORAL phenomenon — "is this value's trajectory abnormal vs its rolling baseline." A top-N z-score table is strictly WORSE than the existing time-series panel at answering that. The z-score is a slow (5-min, rolling-baseline) metric; time-series is the correct visualization.
- **YAGNI / cost.** Option A (endpoint + Pydantic model + OpenAPI regen + schema.ts + drill-test-mock cascade + hook + types + widget + drill + tests — ~8 files + dual-signoff refinement) would build an INFERIOR DUPLICATE of an existing surface, for the epic's LOWEST-priority stage. Rejected.
- Option A would only be right if the user had a stated requirement for native in-app anomaly browsing without leaving the integration page — no such requirement exists; the user delegated the call.

**D-ANOMALY-GRAFANA-LINK → YES, add a cheap in-app deep-link affordance (locked).** Pure docs would make the stage a near-no-op and not improve single-user discoverability. Add a small SECONDARY "View anomalies in Grafana →" link on the HA Status tab (`apps/ui/src/routes/integrations/HomeAssistantStatusTab.tsx`) that routes to the in-app Metrics HA tab (`/metrics/home-assistant`), ideally focused on the anomaly panel via Grafana's `?viewPanel=19` (the Grafana proxy `kernel/api/routers/grafana.py` passes querystrings through). Render it as a secondary link, NOT a card/drill, so it reads "this lives in Grafana" rather than mimicking a native data widget.
- **Build wrinkle to resolve:** the embed URL is currently hardcoded `?kiosk` in `MetricsHomeAssistantTab.tsx`. For `?viewPanel=19` to take effect the Metrics tab must forward a search param into the iframe src (or the link targets a URL the tab reads). Validate `viewPanel`-in-kiosk focus behavior on the real rig in Refinement 3a; FALLBACK (still a complete deliverable): link to `/metrics/home-assistant` and name the panel, if kiosk+viewPanel doesn't focus cleanly.

**D-ANOMALY-UI-SCOPE → N/A.** No endpoint, no hook, no drill, no types, no OpenAPI/schema regen. Link affordance + docs only.

**This is NOT a deferral.** Option A (the in-app anomaly widget) was EVALUATED and REJECTED as the wrong tool (time-series belongs in Grafana), not deferred — nothing is dropped, so no tracking stage is needed.

**Backend contract verified (Design):** z-score metric LIVE in VM — 78 series, label `entity_id` only, real anomalies present now (heater power |z|=4.36, AC power 4.09, balance power 4.02). Grafana panel id=19 (`topk(8, abs(homelab_ha_entity_value_zscore))`), dashboard uid `home-assistant`, gridPos y=49 (post-038 shift). 041 (final dashboard review) will inventory/polish this panel regardless of 040 — no hard dependency either way.

**Deliverables (Build):** (1) docs note designating Grafana as the anomaly home (a real docs file — Build chooses the cheapest correct location, e.g. an HA integration/monitoring doc); (2) the in-app deep-link affordance in `HomeAssistantStatusTab.tsx`; (3) CHANGELOG entry. **FRONTEND stage** → Refinement is 3a Desktop+Mobile USER PROMPT (the deep-link is a real UI change); frontend-only, NO 3b (no host-integration delta). **Reverted (anomaly deep-link, removed):** `apps/ui/src/router.tsx` (validateSearch removed), `apps/ui/src/routes/metrics/MetricsHomeAssistantTab.tsx` (useSearch + conditional src removed — back to the static kiosk embed), `apps/ui/src/routes/integrations/HomeAssistantStatusTab.tsx` (Link import + footer affordance removed), both test files restored to their simple pre-040 form (no RouterProvider wrapper), and the new `docs/integrations/home-assistant.md` DELETED. The UI is back to its pre-040 state.

**Delivered instead (the actual functional fix — pre-existing `/metrics/*` SPA-refresh 404 bug):** Hard-refreshing any `/metrics/*` client route (e.g. `/metrics/system`, `/metrics/home-assistant`) returned `{"error":{"code":"not_found","message":"Not Found","details":null}}` instead of the app. Root cause: the SPA catch-all guard in `apps/monitor/homelab_monitor/kernel/api/app.py` used `spa_path.startswith(("api/", "metrics"))` — `"metrics/system".startswith("metrics")` is True, so the WHOLE `/metrics/*` client subtree 404'd. The intent was to protect only the EXACT backend Prometheus scrape path `/metrics` (owned by `observability.router`, registered before the catch-all, so the prefix match was both wrong AND unnecessary). Fixed to `spa_path.startswith("api/") or spa_path == "metrics"`. The stale test `test_spa_fallback_rejects_metrics_misses` (which codified the bug) was repurposed to `test_spa_fallback_serves_metrics_client_routes` asserting `/metrics/system` + `/metrics/home-assistant` now return 200 index.html.

**Real-rig validation (3a, user-approved Desktop + Mobile):** monitor rebuilt + recreated; data-level checks: `/metrics/system` → 200 text/html (was JSON 404), `/metrics/home-assistant` → 200 text/html, exact `/metrics` → 200 Prometheus text (scrape endpoint intact), `/api/<nonexistent>` → 404 JSON (API 404 preserved). User confirmed the refresh fix works and the link is gone, and APPROVED both viewports. `make verify` GREEN (backend 4909 pass / 1 skip / 100% kernel coverage; vitest 1527 pass incl. the reverted-to-simple UI tests; tsc/eslint/UI-build clean). Frontend-only revert → 3a only, no 3b.
