## Overview

## Concrete deliverables

### 1. Push-back firer

A mechanism that, on a monitor alert firing (and optionally resolving), POSTs to HA's event bus:
`ctx.ha`-based call to `/api/events/<event_type>` (the REST client gets a `fire_event(event_type,
data)` method, or reuses `call_service`/a direct POST â€” Design confirms). The event carries the alert
fingerprint, name, severity, and a deep-link in its data so HA automations can branch. Default
`event_type` like `homelab_alert` (configurable).

### 2. Per-alert opt-in configuration

Push-back is OFF by default; it's enabled per alert (via routing config / a tag / the alert's
labels â€” Design decides, consistent with the minimal routing from 005-018 and forward-compatible with
EPIC-012's routing). Only alerts explicitly opted in fire HA events (no surprise automation triggers).

### 3. Tests

Mock the HA event POST â†’ assert a firing alert with the opt-in fires the correct event payload; a
non-opted-in alert does NOT; HA 5xx is caught/recorded (no crash); token not logged. 3b: a real HA
automation listens for the event and reacts.

## Design Phase

### Design Notes (decided 2026-06-12)

A new outbound dispatch Channel `HAEventChannel` (kind="ha_event"), mirroring 017's HAPushChannel,
wired into the AlertDispatcher. On an opted-in alert firing/resolving, it POSTs an event to HA's event
bus (`POST /api/events/<event_type>`) so HA automations can react. The channel reuses the dispatcher's
failure-isolation + the 018 `accepts()` gate; EPIC-012 can later select it via routing_rules.

**D-PUSHBACK-SHAPE â†’ CHOSEN: new `HAEventChannel` (kind="ha_event") in the dispatcher.** The only trigger
is an alert firing/resolving, which is exactly what `AlertDispatcher.dispatch(event)` already fans out â€”
so a channel is the minimal correct shape and the clean mirror of 017. Gets firing AND resolved events
for free, reuses per-channel failure isolation (one channel raising never breaks others; dispatcher
catches + records a DeliveryResult) and the mandatory `accepts()` gate. Rejected: firing outside the
dispatcher (would duplicate the firing/resolved fan-out + re-implement failure isolation for no benefit).

**D-PUSHBACK-MECHANISM â†’ CHOSEN: add `fire_event(event_type: str, data: dict[str, str]) -> None | HaError`
to `HomeAssistantRestClient`, success = `None`.** Mirrors `call_service` but POSTs `/api/events/<event_type>`
via the existing `_request` helper (connect/timeout â†’ HaError already mapped; token via token_provider in
the Authorization header, NEVER logged). HA returns 200 on success; a non-200 â†’ `HaError(reason="http_error",
status=...)` (token-safe message = method+path+status only). Success returns `None` â€” nothing consumes the
fire-event response body (`{"message": "..."}`); the channel only branches `if isinstance(result, HaError):
raise`, and `None`-on-success is the tightest path for 100% coverage (no unused result-dataclass fields, no
body-parse â†’ no `bad_response` branch). Rejected: a new `HaEventResult` dataclass (a `message` field nothing
reads = dead weight + a coverage burden); returning the raw dict (leaks HA's wire shape, still unused).
The asymmetry with `HaServiceResult` (which carries `changed_states`) is justified â€” call_service callers
consume entity state; fire_event has no analogous payload.

**D-PUSHBACK-OPTIN â†’ CHOSEN: TWO gates in `accepts()` â€” global enable AND per-alert opt-in label.**
```
def accepts(self, event: AlertEvent) -> bool:
    if self._event_type == "":                          # gate 1: global enable
        return False
    return event.labels.get("push_to_ha") == "true"     # gate 2: per-alert opt-in
```
Push-back is OFF by default: the channel is globally disabled unless `event_type` is configured (env
unset â†’ `""` â†’ `accepts` always False), AND an individual alert only fires if it carries the opt-in label
`push_to_ha == "true"`. BOTH gates live in `accepts()` (NOT in `deliver()`) so a disabled/non-opted channel
produces ZERO `DeliveryResult` (the dispatcher just `continue`s â€” no "delivered to a no-op" noise), and
`deliver()` has no unreachable no-op line (100% coverage). The opt-in is a per-alert LABEL (operator adds
`push_to_ha: "true"` in the vmalert rule / Alertmanager config) â€” self-contained, no new config table, and
forward-compatible with EPIC-012 (a label is trivially a `tag_match` routing rule; this is exactly the
"tag / label" mechanism the 018 hand-off pointed at). Rejected: severity-based gate (the card says per-alert
opt-in, NOT severity â€” severity would fire for ALL error/critical, not "explicitly opted in"); a config-driven
allow-list (routing-table territory, EPIC-012 owns it, violates YAGNI).
Default `event_type`: `""` (open-source-safe OFF). Recommended value: `"homelab_monitor_alert"` (fully-qualified
to attribute the event to this service + avoid colliding with a hand-rolled `homelab_alert` automation). Env:
`HOMELAB_MONITOR_HA_EVENT_TYPE`. Label name `push_to_ha` (names the intent/action, not the destination internals).

**D-PUSHBACK-PAYLOAD â†’ CHOSEN: flat `dict[str, str]` (the fire-event body IS the event data; no title/message
envelope unlike 017's notify payload).** One shared `_build_data(event)` helper builds both firing and resolved:
`{"status": "firing"|"resolved", "fingerprint": event.fingerprint, "alertname": event.labels.get("alertname",
""), "severity": event.severity.value, "summary": event.annotations.get("summary", ""), "source_tool":
event.source_tool, "url": <deep-link>}`. `status` is the single branch point HA needs; `fingerprint` is the
stable correlation key (an automation turns OFF on resolve the scene it turned ON for the same fingerprint);
`alertname`+`severity` are what automations branch on; `summary` is human-readable for TTS; `source_tool`
tags the origin; `url` is the dashboard deep-link via `get_public_url()`. â˜… BUILD NOTE: `get_public_url()`
returns `str | None` â€” to keep the payload a clean `dict[str, str]`, OMIT the `url` key when
`get_public_url()` is None (mirror 017's deep-link guard), do NOT put `None` into a `dict[str, str]`.
Excluded (YAGNI): full labels/annotations dicts (can't be flat str values), alert_id, timestamps (HA stamps
its own event time).

**D-PUSHBACK-RESOLVED â†’ CHOSEN: fire on BOTH firing and resolved, distinguished by the `status` field.**
A resolved event closes the loop (HA turns the alarm scene back off). The channel receives both event types
from the dispatcher for free; suppressing resolved would mean ADDING a type-check to drop them (more code,
less function). Both pass the same `accepts()` gate (the resolved event carries the same `labels` including
`push_to_ha`, since Alertmanager preserves labels through to resolved). One event_type; `status` is the branch.

**D-LOOP-SAFETY â†’ CONFIRMED no loop.** This is the inverse of the loop 019 avoided: 020 fires an HA event in
response to an alert event; 019 (inbound webhook) writes incoming HA events to `audit_log` ONLY and
deliberately does NOT create alerts. So 020 â†’ HA event â†’ (an HA automation may POST back to 019) â†’ audit_log
â†’ DEAD END (no alert created â†’ nothing re-enters `dispatch()` â†’ 020 cannot re-fire). A pathological indirect
loop (020's event â†’ HA state change â†’ a vmalert rule alerts â†’ that alert tagged `push_to_ha` â†’ 020 fires
again) requires three deliberate operator config choices and is the operator's misconfiguration, not a 020
defect; the per-alert opt-in + distinct event_type make accidental loops unlikely. Noted as an operator-config
consideration, no code mitigation. Security: `fire_event` reuses `_request` so the token flows via
token_provider in the Authorization header and is NEVER logged; the token-safe raise surfaces only
`reason`+`status` (`f"ha_event delivery failed: {result.reason} status={result.status}"`), never the body/token;
the payload carries only alert metadata (no credentials).

**Build notes to confirm:** (1) the `url`/None guard above; (2) the firing-vs-resolved discriminator â€” use
