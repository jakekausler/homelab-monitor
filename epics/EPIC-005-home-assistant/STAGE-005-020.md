# STAGE-005-020: Push-back-to-HA event firer

## Status: Design (Not Started)

## Overview

The outbound-event half of bidirectional integration: when a monitor alert fires, optionally fire a
corresponding event INTO Home Assistant (`POST /api/events/<event_type>`), so HA automations can react
— run a scene, announce on speakers, flash lights. Scenario #24. Per-alert opt-in via configuration.

**Vertical slice:** after this stage, a configured alert firing causes the monitor to POST an event to
HA's event bus, which an HA automation can listen for (`trigger: event, event_type: homelab_alert`) and
act on.

**Spec references:** master spec §2 Q10 (push back to HA so HA can run scenes/announce on speakers when
alerts fire). EPIC-005 §"Bidirectional webhook". Scenario #24.

## Cross-cutting standards (from EPIC-005 §Cross-stage acceptance criteria)

- HA token never logged; HA 5xx on the event POST never crashes dispatch (best-effort, recorded as a
  failure like any channel delivery).

## Concrete deliverables

### 1. Push-back firer

A mechanism that, on a monitor alert firing (and optionally resolving), POSTs to HA's event bus:
`ctx.ha`-based call to `/api/events/<event_type>` (the REST client gets a `fire_event(event_type,
data)` method, or reuses `call_service`/a direct POST — Design confirms). The event carries the alert
fingerprint, name, severity, and a deep-link in its data so HA automations can branch. Default
`event_type` like `homelab_alert` (configurable).

### 2. Per-alert opt-in configuration

Push-back is OFF by default; it's enabled per alert (via routing config / a tag / the alert's
labels — Design decides, consistent with the minimal routing from 005-018 and forward-compatible with
EPIC-012's routing). Only alerts explicitly opted in fire HA events (no surprise automation triggers).

### 3. Tests

Mock the HA event POST → assert a firing alert with the opt-in fires the correct event payload; a
non-opted-in alert does NOT; HA 5xx is caught/recorded (no crash); token not logged. 3b: a real HA
automation listens for the event and reacts.

## Out of scope for this stage

- The inbound webhook (005-019).
- Defining the HA automations that listen (that's the user's HA config, documented as guidance, not
  shipped here).
- Full routing UI (EPIC-012) — opt-in here is minimal/config-driven.

## Decisions in this stage

### Design phase will confirm

- **D-PUSHBACK-MECHANISM** — `fire_event` method on the REST client vs `call_service` vs direct POST to
  `/api/events/<type>`.
- **D-PUSHBACK-OPTIN** — how an alert opts in (routing config / tag / label) and the default
  `event_type`.
- **D-PUSHBACK-PAYLOAD** — what alert fields the event carries (fingerprint, name, severity, deep-link).

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-PUSHBACK-OPTIN-DEFAULT-OFF** — push-back is per-alert opt-in, OFF by default (no surprise HA
  automation triggers).
- **D-BIDIRECTIONAL-COMPLETE** — push-back (out) completes the bidirectional HA decision alongside the
  webhook ingester (in, 005-019) and the push channel (005-017).
