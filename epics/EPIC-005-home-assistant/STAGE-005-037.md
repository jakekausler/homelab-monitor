# STAGE-005-037: HA entity-registry WS fetch + cache; exclude disabled/hidden from availability + z-score

## Status: Complete

## Overview

No collector today can see an entity's `disabled_by` / `hidden_by` / `entity_category` because `GET /api/states` does not carry registry metadata; only the WS command `config/entity_registry/list` does. As a result, disabled and hidden entities (which legitimately report `unavailable`) are counted as availability problems by `ha_entity_available.py`, inflating the unavailable count to ~50% of 1906 entities (measured in STAGE-005-015 Refinement 3b) and making any blanket `homelab_ha_entity_available == 0` alert (~924 firing instances) useless. This stage adds a one-time-per-refresh WS fetch of the entity registry into an in-memory cache, and uses that cache to EXCLUDE disabled and hidden entities (and OPTIONALLY `entity_category in {diagnostic, config}`) from the `homelab_ha_entity_available` family and from z-score eligibility. VM-selection/count semantics shift to "registry-visible entities only," which is precisely the dataset GAP F needs to activate availability/stale alerts. This is the KEYSTONE dependency for STAGE-005-039 (GAP F).

The registry data lives behind the WS API; the WS client (`HomeAssistantWebsocketClient.send_command`) and the live-HA enrichment-index pattern (`_ha_states_index` / `_ha_repairs_index` in the endpoint router) are the proven precedents for "fetch a WS snapshot, build an index, degrade to empty on HaError." The central open question is WHERE the registry cache lives (a kernel service shared by collectors, vs a stateful collector, vs folded into each collector) and HOW OFTEN it refreshes (registry changes rarely vs the 30s availability tick).

**Depends on:** STAGE-005-034 (soft — clean idle data co-located concern)

## Stage Flags

- Has Input Forms: No.
- Touches secrets / production data: Possibly — registry entries carry entity names/areas; the cache is in-memory and the only EMITTED change is fewer/labelled series. Apply the 031/029 never-logged posture to any registry free-text (names must NOT become metric labels or log lines).
- Host-integration concerns: Yes — live WS registry fetch against the real HA instance; collector output changes against real entity counts; Refinement 3a + 3b (3b is load-bearing: the whole point is the real ~50%-unavailable dataset).

## Concrete deliverables

1. **Registry fetch + parse helper** (kernel-side, mirror `kernel/ha/enrichment.py`): a function over `send_command("config/entity_registry/list") -> list[object] | HaError` that builds `dict[entity_id, RegistryEntry]` where `RegistryEntry` is a frozen dataclass exposing at least `disabled_by: str | None`, `hidden_by: str | None`, `entity_category: str | None` (HA registry entries carry these keys; `disabled_by`/`hidden_by` non-null ⇒ disabled/hidden). Coerce untyped values through `isinstance`-guarded readers (the `attr_str` discipline). Degrade to empty dict on HaError. NEVER log entity names.
2. **A cache + refresh mechanism** — DECISION (see Design decisions): either (a) a kernel service on `app.state` (e.g. `ha_entity_registry_cache`) refreshed on a timer / on demand and READ by collectors via `CollectorContext`, OR (b) a stateful collector that periodically refreshes its own registry snapshot and exposes it to siblings, OR (c) each collector fetches the registry itself (rejected — duplicate WS load every 30s). The cache must expose a fast `is_excluded(entity_id) -> bool` (or `entry_for(entity_id)`).
3. **`ha_entity_available.py` change** (per-state loop 110-127): after the domain allow/deny check, ALSO skip (or label) entities whose registry entry is disabled/hidden (and optionally diagnostic/config category). Net: `homelab_ha_entity_available` carries only registry-visible entities. Confirm in Design whether excluded entities are DROPPED entirely or kept with an added label (`registry_visible="0"`) so a user-rule could still target them — see D-REGISTRY-EXCLUDE-VS-LABEL.
4. **`ha_anomaly_zscore.py` change** (`_is_eligible` 65-84): add a registry-exclusion check so disabled/hidden entities never feed the rolling window. Precedence: registry-exclusion should compose with the existing `excluded_entity_ids` / `extra_entity_ids` overrides — define the order in Design.
5. **Config** (`kernel/config.py`, mirror `AnomalyZscoreConfig` / `load_anomaly_zscore_config`): a `HaRegistryConfig` (or extend `HaConfig`) with toggles: `exclude_disabled` (default true), `exclude_hidden` (default true), `exclude_categories` (default — confirm: `{}` vs `{diagnostic, config}`), and a `refresh_interval` for the cache. Open-source-safe defaults (observe-don't-surprise) — confirm defaults in Design.
6. **Self-metric** (mandatory plugin self-observation): emit a gauge for registry-cache health, e.g. `homelab_ha_entity_registry_entries` (count) + `homelab_ha_entity_registry_last_fetch_timestamp_seconds`, and an error counter on fetch failure, so a stale/failed registry fetch is observable (and so GAP F can know whether filtering is active).
7. **Graceful degradation**: registry fetch failure (HaError) → empty cache → collectors fall back to current (unfiltered) behavior (fail-OPEN: better to over-count than to silently drop everything when the registry is unavailable). Confirm fail-open vs fail-closed in Design.
8. Tests: registry parse (list shape, dict-wrapped shape, HaError → empty); `is_excluded` for disabled/hidden/diagnostic/config/visible; `ha_entity_available` excludes disabled/hidden and keeps visible (count assertion against a mixed fixture); z-score `_is_eligible` excludes disabled, honors override precedence; cache refresh cadence; never-logged sentinel for registry names; self-metric emission.
9. real-HA 3a + 3b: validate the unavailable count drops materially (target: the ~50% chronic figure shrinks toward the actionable subset) and that no genuinely-visible-but-unavailable entity is wrongly excluded.

## Decisions in this stage

### Design phase will confirm

- D-REGISTRY-CACHE-HOME — kernel service on `app.state` read via `CollectorContext` vs stateful collector vs per-collector fetch. (Lean: kernel service — both collectors need it; matches the `app.state.ha_client` / `_ha_states_index` precedent; avoids duplicate WS load.) Requires deciding how `CollectorContext` exposes the cache (new optional field, mirror `ctx.ha`).
- D-REGISTRY-REFRESH-CADENCE — how often the registry refreshes (registry changes rarely; a 30s tick is wasteful). Lean: 5-15 min, or on-demand-with-TTL, decoupled from the 30s availability tick.
- D-REGISTRY-EXCLUDE-CATEGORIES — exclude `entity_category in {diagnostic, config}` by default, or only disabled/hidden? (Diagnostic entities are often legitimately monitored — e.g. signal strength — so excluding them may hide real problems. Lean: disabled+hidden default ON, categories default OFF/configurable.)
- D-REGISTRY-EXCLUDE-VS-LABEL — DROP excluded entities from `homelab_ha_entity_available` entirely, OR keep them with a `registry_visible="0"` label so user-rules can still opt-in? (Drop = simplest + smallest cardinality + directly fixes GAP F; label = preserves opt-in targeting but keeps the noisy series. Lean: drop from the default family; this is what makes GAP F's blanket rule viable.) This decision directly shapes GAP F's expr.
- D-REGISTRY-FAIL-MODE — fail-open (empty cache → unfiltered, over-count) vs fail-closed (empty cache → drop nothing OR everything). Lean: fail-open (never silently blind the availability metric on a transient WS hiccup).
- D-REGISTRY-ZSCORE-PRECEDENCE — order of registry-exclusion vs `excluded_entity_ids`/`extra_entity_ids` in `_is_eligible`. Lean: hard-exclude (registry-disabled) wins over force-include? OR force-include wins? (A user force-including a disabled entity is an explicit override — lean: `extra_entity_ids` wins over registry-exclusion; `excluded_entity_ids` still wins over everything.)
- D-REGISTRY-CONTEXT-WIRING — how the cache reaches collectors: new `CollectorContext` field vs accessed via `ctx.ha_ws`/a provider. Must keep collectors testable (inject a fake cache).

## Design Phase

### Design Notes

KEYSTONE backend-infra stage. All 9 card deliverables are in scope; nothing deferred.

**D-REGISTRY-CACHE-HOME / D-REGISTRY-REFRESH-CADENCE → Option A (self-fetching kernel cache service).** A new `HaEntityRegistryCache` (kernel `kernel/ha/entity_registry.py`) holds a ref to the existing `ha_ws_client`, owns its own `asyncio.Task` sleep-loop (mirroring the WS client's `start_task()`/`stop_task()` + the image-events loop), fetches `config/entity_registry/list` on a cadence decoupled from the 30s tick, and parses into an IMMUTABLE `RegistrySnapshot` (frozen dataclass: `entries: dict[entity_id, RegistryEntry]`, `fetched_at: datetime | None`, `is_populated` property). Constructed in `lifespan.py` right after `ha_ws_client`, set on `app.state.ha_entity_registry`, torn down in the finally block alongside `ha_ws_client.stop_task()`. Refresh-FIRST-then-sleep so the snapshot populates ~immediately after boot. Default cadence 600s, env `HOMELAB_MONITOR_HA_REGISTRY_REFRESH_SECONDS` clamped ≥60. The cache does its OWN WS fetch (does NOT depend on a collector). Rejected Option B (stateful registry collector publishing to siblings via app.state) — introduces a novel inter-collector coupling + a cold-start ordering race + wasteful 30s no-op ticks. Rejected Option C (event-driven `entity_registry_updated` subscription) — premature for rarely-changing data; far higher test surface.

**D-REGISTRY-EXCLUDE-VS-LABEL → DROP (shapes STAGE-005-039).** Registry-excluded (disabled/hidden) entities emit NO `homelab_ha_entity_available` series at all (the exclusion gate `continue`s after the domain allow/deny check, before the availability computation). Rejected LABEL (`registry_visible="0/1"`) because the downstream STAGE-005-039 `HaEntityUnavailable` consumer is an UNFILTERED `homelab_ha_entity_available == 0` expr — per the locked project lesson "when the downstream query is unfiltered, the collector's emit condition IS the filter; prefer the positive whitelist gate over a label." DROP solves the measured ~924-firing-instance cardinality problem at source and keeps 039's rule maximally simple (no label any future query can forget). 039 inherits the bare rule unchanged; if anyone ever switches 037 to LABEL, 039's expr must gain `{registry_visible="1"}` — to be documented in 039.

**D-REGISTRY-EXCLUDE-CATEGORIES → only `disabled_by`/`hidden_by`; `entity_category` exclusion OFF by default** (env-tunable `HOMELAB_MONITOR_HA_REGISTRY_EXCLUDE_CATEGORIES`, ships empty frozenset). Diagnostic entities (battery, signal strength, uptime) are frequently the things worth monitoring; excluding them by default would blind legitimate alerts.

**D-REGISTRY-ZSCORE-PRECEDENCE.** In `ha_anomaly_zscore.py` `_is_eligible`, the new order is: (0) `extra_entity_ids` force-include wins over ALL (explicit operator intent beats automatic registry inference) → (1) registry-exclude when `snapshot.is_populated` → not eligible → (2) `excluded_entity_ids` hard-exclude → not eligible → (3) measurement/device-class heuristic. z-score family also DROPs registry-excluded entities (consistent with availability; eligibility is an internal whitelist gate, no unfiltered-consumer tension).

**D-REGISTRY-FAIL-MODE → fail-OPEN, gated on `snapshot.is_populated`.** A never-populated or failed-fetch snapshot applies NO exclusion in either collector — the metric stays unfiltered, NEVER silently blinded. Observability via mandatory self-metrics emitted from the cache loop: `homelab_ha_entity_registry_entries` (gauge), `homelab_ha_entity_registry_last_fetch_timestamp_seconds` (gauge), and a fetch error counter (`homelab_ha_entity_registry_fetch_total{result="ok|error"}` or equivalent) — so a stale/failed registry fetch is directly observable and a future alert can fire on it.

**D-REGISTRY-CONTEXT-WIRING.** Add one optional field `ha_registry: HaEntityRegistryCache | None = None` to `CollectorContext` (declared AFTER `ha`, preserving the slots default-ordering rule). `ctx_factory` in `lifespan.py` sets it from `app.state.ha_entity_registry`. Tests inject via the existing `SimpleNamespace` pattern with a fake exposing `.snapshot()`.

**Module split (preserves enrichment.py purity).** Pure parse helpers `extract_registry` + `build_registry_index` go in `kernel/ha/enrichment.py` next to `build_repairs_index` (isinstance-guarded, never-log, never-emit, never-raise). The stateful cache + self-metrics + ws_client ref live in a NEW `kernel/ha/entity_registry.py` (kernel-only, NO plugin imports). `RegistryEntry` is a frozen dataclass exposing `entity_id`, `disabled_by: str | None`, `hidden_by: str | None`, `entity_category: str | None`.

**Config.** New `HaRegistryConfig` (frozen slots) + `load_ha_registry_config()` mirroring `AnomalyZscoreConfig` / `load_anomaly_zscore_config`, env prefix `HOMELAB_MONITOR_HA_REGISTRY_*`, with toggles `enabled` (default true — a kill-switch), `exclude_disabled` (default true), `exclude_hidden` (default true), `exclude_categories` (default empty frozenset), `refresh_seconds` (default 600, clamp ≥60).

**Privacy.** Registry entity NAMES must NEVER become metric labels or log lines — a sentinel test will lock this (mirrors the 031/enrichment never-logged posture).

**OpenAPI.** No API response-model change (config + self-metric + collector-internal only) → expected NO OpenAPI regen; confirm in Build.

**Contract verified (Design):** the WS command `config/entity_registry/list` is issued via the proven `ha_ws_client.send_command(...)` (returns `list[object] | HaError`, mirrors the repairs `send_command("repairs/list_issues")` caller). HA registry entries carry `disabled_by` / `hidden_by` / `entity_category` keys (non-null disabled_by/hidden_by ⇒ disabled/hidden). The collector gate location (`ha_entity_available.py` after the domain check, before availability emit) and the `_is_eligible` precedence (`ha_anomaly_zscore.py`) were confirmed against the real source.

## Refinement Phase

### Refinement Notes

Backend stage with host-integration concerns — real-HA validation (3a synthetic subsumed by Build's unit suite; 3b against the live HA instance ~3031 registry entries / ~1906 states). No viewport prompt.

**Two real-HA bugs found and fixed during Refinement (each re-verified with a full `make verify` + rig redeploy):**

1. **Registry-cache startup race.** The cache's refresh-first-on-start ran ~2s after boot, before the WS client finished connect+auth → `send_command` returned `HaError(unreachable)` → the loop then waited the full `refresh_seconds` (600s) before retrying, leaving the registry empty (exclusion fail-open/inert) for up to 10 min after every boot. FIX (`kernel/ha/entity_registry.py` `_run_loop`): retry on a short initial backoff (2s→4s→…cap 30s) UNTIL the first populated fetch, then settle into the normal cadence; new test `test_run_loop_retries_on_initial_backoff_until_populated` covers the branch.

2. **WS frame-size limit (the load-bearing bug).** `config/entity_registry/list` for a real ~3031-entry registry exceeds the `websockets` library's default 1 MiB `max_size` → HA sends a 1009 "message too big" close → connection drops → registry NEVER populates on any real HA. FIX (`kernel/ha/websocket.py`): added `_WS_MAX_FRAME_BYTES = 32 MiB` (Final constant) and passed `max_size=_WS_MAX_FRAME_BYTES` to `connect()` — keeps a finite OOM guard (not `max_size=None`). This is a shared-WS-client change but is in scope for 037 (the keystone that introduces the first large WS payload). Updated `test_default_connect_uses_websockets_connect` to assert the kwarg.

**Real-HA 3b validation (DECISIVE, after both fixes deployed):** registry healthy — `homelab_ha_entity_registry_entries` = 3031, `fetch_total{result="ok"}` = 1 (and `error` = 1 from the one startup-race miss, NOT climbing), recent `last_fetch_timestamp`; `homelab_ha_websocket_connected` stable at 1 (no 1009 drop). Exclusion ACTIVE and de-noising real data: `count(homelab_ha_entity_available)` stepped 1138 → 1116 (−22 disabled/hidden entities that leaked from `/api/states` now DROPPED), `count(== 0)` 202 → 198; `count(homelab_ha_entity_last_changed_seconds)` == 1116 (identical cardinality → confirms DROP, not value-flip); the step-down was temporally aligned (~37s) with the registry populating; the authenticated `/entities?filter=unavailable` endpoint returned total=198, matching VM exactly.

**Note on magnitude:** the de-noise from 037's exclusion alone is modest (−22 series) because the much larger 1907→1138 reduction was an independent HA-side entity cleanup at ~20:05 UTC (not 037). The slate spec's "~50% of 1906" estimate dates from the STAGE-005-015 measurement era when the unavailable population was far larger; the current real HA simply has fewer disabled/hidden-AND-in-states entities. The exclusion mechanism is verified correct — it drops exactly the disabled/hidden entities present.
