# STAGE-005-015: vmalert-METRICS rules

## Status: Complete

## Overview

Ship the default vmalert-METRICS rule set for the HA collector metrics (Waves B). This is where most
of the "HA health" scenarios become actual alerts. One rule file covering: entity-unavailable-for-long
(#1), stale/frozen-entity per device-class (#2), battery 20%/10% (#5), update-available roundup (#13),
device-down rollup (#7), HA reachability `homelab_ha_up` (#18), anomaly z-score (#4), automation
idle-too-long (#16), and automation-disabled-unexpectedly (#17).

**Vertical slice:** after this stage, the HA metrics emitted in Waves B drive real alerts — e.g. a
battery dropping below 10% pages, an entity unavailable for >N minutes fires, a frozen freezer sensor
fires the staleness rule (which z-score can't catch), and HA being unreachable fires `homelab_ha_up`.

**Spec references:** master spec §6.2 (HA metrics), §17 (vmalert rules under `deploy/vmalert/metrics/`),
§8.2 (severity). EPIC-005 §"Stage decomposition" Wave C. Scenarios #1,#2,#4,#5,#7,#13,#16,#17,#18.

## Concrete deliverables

### 1. `deploy/vmalert/metrics/home-assistant.yaml`

A rule group covering:

- **Entity unavailable for long** (#1) — `homelab_ha_entity_available == 0 for Nm`. Severity scaled by
  domain/importance where feasible.
- **Stale/frozen entity** (#2) — `homelab_ha_entity_last_changed_seconds > threshold`, per device-class
  (a thermostat vs a flood sensor have different expected change cadences — Design provides a
  per-class default map). This catches the zero-variance "frozen sensor" case z-score cannot.
- **Battery low** (#5) — `homelab_ha_battery_level < 20` (warning) / `< 10` (error).
- **Update-available roundup** (#13) — a digest-style/grouped rule on `homelab_ha_update_available == 1`
  (info/warning; likely grouped, not one-per-update, to avoid 41 separate alerts).
- **Device-down rollup** (#7) — group `homelab_ha_entity_available == 0` by device (via the device
  label from 005-006/007) so "device X down" fires once instead of 8 entity alerts.
- **HA reachability `homelab_ha_up`** (#18) — HA itself unreachable (the collector can't reach
  `/api/config`); first-class `critical` alert. Confirm the metric is emitted (005-003 ping seed or
  add it here).
- **Anomaly z-score** (#4) — `abs(homelab_ha_entity_value_zscore) > N for Mm`.
- **Automation idle-too-long** (#16) — `homelab_ha_automation_last_triggered_seconds >
  expected_max_idle` for automations with a configured expectation (from 005-009).
- **Automation disabled unexpectedly** (#17) — `homelab_ha_automation_enabled == 0` for automations
  expected on (low severity; likely opt-in to avoid noise).

### 2. Test mirror + tuning

Shortened-window mirror under `deploy/vmalert/metrics/__tests__/`; integration tests plant metric
values to fire each rule and confirm baseline silence. Tune `for`/threshold against the real scope in
3b.

## Decisions in this stage

### Design Decisions (RESOLVED)

**D-DEVICE-ROLLUP (#7) → CHOSEN: DEFER the device-down rollup.** No `device` label exists on any HA metric today (verified across all collectors: entity-available, battery, etc. carry only `entity_id`/`domain`). A true group-by-device rule requires sourcing the entity→device map from HA `core.device_registry` via WS plus a `CollectorContext` change — a multi-file Python collector facility, which is the wrong nature for a static-YAML rules stage, and inserting a new lower-numbered stage mid-epic is disruptive. Instead, 015 ships the per-entity `HaEntityUnavailable` rule (`homelab_ha_entity_available == 0 for: 5m`) and relies on Alertmanager's existing `group_by: ['alertname','target_kind']` to collapse simultaneous entity outages into a single notification (~80% of the rollup's operator value at 0% risk). The `home-assistant.yaml` header documents that the true device rollup requires the deferred device-label facility (a future EPIC-005 stage, numbered after 015). Rejected: (A) build the device_registry WS facility in 015 (scope/nature violation); (B) group by `domain` (too coarse — all sensors share one domain).

**D-STALE-PER-CLASS-MAP (#2) → CHOSEN: OPT-IN commented exemplar, no default-on rule.** `device_class` (the only axis on which a staleness default could be safe) is NOT a metric label; `homelab_ha_entity_last_changed_seconds` carries only `{entity_id,domain}`, and `domain` is too coarse (a flood `binary_sensor` legitimately never changes for weeks while a motion `binary_sensor` changes constantly). Any default rule fires blind and would be noisy — violating the "a noisy rule is a failed rule" acceptance criterion. The frozen-freezer case is inherently per-entity (you must know which sensor and its legitimate cadence), which is exactly the 005-005 user-rule machinery's job. Ship a commented `HaEntityStale` exemplar in `home-assistant.yaml` scoped to a single example entity, pointing the user at 005-005 for per-entity scoping. Rejected: (A) single global threshold (useless for frozen-freezer); (B) per-domain thresholds (domain too coarse).

**D-UPDATE-ROUNDUP-SHAPE (#13) → CHOSEN: single aggregated digest.** `expr: count(homelab_ha_update_available == 1) > 0`, `severity: info`, `for: 15m`, count surfaced via `{{ $value }}` in the annotation. `count()` collapses all update-available series to one labelless value → exactly ONE alert instance regardless of how many entities have updates (avoids 41 separate alerts). Rejected: per-entity + Alertmanager grouping (still 41 alert instances); count-threshold > N (a single pending update is still worth a low-priority digest).

**D-HA-UP-METRIC (#18) → CHOSEN: confirmed emitted; rule `homelab_ha_up == 0 for: 2m` → critical.** `homelab_ha_up{}` (no labels) is emitted by the HaUp collector (`ha_up.py:46`), 1=reachable/0=down — NOT seeded by 005-003. `for: 2m` rides out routine HA restarts (typically <60s) and single-scrape misses while still paging within ~2min of a real outage.

**D-IDLE-DISABLED-OPT-IN (#16/#17) → CHOSEN: both OFF (commented exemplars) with default thresholds encoded.** Automation idle-too-long (#16, `homelab_ha_automation_last_triggered_seconds > threshold`) and automation-disabled-unexpectedly (#17, `homelab_ha_automation_enabled == 0`) are noisy by nature — seasonal automations are legitimately idle for months, and disabling an automation is almost always an intentional user action. Per 005-009's locked decision, the default idle threshold lives in 015's YAML (satisfied by a commented exemplar carrying the default value, e.g. `> 86400`); per-automation overrides come from the 005-005 user-rule machinery. Shipping these default-on would violate "a noisy rule is a failed rule." Rejected: both default-on; #16-on/#17-off (7d idle still false-fires on seasonal automations).

**Smaller decisions:**
- **Config-entry STATE rule → INCLUDE** (`HaConfigEntryError`, `homelab_ha_config_entry_setup_error == 1 for: 5m`, severity error). Distinct alertname + `source_tool: vmalert-metrics` (vs 014's `HaIntegrationSetupFailure`/`vmalert-logs`) — no Alertmanager dedup collision. This is the persistent-STATE counterpart to 014's setup-failure EVENT; `for: 5m` is the EVENT-vs-STATE differentiator (014 fires immediately on transient `ConfigEntryNotReady`, this waits for the state to persist). Uses `setup_error == 1` not `loaded == 0` (the latter conflates intentional unloads with failures).
- **Critical-repairs rule → SKIP default** (repairs are panel-surfaced; `homelab_ha_repair_issue`'s `severity` label is HA's own taxonomy, not the project's locked vocab). Leave a one-line comment noting repairs are panel-only.
- **Battery #5 → TWO band-guarded rules.** `HaBatteryCritical` (`homelab_ha_battery_level < 10 for: 10m`, error) and `HaBatteryLow` (`homelab_ha_battery_level >= 10 and homelab_ha_battery_level < 20 for: 10m`, warning). The `>= 10 and < 20` band on the warning prevents a dying battery from firing BOTH alerts. `for: 10m` rides out jittery battery reads.
- **File structure → single `deploy/vmalert/metrics/home-assistant.yaml`** with groups `home_assistant_availability`, `home_assistant_health`, `home_assistant_updates`, `interval: 30s`, NO `type: vlogs` (MetricsQL against the VictoriaMetrics datasource). 016's safety rules slot in as a `home_assistant_safety` group (same file).
- **Test format → vmalert unit-test fixtures** at `deploy/vmalert/metrics/__tests__/home-assistant.tests.yaml` (`rule_files: [../home-assistant.yaml]`, `input_series` with `values: 'N+0xM'`, `alert_rule_test`/`exp_alerts`). NOT a shortened-window mirror — plants series and asserts firing against the production rule directly. Fire-on-planted + no-fire-on-baseline cases per default-on rule.
- **Labels (all rules):** `severity`, `source_tool: vmalert-metrics`, `target_kind: container`, `integration: home_assistant` (matching 014's conventions with vmalert-metrics).

### Locked Design Decisions (inherited from brainstorming 2026-06-10)

- **D-ALL-25-COVERED** — EPIC-005 covers all 25 scenarios; this stage is where the bulk of the metric-
  based ones land.
- **D-STALE-NOT-ZSCORE** — frozen/stuck sensors (#2) are caught by a staleness rule here, NOT by
  z-score (#4); the two are complementary.
- **D-HA-UP-FIRST-CLASS** — "HA is down" (#18) is a first-class `critical` rule, not just a quarantined
  collector (which also leans on EPIC-003 Docker monitoring of the HA container).
